## 🎬 开场：一个引子（3 min）

> **主讲人说**：  
> "假设我们要开发一套企业运营体系——管理市场、研发、生产、售后和经营。几十个人分工协作，做一两年。  
> **问题来了**：
> 1. 每个人的工作怎么对齐？
> 2. 今天设计的东西，三个月后还能追溯到当初为什么要这么做吗？
> 3. 新人加入，怎么快速理解整套设计？

> 传统的做法：画一堆 PPT、写一堆 Word 文档，然后它们迅速过时，谁也说不清全貌。  
> 这个工程就是为解决这个问题而生的——**用一套结构化的方法，把系统设计变成一条流水线**。"

> **抛出今天的三问**（写在白板上）：
> - 这个仓库里到底有几样东西？
> - "五层"是哪五层？为什么要五层？
> - 我做为团队一员，我的工作对应哪里？

## Part 1：基本概念澄清（18 min）

### 1.1 先统一语言：几个你必须懂的关键词（5 min）

> **讨论方式**：主讲人先提问，引导团队自己说，然后给出精确定义。

**Q1：什么是"需求"？什么是"方案"？**

> 先让团队成员用自己的话回答。

- **需求（Requirement）**：系统应该做什么——What。  
  *例："系统必须支持用户登录"*
- **方案（Solution）**：怎么实现需求——How。  
  *例："采用 JWT 令牌认证，前端弹出登录页，后端验证用户名密码"*
- **核心关系**：需求是对方案的要求，方案是对需求的满足。

✅ **关键认知**：**同一个文档，对上层来说是"方案"，对下层来说可能是"需求"**。这就是"文档的双重身份"——下面会反复提到。

**Q2：什么是"架构末级节点"？**

> 比喻：一棵树，从树干分到树枝、树枝分到小枝，**最小的那个枝条末端**就是"末级节点"。

- **架构定义** = 整棵树的形状（树干→树枝→小枝）
- **架构末级节点** = 最小的枝条末端（不再继续分叉）
- **详细定义** = 这根枝条上长了多少片叶子（对末级节点的展开）

✅ **关键认知**：需求映射只发生在**末级节点**层面，不发生在中间层。

**Q3：什么是产品开发四阶段？**

| 阶段 | 名字 | 一句话 |
|------|------|--------|
| 阶段1 | **系统设计** | 从需求推导出产品架构（我们正在做的） |
| 阶段2 | **构件开发** | 按架构去开发每个构件 |
| 阶段3 | **集成交付** | 把构件拼起来，交付给用户 |
| 阶段4 | **系统运维** | 上线后维护 |

> **注意**：这个工程**只覆盖阶段1（系统设计）**——因为系统设计是产品开发中最重要、也最难的一步。架构没设计好，后面全是白搭。

### 1.2 三种设计场景的直觉理解（3 min）

> 不用记细节，先建立直觉。

| 场景 | 什么时候用 | 举个生活中的例子 |
|------|-----------|----------------|
| **瀑布式** | 全新开始，需求清楚 | 你要建一栋新房，图纸全部画好再施工 |
| **敏捷式** | 已有系统，加新功能 | 你的房子已经住进去了，现在要加一个阳光房 |
| **逆向工程** | 房子建好了，但图纸丢了 | 你要买一栋二手房，但前任房主没给图纸，你找人重新测绘 |
| **DevOps** | 出问题了，快速修复 | 屋顶漏水了，赶紧补上，不用重新设计整栋房子 |

> **讨论点**：让团队想想自己工作中遇到的场景，分别对应哪种？

### 1.3 三个关键角色（2 min）

| 角色 | 一句话 | 在这个工程里做什么 |
|------|--------|------------------|
| **流水线开发者** | 造工具的人 | 设计准则、写指南、定义 AI 怎么工作 |
| **AI** | 自动化执行的人 | 按照任务定义自动生成文档、检查一致性 |
| **AOS 开发者** | 使用工具的人 | 用这套方法去设计企业运营体系 |

> **关键问题**：**我们团队目前是哪个角色？**  
> → 答案是：流水线开发者（我们正在造这条"流水线"本身）。

### 1.4 最容易混淆的一对概念：功能需求 vs 非功能需求（2 min）

- **功能需求（FR）**：系统"做什么"——用户登录、生成报表、发送邮件
- **非功能需求（NFR）**：系统"做得怎么样"——登录要在 1 秒内完成、系统 99.9% 时间可用

**为什么非功能需求要走单独的路径？**  
因为非功能需求**不经过业务架构**（业务架构只描述业务流程，不描述系统性能）。所以：

```
功能路径：SR → BA → SysReq → PA
非功能路径：SR-NFR → SysReq-NFR → PA（跳过 BA）
```

> ✅ **每条需求都由 "需求描述 + 性能指标" 两部分组成**——功能需求有性能指标，非功能需求也有。

### 1.5 一个常见困惑：为什么产品L止于文档，产品M必须是信息系统？（3 min）

> **讨论方式**：抛出问题让大家先猜，再揭示答案。

**问题**：产品L和产品M用的是同一套五层方法论，为什么产品L的产品架构（PA）是"准则、指南、AI辅助、任务定义"这些文档，而产品M的PA必须是"前端、后端、数据、基础设施"这些软件构件？

**答案**：不是因为方法论不同，而是因为**产品类型和消费方式不同**：

```
产品L → 知识/方法类产品 → 用户是系统设计人员（人+AI）
     → 消费方式：阅读→理解→遵循
     → PA 形态：文档集 + AI Skill 定义就够了

产品M → 企业管理信息系统 → 用户是企业员工
     → 消费方式：运行→交互→执行业务操作
     → PA 形态：可运行的信息系统
```

**产品形态连续谱**：从纯知识产品到纯软件产品之间存在混合形态。产品L不是"不能"做成信息系统（完全可以做成低代码设计平台），而是当前阶段"文档+AI"对运营体系开发者**刚好够用**。未来可根据需要沿谱右移。

> **一句话记住**：**消费方式决定产品形态**——用户怎么"用"，决定了产品长什么样。

### 1.6 再深一层：信息处理类型决定消费方式（3 min）

> **"消费方式决定产品形态"是对的。但什么决定了消费方式？——业务的信息处理类型。"**

**13种信息处理模式**（取代原有的五分类，更精确地刻画业务的信息原子）：

前9类为**跨领域通用**的处理模式：

| 模式 | 核心特征 |
|------|---------|
| **计划事务** | 基于约束条件生成资源/时间分配方案 |
| **工单事务** | 创建→派发→执行→反馈→关闭的闭环处理 |
| **审批流程** | 按规则链路由、汇集决策意见、形成审批结论 |
| **合规事务** | 对照规则库检查行为/数据、判定合规性、记录证据链 |
| **规范知识** | 结构化编撰准则/规程/指南，供人员或AI遵循 |
| **指标看板** | 从多源数据聚合指标并以可视化形式呈现 |
| **进展追踪** | 跟踪状态变更、比对计划与实际、标识偏差 |
| **信息台账** | 对信息进行登记、维护、检索和状态管理 |
| **xBOM** | 管理产品结构及配置关系，追踪物料清单的版本和变更 |

后4类为**领域特有**的处理模式——不同领域有各自的分析方法和专用算法：

| 模式 | 核心特征 |
|------|---------|
| **设备IO** | 与物理设备进行实时数据交换，采集和控制IO信号 |
| **领域分析** | 特定领域公式/模型驱动的计算分析 |
| **领域建模** | 构建领域模型、运行仿真场景、输出预测结果 |
| **领域算法** | 封装特定领域逻辑的算法执行 |

**判断原则**：不同模式→分离；同模式→可共享。但后4类即使模式相同，如果领域不同，仍然需要分离。

> **实践验证**：前9类跨领域通用的处理模式可以统一到同一个信息化平台中。

**回到产品L和产品M**：

```
产品L → 知识创作型 → 用户阅读遵循 → 文档形态     ← 消费方式匹配信息本质
产品M → 混合型       → 多种消费方式  → 信息系统
                                  → 内部还需进一步分解为多子系统
```

> **产品L为什么止于文档？——不是"恰好不需要做成交互系统"，而是它的信息本质就是知识创作型，原生输出形态就是文档。**

## Part 2：三层工作层面全景（10 min）

> **讨论方式**：在白板上画三个圈，一层层展开。

### 2.1 先画出两个产品和两个团队

```
                    ┌──────────────────────┐
                    │      团队 B          │
                    │  (流水线开发者 + AI)  │
                    └────────┬─────────────┘
                             │ 开发
                             ▼
                    ┌──────────────────────┐
                    │   产品 B              │
                    │   系统设计流水线       │
                    │   (设计工具)           │
                    └────────┬─────────────┘
                             │ 交付工具
                             ▼
                    ┌──────────────────────┐
                    │   产品 A              │
                    │   AOS 企业运营体系     │
                    │   (目标系统)           │
                    └──────────────────────┘
                             ▲
                             │ 使用工具设计
                    ┌────────┴─────────────┐
                    │      团队 A          │
                    │   (AOS 开发者)         │
                    └──────────────────────┘
```

### 2.2 再展开为三层工作层面

**第1层：自指设计（Self-Referential Design）**

> "运营体系包含设计自身的工具，产品L进行自我设计。"

- 团队L拿着五层结构，对产品L自身的需求做系统设计
- 产出：产品L的9份产品数据文档（原始需求 → 产品架构）
- 作用：验证方法论是否可行，同时把工具先造出来

**第2层：产品L的交付物构建**

> "产品L造完后，交付给团队M的是什么？"

- 不是代码、不是软件——而是**准则、指南、AI辅助文档、任务定义**
- 这些东西放在 `50-aos-design/` 下（因为它们是团队M要用的工具）
- 这就是产品L的产品架构末级节点对应的**实际交付物**

**第3层：使用产品L设计产品M**

> "团队M拿到了工具，开始设计企业运营体系。"

- 使用第2层提供的准则和指南
- 按第2层提供的任务定义操作
- 产出：产品M的9份产品数据文档（放在 `10-aos-system-product-data/`）

### 2.3 一个比喻帮大家记住

> **产品L = 建造"设计工具箱"**  
> **第1层：造箱子的过程**（团队L自己设计箱子的图纸）  
> **第2层：把各类工具放进箱子**（准则、指南、AI辅助、任务定义）  
> **第3层：团队M拿到工具箱，用里面的工具去设计产品M**

### 2.4 三个视角看同一件事（2 min）

> **关键认知**：我们前面说的"产品关系"、"三层工作层面"、"自指关系"——不是三件事，是**同一件事的三个互补视角**。

| 视角 | 焦点 | 一句话 |
|------|------|--------|
| **产品关系**（静态） | "谁是什么" | 产品L是设计工具，产品M是目标系统 |
| **三层工作层面**（动态） | "谁在做什么" | 团队L在第1层自指设计、第2层构建交付物；团队M在第3层使用工具 |
| **自指关系**（递归） | "谁包含谁" | 运营体系包含设计自身的规则和工具——产品L是支撑"设计运营体系本身"这项业务的工具 |

> **推论**：开发产品L所用的准则、规范、流程自然地适用于开发整个运营体系——因为运营体系的设计本身，就是运营体系包含的一项业务。这不是巧合，而是自指的必然结果。

## Part 3：五层结构方法论（14 min）

> 这是最核心、也最容易被绕晕的部分。  
> **建议**：在白板上一步步画出五层结构和映射关系。

### 3.1 五层全景图（5 min）

> **主讲人一边画一边讲**：

```
第1层：原始需求 (OR)
    │  收集、规范化、分解到末级
    ▼
第2层：相关方需求 (SR)
    ├── 功能部分 (SR-F)
    │       │
    │       ▼
    │  第3层：业务架构 (BA)  ← 只有架构（IPO清单），没有详细定义
    │       │
    │       ▼
    ├── 第4层：系统需求 (SysReq)
    │   ├── 功能部分 (SysReq-F)  ← 承接 BA 的 IPO
    │   └── 非功能部分 (SysReq-NFR) ← 直接承接 SR-NFR（不经过 BA）
    │       │
    │       ▼
    └── 第5层：产品架构 (PA)
         └── 产品构件清单 ← 阶段2 构件开发的输入
```

**每一层的文档构成**：

```
第2层 SR：架构定义（树形结构）+ 详细定义（对每个末级节点的展开）
第3层 BA：只有架构定义（IPO 清单），没有详细定义  ★特例★
第4层 SysReq：架构定义（树形结构）+ 详细定义（用例分解等）
第5层 PA：架构定义（构件树）+ 详细定义（构件的实现方案）
```

> **讨论**：为什么 BA 层不需要详细定义？  
> → 因为 BA 描述的是"业务流程"，IPO 本身就是最小单元。它的每个 IPO 直接映射到 SysReq 的一个架构末级节点，不需要再分解。

### 3.2 三条核心规则（3 min）

> **这三条规则是一切设计的"交通规则"——写在了通用准则里。**

**规则1：1:1 分配约束**
- 每条详细定义末级节点 → 只能分配到 **唯一** 一个下层架构末级节点
- *类比：每个人只能有一个直属上级*

**规则2：N:1 承接支持**
- 一个下层架构末级节点 → 可以承接 **多条** 上层详细定义末级节点
- *类比：一个经理可以带多个下属*

**规则3：双向追溯**
- 从需求可以找到分配给了哪个方案节点（正向）
- 从方案节点可以找到承接了哪些需求（反向）
- *类比：你能从员工找到经理，也能从经理找到所有下属*

### 3.3 组织资产优先原则（1 min）

> **新增一条设计原则——不是方法变了，而是多了一条决策红线。**

**核心优先级**：
```
第1优先 ── 复用（Reuse）
第2优先 ── 改进（Improve：Extend/Split/Merge）
第3优先 ── 新增（Add）
特殊情况 ── 废弃（Deprecate，标注但不删除）
```

**一句话**：每次要新增架构节点前，先问自己三遍——"现有的能不能改一改就用？"

**为什么要这条原则**？每个架构节点都是已投入设计工作的组织资产。动辄新增会导致架构膨胀和成本失控。信息处理模式是不可逾越的边界——不同模式的需求不能硬塞进同一节点。

### 3.4 开发六步法（3 min）

| 步骤 | 做什么 | 输入 | 输出 |
|------|--------|------|------|
| Step 1 | 分析原始需求，定义 SR 架构 | 原始需求原文 | 原始需求详细定义 + SR 架构定义 |
| Step 2 | SR 详细定义 + BA 架构 | SR 架构 | SR 详细定义 + BA 的 IPO 清单 |
| Step 3 | BA → SysReq 功能部分 | BA 的 IPO 清单 | SysReq 功能架构 |
| Step 4 | SR-NFR → SysReq-NFR | SR 非功能需求 | SysReq 非功能架构 |
| Step 5 | SysReq 详细定义 + PA 架构 | SysReq 架构 | SysReq 详细定义 + 产品架构 |
| Step 6 | 双向追溯验证 | 全部前序产出 | 追溯矩阵 + 验证报告 |

> **关键认知**：六步法不是串行的——**Step 2 和 Step 1 的后半段是同步进行的**（同步设计原则）。先快速建立结构骨架，再并行填充内容。

### 3.5 "同构性"——最重要的一个概念（1 min）

> **产品同构性**：产品L和产品M用的**是同一种设计方法**，所以它们的文档结构长得一样。

```
产品L 的产品数据 → 01-OR, 02/03-SR, 04-BA, 05/06-SysReq, 07-PA, 08-追溯, 09-验证
产品M 的产品数据 → 01-OR, 02/03-SR, 04-BA, 05/06-SysReq, 07-PA, 08-追溯, 09-验证
```

**同构 ≠ 相同**——结构一样，但内容完全不同。一个是"流水线"，一个是"运营体系"。

**为什么同构性重要**？因为产品L的准则、指南可以**直接参考**给产品M用，不需要从零开始。

### 3.6 PA 分解的三个判断因素（2 min）

> **讨论**：产品M包含多种业务（订单、生产、设计、分析……）。什么时候应该放在同一系统，什么时候必须分开？

**三个因素共同决定**：

| 因素 | 共享的条件 | 分离的条件 |
|------|-----------|-----------|
| **① 信息模型内聚性** | 操作同一组核心实体（共享主数据） | 实体同管理目的不同（设计BOM vs 生产BOM） |
| **② NFR Profile** | 响应时间、可用性等约束相容 | 毫秒级产线 vs 分钟级分析 → 冲突 |
| **③ 信息处理类型** | 同类型（计划事务+工单事务） | 不同类型（规范知识 vs 工单事务 → 信息原子不同） |

> **核心认知**：PA 分解不是从零设计，而是读取 BA 层已经隐含的业务边界。架构质量取决于能否读出这些自然边界，而非人为统一。

## 🎯 总结与 Q&A（预留缓冲时间）

### 五个要点回顾

1. **这个工程在做什么？** — 造一条"系统设计流水线"，让系统设计从手工作坊变成工业化生产
2. **凭什么造？** — 五层结构（OR→SR→BA→SysReq→PA）+ 三条规则（1:1、N:1、双向追溯）+ **组织资产优先原则**（复用 > 改进 > 新增）
3. **产品M和产品L为什么长不一样？** — 产品类型不同（知识工具 vs 信息系统），消费方式不同，所以PA形态不同。方法相同，形态不同。
4. **为什么产品L止于文档就够了？** — 信息本质是知识创作型，原生输出就是文档。不是"不能"，而是"刚好够用"。
5. **我们是谁？** — 团队L（流水线开发者），当前在做第1层（自指设计）和第2层（交付物构建）

### 让大家带走的一句话

> **"运营体系包含设计自身的规则和工具。它用五层结构先自指设计了自身，再以同构性去支撑产品M的设计。方法相同，形态不同。信息本质决定了消费方式，消费方式决定了产品长什么样。这个仓库里，一半是'造工具的过程'，一半是'工具交付后的使用结果'。"**
