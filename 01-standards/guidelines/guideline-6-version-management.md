# 准则 6：版本管理和变更控制

**目的**：建立统一的版本管理和变更控制规范，确保设计文档的可控性和可追溯性

**版本**：v1.0  
**最后更新**：2026-05-12

---

## 一、版本管理规范

### 1.1 版本号格式

**格式**：`v[主版本].[次版本]`

**示例**：
- v1.0 - 初始版本
- v1.1 - 小的改进
- v2.0 - 重大变化

### 1.2 版本号更新规则

| 情况 | 主版本 | 次版本 | 示例 |
|------|--------|--------|------|
| 初始发布 | 1 | 0 | v1.0 |
| 小的改进或修复 | 不变 | +1 | v1.0 → v1.1 |
| 重大功能变化 | +1 | 0 | v1.x → v2.0 |
| 相关方需求有重大变化 | +1 | 0 | v1.x → v2.0 |
| 架构有重大调整 | +1 | 0 | v1.x → v2.0 |

**判断标准**：

**主版本更新**（v1.0 → v2.0）：
- ✅ 相关方需求有重大变化（新增/删除/修改关键需求）
- ✅ 业务架构有重大调整（新增/删除/修改关键组件）
- ✅ 系统架构有重大变化（新增/删除/修改关键模块）
- ✅ 产品架构有重大重构

**次版本更新**（v1.0 → v1.1）：
- ✅ 相关方需求有小的改进（新增/修改非关键需求）
- ✅ 业务架构有小的调整（新增/修改非关键组件）
- ✅ 系统需求有小的改进（新增/修改非关键需求）
- ✅ 产品架构有小的优化

## 二、变更控制规范

### 2.1 变更类型

| 变更类型 | 定义 | 示例 |
|---------|------|------|
| **新增** | 添加新的需求、架构、组件等 | 新增 SR-F-001 |
| **修改** | 修改现有的需求、架构、组件等 | 修改 SR-F-001 的描述 |
| **删除** | 删除现有的需求、架构、组件等 | 删除 SR-F-002 |
| **重构** | 重新组织或重新设计 | 重构业务架构 |
| **优化** | 改进性能、可维护性等 | 优化产品架构 |
| **修复** | 修复错误或不一致 | 修复映射关系错误 |

---

### 2.2 变更记录格式

**changelog.md 格式**：

```markdown
# 变更日志

## v1.1 (2026-05-15)

### 新增
- SR-F-006: 新增需求规范化检查清单
- BA-NODE-005: 新增业务组件

### 修改
- SR-F-001: 更新需求描述
- BA-NODE-001: 调整业务流程

### 删除
- SR-F-003: 删除已过时的需求

### 修复
- 修复 SR → BA 映射关系错误
- 修复系统需求文档格式问题

### 优化
- 优化产品架构性能
- 改进系统需求的清晰性

---

## v1.0 (2026-05-12)

### 新增
- 初始版本
- 相关方需求定义
- 业务架构设计
- 系统需求规格
- 产品架构设计
```

**变更记录表格式**：

```markdown
| 版本 | 变更日期 | 变更类型 | 变更内容 | 影响范围 | 编制人 |
|------|---------|---------|--------|--------|------|
| v1.1 | 2026-05-15 | 新增 | 新增 SR-F-006 | 相关方需求 | 张三 |
| v1.1 | 2026-05-15 | 修改 | 修改 SR-F-001 | 相关方需求、业务架构 | 李四 |
| v1.0 | 2026-05-12 | 新增 | 初始版本 | 全部 | 王五 |
```

### 2.3 变更影响分析

**影响范围分析**：

| 层级 | 影响 | 说明 |
|------|------|------|
| **相关方需求** | 是否受影响？ | [说明] |
| **业务架构** | 是否受影响？ | [说明] |
| **系统需求** | 是否受影响？ | [说明] |
| **产品架构** | 是否受影响？ | [说明] |
| **映射关系** | 是否需要更新？ | [说明] |

**影响分析示例**：

```
变更：修改 SR-F-001 的需求描述

影响分析：
- 相关方需求：直接影响（SR-F-001 被修改）
- 业务架构：间接影响（BA-NODE-001 需要重新评估）
- 系统需求：间接影响（SysReq-NODE-001 需要重新评估）
- 产品架构：间接影响（PA-COMPONENT-001 需要重新评估）
- 映射关系：需要更新 SR → BA 映射

总体影响：中等
```

## 三、变更流程

### 3.1 变更申请流程

```
变更需求产生
  ↓
【填写】变更申请表
  ├─ 变更类型
  ├─ 变更内容
  ├─ 变更原因
  ├─ 影响范围
  └─ 建议版本号
  ↓
【评审】变更申请
  ├─ 相关方评审
  ├─ 架构师评审
  └─ IT管理部门评审
  ↓
【决策】是否批准变更？
  ├─ 批准 → 进入实施阶段
  └─ 拒绝 → 记录原因，结束
  ↓
【实施】变更
  ├─ 更新相关文档
  ├─ 更新映射关系
  ├─ 验证一致性
  └─ 记录变更
  ↓
【验证】变更是否完成？
  ├─ 是 → 发布新版本
  └─ 否 → 继续实施
  ↓
【发布】新版本
  ├─ 更新版本号
  ├─ 更新 changelog.md
  ├─ 通知相关方
  └─ 归档旧版本
```

---

## 四、版本发布规范

### 4.1 发布前检查清单

**发布前必须完成**：

- [ ] 所有变更都已实施
- [ ] 所有文档都已更新
- [ ] 所有映射关系都已更新
- [ ] 一致性验证已通过
- [ ] 质量检查已通过
- [ ] changelog.md 已更新
- [ ] 版本号已更新
- [ ] 相关方已确认

---

### 4.2 版本发布流程

```
发布前检查
  ↓
【检查】所有检查项是否都已完成？
  ├─ 是 → 进入发布阶段
  └─ 否 → 返回修复，重新检查
  ↓
【更新】版本号和发布日期
  ├─ 更新文档头部的版本号
  ├─ 更新文档信息表的版本号
  └─ 更新 changelog.md
  ↓
【通知】相关方
  ├─ 发送发布通知
  ├─ 说明变更内容
  └─ 提供文档链接
  ↓
【归档】旧版本
  ├─ 保存旧版本文档
  ├─ 记录版本历史
  └─ 建立版本索引
  ↓
【发布】新版本
  ├─ 更新文档库
  ├─ 更新版本索引
  └─ 完成发布 ✓
```

## 五、版本管理的最佳实践

### 5.1 定期发布版本

**最佳实践**：
- ✅ 定期发布版本（建议每月一次）
- ✅ 避免过于频繁的版本发布
- ✅ 避免版本号跳跃
- ✅ 保持版本历史的完整性

**发布周期**：
- 主版本：每季度一次（或根据需要）
- 次版本：每月一次（或根据需要）

---

### 5.2 清晰的变更记录

**最佳实践**：
- ✅ 详细记录每个变更
- ✅ 说明变更的原因
- ✅ 分析变更的影响
- ✅ 记录变更的审批过程

**记录内容**：
- 变更类型
- 变更内容
- 变更原因
- 影响范围
- 编制人
- 审核人
- 批准人

---

### 5.3 版本追溯

**最佳实践**：
- ✅ 保存所有版本的历史记录
- ✅ 建立版本索引
- ✅ 支持版本回溯
- ✅ 记录版本间的差异

**版本索引格式**：

```markdown

| 版本 | 发布日期 | 主要变化 | 文档链接 |
```

## 六、版本管理工具

### 6.1 使用 Git 进行版本管理

**Git 标签**：
```bash
# 创建版本标签
git tag -a v1.0 -m "Initial release"

# 查看所有标签
git tag -l

# 查看标签详情
git show v1.0

# 推送标签到远程
git push origin v1.0
```

**Git 分支**：
```bash
# 创建发布分支
git checkout -b release/v1.1

# 在发布分支上进行最后的修改
# ...

# 合并回主分支
git checkout main
git merge release/v1.1

# 创建标签
git tag -a v1.1 -m "Release v1.1"
```

## 七、常见问题

### Q1：如何处理紧急变更？

**A**：
1. 填写变更申请表
2. 标记为"紧急"
3. 加快评审流程
4. 实施变更
5. 发布新版本
6. 事后进行完整的评审

### Q2：如何处理版本冲突？

**A**：
1. 识别冲突的版本
2. 分析冲突的原因
3. 决定保留哪个版本
4. 更新相关文档
5. 发布新版本

### Q3：如何回溯到旧版本？

**A**：
1. 查看版本历史
2. 找到需要的版本
3. 检出该版本
4. 进行必要的修改
5. 发布新版本

### Q4：如何管理多个项目的版本？

**A**：
- 每个项目独立管理版本
- 建立统一的版本管理规范
- 定期同步版本信息
- 建立版本管理中心
