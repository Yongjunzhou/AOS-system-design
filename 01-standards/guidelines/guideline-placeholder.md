# 占位符规范与指南

**目的**：定义占位符的定义、创建规范、验证规则，确保占位符的一致性和有效性

**版本**：v1.0  
**最后更新**：2026-05-12

---

## 一、占位符的定义

### 1.1 什么是占位符

**占位符** = 在下游文档中预先创建的"空壳"节点，用于承接上游文档的末级节点。

**特征**：
- ✅ 有唯一的 ID 和名称
- ✅ 有明确的上游映射关系
- ✅ 具体内容用 `[待设计]` 占位
- ✅ 标记为"占位符"状态

### 1.2 占位符的作用

1. **建立映射关系**：清晰地建立上下游的对应关系
2. **预留承接点**：为下游文档预留承接点
3. **支持并行工作**：使下游团队可以独立工作
4. **避免遗漏**：确保每个上游末级节点都有对应的下游承接点

### 1.3 占位符的生命周期

```
创建 → 冻结 → 填充 → 完成
```

| 阶段 | 状态 | 内容 | 责任人 |
|------|------|------|--------|
| 创建 | 占位符 | ID、名称、映射关系 | 下游团队 |
| 冻结 | 占位符 ✓ | 占位符结构确认 | 下游团队 + 上游团队 |
| 填充 | 设计中 | 具体内容逐步填充 | 下游团队 |
| 完成 | 完成 ✓ | 所有内容已填充 | 下游团队 |

## 二、占位符的创建规范

### 2.1 占位符的标准格式

#### 2.1.1 占位符总览表

```markdown
| 节点ID | 节点名称 | 承接的上游节点 | 节点描述 | 状态 |
|--------|--------|-------------|--------|------|
| BA-NODE-001 | 订单创建流程 | SR-F-001 | [待设计] | 占位符 |
| BA-NODE-002 | 订单审批流程 | SR-F-002 | [待设计] | 占位符 |
```

**字段说明**：
- **节点ID**：唯一标识符，格式为 `[层级缩写]-NODE-[序号]`
- **节点名称**：清晰、简洁的节点名称
- **承接的上游节点**：上游末级节点的 ID
- **节点描述**：占位符阶段用 `[待设计]` 占位
- **状态**：占位符、设计中、完成

#### 2.1.2 占位符详情表

```markdown
| 项目 | 内容 |
|------|------|
| 节点ID | BA-NODE-001 |
| 节点名称 | 订单创建流程 |
| 承接的相关方需求 | SR-F-001 |
| 节点描述 | [待设计] |
| 参与角色 | [待设计] |
| 业务流程 | [待设计] |
| 输入 | [待设计] |
| 输出 | [待设计] |
| 关键决策点 | [待设计] |
| 异常处理 | [待设计] |
| 分解的系统需求 | SysReq-NODE-001 |
```

**关键点**：
- ✅ 只填充 ID、名称、映射关系
- ✅ 其他内容用 `[待设计]` 占位
- ✅ 预留下一层级的映射关系

### 2.2 占位符的命名规范

#### 2.2.1 节点 ID 命名

**格式**：`[层级缩写]-NODE-[序号]`

| 层级 | 缩写 | 示例 |
|------|------|------|
| 业务架构 | BA | BA-NODE-001 |
| 系统需求 | SysReq | SysReq-NODE-001 |
| 非功能需求 | NFR | NFR-NODE-001 |
| 产品架构 | PA | PA-COMPONENT-001 |

**规则**：
- ✅ 序号从 001 开始，逐个递增
- ✅ 同一层级内 ID 唯一
- ✅ 不跳号，不重复

#### 2.2.2 节点名称命名

**规则**：
- ✅ 清晰、简洁、名词性结构
- ✅ 避免模糊词汇（如"处理"、"管理"等）
- ✅ 与上游节点的名称相关联

**示例**：

| 上游节点 | 下游节点 | 说明 |
|--------|--------|------|
| SR-F-001: 订单创建 | BA-NODE-001: 订单创建流程 | 添加"流程"后缀 |
| SR-F-002: 订单审批 | BA-NODE-002: 订单审批流程 | 添加"流程"后缀 |
| SR-NF-001: 响应时间 | NFR-NODE-001: 订单创建响应时间 | 更具体的名称 |

### 2.3 占位符的映射关系

#### 2.3.1 映射表的标准格式

```markdown
## 映射表

| 上游ID | 上游名称 | 下游ID | 下游名称 | 状态 | 备注 |
|--------|--------|--------|--------|------|------|
| SR-F-001 | 订单创建 | BA-NODE-001 | 订单创建流程 | 占位符 ✓ | - |
| SR-F-002 | 订单审批 | BA-NODE-002 | 订单审批流程 | 占位符 ✓ | - |
| SR-NF-001 | 响应时间 | NFR-NODE-001 | 响应时间 | 占位符 ✓ | - |
```

#### 2.3.2 映射关系的验证规则

- ✅ 每个上游末级节点都有对应的下游节点
- ✅ 每个下游节点都有对应的上游节点
- ✅ 映射关系是一对一的（N:1 多对一）
- ✅ 不存在遗漏或重复

## 三、占位符的验证规范

### 3.1 占位符创建完成后的验证

#### 3.1.1 完整性验证

**检查项**：
- [ ] 是否为每个上游末级节点创建了对应的占位符？
- [ ] 是否存在遗漏的末级节点？
- [ ] 占位符的 ID 是否唯一？
- [ ] 占位符的名称是否清晰？

**验证方法**：
1. 逐个对比上游末级节点和下游占位符
2. 确保一一对应
3. 检查是否有遗漏或重复

#### 3.1.2 一致性验证

**检查项**：
- [ ] 映射关系是否正确？
- [ ] 占位符的名称是否与上游节点相关联？
- [ ] 占位符的描述是否用 `[待设计]` 占位？

**验证方法**：
1. 检查映射表中的映射关系
2. 确保映射关系与实际占位符一致
3. 检查占位符的格式是否规范

#### 3.1.3 规范性验证

**检查项**：
- [ ] 占位符的 ID 是否符合命名规范？
- [ ] 占位符的名称是否符合命名规范？
- [ ] 占位符的格式是否符合标准格式？

**验证方法**：
1. 检查 ID 的格式和序号
2. 检查名称的清晰性和简洁性
3. 检查表格的格式和内容

### 3.2 占位符填充完成后的验证

#### 3.2.1 覆盖性验证

**检查项**：
- [ ] 下游节点的内容是否完全覆盖上游节点的需求？
- [ ] 是否存在遗漏的需求？
- [ ] 是否存在超出范围的内容？

**验证方法**：
1. 逐个对比上游节点和下游节点的内容
2. 确保下游节点的内容完全覆盖上游节点的需求
3. 检查是否有遗漏或超出范围

#### 3.2.2 一致性验证

**检查项**：
- [ ] 下游节点的内容是否与上游节点的需求一致？
- [ ] 是否存在冲突或不一致？

**验证方法**：
1. 逐个检查下游节点的内容
2. 确保与上游节点的需求一致
3. 解决任何冲突或不一致

#### 3.2.3 完整性验证

**检查项**：
- [ ] 下游节点的所有必填字段是否都已填充？
- [ ] 是否还有 `[待设计]` 的占位符？

**验证方法**：
1. 检查每个下游节点的所有字段
2. 确保没有遗漏的 `[待设计]` 占位符
3. 确保所有必填字段都已填充

## 四、占位符的状态管理

### 4.1 占位符的状态定义

| 状态 | 定义 | 标记 |
|------|------|------|
| 占位符 | 已创建占位符，待填充 | 占位符 |
| 设计中 | 正在填充具体内容 | 设计中 |
| 完成 | 所有内容已填充 | ✓ |

### 4.2 占位符的状态转换

```
创建 → 占位符 → 冻结 → 设计中 → 完成 ✓
```

**转换规则**：
- 占位符 → 冻结：占位符创建完成，通过验证
- 冻结 → 设计中：开始填充具体内容
- 设计中 → 完成：所有内容已填充，通过验证

### 4.3 占位符的状态标记

**在文档中的标记方式**：

```markdown
| 节点ID | 节点名称 | 状态 |
|--------|--------|------|
| BA-NODE-001 | 订单创建流程 | 占位符 |
| BA-NODE-002 | 订单审批流程 | 设计中 |
| BA-NODE-003 | 订单履行流程 | ✓ |
```

## 五、占位符的最佳实践

### 5.1 创建占位符时的最佳实践

1. **快速完成**
   - 占位符创建应该快速完成（1-2 小时）
   - 不要过度设计或思考
   - 只需要创建基本的占位符结构

2. **清晰标记**
   - 使用 `[待设计]` 或其他清晰的标记
   - 在状态列中标记为"占位符"
   - 在映射表中清晰地显示映射关系

3. **建立映射**
   - 清晰地建立上下游的映射关系
   - 创建映射表，便于追踪
   - 确保映射关系的完整性

4. **获得确认**
   - 占位符创建完成后，获得上游团队的确认
   - 确保映射关系的正确性
   - 冻结占位符结构

### 5.2 填充占位符时的最佳实践

1. **充分理解**
   - 在开始填充前，充分理解上游文档
   - 理解每个占位符与上游节点的映射关系
   - 记录关键信息和疑问

2. **逐个填充**
   - 逐个填充占位符，不要一次性填充所有内容
   - 填充一个后，立即验证与上游文档的一致性
   - 及时解决任何不一致

3. **及时验证**
   - 在填充过程中及时验证与上游文档的一致性
   - 定期检查覆盖性和完整性
   - 及时发现和解决问题

4. **预留下游**
   - 在填充完成后，为下游文档预留占位符
   - 建立下一层级的映射表
   - 为下游团队做好准备

### 5.3 管理占位符时的最佳实践

1. **及时更新**
   - 在占位符创建和填充时及时更新映射表
   - 及时更新占位符的状态
   - 保持文档的最新状态

2. **清晰记录**
   - 清晰地记录映射关系和状态
   - 记录占位符的创建时间和填充时间
   - 记录任何变更和原因

3. **定期验证**
   - 定期验证映射表的完整性和正确性
   - 定期检查占位符的状态
   - 及时发现和解决问题

## 六、常见问题

### Q1：占位符和最终的节点有什么区别？

**A**：
- **占位符**：只有 ID、名称、映射关系，其他内容用 `[待设计]` 占位
- **最终节点**：所有内容都已填充，可以用于实际工作

### Q2：占位符创建时需要多少细节？

**A**：
- 只需要 ID、名称、映射关系
- 不需要具体的业务流程、系统需求等
- 这些内容在第二阶段填充

### Q3：如果上游文档修改了怎么办？

**A**：
1. 如果修改发生在占位符创建之前，则重新创建占位符
2. 如果修改发生在占位符创建之后，则需要更新占位符和映射表
3. 如果修改影响了下游的具体设计，则需要迭代修改

### Q4：占位符是否需要在版本控制系统中管理？

**A**：
- 可以在版本控制系统中创建分支来管理占位符版本
- 也可以在文档中标记占位符的状态
- 建议使用版本控制系统来管理，便于追踪变更

### Q5：占位符创建是否需要上游团队的参与？

**A**：
- 不一定需要上游团队的参与
- 但建议在占位符创建完成后，获得上游团队的确认
- 这样可以确保映射关系的正确性
