# 准则 5：反向迭代机制

**目的**：定义当产品架构无法满足非功能约束时的反向迭代流程，确保完整的追溯链一致性

**版本**：v1.0  
**最后更新**：2026-05-12

---

## 一、反向迭代的定义

### 1.1 什么是反向迭代？

**反向迭代**：当下游层级（产品架构）无法满足上游层级（系统需求、业务架构、相关方需求）的要求时，向上游层级反馈问题，并进行相应的调整和优化。

**核心特点**：
- ✅ 从下游向上游反馈问题
- ✅ 逐层调整和优化
- ✅ 确保完整的追溯链一致性
- ✅ 保证需求的可实现性

### 1.2 反向迭代的必要性

**为什么需要反向迭代？**

1. **发现不可实现的需求**
   - 产品架构设计时发现某些系统需求无法实现
   - 系统需求设计时发现某些业务架构无法支持
   - 业务架构设计时发现某些相关方需求无法满足

2. **发现非功能约束冲突**
   - 产品架构无法同时满足所有非功能约束
   - 需要在约束间进行权衡和调整

3. **优化设计方案**
   - 发现更优的设计方案
   - 降低实现成本
   - 提高系统性能

4. **保证设计质量**
   - 确保需求的完整性和一致性
   - 避免在开发阶段才发现问题
   - 降低项目风险

## 二、反向迭代的触发条件

### 2.1 产品架构层级的触发条件

**产品架构 → 系统需求反向迭代**

触发条件：
- [ ] 产品架构无法实现某个系统需求
- [ ] 产品架构无法满足某个非功能约束
- [ ] 产品架构的实现成本超过预期
- [ ] 产品架构的性能无法达到要求
- [ ] 产品架构存在技术风险

**示例**：
```
系统需求：响应时间 < 100ms
产品架构设计：采用同步调用方式
问题：同步调用无法达到 100ms 的响应时间
触发反向迭代：需要调整系统需求或产品架构
```

---

### 2.2 系统需求层级的触发条件

**系统需求 → 业务架构反向迭代**

触发条件：
- [ ] 系统需求无法由业务架构支持
- [ ] 系统需求的实现成本超过预期
- [ ] 系统需求存在技术风险
- [ ] 系统需求与其他需求冲突

**示例**：
```
业务架构：订单处理流程（线性流程）
系统需求：支持订单并行处理
问题：线性流程无法支持并行处理
触发反向迭代：需要调整业务架构或系统需求
```

---

### 2.3 业务架构层级的触发条件

**业务架构 → 相关方需求反向迭代**

触发条件：
- [ ] 业务架构无法满足相关方需求
- [ ] 业务架构的实现成本超过预期
- [ ] 业务架构存在技术风险
- [ ] 业务架构与其他需求冲突

**示例**：
```
相关方需求：支持 1000 并发用户
业务架构：单机部署
问题：单机部署无法支持 1000 并发用户
触发反向迭代：需要调整业务架构或相关方需求
```

## 三、反向迭代的流程

### 3.1 完整的反向迭代流程

```
产品架构设计完成
  ↓
【检查】产品架构是否满足所有系统需求？
  ├─ 是 → 完成 ✓
  └─ 否 → 进入反向迭代
     ↓
【识别】无法满足的系统需求
  ├─ 功能需求无法实现
  ├─ 非功能约束无法满足
  └─ 其他问题
     ↓
【分析】问题的根本原因
  ├─ 系统需求设计不合理
  ├─ 业务架构无法支持
  ├─ 相关方需求不合理
  └─ 其他原因
     ↓
【决策】反向迭代的方向
  ├─ 调整产品架构
  ├─ 调整系统需求
  ├─ 调整业务架构
  ├─ 调整相关方需求
  └─ 组合调整
     ↓
【实施】反向迭代
  ├─ 更新相关文档
  ├─ 更新映射关系
  ├─ 验证一致性
  └─ 记录变更
     ↓
【验证】反向迭代是否解决了问题？
  ├─ 是 → 完成 ✓
  └─ 否 → 继续迭代
```

---

### 3.2 反向迭代的具体步骤

#### 步骤 1：问题识别

**任务**：识别产品架构无法满足的系统需求

**检查清单**：
- [ ] 是否存在无法实现的功能需求？
- [ ] 是否存在无法满足的非功能约束？
- [ ] 是否存在技术风险？
- [ ] 是否存在成本超支？

**输出**：问题清单

**示例**：
```
问题1：响应时间无法达到 100ms
问题2：并发用户数无法达到 1000
问题3：数据一致性无法保证
```

---

#### 步骤 2：问题分析

**任务**：分析问题的根本原因

**分析方法**：
1. 逐条系统需求检查
2. 对比产品架构设计
3. 识别不匹配的地方
4. 分析根本原因

**输出**：问题分析报告

**示例**：
```
问题1：响应时间无法达到 100ms
原因：采用同步调用方式，网络延迟导致
根本原因：系统需求设计时未考虑网络延迟

问题2：并发用户数无法达到 1000
原因：单机部署，资源有限
根本原因：业务架构设计时未考虑并发需求
```

---

#### 步骤 3：反向迭代决策

**任务**：决定反向迭代的方向

**可选方向**：

| 方向 | 说明 | 适用场景 |
|------|------|--------|
| **调整产品架构** | 改进产品架构设计，满足系统需求 | 产品架构设计不合理 |
| **调整系统需求** | 修改系统需求，降低对产品架构的要求 | 系统需求过高 |
| **调整业务架构** | 修改业务架构，支持新的系统需求 | 业务架构无法支持 |
| **调整相关方需求** | 修改相关方需求，降低对系统的要求 | 相关方需求不合理 |
| **组合调整** | 多层级同时调整 | 问题复杂 |

**决策标准**：
1. 优先调整下游层级（产品架构）
2. 如果下游无法调整，再调整上游层级
3. 尽量避免调整相关方需求
4. 记录所有调整决策

**输出**：反向迭代决策

**示例**：
```
问题1：响应时间无法达到 100ms
决策：调整产品架构，采用异步调用方式

问题2：并发用户数无法达到 1000
决策：调整业务架构，支持分布式部署

问题3：数据一致性无法保证
决策：调整系统需求，允许最终一致性
```

---

#### 步骤 4：反向迭代实施

**任务**：实施反向迭代决策

**实施内容**：

1. **更新产品架构**（如果需要）
   - 修改架构设计
   - 更新组件设计
   - 更新通信方式

2. **更新系统需求**（如果需要）
   - 修改功能需求
   - 修改非功能约束
   - 更新验证方法

3. **更新业务架构**（如果需要）
   - 修改流程设计
   - 修改组件定义
   - 更新参与角色

4. **更新相关方需求**（如果需要）
   - 修改需求描述
   - 更新验收标准
   - 获得相关方确认

5. **更新映射关系**
   - 更新 SR → BA 映射
   - 更新 BA → SysReq 映射
   - 更新 SysReq → PA 映射

6. **记录变更**
   - 更新 changelog.md
   - 记录变更原因
   - 记录影响范围

**输出**：更新后的文档和映射关系

---

#### 步骤 5：反向迭代验证

**任务**：验证反向迭代是否解决了问题

**验证内容**：
- [ ] 产品架构是否满足所有系统需求？
- [ ] 系统需求是否满足所有业务架构？
- [ ] 业务架构是否满足所有相关方需求？
- [ ] 映射关系是否完整和一致？
- [ ] 是否存在新的问题？

**验证方法**：
1. 逐条检查系统需求
2. 对比产品架构设计
3. 验证映射关系
4. 进行一致性检查

**输出**：验证报告

**如果验证通过**：
- 反向迭代完成 ✓
- 冻结文档
- 进入下一阶段

**如果验证不通过**：
- 继续反向迭代
- 返回步骤 1

## 五、反向迭代的最佳实践

### 5.1 及早发现问题

**最佳实践**：
- ✅ 在产品架构设计初期进行可行性评估
- ✅ 定期进行一致性检查
- ✅ 及早识别潜在问题
- ✅ 避免在开发阶段才发现问题

**检查清单**：
- [ ] 产品架构是否满足所有系统需求？
- [ ] 系统需求是否满足所有业务架构？
- [ ] 业务架构是否满足所有相关方需求？
- [ ] 是否存在技术风险？
- [ ] 是否存在成本超支？

---

### 5.2 系统化的反向迭代

**最佳实践**：
- ✅ 建立反向迭代的标准流程
- ✅ 记录所有反向迭代
- ✅ 分析反向迭代的原因
- ✅ 持续改进设计流程

**改进方向**：
- 优化需求规范化流程
- 改进业务架构设计方法
- 完善系统需求设计规范
- 提高产品架构设计质量

---

### 5.3 有效的沟通和协调

**最佳实践**：
- ✅ 及时通知相关方
- ✅ 清晰解释反向迭代的原因
- ✅ 获得相关方的确认
- ✅ 记录所有决策和意见

**沟通方式**：
- 定期同步会议
- 书面通知
- 决策记录
- 影响分析报告

## 六、反向迭代的案例

### 案例 1：产品架构无法满足性能约束

**背景**：
- 系统需求：响应时间 < 100ms
- 产品架构：采用同步调用方式
- 问题：实际响应时间 > 500ms

**反向迭代过程**：

1. **问题识别**
   - 产品架构无法满足性能约束

2. **问题分析**
   - 原因：同步调用导致网络延迟累积
   - 根本原因：系统需求设计时未充分考虑网络延迟

3. **反向迭代决策**
   - 调整产品架构：采用异步调用 + 缓存方式
   - 调整系统需求：明确缓存策略和一致性要求

4. **反向迭代实施**
   - 更新产品架构设计
   - 更新系统需求文档
   - 更新映射关系

5. **反向迭代验证**
   - 验证响应时间是否 < 100ms
   - 验证数据一致性是否满足要求

**结果**：
- ✅ 响应时间达到 < 100ms
- ✅ 数据一致性满足要求
- ✅ 反向迭代完成

---

### 案例 2：业务架构无法支持并发需求

**背景**：
- 相关方需求：支持 1000 并发用户
- 业务架构：单机部署
- 问题：单机无法支持 1000 并发用户

**反向迭代过程**：

1. **问题识别**
   - 业务架构无法支持并发需求

2. **问题分析**
   - 原因：单机部署资源有限
   - 根本原因：业务架构设计时未考虑并发需求

3. **反向迭代决策**
   - 调整业务架构：支持分布式部署
   - 调整系统需求：明确分布式架构要求

4. **反向迭代实施**
   - 更新业务架构设计
   - 更新系统需求文档
   - 更新产品架构设计

5. **反向迭代验证**
   - 验证是否支持 1000 并发用户
   - 验证数据一致性

**结果**：
- ✅ 支持 1000 并发用户
- ✅ 数据一致性满足要求
- ✅ 反向迭代完成

## 七、常见问题

### Q1：反向迭代会不会导致项目延期？

**A**：
- 及早发现问题可以避免在开发阶段才发现，从而节省时间
- 反向迭代的成本通常远低于在开发阶段修复问题的成本
- 建议在设计阶段进行充分的反向迭代，而不是在开发阶段

### Q2：如何判断是否需要进行反向迭代？

**A**：
- 产品架构无法实现系统需求
- 产品架构无法满足非功能约束
- 系统需求无法由业务架构支持
- 业务架构无法满足相关方需求
- 存在技术风险或成本超支

### Q3：反向迭代应该由谁来主导？

**A**：
- 产品架构层级的反向迭代：由系统设计团队主导
- 系统需求层级的反向迭代：由系统设计团队和业务架构团队共同主导
- 业务架构层级的反向迭代：由业务架构团队和相关方共同主导

### Q4：如何避免过度的反向迭代？

**A**：
- 在设计初期进行充分的评估和验证
- 建立清晰的反向迭代触发条件
- 记录所有反向迭代，分析原因
- 持续改进设计流程，减少反向迭代的需要
