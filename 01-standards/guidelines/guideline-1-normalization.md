# 准则 1：相关方需求规范化与分类准则

**目的**：将原始需求转化为规范化的、分解的、分类的相关方需求，区分功能需求和非功能需求，为后续的业务架构设计和系统需求设计奠定基础

**适用场景**：新增相关方需求的处理

---

## 一、核心概念

### 1.1 功能需求 vs 非功能需求

**功能需求**：
- 定义：系统应该做什么，描述系统的具体功能和业务流程
- 特征：与业务流程、业务规则、业务操作相关
- 承接路线：相关方需求 → 业务架构 → 系统需求(功能) → 产品架构
- 示例：订单创建、支付处理、库存查询

**非功能需求**：
- 定义：系统应该如何做，描述系统的质量属性和约束条件
- 特征：与性能、安全、可用性、可扩展性等相关
- 承接路线：相关方需求 → 系统需求(非功能) → 产品架构约束
- 示例：响应时间<2s、支持100并发、数据加密传输

**关键区别**：
| 维度 | 功能需求 | 非功能需求 |
|------|---------|----------|
| **是否经过业务架构** | ✅ 是 | ❌ 否（直接到系统需求） |
| **是否分配到产品架构** | ✅ 是 | ❌ 否（作为约束） |
| **是否需要初步设计BA** | ✅ 是 | ❌ 否 |
| **验收方式** | 功能实现 | 指标达成 |

### 1.2 业务组件与引擎组件

**BA承接层级节点的两大类**：

| 类型 | 定义 | 承接对象 | 实现方式 |
|------|------|--------|--------|
| **业务组件** | 由引擎组件以配置方式生成的业务实体 | 功能性相关方需求 | UI配置 |
| **引擎组件** | 业务平台提供的基础引擎 | 平台功能相关的相关方需求 | 代码开发 |

**引擎组件类型示例**：
- 流程引擎（Process Engine）
- 计划引擎（Plan Engine）
- 工单引擎（Work Order Engine）
- 指标引擎（Metric Engine）
- 表单引擎（Form Engine）
- 关系引擎（Relationship Engine）
- 其他引擎...

**业务组件类型示例**（由对应引擎配置生成）：
- 流程定义（Process Definition）- 由流程引擎配置生成
- 计划定义（Plan Definition）- 由计划引擎配置生成
- 工单定义（Work Order Definition）- 由工单引擎配置生成
- 指标定义（Metric Definition）- 由指标引擎配置生成
- 表单定义（Form Definition）- 由表单引擎配置生成
- 关系定义（Relationship Definition）- 由关系引擎配置生成
- 其他业务组件...

**业务组件的生命周期**：

```
引擎组件（基础层）
   ↓ [以UI配置方式生成]
业务组件定义（配置层）
   ↓ [实例化为具体业务]
业务实例（运行层）
```

**关键特征**：
- **业务组件定义**：通过引擎提供的配置界面定义业务规则、流程、指标等
- **业务实例**：业务组件定义必须实例化为具体的业务才能运行
  - 例如：流程定义 → 订单审批流程实例、采购审批流程实例等
  - 例如：计划定义 → 销售计划实例、生产计划实例等
  - 例如：工单定义 → 维修工单实例、服务工单实例等
- **配置优于开发**：业务组件通过配置生成，无需代码开发，快速迭代

**关键关系**：
```
引擎组件（基础层）
   ↓ [以UI配置方式生成]
业务组件定义（配置层）
   ↓ [实例化为具体业务]
业务实例（运行层）
```

### 1.3 需求分类矩阵

```
相关方需求
├─ 功能需求
│  ├─ 业务相关功能需求 → 业务组件 → 配置改进/引擎改进/引擎开发
│  └─ 平台功能相关需求 → 引擎组件 → 代码开发/改进
│
└─ 非功能需求
   ├─ 性能需求 → 系统约束 → 产品架构调整
   ├─ 安全需求 → 系统约束 → 产品架构调整
   ├─ 可用性需求 → 系统约束 → 产品架构调整
   └─ 其他质量属性 → 系统约束 → 产品架构调整
```

## 二、规范化规则

### 2.1 功能需求规范化

每条规范化后的功能需求必须包含以下内容：

| 字段 | 要求 | 说明 |
|------|------|------|
| **需求 ID** | 唯一标识符，格式 SR-F-XXX | 示例：SR-F-001, SR-F-002（F 表示 Functional） |
| **需求标题** | 一句话概括 | 清晰、简洁、名词性结构 |
| **相关方** | 提出需求的部门/角色 | 明确业务来源 |
| **优先级** | P0/P1/P2/P3 | P0：必须；P1：应该；P2：可以；P3：可选 |
| **需求描述** | 完整的需求叙述 | 避免模糊词汇；使用"系统应该..." |
| **背景/原因** | 为什么需要这个需求 | 业务背景、痛点、驱动因素 |
| **验收标准** | 可测试的标准 | 必须包含具体、可测量的标准 |
| **约束条件** | 已知的限制条件 | 例如：仅限移动端、特定操作系统等 |
| **相关需求** | 关联的其他需求 ID | 说明关联类型（依赖/冲突/补充） |

### 2.2 非功能需求规范化

每条规范化后的非功能需求必须包含以下内容：

| 字段 | 要求 | 说明 |
|------|------|------|
| **需求 ID** | 唯一标识符，格式 SR-NF-XXX | 示例：SR-NF-001, SR-NF-002（NF 表示 Non-Functional） |
| **需求标题** | 一句话概括 | 清晰、简洁、指标性结构 |
| **需求类型** | 性能/安全/可用性/可扩展性/其他 | 明确非功能需求的类别 |
| **相关方** | 提出需求的部门/角色 | 明确来源 |
| **优先级** | P0/P1/P2/P3 | P0：必须；P1：应该；P2：可以；P3：可选 |
| **需求描述** | 完整的需求叙述 | 必须包含具体的量化指标 |
| **背景/原因** | 为什么需要这个需求 | 业务背景、技术考虑、合规要求等 |
| **量化指标** | 具体的可测量指标 | 例如：响应时间<2s、可用性99.9%、支持100并发 |
| **相关需求** | 关联的其他需求 ID | 说明关联类型 |

### 2.3 需求清晰性检查清单

**功能需求**：
- [ ] 需求是否避免了模糊表述（如"尽可能"、"大约"等）？
- [ ] 需求是否使用了统一的术语？
- [ ] 需求是否明确了相关方？
- [ ] 需求是否明确了约束条件？
- [ ] 验收标准是否具体和可量化？

**非功能需求**：
- [ ] 需求类型是否明确（性能/安全/可用性等）？
- [ ] 量化指标是否具体和可测量？
- [ ] 指标是否有明确的测量方法？
- [ ] 指标是否有时间或条件限制？
- [ ] 是否包含了背景和原因？

## 三、分解规则

### 3.1 功能需求分解

**分解原则**：
- 每个分解后的需求应相对独立
- 分解后的需求应能够映射到业务架构的唯一节点
- 分解应保持原需求的完整语义

**分解标准**：

| 情况 | 示例 | 分解方式 |
|------|------|--------|
| **多个功能的组合** | "系统应支持订单创建、修改和删除" | 分解为 3 个独立需求 |
| **多个角色的需求** | "销售和财务都需要查看订单信息" | 分解为 2 个独立需求 |
| **多个流程的需求** | "系统应支持订单的完整生命周期管理" | 分解为多个流程需求 |
| **多个条件的需求** | "系统应支持按订单号、客户名、日期查询" | 分解为多个查询需求 |

**分解深度**：
- 分解应达到"原子需求"的级别
- 原子需求是指不能再分解的最小需求单位
- 原子需求应能够映射到业务架构的唯一节点

### 3.2 非功能需求分解

**分解原则**：
- 按需求类型分解（性能、安全、可用性等）
- 按影响范围分解（系统级、模块级、组件级）
- 每个分解后的需求应有明确的量化指标

**分解标准**：

| 情况 | 示例 | 分解方式 |
|------|------|--------|
| **多个质量属性** | "系统应该快速、安全、可靠" | 分解为性能、安全、可用性需求 |
| **多个影响范围** | "整个系统和支付模块都要快速" | 分解为系统级和模块级性能需求 |
| **多个指标维度** | "支持高并发和快速响应" | 分解为并发数和响应时间需求 |

**分解深度**：
- 分解应达到"可测量"的级别
- 每个分解后的需求应有明确的量化指标
- 指标应能够在产品架构中验证

### 3.3 分解记录格式

```markdown
### 功能需求分解

原始需求：[原始需求]

分解后的需求：
1. SR-F-NEW-001: [分解需求1]
2. SR-F-NEW-002: [分解需求2]
3. SR-F-NEW-003: [分解需求3]

分解决策：
- [分解决策1]
- [分解决策2]

### 非功能需求分解

原始需求：[原始需求]

分解后的需求：
1. SR-NF-NEW-001: [性能需求]
2. SR-NF-NEW-002: [安全需求]
3. SR-NF-NEW-003: [可用性需求]

分解决策：
- [分解决策1]
- [分解决策2]
```

## 四、非功能需求标记为系统约束

### 4.1 标记原则

**非功能需求的处理方式**：
- 非功能需求不经过业务架构
- 非功能需求直接分配到系统需求架构中的非功能节点
- 系统需求中的非功能需求作为**产品架构的约束条件**，而非分配关系

### 4.2 约束标记

每条非功能需求应标记以下信息：

| 标记项 | 说明 | 示例 |
|--------|------|------|
| **约束类型** | 性能/安全/可用性/可扩展性/其他 | 性能约束 |
| **约束范围** | 系统级/模块级/组件级 | 系统级 |
| **量化指标** | 具体的可测量指标 | 响应时间<2s |
| **验证方法** | 如何验证该约束 | 性能测试 |
| **优先级** | P0/P1/P2/P3 | P0 |

### 4.3 约束记录格式

```markdown
### 非功能需求约束标记

| 需求ID | 约束类型 | 约束范围 | 量化指标 | 验证方法 | 优先级 |
|--------|---------|---------|---------|---------|--------|
| SR-NF-001 | 性能 | 系统级 | 响应时间<2s | 性能测试 | P0 |
| SR-NF-002 | 安全 | 系统级 | 数据加密传输 | 安全审计 | P0 |
| SR-NF-003 | 可用性 | 系统级 | 可用性99.9% | 可用性测试 | P1 |
| SR-NF-004 | 可扩展性 | 模块级 | 支持100并发 | 压力测试 | P1 |
```

## 五、初步设计BA承接层（仅功能需求）

### 5.1 初步设计的定义

**初步设计**：在规范化功能需求的同时，对已有的BA进行初步设计，以设计出能够承接功能需求末级节点的BA承接层节点

**关键特点**：
- 仅针对功能需求进行
- 与规范化功能需求同步进行
- 最小化设计，只设计必要的承接层节点
- 为后续的详细设计奠定基础

### 5.2 功能需求的处理优先级

当功能需求找不到现有的业务组件时，按以下优先级处理：

#### 【第一优先级】配置改进（推荐）

**方式**：以现有的引擎组件配置改进现有的业务组件或生成新的业务组件

**特点**：
- ✅ 无需代码开发
- ✅ 快速实现
- ✅ 风险最低
- ✅ 灵活性最高

**示例**：
```
SR-F-001: 订单创建流程
→ 查找现有业务组件：没有找到
→ 第一优先级：以现有流程引擎配置生成新的业务组件
→ 决策：可以 ✅
→ 处理方式：通过UI配置流程引擎生成新的订单创建业务组件
```

#### 【第二优先级】引擎改进（次选）

**方式**：改进已有引擎组件，以生成可以承接功能需求的业务组件

**特点**：
- 需要对引擎组件进行代码开发/补充/改进
- 然后通过UI配置生成新的业务组件
- 开发工作量中等
- 复用现有引擎框架

**示例**：
```
SR-F-002: 订单审批流程（包含复杂的条件分支）
→ 查找现有业务组件：没有找到
→ 第一优先级：以现有流程引擎配置生成
→ 决策：不可以（现有流程引擎不支持复杂条件分支）
→ 第二优先级：改进流程引擎
→ 处理方式：对流程引擎进行代码开发，增加复杂条件分支支持
         → 然后通过UI配置生成新的订单审批业务组件
```

#### 【第三优先级】引擎开发（最后选择）

**方式**：开发新的引擎组件，以生成新的类型的业务组件

**特点**：
- 完全满足需求
- 开发工作量最大
- 风险最高
- 灵活性最低

**示例**：
```
SR-F-003: 订单决策流程（需要复杂的AI决策）
→ 查找现有业务组件：没有找到
→ 第一优先级：以现有引擎配置生成
→ 决策：不可以
→ 第二优先级：改进已有引擎
→ 决策：不可以（现有引擎无法支持AI决策）
→ 第三优先级：开发新引擎
→ 处理方式：开发新的"决策引擎"
         → 然后通过UI配置生成新的订单决策业务组件
```

### 5.3 初步设计的检查清单

- [ ] 是否仅针对功能需求进行了初步设计？
- [ ] 是否区分了业务相关和平台功能相关的功能需求？
- [ ] 对于功能需求，是否按优先级进行了评估？
- [ ] 是否标记了需要进行的处理方式（配置改进/引擎改进/引擎开发）？
- [ ] 是否确保了新增BA承接层节点有完整的上级节点链，直到BA的根节点？
- [ ] 是否记录了初步设计的决策和原因？

### 5.4 初步设计记录格式

```markdown
### 初步设计 BA 承接层（功能需求）

| SR-F ID | SR末级节点 | 类型 | 现有组件 | 处理方式 | 涉及组件 | 备注 |
|---------|-----------|------|--------|--------|--------|------|
| SR-F-001 | 订单创建流程 | 业务相关 | 无 | 配置改进 | 流程引擎 | 以现有流程引擎配置生成 |
| SR-F-002 | 订单审批流程 | 业务相关 | 无 | 引擎改进 | 流程引擎 | 需要对流程引擎进行开发 |
| SR-F-003 | 订单权限控制 | 平台功能 | 无 | 引擎开发 | 权限引擎 | 需要开发新的权限引擎 |
```

## 六、冲突检测规则

### 6.1 冲突定义

**冲突**：新增相关方需求与已有需求在语义、功能或业务规则上产生矛盾或冲突

**注意**：冲突检测仅针对功能需求，非功能需求的冲突在产品架构调整时处理

### 6.2 冲突类型

| 冲突类型 | 定义 | 例子 | 处理方式 |
|---------|------|------|--------|
| **直接冲突** | 新需求与已有需求直接矛盾 | 已有"禁止删除" vs 新增"允许删除" | 需要相关方明确选择 |
| **间接冲突** | 新需求与已有需求间接矛盾 | 已有"不能修改" + 新增"支持修改" | 需要相关方确认是否修改规则 |
| **可合并冲突** | 新需求与已有需求看似冲突，实际可合并 | 已有"按订单号查询" + 新增"按客户名查询" | 建议合并为"多条件查询" |

### 6.3 冲突检测清单

- [ ] 是否存在相反的功能要求？
- [ ] 是否存在相互矛盾的业务规则？
- [ ] 是否存在时间或流程上的矛盾？
- [ ] 是否存在权限或访问权限上的矛盾？

### 5.4 冲突检测记录格式

```markdown
### 冲突检测结果

| 新增需求 | 已有需求 | 冲突类型 | 冲突描述 | AI 建议 | 人工确认 |
|---------|---------|--------|--------|--------|--------|
| SR-NEW-001 | SR-005 | 直接冲突 | "禁止删除" vs "允许删除" | 需要明确选择 | [ ] 同意 / [ ] 不同意 |
| SR-NEW-002 | SR-008 | 间接冲突 | [冲突描述] | [处理建议] | [ ] 同意 / [ ] 不同意 |
```

## 七、重复检测规则（仅功能需求）

### 7.1 重复定义

**重复**：新增功能需求与已有功能需求在功能、目的或表述上存在重复或相似

**注意**：重复检测仅针对功能需求，非功能需求的重复在系统需求层面处理

### 7.2 重复类型

| 重复类型 | 定义 | 例子 | 处理方式 |
|---------|------|------|--------|
| **完全重复** | 新需求与已有需求完全相同 | 已有"系统应支持用户登录" = 新增"系统应支持用户登录" | 删除新需求或标记为冗余 |
| **部分重复** | 新需求包含已有需求的功能，但有所扩展 | 已有"支持登录" + 新增"支持登录和注册" | 建议扩展已有需求 |
| **相似重复** | 新需求与已有需求表述不同，但功能相似 | 已有"用户可以查看订单列表" vs 新增"系统应显示用户的所有订单" | 确认是否为同一需求 |

### 6.3 重复检测清单

- [ ] 是否与已有功能需求的标题完全相同？
- [ ] 是否与已有功能需求的功能完全相同？
- [ ] 是否与已有功能需求具有相同的核心目标？
- [ ] 是否与已有功能需求具有相同的验收标准？

### 6.4 重复检测记录格式

```markdown
### 重复检测结果

| 新增需求 | 已有需求 | 重复类型 | 重复描述 | AI 建议 | 人工确认 |
|---------|---------|--------|--------|--------|--------|
| SR-NEW-001 | SR-005 | 部分重复 | 都支持订单查询，新增增加了修改功能 | 建议扩展 SR-005 | [ ] 同意 / [ ] 不同意 |
| SR-NEW-002 | SR-008 | 相似重复 | [重复描述] | [处理建议] | [ ] 同意 / [ ] 不同意 |
```

## 八、分类与确认

### 8.1 分类标准

| 分类 | 定义 | 处理方式 | 记录 |
|------|------|--------|------|
| **新增** | 与已有需求完全不同 | 直接添加新需求 | 记录为新增 SR-F-XXX 或 SR-NF-XXX |
| **改进** | 对已有需求的扩展或增强 | 合并到已有需求，更新需求描述 | 更新原需求 ID，记录变更 |
| **重复** | 与已有需求完全或高度重复 | 删除新需求，或标记为冗余 | 记录冗余关系 |
| **冲突** | 与已有需求冲突（仅功能需求） | 需要相关方明确选择 | 记录冲突，等待相关方决策 |

### 8.2 人工确认内容

**确认责任人**：相关方需求评审团队 + 系统架构师团队

**确认内容**：

- [ ] 相关方确认分解是否合理
- [ ] 系统架构师确认初步设计是否合理（仅功能需求）
- [ ] 系统架构师确认处理优先级是否合适（仅功能需求）
- [ ] 相关方对冲突有明确决策（仅功能需求）
- [ ] 相关方对重复合并有明确意见（仅功能需求）

## 九、完整处理流程

```
原始需求
   ↓
规范化（检查完整性、清晰性）
   ↓
分解（达到原子需求级别）
   ↓ 【同步进行】
区分功能需求和非功能需求
   ├─ 功能需求：
   │  ├─ 初步设计 BA 承接层
   │  ├─ 区分业务相关和平台功能相关
   │  ├─ 对业务相关的 SR 按优先级评估
   │  └─ 标记处理方式（配置改进/引擎改进/引擎开发）
   │
   └─ 非功能需求：
      └─ 标记为系统约束（约束类型、范围、量化指标、验证方法）
   ↓
冲突检测（仅功能需求，检查与已有功能需求的冲突）
   ↓
重复检测（仅功能需求，检查与已有功能需求的重复）
   ↓
分类（新增/改进/重复/冲突）
   ↓
给出建议
   ↓
人工确认（相关方需求评审团队 + 系统架构师团队）
   ↓
完成
```

## 十、常见问题

### Q1：如何判断需求是否需要分解？

**A**：如果需求中包含多个独立的功能、角色、流程或条件，就应该分解。判断标准是：分解后的每个需求是否都能映射到业务架构承接层的唯一节点（仅功能需求）或系统需求承接层的唯一节点（非功能需求）。

### Q2：冲突和重复有什么区别？

**A**：
- **冲突**：两个功能需求在功能或规则上相互矛盾（例如"允许"vs"禁止"）
- **重复**：两个功能需求在功能或目的上相同或相似（例如两个都是"支持查询"）

### Q3：如何判断SR是业务相关还是平台功能相关？

**A**：
- **业务相关**：与业务流程、业务规则、业务操作相关，可以通过配置业务组件来实现
- **平台功能相关**：与平台功能、技术能力相关，需要对引擎组件进行开发或改进

### Q4：初步设计BA承接层时，谁来做决策？

**A**：由相关方需求评审团队和系统架构师团队共同决策。相关方提供业务视角，架构师提供技术视角。

### Q5：如何处理相关方对分类有异议的情况？

**A**：记录相关方的异议，进行讨论和协商，达成一致后再继续后续步骤。

### Q6：功能需求和非功能需求如何区分？

**A**：
- **功能需求**：描述系统应该做什么，与业务流程、业务规则、业务操作相关
- **非功能需求**：描述系统应该如何做，与性能、安全、可用性、可扩展性等质量属性相关

### Q7：非功能需求为什么不经过业务架构？

**A**：因为业务架构是用户视角下的业务场景架构，代表业务流程和业务规则。非功能需求（如性能、安全等）是系统的质量属性，不是业务流程的一部分，因此不需要在业务架构中体现，而是直接作为系统需求的约束条件。

### Q8：非功能需求的量化指标如何确定？

**A**：非功能需求必须包含具体的、可测量的量化指标，例如：
- 性能：响应时间<2s、吞吐量>1000 req/s
- 安全：数据加密传输、支持 HTTPS
- 可用性：可用性 99.9%、RTO<1小时
- 可扩展性：支持 100 并发、支持 1000 万条数据
