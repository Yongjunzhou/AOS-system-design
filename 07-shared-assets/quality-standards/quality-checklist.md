# 质量检查清单

**目的**：提供可操作的质量检查工具，确保所有输出文档符合准则和模板要求

**版本**：v1.0  
**最后更新**：2026-05-12

## 一、相关方需求质量检查清单

### 1.1 功能需求检查清单

**检查人**：企业架构团队 / 系统架构师

**检查时机**：需求规范化完成后

#### 完整性检查

- [ ] 需求ID是否唯一且格式正确（SR-F-XXX）？
- [ ] 需求标题是否清晰、简洁？
- [ ] 相关方是否明确？
- [ ] 优先级是否标记（P0/P1/P2/P3）？
- [ ] 需求描述是否完整？
- [ ] 背景/原因是否说明？
- [ ] 验收标准是否具体、可测量？
- [ ] 约束条件是否列出？
- [ ] 相关需求是否关联？

**检查方法**：
```
逐条检查每个功能需求，确保所有9个必填字段都已填充
```

#### 清晰性检查

- [ ] 需求是否避免了模糊表述（如"尽可能"、"大约"等）？
- [ ] 需求是否使用了统一的术语？
- [ ] 需求是否明确了操作对象和操作方式？
- [ ] 验收标准是否可以被客观地验证？

**检查方法**：
```
阅读需求描述，确保任何人都能理解需求的含义
```

#### 分解性检查

- [ ] 需求是否已分解到原子级别？
- [ ] 是否存在可以进一步分解的需求？
- [ ] 分解后的每个需求是否都能映射到唯一的BA节点？

**检查方法**：
```
对每个需求问：这个需求是否还能再分解？
如果能，则需要继续分解
```

#### 冲突检测

- [ ] 是否存在与已有需求直接矛盾的需求？
- [ ] 是否存在与已有需求间接矛盾的需求？
- [ ] 是否存在可合并的冲突需求？

**检查方法**：
```
对比新增需求与已有需求，检查是否存在冲突
```

#### 重复检测

- [ ] 是否存在与已有需求完全相同的需求？
- [ ] 是否存在与已有需求部分重复的需求？
- [ ] 是否存在与已有需求相似的需求？

**检查方法**：
```
对比新增需求与已有需求，检查是否存在重复
```

### 1.2 非功能需求检查清单

**检查人**：企业架构团队 / 系统架构师

**检查时机**：需求规范化完成后

#### 完整性检查

- [ ] 需求ID是否唯一且格式正确（SR-NF-XXX）？
- [ ] 需求标题是否清晰、简洁？
- [ ] 需求类型是否明确（性能/安全/可用性等）？
- [ ] 相关方是否明确？
- [ ] 优先级是否标记（P0/P1/P2/P3）？
- [ ] 需求描述是否完整？
- [ ] 背景/原因是否说明？
- [ ] 量化指标是否具体、可测量？
- [ ] 相关需求是否关联？

**检查方法**：
```
逐条检查每个非功能需求，确保所有必填字段都已填充
```

#### 量化指标检查

- [ ] 指标是否具体（不是"快速"、"安全"等模糊词汇）？
- [ ] 指标是否可测量（有具体的数值或标准）？
- [ ] 指标是否有时间或条件限制？
- [ ] 指标是否有验证方法？

**检查方法**：
```
对每个指标问：如何验证这个指标是否满足？
如果无法验证，则指标不够具体
```

#### 分类检查

- [ ] 非功能需求是否正确分类（质量特性/环境适应性/可实现性）？
- [ ] 非功能需求是否正确分类（性能/安全/可用性等）？

**检查方法**：
```
根据非功能需求的定义，确认分类是否正确
```

## 二、业务架构质量检查清单

### 2.1 占位符创建检查清单

**检查人**：企业架构团队 + 系统设计团队

**检查时机**：占位符创建完成后

#### 完整性检查

- [ ] 是否为每个SR末级节点创建了对应的BA占位符？
- [ ] 是否存在遗漏的末级节点？
- [ ] 占位符的ID是否唯一？
- [ ] 占位符的名称是否清晰？
- [ ] 映射表是否完整？

**检查方法**：
```
逐个对比SR末级节点和BA占位符，确保一一对应
```

#### 一致性检查

- [ ] 映射关系是否正确？
- [ ] 占位符的名称是否与SR节点相关联？
- [ ] 占位符的描述是否用[待设计]占位？

**检查方法**：
```
检查映射表中的映射关系，确保与实际占位符一致
```

#### 规范性检查

- [ ] 占位符的ID是否符合命名规范（BA-NODE-XXX）？
- [ ] 占位符的名称是否符合命名规范？
- [ ] 占位符的格式是否符合标准格式？

**检查方法**：
```
检查ID的格式和序号，检查名称的清晰性和简洁性
```

### 2.2 具体设计检查清单

**检查人**：企业架构团队 + 系统设计团队

**检查时机**：BA具体设计完成后

#### 覆盖性检查

- [ ] 是否存在遗漏的SR需求？

**检查方法**：
```
逐条SR验证是否都被BA覆盖
```

#### 一致性检查

- [ ] BA节点的内容是否与SR需求一致？
- [ ] 是否存在冲突或不一致？

**检查方法**：
```
逐个检查BA节点的内容，确保与SR需求一致
```

#### 完整性检查

- [ ] BA节点的所有必填字段是否都已填充？
- [ ] 是否还有[待设计]的占位符？
- [ ] 业务流程是否清晰？
- [ ] 参与角色是否明确？
- [ ] 关键决策点是否标记？

**检查方法**：
```
检查每个BA节点的所有字段，确保没有遗漏的[待设计]占位符
```

#### 规范性检查

- [ ] BA节点的ID是否符合命名规范？
- [ ] BA节点的名称是否清晰、简洁？
- [ ] 业务流程的描述是否规范？
- [ ] 是否使用了Mermaid图表？

**检查方法**：
```

```

## 八、检查流程

### 8.1 完整的质量检查流程

```
需求规范化完成
  ↓
【检查】相关方需求质量检查
  ├─ 功能需求检查
  └─ 非功能需求检查
  ↓
业务架构设计完成
  ↓
【检查】业务架构质量检查
  ├─ 占位符创建检查
  └─ 具体设计检查
  ↓
系统需求设计完成
  ↓
【检查】系统需求质量检查
  ├─ 占位符创建检查
  └─ 具体设计检查
  ↓
产品架构设计完成
  ↓
【检查】产品架构质量检查
  ├─ 占位符创建检查
  └─ 具体设计检查
  ↓
【检查】映射关系质量检查
  ├─ 映射完整性检查
  └─ 映射一致性检查
  ↓
【检查】文档规范检查
  ├─ 格式检查
  ├─ 内容检查
  └─ 版本管理检查
  ↓
质量检查完成 ✓
```
