# DevOps 系统设计：提示词模板集
**DevOps System Design: Workflow Prompt Templates**

**用途**：AI 按 4 步流程依次执行，根据问题影响范围选择对应的处理模式。AI 将提示词中的 `[占位符]` 替换为实际内容后执行。

## 依据B说明

本提示词模板集适用于产品B（系统设计流水线）的DevOps场景，默认演示"软件产品"类型的快速修复和局部设计过程。使用时可根据目标产品类型选择相应的PA分类体系：

- **软件产品类型**（如产品A/AOS运营体系）：PA分类为 `前端/后端/数据/基础设施`
- **方法论/工具产品类型**（如产品B/流水线）：PA分类为 `规范与准则/设计指南/AI辅助文档/任务定义`

所有提示词模板中的 `[占位符]` 在执行时替换为实际内容。

## 第1步：快速影响范围评估

### 目标
快速判断问题的影响范围，选择合适的处理模式（15 分钟内完成）。

### 输入
- 问题报告或紧急需求描述
- 现有完整系统设计文档集

### 涉及规范
- [DevOps 设计指南 §快速决策树](../../02-design-guidelines/04-devops-design-guide.md#-快速决策树)

### 提示词模板

#### 1a. 问题分类与影响评估

```
请对以下生产问题进行快速影响范围评估。

问题描述：
[粘贴问题描述/bug报告/错误日志/用户反馈]

现有设计文档：
[粘贴或说明受影响模块的文档路径]

请执行：
1. 问题分类 — Bug 修复 vs 功能需求 vs 性能优化
2. 紧急程度评估 — P0（立即修复）vs P1（今天修复）vs P2（本周修复）
3. 定位受影响层——逐层检查：

   【第5层·产品架构】是否受影响？
   - 检查：是否有具体的 PA 组件出现异常？
   - 如确认 → 记录受影响的 PA 组件编号和名称

   【第4层·系统需求】是否受影响？
   - 检查：是否有 SysReq 的定义、性能指标或场景活动需要修改？
   - 如确认 → 记录受影响的 SysReq 节点编号

   【第3层·业务架构】是否受影响？
   - 检查：是否有 BA IPO 的业务逻辑需要调整？
   - 如确认 → 记录受影响的 BA IPO 编号

   【第2层·相关方需求】是否受影响？
   - 检查：是否有 SR 的需求描述或指标需要修改？
   - 如确认 → 记录受影响的 SR 节点编号

输出格式：
## 问题分类报告
├─ 问题类型：[Bug修复/功能需求/性能优化]
├─ 紧急程度：[P0/P1/P2] ⚠️ 待确认
├─ 问题摘要：[一句话描述]
├─ 现象：[具体现象]

## 影响范围
├─ 受影响的产品架构（PA）：PA-XXX（[原因]）
├─ 受影响的系统需求（SysReq）：SysReq-F-XXX（[原因]）
├─ 受影响的业务架构（BA）：BA-XXX（[原因]）
├─ 受影响的相关方需求（SR）：SR-F-XXX（[原因]）
└─ 受影响程度：[仅PA / PA+SysReq / 跨多层]
```

#### 1b. 处理模式选择（决策树）

```
请根据影响范围评估结果，使用决策树选择处理模式。

影响范围评估结果：
[粘贴1a的输出]

决策树：
1. 问题是否仅影响产品架构（PA）？
   ├─ 是 → 【模式1：快速修复】
   │       仅修改 PA 组件定义，几小时完成
   │
   └─ 否 → 2. 问题是否影响系统需求（SysReq）？
            ├─ 是 → 3. 问题是否影响业务架构（BA）或相关方需求（SR）？
            │       ├─ 否 → 【模式2：局部设计】
            │       │       修改 SysReq + PA，1天完成
            │       │
            │       └─ 是 → 【模式3：完整设计】
            │               修改多层（SR/BA/SysReq/PA），2-3天完成
            │
            └─ 否 → 需要重新评估问题

输出格式：
## 处理模式选择
├─ 选定模式：[模式1/模式2/模式3]
├─ 选择依据：[为什么选择该模式]
├─ 预计工时：[几小时/1天/2-3天]
├─ 涉及文档：[列出需要修改的文档]
└─ 同步设计范围：[列出需要同步更新的映射关系]
```

### 预期产出
- 问题分类和优先级报告
- 影响范围评估
- 选定的处理模式

## 第2步：模式1 - 快速修复

### 目标
快速修复仅影响产品架构的问题，几小时内完成。

### 适用前提
- 第1步已选定模式1（仅 PA 受影响）
- 已有完整的 PA 文档

### 涉及规范
- [产品架构规范 §3 详细定义规范](../../01-system-design-standards/06-product-architecture-specification.md#三详细定义规范)

### 提示词模板

#### 2a. 问题定位与 PA 组件定位

```
请在 PA 文档中定位到具体的组件和条目。

问题描述：
[粘贴1a的问题摘要]

受影响组件：
[粘贴1a中记录的PA组件]

PA 文档相关部分：
[粘贴受影响 PA 组件的当前定义]

请执行：
1. 确认受影响的具体 PA 组件和条目
2. 结合代码或配置，理解问题的根本原因
3. 定位到需要修改的具体字段（定义、实现细节、性能指标等）

输出格式：
## 问题定位结果
├─ 组件：PA-XXX [组件名称]
├─ 问题根因：[在代码/配置中的具体位置和原因]
├─ 需修改字段：[定义/实现细节/性能指标/接口定义/部署方式等]
└─ 修改类型：[~修改 / +新增字段]
```

#### 2b. 分析活动：影响验证（模式1）

```
请在开始修改前，对定位结果执行影响验证。

问题定位结果： [粘贴2a的输出]

需求说明符合性分析（模式1）：
1. 修改是否影响该 PA 组件所承接的 SysReq 9 级场景活动的覆盖完整性？
2. 是否有功能因修改而缺失？
3. 树形结构是否需要调整？

需求性能符合性分析（模式1）：
1. 修改是否引入了性能退化？
2. 响应时间、吞吐量是否仍在 SysReq 要求的范围内？

输出格式：
## 影响验证报告（模式1）
├─ 功能覆盖完整：[通过/未通过，说明原因]
├─ 树形结构调整：[无需调整/需调整，说明]
├─ 性能回退风险：[无风险/有风险，说明]
└─ ⚠️ 待确认项：[如有]
```

#### 2c. 快速修复（模式1）

```
请根据影响验证结果，生成 PA 组件的修复条目。

影响验证报告： [粘贴2b的输出]

修复说明：
[粘贴问题根因和修改方案]

简化条目格式（仅列出变更内容，使用 ~ 前缀）：

~ PA-XXX: [产品组件名称]（3级·架构末级节点）
├─ 原问题：[问题描述]
├─ 修改内容：[对实现方案的具体修改]
├─ 承接的系统需求：[SysReq-XXX]（保持原有映射关系）
├─ 符合性分析：[修改后如何满足所承接的系统需求]（必须保留）
├─ 实现细节：[更新后的技术方案]
└─ 验证方法：[如何验证修复]

如需新增组件（极少使用）：
+ PA-XXX: [新组件名称]（3级·架构末级节点）
├─ 定义：[组件定义]
├─ 组件类型：[根据产品类型选择：软件产品→前端/后端/数据/基础设施；方法论/工具产品→规范与准则/设计指南/AI辅助文档/任务定义]
├─ 承接SysReq：[SysReq-XXX]
├─ 符合性分析：[如何满足]
├─ 实现细节：[技术栈、关键模块]
└─ 性能指标：[指标值]

同步更新：
1. 架构定义文档 → 更新映射关系表中对应的符合性分析
2. 详细定义文档 → 更新该组件的实现细节和技术方案描述
```

#### 2d. 快速验证与发布（模式1）

```
请验证修复结果并准备发布。

修复条目： [粘贴2c的输出]

验证项：
1. [ ] 问题是否完全解决
2. [ ] 是否有副作用（回归测试通过）
3. [ ] 性能是否在 SysReq 指标范围内
4. [ ] 符合性分析已更新
5. [ ] 产品架构文档的架构定义和详细定义同步更新
6. [ ] 映射关系表中的 PA 组件映射保持不变

变更日志：
## 变更日志 v[版本号]
├─ 变更类型：快速修复
├─ 问题：[问题描述]
├─ 修复内容：[修改简述]
├─ 涉及组件：PA-XXX
├─ 涉及文档：产品架构文档
└─ 版本号：[更新后的版本号]
```

### 预期产出
- 修复后的产品架构文档（架构定义 + 详细定义）
- 验证报告
- 变更日志

## 第3步：模式2 - 局部设计

### 目标
快速设计和实现影响系统需求和产品架构的问题，1 天内完成。

- [产品架构规范 §2 架构定义规范](../../01-system-design-standards/06-product-architecture-specification.md#二架构定义规范)

### 提示词模板

#### 3a. 分析活动：影响范围分析（模式2）

## 第4步：模式3 - 完整设计

### 目标
完整设计和实现影响多层的问题，2-3 天内完成。

### 适用前提
- 第1步已选定模式3（影响跨多层，涉及 BA 或 SR）
- 已有完整的 5 层设计文档

### 涉及规范
- 全部 02-06 层规范（依影响层级选择）

### 提示词模板

#### 4a. 分析活动：快速跨层影响分析

```
请对模式3的修改执行快速跨层影响分析。

问题描述：
[粘贴1a的问题摘要]

受影响范围（跨多层）：
[粘贴1a中记录的受影响层和节点]

现有 5 层文档：
[说明各层文档路径]

需求说明符合性分析（模式3）：
1. 问题跨层影响到哪些架构末级节点？
2. 逐层追溯完整性——从 SR 到 PA，受影响的节点链是什么？
3. 修改后是否需要新增节点或修改现有节点？

需求性能符合性分析（模式3）：
1. 跨层的性能指标传递是否一致？
2. 下层方案是否仍能满足上层的性能要求？

处理原则：
- 遵循敏捷式系统设计的增量标记约定（`~` 修改 / `+` 新增）
- 使用两阶段工作流程：先建立结构骨架，再填充详细内容
- 更新顺序按影响层级从高到低：SR → BA → SysReq → PA
- 上层详细定义修改后，下层架构必须同步调整（同步设计原则）

输出格式：
## 跨层影响分析报告（模式3）
├─ SR 影响：[受影响的SR节点]
├─ BA 影响：[受影响的BA IPO]
├─ SysReq 影响：[受影响的SysReq节点]
├─ PA 影响：[受影响的PA组件]
├─ 节点链：[SR-XXX → BA-XXX → SysReq-XXX → PA-XXX]
├─ 变更类型：[~修改 / +新增 / 混合]
└─ ⚠️ 待确认项：[如有]
```

#### 4b. 按层修改（模式3 - 从高到低依次执行）

```
请根据跨层影响分析结果，从高层到低层逐层修改。

跨层影响分析报告： [粘贴4a的输出]

请按照以下顺序修改（每步完成后再进入下一步）：

【第1步】相关方需求（SR）修改：
~ SR-F-XXX: [名称]（架构末级节点）
├─ 原问题：[问题描述]
├─ 修改内容：[对需求定义或性能指标的修改]
```
