# 各层产品数据规范编写规范（规范之规范）
**Specification for Writing Layer Product Data Specifications — A Meta-Specification**

**文档版本**：v1.0
**创建日期**：2026-05-15
**目的**：为编写系统设计流水线各层级产品数据规范提供统一的结构、内容和风格指南，确保各层产品数据规范在质量、可读性和可操作性上保持一致

---

## 📌 核心原则

### 原则1：每份产品数据规范都应该"自包含"

读者阅读一份产品数据规范时，应该能独立理解该层的完整设计方法，无需频繁跳转到其他文档。少量必要的交叉引用（指向设计准则总纲和相关层产品数据规范）是允许的。

### 原则2：内容按"概念 → 方法 → 验证 → 常见问题"四段式组织

这是经过对比验证的最优结构（03规范已验证其有效性）：

```
概念层（What & Why）→ 方法层（How）→ 验证层（Check）→ 常见问题层（FAQ）
```

每段的占比建议：
- 概念段 20% — 建立共识
- 方法段 50% — 核心操作指导
- 验证段 15% — 质量和验收标准
- 常见问题段 15% — 解答真实困惑

### 原则3：用对比示例代替纯文字解释

一个 ✅/❌ 对比示例的效果胜过三段文字说明。关键判断标准处都应使用对比示例。

### 原则4：每份产品数据规范控制在 2000 行以内

超过 2000 行文档将变得难以维护和阅读；过短的规范则深度不足。各层规范的行数应在合理范围内，以完整覆盖该层的设计方法为准。


```
  1.3 本层特有的关键概念（2-4个）

  2.3 承接的上层需求及其映射

三、详细定义规范（方案文档层适用）
  3.1 详细定义末级节点定义
  3.2 分解标准和终止准则

  4.2 关键设计规则

五、验收标准
  5.1 架构定义验收
  5.2 详细定义验收
  5.3 映射关系验收

六、检查清单
  6.1 分解充分性检查
  6.2 映射关系检查
  6.3 符合性分析检查

七、常见错误和最佳实践
  7.1 常见错误（含正确与错误对比）
  7.2 最佳实践

八、FAQ
  8.1 常见疑问解答（3-8个）

九、与其他规范的关系
  9.1 输入规范（上层）
  9.2 输出规范（下层）
  9.3 与设计准则总纲的关系
```

### 各层特殊的结构调整

| 层 | 调整说明 |
|----|---------|
| **第1层-原始需求** | 无"架构定义"（第2章替换为"收集和格式规范"）；"详细定义"替换为"分解规范"；增加"冲突和重复检测" |
| **第2层-相关方需求** | 按标准模板执行，无需特殊调整 |
| **第3层-业务架构** | 无"详细定义"（删除第3章）；需增加"IPO模型"独立章节；需增加"去重机制" |
| **第4层-系统需求** | "架构定义"需拆分"功能分支"和"非功能分支"两个子章节；需包含0-9级和0-6级层级定义 |
| **第5层-产品架构** | 需增加"组件分类"（前端/后端/数据/基础设施）；"详细定义"聚焦技术实现细节 |

## 三、每个章节的内容指南

### 第一章：核心概念

**必须回答的问题**：
- 本层是什么？（一句话定义）
- 在本层中，"架构末级节点"是什么？
- 在本层中，"详细定义末级节点"是什么（如有）？
- 本层在5层结构中充当什么角色？
- 本层特有的关键概念是什么？

**必须包含的内容**：
- ✅ 本层在5层结构中的位置图（文本流程图）
- ✅ 本层与上、下层的承接/分配关系
- ✅ 本层的"分析"是什么（设计准则要求的分析活动）
- ❌ 不要在此处展开详细的操作步骤

**最佳实践**（来自03规范）：
> 在概念层就把"功能需求和非功能需求的不同处理方式"区分清楚，避免读者在阅读后续章节时混淆。

---

### 第二~三章：架构定义 / 详细定义规范

**核心检查清单**：

- [ ] 给出了末级节点的属性定义表（如ID、定义、性能指标等）
- [ ] 给出了适当的粒度判断标准（什么是"太粗"、"太细"、"正好"）
- [ ] 给出了"1:1约束"的实际判断方法（"问自己：这个条目能映射到下层的一个节点吗？"）
- [ ] 至少包含1个完整的树形结构示例和1个映射视图示例
- [ ] 每个重要判断标准都配有 ✅/❌ 对比示例

**避免的写法**（来自04规范的反面教训）：
```
❌ IPO的体量主要取决于所期望输出物的颗粒度
```
改为：
```
✅ IPO的体量判断标准：
   - 期望输出是技术文档 → 一个IPO对应一份技术文档
   - 期望输出是业务流程 → 一个IPO对应一个完整流程
   - 判断：能否分配到SysReq的唯一一个功能组件？
```

---

### 第四章：设计方法

**必须包含**：
- ✅ 本层的完整设计流程（至少3步）
- ✅ 如何与上下层同步设计的说明
- ✅ 适用该层的具体设计规则

**推荐的流程描述方式**（来自05规范的最佳实践）：
```
第1步：IPO去重
  └─ 动作：[具体做什么]
     └─ 输出：[产生什么]

第2步：建立架构
  └─ ...
```
而不是纯文字段落描述。

---

### 第五~六章：验收标准和检查清单

**两类内容的区分**：

| 内容 | 验收标准 | 检查清单 |
|------|---------|---------|
| **定位** | 量化指标 | 操作验证 |
| **格式** | 表格（检查项+标准+说明） | 列表（✅ - 项） |
| **作用** | 衡量完成质量 | 确保无遗漏 |
| **示例** | 分解覆盖率100% | [ ] 所有需求已分解到末级 |

**注意事项**：
- ❌ 验收标准和检查清单不要内容重叠（05规范的§9和§10重叠严重，是反面教训）
- ✅ 验收标准提供量化指标（如"覆盖率100%"）
- ✅ 检查清单提供操作指导（如"[ ] 检查所有需求是否已分解"）

---

### 第七章：常见错误和最佳实践

**推荐的"四段式"结构**（来自03规范）：

```markdown
**❌ 错误：[错误示例的标题]**

错误的做法：
```
[...错误代码/示例...]
```

问题：[分析为什么错了]
后果：[不修正会怎样]

**✅ 正确做法：**
```
[...正确的示例...]
```
说明：[为什么这样做是对的]
```

**注意事项**：
- 每个错误必须配一个正确的做法对比
- 错误数量控制在4-6个（太多记不住，太少不够全面）
- 最佳实践数量控制在3-5个

---

### 第八章：FAQ

**FAQ提问指南**：

| 提问类型 | 推荐 | 例子 |
|---------|------|------|
| 决策型 | ✅ | "如果无法1:1映射怎么办？" |
| 数量型 | ✅ | "详细定义最多可以有多少层？" |
| 流程型 | ✅ | "详细定义和架构设计可以并行吗？" |
| 重复型 | ❌ | 已在正文中明确说明的内容不要再问 |
| 主观型 | ❌ | "哪种方法更好？"（正文应已直接给出结论） |

**FAQ数量**：3-8个为最佳区间。

### 第九章：与其他规范的关系

**推荐的格式**（来自03规范的示例）：

```markdown
### 与[上层产品数据]的关系

| 方面 | [上层产品数据] | [本产品数据] |
|------|--------------|-------------|
| **定位** | [一句话] | [一句话] |
| **映射方向** | [方向说明] | [承接方向说明] |
| **关系** | [关系说明] | [关系说明] |

[类似结构]
```

## 四、风格指南

### 4.1 术语一致性

| 术语 | 推荐用法 | 不推荐用法 |
|------|---------|-----------|
| 架构末级节点 | 全文统一使用 | 架构节点/末级节点/承接点混用 |
| 详细定义末级节点 | 全文统一使用 | 分解条目/详细节点/末级条目混用 |
| 1:1分配约束 | 全文统一使用 | 一对一约束/分配规则混用 |
| N:1承接支持 | 全文统一使用 | 多对一映射/承接关系混用 |

### 4.2 对比示例的使用规范

每处关键判断标准都应有对比示例：

```
✅ 正确（格式正确/粒度合适/做法正确）：
  [具体示例]

❌ 错误（格式错误/粒度过大/做法错误）：
  [具体示例]
```

### 4.3 流程图的使用

推荐使用纯文本树形结构（非Mermaid），确保在纯文本环境和AI环境中均可读：

```
第1层：XXX
   ↓ N:1 分配
第2层：XXX
  ├─ 架构末级节点 ↔ 详细定义末级节点
  └─ ↓ N:1 分配
      第3层：XXX
```

不要用 `→` 和 `←` 混合方向，统一用：
- `↓` 表示分配的流向（上层→下层）
- `←` 表示承接的方向（下层←上层）

### 4.4 章节编号

| 层级 | 编号规范 | 示例 |
|------|---------|------|
| 章 | 一、二、三... | `## 一、核心概念` |
| 节 | 1.1, 1.2, 1.3... | `### 1.1 本层的定义` |
| 小节 | 1.1.1, 1.1.2... | `#### 1.1.1 子概念A` |

## 五、质量检查清单（规范自身的验收）

完成一份产品数据规范后，用此清单自检：

### 结构完整性

- [ ] 封面信息完整（标题、目的、版本、日期、位置）
- [ ] 有快速导航（按角色或按需求）
- [ ] 核心概念独立成章
- [ ] 架构定义和详细定义（如有）分别说明
- [ ] 设计方法独立成章
- [ ] 验收标准与检查清单不重叠
- [ ] 常见错误用四段式结构
- [ ] 有FAQ章节
- [ ] 明确标注了与上下层产品数据规范的关系
- [ ] 总行数不超过 2000 行（层规范适用；设计准则总纲不受此限）

### 内容完整性

- [ ] 明确说明了本层在5层结构中的位置
- [ ] 明确了本层"架构末级节点"是什么
- [ ] 明确了本层"详细定义末级节点"是什么（如有）
- [ ] 明确了本层的"分析"活动是什么
- [ ] 1:1分配约束和N:1承接支持都有说明
- [ ] 给出了完整的格式示例
- [ ] 每个关键判断标准都有✅/❌对比示例
- [ ] 有具体的操作步骤（不只有概念说明）
- [ ] 给出了可复制使用的模板

### 风格一致性

- [ ] 所有[ ] 占位符已填充
- [ ] 术语全文统一
- [ ] 对比示例格式统一
- [ ] 流程图方向统一
- [ ] 没有冗余的内容重复
- [ ] 没有与设计准则总纲矛盾的内容
- [ ] 超链接格式正确

## 六、示例：本规范的实际应用

以下以04-业务架构规范为例，展示如何应用本规范改进一份层产品数据规范。

### 步骤1：诊断现有问题

规范之规范的"二、推荐目录结构"提供了9章标准模板。对照模板诊断04：

```
04（修订前）：
- 13章碎化严重，超出9章模板建议
- §8和§9内容重叠（验收标准与检查清单混用）
- 缺少FAQ章节
- 缺少✅/❌对比示例
- 缺少"本层的分析活动"说明
```

### 步骤2：按本规范重组结构

按模板的9章结构和各层特殊调整指南，重组为：

```
04（修订后）：
一、核心概念（模板第一章）
二、架构定义规范（模板第二章——含层级结构、组件分类）
三、IPO模型（模板建议新增的独立章节——第3层特殊调整）
四、设计方法（模板第四章）
五、验收标准（模板第五章——与检查清单分离）
六、检查清单（模板第六章）
七、常见错误和最佳实践（模板第七章——含✅/❌对比）
八、FAQ（模板第八章——5个问答）
九、与其他规范的关系（模板第九章——与SR/SysReq对应）
```

### 步骤3：补充缺失内容

按模板的章节指南和质量检查清单补充：

- 在§7中新增✅/❌四段式对比示例（4个常见错误，每个含错误做法/问题/后果/正确做法）
- 新增FAQ章节（5个问答：为何无详细定义、粒度判断、去重必要性、承接数量、与流程图区别）
- 在§1.4中新增"本层的分析活动"说明（需求说明+性能符合性分析）
- 按模板§4.4章节编号规范统一全文标题层级
