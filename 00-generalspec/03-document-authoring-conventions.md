## 一、文档元数据

每份文档的头部应包含以下信息：

```markdown
# 文档标题
**English Title**

**文档版本**：v1.0
**创建日期**：2026-05-29
**修订日期**：2026-05-31
**用途**：一句话说明文档用途及覆盖范围（可选字段：历史、来源文档、状态）
```

**字段说明**：

| 字段 | 必填 | 说明 |
|------|------|------|
| **文档版本** | ✅ | 格式 `v[主].[次]`（v1.0 初版 / v2.0 重构 / v1.1+ 小修） |
| **创建日期** | ✅ | 文档首次创建的日期 |
| **修订日期** | ✅ | 最近一次修订的日期 |
| **用途** | ✅ | 一句话定义文档目的和适用范围 |
| **历史** | 可选 | 重要变更记录的简要列表 |
| **来源文档** | 可选 | 产品数据文档使用，标注上游来源 |
| **状态** | 可选 | 产品数据文档使用（草稿/已冻结/待确认等） |

元数据块与正文之间用 `---` 分隔。

## 二、章节编号体系

### 2.1 层级规则

| 标题层级 | 编号格式 | 示例 |
|---------|---------|------|
| **H1** | 无编号，纯文档标题 | `# 通用系统设计规范` |
| **H2** | 大分区名称，用冒号或直接命名 | `## 第一部分：方法论与设计规则` |
| **H3** | 中文数字 + 顿号 | `### 一、基本术语与概念` |
| **H4** | 阿拉伯数字父子编号 | `#### 1.1 基本概念`、`#### 3.1 需求分解及分析` |
| **H5** | 三级数字编号 | `##### 1.3.1 架构定义子文档`、`##### 10.1.1 模式的 PACE 特征` |
| **H6** | **不编号**，仅文字标题 | `###### 条件性升级规则` |

### 2.2 附录编号

| 标题层级 | 编号格式 | 示例 |
|---------|---------|------|
| H3 附录标题 | `### 附录[字母]、[标题]` | `### 附录A、方法论底层推导链` |
| H4 附录子节 | `#### [字母].[数字] [标题]` | `#### A.1 第〇层：先验约束` |
| H5 附录子节 | `##### [字母].[数字].[数字] [标题]` | `##### A.1.1 范围限定：什么是"设计"？` |

> **注**：附录的 H5 沿用 `字母.数字.数字` 编号，这是附录的标准做法，与正文 H5 的纯数字编号在形式上一致、在体系上并行。H6 保持不编号。

### 2.3 编号的稳定性

- 章节编号一旦分配，不因树形结构调整而改变
- 新增 H4/H5 子节 → 在所属父节下追加序号（如 `#### 1.4` 之后追加 `#### 1.5`）
- 中间层级重新组织 → 不改变 H4/H5 的编号

## 三、表格格式

### 3.1 通用规则

- 使用标准 Markdown 管道表格：`| 列1 | 列2 |`
- **不使用** `:-` 或 `-:` 对齐标记（全部默认左对齐）
- 表头用 `|------|------|` 分隔，长度与列内容对齐即可
- 表头文字不加粗；术语表中术语名加粗

### 3.2 术语定义表（规范正文）

用于规范正文中定义术语：

```markdown
| 术语 | 定义 | 详见 |
|------|------|------|
| **术语名** (English) | 术语的精确定义 | §X.X, §Y |
```

### 3.3 术语对照表（独立术语表文档）

用于独立的术语对照文档：

```markdown
| 中文 | 英文 | 说明 |
|------|------|------|
| 架构末级节点 | Architecture Leaf Node | 方案分类和组织的最大条目... |
```

### 3.4 说明类表格

用于展示规则、条件、对比等。表头随内容而定，不拘泥于固定格式。示例：

```markdown
| 条件 | 说明 | 违规信号 |
|------|------|---------|
| **覆盖完整性** | 父需求中的每个功能点... | 遗漏 → ❌ |
```

## 四、引用格式

### 4.1 层内引用（同一文档内）

| 场景 | 格式 | 示例 |
|------|------|------|
| 引用章节 | `§[编号]` | `§3.1`、`§七` |
| 引用子节 | `§[父].[子]` | `§2.3.1`、`§4.2.2` |
| 附录引用 | `附录[字母]` | `附录A`、`附录B` |
| 括号内引用 | `（§编号）` | `（§1.1 "需求/方案相对"）` |
| 中文后引用 | 中文空格§编号 | `见 §3.3`、`详见 §4.4.1` |

> **注**：`§` 后**不加空格**。`§` 与前面中文之间**保留空格**（标准中文字距排版）。

### 4.2 跨文档引用

使用 Markdown 相对路径链接，并在链接文本中标注目标文档的简称：

```
[通用设计规范 §一~§二](../../../00-general/10-general-system-design-standards/01-general-system-design-specification.md)
[EOS系统设计规范 §一](../../20-eos-design/10-eos-system-design/10-eos-system-design-specification/01-eos-system-design-specification.md)
```

### 4.3 术语引用

正文中首次出现关键术语时，可加粗或附带英文：

```
**需求**（Requirement）是指...
```

## 五、注记与说明块

根据内容的长度和重要程度选择不同格式：

| 类型 | 格式 | 适用场景 |
|------|------|---------|
| **简短注记**（1-2行） | `**注**：说明文字` | 在段落中嵌入简短补充 |
| **重要说明**（3行以上） | `> **注**：说明文字` | 需要独立成段的补充说明 |
| **专题说明** | `> **专题名称**：说明文字` | 有特定主题名称的说明 |
| **操作提示** | `> 提示内容` | 无标签的纯操作指引 |
| **反面教材** | `> **反面教材**："空洞声明"` | 举反例时使用 |

**专题说明示例**：

```markdown
> **约束继承说明**：父需求的约束不要求出现在每一条子需求中...
```

**反面教材示例**：

```markdown
> **反面教材**："本节点满足上述所有需求。"——没有说明如何满足，等于没有分析。
```

## 六、代码块与图表

### 6.1 代码块

- 使用 fenced code block（`` ``` ``）
- **不加语言标签**（不写 ```` ```markdown ````、```` ```text ```` 等）
- 用于展示：架构树、流程图、示例内容、模板

### 6.2 流程图

使用纯文本 ASCII 树（非 Mermaid），确保纯文本环境和 AI 环境中均可读：

```
第1层：原始需求及分析 [需求]
    ↓ N:1 分配
第2层：相关方需求 [需求/方案]
  ├─ 架构末级节点 ↔ 详细定义末级节点
  └─ ↓ N:1 分配
      第3层：业务架构 [方案]
```

**箭头方向统一规则**：
- `↓` 表示**分配**的流向（上层→下层）
- `→` 表示正向推导方向
- `←` 表示承接方向（下层←上层）
- `───→` 用于强调箭头（附录中的因果关系链）

### 6.3 架构树

```文本
├── 纯分类节点
│   ├── 架构末级节点 A
│   └── 架构末级节点 B
└── 集成实体节点
    ├── 架构末级节点 C
    └── 架构末级节点 D
```

## 八、表达风格

本文档是**工程规范**（Engineering Specification），以"查"和"引用"为阅读目的，而非从头到尾顺序学习。以下规则确保语气一致、内容可预测。

### 8.1 文档定位

内容功能与表达方式的对应关系：

| 内容功能 | 表达方式 | 示例 |
|---------|---------|------|
| **定义概念** | `X是Y` 判断句 | "架构末级节点是架构树的末梢节点" |
| **规定规则** | `应/不应/不得` 祈使句 | "分配应遵循1:1约束" |
| **描述操作** | 零人称命令句 | "将父需求分解为子需求，执行四项自检" |
| **推理论证** | `因为/所以/因此` 论证句 | "因此可追溯性不是选项，而是必要条件" |

> **注**：附录中的推理论证（如方法论底层推导链）可使用更具哲学推演色彩的表达方式（反问、否认推论、强调因果必然性），这是附录的自身功能，与正文的工程规范语气不同。附录的语气差异化是合理的——读者进入附录时已了解其性质不同于正文。

### 8.2 人称与视角

| 规则 | 说明 | ✅ 正确 | ❌ 避免 |
|------|------|--------|--------|
| 规则和操作 | 零人称祈使句 | "将父需求分解为子需求" | "你要将父需求分解" |
| 概念定义 | 第三人称陈述句 | "架构末级节点是..." | "你可以把架构末级节点理解为..." |
| 禁止事项 | 不得/不应 | "不得将同一需求分配到多个节点" | "最好不要这么做" |
| 规则表述 | 应/须 | "分配应遵循1:1约束" | "建议考虑1:1约束" |

禁止使用第二人称"你"、"我们"作为叙事主体。角色名称（如"设计者"、"用户"）在定义需求或描述场景时正常使用，不受此限。"读者"一词在元说明（如"详见第X章"）中也不受此限。

### 8.3 规则表达硬度

| 硬度 | 措辞 | 含义 |
|------|------|------|
| 强制性 | 应 / 须 | 必须遵守，无例外 |
| 禁止性 | 不得 / 不应 | 严格禁止 |
| 推荐性 | —（直接陈述） | "优先级：复用→改进→新增"（不需"建议"二字） |

避免"可以"、"建议"、"最好"等模糊措辞。如果某条不是强制要求，用陈述句说明"什么情况下用什么做法"而非"建议做什么"。

### 8.4 解释方式

| ✅ 允许 | ❌ 禁止 |
|--------|--------|
| 直接陈述定义 | "这就像..."（比喻） |
| 因果解释（"这是因为..."） | "你可以理解为..."（辅导口吻） |
| 条件说明（"当X时，应Y"） | "相当于..."（类比） |

### 8.5 禁止元评论

文档**不谈论自身**。以下写法严禁出现：

- ❌ "本文的核心内容" → ✅ 直接陈述："本章规定..."
- ❌ "如下是重点" → 重要性由章节层级传达，不需旁白
- ❌ "如前所述" → 改为"见§X.X"
- ❌ "首先我们来看..." → 直接进入内容

### 8.6 对比示例的标准写法

工程规范中的对比示例用于消除歧义，而非"教学"。标准格式：

```markdown
✅ 正确做法：
- 具体示例

❌ 错误做法：
- 具体示例
```

对比示例仅出现在需要消除理解歧义的关键判断点，不滥用。

### 8.7 FAQ 与常见错误的写法

FAQ 和常见错误章节使用"错误+正确"对比式表达，**不采用 Q/A（Question/Answer）格式**：

```markdown
**错误：分解粒度不当**

分解粒度过粗或过细是各层最常见的错误。

| 问题 | 表现 | 后果 |
|------|------|------|
| **粒度过粗** | 一条末级条目需要分配到下层多个节点 | 违反1:1约束 |
| **粒度过细** | 功能拆成过多条目 | 管理复杂，架构碎片化 |

✅ 正确做法：分解到每条末级只需下层一个架构末级节点承接。
```

FAQ 使用"问/答"的中文标记取代"Q/A"：

```markdown
**问：一个架构末级节点承接多少条上层条目合理？**

**答：** 建议每个架构末级节点承接2-5条（2-3条最优）。
```

常见错误和 FAQ 的语气可以比正文稍放松，但不得滑入培训腔——仍是规范文档的组成部分，只是面向排查场景的语言略有调整。

### 8.8 句子与段落可读性

以下规则确保文档读起来通顺、不绕、不啰嗦：

**单句单意。** 一句话表达一个核心意思。当一句话需要两个以上破折号（`——`）串联多个子句时，应拆分为多个独立的短句。

```markdown
❌ BA 可采用用户角色架构锚定法——以目标产品的全量用户角色集为推导锚点——也可采用输出产品架构锚定法——以输出产品的 PA 节点类型为推导锚点。

✅ BA 可采用用户角色架构锚定法，以目标产品的全量用户角色集为推导锚点。也可采用输出产品架构锚定法，以输出产品的 PA 节点类型为推导锚点。
```

**慎用"进行""加以"。** 这两个词在多数情况下可删除或替换为更直接的动词。

```markdown
❌ 对架构末级节点进行细化分解 → ✅ 细化分解架构末级节点
❌ 从以下维度进行验证 → ✅ 从以下维度验证
❌ 在 OR 层进行分离 → ✅ 在 OR 层分离
```

**避免重复表述。** 同一概念在相邻段落中不应重复定义或解释。确需指向已有定义时使用"见§X.X"。

```markdown
❌ BA 是功能需求完整性保障的系统性方法。BA 之所以作为五层中的独立层存在，是因为它为功能需求的完整性提供了系统性保障方法。
→ 两句意思相同，合并。

✅ BA 是功能需求完整性保障的系统性方法（非功能需求没有等价方法——路径差异见 §1.2）。
```

**避免教科书式铺垫。** 不写"随着...的深入"、"在...方面"、"众所周知"等背景引导句，直接进入内容。

```markdown
❌ 随着 AI 辅助设计的深入，AI 助理在各模式中的参与方式呈现系统性差异。
✅ AI 助理在各模式中的参与方式呈现系统性差异。
```

**缩写首次出现时标注全称。** 每章或每节中，缩写首次出现时应标注全称，如"系统需求（SysReq）"。同一节内重复出现时不再标注。

## 九、验收标准与检查清单的编写

### 9.1 适用范围标注

每节的适用范围标注统一使用斜体 `适用步骤：§X.X`：

```markdown
*适用步骤：§6.3（SR架构）、§6.4（BA架构）*
```

### 9.2 验收标准

使用表格格式，包含检查项、标准和说明三列：

```markdown
| 检查项 | 标准 | 说明 |
|--------|------|------|
| **架构末级节点定义完整率** | 100% | 每个架构末级节点须具备四要素 |
```

### 9.3 检查清单

使用任务列表格式：

```markdown
- [ ] 遍历每层方案文档的架构树
- [ ] 检查架构定义子文档的第二部分（映射关系表）
```

> **区分原则**：验收标准提供量化指标（"覆盖率 100%"），检查清单提供操作指导（"遍历每层架构树"），两者不重叠。
