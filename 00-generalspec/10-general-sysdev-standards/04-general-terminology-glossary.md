# 通用术语对照表
**General Terminology Glossary**

**文档版本**：v2.4
**创建日期**：2026-05-18
**修订日期**：2026-06-01
**用途**：统一与具体产品无关的核心设计术语，便于国际化和技术交流。本文档与通用系统设计规范 v1.0 配套使用——各术语的完整定义和上下文以规范正文为准。

## 一、文档类型

| 中文 | 英文 | 说明 |
|------|------|------|
| 准则 | Standards / Principles | 理论基础、规范要求、设计原则 |
| 指南 | Guide / Handbook | 实践指导、操作步骤、工作手册 |
| 参考卡 | Quick Reference Card | 快速查阅、速记、便携版 |
| 索引 | Index | 导航、查找工具 |
| 规范 | Specification | 针对特定设计层的详细规范和标准 |
| 任务定义 | Task Definition | 描述特定操作规程的执行步骤和输入输出 |
| AI辅助文档 | AI Support Document | 为 AI 成员提供的系统设计辅助指引 |
| 产品数据 | Product Data | 按产品结构模板填入具体内容产生的设计输出 |

## 二、产品开发阶段

| 中文 | 英文 | 说明 |
|------|------|------|
| 系统设计 | System Design | 阶段1，从原始需求到产品架构的完整设计过程 |
| 构件开发 | Component Development | 阶段2，对产品构件根据需求定义开发制品 |
| 系统集成与交付 | System Integration & Delivery | 阶段3，沿架构树形层级从叶子节点逐级向上集成至根节点 |
| 系统运维 | System Operation & Maintenance | 阶段4，产品交付后的持续运行、监控、维护和优化 |
| 产品构件 | Product Component | PA 架构末级节点的具体指代对象——可独立分配、可独立替换、可独立验证的产品组成部分（§6.2） |

## 三、五层结构

| 中文 | 英文 | 说明 |
|------|------|------|
| 原始需求 | Original Requirements (OR) | 第1层，用户或相关方的原始需求——纯需求文档，无架构定义、无详细定义（设计链起点） |
| 相关方需求 | Stakeholder Requirements (SR) | 第2层，原始需求的方案和后续需求——功能（SR-F）和非功能（SR-NFR）平行架构 |
| 业务架构 | Business Architecture (BA) | 第3层，相关方需求功能部分的方案——运行概念说明（OpsCon），以 IPO 为基本描述单元 |
| 系统需求 | System Requirements (SysReq) | 第4层，业务架构和非功能需求的方案——功能（SysReq-F）和非功能（SysReq-NFR）平行架构 |
| 产品架构 | Product Architecture (PA) | 第5层，系统需求的方案和实现——纯方案，设计链终点 |

## 四、基本术语

| 中文 | 英文 | 说明 |
|------|------|------|
| **质量** | Quality | 一组固有特性满足需求的程度（ISO 9000:2015, 3.6.2）。本方法论中，质量特指系统设计质量——各层设计输出（方案）满足其所承接上层需求的程度，通过符合性分析（语义符合性 + 指标符合性）来度量和保证（§1.1, §三） |
| **需求** | Requirement | 明示的、通常隐含的或必须履行的需要或期望（ISO 9000:2015, 3.6.4）。本方法论中，需求指对系统应具备的功能或应满足的质量约束的描述，由需求描述和性能指标两部分构成——不可衡量则不成为需求。**类型维度**：方案的对立面——方案回答 How，需求回答 What。**角色维度**：设计链中需要被下层方案承接的设计输入。两种基本类型：功能需求（FR）和非功能需求（NFR）（§1.1-§1.2） |
| **方案** | Solution | 在特定上下文中满足需求的一种具体方式（BABOK® v3 / BACCM）。本方法论中，方案指对上层需求的设计实现，承载于方案文档之中；方案文档由架构定义子文档和详细定义子文档组成。**类型维度**：需求的对立面——需求回答 What，方案回答 How。**角色维度**：设计链中对上层需求的满足输出；其详细定义末级条目在向下传递时承担需求角色，这是"文档的双重属性"的机制来源（§1.1, §2.2） |
| **架构** | Architecture | 系统在其环境中的基本概念或属性，体现于其元素、关系以及设计和演化的原则之中（ISO/IEC/IEEE 42010:2011, 3.2）。本方法论中，架构特指各层设计内容的基本组织形态——以树形结构组织设计内容，其末级节点是所承接上层需求的解决方案实体（§1.1, §2.2） |
| **需求/方案相对** | Requirement-Solution Relativity | "需求"和"方案"不是文档的固有标签，而是文档在设计链中相对于相邻文档扮演的角色——同一文档在不同设计层位上兼具不同身份：向上是方案（满足上层需求），向下是需求（驱动下层设计）。本条是§2.1**需求相对原则**的概念基础，两者定义一致——"需求/方案相对"是概念表述，"需求相对"是原则名称（§1.1, §2.1） |
| **需求文档** | Requirements Document | 文档在结对中扮演需求方角色时即为需求文档，描述系统应该做什么（What），包含功能需求和非功能需求（§1.4） |
| **方案文档** | Solution Document | 文档在结对中扮演方案方角色时即为方案文档，描述如何满足需求（How），涵盖功能需求和非功能需求的实现方案（§1.4） |
| **需求相对原则** | Requirement-Solution Relativity Principle | 四条工程化设计原则之一。"需求"和"方案"不是文档的固有标签，而是文档在设计链中相对于相邻文档扮演的角色，同一文档在不同设计层位上兼具不同身份。解决的问题：设计链如何跨层衔接（§2.1） |
| **详概分离原则** | Separation Principle | 四条工程化设计原则之一。为落实分解和分配而设——每份需求或方案文档由**架构定义子文档**（粗粒度树形结构）和**详细定义子文档**（细粒度条目平铺列表）两个独立子文档构成。解决的问题：单层文档如何组织（§2.2）。BA 例外：仅有架构定义、无详细定义；OR 例外：仅条目清单、无子文档结构 |
| **文档结对 / 设计文档结对** | Design Document Pairing | 系统设计的基本组织单元——上层需求详细定义末级条目与下层方案架构定义末级节点之间的配对关系。不同于一般意义的"需求—方案配对"，结对特指详细定义末级与架构末级之间的精确链接。有需求必有方案，有方案必有需求，保证设计链的完整闭合（§1.4, §2.3） |
| **结对设计原则** | Paired Design Principle | 四条工程化设计原则之一。结对发生在上层需求详细定义末级条目与下层方案架构定义末级节点之间，要求同步设计而非串行执行——分解时先约定下层方案架构骨架并建立初步映射，再并行填充详细内容，避免全部分解完才发现无目标可承接的返工。解决的问题：跨层文档如何协同（§2.3） |
| **两阶段工作法** | Two-phase Workflow | 结对设计的具体执行方式：**骨架阶段**——先快速约定下层方案架构定义的节点分类和末级标题并建立初步映射；**填充阶段**——上下层各自并行填充（上层完善需求详细定义，下层完善方案架构节点四要素）（§2.3.1） |
| **基于资产原则** | Asset-Based Principle | 四条工程化设计原则之一。架构节点不是凭空产生的，而是基于组织已有资产按**复用→改进→新增**优先级生成。解决的问题：架构节点如何生成。改进分三种操作手势：Extend（扩展）、Split（拆分）、Merge（合并）；新增后按成本评估实现路径（自研、外购或外包）；特殊情况：Deprecate（废弃）——标记"无上游需求，已废弃"但保留节点（§2.4） |
| **架构定义子文档** | Architecture Definition Document | 需求/方案文档的子文档之一。以树形结构组织文档层级，以架构末级节点为需求映射的落脚点，建立对上层需求的 N:1 承接关系。由架构设计结果（树形结构）和需求映射视图（映射关系表）两部分组成（§2.2.1） |
| **详细定义子文档** | Detailed Definition Document | 需求/方案文档的子文档。对架构末级节点逐层细化，产出可分配的原子粒度条目。采用平铺列表而非树形结构，每条条目包含编号、名称、需求描述、性能指标、验收标准和映射目标（§2.2.2） |
| **架构末级节点** | Architecture Leaf Node | 架构定义子文档树形结构的末梢节点，是所承接上层需求的**解决方案实体**。与纯分类节点（仅用于组织层级、无实体含义）及集成实体节点（代表由子节点集成的整体、有聚合级需求定义）的本质区别在于：末级节点以 1:1 粒度直接承载上层需求的映射及方案内容。颗粒度标准：一个节点承接的条目数应合理（N:1），不过大、不过小、语义独立。各层具体指代：SR-F 架构末级 = 一个端到端业务闭环（§4.2）；SR-NFR 架构末级 = 一个非功能特性维度（§4.3）；BA IPO = 一个业务行为单元（§4.4）；SysReq-F = 一个系统功能点（§4.5）；SysReq-NFR = 一个非功能特性维度（跨角色归并）（§4.6）；PA = 一个产品构件（§6.2） |
| **架构末级节点四要素** | Four Elements of Architecture Leaf Node | 每个架构末级节点包含四项信息：**节点定义**、**上层需求映射**（承接了上层哪些末级节点）、**符合性分析说明**（如何定性定量满足承接需求）、**节点关系说明**（与同层其他节点之间的关联——上下游衔接、功能依赖、数据交换或接口调用）。PA 层例外：PA 是设计链终点，其第四要素替换为**下游设计指导**（对组件开发阶段的指导——技术选型方向、接口约定、关键实现约束）（§2.2.1, §6.2.2） |
| **纯分类节点** | Pure Classification Node | 架构树中仅用于对架构进行分类和组织的非末级节点，无实体含义，不承载需求映射（如 SR 层"功能需求"分支）（§1.3.1, §2.2.1） |
| **集成实体节点** | Integrated Entity Node | 架构树中代表由其子节点功能集成的真实实体的非末级节点——节点本身在架构中有意义，负责聚合级需求定义和符合性分析，但 N:1 映射仍落在其子节点上（如 BA 层业务域、PA 层服务组）（§1.3.1, §2.2.1） |
| **详细定义末级节点** | Detailed Definition Leaf Node | 对架构末级节点逐层分解后达到的原子粒度条目。分解终止标准：该条目已细化到可依 1:1 约束分配到下层方案架构定义子文档的架构末级节点（§1.3.2, §2.2.2） |
| **详细定义的双重身份** | Dual Identity of Detailed Definition | 详细定义子文档中的每个末级节点具有双重身份：向内是对当前层架构末级节点的细化（定义具体职责和实现策略），向下是作为需求输入传递给下层方案（其架构末级节点需要承接这些条目）——本层设计的输出末端，同时是下层设计的需求源头（§1.4） |
| **系统设计链路** | System Design Chain | 系统设计阶段中从原始需求到产品架构的完整设计链条，共五层：OR → SR → BA → SysReq → PA，由相邻层之间的文档结对依次衔接而成。其中 FR 沿 OR→SR→BA→SysReq→PA 通路分配，NFR 沿 OR→SR-NFR→SysReq-NFR→PA 通路分配（NFR 平行路径）（§1.5, §一） |
| **需求分解** | Requirements Decomposition | 将父需求分解为子需求的过程，语义和性能指标两个维度同步进行。**语义分解**将父需求的功能描述无重叠无遗漏地切分为子需求，执行五项操作（覆盖完整性、不超出性、逻辑一致性、约束继承、NFR→FR 派生）；**指标分解**将父需求的性能指标分配到子需求，按三类场景（可拆分/下放/暂不分解）判定处理（§3.1） |
| **需求分配** | Requirements Allocation | 对需求分解产出的详细定义末级节点，按 1:1 约束逐一映射到下层方案架构定义子文档的架构末级节点。分配后执行三项自检：全覆盖、1:1、N:1 合理性（§3.2） |
| **符合性分析** | Compliance Analysis | 验证架构定义子文档的架构末级节点是否充分满足所承接的上层需求集合。分两个维度：**语义符合性**（方案定义是否满足所承接需求的全部语义）和**指标符合性**（方案指标不低于需求指标的要求值）。分析深度分三级：L1 快速自检、L2 语义符合性、L3 指标符合性（§3.3） |
| **1:1分配约束** | One-to-One Allocation Constraint | 每条详细定义末级节点只能分配到下层方案中唯一一个架构末级节点。例外处理见 §3.2.2 |
| **N:1承接支持** | Many-to-One Acceptance Support | 下层方案中一个架构末级节点可以承接上层需求的多个详细定义末级节点。合理合并条件见 §3.2.3 |
| **需求性能指标** | Requirement Performance Metric | 需求被满足的程度和测量方式，分为定量指标（响应时间 < 200ms、可用性 99.9% 等）和二元判定指标（通过认证、满足标准等）两类。每条需求必须同时包含需求描述和性能指标——不可衡量则不成为需求（§1.2） |
| **指标分解三场景** | Three Metric Decomposition Scenarios | 父指标与子指标之间的三种运算关系判定：**可拆分**（存在可计算的函数关系——加和、乘积、极值等，按该关系分配子指标值）、**下放**（所有子需求各自独立满足同一指标值——同值继承）、**暂不分解**（独立约束无法运算分解——标注"向下一层传递"）（§3.1.2） |
| **需求语义分解五项操作** | Five Semantic Decomposition Checks | 覆盖完整性（不漏）、不超出性（不增）、逻辑一致性（不矛盾）、约束继承（不失）、NFR→FR 派生（将非功能的功能含义提取到 FR 路径）（§3.1.1） |
| **四单一原则** | Four Single Principles | OR 条目拆分的四项原则——单一角色（仅涉及一个用户角色）、单一语义（仅含一个独立功能或关注点）、单一路径（FR/NFR 分离）、单一映射（拆分后可 1:1 映射到 SR 架构末级节点）。拆分终止标准：四条同时满足（§7.3） |
| **NFR 平行路径** | NFR Parallel Path | 非功能需求在五层设计链中不经过 BA 层，沿 OR→SR-NFR→SysReq-NFR→PA 独立分配。分类视图在 SR-NFR→SysReq-NFR 间发生切换——SR-NFR 按相关方角色组织以便验证完备性，SysReq-NFR 按三级分类组织以便工程师查阅全量约束（§1.2, §4.3.4, §4.6.4） |
| **纯方案** | Pure Solution | 五层中仅承担方案角色、不向下传递需求的层级。BA（第3层）是纯方案——IPO 条目自身承担"详细定义"角色；PA（第5层）是纯方案——设计链终点，其详细定义描述"如何实现"而非还需什么（§5.1, §6.1） |
| **桥 / 桥接** | Bridge | BA 在五层设计链中的角色定位——承上启下的桥接层。对上游（SR）是方案方，对下游（SysReq）是功能需求输入源。BA IPO 去重后直接映射到 SysReq-F 架构定义，每个唯一 IPO 对应一个 SysReq-F 架构末级节点（§5.1） |
| **产品构件** | Product Component | PA 架构末级节点的具体指代对象——可独立分配、可独立替换、可独立验证的产品组成部分。五层设计链的终点，直接对接组件开发阶段（§6.2） |
| **三独立判定原则** | Three Independence Principles | PA 产品构件划分的判定标准：**可独立分配**（能否作为一个独立工作包分配给一个团队开发）、**可独立替换**（更换技术方案时能否只更换该构件而不牵连其他）、**可独立验证**（能否独立验证其是否符合 SysReq）。三项必须同时满足（§6.2.1） |
| **IPO（输入→处理→输出）** | Input-Process-Output | 业务架构（BA）的基本描述单元，每条 IPO 描述产品处理一个输入输出单元的行为——I 由外部角色提交的输入单元，P 将输入转换为输出的过程（可含自动化处理和人参与），O 交付给外部角色的输出单元。BA 仅架构定义、无详细定义，IPO 自身承担"详细定义"角色（§4.4, §5.1） |
| **运行概念说明** | Operational Concept (OpsCon) | 业务架构（BA）的本质——各类用户角色使用目标产品开展期望业务的运行概念说明（ISO/IEC/IEEE 29148:2018）。本方法论以 IPO 作为 OpsCon 的结构化载体（§5.1） |
| **目标产品** | Target Product (Product A) | 正在被系统设计的产品，统一称谓 |
| **输出产品** | Output Product (Product B) | 流水线生产出来的产品，仅流水线类产品有此概念 |
| **用户角色** | User Role | 与目标产品交互的所有外部实体（操作用户、外部系统、保障者、自然环境等）。BA 开发的核心锚点（§5.2.1） |
| **用户角色架构锚定法** | User Role Architecture Anchoring Method | BA 开发的通用方法——以目标产品的全量用户角色集为推导锚点，适用于所有产品类型（§5.2.1） |
| **输出产品架构锚定法** | Output Product Architecture Anchoring Method | 流水线类产品的 BA 特化路径——以输出产品的 PA 节点类型为推导锚点，省略通用方法的角色遍历。其有效性源于：当产品 B 的 PA 节点类型被穷举时，为其定义的生产操作步骤就覆盖了产品 A 的全部操作场景（§5.2.2） |
| **SR-F 三步法** | Three-Step Method for SR-F | SR-F 架构末级节点的识别方法——第一步：识别用户角色类型；第二步：按角色类型识别触发，定义端到端业务；第三步：用分解分配分析验证候选节点（§4.2.3） |
| **SR-NFR 角色→分类归并法** | Role→Classification Method for SR-NFR | SR-NFR 架构末级节点的识别方法——按相关方角色类型识别 NFR 关注点，再按附录B三级分类归并关注点到末级节点（§4.3.3） |
| **SysReq-F 视角转换法** | Perspective Conversion Method for SysReq-F | 将 BA IPO 从业务视角转换为系统功能视角的四步转换：命名转换、I 转换（业务输入→系统接收）、P 转换（人机分工明确化）、O 转换（业务输出→系统输出），每个唯一 BA IPO 一对一映射为一个 SysReq-F 架构末级节点（§4.5.3） |
| **SysReq-NFR 去重归并法** | Deduplication & Merge Method for SysReq-NFR | 从"按角色组织"切换到"按三级分类组织"——同一分类下、来源于不同角色的 SR-NFR 节点合并为一个 SysReq-NFR 节点，量化指标取各来源中最严格的要求（§4.6.3） |
| **输入输出流转链法** | IO Flow Chain Method | BA IPO 的识别方法——对每个 SR-F 端到端业务闭环，按业务时序识别产品全过程中接收的输入单元和产出的输出单元，每个"接收→处理→输出"节点定义一个 IPO（§4.4, §5.2.1） |
| **三层工作层面** | Three Work Levels | 第1层流水线设计（用通用方法论设计设计线自身）、第2层交付物构建（设计线的构件交付物）、第3层使用工具设计 EOS。本概念为 EOS 流水线专有，在通用设计规范中属于实例化应用 |
| **信息处理模式** | Information Processing Mode | 业务活动处理信息的方式分类。不同信息处理模式的需求不能共享同一架构节点（信息处理模式硬边界），该边界优先级高于资产优先原则（§3.2.3 N:1 合理合并条件之一）。具体分类在相关产品级设计中展开 |
| **PACE** | Precision, Adaptability, Completeness, Efficiency | 刻画四种设计模式差异的四维度——**Precision**（精度：产出精细化程度）、**Adaptability**（适应性：对需求变化的响应能力）、**Completeness**（完备度：设计文档完整程度）、**Efficiency**（效率：设计执行时效性）（§11.1.1） |
| **L1/L2/L3 分析深度** | Compliance Analysis Depth Levels | 符合性分析的三个深度层次——**L1**快速自检（分解四检+分配三检）、**L2**语义符合性（覆盖度/映射正确性/分析具体性）、**L3**指标符合性（指标对应完整性/比对方法正确性/可验证性）。不同设计模式采用不同分析深度组合（§3.3, §12.4） |
| **变更影响分析** | Change Impact Analysis | 敏捷式和 DevOps 模式中确定变更范围的方法：变更源定位 → 影响扩散分析 → 影响范围分级 → 变更路径确定（§12.2.5） |
| **增量映射三个手势** | Incremental Mapping Gestures | 敏捷式增量分配中基于资产优先原则的三个操作手势——**Extend**（扩展到已有节点）、**Split**（从已有节点拆分出新节点）、**Add**（新增节点），优先级依次递减。优先级更高的是"复用"（Reuse，无需手势的直接映射）（§12.3.3） |
| **架构债务** | Architecture Debt | 当前版本无法满足某 NFR 时，记录差距并纳入后续版本规划的决策项。三类情形：调整架构后仍无法满足、协商调整 NFR 指标后仍有差距、非功能权衡后的选择（§7.7） |

## 五、需求类型

| 中文 | 英文 | 说明 |

| 功能需求 | Functional Requirements (FR) | 系统必须执行的功能、行为或服务——系统对输入的反应和产生的输出（ISO/IEC/IEEE 29148:2018） |

## 六、核心约束

| 中文 | 英文 | 说明 |
|------|------|------|
| 1:1分配约束 | One-to-One Allocation Constraint | 每条详细定义末级节点只能分配到下层方案中唯一一个架构末级节点 |
| N:1承接支持 | Many-to-One Acceptance Support | 下层方案中一个架构末级节点可以承接上层需求的多个详细定义末级节点 |
| 双向追溯 | Bidirectional Traceability | 支持从上层需求追溯到下层方案，也支持从下层方案反向追溯到上层需求 |
| 符合性分析 | Compliance Analysis | 通过分析验证方案如何满足所承接的上层需求 |
| 映射关系 | Mapping Relationship | 需求和方案之间的对应关系 |

## 七、四种设计方法

| 中文 | 英文 | 说明 |
|------|------|------|
| 瀑布式 | Waterfall | 从原始需求开始的完整正向系统设计（正向完整设计） |
| 敏捷式 | Agile | 在现有设计基础上处理增量需求的迭代设计（正向增量） |
| 逆向工程 | Reverse Engineering | 从已有产品（代码/图纸/PA 架构定义）反向推导系统设计（反向推导） |
| DevOps | DevOps | 快速响应生产环境问题和紧急需求（最小正向变更） |
| 正向完整设计 | Forward Complete Design | 瀑布式：逐层正向推导（OR → SR → BA → SysReq → PA） |
| 正向增量 | Forward Incremental | 敏捷式：分析影响范围 → 增量更新受影响的文档 |
| 反向推导 | Reverse Derivation | 逆向工程：从 PA 节点需求定义反向推导至原始需求 |
| 最小正向变更 | Minimal Forward Change | DevOps：定位 → 修复 → 验证的最小化变更路径 |

## 八、操作流程

| 中文 | 英文 | 说明 |
|------|------|------|
| 需求分解 | Requirements Decomposition | 将父需求逐层分解为更细的子需求——语义分解（无重叠无遗漏切分）和指标分解（运算关系分配）双维度同步进行 |
| 需求分配 | Requirements Allocation | 将详细定义末级节点按1:1约束分配到下层方案架构末级节点 |
| 符合性分析 | Compliance Analysis | 验证方案的架构末级节点是否充分满足所承接的上层需求（语义+指标双维度） |
| NFR→FR 派生 | NFR-to-FR Derivation | 从 NFR 子需求中识别其驱动的功能含义，生成为独立的 FR 条目（§3.1.1 第5步） |
| 需求映射 | Requirements Mapping | 将需求映射到方案 |
| 影响分析 | Impact Analysis | 分析变更的影响范围 |
| 完整性验证 | Completeness Verification | 验证所有需求都被映射 |
| 一致性验证 | Consistency Verification | 验证映射关系的一致性 |

## 九、节点编号规范

| 中文 | 英文 | 说明 |
|------|------|------|
| 架构末级节点编号 | Architecture Leaf Node Numbering | 格式：`{层前缀}[-{类型后缀}]-{序号}`，如 SR-F-001 |
| 详细定义末级节点编号 | Detailed Definition Leaf Node Numbering | 格式：`{所归属的架构末级节点ID}.{层内序号}`，如 SR-F-001.01 |

| 编号的稳定性 | ID Stability | 编号一旦分配即固定，不因树形结构调整而改变 |
