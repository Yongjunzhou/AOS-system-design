## 一、文档元数据

每份文档的头部应包含以下信息：

```markdown
# 文档标题
**English Title**

**文档版本**：v1.0
**创建日期**：2026-05-29
**修订日期**：2026-05-31
**用途**：一句话说明文档用途及覆盖范围（可选字段：历史、来源文档、状态）
```

**字段说明**：

| 字段 | 必填 | 说明 |
|------|------|------|
| **文档版本** | ✅ | 格式 `v[主].[次]`（v1.0 初版 / v2.0 重构 / v1.1+ 小修） |
| **创建日期** | ✅ | 文档首次创建的日期 |
| **修订日期** | ✅ | 最近一次修订的日期 |
| **用途** | ✅ | 一句话定义文档目的和适用范围 |
| **历史** | 可选 | 重要变更记录的简要列表 |
| **来源文档** | 可选 | 产品数据文档使用，标注上游来源 |
| **状态** | 可选 | 产品数据文档使用（草稿/已冻结/待确认等） |

元数据块与正文之间用 `---` 分隔。

## 二、章节编号体系

### 2.1 层级规则

| 标题层级 | 编号格式 | 示例 |
|---------|---------|------|
| **H1** | 无编号，纯文档标题 | `# 通用系统设计规范` |
| **H2** | 大分区名称，用冒号或直接命名 | `## 第一部分：方法论与设计规则` |
| **H3** | 中文数字 + 顿号 | `### 一、基本术语与概念` |
| **H4** | 阿拉伯数字父子编号 | `#### 1.1 基本概念`、`#### 3.1 需求分解及分析` |
| **H5** | 三级数字编号 | `##### 1.3.1 架构定义子文档`、`##### 10.1.1 模式的 PACE 特征` |
| **H6** | **不编号**，仅文字标题 | `###### 条件性升级规则` |

### 2.2 附录编号

| 标题层级 | 编号格式 | 示例 |
|---------|---------|------|
| H3 附录标题 | `### 附录[字母]、[标题]` | `### 附录A、方法论底层推导链` |
| H4 附录子节 | `#### [字母].[数字] [标题]` | `#### A.1 第〇层：先验约束` |
| H5 附录子节 | `##### [字母].[数字].[数字] [标题]` | `##### A.1.1 范围限定：什么是"设计"？` |

> **注**：附录的 H5 沿用 `字母.数字.数字` 编号，这是附录的标准做法，与正文 H5 的纯数字编号在形式上一致、在体系上并行。H6 保持不编号。

### 2.3 编号的稳定性

- 章节编号一旦分配，不因树形结构调整而改变
- 新增 H4/H5 子节 → 在所属父节下追加序号（如 `#### 1.4` 之后追加 `#### 1.5`）
- 中间层级重新组织 → 不改变 H4/H5 的编号

## 三、表格格式

### 3.1 通用规则

- 使用标准 Markdown 管道表格：`| 列1 | 列2 |`
- **不使用** `:-` 或 `-:` 对齐标记（全部默认左对齐）
- 表头用 `|------|------|` 分隔，长度与列内容对齐即可
- 表头文字不加粗；术语表中术语名加粗

### 3.2 术语定义表（规范正文）

用于规范正文中定义术语：

```markdown
| 术语 | 定义 | 详见 |
|------|------|------|
| **术语名** (English) | 术语的精确定义 | §X.X, §Y |
```

### 3.3 术语对照表（独立术语表文档）

用于独立的术语对照文档：

```markdown
| 中文 | 英文 | 说明 |
|------|------|------|
| 架构末级节点 | Architecture Leaf Node | 方案分类和组织的最大条目... |
```

### 3.4 说明类表格

用于展示规则、条件、对比等。表头随内容而定，不拘泥于固定格式。示例：

```markdown
| 条件 | 说明 | 违规信号 |
|------|------|---------|
| **覆盖完整性** | 父需求中的每个功能点... | 遗漏 → ❌ |
```

## 四、引用格式

### 4.1 层内引用（同一文档内）

| 场景 | 格式 | 示例 |
|------|------|------|
| 引用章节 | `§[编号]` | `§3.1`、`§七` |
| 引用子节 | `§[父].[子]` | `§2.3.1`、`§4.2.2` |
| 附录引用 | `附录[字母]` | `附录A`、`附录B` |
| 括号内引用 | `（§编号）` | `（§1.1 "需求/方案相对"）` |
| 中文后引用 | 中文空格§编号 | `见 §3.3`、`详见 §4.4.1` |

> **注**：`§` 后**不加空格**。`§` 与前面中文之间**保留空格**（标准中文字距排版）。

### 4.2 跨文档引用

使用 Markdown 相对路径链接，并在链接文本中标注目标文档的简称：

```
[通用设计规范 §一~§二](../../../00-general/10-general-system-design-standards/01-general-system-design-specification.md)
[EOS系统设计规范 §一](../../20-eos-design/10-eos-system-design/10-eos-system-design-specification/01-eos-system-design-specification.md)
```

### 4.3 术语引用

正文中首次出现关键术语时，可加粗或附带英文：

```
**需求**（Requirement）是指...
```

## 五、注记与说明块

根据内容的长度和重要程度选择不同格式：

| 类型 | 格式 | 适用场景 |
|------|------|---------|
| **简短注记**（1-2行） | `**注**：说明文字` | 在段落中嵌入简短补充 |
| **重要说明**（3行以上） | `> **注**：说明文字` | 需要独立成段的补充说明 |
| **专题说明** | `> **专题名称**：说明文字` | 有特定主题名称的说明 |
| **操作提示** | `> 提示内容` | 无标签的纯操作指引 |
| **反面教材** | `> **反面教材**："空洞声明"` | 举反例时使用 |

**专题说明示例**：

```markdown
> **约束继承说明**：父需求的约束不要求出现在每一条子需求中...
```

**反面教材示例**：

```markdown
> **反面教材**："本节点满足上述所有需求。"——没有说明如何满足，等于没有分析。
```

## 六、代码块与图表

### 6.1 代码块

- 使用 fenced code block（`` ``` ``）
- 示例内容、模板、架构树、流程图默认不加语言标签
- YAML、JSON、Shell 等语法敏感内容可加语言标签，以提升可读性和复制安全性
- 不为普通文本示例添加 `text`、`markdown` 等无必要语言标签

### 6.2 流程图

使用纯文本 ASCII 树（非 Mermaid），确保纯文本环境和 AI 环境中均可读：

```
第1层：原始需求及分析 [需求]
    ↓ N:1 分配
第2层：相关方需求 [需求/方案]
  ├─ 架构末级节点 ↔ 详细定义末级节点
  └─ ↓ N:1 分配
      第3层：业务架构 [方案]
```

**箭头方向统一规则**：
- `↓` 表示**分配**的流向（上层→下层）
- `→` 表示正向推导方向
- `←` 表示承接方向（下层←上层）
- `───→` 用于强调箭头（附录中的因果关系链）

### 6.3 架构树

```文本
├── 纯分类节点
│   ├── 架构末级节点 A
│   └── 架构末级节点 B
└── 集成实体节点
    ├── 架构末级节点 C
    └── 架构末级节点 D
```

## 八、表达风格

本文档是**工程规范**（Engineering Specification），以"查"和"引用"为阅读目的，而非从头到尾顺序学习。以下规则确保语气一致、内容可预测。

### 8.1 文档定位

内容功能与表达方式的对应关系：

| 内容功能 | 表达方式 | 示例 |
|---------|---------|------|
| **定义概念** | `X是Y` 判断句 | "架构末级节点是架构树的末梢节点" |
| **规定规则** | `应/不应/不得` 祈使句 | "分配应遵循1:1约束" |
| **描述操作** | 零人称命令句 | "将父需求分解为子需求，执行四项自检" |
| **推理论证** | `因为/所以/因此` 论证句 | "因此可追溯性不是选项，而是必要条件" |

> **注**：附录中的推理论证（如方法论底层推导链）可使用更具哲学推演色彩的表达方式（反问、否认推论、强调因果必然性），这是附录的自身功能，与正文的工程规范语气不同。附录的语气差异化是合理的——读者进入附录时已了解其性质不同于正文。

### 8.2 人称与视角

| 规则 | 说明 | ✅ 正确 | ❌ 避免 |
|------|------|--------|--------|
| 规则和操作 | 零人称祈使句 | "将父需求分解为子需求" | "你要将父需求分解" |
| 概念定义 | 第三人称陈述句 | "架构末级节点是..." | "你可以把架构末级节点理解为..." |
| 禁止事项 | 不得/不应 | "不得将同一需求分配到多个节点" | "最好不要这么做" |
| 规则表述 | 应/须 | "分配应遵循1:1约束" | "建议考虑1:1约束" |

禁止使用第二人称"你"、"我们"作为叙事主体。角色名称（如"设计者"、"用户"）在定义需求或描述场景时正常使用，不受此限。"读者"一词在元说明（如"详见第X章"）中也不受此限。

### 8.3 规则表达硬度

| 硬度 | 措辞 | 含义 |
|------|------|------|
| 强制性 | 应 / 须 | 必须遵守，无例外 |
| 禁止性 | 不得 / 不应 | 严格禁止 |
| 推荐性 | —（直接陈述） | "优先级：复用→改进→新增"（不需"建议"二字） |

避免"可以"、"建议"、"最好"等模糊措辞。如果某条不是强制要求，用陈述句说明"什么情况下用什么做法"而非"建议做什么"。

### 8.4 解释方式

| ✅ 允许 | ❌ 禁止 |
|--------|--------|
| 直接陈述定义 | "这就像..."（比喻） |
| 因果解释（"这是因为..."） | "你可以理解为..."（辅导口吻） |
| 条件说明（"当X时，应Y"） | "相当于..."（类比） |

### 8.5 禁止元评论

文档**不谈论自身**。以下写法严禁出现：

- ❌ "本文的核心内容" → ✅ 直接陈述："本章规定..."
- ❌ "如下是重点" → 重要性由章节层级传达，不需旁白
- ❌ "如前所述" → 改为"见§X.X"
- ❌ "首先我们来看..." → 直接进入内容

### 8.6 对比示例的标准写法

工程规范中的对比示例用于消除歧义，而非"教学"。标准格式：

```markdown
✅ 正确做法：
- 具体示例

❌ 错误做法：
- 具体示例
```

对比示例仅出现在需要消除理解歧义的关键判断点，不滥用。

### 8.7 FAQ 与常见错误的写法

FAQ 和常见错误章节使用"错误+正确"对比式表达，**不采用 Q/A（Question/Answer）格式**：

```markdown
**错误：分解粒度不当**

分解粒度过粗或过细是各层最常见的错误。

| 问题 | 表现 | 后果 |
|------|------|------|
| **粒度过粗** | 一条末级条目需要分配到下层多个节点 | 违反1:1约束 |
| **粒度过细** | 功能拆成过多条目 | 管理复杂，架构碎片化 |

✅ 正确做法：分解到每条末级只需下层一个架构末级节点承接。
```

FAQ 使用"问/答"的中文标记取代"Q/A"：

```markdown
**问：一个架构末级节点承接多少条上层条目合理？**

**答：** 建议每个架构末级节点承接2-5条（2-3条最优）。
```

常见错误和 FAQ 的语气可以比正文稍放松，但不得滑入培训腔——仍是规范文档的组成部分，只是面向排查场景的语言略有调整。

### 8.8 句子与段落可读性

以下规则确保文档读起来通顺、不绕、不啰嗦：

**单句单意。** 一句话表达一个核心意思。当一句话需要两个以上破折号（`——`）串联多个子句时，应拆分为多个独立的短句。

```markdown
❌ BA 可采用用户角色架构锚定法——以目标产品的全量用户角色集为推导锚点——也可采用输出产品架构锚定法——以输出产品的 PA 节点类型为推导锚点。

✅ BA 可采用用户角色架构锚定法，以目标产品的全量用户角色集为推导锚点。也可采用输出产品架构锚定法，以输出产品的 PA 节点类型为推导锚点。
```

**慎用"进行""加以"。** 这两个词在多数情况下可删除或替换为更直接的动词。

```markdown
❌ 对架构末级节点进行细化分解 → ✅ 细化分解架构末级节点
❌ 从以下维度进行验证 → ✅ 从以下维度验证
❌ 在 OR 层进行分离 → ✅ 在 OR 层分离
```

**避免重复表述。** 同一概念在相邻段落中不应重复定义或解释。确需指向已有定义时使用"见§X.X"。

```markdown
❌ BA 是功能需求完整性保障的系统性方法。BA 之所以作为五层中的独立层存在，是因为它为功能需求的完整性提供了系统性保障方法。
→ 两句意思相同，合并。

✅ BA 是功能需求完整性保障的系统性方法（非功能需求没有等价方法——路径差异见 §1.2）。
```

**避免教科书式铺垫。** 不写"随着...的深入"、"在...方面"、"众所周知"等背景引导句，直接进入内容。

```markdown
❌ 随着 AI 辅助设计的深入，AI 助理在各模式中的参与方式呈现系统性差异。
✅ AI 助理在各模式中的参与方式呈现系统性差异。
```

**缩写首次出现时标注全称。** 每章或每节中，缩写首次出现时应标注全称，如"系统需求（SysReq）"。同一节内重复出现时不再标注。

## 九、验收标准与检查清单的编写

### 9.1 适用范围标注

每节的适用范围标注统一使用斜体 `适用步骤：§X.X`：

```markdown
*适用步骤：§6.3（SR架构）、§6.4（BA架构）*
```

### 9.2 验收标准

使用表格格式，包含检查项、标准和说明三列：

```markdown
| 检查项 | 标准 | 说明 |
|--------|------|------|
| **架构末级节点定义完整率** | 100% | 每个架构末级节点须具备四要素 |
```

### 9.3 检查清单

使用任务列表格式：

```markdown
- [ ] 遍历每层方案文档的架构树
- [ ] 检查架构定义子文档的第二部分（映射关系表）
```

> **区分原则**：验收标准提供量化指标（"覆盖率 100%"），检查清单提供操作指导（"遍历每层架构树"），两者不重叠。

## 十二、预处理文档两区结构约定

本章适用于预处理链六个 Skill（ort00~ort05）中处理的中间文档，包括文本化文档 `{原始文件名}.md` 和切分文件 `-sub-{seq}.md`。系统设计过程文档和资产文档按 §十四 执行。

预处理文档分为管理信息区和文档正文区两个区，不设置历史归档区。历史版本由 Git 管理。文档内仅保留当前正文、状态摘要、变更记录、迭代记录和终态决策记录。

### 12.1 两区结构总览

```
┌─ 管理信息区 ──────────────┐  文档第一部分
│ 文档属性（frontmatter）     │  机器读
│ 状态摘要                   │  人类读
│ 变更记录表                  │  人类操作 / AI 转写
│ 迭代记录块                  │  执行追溯摘要
│ 决策记录/废弃说明            │  决策追溯
├─ 文档正文区 ──────────────┤  文档第二部分
│ 正文内容（当前版本）          │
└───────────────────────────┘
```

两个区依次排列，管理信息区在上，文档正文区在下。

**两区划分原则**：

| 区域 | 承载内容 | 示例 |
|------|---------|------|
| **管理信息区** | 关于文档的管理信息、操作入口和过程摘要 | 状态摘要、变更记录表、迭代记录、决策记录 |
| **文档正文区** | 当前版本的正文内容 | 文本化后的原始材料原文、澄清后的当前正文 |

### 12.2 管理信息区各区块格式

#### 12.2.1 文档属性（frontmatter）

仅切分文件包含，位于文件最顶部，`---` 分隔：

```yaml
---
chunk:
  source: {原始文件名}.md
  original_source: {原始文件名}.{ext}
  seq: {序号}
  total: {总片数}
  label: {语义标签}
  range: {原文范围}
status: raw
---
```

`status` 沿链递进：`raw` → `annotated` → `clarified` → `normalized`。文本化文档无此区块。

#### 12.2.2 状态摘要

位于文档属性（如有）之后、正文之前。固定框体，完全替换旧内容。

**格式：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
文件状态
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段：[当前阶段名]
AI建议：[建议值 | —]
待处理：[简述 | 无]
操作指引：[人类下一步应做的事]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**字段说明：**

| 字段 | 说明 |
|------|------|
| 阶段 | 当前文件所处的处理环节（如切分完成、首轮标注完成、已澄清、已规范化、已废弃） |
| AI建议 | AI 对当前阶段的判断结论。无建议值时填“—” |
| 待处理 | 人类待办事项概述（项数或“无”） |
| 操作指引 | 人类下一步应执行的操作 |

**写入规则：** 每次完全覆盖旧摘要，不累计不追加。各 Skill 的写入变体见各自的操作规程。

#### 12.2.3 变更记录表

位于状态摘要之后。5 列固定格式：

```
## 变更记录

| 日期 | 操作者 | 操作 | 说明 | 状态 |
|------|--------|------|------|------|
| {YYYY-MM-DD} | AI | {操作名} | {说明文字} | 待确认 |
| {YYYY-MM-DD} | AI | 切分建议 | 确认切分 → 按当前标记执行切分（3片） | 待确认 |
```

**状态取值：** 待确认（初始写入）、已完成（AI 执行后）、已终止（人类或 AI 废弃后）。

**操作提示：** 变更记录表末尾应提示人类可通过对话确认保留项，AI 将对话反馈转写到变更记录表；也可手工编辑变更记录表作为备选入口。

#### 12.2.4 迭代记录块

位于变更记录表之后。记录每次执行的摘要。`##` 为记录类别标题，`###` 为单次执行记录。

**切分迭代记录格式：**

```
## 切分迭代记录

### 执行 #2 · 2026-06-07 · 重切 · 批次 20260607-v2
| 片序号 | 切分文件 | 语义标签 | 范围 |
|--------|---------|---------|------|

### 执行 #1 · 2026-06-06 · 首次切分 · 批次 20260606-v1
| 片序号 | 切分文件 | 语义标签 | 范围 |
|--------|---------|---------|------|
```

**规范化迭代记录格式：**

```
## 规范化迭代记录

### 执行 #2 · 2026-06-07 · 重规范化 · 批次 20260607-v2
| OR 编号区间 | 临时编号区间 | 源切分文件 | 写入位置 |
|------------|-------------|--------|---------|
```

**插入规则：** 重迭代时在已有记录之前插入新记录（最新记录最上）。迭代记录只写摘要，不复制旧正文全文。

#### 12.2.5 决策记录

位于变更记录表之后。仅在达到终态时写入一次。

**格式：**

```
本步关键决策记录：
├─ 文档：[文件名]
├─ 处理记录：
│  ├─ 文件类型：[原始文件类型]
│  ├─ 初处理日期：[YYYY-MM-DD]
│  └─ 最终判定日期：[YYYY-MM-DD]
├─ 生命周期状态：[当前值]
└─ 变更记录表操作：[幸存行] + [检测情形]
```

#### 12.2.6 废弃说明块

位于决策记录之后（管理区最底部），仅在废弃场景写入。

**格式：**

```
> **【废弃说明】**
> | 项目 | 内容 |
> |------|------|
> | **废弃时间** | [YYYY-MM-DD] |
> | **决策人** | [姓名] |
> | **废弃原因** | [说明] |
> | **当前生命周期状态** | [废弃前状态] |
```

### 12.3 写入规则

| 规则 | 说明 |
|------|------|
| **完全覆盖** | 状态摘要每次写入完全替换旧内容，不累计、不追加 |
| **谁写谁维护** | 每个往文件中写入内容的 Skill 同时更新管理区中对应的区块 |
| **两区排列** | 管理信息区是文档的第一部分，文档正文区是第二部分。管理区内区块排列：文档属性 → 状态摘要 → 变更记录表 → 迭代记录块 → 决策记录 → 废弃说明块（可选） |
| **不保留历史区** | 旧正文、旧注释、旧分析块不在文档内归档；历史版本由 Git 管理 |

### 12.4 正文区

正文区位于管理信息区之后，是文档的第二部分，承载**当前版本**的正文内容。正文区仅包含当前正文内容，不附带历史版本。

| 块 | 说明 | 写入者 |
|----|------|--------|
| **正文内容** | 当前版本的原文（澄清后可被 AI 更新为新版本） | ort00（首次写入）、ort03（澄清后更新）、ort05（规范化不更新正文） |

**写入规则**：
- ort00 首次写入：文本化原始材料原文
- ort03 更新正文：AI 根据人类追加的澄清注释生成新正文，直接替换正文区旧内容；旧版本由 Git 追溯
- ort05 不更新正文：规范化不改变原文内容
- ort04 的分析建议块为工作态叠加，步骤结束后清理；分析结果若有留痕价值，应摘要写入管理信息区

### 12.5 文档类型差异

| 区块 | 文本化文档（`.md`） | 切分文件（`-sub-*.md`） |
|------|-------------------|------------------------|
| 文档属性 | 无 | `---` 分隔的 YAML，含 `status` |
| 状态摘要 | ✅（由 ort00/ort01 维护） | ✅（由 ort01~ort05 维护） |
| 变更记录表 | 切分确认 / 形态② | 澄清确认 / 规范化确认 / 形态② |
| 迭代记录块 | 切分迭代记录 | 规范化迭代记录 |
| 决策记录 | 切分决策记录 | 澄清/规范化终态决策记录 |
| 废弃说明 | 可选 | 可选 |
| **正文内容** | **✅（当前版本原文）** | **✅（当前版本原文片段）** |
| **历史区** | 不设置 | 不设置 |

### 12.6 工作态叠加与清理

预处理链各步骤执行期间，AI 可将工作态内容临时叠加到正文区（行内标记、分析建议块等），服务于当前步骤的工作需要。步骤完成后，由步骤的 execute 端 Skill（ort01/ort03/ort05）清理这些叠加内容。需留痕的信息应摘要写入管理信息区，不得将旧正文或工作态全文转入历史区。

**叠加与清理对应表**：

| 步骤 | 叠加内容 | 叠加与处理方式 | 处理者 | 处理时机 |
|------|---------|--------------|--------|---------|
| ort00~ort01（切分） | 行内切分标记 | **清理**：删除标记，不留历史 | ort01 | 切分执行后 |
| ort02~ort03（澄清） | 行内模糊标注、模糊标注报告、人类追加的澄清注释 | **清理并摘要留痕**：AI 据注释生成新正文，正文区替换为当前版本；澄清摘要写入管理信息区 | ort03 | 状态转迁至已澄清时 |
| ort04~ort05（规范化） | 条目拆分方案、冲突检测结果 | **清理**：删除分析建议块（信息已写入 OR 基线，正文区保持原文） | ort05 | 写入 OR 基线后 |

**终态保证**：任一预处理步骤完成后，正文区内容=当前版本的原文，不残留工作态产物。

**管理区留痕**：管理区的变更记录表、迭代记录、决策记录作为操作历史永久保留。需追溯的处理过程写摘要，不复制旧正文全文。

## 十三、Skill 文档操作流程 Step 布局约定

### 13.1 适用范围

本约定适用于所有 Skill 文档（wft01、wft02、ort01 等）中 **§三 操作流程** 的 Step 布局描述。每个 Skill 的操作流程应以 Step 布局总览表开头，提供全流程的快速导航。

### 13.2 Step 布局总览表格式

每个 Skill 的 §三 操作流程首段应为以下格式的总览表：

```
| Step | 名称 | 职责 | 执行模式 | 产出 | 写资产 |
|------|------|------|---------|------|-------|
| Start | {名称} | {一句话职责} | {执行模式} | {产出物} | {目标资产或—} |
| 1 | {名称} | {一句话职责} | {执行模式} | {产出物} | {目标资产或—} |
| ... | ... | ... | ... | ... | ... |
| End | {名称} | {输出交互} | 人类 | 行动清单 | — |
```

列定义：

| 列 | 必填 | 说明 |
|----|------|------|
| Step | 是 | Start, 1, 2, ..., N, End。不跳号，不重复 |
| 名称 | 是 | 2~6 字，动词性短语（如"资产加载""分析活动""方案反馈处理"）|
| 职责 | 是 | 一句话描述本步核心职责（~25 字以内），不含过程细节 |
| 执行模式 | 是 | 统一 / 两链并行 / 仅某链 / 人类 等 |
| 产出 | 是 | 本步的关键产出物名称 |
| 写资产 | 否 | 本步**直接写入**的资产文件（多个时用顿号分隔）。不直接写资产则填"—" |

### 13.3 约束规则

1. **Step 编号体系**：Start → 1 → 2 → ... → N → End。编号连续不跳号
2. **每个 Step 一个职责**：不把多个无关操作合并到一个 Step。识别→匹配→写资产应各自独立或按资产维度垂直切分
3. **资产管理 Step 的特殊地位**：如果某个 Step 直接写资产文件（"写资产"列非"—"），该 Step 应自包含"读源→匹配→决策→写资产"的完整闭环，不依赖后续 Step 代写
4. **执行模式一致性**："写资产"列有值的 Step 应标注具体的执行模式（仅业务链 / 仅平台链 / 两链交叉等），便于人类理解谁在什么时机写资产
5. **总览表之后**：总览表之后按 Step 展开详细操作表（格式见 13.4），每个普通 Step 对应一个 `### Step {N}` 小节
6. **End 行处理**：End 行必须出现在 Step 布局总览表中。运行结束输出（Step End）可独立成章，不必作为 §三 下的普通 Step 小节展开

### 13.4 Step 详细操作表格式

总览表之后，每个 Step 展开为一个详细的操作用户故事表格：

```
### Step {N} · {名称}（{执行模式补充}）

> {可选：本步的补充说明、源和目标资产}

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| {序号} | {操作名} | AI/人类 | {操作的具体说明} |
```

- 如果该 Step 写资产，在说明中注明 `**直接写入 {资产文件名}**`
- 如果该 Step 有前置依赖（需上一步产出），在补充说明中注明"**源**：..."
- Step End 若独立成章，应在 Step 布局总览表的 End 行中注明"独立成章，见运行结束输出章节"

## 十四、树形资产文档布局规范

本章适用于系统设计过程文档和资产文档，包括 `01~09` 系统设计数据文档，以及 `21~26` 等资产文档。预处理链中间文档仍按 §十二 执行。

系统设计过程文档和资产文档本质上是树形资产。文档应支持人类完整预览，也应支持 Skill 按索引和节点块局部读取。

### 14.1 统一布局

树形资产文档应采用以下布局：

```
0. 树画像 / 资产全貌
1. 状态与处理队列
2. 节点索引 / 分片索引
3. 规则区 / 口径区（如适用）
4. 正文节点块
5. 追溯关系
6. 反馈区 / 建议区
9. 文档版本与变更记录
```

各区含义如下：

| 区域 | 作用 | Skill 读取方式 |
|------|------|---------------|
| 树画像 / 资产全貌 | 提供资产整体结构、节点规模、主要分片、设计风格摘要 | Step Start 必读 |
| 状态与处理队列 | 标识本轮可处理对象、状态、人类决策、入口块 | Step Start 必读 |
| 节点索引 / 分片索引 | 将节点 ID、父子关系、状态、块 ID 建立可定位索引 | 按目标对象读取 |
| 规则区 / 口径区 | 记录识别规则、边界口径、命名原则、归并原则 | 按 Skill 需要读取 |
| 正文节点块 | 存放节点定义、详细字段、承接关系、局部反馈区 | 只读目标块和相关块 |
| 追溯关系 | 记录跨文档、跨层级的来源和承接关系 | 按目标对象读取 |
| 反馈区 / 建议区 | 承载人类反馈、AI 建议、待确认项 | 按目标对象读取和写回 |
| 文档版本与变更记录 | 记录轻量变更历史，按时间倒序排列 | 仅在版本检查时读取 |

### 14.2 树画像与资产全貌

树画像 / 资产全貌是 AI 建立全局资产感知的入口。该区只保留轻量摘要，不写完整定义。

设计产物型文档应提供树画像，例如：

```
| 节点ID | 名称 | 节点类型 | 父节点ID | 层级 | 状态 | 一句话边界 | 块ID |
|--------|------|----------|----------|------|------|------------|------|
| BA-IPO-001 | 采购申请处理 | IPO | BA-PROC | 5 | 待系统需求设计 | 处理采购申请的接收、判断和处置 | BA-IPO-001 |
```

资产台账型文档应提供资产全貌，例如：

```
| ID | 名称 | 层级 | 父级ID | 业务域 | 状态 | 一句话边界 | 典型文档 | 块ID |
|----|------|------|--------|--------|------|------------|----------|------|
| @node-purchase-order | 采购订单管理 | 构件类型 | @prod-procurement-product | 采购管理 | 已确认 | 管理采购订单生成、变更和跟踪 | 采购订单 | PROD-PROCUREMENT |
```

产品类型 + 构件类型数量在可控规模内时，可作为产品架构类资产的全貌层。末级节点、详细定义、完整 I/O、历史记录不得写入全貌层。

### 14.3 分片规则

当节点规模较大时，树画像 / 资产全貌应分层组织：

```
L0 全局画像：节点总数、层级结构、分片目录、设计风格摘要
L1 分片目录：按业务域、产品类型、架构层级或状态列出分片
L2 分片摘要：描述目标分片内的节点类型、边界、命名和典型对象
L3 节点正文块：存放具体节点定义
```

分片原则：

1. 业务架构节点达到大几千规模时，应按业务域、产品类型或场景域分片。
2. 产品架构节点达到小几千规模时，应按产品类型、构件类型、平台链 / 业务链或引擎类型分片。
3. Skill 默认读取 L0 + L1，再按本轮对象读取候选 L2 和目标 L3。
4. Skill 不得默认全量读取大文档正文。

### 14.4 节点正文块

正文节点应使用稳定块标识，便于 AI 定位、读取和写回。

```
<!-- BLOCK: BA-IPO-001 -->
### BA-IPO-001 采购申请处理

#### 基本信息

#### 节点定义

#### 承接关系

#### 反馈区
<!-- 人类对话反馈由 AI 转写到这里 -->
<!-- /BLOCK: BA-IPO-001 -->
```

块标识规则：

1. `BLOCK` ID 应稳定，不随标题名称微调而改变。
2. 节点索引中的 `块ID` 必须能定位到对应正文块。
3. 一个节点的状态、定义、承接关系和反馈区应尽量放在同一正文块内。
4. 跨文档关系只记录引用，不复制对方正文。

### 14.5 Skill 局部读取规则

Skill 应按以下顺序读取树形资产文档：

1. 读取树画像 / 资产全貌，建立全局资产感知。
2. 读取状态与处理队列，确定本轮处理对象。
3. 读取节点索引 / 分片索引，定位目标块和候选相关块。
4. 读取目标正文块。
5. 根据追溯关系读取必要的上游、下游或相邻块。

禁止事项：

1. 不得默认全量读取大文档正文。
2. 不得只读目标块而跳过全貌层。
3. 不得将历史版本、废弃正文或旧版归档作为当前资产参与设计判断。

### 14.6 反馈与留痕

人类可在 Markdown 预览模式阅读文档，并通过对话向 AI 提交 `[同意] / [修改] / [驳回]` 及具体意见。AI 必须先将对话反馈转写到目标节点反馈区，再执行状态推进、方案修改、资产写回或回退。

规则：

1. 对话是反馈输入通道，文档反馈区是权威留痕。
2. 反馈区应位于目标节点正文块内，或能通过节点索引定位。
3. AI 不得只凭对话上下文推进状态。
4. 文档反馈区保留为人类手工填写的备选入口。

### 14.7 不保留历史区

系统设计过程文档和资产文档不在正文内保留历史区、归档区或旧版正文。

历史版本由 Git 管理。文档内仅保留轻量变更记录，且变更记录按时间倒序排列。

若节点不再使用，应在节点状态中标记为 `已废弃`，并在节点反馈区或变更记录中说明原因。不得复制整版旧内容到历史区。

## 十五、资产写回协议

本章适用于所有 Skill 对资产文档的写入操作。Skill 写回资产时必须遵守以下规则，不得重写资产全文。

### 15.1 幂等与去重

1. 同一来源（STR 节点 ID + 原子单元名称）对同一资产条目的引用不得重复追加。
2. 写回前检查目标资产块是否已有同源引用，已有引用时更新摘要信息，不重复追加新行。
3. 同一产品类型 / 构件类型 / 文档定义 / 分类维度下的引用列表应去重维护。

### 15.2 局部写回

1. Skill 只更新本轮相关资产块，不得重写资产全文。
2. 写入范围限定为：目标正文块（引用追加或定义更新）+ 相关索引行 + 相关全貌摘要行 + 追溯引用行。
3. 不因写回而默认读取资产全文；只在索引缺失或块定位失败时扩大读取。

### 15.3 全貌同步

所有新增或变更的资产块必须同步维护以下内容：

| 维护项 | 新增资产块时 | 变更已有资产块时 |
|--------|-------------|----------------|
| 树画像 / 资产全貌 | 新增摘要行（含 ID / 名称 / 层级 / 父级 / 业务域 / 状态 / 一句话边界 / 块ID） | 更新对应摘要行（状态 / 边界描述 / 典型文档等） |
| 分片索引 | 新增索引行 | 更新索引摘要 |
| 状态队列 | 写入初始状态 | 更新状态 |
| 正文块 | 新增稳定 `BLOCK`（含块标识注释） | 局部更新正文内容 |
| 追溯引用 | 双向记录（资产侧 ← 来源 STR 节点 ID，来源文档侧 → 资产块 ID） | 追加引用链 |

### 15.4 写回摘要

Skill 完成资产写回后，应在来源文档（如 `02-*.md`）的本轮节点块中记录写回摘要：

```text
### 资产写回记录

| 日期 | 原子单元 | 目标资产 | 目标资产块 | 动作 | 判定 |
|------|---------|---------|-----------|------|------|
| 2026-06-23 | IC-采购申请单 | 23 | @comp-采购管理:采购申请 | 追加引用 | 23-Q1 |
| 2026-06-23 | IC-采购申请单 | 26 | @doc-采购申请单 | 更新归属 | 26-Q1 |
```

写回记录字段：
- **日期**：写回日期
- **原子单元**：被写回的原子单元名称（信息载体 / 配置单元 / 量化指标）
- **目标资产**：23 / 25 / 26 / 22
- **目标资产块**：写入的资产块 ID
- **动作**：追加引用 / 更新归属 / 新增块 / 扩展定义
- **判定**：Q1 / Q2 / Q3（与对准结果的判定一致）

### 15.5 资产建议写回

Q2 或 Q3 项应先形成正式资产建议，写入来源文档节点的 `### 资产建议反馈区`，等人类确认后再执行写回。不得在人类确认前写入资产文档正文。

资产建议至少包含：建议 ID、类型（Q2 扩展 / Q3 新增）、内容（建议的新增/扩展项名称和定义）、理由（引用来源和依据）、候选资产块 ID（如有）。
