## 一、核心概念

### 1.1 角色定位

#### 什么是 AOS

AOS（Enterprise Operation System）是企业运营体系的简称。本准则所关注的企业，企业范围涵盖机载设备的市场、研发、生产、售后及经营管理。

#### 什么是「运营体系开发之系统设计流水线」

该流水线是 AOS 开发者在开发企业运营体系时，用于将原始需求逐层推导至产品架构的系统设计工具。它定义了完整的 5 层设计链路、3 条核心规则和标准化的操作流程。

#### 三个关键角色

| 角色 | 职责 | 参与方式 | 关注的目录 |
|------|------|---------|-----------|
| **流水线开发者** | 开发、维护和优化系统设计流水线本身。定义设计准则、实践指南和 AI 辅助开发文档 | 开发（构建流水线） | `01-design-standards/`<br>`02-design-guidelines/`<br>`03-ai-support/` |
| **AI** | 按流水线开发者的指令执行自动化任务，生成文档初稿、检查一致性、验证映射关系 | 辅助（协助开发流水线） | `01-design-standards/`<br>`02-design-guidelines/`<br>`03-ai-support/` |
| **AOS开发者** | 使用系统设计流水线进行企业运营体系的系统设计（含需求提出者、业务流程开发者、信息系统开发者三类） | 使用（利用流水线做设计） | `01-design-standards/`<br>`02-design-guidelines/` |

#### 03-ai-support 的定位

`03-ai-support/` 目录是给**流水线开发者**（而非 AOS 开发者）使用的 AI 辅助开发文档，用于在开发流水线的过程中借助 AI 能力提升效率和质量。AOS 开发者在使用流水线时无需直接阅读该目录。

### 1.2 系统设计流水线的应用对象

本准则用于设计"运营体系开发之系统设计流水线"这个产品。该流水线是为企业运营体系（AOS）的开发者设计的，本准则所关注企业的业务范围包括机载设备的市场、研发、生产、售后以及经营管理。

流水线的应用对象是企业运营体系的开发者，主要包括三类人员：

| 开发者类型 | 角色 | 关注点 |
|-----------|------|--------|
| **需求提出者** | 提出运营体系建设、优化或改进需求 | 需求的完整性、清晰性、可追溯性 |
| **业务流程开发者** | 设计和实现业务流程 | 流程的合理性、可执行性、效率 |
| **信息系统开发者** | 设计和实现信息系统 | 系统的可实现性、性能、可维护性 |

**注**：本准则同时服务于流水线开发者、AI 和三类 AOS 开发者。在应用准则设计流水线时，应充分考虑这三类参与者的不同角色和关注点。

### 1.3 基本术语

| 术语 | 定义 | 详见 |
|------|------|------|
| **需求文档** | 描述系统应该做什么（What），包含功能需求和非功能需求 | §1.6.1 |
| **方案文档** | 描述如何满足需求（How），由架构定义和详细定义两个子文档组成 | §1.6.2 |
| **需求—方案配对** | 需求文档与方案文档成对出现，需求是对方案的要求，方案是对需求的满足 | §1.6.1 |
| **文档的双重属性** | 一个文档可同时扮演两种角色——对上层的方案和对下层的需求 | §1.6.1 |
| **架构定义** | 以架构末级节点为基本单位，构建方案的层级结构并建立对上层需求的映射 | §1.6.2 |
| **详细定义** | 对架构末级节点进行详细设计分解，使每条末级条目在后续方案的架构中有唯一的承接点 | §1.6.2 |
| **架构末级节点** | 被方案分类和组织的最大条目，是需求映射的关键层级 | §1.7 |
| **详细定义末级节点** | 对架构末级节点分解后得到的末级条目，是分配到后续方案架构末级节点的源头 | §1.5.1 |
| **详细定义的双重身份** | 详细定义既是当前方案架构定义的细化（向内），又是后续方案的需求（向下） | §1.5.2 |
| **同步设计** | 在开展上层详细定义的同时，同步开展下层方案架构定义（即使只是结构骨架层级） | §1.5.3 |
| **两阶段工作流程** | 先快速约定结构骨架并建立映射关系，再并行填充详细内容的工作模式 | §1.5.3 |
| **1:1分配约束** | 每条详细定义末级节点只能分配到下层方案中唯一一个架构末级节点 | §3.2·规则1 |
| **N:1承接支持** | 下层方案中一个架构末级节点可以承接上层需求的多个详细定义末级节点 | §3.2·规则2 |

### 1.4 系统设计流水线的目标

建立从**原始需求**到**产品架构**的完整设计链路，确保：
- ✅ 每条需求都有明确的承接点（不遗漏）
- ✅ 每个方案都充分满足上层需求（不过度设计）
- ✅ 完整的端到端追溯链路（可追踪）
- ✅ 发现并消除设计缺陷（高质量）

### 1.5 架构定义与详细定义的分离

**核心理念**：每份方案应包含两个子文档——架构定义文档和详细定义文档

| 文档 | 定义 | 作用 |
|------|------|------|
| **架构定义文档** | 以架构末级节点为基本单位，构建方案的层级结构并建立对上层需求的映射 | 是需求映射的关键层级 |
| **详细定义文档** | 对架构末级节点进行详细设计分解，使每条末级条目在后续方案的架构中有唯一的承接点 | 是上下层映射的衔接点 |

**分解标准**：一直分解到任意详细定义末级节点都只需找到**后续文档架构的一个末级节点**来承接即可

#### 1.5.1 架构末级节点的分解、分配与分析

**分解**（Decomposition）
对架构末级节点进行详细设计，逐层分解为更细的末级条目，直到每条都只需后续方案架构中的一个末级节点来承接。

**分配**（Allocation）
对详细定义末级节点，按1:1约束分配到后续方案的架构末级节点，确保每条末级条目都有唯一的承接点。

**分析**（Analysis）
通过两类分析——需求说明符合性分析和需求性能符合性分析——来验证分解的充分性和分配的正确性，发现潜在设计缺陷，确保结果合理可执行。

**需求说明符合性分析**：通过覆盖度检查、一致性校验和追溯验证等手段，确认每个架构末级节点是否充分满足所承接的上层需求，检查映射关系是否完整、正确、无矛盾。

**需求性能符合性分析**：通过建模仿真、理论计算等手段，验证设计方案能否达到所承接的性能指标要求（响应时间、吞吐量、并发数、可用性等），确保性能需求在架构层面可实现。

**例外——业务架构的独立存在**：
- 业务架构（第3层）暂时只需要**架构定义**（IPO清单），不需要详细定义
- 业务架构**独立存在**，不参与"详细定义的双重身份"递进链条
- 业务架构的架构末级节点（IPO）自身承担了"需求"的角色，直接映射到系统需求架构末级节点
- 这说明：各层方案文档是否都包含详细定义，取决于该层的设计需要

**这样做的好处**：
- 清晰地区分高层设计和具体实现
- 支持多个团队的并行工作
- 便于需求的追溯和映射
- 提高设计的可维护性

**对比示例**：
```
✅ 正确的分析时机：
  完成第2层架构定义 → 立即做符合性分析（检查SR映射完整性）
  → 发现遗漏 → 补充后再进入详细定义
  分析在详细定义之前完成，问题在早期被发现

❌ 错误的分析时机：
  第2层全部完成（架构+详细）→ 才开始做分析
  → 发现第3层无承接目标 → 需要回溯调整第2层详细定义
  分析滞后导致返工范围扩大
```

#### 1.5.2 详细定义的双重身份（流水线递进的核心机制）

**详细定义**在流水线中扮演着双重角色，这是设计能够层层递进的关键：

- **本层身份**：作为当前方案中**架构定义的细化**（向内分解）
- **下层身份**：作为流水线上**后续文档——架构定义的需求**（向下传递）

```
第2层：相关方需求（需求文档）
  ↓ 详细定义分配到第3层架构末级（1:1约束）
第3层：业务架构（方案文档——独立存在）
  └─ 架构定义：IPO清单 ← 承接第2层详细定义
       ↓ IPO直接映射到第4层架构末级（1:1约束）
第4层：系统需求（方案文档）
  ├─ 架构定义：系统需求条目清单
  └─ 详细定义：系统需求分解 ← 延续至第5层
```

**理解要点**：
- 业务架构独立存在，其架构末级节点（IPO）直接映射到系统需求架构末级节点，不需要经过详细定义的中转
- 系统需求的详细定义（用例分解），同时也是产品架构定义的输入
- 这种层层递进的关系，使流水线中的每一层都能承接上一层的细化结果

#### 1.5.3 详细定义与下层架构定义的同步设计

**核心原则**：详细定义末级节点需要分配到下层方案架构末级节点（1:1约束）。因此，在开展上层需求详细定义的同时，应同步开展下层方案架构定义——即使只是结构骨架层级。

**为什么要同步**：

- **提供分解的"靶子"**：下层架构定义的存在，使分解过程有明确的分配目标，确保每条详细定义末级节点在分解时就能判断"是否只需要一个下层架构末级节点承接"
- **避免无目标返工**：如果下层架构定义尚未建立，详细定义分解到末级后可能发现没有对应的下层目标可承接，导致需要调整分解粒度，造成返工
- **为并行设计创造条件**：同步建立架构结构骨架（即使只是末级节点名称和映射关系），上下游团队可并行填充详细内容，缩短整体设计周期

**实际操作**：

1. **同步建立**：在开始上层详细定义分解时，同步在下层方案文档中建立架构末级节点的结构骨架
2. **边分解边映射**：每完成一条详细定义末级节点，立即检查是否能分配到下层的一个架构末级节点；如果发现不能（需要多个），则说明分解不够，继续分解
3. **同步填充**：结构骨架冻结后，上下层同时进入详细设计阶段

> **实现方式**：两阶段工作流程（先约定结构骨架、再填充详情）是实现同步设计的具体操作方法。详见系统设计指南中关于两阶段工作流程的描述。

**对比示例**：
```
✅ 正确的同步设计：
  第2层详细定义（做骨架）↔ 第3层架构定义（做骨架）
  → 骨架完成后并行填充内容，周期缩短40%

❌ 错误的串行设计：
  第2层全部完成 → 才开始第3层
  → 发现第3层无法承接某些条目，回溯调整第2层，周期延长
```

### 1.6 文档类型定义

#### 1.6.1 需求—方案配对

系统设计流水线中的文档以**"需求—方案"配对**为基本组织单位：

| 文档类型 | 定义 | 示例 |
|---------|------|------|
| **需求文档** | 描述系统应该做什么（What），包含功能需求和非功能需求 | 原始需求、相关方需求 |
| **方案文档** | 描述如何满足需求（How），由**架构定义**和**详细定义**两个子文档组成 | 业务架构、系统需求、产品架构 |

**核心关系**：
- **需求是对方案的要求**，**方案则是对需求的满足**
- 有方案文档则必定有对应的需求文档；有需求文档也应该有对应的方案文档
- **例**：相关方需求体现了**文档的双重属性**——对原始需求来说是"方案"，对业务架构来说又是"需求"

#### 1.6.2 方案文档的构成

每份方案原则上由两个子文档构成：

| 子文档 | 定义 | 包含内容 | 示例 |
|-------|------|---------|------|
| **架构定义文档** | 以架构末级节点为基本单位，构建方案的层级结构并建立对上层需求的映射 | 架构末级节点定义、承接的需求、映射视图 | IPO清单、组件清单 |
| **详细定义文档** | 对架构末级节点进行详细设计分解，使每条末级条目在后续方案的架构中有唯一的承接点 | 实现细节、接口定义、数据分解 | I/P/O说明、用例分解 |

> 架构定义 + 详细定义 = 方案（原则上如此，业务架构为特例）

### 1.7 架构定义的两部分结构

每份架构定义都包含两个部分：

#### 1.7.1 第一部分：架构设计结果（树形结构）

**结构**：从根节点直到末级节点的完整树形结构

**节点分类**：
- **分类和组织节点**：末级节点之上的所有节点，用于对架构进行分类和组织
- **架构末级节点**：被分类和组织的最大条目，是需求映射的关键层级

**每个架构末级节点包含**：
- 节点定义：该架构末级节点的具体定义
- 承接的需求末级节点：该架构末级节点承接了上层需求的哪些末级节点（N:1映射）
- 符合性分析：该架构末级节点如何满足所承接的上层需求
- 为下层提供的设计指导：对下层方案设计的指导

**作用**：
- 清晰地展示整体架构的分类和组织
- 支持需求的追溯和映射
- 便于团队分工和协作

**示例结构**：
```
业务架构（根节点—0级）
└─ 科研管理业务域（业务域—1级）
   └─ 产品开发项目（端到端项目—2级）
      └─ 系统研制与交付（项目业务对象架构节点端到端业务—3级）
         └─ 系统设计工作包（工作包—4级）
            └─ 产品架构定义（计划组件—5级·架构末级节点）
               ├─ 定义：[业务组件的具体定义]
               ├─ 承接的需求末级：SR-F-001, SR-F-002
               └─ 为下层提供的设计指导：[对系统需求的指导]
```

---

#### 1.7.2 第二部分：需求映射视图（映射关系表）

**内容**：所承接需求的所有末级节点，以及承接该需求末级节点的架构末级节点

**结构**：
```
需求末级节点 | 承接的架构末级节点 | 超链接
SR-001      | 业务组件 1.1      | [跳转到第一部分]
SR-002      | 业务组件 1.1      | [跳转到第一部分]
SR-003      | 业务组件 1.2      | [跳转到第一部分]
...
```

**作用**：
- 清晰地展示需求和方案的对应关系
- 支持从需求视角快速定位到架构设计
- 便于验证映射的完整性和正确性

**导航方式**：
- 单击此处的架构末级节点，则直接跳到第一部分的对应架构末级节点
- 支持双向追溯：需求 ↔ 架构

---

#### 1.7.3 两部分的关系

| 部分 | 视角 | 结构 | 用途 |
|------|------|------|------|
| **第一部分** | 架构视角 | 树形结构（根→末级） | 展示架构的分类和组织 |
| **第二部分** | 需求视角 | 映射关系表 | 展示需求和架构的对应关系 |

**映射流向**：
- 第一部分的架构末级节点 ← 承接上层需求末级节点（N:1）
- 第一部分的架构末级节点 → 分配到下层方案的架构末级节点（N:1）

## 二、5层结构概要

### 2.1 完整流程图

```
第1层：原始需求及分析
    ↓ N:1 分配
第2层：相关方需求
    ├─ 架构末级节点 ↔ 详细定义末级节点
    │
    ├─ 功能部分 ↓ N:1 分配
    │   第3层：业务架构（独立存在）
    │   └─ 架构末级节点（IPO）—— 仅架构定义，无详细定义
    │        ↓ IPO直接映射（去重后）
    │   第4层：系统需求
    │   ├─ 架构末级节点 ↔ 详细定义末级节点
    │   │
    │   └─ ↓ N:1 分配
    │       第5层：产品架构
    │       ├─ 架构末级节点 ↔ 详细定义末级节点
    │
    └─ 非功能部分 ↓ N:1 分配
        第4层：系统需求（非功能）
        └─ ↓ N:1 分配
            第5层：产品架构
```

> **图例说明**：`↔` 表示**共存关系**（非双向映射），即该层同时存在两种或多种类型的末级节点。例如 `架构末级节点 ↔ 详细定义末级节点` 表示该层既有架构末级节点也有详细定义末级节点。

### 2.2 层级对照表

| 层级 | 文档名称 | 架构末级节点 | 详细定义末级节点 | 承接关系 |
|------|---------|------------|----------------|---------|
| 1 | 原始需求及分析 | - | 末级需求 | - |
| 2 | 相关方需求 | 相关方需求条目 | 对条目的分解 | 承接原始需求末级（N:1） |
| 3 | 业务架构 | IPO（业务组件） | —（暂无） | 承接相关方需求详细定义末级（N:1） |
| 4 | 系统需求 | 系统需求条目 | 对条目的分解 | 承接业务架构末级（N:1） |
| 5 | 产品架构 | 产品组件 | 实现细节 | 承接系统需求详细定义末级（N:1） |

---

### 2.3 五层定义

每一层的完整定义，提炼自对应的层规范。这是理解整个流水线的核心。

| 层级 | 名称 | 定义 | 本质 | 文档构成 |
|------|------|------|------|---------|
| **1** | **原始需求** | 未经规范化的、来自各相关方的原始表述，是系统设计流水线的起点 | "原材料"——只有需求，没有方案 | 原始需求清单 + 末级需求分解 |
| **2** | **相关方需求** | 对原始需求的规范化整理和组织，同时是业务架构的需求输入 | 既是方案（对第1层）又是需求（对第3层） | 架构定义 + 详细定义 |
| **3** | **业务架构** | 用户使用系统开展业务的运行概念（Operational Concept），即业务过程说明 | 独立存在的方案——仅架构定义（IPO清单），无详细定义 | 仅架构定义（IPO） |
| **4** | **系统需求** | 将业务操作过程转化为系统可实现的功能需求，并将相关方非功能需求转化为系统非功能需求 | 双重来源的方案（功能经BA、非功能直接承接SR-NFR） | 架构定义（0-9级/0-6级）+ 详细定义 |
| **5** | **产品架构** | 将系统需求转化为具体技术实现方案，为开发团队提供技术蓝图 | 技术实现方案——产品研发的最终输入 | 架构定义（组件清单）+ 详细定义（实现方案） |

**关键要点**：
- 第1层是唯一**只有需求、没有方案**的层，不遵循"需求—方案配对"模型
- 第3层是唯一**只有架构定义、没有详细定义**的层，其IPO自身承担"需求"角色
- 第2、4、5层都是完整的方案文档（架构定义 + 详细定义）
- 第4层有两套架构体系：功能需求（0-9级）和非功能需求（0-6级，平行架构）

## 三、设计方法

### 3.1 系统设计流水线开发六步法

本节描述了开发系统设计流水线产品时的六步工作流程。该流程将5层结构中的功能需求分支和非功能需求分支分离处理，实现上下游文档的解耦和并行推进。

#### 完整流程图

```
流水线原始需求
    │
    ▼
[第1步] 原始需求分析及相关方需求架构定义
    ├─ 输出：原始需求详细定义
    │       相关方需求架构定义（功能+非功能）
    │
    ├─ 功能分支 ↓
    │  [第2步] 相关方需求功能部分详细定义及业务架构定义
    │  ├─ 输出：相关方需求功能部分详细定义
    │  │       业务架构定义（IPO）
    │  │
    │  └─ [第3步] 系统需求功能部分架构定义
    │     ├─ 处理：业务架构IPO去重并分类组织
    │     └─ 输出：系统需求功能部分架构定义
    │
    └─ 非功能分支 ↓
       [第4步] 相关方需求非功能部分详细定义及系统需求非功能部分架构定义
       └─ 输出：系统需求非功能部分架构定义

[第5步] 系统需求详细定义及产品架构定义
    ├─ 功能部分：系统需求详细定义分解到9级 → 分配到产品架构末级节点（1:1约束）
    └─ 非功能部分：非功能需求约束产品架构对非功能需求的符合性

[第6步] 双向追溯验证
    ├─ 输出：需求追溯矩阵
    │       验证报告
    └─ 检查：1:1约束、N:1支持、双向追溯
```

#### 分步说明

##### 第1步：原始需求分析及相关方需求架构定义

**目标**：从原始需求产出原始需求详细定义和相关方需求架构定义

**输入**：流水线原始需求（未经规范化的原始表述）

**处理**：
1. 原始需求**初步整理**（碎片归并、补全语境、标注依赖、过滤无效）——对来自多渠道的零散需求先合拢归类，为后续处理提供合理的输入粒度
2. 原始需求规范化（格式统一、术语标准化、清晰性检查）
3. 原始需求分解到末级（原子性、不可再分）
4. 基于末级需求，分类形成相关方需求架构——包括功能需求架构和非功能需求架构

**输出**：
- 原始需求详细定义（第1层文档）
- 相关方需求架构定义（第2层架构定义文档——树形结构 + 映射关系表）

---

##### 第2步：相关方需求功能部分详细定义及业务架构定义

**目标**：沿功能分支推进，产出相关方需求功能部分详细定义和业务架构定义

**输入**：第1步产出的相关方需求架构定义（功能部分）

**处理**：
1. 对相关方需求架构中的功能部分架构末级节点进行详细设计分解
2. 分解到每条详细定义末级节点只需映射到业务架构的一个IPO（1:1约束）
3. 根据详细定义结果，识别并定义业务架构IPO（Input/Process/Output）
4. IPO去重：复用 → 改进 → 新增

**输出**：
- 相关方需求功能部分详细定义（第2层详细定义文档）
- 业务架构定义（第3层文档——IPO清单）

---

##### 第3步：系统需求功能部分架构定义

**目标**：将业务架构IPO去重并分类组织，形成系统需求功能部分的架构定义

**输入**：第2步产出的业务架构定义（IPO清单）

**处理**：
1. 对业务架构中的所有IPO进行去重
2. 将去重后的IPO按系统功能分类组织
3. 形成系统需求功能部分架构

**输出**：
- 系统需求功能部分架构定义（第4层架构定义文档——功能部分）

---

##### 第4步：相关方需求非功能部分详细定义及系统需求非功能部分架构定义

**目标**：沿非功能分支独立推进，产出系统需求非功能部分架构定义

**输入**：第1步产出的相关方需求架构定义（非功能部分）

**处理**：
1. 对相关方需求架构中的非功能部分架构末级节点进行详细设计分解
2. 将非功能需求按三大类组织（质量特性、环境适应性、可实现性）
3. 形成系统需求非功能部分架构

**输出**：
- 相关方需求非功能部分详细定义（第2层详细定义文档——非功能部分）
- 系统需求非功能部分架构定义（第4层架构定义文档——非功能部分）

---

##### 第5步：系统需求详细定义及产品架构定义

**目标**：合并功能和非功能分支，产出系统需求详细定义和产品架构定义

**输入**：
- 第3步产出的系统需求功能部分架构定义
- 第4步产出的系统需求非功能部分架构定义
- 第2步产出的相关方需求功能部分详细定义（作为符合性分析的参考）

**处理**：
1. **功能部分**：对系统需求功能部分架构末级节点进行详细设计分解至9级场景活动，每条详细定义末级节点分配到产品架构末级节点（1:1约束）
2. **非功能部分**：以系统需求非功能需求约束产品架构对非功能需求的符合性
3. 综合形成产品架构定义（前后端组件、基础设施组件、实现细节、性能指标）

**输出**：
- 系统需求详细定义（第4层详细定义文档）
- 产品架构定义（第5层文档）

---

##### 第6步：双向追溯验证

**目标**：验证完整的追溯链路

**输入**：全部6份已生成的设计文档

**处理**：
1. 1:1分配约束验证——每条详细定义末级节点是否分配到唯一的方案架构末级节点
2. N:1承接支持验证——每个方案架构末级节点的承接数量是否合理
3. 双向追溯验证——从原始需求到产品架构正向可追溯、反向可追踪

**输出**：
- 需求追溯矩阵（端到端追溯）
- 验证报告（完整性和一致性）

---

### 3.2 三条核心规则

#### 规则1：1:1 分配约束

**定义**：每条详细定义末级节点都应分配到下层方案中**唯一一个**架构末级节点

**应用范围**：
- 原始需求末级 → 相关方需求架构末级（1:1）
- 相关方需求详细定义末级 → 业务架构末级（1:1）
- 业务架构末级（IPO） → 系统需求架构末级（1:1）
- 系统需求详细定义末级 → 产品架构末级（1:1）

> **注**：业务架构无详细定义，其架构末级节点（IPO）在映射至系统需求时，自身承担了"详细定义末级"的角色，因此适用1:1约束。

**判断标准**：

- **必要性**（何时必须继续分解）：❌ 一个详细定义末级节点需要分配到下层方案的**多个**架构末级节点 → 说明分解不充分，这个条目还需要继续分解
- **充分性**（何时可以停止分解）：✅ 一个详细定义末级节点只需分配到下层方案的**一个**架构末级节点 → 说明分解已充分，可以停止

**实际操作**：
1. 设计详细定义末级节点时，问自己："这个条目能映射到下层方案的一个架构末级节点吗？"
2. 如果答案是"不能，需要多个"，说明分解不够，继续分解
3. 如果答案是"可以"，则分解完成，停止分解

**对比示例**：
```
✅ 分解充分（满足1:1约束）：
  SR-F-001.1（需求收集）→ BA-需求收集流程
  SR-F-001.2（需求规范化）→ BA-规范化流程
  → 每条详细定义末级只需一个架构节点承接

❌ 分解不充分（违反1:1约束）：
  SR-F-001.1（需求处理）→ BA-需求收集流程 + BA-规范化流程
  → 一条详细定义需要两个架构节点承接，继续分解
```

> **注**：规则1与规则2描述的是同一映射关系的两个视角——规则1从**源头**出发（向下分配，1:1约束），规则2从**目标**出发（向上承接，N:1支持）。两者共同定义了映射的双向约束。

---

#### 规则2：N:1 承接支持

**定义**：下层方案中一个架构末级节点可以承接上层需求的**多个**详细定义末级节点

**应用范围**：
- 原始需求末级 → 相关方需求架构末级（N:1）
- 相关方需求详细定义末级 → 业务架构末级（N:1）
- 业务架构末级（IPO） → 系统需求架构末级（N:1）
- 系统需求详细定义末级 → 产品架构末级（N:1）

**实际操作**：
1. 设计架构末级节点时，问自己："这个方案条目应该承接多少个上层需求？"
2. 如果承接的需求太少（1个），考虑是否可以合并
3. 如果承接的需求太多（>5个），考虑是否需要拆分

**对比示例**：
```
✅ 合理的N:1承接：
  BA-001（订单创建）承接 SR-F-001.1, SR-F-002.1, SR-F-003.1
  → 3条需求，均与订单创建相关，逻辑内聚

❌ 不合理的N:1承接：
  BA-001（系统管理）承接 SR-F-001.1, SR-F-005.1, SR-NFR-001, ...
  → 承接的需求之间缺乏逻辑关联，应拆分或重新分组
```

---

#### 规则3：双向追溯

**定义**：支持从上层需求追溯到下层方案，也支持从下层方案反向追溯到上层需求

**实现方式**：
- 在每个方案条目中标注"承接了哪些上层需求"
- 在每条上层需求中标注"分配到哪个下层方案条目"
- 通过超链接实现快速导航

**关键原则**：
- ✅ 上层详细定义末级节点分配到**下层方案的架构末级节点**（不是下层方案的详细定义末级）
- ✅ 这样可以保持需求和方案的对应关系清晰
- ✅ 架构末级节点是需求映射的关键层级

**实际操作**：
1. 每个方案条目都要明确标注"承接的上层需求"
2. 每条上层需求都要明确标注"分配到的下层方案条目"
3. 使用超链接建立双向导航

**对比示例**：
```
✅ 双向追溯完整：
  SR-F-001.1 → 标注"分配到 BA-需求收集流程"
  BA-需求收集流程 → 标注"承接 SR-F-001.1"
  → 正向反向均可追溯

❌ 只追溯了一半：
  SR-F-001.1 → 标注了分配目标（有）
  BA-需求收集流程 → 未标注承接来源（缺失）
  → 从下往上的追溯断裂
```

## 四、每层的具体要求

### 4.1 第1层：原始需求及分析

**定义**：原始需求是系统设计流水线的起点，是未经规范化的、来自各相关方的原始表述（会议纪要、邮件、已有文档等）。它是唯一不遵循"需求—方案配对"模型的层——只有需求，没有方案。

**核心特征**：来源多样、格式不一、质量不均、缺乏分类。原始需求是"原材料"，经过规范化、分解、分类等加工后，转化为结构化的相关方需求。

**本层的分析活动**：

- **需求说明符合性分析**：冲突检测（识别语义直接/间接矛盾的需求）+ 重复检测（识别完全/部分/相似重复的需求）
- **需求性能符合性分析**：验证性能指标是否明确量化，检查指标是否有明确的适用条件和验证方法

分析活动在规范化完成后、映射到相关方需求前集中执行。

**这一层的目的**：收集和分解原始需求，为后续设计提供清晰的输入

**处理流程**：
原始需求从收集到输出，按顺序经历四个阶段：

| 阶段 | 说明 | 主要工作 |
|------|------|---------|
| **① 收集** | 从各渠道获取原始表述 | 记录原始文本、提出方、来源渠道 |
| **② 初步整理** | 对零散碎片需求进行归并整理 | 碎片归并、补全语境、标注依赖、过滤无效 |
| **③ 规范化+分解** | 将整理后的需求标准化并分解到末级 | 四项检查、功能/非功能分离、原子性分解 |
| **④ 冲突/重复检测** | 检查需求之间的矛盾和冗余 | 语义对比、相似度分析 |

> **注**：初步整理是**可选但建议执行**的环节——当原始需求来自多个渠道、粒度参差不齐、或存在碎片化描述时，建议先归并整理再进入规范化分解，以提高处理效率。

**初步整理的四种方法**：

| 方法 | 适用场景 | 操作要点 |
|------|---------|---------|
| **血缘归并** | 碎片描述同一功能的不同切面 | 按业务血缘聚类，还原完整功能表述 |
| **角色聚合** | 不同角色对同一功能的诉求 | 合并为一条需求，标注角色差异 |
| **流程串联** | 碎片是同一流程的不同步骤 | 按操作流程串联，保持流程完整性 |
| **父子补全** | 一条主需求 + 多条补充细节 | 以主需求为载体，补充细节作为子约束 |

**初步整理的五条规则**：

1. **不做过度合并** — 不同功能域的碎片各自独立，不强行合并为"系统功能管理"类宽泛条目。判断标准：合并后能否用一句话说清核心功能
2. **保留原始出处** — 每条碎片合并后记录提出方、原始表述、日期，确保合并可回溯。后续发现合并有误时可回退
3. **性能指标不合并** — 功能描述可以合并，但性能指标单独提取。因为性能指标在后续流程中走不同的处理分支
4. **相互矛盾的碎片标注不合并** — 语义直接矛盾的碎片保持独立，标注"冲突待确认"，由相关方决策
5. **无法判断的标记为"待补全"** — 信息量不足的碎片不猜测，由提出方补充上下文

**整理后的输出格式**：

```
原始需求 [OR-XXX]
├─ 需求标题：[一句话概括]
├─ 需求描述：[合并后的完整需求描述]
├─ 原始碎片清单：
│  ├─ [碎片1原始表述] — [提出方] [日期]
│  └─ [碎片2原始表述] — [提出方] [日期]
├─ 包含的性能指标：[如有，单独列出]
├─ 依赖/冲突关系：[如有，标注]
└─ 整理说明：[归并依据]
```

**内容结构**：
```
原始需求 1
├─ 需求描述：[具体的需求定义]
├─ 末级需求分解：
│  ├─ 末级需求 1.1 → [分配到相关方需求架构末级]
│  ├─ 末级需求 1.2 → [分配到相关方需求架构末级]
│  └─ 末级需求 1.3 → [分配到相关方需求架构末级]
└─ 优先级：[高/中/低]
```

**关键要求**：
- ✅ 每条原始需求都应分解到末级需求
- ✅ 末级需求应该是原子性的、不可再分的
- ✅ 每条末级需求都应分配到相关方需求架构末级节点
- ✅ 文档中只保留纯需求清单，去除解释性内容
- ✅ 来自多渠道的零散需求在规范化前应先进行初步整理（碎片归并、补全语境、标注依赖、过滤无效）

---

### 4.2 第2层：相关方需求

**定义**：相关方需求是对原始需求（第1层）的方案文档，同时又是业务架构（第3层）的需求输入。由两份子文档构成——架构定义文档（对需求的分类和组织）和详细定义文档（对架构末级节点的分解）。

**本层的分析活动**：

- **需求说明符合性分析**：检查每个架构末级节点是否充分满足所承接的原始需求末级，验证N:1映射的完整性，验证功能与非功能需求的分离正确性
- **需求性能符合性分析**：验证性能指标是否从原始需求正确继承和量化，检查指标是否可测量、可验证

分析活动在架构定义完成后和详细定义开始前集中执行。

**功能 vs 非功能的处理区别**：

| 方面 | 功能需求 | 非功能需求 |
|------|---------|----------|
| 映射目标 | 业务架构的IPO | 系统约束（不映射到业务架构） |
| 分解方式 | 支持多层分解 | 不需要分层分解，直接到末级 |
| 分解终点 | 能映射到一个IPO | 能独立验证的约束条件 |

**这一层的目的**：将原始需求转化为相关方能理解的需求规范

**相关方需求的本质**：
- 是对原始需求的方案
- 由两份子文档构成：架构定义文档（对需求的分类和组织）和详细定义文档（对架构末级节点的分解）
- 分解标准：直到每个详细定义末级节点只需映射到业务架构的一个IPO

**内容结构**：

**架构定义文档**

**第一部分：架构设计结果（树形结构）**

从根节点直到末级节点的完整树形结构：
```
相关方需求（根节点）
├─ 功能需求类（分类节点）
│  ├─ SR-F-001: [相关方需求名称]（架构末级节点）
│  │  ├─ 定义：[需求的定义]
│  │  ├─ 承接的原始需求末级：原始需求 1.1, 1.2, 1.3
│  │  ├─ 符合性分析：[如何满足所承接的需求]
│  │  └─ 为下层提供的设计指导：[对业务架构的指导]
│  └─ SR-F-002: [相关方需求名称]（架构末级节点）
└─ 非功能需求类（分类节点）
   └─ SR-NFR-001: [相关方非功能需求]（架构末级节点）
```

**第二部分：需求映射视图（映射关系表）**

展示所有原始需求末级到相关方需求的对应关系：
```
原始需求末级 | 承接的相关方需求 | 超链接
原始需求 1.1  | SR-F-001        | [跳转到第一部分]
原始需求 1.2  | SR-F-001        | [跳转到第一部分]
原始需求 1.3  | SR-F-002        | [跳转到第一部分]
```

**详细定义文档**

**详细定义末级节点示例**

对架构末级节点的详细分解：
```
SR-F-001.1: [对SR-F-001的分解条目]
├─ 描述：[详细说明]
├─ 验收标准：[验收条件]
├─ 分配到：[业务架构末级节点BA-001]
└─ 符合性说明：[如何满足SR-F-001]
```

**关键要求**：
- ✅ 功能需求和非功能需求分别标识
- ✅ 架构末级节点是最大颗粒度的相关方需求条目
- ✅ 建立双向追溯的超链接（点击映射视图中的相关方需求，直接跳到第一部分对应节点）

### 4.3 第3层：业务架构

**定义**：业务架构是系统的运行概念（Operational Concept）——即用户使用系统开展业务的过程说明。从用户视角描述如何使用系统，而非系统如何实现。

**关键特性**：业务架构是唯一"独立存在"的层——仅包含架构定义（IPO清单），无详细定义。其架构末级节点自身承担"需求"角色，直接映射到系统需求架构末级节点。

**层级结构（0-5级）**：

| 层级 | 名称 | 说明 |
|------|------|------|
| 0级 | 架构根节点 | 整个系统的顶层根节点（唯一） |
| 1级 | 业务域 | 按业务领域划分（3-10个） |
| 2级 | 端到端项目 | 跨越多个业务域的项目或流程 |
| 3级 | 项目业务对象架构 | 业务对象和其架构 |
| 4级 | 工作包（WBS） | 业务对象的工作分解结构 |
| **5级** | **业务组件（承接层）** | **★ 末级，不再分解；使用IPO模型完整定义** |

**本层的分析活动**：

- **需求说明符合性分析**：检查每个业务组件（IPO）是否充分满足所承接的相关方需求，验证1:1分配约束，验证N:1承接合理性（2-5条合理），验证IPO三维定义的完整性
- **需求性能符合性分析**：验证性能指标是否从相关方需求正确继承，评估设计方案能否达到性能指标要求

**这一层的目的**：定义用户使用产品开展业务的过程

**业务架构的本质**：
- 是**运行概念**（Operational Concept）
- 是用户使用待开发产品开展业务的过程说明
- **独立存在**，仅包含架构定义（IPO清单），无详细定义
- 业务架构末级节点（IPO）直接作为需求映射到系统需求架构末级节点

> 详见《业务架构设计规范》(`04-business-architecture-specification.md`)——业务架构的完整规范，包括本质、层级结构、设计原则、设计方法和质量标准

**IPO模型（Input-Process-Output）**：

业务架构的架构末级节点应该是一个IPO模型：
- **Input（输入）**：产品和/或系统接收的输入
- **Process（处理）**：对输入进行的处理
- **Output（输出）**：生成的输出

每个IPO对应一个系统功能。

**IPO的颗粒度**：
- IPO的体量主要取决于所期望输出物的颗粒度
- 例如：对于系统设计流水线产品，期望的输出是各技术文档
- 那么一个IPO应该对应一份技术文档（或文档的一个可以由单人完成的独立部分）

**内容结构**：

**仅包含：业务架构末级节点（IPO清单）**
```
BA-001: [IPO名称/业务组件名称]
├─ 定义：[业务组件的定义]
├─ 承接的相关方需求详细定义末级：
│  ├─ SR-F-001.1
│  ├─ SR-F-001.2
│  └─ SR-F-002.1
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对系统需求的指导]
```

> **说明**：业务架构暂不需要详细定义（I/P/O详细说明），其IPO定义自身已足够承接相关方需求并指导系统需求设计。如果后续需要更细化的业务描述，可补充详细定义。

**关键要求**：
- ✅ 业务架构末级节点（IPO）是最大颗粒度的业务组件
- ✅ 每个IPO都应该有清晰的Input、Process、Output定义
- ✅ 建立与相关方需求的映射关系（N:1支持）
- ✅ 为系统需求提供清晰的设计指导
- ✅ 业务架构独立存在，IPO直接作为系统需求的输入（去重后）

**IPO的去重机制**（重要！）：

在业务架构设计时，当在某个分支需要一个IPO时，应该：
1. **首先检查** - 现有业务架构中是否已有相同或相似的IPO
2. **复用优先** - 如果存在，直接复用该IPO
3. **改进次之** - 如果现有IPO改进后可用，则改进后复用
4. **新增最后** - 确实需要新的IPO才新增

### 4.4 第4层：系统需求

**定义**：系统需求是将业务操作过程（业务架构）转化为系统可实现的需求规范。它回答"系统应该提供什么功能来支持业务操作？系统应该满足哪些质量属性？"

**双重来源**：系统需求有两条独立的输入来源：

| 来源 | 说明 | 处理方式 |
|------|------|---------|
| **功能需求** | 业务架构IPO去重清单 | IPO去重后按业务组件类型分类组织，形成0-9级架构 |
| **非功能需求** | 相关方非功能需求（平行架构，不经过BA） | 按三大类（质量特性/环境适应性/可实现性）分类组织，形成0-6级架构 |

**架构层级**：

功能需求使用**0-9级**结构，其中5级（组件实例）承接BA 5级，9级（场景活动）分配到PA末级：
```
5级：组件实例 ←── 承接BA 5级（N:1，架构末级）
6级：功能模块（Tab/页面）
7级：系统用例（按钮/菜单）
8级：用例场景（正常/异常/替代）
9级：场景活动 ──→ 分配到PA末级（1:1，详细定义末级）
```

非功能需求使用**0-6级**平行架构，独立于功能分支：根节点 → 需求类别（三大类）→ 需求类型 → 需求子类型 → 需求细分 → **5级承接层（架构末级）** → 6级验证标准。

**本层的分析活动**：

- **需求说明符合性分析**：覆盖度检查（所有BA IPO是否都有对应的SysReq 5级节点承接），一致性校验（IPO去重后语义是否一致），追溯验证（每个SysReq节点是否标注了承接来源）
- **需求性能符合性分析**：性能继承验证（系统需求是否继承了上层指标），可行性评估（在系统层面评估指标可实现性），指标追溯

**这一层的目的**：将业务架构转化为系统可以实现的需求规范

**系统需求的本质**：
- 是业务架构和相关方需求非功能部分的方案
- 由两份子文档构成：架构定义文档（对业务架构和相关方非功能需求的分类和组织）和详细定义文档（对架构末级节点的分解）
- 分解标准：直到每个详细定义末级节点只需映射到产品架构的一个组件

**系统需求的来源**（重要！）：

1. **功能需求来源** - 业务架构中所有IPO的去重清单
   - 多个业务流程中重复的IPO，在系统需求中只需定义一次
   - 这确保了系统功能的完整性和无重复性

2. **非功能需求来源** - 相关方需求中的非功能部分
   - 直接承接相关方需求的非功能需求

**内容结构**：

**架构定义文档**

**系统需求架构末级节点示例**
```
SysReq-001: [系统需求名称]（架构末级节点）
├─ 定义：[系统需求的定义]
├─ 类型：[功能需求/非功能需求]
├─ 承接的业务架构/相关方需求：
│  ├─ BA-001
│  ├─ BA-002
│  └─ SR-NF-001
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对产品架构的指导]
```

**详细定义文档**

**系统需求详细定义末级节点示例**
```
SysReq-001.1: [具体的系统需求]
├─ 描述：[详细说明]
├─ 验收标准：[验收条件]
├─ 分配到：[产品架构末级节点]
└─ 符合性说明：[如何满足系统需求]
```

**关键要求**：
- ✅ 系统需求架构末级节点是最大颗粒度的系统需求条目
- ✅ 功能需求和非功能需求分别处理
- ✅ 功能需求来自业务架构IPO的去重清单
- ✅ 每条系统需求都应该有清晰的定义和验收标准
- ✅ 建立与上层需求的映射关系（N:1支持）

### 4.5 第5层：产品架构

**定义**：产品架构是将系统需求转化为具体技术实现方案的设计结果。它回答的是"系统应该如何被构建？组件之间如何交互？"——是产品研发的最终蓝图。

**组件组成**：产品架构通常由四部分组成——前端组件（用户界面）、后端组件（业务逻辑）、数据层（数据存储）、基础设施（支撑能力）。

**组件分类体系**：

| 类型 | 子类型 |
|------|--------|
| **前端组件** | 页面组件、功能组件、交互组件 |
| **后端组件** | API服务、业务引擎、数据处理、消息处理 |
| **数据层** | 关系型数据库、NoSQL、文件存储、消息队列 |
| **基础设施** | API网关、认证授权、监控告警、日志管理、CI/CD |

**组件划分原则**：每个组件应有明确的职责边界和接口定义。粒度以"一个团队可在1-2周内完成开发"和"可独立部署和运行"为双重标准。

**本层的分析活动**：

- **需求说明符合性分析**：覆盖度检查（所有SysReq 9级节点是否都有对应的PA组件承接），1:1分配验证（每个SysReq 9级分配到唯一的PA组件），N:1承接合理性（每个PA组件承接2-5个合理）
- **需求性能符合性分析**：性能继承验证（PA组件性能指标不低于所承接SysReq和NFR的要求），可实现性评估，权衡决策记录（当多个NFR冲突时记录决策）

**这一层的目的**：定义产品的具体实现方案

**产品架构的本质**：
- 是系统需求的方案
- 由两份子文档构成：架构定义文档（产品组件分类和组织）和详细定义文档（实现细节和性能指标）

**内容结构**：

**架构定义文档**

**产品架构末级节点示例**
```
PA-001: [产品组件名称]（架构末级节点）
├─ 定义：[产品组件的定义]
├─ 承接的系统需求详细定义末级：
│  ├─ SysReq-001.1
│  ├─ SysReq-001.2
│  └─ SysReq-002.1
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现细节：[具体的技术实现方案]
```

**详细定义文档**

**产品架构详细定义末级节点示例**
```
PA-001-Detail: [实现细节、性能指标、技术栈等]
```

**关键要求**：
- ✅ 产品架构末级节点是最大颗粒度的产品组件
- ✅ 每个产品组件都应该有清晰的实现方案
- ✅ 包含具体的技术细节和性能指标
- ✅ 建立与系统需求的映射关系（N:1支持）

## 五、验收标准

### 5.1 架构定义质量验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **架构末级节点定义完整率** | 100% | 每层方案文档的架构末级节点应有清晰的定义、承接关系和符合性分析 |
| **架构层级结构完整率** | 100% | 从根节点到末级节点的完整树形结构（第3层0-5级，第4层功能0-9级/非功能0-6级） |
| **分类合理性** | 通过评审 | 节点分类符合业务逻辑，同层节点在同一抽象级别 |

### 5.2 详细定义质量验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **详细定义分解完成率** | 100% | 所有架构末级节点都已分解到详细定义末级 |
| **原子性满足率** | 100% | 每条详细定义末级节点只需分配到下层架构的一个末级节点 |
| **验收标准完整率** | 100% | 每条详细定义末级都有可测量的验收标准 |
| **性能指标量化率** | 100% | 非功能需求必须有具体数值、单位和测量条件 |

### 5.3 映射关系质量验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **1:1约束满足率** | 100% | 每条详细定义末级只分配到下层唯一的架构末级节点 |
| **N:1承接合理性** | 2-5条/节点 | 每个架构末级节点承接的详细定义末级数量合理 |
| **映射覆盖率** | 100% | 所有上层末级需求都已映射到下层架构末级节点 |
| **去重完成率** | 100% | BA→SysReq的IPO去重已完成 |

### 5.4 追溯完整性验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **正向追溯完整率** | 100% | 从原始需求末级→产品架构末级正向可追溯 |
| **反向追溯完整率** | 100% | 从产品架构末级→原始需求末级反向可追溯 |
| **符合性分析完整率** | 100% | 每个方案节点都说明了如何满足所承接的上层需求 |

## 六、检查清单

### 分解充分性检查清单

在设计每一层时，检查分解是否充分：

- [ ] 每条上层需求都分解到末级需求
- [ ] 末级需求是原子性的、不可再分的
- [ ] 每条末级需求都有清晰的定义和验收标准
- [ ] 每条末级需求都分配到下层方案的一个架构末级节点
- [ ] 没有遗漏或重复的映射
- [ ] 在开展详细定义的同时，已同步建立下层方案架构末级节点的结构骨架（或确认下层架构已存在）

### 分解标准检查清单

检查分解是否符合标准（只需一个下层方案架构末级节点承接）：

- [ ] 相关方需求详细定义末级节点只需分配到业务架构的一个末级节点
- [ ] 业务架构末级节点（IPO）只需分配到系统需求架构的一个末级节点
- [ ] 系统需求详细定义末级节点只需分配到产品架构的一个末级节点
- [ ] 如果某个条目需要分配到多个下层架构末级节点，说明还需要进一步分解

### 映射关系检查清单

检查映射关系是否满足1:1约束和N:1支持：

- [ ] 每条上层需求都分配到下层方案的唯一一个条目（1:1 约束）
- [ ] 每个下层方案条目都承接了上层的多条需求（N:1 支持）
- [ ] 承接的需求之间有逻辑关联
- [ ] 映射关系符合业务逻辑

### 符合性分析检查清单

检查每个方案条目是否充分说明了如何满足上层需求：

- [ ] 每个方案条目都包含符合性分析
- [ ] 符合性分析清晰、具体、可验证
- [ ] 符合性分析说明了如何满足所承接的需求
- [ ] 符合性分析没有遗漏任何承接的需求
- [ ] 符合性分析的逻辑清晰、易于理解

### 双向追溯检查清单

检查追溯链路是否完整和有效：

- [ ] 每条上层需求都标注了分配到的下层方案条目
- [ ] 每个下层方案条目都标注了承接的上层需求
- [ ] 所有超链接格式正确
- [ ] 所有超链接都指向有效的锚点
- [ ] 所有超链接都指向正确的目标
- [ ] 用户能够快速导航和追溯

### 文档规范化检查清单

检查文档是否符合规范：

- [ ] 文档结构清晰、一致
- [ ] 各层文档结构符合本层定义（架构定义文档含架构末级节点 + 详细定义文档含详细定义末级节点，或如业务架构仅含架构定义文档）
- [ ] 文档中只保留必要的内容，去除冗余信息
- [ ] 文档易于维护和更新
- [ ] 文档版本号正确更新

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误1：混淆架构末级节点和详细定义末级节点的角色**

```
错误的做法：
在相关方需求层，将"需求收集"直接作为架构末级节点（SR-F-001），
但"需求收集"其实是 SR-F-001（需求规范化功能）的一个详细定义条目。

问题：架构末级节点应该是"需求分类体系中的最大条目"，
      "需求收集"是功能的一个具体环节，属于详细定义层次。
后果：架构层级扁平化，失去分类和组织能力。同时"需求收集"需要作为独立
      架构末级节点去承接原始需求末级，粒度不匹配导致映射混乱。
```

```
✅ 正确做法：
架构末级节点：SR-F-001（需求规范化功能）—— 代表一个完整的业务能力
详细定义末级：SR-F-001.1（需求收集）—— 功能下的具体环节
→ 架构末级节点负责分类和组织，详细定义节点负责细化分解
```

**❌ 错误2：同步设计退化为串行设计**

```
错误的做法：
先完成所有层的详细定义，再开始下层架构定义
→ 第2层详细定义全部完成 → 第3层架构定义 → 第3层详细定义...

问题：每层之间完全串行，详细定义完成后才发现没有下层目标可承接
后果：需要反复调整详细定义的粒度，造成大量返工，整体周期延长50%以上
```

```
✅ 正确做法：
第一轮：快速约定各层架构骨架（占位符 + 映射关系）
第二轮：并行填充各层详细内容
第三轮：验证映射的一致性
→ 两阶段工作流程使并行成为可能，缩短整体周期30-50%
```

**❌ 错误3：验收标准和检查清单混为一谈**

```
错误的做法：
验收标准和检查清单放在同一节，列出的全是"是否已检查"类清单项，
但没有量化指标来衡量设计质量。

问题：无法判断"做完了"和"做好了"的区别
后果：团队以"清单全部打勾"为目标，而非以"质量达标"为目标
```

```
✅ 正确做法：
验收标准（量化指标）："映射覆盖率100%，1:1约束满足率100%"
检查清单（操作验证）："[ ] 检查所有需求是否已分解到末级"
→ 验收标准回答"多好算好"，检查清单回答"做了没有"
```

**❌ 错误4：分析活动被遗漏或延后**

```
错误的做法：
需求说明符合性分析和需求性能符合性分析在全部设计完成后才做，
或者根本没做。

问题：设计过程中的问题没有被及时发现
后果：到追溯验证阶段才发现大量映射错误，需要大范围返工
```

```
✅ 正确做法：
分析活动贯穿设计全过程——在每个架构定义完成后集中执行分析，
在详细定义开始前完成验证。
→ 尽早发现设计缺陷，降低返工成本
```

### 7.2 最佳实践

**实践1：始终在**开始详细定义前**完成本层架构定义的分析活动。先确认承接关系和1:1约束，再进行详细分解。

**实践2：同步设计优先于完美设计**。不要试图一次性完成所有层的精确定义。先用占位符建立骨架关系，再逐步迭代细化。

**实践3：每完成一层详细定义的分解**，立即验证1:1约束是否满足。不要等到全部完成后再验证，这会掩盖粒度问题。

**实践4：使用两阶段工作流程**组织团队协作。第一阶段（1-2小时）专注于骨架和映射，第二阶段（2-6天）专注内容填充。两个阶段之间有清晰的交付物和验收标准。

## 八、FAQ

**Q1：系统设计准则和各层规范（02-06）是什么关系？**

系统设计准则是总纲，定义"为什么这样做"和"整体结构是什么"。各层规范是准则的具体化，定义"具体怎么做"。准则提供全景视图和通用规则，层规范提供针对该层的详细操作指导。在阅读时，建议先读准则了解全局，再根据需要查阅特定层规范。

**Q2：六步法中的步骤是否可以并行执行？**

可以，并且推荐并行。六步法中的第2步（功能分支）和第4步（非功能分支）是独立并行的。第2步完成后可立即开始第3步和第5步的功能部分，无需等待第4步。两阶段工作流程是实现并行化的具体操作方法。

**Q3：1:1分配约束和N:1承接支持是否是同一个规则？**

描述的是同一映射关系的两个视角——1:1从**源头**出发（向下分配），约束每个上层条目只能分配到一个下层节点；N:1从**目标**出发（向上承接），允许一个下层节点承接多个上层条目。两者共同定义了映射的双向约束，缺一不可。

**Q4：当详细定义无法1:1映射到下层架构时，应该调整哪一方？**

先检查详细定义是否还能进一步分解——通常这是最直接的原因。如果详细定义已到原子级仍然无法1:1，则说明下层架构设计不合理，需要补充或调整下层架构末级节点。经验法则是：先怀疑分解不充分，再怀疑架构不完整。

**Q5：非功能需求在准则中为什么有独立的处理路径？**

非功能需求（性能、安全、可用性等）是与功能需求平行的独立体系。功能需求描述"系统做什么"，经过BA层传递；非功能需求描述"做得多好"，不经过BA层，直接在系统需求层面承接相关方非功能需求。这种分离设计避免了非功能需求在BA层被稀释或丢失。

**Q6：两阶段工作流程和同步设计是什么关系？**

两阶段工作流程是**实现方式**，同步设计是**设计原则**。同步设计要求上下游在设计时对齐，两阶段工作流程提供了具体的操作方法：第一阶段快速建立骨架和映射（对齐结构），第二阶段并行填充内容（对齐细节）。简言之：同步设计是"为什么要并行"，两阶段是"怎么并行"。

**Q7：如何判断一层设计的"完成"状态？**

完成的标准是：① 架构末级节点都完成了符合性分析；② 详细定义末级节点都满足1:1约束；③ 映射覆盖率100%；④ 检查清单全部通过。满足这些条件后，该层可以冻结，进入下一层设计。

**Q8：准则的版本更新策略是什么？**

v1.x 为小修小补（修正错误、补充示例、优化表述），v2.x 为结构或内容大幅调整（如本次从9章改为章节编号结构）。版本号变动意味着准则发生了变化，使用者应关注 changelog 中的变更说明。
