## 一、核心概念

### 1.1 本层的定义

**产品架构（Product Architecture）** 是将系统需求转化为具体技术实现方案的设计结果。它回答的问题是：**"系统应该如何被构建？组件之间如何交互？"**

产品架构在系统设计流水线中扮演着**方案文档**的角色：
- **是系统需求（第4层）的方案**：将功能和非功能要求转化为技术方案
- **是产品研发的蓝图**：为开发团队提供明确的技术实现指导

### 1.2 在5层结构中的位置

```
系统需求（第4层）
  ├─ 功能：SysReq-F 9级（场景活动）
  └─ 非功能：SysReq-NFR（系统约束）
       ↓ N:1 映射

产品架构（第5层） ★ 本规范
  ├─ 架构定义：产品组件清单 ← 承接SysReq 9级
  └─ 详细定义：实现方案、技术栈、性能指标
       ↓
产品研发（编码实现）
```

| 维度 | 说明 |
|------|------|
| **文档类型** | 方案文档（架构定义 + 详细定义） |
| **承接对象** | SysReq 9级场景活动（N:1）+ SysReq-NFR |
| **输出对象** | 产品研发团队（开发实现） |

### 1.3 产品架构的组成

产品架构通常由以下部分组成：

| 组成 | 说明 | 示例 |
|------|------|------|
| **前端组件** | 用户界面和交互逻辑 | Web页面、移动端界面 |
| **后端组件** | 业务逻辑和数据处理 | API服务、业务引擎 |
| **数据层** | 数据存储和管理 | 数据库、缓存、文件存储 |
| **基础设施** | 支撑系统运行的基础能力 | 网关、认证、监控、消息队列 |

### 1.4 本层的分析活动

**需求说明符合性分析**：
- 覆盖度检查 — 所有 SysReq 9 级节点是否都有对应的 PA 组件承接
- 1:1 分配验证 — 每个 SysReq 9 级节点是否分配到唯一的 PA 组件
- N:1 承接合理性 — 每个 PA 组件承接 2-5 个 SysReq 9 级节点为合理

**需求性能符合性分析**：
- 性能继承验证 — PA 组件的性能指标不低于所承接 SysReq 和 NFR 的要求
- 可实现性评估 — 结合技术栈和架构方案评估性能指标的可实现性
- 权衡决策记录 — 当多个 NFR 冲突时，权衡决策是否已记录

## 二、架构定义规范

### 2.1 架构末级节点

产品架构的末级节点是**产品组件**，每个组件应有清晰的职责边界和接口定义。

每个产品架构末级节点包含：

| 属性 | 说明 | 示例 |
|------|------|------|
| **组件ID** | 唯一标识 | PA-001 |
| **组件名称** | 组件名称 | 用户认证服务 |
| **组件类型** | 前端/后端/数据/基础设施 | 后端 |
| **定义** | 组件的职责和功能描述 | 提供用户身份认证和授权管理 |
| **承接的系统需求** | 承接了哪些SysReq 9级节点 | SysReq-F-001.1.1.1 |
| **符合性分析** | 如何满足所承接的系统需求 | 通过 OAuth 2.0 实现安全认证 |
| **实现细节** | 具体的技术实现方案 | 技术栈、关键模块、部署方式 |
| **性能指标** | 该组件需要满足的性能目标 | 响应时间 < 50ms |

### 2.2 层级结构

产品架构的层级相对灵活，一般遵循以下模式：

| 层级 | 名称 | 说明 |
|------|------|------|
| **0级** | 产品根节点 | 整个产品的顶层 |
| **1级** | 架构分层 | 按前端/后端/基础设施划分 |
| **2级** | 子系统/模块 | 按功能领域划分 |
| **3级** | **组件（承接层）** | **★ 末级，承接SysReq 9级节点** |

**注意**：产品架构的层级数可根据产品复杂度适当调整。

### 2.3 组件分类

按角色分为四类：

**前端组件**：

| 子类型 | 说明 | 技术示例 |
|--------|------|---------|
| 页面组件 | 完整的页面或视图 | 首页、详情页、管理后台 |
| 功能组件 | 可复用的功能模块 | 搜索组件、表格组件、图表组件 |
| 交互组件 | 用户交互元素 | 按钮、表单、对话框 |

**后端组件**：

| 子类型 | 说明 | 技术示例 |
|--------|------|---------|
| API服务 | 提供RESTful/gRPC接口 | 用户服务、订单服务 |
| 业务引擎 | 核心业务逻辑处理 | 流程引擎、规则引擎 |
| 数据处理 | 数据加工和处理 | ETL管道、批量处理服务 |
| 消息处理 | 异步消息处理 | 消息消费者、事件处理器 |

**数据层**：

| 子类型 | 技术示例 |
|--------|---------|
| 关系型数据库 | PostgreSQL、MySQL |
| NoSQL数据库 | MongoDB、Redis |
| 文件存储 | MinIO、S3 |
| 消息队列 | Kafka、RabbitMQ |

**基础设施**：

| 子类型 | 技术示例 |
|--------|---------|
| API网关 | Nginx、Kong |
| 认证授权 | OAuth 2.0、RBAC |
| 监控告警 | Prometheus、Grafana |
| 日志管理 | ELK Stack |
| CI/CD | Jenkins、GitLab CI |

### 2.4 架构定义的文档结构

**第一部分：架构设计结果（树形结构）**

```
产品架构（0级）
├─ 前端层（1级）
│  └─ 用户管理子系统（2级）
│     └─ 用户登录页面（3级·组件/末级）
│        ├─ 定义：提供用户登录界面
│        ├─ 承接SysReq：SysReq-F-001.1.1.1
│        ├─ 符合性分析：[说明如何满足SysReq]
│        └─ 实现细节：React + Ant Design
│
├─ 后端层（1级）
│  └─ 用户服务（2级）
│     └─ 用户认证API（3级·组件/末级）
│        ├─ 定义：提供用户身份认证
│        └─ 性能指标：响应时间 < 50ms
│
└─ 基础设施层（1级）
   └─ API网关（2级）
      └─ 网关服务（3级·组件/末级）
```

**第二部分：需求映射视图（映射关系表）**

```
| SysReq 9级节点 | 承接的PA组件 | 超链接 |
|----------------|-------------|--------|
| SysReq-F-001.1.1.1 | PA-101 用户登录页面 | [→] |
| SysReq-F-001.1.1.2 | PA-101 用户登录页面 | [→] |
| SysReq-F-002.1.1.1 | PA-201 用户认证API | [→] |
```

### 2.5 组件间关系定义

| 关系类型 | 说明 | 表示方式 |
|---------|------|---------|
| **依赖** | 组件A依赖组件B | PA-001 → PA-002 |
| **调用** | 组件A调用组件B的接口 | PA-001 → API(PA-002) |
| **组合** | 组件A由组件B和C组成 | PA-001 = {PA-002, PA-003} |
| **关联** | 组件A与组件B存在逻辑关联 | PA-001 ↔ PA-002 |

### 2.6 组件划分原则

```
✅ 合适的组件划分：
  PA-001: 用户认证服务（单一职责，高内聚）
  PA-002: 订单管理API（独立的业务边界）
  PA-003: 日志收集服务（可独立部署）
  → 每个组件有明确的职责边界、独立部署能力

❌ 不合适的组件划分：
  PA-001: 整个后端系统（粒度过粗，职责不清）
  PA-002: 用户名的显示格式（粒度过细，管理成本高）
  → 粒度过粗或过细都会影响设计质量和开发效率
```

## 三、详细定义规范

### 3.1 详细定义的本质

产品架构详细定义是对每个产品组件进行深入的实现方案描述，使开发团队能够直接进入编码阶段。

每个产品组件的详细定义应包含：

| 内容 | 说明 | 示例 |
|------|------|------|
| **技术栈** | 使用的编程语言、框架、工具 | Python + FastAPI + PostgreSQL |
| **关键模块** | 组件的内部模块划分 | 输入验证模块、业务处理模块 |
| **接口定义** | 对外提供的API接口 | GET /api/users |
| **数据模型** | 涉及的数据结构和存储 | User {id, name, email, role} |
| **部署方式** | 如何部署和运行 | Docker + Kubernetes |
| **性能指标** | 具体的性能目标 | 响应时间 < 200ms |
| **安全要求** | 安全相关的要求 | 接口认证、数据加密 |
| **监控告警** | 运维监控指标 | CPU > 80% 告警 |

### 3.2 详细定义模板（YAML）

```yaml
PA-001: 用户认证服务
├─ 技术栈
│   ├─ 语言: Go 1.21
│   ├─ 框架: Gin
│   └─ 数据库: Redis + PostgreSQL
├─ 关键模块
│   ├─ 用户注册: 处理用户注册逻辑
│   ├─ 用户登录: 处理用户登录和Token生成
│   └─ 权限验证: 处理接口访问权限检查
├─ 接口定义
│   ├─ POST /api/v1/login: 用户登录
│   ├─ POST /api/v1/register: 用户注册
│   └─ POST /api/v1/verify: 验证Token
├─ 部署方式
│   ├─ 容器化: Docker
│   └─ 编排: Kubernetes（3副本）
├─ 性能指标
│   ├─ 响应时间: < 50ms（P99）
│   └─ 吞吐量: > 5000 请求/秒
├─ 安全要求
│   ├─ 密码加密: bcrypt
│   ├─ Token: JWT（过期时间24h）
│   └─ 传输加密: TLS 1.3
└─ 监控告警
    ├─ 错误率 > 1%: P1告警
    └─ 响应时间 > 200ms: P2告警
```

### 3.3 详细定义深度对比

```
✅ 充分的详细定义：
  技术栈: Go 1.21 + Gin + PostgreSQL
  关键模块: 用户注册（处理注册逻辑）
  接口: POST /api/v1/login（接收username+password，返回JWT Token）
  → 开发团队可以直接进入编码

❌ 不充分的详细定义：
  技术栈: Go
  接口: 登录接口
  → 信息太少，开发团队需要大量沟通才能开始编码
```

## 四、设计方法

### 4.1 总体设计流程

```
第1步：分析系统需求
  └─ 收集所有SysReq 9级节点和SysReq-NFR
     └─ 识别需求之间的关联关系
        └─ 输出：SysReq 9级清单 + NFR约束清单

第2步：识别组件候选
  └─ 按功能相关性分组SysReq 9级节点
     └─ 识别可能的前端页面、后端服务、数据存储
        └─ 评估组件的粒度和边界

第3步：建立架构骨架
  └─ 确定前端/后端/基础设施的分层结构
     └─ 创建组件占位符，建立SysReq→PA初步映射
        └─ 输出：架构骨架 + 映射关系

第4步：详细组件设计
  └─ 定义每个组件的职责、接口、技术栈
     └─ 定义组件间的关系
        └─ 输出：完整的架构定义 + 详细定义

第5步：非功能约束验证
  └─ 检查每个NFR是否有对应的组件设计体现
     └─ 验证性能指标是否满足NFR要求
        └─ 输出：NFR符合性检查报告

第6步：映射验证
  └─ SysReq→PA覆盖率 100%、1:1约束 100%
     └─ 输出：映射验证报告
```

### 4.2 两阶段工作流程

```
第一阶段：骨架建立（快速，1-2小时）
  1. 扫描SysReq 9级节点
  2. 创建PA组件占位符
  3. 建立SysReq→PA骨架映射
  4. 冻结骨架结构
  输出：组件占位符 + 映射关系

第二阶段：详细填充（深入，2-4天）
  1. 详细定义每个组件的实现方案
  2. 填充技术栈、接口定义、性能指标
  3. 验证非功能约束的可满足性
  4. 验证映射的完整性和一致性
  5. 冻结产品架构文档
  输出：完整的产品架构定义 + 详细定义
```

### 4.3 同步设计

产品架构的初步结构应与系统需求详细定义**同步开展**。

**实际操作**：

```
1. SysReq 5级节点确定后 → 快速建立PA结构骨架（占位符）
2. SysReq 9级节点确定时 → 同步映射到PA组件
3. 冻结映射关系后 → 并行填充详细内容
4. 非功能约束 → 在组件设计阶段同步验证
```

**为什么同步**：SysReq 9级场景活动需要分配到PA组件。如果PA尚未建立，SysReq 9级分解就没有分配目标，可能导致分解粒度过粗或过细而需要返工。

### 4.4 非功能权衡决策

当多个非功能需求冲突时，需要进行权衡决策：

| 冲突场景 | 权衡决策 | 记录方式 |
|---------|---------|---------|
| 性能 vs 安全 | 加密运算增加延迟 | 记录性能基准和安全要求 |
| 可用性 vs 成本 | 多副本部署增加成本 | 记录可用性目标和预算约束 |
| 可维护性 vs 性能 | 抽象层影响性能 | 记录架构决策记录(ADR) |

### 4.5 设计原则

**N:1承接支持**：一个PA组件可承接多个SysReq 9级节点。2-5个合理，1个需考虑合并，6个以上需考虑拆分。

**1:1分配约束**：每个SysReq 9级节点分配到PA中唯一的组件。如果需要多个组件，说明9级活动还需继续分解。

**分层架构**：按展示层→业务层→数据层→基础设施层组织，职责清晰、独立演化、便于复用。

**非功能约束满足**：每个PA组件标注受哪些NFR约束，组件的性能指标不低于SysReq的要求。

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误1：组件粒度过大或过小**

```
错误的做法：
PA-001: 整个后端系统（粒度过粗，无法聚焦）
PA-002: 用户名的显示格式（粒度过细，管理成本高）

问题：粒度过粗导致职责不清，粒度过细导致管理复杂
后果：开发团队无法明确分工，组件边界模糊
```

```
✅ 正确做法：
PA-001: 用户认证服务（粒度适中——一个独立业务能力）
PA-002: 订单管理API（粒度适中——一个完整功能域）
→ 每个组件是一个完整服务，有清晰边界
```

**❌ 错误2：映射关系不完整**

```
错误的做法：
SysReq有20个9级节点，但PA只定义了15个组件承接
→ 5个SysReq 9级节点未映射

问题：部分功能需求没有对应的组件实现
后果：开发时遗漏功能或临时添加导致架构混乱
```

```
✅ 正确做法：
建立一个SysReq 9级 → PA组件的完整映射表
逐条验证：每个SysReq 9级都有对应的PA组件
→ 映射覆盖率 100%
```

**❌ 错误3：忽略非功能约束**

```
错误的做法：
PA组件定义中只写了功能实现，没有性能指标
→ "用户登录页面：提供登录功能"

问题：开发时不知道性能目标，上线后可能不达标
后果：性能不达标需要返工重构
```

```
✅ 正确做法：
PA组件定义中包含明确的性能指标
→ "用户登录页面：响应时间 < 200ms（P99），支持 5000 并发"
→ 开发时有明确的性能目标
```

### 7.2 最佳实践

**实践1：先骨架后填充**：先快速建立架构骨架（占位符），同步建立映射关系，再并行填充详细内容。

**实践2：组件以"服务"为粒度**：每个组件应是可独立部署和运行的"服务"，粒度以"一个团队可在1-2周内完成开发"为标准。

**实践3：将NFR纳入组件定义**：每个组件定义中都包含可测量、可验证的性能指标，在详细定义中说明NFR的满足方案。

**实践4：记录架构决策**：关键技术选择记录决策理由，NFR权衡决策记录在案，使用ADR格式。

## 八、FAQ

**Q1：产品架构的层级数是固定的吗？**

不是。产品架构的层级可根据产品复杂度适当调整。简单产品可能只有 2 级（根节点 → 组件），复杂产品可能需要 4 级或更多。关键是**保持层级的一致性**——同层节点应在同一抽象级别。

**Q2：一个PA组件承接多少个SysReq 9级节点合理？**

建议每个PA组件承接 2-5 个 SysReq 9 级节点（2-3 条最优，1 条需考虑合并，4-5 条可接受需检查过度聚合，6 条以上说明粒度过粗需考虑拆分）。承接 0 个说明组件可能多余。

**Q3：组件划分到多细才算合适？**

以"一个团队可在 1-2 周内完成开发"和"可独立部署和运行"为双重标准。如果组件太大（需要多人周）、太依赖其他组件才能运行，说明还需拆分。

**Q4：前端组件和后端组件是否需要在不同层级中定义？**

建议在同一架构层级中定义，但通过"组件类型"字段区分。例如 2 级可以是"用户管理子系统"，下面同时包含"用户登录页面（前端）"和"用户认证API（后端）"。

**Q5：当性能和安全要求冲突时怎么办？**

记录权衡决策。例如加密运算会增加延迟，需记录"在 P99 响应时间要求下，选择 AES-256 加密，预计增加 10ms 延迟，仍满足 200ms 目标"。这是架构决策记录（ADR）的标准用法。
