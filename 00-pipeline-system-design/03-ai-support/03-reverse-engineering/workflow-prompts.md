# 逆向工程系统设计：提示词模板集
**Reverse Engineering System Design: Workflow Prompt Templates**

**用途**：AI 按 8 步流程依次执行逆向推导，每个步骤使用对应的提示词模板。AI 将提示词中的 `[占位符]` 替换为实际内容后执行。

---

## 第1步：代码和图纸分析

### 目标
理解产品的整体架构和功能模块，从代码和图纸中提取产品架构组件候选，产出产品架构草图和功能模块清单。

### 输入
- 产品代码库（目录结构、关键模块文件）
- 架构图纸（系统架构图、模块图、数据流图、部署图）
- 现有技术文档（如有，作为参考）

### 涉及规范
- [产品架构规范 §2 架构定义规范](../../01-design-standards/06-product-architecture-specification.md#二架构定义规范)

### 提示词模板

#### 1a. 代码结构分析和模块划分

```
请对以下产品代码和架构图纸进行分析，识别主要组件和模块。

代码目录结构：
[粘贴项目的目录树或关键目录结构]

关键类/文件清单：
[列出核心类、控制器、服务、模型等关键文件]

架构图纸描述：
[如有架构图，描述图中的模块划分和依赖关系]

请执行：
1. 分析目录结构和模块划分，识别主要功能区域
2. 按 4 大组件分类预分类（前端/后端/数据/基础设施）
3. 识别组件间的依赖关系和调用链
4. 记录关键技术决策（语言、框架、中间件、数据库等）

遵循原则：
- 每个组件有明确的职责边界，粒度适中
- 与代码的实际模块划分保持一致
- 基础设施组件（网关、认证等）从代码配置中识别

输出格式：
## 组件识别清单
| 候选组件 | 组件类型 | 代码位置 | 依赖 | 说明 |
|---------|---------|---------|------|------|
| [名称] | [前端/后端/数据/基础设施] | [路径] | [依赖的组件] | [职责说明] |

## 关键技术决策记录
- 后端语言/框架：[如 Java Spring Boot]
- 前端技术栈：[如 React + Ant Design]
- 数据库：[如 PostgreSQL]
- 中间件/消息队列：[如有]
- 部署方式：[如 Docker + Kubernetes]
```

#### 1b. 功能清单提取

```
请从代码中提取所有实现的功能，按功能模块分类整理。

代码结构和模块分析结果：
[粘贴1a的输出]

关键界面/API清单：
[列出主要的页面路由、API 端点]

请执行：
1. 从前端路由和页面文件提取用户可见功能
2. 从后端 API 提取服务端功能
3. 从数据库 schema 提取数据实体
4. 将功能按模块分类并记录实现方式

输出格式：
## 功能模块清单
| 模块 | 功能 | 实现方式 | 关键文件 | 关联组件 |
|------|------|---------|---------|---------|
| [模块名] | [功能描述] | [API/页面/服务] | [文件路径] | [候选组件ID] |

## 功能-组件映射（初步）
| 候选组件 | 承担的功能 | 功能数 |
|---------|-----------|-------|
| [组件名] | [功能列表] | [N] |
```

#### 1c. 分析活动：初版架构评估

```
请对初步提取的组件和功能清单执行架构评估分析。

组件识别结果： [粘贴1a的输出]
功能模块清单： [粘贴1b的输出]

需求说明符合性分析：
1. 提取的功能模块是否构成一个完整的产品架构？是否有明显缺失的组件？
2. 功能模块清单是否覆盖了代码中的所有主要功能路径？
3. 按 4 大组件分类的预分类是否合理？

需求性能符合性分析：
1. 从代码实现特征（缓存、异步、连接池等）反推性能指标范围
2. 从 API 响应设计反推响应时间预期
3. 从部署配置反推可用性和容量要求

注意：所有反推的性能指标须标注"反推值"，供后续人工确认

输出格式：
## 初版架构评估报告
├─ 组件完整度：[完整/基本完整/有缺失，说明缺失项]
├─ 功能覆盖度：[覆盖率百分比，说明未覆盖功能]
├─ 组件分类合理性：[通过/需调整，说明调整项]
├─ 性能指标反推：
│  ├─ [指标1]: [值] 反推值 ⚠️ 待确认
│  └─ [指标2]: [值] 反推值 ⚠️ 待确认
└─ ⚠️ 待确认项清单
```

### 预期产出
- 产品架构草图（组件识别清单 + 功能模块清单）
- 关键技术决策记录
- 初版架构评估报告（含反推性能指标）

## 第2步：产品架构文档编写

### 目标
编写符合准则的产品架构文档，包括架构定义（树形结构 + 映射关系表骨架）和详细定义。

### 输入
- 第1步产出的产品架构草图和组件识别清单

### 涉及规范
- [产品架构规范 §2 架构定义规范](../../01-design-standards/06-product-architecture-specification.md#二架构定义规范)
- [产品架构规范 §3 详细定义规范](../../01-design-standards/06-product-architecture-specification.md#三详细定义规范)

### 提示词模板

#### 2a. 产品组件正式定义

```
请根据第1步的组件识别结果，正式定义产品架构组件。

组件识别清单：
[粘贴1a的组件识别清单]

关键技术决策记录：
[粘贴1a的技术决策记录]

请执行：
1. 确认组件粒度和边界（可独立部署，1-2周开发周期）
2. 按 4 大组件分类组织树形结构
3. 每个组件包含完整的定义、组件类型、实现细节、性能指标
4. 为每个组件编写符合性分析（从代码实现角度说明该组件的职责满足程度）

4 大组件分类：
- 前端层：用户界面组件
- 后端层：业务逻辑和服务组件
- 数据层：数据库和存储组件
- 基础设施层：网关、认证、监控等支撑组件

性能指标标注规则：
- 从代码可直接推断的（如缓存配置 → 响应时间预期）：标注"反推值"

输出格式：
产品架构（0级）
├─ 前端层（1级）
│  └─ [子系统]（2级）
│     └─ PA-XXX: [组件名称]（3级·组件/末级）
│          ├─ 定义：[从代码反推的组件职责]
│          ├─ 组件类型：前端
│          ├─ 承接SysReq：（待第3步完成后补充）
│          ├─ 符合性分析：[该组件的代码实现如何满足设计意图]
│          ├─ 实现细节：[代码位置、技术栈、关键类]
│          └─ 性能指标：[从代码实现反推的性能特征，标注反推值]
├─ 后端层（1级）
│  └─ ...
├─ 数据层（1级）
│  └─ ...
└─ 基础设施层（1级）
   └─ ...
```

## 第3步：系统需求反向推导

### 目标
从产品架构反向推导系统需求，产出系统需求文档（功能 0-9 级 + 非功能 0-6 级）。

### 输入
- 第2步产出的产品架构文档

### 涉及规范
- [系统需求规范 §2 架构定义规范](../../01-design-standards/05-system-requirements-specification.md#二架构定义规范)
- [系统需求规范 §3 详细定义规范](../../01-design-standards/05-system-requirements-specification.md#三详细定义规范)

### 提示词模板

#### 3a. 分析活动：PA→SysReq 追溯验证

```
请在开始系统需求反向推导前，对产品架构执行追溯验证分析。

产品架构文档：
[粘贴2a和2b的输出]

需求说明符合性分析（逆向专用）：
1. 每个 PA 组件实现的每组功能，能否合理反推出一条或多条 SysReq 9 级场景活动？
2. 是否存在功能实现了但理论上应该有对应 SysReq 但却推导不出来的情况？
3. 前端页面 + 后端服务 + 数据存储能否串联成完整的场景活动链？

需求性能符合性分析（逆向专用）：
1. 从 PA 组件的缓存策略、连接池配置、异步处理机制反推 SysReq 的性能要求
2. 从数据库索引和查询优化反推 SysReq 对查询性能的要求
3. 反推的性能指标标注"反推值"

输出格式：
## PA→SysReq 追溯验证报告
├─ 组件覆盖度：[所有 PA 组件是否都能反推出 SysReq / 覆盖率 %]
├─ 场景完整性：[前端→后端→数据的活动链是否完整 / 完整/有断点]
├─ 性能指标反推清单：
│  ├─ [SysReq候选] → 反推性能指标：[值]（来源于：[PA组件]的[代码特征]）
│  └─ ...
└─ ⚠️ 待确认项：[列出需要人工判断的事项]
```

#### 3b. 系统需求反向推导（功能部分）

```
请从产品组件反推系统功能性需求，按 0-9 级层级结构组织。

产品架构文档： [粘贴2a的输出]
PA→SysReq 验证报告： [粘贴3a的输出]

反向推导方法：
1. 分析每个 PA 组件实现了什么功能
2. 从最细粒度的操作开始（9级：场景活动，对应到 PA 组件的具体操作）
3. 按场景聚合 9 级为 8 级 → 7 级 → 6 级，最终形成 5 级承接层节点
4. 5 级节点是架构末级，承接第4步的 BA IPO

9 级场景活动命名规则：
- 使用"动词+名词"格式，如"创建需求"、"提交审批"、"查询列表"
- 与代码中的 API 或页面操作一一对应

5 级节点（组件实例）命名规则：
- 使用"名词+模块/功能"格式，反映一个内聚的功能集合
- 如"需求管理功能模块"、"架构设计功能模块"

架构定义文档两个部分：

第一部分：树形结构
系统需求（0级·根）
└─ [系统名称]（1级·独立信息系统）
   └─ 功能需求分支（2级）
      └─ [业务组件类型]（3级）
         └─ [业务组件小类]（4级）
            └─ SysReq-F-XXX: [名称]（5级·架构末级）
                 ├─ 定义：[从PA反推的系统需求定义]
                 ├─ 对应的PA组件：PA-XXX, PA-YYY
                 ├─ 符合性分析：[PA组件如何满足该系统需求]
                 ├─ 设计指导：[对业务架构的指导——注意是逆向，从实现反推业务意图]
                 └─ 性能指标：[从PA反推，标注反推值]

详细定义（5级→9级分解）：
SysReq-F-XXX: [名称]（5级）
├─ 6级：SysReq-F-XXX.X [功能模块]
│  └─ 7级：SysReq-F-XXX.X.X [系统用例]
│     ├─ 8级：SysReq-F-XXX.X.X.1 [正常场景]
│     │  ├─ 9级：SysReq-F-XXX.X.X.1.1 [活动步骤] → PA-XXX
│     │  └─ 9级：SysReq-F-XXX.X.X.1.2 [活动步骤] → PA-YYY
│     └─ 8级：SysReq-F-XXX.X.X.2 [异常场景]
│        └─ 9级：SysReq-F-XXX.X.X.2.1 [异常处理] → PA-XXX

第二部分：映射关系表
| SysReq 9级节点 | 来源PA组件 | 关联说明 | 超链接 |
|---------------|-----------|--------|--------|
| SysReq-F-XXX.X.X.X.X | PA-XXX | [反推依据] | [→] |
```

#### 3c. 系统需求反向推导（非功能部分）

```
请从产品代码和架构配置反推系统非功能需求，按 0-6 级平行架构组织。

产品架构文档： [粘贴2a和2b的输出]
关键技术决策记录： [粘贴1a的技术决策记录]

反推方法：
1. 从代码性能特征反推：
   - 缓存配置 → 响应时间要求
   - 连接池配置 → 并发处理能力
   - 异步队列 → 吞吐量要求
2. 从安全配置反推：
   - 认证方式 → 安全需求
   - 加密配置 → 数据安全需求
3. 从部署配置反推：
   - 多副本/多活 → 可用性需求
   - 资源限制 → 容量需求
4. 从代码质量特征反推：
   - 单元测试覆盖 → 可测试性需求
   - 日志/监控 → 可运维性需求

0-6 级平行架构：
0级：非功能需求根节点
1级：需求类别（三大类：质量特性/环境适应性/可实现性）
2级：需求类型（性能/安全/可测试性/可部署性等）
3级：需求子类型
4级：需求细分
5级：非功能需求承接层 ←── 承接相关方非功能需求（架构末级）
6级：操作序列/验证标准

输出格式：
系统非功能需求（0级·根）
└─ [需求类别]（1级）
   └─ [需求类型]（2级）
      └─ [需求子类型]（3级）
         └─ [需求细分]（4级）
            └─ SysReq-NFR-XXX: [名称]（5级·架构末级）
                 ├─ 定义：[从代码反推的非功能需求描述]
                 ├─ 类型：[所属类别，如：性能/安全/可测试性]
                 ├─ 量化指标：[从代码配置/实现反推，标注反推值 ⚠️ 待确认]
                 ├─ 验证方法：[如何验证该指标]
                 └─ 约束条件：[对产品架构的约束说明]

量化指标标注：
- 从代码直接配置中提取的：标注"取自配置"
- 从代码实现特征反推的：标注"反推值"
- 无法推断的：标注"待补充"
```

#### 3d. 同步建立业务架构骨架

```
请在完成系统需求定义后，同步建立业务架构 5 级节点占位符。

SysReq 5 级节点清单：
[粘贴3b输出的5级节点]

请执行：
1. 为每个 SysReq 5 级节点创建 BA IPO 候选占位符
2. 建立 SysReq → BA 的骨架映射关系
3. 骨架只需节点名称和映射关系，不需要详细内容

输出格式：
业务架构骨架：
├─ [IPO候选名称] ← 承接：[SysReq-F-XXX]
├─ [IPO候选名称] ← 承接：[SysReq-F-YYY]
└─ ...

映射关系表：
| SysReq 5级 | 承接的BA候选IPO | 状态 |
|-----------|----------------|------|
| SysReq-F-XXX | BA-[候选名称] | 骨架 |
```

### 预期产出
- 系统需求功能部分架构定义（树形结构 + 详细定义 + 映射关系表）
- 系统需求非功能部分架构定义（0-6级平行架构）
- 业务架构骨架（同步建立）

## 第4步：业务架构反向推导

### 目标
从系统需求反向推导业务架构，产出业务架构 IPO 清单。

### 输入
- 第3步产出的系统需求文档
- 第3步同步建立的业务架构骨架

### 涉及规范
- [业务架构规范 §2 架构定义规范](../../01-design-standards/04-business-architecture-specification.md#二架构定义规范)
- [业务架构规范 §3 IPO模型](../../01-design-standards/04-business-architecture-specification.md#三ipo模型)
- [业务架构规范 §4.3 去重机制](../../01-design-standards/04-business-architecture-specification.md#43-去重机制)

### 提示词模板

#### 4a. 分析活动：SysReq→BA 追溯验证

```
请在开始业务架构反向推导前，对系统需求执行追溯验证分析。

系统需求功能部分： [粘贴3b的输出]

需求说明符合性分析（逆向专用）：
1. 从 SysReq 5 级节点能否推导出合理的业务操作过程（IPO）？
2. 是否存在 SysReq 有对应的代码实现但在 BA 层面说不清业务语义的情况？
3. 多个 SysReq 节点是否可能对应同一个业务操作（需要去重）？

需求性能符合性分析（逆向专用）：
1. 业务层面的性能指标是否覆盖了 SysReq 的性能要求？
2. IPO 的处理步骤是否合理反映了业务操作的复杂度？
```
