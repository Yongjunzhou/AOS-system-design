# 需求一致性检查技能
**Skill: Requirement Consistency Check**

**编号**：05
**所属步骤**：第 1 步 — 原始需求分析及相关方需求架构定义
**引用准则**：系统设计准则 v2.3
**引用指南**：瀑布式设计指南 v2.0 第 1 步

---

## 用途

检测需求之间的冲突和重复，提供解决和合并建议。

## 输入

- 新需求（规范化并分解后的末级需求）
- 现有需求列表（如有）

## 一、冲突检测

### 冲突类型

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| 直接冲突 | 两条需求语义直接矛盾，无法同时满足 | 由流水线开发者确认保留哪条或如何调和 |
| 间接冲突 | 不直接矛盾，但同时实现可能导致系统行为不一致 | 由流水线开发者确认是否需要调整业务规则 |
| 可合并冲突 | 有冲突但可以通过调整合并解决 | 建议合并为一条更完整的需求 |

### 冲突检测规则

1. **语义冲突**：对比需求描述的语义，识别直接矛盾和间接矛盾
2. **业务规则冲突**：检查需求是否违反现有业务规则或流程
3. **技术约束冲突**：检查需求是否违反现有技术约束或架构

## 二、重复检测

### 重复类型

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| 完全重复 | 新需求与现有需求完全相同 | 标记为"已存在"，不重复创建 |
| 部分重复 | 新需求与现有需求部分重叠 | 建议合并为更完整的描述 |
| 相似重复 | 表达不同但核心功能可能相同 | 由流水线开发者确认是否为同一需求 |

### 重复检测规则

1. **语义相似度**：核心功能、输入输出、验收标准是否相同
2. **表达相似度**：使用相同关键词，描述相同业务流程
3. **范围重叠度**：需求范围有交集，新需求是现有需求的子集或超集

## 输出格式

```
| 需求 ID | 检查对象 | 问题类型 | 问题描述 | 建议 | 优先级 |
|----------|---------|---------|---------|------|-------|
| NR-001   | ER-005  | 直接冲突 | 一个允许删除，一个禁止删除 | 需确认保留哪条 | 高 |
| NR-002   | ER-003  | 部分重复 | 新需求 50% 功能已被覆盖 | 建议合并 | 中 |
```

## 质量检查

- [ ] 所有新需求已完成冲突检测和重复检测
- [ ] 新需求之间已交叉检测
- [ ] 每个问题已分类并标注优先级
- [ ] 每个问题有明确的解决/合并建议
