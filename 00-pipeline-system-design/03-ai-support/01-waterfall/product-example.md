# 瀑布式系统设计：产品数据范例
**Waterfall System Design: Product Data Example**

**用途**：展示一个完整的瀑布式系统设计过程的输入和输出范例，AI 参照此范例生成自己的设计文档。

**范例项目**：运营体系开发之系统设计流水线（System Design Pipeline）

---

## 一、第1步产出范例

### 1.1 原始需求收集和初步整理

```
OR-001: 原始需求录入与规范化
├─ 需求描述：系统应支持原始需求的录入、规范化处理、四项检查和功能/非功能分离
├─ 原始碎片清单：
│  ├─ "录入需求的界面要简洁，必填字段清晰" — 流水线开发者 2026-05-01
│  └─ "希望能自动检查需求的完整性和清晰度，减少人工审核工作量" — AOS开发者 2026-05-03
├─ 包含的性能指标：需求保存响应时间 < 1s
├─ 依赖/冲突关系：无
└─ 整理说明：将流水线开发者和AOS开发者的碎片合并，归结为需求录入和规范化功能

OR-002: 需求分解与质量检测
├─ 需求描述：系统应支持原始需求的末级分解、冲突检测和重复检测，确保需求质量
├─ 包含的性能指标：分解操作响应时间 < 2s
├─ 依赖/冲突关系：依赖 OR-001（需要先有规范化需求才能分解）
└─ 整理说明：单一来源，直接保留

OR-003: 分层架构定义管理
├─ 需求描述：系统应支持 SR（相关方需求）、BA（业务架构）、SysReq（系统需求）、PA（产品架构）四层架构定义
├─ 原始碎片清单：
│  ├─ "架构树的层级缩进要清晰，节点可以拖拽排序" — 流水线开发者 2026-05-05
│  ├─ "BA的IPO定义要有Input/Process/Output三个部分" — AOS架构师 2026-05-06
│  └─ "SysReq要支持0-9级功能架构和0-6级非功能架构两种模式" — 系统设计者 2026-05-07
├─ 包含的性能指标：架构树加载 < 2s
├─ 依赖/冲突关系：依赖 OR-002（需要先有需求才能设计架构）
└─ 整理说明：来自三个角色的碎片合并，体现四层架构的完整需求

OR-004: 需求映射与追溯
├─ 需求描述：系统应支持各层间的 N:1 映射关系管理、双向追溯验证和验证报告生成
├─ 原始碎片清单：
│  ├─ "映射关系要支持批量建立，不同层之间的节点要能快速关联" — 流水线开发者 2026-05-08
│  └─ "追溯矩阵要能一键生成，展示从原始需求到产品组件的完整链路" — AOS开发者 2026-05-09
├─ 包含的性能指标：追溯矩阵生成时间 < 5s
├─ 依赖/冲突关系：依赖 OR-003（需要所有架构定义完成才能建立映射）
└─ 整理说明：流水线开发者关注映射操作效率，AOS开发者关注结果展示，合并为映射与追溯需求

OR-005: AI协同设计支持
├─ 需求描述：系统应支持 AI 辅助生成各层设计文档、智能检测设计一致性
├─ 包含的性能指标：AI 生成响应时间 < 30s
├─ 依赖/冲突关系：无
└─ 整理说明：单一来源，直接保留

OR-NFR-001: 系统性能与数据安全要求
├─ 需求描述：系统应保障多人并发设计时的响应速度，确保设计数据的访问安全
├─ 包含的性能指标：
│  - 平均响应时间 < 500ms，P99 < 2s
│  - 支持 100 并发设计者
│  - 设计数据加密存储和传输
└─ 整理说明：来自信息技术部的总体非功能要求
```

### 1.2 原始需求规范化和末级分解

```
OR-001: 原始需求录入与规范化
├─ 规范化描述：系统提供原始需求的录入界面，自动执行四项检查和功能/非功能分离
├─ 类型：功能需求
├─ 优先级：P0 ⚠️ 待确认
├─ 验收标准：需求录入后系统在 1s 内完成规范化检查和分类
├─ 末级分解：
│  ├─ OR-001.1: 原始需求录入与编辑 → SR-F 承接
│  ├─ OR-001.2: 需求四项检查（完整性/清晰性/可测试性/独立性）→ SR-F 承接
│  └─ OR-001.3: 功能需求与非功能需求自动分离 → SR-F 承接

OR-002: 需求分解与质量检测
├─ 规范化描述：系统支持原始需求的末级分解、冲突检测和重复检测
├─ 类型：功能需求
├─ 优先级：P0 ⚠️ 待确认
├─ 验收标准：分解后自动验证 1:1 约束，冲突/重复检测准确率 > 90%
├─ 末级分解：
│  ├─ OR-002.1: 需求逐级分解到末级 → SR-F 承接
│  ├─ OR-002.2: 需求间冲突检测 → SR-F 承接
│  └─ OR-002.3: 需求间重复检测 → SR-F 承接

OR-003: 分层架构定义管理
├─ 规范化描述：系统提供 SR/BA/SysReq/PA 四层架构定义能力，每层支持架构树创建和节点管理
├─ 类型：功能需求
├─ 优先级：P0 ⚠️ 待确认
├─ 验收标准：架构树支持拖拽排序和自动缩进，IPO 定义包含完整的 Input/Process/Output
├─ 末级分解：
│  ├─ OR-003.1: SR架构树创建与管理 → SR-F 承接
│  ├─ OR-003.2: BA IPO 定义与管理 → SR-F 承接
│  ├─ OR-003.3: SysReq 0-9级/0-6级架构定义 → SR-F 承接
│  └─ OR-003.4: PA 四层组件架构定义 → SR-F 承接

OR-004: 需求映射与追溯
├─ 规范化描述：系统支持层间 N:1 映射关系管理、双向追溯验证和追溯矩阵生成
├─ 类型：功能需求
├─ 优先级：P1 ⚠️ 待确认
├─ 验收标准：映射覆盖率 100%，追溯矩阵 5s 内生成
├─ 末级分解：
│  ├─ OR-004.1: 层间 N:1 映射关系建立 → SR-F 承接
│  ├─ OR-004.2: 双向追溯链路验证 → SR-F 承接
│  └─ OR-004.3: 验证报告与追溯矩阵生成 → SR-F 承接

OR-005: AI协同设计支持
├─ 规范化描述：系统集成 AI 能力，支持辅助生成设计文档、智能检测一致性和设计建议
├─ 类型：功能需求
├─ 优先级：P1 ⚠️ 待确认
├─ 验收标准：AI 生成文档完整可编辑，一致性检测覆盖所有映射关系
├─ 末级分解：
│  ├─ OR-005.1: AI 辅助生成设计文档 → SR-F 承接
│  └─ OR-005.2: AI 智能检测与建议 → SR-F 承接

OR-NFR-001: 系统性能与数据安全要求
├─ 规范化描述：系统在 100 人并发设计时应快速响应，设计数据加密存储和传输
├─ 类型：非功能需求
├─ 优先级：P0 ⚠️ 待确认
├─ 性能指标：平均响应时间 < 500ms，P99 < 2s，100 并发用户，数据加密率 100%
├─ 末级分解：
│  ├─ OR-NFR-001.1: 100 并发下平均响应时间 < 500ms → SR-NFR 承接
│  └─ OR-NFR-001.2: 设计数据加密存储与传输 → SR-NFR 承接
```

## 二、第2步产出范例

### 2.1 相关方需求功能部分详细定义

```
架构末级节点：SR-F-001（需求管理功能）
详细定义末级节点：
  - ID: SR-F-001.1
    标题: 原始需求录入与编辑
    描述: 设计者录入需求的标题、描述、来源、提出方等信息，系统自动分配唯一需求编号
    分配目标: BA-原始需求录入 [1:1映射]
    验收标准: 录入界面必填字段完整，提交后自动生成 OR 编号，响应时间 < 1s
    对上层的符合性: 直接满足 SR-F-001 对需求录入的需求

  - ID: SR-F-001.2
    标题: 需求规范化检查
    描述: 系统自动对录入的需求执行四项检查，标注未通过项并给出修改建议
    分配目标: BA-需求规范化处理 [1:1映射]
    验收标准: 四项检查自动执行，未通过项以高亮标注，修改建议可参考
    对上层的符合性: 满足 SR-F-001 对规范化处理的需求

  - ID: SR-F-001.3
    标题: 功能与非功能分离
    描述: 系统自动识别需求中的功能描述和性能约束，分别归类并标记关联关系
    分配目标: BA-功能非功能分离 [1:1映射]
    验收标准: 分离准确率 > 90%，分离后标注关联关系
    对上层的符合性: 满足 SR-F-001 对需求分类的需求

  - ID: SR-F-001.4
    标题: 需求末级分解
    描述: 设计者对规范化后的原始需求进行逐级分解，系统自动验证 1:1 映射约束
    分配目标: BA-需求末级分解 [1:1映射]
    验收标准: 分解操作响应 < 2s，1:1 约束自动校验，违反时提示
    对上层的符合性: 满足 SR-F-001 对需求分解的需求

  - ID: SR-F-001.5
    标题: 冲突与重复检测
    描述: 系统自动检测末级需求之间的冲突和重复，分类展示检测结果
    分配目标: BA-冲突重复检测 [1:1映射]
    验收标准: 检测完成后自动分类（直接/间接/重复），准确率 > 90%
    对上层的符合性: 满足 SR-F-001 对质量检测的需求

架构末级节点：SR-F-002（架构设计管理功能）
详细定义末级节点：
  - ID: SR-F-002.1
    标题: SR架构树管理
    描述: 设计者创建和管理 SR-F 及 SR-NFR 架构树，定义架构末级节点
    分配目标: BA-SR架构树创建 [1:1映射]
    验收标准: 架构树支持多级缩进、节点拖拽、末级节点标注
    对上层的符合性: 满足 SR-F-002 对 SR 架构定义的需求

  - ID: SR-F-002.2
    标题: BA IPO定义
    描述: 设计者使用 IPO 模型定义业务组件，支持去重检查
    分配目标: BA-BA IPO定义 [1:1映射]
    验收标准: IPO 包含完整的 Input/Process/Output 定义，去重后显示重复提示
    对上层的符合性: 满足 SR-F-002 对 BA 架构定义的需求

  - ID: SR-F-002.3
    标题: SysReq架构定义
    描述: 设计者定义 0-9 级功能需求架构和 0-6 级非功能需求架构
    分配目标: BA-SysReq架构定义 [1:1映射]
    验收标准: 支持两种架构模式切换，5 级承接层自动关联 BA IPO
    对上层的符合性: 满足 SR-F-002 对 SysReq 架构定义的需求

  - ID: SR-F-002.4
    标题: PA架构定义
    描述: 设计者定义前端/后端/数据/基础设施四层组件架构
    分配目标: BA-PA架构定义 [1:1映射]
    验收标准: 组件分类清晰，承接 SysReq 9 级后自动校验 1:1 约束
    对上层的符合性: 满足 SR-F-002 对 PA 架构定义的需求

架构末级节点：SR-F-003（映射与追溯管理功能）
详细定义末级节点：
  - ID: SR-F-003.1
    标题: 映射关系建立
    描述: 设计者建立各层间的 N:1 映射关系，标注符合性分析
    分配目标: BA-映射关系建立 [1:1映射]
    验收标准: 支持批量映射和手动调整，映射覆盖率实时显示
    对上层的符合性: 满足 SR-F-003 对映射管理的需求

  - ID: SR-F-003.2
    标题: 双向追溯验证
    描述: 系统自动验证正向（OR→PA）和反向（PA→OR）追溯链路的完整性
    分配目标: BA-双向追溯验证 [1:1映射]
