## 一、跨步骤上下文管理

### 1.1 上下文传递方式

6 步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

### 1.2 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**（见本章末尾）中，供后续步骤引用。

### 1.3 回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见 AI决策框架 §7.2 跨步骤影响范围表）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

## 二、AI 经济原则

本步操作以下列顺序执行，AI 需注意：

1. **执行顺序不可逆**：分支合并 → 分析活动 → SysReq 5→9 级分解 → PA 架构定义（资产优先+同步）→ NFR 约束映射 → 闭环检查
   核心驱动力：SysReq 9 级场景活动逐条在现有 PA 架构中找宿主。PA 架构的调整（扩展/新增）由 SysReq 驱动，不是预设的
2. **基于资产优先**：PA 架构调整决策应先检查已有资产（t03 骨架 + 现有 PA 架构），确实需要修改时再改进，最后才新增
3. **资产路径已知**：本步的资产是 `90-eos-pipeline-system-product-data/07-eos-pipeline-product-architecture.md`（v1 基线区）+ t03 同步建立的 PA 骨架。`05-eos-pipeline-system-requirements-architecture.md` 是本步的输入数据（待分解的 SysReq），不是"资产"
4. **分支合并前提**：开始执行前必须确认功能分支和非功能分支均已完成。如有冲突，先记录后处理

### 3.3 SysReq 9 级 → PA（资产优先 + 同步映射）

> 先了解资产再动手——映射前加载现有 PA 架构（v1 基线 + t03 骨架），理解每个 PA 组件的职责边界和覆盖范围。然后面对每条 SysReq 9 级活动时，已经知道"哪个组件可以接、哪个方向还空着"。

```

—— 第一步：加载并理解 PA 架构资产（先做到心中有数）——

资产来源一：t03 同步建立的 PA 骨架
├─ 已有 PA 组件占位符名称和分类归属
├─ 已有 SysReq 5 级 → PA 骨架映射关系
└─ PA 骨架的分类体系（知识/方法类：规范与准则/设计指南/AI辅助文档/任务定义）

资产来源二：07-eos-pipeline-product-architecture.md（v1 基线·只读）
├─ 加载所有现有 PA 组件
├─ 理解每个组件的职责边界、技术栈、部署方式
└─ 评估与 SysReq 9 级活动的匹配程度

整体理解：
├─ PA 架构按什么分类体系组织？（知识/方法类 or 软件产品类）
├─ 哪些区域已有组件覆盖？哪些还空着？
└─ 现有 PA 组件粒度是否合理（1-2 周可独立完成开发）

⚠️ PA 分类体系选择：
  知识/方法类产品 → 规范与准则 / 设计指南 / AI辅助文档 / 任务定义
  软件类产品 → 前端/后端/数据/基础设施

—— 第二步：逐条 SysReq 9 级活动找 PA 宿主 ——

0. 【前置判断】这条 9 级活动涉及多个 PA 组件范围吗？
├─ 如果是 → 继续分解（9级尚未到原子粒度）
└─ 如果不是 → 进入步骤1

1. 【找宿主】用三个维度判断：

   ├─ Q1 功能职责匹配：此 SysReq 9 级活动的核心功能，属于哪个 PA 组件的职责范围？
   │  例："创建设计准则框架" — PA-准则文档? PA-模板库? PA-设计指南?
   │  判断依据：PA 组件定义中描述的职责边界和功能范围
   │
   ├─ Q2 覆盖范围偏差：此 9 级活动描述的具体内容，是否在 PA 组件定义的边界之内？
   │  例：PA-准则文档的定义是"制定和维护系统设计准则" → 
   │  "创建准则框架"属于 PA-准则文档的边界内
   │
   └─ Q3 同类承接情况：此 PA 组件是否已经在承接同类 SysReq 9 级活动？

   根据三个问题的答案做决策：
   ├─ 条件A — 复用骨架或现有 PA 组件：
   │  Q1命中 + Q2在边界内 + Q3同类
   │  → SysReq-9级 → PA-YYY（复用骨架或现有组件）
   │
   ├─ 条件B — 扩展 PA 组件定义后承接：
   │  Q1命中 + Q2略超边界 + Q3同类
   │  → 标记 PA-YYY（改进-扩展），记录扩展理由
   │
   ├─ 条件C — 新增 PA 组件：
   │  Q1无命中 + 不在任何 PA 组件边界内
   │  → 记录需新增 PA 组件，写明新增原因和所属分类
   │
   └─ 条件D — ❓ 无法判断：
       Q1命中了多个方向 / Q2边界完全模糊 / Q3信息不足以对比
       → 标记❓，说明卡在哪个维度上，**不要强行推测**，留给人类

2. 【记录】同步记录本步决策：
├─ 承接的 PA 组件：[PA-YYY / 新增]
├─ 资产决策：[复用骨架 / 改进-扩展 / 新增]
└─ 决策理由：引用以上三个判断维度的结果

⚠️ 关键判断原则：
- "功能职责匹配"的含义：9 级活动的类型与 PA 组件的构件类型对应
  （如"定义规范"→规范与准则构件、"编写操作步骤"→任务定义构件）
- ⚠️ **扩展和新增的强内聚性检查**：
  ├─ 扩展前问"扩展后这个 PA 组件还描述一个单一功能模块吗？"
  │  例：PA-准则文档扩展为"准则+模板+工具"→ 三个不同类型，应新增而非扩展
  ├─ 新增前问"这个新组件和同级兄弟组件的颗粒度一致吗？"
  │  例：相邻组件都是"独立交付文档/模块"的粒度，新组件不能是一个"子系统"
  └─ 颗粒度标准：组件粒度以"一个团队在 1-2 周内可完成开发"为标准
```

```
├─ 待传递的约束：[第6步追溯验证的关键检查点]
```

## 五、审查修复流程

### 5.1 审查输入

AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

### 5.2 AI 修复流程

收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）→ 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题** → 参考AI决策框架的跨步骤影响范围表进行影响范围评估，列出所有受影响的下游节点

### 5.3 影响范围评估

| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 某步的架构定义 | 本层 + 下游已建立映射的节点 | 修改后同步更新映射 |
| 详细定义 | 下层方案 | 修改后检查下层是否受影响 |
| 映射关系 | 所有关联节点 | 需重新运行追溯验证 |
| 编号变更 | 全链路 | 需逐层更新引用，重新验证 |
