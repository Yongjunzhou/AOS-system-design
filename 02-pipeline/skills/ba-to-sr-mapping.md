# Skill: 业务架构→系统需求映射

## 目的
将业务架构的节点分解为系统需求，确保每个业务架构节点有唯一的系统需求承接点。

## 输入
- 业务架构节点
- 现有系统需求

## 输出
- 映射关系（BA Node → SysReq）
- 映射验证结果
- 无法映射的节点和建议

## 映射规则

### 1. 多对一映射
- **原则**：N 条系统需求 → 1 个业务架构节点
- **约束**：1 个业务架构节点 → 1 个系统需求架构节点（业务承接层）
- **验证**：每个业务架构节点必须有且仅有一个对应的系统需求承接点

### 2. 语义覆盖验证
- 系统需求是否完全承接业务架构节点的语义
- 系统需求是否能够实现业务架构节点的所有功能
- 是否存在业务架构节点功能未被系统需求承接

### 3. 无法映射的处理
- 如果业务架构节点无法映射到任何现有系统需求：
  - 选项1：对业务架构节点进行再分解
  - 选项2：创建新的系统需求
  - 选项3：迭代业务架构的设计
- 决策应由架构师确认

## 映射模板

```markdown
### 业务架构→系统需求映射

| BA Node ID | BA Node 名称 | SysReq ID | 系统需求 | 映射状态 |
|-----------|-----------|----------|--------|--------|
| BA-NODE-001 | 订单处理节点 | SysReq-001, SysReq-002 | 订单创建、订单修改 | ✓ |
| BA-NODE-002 | 支付处理节点 | SysReq-003 | 支付处理 | ✓ |
| BA-NODE-003 | [节点] | [无映射] | N/A | ✗ 待处理 |

### 无法映射的节点处理

| BA Node ID | 节点名称 | 原因 | 建议处理方式 |
|-----------|---------|------|-----------|
| BA-NODE-003 | [节点] | [原因] | [处理建议] |
```

## 使用场景
- 新增业务架构节点的系统需求分解
- 系统需求的设计和规划
- 准则3的实现（业务架构与系统需求同步设计）
