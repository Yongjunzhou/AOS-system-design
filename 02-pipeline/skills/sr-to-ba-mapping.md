# Skill: 相关方需求→业务架构映射

## 目的
将规范化和分解后的相关方需求映射到业务架构的节点，确保每条相关方需求有唯一的业务承接点。

## 输入
- 相关方需求
- 业务架构

## 输出
- 映射关系（SR → BA Node）
- 映射验证结果
- 无法映射的需求和建议

## 映射规则

### 1. 多对一映射
- **原则**：N 条相关方需求 → 1 个业务架构节点
- **约束**：1 条相关方需求 → 1 个业务架构节点（一对一）
- **验证**：每条相关方需求必须有且仅有一个对应的业务架构节点

### 2. 语义覆盖验证
- 业务架构节点的职责是否完全覆盖相关方需求
- 业务架构节点的输出是否能够满足相关方需求
- 是否存在相关方需求未被承接

### 3. 无法映射的处理
- 如果相关方需求无法映射到任何现有业务架构节点：
  - 选项1：对相关方需求进行再分解
  - 选项2：创建新的业务架构节点
  - 选项3：迭代业务架构的设计
- 决策应由相关方和架构师共同确认

## 映射模板

```markdown
### 相关方需求→业务架构映射

| SR ID | 相关方需求 | BA Node ID | BA Node 名称 | 映射状态 |
|-------|----------|-----------|-----------|--------|
| SR-001 | 支持在线订单 | BA-NODE-001 | 订单处理节点 | ✓ |
| SR-002 | 支持支付 | BA-NODE-002 | 支付处理节点 | ✓ |
| SR-003 | [需求] | [无映射] | N/A | ✗ 待处理 |

### 无法映射的需求处理

| SR ID | 需求 | 原因 | 建议处理方式 |
|-------|------|------|-----------|
| SR-003 | [需求] | [原因] | [处理建议] |
```

## 使用场景
- 新增相关方需求的映射
- 业务架构的验证
- 准则2的实现（相关方需求与业务架构同步设计）
