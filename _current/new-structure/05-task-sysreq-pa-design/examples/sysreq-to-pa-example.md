# Task 5：SysReq → PA 映射 - 完整示例

**场景**：电商平台订单创建功能的产品架构设计

---

## 第一部分：系统需求（SysReq 9 级）

### 订单创建流程的场景活动

#### 场景 1：正常选择商品

```
SysReq-9-001：浏览商品列表
SysReq-9-002：搜索商品
SysReq-9-003：查看商品详情
SysReq-9-004：添加到购物车
```

#### 场景 2：填写收货地址

```
SysReq-9-005：输入地址信息
SysReq-9-006：验证地址
SysReq-9-007：保存地址
```

#### 场景 3：确认订单

```
SysReq-9-008：显示订单摘要
SysReq-9-009：计算订单金额
SysReq-9-010：生成订单号
```

#### 场景 4：提交订单

```
SysReq-9-011：保存订单信息
SysReq-9-012：扣减库存
SysReq-9-013：触发支付流程
```

## 第四部分：非功能权衡决策

### 性能权衡

**冲突**：
- 需要支持 1000+ 并发用户
- 需要平均响应时间 < 200ms

**权衡决策**：
- 使用 Redis 缓存购物车数据（PA-DB-002）
- 使用 CDN 加速静态资源
- 使用数据库连接池优化数据库访问
- 使用消息队列异步处理订单

**影响**：
- 增加缓存成本
- 提高系统复杂度
- 提升用户体验

### 安全权衡

**冲突**：
- 需要保护用户数据安全
- 需要快速响应

**权衡决策**：
- 使用 HTTPS 加密传输
- 使用 AES-256 加密敏感数据
- 使用 JWT 进行身份认证
- 使用 SQL 参数化防止 SQL 注入

**影响**：
- 增加计算成本
- 提高数据安全性
- 增加代码复杂度

### 可用性权衡

**冲突**：
- 需要 99.9% 的系统可用性
- 需要控制成本

**权衡决策**：
- 使用主从复制实现数据备份
- 使用负载均衡实现高可用
- 使用自动故障转移
- 定期进行灾难恢复演练

**影响**：
- 增加基础设施成本
- 提高系统可用性
- 增加运维复杂度

## 常见错误和最佳实践

### ❌ 常见错误

1. **一个 SysReq 9 级节点映射到多个 PA 末级节点**
   - ❌ 错误：SysReq-9-001 同时映射到 PA-FE-001 和 PA-BE-001
   - ✅ 正确：SysReq-9-001 只映射到 PA-FE-001（前端展示）

2. **忽视非功能权衡**
   - ❌ 错误：同时要求高性能、高安全性、低成本
   - ✅ 正确：明确权衡决策，如"选择性能优先"

3. **PA 末级节点定义不清**
   - ❌ 错误：订单组件
   - ✅ 正确：订单 API（明确的组件类型和职责）

### ✅ 最佳实践

1. **使用结构化格式**
   - 使用 YAML 或表格格式
   - 包含所有必填字段

2. **明确组件职责**
   - 每个 PA 末级节点都应该有明确的职责
   - 职责应该与 SysReq 9 级节点对应

3. **记录权衡决策**
   - 明确记录权衡决策及其原因
   - 便于后续的维护和优化
