# 性能指标与需求结构的关键改进

**日期**：2026-05-13  
**版本**：v1.0

---

## 问题背景

在之前的 AOS 架构中，存在一个关键的概念混淆：

**旧的理解**：
```
功能需求（SR-F）：用户登录
非功能需求（SR-NFR）：响应时间 < 2s

问题：
- 功能需求缺少性能指标
- 非功能需求只有指标，没有需求描述
- 性能指标被当作需求本身
```

## 改进方案

**新的理解**：
```
功能需求（SR-F）：用户登录
  ├─ 需求描述：用户输入用户名和密码，系统验证后登录
  └─ 性能指标：响应时间 < 2s，支持 1000+ 并发

非功能需求（SR-NFR）：系统响应时间
  ├─ 需求描述：系统在正常业务负载下应该快速响应
  └─ 性能指标：平均 < 200ms，P99 < 500ms，在 1000 并发下 < 300ms
```

**关键改进**：
- ✅ 性能指标是需求的属性，不是需求本身
- ✅ 每条需求都包含需求描述和性能指标
- ✅ 性能指标随需求一起追溯，确保不会遗漏

## 影响范围

### 1. 需求结构定义

**功能需求（SR-F）**：
| 字段 | 说明 | 示例 |
|------|------|------|
| 需求描述 | 系统应该提供什么功能 | 用户输入用户名和密码，系统验证后登录 |
| 性能指标 | 该功能的性能约束 | 响应时间 < 2s，支持 1000+ 并发用户 |

**非功能需求（SR-NFR）**：
| 字段 | 说明 | 示例 |
|------|------|------|
| 需求描述 | 系统应该满足什么质量特性 | 系统在正常业务负载下应该快速响应 |
| 性能指标 | 该质量特性的具体量化指标 | 平均 < 200ms，P99 < 500ms，在 1000 并发下 < 300ms |

### 2. 映射关系

**功能需求流**：
```
SR-F（需求描述 + 性能指标）
  ↓ N:1 映射
BA-5（需求描述 + 性能指标）
  ↓ N:1 映射
SysReq-5（需求描述 + 性能指标）
  ↓ N:1 映射
SysReq-9（需求描述 + 性能指标）
  ↓ N:1 映射
PA 末级（需求描述 + 性能指标）
```

**非功能需求流**：
```
SR-NFR（需求描述 + 性能指标）
  ↓ N:1 映射
SysReq-NFR-5（需求描述 + 性能指标）
  ↓ N:1 映射
PA 末级（需求描述 + 性能指标）
```

### 3. 追溯矩阵

**改进前**：
```
| SR ID | SR 名称 | 性能指标 | BA 5级 | SysReq 5级 | PA 末级 | 状态 |
|-------|--------|---------|--------|-----------|--------|------|
| SR-F-001 | 用户登录 | [无] | BA-5-001 | SysReq-5-001 | PA-FE-001 | ✓ |
| SR-NFR-001 | 响应时间 | < 2s | - | SysReq-NFR-5-001 | PA-CACHE-001 | ✓ |
```

**改进后**：
```
| SR ID | SR 名称 | 性能指标 | BA 5级 | SysReq 5级 | PA 末级 | 状态 |
|-------|--------|---------|--------|-----------|--------|------|
| SR-F-001 | 用户登录 | 响应时间 < 2s | BA-5-001 | SysReq-5-001 | PA-FE-001 | ✓ |
| SR-NFR-001 | 系统响应时间 | 平均 < 200ms | - | SysReq-NFR-5-001 | PA-CACHE-001 | ✓ |
```

## 常见问题

**Q：功能需求一定要有性能指标吗？**

A：是的。每条功能需求都应该有性能指标，即使是最小的指标（如"响应时间 < 5s"）。这样可以确保设计团队清楚地了解每个功能的性能约束。

**Q：非功能需求的需求描述应该包含什么？**

A：非功能需求的需求描述应该说明系统应该满足什么质量特性，为什么需要这个特性，以及这个特性的业务背景。例如：
- ❌ 错误："响应时间 < 200ms"
- ✅ 正确："系统在正常业务负载下应该快速响应，确保良好的用户体验。平均响应时间 < 200ms，P99 < 500ms"

**Q：性能指标应该在哪个阶段补充？**

A：性能指标应该在 Task 1（需求规范化）阶段就补充。这样可以确保所有下游的设计工作都基于完整的需求。

**Q：如果需求没有性能指标怎么办？**

A：这表示需求不完整。应该与相关方沟通，确定合理的性能指标。如果确实没有特殊的性能要求，也应该明确说明（如"无特殊性能要求"）。

## 总结

这个改进通过明确性能指标是需求的属性而不是需求本身，使得：

1. **需求更完整**：每个需求都包含描述和性能指标
2. **追溯更清晰**：性能指标随需求一起追溯，不会遗漏
3. **设计更准确**：设计团队可以清楚地看到每个需求的性能约束
4. **验证更有效**：可以清楚地验证每个需求是否满足性能指标
5. **文档更一致**：功能需求和非功能需求的结构更一致

这是 AOS 架构的一个重要改进，应该立即实施。
