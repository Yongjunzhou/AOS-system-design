## 一、常见错误

### 错误 1：混合功能需求和非功能需求

**❌ 错误示例**
```
需求：用户需要能够登录系统，并且系统要快速响应
```

**问题**：
- 这个需求混合了两个不同的需求
- 功能需求：用户登录
- 非功能需求：系统响应时间
- 混合需求会导致后续映射困难

**✅ 正确做法**
```
需求 1（功能）：用户登录
- 需求ID：SR-F-001
- 需求描述：用户应该能够通过用户名和密码登录系统
- 验收标准：登录成功率 > 99%

需求 2（非功能）：系统响应时间
- 需求ID：SR-NFR-001
- 需求描述：系统应该快速响应用户请求
- 量化指标：平均响应时间 < 200ms
```

**检查方法**：
- 问自己：这个需求是在描述"系统应该做什么"还是"系统应该如何做"？
- 如果既有"做什么"又有"如何做"，就需要分解

---

### 错误 2：需求表述不清

**❌ 错误示例**
```
需求：系统要快速响应
需求：支持多种支付方式
需求：数据要安全
```

**问题**：
- "快速"、"多种"、"安全"都是模糊的表述
- 无法确定验收标准
- 无法进行有效的测试

**✅ 正确做法**
```
需求 1：系统响应时间
- 量化指标：平均响应时间 < 200ms，P99 < 1000ms
- 验证方法：使用 JMeter 进行性能测试

需求 2：支持多种支付方式
- 具体方式：支付宝、微信、银行卡、余额支付
- 验收标准：支持至少 4 种方式，支付成功率 > 99%

需求 3：数据安全
- 具体措施：AES-256 加密、TLS 1.2+ 传输、行级加密
- 验证方法：进行安全审计和渗透测试
```

**检查方法**：
- 问自己：一个新人能否根据这个需求进行开发或测试？
- 如果不能，就需要补充更多细节

---

### 错误 3：遗漏验收标准

**❌ 错误示例**
```
需求：用户应该能够创建订单
```

**问题**：
- 没有明确的验收标准
- 无法判断需求是否完成
- 容易产生争议

**✅ 正确做法**
```
需求：用户应该能够创建订单
- 验收标准：
  1. 用户能够成功创建订单
  2. 订单号唯一且格式正确
  3. 订单信息完整保存
  4. 创建过程不超过 5 步
  5. 创建成功率 > 99%
```

**检查方法**：
- 问自己：如何判断这个需求是否完成？
- 如果无法回答，就需要补充验收标准

---

### 错误 4：没有检测重复需求

**❌ 错误示例**
```
需求 1：系统要快速响应
需求 2：系统要快速响应
需求 3：支持多种支付方式
需求 4：支持多种支付方式
```

**问题**：
- 重复需求会导致重复工作
- 浪费资源
- 容易产生不一致

**✅ 正确做法**
```
需求 1：系统响应时间
- 量化指标：平均响应时间 < 200ms，P99 < 1000ms

需求 2：支持多种支付方式
- 具体方式：支付宝、微信、银行卡、余额支付
```

**检查方法**：
- 使用重复检测 Skill
- 对比新增需求与已有需求
- 如果发现重复，进行合并

---

### 错误 5：没有检测冲突需求

**❌ 错误示例**
```
需求 1：系统要支持 1000+ 并发用户
需求 2：系统要快速响应（平均响应时间 < 100ms）
```

**问题**：
- 这两个需求可能存在冲突
- 高并发和快速响应可能无法同时满足
- 需要进行权衡和确认

**✅ 正确做法**
```
需求 1：系统性能
- 并发支持：1000+ 并发用户
- 响应时间：平均 < 200ms，P99 < 1000ms
- 说明：在 1000+ 并发下，平均响应时间应该 < 200ms
```

**检查方法**：
- 使用冲突检测 Skill
- 对比新增需求与已有需求
- 如果发现冲突，进行标记和确认

---

### 错误 6：需求分解不充分

**❌ 错误示例**
```
需求：订单管理系统
```

**问题**：
- 需求太宽泛，无法映射到具体的业务组件
- 无法进行有效的设计和开发

**✅ 正确做法**
```
需求 1：订单创建
需求 2：订单修改
需求 3：订单取消
需求 4：订单查询
需求 5：订单支付
需求 6：订单发货
需求 7：订单追踪
```

**检查方法**：
- 问自己：这个需求是否还能再分解？
- 如果能，就需要继续分解
- 分解后的每个需求应该能映射到唯一的业务组件

---

### 错误 7：缺少相关方信息

**❌ 错误示例**
```
需求：用户应该能够创建订单
```

**问题**：
- 没有明确相关方
- 无法确定需求的优先级
- 无法进行有效的沟通

**✅ 正确做法**
```
需求：用户应该能够创建订单
- 相关方：销售部门、客户、财务部门
- 优先级：P0（高）
- 背景/原因：电商平台的核心功能，支撑销售业务
```

**检查方法**：
- 问自己：谁提出了这个需求？
- 问自己：这个需求对谁很重要？
- 如果无法回答，就需要补充相关方信息

## 二、最佳实践

### 最佳实践 1：使用结构化格式

**推荐格式**：YAML 或表格

```yaml
需求ID: SR-F-001
需求名称: 订单创建
相关方: 销售部门、客户
优先级: P0（高）
需求描述: |
  用户应该能够通过系统创建订单，包括选择商品、数量、收货地址等信息。
背景/原因: |
  电商平台的核心功能，支撑销售业务。
验收标准: |
  1. 用户能够成功创建订单
  2. 订单号唯一且格式正确
  3. 订单信息完整保存
约束条件: |
  - 订单必须包含至少一个商品
  - 收货地址必须有效
相关需求: SR-F-002, SR-F-003
```

**优势**：
- 信息完整
- 易于解析
- 易于版本控制

---

### 最佳实践 2：量化指标

**功能需求的量化**：
```
需求：支持多种支付方式
- 具体方式：支付宝、微信、银行卡、余额支付（4 种）
- 支付成功率：> 99%
- 支付流程步数：不超过 3 步
```

**非功能需求的量化**：
```
需求：系统响应时间
- 平均响应时间：< 200ms
- P95 响应时间：< 500ms
- P99 响应时间：< 1000ms
```

**优势**：
- 明确验收标准
- 便于测试
- 避免争议

---

### 最佳实践 3：关联需求

**为什么要关联需求**：
- 帮助发现隐藏的冲突
- 帮助发现隐藏的重复
- 便于追踪需求链路

**如何关联需求**：
```
需求：订单创建
- 相关需求：SR-F-002（支付方式）, SR-F-003（订单通知）, SR-NFR-001（响应时间）
```

**检查方法**：
- 问自己：这个需求依赖于哪些其他需求？
- 问自己：哪些其他需求依赖于这个需求？

---

## 四、常见问题解答

**Q：如何判断需求是否分解充分？**

A：分解充分的标准是：
- 每个需求都能映射到唯一的业务组件
- 每个需求都能进行独立的设计和开发
- 每个需求都有明确的验收标准

**Q：如何处理相互冲突的需求？**

A：
1. 标记冲突
2. 记录冲突的原因
3. 等待人工确认
4. 根据确认结果进行调整（合并、删除或修改）

**Q：如何处理重复的需求？**

A：
1. 识别重复
2. 合并为一个需求
3. 保留更详细的描述
4. 记录合并的原因

**Q：占位符应该如何标记？**

A：使用 `[TODO: ...]` 标记待确认项，例如：
```
需求描述：用户应该能够通过 [TODO: 确认支付方式] 进行支付
```

**Q：如何确定需求的优先级？**

A：
- P0（高）：核心功能，必须实现
- P1（中）：重要功能，应该实现
- P2（低）：辅助功能，可以延后实现
- P3（极低）：可选功能，可以不实现
