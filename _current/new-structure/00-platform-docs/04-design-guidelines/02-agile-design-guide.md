# 敏捷式系统设计指南
**Agile System Design Guide**

**文档版本**：v1.0  
**创建日期**：2026-05-14  
**基于准则版本**：系统设计准则 v2.1  
**适用场景**：在已有产品技术文档的基础上，增加新需求或修订已有需求，开展增量系统设计

---

## 📌 场景描述

### 适用条件

- ✅ 已有完整的系统设计文档（5层）
- ✅ 已建立需求映射矩阵
- ✅ 需要增加新需求或修订已有需求
- ✅ 需要快速响应需求变更
- ✅ 适合产品迭代和敏捷开发

### 不适用条件

- ❌ 没有现有的系统设计文档
- ❌ 需要从头开始进行完整设计
- ❌ 需求变更影响范围超过50%

### 流水线应用对象的考虑

本指南用于设计系统设计流水线产品。在应用本指南时，应考虑流水线的三类应用对象（企业运营体系的开发者）：

- **需求提出者**
  - 需要支持增量需求的规范化和分解
  - 需要支持需求变更的影响分析
  - 需要支持需求的版本管理

- **业务流程开发者**
  - 需要支持业务流程的增量设计
  - 需要支持业务流程变更的影响分析
  - 需要支持多个业务流程的并行设计

- **信息系统开发者**
  - 需要支持系统需求的增量设计
  - 需要支持产品架构的增量设计
  - 需要支持多个团队的并行工作

## 🎯 核心目标

高效地将新需求集成到现有的系统设计中，保持文档的一致性和完整性，最小化对现有设计的影响

---

## 📊 关键概念

### 两阶段工作流程

**第一阶段：占位符创建（快速，1-2小时）**
- 快速扫描上游文档
- 识别所有末级节点
- 在下游文档中创建占位符
- 建立上下游的映射关系
- 冻结占位符结构

**第二阶段：具体设计（深入，2-6天）**
- 填充占位符的具体内容
- 设计业务流程、系统需求、产品架构等
- 为下一层级预留占位符
- 验证设计的完整性和一致性
- 冻结下游文档

**优势**：
- ✅ 上下游文档相对独立
- ✅ 多个团队可以并行工作
- ✅ 时间缩短30-50%
- ✅ 避免过度的同步迭代

---

## 📋 完整操作流程（8步）

### 第1步：新需求分析和分类（1天）

**目标**：分析新需求的性质和影响范围

**具体操作**：

1. **需求分类**
   - 功能需求 vs 非功能需求
   - 新增需求 vs 修改需求 vs 删除需求
   - 独立需求 vs 相关需求

2. **影响范围评估**
   - 是否影响现有的相关方需求？
   - 是否影响现有的业务架构？
   - 是否影响现有的系统需求？
   - 是否影响现有的产品架构？

3. **优先级评估**
   - 高优先级：影响范围大，需要立即处理
   - 中优先级：影响范围中等，可以在下一个迭代处理
   - 低优先级：影响范围小，可以延后处理

**输出**：
- 新需求分析报告
- 影响范围评估
- 优先级排序

---

### 第2步：新需求分解（1-2天）

**目标**：对新需求进行分解，得到末级需求

**具体操作**：

1. **需求分解**
   - 对每条新原始需求进行分解
   - 确保末级需求是原子性的、不可再分的
   - 末级需求的粒度应该与现有需求保持一致

2. **末级需求定义**
   ```
   新原始需求 N
   ├─ 末级需求 N.1：[具体的功能需求]
   ├─ 末级需求 N.2：[具体的功能需求]
   └─ 末级需求 N.3：[具体的功能需求]
   ```

3. **原始需求文档更新**
   - 在原始需求文档中添加新需求
   - 更新末级需求清单
   - 更新文档版本号

**输出**：
- 更新后的原始需求文档
- 新末级需求清单

---

### 第3步：相关方需求映射（2-3天）

**目标**：将新末级需求映射到相关方需求

**具体操作**：

1. **映射策略选择**
   
   **选项A：映射到现有相关方需求**
   - 如果新需求与现有相关方需求相关
   - 直接映射到现有的相关方需求条目
   - 更新相关方需求的承接需求清单
   - 更新符合性分析

   **选项B：创建新的相关方需求**
   - 如果新需求无法映射到现有相关方需求
   - 创建新的相关方需求条目
   - 定义新相关方需求的内容
   - 建立与新末级需求的映射

2. **映射验证**
   - 验证1:1分配约束
   - 验证N:1承接支持
   - 验证映射的合理性

3. **相关方需求文档更新**
   - 添加新的相关方需求条目（如果有）
   - 更新现有相关方需求的承接需求清单
   - 更新符合性分析
   - 更新原始需求分配说明
   - 更新文档版本号

**输出**：
- 更新后的相关方需求文档
- 新的需求映射矩阵（原始需求 → 相关方需求）

---

### 第4步：业务架构影响分析（2-3天）

**目标**：分析新需求对业务架构的影响

**具体操作**：

1. **影响分析**
   - 新相关方需求是否影响现有业务架构？
   - 是否需要修改现有业务组件？
   - 是否需要创建新的业务组件？

2. **业务架构更新策略**
   
   **策略A：映射到现有业务架构**
   - 如果新相关方需求可以由现有业务组件承接
   - 更新业务组件的承接需求清单
   - 更新符合性分析

   **策略B：修改现有业务架构**
   - 如果新相关方需求需要修改现有业务组件
   - 分析修改的影响范围
   - 更新业务组件的定义和描述
   - 更新符合性分析

   **策略C：创建新的业务架构**
   - 如果新相关方需求需要新的业务组件
   - 创建新的业务组件条目
   - 定义新业务组件的内容
   - 建立与新相关方需求的映射

3. **业务架构文档更新**
   - 添加新的业务架构条目（如果有）
   - 更新现有业务架构的承接需求清单
   - 更新符合性分析
   - 更新相关方需求分配说明
   - 更新文档版本号

**输出**：
- 业务架构影响分析报告
- 更新后的业务架构文档
- 新的需求映射矩阵（相关方需求 → 业务架构）

---

### 第5步：系统需求影响分析（2-3天）

**目标**：分析新需求对系统需求的影响

**具体操作**：

1. **影响分析**
   - 新业务架构是否影响现有系统需求？
   - 是否需要修改现有系统需求？
   - 是否需要创建新的系统需求？

2. **系统需求更新策略**
   
   **策略A：映射到现有系统需求**
   - 如果新业务架构可以由现有系统需求承接
   - 更新系统需求的承接需求清单
   - 更新符合性分析

   **策略B：修改现有系统需求**
   - 如果新业务架构需要修改现有系统需求
   - 分析修改的影响范围
   - 更新系统需求的定义和描述
   - 更新符合性分析

   **策略C：创建新的系统需求**
   - 如果新业务架构需要新的系统需求
   - 创建新的系统需求条目
   - 定义新系统需求的内容
   - 建立与新业务架构的映射

3. **系统需求文档更新**
   - 添加新的系统需求条目（如果有）
   - 更新现有系统需求的承接需求清单
   - 更新符合性分析
   - 更新业务架构分配说明
   - 更新文档版本号

**输出**：
- 系统需求影响分析报告
- 更新后的系统需求文档
- 新的需求映射矩阵（业务架构 → 系统需求）

---

### 第6步：产品架构影响分析（3-5天）

**目标**：分析新需求对产品架构的影响

**具体操作**：

1. **影响分析**
   - 新系统需求是否影响现有产品架构？
   - 是否需要修改现有产品组件？
   - 是否需要创建新的产品组件？

2. **产品架构更新策略**
   
   **策略A：映射到现有产品架构**
   - 如果新系统需求可以由现有产品组件承接
   - 更新产品组件的承接需求清单
   - 更新符合性分析

   **策略B：修改现有产品架构**
   - 如果新系统需求需要修改现有产品组件
   - 分析修改的影响范围
   - 更新产品组件的定义和描述
   - 更新实现细节和性能指标
   - 更新符合性分析

   **策略C：创建新的产品架构**
   - 如果新系统需求需要新的产品组件
   - 创建新的产品组件条目
   - 定义新产品组件的内容
   - 建立与新系统需求的映射

3. **产品架构文档更新**
   - 添加新的产品架构条目（如果有）
   - 更新现有产品架构的承接需求清单
   - 更新符合性分析
   - 更新系统需求分配说明
   - 更新文档版本号

**输出**：
- 产品架构影响分析报告
- 更新后的产品架构文档
- 新的需求映射矩阵（系统需求 → 产品架构）

---

### 第7步：完整性和一致性验证（1-2天）

**目标**：验证增量更新后的系统设计的完整性和一致性

**具体操作**：

1. **1:1 分配约束验证**
   - 验证所有新需求都分配到唯一的方案条目
   - 验证没有遗漏或重复的映射

2. **N:1 承接支持验证**
   - 验证每个方案条目承接的需求数量合理
   - 发现过度分解或过度聚合的情况

3. **双向追溯验证**
   - 验证所有超链接的有效性
   - 验证追溯链路的完整性

4. **符合性分析验证**
   - 验证每个方案条目的符合性分析完整
   - 验证符合性分析的逻辑清晰

5. **版本一致性验证**
   - 验证所有文档的版本号一致
   - 验证所有文档的更新时间一致

**输出**：
- 验证报告
- 问题清单和改进建议

---

### 第8步：文档完善和发布（1-2天）

**目标**：完善文档，发布更新后的系统设计文档

**具体操作**：

1. **文档完善**
   - 补充缺失的内容
   - 修正发现的问题
   - 优化文档结构和表述

2. **文档审查**
   - 邀请相关人员进行审查
   - 收集反馈意见
   - 进行最后的修改

3. **文档发布**
   - 确定文档版本号
   - 发布更新版本
   - 更新变更日志

4. **团队沟通**
   - 向开发团队说明新需求的影响
   - 向产品团队说明设计的变更
   - 向相关方说明需求的处理结果

**输出**：
- 更新后的系统设计文档集（5层）
- 更新后的需求映射矩阵
- 变更日志
- 影响分析报告

## 🎯 增量处理策略

### 1. 优先级处理

**原则**：
- 优先处理高优先级需求
- 分阶段处理，避免一次性大量变更
- 每个阶段都进行验证和审查

**实施**：
```
第一阶段：处理高优先级需求
  ├─ 分析和分解
  ├─ 映射到各层
  ├─ 验证和审查
  └─ 发布更新

第二阶段：处理中优先级需求
  ├─ 分析和分解
  ├─ 映射到各层
  ├─ 验证和审查
  └─ 发布更新

第三阶段：处理低优先级需求
  ├─ 分析和分解
  ├─ 映射到各层
  ├─ 验证和审查
  └─ 发布更新
```

### 2. 影响范围最小化

**原则**：
- 优先选择映射到现有方案条目
- 避免不必要的修改
- 避免过度分解或过度聚合

**实施**：
```
映射决策树：
  新需求是否可以映射到现有相关方需求？
    ├─ 是 → 映射到现有相关方需求
    │       └─ 是否可以映射到现有业务架构？
    │           ├─ 是 → 映射到现有业务架构
    │           │       └─ 是否可以映射到现有系统需求？
    │           │           ├─ 是 → 映射到现有系统需求
    │           │           │       └─ 是否可以映射到现有产品架构？
    │           │           │           ├─ 是 → 完成映射
    │           │           │           └─ 否 → 创建新产品架构
    │           │           └─ 否 → 创建新系统需求
    │           └─ 否 → 创建新业务架构
    └─ 否 → 创建新相关方需求
```

### 3. 两阶段工作流程

**第一阶段：占位符创建（1-2小时）**
- 快速扫描上游文档
- 识别所有末级节点
- 在下游文档中创建占位符
- 建立上下游的映射关系
- 冻结占位符结构

**第二阶段：具体设计（2-6天）**

- 为下一层级预留占位符

## 📊 质量指标

| 指标 | 优秀 | 良好 | 需要改进 |
|------|------|------|---------|
| 映射覆盖率 | 100% | 95-99% | <95% |
| 1:1 约束满足率 | 100% | 95-99% | <95% |
| 平均承接需求数 | 2-3 | 1-4 | >4 或 <1 |
| 符合性分析完整率 | 100% | 95-99% | <95% |
| 超链接有效率 | 100% | 95-99% | <95% |

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 新需求与现有需求冲突 | 需要重新设计 | 进行需求冲突分析，协调相关方 |
| 新需求影响范围大 | 需要大量修改 | 分阶段处理，优先处理高优先级需求 |
| 现有设计无法承接新需求 | 需要重新架构 | 进行架构评估，考虑是否需要重构 |
| 团队对变更的理解不一致 | 实现偏差 | 进行充分的沟通和审查 |

## 📝 使用建议

1. **需求到达时**
   - 进行快速的影响范围评估
   - 确定优先级
   - 决定处理时机

2. **设计过程中**
   - 严格按照8步流程执行
   - 定期进行质量检查
   - 及时处理发现的问题

3. **设计完成后**
   - 进行完整性和一致性验证
   - 更新需求映射矩阵
   - 进行团队沟通
