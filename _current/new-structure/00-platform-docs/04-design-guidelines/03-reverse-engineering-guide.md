# 逆向工程指南
**Reverse Engineering Guide**

**文档版本**：v1.0  
**创建日期**：2026-05-14  
**基于准则版本**：系统设计准则 v2.1  
**适用场景**：有完整的产品代码或图纸，系统设计流水线中各技术文档质量不高或缺失，需要生成系统设计的整套技术文档

---

## 📌 场景描述

### 适用条件

- ✅ 产品已开发完成或接近完成
- ✅ 有详细的代码实现或架构图纸
- ✅ 技术文档不全或不符合设计准则
- ✅ 缺少从需求到架构的完整追溯链路
- ✅ 需要生成符合准则的系统设计文档

### 不适用条件

- ❌ 产品代码不清晰或文档极度缺失
- ❌ 原开发人员无法参与
- ❌ 需要快速完成（时间紧张）

---

## 🎯 核心目标

通过反向工程，从代码和图纸中提取设计信息，重建符合设计准则的系统设计文档

```
产品代码/图纸 → 产品架构 → 系统需求 → 业务架构 → 相关方需求 → 原始需求
   第5层        第4层       第3层       第2层       第1层
```

---

## 📋 完整操作流程（8步）

### 第1步：代码和图纸分析（1-2天）

**目标**：理解产品的整体架构和功能模块

**具体操作**：

1. **架构图纸分析**
   - 收集所有架构图纸（系统架构图、模块图、数据流图等）
   - 识别主要的系统组件和模块
   - 识别组件之间的依赖关系
   - 记录关键的技术决策和设计考虑

2. **代码结构分析**
   - 分析代码的目录结构和模块划分
   - 识别主要的功能模块和子模块
   - 分析模块之间的调用关系
   - 识别关键的业务流程和数据流

3. **功能清单提取**
   - 从代码中提取所有实现的功能
   - 按照功能模块进行分类
   - 记录每个功能的实现方式和关键参数

**输出**：
- 产品架构图（第5层）
- 功能模块清单
- 关键技术决策记录

---

### 第2步：产品架构文档编写（2-3天）

**目标**：编写符合设计准则的产品架构文档

**具体操作**：

1. **产品架构条目定义**
   ```
   PA-001: [产品组件名称]
   ├─ 定义：[组件的定义]
   ├─ 描述：[实现说明]
   ├─ 实现细节：[代码位置、关键类/函数]
   ├─ 性能指标：[响应时间、吞吐量等]
   └─ 技术栈：[使用的技术和框架]
   ```

2. **产品架构文档结构**
   - 第一部分：条目化的产品架构（所有PA条目）
   - 第二部分：系统需求分配说明（待补充）

**输出**：
- 产品架构文档（初版）

---

### 第3步：系统需求反向推导（3-5天）

**目标**：从产品架构反向推导系统需求

**具体操作**：

1. **系统需求识别**
   - 分析每个产品组件实现了什么功能
   - 识别功能性需求（做什么）
   - 识别非功能性需求（性能、安全、可靠性等）
   - 记录每个系统需求的定义和验收标准

2. **系统需求条目定义**
   ```
   SysReq-001: [系统需求名称]
   ├─ 定义：[需求定义]
   ├─ 描述：[详细说明]
   ├─ 承接的产品架构：PA-001, PA-002, ...
   ├─ 符合性分析：[如何通过产品架构实现]
   └─ 验收标准：[验收条件]
   ```

3. **系统需求文档结构**
   - 第一部分：条目化的系统需求
   - 第二部分：产品架构分配说明

**输出**：
- 系统需求文档（初版）

---

### 第4步：业务架构反向推导（3-5天）

**目标**：从系统需求反向推导业务架构

**具体操作**：

1. **业务架构识别**
   - 分析系统需求的业务含义
   - 识别业务组件和业务流程
   - 识别业务组件之间的关系
   - 记录每个业务组件的定义和职责

2. **业务架构条目定义**
   ```
   BA-001: [业务组件名称]
   ├─ 定义：[业务组件定义]
   ├─ 描述：[业务流程说明]
   ├─ Input/Process/Output：[IPO详细说明]
   ├─ 承接的系统需求：SysReq-001, SysReq-002, ...
   ├─ 符合性分析：[如何通过系统需求实现]
   └─ 业务流程：[关键的业务流程]
   ```

3. **业务架构文档结构**
   - 第一部分：条目化的业务架构
   - 第二部分：系统需求分配说明

**输出**：
- 业务架构文档（初版）

---

### 第5步：相关方需求反向推导（2-3天）

**目标**：从业务架构反向推导相关方需求

**具体操作**：

1. **相关方需求识别**
   - 分析业务架构的相关方需求
   - 识别功能需求和非功能需求
   - 记录每条相关方需求的定义和验收标准

2. **相关方需求条目定义**
   ```
   SR-F-001: [相关方需求名称]
   ├─ 定义：[需求定义]
   ├─ 描述：[详细说明]
   ├─ 承接的业务架构：BA-001, BA-002, ...
   ├─ 符合性分析：[如何通过业务架构实现]
   └─ 验收标准：[验收条件]
   ```

3. **相关方需求文档结构**
   - 第一部分：条目化的相关方需求
   - 第二部分：业务架构分配说明

**输出**：
- 相关方需求文档（初版）

---

### 第6步：原始需求反向推导（1-2天）

**目标**：从相关方需求反向推导原始需求

**具体操作**：

1. **原始需求识别**
   - 分析相关方需求的来源
   - 识别原始需求的末级需求
   - 记录每条原始需求的定义

2. **原始需求文档结构**
   - 原始需求清单
   - 末级需求分解
   - 相关方需求分配说明

**输出**：
- 原始需求文档（初版）

---

### 第7步：完整性和一致性验证（2-3天）

**目标**：验证反向工程的完整性和一致性

**具体操作**：

1. **1:1 分配约束验证**
   - 验证每条需求都分配到唯一的方案条目
   - 发现遗漏或重复的映射

2. **N:1 承接支持验证**
   - 验证每个方案条目承接的需求数量合理
   - 发现过度分解或过度聚合的情况

3. **双向追溯验证**
   - 验证所有超链接的有效性
   - 验证追溯链路的完整性

4. **符合性分析验证**
   - 验证每个方案条目的符合性分析完整
   - 验证符合性分析的逻辑清晰

**输出**：
- 验证报告
- 问题清单和改进建议

---

### 第8步：文档完善和发布（2-3天）

**目标**：完善文档，发布符合设计准则的系统设计文档

**具体操作**：

1. **文档完善**
   - 补充缺失的内容
   - 修正发现的问题
   - 优化文档结构和表述

2. **文档审查**
   - 邀请相关人员进行审查
   - 收集反馈意见
   - 进行最后的修改

3. **文档发布**
   - 确定文档版本号
   - 发布最终版本
   - 建立文档维护机制

**输出**：
- 完整的系统设计文档集（5层）
- 需求映射矩阵
- 验证报告

## ⏱️ 时间估算

| 步骤 | 时间 | 说明 |
|------|------|------|
| 第1步：代码和图纸分析 | 1-2天 | 取决于产品复杂度 |
| 第2步：产品架构文档 | 2-3天 | 取决于产品规模 |
| 第3步：系统需求反向推导 | 3-5天 | 取决于需求复杂度 |
| 第4步：业务架构反向推导 | 3-5天 | 取决于业务复杂度 |
| 第5步：相关方需求反向推导 | 2-3天 | 取决于相关方数量 |
| 第6步：原始需求反向推导 | 1-2天 | 通常较快 |
| 第7步：完整性和一致性验证 | 2-3天 | 取决于问题数量 |
| 第8步：文档完善和发布 | 2-3天 | 取决于反馈数量 |
| **总计** | **16-26天** | **约3-4周** |

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 代码文档不清晰 | 难以理解设计意图 | 邀请原开发人员参与，进行代码审查 |
| 架构图纸过时 | 反向推导的架构不准确 | 与代码实现进行对比，发现差异 |
| 需求信息遗漏 | 反向推导的需求不完整 | 进行用户访谈，补充缺失的需求 |
| 映射关系复杂 | 难以建立清晰的映射 | 进行需求分解，简化映射关系 |

## 📊 质量指标

| 指标 | 优秀 | 良好 | 需要改进 |
|------|------|------|---------|
| 映射覆盖率 | 100% | 95-99% | <95% |
| 1:1 约束满足率 | 100% | 95-99% | <95% |
| 平均承接需求数 | 2-3 | 1-4 | >4 或 <1 |
| 符合性分析完整率 | 100% | 95-99% | <95% |
| 超链接有效率 | 100% | 95-99% | <95% |

## 📝 使用建议

1. **项目启动前**
   - 收集所有代码和图纸
   - 组建跨职能的分析团队
   - 邀请原开发人员参与

2. **分析过程中**
   - 严格按照8步流程执行
   - 定期进行质量检查
   - 及时处理发现的问题

3. **分析完成后**
   - 进行完整性和一致性验证
   - 生成需求映射矩阵
   - 建立文档维护机制
