# 系统设计实践指南：三种场景的操作流程
**System Design Handbook: Operational Procedures for Three Scenarios**

**文档版本**：v1.0  
**创建日期**：2026-05-13  
**目标受众**：产品经理、架构师、需求分析师、开发团队

---

## 📌 指南概览

本指南基于《产品开发系统设计准则》，为三种典型场景提供详细的操作流程和最佳实践。

**三种场景**：
1. **场景1**：已开发产品 — 技术文档不全，但有详细代码或图纸
2. **场景2**：新研产品 — 只有原始需求
3. **场景3**：在开发产品 — 系统设计已符合准则，增加新需求

---

## 🔄 场景1：已开发产品的反向工程

### 场景描述

- ✅ 产品已开发完成或接近完成
- ✅ 有详细的代码实现或架构图纸
- ❌ 技术文档不全或不符合设计准则
- ❌ 缺少从需求到架构的完整追溯链路

### 目标

通过反向工程，从代码和图纸中提取设计信息，重建符合设计准则的系统设计文档。

### 操作流程

#### 第1步：代码和图纸分析（1-2天）

**目标**：理解产品的整体架构和功能模块

**具体操作**：

1. **架构图纸分析**
   - 收集所有架构图纸（系统架构图、模块图、数据流图等）
   - 识别主要的系统组件和模块
   - 识别组件之间的依赖关系
   - 记录关键的技术决策和设计考虑

2. **代码结构分析**
   - 分析代码的目录结构和模块划分
   - 识别主要的功能模块和子模块
   - 分析模块之间的调用关系
   - 识别关键的业务流程和数据流

3. **功能清单提取**
   - 从代码中提取所有实现的功能
   - 按照功能模块进行分类
   - 记录每个功能的实现方式和关键参数

**输出**：
- 产品架构图（第5层）
- 功能模块清单
- 关键技术决策记录

#### 第2步：产品架构文档编写（2-3天）

**目标**：编写符合设计准则的产品架构文档

**具体操作**：

1. **产品架构条目定义**
   ```
   PA-001: [产品组件名称]
   ├─ 定义：[组件的定义]
   ├─ 描述：[实现说明]
   ├─ 实现细节：[代码位置、关键类/函数]
   ├─ 性能指标：[响应时间、吞吐量等]
   └─ 技术栈：[使用的技术和框架]
   ```

2. **产品架构文档结构**
   - 第一部分：条目化的产品架构（所有PA条目）
   - 第二部分：系统需求分配说明（待补充）

**输出**：
- 产品架构文档（初版）

#### 第3步：系统需求反向推导（3-5天）

**目标**：从产品架构反向推导系统需求

**具体操作**：

1. **系统需求识别**
   - 分析每个产品组件实现了什么功能
   - 识别功能性需求（做什么）
   - 识别非功能性需求（性能、安全、可靠性等）
   - 记录每个系统需求的定义和验收标准

2. **系统需求条目定义**
   ```
   SysReq-001: [系统需求名称]
   ├─ 定义：[需求定义]
   ├─ 描述：[详细说明]
   ├─ 承接的产品架构：PA-001, PA-002, ...
   ├─ 符合性分析：[如何通过产品架构实现]
   └─ 验收标准：[验收条件]
   ```

3. **系统需求文档结构**
   - 第一部分：条目化的系统需求
   - 第二部分：产品架构分配说明

**输出**：
- 系统需求文档（初版）

#### 第4步：业务架构反向推导（3-5天）

**目标**：从系统需求反向推导业务架构

**具体操作**：

1. **业务架构识别**
   - 分析系统需求的业务含义
   - 识别业务组件和业务流程
   - 识别业务组件之间的关系
   - 记录每个业务组件的定义和职责

2. **业务架构条目定义**
   ```
   BA-001: [业务组件名称]
   ├─ 定义：[业务组件定义]
   ├─ 描述：[业务流程说明]
   ├─ 承接的系统需求：SysReq-001, SysReq-002, ...
   ├─ 符合性分析：[如何通过系统需求实现]
   └─ 业务流程：[关键的业务流程]
   ```

3. **业务架构文档结构**
   - 第一部分：条目化的业务架构
   - 第二部分：系统需求分配说明

**输出**：
- 业务架构文档（初版）

#### 第5步：相关方需求反向推导（2-3天）

**目标**：从业务架构反向推导相关方需求

**具体操作**：

1. **相关方需求识别**
   - 分析业务架构的相关方需求
   - 识别功能需求和非功能需求
   - 记录每条相关方需求的定义和验收标准

2. **相关方需求条目定义**
   ```
   SR-F-001: [相关方需求名称]
   ├─ 定义：[需求定义]
   ├─ 描述：[详细说明]
   ├─ 承接的业务架构：BA-001, BA-002, ...
   ├─ 符合性分析：[如何通过业务架构实现]
   └─ 验收标准：[验收条件]
   ```

3. **相关方需求文档结构**
   - 第一部分：条目化的相关方需求
   - 第二部分：业务架构分配说明

**输出**：
- 相关方需求文档（初版）

#### 第6步：原始需求反向推导（1-2天）

**目标**：从相关方需求反向推导原始需求

**具体操作**：

1. **原始需求识别**
   - 分析相关方需求的来源
   - 识别原始需求的末级需求
   - 记录每条原始需求的定义

2. **原始需求文档结构**
   - 原始需求清单
   - 末级需求分解
   - 相关方需求分配说明

**输出**：
- 原始需求文档（初版）

#### 第7步：完整性和一致性验证（2-3天）

**目标**：验证反向工程的完整性和一致性

**具体操作**：

1. **1:1 分配约束验证**
   - 验证每条需求都分配到唯一的方案条目
   - 发现遗漏或重复的映射

2. **N:1 承接支持验证**
   - 验证每个方案条目承接的需求数量合理
   - 发现过度分解或过度聚合的情况

3. **双向追溯验证**
   - 验证所有超链接的有效性
   - 验证追溯链路的完整性

4. **符合性分析验证**
   - 验证每个方案条目的符合性分析完整
   - 验证符合性分析的逻辑清晰

**输出**：
- 验证报告
- 问题清单和改进建议

#### 第8步：文档完善和发布（2-3天）

**目标**：完善文档，发布符合设计准则的系统设计文档

**具体操作**：

1. **文档完善**
   - 补充缺失的内容
   - 修正发现的问题
   - 优化文档结构和表述

2. **文档审查**
   - 邀请相关人员进行审查
   - 收集反馈意见
   - 进行最后的修改

3. **文档发布**
   - 确定文档版本号
   - 发布最终版本
   - 建立文档维护机制

**输出**：
- 完整的系统设计文档集（5层）
- 需求映射矩阵
- 验证报告

### 时间估算

| 步骤 | 时间 | 说明 |
|------|------|------|
| 第1步：代码和图纸分析 | 1-2天 | 取决于产品复杂度 |
| 第2步：产品架构文档 | 2-3天 | 取决于产品规模 |
| 第3步：系统需求反向推导 | 3-5天 | 取决于需求复杂度 |
| 第4步：业务架构反向推导 | 3-5天 | 取决于业务复杂度 |
| 第5步：相关方需求反向推导 | 2-3天 | 取决于相关方数量 |
| 第6步：原始需求反向推导 | 1-2天 | 通常较快 |
| 第7步：完整性和一致性验证 | 2-3天 | 取决于问题数量 |
| 第8步：文档完善和发布 | 2-3天 | 取决于反馈数量 |
| **总计** | **16-26天** | **约3-4周** |

### 关键风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 代码文档不清晰 | 难以理解设计意图 | 邀请原开发人员参与，进行代码审查 |
| 架构图纸过时 | 反向推导的架构不准确 | 与代码实现进行对比，发现差异 |
| 需求信息遗漏 | 反向推导的需求不完整 | 进行用户访谈，补充缺失的需求 |
| 映射关系复杂 | 难以建立清晰的映射 | 进行需求分解，简化映射关系 |

## 🚀 场景2：新研产品的系统设计

### 场景描述

- ✅ 产品处于规划阶段
- ✅ 有明确的原始需求
- ❌ 没有任何设计文档或代码实现
- ❌ 需要从零开始进行系统设计

### 目标

从原始需求开始，按照设计准则，逐步完成系统设计的5层文档。

### 操作流程

#### 第1步：原始需求分解（2-3天）

**目标**：对原始需求进行分解，得到末级需求

**具体操作**：

1. **需求分解**
   - 对每条原始需求进行分解
   - 确保末级需求是原子性的、不可再分的
   - 末级需求的粒度应该合理（平均分解数2-3条）

2. **末级需求定义**
   ```
   原始需求 1
   ├─ 末级需求 1.1：[具体的功能需求]
   ├─ 末级需求 1.2：[具体的功能需求]
   └─ 末级需求 1.3：[具体的功能需求]
   ```

3. **原始需求文档编写**
   - 第一部分：条目化的原始需求和末级需求分解
   - 第二部分：相关方需求分配说明（待补充）

**输出**：
- 原始需求文档（初版）
- 末级需求清单

#### 第2步：相关方需求设计（3-5天）

**目标**：设计相关方需求，作为原始需求的方案

**具体操作**：

1. **相关方需求识别**
   - 分析末级需求的共性和关联性
   - 进行需求聚合，形成相关方需求
   - 分离功能需求和非功能需求

2. **相关方需求条目定义**
   ```
   SR-F-001: [相关方需求名称]
   ├─ 定义：[需求定义]
   ├─ 描述：[详细说明]
   ├─ 承接的末级需求：1.1, 1.2, 1.3
   ├─ 符合性分析：[如何满足所承接的需求]
   └─ 验收标准：[验收条件]
   ```

3. **相关方需求文档编写**
   - 第一部分：条目化的相关方需求
   - 第二部分：原始需求分配说明

**输出**：
- 相关方需求文档（初版）
- 需求映射矩阵（原始需求 → 相关方需求）

#### 第3步：业务架构设计（5-7天）

**目标**：设计业务架构，作为相关方需求功能部分的方案

**具体操作**：

1. **业务架构识别**
   - 分析相关方需求的业务含义
   - 识别业务组件和业务流程
   - 进行业务架构设计

2. **业务架构条目定义**
   ```
   BA-001: [业务组件名称]
   ├─ 定义：[业务组件定义]
   ├─ 描述：[业务流程说明]
   ├─ 承接的相关方需求：SR-F-001, SR-F-002
   ├─ 符合性分析：[如何满足所承接的需求]
   └─ 为下层提供的设计指导：[对系统需求的指导]
   ```

3. **业务架构文档编写**
   - 第一部分：条目化的业务架构
   - 第二部分：相关方需求分配说明

**输出**：
- 业务架构文档（初版）
- 需求映射矩阵（相关方需求 → 业务架构）

#### 第4步：系统需求设计（5-7天）

**目标**：设计系统需求，作为业务架构和相关方需求非功能部分的方案

**具体操作**：

1. **系统需求识别**
   - 分析业务架构的系统实现需求
   - 分析相关方需求的非功能需求
   - 进行系统需求设计

2. **系统需求条目定义**
   ```
   SysReq-001: [系统需求名称]
   ├─ 定义：[需求定义]
   ├─ 描述：[详细说明]
   ├─ 承接的业务架构/相关方需求：BA-001, SR-NF-001
   ├─ 符合性分析：[如何满足所承接的需求]
   └─ 为下层提供的设计指导：[对产品架构的指导]
   ```

3. **系统需求文档编写**
   - 第一部分：条目化的系统需求
   - 第二部分：业务架构/相关方需求分配说明

**输出**：
- 系统需求文档（初版）
- 需求映射矩阵（业务架构 → 系统需求）

#### 第5步：产品架构设计（7-10天）

**目标**：设计产品架构，作为系统需求的方案

**具体操作**：

1. **产品架构识别**
   - 分析系统需求的技术实现方案
   - 进行产品架构设计
   - 确定前后端组件划分

2. **产品架构条目定义**
   ```
   PA-001: [产品组件名称]
   ├─ 定义：[组件定义]
   ├─ 描述：[实现说明]
   ├─ 承接的系统需求：SysReq-001, SysReq-002
   ├─ 符合性分析：[如何满足所承接的需求]
   ├─ 实现细节：[技术实现方案]
   └─ 性能指标：[响应时间、吞吐量等]
   ```

3. **产品架构文档编写**
   - 第一部分：条目化的产品架构
   - 第二部分：系统需求分配说明

**输出**：
- 产品架构文档（初版）
- 需求映射矩阵（系统需求 → 产品架构）

#### 第6步：完整性和一致性验证（2-3天）

**目标**：验证系统设计的完整性和一致性

**具体操作**：

1. **1:1 分配约束验证**
2. **N:1 承接支持验证**
3. **双向追溯验证**
4. **符合性分析验证**

**输出**：
- 验证报告
- 问题清单和改进建议

#### 第7步：文档完善和发布（2-3天）

**目标**：完善文档，发布符合设计准则的系统设计文档

**输出**：
- 完整的系统设计文档集（5层）
- 需求映射矩阵
- 验证报告

### 时间估算

| 步骤 | 时间 | 说明 |
|------|------|------|
| 第1步：原始需求分解 | 2-3天 | 取决于需求数量 |
| 第2步：相关方需求设计 | 3-5天 | 取决于需求复杂度 |
| 第3步：业务架构设计 | 5-7天 | 取决于业务复杂度 |
| 第4步：系统需求设计 | 5-7天 | 取决于需求复杂度 |
| 第5步：产品架构设计 | 7-10天 | 取决于产品复杂度 |
| 第6步：完整性和一致性验证 | 2-3天 | 取决于问题数量 |
| 第7步：文档完善和发布 | 2-3天 | 取决于反馈数量 |
| **总计** | **26-41天** | **约4-6周** |

### 关键成功因素

| 因素 | 说明 |
|------|------|
| 需求清晰 | 原始需求应该清晰、完整、无冲突 |
| 团队协作 | 需要产品、架构、开发等多个角色的协作 |
| 迭代反馈 | 每个阶段都应该进行审查和反馈 |
| 工具支持 | 使用需求管理工具进行追溯和验证 |
| 文档规范 | 严格遵循设计准则的文档规范 |

## 📝 场景3：在开发产品的增量需求处理

### 场景描述

- ✅ 产品处于开发阶段
- ✅ 系统设计的所有技术报告已经符合设计准则
- ✅ 已建立完整的5层设计文档和需求映射矩阵
- ✅ 增加了新的原始需求
- ❌ 需要将新需求集成到现有的系统设计中

### 目标

高效地将新需求集成到现有的系统设计中，保持文档的一致性和完整性，最小化对现有设计的影响。

### 操作流程

#### 第1步：新需求分析和分类（1天）

**目标**：分析新需求的性质和影响范围

**具体操作**：

1. **需求分类**
   - 功能需求 vs 非功能需求
   - 新增需求 vs 修改需求 vs 删除需求
   - 独立需求 vs 相关需求

2. **影响范围评估**
   - 是否影响现有的相关方需求？
   - 是否影响现有的业务架构？
   - 是否影响现有的系统需求？
   - 是否影响现有的产品架构？

3. **优先级评估**
   - 高优先级：影响范围大，需要立即处理
   - 中优先级：影响范围中等，可以在下一个迭代处理
   - 低优先级：影响范围小，可以延后处理

**输出**：
- 新需求分析报告
- 影响范围评估
- 优先级排序

#### 第2步：新需求分解（1-2天）

**目标**：对新需求进行分解，得到末级需求

**具体操作**：

1. **需求分解**
   - 对每条新原始需求进行分解
   - 确保末级需求是原子性的、不可再分的
   - 末级需求的粒度应该与现有需求保持一致

2. **末级需求定义**
   ```
   新原始需求 N
   ├─ 末级需求 N.1：[具体的功能需求]
   ├─ 末级需求 N.2：[具体的功能需求]
   └─ 末级需求 N.3：[具体的功能需求]
   ```

3. **原始需求文档更新**
   - 在原始需求文档中添加新需求
   - 更新末级需求清单
   - 更新文档版本号

**输出**：
- 更新后的原始需求文档
- 新末级需求清单

#### 第3步：相关方需求映射（2-3天）

**目标**：将新末级需求映射到相关方需求

**具体操作**：

### 关键风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 新需求与现有需求冲突 | 需要重新设计 | 进行需求冲突分析，协调相关方 |

| 团队对变更的理解不一致 | 实现偏差 | 进行充分的沟通和审查 |

### 最佳实践

#### 1. 增量处理策略

**原则**：
- 优先处理高优先级需求
- 分阶段处理，避免一次性大量变更
- 每个阶段都进行验证和审查

**实施**：
```
第一阶段：处理高优先级需求
  ├─ 分析和分解
  ├─ 映射到各层
  ├─ 验证和审查
  └─ 发布更新

第二阶段：处理中优先级需求
  ├─ 分析和分解
  ├─ 映射到各层
  ├─ 验证和审查
  └─ 发布更新

第三阶段：处理低优先级需求
  ├─ 分析和分解
  ├─ 映射到各层
  ├─ 验证和审查
  └─ 发布更新
```

#### 2. 影响范围最小化

**原则**：
- 优先选择映射到现有方案条目
- 避免不必要的修改
- 避免过度分解或过度聚合

**实施**：
```
映射决策树：
  新需求是否可以映射到现有相关方需求？
    ├─ 是 → 映射到现有相关方需求
    │       └─ 是否可以映射到现有业务架构？
    │           ├─ 是 → 映射到现有业务架构
    │           │       └─ 是否可以映射到现有系统需求？
    │           │           ├─ 是 → 映射到现有系统需求
    │           │           │       └─ 是否可以映射到现有产品架构？
    │           │           │           ├─ 是 → 完成映射
    │           │           │           └─ 否 → 创建新产品架构
    │           │           └─ 否 → 创建新系统需求
    │           └─ 否 → 创建新业务架构
    └─ 否 → 创建新相关方需求
```

#### 3. 文档版本管理

**原则**：
- 每次更新都要更新版本号
- 记录每次更新的内容和原因
- 保持版本历史

**实施**：
```
版本号格式：v[主版本].[次版本]

更新规则：
- 重大变更（影响范围大）：主版本号 +1
- 小幅变更（影响范围小）：次版本号 +1

变更日志：
- 记录每次更新的日期
- 记录每次更新的内容
- 记录每次更新的原因
- 记录每次更新的影响范围
```

#### 4. 团队沟通

**原则**：
- 及时沟通新需求的影响
- 透明地说明设计的变更
- 获得相关方的认可

**实施**：
```
沟通计划：
1. 需求分析阶段
   - 向产品团队说明新需求的分析结果
   - 获得产品团队的确认

2. 映射阶段
   - 向架构团队说明映射策略
   - 获得架构团队的认可

3. 验证阶段
   - 向开发团队说明设计的变更
   - 获得开发团队的反馈

4. 发布阶段
   - 向所有相关方发布更新
   - 说明变更的内容和影响
```

## 🎯 通用检查清单

### 需求分解检查清单

- [ ] 每条原始需求都分解到末级需求
- [ ] 末级需求是原子性的、不可再分的
- [ ] 末级需求的粒度合理（平均分解数2-3条）
- [ ] 每条末级需求都有清晰的定义和验收标准
- [ ] 每条末级需求都分配到相关方需求的某个条目
- [ ] 没有遗漏或重复的映射

### 需求映射检查清单

- [ ] 每条上层需求都分配到下层方案的唯一一个条目（1:1约束）
- [ ] 每个下层方案条目都承接了上层的多条需求（N:1支持）
- [ ] 承接的需求数量合理（通常2-4条）
- [ ] 承接的需求之间有逻辑关联
- [ ] 映射关系符合业务逻辑

### 符合性分析检查清单

- [ ] 每个方案条目都包含符合性分析
- [ ] 符合性分析清晰、具体、可验证
- [ ] 符合性分析说明了如何满足所承接的需求
- [ ] 符合性分析没有遗漏任何承接的需求
- [ ] 符合性分析的逻辑清晰、易于理解

### 双向追溯检查清单

- [ ] 每条上层需求都标注了分配到的下层方案条目
- [ ] 每个下层方案条目都标注了承接的上层需求
- [ ] 所有超链接格式正确
- [ ] 所有超链接都指向有效的锚点
- [ ] 所有超链接都指向正确的目标
- [ ] 用户能够快速导航和追溯

### 文档规范化检查清单

- [ ] 文档结构清晰、一致
- [ ] 每层报告都包含两个部分（条目化内容+映射说明）
- [ ] 文档中只保留必要的内容，去除冗余信息
- [ ] 文档易于维护和更新
- [ ] 文档版本号正确更新
