## 一、核心概念

### 1.1 详细定义 vs 架构定义

在相关方需求的设计中，我们需要区分两个不同的层次：

| 维度 | 架构定义 | 详细定义 |
|------|---------|---------|
| **目的** | 对相关方需求的分类和组织 | 对架构末级节点的详细设计分解 |
| **基本单位** | 相关方需求条目（末级节点）| 对条目的分解项目 |
| **示例ID** | SR-F-001, SR-NFR-001 | SR-F-001.1, SR-F-001.1.1 |
| **作用** | 需求的分类体系 | 需求的具体化说明 |
| **承接关系** | 承接原始需求末级 | 分配到业务架构末级 |
| **颗粒度** | 相对粗粒度 | 原子级（不可再分） |
| **验收方式** | 结构完整性 | 映射唯一性 |

**关键区别**：
- **架构定义**是对需求的"分类"——回答"有什么需求"
- **详细定义**是对需求的"分解"——回答"每个需求具体包括什么"

---

### 1.2 详细定义的双重身份

详细定义在流水线中扮演双重角色，这是设计能够层层递进的关键：

```
本层身份 ↔ 下层身份

详细定义是：
1. 本层身份：对相关方需求架构末级节点的细化分解（向内分解）
2. 下层身份：对业务架构的需求输入（向下传递）
```

**实际含义**：
- 一条相关方需求（如SR-F-001）可能分解为多条详细定义末级节点（SR-F-001.1, SR-F-001.2等）
- 这些详细定义末级节点集合 = 对SR-F-001的完整说明
- 每条详细定义末级节点 = 业务架构设计时需要承接的具体需求

**示例**：
```
架构末级：SR-F-001（需求规范化功能）
   ↓ [分解为]
详细定义末级：
   SR-F-001.1（原始需求收集）
   SR-F-001.2（需求规范化处理）
   SR-F-001.3（需求分解）
   SR-F-001.4（冲突和重复检测）
```

这些SR-F-001.1到SR-F-001.4既是对SR-F-001的分解说明，又是任务2中对业务架构的需求。

---

### 1.3 分解标准：1:1映射约束

详细定义的分解应遵循1:1映射约束：

**核心原则**：
每条详细定义末级节点只应分配到业务架构中**唯一一个**末级节点（IPO）

**判断标准**：

- ❌ **分解不充分**：一条详细定义末级节点需要分配到业务架构的**多个**末级节点
  - 原因：该条目还能继续分解
  - 处理：继续分解，直到每条只需一个下层节点承接

- ✅ **分解充分**：一条详细定义末级节点只需分配到业务架构的**一个**末级节点
  - 原因：该条目已经是原子性的
  - 处理：停止分解，该条目可以成为分配目标

**实际应用**：
在设计每条详细定义末级节点时，问自己：
> "这个需求能映射到业务架构的**一个**IPO吗？"
> - 如果答案是"不能，需要多个"，说明分解不够，继续分解
> - 如果答案是"可以"，则分解完成

---

### 1.4 功能 vs 非功能需求的处理区别

详细定义对两类需求有**不同的处理方式**：

| 方面 | 功能需求 | 非功能需求 |
|------|---------|----------|
| **分解方式** | 支持多层分解（SR-F-001.1.1.1等） | 不需要分层分解，直接到末级 |
| **映射目标** | 业务架构的IPO | 系统约束（不映射到业务架构） |
| **详细定义内容** | 具体的功能描述 | 量化指标和验证方法 |
| **分解终点** | 能映射到一个IPO | 能独立验证的约束条件 |
| **验收方式** | 功能完成 | 指标达成 |

---

### 1.5 相关方需求的两份子文档结构

根据系统设计准则，相关方需求由**两份独立的子文档**组成，形成完整的需求设计规范：

#### 第一份：架构定义文档

**定义**：对相关方需求的分类和组织

**内容包括**：
- 相关方需求的树形分类体系（根→分类→末级节点）
- 架构末级节点的定义和说明
- 原始需求末级到架构末级节点的映射关系
- 双向追溯的超链接

**作用**：
- 建立需求的分类框架
- 明确每个架构末级节点承接的原始需求（N:1关系）
- 支持需求的端到端追溯

**设计标准**：详见本规范"第二章 架构定义规范"

#### 第二份：详细定义文档

**定义**：对架构末级节点的详细设计分解

**内容包括**：
- 对每个架构末级节点的分层分解
- 分解到末级节点只需映射到业务架构的一个IPO（1:1约束）
- 每个详细定义末级节点的完整说明
- 验收标准和符合性分析

**作用**：
- 细化架构末级节点的定义
- 提供业务架构设计的具体输入
- 支持后续的业务架构末级节点设计

**设计标准**：详见本规范"第三章 详细定义规范"

#### 两份文档的关系

```
相关方需求

├─ 架构定义文档
│  ├─ 分类体系（树形结构）
│  │  └─ 架构末级节点 SR-F-001, SR-F-002, SR-NFR-001...
│  │     ├─ 承接原始需求末级：N:1关系
│  │     └─ 为下层提供设计指导
│  │
│  └─ 映射视图（表格）
│     └─ 原始需求末级 ↔ 架构末级节点的对应关系

└─ 详细定义文档
   └─ 对每个架构末级节点的分解
      ├─ SR-F-001.1, SR-F-001.2, SR-F-001.3...
      │  └─ 分配到业务架构的IPO：1:1关系
      │
      └─ SR-NFR-001.1, SR-NFR-001.2...
         └─ 标记为系统约束
```

### 1.6 详细定义与下层架构定义的同步设计

详细定义末级节点需要分配到下层方案架构末级节点（1:1约束）。因此，在开展上层需求详细定义的同时，应同步开展下层方案架构定义——即使只是结构骨架层级。

**核心原因**：
1. **提供分解的"靶子"**：下层架构定义的存在，使分解过程有明确的分配目标，确保每条详细定义末级节点在分解时就能判断"是否只需要一个下层架构末级节点承接"
2. **避免无目标返工**：如果下层架构定义尚未建立，详细定义分解到末级后可能发现没有对应的下层目标可承接，导致需要调整分解粒度
3. **为并行设计创造条件**：同步建立架构结构骨架（即使只是末级节点名称和映射关系），上下游团队可并行填充详细内容，缩短整体设计周期

**实际操作方式**：
1. **同步建立**：在开始上层详细定义分解时，同步在下层方案文档中建立架构末级节点的结构骨架
2. **边分解边映射**：每完成一条详细定义末级节点，立即检查是否能分配到下层的一个架构末级节点；如果发现不能（需要多个），则说明分解不够，继续分解
3. **同步填充**：结构骨架冻结后，上下层同时进入详细设计阶段

**实现方式——两阶段工作流程**：
第一阶段快速约定结构骨架（末级节点名称和映射关系），第二阶段并行填充详细内容。详见本规范"第九章 完整的两层设计流程"中的迭代同步说明。

### 1.7 性能指标的跨层传递

性能指标**不是独立的需求**，而是需求的一个属性。每条需求都应包含：
- **需求描述**：系统应该做什么或满足什么质量特性
- **性能指标**：该需求的具体性能约束或量化指标

性能指标随需求一起，在流水线的各层之间逐层传递：

```
第2层：相关方需求（SR）  ← 定义性能指标的起点
  │  每条SR需求包含：需求描述 + 性能指标
  ↓  N:1分配
第3层：业务架构（BA）     ← 继承并细化性能指标
  │  每个BA的IPO包含：需求描述 + 性能指标（继承自SR）
  ↓  IPO去重后作为系统需求输入
第4层：系统需求（SysReq） ← 继承并技术化性能指标
  │  功能SysReq包含：需求描述 + 性能指标（继承自BA）
  │  非功能SysReq包含：需求描述 + 量化指标（继承自SR-NFR）
  ↓  N:1分配
第5层：产品架构（PA）     ← 最终实现并验证性能指标
    每个PA组件包含：需求描述 + 性能指标（继承自SysReq）
```

- **功能需求性能指标**：响应时间、吞吐量、并发数、成功率等
- **非功能需求性能指标**：验证条件、测试方法、具体数值、适用范围等

本规范聚焦于第2层（相关方需求）的性能指标定义。在架构定义阶段为每个架构末级节点明确性能指标，在详细定义阶段为每条详细定义末级节点细化验收标准中的性能要求。后续层级的性能指标定义由相应准则规范覆盖。

---

### 1.8 分析：验证设计的两种符合性

分析（Analysis）是验证分解充分性和分配正确性的关键活动。在流水线设计中，分析分为两个维度：

**需求说明符合性分析**：确认每个架构末级节点是否充分满足所承接的上层需求，检查映射关系是否完整、正确、无矛盾。
- 手段：覆盖度检查、一致性校验、追溯验证
- 作用场景：架构定义完成后、详细定义开始时

**需求性能符合性分析**：验证设计方案能否达到所承接的性能指标要求，确保性能需求在架构层面可实现。
- 手段：建模仿真、理论计算等
- 作用场景：性能指标定义后、方案设计过程中

这两种分析分别对应需求的两个属性——**需求描述**和**性能指标**——确保两者都得到充分验证。详见系统设计准则中"架构末级节点的分解、分配与分析"。

## 二、架构定义规范

### 2.1 相关方需求架构末级节点的定义

**架构末级节点**是相关方需求设计中最重要的概念，是最大颗粒度的相关方需求条目。

#### 定义和特征

**什么是架构末级节点**：
- 相关方需求分类体系中的最细粒度节点
- 承接原始需求末级的关键层级
- 对应一个有明确界限的相关方需求
- 为业务架构设计提供清晰的设计指导

**架构末级节点的特征**：
- ✅ **可独立定义**：有清晰、完整的定义和描述
- ✅ **可独立验证**：有明确的验收标准和评估方式
- ✅ **相对稳定**：不会因为实现细节而频繁变更
- ✅ **承接多条原始需求**：体现N:1的映射关系
- ✅ **分配明确**：每条承接需求都有明确的分配关系

#### 功能需求的架构末级节点

**功能需求架构末级节点（SR-F-XXX）的标准**：

| 属性 | 要求 | 说明 |
|------|------|------|
| **ID格式** | SR-F-[号] | 示例：SR-F-001, SR-F-002 |
| **标题** | 一个完整的功能或业务需求 | 例如：需求规范化功能、系统需求映射功能 |
| **定义** | 清晰的需求叙述 | 描述用户/系统应该能够做什么 |
| **承接需求** | N条原始需求末级 | 说明该节点承接了哪些原始需求 |
| **符合性分析** | 如何满足所承接的需求 | 说明该节点如何满足上层需求 |
| **设计指导** | 为业务架构的指导 | 暗示业务架构应该如何设计 |
| **性能指标** | 功能相关的性能要求 | 例如：处理效率、准确率等 |

**功能需求架构末级节点的颗粒度标准**：

✅ **适当的粒度**（应该分为末级节点）：
- 代表一个完整的业务能力
- 用户可以单独使用的功能
- 可以有相对独立的业务规则
- 涉及明确的业务流程或交互

❌ **粒度过粗**（应该进一步分类）：
- 涵盖多个明显不同的功能
- 需要多个不同的业务流程来实现
- 跨越多个业务域或角色

❌ **粒度过细**（应该合并到上级）：
- 只是某个功能的一个小部分
- 无法独立定义和验证
- 与其他需求紧密耦合

**示例**：

```
✅ 合适的功能需求架构末级节点：
SR-F-001: 需求规范化功能
  定义：开发者能够收集、规范化、分解、验证相关方需求
  承接需求：原始需求-001, 原始需求-003, 原始需求-005
  性能指标：处理效率高（支持批量处理），准确率>95%

SR-F-002: 业务架构设计功能
  定义：系统支持业务架构的定义、设计、验证
  承接需求：原始需求-002, 原始需求-004
  性能指标：设计周期缩短50%，设计一致性>90%

❌ 不合适的例子：
SR-F-001: 系统功能管理  ← 太粗糙，应该分为多个
SR-F-002: 需求ID生成   ← 太细化，应该并入需求规范化功能
```

---

#### 非功能需求的架构末级节点

**非功能需求架构末级节点（SR-NFR-XXX）的标准**：

| 属性 | 要求 | 说明 |
|------|------|------|
| **ID格式** | SR-NFR-[号] | 示例：SR-NFR-001, SR-NFR-002 |
| **需求类型** | 性能/安全/可用性等 | 明确的非功能需求类型 |
| **需求类别** | 质量特性/环境适应性/可实现性 | 三大类之一 |
| **定义** | 系统应满足的质量属性 | 例如：系统应快速响应、数据应加密等 |
| **承接需求** | N条原始非功能需求末级 | 说明该节点承接的原始需求 |
| **符合性分析** | 如何满足所承接的需求 | 说明该节点如何满足上层需求 |
| **量化指标** | 具体的、可测量的指标 | 包含数值、单位、条件 |

**非功能需求架构末级节点的颗粒度标准**：

✅ **适当的粒度**（应该分为末级节点）：
- 一个完整的质量属性维度
- 有独立的、可测量的指标体系
- 可以独立进行测试和验证
- 有明确的责任所有者和验证方法

❌ **粒度过粗**（应该进一步细化）：
- 混合多个质量属性
- 指标不具体或无法测量

❌ **粒度过细**（应该合并到上级）：
- 只是某个质量属性的一个子指标
- 无法独立定义和验证

**示例**：

```
✅ 合适的非功能需求架构末级节点：
SR-NFR-001: 系统响应时间性能需求
  类型：性能
  定义：系统应在合理时间内响应用户请求
  量化指标：平均<200ms，P99<1000ms
  承接需求：原始需求-006, 原始需求-007

SR-NFR-002: 数据安全需求
  类型：安全
  定义：系统应保护用户数据安全
  量化指标：TLS传输、AES-256存储加密
  承接需求：原始需求-008

❌ 不合适的例子：
SR-NFR-001: 性能和安全   ← 混合两个不同的质量属性
SR-NFR-002: 响应快       ← 没有具体量化指标
```

### 2.2 原始需求到架构末级节点的映射（N:1关系）

**映射关系的含义**：
- 一个架构末级节点可以承接**多条（N条）**原始需求末级
- 多条相似或相关的原始需求可以合并到一个架构末级节点
- 这样可以简化需求结构，提高设计效率

**映射的设计原则**：

1. **关联性原则**：承接的原始需求之间应该有逻辑关联
   - ✅ 正确：需求收集、需求规范化、需求分解 → 相关方需求管理功能
   - ❌ 错误：相关方需求管理、系统访问控制 → 系统管理功能（太宽泛）

2. **完整性原则**：架构末级节点应该完整覆盖其承接的所有需求
   - ✅ 确保所有原始需求末级都被映射到某个架构末级节点
   - ❌ 不能遗漏原始需求末级

3. **清晰性原则**：映射关系应该清晰易懂
   - ✅ 在文档中明确标注映射关系
   - ✅ 使用双向追溯链接
   - ❌ 隐含的、难以理解的映射

**映射的操作方法**：

**第一步：列举所有原始需求末级**
```
原始需求末级清单：
- 需求-001: 需求收集和输入
- 需求-002: 需求规范化处理
- 需求-003: 需求分解分析
- 需求-004: 业务架构设计
- 需求-005: 架构验证
```

**第二步：按关联性分组**
```
分组结果：
组1（规范化相关）：需求-001, 002, 003
组2（架构相关）：需求-004, 005
```

**第三步：为每组定义架构末级节点**
```
架构末级节点：
SR-F-001: 需求规范化功能
  承接：需求-001, 需求-002, 需求-003
  
SR-F-002: 业务架构设计功能
  承接：需求-004, 需求-005
```

### 2.6 两层架构设计的协作（架构定义与详细定义）

在架构定义确定之后，详细定义会进一步细化每个架构末级节点：

```
第一层：架构定义（组织和分类）
  SR-F-001: 相关方需求规范化功能 ← 架构末级节点
    ↓ 承接原始需求末级：需求-001, 002, 003
    └─ 设计指导：...

第二层：详细定义（分解和细化）
  SR-F-001.1: 需求收集 ← 详细定义末级节点
  SR-F-001.2: 需求规范化
  SR-F-001.3: 需求分解
  SR-F-001.4: 冲突检测
    ↓ 每条都分配到业务架构的一个IPO（1:1）
    └─ BA-需求收集流程、BA-规范化流程等
```

**关键是两层的明确分工**：
- **架构定义层**：按逻辑关系将原始需求分组（N:1）
- **详细定义层**：按功能细节进一步分解架构末级（1:N）

## 三、功能需求的详细定义规范

### 3.1 详细定义末级节点的结构

每条功能需求详细定义末级节点应包含以下内容：

| 字段 | 要求 | 说明 |
|------|------|------|
| **ID** | 唯一标识符，格式SR-F-[上级]-[分解号] | 示例：SR-F-001.1, SR-F-001.1.1 |
| **标题** | 一句话概括 | 清晰、简洁、名词性或动词-名词结构 |
| **详细描述** | 完整的需求叙述 | 避免模糊词汇；使用"系统应该..."或"用户能够..." |
| **分解依据** | 为什么进行这样的分解 | 说明该条目的独立意义 |
| **分配目标** | 映射到业务架构的末级节点 | 标注为BA-XXX或IPO名称 |
| **验收标准** | 可测试的标准 | 功能实现后如何验证 |
| **对上层的符合性** | 说明如何满足上层需求 | 链接到上层架构末级节点 |

### 3.2 详细定义末级节点的示例

#### 示例 1：单层分解

```
架构末级节点：SR-F-001（相关方需求规范化功能）
  定义：用户能够完整处理相关方需求规范化的全流程

详细定义末级节点：
  SR-F-001.1（需求收集功能）
  ├─ 描述：用户能够输入需求信息（需求提出者、需求类型、分解粒度等）并创建新需求记录
  ├─ 分解依据：需求处理流程的第一个环节，是独立的业务流程
  ├─ 分配目标：BA-需求收集流程 [1:1映射]
  ├─ 验收标准：需求收集成功率 > 99%，需求ID唯一且格式正确
  └─ 符合性：直接满足SR-F-001中对需求收集的需求
```

#### 示例 2：多层分解

```
架构末级节点：SR-F-002（业务架构设计功能）
  定义：系统支持多种业务模式的架构设计映射

详细定义末级节点第一层：
  SR-F-002.1（架构映射管理）
  ├─ 描述：架构师在设计完相关方需求后，系统进行架构映射
  ├─ 分配目标：BA-架构映射流程
  └─ [需继续分解...]

详细定义末级节点第二层：
  SR-F-002.1.1（架构节点定义）
  ├─ 描述：架构师在设计时能够定义不同类型的架构节点（业务组件、引擎组件等）
  ├─ 分配目标：BA-架构节点定义 [1:1映射]
  ├─ 验收标准：支持至少3种架构节点类型，设计者能明确看到节点选项
  └─ 符合性：满足SR-F-002对多种架构模式的需求

  SR-F-002.1.2（映射关系验证）
  ├─ 描述：架构设计完成后，系统记录映射关系并进行一致性验证
  ├─ 分配目标：BA-映射验证流程 [1:1映射]
  ├─ 验收标准：映射验证成功率 > 99.5%，用户在5秒内收到验证反馈
  └─ 符合性：满足SR-F-002对架构设计完整性的需求
```

### 3.3 功能需求详细定义的分解标准

**何时需要分解**：

- ✅ 一个架构末级节点涉及**多个独立的功能流程**
  - 示例：需求规范化 = 收集+规范化+分解+冲突检测，应分解为4个
  
- ✅ 一个需求涉及**多个用户角色的交互**
  - 示例：架构映射 = 需求分析+节点定义+映射关系标注，应按角色分解
  
- ✅ 一个需求涉及**多个业务规则或条件**
  - 示例：冲突检测 = 直接冲突检测+间接冲突检测，应分解为两个

- ✅ 一个需求涉及**多个数据操作或查询条件**
  - 示例：需求查询 = 按需求ID查询+按需求类型查询+按优先级查询，应分解为三个

**何时可以停止分解**：

- ✅ 该条目能映射到业务架构的**一个**IPO或业务流程
- ✅ 该条目是**原子性的**、不可再分的业务单元
- ✅ 该条目有**清晰的验收标准**
- ✅ 该条目**不涉及多个独立的业务流程或角色**

### 3.4 详细定义的标注格式

**YAML格式（推荐用于文档编写）**：

```yaml
相关方需求: SR-F-001
架构末级节点: 相关方需求规范化功能

详细定义末级节点:
  - ID: SR-F-001.1
    标题: 需求收集
    描述: |
      用户能够输入需求基本信息（提出者、需求类型、优先级等）
      并创建新需求记录，系统生成唯一的需求ID
    分解依据: 需求收集是需求处理生命周期的第一个独立环节
    分配目标: 业务架构末级节点BA-需求收集流程
    验收标准: |
      1. 收集成功率 > 99%
      2. 需求ID唯一且格式正确
      3. 需求信息完整保存
    对上层的符合性: 直接满足SR-F-001中对需求收集的要求

  - ID: SR-F-001.2
    标题: 需求规范化
    描述: |
      用户能够对已收集的需求进行规范化处理（格式统一、术语标准化等）
    分解依据: 用户需要在分解前对需求进行规范化处理
    分配目标: 业务架构末级节点BA-规范化流程
    验收标准: |
      1. 规范化成功率 > 99%
      2. 规范化历史可追踪
    对上层的符合性: 满足SR-F-001中对需求规范化的要求
```

**表格格式（用于总览）**：

| 详细ID | 标题 | 描述 | 分配目标 | 验收标准 |
|--------|------|------|---------|---------|
| SR-F-001.1 | 需求收集 | 输入需求信息并创建新需求记录 | BA-需求收集流程 | 收集成功率>99% |
| SR-F-001.2 | 需求规范化 | 对已收集的需求进行规范化处理 | BA-规范化流程 | 规范化成功率>99% |
| SR-F-001.3 | 需求分解 | 对需求进行逻辑分解 | BA-分解流程 | 分解响应<2s |

## 四、非功能需求的详细定义规范

### 4.1 非功能需求的处理路线

**关键特点**：非功能需求不经过业务架构，直接标记为系统约束

```
相关方非功能需求（SR-NFR-001）
   ↓ [不经过业务架构]
详细定义末级节点 = 量化指标的明确定义
   ↓
系统约束（在系统需求层面标记）
```

**为什么不经过业务架构**：
- 非功能需求（如性能、安全、可用性）是系统的**质量属性**，不是业务流程的一部分
- 业务架构代表**用户视角**下的业务场景，而非功能需求是**系统级**的约束条件
- 因此非功能需求直接作为系统需求的约束，而非业务架构的输入

### 4.2 非功能需求详细定义末级节点的结构

非功能需求的详细定义 = **量化指标的完整定义**

每条非功能需求详细定义末级节点应包含：

| 字段 | 要求 | 说明 |
|------|------|------|
| **ID** | 唯一标识符，格式SR-NFR-[号] | 示例：SR-NFR-001, SR-NFR-002 |
| **需求类型** | 性能/安全/可用性/可扩展性等 | 明确非功能需求的分类 |
| **需求类别** | 质量特性/环境适应性/可实现性 | 三大类之一 |
| **需求描述** | 完整的需求叙述 | 说明系统应该满足什么质量特性 |
| **量化指标** | 具体的、可测量的指标 | 必须包含具体数值和单位 |
| **测量方法** | 如何测量/验证该指标 | 例如：性能测试、安全审计、可用性测试 |
| **应用范围** | 指标适用的场景/条件 | 例如：正常业务负载、高并发场景等 |
| **优先级** | P0/P1/P2/P3 | 需求的优先级 |
| **备注** | 任何附加说明 | 例如：依赖条件、约束、风险等 |

### 4.3 非功能需求详细定义的示例

#### 示例 1：性能需求

```
SR-NFR-001: 系统响应时间性能需求

需求类型：性能
需求类别：质量特性需求
描述：流水线在正常需求处理负载下应快速响应用户操作
量化指标：
  - 平均响应时间：< 200ms
  - P95响应时间：< 500ms
  - P99响应时间：< 1000ms
  - 在1000并发用户下：< 300ms
测量方法：使用JMeter进行性能测试，记录各百分位数的响应时间
应用范围：所有关键流程操作（需求收集、规范化、架构设计等）
优先级：P0（必须满足）
备注：该指标基于用户体验最佳实践，超过1000ms设计者会明显感受到延迟
```

#### 示例 2：安全需求

```
SR-NFR-002: 系统设计数据安全需求

需求类型：安全
需求类别：质量特性需求
描述：流水线应保护需求设计数据和架构信息的安全
量化指标：
  - 传输层：TLS 1.2+加密
  - 存储层：AES-256加密敏感数据（架构设计、性能指标等）
  - 认证：支持OAuth2.0、MFA
  - 审计：100%操作审计日志记录
测量方法：安全审计、渗透测试、代码审查
应用范围：所有涉及架构设计和需求信息的功能
优先级：P0（必须满足）
备注：符合企业数据安全管理规范
```

#### 示例 3：可用性需求

```
SR-NFR-003: 流水线可用性需求

需求类型：可用性
需求类别：质量特性需求
描述：流水线应在大部分时间内可用并快速恢复故障
量化指标：
  - 可用性：99.9%（全年停机时间< 8.76小时）
  - 故障恢复时间(RTO)：< 1小时
  - 数据恢复点(RPO)：< 15分钟
  - 故障检测时间：< 5分钟
测量方法：监控流水线运行时间、故障恢复记录、备份恢复演练
应用范围：所有生产环境
优先级：P0（必须满足）
备注：需要多机房部署和故障转移能力
```

### 4.4 非功能需求详细定义的分解标准

非功能需求通常不需要多层分解，但可能需要按**维度**细化：

**按需求类型细化**：
```
SR-NFR-001（性能需求）
   ↓ [按维度细化]
SR-NFR-001.1（响应时间）
SR-NFR-001.2（吞吐量）
SR-NFR-001.3（并发数）
```

**按应用范围细化**：
```
SR-NFR-002（安全需求）
   ↓ [按范围细化]
SR-NFR-002.1（传输层安全）
SR-NFR-002.2（存储层安全）
SR-NFR-002.3（应用层安全）
```

**细化标准**：
- ✅ 每个细化后的需求有**独立的、可测量的指标**
- ✅ 每个细化后的需求有**明确的验证方法**
- ✅ 所有细化的需求**合集 = 上层需求的完整定义**

## 五、详细定义与架构定义的协作

### 5.1 架构末级节点与详细定义末级节点的关系

```
一对多关系（1:N）：

架构末级节点（相关方需求条目）
       ↓ [分解为]
多条详细定义末级节点

例如：
SR-F-001（相关方需求规范化功能）
   ├─ SR-F-001.1（需求收集）
   ├─ SR-F-001.2（需求规范化）
   ├─ SR-F-001.3（需求分解）
   └─ SR-F-001.4（冲突检测）

所有详细定义末级节点合集 = 对架构末级节点的完整说明
```

### 5.2 映射标注标准

#### 双向标注

**在架构末级节点中标注**：
```
SR-F-001: 相关方需求规范化功能
├─ 定义：[需求定义]
├─ 承接的原始需求末级：[原始需求ID]
├─ 详细定义末级节点：
│  ├─ SR-F-001.1
│  ├─ SR-F-001.2
│  ├─ SR-F-001.3
│  └─ SR-F-001.4
└─ 分配到业务架构末级节点：BA-规范化管理流程
```

**在详细定义末级节点中标注**：
```
SR-F-001.1: 需求收集
├─ 描述：[具体描述]
├─ 上级架构末级节点：SR-F-001（相关方需求规范化功能）
├─ 分配到：BA-需求收集流程 [1:1映射]
└─ 符合性分析：[说明如何满足SR-F-001]
```

#### 符合性分析

每条详细定义末级节点都应包含**符合性分析**，说明：
1. 如何满足上级架构末级节点的定义
2. 如何满足原始需求
3. 与其他详细定义末级节点的关系

**示例**：
```
SR-F-001.1: 需求收集
符合性分析：
  - 满足SR-F-001中对"需求收集"功能的要求
  - 与SR-F-001.2（需求规范化）配合，支持完整的规范化流程
  - 是需求处理的第一个环节，后续环节依赖于此
  - 生成的需求ID为规范化、分解等后续操作的基础
```

## 六、详细定义的验收标准

### 6.1 分解充分性检查

验证详细定义是否能满足1:1映射约束：

- [ ] 每条详细定义末级节点都能映射到业务架构的**唯一一个**末级节点
- [ ] 如果某条详细定义末级节点需要映射到**多个**业务架构末级节点，则说明分解不充分，需要继续分解
- [ ] 所有详细定义末级节点的集合能够**完整覆盖**上级架构末级节点的定义
- [ ] 没有**冗余或重复**的详细定义末级节点

### 6.2 清晰性检查

验证详细定义是否清晰易懂：

- [ ] 每条详细定义末级节点的**标题**是否清晰简洁？
- [ ] 每条详细定义末级节点的**描述**是否避免了模糊词汇（"大约"、"尽可能"等）？
- [ ] 每条详细定义末级节点的**验收标准**是否具体和可测量？
  - 功能需求：验收标准能否通过测试验证？
  - 非功能需求：验收标准是否包含具体的数值、单位、方法？
- [ ] **术语**是否统一？不同的需求是否使用了相同的术语来表达相同的概念？

### 6.3 映射关系检查

验证映射关系是否正确：

- [ ] 每条详细定义末级节点都标注了**分配目标**（业务架构末级节点）
- [ ] 每条详细定义末级节点都说明了**如何满足上级架构末级节点**
- [ ] 所有映射关系是否满足**1:1约束**（一条详细定义 → 一个业务架构末级）
- [ ] 所有映射关系是否支持**N:1承接**（一个业务架构末级 ← 多条详细定义）
- [ ] 映射关系是否形成了**完整的追溯链**

### 6.4 完整性检查

验证详细定义是否完整：

- [ ] 功能需求和非功能需求是否都有详细定义？
- [ ] 是否有**遗漏**的需求（架构末级节点未被详细定义覆盖）？
- [ ] 是否有**孤立**的详细定义末级节点（没有分配目标）？
- [ ] 所有详细定义末级节点的**验收标准**是否都包含？
- [ ] 非功能需求的**量化指标**是否都包含？

### 6.5 检查清单

在完成详细定义后，使用此清单进行自检：

**分解标准检查**
- [ ] 每条功能需求详细定义末级节点只需分配到业务架构的一个IPO
- [ ] 每条非功能需求都有具体的量化指标
- [ ] 详细定义末级节点是原子性的、不可再分的

**清晰性检查**
- [ ] 所有需求表述避免模糊词汇
- [ ] 所有术语使用一致
- [ ] 所有验收标准具体、可测量
- [ ] 所有描述使用"系统应该..."或"用户能够..."的标准表述

**映射关系检查**
- [ ] 每条详细定义末级节点都标注了分配目标
- [ ] 所有映射关系符合1:1约束
- [ ] 映射关系支持N:1承接（无遗漏、无孤立）

**完整性检查**
- [ ] 没有遗漏任何架构末级节点的详细定义
- [ ] 功能和非功能需求都已覆盖
- [ ] 所有详细定义末级节点都有验收标准

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误 1：分解粒度不当**

```
错误示例：
SR-F-001: 相关方需求规范化功能
  ├─ SR-F-001.1: 需求规范化
  ├─ SR-F-001.2: 流程处理
  └─ SR-F-001.3: 系统管理

问题：分解标题与上层相同或更模糊，没有真正分解
```

```
正确做法：
SR-F-001: 相关方需求规范化功能
  ├─ SR-F-001.1: 需求收集
  ├─ SR-F-001.2: 需求规范化
  ├─ SR-F-001.3: 需求分解
  └─ SR-F-001.4: 冲突检测

说明：明确了规范化处理的四个具体环节，每个都能映射到独立的业务流程
```

---

**❌ 错误 2：多个详细定义映射到同一业务架构**

```
错误示例：
SR-F-001.1: 需求收集
  分配到：BA-需求处理流程

SR-F-001.2: 需求规范化
  分配到：BA-需求处理流程

SR-F-001.3: 需求分解
  分配到：BA-需求处理流程

问题：三个不同的功能映射到同一个业务流程，可能意味着分解不充分
       或者业务架构设计不合理
```

```
正确做法：
要么调整详细定义（合并相关项），要么调整业务架构（拆分成三个流程）

方案A（调整详细定义）：
SR-F-001.1: 需求生命周期管理
  分配到：BA-需求处理流程

方案B（调整业务架构）：
SR-F-001.1: 需求收集 → BA-需求收集流程
SR-F-001.2: 需求规范化 → BA-规范化流程
SR-F-001.3: 需求分解 → BA-分解流程
```

---

**❌ 错误 3：非功能需求缺少量化指标**

```
错误示例：
SR-NFR-001: 系统应该快速响应
量化指标：系统响应快

问题："快速"、"快"都是模糊的主观词汇，无法验证
```

```
正确做法：
SR-NFR-001: 系统响应时间性能需求
量化指标：
  - 平均响应时间 < 200ms
  - P99响应时间 < 1000ms
  - 在1000并发用户下 < 300ms

说明：具体的数字和单位，可以通过性能测试验证
```

---

### 7.2 最佳实践

**✅ 最佳实践 1：使用标准化的分解标准**

```
建议：对各类需求制定标准的分解粒度

示例：
相关方需求相关：分解为 收集/规范化/分解/冲突检测/重复检测
架构设计相关：分解为 定义/映射/验证/优化
流水线功能相关：分解为 需求输入/处理/输出/追溯

优点：
  - 保持分解的一致性
  - 减少重复设计
  - 降低理解成本
```

---

**✅ 最佳实践 2：建立详细定义-业务架构的双向追溯**

```
建议：在详细定义和业务架构之间建立双向链接

实现方式：
  1. 详细定义中标注分配目标（向下）
  2. 业务架构中标注承接的详细定义（向上）
  3. 使用超链接建立快速导航

优点：
  - 快速定位详细定义和业务架构的对应关系
  - 便于验证映射的完整性
  - 支持需求的端到端追溯
```

---

**✅ 最佳实践 3：非功能需求的量化指标要包含上下文**

```
建议：量化指标应该包含：数值 + 单位 + 测量条件 + 应用范围

示例（完整的量化指标）：
  ❌ 不完整：响应时间 < 2秒
  ✅ 完整：平均响应时间 < 200ms（在1000并发、正常数据量下）
          P99响应时间 < 1秒（在1000并发、最大数据量下）
          
  说明：
    - 数值：200ms、1秒
    - 单位：毫秒、秒
    - 测量条件：1000并发、正常/最大数据量
    - 应用范围：所有查询操作

优点：
  - 指标清晰具体，无歧义
  - 便于测试和验证
  - 避免因条件不清而产生争议
```

---

**✅ 最佳实践 4：详细定义与下层的同步设计**

> 本实践是 §1.6"详细定义与下层架构定义的同步设计"的具体应用。核心概念详见 §1.6。

```
建议：在进行详细定义的同时，同步考虑业务架构的设计

流程：
  1. 对架构末级节点进行初步的详细定义分解
  2. 同时对业务架构进行初步的设计（至少确定IPO的名称和范围）
  3. 验证是否每条详细定义末级都能1:1映射到一个IPO
  4. 如果不能，调整详细定义或业务架构
  5. 确认映射关系后，再进行详细的内容填充

优点：
  - 避免详细定义后发现无法映射的问题
  - 减少返工
  - 支持上下游团队的并行工作
```

## 八、常见问题

**Q1：详细定义可以有多少层？**

A：没有固定限制，但应遵循1:1映射约束。一般来说：
- 功能需求：1-3层（SR-F-001 → SR-F-001.1 → SR-F-001.1.1）
- 非功能需求：1-2层（SR-NFR-001 → SR-NFR-001.1）

关键是**每层都应该有明确的分解目的和分解依据**，而不是盲目分解。

---

**Q2：详细定义末级节点的数量有没有上限？**

A：没有硬性上限，但建议：
- 单个架构末级节点的详细定义末级数量：3-10个（过少说明分解不足，过多说明粒度不当）
- 如果某个架构末级节点有超过10条详细定义末级，考虑是否需要重新组织分类

---

**Q3：如果详细定义末级节点无法1:1映射到业务架构末级节点怎么办？**

A：有两种可能：
1. **详细定义分解不充分**：该条目需要继续分解，直到能1:1映射
2. **业务架构设计不合理**：业务架构中缺少对应的末级节点，需要补充

处理方法：
- 首先检查详细定义是否还能进一步分解
- 如果已经是原子级，则调整业务架构设计
- 在两者之间找到平衡点

---

**Q4：非功能需求一定要细化到具体数字吗？**

A：是的，**非功能需求的详细定义本质上就是具体化量化指标**。

如果无法提供具体数字，可能原因：
1. 不了解用户的真实需求 → 需要与相关方沟通
2. 技术上还未确定 → 可先提出预期，后续调整
3. 需求本身不合理 → 需要重新审视需求

但无论如何，**最终的详细定义中必须包含可测量的指标**，否则无法验证是否满足需求。

---

**Q5：详细定义和业务架构设计可以并行吗？**

A：可以，并且**推荐并行**。这样做的好处：
- 详细定义的设计可以及时得到业务架构设计的反馈
- 业务架构设计可以明确详细定义的分配目标
- 减少单向的设计，提高效率

具体做法：
1. 第一阶段：快速约定结构（详细定义的ID和标题，业务架构的IPO名称）
2. 第二阶段：并行填充内容（详细定义的描述和验收标准，业务架构的IPO定义）
3. 第三阶段：验证映射（确保每条详细定义都能1:1映射）

---

## 九、完整的两层设计流程

本规范完整覆盖相关方需求的两层设计过程：

### 9.1 第一层：架构定义设计流程

**目标**：建立相关方需求的分类体系，对原始需求进行分组和组织

**输入**：
- 原始需求末级清单（来自系统设计准则第1层）

**处理步骤**：
1. **分析原始需求的关联性**：找出相关、相似的原始需求
2. **按业务逻辑分组**：将相关需求分为若干组
3. **定义架构末级节点**：为每组定义一个相关方需求（SR-F-XXX或SR-NFR-XXX）
4. **明确映射关系**：在文档中标注N:1的映射关系

**输出**：
- 架构定义文档（树形结构 + 映射视图）
  - 功能需求架构末级节点清单
  - 非功能需求架构末级节点清单
  - 原始需求末级 ↔ 架构末级节点的映射表

**检查清单**：
- [ ] 所有原始需求末级都映射到某个架构末级节点
- [ ] 承接同一架构末级节点的原始需求有逻辑关联
- [ ] 每个架构末级节点的定义清晰、完整
- [ ] 功能需求和非功能需求已分离
- [ ] 双向追溯的映射关系正确

---

### 9.2 第二层：详细定义设计流程

**目标**：对每个架构末级节点进行详细设计分解，为业务架构设计提供输入

**输入**：
- 架构定义文档（特别是架构末级节点）

**处理步骤**：
1. **选择一个架构末级节点**：从功能需求或非功能需求中选择
2. **分解成详细定义末级**：
   - 功能需求：按流程、角色、条件等进一步分解
   - 非功能需求：按维度、范围等细化指标
3. **检查1:1映射约束**：每条详细定义末级是否只需一个下层节点承接
4. **标注分配目标**：明确每条详细定义末级分配到业务架构的哪个IPO
5. **补充验收标准**：为每条详细定义末级补充具体的验收标准

**输出**：
- 详细定义文档（对每个架构末级节点的分解）
  - 详细定义末级节点清单（SR-F-001.1, SR-F-001.1.1等）
  - 业务架构IPO的设计建议
  - 验收标准和符合性分析

**检查清单**：
- [ ] 分解是否满足1:1映射约束
- [ ] 所有详细定义末级都有清晰的验收标准
- [ ] 非功能需求都有具体的量化指标
- [ ] 映射关系是否正确
- [ ] 所有详细定义末级都能为业务架构提供清晰的设计指导

---

### 9.3 两层设计的迭代和同步

在实际设计中，两层设计通常是**迭代和同步**进行的：

```
第一轮：快速确定架构结构
  第1层：快速定义架构末级节点（可能还不完整）
  第2层：快速定义详细定义末级节点（确定粒度）
  → 目标：建立映射关系骨架

第二轮：详细填充内容
  第1层：补充架构末级节点的完整定义
  第2层：补充详细定义末级节点的完整说明
  → 目标：补充设计细节

第三轮：验证和调整
  检查：映射关系是否满足1:1约束
  调整：如果不满足，调整分解粒度或架构结构
  → 目标：确保设计的充分性
```

## 十、与其他准则的关系

本规范在系统设计流水线中的位置：

```
准则 1（需求规范化）
    ↓ [输出：原始需求末级清单]
★ 准则 2（相关方需求设计规范）本规范
    ├─ 第一层：架构定义
    │  ├─ 原始需求末级 → 相关方需求架构末级节点（N:1）
    │  └─ [输出：架构定义文档]
    │
    └─ 第二层：详细定义
       ├─ 相关方需求架构末级节点 → 详细定义末级节点（1:N）
       └─ [输出：详细定义文档]
            ↓
准则 3（SR→BA映射）
    ↓ [输入：相关方需求详细定义末级节点]
准则 4（BA→SysReq映射）
    ↓
... 后续流程
```

**与准则 1 的关系**：
- 准则 1 的输出（原始需求末级清单）是本规范的输入
- 准则 1 关注原始需求的"规范化和分解"
- 本规范关注相关方需求的"架构定义和详细定义"
- 两者共同确保需求从原始状态到架构化的完整转换

**与准则 3 和业务架构规范的关系**：
- 本规范的输出（详细定义末级节点）是准则 3 和业务架构规范的直接输入
- 准则 3 将详细定义末级节点（1:1）映射到业务架构末级节点（IPO）
- 业务架构规范（`04-business-architecture-specification.md`）提供业务架构设计的完整指导
- 本规范确保详细定义满足1:1映射约束，这是后续业务架构设计的前提条件

**本规范的完整性**：
