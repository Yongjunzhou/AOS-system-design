## 📋 背景信息：系统设计流水线的应用对象

本准则用于设计"产品开发之系统设计流水线"这个产品。该流水线是为企业运营体系（AOS）的开发者设计的，企业业务范围包括机载设备的市场、研发、生产、售后以及经营管理。

流水线的应用对象是企业运营体系的开发者，主要包括三类人员：

| 开发者类型 | 角色 | 关注点 |
|-----------|------|--------|
| **需求提出者** | 提出运营体系建设、优化或改进需求 | 需求的完整性、清晰性、可追溯性 |
| **业务流程开发者** | 设计和实现业务流程 | 流程的合理性、可执行性、效率 |
| **信息系统开发者** | 设计和实现信息系统 | 系统的可实现性、性能、可维护性 |

**注**：本准则主要针对这三类开发者。在应用准则设计流水线时，应充分考虑这三类开发者的不同需求和关注点。

## 🎯 核心概念（为什么这样做）

### 系统设计流水线的目标

建立从**原始需求**到**产品架构**的完整设计链路，确保：
- ✅ 每条需求都有明确的承接点（不遗漏）
- ✅ 每个方案都充分满足上层需求（不过度设计）
- ✅ 完整的端到端追溯链路（可追踪）
- ✅ 发现并消除设计缺陷（高质量）

### 架构与详细设计的分离

**核心理念**：每份技术文档都包含两个层级

| 层级 | 定义 | 作用 |
|------|------|------|
| **架构末级节点** | 被分类和组织的最大颗粒度的需求/方案条目 | 是需求映射的关键层级 |
| **详细设计末级节点** | 对架构末级节点的进一步分解 | 是1:1映射的源头 |

**分解标准**：一直分解到任意末级条目都只需找到**后续文档架构的一个末级节点**来承接即可

**这样做的好处**：
- 清晰地区分高层设计和具体实现
- 支持多个团队的并行工作
- 便于需求的追溯和映射
- 提高设计的可维护性

---

### 文档类型定义

系统设计流水线中的文档分为三大类：

| 文档类型 | 定义 | 包含内容 | 示例 |
|---------|------|---------|------|
| **需求文档** | 描述系统应该做什么，包含功能需求和非功能需求 | 需求描述、验收标准、性能指标 | 原始需求、相关方需求 |
| **方案（架构）文档** | 描述如何满足需求，包含架构末级节点和详细设计末级节点两个部分 | 架构设计、组件定义、映射关系 | 业务架构、系统需求、产品架构 |
| **详细设计文档** | 对方案的进一步细化，包含实现细节、接口定义、数据结构等 | 实现细节、代码框架、配置说明 | 代码设计、数据库设计、API文档 |

**关键区别**：
- **需求文档** = "做什么"（What）
- **方案文档** = "怎么做"（How）的高层设计
- **详细设计文档** = "怎么做"（How）的具体实现

---

### 方案（架构）文档的两部分结构

每份方案（架构）文档都包含两个部分：

#### 第一部分：架构设计结果（树形结构）

**结构**：从根节点直到末级节点的完整树形结构

**节点分类**：
- **分类和组织节点**：末级节点之上的所有节点，用于对架构进行分类和组织
- **架构末级节点**：被分类和组织的最大条目，是需求映射的关键层级

**每个架构末级节点包含**：
- 节点定义：该架构末级节点的具体定义
- 承接的需求末级节点：该架构末级节点承接了上层需求的哪些末级节点（N:1映射）
- 为下层提供的设计指导：对下层方案设计的指导

**作用**：
- 清晰地展示整体架构的分类和组织
- 支持需求的追溯和映射
- 便于团队分工和协作

**示例结构**：
```
业务架构（根节点）
├─ 业务域 1（分类节点）
│  ├─ 业务组件 1.1（架构末级节点）
│  │  ├─ 定义：[具体定义]
│  │  ├─ 承接的需求末级：SR-F-001, SR-F-002
│  │  └─ 设计指导：[对系统需求的指导]
│  └─ 业务组件 1.2（架构末级节点）
└─ 业务域 2（分类节点）
   └─ 业务组件 2.1（架构末级节点）
```

---

#### 第二部分：需求映射视图（映射关系表）

**内容**：所承接需求的所有末级节点，以及承接该需求末级节点的架构末级节点

**结构**：
```
需求末级节点 | 承接的架构末级节点 | 超链接
SR-001      | 业务组件 1.1      | [跳转到第一部分]
SR-002      | 业务组件 1.1      | [跳转到第一部分]
SR-003      | 业务组件 1.2      | [跳转到第一部分]
...
```

**作用**：
- 清晰地展示需求和方案的对应关系
- 支持从需求视角快速定位到架构设计
- 便于验证映射的完整性和正确性

**导航方式**：
- 单击此处的架构末级节点，则直接跳到第一部分的对应架构末级节点
- 支持双向追溯：需求 ↔ 架构

---

#### 两部分的关系

| 部分 | 视角 | 结构 | 用途 |
|------|------|------|------|
| **第一部分** | 架构视角 | 树形结构（根→末级） | 展示架构的分类和组织 |
| **第二部分** | 需求视角 | 映射关系表 | 展示需求和架构的对应关系 |

**映射流向**：
- 第一部分的架构末级节点 ← 承接上层需求末级节点（N:1）
- 第一部分的架构末级节点 → 分配到下层方案的架构末级节点（N:1）

## 📊 5层结构速查表

### 完整流程图

```
第1层：原始需求及分析
    ↓ N:1 分配
第2层：相关方需求
    ├─ 架构末级节点 ↔ 详细设计末级节点
    │
    ├─ 功能部分 ↓ N:1 分配
    │   第3层：业务架构
    │   ├─ 架构末级节点（IPO） ↔ 详细设计末级节点（I/P/O）
    │   │
    │   ├─ 去重后 ↓ N:1 分配
    │   │   第4层：系统需求
    │   │   ├─ 架构末级节点 ↔ 详细设计末级节点
    │   │   │
    │   │   └─ ↓ N:1 分配
    │   │       第5层：产品架构
    │   │       ├─ 架构末级节点 ↔ 详细设计末级节点
    │
    └─ 非功能部分 ↓ N:1 分配
        第4层：系统需求（非功能）
        └─ ↓ N:1 分配
            第5层：产品架构
```

### 层级对照表

| 层级 | 文档名称 | 架构末级节点 | 详细设计末级节点 | 承接关系 |
|------|---------|------------|----------------|---------|
| 1 | 原始需求及分析 | - | 末级需求 | - |
| 2 | 相关方需求 | 相关方需求条目 | 对条目的分解 | 承接原始需求末级（N:1） |
| 3 | 业务架构 | IPO（业务组件） | I、P、O详细说明 | 承接相关方需求详细设计末级（N:1） |
| 4 | 系统需求 | 系统需求条目 | 对条目的分解 | 承接业务架构详细设计末级（N:1） |
| 5 | 产品架构 | 产品组件 | 实现细节 | 承接系统需求详细设计末级（N:1） |

## 🎯 三条核心规则（怎么做）

### 规则1：1:1 分配约束

**定义**：需求中的每一条都应分配至方案中**唯一一个条目**

**应用范围**：
- 相关方需求详细设计末级 → 业务架构末级（1:1）
- 业务架构详细设计末级 → 系统需求架构末级（1:1）
- 系统需求详细设计末级 → 产品架构末级（1:1）

**判断标准**（如何检查是否满足）：
- ❌ 一个条目需要分配到下层方案的**多个**架构末级节点 → 说明这个条目还需要进一步分解
- ✅ 一个条目只需要分配到下层方案的**一个**架构末级节点 → 说明这个条目已经是详细设计末级了

**实际操作**：
1. 设计详细设计末级节点时，问自己："这个条目能映射到下层的一个方案条目吗？"
2. 如果答案是"不能，需要多个"，那就继续分解
3. 如果答案是"可以"，那就停止分解

---

### 规则2：N:1 承接支持

**定义**：方案中一个条目可以承接需求的**多个条目**

**应用范围**：
- 原始需求末级 → 相关方需求架构末级（N:1）
- 相关方需求详细设计末级 → 业务架构末级（N:1）
- 业务架构详细设计末级 → 系统需求架构末级（N:1）
- 系统需求详细设计末级 → 产品架构末级（N:1）

**合理范围**（质量指标）：
- 最少：1条需求（某些特殊需求可能独立承接）
- 最多：4-5条需求（超过5条说明方案条目定义过于宽泛）
- 平均：2-3条需求（表示设计的合理聚合）

**实际操作**：
1. 设计架构末级节点时，问自己："这个方案条目应该承接多少个上层需求？"
2. 如果承接的需求太少（1个），考虑是否可以合并
3. 如果承接的需求太多（>5个），考虑是否需要拆分

---

### 规则3：双向追溯

**定义**：支持从上层需求追溯到下层方案，也支持从下层方案反向追溯到上层需求

**实现方式**：
- 在每个方案条目中标注"承接了哪些上层需求"
- 在每条上层需求中标注"分配到哪个下层方案条目"
- 通过超链接实现快速导航

**关键原则**：
- ✅ 条目化需求分配到**下层方案的架构末级节点**（不是详细设计末级）
- ✅ 这样可以保持需求和方案的对应关系清晰
- ✅ 架构末级节点是需求映射的关键层级

**实际操作**：
1. 每个方案条目都要明确标注"承接的上层需求"
2. 每条上层需求都要明确标注"分配到的下层方案条目"
3. 使用超链接建立双向导航

## 📋 每层的具体要求

### 第1层：原始需求及分析

**这一层的目的**：收集和分解原始需求，为后续设计提供清晰的输入

**内容结构**：
```
原始需求 1
├─ 需求描述：[具体的需求定义]
├─ 末级需求分解：
│  ├─ 末级需求 1.1 → [分配到相关方需求架构末级]
│  ├─ 末级需求 1.2 → [分配到相关方需求架构末级]
│  └─ 末级需求 1.3 → [分配到相关方需求架构末级]
└─ 优先级：[高/中/低]
```

**关键要求**：
- ✅ 每条原始需求都应分解到末级需求
- ✅ 末级需求应该是原子性的、不可再分的
- ✅ 每条末级需求都应分配到相关方需求架构末级节点
- ✅ 文档中只保留纯需求清单，去除解释性内容

**质量指标**：
- 末级需求平均分解数：2-3条（表示中粒度分解）
- 映射覆盖率：100%（所有末级需求都被映射）

---

### 第2层：相关方需求

**这一层的目的**：将原始需求转化为相关方能理解的需求规范

**相关方需求的本质**：
- 是对原始需求的方案
- 包含两个层级：架构末级节点（分类）和详细设计末级节点（分解）
- 分解标准：直到每个末级条目只需映射到业务架构的一个IPO

**内容结构**：

**第一部分：架构设计结果（树形结构）**

从根节点直到末级节点的完整树形结构：
```
相关方需求（根节点）
├─ 功能需求类（分类节点）
│  ├─ SR-F-001: [相关方需求名称]（架构末级节点）
│  │  ├─ 定义：[需求的定义]
│  │  ├─ 承接的原始需求末级：原始需求 1.1, 1.2, 1.3
│  │  ├─ 符合性分析：[如何满足所承接的需求]
│  │  └─ 为下层提供的设计指导：[对业务架构的指导]
│  └─ SR-F-002: [相关方需求名称]（架构末级节点）
└─ 非功能需求类（分类节点）
   └─ SR-NFR-001: [相关方非功能需求]（架构末级节点）
```

**第二部分：需求映射视图（映射关系表）**

展示所有原始需求末级到相关方需求的对应关系：
```
原始需求末级 | 承接的相关方需求 | 超链接
原始需求 1.1  | SR-F-001        | [跳转到第一部分]
原始需求 1.2  | SR-F-001        | [跳转到第一部分]
原始需求 1.3  | SR-F-002        | [跳转到第一部分]
```

**关键要求**：
- ✅ 功能需求和非功能需求分别标识
- ✅ 架构末级节点是最大颗粒度的相关方需求条目
- ✅ 建立双向追溯的超链接（点击映射视图中的相关方需求，直接跳到第一部分对应节点）

**质量指标**：
- 相关方需求架构末级节点数：通常 10-20 个
- 平均每个架构末级节点承接原始需求末级数：2-3 条
- 相关方需求详细设计末级节点数：通常 20-40 个
- 平均每个详细设计末级节点分配到业务架构末级数：1 个（1:1 约束）
- 映射覆盖率：100%（所有原始需求末级都被映射）

---

### 第3层：业务架构

**这一层的目的**：定义用户使用产品开展业务的过程

**业务架构的本质**：
- 是**运行概念**（Operational Concept）
- 是用户使用待开发产品开展业务的过程说明
- 包含两个层级：架构末级节点（IPO）和详细设计末级节点（I/P/O详细说明）

**IPO模型（Input-Process-Output）**：

业务架构的架构末级节点应该是一个IPO模型：
- **Input（输入）**：产品和/或系统接收的输入
- **Process（处理）**：对输入进行的处理
- **Output（输出）**：生成的输出

每个IPO对应一个系统功能。

**IPO的颗粒度**：
- IPO的体量主要取决于所期望输出物的颗粒度
- 例如：对于系统设计流水线产品，期望的输出是各技术文档
- 那么一个IPO应该对应一份技术文档（或文档的一个可以由单人完成的独立部分）

**内容结构**：

**第一部分：业务架构末级节点（IPO清单）**
```
BA-001: [IPO名称/业务组件名称]
├─ 定义：[业务组件的定义]
├─ 承接的相关方需求详细设计末级：
│  ├─ SR-F-001.1
│  ├─ SR-F-001.2
│  └─ SR-F-002.1
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对系统需求的指导]
```

**第二部分：业务架构详细设计末级节点（I、P、O详细说明）**
```
BA-001-Input: [输入的具体形式、数据结构、来源]
BA-001-Process: [处理的规则、算法、步骤、约束条件]
BA-001-Output: [输出的具体形式、数据结构、验收标准]
```

**关键要求**：
- ✅ 业务架构末级节点是最大颗粒度的IPO
- ✅ 每个IPO都应该有清晰的Input、Process、Output定义
- ✅ 建立与相关方需求的映射关系（N:1支持）
- ✅ 为系统需求提供清晰的设计指导

**IPO的去重机制**（重要！）：

在业务架构设计时，当在某个分支需要一个IPO时，应该：
1. **首先检查** - 现有业务架构中是否已有相同或相似的IPO
2. **复用优先** - 如果存在，直接复用该IPO
3. **改进次之** - 如果现有IPO改进后可用，则改进后复用
4. **新增最后** - 确实需要新的IPO才新增

**质量指标**：
- 业务架构末级节点数（IPO数）：通常 10-20 个
- 平均每个IPO承接相关方需求详细设计末级数：2-3 条（N:1支持）
- 1:1 分配约束满足率：100%（每条相关方需求详细设计末级分配到唯一的IPO）
- IPO详细程度：能指导系统需求定义，能与用户沟通

### 第4层：系统需求

**这一层的目的**：将业务架构转化为系统可以实现的需求规范

**系统需求的本质**：
- 是业务架构和相关方需求非功能部分的方案
- 包含两个层级：架构末级节点（分类）和详细设计末级节点（分解）
- 分解标准：直到每个末级条目只需映射到产品架构的一个组件

**系统需求的来源**（重要！）：

1. **功能需求来源** - 业务架构中所有IPO的去重清单
   - 多个业务流程中重复的IPO，在系统需求中只需定义一次
   - 这确保了系统功能的完整性和无重复性

2. **非功能需求来源** - 相关方需求中的非功能部分
   - 直接承接相关方需求的非功能需求

**内容结构**：

**第一部分：系统需求架构末级节点**
```
SysReq-001: [系统需求名称]
├─ 定义：[系统需求的定义]
├─ 类型：[功能需求/非功能需求]
├─ 承接的业务架构/相关方需求：
│  ├─ BA-001
│  ├─ BA-002
│  └─ SR-NF-001
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对产品架构的指导]
```

**第二部分：系统需求详细设计末级节点**
```
SysReq-001.1: [具体的系统需求]
├─ 描述：[详细说明]
├─ 验收标准：[验收条件]
├─ 分配到：[产品架构末级节点]
└─ 符合性说明：[如何满足系统需求]
```

**关键要求**：
- ✅ 系统需求架构末级节点是最大颗粒度的系统需求条目
- ✅ 功能需求和非功能需求分别处理
- ✅ 功能需求来自业务架构IPO的去重清单
- ✅ 每条系统需求都应该有清晰的定义和验收标准
- ✅ 建立与上层需求的映射关系（N:1支持）

**质量指标**：
- 系统需求架构末级节点数：通常 10-20 个
- 平均每个架构末级节点承接业务架构/相关方需求数：2-3 条
- 系统需求详细设计末级节点数：通常 20-40 个
- 平均每个详细设计末级节点分配到产品架构末级数：1 个（1:1 约束）
- 1:1 分配约束满足率：100%（每条业务架构/相关方需求分配到唯一的系统需求）
- 功能需求去重率：100%（没有重复的功能定义）

### 第5层：产品架构

**这一层的目的**：定义产品的具体实现方案

**产品架构的本质**：
- 是系统需求的方案
- 包含两个层级：架构末级节点（组件）和详细设计末级节点（实现细节）

**内容结构**：

**第一部分：产品架构末级节点**
```
PA-001: [产品组件名称]
├─ 定义：[产品组件的定义]
├─ 承接的系统需求详细设计末级：
│  ├─ SysReq-001.1
│  ├─ SysReq-001.2
│  └─ SysReq-002.1
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现细节：[具体的技术实现方案]
```

**第二部分：产品架构详细设计末级节点**
```
PA-001-Detail: [实现细节、性能指标、技术栈等]
```

**关键要求**：
- ✅ 产品架构末级节点是最大颗粒度的产品组件
- ✅ 每个产品组件都应该有清晰的实现方案
- ✅ 包含具体的技术细节和性能指标
- ✅ 建立与系统需求的映射关系（N:1支持）

**质量指标**：
- 产品架构末级节点数：通常 15-30 个
- 平均每个产品组件承接系统需求详细设计末级数：1-2 条
- 1:1 分配约束满足率：100%（每条系统需求详细设计末级分配到唯一的产品组件）

## ✅ 操作检查清单

### 分解充分性检查清单

在设计每一层时，检查分解是否充分：

- [ ] 每条上层需求都分解到末级需求
- [ ] 末级需求是原子性的、不可再分的
- [ ] 末级需求的粒度合理（平均分解数 2-3 条）
- [ ] 每条末级需求都有清晰的定义和验收标准
- [ ] 每条末级需求都分配到下层方案的一个架构末级节点
- [ ] 没有遗漏或重复的映射

### 分解标准检查清单

检查分解是否符合标准（只需一个下层方案架构末级节点承接）：

- [ ] 相关方需求详细设计末级节点只需分配到业务架构的一个末级节点
- [ ] 业务架构详细设计末级节点只需分配到系统需求的一个末级节点
- [ ] 系统需求详细设计末级节点只需分配到产品架构的一个末级节点
- [ ] 如果某个条目需要分配到多个下层架构末级节点，说明还需要进一步分解

### 映射关系检查清单

检查映射关系是否满足1:1约束和N:1支持：

- [ ] 每条上层需求都分配到下层方案的唯一一个条目（1:1 约束）
- [ ] 每个下层方案条目都承接了上层的多条需求（N:1 支持）
- [ ] 承接的需求数量合理（通常 2-4 条）
- [ ] 承接的需求之间有逻辑关联
- [ ] 映射关系符合业务逻辑

### 符合性分析检查清单

检查每个方案条目是否充分说明了如何满足上层需求：

- [ ] 每个方案条目都包含符合性分析
- [ ] 符合性分析清晰、具体、可验证
- [ ] 符合性分析说明了如何满足所承接的需求
- [ ] 符合性分析没有遗漏任何承接的需求
- [ ] 符合性分析的逻辑清晰、易于理解

### 双向追溯检查清单

检查追溯链路是否完整和有效：

- [ ] 每条上层需求都标注了分配到的下层方案条目
- [ ] 每个下层方案条目都标注了承接的上层需求
- [ ] 所有超链接格式正确
- [ ] 所有超链接都指向有效的锚点
- [ ] 所有超链接都指向正确的目标
- [ ] 用户能够快速导航和追溯

### 文档规范化检查清单

检查文档是否符合规范：

- [ ] 文档结构清晰、一致
- [ ] 每层报告都包含架构末级节点和详细设计末级节点
- [ ] 文档中只保留必要的内容，去除冗余信息
- [ ] 文档易于维护和更新
- [ ] 文档版本号正确更新
