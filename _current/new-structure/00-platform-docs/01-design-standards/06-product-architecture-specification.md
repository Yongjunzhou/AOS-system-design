## 一、核心概念

### 1.1 产品架构的定义

**产品架构（Product Architecture）** 是将系统需求转化为具体技术实现方案的设计结果。它回答的问题是："系统应该如何被构建？组件之间如何交互？"

产品架构在系统设计流水线中扮演着**方案文档**的角色：

| 角色 | 说明 |
|------|------|
| **是系统需求（第4层）的方案** | 将系统需求的功能和非功能要求转化为技术方案 |
| **是产品研发的蓝图** | 为开发团队提供明确的技术实现指导 |

### 1.2 产品架构的组成

产品架构通常由以下部分组成：

| 组成 | 说明 | 示例 |
|------|------|------|
| **前端组件** | 用户界面和交互逻辑 | Web页面、移动端界面 |
| **后端组件** | 业务逻辑和数据处理 | API服务、业务引擎 |
| **数据层** | 数据存储和管理 | 数据库、缓存、文件存储 |
| **基础设施** | 支撑系统运行的基础能力 | 网关、认证、监控、消息队列 |

### 1.3 产品架构的地位

产品架构是系统设计流水线的**第5层**（最后一层），位于系统需求之上：

| 层级 | 文档类型 | 角色 |
|------|---------|------|
| 第3层：业务架构 | 方案文档（仅架构定义） | 用户使用产品的过程 |
| 第4层：系统需求 | 方案文档（架构定义 + 详细定义） | 系统实现的要求 |
| **第5层：产品架构** | **方案文档（架构定义 + 详细定义）** | **产品的技术方案** |

### 1.4 产品架构的双重结构

产品架构作为方案文档，包含两个子文档：

| 子文档 | 定义 | 包含内容 |
|-------|------|---------|
| **架构定义文档** | 以产品组件为基本单位，构建产品的技术架构并建立对系统需求的映射 | 组件清单、组件定义、组件关系、映射关系 |
| **详细定义文档** | 对每个产品组件进行详细的实现方案描述 | 实现细节、技术栈、接口定义、性能指标 |

### 1.5 产品架构层的分析

产品架构层通过两类符合性分析来验证设计的完整性和正确性：

**需求说明符合性分析**
- **分析对象**：产品架构组件与所承接的系统需求详细定义末级（SysReq 9级）之间的映射关系
- **分析手段**：
  - 覆盖度检查 — 所有 SysReq 9 级节点是否都有对应的 PA 组件承接
  - 1:1 分配验证 — 每个 SysReq 9 级节点是否分配到唯一的 PA 组件
  - N:1 承接合理性 — 每个 PA 组件承接的 SysReq 9 级节点数量是否合理（2-5 个为宜）
- **目的**：确保产品架构的组件定义完整覆盖了所有系统需求，映射关系准确无遗漏

**需求性能符合性分析**
- **分析对象**：产品架构组件的性能指标与所承接系统需求性能约束之间的关系
- **分析手段**：
  - 性能继承验证 — PA 组件的性能指标是否不低于所承接 SysReq 和 NFR 的要求
  - 可实现性评估 — 结合技术栈和架构方案评估性能指标的可实现性
  - 权衡决策记录 — 当多个 NFR 冲突时，权衡决策是否已记录
- **目的**：确保性能指标在设计链路末端得到落实，产品架构具备满足非功能约束的能力

## 二、架构定义规范

### 2.1 架构定义的本质

**产品架构定义**是以产品组件为基本单位，构建产品的技术架构并建立对系统需求的映射。

架构定义回答的问题是："产品由哪些组件构成？每个组件承担什么职责？组件之间如何交互？"

### 2.2 架构末级节点

产品架构的末级节点是**产品组件**，每个组件应有清晰的职责边界和接口定义。

每个产品架构末级节点包含：

| 属性 | 说明 | 示例 |
|------|------|------|
| **组件ID** | 唯一标识 | PA-001 |
| **组件名称** | 组件名称 | 用户认证服务 |
| **组件类型** | 前端/后端/数据/基础设施 | 后端 |
| **定义** | 组件的职责和功能描述 | 提供用户身份认证和授权管理 |
| **承接的系统需求** | 承接了哪些SysReq详细定义末级（9级） | SysReq-F-001.1.1.1, SysReq-NFR-003 |
| **符合性分析** | 如何满足所承接的系统需求 | 通过OAuth 2.0实现安全认证... |
| **实现细节** | 具体的技术实现方案 | 技术栈、关键模块、部署方式 |
| **性能指标** | 该组件需要满足的性能目标 | 响应时间 < 50ms, 可用性 > 99.9% |

### 2.3 架构定义的层级

产品架构的层级结构相对灵活，一般遵循以下模式：

| 层级 | 名称 | 说明 |
|------|------|------|
| **0级** | 产品根节点 | 整个产品的顶层 |
| **1级** | 架构分层 | 按架构层次划分（前端/后端/基础设施） |
| **2级** | 子系统/模块 | 按功能领域划分的子系统 |
| **3级** | **组件（承接层）** | **★ 末级，承接SysReq 9级节点** |

**注意**：产品架构的层级数不是固定的，可以根据产品复杂度适当调整。

### 2.4 架构定义的两部分结构

#### 第一部分：架构设计结果（树形结构）

```
产品架构（0级）
 ├─ 前端层（1级）
 │   ├─ 用户管理子系统（2级）
 │   │   └─ 用户登录页面（3级·组件/末级）
 │   │       ├─ 定义：提供用户登录界面
 │   │       ├─ 承接的SysReq：SysReq-F-001.1.1.1
 │   │       ├─ 符合性分析：... 
 │   │       └─ 实现细节：React + Ant Design
 │   └─ 订单管理子系统（2级）
 │       └─ 订单列表页面（3级·组件/末级）
 ├─ 后端层（1级）
 │   ├─ 用户服务（2级）
 │   │   └─ 用户认证API（3级·组件/末级）
 │   └─ 订单服务（2级）
 │       └─ 订单管理API（3级·组件/末级）
 └─ 基础设施层（1级）
     ├─ 数据存储（2级）
     └─ API网关（2级）
         └─ 网关服务（3级·组件/末级）
```

#### 第二部分：需求映射视图（映射关系表）

```
SysReq 9级节点 | 承接的PA组件 | 超链接
SysReq-F-001.1.1.1 | PA-101 | [跳转到第一部分]
SysReq-F-001.1.1.2 | PA-101 | [跳转到第一部分]
SysReq-F-002.1.1.1 | PA-102 | [跳转到第一部分]
```

### 2.5 组件间关系定义

除了每个组件本身的定义外，产品架构定义还应描述组件之间的关系：

| 关系类型 | 说明 | 表示方式 |
|---------|------|---------|
| **依赖** | 组件A依赖组件B | PA-001 → PA-002 |
| **调用** | 组件A调用组件B的接口 | PA-001 → API(PA-002) |
| **组合** | 组件A由组件B和C组成 | PA-001 = {PA-002, PA-003} |
| **关联** | 组件A与组件B存在逻辑关联 | PA-001 ↔ PA-002 |

## 三、详细定义规范

### 3.1 详细定义的本质

**产品架构详细定义**是对每个产品组件进行深入的实现方案描述，使开发团队能够直接进入编码阶段。

详细定义回答的问题是："每个组件具体如何实现？使用什么技术栈？性能目标是多少？"

### 3.2 详细定义的内容

每个产品组件的详细定义应包含：

| 内容 | 说明 | 示例 |
|------|------|------|
| **技术栈** | 使用的编程语言、框架、工具 | Python + FastAPI + PostgreSQL |
| **关键模块** | 组件的内部模块划分 | 输入验证模块、业务处理模块 |
| **接口定义** | 对外提供的API接口 | GET /api/users, POST /api/users |
| **数据模型** | 涉及的数据结构和存储 | User {id, name, email, role} |
| **部署方式** | 如何部署和运行 | Docker容器化，Kubernetes编排 |
| **性能指标** | 具体的性能目标 | 响应时间 < 200ms，吞吐量 > 1000qps |
| **安全要求** | 安全相关的要求 | 接口认证、数据加密 |
| **监控告警** | 运维监控指标 | CPU > 80%告警、错误率 > 1%告警 |

### 3.3 详细定义的格式

```yaml
PA-001: 用户认证服务
├─ 技术栈
│   ├─ 语言：Go 1.21
│   ├─ 框架：Gin
│   └─ 数据库：Redis + PostgreSQL
├─ 关键模块
│   ├─ 用户注册模块：处理用户注册逻辑
│   ├─ 用户登录模块：处理用户登录和Token生成
│   └─ 权限验证模块：处理接口访问权限检查
├─ 接口定义
│   ├─ POST /api/v1/login：用户登录
│   ├─ POST /api/v1/register：用户注册
│   └─ POST /api/v1/verify：验证Token
├─ 数据模型
│   └─ User { id, username, password_hash, role, created_at }
├─ 部署方式
│   ├─ 容器化：Docker
│   └─ 编排：Kubernetes（3副本）
├─ 性能指标
│   ├─ 响应时间：< 50ms（P99）
│   └─ 吞吐量：> 5000 请求/秒
├─ 安全要求
│   ├─ 密码加密：bcrypt
│   ├─ Token：JWT（过期时间24h）
│   └─ 传输加密：TLS 1.3
└─ 监控告警
    ├─ 错误率 > 1% → P1告警
    └─ 响应时间 > 200ms → P2告警
```

## 四、产品架构的组成

### 4.1 组件分类

产品组件按其在系统中的角色可分为以下几类：

#### 前端组件

| 子类型 | 说明 | 技术示例 |
|--------|------|---------|
| **页面组件** | 完整的页面或视图 | 首页、详情页、管理后台 |
| **功能组件** | 可复用的功能模块 | 搜索组件、表格组件、图表组件 |
| **交互组件** | 用户交互元素 | 按钮、表单、对话框 |

#### 后端组件

| 子类型 | 说明 | 技术示例 |
|--------|------|---------|
| **API服务** | 提供RESTful/gRPC接口 | 用户服务、订单服务 |
| **业务引擎** | 核心业务逻辑处理 | 流程引擎、规则引擎 |
| **数据处理** | 数据加工和处理 | ETL管道、批量处理服务 |
| **消息处理** | 异步消息处理 | 消息消费者、事件处理器 |

#### 数据层

| 子类型 | 说明 | 技术示例 |
|--------|------|---------|
| **关系型数据库** | 结构化数据存储 | PostgreSQL、MySQL |
| **NoSQL数据库** | 非结构化数据存储 | MongoDB、Redis |
| **文件存储** | 文件和数据对象存储 | MinIO、S3 |
| **消息队列** | 异步消息通信 | Kafka、RabbitMQ |

#### 基础设施

| 子类型 | 说明 | 技术示例 |
|--------|------|---------|
| **API网关** | 请求路由和治理 | Nginx、Kong |
| **认证授权** | 身份认证和权限控制 | OAuth 2.0、RBAC |
| **监控告警** | 系统监控和告警 | Prometheus、Grafana |
| **日志管理** | 日志收集和分析 | ELK Stack |
| **CI/CD** | 持续集成和交付 | Jenkins、GitLab CI |

### 4.2 组件划分原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个组件应该有明确的职责边界 |
| **高内聚** | 组件内部的元素应该紧密相关 |
| **低耦合** | 组件之间的依赖应该最小化 |
| **可替换** | 组件应该可以被替换而不影响其他组件 |
| **可测试** | 组件应该易于独立测试 |

## 五、设计原则

### 5.1 N:1承接支持原则

**定义**：一个产品架构组件可以承接多个SysReq 9级节点

**检查标准**：
- ✅ 承接2-5个SysReq 9级节点：合理
- ⚠ 承接1个：检查是否可以合并
- ⚠ 承接6+个：考虑是否需要拆分

### 5.2 1:1分配约束原则

**定义**：每个SysReq 9级节点分配到产品架构中**唯一**的组件

**应用范围**：
- SysReq 9级（场景活动）→ PA末级组件（1:1）

**判断标准**：
- 一个9级节点如果需要在多个组件中实现 → 继续分解
- 一个9级节点只需在一个组件中实现 → 停止分解

### 5.3 分层架构原则

**定义**：产品架构应按照清晰的分层结构组织

**推荐分层**：
- **展示层**（前端）：用户界面和交互逻辑
- **业务层**（后端）：业务逻辑和数据处理
- **数据层**：数据存储和管理
- **基础设施层**：跨切面的基础能力

**分层的好处**：
- 职责清晰：每层聚焦于特定职责
- 独立演化：每层可以独立变化
- 便于复用：同层组件可以互相调用

### 5.4 非功能约束满足原则

**定义**：产品架构设计必须满足系统需求中定义的非功能约束

**处理方式**：
- 每个PA组件标注受哪些NFR约束
- 组件设计体现NFR满足方案（如缓存满足性能、冗余满足可靠性）
- 在组件详细定义中列出具体的性能指标目标

### 5.5 同步设计原则

**定义**：产品架构的初步结构应与系统需求详细定义同步开展

**实际操作**：
1. SysReq 5级节点确定后，快速建立PA结构骨架（占位符）
2. SysReq 9级节点确定时，同步映射到PA组件
3. 冻结映射关系后，并行填充详细内容

## 六、设计方法

### 6.1 整体设计流程

产品架构的设计遵循以下流程：

#### 第1步：分析系统需求

- 收集所有SysReq 9级节点（场景活动）
- 收集SysReq-NFR节点（非功能约束）
- 识别需求之间的关联关系

#### 第2步：识别组件候选

- 按功能相关性分组SysReq 9级节点
- 识别可能的前端页面、后端服务、数据存储
- 评估组件的粒度和边界

#### 第3步：建立架构骨架

- 确定前端/后端/基础设施的分层结构
- 创建组件的占位符
- 建立SysReq → PA的初步映射

#### 第4步：详细组件设计

- 定义每个组件的职责和边界
- 定义组件间的接口和关系
- 确定技术栈和实现方案
- 定义性能指标和安全要求

#### 第5步：非功能约束验证

- 检查每个NFR是否有对应的组件设计体现
- 验证组件的性能指标是否满足NFR要求
- 记录NFR权衡决策

#### 第6步：映射验证

- 验证SysReq → PA映射的完整性（100%）
- 验证1:1分配约束
- 生成映射验证报告

### 6.2 两阶段工作流程

```
第一阶段：骨架建立（快速，1-2小时）
  1. 扫描SysReq 9级节点
  2. 创建PA组件占位符
  3. 建立SysReq → PA的骨架映射
  4. 冻结骨架结构

第二阶段：详细填充（深入，2-4天）
  1. 详细定义每个组件的实现方案
  2. 填充技术栈、接口定义、性能指标
  3. 验证非功能约束的可满足性
  4. 验证映射的完整性和一致性
  5. 冻结产品架构文档
```

### 6.3 非功能权衡决策

当多个非功能需求之间存在冲突时（如性能 vs 安全性），需要进行权衡决策：

| 冲突场景 | 权衡决策 | 记录方式 |
|---------|---------|---------|
| 性能 vs 安全 | 加密运算增加延迟 | 记录性能基准和安全要求 |
| 可用性 vs 成本 | 多副本部署增加成本 | 记录可用性目标和预算约束 |
| 可维护性 vs 性能 | 抽象层影响性能 | 记录架构决策记录(ADR) |

## 七、验收标准

### 7.1 架构定义验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **SysReq映射覆盖率** | 100% | 所有SysReq 9级节点都映射到PA组件 |
| **组件定义清晰** | 100% | 每个组件都有明确的职责和边界定义 |
| **组件分类合理** | 100% | 前端/后端/基础设施分类清晰 |
| **组件间关系明确** | 100% | 组件间的依赖和接口关系已定义 |

### 7.2 详细定义验收

| 检查项 | 标准 | 说明 |
|--------|------|------|

## 九、常见错误和最佳实践

### 常见错误

**错误1：组件粒度过粗或过细**
- ❌ 错误：PA-001: "整个后端系统"（粒度过粗）
- ❌ 错误：PA-001: "用户名的显示格式"（粒度过细）
- ✅ 正确：PA-001: "用户认证服务"（粒度适中）
- **影响**：粒度过粗导致职责不清，粒度过细导致管理复杂度增加

**错误2：忽略非功能约束**
- ❌ 错误：只关注功能实现，未定义性能指标
- ✅ 正确：每个组件都标注性能指标和安全要求
- **影响**：系统上线后发现性能不达标，需要返工

**错误3：组件间关系不清晰**
- ❌ 错误：组件定义中未说明依赖哪些其他组件
- ✅ 正确：每个组件明确标注依赖关系和接口调用
- **影响**：集成测试时发现接口不匹配，需要大量调整

**错误4：映射关系不完整**
- ❌ 错误：部分SysReq 9级节点未映射到PA组件
- ✅ 正确：所有SysReq 9级节点都有PA组件承接
- **影响**：部分功能在开发中被遗漏

**错误5：技术栈选择未说明理由**
- ❌ 错误：直接列出技术栈名称，不说明选择理由
- ✅ 正确：说明选择该技术栈的理由和备选方案
- **影响**：后期难以评估技术栈变更的影响

### 最佳实践

**实践1：先骨架后填充**
- 先快速建立架构骨架（占位符）
- 同步建立映射关系
- 再并行填充详细内容

**实践2：组件以"服务"为粒度**
- 每个组件应该是可独立部署和运行的"服务"
- 组件的粒度以"一个团队可以在1-2周内完成开发"为标准
- 避免组件粒度过大或过小

**实践3：将NFR纳入组件定义**
- 每个组件定义中都包含性能指标
- 性能指标应该可测量、可验证
- 在组件详细定义中说明NFR的满足方案

**实践4：记录架构决策**
- 关键的技术选择记录决策理由
- NFR权衡决策记录在案
- 使用架构决策记录（ADR）格式

**实践5：保持架构文档与代码一致**
- 架构文档随代码迭代同步更新
- 重大架构变更前先更新架构文档
- 使用工具自动验证架构与实际的一致性
