## 一、核心概念

### 1.1 原始需求的定义

**原始需求**（Original Requirements）是系统设计流水线的起点，是未经规范化的、来自各相关方的原始表述。它可能是会议纪要中的一句话、邮件中的一段描述、或需求文档中的一条条目。

**核心特征**：
- **来源多样**：来自不同相关方、不同渠道
- **格式不一**：自然语言表述，风格和粒度参差不齐
- **质量不均**：可能包含模糊、矛盾、重复等质量问题
- **缺乏分类**：功能和性能常常混在一起表述

**原始需求的本质**：
原始需求是"原材料"，不是"成品"。它的作用是作为系统设计的输入，经过规范化、分解、分类等加工步骤后，转化为结构化的相关方需求。

### 1.2 在系统设计流水线中的位置

原始需求是系统设计流水线的**第1层**，位于整个设计链路的最前端：

```
第1层：原始需求及分析
   ↓ 规范化、分解、分类
第2层：相关方需求
   ├─ 功能部分 → 第3层：业务架构
   └─ 非功能部分 → 第4层：系统需求（非功能分支）
```

**地位和角色**：

| 维度 | 说明 |
|------|------|
| **在5层结构中的位置** | 第1层 |
| **文档类型** | 需求文档（描述系统应该做什么） |
| **主要作用** | 为后续所有设计层提供输入 |
| **承接关系** | 承接原始输入（未经规范化的表述） |
| **分配方向** | 分配到第2层相关方需求架构末级节点（N:1） |
| **是否需要详细定义** | 需要——原始需求详细定义（末级需求分解） |

**关键特点**：
- 原始需求是唯一不遵循"需求—方案配对"模型的层——它只有需求，没有方案
- 原始需求详细定义中的末级需求是分配的源头，分配到相关方需求架构末级节点

### 1.3 原始需求与相关方需求的关系

原始需求和相关方需求是两个不同的概念，不能混淆：

| 维度 | 原始需求 | 相关方需求 |
|------|---------|-----------|
| **性质** | 原始表述，规范化前 | 规范化后的需求 |
| **格式** | 自由格式，风格不一 | 标准格式，统一规范 |
| **粒度** | 粗细不一 | 分解到末级（原子性） |
| **分类** | 功能/非功能混在一起 | 明确分离功能和性能 |
| **质量** | 可能有模糊、冲突、重复 | 经过检查和清理 |
| **结构** | 平铺列表 | 架构树形结构 |

### 1.4 原始需求详细定义的双重角色

原始需求详细定义在流水线中扮演着与上层不同的角色：

- **本层身份**：对原始需求的细化和分解（向内）
- **下层身份**：作为相关方需求架构定义的输入（向下传递）

```
原始输入（邮件、会议纪要、文档...）
   ↓ 收集、整理
原始需求清单
   ↓ 规范化、分解
原始需求详细定义（末级需求）
   ↓ N:1 分配
相关方需求架构末级节点
```

**理解要点**：
- 原始需求详细定义中的每条末级需求都是原子性的、不可再分的
- 每条末级需求只需分配到相关方需求架构的一个末级节点
- 原始需求本身不需要"方案"——它是整个设计链路的起点

---

### 1.5 原始需求层的分析

按系统设计准则的定义，分析（Analysis）是验证分解充分性和分配正确性的关键活动，涵盖**需求说明符合性分析**和**需求性能符合性分析**两个维度。

在原始需求层，分析活动体现在以下环节：

| 分析类型 | 体现环节 | 手段 |
|---------|---------|------|
| **需求说明符合性分析** | §3.2 四项基本检查（完整性、清晰性、可测试性、独立性） | 人工审查、AI 辅助检查 |
| | §4.4 分解充分性判断 | 1:1 映射判断法 |
| | §五 冲突和重复检测 | 语义对比、业务规则检查 |
| **需求性能符合性分析** | §3.3 功能/非功能分离中的性能指标提取 | 指标量化、建模仿真、理论计算 |
| | 规范化输出中的验收标准定义 | 指标可测性验证 |

**理解要点**：
- 原始需求层的分析不是在分解分配完成之后才进行，而是**贯穿整个处理过程**
- 每完成一个环节（收集→规范化→分解→冲突检测），即对前序环节的结果进行分析验证
- 分析的结果直接影响原始需求的质量，进而影响下游所有设计的质量

## 二、原始需求的收集和格式规范

### 2.1 收集渠道

原始需求可以来自多种渠道：

| 渠道 | 说明 | 注意事项 |
|------|------|---------|
| **需求会议** | 会议纪要、讨论记录 | 确认参会方和记录准确性 |
| **邮件沟通** | 相关方的邮件需求 | 确认邮件中的需求是否代表最终意见 |
| **需求文档** | 已有的需求规格说明书 | 确认文档版本和时效性 |
| **用户反馈** | 用户投诉、建议、改进意见 | 确认反馈的普遍性和优先级 |
| **竞品分析** | 竞品功能对比和分析 | 确认竞品功能是否真正适合自身业务 |
| **法规合规** | 法律法规、行业标准 | 确认合规要求的强制性和时间节点 |
| **技术演进** | 新技术引入、架构升级 | 确认技术方案的成熟度和可行性 |

### 2.2 原始需求的格式规范

每条原始需求应记录以下信息：

| 字段 | 要求 | 说明 |
|------|------|------|
| **需求编号** | 唯一标识符，格式 OR-XXX | 示例：OR-001, OR-002 |
| **需求标题** | 一句话概括 | 清晰、简洁 |
| **提出方** | 提出需求的部门/角色 | 明确业务来源 |
| **提出时间** | 需求的提出日期 | 用于追溯和版本管理 |
| **原始表述** | 原始的、未经修改的表述 | 保留原始措辞，便于追溯 |
| **性能指标** | 该需求涉及的性能约束（如有） | 示例：响应时间 < 2s，支持 1000+ 并发。功能需求包含操作性能约束时填写；非功能需求必须填写 |
| **背景/原因** | 为什么需要这个需求 | 业务背景、痛点、驱动因素 |
| **优先级** | P0/P1/P2/P3 | P0：必须；P1：应该；P2：可以；P3：可选 |
| **来源渠道** | 需求来自哪个渠道 | 会议/邮件/文档/反馈/其他 |
| **关联需求** | 与其他需求的关联关系 | 依赖/冲突/补充 |

### 2.3 原始需求收集检查清单

- [ ] 所有相关方都已覆盖
- [ ] 每条需求都有明确的提出方
- [ ] 每条需求都有原始的、未经修改的表述
- [ ] 每条需求都有背景或原因说明
- [ ] 优先级标注清晰、合理
- [ ] 来源渠道明确
- [ ] 需求之间已标注初步的关联关系

### 2.4 原始需求的初步整理

当原始需求来自多个渠道、粒度参差不齐或存在碎片化描述时，建议先进行初步整理再进入规范化分解，以提高处理效率。

#### 四种整理方法

| 方法 | 适用场景 | 操作要点 |
|------|---------|---------|
| **血缘归并** | 碎片描述同一功能的不同切面 | 按业务血缘聚类，还原完整功能表述 |
| **角色聚合** | 不同角色对同一功能的诉求 | 合并为一条需求，标注角色差异 |
| **流程串联** | 碎片是同一流程的不同步骤 | 按操作流程串联，保持流程完整性 |
| **父子补全** | 一条主需求 + 多条补充细节 | 以主需求为载体，补充细节作为子约束 |

#### 五条整理规则

1. **不做过度合并** — 不同功能域的碎片各自独立，不强行合并。判断标准：合并后能否用一句话说清核心功能
2. **保留原始出处** — 每条碎片合并后记录提出方、原始表述、日期，确保可回溯
3. **性能指标不合并** — 功能描述可合并，性能指标单独提取（后续走不同的处理分支）
4. **相互矛盾的碎片标注不合并** — 语义直接矛盾的碎片保持独立，标注"冲突待确认"，由相关方决策
5. **无法判断的标记为"待补全"** — 信息量不足的碎片不猜测，由提出方补充上下文

#### 整理后的输出格式

```
原始需求 [OR-XXX]
├─ 需求标题：[一句话概括]
├─ 需求描述：[合并后的完整需求描述]
├─ 原始碎片清单：
│  ├─ [碎片1原始表述] — [提出方] [日期]
│  └─ [碎片2原始表述] — [提出方] [日期]
├─ 包含的性能指标：[如有，单独列出]
├─ 依赖/冲突关系：[如有，标注]
└─ 整理说明：[归并依据]
```

#### 初步整理检查清单

- [ ] 碎片是否按业务血缘或流程进行了合理归并？
- [ ] 合并后的需求能否用一句话说清核心功能？
- [ ] 每条碎片是否保留了原始出处？
- [ ] 性能指标是否已从功能描述中分离提取？
- [ ] 相互矛盾的碎片是否已标注"冲突待确认"？
- [ ] 信息量不足的碎片是否已标记为"待补全"？

## 三、规范化规则

### 3.1 规范化的定义

**规范化**（Normalization）是将原始需求的自然语言表述转化为结构化、标准化、可测试的需求描述的过程。

### 3.2 四项基本检查

#### 3.2.1 完整性检查

检查每条需求是否包含必要的信息：

- 是否有清晰的**需求描述**（系统应该做什么）
- 是否有**提出方**（谁提出的需求）
- 是否有**优先级**（P0/P1/P2/P3）
- 是否有**类型**（功能需求/非功能需求）

#### 3.2.2 清晰性检查

检查需求的表述是否清晰、无歧义：

- 无模糊术语（"尽可能"、"大概"、"必要时"、"尽量"等）
- 术语一致（同一概念使用同一术语）
- 相关方清晰（谁使用、谁受益）
- 约束清晰（技术、时间、资源约束）

**模糊术语替换对照表**：

| 模糊术语 | 应替换为 |
|---------|---------|
| 尽可能、尽量 | 具体数值或范围（如：响应时间 < 2s） |
| 大概、大约 | 精确描述（如：支持 1000 并发用户） |
| 必要时 | 触发条件（如：当请求量 > 1000/s 时） |
| 良好的 | 具体指标（如：可用性 99.9%） |
| 实时的 | 具体延迟要求（如：延迟 < 100ms） |
| 大量的 | 具体数量（如：支持 10000 条数据） |
| 快速的 | 具体时间（如：响应时间 < 1s） |

#### 3.2.3 可测试性检查

**功能需求**的可测试性：
- 是否有明确的输入（用户输入了什么？）
- 是否有明确的处理逻辑（系统怎么处理？）
- 是否有明确的输出（系统返回了什么？）
- 验收标准是否可量化（成功的标志是什么？）

**非功能需求**的可测试性：
- 是否有具体的量化指标（响应时间、吞吐量、可靠率等）
- 指标是否有明确的适用条件（在什么负载/条件下）
- 是否有验证方法（如何测试和验证）

#### 3.2.4 独立性检查

- 需求是否相对独立
- 是否与其他需求过度耦合
- 如果过于复杂，标注需要进一步分解

### 3.3 功能与非功能需求分离

#### 3.3.1 分离规则

在规范化过程中，必须将功能需求和非功能需求明确分离，分别创建独立的规范化需求记录：

**功能需求特征**（应该生成 SR-F）：
- 描述"系统应该提供什么功能"或"系统应该怎样处理"
- 包含 IPO（Input-Process-Output）的明确描述
- 示例：订单创建、数据查询、权限验证

**非功能需求特征**（应该生成 SR-NFR）：
- 描述"系统应该满足什么质量特性"或"系统应该有什么性能表现"
- 属于三大类之一：质量特性、环境适应性、可实现性
- 不包含具体的业务操作步骤，而是对操作的约束
- 示例：响应时间 < 2s、支持 10000 并发用户、99.9% 可用性

#### 3.3.2 分离方法

1. 识别原始需求中的"做什么"部分 → 作为功能需求
2. 识别原始需求中的"有多好"部分 → 作为非功能需求
3. 分别输出不同的规范化需求，但标注相互关联关系

#### 3.3.3 分离示例

原始需求：
```
用户登录系统时，系统应该快速响应，在 2 秒内完成验证。
```

规范化后：
```
功能需求（SR-F）：
- 描述：用户输入用户名和密码后，系统验证身份并返回登录结果
- 验收标准：验证成功返回用户信息和 Token，验证失败返回错误信息

非功能需求（SR-NFR）：
- 描述：登录功能应该快速响应，提供良好的用户体验
- 性能指标：响应时间 < 2s（包括网络延迟），在 1000 并发用户下平均 < 500ms
```

### 3.4 规范化后的输出格式

每条规范化后的需求输出为：

```markdown
- 原始文本：[原始描述]
- 规范化描述：[规范化后的描述]
- 提出方：[需求提出者]
- 优先级：[P0/P1/P2/P3]
- 类型：[功能需求/非功能需求]
- 性能指标：[具体的量化指标，功能需求含性能约束时填写；非功能需求必须填写]
- 验收标准：[可量化的验收条件]
- 规范化说明：[做了哪些规范化处理]
```

### 3.5 规范化检查清单

- [ ] 所有需求通过四项检查（完整性、清晰性、可测试性、独立性）
- [ ] 模糊术语已替换为精确表述
- [ ] 术语在全部需求中保持一致
- [ ] 验收标准可量化
- [ ] 性能指标已提取并填入"性能指标"字段（非功能需求必须有量化指标）
- [ ] 功能需求和非功能需求已明确分离
- [ ] 功能需求和非功能需求的关联关系已标注
- [ ] 优先级标注清晰

## 四、原始需求分解规则

### 4.1 分解的定义

**分解**（Decomposition）是将复杂的原始需求逐层拆解为原子性的末级需求的过程。每条末级需求应该是独立的、不可再分的，且只需分配到相关方需求架构的一个末级节点。

### 4.2 分解原则

1. **原子性原则**：每条分解后的末级需求应是原子性的、不可再分的
2. **1:1映射原则**：每条末级需求只需映射到相关方需求架构的一个末级节点
3. **语义完整性原则**：分解后不丢失原始需求的语义，所有末级需求之和应能完整代表原始需求
4. **粒度一致性原则**：末级需求的粒度应与同层其他末级需求保持一致

### 4.3 触发分解的条件

遇到以下情况时需求需要分解：

| 条件 | 示例 | 分解方式 |
|------|------|---------|
| 包含"和/并且/同时"连接多个独立功能 | "系统应支持订单创建、修改和删除" | 分解为 3 个独立需求 |
| 涉及多个独立的操作或流程 | "系统应支持订单的完整生命周期管理" | 分解为创建、处理、完成等流程需求 |
| 涉及多个角色或相关方 | "销售和财务都需要查看订单信息" | 按角色分解 |
| 需要映射到多个相关方需求架构末级节点 | 一个需求描述了多个不同分类的功能 | 按功能分类分解 |

### 4.4 分解的停止准则

**判断方法**：在设计每条末级需求时，问自己：
"这个条目能映射到相关方需求架构的一个末级节点吗？"

- ❌ **需要多个节点承接** → 分解不充分，继续分解
- ✅ **只需一个节点承接** → 分解完成，停止分解

### 4.5 分解示例

```
原始需求：系统应支持订单全生命周期管理
├─ 末级需求 1.1：系统应支持订单创建功能
├─ 末级需求 1.2：系统应支持订单修改功能
├─ 末级需求 1.3：系统应支持订单取消功能
└─ 末级需求 1.4：系统应支持订单查询功能
```

### 4.6 分解记录格式

## 五、冲突和重复检测

### 5.1 冲突检测

#### 5.1.1 冲突类型

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| **直接冲突** | 两条需求语义直接矛盾，无法同时满足 | 由相关方确认保留哪条或如何调和 |
| **间接冲突** | 不直接矛盾，但同时实现可能导致系统行为不一致 | 由相关方确认是否需要调整业务规则 |
| **可合并冲突** | 有冲突但可以通过调整合并解决 | 建议合并为一条更完整的需求 |

#### 5.1.2 冲突检测规则

1. **语义冲突**：对比需求描述的语义，识别直接矛盾和间接矛盾
2. **业务规则冲突**：检查需求是否违反现有业务规则或流程
3. **技术约束冲突**：检查需求是否违反现有技术约束或架构

### 5.2 重复检测

#### 5.2.1 重复类型

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| **完全重复** | 新需求与现有需求完全相同 | 标记为"已存在"，不重复创建 |
| **部分重复** | 新需求与现有需求部分重叠 | 建议合并为更完整的描述 |
| **相似重复** | 表达不同但核心功能可能相同 | 由相关方确认是否为同一需求 |

#### 5.2.2 重复检测规则

1. **语义相似度**：核心功能、输入输出、验收标准是否相同
2. **表达相似度**：使用相同关键词，描述相同业务流程
3. **范围重叠度**：需求范围有交集，新需求是现有需求的子集或超集

### 5.3 检测记录格式

```markdown
| 需求 ID | 检查对象 | 问题类型 | 问题描述 | 建议 | 优先级 |
|---------|---------|---------|---------|------|--------|
| OR-001  | OR-005  | 直接冲突 | 一个允许删除，一个禁止删除 | 需确认保留哪条 | 高 |
| OR-002  | OR-003  | 部分重复 | 新需求 50% 功能已被覆盖 | 建议合并 | 中 |
```

### 5.4 冲突和重复检测检查清单

- [ ] 所有新需求已完成冲突检测和重复检测
- [ ] 新需求之间已交叉检测
- [ ] 每个问题已分类并标注优先级
- [ ] 每个问题有明确的解决/合并建议
- [ ] 检测结果已记录，待相关方确认

## 六、到相关方需求的映射

### 6.1 映射路径

原始需求经过规范化、分解后，末级需求通过 N:1 关系映射到相关方需求架构的末级节点：

```
原始需求末级需求清单
   │
   ├─ 功能类末级需求 ──N:1──→ SR-F 架构末级节点
   │
   └─ 非功能类末级需求 ──N:1──→ SR-NFR 架构末级节点
```

### 6.2 映射规则

| 规则 | 说明 |
|------|------|
| **N:1 承接支持** | 多个原始需求末级可映射到同一个相关方需求架构末级节点 |
| **功能与非功能分离** | 功能类末级需求映射到 SR-F 分支，非功能类映射到 SR-NFR 分支 |
| **1:1 分配约束** | 每条原始需求末级只能分配到唯一一个相关方需求架构末级节点 |
| **完全映射** | 所有原始需求末级都必须映射到相关方需求架构，无遗漏 |

### 6.3 映射关系表示例

```
| 原始需求末级 | 相关方需求架构末级 | 类型 | 符合性说明 |
|-------------|-------------------|------|-----------|
| OR-001.1    | SR-F-001          | 功能 | 末级需求的规范化描述与 SR 定义一致 |
| OR-002.1    | SR-F-001          | 功能 | 两个末级需求合并到同一 SR 节点 |
| OR-003.1    | SR-NFR-001        | 非功能 | 末级需求的性能指标在 SR 中量化 |
```

### 6.4 映射检查清单

- [ ] 所有原始需求末级已映射到相关方需求架构
- [ ] 功能需求和非功能需求已分别映射到正确的分支
- [ ] 每条末级需求只映射到一个相关方需求架构末级节点（1:1 约束）
- [ ] N:1 映射的承接数量合理（平均 2-3 条）
- [ ] 映射关系已记录，支持双向追溯

## 八、常见错误和最佳实践

### 8.1 常见错误

#### 错误 1：跳过规范化直接分解

**表现**：不经过规范化处理，直接对原始表述进行分解，导致分解结果质量参差不齐

**后果**：模糊表述在分解后依然模糊，问题在后续层被放大

**正确做法**：先规范化再分解——先通过四项检查确保需求质量，然后再进行分解

#### 错误 2：功能和非功能需求未分离

**表现**：将功能需求和非功能需求混在一起处理，没有分别记录和分类

**后果**：非功能需求可能被当作功能需求处理，经过业务架构层，导致设计混乱

**正确做法**：在规范化阶段就明确分离功能和非功能需求，分别记录、分别处理

#### 错误 3：分解不充分

**表现**：一条末级需求需要映射到多个相关方需求架构末级节点，但没有继续分解

**后果**：导致 1:1 分配约束无法满足，在后续层造成追溯断裂

**正确做法**：使用 1:1 映射约束作为分解停止的判断标准——如果不能映射到唯一节点，继续分解

#### 错误 4：分解过度

**表现**：将需求分解得过细，一条末级需求粒度远小于同层其他需求

**后果**：导致大量末级需求需要合并到同一个相关方需求节点，增加管理成本

**正确做法**：保持粒度和同层其他需求一致，通过 N:1 映射检查承接数量是否合理

#### 错误 5：忽视冲突和重复检测

**表现**：完成规范化后直接进入相关方需求设计，不做冲突和重复检测

**后果**：下游设计时才发现需求之间的矛盾或冗余，造成返工

**正确做法**：将冲突和重复检测作为规范化后的标准步骤，固定在设计流程中

### 8.2 最佳实践

#### 实践 1：使用结构化格式记录原始需求

**说明**：每条原始需求使用统一的字段格式记录，确保信息的完整性和一致性

**好处**：
- 便于自动化和批量处理
- 减少信息遗漏
- 支持后续的追溯分析

#### 实践 2：在规范化阶段就考虑分类

**说明**：在规范化处理的同時，就开始思考每条需求属于功能还是非功能、业务相关还是平台功能相关

**好处**：
- 提前识别分类边界
- 为后续工序准备好分类依据
- 减少后续的分类争议

#### 实践 3：边分解边考虑映射

**说明**：在分解每条需求时，同步考虑它应该映射到相关方需求架构的哪个节点

**好处**：
- 以映射为导向，确保分解粒度的合理性
- 避免分解完才发现无法映射
- 提高分解效率

#### 实践 4：保留原始表述

**说明**：无论规范化后的描述如何优化，始终保留原始表述作为追溯依据

**好处**：
- 需要时可以回查原始信息来源
- 支持争议时的溯源
- 便于审计和合规

#### 实践 5：在规范化阶段就记录分解决策

**说明**：在规范化处理时，同步记录分解的依据和决策，而不是事后补充

**好处**：
- 决策依据更清晰
- 减少遗忘
- 便于其他人理解和审查

## 九、与其他规范的关系

### 9.1 与相关方需求规范的关系

本规范的输出（规范化、分解、确认后的末级需求）是相关方需求规范的直接输入：

- 原始需求末级 → N:1 → 相关方需求架构末级节点（SR-F / SR-NFR）
- 原始需求的质量直接影响相关方需求的质量
- 原始需求分解的充分性决定相关方需求映射的完整性

### 9.2 与系统设计准则的关系

本规范是系统设计准则第 1 层（原始需求及分析）的具体化：

- 系统设计准则定义了原始需求在 5 层结构中的位置和角色
- 本规范定义了原始需求的具体处理方法和标准
- 本规范中的规则和检查清单与系统设计准则保持一致
