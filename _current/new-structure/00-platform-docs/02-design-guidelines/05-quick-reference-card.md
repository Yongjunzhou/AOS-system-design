## 📋 核心约束速记

### 约束1：1:1 分配约束
```
每条上层需求 → 唯一一个下层方案条目

检查方法：
✓ 需求是否分配了？
✓ 是否只分配到一个条目？
✓ 是否有多个条目都承接了这条需求？

常见问题：
❌ 一条需求分配到多个条目 → 需求分解不足
❌ 一条需求无法分配 → 方案设计不完整
```

### 约束2：N:1 承接支持
```
每个下层方案条目 ← 多条上层需求

合理范围：
- 最少：1条需求
- 最多：4-5条需求
- 平均：2-3条需求

检查方法：
✓ 承接的需求数量是否合理？
✓ 承接的需求之间是否有逻辑关联？
```

### 约束3：双向追溯
```
需求 ↔ 方案（通过超链接）

检查方法：
✓ 每条需求都标注了分配的方案条目？
✓ 每个方案条目都标注了承接的需求？
✓ 所有超链接都有效？
```

## 🏗️ 五层结构速记

```
第1层：原始需求及分析
  ↓ 分解到末级需求
第2层：相关方需求（原始需求→方案）
  ├─ 架构定义子文档（分类组织，架构末级节点=映射关键层级）
  └─ 详细定义子文档（条目化需求，详细定义末级节点=分配源头）
  ↓ N:1
第3层：业务架构（相关方需求功能部分的方案）
  └─ 架构定义（IPO清单，无独立详细定义子文档）
  ↓ N:1
第4层：系统需求（业务架构+相关方非功能需求的方案）
  ├─ 架构定义子文档（分类组织）
  │  ├─ 功能需求分支（0-9级，5级=承接层，9级=场景活动→PA）
  │  └─ 非功能需求分支（0-6级，平行体系）
  └─ 详细定义子文档（条目化需求，含符合性分析）
  ↓ N:1
第5层：产品架构（系统需求的方案）
  ├─ 架构定义子文档（分类组织）
  └─ 详细定义子文档（条目化组件，含实现细节和性能指标）
```

> **同步设计原则**：在开展上层详细定义的同时，同步建立下层方案的架构结构骨架（末级节点名称+映射关系），确保每条详细定义末级节点都有明确的分配目标。详见设计准则。

### 每层的角色

| 层 | 名称 | 角色 | 文档结构 |
|----|------|------|---------|
| 1 | 原始需求 | 需求 | 条目化需求 + 末级分解 |
| 2 | 相关方需求 | 方案→需求 | 架构定义子文档 + 详细定义子文档 |
| 3 | 业务架构 | 方案 | 架构定义（IPO清单，无详细定义） |
| 4 | 系统需求 | 方案→需求 | 架构定义子文档 + 详细定义子文档 |
| 5 | 产品架构 | 方案 | 架构定义子文档 + 详细定义子文档 |

## ✅ 操作检查清单（精简版）

### 需求分解
- [ ] 末级需求是原子性的
- [ ] 粒度合理（平均2-3条）
- [ ] 每条都有定义和验收标准
- [ ] 每条都分配到下层方案

### 需求映射
- [ ] 1:1约束满足（每条需求唯一分配）
- [ ] N:1支持满足（每个方案承接2-4条需求）
- [ ] 映射关系符合业务逻辑

### 符合性分析
- [ ] 每个方案都有符合性分析
- [ ] 说明了如何满足所承接的需求
- [ ] 逻辑清晰、可验证

### 双向追溯
- [ ] 需求标注了分配的方案
- [ ] 方案标注了承接的需求
- [ ] 所有超链接有效

### 同步设计
- [ ] 上层详细定义阶段同步建立下层架构结构骨架
- [ ] 每条详细定义末级节点都有明确的分配目标
- [ ] 结构骨架冻结后方可填充具体内容

### 文档规范
- [ ] 结构清晰一致
- [ ] 包含两部分（架构定义子文档 + 详细定义子文档）
- [ ] 版本号正确更新

## 🚀 场景A：逆向工程（快速流程）

```
第1步：代码和图纸分析（1-2天）
  ├─ 分析架构图纸
  ├─ 分析代码结构
  └─ 提取功能清单

第2步：产品架构文档编写（2-3天）
  └─ 编写PA条目（先详细定义，同步预留架构骨架）

第3步：系统需求反向推导（3-5天）
  ├─ 从PA推导SysReq
  └─ 同步补充PA架构定义子文档的映射视图

第4步：业务架构反向推导（3-5天）
  ├─ 从SysReq推导BA
  └─ 注意：BA只有架构定义（IPO清单）

第5步：相关方需求反向推导（2-3天）
  └─ 从BA推导SR

第6步：原始需求反向推导（1-2天）
  └─ 从SR推导OR

第7步：验证（2-3天）
  ├─ 1:1约束验证
  ├─ N:1支持验证
  ├─ 双向追溯验证
  └─ 符合性分析验证

第8步：文档完善和发布（2-3天）

总时间：16-26天（约3-4周）
```

## 🎯 场景B：瀑布式/新研产品（快速流程）

```
第1步：原始需求分解（2-3天）
  └─ 分解到末级需求

第2步：相关方需求设计（5-8天）
  ├─ 阶段一：架构定义（2-3天）
  └─ 阶段二：详细定义（3-5天）
  ※ 同步建立BA架构结构骨架

第3步：业务架构设计（5-7天）
  └─ 设计BA-IPO条目
  ※ 同步建立SysReq架构结构骨架

第4步：系统需求架构设计（2-3天）
  └─ 分类组织（功能+非功能）
  ※ 同步建立PA架构结构骨架

第5步：系统需求详细设计（5-7天）
  └─ 详细定义SysReq条目

第6步：产品架构设计（7-10天）
  └─ 设计PA条目

第7步：完整性和一致性验证（2-3天）
  ├─ 1:1约束验证
  ├─ N:1支持验证
  ├─ 双向追溯验证
  └─ 符合性分析验证

第8步：需求映射矩阵生成（1-2天）

第9步：文档完善和发布（2-3天）

总时间：30-48天（约4-7周）
```

## 📝 场景C：敏捷式/增量需求（快速流程）

```
第1步：新需求分析和分类（1天）
  ├─ 分类（功能/非功能、新增/修改/删除）
  ├─ 影响范围评估
  └─ 优先级排序

第2步：新需求分解（1-2天）
  └─ 分解到末级需求

第3步：相关方需求映射（2-3天）
  ├─ 选择映射策略（映射/修改/新建）
  └─ 使用增量标记（+ / ~）

第4步：业务架构影响分析（2-3天）
  ├─ 选择更新策略（映射/修改/新建）
  └─ 使用增量标记

第5步：系统需求影响分析（2-3天）
  ├─ 选择更新策略
  └─ 使用增量标记

第6步：产品架构影响分析（3-5天）
  ├─ 选择更新策略
  └─ 使用增量标记

第7步：验证（1-2天）
  ├─ 1:1约束验证
  ├─ N:1支持验证
  ├─ 双向追溯验证
  └─ 符合性分析验证

第8步：文档完善和发布（1-2天）
  ├─ 文档完善
  ├─ 团队沟通
  └─ 发布更新

※ 每步采用两阶段工作流程：结构骨架约定（1-2h）→ 具体设计（2-6天）

总时间：13-21天（约2-3周）
```

## ⚡ 场景D：DevOps/快速修复（快速流程）

```
第1步：快速影响范围评估（15分钟）
  ├─ 问题分类（bug/需求/优化）
  ├─ 优先级评估（P0/P1/P2）
  └─ 选择处理模式

  影响仅PA → 模式1：快速修复（几小时）
  影响PA+SysReq → 模式2：局部设计（1天）
  影响多层 → 模式3：完整设计（2-3天）

第2步：快速修复（模式1，几小时）
  ├─ 定位问题组件（PA-XXX）
  ├─ 直接修改PA条目
  └─ 验证并发布

第3步：局部设计（模式2，1天）
  ├─ 修改SysReq + PA
  ├─ 同步维护架构+详细定义子文档
  └─ 验证并发布

第4步：完整设计（模式3，2-3天）
  └─ 按敏捷式8步流程快速执行

※ 核心原则：快速响应、最小化变更、保持一致性、充分验证

总时间：几小时 - 2-3天（取决于模式选择）
```

## 🔍 常见问题速答

### Q1：末级需求应该分解到什么程度？
**A**：
- ✅ 原子性的、不可再分的
- ✅ 能够分配到下层方案的某个条目
- ✅ 粒度合理（平均分解数2-3条）

### Q2：一个方案条目应该承接多少条需求？
**A**：
- 最少：1条
- 最多：4-5条
- 平均：2-3条

### Q3：如何判断映射是否合理？
**A**：
- ✅ 1:1约束满足（每条需求唯一分配）
- ✅ N:1支持满足（每个方案承接2-4条需求）
- ✅ 承接的需求之间有逻辑关联

### Q4：如何编写有效的符合性分析？
**A**：
1. 列出所有承接的需求
2. 说明如何满足每条需求
3. 说明验证方法
4. 说明是否有限制或特殊情况

### Q5：如何建立有效的双向追溯？
**A**：
1. 在方案条目中标注承接的需求
2. 在需求中标注分配的方案条目
3. 使用超链接实现快速导航
4. 定期验证所有超链接的有效性

### Q6：架构末级节点和详细定义末级节点有什么关系？
**A**：
- **架构末级节点**：方案文档的架构定义子文档中的末级节点，是需求映射的关键层级
- **详细定义末级节点**：方案文档的详细定义子文档中的条目，是分配到下层方案架构末级节点的源头
- 两者共同构成方案文档的双层结构

## 🎓 最佳实践速记

### 场景A：逆向工程
- ✅ 邀请原开发人员参与，进行代码审查
- ✅ 与代码实现进行对比，发现差异
- ✅ 进行用户访谈，补充缺失的需求
- ✅ 注意BA只有架构定义（IPO清单），没有详细定义

### 场景B：瀑布式/新研产品
- ✅ 确保原始需求清晰完整
- ✅ 采用同步设计：开展上层详细定义时同步建立下层架构骨架
- ✅ 每个阶段都进行审查
- ✅ 使用两阶段工作流程（结构骨架→具体设计）

### 场景C：敏捷式/增量需求
- ✅ 优先处理高优先级需求
- ✅ 分阶段处理，避免一次性大量变更
- ✅ 优先映射到现有方案，最小化影响
- ✅ 使用增量标记（+新增/~修改）追踪变更
- ✅ 每步采用两阶段工作流程

### 场景D：DevOps/快速修复
- ✅ 15分钟快速决策，选择最简模式
- ✅ 只修改必要的部分，避免过度设计
- ✅ 充分验证副作用
- ✅ 事后补全文档，保持追溯链路完整
