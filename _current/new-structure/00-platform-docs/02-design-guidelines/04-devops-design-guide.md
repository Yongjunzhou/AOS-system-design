# DevOps系统设计指南
**DevOps System Design Guide**

**文档版本**：v2.0  
**创建日期**：2026-05-15  
**基于准则版本**：系统设计准则 v2.3  
**适用场景**：在产品使用过程中发现新的需求或bug，需要快速响应和最小化变更

---

## 📌 场景描述

### 适用条件

- ✅ 产品已上线运行
- ✅ 在使用过程中发现bug或紧急需求
- ✅ 需要快速响应（几小时到1-2天）
- ✅ 需要最小化变更，避免影响现有功能
- ✅ 适合DevOps团队快速迭代

### 不适用条件

- ❌ 需要进行大规模重构
- ❌ 影响范围超过50%
- ❌ 需要完整的系统设计

### 参与角色

本指南涉及三个核心参与者，在DevOps设计流程中各司其职：

| 角色 | 职责 | 参与环节 |
|------|------|---------|
| **流水线开发者** | 定义设计准则和指南，配置 AI 辅助工具，审核 AI 输出 | 所有步骤的审核确认 |
| **AI** | 按指令执行影响范围评估、快速修复生成、映射关系更新等自动化任务 | 所有步骤的自动生成和检查 |
| **AOS开发者** | 提出需求、确认设计、审核方案（含需求提出者、业务流程开发者、信息系统开发者三类） | 问题定位、模式选择、修复验证 |

## 🎯 核心目标

快速响应生产环境的问题，通过最小化变更来修复bug或实现紧急需求，同时保持系统设计的一致性

---

## 📊 三种处理模式

### 模式1：快速修复（几小时）
**适用**：仅影响产品架构的bug或小改动

```
发现问题 → 快速定位 → 直接修改产品架构 → 快速验证 → 发布
```

**特点**：
- ✅ 不需要改动上层设计
- ✅ 最快的响应时间
- ✅ 最小的变更范围

**示例**：
- 修复UI显示bug
- 调整性能参数
- 修复前端逻辑错误

### 模式2：局部设计（1天）
**适用**：影响产品架构和系统需求的问题

```
发现问题 → 评估影响 → 快速设计 → 实现 → 验证 → 发布
```

**特点**：
- ✅ 需要修改系统需求和产品架构
- ✅ 不影响业务架构
- ✅ 相对快速的响应时间

**示例**：
- 修复系统功能bug
- 实现小功能需求
- 优化系统性能

### 模式3：完整设计（2-3天）
**适用**：影响多层设计的问题

```
发现问题 → 完整评估 → 完整设计 → 实现 → 验证 → 发布
```

**特点**：
- ✅ 需要修改多层设计
- ✅ 可能影响业务架构
- ✅ 需要完整的系统设计流程

**示例**：
- 修复业务流程bug
- 实现重要功能需求
- 架构级别的优化

## 📋 快速决策树

```
发现问题
  ↓
【第1步】快速影响范围评估（15分钟）
  ↓
问题是否仅影响产品架构？
  ├─ 是 → 【模式1：快速修复】
  │       └─ 直接修改PA → 验证 → 发布（几小时）
  │
  └─ 否 → 问题是否影响系统需求？
      ├─ 是 → 问题是否影响业务架构？
      │       ├─ 否 → 【模式2：局部设计】
      │       │       └─ 修改SysReq + PA → 验证 → 发布（1天）
      │       │
      │       └─ 是 → 【模式3：完整设计】
      │               └─ 完整设计流程 → 验证 → 发布（2-3天）
      │
      └─ 否 → 问题是否影响相关方需求？
          ├─ 是 → 【模式3：完整设计】
          │       └─ 完整设计流程 → 验证 → 发布（2-3天）
          │
          └─ 否 → 需要重新评估问题
```

## 📋 完整操作流程

### 第1步：快速影响范围评估（15分钟）

**目标**：快速判断问题的影响范围，选择合适的处理模式

**具体操作**：

1. **问题分类**
   - Bug修复 vs 功能需求 vs 性能优化
   - 紧急程度：P0（立即修复）vs P1（今天修复）vs P2（本周修复）

2. **影响范围快速评估**
   - 问题是否仅影响产品架构（PA）？
   - 是否影响系统需求（SysReq）？
   - 是否影响业务架构（BA）？
   - 是否影响相关方需求（SR）？

3. **处理模式选择**
   - 仅PA → 模式1（快速修复）
   - PA + SysReq → 模式2（局部设计）
   - 跨多层 → 模式3（完整设计）

**输出**：
- 问题分类和优先级
- 影响范围评估
- 选定的处理模式

---

### 第2步：快速修复（模式1，几小时）

**目标**：快速修复仅影响产品架构的问题

**具体操作**：

1. **问题定位**
   - 定位到具体的产品组件（PA）
   - 理解问题的根本原因

2. **快速修改**
   ```
   PA-XXX: [产品组件名称]
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[具体修改]
   ├─ 影响分析：[是否影响其他组件]
   ├─ 符合性分析：[如何满足需求]
   └─ 验证方法：[如何验证修复]
   ```
   **说明**：在快速修复场景下，PA 条目采用简化格式（聚焦原问题和修改内容），但应包含符合性分析以保持追溯链路的完整性。事后应根据需要在产品架构文档中补充**架构定义子文档**（分类组织）和**详细定义子文档**（条目化定义）。

3. **快速验证**
   - 验证问题是否解决
   - 验证是否有副作用
   - 验证性能是否受影响

4. **发布**
   - 更新产品架构文档
   - 发布修复

**质量检查**：
- [ ] 问题是否完全解决
- [ ] 是否有副作用
- [ ] 是否需要更新文档

**输出**：
- 修复后的产品架构
- 验证报告

---

### 第3步：局部设计（模式2，1天）

**目标**：快速设计和实现影响系统需求的问题

**具体操作**：

1. **影响范围分析**
   - 问题影响哪些系统需求（SysReq）？
   - 是否需要修改现有SysReq？
   - 是否需要创建新的SysReq？

2. **快速设计**
   ```
   SysReq-XXX: [系统需求名称]
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[具体修改]
   ├─ 对应的产品架构：[PA-XXX]
   ├─ 符合性分析：[如何满足需求]
   └─ 验证方法：[如何验证]
   
   PA-XXX: [产品组件名称]
   ├─ 修改内容：[具体修改]
   ├─ 承接的系统需求：[SysReq-XXX]
   └─ 验证方法：[如何验证]
   ```

3. **快速验证**
   - 验证映射关系是否正确
   - 验证是否有遗漏的影响
   - 验证功能是否正确实现

4. **发布**
   - 更新系统需求和产品架构文档
   - 发布修复

   **文档结构**：更新后的系统需求文档和产品架构文档应同步维护 **架构定义子文档**（以架构末级节点为基本单位的分类组织）和 **详细定义子文档**（以详细定义末级节点为基本单位的条目化定义），保持与 v2.3 准则一致。

**质量检查**：
- [ ] 映射关系是否正确
- [ ] 是否有遗漏的影响
- [ ] 是否需要更新上层设计

**输出**：
- 修改后的系统需求和产品架构
- 验证报告

---

### 第4步：完整设计（模式3，2-3天）

**目标**：完整设计和实现影响多层的问题

**具体操作**：

按照敏捷式系统设计指南的8步流程执行，但加快速度。以下提供各层的快速条目模板（增量标记）：

1. **快速影响分析**（1小时）
   - 分析问题影响的所有层级
   - 确定需要修改的文档

2. **快速设计**（1天）
   - 从影响点开始，逐层设计
   - 使用两阶段工作流程（结构骨架 + 具体设计）
   - 遵循同步设计原则：在修改上层详细定义的同时，同步调整下层方案的架构结构骨架（保持一致）

   **相关方需求（SR）快速模板**：
   ```
   ~ SR-F-XXX: [修改的相关方需求名称]
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对需求定义的修改]
   ├─ 对应的业务架构：[BA-XXX]
   └─ 更新符合性分析：[更新说明]
   ```

   **业务架构（BA）快速模板**：
   ```
   ~ BA-XXX: [修改的业务组件名称]
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对IPO定义的修改]
   ├─ Input/Process/Output：[更新后的I/P/O]
   ├─ 对应的系统需求：[SysReq-XXX]
   └─ 更新符合性分析：[更新说明]
   ```
   > **业务架构独立存在说明**：BA只有架构定义（IPO清单），没有独立的详细定义子文档。BA的"详细定义"实质上是架构定义的扩展（IPO详细说明），与SysReq/PA的详细定义性质不同。在DevOps快速修复中，BA修改仅涉及IPO定义本身，不需要维护独立的详细定义文档结构。

   **系统需求（SysReq）快速模板**：
   ```
   ~ SysReq-XXX: [修改的系统需求名称]
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对需求定义的修改]
   ├─ 类型：[功能需求/非功能需求]
   ├─ 对应的产品架构：[PA-XXX]
   └─ 更新符合性分析：[更新说明]
   ```

   **产品架构（PA）快速模板**：
   ```
   ~ PA-XXX: [修改的产品组件名称]
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对实现方案的修改]
   ├─ 承接的系统需求：[SysReq-XXX]
   └─ 更新符合性分析：[更新说明]
   ```

   > **标记说明**：~ 前缀表示修改现有条目，+ 前缀表示新增条目。完整模板格式请参考敏捷式系统设计指南。

   **文档结构**：所有更新的文档应同步维护**架构定义子文档**（以架构末级节点为基本单位的分类组织）和**详细定义子文档**（以详细定义末级节点为基本单位的条目化定义），确保追溯链路的完整性。

3. **快速验证**（半天）
   - 验证映射关系
   - 验证完整性和一致性

4. **发布**（半天）
   - 更新所有相关文档
   - 发布修复

**输出**：
- 修改后的系统设计文档
- 验证报告

## 🎯 问题类型处理指南

### Bug修复

**快速诊断**：
1. 问题现象是什么？
2. 在哪个组件中出现？
3. 根本原因是什么？

**处理流程**：
- 仅UI bug → 模式1
- 系统功能bug → 模式2
- 业务流程bug → 模式3

**验证方法**：
- 问题是否完全解决
- 是否有副作用
- 是否影响其他功能

### 紧急需求

**快速评估**：
1. 需求的优先级是什么？
2. 需要修改哪些层级？
3. 预计工作量是多少？

**处理流程**：
- 小功能需求 → 模式2
- 大功能需求 → 模式3

**验证方法**：
- 功能是否正确实现
- 是否满足需求
- 是否有遗漏

### 性能优化

**快速诊断**：
1. 性能瓶颈在哪里？
2. 是否需要修改架构？
3. 优化的收益是多少？

**处理流程**：
- 参数调整 → 模式1
- 算法优化 → 模式2
- 架构优化 → 模式3

**验证方法**：
- 性能是否改善
- 是否有副作用
- 是否满足目标

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 快速修复引入新bug | 问题扩大 | 充分的验证和测试 |

| 模式选择错误 | 工作量增加 | 充分的评估 |

## 📝 使用建议

### 1. 问题到达时（5分钟）
- 快速分类问题
- 评估优先级
- 选择处理模式

### 2. 设计过程中（几小时到2-3天）
- 按照选定的模式执行
- 定期进行质量检查
- 及时处理发现的问题

### 3. 发布前（30分钟）
- 使用快速验证清单进行检查
- 更新相关文档
- 准备发布

### 4. 发布后（1小时）
- 监控系统运行状态
- 收集用户反馈
- 记录问题和解决方案

## 🎯 与其他场景的关系

### vs 敏捷式系统设计
- **相似**：都是在已有设计基础上的增量设计
- **差异**：DevOps更强调快速响应和最小化变更
- **关系**：DevOps是敏捷式的快速版本

### vs 瀑布式系统设计
- **相似**：都遵循系统设计准则
- **差异**：DevOps不需要完整的设计流程
- **关系**：DevOps是瀑布式的简化版本

### vs 逆向工程
- **相似**：都需要理解现有设计
- **差异**：DevOps是在理解基础上的快速修改
- **关系**：DevOps假设已有完整的设计文档

---

## 📌 关键原则

1. **快速响应**
   - 优先级高的问题要快速处理
   - 不要过度设计

2. **最小化变更**
   - 只修改必要的部分
   - 避免不必要的改动

3. **保持一致性**
   - 遵循系统设计准则
   - 更新相关文档

4. **充分验证**
   - 验证问题是否解决
   - 验证是否有副作用

5. **双向追溯**
   - 每次修改后更新映射关系（1:1 分配约束 + N:1 承接支持）
   - 维护从原始需求到产品架构的完整追溯链路
   - 确保修改层级的符合性分析反映在相邻层级的映射中
