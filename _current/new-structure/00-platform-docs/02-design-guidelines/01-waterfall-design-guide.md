# 瀑布式系统设计指南
**Waterfall System Design Guide**

**文档版本**：v2.0
**创建日期**：2026-05-15
**基于准则版本**：系统设计准则 v2.3
**适用场景**：原始需求全部齐备，从原始需求分析直到产品架构的完整系统设计

---

## 📌 场景描述

### 适用条件

- ✅ 原始需求已经完全收集和确认
- ✅ 需要从头到尾进行完整的系统设计
- ✅ 设计周期相对固定，需求变更较少
- ✅ 适合大型项目或关键系统

### 不适用条件

- ❌ 需求还在不断变化
- ❌ 已有部分设计文档需要继承
- ❌ 需要快速迭代和反馈

### 参与角色

本指南涉及三个核心参与者，在瀑布式设计流程中各司其职：

| 角色 | 职责 | 参与环节 |
|------|------|---------|
| **流水线开发者** | 定义设计准则和指南，配置 AI 辅助工具，审核 AI 输出 | 所有步骤的审核确认 |
| **AI** | 按指令执行规范化、分解、映射、验证等自动化任务 | 所有步骤的自动生成和检查 |
| **AOS开发者** | 提出需求、确认设计、审核方案（含需求提出者、业务流程开发者、信息系统开发者三类） | 需求提出、设计确认、方案审核 |

---

## 🎯 核心目标

建立从**原始需求**到**产品架构**的完整、清晰、可追溯的设计链路


> **工作方式**：采用系统设计准则中定义的六步法流程——第1步统一处理原始需求和相关方需求架构，第2-3步走功能分支，第4步走非功能分支，第5步合并，第6步验证。

---

## 📋 完整操作流程（6步）

### 第1步：原始需求分析及相关方需求架构定义（3-5天）

**目标**：从原始需求产出原始需求详细定义和相关方需求架构定义

**参考**：设计准则 开发六步法 第1步

**具体操作**：

1. **原始需求收集和整理**
   - 收集所有原始需求
   - 去除重复和冲突
   - 按照优先级排序

2. **原始需求规范化**
   - 格式统一（需求描述、提出方、优先级、类型）
   - 术语标准化
   - 补充性能指标

3. **原始需求分解到末级**
   ```
   原始需求 1
   ├─ 需求描述：[具体的需求定义]
   ├─ 末级需求分解：
   │  ├─ 末级需求 1.1 → [分配到相关方需求架构末级]
   │  ├─ 末级需求 1.2 → [分配到相关方需求架构末级]
   │  └─ 末级需求 1.3 → [分配到相关方需求架构末级]
   └─ 优先级：[高/中/低]
   ```

4. **相关方需求架构定义**
   - 分析末级需求的共性和关联性
   - 按功能需求和非功能需求分别分类
   - 定义相关方需求架构末级节点（含SR-F和SR-NFR分支）
   ```
   相关方需求（根节点）
   ├─ 功能需求类（分类节点）
   │  └─ SR-F-001: [相关方需求名称]（架构末级节点）
   │       ├─ 定义：[需求的定义]
   │       ├─ 承接的原始需求末级：原始需求 1.1, 1.2
   │       ├─ 符合性分析：[如何满足所承接的需求]
   │       └─ 为下层提供的设计指导：[对业务架构的指导]
   └─ 非功能需求类（分类节点）
      └─ SR-NFR-001: [相关方非功能需求]（架构末级节点）
   ```

**质量检查**：
- [ ] 每条原始需求都分解到末级需求
- [ ] 末级需求是原子性的、不可再分的
- [ ] 所有原始需求末级都已映射到相关方需求架构（覆盖率100%）
- [ ] 功能需求和非功能需求分别分类

**输出**：
- 原始需求文档（含末级需求分解）
- 相关方需求架构定义文档（树形结构 + 映射关系表）

---

### 第2步：相关方需求功能部分详细定义及业务架构定义（5-7天）

**目标**：沿功能分支推进，产出相关方需求功能部分详细定义和业务架构定义

**参考**：设计准则 开发六步法 第2步

**具体操作**：

1. **相关方需求功能部分详细定义**
   - 对SR-F架构末级节点进行详细设计分解
   - 分解到每条详细定义末级节点只需映射到业务架构的一个IPO（1:1约束）
   ```
   SR-F-001.1: [对SR-F-001的分解条目]
   ├─ 描述：[详细说明]
   ├─ 验收标准：[验收条件]
   ├─ 分配到：[业务架构末级节点BA-001]
   └─ 符合性说明：[如何满足SR-F-001]
   ```

2. **业务架构IPO定义**
   - 根据详细定义结果识别IPO
   - 为每个IPO定义I（输入）/ P（处理）/ O（输出）
   - IPO去重：复用 → 改进 → 新增
   ```
   BA-001: [IPO名称/业务组件名称]
   ├─ 定义：[业务组件的定义]
   ├─ Input（输入）：[具体形式、数据结构、来源]
   ├─ Process（处理）：[规则、算法、步骤、约束条件]
   ├─ Output（输出）：[具体形式、数据结构、验收标准]
   ├─ 承接的相关方需求：SR-F-001.1, SR-F-001.2
   ├─ 符合性分析：[如何满足所承接的需求]
   └─ 为下层提供的设计指导：[对系统需求的指导]
   ```

3. **同步建立下层架构骨架**
   - 在填充业务架构的同时，同步在下层（系统需求）建立功能部分架构末级节点的结构骨架
   - 边分解边映射

**质量检查**：
- [ ] 1:1分配约束满足率100%
- [ ] 每个IPO都有清晰的I/P/O定义
- [ ] IPO去重率100%
- [ ] 同步建立的系统需求架构骨架与IPO清单匹配

**输出**：
- 相关方需求功能部分详细定义
- 业务架构定义（IPO清单 + 映射关系表）
- 系统需求功能部分架构骨架（同步建立）

---

### 第3步：系统需求功能部分架构定义（2-3天）

**目标**：将业务架构IPO去重并分类组织，形成系统需求功能部分的架构定义

**参考**：设计准则 开发六步法 第3步

**具体操作**：

1. **IPO去重**
   - 对业务架构中的所有IPO进行去重
   - 确保功能完整性和无重复性

2. **功能需求架构组织**
   - 将去重后的IPO按系统功能分类组织
   - 形成系统需求功能部分架构末级节点
   ```
   SysReq-F-001: [系统功能需求名称]
   ├─ 定义：[系统需求的定义]
   ├─ 类型：功能需求
   ├─ 承接的业务架构：BA-001, BA-002
   ├─ 符合性分析：[如何满足所承接的需求]
   └─ 为下层提供的设计指导：[对产品架构的指导]
   ```

**质量检查**：
- [ ] 功能需求来自IPO去重清单（100%覆盖）
- [ ] 功能分类清晰
- [ ] 无重复功能定义

**输出**：
- 系统需求功能部分架构定义

---

### 第4步：相关方需求非功能部分详细定义及系统需求非功能部分架构定义（3-5天）

**目标**：沿非功能分支独立推进，产出系统需求非功能部分架构定义

**参考**：设计准则 开发六步法 第4步

**具体操作**：

1. **相关方需求非功能部分详细定义**
   - 对SR-NFR架构末级节点进行详细设计分解
   - 为每条非功能需求定义量化指标和验证方法
   ```
   SR-NFR-001.1: [对SR-NFR-001的分解条目]
   ├─ 描述：[详细说明]
   ├─ 量化指标：[具体的性能指标]
   ├─ 验证方法：[如何验证]
   └─ 分配到：[系统需求非功能架构末级节点SysReq-NFR-001]
   ```

2. **非功能需求架构设计**
   - 按三大类组织：质量特性、环境适应性、可实现性
   - 形成系统需求非功能部分架构末级节点
   ```
   SysReq-NFR-001: [系统非功能需求名称]
   ├─ 定义：[非功能需求的定义]
   ├─ 类型：[性能/可靠性/安全性/...]
   ├─ 承接的相关方非功能需求：SR-NFR-001.1, SR-NFR-001.2
   ├─ 符合性分析：[如何满足所承接的需求]
   └─ 为下层提供的设计指导：[对产品架构的约束]
   ```

**质量检查**：
- [ ] 所有非功能需求都有明确的量化指标
- [ ] 非功能需求分类体系清晰
- [ ] 与功能分支的接口定义清晰

**输出**：
- 相关方需求非功能部分详细定义
- 系统需求非功能部分架构定义

---

### 第5步：系统需求详细定义及产品架构定义（7-10天）

**目标**：合并功能和非功能分支，产出系统需求详细定义和产品架构定义

**参考**：设计准则 开发六步法 第5步

**具体操作**：

1. **系统需求功能部分详细定义**
   - 对SysReq-F架构末级节点进行详细设计分解至9级
   - 每条详细定义末级节点分配到产品架构末级节点（1:1约束）
   ```
   SysReq-F-001.1: [具体的系统功能需求]
   ├─ 描述：[详细说明]
   ├─ 验收标准：[验收条件]
   ├─ 分配到：[产品架构末级节点PA-001]
   └─ 符合性说明：[如何满足SysReq-F-001]
   ```

2. **产品架构定义**
   - 根据系统需求详细定义，识别前后端组件
   - 定义组件职责、实现细节、性能指标
   - 非功能需求约束产品架构的符合性
   ```
   PA-001: [产品组件名称]
   ├─ 定义：[组件定义和职责]
   ├─ 承接的系统需求：SysReq-001.1, SysReq-001.2
   ├─ 符合性分析：[如何满足所承接的需求]
   ├─ 实现细节：[技术栈、关键模块、数据存储]
   └─ 性能指标：[响应时间、吞吐量等]
   ```

**质量检查**：
- [ ] 1:1分配约束满足率100%
- [ ] 前后端组件划分清晰
- [ ] 每个组件包含实现细节和性能指标
- [ ] 非功能部分对产品架构的约束已体现

**输出**：
- 系统需求详细定义（第4层详细定义文档）
- 产品架构定义（第5层文档）

---

### 第6步：双向追溯验证（2-3天）

**目标**：验证完整的追溯链路，产出追溯矩阵和验证报告

**参考**：设计准则 开发六步法 第6步

**具体操作**：

1. **1:1分配约束验证**
   - 验证每条详细定义末级节点都分配到唯一的方案架构末级节点
   - 发现遗漏或重复的映射

2. **N:1承接支持验证**
   - 验证每个方案架构末级节点的承接数量合理（平均2-3条，最多5条）
   - 发现过度分解或过度聚合的情况

3. **双向追溯验证**
   - 正向：从原始需求追溯到产品架构
   - 反向：从产品架构追溯到原始需求
   - 验证追溯链路的完整性

4. **文档完善和发布**
   - 补充缺失的内容
   - 更新版本号和日期
   - 生成需求追溯矩阵和验证报告

**质量检查**：
- [ ] 1:1约束满足率100%
- [ ] N:1支持合理
- [ ] 映射覆盖率100%
- [ ] 符合性分析完整率100%
- [ ] 超链接有效率100%

**输出**：
- 需求追溯矩阵
- 验证报告

## ⏱️ 时间估算

| 步骤 | 时间 | 说明 |
|------|------|------|
| 第1步：原始需求分析及SR架构定义 | 3-5天 | 取决于需求数量和复杂度 |
| 第2步：SR功能详细定义及业务架构 | 5-7天 | 功能分支，取决于业务复杂度 |
| 第3步：系统需求功能架构定义 | 2-3天 | 功能分支，IPO去重 |
| 第4步：SR非功能详细定义及系统非功能架构 | 3-5天 | 非功能分支，独立并行 |
| 第5步：系统需求详细定义及产品架构 | 7-10天 | 合并分支，取决于产品复杂度 |
| 第6步：双向追溯验证 | 2-3天 | 端到端验证 |
| **总计** | **22-33天** | **约3-5周** |

> **注**：第2步和第4步可并行执行（功能分支和非功能分支解耦），实际日历时间可缩短。时间估算为串行执行情况。

## 🎯 关键检查点

### 原始需求及架构定义阶段
- [ ] 原始需求是否完全收集和规范化
- [ ] 末级需求是否原子性
- [ ] 相关方需求架构分类是否清晰
- [ ] 功能和非功能需求是否分别分类

### 功能分支阶段
- [ ] SR-F详细定义是否满足1:1约束
- [ ] IPO定义是否清晰完整
- [ ] IPO去重是否有效
- [ ] 系统需求功能架构是否完整

### 非功能分支阶段
- [ ] SR-NFR详细定义是否有量化指标
- [ ] 非功能需求分类是否合理
- [ ] 对产品架构的约束是否明确

### 合并阶段
- [ ] 系统需求详细定义是否分配到产品架构（1:1）
- [ ] 产品组件划分是否合理
- [ ] 非功能约束是否在产品架构中体现

### 验证阶段
- [ ] 1:1约束是否满足
- [ ] N:1支持是否合理
- [ ] 追溯链路是否完整

## 🚀 使用建议

### 1. 项目启动前
- 确认原始需求已完全收集
- 明确三个参与角色的分工（流水线开发者、AI、AOS开发者）
- 制定详细的项目计划

### 2. 设计过程中
- 严格按照6步流程执行
- 第2步和第4步可并行推进（功能/非功能解耦）
- 定期进行质量检查

### 3. 设计完成后
- 进行端到端追溯验证
- 生成追溯矩阵和验证报告
- 建立文档维护机制
