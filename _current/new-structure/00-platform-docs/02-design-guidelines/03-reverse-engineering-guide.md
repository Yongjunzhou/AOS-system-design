# 逆向工程指南
**Reverse Engineering Guide**

**文档版本**：v2.0  
**创建日期**：2026-05-15  
**基于准则版本**：系统设计准则 v2.3  
**适用场景**：有完整的产品代码或图纸，系统设计流水线中各技术文档质量不高或缺失，需要生成系统设计的整套技术文档

---

## 📌 场景描述

### 适用条件

- ✅ 产品已开发完成或接近完成
- ✅ 有详细的代码实现或架构图纸
- ✅ 技术文档不全或不符合设计准则
- ✅ 缺少从需求到架构的完整追溯链路
- ✅ 需要生成符合准则的系统设计文档

### 不适用条件

- ❌ 产品代码不清晰或文档极度缺失
- ❌ 原开发人员无法参与
- ❌ 需要快速完成（时间紧张）

### 参与角色

本指南涉及三个核心参与者，在逆向工程流程中各司其职：

| 角色 | 职责 | 参与环节 |
|------|------|---------|
| **流水线开发者** | 定义设计准则和指南，配置 AI 辅助工具，审核 AI 输出 | 所有步骤的审核确认 |
| **AI** | 按指令执行代码分析、需求反向推导、映射关系建立等自动化任务 | 所有步骤的自动生成和检查 |
| **AOS开发者** | 提出需求、确认设计、审核方案（含需求提出者、业务流程开发者、信息系统开发者三类） | 代码分析、文档编写、方案审核 |

## 🎯 核心目标

通过反向工程，从代码和图纸中提取设计信息，重建符合设计准则的系统设计文档

```
产品代码/图纸 → 产品架构 → 系统需求 → 业务架构 → 相关方需求 → 原始需求
   第5层        第4层       第3层       第2层       第1层
```

---

## 📋 完整操作流程（8步）

### 第1步：代码和图纸分析（1-2天）

**目标**：理解产品的整体架构和功能模块

**具体操作**：

1. **架构图纸分析**
   - 收集所有架构图纸（系统架构图、模块图、数据流图等）
   - 识别主要的系统组件和模块
   - 识别组件之间的依赖关系
   - 记录关键的技术决策和设计考虑

2. **代码结构分析**
   - 分析代码的目录结构和模块划分
   - 识别主要的功能模块和子模块
   - 分析模块之间的调用关系
   - 识别关键的业务流程和数据流

3. **功能清单提取**
   - 从代码中提取所有实现的功能
   - 按照功能模块进行分类
   - 记录每个功能的实现方式和关键参数

**输出**：
- 产品架构图（第5层）
- 功能模块清单
- 关键技术决策记录

> **分析要点**：代码分析中识别的每个功能模块对应一个或多个产品架构组件。功能模块清单中的条目将作为第2步中PA条目的输入，建议先对功能模块进行分类聚合，再映射到PA条目。

---

### 第2步：产品架构文档编写（2-3天）

**目标**：编写符合设计准则的产品架构文档

**具体操作**：

1. **产品架构条目定义**
   ```
   PA-001: [产品组件名称]
   ├─ 定义：[组件的定义]
   ├─ 描述：[实现说明]
   ├─ 实现细节：[代码位置、关键类/函数]
   ├─ 性能指标：[响应时间、吞吐量等]
   └─ 技术栈：[使用的技术和框架]
   ```

2. **产品架构文档结构**
   - 第一部分：架构设计结果（树形结构，从根节点直到末级节点）
   - 第二部分：需求映射视图（预留，将在第3步系统需求反向推导完成后填入，显示每个产品架构末级节点对应的系统需求末级节点）
   
   **文档结构说明**：产品架构文档包含**详细定义子文档**（条目化PA定义，如上述模板）和**架构定义子文档**（分类组织）。详细定义子文档在Step 2编写，架构定义子文档中的映射视图待Step 3完成后补充。

   > **同步设计参考**：上述"先编写详细定义，后补充映射视图"的方式，符合同步设计原则（详细定义与下层架构定义的同步设计）的两阶段工作流程——Step 2先建立PA详细定义条目并创建到SysReq的结构骨架（预留映射空间），Step 3在反向推导SysReq时同步填充PA的架构定义子文档中的映射视图。详见系统设计准则中"详细定义与下层架构定义的同步设计"章节。

**输出**：
- 产品架构文档（初版）

---

### 第3步：系统需求反向推导（3-5天）

**目标**：从产品架构反向推导系统需求

**具体操作**：

1. **系统需求识别**
   - 分析每个产品组件实现了什么功能
   - 识别功能性需求（做什么）
   - 识别非功能性需求（性能、安全、可靠性等）
   - 记录每个系统需求的定义和验收标准

2. **系统需求条目定义**
   ```
   SysReq-001: [系统需求名称]
   ├─ 定义：[需求定义]
   ├─ 描述：[详细说明]
   ├─ 对应的产品架构：PA-001, PA-002, ...
   ├─ 符合性分析：[如何通过产品架构实现]
   └─ 验收标准：[验收条件]
   ```

   **文档结构**：系统需求文档包含**架构定义子文档**（分类组织）和**详细定义子文档**（条目化需求），反向推导出的系统需求应同步维护两个子文档。

3. **系统需求文档结构**
   - 第一部分：架构设计结果（树形结构，从根节点直到末级节点）
   - 第二部分：需求映射视图（映射关系表，显示每个系统需求末级节点对应的产品架构末级节点）

**输出**：
- 系统需求文档（初版）

---

### 第4步：业务架构反向推导（3-5天）

**目标**：从系统需求反向推导业务架构

**具体操作**：

1. **业务架构识别**
   - 分析系统需求的业务含义
   - 识别业务组件和业务流程
   - 识别业务组件之间的关系
   - 记录每个业务组件的定义和职责

2. **业务架构条目定义**
   ```
   BA-001: [业务组件名称]
   ├─ 定义：[业务组件定义]
   ├─ 描述：[业务流程说明]
   ├─ Input/Process/Output：[IPO详细说明]
   ├─ 对应的系统需求：SysReq-001, SysReq-002, ...
   ├─ 符合性分析：[如何通过系统需求实现]
   └─ 业务流程：[关键的业务流程]
   ```

   **文档结构**：业务架构文档包含**架构定义子文档**（分类组织）和**详细定义子文档**（条目化BA/IPO定义），反向推导出的业务架构应同步维护两个子文档。

   > **业务架构独立存在**：业务架构（BA）的"详细定义子文档"本质上是架构定义的扩展内容（IPO详细定义），而不是独立于架构定义之外的详细设计。BA只有架构定义（IPO清单），没有独立的详细定义子文档。这与系统需求（SysReq）和产品架构（PA）的"详细定义是独立于架构定义的详细设计"不同。在反向推导中，BA条目应聚焦于IPO定义本身，不应引入额外的详细设计层级。

3. **业务架构文档结构**
   - 第一部分：架构设计结果（树形结构，从根节点直到末级节点IPO）
   - 第二部分：需求映射视图（映射关系表，显示每个业务架构末级节点对应的系统需求末级节点）

**输出**：
- 业务架构文档（初版）

---

### 第5步：相关方需求反向推导（2-3天）

**目标**：从业务架构反向推导相关方需求

**具体操作**：

1. **相关方需求识别**
   - 分析业务架构的相关方需求
   - 识别功能需求和非功能需求
   - 记录每条相关方需求的定义和验收标准

2. **相关方需求条目定义**
   ```
   SR-F-001: [相关方需求名称]
   ├─ 定义：[需求定义]
   ├─ 描述：[详细说明]
   ├─ 对应的业务架构：BA-001, BA-002, ...
   ├─ 符合性分析：[如何通过业务架构实现]
   └─ 验收标准：[验收条件]
   ```

   **文档结构**：相关方需求文档包含**架构定义子文档**（分类组织）和**详细定义子文档**（条目化需求），反向推导出的相关方需求应同步维护两个子文档。注意：非功能需求（SR-NF）应与功能需求（SR-F）分开处理，采用平行的非功能需求架构。

3. **相关方需求文档结构**
   - 第一部分：架构设计结果（树形结构，从根节点直到末级节点）
   - 第二部分：需求映射视图（映射关系表，显示每个相关方需求末级节点对应的业务架构末级节点）

**输出**：
- 相关方需求文档（初版）

---

### 第6步：原始需求反向推导（1-2天）

**目标**：从相关方需求反向推导原始需求

**具体操作**：

1. **原始需求识别**
   - 分析相关方需求的来源
   - 识别原始需求的末级需求
   - 记录每条原始需求的定义

2. **原始需求条目定义**
   ```
   OR-001: [原始需求名称]
   ├─ N.1：[末级需求定义]
   ├─ N.2：[末级需求定义]
   ├─ 原始需求描述：[原始需求的完整描述]
   └─ 对应的相关方需求：SR-F-001, SR-F-002, ...

   末级需求 N.1 → SR-F-002
   末级需求 N.2 → SR-F-003
   ```
   末级需求分解后，建立与相关方需求的分配关系。

3. **原始需求文档结构**
   - 第一部分：原始需求条目
   - 末级需求分解
   - 第二部分：**预留**末级需求 → 相关方需求分配说明空间（将在第5步完成后填入，建立完整追溯链路）

   > **注意**：原始需求文档的需求分配说明与其它文档的映射视图类似，需要在所有下游文档完成后才能填写完整。建议先建立结构骨架，最后补充映射内容。

**输出**：
- 原始需求文档（初版）

---

### 第7步：完整性和一致性验证（2-3天）

**目标**：验证反向工程的完整性和一致性

**具体操作**：

1. **1:1 分配约束验证**
   - 验证每条需求都分配到唯一的方案条目
   - 发现遗漏或重复的映射

2. **N:1 承接支持验证**
   - 验证每个方案条目承接的需求数量合理
   - 发现过度分解或过度聚合的情况

3. **双向追溯验证**
   - 验证所有超链接的有效性
   - 验证追溯链路的完整性

4. **符合性分析验证**
   - 验证每个方案条目的符合性分析完整
   - 验证符合性分析的逻辑清晰

**输出**：
- 验证报告
- 问题清单和改进建议

---

### 第8步：文档完善和发布（2-3天）

**目标**：完善文档，发布符合设计准则的系统设计文档

**具体操作**：

1. **文档完善**
   - 补充缺失的内容
   - 修正发现的问题
   - 优化文档结构和表述

2. **文档审查**
   - 邀请相关人员进行审查
   - 收集反馈意见
   - 进行最后的修改

3. **文档发布**
   - 确定文档版本号
   - 发布最终版本
   - 建立文档维护机制

**输出**：
- 完整的系统设计文档集（5层）
- 需求映射矩阵
- 验证报告

## ⏱️ 时间估算

| 步骤 | 时间 | 说明 |
|------|------|------|
| 第1步：代码和图纸分析 | 1-2天 | 取决于产品复杂度 |
| 第2步：产品架构文档 | 2-3天 | 取决于产品规模 |
| 第3步：系统需求反向推导 | 3-5天 | 取决于需求复杂度 |
| 第4步：业务架构反向推导 | 3-5天 | 取决于业务复杂度 |
| 第5步：相关方需求反向推导 | 2-3天 | 取决于相关方数量 |
| 第6步：原始需求反向推导 | 1-2天 | 通常较快 |
| 第7步：完整性和一致性验证 | 2-3天 | 取决于问题数量 |
| 第8步：文档完善和发布 | 2-3天 | 取决于反馈数量 |
| **总计** | **16-26天** | **约3-4周** |

## 🎯 关键检查点

### 代码和图纸分析阶段
- [ ] 所有架构图纸是否已收集和分析
- [ ] 代码结构是否已全面审查
- [ ] 功能模块清单是否完整

### 产品架构阶段
- [ ] PA条目是否准确反映了代码实现
- [ ] 技术栈和实现细节是否完整
- [ ] 组件划分是否合理

### 系统需求阶段
- [ ] 从PA反向推导的SysReq是否完整
- [ ] 功能需求和非功能需求是否都已识别
- [ ] PA ↔ SysReq映射关系是否正确

### 业务架构阶段
- [ ] 从SysReq反向推导的BA是否合理
- [ ] IPO定义是否清晰
- [ ] SysReq ↔ BA映射关系是否正确

### 相关方需求阶段
- [ ] 功能需求与非功能需求是否分离
- [ ] 反向推导的SR是否覆盖所有BA
- [ ] 映射关系是否正确

### 原始需求阶段
- [ ] 从SR反向推导的原始需求是否完整
- [ ] 末级需求分解是否原子性
- [ ] 追溯链路是否完整

### 验证阶段
- [ ] 1:1约束是否满足
- [ ] N:1支持是否合理
- [ ] 双向追溯链路是否完整

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 代码文档不清晰 | 难以理解设计意图 | 邀请原开发人员参与，进行代码审查 |
| 架构图纸过时 | 反向推导的架构不准确 | 与代码实现进行对比，发现差异 |
| 需求信息遗漏 | 反向推导的需求不完整 | 进行用户访谈，补充缺失的需求 |
| 映射关系复杂 | 难以建立清晰的映射 | 进行需求分解，简化映射关系 |

## 📝 使用建议

1. **项目启动前**
   - 收集所有代码和图纸
   - 组建跨职能的分析团队
   - 邀请原开发人员参与

2. **分析过程中**
   - 严格按照8步流程执行
   - 定期进行质量检查
   - 及时处理发现的问题

3. **分析完成后**
   - 进行完整性和一致性验证
   - 生成需求映射矩阵
   - 建立文档维护机制
