# 系统设计准则审查报告

**审查日期**：2026-05-14  
**审查范围**：系统设计准则的一致性和系统性  
**审查结果**：发现多个需要修订的问题

---

## 🔍 发现的问题

### 问题1：架构末级节点的定义不够清晰

**位置**：第2层（相关方需求架构）、第4层（系统需求架构）

**问题描述**：
- 相关方需求架构的末级节点定义为"能够承接原始需求末级节点"
- 但没有明确说明：这个末级节点具体是什么？是分类节点吗？还是相关方需求条目本身？
- 系统需求架构的末级节点定义为"业务架构中的 IPO"
- 但这意味着系统需求架构的末级节点就是业务架构的末级节点，那系统需求架构的作用是什么？

**影响**：
- 不清楚如何设计相关方需求架构
- 不清楚相关方需求架构和相关方需求（详细设计）的关系

### 问题2：架构和详细设计的关系不清晰

**位置**：第2层和第2.5层、第4层和第4.5层

**问题描述**：
- 相关方需求架构（第2层）的末级节点和相关方需求（第2.5层）的末级节点是什么关系？
- 是一对一还是一对多？
- 系统需求架构（第4层）的末级节点和系统需求（第4.5层）的末级节点是什么关系？

**影响**：
- 不清楚如何从架构末级节点过渡到详细设计末级节点
- 不清楚映射关系的完整链路

### 问题3：映射关系的描述不够完整

**位置**：核心约束和规则部分

**问题描述**：
- 规则2中说"原始需求末级 → 相关方需求架构末级（N:1）"
- 但没有说明"相关方需求架构末级 → 相关方需求末级"的关系
- 这导致映射链路不完整

**影响**：
- 不清楚完整的映射链路是什么
- 不清楚如何进行需求追溯

### 问题4：相关方需求架构的作用不明确

**位置**：第2层（相关方需求架构）

**问题描述**：
- 相关方需求架构被定义为"对相关方需求的分类和组织"
- 但相关方需求本身也是对原始需求的分类和组织
- 那相关方需求架构和相关方需求有什么区别？

**影响**：
- 不清楚为什么需要相关方需求架构
- 不清楚相关方需求架构的实际用途

### 问题5：系统需求架构的末级节点定义有矛盾

**位置**：第4层（系统需求架构）

**问题描述**：
- 说"功能需求末级节点 = 业务架构中的 IPO"
- 但业务架构是第3层，系统需求架构是第4层
- 这意味着系统需求架构的末级节点就是业务架构的末级节点
- 那系统需求架构的作用是什么？只是对IPO的分类吗？

**影响**：
- 不清楚系统需求架构的实际含义
- 不清楚系统需求架构和业务架构的关系

### 问题6：质量指标的一致性问题

**位置**：各层的质量指标

**问题描述**：
- 第2层（相关方需求架构）的质量指标说"平均每个架构末级节点承接原始需求末级数：2-3 条"
- 第2.5层（相关方需求）的质量指标说"平均每个相关方需求承接原始需求末级数：2-3 条"
- 这两个指标是什么关系？是否重复了？

**影响**：
- 不清楚应该如何评估设计质量
- 不清楚两个指标是否应该同时满足

### 问题7：条目化需求分配的对象不清晰

**位置**：规则3（双向追溯）

**问题描述**：
- 说"条目化需求分配到架构末级节点"
- 但没有明确说明：条目化需求是什么？是相关方需求条目吗？还是系统需求条目？
- 架构末级节点是什么？是相关方需求架构末级节点吗？还是业务架构末级节点？

**影响**：
- 不清楚如何进行需求分配
- 不清楚映射关系的具体含义

---

## 💡 根本原因分析

我认为这些问题的根本原因是：

1. **对"架构"的理解还不够深入**
   - 架构应该是什么？是分类体系吗？还是组织结构？
   - 架构末级节点应该是什么？是分类节点吗？还是具体的条目？

2. **对"架构和详细设计的关系"理解不清**
   - 架构末级节点和详细设计末级节点是什么关系？
   - 是一对一还是一对多？

3. **对"映射关系的完整链路"理解不完整**
   - 从原始需求到产品架构的完整映射链路是什么？
   - 每一层之间的映射关系是什么？

## 🔧 修订建议

### 建议1：明确架构末级节点的定义

**修订内容**：

对于相关方需求架构：
- 末级节点应该是相关方需求的**分类节点**
- 这些分类节点对应相关方需求（详细设计）中的多个需求条目
- 一个相关方需求架构末级节点可能对应多个相关方需求条目（1:N）

对于系统需求架构：
- 功能需求末级节点应该是**功能的分类节点**（不是IPO本身）
- 这些分类节点对应系统需求（详细设计）中的多个需求条目
- 非功能需求末级节点应该是**非功能需求的分类节点**

### 建议2：明确架构和详细设计的关系

**修订内容**：

- 架构末级节点是对需求的**分类和组织**
- 详细设计末级节点是对架构末级节点的**具体说明**
- 一个架构末级节点可能对应多个详细设计末级节点（1:N）
- 多个详细设计末级节点可能分配到一个下层方案的架构末级节点（N:1）

### 建议3：完整描述映射关系链路

**修订内容**：

```
原始需求末级 ──N:1──→ 相关方需求架构末级（分类）
    ↓
相关方需求架构末级 ──1:N──→ 相关方需求末级（条目）
    ↓
相关方需求末级 ──N:1──→ 业务架构末级（IPO）
    ↓
业务架构末级 ──去重后1:1──→ 系统需求架构末级（分类）
    ↓
系统需求架构末级 ──1:N──→ 系统需求末级（条目）
    ↓
系统需求末级 ──N:1──→ 产品架构末级（组件）
```

### 建议4：明确条目化需求分配的对象

**修订内容**：

- 条目化需求 = 详细设计末级节点（相关方需求条目、系统需求条目等）
- 架构末级节点 = 架构层的末级节点（相关方需求架构末级、业务架构末级等）
- 条目化需求分配到**下层方案的架构末级节点**，而不是详细设计末级节点

### 建议5：调整质量指标

**修订内容**：

对于相关方需求架构：
- 架构末级节点数：通常 5-10 个（分类节点）
- 平均每个架构末级节点对应相关方需求条目数：2-4 条

对于相关方需求（详细设计）：
- 相关方需求条目数：通常 20-40 条
- 平均每个相关方需求条目承接原始需求末级数：2-3 条
