# 产品开发系统设计准则
**System Design Standards for Product Development**

**文档版本**：v1.0  
**创建日期**：2026-05-13  
**目标受众**：产品经理、架构师、需求分析师、开发团队

---

## 📌 准则概览

本准则基于**系统设计流水线方法论**，为产品开发的系统设计过程提供可操作的指导。通过建立从原始需求到产品架构的完整设计链路，确保需求的完整性、方案的充分性和设计的一致性。

**核心目标**：
- ✅ 确保每条需求都有明确的承接点
- ✅ 确保每个方案都充分满足上层需求
- ✅ 建立完整的端到端追溯链路
- ✅ 发现并消除设计缺陷

---

## 🏗️ 系统设计流水线的5层结构

### 层级定义

```
第1层：原始需求及分析
    ↓ 作为需求
第2层：相关方需求（方案）
    ├─ 功能部分 ↓ 作为需求
    │   第3层：业务架构（方案）
    │       ↓ 作为需求
    │       第4层：系统需求-功能部分（方案）
    │           ↓ 作为需求
    │           第5层：产品架构（方案）
    │
    └─ 非功能部分 ↓ 作为需求
        第4层：系统需求-非功能部分（方案）
            ↓ 作为需求
            第5层：产品架构（方案）
```

### 每层的角色定义

| 层级 | 报告名称 | 角色 | 承接关系 | 输出形式 |
|------|---------|------|---------|---------|
| 1 | 原始需求及分析 | 需求 | - | 需求清单 |
| 2 | 相关方需求 | 原始需求的方案 + 后续的需求 | 承接原始需求 | 条目化需求 + 映射说明 |
| 3 | 业务架构 | 相关方需求功能部分的方案 + 后续的需求 | 承接相关方需求功能部分 | 条目化方案 + 映射说明 |
| 4 | 系统需求 | 业务架构的方案 + 相关方需求非功能部分的方案 + 后续的需求 | 承接BA + 相关方需求非功能部分 | 条目化需求 + 映射说明 |
| 5 | 产品架构 | 系统需求的方案 | 承接系统需求 | 条目化方案 + 实现细节 |

## 🎯 核心约束和规则

### 规则1：1:1 分配约束

**定义**：需求中的每一条都应分配至方案中**唯一一个条目**。

**目的**：
- 确保每条需求都有明确的承接点
- 发现需求分解不足或方案设计不完整的问题
- 防止需求遗漏或重复分配

**检查方法**：
```
对于每条上层需求：
  ✓ 是否分配到了下层方案的某个条目？
  ✓ 是否只分配到了唯一一个条目？
  ✓ 是否有多个条目都承接了这条需求？
```

**常见问题**：
- ❌ 一条需求分配到多个方案条目 → 说明需求分解不足
- ❌ 一条需求无法分配到任何方案条目 → 说明方案设计不完整
- ❌ 一条需求分配到的方案条目不清晰 → 说明需求或方案定义不够具体

### 规则2：N:1 承接支持

**定义**：方案中一个条目可以承接需求的**多个条目**。

**目的**：
- 支持需求的聚合和整合
- 体现设计的高效性和合理性
- 避免过度分解导致的复杂性

**检查方法**：
```
对于每个下层方案条目：
  ✓ 承接了多少条上层需求？
  ✓ 这些需求之间是否有逻辑关联？
  ✓ 承接的需求数量是否合理（通常2-4条）？
```

**合理范围**：
- 最少：1条需求（某些特殊需求可能独立承接）
- 最多：4-5条需求（超过5条说明方案条目定义过于宽泛）
- 平均：2-3条需求（表示设计的合理聚合）

### 规则3：双向追溯

**定义**：支持从上层需求追溯到下层方案，也支持从下层方案反向追溯到上层需求。

**实现方式**：
- 在每个方案条目中标注"承接了哪些上层需求"
- 在每条上层需求中标注"分配到哪个下层方案条目"
- 通过超链接实现快速导航

**检查方法**：
```
对于每条需求：
  ✓ 能否快速找到它分配到的方案条目？
  ✓ 能否从方案条目反向找到它？

对于每个方案条目：
  ✓ 能否清晰看到它承接了哪些需求？
  ✓ 能否从需求反向找到这个方案条目？
```

## 📋 每层的具体要求

### 第1层：原始需求及分析

**内容结构**：
```
原始需求 1
├─ 需求描述：[具体的需求定义]
├─ 末级需求分解：
│  ├─ 末级需求 1.1 → [分配到相关方需求]
│  ├─ 末级需求 1.2 → [分配到相关方需求]
│  └─ 末级需求 1.3 → [分配到相关方需求]
└─ 优先级：[高/中/低]
```

**关键要求**：
- ✅ 每条原始需求都应分解到末级需求
- ✅ 末级需求应该是原子性的、不可再分的
- ✅ 每条末级需求都应分配到相关方需求的某个条目
- ✅ 文档中只保留纯需求清单，去除解释性内容

**质量指标**：
- 末级需求平均分解数：2-3条（表示中粒度分解）
- 映射覆盖率：100%（所有末级需求都被映射）

### 第2层：相关方需求

**内容结构**：

**第一部分：条目化的相关方需求**
```
SR-F-001: [需求名称]

├─ 验收标准：[该需求的验收条件]
├─ 承接的原始需求：
│  ├─ 原始需求 1.1
│  ├─ 原始需求 1.2
│  └─ 原始需求 1.3
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对业务架构的指导]
```

**第二部分：原始需求分配说明**
```
原始需求 1.1
├─ 分配到：SR-F-001
├─ 符合性说明：[SR-F-001 如何满足这条需求]
└─ 超链接：[指向 SR-F-001 的链接]
```

**关键要求**：
- ✅ 功能需求和非功能需求分别标识
- ✅ 每条相关方需求都包含完整的内容结构
- ✅ 符合性分析应该清晰、具体
- ✅ 建立双向追溯的超链接

**质量指标**：
- 相关方需求总数：通常 20-40 条
- 平均每个相关方需求承接末级需求数：2-3 条
- 符合性分析完整率：100%

## ✅ 操作检查清单

### 需求分解检查清单

- [ ] 每条原始需求都分解到末级需求
- [ ] 末级需求是原子性的、不可再分的
- [ ] 末级需求的粒度合理（平均分解数 2-3 条）
- [ ] 每条末级需求都有清晰的定义和验收标准
- [ ] 每条末级需求都分配到相关方需求的某个条目
- [ ] 没有遗漏或重复的映射

### 需求映射检查清单

- [ ] 每条上层需求都分配到下层方案的唯一一个条目（1:1 约束）
- [ ] 每个下层方案条目都承接了上层的多条需求（N:1 支持）
- [ ] 承接的需求数量合理（通常 2-4 条）
- [ ] 承接的需求之间有逻辑关联
- [ ] 映射关系符合业务逻辑

### 符合性分析检查清单

- [ ] 每个方案条目都包含符合性分析
- [ ] 符合性分析清晰、具体、可验证
- [ ] 符合性分析说明了如何满足所承接的需求
- [ ] 符合性分析没有遗漏任何承接的需求
- [ ] 符合性分析的逻辑清晰、易于理解

### 双向追溯检查清单

- [ ] 每条上层需求都标注了分配到的下层方案条目
- [ ] 每个下层方案条目都标注了承接的上层需求
- [ ] 所有超链接格式正确
- [ ] 所有超链接都指向有效的锚点
- [ ] 所有超链接都指向正确的目标
- [ ] 用户能够快速导航和追溯

### 文档规范化检查清单

- [ ] 文档结构清晰、一致
- [ ] 每层报告都包含两个部分（条目化内容 + 映射说明）
- [ ] 文档中只保留必要的内容，去除冗余信息
- [ ] 文档易于维护和更新
- [ ] 文档版本号正确更新

## 🔍 常见问题和最佳实践

### Q1：如何判断需求分解是否充分？

**答**：
- ✅ 末级需求应该是原子性的、不可再分的
- ✅ 末级需求应该能够分配到下层方案的某个条目
- ✅ 末级需求的粒度应该合理（平均分解数 2-3 条）
- ❌ 如果末级需求无法分配到任何方案条目，说明分解不足

**最佳实践**：
- 使用"能否进一步分解"的问题来判断是否已经到达末级
- 使用"是否能够分配到下层方案"的问题来验证分解的充分性

### Q2：如何判断方案设计是否完整？

**答**：
- ✅ 所有上层需求都应该分配到下层方案的某个条目
- ✅ 没有遗漏的需求
- ✅ 没有无法承接的需求
- ❌ 如果某条需求无法分配到任何方案条目，说明方案设计不完整

**最佳实践**：
- 使用"是否所有需求都被承接"的问题来验证方案的完整性
- 使用需求映射矩阵来发现遗漏的映射

### Q3：一个方案条目应该承接多少条需求？

**答**：
- 最少：1 条需求（某些特殊需求可能独立承接）
- 最多：4-5 条需求（超过 5 条说明方案条目定义过于宽泛）
- 平均：2-3 条需求（表示设计的合理聚合）

**最佳实践**：
- 如果某个方案条目承接超过 5 条需求，考虑是否需要进一步分解
- 如果某个方案条目只承接 1 条需求，考虑是否可以与其他条目合并

### Q4：如何编写有效的符合性分析？

**答**：符合性分析应该包含以下内容：
1. **承接的需求清单**：列出所有承接的需求
2. **满足方式**：说明如何满足每条需求
3. **验证方法**：说明如何验证符合性
4. **特殊说明**：说明是否有任何限制或特殊情况

**最佳实践**：
```
符合性分析示例：
- 承接需求 SR-F-001：通过提供用户登录功能满足
  验证方法：用户能够使用用户名和密码登录系统
- 承接需求 SR-F-002：通过提供权限管理功能满足
  验证方法：系统能够根据用户角色限制功能访问
```

### Q5：如何建立有效的双向追溯？

**答**：
1. **在方案条目中标注承接的需求**：
   ```
   BA-001: 用户管理
   承接的相关方需求：SR-F-001, SR-F-002, SR-F-003
   ```

2. **在需求中标注分配的方案条目**：
   ```
   SR-F-001: 用户登录
   分配到：[BA-001](business-architecture.md#ba-001)
   ```

3. **使用超链接实现快速导航**：
   - 格式：`[文本](文件名.md#锚点)`
   - 确保锚点与目标文档中的 ID 对应

**最佳实践**：
- 使用一致的超链接格式
- 定期验证所有超链接的有效性
- 在文档更新时同时更新相关的超链接

## 🎓 应用场景和示例

### 场景1：新产品开发

**流程**：
1. 收集原始需求（第1层）
2. 设计相关方需求（第2层）
3. 设计业务架构（第3层）
4. 设计系统需求（第4层）
5. 设计产品架构（第5层）

**关键点**：
- 每层都要进行 1:1 分配约束验证
- 每层都要进行 N:1 承接支持验证
- 每层都要建立双向追溯

### 场景2：需求变更管理

**流程**：
1. 识别变更的原始需求
2. 追溯到相关的相关方需求
3. 追溯到相关的业务架构
4. 追溯到相关的系统需求
5. 追溯到相关的产品架构
6. 评估变更的影响范围

**关键点**：
- 使用双向追溯快速定位影响范围
- 使用 1:1 约束验证变更的完整性
- 使用符合性分析验证变更的有效性

### 场景3：设计审查

**流程**：
1. 检查每层的 1:1 分配约束
2. 检查每层的 N:1 承接支持
3. 检查每层的符合性分析
4. 检查双向追溯的完整性
5. 检查文档的规范化程度

**关键点**：
- 使用检查清单进行系统的审查
- 使用质量指标进行量化评价
- 使用需求映射矩阵发现问题
