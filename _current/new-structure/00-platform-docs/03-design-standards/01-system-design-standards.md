# 产品开发系统设计准则（修订版）
**System Design Standards for Product Development - Revised**

**文档版本**：v2.0  
**创建日期**：2026-05-14  
**目标受众**：产品经理、架构师、需求分析师、开发团队

---

## 📌 准则概览

本准则基于**系统设计流水线方法论**，为产品开发的系统设计过程提供可操作的指导。通过建立从原始需求到产品架构的完整设计链路，确保需求的完整性、方案的充分性和设计的一致性。

**核心目标**：
- ✅ 确保每条需求都有明确的承接点
- ✅ 确保每个方案都充分满足上层需求
- ✅ 建立完整的端到端追溯链路
- ✅ 发现并消除设计缺陷

---

## 🏗️ 系统设计流水线的5层结构

### 核心理念

系统设计流水线采用**架构 + 详细设计**的分离模式。每份技术文档都包含两个层级：

1. **架构末级节点**：被分类和组织的**最大颗粒度**的需求/方案条目
2. **详细设计末级节点**：对架构末级节点的**进一步分解**

分解的标准是：**一直分解到任意末级条目都只需找到后续文档架构的一个末级节点来承接即可**。

这样可以：
- ✅ 清晰地区分高层设计和具体实现
- ✅ 支持多个团队的并行工作
- ✅ 便于需求的追溯和映射
- ✅ 提高设计的可维护性

### 层级定义

```
第1层：原始需求及分析
    ↓ N:1 分配
第2层：相关方需求
    ├─ 架构末级节点：最大颗粒度的相关方需求条目
    ├─ 详细设计末级节点：对架构末级节点的进一步分解
    │
    ├─ 功能部分 ↓ N:1 分配
    │   第3层：业务架构
    │   ├─ 架构末级节点：最大颗粒度的IPO（业务组件）
    │   ├─ 详细设计末级节点：对IPO的I、P、O详细说明
    │   │
    │   ├─ 去重后 ↓ N:1 分配
    │   │   第4层：系统需求
    │   │   ├─ 架构末级节点：最大颗粒度的系统需求条目
    │   │   ├─ 详细设计末级节点：对系统需求的进一步分解
    │   │   │
    │   │   └─ ↓ N:1 分配
    │   │       第5层：产品架构
    │   │       ├─ 架构末级节点：最大颗粒度的产品组件
    │   │       └─ 详细设计末级节点：对产品组件的实现细节
    │
    └─ 非功能部分 ↓ N:1 分配
        第4层：系统需求（非功能）
        └─ ↓ N:1 分配
            第5层：产品架构
```

### 每层的角色定义

| 层级 | 报告名称 | 架构末级节点 | 详细设计末级节点 | 承接关系 |
|------|---------|------------|----------------|---------|
| 1 | 原始需求及分析 | - | 末级需求 | - |
| 2 | 相关方需求 | 相关方需求条目 | 对条目的分解 | 承接原始需求末级（N:1） |
| 3 | 业务架构 | IPO（业务组件） | I、P、O详细说明 | 承接相关方需求详细设计末级（N:1） |
| 4 | 系统需求 | 系统需求条目 | 对条目的分解 | 承接业务架构详细设计末级（N:1） |
| 5 | 产品架构 | 产品组件 | 实现细节 | 承接系统需求详细设计末级（N:1） |

## 🎯 核心约束和规则

### 规则1：1:1 分配约束

**定义**：需求中的每一条都应分配至方案中**唯一一个条目**。

**应用范围**：
- 相关方需求详细设计末级 → 业务架构末级（1:1）
- 业务架构详细设计末级 → 系统需求架构末级（1:1）
- 系统需求详细设计末级 → 产品架构末级（1:1）

**判断标准**：
- 如果一个条目需要分配到下层方案的**多个**架构末级节点 → 说明这个条目还需要进一步分解
- 如果一个条目只需要分配到下层方案的**一个**架构末级节点 → 说明这个条目已经是详细设计末级了

### 规则2：N:1 承接支持

**定义**：方案中一个条目可以承接需求的**多个条目**。

**应用范围**：
- 原始需求末级 → 相关方需求架构末级（N:1）
- 相关方需求详细设计末级 → 业务架构末级（N:1）
- 业务架构详细设计末级 → 系统需求架构末级（N:1）
- 系统需求详细设计末级 → 产品架构末级（N:1）

**合理范围**：
- 最少：1条需求（某些特殊需求可能独立承接）
- 最多：4-5条需求（超过5条说明方案条目定义过于宽泛）
- 平均：2-3条需求（表示设计的合理聚合）

### 规则3：双向追溯

**定义**：支持从上层需求追溯到下层方案，也支持从下层方案反向追溯到上层需求。

**实现方式**：
- 在每个方案条目中标注"承接了哪些上层需求"
- 在每条上层需求中标注"分配到哪个下层方案条目"
- 通过超链接实现快速导航

**关键原则**：
- ✅ 条目化需求分配到**下层方案的架构末级节点**
- ✅ 这样可以保持需求和方案的对应关系清晰
- ✅ 架构末级节点是需求映射的关键层级

## 📋 每层的具体要求

### 第1层：原始需求及分析

**内容结构**：
```
原始需求 1
├─ 需求描述：[具体的需求定义]
├─ 末级需求分解：
│  ├─ 末级需求 1.1 → [分配到相关方需求架构末级]
│  ├─ 末级需求 1.2 → [分配到相关方需求架构末级]
│  └─ 末级需求 1.3 → [分配到相关方需求架构末级]
└─ 优先级：[高/中/低]
```

**关键要求**：
- ✅ 每条原始需求都应分解到末级需求
- ✅ 末级需求应该是原子性的、不可再分的
- ✅ 每条末级需求都应分配到相关方需求架构末级节点
- ✅ 文档中只保留纯需求清单，去除解释性内容

**质量指标**：
- 末级需求平均分解数：2-3条（表示中粒度分解）
- 映射覆盖率：100%（所有末级需求都被映射）

---

### 第2层：相关方需求

**相关方需求的本质**：

相关方需求是对原始需求的方案。它包含两个层级：
- **架构末级节点**：被分类和组织的最大颗粒度的相关方需求条目
- **详细设计末级节点**：对架构末级节点的进一步分解

**分解标准**：

一直分解到任意末级条目都只需找到**业务架构的一个末级节点（IPO）**来承接即可。

**内容结构**：

**第一部分：相关方需求架构末级节点**
```
SR-F-001: [相关方需求名称]
├─ 定义：[需求的定义]
├─ 承接的原始需求末级：
│  ├─ 原始需求 1.1
│  ├─ 原始需求 1.2
│  └─ 原始需求 1.3
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对业务架构的指导]
```

**第二部分：相关方需求详细设计末级节点**
```
SR-F-001.1: [具体的功能需求]
├─ 描述：[详细说明]
├─ 验收标准：[验收条件]
├─ 分配到：[业务架构末级节点]
└─ 符合性说明：[如何满足相关方需求]
```

**第三部分：原始需求分配说明**
```
原始需求 1.1
├─ 分配到：SR-F-001
├─ 符合性说明：[SR-F-001 如何满足这条需求]
└─ 超链接：[指向 SR-F-001 的链接]
```

**关键要求**：
- ✅ 功能需求和非功能需求分别标识
- ✅ 架构末级节点是最大颗粒度的相关方需求条目
- ✅ 详细设计末级节点分解到只需分配到业务架构的一个末级节点
- ✅ 建立双向追溯的超链接

**质量指标**：
- 相关方需求架构末级节点数：通常 10-20 个
- 平均每个架构末级节点承接原始需求末级数：2-3 条
- 相关方需求详细设计末级节点数：通常 20-40 个
- 平均每个详细设计末级节点分配到业务架构末级数：1 个（1:1 约束）
- 映射覆盖率：100%（所有原始需求末级都被映射）

### 第3层：业务架构

**业务架构的本质**：

业务架构是**运行概念**（Operational Concept），是用户使用待开发产品开展业务的过程说明。它包含两个层级：
- **架构末级节点**：最大颗粒度的IPO（业务组件）
- **详细设计末级节点**：对IPO的I、P、O详细说明

**IPO模型（Input-Process-Output）**：

业务架构的架构末级节点应该是一个IPO模型：
- **Input（输入）**：产品和/或系统接收的输入
- **Process（处理）**：对输入进行的处理
- **Output（输出）**：生成的输出

每个IPO对应一个系统功能。

**IPO的颗粒度**：

IPO的体量主要取决于所期望输出物的颗粒度。例如：
- 对于系统设计流水线产品，期望的输出是各技术文档
- 那么一个IPO应该对应一份技术文档（或文档的一个可以由单人完成的独立部分）

**内容结构**：

**第一部分：业务架构末级节点（IPO清单）**
```
BA-001: [IPO名称/业务组件名称]
├─ 定义：[业务组件的定义]
├─ 承接的相关方需求详细设计末级：
│  ├─ SR-F-001.1
│  ├─ SR-F-001.2
│  └─ SR-F-002.1
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对系统需求的指导]
```

**第二部分：业务架构详细设计末级节点（I、P、O详细说明）**
```
BA-001-Input: [输入的具体形式、数据结构、来源]
BA-001-Process: [处理的规则、算法、步骤、约束条件]
BA-001-Output: [输出的具体形式、数据结构、验收标准]
```

**第三部分：相关方需求分配说明**
```
SR-F-001.1
├─ 分配到：BA-001
├─ 符合性说明：[BA-001 如何满足这条需求]
└─ 超链接：[指向 BA-001 的链接]
```

**关键要求**：
- ✅ 业务架构末级节点是最大颗粒度的IPO
- ✅ 每个IPO都应该有清晰的Input、Process、Output定义
- ✅ 详细设计末级节点分解到只需分配到系统需求的一个末级节点
- ✅ 建立与相关方需求的映射关系（N:1支持）
- ✅ 为系统需求提供清晰的设计指导

**IPO的去重机制**：

在业务架构设计时，当在某个分支需要一个IPO时，应该：
1. **首先检查** - 现有业务架构中是否已有相同或相似的IPO
2. **复用优先** - 如果存在，直接复用该IPO
3. **改进次之** - 如果现有IPO改进后可用，则改进后复用
4. **新增最后** - 确实需要新的IPO才新增

**质量指标**：
- 业务架构末级节点数（IPO数）：通常 10-20 个
- 平均每个IPO承接相关方需求详细设计末级数：2-3 条（N:1支持）
- 1:1 分配约束满足率：100%（每条相关方需求详细设计末级分配到唯一的IPO）
- IPO详细程度：能指导系统需求定义，能与用户沟通

### 第4层：系统需求

**系统需求的本质**：

系统需求是业务架构和相关方需求非功能部分的方案。它包含两个层级：
- **架构末级节点**：被分类和组织的最大颗粒度的系统需求条目
- **详细设计末级节点**：对架构末级节点的进一步分解

**分解标准**：

一直分解到任意末级条目都只需找到**产品架构的一个末级节点（产品组件）**来承接即可。

**系统需求的来源**：

1. **功能需求来源** - 业务架构中所有IPO的去重清单
   - 多个业务流程中重复的IPO，在系统需求中只需定义一次
   - 这确保了系统功能的完整性和无重复性

2. **非功能需求来源** - 相关方需求中的非功能部分
   - 直接承接相关方需求的非功能需求

**内容结构**：

**第一部分：系统需求架构末级节点**
```
SysReq-001: [系统需求名称]
├─ 定义：[系统需求的定义]
├─ 类型：[功能需求/非功能需求]
├─ 承接的业务架构/相关方需求：
│  ├─ BA-001
│  ├─ BA-002
│  └─ SR-NF-001
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对产品架构的指导]
```

**第二部分：系统需求详细设计末级节点**
```
SysReq-001.1: [具体的系统需求]
├─ 描述：[详细说明]
├─ 验收标准：[验收条件]
├─ 分配到：[产品架构末级节点]
└─ 符合性说明：[如何满足系统需求]
```

**第三部分：业务架构/相关方需求分配说明**
```
BA-001
├─ 分配到：SysReq-001
├─ 符合性说明：[SysReq-001 如何满足这个IPO]
└─ 超链接：[指向 SysReq-001 的链接]
```

**关键要求**：
- ✅ 系统需求架构末级节点是最大颗粒度的系统需求条目
- ✅ 功能需求和非功能需求分别处理
- ✅ 功能需求来自业务架构IPO的去重清单
- ✅ 详细设计末级节点分解到只需分配到产品架构的一个末级节点
- ✅ 每条系统需求都应该有清晰的定义和验收标准
- ✅ 建立与上层需求的映射关系（N:1支持）

**质量指标**：
- 系统需求架构末级节点数：通常 10-20 个
- 平均每个架构末级节点承接业务架构/相关方需求数：2-3 条
- 系统需求详细设计末级节点数：通常 20-40 个
- 平均每个详细设计末级节点分配到产品架构末级数：1 个（1:1 约束）
- 1:1 分配约束满足率：100%（每条业务架构/相关方需求分配到唯一的系统需求）
- 功能需求去重率：100%（没有重复的功能定义）

### 第5层：产品架构

**产品架构的本质**：

产品架构是系统需求的方案。它包含两个层级：
- **架构末级节点**：最大颗粒度的产品组件
- **详细设计末级节点**：对产品组件的实现细节

**内容结构**：

**第一部分：产品架构末级节点**
```
PA-001: [产品组件名称]
├─ 定义：[产品组件的定义]
├─ 承接的系统需求详细设计末级：
│  ├─ SysReq-001.1
│  ├─ SysReq-001.2
│  └─ SysReq-002.1
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现细节：[具体的技术实现方案]
```

**第二部分：产品架构详细设计末级节点**
```
PA-001-Detail: [实现细节、性能指标、技术栈等]
```

**第三部分：系统需求分配说明**
```
SysReq-001.1
├─ 分配到：PA-001
├─ 符合性说明：[PA-001 如何满足这条需求]
└─ 超链接：[指向 PA-001 的链接]
```

**关键要求**：
- ✅ 产品架构末级节点是最大颗粒度的产品组件
- ✅ 每个产品组件都应该有清晰的实现方案
- ✅ 包含具体的技术细节和性能指标
- ✅ 建立与系统需求的映射关系（N:1支持）

**质量指标**：
- 产品架构末级节点数：通常 15-30 个
- 平均每个产品组件承接系统需求详细设计末级数：1-2 条
- 1:1 分配约束满足率：100%（每条系统需求详细设计末级分配到唯一的产品组件）

## ✅ 操作检查清单

### 分解充分性检查清单

- [ ] 每条上层需求都分解到末级需求
- [ ] 末级需求是原子性的、不可再分的
- [ ] 末级需求的粒度合理（平均分解数 2-3 条）
- [ ] 每条末级需求都有清晰的定义和验收标准
- [ ] 每条末级需求都分配到下层方案的一个架构末级节点
- [ ] 没有遗漏或重复的映射

### 分解标准检查清单

- [ ] 相关方需求详细设计末级节点只需分配到业务架构的一个末级节点
- [ ] 业务架构详细设计末级节点只需分配到系统需求的一个末级节点
- [ ] 系统需求详细设计末级节点只需分配到产品架构的一个末级节点
- [ ] 如果某个条目需要分配到多个下层架构末级节点，说明还需要进一步分解

### 映射关系检查清单

- [ ] 每条上层需求都分配到下层方案的唯一一个条目（1:1 约束）
- [ ] 每个下层方案条目都承接了上层的多条需求（N:1 支持）
- [ ] 承接的需求数量合理（通常 2-4 条）
- [ ] 承接的需求之间有逻辑关联
- [ ] 映射关系符合业务逻辑

### 符合性分析检查清单

- [ ] 每个方案条目都包含符合性分析
- [ ] 符合性分析清晰、具体、可验证
- [ ] 符合性分析说明了如何满足所承接的需求
- [ ] 符合性分析没有遗漏任何承接的需求
