# 产品开发系统设计准则
**System Design Standards for Product Development**

**文档版本**：v1.0  
**创建日期**：2026-05-13  
**目标受众**：产品经理、架构师、需求分析师、开发团队

---

## 📌 准则概览

本准则基于**系统设计流水线方法论**，为产品开发的系统设计过程提供可操作的指导。通过建立从原始需求到产品架构的完整设计链路，确保需求的完整性、方案的充分性和设计的一致性。

**核心目标**：
- ✅ 确保每条需求都有明确的承接点
- ✅ 确保每个方案都充分满足上层需求
- ✅ 建立完整的端到端追溯链路
- ✅ 发现并消除设计缺陷

---

## 🏗️ 系统设计流水线的7层结构

### 核心理念

系统设计流水线采用**架构 + 详细设计**的分离模式。每份技术文档都应该先有架构（高层结构和组织方式），再有详细设计（对架构的细化和实现）。这样可以：
- ✅ 清晰地区分高层设计和具体实现
- ✅ 支持多个团队的并行工作
- ✅ 便于需求的追溯和映射
- ✅ 提高设计的可维护性

### 层级定义

```
第1层：原始需求及分析
    ↓ N:1 分配
第2层：相关方需求架构（架构）
    ├─ 末级节点原则：能够完全承接原始需求末级节点
    │
    ├─ 功能部分 ↓
    │   第2.5层：相关方需求-功能（详细设计）
    │   末级节点 ↓ N:1 分配
    │   第3层：业务架构（架构）
    │       ↓ 去重后得到功能清单
    │       第4层：系统需求架构-功能（架构）
    │       末级节点 = 业务架构中的 IPO（功能清单）
    │           ↓
    │           第4.5层：系统需求-功能（详细设计）
    │           末级节点 ↓ N:1 分配
    │           第5层：产品架构（架构 + 详细设计）
    │
    └─ 非功能部分 ↓
        第2.5层：相关方需求-非功能（详细设计）
        末级节点 ↓ N:1 分配
        第4层：系统需求架构-非功能（架构）
        末级节点 = 非功能需求分类
            ↓
            第4.5层：系统需求-非功能（详细设计）
            末级节点 ↓ N:1 分配
            第5层：产品架构（架构 + 详细设计）
```

### 每层的角色定义

| 层级 | 报告名称 | 角色 | 承接关系 | 输出形式 |
|------|---------|------|---------|---------|
| 1 | 原始需求及分析 | 需求 | - | 需求清单 |
| 2 | 相关方需求架构 | 原始需求的架构分类 | 承接原始需求（N:1） | 架构末级节点 |
| 2.5 | 相关方需求 | 原始需求的详细设计 + 后续的需求 | 承接原始需求 | 条目化需求 + 映射说明 |
| 3 | 业务架构 | 相关方需求功能部分的方案 | 承接相关方需求功能部分 | IPO清单 + 映射说明 |
| 4 | 系统需求架构 | 业务架构和相关方需求非功能部分的架构分类 | 承接BA + 相关方需求非功能部分 | 架构末级节点 |
| 4.5 | 系统需求 | 业务架构和相关方需求非功能部分的详细设计 + 后续的需求 | 承接BA + 相关方需求非功能部分 | 条目化需求 + 映射说明 |
| 5 | 产品架构 | 系统需求的方案 | 承接系统需求 | 条目化方案 + 实现细节 |

## 🎯 核心约束和规则

### 规则1：1:1 分配约束

**定义**：需求中的每一条都应分配至方案中**唯一一个条目**。

**目的**：
- 确保每条需求都有明确的承接点
- 发现需求分解不足或方案设计不完整的问题
- 防止需求遗漏或重复分配

**应用范围**：
- 相关方需求末级 → 业务架构末级（1:1）
- 系统需求末级 → 产品架构末级（1:1）

**检查方法**：
```
对于每条上层需求：
  ✓ 是否分配到了下层方案的某个条目？
  ✓ 是否只分配到了唯一一个条目？
  ✓ 是否有多个条目都承接了这条需求？
```

**常见问题**：
- ❌ 一条需求分配到多个方案条目 → 说明需求分解不足
- ❌ 一条需求无法分配到任何方案条目 → 说明方案设计不完整
- ❌ 一条需求分配到的方案条目不清晰 → 说明需求或方案定义不够具体

### 规则2：N:1 承接支持

**定义**：方案中一个条目可以承接需求的**多个条目**。

**目的**：
- 支持需求的聚合和整合
- 体现设计的高效性和合理性
- 避免过度分解导致的复杂性

**应用范围**：
- 原始需求末级 → 相关方需求架构末级（N:1）
- 相关方需求末级 → 业务架构末级（N:1）
- 业务架构末级 → 系统需求架构末级（去重后 1:1）
- 系统需求末级 → 产品架构末级（N:1）

**检查方法**：
```
对于每个下层方案条目：
  ✓ 承接了多少条上层需求？
  ✓ 这些需求之间是否有逻辑关联？
  ✓ 承接的需求数量是否合理（通常2-4条）？
```

**合理范围**：
- 最少：1条需求（某些特殊需求可能独立承接）
- 最多：4-5条需求（超过5条说明方案条目定义过于宽泛）
- 平均：2-3条需求（表示设计的合理聚合）

### 规则3：双向追溯

**定义**：支持从上层需求追溯到下层方案，也支持从下层方案反向追溯到上层需求。

**实现方式**：
- 在每个方案条目中标注"承接了哪些上层需求"
- 在每条上层需求中标注"分配到哪个下层方案条目"
- 通过超链接实现快速导航

**关键原则**：
- ✅ 条目化需求分配到**架构末级节点**，而不是详细设计末级节点
- ✅ 这样可以保持需求和方案的对应关系清晰
- ✅ 架构末级节点是需求映射的关键层级

**检查方法**：
```
对于每条需求：
  ✓ 能否快速找到它分配到的方案条目？
  ✓ 能否从方案条目反向找到它？

对于每个方案条目：
  ✓ 能否清晰看到它承接了哪些需求？
  ✓ 能否从需求反向找到这个方案条目？
```

## 📋 每层的具体要求

### 第1层：原始需求及分析

**内容结构**：
```
原始需求 1
├─ 需求描述：[具体的需求定义]
├─ 末级需求分解：
│  ├─ 末级需求 1.1 → [分配到相关方需求]
│  ├─ 末级需求 1.2 → [分配到相关方需求]
│  └─ 末级需求 1.3 → [分配到相关方需求]
└─ 优先级：[高/中/低]
```

**关键要求**：
- ✅ 每条原始需求都应分解到末级需求
- ✅ 末级需求应该是原子性的、不可再分的
- ✅ 每条末级需求都应分配到相关方需求的某个条目
- ✅ 文档中只保留纯需求清单，去除解释性内容

**质量指标**：
- 末级需求平均分解数：2-3条（表示中粒度分解）
- 映射覆盖率：100%（所有末级需求都被映射）

### 第2层：相关方需求架构

**相关方需求架构的本质**：

相关方需求架构是对相关方需求的分类和组织，是原始需求的高层架构。通过对相关方需求进行分类和组织，帮助理解需求的结构和关系。

**末级节点原则**：

相关方需求架构的末级节点应该能够完全承接原始需求的末级节点。也就是说，每个相关方需求架构末级节点应该对应一个或多个原始需求末级节点（N:1 映射）。

**内容结构**：

相关方需求架构通常包含以下层级：
- 第1级：需求类别（功能需求、非功能需求等）
- 第2级：需求分类（按业务域、按功能模块等）
- 第3级：需求子分类（进一步细分）
- 末级：相关方需求架构末级节点（能够承接原始需求末级节点）

**关键要求**：
- ✅ 架构末级节点能够完全承接原始需求末级节点
- ✅ 架构结构清晰、易于理解
- ✅ 支持 N:1 映射：多个原始需求末级可映射到一个架构末级节点
- ✅ 为下层的相关方需求详细设计提供指导

**质量指标**：
- 相关方需求架构末级节点数：通常 10-20 个
- 平均每个架构末级节点承接原始需求末级数：2-3 条
- 映射覆盖率：100%（所有原始需求末级都被映射）

---

### 第2.5层：相关方需求（详细设计）

**内容结构**：

**第一部分：条目化的相关方需求**
```
SR-F-001: [需求名称]
├─ 需求描述：[详细的需求定义]
├─ 验收标准：[该需求的验收条件]
├─ 承接的原始需求：
│  ├─ 原始需求 1.1
│  ├─ 原始需求 1.2
│  └─ 原始需求 1.3
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对业务架构的指导]
```

**第二部分：原始需求分配说明**
```
原始需求 1.1
├─ 分配到：SR-F-001
├─ 符合性说明：[SR-F-001 如何满足这条需求]
└─ 超链接：[指向 SR-F-001 的链接]
```

**关键要求**：
- ✅ 功能需求和非功能需求分别标识
- ✅ 每条相关方需求都包含完整的内容结构
- ✅ 符合性分析应该清晰、具体
- ✅ 建立双向追溯的超链接
- ✅ 相关方需求末级节点分配到业务架构末级节点（N:1 映射）

**质量指标**：
- 相关方需求总数：通常 20-40 条
- 平均每个相关方需求承接原始需求末级数：2-3 条
- 平均每个相关方需求分配到业务架构末级数：1-2 个
- 符合性分析完整率：100%

### 第3层：业务架构

**业务架构的本质**：

业务架构是**运行概念**（Operational Concept），是用户使用待开发产品以及由该产品与其运行环境组成的系统开展业务并实现业务目标的使用手册，也是用户开展业务的过程说明。

**业务架构的作用**：

通过对业务架构的开发、分析和确认，帮助产品开发者和产品应用者能够尽可能全地识别并定义待开发产品的功能清单。

**IPO模型（Input-Process-Output）**：

业务架构的末级节点应该是一个IPO模型：
- **Input（输入）**：产品和/或系统接收的输入
- **Process（处理）**：对输入进行的处理
- **Output（输出）**：生成的输出

每个IPO对应一个系统功能。

**IPO的颗粒度**：

IPO的体量主要取决于所期望输出物的颗粒度。例如：
- 对于系统设计流水线产品，期望的输出是各技术文档
- 那么一个IPO应该对应一份技术文档（或文档的一个可以由单人完成的独立部分）

**内容结构**：

**第一部分：条目化的业务架构（IPO清单）**
```
BA-001: [业务组件名称/IPO名称]
├─ 定义：[业务组件的定义]
├─ Input（输入）：
│  ├─ 输入的具体形式
│  ├─ 数据结构或内容
│  └─ 输入的来源
├─ Process（处理）：
│  ├─ 处理的规则或算法
│  ├─ 处理的步骤
│  └─ 处理的约束条件
├─ Output（输出）：
│  ├─ 输出的具体形式
│  ├─ 数据结构或内容
│  └─ 输出的验收标准
├─ 承接的相关方需求：
│  ├─ SR-F-001
│  ├─ SR-F-002
│  └─ SR-F-003
├─ 符合性分析：[如何满足所承接的需求]
└─ 为下层提供的设计指导：[对系统需求的指导]
```

**第二部分：相关方需求分配说明**
```
SR-F-001
├─ 分配到：BA-001
├─ 符合性说明：[BA-001 如何满足这条需求]
└─ 超链接：[指向 BA-001 的链接]
```

**关键要求**：
- ✅ 业务架构应该是相关方需求功能部分的方案
- ✅ 每个IPO都应该有清晰的Input、Process、Output定义
- ✅ IPO不再向下分解，只对I、P、O进行详细说明
- ✅ 建立与相关方需求的映射关系（N:1支持）
- ✅ 为系统需求提供清晰的设计指导

**IPO的去重机制**：

在业务架构设计时，当在某个分支需要一个IPO时，应该：
1. **首先检查** - 现有业务架构中是否已有相同或相似的IPO
2. **复用优先** - 如果存在，直接复用该IPO
3. **改进次之** - 如果现有IPO改进后可用，则改进后复用
4. **新增最后** - 确实需要新的IPO才新增

**质量指标**：
- 业务架构条目数（IPO数）：通常 10-20 个
- 平均每个IPO承接相关方需求数：2-3 条（N:1支持）
- 1:1 分配约束满足率：100%（每条相关方需求分配到唯一的IPO）
- IPO详细程度：能指导系统需求定义，能与用户沟通

### 第4层：系统需求架构

**系统需求架构的本质**：

系统需求架构是对系统需求的分类和组织，是业务架构和相关方需求非功能部分的高层架构。

**功能需求架构**：

系统需求架构的功能需求末级节点就是业务架构中的 IPO，也就是功能清单中的每一项功能。上级节点是对功能节点的分类和组织。

**非功能需求架构**：

系统需求架构的非功能需求采用平行的架构体系，独立于功能需求设计。非功能需求架构末级节点是具体的非功能需求承接层节点。

**内容结构**：

系统需求架构通常包含以下层级：

**功能需求分支**：
- 第1级：功能需求根节点
- 第2级：业务组件类型（流程、工单、计划、指标等）
- 第3级：业务组件小类
- 末级：IPO 节点（功能清单中的每一项）

**非功能需求分支**：
- 第1级：非功能需求根节点
- 第2级：需求类别（质量特性、环境适应性、可实现性）
- 第3级：需求类型（性能、安全、可测试性等）
- 第4级：需求子类型
- 末级：非功能需求承接层节点

**关键要求**：
- ✅ 功能需求末级节点 = 业务架构中的 IPO
- ✅ 非功能需求采用平行的架构体系
- ✅ 架构结构清晰、易于理解
- ✅ 为下层的系统需求详细设计提供指导

**质量指标**：
- 功能需求末级节点数 = 业务架构 IPO 去重后的数量
- 非功能需求末级节点数：通常 10-20 个
- 映射覆盖率：100%（所有业务架构 IPO 都被映射）

---

### 第4.5层：系统需求（详细设计）

**系统需求的本质**：

系统需求是业务架构和相关方需求非功能部分的详细设计。系统需求中的功能清单是通过对业务架构中所有 IPO 进行去重后得到的。

**系统需求的来源**：

1. **功能需求来源** - 业务架构中所有 IPO 的去重清单
   - 多个业务流程中重复的 IPO，在系统需求中只需定义一次
   - 这确保了系统功能的完整性和无重复性

2. **非功能需求来源** - 相关方需求中的非功能部分
   - 直接承接相关方需求的非功能需求

**内容结构**：

**第一部分：条目化的系统需求**
```
SysReq-001: [系统需求名称]
├─ 定义：[系统需求的定义]
├─ 描述：[详细的系统功能或特性说明]
├─ 类型：[功能需求/非功能需求]
├─ 承接的业务架构/相关方需求：
│  ├─ BA-001
│  ├─ BA-002
│  └─ SR-NF-001
├─ 符合性分析：[如何满足所承接的需求]
├─ 验收标准：[该需求的验收条件]
└─ 为下层提供的设计指导：[对产品架构的指导]
```

**第二部分：业务架构/相关方需求分配说明**
```
BA-001
├─ 分配到：SysReq-001
├─ 符合性说明：[SysReq-001 如何满足这个 IPO]
└─ 超链接：[指向 SysReq-001 的链接]
```

**关键要求**：
- ✅ 系统需求应该是业务架构和相关方需求非功能部分的详细设计
- ✅ 功能需求和非功能需求分别处理
- ✅ 功能需求来自业务架构 IPO 的去重清单
- ✅ 每条系统需求都应该有清晰的定义和验收标准
- ✅ 建立与上层需求的映射关系（N:1 支持）
- ✅ 系统需求末级节点分配到产品架构末级节点（N:1 映射）

**质量指标**：
- 系统需求总数：通常 10-20 个
- 平均每个系统需求承接业务架构/相关方需求数：2-3 条
- 1:1 分配约束满足率：100%（每条业务架构/相关方需求分配到唯一的系统需求）
- 功能需求去重率：100%（没有重复的功能定义）
- 平均每个系统需求分配到产品架构末级数：1-2 个

### 第5层：产品架构

**产品架构的本质**：

产品架构是系统需求的详细设计和实现方案，包含具体的技术实现细节和性能指标。

**内容结构**：

**第一部分：条目化的产品架构**
```
PA-001: [产品组件名称]
├─ 定义：[产品组件的定义]
├─ 描述：[详细的实现说明]
├─ 承接的系统需求：
│  ├─ SysReq-001
│  ├─ SysReq-002
│  └─ SysReq-003
├─ 符合性分析：[如何满足所承接的需求]
├─ 实现细节：[具体的技术实现方案]
└─ 性能指标：[响应时间、吞吐量等]
```

**第二部分：系统需求分配说明**
```
SysReq-001
├─ 分配到：PA-001
├─ 符合性说明：[PA-001 如何满足这条需求]
└─ 超链接：[指向 PA-001 的链接]
```

**关键要求**：
- ✅ 产品架构应该是系统需求的完整方案
- ✅ 每个产品组件都应该有清晰的实现方案
- ✅ 包含具体的技术细节和性能指标
- ✅ 建立与系统需求的映射关系（N:1 支持）
- ✅ 产品架构末级节点承接系统需求末级节点

**质量指标**：
- 产品架构条目数：通常 15-30 个
- 平均每个产品组件承接系统需求数：1-2 条
- 1:1 分配约束满足率：100%（每条系统需求分配到唯一的产品组件）

## 🔍 常见问题和最佳实践

### Q1：如何判断需求分解是否充分？

**答**：
- ✅ 末级需求应该是原子性的、不可再分的
- ✅ 末级需求应该能够分配到下层方案的某个条目
- ✅ 末级需求的粒度应该合理（平均分解数 2-3 条）
- ❌ 如果末级需求无法分配到任何方案条目，说明分解不足

**最佳实践**：
- 使用"能否进一步分解"的问题来判断是否已经到达末级
- 使用"是否能够分配到下层方案"的问题来验证分解的充分性

### Q2：如何判断方案设计是否完整？

**答**：
- ✅ 所有上层需求都应该分配到下层方案的某个条目
- ✅ 没有遗漏的需求
- ✅ 没有无法承接的需求
- ❌ 如果某条需求无法分配到任何方案条目，说明方案设计不完整

**最佳实践**：
- 使用"是否所有需求都被承接"的问题来验证方案的完整性
- 使用需求映射矩阵来发现遗漏的映射

### Q3：一个方案条目应该承接多少条需求？

**答**：
- 最少：1 条需求（某些特殊需求可能独立承接）
- 最多：4-5 条需求（超过 5 条说明方案条目定义过于宽泛）
- 平均：2-3 条需求（表示设计的合理聚合）

**最佳实践**：
- 如果某个方案条目承接超过 5 条需求，考虑是否需要进一步分解
- 如果某个方案条目只承接 1 条需求，考虑是否可以与其他条目合并

### Q4：如何编写有效的符合性分析？

**答**：符合性分析应该包含以下内容：
1. **承接的需求清单**：列出所有承接的需求
2. **满足方式**：说明如何满足每条需求
3. **验证方法**：说明如何验证符合性
4. **特殊说明**：说明是否有任何限制或特殊情况

**最佳实践**：
```
符合性分析示例：
- 承接需求 SR-F-001：通过提供用户登录功能满足
  验证方法：用户能够使用用户名和密码登录系统
- 承接需求 SR-F-002：通过提供权限管理功能满足
  验证方法：系统能够根据用户角色限制功能访问
```

### Q5：如何建立有效的双向追溯？

**答**：
1. **在方案条目中标注承接的需求**：
   ```
   BA-001: 用户管理
   承接的相关方需求：SR-F-001, SR-F-002, SR-F-003
   ```

2. **在需求中标注分配的方案条目**：
   ```
   SR-F-001: 用户登录
   分配到：[BA-001](business-architecture.md#ba-001)
   ```

3. **使用超链接实现快速导航**：
   - 格式：`[文本](文件名.md#锚点)`
   - 确保锚点与目标文档中的 ID 对应

**最佳实践**：
- 使用一致的超链接格式
- 定期验证所有超链接的有效性
- 在文档更新时同时更新相关的超链接

## 🎓 应用场景和示例

### 场景1：新产品开发

**流程**：
1. 收集原始需求（第1层）
2. 设计相关方需求（第2层）
3. 设计业务架构（第3层）
4. 设计系统需求（第4层）
5. 设计产品架构（第5层）

**关键点**：
- 每层都要进行 1:1 分配约束验证
- 每层都要进行 N:1 承接支持验证
- 每层都要建立双向追溯

### 场景2：需求变更管理

**流程**：
1. 识别变更的原始需求
2. 追溯到相关的相关方需求
3. 追溯到相关的业务架构
4. 追溯到相关的系统需求
5. 追溯到相关的产品架构
6. 评估变更的影响范围

**关键点**：
- 使用双向追溯快速定位影响范围
- 使用 1:1 约束验证变更的完整性
- 使用符合性分析验证变更的有效性

### 场景3：设计审查

**流程**：
1. 检查每层的 1:1 分配约束
2. 检查每层的 N:1 承接支持
3. 检查每层的符合性分析
4. 检查双向追溯的完整性
5. 检查文档的规范化程度

**关键点**：
- 使用检查清单进行系统的审查
