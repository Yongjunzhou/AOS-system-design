# 需求规范化技能
**Skill: Requirement Normalization**

**编号**：03
**所属步骤**：第 1 步 — 原始需求分析及相关方需求架构定义
**引用准则**：系统设计准则 v2.3 — 第 1 层原始需求及分析
**引用指南**：瀑布式设计指南 v2.0 第 1 步

---

## 用途

将原始需求文本（邮件、会议纪要、需求文档）转化为规范化需求，确保每条需求完整性、清晰性和可测试性。

## 输入

- 原始需求文本（未经规范化的表述）
- 术语表（01-design-standards/02-terminology-glossary.md）

## 处理规则

执行以下四项检查：

### 1. 完整性检查
- 是否有清晰的**需求描述**（系统应该做什么）
- 是否有**提出方**（谁提出的需求）
- 是否有**优先级**（高/中/低）
- 是否有**类型**（功能需求/非功能需求）

### 2. 清晰性检查
- 无模糊术语（"尽可能"、"大概"、"必要时"、"尽量"）
- 术语一致（同一概念使用同一术语）
- 相关方清晰（谁使用、谁受益）
- 约束清晰（技术、时间、资源约束）

### 3. 可测试性检查

**功能需求**的可测试性：
- 是否有明确的输入（用户输入了什么？）
- 是否有明确的处理逻辑（系统怎么处理？）
- 是否有明确的输出（系统返回了什么？）
- 验收标准是否可量化（成功的标志是什么？）

**非功能需求**的可测试性：
- 是否有具体的量化指标（响应时间、吞吐量、可靠率、安全等级等）
- 指标是否有明确的适用条件（在什么负载/条件下）
- 是否有验证方法（如何测试和验证）

### 4. 独立性检查
- 需求是否相对独立
- 是否与其它需求过度耦合
- 如果过于复杂，标注建议分解

### 5. 功能与非功能需求分离规则（新增）

在规范化过程中，必须将功能需求和非功能需求明确分离，分别创建独立的规范化需求记录：

**功能需求特征**（应该生成 SR-F）：
- 描述"系统应该提供什么功能"或"系统应该怎样处理"
- 包含 IPO（Input-Process-Output）的明确描述
- 示例：订单创建、数据查询、权限验证

**非功能需求特征**（应该生成 SR-NFR）：
- 描述"系统应该满足什么质量特性"或"系统应该有什么性能表现"
- 属于三大类之一：质量特性、环境适应性、可实现性
- 不包含具体的业务操作步骤，而是对操作的约束
- 示例：响应时间 < 2s、支持 10000 并发用户、99.9% 可用性

**分离方法**：
1. 识别需求中的"做什么"部分 → 作为功能需求
2. 识别需求中的"有多好"部分 → 作为非功能需求
3. 不同的规范化需求，分别输出，但标注相互关联关系

**示例**：

原始需求：
```
用户登录系统时，系统应该快速响应，在 2 秒内完成验证。
```

规范化后：
```
功能需求（SR-F）：
- 描述：用户输入用户名和密码后，系统验证身份并返回登录结果
- 验收标准：验证成功返回用户信息和 Token，验证失败返回错误信息

非功能需求（SR-NFR）：
- 描述：登录功能应该快速响应，提供良好的用户体验
- 性能指标：响应时间 < 2s（包括网络延迟），在 1000 并发用户下平均 < 500ms
```

## 输出格式

每条规范化需求输出为：

```
- 原始文本：[原始描述]
- 规范化描述：[规范化后的描述]
- 提出方：[需求提出者]
- 优先级：[高/中/低]
- 类型：[功能需求/非功能需求]
- 验收标准：[可量化的验收条件]
- 规范化说明：[做了哪些规范化处理]
```
