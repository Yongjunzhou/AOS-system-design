### 冲突 2：非功能需求处理方式

#### 冲突描述

**时间线**：
- **2026-05-11**（初期）：非功能需求与功能需求混合处理
  - 所有需求都经过完整的四层架构
  - 非功能需求也映射到业务架构
  
- **2026-05-12**（后期）：非功能需求采用独立的两轨处理模型
  - 功能需求流：SR-F → BA → SysReq-F → PA
  - 非功能需求流：SR-NF → SysReq-NF → PA（不经过 BA）

**冲突原因**：
- 初期对非功能需求的处理方式不够清晰
- 后期发现非功能需求不应该映射到业务架构
- 非功能需求应该作为系统约束条件

#### 冲突影响

| 方面 | 影响 |
|------|------|
| 架构设计 | 需要重新定义非功能需求的处理流程 |
| 模板设计 | 需要创建独立的非功能需求模板 |
| 文档结构 | 需要调整系统需求的层级结构 |
| 映射规则 | 需要重新定义映射关系 |

#### 解决方案

**决策**：以后者为主（采用两轨处理模型）

**功能需求流**：
```
SR-F（相关方功能需求）
  ↓ [准则1：规范化、分解、初步BA设计]
BA（业务架构）
  ↓ [准则2：分配]
SysReq-F（系统功能需求）
  ↓ [准则3：分配]
PA（产品架构）
```

**非功能需求流**：
```
SR-NF（相关方非功能需求）
  ↓ [准则1：规范化、分解、标记为约束]
SysReq-NF（系统约束）
  ↓ [准则3：标记为约束]
PA（产品架构）
  ↓ [准则4：验证约束满足]
  ↓ [如不满足 → 反向迭代]
```

**执行步骤**：
1. 重新定义系统需求的层级结构
2. 创建独立的非功能需求模板
3. 更新所有设计准则
4. 更新所有示例文档

#### 最终状态

✅ 两轨处理模型已完全实现
- 功能需求和非功能需求有清晰的处理流程
- 非功能需求作为系统约束条件
- 支持反向迭代机制

### 冲突 3：业务架构层级定义

#### 冲突描述

**时间线**：
- **2026-05-11**（初期）：业务架构层级定义不够清晰
  - 只有模糊的"业务域"、"项目"等概念
  - 没有明确的层级编号
  
- **2026-05-12**（后期）：明确定义为 0-5 级
  - 0 级：架构根节点
  - 1 级：业务域
  - 2 级：端到端项目
  - 3 级：项目业务对象架构节点
  - 4 级：工作包（WBS）
  - 5 级：业务组件（承接层，末级）

**冲突原因**：
- 初期对业务架构的分层不够系统
- 后期通过实践发现需要更清晰的层级定义
- 需要明确哪一级是需求承接层

#### 冲突影响

| 方面 | 影响 |
|------|------|
| 架构设计 | 需要重新定义业务架构的层级 |
| 模板设计 | 需要调整业务架构模板 |
| 示例文档 | 需要更新所有示例中的层级定义 |
| 映射规则 | 需要明确 BA 5 级是需求承接层 |

#### 解决方案

**决策**：以后者为主（采用 0-5 级明确定义）

**业务架构层级结构**：
```
0 级：架构根节点
  ├─ 1 级：业务域
  │   ├─ 2 级：端到端项目
  │   │   ├─ 3 级：项目业务对象架构节点
  │   │   │   ├─ 4 级：工作包（WBS）
  │   │   │   │   └─ 5 级：业务组件（承接层，末级）
```

**关键特点**：
- 5 级是需求承接层（末级，不再分解）
- 每条相关方需求映射到唯一的 5 级节点
- 支持 N:1 多对一映射

#### 最终状态

✅ 业务架构层级已明确定义
- 所有示例都遵循 0-5 级结构
- 5 级是需求承接层
- 映射关系清晰

### 冲突 5：性能指标的性质

#### 冲突描述

**时间线**：
- **2026-05-11**（初期）：性能指标被视为独立的需求
  - SR-NFR-001：系统响应时间 < 200ms
  - SR-NFR-002：支持 1000+ 并发用户
  
- **2026-05-13**（后期）：性能指标是需求的属性
  - SR-F-001：产品设计文档创建
    - 描述：工程师能够创建设计文档
    - 性能指标：响应时间 < 500ms，支持 500+ 并发用户

**冲突原因**：
- 初期对需求结构的理解不够深入
- 后期通过实践发现性能指标应该是需求的属性
- 这样可以形成更清晰的需求结构

#### 冲突影响

| 方面 | 影响 |
|------|------|
| 需求结构 | 需要重新定义需求的结构 |
| 模板设计 | 需要调整所有需求模板 |
| 示例文档 | 需要更新所有示例中的需求定义 |
| 映射规则 | 需要调整映射关系的定义 |

#### 解决方案

**决策**：以后者为主（性能指标是需求的属性）

**新的需求结构**：
```
功能需求 = 需求描述 + 性能指标
  - 需求描述：系统应该提供什么功能
  - 性能指标：该功能的性能约束（响应时间、吞吐量、并发数等）

非功能需求 = 需求描述 + 性能指标
  - 需求描述：系统应该满足什么质量特性
  - 性能指标：该质量特性的具体量化指标
```

**执行步骤**：
1. 更新 CLAUDE.md 中的需求结构定义
2. 更新所有需求模板
3. 更新所有示例文档
4. 更新所有设计准则

#### 最终状态

✅ 需求结构已明确定义
- 性能指标是需求的属性
- 需求结构更清晰
- 映射关系更准确
- 追溯链路更完整
