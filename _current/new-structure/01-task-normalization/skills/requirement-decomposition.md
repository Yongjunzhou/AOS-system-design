# Skill: 需求分解

## 目的
将复杂的需求分解为多个独立的、相对简单的需求，便于后续的映射和实现。

## 输入
- 规范化的需求
- 相关的业务背景

## 输出
- 分解后的需求列表
- 分解过程中的决策记录

## 分解规则

### 1. 分解原则
- 每个分解后的需求应相对独立
- 分解后的需求应能够映射到业务架构的唯一节点
- 分解应保持原需求的完整语义

### 2. 分解标准
- [ ] 是否存在"和"连接的多个需求（应分解）
- [ ] 是否存在多个独立的功能（应分解）
- [ ] 是否存在多个独立的角色（应分解）
- [ ] 是否存在多个独立的流程（应分解）

### 3. 分解深度
- 分解应达到"原子需求"的级别
- 原子需求是指不能再分解的最小需求单位
- 原子需求应能够映射到业务架构的唯一节点

## 分解模板

```markdown
### 分解结果

原始需求：[原始需求]

分解后的需求：
1. [分解需求1]
2. [分解需求2]
3. [分解需求3]

分解决策：
- [分解决策1]
- [分解决策2]
```

## 常见分解场景

### 场景1：多个功能的组合
**原始需求**：系统应支持订单的创建、修改和删除

**分解结果**：
- SR-001: 支持订单创建
- SR-002: 支持订单修改
- SR-003: 支持订单删除

### 场景2：多个角色的需求
**原始需求**：销售和财务都需要查看订单信息

**分解结果**：
- SR-001: 销售可以查看订单信息
- SR-002: 财务可以查看订单信息

### 场景3：多个流程的需求
**原始需求**：系统应支持订单的完整生命周期管理

**分解结果**：
- SR-001: 支持订单创建流程
- SR-002: 支持订单处理流程
- SR-003: 支持订单完成流程
