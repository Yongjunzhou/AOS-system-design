# AOS 架构改进总结：性能指标与需求结构

**完成日期**：2026-05-13  
**改进类型**：核心架构优化  
**影响范围**：全系统（Task 1-6）

---

## 🎯 改进概述

这次改进解决了 AOS 架构中一个关键的概念混淆：**性能指标的性质**。

### 问题

在之前的架构中：
- 功能需求（SR-F）只有需求描述，缺少性能指标
- 非功能需求（SR-NFR）只有指标值，缺少需求描述
- 性能指标被当作需求本身，而不是需求的属性

### 解决方案

**性能指标是需求的属性，不是需求本身**：
- 功能需求 = 需求描述 + 性能指标
- 非功能需求 = 需求描述 + 性能指标
- 性能指标随需求一起流转，从 SR → BA → SysReq → PA

## 📊 具体改进

### 1. 功能需求结构

**改进前**：
```yaml
SR-F-001: 用户登录
描述：用户输入用户名和密码，系统验证后登录
# 缺少性能指标
```

**改进后**：
```yaml
SR-F-001: 用户登录
描述：用户输入用户名和密码，系统验证后登录
性能指标：
  - 响应时间：< 2s（在 1000 并发下）
  - 成功率：> 99.9%
  - 支持并发用户：1000+
```

### 2. 非功能需求结构

**改进前**：
```yaml
SR-NFR-001: 响应时间 < 200ms
# 只有指标，没有需求描述
```

**改进后**：
```yaml
SR-NFR-001: 系统响应时间
描述：系统在正常业务负载下应该快速响应，提供良好的用户体验
性能指标：
  - 平均响应时间：< 200ms
  - P99 响应时间：< 500ms
  - 在 1000 并发下的响应时间：< 300ms
```

### 3. 映射关系

**改进前**：
```
SR-F（描述）→ BA（描述）→ SysReq（描述）→ PA（描述）
SR-NFR（指标）→ SysReq-NFR（指标）→ PA（指标）
# 功能需求和非功能需求的追溯是分离的
```

**改进后**：
```
SR-F（描述 + 指标）→ BA（描述 + 指标）→ SysReq（描述 + 指标）→ PA（描述 + 指标）
SR-NFR（描述 + 指标）→ SysReq-NFR（描述 + 指标）→ PA（描述 + 指标）
# 性能指标随需求一起追溯，确保不会遗漏
```

## 💡 改进的价值

### 1. 需求更完整
- 每个需求都包含需求描述和性能指标
- 设计团队可以清楚地了解每个需求的完整内容

- 所有需求都遵循相同的结构（描述 + 指标）

## ❓ 常见问题

**Q：功能需求一定要有性能指标吗？**

A：是的。每条功能需求都应该有性能指标，即使是最小的指标（如"响应时间 < 5s"）。这样可以确保设计团队清楚地了解每个功能的性能约束。

**Q：非功能需求的需求描述应该包含什么？**

A：非功能需求的需求描述应该说明系统应该满足什么质量特性，为什么需要这个特性，以及这个特性的业务背景。

**Q：性能指标应该在哪个阶段补充？**

A：性能指标应该在 Task 1（需求规范化）阶段就补充。这样可以确保所有下游的设计工作都基于完整的需求。

**Q：如果需求没有性能指标怎么办？**

A：这表示需求不完整。应该与相关方沟通，确定合理的性能指标。如果确实没有特殊的性能要求，也应该明确说明（如"无特殊性能要求"）。
