## 一、非功能需求三大类详解

### 1.1 质量特性需求（Quality Attributes）

**定义**：系统应该具备的质量属性

**常见类型**：

#### 性能（Performance）
- **响应时间**：用户请求到收到响应的时间
  - 示例：平均响应时间 < 200ms，P99 < 1000ms
- **吞吐量**：系统单位时间内处理的请求数
  - 示例：支持 10000+ RPS（每秒请求数）
- **并发能力**：系统能同时处理的用户数
  - 示例：支持 1000+ 并发用户

#### 可靠性（Reliability）
- **平均无故障时间（MTBF）**：系统平均能运行多长时间不出故障
  - 示例：MTBF > 720 小时
- **故障率**：系统出故障的频率
  - 示例：故障率 < 0.1%

#### 安全性（Security）
- **数据加密**：保护数据的机密性
  - 示例：AES-256 加密、TLS 1.2+ 传输
- **访问控制**：保护数据的完整性和可用性
  - 示例：基于角色的访问控制（RBAC）
- **审计日志**：记录系统操作
  - 示例：所有敏感操作都有审计日志

#### 可用性（Availability）
- **系统可用性**：系统能正常工作的时间比例
  - 示例：系统可用性 > 99.9%（每月停机时间 < 43 分钟）
- **故障恢复时间（RTO）**：系统从故障到恢复的时间
  - 示例：RTO < 30 分钟
- **故障恢复点（RPO）**：系统能恢复到的最近数据点
  - 示例：RPO < 5 分钟

#### 可维护性（Maintainability）
- **代码质量**：代码的可读性、可理解性
  - 示例：圈复杂度 < 10，代码覆盖率 > 80%
- **文档完整性**：文档的完整程度
  - 示例：文档完整性 > 90%

---

### 1.2 环境适应性需求（Environmental Adaptation）

**定义**：系统应该适应的环境和条件

**常见类型**：

#### 兼容性（Compatibility）
- **浏览器兼容**：支持不同的浏览器
  - 示例：支持 Chrome、Firefox、Safari、Edge 最新两个版本
- **操作系统兼容**：支持不同的操作系统
  - 示例：支持 Windows、macOS、Linux
- **移动端兼容**：支持移动设备
  - 示例：支持 iOS 12.0+、Android 8.0+

#### 可扩展性（Scalability）
- **水平扩展**：通过增加服务器来提高性能
  - 示例：支持自动负载均衡和水平扩展
- **垂直扩展**：通过增加单个服务器的资源来提高性能
  - 示例：支持动态资源分配

#### 可移植性（Portability）
- **跨平台支持**：系统能在不同平台上运行
  - 示例：支持 Docker 容器化部署

#### 集成适应性（Integration Adaptability）
- **第三方系统集成**：与其他系统的集成能力
  - 示例：支持与支付宝、微信等第三方支付平台集成

---

### 1.3 可实现性需求（Implementability）

**定义**：系统应该满足的实现相关需求

**常见类型**：

#### 可生产性（Producibility）
- **构建自动化**：自动化构建过程
  - 示例：支持 Maven、Gradle 等构建工具
- **依赖管理**：管理项目依赖
  - 示例：使用 Maven Central 或 npm Registry

#### 可测试性（Testability）
- **单元测试**：测试单个函数或类
  - 示例：单元测试覆盖率 > 80%
- **集成测试**：测试多个组件的集成
  - 示例：集成测试覆盖率 > 60%
- **端到端测试**：测试完整的业务流程
  - 示例：关键业务流程的端到端测试覆盖率 > 80%

#### 可交付性（Deliverability）
- **交付物完整性**：交付物包含所有必需的内容
  - 示例：包含源代码、文档、测试用例等

#### 可部署性（Deployability）
- **部署自动化**：自动化部署过程
  - 示例：支持 CI/CD 流水线
- **零停机部署**：部署时不中断服务
  - 示例：支持滚动部署、蓝绿部署

#### 可操作性（Operability）
- **监控能力**：系统的可监控性
  - 示例：支持实时监控和告警
- **日志记录**：系统的日志记录能力
  - 示例：日志保留 > 30 天，支持日志查询

#### 可运维性（Maintainability in Operations）
- **故障诊断**：快速定位和解决问题
  - 示例：支持详细的错误日志和堆栈跟踪
- **性能优化**：系统的性能优化能力
  - 示例：支持性能分析和优化工具

#### 可退役性（Retirability）
- **数据迁移**：系统下线时的数据迁移
  - 示例：支持数据导出和迁移工具

## 二、常见错误

### 错误 1：非功能需求表述不清

**❌ 错误示例**
```
需求：系统要快速响应
需求：系统要稳定可靠
需求：系统要安全
```

**问题**：
- "快速"、"稳定"、"安全"都是模糊的表述
- 无法确定验收标准
- 无法进行有效的测试

**✅ 正确做法**
```
需求：系统响应时间
- 平均响应时间 < 200ms
- P95 响应时间 < 500ms
- P99 响应时间 < 1000ms

需求：系统可靠性
- 平均无故障时间（MTBF）> 720 小时
- 故障恢复时间（RTO）< 30 分钟

需求：数据安全
- 支付信息采用 AES-256 加密
- 传输层采用 TLS 1.2+ 加密
```

---

### 错误 2：缺少验证方法

**❌ 错误示例**
```
需求：系统可用性 > 99.9%
```

**问题**：
- 没有说明如何验证这个需求
- 无法进行有效的测试

**✅ 正确做法**
```
需求：系统可用性 > 99.9%
- 验证方法：监控系统运行时间，记录停机时间
- 验证工具：Prometheus、Grafana
- 验证标准：每月停机时间 < 43 分钟
```

---

### 错误 3：忽视权衡关系

**❌ 错误示例**
```
需求 1：系统响应时间 < 100ms
需求 2：支持 10000+ 并发用户
需求 3：采用 AES-256 加密
需求 4：系统成本 < 10 万元
```

**问题**：
- 这些需求之间可能存在冲突
- 高性能需要更多的硬件投入
- 强加密会增加 CPU 开销
- 低成本可能无法满足高性能需求

**✅ 正确做法**
```
需求 1：系统响应时间 < 200ms（在 1000 并发下）
需求 2：支持 1000+ 并发用户
需求 3：采用 TLS 1.2 + AES-128 加密（平衡性能和安全性）
需求 4：系统成本 < 50 万元

权衡决策：
- 选择性能优先，因为用户体验是核心竞争力
- 选择 AES-128 而不是 AES-256，以平衡性能和安全性
- 使用云服务的自动扩展功能，以降低成本
```

---

### 错误 4：非功能需求分解不充分

**❌ 错误示例**
```
需求：系统要高可用
```

**问题**：
- 需求太宽泛，无法映射到具体的系统设计
- 无法进行有效的测试

**✅ 正确做法**
```
需求：系统可用性
├── 系统可用性 > 99.9%
├── 平均无故障时间（MTBF）> 720 小时
├── 故障恢复时间（RTO）< 30 分钟
└── 故障恢复点（RPO）< 5 分钟
```

---

### 错误 5：缺少相关方信息

**❌ 错误示例**
```
需求：系统响应时间 < 200ms
```

**问题**：
- 没有明确相关方
- 无法确定需求的优先级
- 无法进行有效的沟通

**✅ 正确做法**
```
需求：系统响应时间 < 200ms
- 相关方：用户、前端团队、后端团队、性能团队
- 优先级：P0（高）
- 背景/原因：用户体验是核心竞争力，快速响应能提高用户满意度
```

## 三、最佳实践

### 最佳实践 1：使用结构化格式

**推荐格式**：YAML 或表格

```yaml
需求ID: SysReq-NFR-001
需求名称: 系统响应时间
需求类别: 质量特性需求
需求类型: 性能
需求子类型: 响应时间

需求描述: |
  系统应该快速响应用户请求，确保良好的用户体验。

量化指标: |
  - 平均响应时间 < 200ms
  - P95 响应时间 < 500ms
  - P99 响应时间 < 1000ms

验证方法: |
  使用 JMeter 进行性能测试，在正常负载和峰值负载下测试

优先级: P0（高）
相关方: 用户、前端团队、后端团队、性能团队
```

**优势**：
- 信息完整
- 易于解析
- 易于版本控制

---

### 最佳实践 2：量化指标

**所有非功能需求都应该有量化指标**

```
❌ 错误：系统要快速响应
✅ 正确：平均响应时间 < 200ms，P99 < 1000ms

❌ 错误：系统要稳定可靠
✅ 正确：系统可用性 > 99.9%，MTBF > 720 小时

❌ 错误：系统要安全
✅ 正确：采用 AES-256 加密，TLS 1.2+ 传输
```

**优势**：
- 明确验收标准
- 便于测试
- 避免争议

---

### 最佳实践 3：分层设计

**使用 5 级层级结构，从抽象到具体**

```
0 级：非功能需求根节点
  ↓
1 级：需求类别（质量特性、环境适应性、可实现性）
  ↓
2 级：需求类型（性能、安全、可用性等）
  ↓
3 级：需求子类型（响应时间、吞吐量、数据加密等）
  ↓
4 级：需求细分（登录响应时间、订单响应时间等）
  ↓
5 级：非功能需求承接层（具体的量化指标）
```

**优势**：
- 结构清晰
- 易于理解
- 易于维护

---

### 最佳实践 4：权衡决策

**明确记录权衡决策及其原因**

```
权衡 1：性能 vs 安全性
- 冲突：强加密会增加 CPU 开销，降低性能
- 决策：选择 TLS 1.2 + AES-128，平衡性能和安全性
- 原因：用户体验和数据安全都很重要

权衡 2：可用性 vs 成本
- 冲突：高可用性需要多个数据中心，增加成本
- 决策：采用主从复制 + 自动故障转移
- 原因：在成本可控的范围内最大化可用性

权衡 3：功能完整性 vs 移动端性能
- 冲突：完整的功能导致移动端页面加载时间过长
- 决策：实现渐进式增强，使用懒加载技术
- 原因：提供良好的移动端体验
```

**优势**：
- 便于后续的维护和优化
- 便于团队沟通
- 便于决策追溯

---

### 最佳实践 5：关联需求

**为非功能需求关联相关的功能需求和其他非功能需求**

```
需求：系统响应时间 < 200ms
- 映射的 SR-NFR：SR-NFR-001
- 相关的功能需求：SR-F-001（用户登录）、SR-F-002（订单创建）
- 相关的非功能需求：SysReq-NFR-002（高并发支持）
- 相关的产品架构约束：PA-001（缓存层）、PA-002（数据库优化）
```

**优势**：
- 帮助发现隐藏的冲突
- 帮助发现隐藏的重复
- 便于追踪需求链路

## 四、验证方法指南

### 4.1 性能需求的验证

**工具**：JMeter、LoadRunner、Gatling

**步骤**：
1. 准备测试环境（与生产环境相同配置）
2. 设计测试场景（正常负载、峰值负载、压力测试）
3. 执行测试并收集数据
4. 分析结果并生成报告
5. 优化系统并重新测试

**示例**：
```
需求：平均响应时间 < 200ms，P99 < 1000ms

验证步骤：
1. 使用 JMeter 模拟 1000 个并发用户
2. 每个用户发送 100 个请求
3. 记录每个请求的响应时间
4. 计算平均响应时间和 P99 响应时间
5. 验证是否满足需求
```

---

### 4.2 可靠性需求的验证

**工具**：监控系统（Prometheus、Grafana）、日志系统（ELK Stack）

**步骤**：
1. 部署监控系统
2. 记录系统运行时间和停机时间
3. 定期分析故障原因
4. 进行故障恢复演练
5. 优化系统并重新测试

**示例**：
```
需求：系统可用性 > 99.9%，MTBF > 720 小时

验证步骤：
1. 监控系统运行时间和停机时间
2. 每月计算系统可用性
3. 记录故障发生的时间和原因
4. 计算平均无故障时间（MTBF）
5. 验证是否满足需求
```

---

### 4.3 安全需求的验证

**工具**：SSL Labs、Burp Suite、OWASP ZAP

**步骤**：
1. 进行安全审计
2. 进行渗透测试
3. 检查加密算法和密钥强度
4. 检查访问控制和认证机制
5. 进行漏洞扫描

**示例**：
```
需求：采用 TLS 1.2+ 加密，AES-256 加密

验证步骤：
1. 使用 SSL Labs 检查 TLS 版本和加密算法
2. 检查证书的有效性和强度
3. 进行渗透测试，尝试降级攻击
4. 验证 HSTS 头的配置
5. 验证是否满足需求
```

## 六、常见问题解答

**Q：如何判断非功能需求是否分解充分？**

A：分解充分的标准是：
- 每个非功能需求都能映射到唯一的系统设计决策
- 每个非功能需求都有明确的量化指标
- 每个非功能需求都有明确的验证方法

**Q：如何处理相互冲突的非功能需求？**

A：
1. 识别冲突
2. 记录冲突的原因
3. 进行权衡分析
4. 做出权衡决策
5. 记录决策的原因

**Q：如何确定非功能需求的优先级？**

A：
- P0（高）：核心质量属性，必须满足
- P1（中）：重要质量属性，应该满足
- P2（低）：辅助质量属性，可以延后满足
- P3（极低）：可选质量属性，可以不满足

**Q：非功能需求应该如何与功能需求关联？**

A：
- 识别哪些功能需求受到非功能需求的影响
- 在功能需求中标记相关的非功能需求
- 在非功能需求中标记相关的功能需求
- 便于追踪需求链路
