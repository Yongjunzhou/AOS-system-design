## 第一部分：相关方非功能需求（SR-NFR）

### 原始需求

```
1. 系统要快速响应工程师请求
2. 系统要支持高并发工程师
3. 工程师数据要安全
4. 系统要稳定可靠
5. 系统要支持移动端访问
6. 系统要易于维护
```

## 第三部分：SR-NFR → SysReq-NFR 映射

### 映射矩阵

| SR-NFR ID | SR-NFR 名称 | SysReq-NFR 5级 | 映射理由 | 状态 |
|-----------|-----------|----------------|---------|------|
| SR-NFR-001 | 系统响应时间 | SysReq-NFR-5-001, 002, 003 | 分解为登录、订单、搜索三个场景 | ✓ |
| SR-NFR-002 | 高并发支持 | SysReq-NFR-5-004, 005 | 分解为并发工程师和 RPS 两个指标 | ✓ |
| SR-NFR-003 | 数据安全 | SysReq-NFR-5-006, 007, 008, 009 | 分解为传输、存储、支付、认证四个方面 | ✓ |
| SR-NFR-004 | 系统可靠性 | SysReq-NFR-5-010, 011, 012 | 分解为可用性、MTBF、RTO 三个指标 | ✓ |
| SR-NFR-005 | 移动端支持 | SysReq-NFR-5-015, 016, 017 | 分解为浏览器和操作系统兼容性 | ✓ |
| SR-NFR-006 | 可维护性 | SysReq-NFR-5-013, 014, 020, 021 | 分解为代码质量、文档、测试覆盖率 | ✓ |

### 映射关系说明

**SR-NFR-001 → SysReq-NFR-5-001/002/003**
- 原因：系统响应时间需要分解为不同场景的响应时间
- 登录响应时间：工程师首次访问系统的关键指标
- 订单操作响应时间：核心业务流程的关键指标
- 搜索响应时间：工程师查询的关键指标

**SR-NFR-002 → SysReq-NFR-5-004/005**
- 原因：高并发支持需要分解为并发工程师数和吞吐量
- 并发工程师数：系统能同时处理的工程师数
- 每秒请求数（RPS）：系统的吞吐量指标

**SR-NFR-003 → SysReq-NFR-5-006/007/008/009**
- 原因：数据安全需要分解为多个维度的加密和认证
- 传输加密：保护数据在网络传输中的安全
- 存储加密：保护数据在数据库中的安全
- 支付信息加密：保护敏感的支付信息
- 多因素认证：保护工程师账户安全

## 第五部分：权衡决策

### 性能 vs 安全性

**冲突**：
- 强加密算法（如 AES-256）会增加 CPU 开销，可能降低性能
- 高并发支持需要优化，但安全检查可能增加延迟

**权衡决策**：
- 选择 TLS 1.2 + AES-128，平衡性能和安全性
- 使用硬件加速（如 AES-NI）来提高加密性能
- 在关键路径上使用缓存来减少加密开销

### 可用性 vs 成本

**冲突**：
- 高可用性（99.9%）需要多个数据中心和冗余
- 这会增加基础设施成本

**权衡决策**：
- 采用主从复制 + 自动故障转移
- 使用云服务的自动扩展功能
- 定期进行灾难恢复演练

### 功能完整性 vs 移动端性能

**冲突**：
- 完整的功能可能导致移动端页面加载时间过长
- 移动网络环境下，大文件传输会很慢

**权衡决策**：
- 实现渐进式增强（Progressive Enhancement）
- 使用懒加载（Lazy Loading）技术
- 提供移动端优化版本

## 常见错误和最佳实践

### ❌ 常见错误

1. **非功能需求表述不清**
   - ❌ 错误：系统要快速响应
   - ✅ 正确：平均响应时间 < 200ms，P99 < 1000ms

2. **缺少验证方法**
   - ❌ 错误：系统可用性 > 99.9%
   - ✅ 正确：系统可用性 > 99.9%，通过监控系统运行时间验证

3. **忽视权衡关系**
   - ❌ 错误：同时要求高性能、高安全性、低成本
   - ✅ 正确：明确权衡决策，如"选择性能优先"

### ✅ 最佳实践

1. **量化指标**
   - 所有非功能需求都应该有量化指标
   - 指标应该是可测量的、可验证的

2. **分层设计**
   - 使用 5 级层级结构，从抽象到具体
   - 每一级都应该有明确的定义

3. **权衡决策**
   - 明确记录权衡决策及其原因
   - 便于后续的维护和优化
