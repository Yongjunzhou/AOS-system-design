# Task 2：业务架构最佳实践指南

**目的**：帮助团队设计高质量的业务架构，避免常见错误

**版本**：v1.0  
**最后更新**：2026-05-13

---

## 一、业务架构层级结构详解

### 1.1 BA 0 级：架构根节点

**定义**：整个系统的顶层根节点

**特点**：
- 唯一的根节点
- 代表整个系统
- 所有其他节点都是其子节点

**示例**：
```
BA-ROOT：电商平台根节点
```

---

### 1.2 BA 1 级：业务域

**定义**：按业务领域划分的顶层业务单元

**特点**：
- 代表不同的业务领域
- 通常对应不同的部门或业务线
- 一个系统可以有多个业务域

**示例**：
```
BA-1-ECOMMERCE：电商业务域
BA-1-LOGISTICS：物流业务域
BA-1-FINANCE：财务业务域
```

**最佳实践**：
- 业务域应该相对独立
- 业务域之间的依赖关系应该清晰
- 避免过多的业务域（通常 3-10 个）

---

### 1.3 BA 2 级：端到端项目

**定义**：跨越多个业务域的项目

**特点**：
- 代表一个完整的业务项目
- 可能涉及多个业务域
- 通常有明确的目标和范围

**示例**：
```
BA-2-ORDER-PROJECT：订单管理项目（涉及订单、支付、物流等）
BA-2-USER-PROJECT：用户管理项目（涉及用户、认证、授权等）
```

**最佳实践**：
- 项目应该有明确的业务目标
- 项目应该有明确的范围和边界
- 项目应该与业务需求对应

---

### 1.4 BA 3 级：项目业务对象架构

**定义**：端到端项目中的业务对象

**特点**：
- 代表项目中的核心业务对象
- 通常对应业务实体（如订单、商品、用户等）
- 一个项目可以有多个业务对象

**示例**：
```
BA-3-ORDER-OBJECT：订单对象
BA-3-PRODUCT-OBJECT：商品对象
BA-3-USER-OBJECT：用户对象
```

**最佳实践**：
- 业务对象应该代表真实的业务实体
- 业务对象应该有明确的属性和行为
- 避免过多的业务对象（通常 3-10 个）

---

### 1.5 BA 4 级：工作包（WBS）

**定义**：对业务对象的工作分解结构

**特点**：
- 代表业务对象下的工作单元
- 通常对应业务流程或业务活动
- 一个业务对象可以有多个工作包

**示例**：
```
BA-4-ORDER-CREATION：订单创建工作包
BA-4-ORDER-PAYMENT：订单支付工作包
BA-4-ORDER-MODIFICATION：订单修改工作包
```

**最佳实践**：
- 工作包应该代表完整的业务活动
- 工作包应该有明确的输入和输出
- 工作包应该相对独立

---

### 1.6 BA 5 级：业务组件（承接层）

**定义**：承接相关方需求的关键层级

**特点**：
- 是 SR → BA 映射的承接点
- 代表具体的业务组件或配置业务
- 末级，不再分解

**示例**：
```
BA-5-001：订单创建组件
BA-5-002：支付处理组件
BA-5-003：订单修改组件
```

**最佳实践**：
- 每个 BA 5 级节点应该对应一个 SR
- 每个 BA 5 级节点应该有明确的业务功能
- 每个 BA 5 级节点应该有明确的实现方式

---

## 二、业务组件的两大类

### 2.1 配置方式实现的业务组件

**定义**：通过引擎提供的配置界面定义业务规则

**特点**：
- 无需代码开发
- 快速迭代
- 支持业务人员自助配置

**引擎类型**：
- 流程引擎（Process Engine）
- 计划引擎（Plan Engine）
- 工单引擎（Work Order Engine）
- 指标引擎（Metric Engine）
- 表单引擎（Form Engine）
- 关系引擎（Relationship Engine）

**业务组件类型**：
- 流程定义（Process Definition）
- 计划定义（Plan Definition）
- 工单定义（Work Order Definition）
- 指标定义（Metric Definition）
- 表单定义（Form Definition）
- 关系定义（Relationship Definition）

**示例**：
```
订单创建流程 = 订单引擎 + 订单创建流程定义
支付处理流程 = 支付引擎 + 支付处理流程定义
```

**优势**：
- 快速实现业务需求
- 支持业务人员自助配置
- 易于维护和扩展

---

### 2.2 开发方式实现的业务组件

**定义**：通过代码开发实现的业务组件

**特点**：
- 需要代码开发
- 实现复杂的业务逻辑
- 通常是平台的基础引擎

**示例**：
```
订单引擎（Order Engine）
支付引擎（Payment Engine）
库存引擎（Inventory Engine）
```

**优势**：
- 支持复杂的业务逻辑
- 性能优化空间大
- 可以实现定制化功能

---

## 三、常见错误

### 错误 1：一个 SR 映射到多个 BA 5 级节点

**❌ 错误示例**
```
SR-F-001：订单创建
├── BA-5-001：订单创建组件
├── BA-5-002：支付处理组件
└── BA-5-003：库存管理组件
```

**问题**：
- 违反了 1:1 约束
- 导致需求追踪困难
- 导致责任不清

**✅ 正确做法**
```
SR-F-001：订单创建 → BA-5-001：订单创建组件
SR-F-002：支持多种支付方式 → BA-5-002：支付处理组件
SR-F-007：库存管理 → BA-5-007：库存管理组件
```

---

### 错误 2：BA 5 级节点定义不清

**❌ 错误示例**
```
BA-5-001：订单组件
BA-5-002：用户组件
BA-5-003：商品组件
```

**问题**：
- 定义太宽泛
- 无法明确业务功能
- 无法进行有效的系统设计

**✅ 正确做法**
```
BA-5-001：订单创建组件（明确的业务功能）
BA-5-002：用户账户管理组件（明确的业务功能）
BA-5-003：商品管理组件（明确的业务功能）
```

---

### 错误 3：忽视业务组件之间的关系

**❌ 错误示例**
```
BA-5-001：订单创建组件
BA-5-007：库存管理组件
（没有记录它们之间的关系）
```

**问题**：
- 无法理解业务流程
- 无法进行有效的系统设计
- 容易遗漏关键的集成点

**✅ 正确做法**
```
BA-5-001：订单创建组件
  相关组件：BA-5-002（支付处理）、BA-5-007（库存管理）、BA-5-004（通知发送）
  业务流程：订单创建 → 库存扣减 → 支付处理 → 通知发送

BA-5-007：库存管理组件
  相关组件：BA-5-001（订单创建）、BA-5-003（订单修改）、BA-5-006（商品管理）
```

---

### 错误 4：BA 层级结构不合理

**❌ 错误示例**
```
BA-1：电商业务域
  BA-2：订单项目
    BA-3：订单对象
      BA-4：订单创建工作包
        BA-5：订单创建组件
          BA-6：订单创建子组件（不应该有）
```

**问题**：
- 层级过深
- 难以理解和维护
- 违反了 5 级层级结构

**✅ 正确做法**
```
BA-1：电商业务域
  BA-2：订单项目
    BA-3：订单对象
      BA-4：订单创建工作包
        BA-5：订单创建组件（末级，不再分解）
```

---

### 错误 5：业务组件实现方式不清

**❌ 错误示例**
```
BA-5-001：订单创建组件
（没有说明是配置方式还是开发方式）
```

**问题**：
- 无法确定实现方式
- 无法进行有效的系统设计
- 无法估算工作量

**✅ 正确做法**
```
BA-5-001：订单创建组件
  实现方式：配置方式（通过订单引擎配置生成）
  引擎类型：订单引擎（Order Engine）
  
BA-5-002：支付处理组件
  实现方式：配置方式（通过支付引擎配置生成）
  引擎类型：支付引擎（Payment Engine）
```

## 四、最佳实践

### 最佳实践 1：使用结构化格式

**推荐格式**：YAML 或表格

```yaml
组件ID: BA-5-001
组件名称: 订单创建组件
业务域: 电商业务域
项目: 订单管理项目
对象: 订单对象
工作包: 订单创建工作包

业务功能: |
  支持用户创建订单，包括选择商品、数量、收货地址等信息。

承接的 SR: SR-F-001
相关的 SR: SR-F-002, SR-F-004, SR-F-007

实现方式: 配置方式（通过订单引擎配置生成）
引擎类型: 订单引擎（Order Engine）

关键业务规则: |
  1. 订单必须包含至少一个商品
  2. 收货地址必须有效
  3. 订单创建后自动生成订单号

关键数据: |
  - 订单号
  - 商品列表
  - 收货地址
  - 订单金额
```

**优势**：
- 信息完整
- 易于解析
- 易于版本控制

---

### 最佳实践 2：明确业务功能

**每个 BA 5 级节点都应该有明确的业务功能**

```
❌ 错误：订单组件
✅ 正确：订单创建组件（明确的业务功能）

❌ 错误：用户组件
✅ 正确：用户账户管理组件（明确的业务功能）

❌ 错误：商品组件
✅ 正确：商品管理组件（明确的业务功能）
```

**优势**：
- 便于理解
- 便于沟通
- 便于系统设计

---

### 最佳实践 3：记录关联关系

**为业务组件记录相关的 SR 和其他业务组件**

```
BA-5-001：订单创建组件
  承接的 SR：SR-F-001
  相关的 SR：SR-F-002, SR-F-004, SR-F-007
  相关的业务组件：BA-5-002（支付处理）、BA-5-007（库存管理）、BA-5-004（通知发送）
```

**优势**：
- 帮助发现隐藏的依赖关系
- 便于追踪需求链路
- 便于系统设计

---

### 最佳实践 4：明确实现方式

**每个 BA 5 级节点都应该明确说明实现方式**

```
BA-5-001：订单创建组件
  实现方式：配置方式（通过订单引擎配置生成）
  引擎类型：订单引擎（Order Engine）
  
BA-5-002：支付处理组件
  实现方式：配置方式（通过支付引擎配置生成）
  引擎类型：支付引擎（Payment Engine）
```

**优势**：
- 便于系统设计
- 便于工作量估算
- 便于技术选型

**常见业务模式**：

## 六、常见问题解答

**Q：如何判断业务域的划分是否合理？**

A：业务域的划分应该满足以下条件：
- 业务域之间相对独立
- 业务域之间的依赖关系清晰
- 业务域的数量适中（通常 3-10 个）
- 业务域与组织结构相对应

**Q：一个 SR 可以映射到多个 BA 5 级节点吗？**

A：不可以。每条 SR 只能映射到唯一的 BA 5 级节点（1:1 约束）。但一个 BA 5 级节点可以承接多条 SR（N:1 关系）。

**Q：如何处理跨越多个业务域的需求？**

A：
1. 识别需求涉及的业务域
2. 为每个业务域创建相应的业务组件
3. 记录业务组件之间的关系
4. 在系统设计阶段处理跨域集成

**Q：业务组件应该如何与系统需求关联？**

A：
- 每个 BA 5 级节点应该映射到一个或多个 SysReq 5 级节点
- 在 Task 3 中进行 BA → SysReq 的映射
- 记录映射关系便于追踪

**Q：如何确定业务组件的实现方式？**

A：
- 如果业务逻辑相对简单，可以使用配置方式
- 如果业务逻辑复杂，需要使用开发方式
- 如果是平台的基础功能，通常使用开发方式
- 如果是业务特定的功能，通常使用配置方式
