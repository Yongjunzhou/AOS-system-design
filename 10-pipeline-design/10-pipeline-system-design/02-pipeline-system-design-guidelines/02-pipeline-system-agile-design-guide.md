## 📌 场景描述

### 适用条件

- ✅ 已有完整的系统设计文档（5层）
- ✅ 已建立需求映射矩阵
- ✅ 需要增加新需求或修订已有需求
- ✅ 需要快速响应需求变更
- ✅ 适合产品迭代和敏捷开发

### 不适用条件

- ❌ 没有现有的系统设计文档
- ❌ 需要从头开始进行完整设计
- ❌ 需求变更影响范围超过50%

### 参与角色

本指南涉及三个核心参与者，在敏捷式设计流程中各司其职：

| 角色 | 职责 | 参与环节 |
|------|------|---------|
| **流水线开发者** | 定义设计准则和指南，配置 AI 辅助工具，审核 AI 输出 | 所有步骤的审核确认 |
| **AI** | 按指令执行规范化、分解、映射、验证等自动化任务 | 所有步骤的自动生成和检查 |
| **AOS开发者** | 提出需求、确认设计、审核方案（含需求提出者、业务流程开发者、信息系统开发者三类） | 需求提出、增量映射、更新验证 |

## 🎯 核心目标

高效地将新需求集成到现有的系统设计中，保持文档的一致性和完整性，最小化对现有设计的影响

## 📊 关键概念

### 架构定义与详细定义的分离

每份方案文档由两个子文档构成（系统设计准则 §1.5）：

| 子文档 | 作用 | 在敏捷中的关注点 |
|-------|------|----------------|
| **架构定义文档** | 以架构末级节点为基本单位，构建方案的层级结构并建立对上层需求的映射 | 增量时检查现有架构末级节点是否能承接新需求 |
| **详细定义文档** | 对架构末级节点进行详细设计分解，使每条末级条目在下层方案中有唯一的承接点 | 增量时检查现有详细定义是否需要扩展或修改 |

### 分析活动的贯穿执行

系统设计准则要求每层在架构定义完成后、详细定义开始前，集中执行两类分析活动。在敏捷增量中，分析活动体现为**影响评估**：

- **需求说明符合性分析**：检查新需求是否能由现有架构末级节点承接，覆盖度是否完整
- **需求性能符合性分析**：检查新需求的性能指标是否在现有架构能力范围内

### 详细定义的双重身份

详细定义在流水线中扮演双重角色（系统设计准则 §1.5.2）：
- **本层身份**：作为当前方案**架构定义的细化**（向内分解）
- **下层身份**：作为流水线上**后续方案的需求**（向下传递）

在敏捷增量中，这一概念帮助判断：更新某层的详细定义时，是否需要同步更新下游方案。

### 两阶段工作流程

两阶段工作流程是系统设计准则中**同步设计原则**（§1.5.3）的具体操作方法：

- **第一阶段：结构骨架约定** — 快速扫描上游文档，识别末级节点，在下游建立结构骨架和映射关系，冻结骨架
- **第二阶段：具体设计** — 并行填充详细内容，验证完整性和一致性

在敏捷场景中，两阶段工作流程表现为：先快速评估影响范围并建立新需求与现有架构的映射骨架，再并行填充各组件的详细更新内容。

## 📋 完整操作流程（8步）

### 第1步：新需求分析和分类

**目标**：分析新需求的性质和影响范围

**涉及规范**：[原始需求规范 §1 核心概念](../01-pipeline-system-design-standards/02-pipeline-original-requirements-specification.md#一核心概念)

**具体操作**：

1. **需求分类**
   - 功能需求 vs 非功能需求——功能需求走 BA 链路，非功能需求走 NFR 平行链路
   - 新增需求 vs 修改需求 vs 删除需求
   - 独立需求 vs 相关需求

2. **影响范围评估**
   - 是否影响现有的相关方需求？
   - 是否影响现有的业务架构？
   - 是否影响现有的系统需求？
   - 是否影响现有的产品架构？

3. **优先级评估**
   - 高优先级：影响范围大，需要立即处理
   - 中优先级：影响范围中等，可以在下一个迭代处理
   - 低优先级：影响范围小，可以延后处理

4. **初步整理（可选，建议执行）**
   - 对来自多渠道的零散需求进行碎片归并、补全语境
   - 遵循五条规则：不过度合并、保留原始出处、性能指标不合并、矛盾碎片不合并、无法判断标记待补全

**输出**：
- 新需求分析报告
- 影响范围评估
- 优先级排序

---

### 第2步：新需求分解

**目标**：对新需求进行规范化处理和分解，得到末级需求

**涉及规范**：[原始需求规范 §3 分解规范](../01-pipeline-system-design-standards/02-pipeline-original-requirements-specification.md#三分解规范)

**具体操作**：

1. **需求规范化**
   - 四项检查：完整性、清晰性、可测试性、独立性
   - 模糊术语替换为精确表述
   - 功能需求和非功能需求分离

2. **需求分解**
   - 对每条新原始需求进行分解
   - **分解停止准则**：每条末级只需映射到相关方需求架构的一个末级节点
   - 末级需求的粒度应与现有末级需求一致
   ```
   新原始需求 N
   ├─ 末级需求 N.1：[具体的功能需求] → 分配到 SR-F-XXX
   ├─ 末级需求 N.2：[具体的功能需求] → 分配到 SR-F-XXX
   └─ 末级需求 N.3：[具体的非功能需求] → 分配到 SR-NFR-XXX
   ```

3. **分析活动：冲突和重复检测**
   - **需求说明符合性分析**：与新需求与现有需求之间的冲突检测、重复检测
   - **需求性能符合性分析**：验证新需求的性能指标是否明确量化

4. **原始需求文档更新**
   - 在原始需求文档中添加新需求
   - 更新末级需求清单

**输出**：
- 更新后的原始需求文档（含末级需求分解）
- 新末级需求清单
- 冲突/重复检测报告

---

### 第3步：相关方需求影响分析

**目标**：将新末级需求映射或更新到相关方需求，分析影响范围

**涉及规范**：
- [相关方需求规范 §2 架构定义规范](../01-pipeline-system-design-standards/03-pipeline-stakeholder-requirements-specification.md#二架构定义规范)
- [相关方需求规范 §3 详细定义规范](../01-pipeline-system-design-standards/03-pipeline-stakeholder-requirements-specification.md#三详细定义规范)

**具体操作**：

1. **分析活动：架构定义验证**
   - **需求说明符合性分析**：检查新末级需求是否能由现有 SR 架构末级节点承接，N:1 映射完整性
   - **需求性能符合性分析**：新需求的性能指标是否可在现有 SR 框架内量化

2. **映射策略选择**
   
   **选项A：映射到现有相关方需求**
   - 如果新需求与现有相关方需求相关
   - 更新相关方需求的承接需求清单
   - 更新符合性分析

   **选项B：创建新的相关方需求**
   - 如果新需求无法映射到现有相关方需求
   - 创建新的相关方需求架构末级节点
   - 定义新节点内容并建立映射

3. **相关方需求条目定义（增量标记）**
   ```
   + SR-F-00N: [新增的相关方需求名称]（架构末级节点）
   ├─ 定义：[需求定义]
   ├─ 承接的原始需求末级：N.1, N.2
   ├─ 符合性分析：[如何满足所承接的需求]
   └─ 设计指导：[对业务架构的指导]

   ~ SR-F-001: [修改的相关方需求名称]
   ├─ 更新说明：[变更内容]
   └─ 更新符合性分析：[更新说明]
   ```
   **标记说明**：`+` 前缀表示新增架构末级节点，`~` 前缀表示修改现有节点；新增条目遵循完整格式，修改条目仅列出变更内容。

4. **同步维护两个子文档**
   - **架构定义文档**（树形结构 + 映射关系表）— 更新承接关系和新增节点
   - **详细定义文档**（对架构末级节点的分解）— 新增或更新详细定义末级节点
   ```
   **详细定义的双重身份**：
   新增/修改的详细定义末级节点既是本层 SR 的细化（向内），
   又是下层业务架构的需求（向下传递）。
   在开展详细定义时，应同步评估对业务架构的影响（详见第4步）。
   ```

**输出**：
- 更新后的相关方需求文档（架构定义 + 详细定义同步更新）
- 新的需求映射关系（原始需求 → 相关方需求）

---

### 第4步：业务架构影响分析

**目标**：分析新需求对业务架构的影响

**涉及规范**：
- [业务架构规范 §2 架构定义规范](../01-pipeline-system-design-standards/04-pipeline-business-architecture-specification.md#二架构定义规范)
- [业务架构规范 §4.1 三阶段工作流程](../01-pipeline-system-design-standards/04-pipeline-business-architecture-specification.md#41-三阶段工作流程)
- [业务架构规范 §4.3 去重机制](../01-pipeline-system-design-standards/04-pipeline-business-architecture-specification.md#43-去重机制)

**具体操作**：

1. **分析活动**
   - **需求说明符合性分析**：检查新的 SR 详细定义末级是否能由现有业务组件承接，1:1 分配约束验证
   - **需求性能符合性分析**：性能指标是否从 SR 正确继承；**指标追溯**（每个 BA IPO 的性能指标可追溯到 SR 详细定义末级的指标要求）

2. **业务架构更新策略**
   - 采用**三阶段工作流程**（详见[业务架构规范 §4.1](../01-pipeline-system-design-standards/04-pipeline-business-architecture-specification.md#41-三阶段工作流程)）：先快速评估创建占位符和骨架映射，再填充/更新IPO内容并去重，最后验证和冻结
   
   **策略A：映射到现有业务架构**
   - 如果新 SR 详细定义可以由现有业务组件（IPO）承接
   - 更新业务组件的承接需求清单和符合性分析

   **策略B：修改现有业务架构**
   - 如果新 SR 详细定义需要修改现有业务组件
   - 分析修改的影响范围，更新 IPO 定义

   **策略C：创建新的业务组件**
   - 如果新 SR 详细定义需要新的业务组件
   - **去重检查**：先检查是否已有相同或相似的 IPO（复用→改进→新增三级策略）
   - 创建新的 IPO 并建立映射

3. **业务架构条目定义（增量标记）**
   ```
   + BA-00N: [新增的业务组件名称]（5级·架构末级节点）
   ├─ 定义：[业务组件的定义]
   ├─ Input（输入）：[具体形式、数据结构、来源]
   ├─ Process（处理）：[规则、算法、步骤、约束条件]
   ├─ Output（输出）：[具体形式、数据结构、验收标准]
   ├─ 承接的SR详细定义末级：SR-F-00N.1
   ├─ 符合性分析：[如何满足所承接的需求]
   └─ 设计指导：[对系统需求的指导]

   ~ BA-001: [修改的业务组件名称]
   ├─ 更新说明：[变更内容]
   └─ 更新符合性分析：[更新说明]
   ```

4. **同步设计**
   - 在更新/新增 IPO 的同时，同步在下层（系统需求）建立 5 级节点占位符或调整现有节点
   - 确保 BA → SysReq 的映射关系同步更新

**输出**：
- 业务架构影响分析报告
- 更新后的业务架构文档
- 新的需求映射关系（相关方需求 → 业务架构）

---

### 第5步：系统需求影响分析

**目标**：分析新需求对系统需求的影响

**涉及规范**：
- [系统需求规范 §2 架构定义规范](../01-pipeline-system-design-standards/05-pipeline-system-requirements-specification.md#二架构定义规范)
- [系统需求规范 §3 详细定义规范](../01-pipeline-system-design-standards/05-pipeline-system-requirements-specification.md#三详细定义规范)

**具体操作**：

1. **分析活动**
   - **需求说明符合性分析**：BA IPO 去重后是否能由现有 SysReq 5 级节点承接，覆盖度检查
   - **需求性能符合性分析**：性能指标是否从 BA 正确继承，**指标追溯**（每个SysReq节点性能指标可追溯到BA IPO或SR-NFR），在系统层面评估可行性

2. **系统需求更新策略**
   
   **功能需求分支**：
   - 采用 **0-9 级层级结构**（5 级为架构末级承接 BA，9 级为详细定义末级分配到 PA）
   - 先对 BA IPO 去重（复用→改进→新增），再映射到 SysReq 5 级

   **非功能需求分支**（平行架构，不经 BA）：
   - 采用 **0-6 级平行架构**（5 级为承接层，6 级为验证标准）
   - 按三大类组织：质量特性 / 环境适应性 / 可实现性

3. **系统需求条目定义（增量标记）**
   ```
   + SysReq-F-00N: [新增的系统功能需求]（5级·架构末级节点）
   ├─ 定义：[系统需求的定义]
   ├─ 类型：功能需求
   ├─ 承接的BA：BA-00N
   ├─ 符合性分析：[如何满足所承接的需求]
   ├─ 设计指导：[对产品架构的指导]
   └─ 性能指标：[继承自BA]

   + SysReq-NFR-00N: [新增的系统非功能需求]（5级·架构末级节点）
   ├─ 定义：[非功能需求的定义]
   ├─ 类型：[性能/安全/可用性等]
   ├─ 承接的SR-NFR：SR-NFR-00N
   ├─ 符合性分析：[如何满足所承接的需求]
   ├─ 量化指标：[具体数值、单位、条件]
   └─ 约束条件：[对产品架构的约束]
   ```

4. **同步维护两个子文档**
   - **架构定义文档**（树形结构 + 映射关系表）— 更新 5 级承接节点
   - **详细定义文档**（6-9 级分解）— 更新分解链直到 9 级场景活动
   - 在开展系统需求详细定义的同时，同步评估对产品架构的影响（详见第6步）

**输出**：
- 系统需求影响分析报告
- 更新后的系统需求文档（架构定义 + 详细定义同步更新）
- 新的需求映射关系（业务架构 → 系统需求）

---

### 第6步：产品架构影响分析

**目标**：分析新需求对产品架构的影响

**涉及规范**：
- [产品架构规范 §2 架构定义规范](../01-pipeline-system-design-standards/06-pipeline-product-architecture-specification.md#二架构定义规范)
- [产品架构规范 §3 详细定义规范](../01-pipeline-system-design-standards/06-pipeline-product-architecture-specification.md#三详细定义规范)

**具体操作**：

1. **分析活动**
   - **需求说明符合性分析**：所有新 SysReq 9 级节点是否都有对应的 PA 构件承接，1:1 分配验证；N:1承接合理性（每个PA构件承接2-5个SysReq 9级节点合理）
   - **需求性能/质量符合性分析**：PA 构件性能/质量指标不低于所承接 SysReq 和 NFR 的要求；**可实现性评估**（结合技术选型和架构方案评估指标的可实现性）；**指标追溯**（每个 PA 构件的性能指标可追溯到 SysReq 9 级节点或 SR-NFR）；NFR 冲突时记录权衡决策

2. **产品架构更新策略**
   
   **策略A：映射到现有产品构件**
   - 如果新系统需求可以由现有产品构件承接
   - 更新构件定义和符合性分析

   **策略B：修改现有产品构件**
   - 如果新系统需求需要修改现有产品构件
   - 更新构件定义、接口、性能指标

   **策略C：创建新的产品构件**
   - 如果新系统需求需要新的产品构件
   - 按**4大分类**（规范与准则、操作指南、AI辅助文档、任务定义）确定构件类型
   - 创建新的构件并建立映射

3. **产品架构条目定义（增量标记）**
   ```
   + PA-00N: [新增的产品构件名称]（3级·构件/末级）
   ├─ 定义：[构件职责和功能描述]
   ├─ 构件类型：[前端/后端/数据/基础设施]（因产品类型而异）
   ├─ 承接的系统需求：SysReq-F-00N.1.1
   ├─ 符合性分析：[如何满足所承接的需求]
   ├─ 实现细节：[技术栈、关键模块、数据存储]
   └─ 性能指标：[响应时间、吞吐量等]

   ~ PA-001: [修改的产品构件名称]
   ├─ 更新说明：[变更内容]
   └─ 更新符合性分析：[更新说明]
   ```

   **示例：知识/工具类产品**增量定义：
   ```
   + PA-00N: [新增的文档/工具构件名称]（3级·构件/末级）
   ├─ 定义：[构件职责和功能描述]
   ├─ 构件类型：[规范与准则/操作指南/AI辅助工具/任务定义]
   ├─ 承接的系统需求：SysReq-F-00N.1.1
   ├─ 符合性分析：[如何满足所承接的需求]
   ├─ 对应文件：[文件名或目录路径]
   └─ 质量指标：[完整性、一致性、清晰性等]

   ~ PA-001: [修改的文档/工具构件名称]
   ├─ 更新说明：[变更内容]
   └─ 更新符合性分析：[更新说明]
   ```

4. **详细定义同步更新**
   - 对新增/修改的产品构件，补充**详细定义**（文档格式、内容结构、版本策略、引用关系、质量指标等）
   - 确保构件的性能/质量指标不低于 SysReq 和 NFR 要求
   - 标注构件受哪些 NFR 约束

**输出**：
- 产品架构影响分析报告
- 更新后的产品架构文档（架构定义 + 详细定义）
- 新的需求映射关系（系统需求 → 产品架构）

### 第7步：完整性和一致性验证

**目标**：验证增量更新后的系统设计的完整性和一致性

**涉及规范**：
- [系统设计准则 §5 验收标准](../01-pipeline-system-design-standards/01-pipeline-system-design-standards.md#五验收标准)
- [系统设计准则 §6 检查清单](../01-pipeline-system-design-standards/01-pipeline-system-design-standards.md#六检查清单)
- [系统设计准则 §3.2 三条核心规则](../01-pipeline-system-design-standards/01-pipeline-system-design-standards.md#32-三条核心规则)

**具体操作**：

1. **1:1 分配约束验证**
   - 验证所有新需求末级都分配到唯一的方案架构末级节点
   - 没有遗漏或重复的映射

2. **N:1 承接支持验证**
   - 验证每个方案架构末级节点的承接数量合理（2-5 条/节点）
   - 发现过度分解或过度聚合的情况

3. **双向追溯验证**
   - **正向**：新需求末级 → SR → BA → SysReq → PA
   - **反向**：PA → SysReq → BA → SR → 新需求末级
   - 所有新增/修改的节点都标注了承接来源和分配目标

4. **符合性分析完整率验证**
   - 所有新增/修改的方案节点都有符合性分析
   - 符合性分析清晰说明了如何满足所承接的需求

5. **同步设计一致性验证**
   - 上层详细定义的更新是否与下层架构定义的更新一致
   - 没有"更新了上层但漏掉了下层"的情况

6. **版本一致性验证**
   - 验证所有文档的版本号一致
   - 验证所有文档的更新时间一致

**输出**：
- 验证报告（完整性和一致性分析）
- 问题清单和改进建议

## 🎯 增量处理策略

### 1. 优先级处理

**原则**：
- 优先处理高优先级需求
- 分阶段处理，避免一次性大量变更
- 每个阶段都进行验证和审查

**实施**：
```
第一阶段：处理高优先级需求
  ├─ 分析和分解 → 映射到各层 → 验证 → 发布

第二阶段：处理中优先级需求
  ├─ 分析和分解 → 映射到各层 → 验证 → 发布

第三阶段：处理低优先级需求
  ├─ 分析和分解 → 映射到各层 → 验证 → 发布
```

### 2. 影响范围最小化

**原则**：
- 优先选择映射到现有方案条目
- 避免不必要的修改
- 避免过度分解或过度聚合

**实施**：
```
映射决策树：
  新需求是否可以映射到现有相关方需求？
    ├─ 是 → 映射到现有 SR 架构末级节点
    │       └─ 是否可以映射到现有业务架构？
    │           ├─ 是 → 映射到现有 BA IPO（先去重检查）
    │           │       └─ 是否可以映射到现有系统需求？
    │           │           ├─ 是 → 映射到现有 SysReq 5 级节点
    │           │           │       └─ 是否可以映射到现有产品架构？
    │           │           │           ├─ 是 → 完成映射
    │           │           │           └─ 否 → 创建新 PA 构件
    │           │           └─ 否 → 创建新 SysReq 节点
    │           └─ 否 → 创建新 BA IPO
    └─ 否 → 创建新 SR 架构末级节点
```

### 3. 两阶段工作流程在敏捷中的应用

**第一阶段：结构骨架评估（快速）**
- 快速扫描上游文档的新增/修改节点
- 在下游文档中建立对应的结构骨架占位符
- 建立上下游的映射关系
- 冻结骨架结构

**第二阶段：详细内容填充（并行）**
- 在结构骨架中填充具体内容
- 各层可并行更新
- 为下一层级同步建立结构骨架
- 验证更新的完整性和一致性

## 🔗 跨步骤一致性检查

以下检查项覆盖所有步骤，确保增量更新的整体一致性：

- [ ] **术语一致性**：全文使用统一定义（架构末级节点/详细定义末级节点/1:1分配约束/N:1承接支持）
- [ ] **映射流向一致**：分配方向统一向下（↓），承接方向统一向上（↑）
- [ ] **层级结构完整**：每层从根节点到末级节点的树形结构完整（BA 0-5级，SysReq 功能0-9级/非功能0-6级）
- [ ] **增量标记规范**：使用标准化的 `+`（新增）/ `~`（修改）标记，与现有文档标记一致
- [ ] **1:1约束满足率100%**：每条详细定义末级分配到唯一的架构末级节点
- [ ] **N:1承接合理性**：修改后的架构末级节点承接 2-5 条详细定义末级为合理（1条需考虑合并，6+条需拆分）
- [ ] **映射覆盖率100%**：逐层映射无遗漏
- [ ] **详细定义的双重身份理解**：更新详细定义后检查下游方案是否同步更新
- [ ] **同步设计执行**：上层详细定义修改后，下层架构已同步调整
- [ ] **符合性分析完整**：修改后的方案节点都更新了符合性分析
- [ ] **超链接有效**：所有映射关系保留了双向导航的可追溯性
- [ ] **版本更新**：所有修改过的文档版本号已更新

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 新需求与现有需求冲突 | 需要重新设计 | 进行需求冲突分析（第2步的分析活动），协调相关方 |
| 新需求影响范围大 | 需要大量修改 | 分阶段处理，优先处理高优先级需求 |
| 现有设计无法承接新需求 | 需要重新架构 | 进行架构评估，考虑是否需要重构 |
| 上层更新后下层未同步更新 | 追溯链路断裂 | 严格执行同步设计原则，每步都检查下层骨架 |
| 增量标记混淆 | 版本管理混乱 | 使用标准化的 `+`/`~` 增量标记，更新变更日志 |

## 📝 使用建议

1. **需求到达时**
   - 进行快速的影响范围评估

   - 每步完成架构定义后先做分析活动，再更新详细定义（避免返工）

3. **设计完成后**
   - 进行完整性和一致性验证
   - 更新需求映射矩阵
   - 进行团队沟通
