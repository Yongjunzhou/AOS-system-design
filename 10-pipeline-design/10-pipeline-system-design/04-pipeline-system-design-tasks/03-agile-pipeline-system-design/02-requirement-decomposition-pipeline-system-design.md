# task-02：新需求分解

**对应场景**：03-agile · 第2步

## 目标

对新需求进行规范化处理和分解，与现有需求进行冲突和重复检查。

## 上下文继承

```
从哪里来：
  - task-01 产出：新需求分析报告（分类 + 影响范围 + 优先级）
  - 参考基线：现有原始需求文档

去哪里：
  - 本步产出 → task-03（SR 影响分析）的新末级需求清单
```

---

## 输入

| 输入 | 说明 |
|------|------|
| 新需求分析报告 | task-01 产出 |

## 操作流程

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 需求规范化 | AI | 四项检查（完整性/清晰性/可测试性/独立性） |
| 2 | 功能/非功能分离 | AI | 将"做什么"和"有多好"分开记录 |
| 3 | 需求分解到末级 | AI | 原子性原则，1:1 约束 |
| 4 | 分析活动：冲突和重复检测 | AI | 需求说明符合性分析（新需求与现有需求的冲突/重复检测）+ 需求性能符合性分析（新需求性能指标是否明确量化） |
| 5 | 结果审核 | 人类 | 确认分解和冲突处理 |

## 人类审核要点

- [ ] 冲突检测的最终决策 — 需相关方确认
- [ ] 需求分解粒度是否与现有需求一致
- [ ] 功能/非功能分离是否正确
- [ ] 分解停止准则是否满足（每条末级只需映射到 SR 架构的一个末级节点）

## 质量门禁

- [ ] 分析活动：冲突和重复检测已执行
- [ ] 新原始需求末级清单完整
- [ ] 功能/非功能分离完成
- [ ] 冲突/重复检测完成
- [ ] 运行 conformance-review 进行符合性审查
- [ ] 按审查修复流程处理问题
