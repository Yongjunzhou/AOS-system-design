## 目标

沿非功能分支独立推进，将 SR-NFR 详细定义并量化，建立系统需求非功能部分架构。

## 上下文继承

```
从哪里来：
  - 第1步产出：相关方需求架构定义（非功能部分 SR-NFR）
  - 涉及规范：系统需求规范 §2.2、设计准则 §3.1 第4步

并行说明：
  - 本步（非功能分支）与 task-02 + task-03（功能分支）完全独立并行
  - 两个分支可同时推进，不受彼此影响
  - task-05 合并时需进行覆盖度检查

去哪里：
  - 本步产出 → task-05（分支合并），约束产品架构的 NFR 要求
```

## 输入

| 输入 | 说明 |
|------|------|
| SR 架构定义（非功能部分 SR-NFR） | task-01 产出 |

## 操作流程

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 分析活动：非功能架构验证 | AI | 需求说明符合性分析（承接完整性）+ 需求性能符合性分析（指标量化/可测量性） |
| 2 | SR-NFR 详细定义 | AI | 每条非功能需求量化（指标+单位+条件+测量方法） |
| 3 | 非功能需求 0-6 级架构设计 | AI | 按三大类（质量特性/环境适应性/可实现性）组织 |
| 4 | 结果审核 | 人类 | 确认量化指标的合理性 |

## 人类审核要点

- [ ] 非功能需求量化指标数值（含数值+单位+条件）——需架构师确认
- [ ] 测量方法是否可行
- [ ] 非功能需求分类是否正确（三大类：质量特性/环境适应性/可实现性）
- [ ] 功能分支与非功能分支之间是否存在冲突

## 质量门禁

- [ ] 分析活动已完成（指标量化率 100%）
- [ ] SR-NFR 详细定义已量化（每条含数值+单位+条件+测量方法）
- [ ] 非功能需求按三大类分类组织
- [ ] 系统需求非功能部分架构定义完整（0-6 级平行架构 + 映射关系表）
- [ ] 与功能分支无冲突
- [ ] 运行 rule-conformance-review 进行符合性审查，按审查修复流程处理问题
