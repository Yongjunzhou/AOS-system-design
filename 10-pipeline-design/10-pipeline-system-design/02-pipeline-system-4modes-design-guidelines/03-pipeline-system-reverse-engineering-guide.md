## 📌 场景描述

### 适用条件

- ✅ 产品已开发完成或接近完成
- ✅ 有详细的代码实现或架构图纸
- ✅ 技术文档不全或不符合设计准则
- ✅ 缺少从需求到架构的完整追溯链路
- ✅ 需要生成符合准则的系统设计文档

### 不适用条件

- ❌ 产品代码不清晰或文档极度缺失
- ❌ 原开发人员无法参与
- ❌ 需要快速完成（时间紧张）

### 参与角色

本指南涉及三个核心参与者，在逆向工程流程中各司其职：

| 角色 | 职责 | 参与环节 |
|------|------|---------|
| **流水线开发者** | 定义设计准则和指南，配置 AI 辅助工具，审核 AI 输出 | 所有步骤的审核确认 |
| **AI** | 按指令执行代码分析、需求反向推导、映射关系建立等自动化任务 | 所有步骤的自动生成和检查 |
| **AOS开发者** | 提出需求、确认设计、审核方案（含需求提出者、业务流程开发者、信息系统开发者三类） | 代码分析、文档编写、方案审核 |

## 🎯 核心目标

通过逆向工程，从代码和图纸中提取设计信息，重建符合设计准则的系统设计文档

```
产品代码/图纸 → 产品架构 → 系统需求 → 业务架构 → 相关方需求 → 原始需求
   第5层        第4层       第3层       第2层       第1层
```

---

## 📊 关键概念

### 逆向工程的流向特点

与正向（瀑布式）设计不同，逆向工程的推导方向是 **自底向上**（第5层→第1层）。这意味着：

- 每步的"分析活动"从"检查完整性"变为"验证推导一致性"——即衍生出的上层需求是否与下层实现一致
- **同步设计原则**在逆向中表现为：在推导上层时，需同步检查下层架构是否能合理承接
- **详细定义的双重身份**（既是本层细化又是下层需求）在逆向中帮助判断：从下层代码能否合理反推出上层需求

### 架构定义与详细定义在逆向中的关注点

每份方案文档由两个子文档构成（系统设计准则 §1.5），在逆向中需注意：

| 子文档 | 正向设计 | 逆向工程 |
|-------|---------|---------|
| **架构定义文档** | 先定义架构再分解 | 从代码的模块结构推导出架构树 |
| **详细定义文档** | 对架构末级节点分解 | 从代码的实现细节反推出分解条目 |

### 分析活动的逆向适配

系统设计准则要求每层在架构定义完成后执行两类分析活动。在逆向中适配为：

- **需求说明符合性分析**：从下层反推上层的需求说明，验证"下层实现是否完整对应于上层需求"
- **需求性能符合性分析**：从代码的实现性能反向推导应满足的性能指标要求

### 两阶段工作流程在逆向中的应用

由于逆向是自底向上推导，两阶段工作流程（§1.5.3）表现为：

- **第一阶段**：先快速从代码提取各层结构骨架（节点名称 + 映射关系），不做深度内容
- **第二阶段**：在各层骨架建立后，再并行填充详细内容
- 这与正向的区别在于：正向是先有需求再设计方案，逆向是先有实现再追溯需求

## 📋 完整操作流程（8步）

### 第1步：代码和图纸分析

**目标**：理解产品的整体架构和功能模块

**涉及规范**：[产品架构规范 §2 架构定义规范](../01-pipeline-system-design-standards/06-pipeline-product-architecture-specification.md#二架构定义规范)

**具体操作**：

1. **架构图纸分析**
   - 收集所有架构图纸（系统架构图、模块图、数据流图等）
   - 识别主要的系统组件和模块
   - 识别组件之间的依赖关系
   - 记录关键的技术决策和设计考虑

2. **代码结构分析**
   - 分析代码的目录结构和模块划分
   - 识别主要的功能模块和子模块
   - 分析模块之间的调用关系
   - 识别关键的业务流程和数据流

3. **功能清单提取**
   - 从代码中提取所有实现的功能
   - 按照功能模块进行分类
   - 记录每个功能的实现方式和关键参数

4. **分析活动：初版架构评估**
   - **需求说明符合性分析**：提取的功能模块是否构成一个完整的产品架构？是否有明显缺失的构件？
   - **需求性能符合性分析**：从代码的性能特征反推产品应满足的性能指标范围；**指标追溯**（每个识别出的功能点或性能特征可关联到产品架构的特定构件）

**输出**：
- 产品架构草图（第5层）
- 功能模块清单
- 关键技术决策记录

> **分析要点**：代码分析中识别的每个功能模块对应一个或多个产品架构构件。功能模块清单中的条目将作为第2步中产品架构构件的输入，建议先对功能模块按**4大分类**（规范与准则、操作指南、AI辅助文档、任务定义）进行归类，再映射到架构末级节点。

### 第2步：产品架构文档编写

**目标**：编写符合准则的产品架构文档

**涉及规范**：
- [产品架构规范 §2 架构定义规范](../01-pipeline-system-design-standards/06-pipeline-product-architecture-specification.md#二架构定义规范)
- [产品架构规范 §3 详细定义规范](../01-pipeline-system-design-standards/06-pipeline-product-architecture-specification.md#三详细定义规范)
- [系统设计准则 §1.5 架构定义与详细定义的分离](../01-pipeline-system-design-standards/01-pipeline-system-design-standards.md#15-架构定义与详细定义的分离)

**具体操作**：

1. **产品构件识别和定义**
   - 从代码模块和功能清单中识别产品构件
   - 按**4大分类**（规范与准则、操作指南、AI辅助文档、任务定义）确定构件类型
   - 按**构件划分原则**判断粒度是否合适（以"可由单人独立完成"且"可独立审查和验收"为双重标准）
   ```
   PA-001: [产品构件名称]（3级·架构末级节点）
   ├─ 定义：[构件的职责和功能描述]
   ├─ 构件类型：[前端/后端/数据/基础设施]（因产品类型而异）
   ├─ 实现细节：[代码位置、关键类/函数、技术栈]
   ├─ 性能指标：[从代码反推的性能特征]
   └─ 承接的系统需求：（待第3步完成后补充）
   ```

2. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构）**
   ```
   产品架构：系统设计流水线（0级）
   ├─ 设计规范与准则（1级）
   │  └─ 系统设计准则体系（2级）
   │     ├─ 系统设计准则总纲（3级·构件/末级）
   │     ├─ 原始需求规范（3级·构件/末级）
   │     ├─ 相关方需求规范（3级·构件/末级）
   │     ├─ 业务架构规范（3级·构件/末级）
   │     ├─ 系统需求规范（3级·构件/末级）
   │     └─ 产品架构规范（3级·构件/末级）
   │
   ├─ 设计指南（1级）
   │  └─ 场景设计指南（2级）
   │     ├─ 瀑布式设计指南（3级·构件/末级）
   │     ├─ 敏捷式设计指南（3级·构件/末级）
   │     ├─ 逆向工程设计指南（3级·构件/末级）
   │     └─ DevOps设计指南（3级·构件/末级）
   │
   ├─ AI辅助文档（1级）
   │  └─ 设计场景AI支持（2级）
   │     ├─ 通用AI辅助（3级·构件/末级）
   │     ├─ 瀑布式AI辅助（3级·构件/末级）
   │     ├─ 敏捷式AI辅助（3级·构件/末级）
   │     ├─ 逆向工程AI辅助（3级·构件/末级）
   │     └─ DevOps AI辅助（3级·构件/末级）
   │
   └─ 任务定义（1级）
      └─ 六步法核心任务（2级）
         ├─ 原始需求分析及SR架构定义（3级·构件/末级）
         ├─ SR详细定义及BA定义（3级·构件/末级）
         ├─ SysReq功能部分架构定义（3级·构件/末级）
         ├─ SR-NFR详细定义及SysReq-NFR架构定义（3级·构件/末级）
         ├─ SysReq详细定义及PA定义（3级·构件/末级）
         └─ 双向追溯验证（3级·构件/末级）
   ```


   **第二部分：需求映射视图（预留）**
   - 映射关系表暂时预留，将在第3步系统需求反向推导完成后填充
   - 显示每个 PA 构件承接了哪些 SysReq 9 级节点
   - 产品的映射关系表以"对应交付文件"替代"构件类型"，详见[产品架构规范 §2.4](../01-pipeline-system-design-standards/06-pipeline-product-architecture-specification.md#24-架构定义的文档结构)

3. **详细定义同步编写**
   - 每个产品构件补充**详细定义**（文档格式、内容结构、版本策略、引用关系、质量指标等）
   - 详细定义是架构末级节点的实现细化，也是后续追溯的依据
   ```
   PA-001 详细定义示例：
   ├─ 文档格式: Markdown + Mermaid
   ├─ 内容结构: 总则→五层结构→核心规则→设计要点→产品构件
   ├─ 关键条目: 五层结构含完整图示、三条规则含正例反例
   ├─ 版本策略: 语义化版本号（主版本·次版本）
   └─ 质量指标: 完整性100%、一致性0矛盾
   ```

**输出**：
- 产品架构文档（架构定义 + 详细定义）
- 待下游步骤填充的映射关系表骨架

### 第3步：系统需求反向推导

**目标**：从产品架构反向推导系统需求

**涉及规范**：
- [系统需求规范 §2 架构定义规范](../01-pipeline-system-design-standards/05-pipeline-system-requirements-specification.md#二架构定义规范)
- [系统需求规范 §3 详细定义规范](../01-pipeline-system-design-standards/05-pipeline-system-requirements-specification.md#三详细定义规范)

**具体操作**：

1. **分析活动：PA→SysReq 追溯验证**
   - **需求说明符合性分析**：每个 PA 构件实现的每个功能，能否反推出一条合理的 SysReq 9 级场景活动？是否有功能实现了但缺少对应的 SysReq？
   - **需求性能符合性分析**：从 PA 构件的实现性能反推 SysReq 的性能指标范围；**指标追溯**（每个 SysReq 节点的性能指标可追溯到 PA 构件的实现特征）

2. **系统需求识别**
   - 分析每个产品构件实现了什么功能
   - 识别功能性需求（做什么）
   - 识别非功能性需求（性能、安全、可靠性等）——采用 **0-6 级平行架构**，按三大类组织
   - 记录每个系统需求的定义和验收标准

3. **系统需求条目定义**

   **功能需求部分（0-9级结构）**：
   - 从 PA 构件反推 SysReq 9 级场景活动（第9级是最细粒度的活动步骤）
   - 多条 9 级活动按场景聚合为 8 级 → 7 级 → 6 级 → 5 级节点
   - **5 级节点（组件实例）** 是架构末级节点，承接后续第4步的业务架构 IPO
   ```
   SysReq-F-001: [系统功能需求名称]（5级·架构末级节点）
   ├─ 定义：[系统需求的定义]
   ├─ 类型：功能需求
   ├─ 对应的PA构件：PA-001, PA-002
   ├─ 符合性分析：[如何通过产品架构实现]
   ├─ 设计指导：[对业务架构的指导]
   └─ 性能指标：[从PA反推的性能要求]

   （详细定义：6级→7级→8级→9级分解链）
   SysReq-F-001.1.1.1.1（9级·场景活动）
   ├─ 描述：[活动步骤说明]
   ├─ 分配到：PA-001（产品构件）
   └─ 验收标准：[验收条件]
   ```

   **非功能需求部分（0-6级平行架构）**：
   - 从 PA 构件的非功能特征反推系统非功能需求
   - 按三大类分类：质量特性 / 环境适应性 / 可实现性
   ```
   SysReq-NFR-001: [系统非功能需求名称]（5级·架构末级节点）
   ├─ 定义：[非功能需求的定义]
   ├─ 类型：[性能/安全/可用性等]
   ├─ 量化指标：[从代码反推的具体数值、单位、条件]
   ├─ 符合性分析：[如何通过产品架构实现]
   └─ 约束条件：[对产品架构的约束]
   ```

4. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构）**
   - 功能分支采用 **0-9 级层级结构**（5 级为架构末级，9 级为详细定义末级）
   - 非功能分支采用 **0-6 级平行架构**（独立分类组织）

   **第二部分：需求映射视图**
   - 显示每个 SysReq 末级节点对应的 PA 构件
   - 同时更新第2步 PA 文档中的映射关系表

5. **同步设计**
   - 完成 SysReq 定义后，更新第2步 PA 文档的"承接系统需求"字段
   - 同步在下层（业务架构）建立 5 级节点占位符（为第4步做准备）

**输出**：
- 系统需求文档（架构定义 + 详细定义）
- 需求映射关系表（PA ↔ SysReq）

### 第4步：业务架构反向推导

**目标**：从系统需求反向推导业务架构

**涉及规范**：
- [业务架构规范 §2 架构定义规范](../01-pipeline-system-design-standards/04-pipeline-business-architecture-specification.md#二架构定义规范)
- [业务架构规范 §3 IPO模型](../01-pipeline-system-design-standards/04-pipeline-business-architecture-specification.md#三ipo模型)
- [业务架构规范 §4.3 去重机制](../01-pipeline-system-design-standards/04-pipeline-business-architecture-specification.md#43-去重机制)

**具体操作**：

1. **分析活动：SysReq→BA 追溯验证**
   - **需求说明符合性分析**：从 SysReq 5 级节点是否能推导出合理的业务操作过程（IPO）？是否存在 SysReq 实现了但 BA 中说不清业务语义的情况？
   - **需求性能符合性分析**：业务层面的性能指标是否覆盖了 SysReq 的性能要求；**指标追溯**（每个 BA IPO 的性能指标可追溯到 SysReq 的性能指标）

2. **业务架构识别**
   - 分析系统需求的业务含义
   - 识别业务组件和业务流程
   - 每个业务组件使用 **IPO 模型**（Input-Process-Output）三维定义

3. **IPO 去重**
   - 逆向推导时，多个 SysReq 节点可能对应相同的业务操作
   - 采用**三阶段工作流程**（详见[业务架构规范 §4.1](../01-pipeline-system-design-standards/04-pipeline-business-architecture-specification.md#41-三阶段工作流程)）：先快速评估创建占位符和骨架映射，再填充/更新IPO内容并去重，最后验证和冻结
   - **去重三级策略**：复用→改进→新增（优先复用已识别的 IPO）
   - 去重后的 IPO 清单才是最终的业务架构输出

4. **业务架构条目定义**
   ```
   BA-001: [业务组件名称]（5级·架构末级节点）
   ├─ 定义：[业务组件的定义]
   ├─ Input（输入）：[输入的具体形式、数据结构、来源]
   ├─ Process（处理）：[规则、算法、步骤、约束条件]
   ├─ Output（输出）：[输出的具体形式、数据结构、验收标准]
   ├─ 对应的SysReq：SysReq-F-001, SysReq-F-002
   ├─ 符合性分析：[如何通过SysReq实现该业务操作]
   └─ 设计指导：[对系统需求的指导]
   ```

   > **业务架构仅包含架构定义（IPO清单），无详细定义**：业务架构的架构末级节点（IPO）自身承担"需求"角色，直接映射到系统需求。这与 SysReq 和 PA 不同——后两者既有架构定义也有详细定义。

5. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构，0-5级）**
   ```
   0级：架构根节点
   1级：业务域
   2级：端到端项目
   3级：项目业务对象架构
   4级：工作包（WBS）
   5级：业务组件（承接层）★ 末级
   └─ BA-001: [IPO名称]
        ├─ 定义：[业务组件定义]
        └─ Input / Process / Output
   ```

   **第二部分：需求映射视图**
   - 显示每个 BA IPO 对应的 SysReq 5 级节点
   - 同时更新第3步 SysReq 文档中的"承接 BA"字段

**输出**：
- 业务架构文档（架构定义 / IPO清单 + 映射关系表）
- 需求映射关系表（SysReq ↔ BA）

### 第5步：相关方需求反向推导

**目标**：从业务架构反向推导相关方需求

**涉及规范**：
- [相关方需求规范 §2 架构定义规范](../01-pipeline-system-design-standards/03-pipeline-stakeholder-requirements-specification.md#二架构定义规范)
- [相关方需求规范 §3 详细定义规范](../01-pipeline-system-design-standards/03-pipeline-stakeholder-requirements-specification.md#三详细定义规范)

**具体操作**：

1. **分析活动：BA→SR 追溯验证**
   - **需求说明符合性分析**：从 BA IPO 能否反推出清晰的相关方需求描述？业务操作是否都有对应的相关方诉求？
   - **需求性能符合性分析**：SR 层面的性能指标是否覆盖了 BA 的性能要求；**指标追溯**（每个 SR 节点的性能指标可追溯到 BA IPO 的性能指标）

2. **相关方需求识别**
   - 分析业务架构的相关方需求
   - 功能需求（SR-F）和非功能需求（SR-NFR）**分别处理**
   - 非功能需求不经过 BA 层，直接从产品代码反推

3. **相关方需求条目定义**

   **功能需求部分（SR-F）**：
   ```
   SR-F-001: [相关方需求名称]（架构末级节点）
   ├─ 定义：[需求的定义]
   ├─ 承接的原始需求末级：（待第6步完成后补充）
   ├─ 符合性分析：[如何通过业务架构实现]
   ├─ 设计指导：[对业务架构的指导]
   └─ 性能指标：[从BA继承的性能要求]

   （详细定义末级节点）
   SR-F-001.01: [需求的分解条目]
   ├─ 描述：[详细说明]
   ├─ 分配到：BA-001（业务架构IPO）
   └─ 验收标准：[验收条件]
   ```

   **非功能需求部分（SR-NFR）**：
   ```
   SR-NFR-001: [相关方非功能需求名称]（架构末级节点）
   ├─ 定义：[非功能需求的定义]
   ├─ 量化指标：[具体数值、单位、条件]
   ├─ 测量方法：[如何验证]
   └─ 分配到：SysReq-NFR-001（系统非功能需求）
   ```

4. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构）**
   - 功能需求分类 + 非功能需求分类（明确分离）

   **第二部分：需求映射视图**
   - 显示每个 SR 架构末级节点对应的 BA IPO
   - 同时更新第4步 BA 文档中的"承接 SR"字段

5. **详细定义的双重身份**
   - SR 的详细定义末级节点既是本层架构的细化（向内），又是 BA 的需求输入（向下传递）
   - 确保每个 SR 详细定义末级都能分配到**唯一**的 BA IPO（1:1 约束）

**输出**：
- 相关方需求文档（架构定义 + 详细定义）
- 需求映射关系表（BA ↔ SR）

### 第6步：原始需求反向推导

**目标**：从相关方需求反向推导原始需求

**涉及规范**：[原始需求规范 §3 分解规范](../01-pipeline-system-design-standards/02-pipeline-original-requirements-specification.md#三分解规范)

**具体操作**：

1. **分析活动：SR→OR 追溯验证**
   - **需求说明符合性分析**：从 SR 架构末级节点能否合理反推出原始需求的表述？每条 SR 是否有真实的业务来源？
   - **需求性能符合性分析**：原始需求的性能指标是否与 SR 的量化指标一致；**指标追溯**（原始需求的指标可从 SR 的量化指标追溯）

2. **原始需求识别**
   - 分析相关方需求的来源
   - 将 SR 架构末级节点聚合为原始需求条目
   - 识别原始需求的末级需求并分配到 SR 架构末级节点

3. **原始需求条目定义**
   ```
   OR-001: [原始需求名称]
   ├─ 规范化描述：[经过规范化处理的描述]
   ├─ 末级需求分解：
   │  ├─ 末级需求 1.1 → SR-F-001
   │  ├─ 末级需求 1.2 → SR-F-002
   │  └─ 末级需求 1.3 → SR-NFR-001
   ├─ 类型：[功能需求/非功能需求]
   └─ 优先级：[P0/P1/P2/P3]
   ```

4. **原始需求文档结构**
   - 原始需求条目清单（规范化后的简洁格式）
   - 末级需求分解清单
   - 末级需求 → 相关方需求的分配关系

5. **同步设计**
   - 完成 OR 定义后，更新第5步 SR 文档中的"承接原始需求末级"字段
   - 至此，所有 5 层文档的映射关系均已闭环

**输出**：
- 原始需求文档（含末级需求分解）
- 完整的映射关系表（OR → SR）

### 第7步：完整性和一致性验证

**目标**：验证逆向工程结果的完整性和一致性

**涉及规范**：
- [系统设计准则 §5 验收标准](../01-pipeline-system-design-standards/01-pipeline-system-design-standards.md#五验收标准)
- [系统设计准则 §6 检查清单](../01-pipeline-system-design-standards/01-pipeline-system-design-standards.md#六检查清单)
- [系统设计准则 §3.2 三条核心规则](../01-pipeline-system-design-standards/01-pipeline-system-design-standards.md#32-三条核心规则)

**具体操作**：

1. **分析活动：全链路合并验证**
   - **需求说明符合性分析汇总**：逐层验证映射覆盖率 100%，从 OR→SR→BA→SysReq→PA 每个节点都有明确的承接/分配关系
   - **需求性能符合性分析汇总**：性能指标从 OR 到 PA 逐层继承验证，一致性确认；**可实现性评估**（产品架构能否满足所有性能要求）；**指标追溯**（端到端性能指标可逐层追溯）

2. **1:1 分配约束验证**
   - 验证每条需求都分配到唯一的方案架构末级节点
   - 范围：OR 末级→SR、SR 详细定义末级→BA IPO、SysReq 9 级→PA
   - 发现遗漏或重复的映射

3. **N:1 承接支持验证**
   - 验证每个方案架构末级节点的承接数量合理（2-5 条/节点）
   - 发现过度分解或过度聚合的情况

4. **双向追溯验证**
   - **正向**：从原始需求末级 → 产品架构（自顶向下验证）
   - **反向**：从产品架构 → 原始需求末级（自底向上验证——这是逆向工程的核心产出）
   - 验证所有超链接的有效性

5. **符合性分析完整率验证**
   - 所有方案节点（SR/BA/SysReq/PA）是否都有符合性分析
   - 每个符合性分析是否清晰说明了如何满足所承接的需求

6. **同步设计一致性验证**
   - 每层的架构定义与详细定义之间的映射是否一致
   - 各层之间没有"推导了上层但漏了下层"的情况

**输出**：
- 验证报告（完整性和一致性分析）
- 问题清单和改进建议

## 🔗 跨步骤一致性检查

以下检查项覆盖所有步骤，确保逆向推导的整体一致性：

- [ ] **映射流向一致**：分配方向统一向下（↓），承接方向统一向上（↑）
- [ ] **术语一致性**：全文使用统一定义（架构末级节点/详细定义末级节点/1:1分配约束/N:1承接支持）
- [ ] **层级结构完整**：每层从根节点到末级节点的树形结构完整（BA 0-5级，SysReq 功能0-9级/非功能0-6级）
- [ ] **1:1约束满足率100%**：每条详细定义末级分配到唯一的架构末级节点
- [ ] **N:1承接合理性**：每个架构末级节点承接 2-5 条详细定义末级为合理（1条需考虑合并，6+条需拆分）
- [ ] **映射覆盖率100%**：逐层映射无遗漏
- [ ] **符合性分析完整率100%**：所有方案节点都有符合性分析
- [ ] **详细定义的双重身份理解**：每一步的详细定义既是本层细化又是下层需求
- [ ] **IPO去重完成**：BA 层已执行去重（复用→改进→新增），无重复 IPO
- [ ] **超链接有效**：所有映射关系建立了双向超链接导航

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 代码文档不清晰 | 难以理解设计意图 | 邀请原开发人员参与，进行代码审查 |
| 架构图纸过时 | 反向推导的架构不准确 | 与代码实现进行对比，发现差异 |
| 需求信息遗漏 | 反向推导的需求不完整 | 进行用户访谈，补充缺失的需求 |
| 映射关系复杂 | 难以建立清晰的映射 | 严格执行1:1约束和N:1承接规则 |
| 性能指标难以逆向 | 无法确定原始指标 | 从代码的实际性能监测数据反推，标注"反推值" |

1. **项目启动前**
   - 收集所有代码和图纸
   - 组建跨职能的分析团队
   - 邀请原开发人员参与

2. **分析过程中**
   - 严格按照 8 步流程执行
   - 每步完成架构定义后先做分析活动，再进入下一步（避免推导偏差累积）
   - 及时记录发现的问题

3. **分析完成后**
   - 进行完整性和一致性验证
   - 生成需求映射矩阵
   - 建立文档维护机制
