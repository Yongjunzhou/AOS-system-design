## 一、核心概念

### 1.1 本层的定义

**系统需求（System Requirements）** 是将业务操作过程（业务架构）转化为系统可实现的需求规范。它回答的问题是：**"系统应该提供什么功能来支持业务操作？系统应该满足哪些质量属性？"**

系统需求在第4层扮演着**方案文档**的角色：
- **是业务架构的方案**：将业务操作的IPO转化为系统功能需求
- **是相关方非功能需求的方案**：将质量属性转化为系统非功能需求
- **是产品架构的需求**：为产品架构设计提供输入和约束

### 1.2 在5层结构中的位置

```
业务架构（第3层，功能部分）
  └─ IPO清单 → 去重 → 系统功能需求架构
       ↓ N:1承接支持

相关方非功能需求（第2层，非功能部分）
  └─ SR-NFR → 分类组织 → 系统非功能需求架构
       ↓ N:1承接支持

系统需求（第4层） ★ 本规范
  ├─ 架构定义文档：功能 SysReq-F（0-9级）+ 非功能 SysReq-NFR（0-6级）
  └─ 详细定义文档：对架构末级节点的分解（6-9级）
       ↓ 1:1分配约束
产品架构（第5层）
```

| 维度 | 说明 |
|------|------|
| **文档类型** | 方案文档（架构定义 + 详细定义） |
| **功能来源** | 业务架构IPO去重清单 |
| **非功能来源** | 相关方非功能需求（平行架构，不经过BA） |
| **承接对象** | BA 5级节点（N:1承接支持）+ SR-NFR（N:1承接支持） |
| **分配目标** | 产品架构末级节点（1:1分配约束） |

### 1.3 系统需求的双重来源

```
功能需求           非功能需求
  │                  │
  BA IPO             SR-NFR（第2层）
  │（去重）           │（平行架构，不经BA）
  ↓                  ↓
SysReq-F 5级 ←── SysReq-NFR 5级
  │                  │
  └────── 汇合 ──────┘
         ↓
     产品架构（第5层）
```

功能需求和非功能需求在系统需求层面**分开设计**，但在产品架构层面**汇合**。

### 1.4 本层的设计活动

三个活动遵循准则 [§1.5.1 需求的分解、分配与分析](./01-pipeline-system-design-standards.md#151-需求的分解分配与分析)（含语义分解四条件、指标分解规则、1:1约束细则等完整规范），本层的具体要点如下。

**需求分解及分析**——本层要点：
- 对 SysReq 5级节点进行 6→9级详细设计分解（功能模块→系统用例→用例场景→场景活动）
- 分解时检查各层级分解维度是否正确、无越层遗漏
- 场景活动完整覆盖用例的所有场景（正常/异常/替代），无遗漏

**需求分配及分析**——本层要点：
- 将 SysReq 9级（场景活动）分配到 PA 架构末级节点，检查 1:1 约束、全覆盖、N:1 合理性
- 每条场景活动都有唯一的 PA 构件承接，无遗漏、无多头分配

**架构承接需求分析**——本层要点：
- SysReq 架构末级节点自检语义覆盖度——所有承接的 BA IPO 是否都有对应的 SysReq 5级节点承接
- IPO 去重后语义是否一致、无遗漏
- 性能指标正确继承自 BA（不降级），每个节点明确标注了承接的 BA / NFR 来源

## 二、架构定义规范

### 2.1 功能需求架构（0-9级）

功能需求分支采用 0-9 级层级结构，其中 **5 级节点（组件实例）是承接 BA 5 级节点的关键层级**：

| 层级 | 名称 | 说明 | 是否末级 |
|------|------|------|---------|
| **0级** | 根节点 | 整个系统的顶层根节点 | 否 |
| **1级** | 独立信息系统 | 一个独立的信息系统或子系统 | 否 |
| **2级** | 功能分支 | 功能需求分支的入口 | 否 |
| **3级** | 业务组件类型 | 流程、工单、计划、指标等 | 否 |
| **4级** | 业务组件小类 | 组件类型下的小类分类 | 否 |
| **5级** | **组件实例（承接层）** | **★ 承接 BA 5 级节点** | **★ 架构末级** |
| **6级** | 功能模块 | 以标签页/页面为载体的功能模块 | 否 |
| **7级** | 系统用例 | 按钮/右键菜单提供的功能 | 否 |
| **8级** | 用例场景 | 正常/异常/替代场景 | 否 |
| **9级** | **场景活动** | **★ 详细定义末级，分配到 PA 末级** | **★ 详细定义末级** |

**各层级关系图**：

```
0级：根节点
 │
1级：独立信息系统（如：需求管理系统）
 │
2级：功能需求分支
 │
3级：业务组件类型（如：需求收集、需求规范化）
 │
4级：业务组件小类（如：流程定义、流程实例）
 │
5级：组件实例 ←── 承接 BA 5 级（N:1承接支持，架构末级）
 │
6级：功能模块（Tab/页面）
 │
7级：系统用例（按钮/菜单）
 │
8级：用例场景（正常/异常/替代）
 │
9级：场景活动 ──→ 分配到 PA 末级（1:1分配约束，详细定义末级）
```

### 2.2 非功能需求架构（0-6级）

非功能需求采用**平行架构**，独立于功能需求分支：

| 层级 | 名称 | 说明 |
|------|------|------|
| **0级** | 非功能需求根节点 | 整个系统的非功能需求顶层根节点 |
| **1级** | 需求类别 | 三大类：质量特性 / 环境适应性 / 可实现性 |
| **2级** | 需求类型 | 具体类型（性能、安全、可测试性等） |
| **3级** | 需求子类型 | 子类型（响应时间、吞吐量、数据加密等） |
| **4级** | 需求细分 | 对第3级的更细分类 |
| **5级** | **非功能需求承接层** | **★ 承接 SR-NFR（架构末级）** |
| **6级** | 操作序列/验证标准 | 验证步骤和标准 |

**三大类分类体系**：

| 类别 | 类型 | 示例指标 |
|------|------|---------|
| **质量特性需求** | 性能、可靠性、安全性、可用性、可维护性 | 响应时间 < 200ms，可用性 > 99.9% |
| **环境适应性需求** | 兼容性、可扩展性、可移植性、集成适应性 | 支持水平扩展，跨平台部署 |
| **可实现性需求** | 可测试性、可部署性、可操作性、可运维性 | 单元测试覆盖率 > 80%，部署时间 < 30min |

### 2.3 架构末级节点定义

**功能需求架构末级节点（SysReq-F 5级）**：

| 属性 | 说明 | 示例 |
|------|------|------|
| **节点ID** | 唯一标识 | SysReq-F-001 |
| **节点名称** | 系统功能名称 | 需求收集管理 |
| **类型** | 功能/非功能 | 功能需求 |
| **定义** | 功能的简要描述 | 提供需求的收集、规范化、分解功能 |
| **承接的上层需求** | 承接了哪些BA IPO | BA-001, BA-002 |
| **符合性分析** | 如何满足所承接的需求 | 通过提供需求管理相关功能... |
| **性能指标** | 从上层继承的性能约束 | 响应时间 < 2s |
| **设计指导** | 对产品架构的指导 | 系统应提供需求录入界面... |

**非功能需求架构末级节点（SysReq-NFR 5级）**：

| 属性 | 说明 | 示例 |
|------|------|------|
| **节点ID** | 唯一标识 | SysReq-NFR-001 |
| **节点名称** | 非功能需求名称 | 系统响应时间要求 |
| **需求描述** | 非功能需求的详细说明 | 系统在正常负载下应快速响应 |
| **类型** | 所属类别 | 性能 |
| **承接的SR-NFR** | 承接了哪些相关方非功能需求 | SR-NFR-001 |
| **量化指标** | 具体的、可测量的指标值 | 平均响应时间 < 200ms |
| **验证方法** | 如何验证指标 | 使用JMeter进行负载测试 |
| **约束条件** | 对产品架构的约束 | 必须使用缓存层 |

### 2.4 架构定义的文档结构

**第一部分：架构设计结果（树形结构）**

```
系统需求（0级·根）

├─ 需求管理系统（1级·独立信息系统）
│  └─ 功能需求分支（2级·功能分支）
│     └─ 需求处理（3级·业务组件类型）
│        └─ 需求收集（4级·业务组件小类）
│           └─ SysReq-F-001: 需求收集管理（5级·架构末级）
│              ├─ 定义：提供需求收集的全生命周期管理
│              ├─ 承接BA：BA-001, BA-002
│              ├─ 符合性分析：通过需求管理引擎实现BA定义的收集流程
│              ├─ 设计指导：基于需求引擎配置，需包含录入/格式化/存储
│              └─ 性能指标：需求保存时间 < 1s，支持 100 并发用户
│
└─ 非功能需求（0级·根）
   └─ 质量特性需求（1级·需求类别）
      └─ 性能需求（2级·需求类型）
         └─ 响应时间（3级·需求子类型）
            └─ 系统响应时间（4级·需求细分）
               └─ SysReq-NFR-001: 系统响应时间（5级·架构末级）
                  ├─ 定义：系统在正常负载下应快速响应
                  ├─ 承接SR-NFR：SR-NFR-001
                  ├─ 量化指标：平均 < 200ms，P99 < 500ms
                  └─ 约束条件：需使用缓存层，数据库索引优化
```

**第二部分：需求映射视图（映射关系表）**

```
| BA/NFR末级节点 | 承接的系统需求架构末级 | 超链接 |
|---------------|---------------------|--------|
| BA-001        | SysReq-F-001        | [→] |
| BA-002        | SysReq-F-001        | [→] |
| BA-003        | SysReq-F-002        | [→] |
| SR-NFR-001    | SysReq-NFR-001      | [→] |
```

## 三、详细定义规范

### 3.1 详细定义末级节点

系统需求详细定义是对架构末级节点进行分解，使每条末级条目（9级场景活动）都能分配到产品架构的唯一末级节点。

| 属性 | 说明 | 示例 |
|------|------|------|
| **节点ID** | 唯一标识 | SysReq-F-001.1.1.1.1 |
| **节点名称** | 具体活动名称 | 填写需求信息 |
| **描述** | 活动步骤的详细说明 | 用户填写需求标题、描述和优先级 |
| **验收标准** | 活动完成的判断标准 | 所有必填项已填写，格式验证通过 |
| **分配到** | 产品架构末级节点 | PA-001 |
| **符合性分析** | 如何满足所承接的系统需求 | 通过前端表单提供需求录入界面 |

### 3.2 分解链（5级→9级）

```
5级：组件实例（承接层）
  └─ SysReq-F-001: 需求管理
      ↓ 按功能模块（Tab）分解
6级：功能模块
  └─ SysReq-F-001.1: 需求列表
      ↓ 按系统用例（按钮/菜单）分解
7级：系统用例
  └─ SysReq-F-001.1.1: 录入需求
      ↓ 按业务场景分解
8级：用例场景
  └─ SysReq-F-001.1.1.1: 正常录入需求
      ↓ 按活动步骤分解
9级：场景活动（末级）
  ├─ SysReq-F-001.1.1.1.1: 填写需求信息
  ├─ SysReq-F-001.1.1.1.2: 选择需求分类
  ├─ SysReq-F-001.1.1.1.3: 提交需求
  └─ 分配到：PA-001（需求录入前端组件）
```

**对比示例**：

```
✅ 分解充分（1:1约束满足）：
  9级活动 SysReq-F-001.1.1.1.1（填写需求信息）→ PA-001（前端表单组件）
  9级活动 SysReq-F-001.1.1.1.2（提交需求数据）→ PA-101（需求API服务）
  → 每个9级活动只需一个PA构件承接

❌ 分解不充分（违反1:1约束）：
  9级活动 SysReq-F-001.1.1（录入需求）→ PA-001（前端）+ PA-101（后端）
  → 一个9级活动需要分配到多个PA构件，需继续分解
```

### 3.3 分解标准

**分解充分性标准**：
- ✅ 一条详细定义条目只需分配到产品架构的**一个**末级节点 → 停止分解
- ❌ 一条详细定义条目需要分配到产品架构的**多个**末级节点 → 继续分解

**分解一致性标准**：

| 当前级 | 目标级 | 分解维度 |
|-------|-------|---------|
| 5级 → 6级 | 组件实例 → 功能模块 | 按Tab/页面分解 |
| 6级 → 7级 | 功能模块 → 系统用例 | 按操作/功能按钮分解 |
| 7级 → 8级 | 系统用例 → 用例场景 | 按业务场景（正常/异常/替代）分解 |
| 8级 → 9级 | 用例场景 → 场景活动 | 按可操作步骤分解 |

### 3.4 格式模板（YAML）

```yaml
系统需求: SysReq-F-001
架构末级节点: 需求管理

详细定义:
  - ID: SysReq-F-001.1
    层级: 6级-功能模块
    名称: 需求列表
    描述: 展示所有待处理需求的列表页面

    - ID: SysReq-F-001.1.1
      层级: 7级-系统用例
      名称: 录入需求
      描述: 用户点击"录入"按钮创建新需求

      - ID: SysReq-F-001.1.1.1
        层级: 8级-用例场景
        名称: 正常录入需求
        描述: 用户填写完整信息成功录入需求

        - ID: SysReq-F-001.1.1.1.1
          层级: 9级-场景活动
          名称: 填写需求信息
          描述: 用户填写需求标题、描述、优先级
          验收标准: 所有必填项已填写
          分配到: PA-001（需求录入前端）
```

## 四、设计方法

### 4.1 总体设计流程

```
步骤1：IPO去重
  └─ 收集所有BA 5级节点（IPO清单）
     └─ 识别相同或相似的IPO，合并重复
        └─ 输出：去重后的IPO清单 + 去重映射关系

步骤2：建立功能需求架构（0-5级）
  └─ 将去重后的IPO按业务组件类型分类（3级）
     └─ 细分为业务组件小类（4级）
        └─ 形成组件实例清单（5级承接层）
           └─ 输出：功能需求架构定义

步骤3：功能需求详细分解（5级→9级）
  └─ 5级→6级（功能模块）→7级（用例）→8级（场景）→9级（活动）
     └─ 输出：详细定义文档（含9级→PA映射）

步骤4：建立非功能需求架构
  └─ 收集SR-NFR → 按三大类分类组织
     └─ 输出：SysReq-NFR 0-5级架构

步骤5：映射验证
  └─ BA → SysReq 映射完整性（100%）
     └─ 9级→PA 1:1分配约束（100%）
        └─ 输出：映射验证报告
```

### 4.2 IPO去重（核心）

系统功能需求的来源是业务架构IPO的**去重清单**。多个BA 5级节点如果描述相同或相似的业务操作，在系统需求中只定义一次。

**示例**：

```
BA-001（常规流程需求）→ 包含"需求收集"IPO
BA-002（紧急流程需求）→ 包含"需求收集"IPO

去重后：
SysReq-F-001: 需求收集管理（同时承接BA-001和BA-002）
→ 系统需求中"需求收集"只需实现一次
```

### 4.3 两阶段工作流程

> **与准则的关系**：本节描述的两阶段直接对应准则 §1.5.3 定义的跨层"两阶段工作流程"——第一阶段快速约定结构骨架并建立映射关系，第二阶段并行填充详细内容。系统需求层作为第4层，其两阶段工作流程同时涉及向上承接（BA→SysReq）和向下分配（SysReq→PA）。

```
第一阶段：骨架建立（快速，占位符创建）
  1. BA IPO去重
  2. 创建SysReq 5级节点占位符
  3. 同步创建PA末级节点结构骨架
  4. 建立BA → SysReq映射关系
  5. 冻结骨架结构
  输出：架构骨架 + 映射关系

第二阶段：详细填充（并行进行）
  1. 详细定义SysReq 6-9级分解
  2. 同步填充PA末级节点详细内容
  3. 建立9级→PA的映射关系
  4. 验证映射的一致性
  5. 冻结系统需求文档
  输出：完整的系统需求文档
```

### 4.4 同步设计

在开展系统需求详细定义的同时，应同步开展产品架构定义。

**原因**：系统需求详细定义（9级场景活动）需要分配到产品架构末级节点。如果产品架构尚未建立，9级分解就没有分配目标，可能导致返工。

**实际操作**：
1. 先快速建立产品架构的结构骨架（占位符）
2. 边分解9级活动边映射到PA占位符
3. 冻结映射关系后，并行填充详细内容

### 4.5 设计原则

| 原则 | 定义 | 应用范围 |
|------|------|---------|
| **IPO去重** | 多个BA 5级节点相同→SysReq只定义一次 | BA 5级 → SysReq 5级 |
| **N:1承接支持** | 一个SysReq 5级节点可承接多个BA IPO | BA 5级 → SysReq 5级（2-5个合理） |
| **1:1分配约束** | 每条9级活动分配到唯一的PA末级节点 | SysReq 9级 → PA末级 |
| **功能/非功能分离** | 两条独立架构体系，分别设计 | 功能走BA链路，非功能走NFR平行链路 |
| **同步设计** | 系统需求详细定义与产品架构定义同步进行 | SysReq 9级 ↔ PA末级 |
| **组织资产优先** | 优先复用现有SysReq节点，其次改进，最后新增 | 分配到PA时按复用→改进→新增顺序检查 |

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误1：IPO去重不完整**

```
错误的做法：
直接将BA IPO清单复制为SysReq清单，未进行去重
→ BA-001（需求收集）+ BA-002（需求收集）在SysReq中出现两次

问题：两个功能相同的IPO在系统需求中被重复定义
后果：导致后续产品架构设计中出现重复构件
```

```
✅ 正确做法：
跨BA流程检查重复IPO，合并后形成唯一的功能清单
BA-001和BA-002的"需求收集"合并为SysReq-F-001
→ SysReq-F-001同时承接BA-001和BA-002
```

**❌ 错误2：9级活动粒度不一致**

```
错误的做法：
SysReq-001的9级活动 → "填写需求信息"（细粒度）
SysReq-002的9级活动 → "整个需求处理流程"（粗粒度）

问题：同一层的9级活动粒度参差不齐
后果：粗粒度活动需要分配到多个PA节点，违反1:1约束
```

```
✅ 正确做法：
所有9级活动保持在"可操作步骤"的粒度
SysReq-001的9级活动 → "填写需求信息"（✅）
SysReq-002的9级活动 → "提交需求"、"审核需求"、"完成需求"（✅）
→ 每个9级活动都只需一个PA节点承接
```

**❌ 错误3：非功能需求缺乏量化指标**

```
错误的做法：
SysReq-NFR-001: "系统应该很快"

问题："很快"是主观词汇，无法测量
后果：产品架构无法确定设计目标，验收时无法判断是否满足
```

```
✅ 正确做法：
SysReq-NFR-001: "平均响应时间 < 200ms，P99 < 500ms"
→ 具体的数字和百分位数，可通过性能测试验证
```

**❌ 错误4：忽略资产优先原则，动辄新增SysReq架构节点**

```
错误的做法：
新的BA IPO到达时，直接新增SysReq 5级节点来承接，
而不检查现有SysReq节点是否可通过扩展承接。

问题：忽视已有SysReq设计资产，导致架构膨胀
后果：SysReq节点数量快速增加，映射关系复杂化
```

```
✅ 正确做法：
按 复用→改进→新增 的优先级依次检查现有SysReq节点。
仅当复用和改进都不可行时，才新增节点。
说明：保护组织已有设计资产，控制架构复杂度
```

### 7.2 最佳实践

**实践1：复用优先的去重流程**：在建立SysReq架构前，先全面扫描BA IPO，使用"复用→改进→新增"三级去重策略，记录去重映射关系。

**实践2：逐级验证的分解流程**：每完成一级分解（5→6→7→8→9），立即验证本级是否满足1:1约束。尽早发现粒度问题，避免累积到9级才返工。

**实践3：功能与非功能的协调设计**：在定义功能需求时，同步考虑非功能需求的约束。在功能模块（6级）定义时，标注受哪些NFR约束。

**实践4：使用标准化模板**：采用统一的YAML模板定义每个层级节点，确保所有节点的属性完整，便于自动化处理和AI辅助生成。

## 八、FAQ

**Q1：为什么功能需求架构有9级，而非功能只有6级？**

功能需求需要从业务组件层层分解到可操作的活动步骤（场景活动），因此需要更多的层级来承载这种分解。非功能需求是质量属性约束，不需要这么细的分解，6级（根→类别→类型→子类型→承接层→验证标准）已经足够。

**Q2：5级节点承接多少个BA IPO合适？**

建议每个5级节点承接 2-5 个 BA 5 级节点（2-3 条最优，1 条需考虑合并，4-5 条可接受需检查过度聚合，6 条以上说明粒度过粗需考虑拆分）。承接 0 个说明有遗漏。

**Q3：如果9级场景活动无法1:1映射到PA构件怎么办？**

有两种可能：① 9级活动粒度太粗，需要继续分解为更细的活动步骤；② 产品架构定义不完整，缺少对应的构件。处理：先尝试继续分解9级活动，如果已经是原子级则补充PA构件。

**Q4：IPO去重后发现承接关系变得复杂，如何处理？**

记录去重映射关系表，明确每个SysReq节点承接了哪些BA IPO。去重不丢失映射信息，只是将多个相同的BA IPO合并到一个SysReq节点。追溯时可通过映射关系表查找到原始的BA来源。

**Q5：非功能需求在系统需求层面如何与功能需求协调？**

两条链路独立设计但在产品架构层面汇合。在定义功能需求时，应在每个功能模块（6级）处标注受哪些NFR约束，确保功能设计不违反非功能约束。
