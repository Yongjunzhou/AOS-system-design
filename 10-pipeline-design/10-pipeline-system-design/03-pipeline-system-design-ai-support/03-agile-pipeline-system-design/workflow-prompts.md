# 敏捷式系统设计：提示词模板集
**Agile System Design: Workflow Prompt Templates**

**用途**：AI 按 8 步流程依次执行，每个步骤使用对应的提示词模板。AI 将提示词中的 `[占位符]` 替换为实际内容后执行。

**重要前提**：本场景假设已有完整的系统设计文档（5 层 + 映射矩阵）。AI 在执行前应确保已加载现有基线文档。

**依据B说明**：本提示词模板是产品B（运营体系系统设计流水线）的"AI辅助文档"构件，是产品B产品架构（PA）的末级节点。模板中的提示词指令（分解方法、映射规则、验证标准）为方法论层面的通用操作，与目标产品类型无关。

AI使用本模板时，应根据目标产品类型选择对应的PA分类体系：
- **软件产品类型**（如产品A·AOS运营体系）：PA分类为`前端/后端/数据/基础设施`
- **方法论/工具产品类型**（如产品B·流水线）：PA分类为`规范与准则/设计指南/AI辅助文档/任务定义`

模板中的示例默认使用产品B的敏捷式迭代场景进行演示，输出实例是产品B的另一个PA末级节点（即面向目标产品的AI辅助文档自定义实例）。为目标产品生成设计文档时，示例和内容应引用目标产品的实际领域。

## 第1步：新需求分析和分类

### 目标
分析新需求的性质和影响范围，确定优先级。

### 输入
- 新需求原始表述（作为本步骤的输入提供）
- 现有完整系统设计文档集（基线）

### 涉及规范
- [敏捷式设计指南 §第1步](../../02-pipeline-system-design-guidelines/03-agile-pipeline-system-design-guide.md#第1步新需求分析和分类)
- [原始需求规范 §1 核心概念](../../01-design-standards/02-original-requirements-specification.md#一核心概念)

### 提示词模板

#### 1a. 新需求收集与分类

```
请对以下新需求进行收集和分类。

新需求原始表述：
[在此处粘贴新需求原文]

现有系统设计文档集：
[说明基线的文档路径]

请执行：
1. 需求分类 — 每条需求的类型：
   - 功能需求 vs 非功能需求
   - 新增需求 vs 修改需求 vs 删除需求
   - 独立需求 vs 相关需求（与现有需求的关系）

2. 碎片归并（可选）：
   - 如果新需求来自多渠道、粒度参差不齐，先进行碎片归并
   - 血缘归并：同一功能不同切面的碎片合并
   - 角色聚合：不同角色对同一功能的诉求合并
   - 流程串联：同一流程不同步骤按顺序串联
   - 父子补全：以主需求为载体，补充细节作为子约束

3. 保留原始出处：
   - 记录提出方、原始表述、日期
   - 矛盾碎片标注"冲突待确认"，不合并

输出格式：
新需求 [OR-XXX]
├─ 需求标题：[一句话概括]
├─ 需求描述：[完整表述]
├─ 原始碎片清单：[如有]
├─ 需求类型：[功能需求/非功能需求]
├─ 变更类型：[新增/修改/删除]
├─ 与现有需求的关系：[相关/独立]
└─ 依赖/冲突关系：[如有，标注]
```

#### 1b. 影响范围评估与优先级

```
请对新需求进行影响范围评估和优先级排序。

新需求清单：[粘贴1a的输出]
现有系统设计文档集：[基线路径]

影响范围评估（逐层检查）：
1. 是否影响现有的相关方需求（SR）？
   - 影响现有 SR 架构末级节点？哪个节点？
   - 需要新增 SR 节点？
2. 是否影响现有的业务架构（BA）？
   - 影响现有 BA IPO？
   - 需要新增 BA IPO？
3. 是否影响现有的系统需求（SysReq）？
   - 影响现有 SysReq 5 级节点？
   - 需要新增 SysReq 节点？
4. 是否影响现有的产品架构（PA）？
   - 影响现有 PA 组件？
   - 需要新增 PA 组件？

优先级评估标准：
- 高优先级：影响范围大，需要立即处理
- 中优先级：影响范围中等，可在下一个迭代处理
- 低优先级：影响范围小，可延后处理

输出格式：
## 影响范围评估报告

### 需求概览
| 需求ID | 标题 | 类型 | 优先级 |
|--------|------|------|--------|
| OR-XXX | [标题] | [功能/NFR] | [高/中/低] ⚠️ 待确认 |

### 影响范围矩阵
| 受影响的层 | 影响类型 | 涉及节点 | 说明 |
|-----------|---------|---------|------|
| SR | [新增/修改/无] | SR-F-XXX | [说明] |
| BA | [新增/修改/无] | BA-XXX | [说明] |
| SysReq | [新增/修改/无] | SysReq-F-XXX | [说明] |
| PA | [新增/修改/无] | PA-XXX | [说明] |
```

### 预期产出
- 新需求分析报告（含分类和碎片归并结果）
- 影响范围评估矩阵
- 优先级排序

## 第2步：新需求分解

### 目标
对新需求进行规范化处理和分解，得到末级需求。

### 输入
- 第1步产出的新需求分析报告

### 涉及规范
- [原始需求规范 §3 分解规范](../../01-design-standards/02-original-requirements-specification.md#三分解规范)
- [敏捷式设计指南 §第2步](../../02-pipeline-system-design-guidelines/03-agile-pipeline-system-design-guide.md#第2步新需求分解)

### 提示词模板

#### 2a. 需求规范化与末级分解

```
请对新需求进行规范化处理和末级分解。

新需求清单：[粘贴1a或1b的输出]

请执行：
1. 四项检查：
   - 完整性：每条需求是否有清晰的描述、类型、优先级？
   - 清晰性：模糊术语替换为精确表述
   - 可测试性：输入/处理/输出是否明确？
   - 独立性：是否与其他需求过度耦合？

2. 功能/非功能分离：
   - 将"做什么"（功能）和"有多好"（非功能）分开记录

3. 需求分解到末级：
   分解原则：
   - 原子性原则：每条末级不可再分
   - 1:1 映射原则：每条末级只需映射到一个 SR 架构末级节点
   - 粒度一致性：末级粒度与现有需求一致
   - 语义完整性：分解后不丢失原始语义

   停止准则：问自己"这个条目能映射到现有相关方需求架构的一个末级节点吗？"
   - 只需一个节点承接 → 停止
   - 需要多个节点承接 → 继续分解

输出格式：
OR-XXX: [需求标题]
├─ 规范化描述：[经过四项检查后的描述]
├─ 类型：[功能需求/非功能需求]
├─ 优先级：[高/中/低] ⚠️ 待确认
├─ 末级分解：
│  ├─ OR-XXX.1: [原子性功能描述] → SR-F 承接
│  ├─ OR-XXX.2: [原子性功能描述] → SR-F 承接
│  └─ OR-XXX.3: [原子性描述] → SR-NFR 承接
└─ 验收标准：[可量化的验收条件]
```

#### 2b. 冲突与重复检测

```
请对新末级需求与现有末级需求之间执行冲突和重复检测。

新末级需求清单：[粘贴2a的输出]
现有末级需求清单：[从基线文档中提取]

冲突检测（新需求 vs 现有需求）：
- 直接冲突：语义直接矛盾，无法同时满足 → 标记"需确认"
- 间接冲突：同时实现可能行为不一致 → 标记"需调整"
- 可合并冲突：可通过调整合并解决 → 标记"建议合并"

重复检测：
- 完全重复：与现有需求完全相同 → 去重（合并到现有需求）
- 部分重复：功能范围部分重叠 → 建议合并
- 相似重复：表达不同但核心相同 → 确认是否为同一需求

输出格式：
| 新需求 | 现有需求 | 问题类型 | 描述 | 建议 |
|-------|---------|---------|------|------|
| OR-XXX.X | OR-YYY.Y | [直接/间接/重复] | [描述] | [建议] ⚠️ |
```

### 预期产出
- 更新后的原始需求文档（添加新需求）
- 新末级需求清单
- 冲突/重复检测报告

## 第3步：相关方需求影响分析

### 目标
将新末级需求映射或更新到相关方需求，输出增量定义。

### 输入
- 第2步产出的新末级需求清单
- 现有相关方需求文档（基线）

### 涉及规范

- [相关方需求规范 §3 详细定义规范](../../01-design-standards/03-stakeholder-requirements-specification.md#三详细定义规范)
- [敏捷式设计指南 §第3步](../../02-pipeline-system-design-guidelines/03-agile-pipeline-system-design-guide.md#第3步相关方需求影响分析)

### 提示词模板

#### 3a. SR 影响分析

```
请执行相关方需求影响分析。

新末级需求清单：[粘贴2a的输出]
现有相关方需求文档：[基线文档路径]

分析活动：
1. 需求说明符合性分析：
   - 检查新末级需求是否能由现有 SR 架构末级节点承接
   - 如果可承接：更新 N:1 映射和符合性分析
   - 如果不可承接：需要新建 SR 节点

2. 需求性能符合性分析：
   - 新需求的性能指标是否可在现有 SR 框架内量化

映射策略选择（决策树）：
选项A：映射到现有 SR → 如果新需求与现有 SR 节点相关
选项B：创建新的 SR → 如果新需求无法映射到现有 SR

输出格式：
## SR 影响分析报告

### 功能需求映射决策
| 新末级需求 | 映射策略 | 目标节点 | 说明 |
|-----------|---------|---------|------|
| OR-XXX.1 | 映射现有 | SR-F-00N | [说明] |
| OR-XXX.2 | 新建 | SR-F-00M | [说明] |

### 非功能需求映射决策
| 新末级需求 | 映射策略 | 目标节点 | 说明 |
|-----------|---------|---------|------|
| OR-XXX.3 | 映射现有 | SR-NFR-00N | [说明] |
| OR-XXX.4 | 新建 | SR-NFR-00M | [说明] |

### 分析结论
├─ 需求说明符合性：[通过/未通过]
└─ 需求性能符合性：[通过/未通过]
```

#### 3b. SR 增量条目定义

```
请根据影响分析结果，生成相关方需求增量定义。

影响分析报告：[粘贴3a的输出]

1. 对映射到现有 SR 节点的：更新承接需求清单和符合性分析

详细定义末级（同样使用增量标记）：
```

## 第4步：业务架构影响分析

### 目标
分析新需求对业务架构的影响，输出增量定义。

### 输入
- 第3步产出的 SR 增量定义
- 现有业务架构文档（基线）

### 涉及规范
- [业务架构规范 §2 架构定义规范](../../01-design-standards/04-business-architecture-specification.md#二架构定义规范)
- [业务架构规范 §4.3 去重机制](../../01-design-standards/04-business-architecture-specification.md#43-去重机制)
- [敏捷式设计指南 §第4步](../../02-pipeline-system-design-guidelines/03-agile-pipeline-system-design-guide.md#第4步业务架构影响分析)

### 提示词模板

#### 4a. BA 影响分析与去重

```
请执行业务架构影响分析。

SR 增量定义：[粘贴3b的输出中详细定义部分]
现有业务架构文档：[基线文档路径]

分析活动：
1. 需求说明符合性分析：
   - 检查新的 SR 详细定义末级是否能由现有 BA IPO 承接
   - 验证 1:1 分配约束

2. 需求性能符合性分析：
   - 性能指标是否从 SR 正确继承

IPO 去重检查（三级策略）：
第1步：检查现有 IPO 中是否已有相同或相似的
第2步：复用优先 — 完全匹配的直接复用，记录映射来源
第3步：改进次之 — 改进后可用的则改进后复用
第4步：新增最后 — 确实需要才新增

更新策略选择：
策略A：映射到现有 BA → 更新承接需求清单和符合性分析
策略B：修改现有 BA → 分析影响范围，更新 IPO 定义
策略C：创建新的 BA → 执行去重后创建新 IPO

输出格式：
## BA 影响分析报告

### 更新策略决策
| SR详细定义末级 | 更新策略 | 目标节点 | 去重说明 |
|---------------|---------|---------|---------|
| SR-F-00N.1   | 映射现有 | BA-00M  | [复用/改进/新增] |
| SR-F-00N.2   | 新建 | BA-00P  | 新增（无重复） |
```

#### 4b. BA 增量条目定义与同步设计

```
请根据影响分析结果，生成业务架构增量定义。

BA 影响分析报告：[粘贴4a的输出]
现有业务架构文档：[基线文档路径]

请执行：
1. 对映射到现有 BA 的：更新承接需求清单和符合性分析
2. 对修改现有 BA 的：更新 IPO 定义
3. 对创建新 BA 的：定义完整的 IPO（I/P/O）
4. 同步设计：在系统需求层建立 5 级节点占位符或调整现有节点

增量标记格式：
+ BA-00N: [新增的业务组件名称]（5级·架构末级节点）
├─ 节点类型：[配置/开发]
├─ 定义：[业务组件的定义]
├─ Input（输入）：[具体形式、数据结构、来源]
├─ Process（处理）：[规则、步骤、约束条件]
├─ Output（输出）：[具体形式、数据结构、验收标准]
├─ 承接的SR详细定义末级：SR-F-00N.1
├─ 符合性分析：[如何满足]
├─ 设计指导：[对系统需求的指导]
└─ 去重说明：[复用/改进/新增]

~ BA-00M: [修改的业务组件名称]
├─ 更新说明：[变更内容]
└─ 更新符合性分析：[更新说明]

> **注意**：BA IPO 名称应使用"动词+名词"结构（如"基线创建冻结""基线差异对比"），准确反映业务活动的核心操作，不得使用模糊词汇。同时确保每个 BA 节点已标注类型（配置方式/开发方式），对应正确的层级结构。

同步设计输出格式：
### BA → SysReq 骨架映射更新
| BA IPO | 承接的SysReq 5级 | 变更类型 |
|--------|----------------|---------|
| BA-00N | SysReq-F-00N  | +新增骨架 |
| BA-00M | SysReq-F-00M  | ~更新 |
```

### 预期产出
- 业务架构影响分析报告
- 更新后的业务架构文档（增量定义）
- BA → SysReq 骨架映射更新

## 第5步：系统需求影响分析

### 目标
分析新需求对系统需求的影响，输出增量定义。

### 输入

## 并行说明

在敏捷式 8 步流程中，以下步骤可以**并行执行**：

| 并行组 | 步骤 | 说明 |
|--------|------|------|
| **增量分析** | 第1-2步 | 新需求分析和分解，连续执行 |
| **功能分支** | 第3-6步 | SR→BA→SysReq→PA 沿功能链路依次推进 |
| **非功能分支** | 第5步（非功能部分） | 独立于功能分支，单独更新 |
| **验证+发布** | 第7-8步 | 合并所有增量后执行 |

实际时间线：
```
第1+2步 → 第3+4+5+6步（功能链路）+ 第5步（非功能）→ 第7步 → 第8步
```
