# 敏捷式系统设计：提示词模板集
**Agile System Design: Workflow Prompt Templates**

**用途**：AI 按 8 步流程依次执行，每个步骤使用对应的提示词模板。AI 将提示词中的 `[占位符]` 替换为实际内容后执行。

**重要前提**：本场景假设已有完整的系统设计文档（5 层 + 映射矩阵）。AI 在执行前应确保已加载现有基线文档。

**依据B说明**：本提示词模板是产品B（运营体系系统设计流水线）的"AI辅助文档"构件，是产品B产品架构（PA）的末级节点。模板中的提示词指令（分解方法、映射规则、验证标准）为方法论层面的通用操作，与目标产品类型无关。

AI使用本模板时，应根据目标产品类型选择对应的PA分类体系：
- **软件产品类型**（如产品A·AOS运营体系）：PA分类为`前端/后端/数据/基础设施`
- **方法论/工具产品类型**（如产品B·流水线）：PA分类为`规范与准则/设计指南/AI辅助文档/任务定义`

模板中的示例默认使用产品B的敏捷式迭代场景进行演示，输出实例是产品B的另一个PA末级节点（即面向目标产品的AI辅助文档自定义实例）。为目标产品生成设计文档时，示例和内容应引用目标产品的实际领域。

---

## 跨步骤上下文管理说明

### 上下文传递方式

8步增量设计流程中，从第2步开始，每步都基于上一步的产出。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 新需求分析结果 | 需求分类、影响范围、优先级 | 第1步输出 |
| 需求末级分解 | 分解后的原子性需求，冲突检测结果 | 第2步输出 |
| 增量定义 | 各层的 +/~ 增量内容 | 各步增量输出 |
| 同步设计 | 上层更新后下层骨架的同步 | 各步的同步设计记录 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |

### 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**中，供后续步骤引用。

### 当需要回溯修正

如果在第 N 步发现问题需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见[AI决策框架 -> 跨步骤影响范围表](../02-ai-decision-framework-pipeline.md#72-跨步骤影响范围表)）
2. 由于增量设计的影响逐层向下传递，修改上层后需检查下游所有增量是否需要同步调整
3. 修改后重新运行受影响步骤的 self-consistency check
4. 标注变更记录

## 第1步：新需求分析和分类

### 目标
分析新需求的性质和影响范围，确定优先级。

### 上下文继承清单

```
从哪里来：
  - 直接输入：新需求原始表述（由流水线开发者/AOS开发者提供）
  - 基线输入：现有完整系统设计文档集（5 层 + 映射矩阵）
  - 参考文档：[产品上下文](../01-product-context-pipeline.md)，了解产品B的背景
  - 涉及规范：原始需求规范 §1

预期产出：
  1. 新需求分析报告（含分类和碎片归并结果）
  2. 影响范围评估矩阵
  3. 优先级排序

本步决策对后续的影响：
  - 需求分类决定后续增量链路的走向（功能/NFR）
  - 影响范围评估控制第3-6步的增量工作范围
  - 优先级排序决定多批次需求的处理顺序
```

### 输入
- 新需求原始表述（作为本步骤的输入提供）
- 现有完整系统设计文档集（基线）

### 涉及规范
- [敏捷式设计指南 §第1步](../../02-pipeline-system-design-guidelines/03-agile-pipeline-system-design-guide.md#第1步新需求分析和分类)
- [原始需求规范 §1 核心概念](../../01-pipeline-system-design-standards/02-pipeline-original-requirements-specification.md#一核心概念)

### 决策框架参考

| 决策场景 | 处理方法 | 参考章节 |
|---------|---------|---------|
| 需求分类 | 功能 vs 非功能、新增 vs 修改 vs 删除 | 按指南执行 |
| 需求歧义（L1/L2/L3） | 可推测则补齐，多选项则列出 | [§4 需求歧义处理策略](../02-ai-decision-framework-pipeline.md#四需求歧义处理策略) |
| 碎片归并 | 血缘归并/角色聚合/流程串联 | 按指南执行 |
| 矛盾碎片 | 标注"冲突待确认"不合并 | 按指南执行 |

### 提示词模板

#### 1a. 新需求收集与分类

```
请对以下新需求进行收集和分类。

新需求原始表述：
[在此处粘贴新需求原文]

现有系统设计文档集：
[说明基线的文档路径]

请执行：
1. 需求分类 — 每条需求的类型：
   - 功能需求 vs 非功能需求
   - 新增需求 vs 修改需求 vs 删除需求
   - 独立需求 vs 相关需求（与现有需求的关系）

2. 碎片归并（可选）：
   - 如果新需求来自多渠道、粒度参差不齐，先进行碎片归并
   - 血缘归并：同一功能不同切面的碎片合并
   - 角色聚合：不同角色对同一功能的诉求合并
   - 流程串联：同一流程不同步骤按顺序串联
   - 父子补全：以主需求为载体，补充细节作为子约束

3. 保留原始出处：
   - 记录提出方、原始表述、日期
   - 矛盾碎片标注"冲突待确认"，不合并

输出格式：
新需求 [OR-XXX]
├─ 需求标题：[一句话概括]
├─ 需求描述：[完整表述]
├─ 原始碎片清单：[如有]
├─ 需求类型：[功能需求/非功能需求]
├─ 变更类型：[新增/修改/删除]
├─ 与现有需求的关系：[相关/独立]
└─ 依赖/冲突关系：[如有，标注]
```

#### 1b. 影响范围评估与优先级

```
请对新需求进行影响范围评估和优先级排序。

新需求清单：[粘贴1a的输出]
现有系统设计文档集：[基线路径]

影响范围评估（逐层检查）：
1. 是否影响现有的相关方需求（SR）？
   - 影响现有 SR 架构末级节点？哪个节点？
   - 需要新增 SR 节点？
2. 是否影响现有的业务架构（BA）？
   - 影响现有 BA IPO？
   - 需要新增 BA IPO？
3. 是否影响现有的系统需求（SysReq）？
   - 影响现有 SysReq 5 级节点？
   - 需要新增 SysReq 节点？
4. 是否影响现有的产品架构（PA）？
   - 影响现有 PA 组件？
   - 需要新增 PA 组件？

优先级评估标准：
- 高优先级：影响范围大，需要立即处理
- 中优先级：影响范围中等，可在下一个迭代处理
- 低优先级：影响范围小，可延后处理

输出格式：
## 影响范围评估报告

### 需求概览
| 需求ID | 标题 | 类型 | 优先级 |
|--------|------|------|--------|
| OR-XXX | [标题] | [功能/NFR] | [高/中/低] ⚠️ 待确认 |

### 影响范围矩阵
| 受影响的层 | 影响类型 | 涉及节点 | 说明 |
|-----------|---------|---------|------|
| SR | [新增/修改/无] | SR-F-XXX | [说明] |
| BA | [新增/修改/无] | BA-XXX | [说明] |
| SysReq | [新增/修改/无] | SysReq-F-XXX | [说明] |
| PA | [新增/修改/无] | PA-XXX | [说明] |
```

- 新需求分析报告（含分类和碎片归并结果）
- 影响范围评估矩阵
- 优先级排序

## 第2步：新需求分解

### 目标
对新需求进行规范化处理和分解，得到末级需求。

### 上下文继承清单

```
从哪里来：
  - 第1步产出：新需求分析报告（含分类、影响范围、优先级）
  - 基线输入：现有原始需求文档

去哪里：
  - 本步产出 → 第3步（SR影响分析）的输入
  - 更新后的原始需求文档将成为增量基线的一部分

特别注意：
  本步是增量设计的起点，分解粒度直接影响后续所有映射
  必须参考现有末级需求的粒度水平，保持一致
```

### 输入
- 第1步产出的新需求分析报告

### 涉及规范
- [原始需求规范 §3 分解规范](../../01-pipeline-system-design-standards/02-pipeline-original-requirements-specification.md#三分解规范)
- [敏捷式设计指南 §第2步](../../02-pipeline-system-design-guidelines/03-agile-pipeline-system-design-guide.md#第2步新需求分解)

### 决策框架参考

| 决策场景 | 处理方法 | 参考章节 |
|---------|---------|---------|
| 需求分解粒度 | 按原子性原则，1:1 约束测试 | [§2.1 原始需求→末级的分解](../02-ai-decision-framework-pipeline.md#21-原始需求--末级的分解) |
| 功能/NFR分离 | 将"做什么"和"有多好"分开 | 按指南执行 |
| 冲突检测 | 直接/间接/可合并三级分类 | 按指南执行 |
| 重复检测 | 完全/部分/相似三级分类 | 按指南执行 |

### 提示词模板

#### 2a. 需求规范化与末级分解

```
请对新需求进行规范化处理和末级分解。

新需求清单：[粘贴1a或1b的输出]

请执行：
1. 四项检查：
   - 完整性：每条需求是否有清晰的描述、类型、优先级？
   - 清晰性：模糊术语替换为精确表述
   - 可测试性：输入/处理/输出是否明确？
   - 独立性：是否与其他需求过度耦合？

2. 功能/非功能分离：
   - 将"做什么"（功能）和"有多好"（非功能）分开记录

3. 需求分解到末级：
   分解原则：
   - 原子性原则：每条末级不可再分
   - 1:1 映射原则：每条末级只需映射到一个 SR 架构末级节点
   - 粒度一致性：末级粒度与现有需求一致
   - 语义完整性：分解后不丢失原始语义

   停止准则：问自己"这个条目能映射到现有相关方需求架构的一个末级节点吗？"
   - 只需一个节点承接 → 停止
   - 需要多个节点承接 → 继续分解

输出格式：
OR-XXX: [需求标题]
├─ 规范化描述：[经过四项检查后的描述]
├─ 类型：[功能需求/非功能需求]
├─ 优先级：[高/中/低] ⚠️ 待确认
├─ 末级分解：
│  ├─ OR-XXX.1: [原子性功能描述] → SR-F 承接
│  ├─ OR-XXX.2: [原子性功能描述] → SR-F 承接
│  └─ OR-XXX.3: [原子性描述] → SR-NFR 承接
└─ 验收标准：[可量化的验收条件]
```

#### 2b. 冲突与重复检测

```
请对新末级需求与现有末级需求之间执行冲突和重复检测。

新末级需求清单：[粘贴2a的输出]
现有末级需求清单：[从基线文档中提取]

冲突检测（新需求 vs 现有需求）：
- 直接冲突：语义直接矛盾，无法同时满足 → 标记"需确认"
- 间接冲突：同时实现可能行为不一致 → 标记"需调整"
- 可合并冲突：可通过调整合并解决 → 标记"建议合并"

重复检测：
- 完全重复：与现有需求完全相同 → 去重（合并到现有需求）
- 部分重复：功能范围部分重叠 → 建议合并
- 相似重复：表达不同但核心相同 → 确认是否为同一需求

输出格式：
| 新需求 | 现有需求 | 问题类型 | 描述 | 建议 |
|-------|---------|---------|------|------|
| OR-XXX.X | OR-YYY.Y | [直接/间接/重复] | [描述] | [建议] ⚠️ |
```

### 预期产出
- 更新后的原始需求文档（添加新需求）
- 新末级需求清单
- 冲突/重复检测报告

### 本步决策记录

```
本步关键决策记录：
├─ 需求分解：
│  ├─ 需求总数：[N] 条 -> 末级 [M] 条
│  ├─ 功能需求末级：[N] 条
│  └─ 非功能需求末级：[N] 条
├─ 冲突检测：发现 [N] 项（直接 [N] / 间接 [N] / 可合并 [N]）
├─ 重复检测：发现 [N] 项（完全 [N] / 部分 [N] / 相似 [N]）
├─ 冲突/重复处理：合并 [N] / 标注待确认 [N] / 保持独立 [N]
├─ ⚠️ 待确认项：[列出]
└─ AI推断记录：[列出所有AI补齐/推断的内容]

## 第3步：相关方需求影响分析

### 目标
将新末级需求映射或更新到相关方需求，输出增量定义。

### 上下文继承清单

```
从哪里来：
  - 第2步产出：新末级需求清单 + 冲突/重复检测报告
  - 基线输入：现有相关方需求文档

去哪里：
  - 本步产出 → 第4步（BA影响分析）的输入
  - 更新后的SR文档将成为增量基线的一部分

特别注意：
  先做分析活动（符合性），再做映射决策
  功能需求和非功能需求分别处理，各自映射
  新增节点ID从现有编号体系延续，不重排已有编号
```

### 输入
- 第2步产出的新末级需求清单
- 现有相关方需求文档（基线）

### 涉及规范
- [相关方需求规范 §2 架构定义规范](../../01-pipeline-system-design-standards/03-pipeline-stakeholder-requirements-specification.md#二架构定义规范)
- [相关方需求规范 §3 详细定义规范](../../01-pipeline-system-design-standards/03-pipeline-stakeholder-requirements-specification.md#三详细定义规范)
- [敏捷式设计指南 §第3步](../../02-pipeline-system-design-guidelines/03-agile-pipeline-system-design-guide.md#第3步相关方需求影响分析)

### 决策框架参考

| 决策场景 | 处理方法 | 参考章节 |
|---------|---------|---------|
| 映射策略选择 | 按映射决策树：复用现有 > 修改现有 > 创建新的 | 按指南执行 |
| 合并/拆分 | N:1 承接数触发的判断 | [§3.1 N:1 承接数触发的判断](../02-ai-decision-framework-pipeline.md#31-n1-承接数触发的判断) |
| 详细定义粒度 | 按1:1约束：每条只需分配到一个BA IPO | [§2.2 SR详细定义→BA IPO的分解](../02-ai-decision-framework-pipeline.md#22-sr-详细定义--ba-ipo-的分解) |

### 提示词模板

#### 3a. SR 影响分析

```
请执行相关方需求影响分析。

新末级需求清单：[粘贴2a的输出]
现有相关方需求文档：[基线文档路径]

分析活动：
1. 需求说明符合性分析：
   - 检查新末级需求是否能由现有 SR 架构末级节点承接

选项A：映射到现有 SR → 如果新需求与现有 SR 节点相关

输出格式：
## SR 影响分析报告

### 功能需求映射决策
| 新末级需求 | 映射策略 | 目标节点 | 说明 |
|-----------|---------|---------|------|
| OR-XXX.1 | 映射现有 | SR-F-00N | [说明] |
| OR-XXX.2 | 新建 | SR-F-00M | [说明] |

### 非功能需求映射决策
| 新末级需求 | 映射策略 | 目标节点 | 说明 |
|-----------|---------|---------|------|
| OR-XXX.3 | 映射现有 | SR-NFR-00N | [说明] |
| OR-XXX.4 | 新建 | SR-NFR-00M | [说明] |

### 分析结论
├─ 需求说明符合性：[通过/未通过]
└─ 需求性能符合性：[通过/未通过]
```

#### 3b. SR 增量条目定义

```
请根据影响分析结果，生成相关方需求增量定义。

影响分析报告：[粘贴3a的输出]
现有相关方需求文档：[基线文档路径]

1. 对映射到现有 SR 节点的：更新承接需求清单和符合性分析

## 审查修复流程

### 审查输入
AI 完成指定步骤后，输出产物供 conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性/增量标记）

### AI 修复流程
收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、增量标记不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **增量标记错误**（`+`/`~` 混用、遗漏标记）→ 以基线为准修正标记
4. **结构性问题**（映射策略不当、粒度不匹配）→ 提出修复方案，标注 ⚠️ 待确认
5. **跨步骤一致性问题** → 参考[AI决策框架 -> 跨步骤影响范围表](../02-ai-decision-framework-pipeline.md#72-跨步骤影响范围表)进行影响范围评估

### 影响范围评估
| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 第1步需求分类偏差 | OR + SR + 后续增量链路 | 修正分类后从第2步重新评估 |
| 第2步需求分解错误 | SR + BA + SysReq + PA 全部增量链路 | 修正分解后从第3步重新评估 |
| 第3步 SR 映射偏差 | BA IPO + SysReq + PA 映射 | 修正映射后从第4步重新评估 |
| 第4步 BA IPO 错误 | SysReq + PA 映射 | 修正后从第5步重新评估 |
| 第5步 SysReq 定义错误 | PA 映射 | 修正后从第6步重新评估 |
| 第6步 PA 定义错误 | 仅影响产品架构层 | 局部修正 |
| 第7步发现问题 | 取决于问题定位 | 定位到对应步骤后回溯修复 |
