# 瀑布式系统设计：提示词模板集
**Waterfall System Design: Workflow Prompt Templates**

**用途**：AI 按 6 步流程依次执行，每个步骤使用对应的提示词模板。AI 将提示词中的 `[占位符]` 替换为实际内容后执行。

**依据B说明**：本提示词模板是产品B（运营体系系统设计流水线）的"AI辅助文档"构件，是产品B产品架构（PA）的末级节点。模板中的提示词指令（分解方法、映射规则、验证标准）为方法论层面的通用操作，与目标产品类型无关。

AI使用本模板时，应根据目标产品类型选择对应的PA分类体系：
- **软件产品类型**（如产品A·AOS运营体系）：PA分类为`前端/后端/数据/基础设施`
- **方法论/工具产品类型**（如产品B·流水线）：PA分类为`规范与准则/设计指南/AI辅助文档/任务定义`

模板中的示例默认使用产品B的瀑布式场景进行演示，输出实例是产品B的另一个PA末级节点（即面向目标产品的AI辅助文档自定义实例）。为目标产品生成设计文档时，示例和内容应引用目标产品的实际领域。

## 第1步：原始需求分析及相关方需求架构定义

### 目标
从原始需求（会议纪要、邮件、文档等原始表述）出发，产出原始需求详细定义和相关方需求架构定义。

### 输入
- 原始需求原始表述（作为本步骤的输入提供）

### 涉及规范
- [原始需求规范 §4.1 完整处理流程](../../01-design-standards/02-original-requirements-specification.md#41-完整处理流程)
- [相关方需求规范 §2 架构定义规范](../../01-design-standards/03-stakeholder-requirements-specification.md#二架构定义规范)
- [系统设计准则 §3.1 第1步](../../01-design-standards/01-system-design-standards.md#第1步原始需求分析及相关方需求架构定义)

### 提示词模板

#### 1a. 原始需求收集和初步整理

```
请对以下原始需求进行初步整理。原始需求来自多个渠道，粒度参差不齐。

原始需求原始表述：
[在此处粘贴原始需求原文]

请执行：
1. 血缘归并 — 将描述同一功能不同切面的碎片合并，还原完整表述
2. 角色聚合 — 不同角色对同一功能的诉求合并为一条，标注角色差异
3. 流程串联 — 同一流程不同步骤的碎片按操作顺序串联
4. 父子补全 — 以主需求为载体，补充细节作为子约束

遵循五条规则：
- 不做过度合并（合并后能用一句话说清核心功能）
- 保留原始出处（记录提出方、原始表述、日期）
- 性能指标不合并（功能可合并，指标单独提取）
- 矛盾碎片标注不合并（标注"冲突待确认"）
- 无法判断的标记为"待补全"

输出格式：
原始需求 [OR-XXX]
├─ 需求标题：[一句话概括]
├─ 需求描述：[合并后的完整需求描述]
├─ 原始碎片清单：
│  ├─ [碎片1原始表述] — [提出方] [日期]
│  └─ [碎片2原始表述] — [提出方] [日期]
├─ 包含的性能指标：[如有，单独列出]
├─ 依赖/冲突关系：[如有，标注]
└─ 整理说明：[归并依据]
```

#### 1b. 原始需求规范化和四项检查

```
请对初步整理后的原始需求进行规范化处理。

原始需求清单：
[粘贴1a的输出]

请执行：
1. 完整性检查 — 每条需求是否有清晰的描述、类型、优先级？
2. 清晰性检查 — 模糊术语替换为精确表述（"实时"→"延迟<100ms"，"快速"→"<2s"）
3. 可测试性检查 — 功能需求：输入/处理/输出是否明确？非功能需求：指标/条件/方法是否明确？
4. 独立性检查 — 需求是否相对独立？是否与其他需求过度耦合？
5. 功能/非功能分离 — 将"做什么"（功能）和"有多好"（非功能）分开记录

输出格式：
OR-XXX: [需求标题]
├─ 规范化描述：[经过四项检查后的描述]
├─ 类型：[功能需求/非功能需求]
├─ 优先级：[P0/P1/P2/P3] ⚠️ 待确认
├─ 验收标准：[可量化的验收条件]
└─ 性能指标：[如有，具体数值和条件]
```

#### 1c. 原始需求分解到末级

```
请将规范化后的原始需求分解到末级。

规范化原始需求清单：
[粘贴1b的输出]

分解原则：
1. 原子性原则 — 每条末级不可再分
2. 1:1映射原则 — 每条末级只需映射到一个SR架构末级节点
3. 语义完整性 — 分解后不丢失原始语义，所有末级之和=原始需求
4. 粒度一致性 — 同层末级粒度相当

触发分解的条件：
- 包含"和/并且/同时"连接多个功能 → 分解
- 涉及多个角色或相关方 → 分解
- 需要映射到多个SR末级 → 分解

停止准则：问自己"这个条目能映射到相关方需求架构的一个末级节点吗？"
- 只需一个节点承接 → 停止
- 需要多个节点承接 → 继续分解

输出格式：
原始需求: OR-XXX（[需求标题]）
末级分解:
  - ID: OR-XXX.X
    描述: [原子性功能描述]
    分配到: SR-F/SR-NFR-[待定]
    验收标准: [可量化条件]
```

#### 1d. 冲突和重复检测

```
请对末级需求清单进行冲突和重复检测。

末级需求清单：
[粘贴1c的输出]

冲突检测：
- 直接冲突：语义直接矛盾，无法同时满足 → 标记"需确认"
- 间接冲突：同时实现可能行为不一致 → 标记"需调整"
- 可合并冲突：可通过调整合并解决 → 标记"建议合并"

重复检测：
- 完全重复：两条完全相同 → 去重
- 部分重复：功能范围部分重叠 → 建议合并
- 相似重复：表达不同但核心相同 → 确认是否为同一需求

输出格式：
| 需求A | 需求B | 问题类型 | 描述 | 建议 |
|-------|-------|---------|------|------|
| OR-XXX | OR-YYY | 直接冲突 | [描述] | 需确认 ⚠️ |
```

#### 1e. 相关方需求架构定义

```
请基于末级需求清单，建立相关方需求架构定义。

末级需求清单： [粘贴1c的输出]
冲突/重复检测结果： [粘贴1d的输出]

请执行：
1. 分析末级需求的共性和关联性
2. 按功能需求（SR-F）和非功能需求（SR-NFR）分别分类
3. 为每组定义架构末级节点
4. 编写符合性分析（如何满足所承接的需求）
5. 编写设计指导（对业务架构的指导）
6. 建立映射关系表

架构定义文档包含两个部分：

第一部分：架构设计结果（树形结构）
相关方需求（0级·根）
├─ 功能需求（1级·分类）
│  └─ [分类名称]（2级·分类）
│     └─ SR-F-XXX: [名称]（架构末级节点）
│          ├─ 定义：[需求的定义]
│          ├─ 承接的原始需求末级：OR-XXX.X, OR-YYY.Y
│          ├─ 符合性分析：[如何满足]
│          ├─ 设计指导：[对业务架构的指导]
│          └─ 性能指标：[继承自原始需求]
│
└─ 非功能需求（1级·分类）
   └─ SR-NFR-XXX: [名称]（架构末级节点）
        ├─ 定义：[非功能需求的定义]
        ├─ 需求类型：[性能/安全/可用性/可维护性等]
        ├─ 需求类别：[质量特性/环境适应性/可实现性]
        ├─ 承接的原始需求末级：[OR-XXX]
        ├─ 符合性分析：[如何满足所承接的非功能需求]
        ├─ 量化指标：[具体可测量的指标，含数值+单位+条件]
        └─ 测量方法：[如何测试和验证]

第二部分：需求映射视图（映射关系表）
| 原始需求末级 | 承接的SR架构末级 | 关联说明 | 超链接 |
|-------------|-----------------|--------|--------|
| OR-001.1    | SR-F-XXX        | [说明]  | [→]    |
```

### 预期产出
- 原始需求文档（含末级需求分解）
- 相关方需求架构定义文档（树形结构 + 映射关系表）
- 冲突和重复检测报告

## 第2步：相关方需求功能部分详细定义及业务架构定义

### 目标
沿功能分支推进，产出相关方需求功能部分详细定义和业务架构定义。

### 输入
- 第1步产出的相关方需求架构定义（功能部分 SR-F）

### 涉及规范
- [相关方需求规范 §3 详细定义规范](../../01-design-standards/03-stakeholder-requirements-specification.md#三详细定义规范)
- [业务架构规范 §2 架构定义规范](../../01-design-standards/04-business-architecture-specification.md#二架构定义规范)

### 提示词模板

#### 2a. 分析活动：架构定义验证（在详细定义开始前执行）

```
请在开始详细定义前，对第1步产出的相关方需求架构定义执行分析活动。

相关方需求架构定义（功能部分）：
[粘贴1e的输出中SR-F部分]

需求说明符合性分析：
1. 检查每个SR-F架构末级节点是否充分满足所承接的原始需求末级
2. 验证N:1映射的完整性（无遗漏原始需求末级）
3. 验证功能与非功能需求的分离正确性

需求性能符合性分析：
1. 验证性能指标是否从原始需求正确继承和量化
2. 检查指标是否可测量、可验证

输出：分析活动报告
├─ 映射完整性：[通过/未通过，如有遗漏列出]
├─ 性能指标量化：[通过/未通过，如有问题列出]
└─ ⚠️ 待确认项：[如有]
```

#### 2b. 相关方需求功能部分详细定义

```
请对相关方需求功能部分（SR-F）的架构末级节点进行详细定义分解。

相关方需求架构定义（功能部分）：
[粘贴1e的输出中SR-F部分]

分解标准：
- 对每个SR-F架构末级节点进行详细设计分解
- 直到每条详细定义末级只需映射到业务架构的一个IPO（1:1约束）

分解方法：
- 按功能流程分解（步骤1 → 步骤2 → 步骤3）
- 按用户角色分解（角色A的操作 → 角色B的操作）
- 按业务规则分解（条件1的处理 → 条件2的处理）

停止准则：问自己"这个条目能映射到业务架构的一个IPO吗？"
- ❌ 需要多个IPO承接 → 继续分解
- ✅ 只需一个IPO承接 → 停止

BA命名规则：IPO 名称应使用"动词+名词"结构（如"立项审批""任务分配""报告归档"），准确反映业务活动的核心操作，不得使用模糊词汇。
分配目标中的 BA 名称由 AI 建议，仅确认归属是否正确（是否映射到正确的 IPO），不因名称措辞标记待确认。

输出格式：
架构末级节点：SR-F-XXX（[名称]）
详细定义末级节点：
  - ID: SR-F-XXX.X
    标题: [具体功能名称]
    描述: [使用"系统应该..."或"用户能够..."句式]
    分配目标: BA-[IPO名称] [1:1映射] ⚠️ 待确认
    验收标准: [可测试的标准]
    对上层的符合性: [如何满足SR-F-XXX]
```

#### 2c. 业务架构 IPO 定义

```
请根据功能部分详细定义结果，建立业务架构 IPO 清单。

相关方需求功能部分详细定义：
[粘贴2b的输出]

请执行：
1. 根据详细定义结果识别业务组件（IPO）
2. 为每个 IPO 定义 I（输入）/ P（处理）/ O（输出）
3. 执行 IPO 去重（复用→改进→新增）

IPO 去重三级策略：
第1步：首先检查 — 现有IPO中是否已有相同或相似的
第2步：复用优先 — 完全匹配的直接复用，记录映射来源
第3步：改进次之 — 改进后可用的则改进后复用，标注变更记录
第4步：新增最后 — 确实需要才新增

输出格式：
BA-XXX: [IPO名称]（5级·架构末级节点）
├─ 节点类型：[配置/开发]
├─ 定义：[业务组件的定义]
├─ Input（输入）：[具体形式、数据结构、来源]
├─ Process（处理）：[规则、步骤、约束条件]
├─ Output（输出）：[具体形式、数据结构、验收标准]
├─ 承接的SR详细定义末级：SR-F-XXX.X, SR-F-YYY.Y
├─ 符合性分析：[如何满足所承接的需求]
├─ 性能指标：[继承自相关方需求]
├─ 设计指导：[对系统需求的指导]
└─ 去重说明：[复用/改进/新增]
```

## 第3步：系统需求功能部分架构定义

### 目标
将业务架构 IPO 去重并分类组织，形成系统需求功能部分的架构定义。

### 输入
- 第2步产出的业务架构定义（IPO清单）
- 第2步同步建立的系统需求架构骨架

### 涉及规范
- [系统需求规范 §2.1 功能需求架构（0-9级）](../../01-design-standards/05-system-requirements-specification.md#21-功能需求架构0-9级)

### 提示词模板

#### 3a. IPO 去重（三级策略，确认版）

```
请对业务架构 IPO 清单执行最终去重确认。

业务架构 IPO 清单：
[粘贴2c的输出]

三级去重策略：
第1步：检查 — 跨所有BA流程，检查是否有相同或相似的IPO
第2步：复用 — 完全相同的IPO合并为一个（记录所有承接来源）
第3步：新增 — 确实唯一的IPO保留独立

输出格式：
去重前IPO数：[N]
去重后IPO数：[M]
去重映射：
| 原始BA IPO | 去重后IPO | 去重说明 |
|-----------|----------|---------|
| BA-001    | SysReq-F-001 | 合并（与BA-002的"订单创建"相同） |
| BA-002    | SysReq-F-001 | 合并（与BA-001的"订单创建"相同） |
| BA-003    | SysReq-F-002 | 唯一，保留 |
```

#### 3b. 功能需求架构组织（0-9级）

```
请将去重后的 IPO 按 0-9 级层级结构组织，形成系统需求功能部分架构定义。

去重后IPO清单：
[粘贴3a的输出]

0-9级层级结构：
0级：根节点
1级：独立信息系统
2级：功能需求分支
3级：业务组件类型（流程/工单/计划/指标等）
4级：业务组件小类
5级：组件实例 ←── 承接BA IPO（架构末级）
6级：功能模块（Tab/页面）
7级：系统用例（按钮/菜单）
8级：用例场景（正常/异常/替代）
9级：场景活动 ──→ 分配到PA末级（详细定义末级——本步只需建立5级架构）

请执行：
1. 按业务组件类型（3级）分类
2. 细分为业务组件小类（4级）
3. 形成组件实例清单（5级承接层）
4. 编写符合性分析和设计指导

架构定义文档两个部分：

第一部分：树形结构
系统需求（0级·根）
└─ [系统名称]（1级·独立信息系统）
   └─ 功能需求分支（2级）
      └─ [业务组件类型]（3级）
         └─ [业务组件小类]（4级）
            └─ SysReq-F-XXX: [名称]（5级·架构末级）
                 ├─ 定义：[系统需求的定义]
                 ├─ 承接BA：BA-XXX, BA-YYY
                 ├─ 符合性分析：[如何满足]
                 ├─ 设计指导：[对产品架构的指导]
                 └─ 性能指标：[继承自BA]

第二部分：映射关系表
| BA IPO | 承接的SysReq架构末级 | 超链接 |
|--------|---------------------|--------|
| BA-XXX | SysReq-F-XXX       | [→]    |
```

#### 3c. 同步建立产品架构骨架

```
请在完成功能需求架构后，同步建立产品架构结构骨架。

SysReq 5级节点清单：
[粘贴3b输出的5级节点]

请执行：
```

## 并行说明

在瀑布式6步流程中，以下分支可以**并行执行**：

| 并行组 | 步骤 | 说明 |
|--------|------|------|
| **功能分支** | 第2步 + 第3步 | 功能分支连续执行（第2步定义SR详细定义+BA IPO，第3步定义SysReq功能架构） |
| **非功能分支** | 第4步 | 独立于功能分支，与第2-3步完全并行 |
| **合并** | 第5步 | 等待功能分支和非功能分支都完成后，合并执行 |

因此实际时间线为：
```
第1步 → 第2+3步（功能） + 第4步（非功能）→ 第5步 → 第6步
```
