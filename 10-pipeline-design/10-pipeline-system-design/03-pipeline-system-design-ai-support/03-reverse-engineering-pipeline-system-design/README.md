# 逆向工程/代码恢复系统设计 Skill
**Reverse Engineering / Code Recovery System Design Skill**

**文档版本**：v1.1
**基于准则版本**：系统设计准则 v3.3
**基于指南版本**：逆向工程系统设计指南 v3.2
**对应场景**：03-reverse-engineering（已有产品代码或图纸，技术文档缺失或不达标）

---

## 一句话场景描述

产品已开发完成或接近完成，有完整的代码实现或架构图纸，但技术文档不全或不符合系统设计准则，需要从代码和图纸中逆向推导出完整的、符合准则的五层系统设计文档集。

## 适用条件

- ✅ 产品已开发完成或接近完成
- ✅ 有详细的代码实现或架构图纸
- ✅ 技术文档不全或不符合设计准则
- ✅ 缺少从需求到架构的完整追溯链路
- ✅ 需要生成符合准则的系统设计文档

## 不适用条件

- ❌ 产品代码不清晰或文档极度缺失（无法进行有效逆向推导）
- ❌ 原开发人员无法参与配合
- ❌ 需要快速完成，时间紧张
- ❌ 产品尚在设计阶段（应使用瀑布式或敏捷式场景）

## 依据B：两层产品关系说明

本目录下的文档是产品B（系统设计流水线）的 **AI辅助文档构件**，其上层产品架构节点为 `03-system-design-ai-support/03-reverse-engineering/`。

核心认知：
- **两套产品类型**：产品B（流水线）为方法论/工具产品，其PA分类为"规范与准则 / 设计指南 / AI辅助文档 / 任务定义"；产品A（AOS运营体系）为IT/AI化运营系统（软件产品），其PA分类为"前端 / 后端 / 数据 / 基础设施"
- **模板复用**：本文档为结构模板，供不同产品类型共用。使用时需根据目标产品类型选择相应的PA分类体系
- **使用者**：流水线开发者（团队B）使用本目录下的文档指导逆向工程系统设计
- **维护者**：团队B（流水线开发者 + AI）负责维护和更新这些模板文档

在本文档中，提示词模板和自检清单默认演示"软件产品"类型的逆向推导过程（从代码到文档），适用范围包括：
- 对产品B（流水线）自身软件实现的逆向推导
- 对产品A（AOS运营体系）或其他软件产品的逆向推导
- 对其他方法论/工具产品的软件实现的逆向推导

## 输入

| 输入 | 说明 | 来源 |
|------|------|------|
| 产品代码和架构图纸 | 系统源码、架构图、模块图、数据流图等 | 项目代码库、文档库 |
| 现有技术文档（如有） | 已有但不完整的各类设计文档 | 项目文档库 |
| 系统设计准则 v3.0 | 总纲，定义 5 层结构、三条核心规则 | `01-system-design-standards.md` |
| 各层设计规范 | 各层的架构定义、详细定义、映射规则 | `02~06 层规范` |
| 逆向工程设计指南 | 8 步操作流程和质量要求 | `03-reverse-engineering-guide.md` |

## 输出

AI 按 8 步流程依次产出以下文档集：

| 步骤 | 产出文档 |
|------|---------|
| 第1步 | 产品架构草图、功能模块清单、关键技术决策记录 |
| 第2步 | 产品架构文档（架构定义 + 详细定义 + 映射关系表骨架） |
| 第3步 | 系统需求文档（功能 0-9 级架构 + 非功能 0-6 级架构 + 详细定义 + 映射关系表） |
| 第4步 | 业务架构文档（IPO清单 + 去重报告 + 映射关系表） |
| 第5步 | 相关方需求文档（架构定义 + 详细定义 + 映射关系表） |
| 第6步 | 原始需求文档（含末级需求分解 + OR→SR 映射关系） |
| 第7步 | 验证报告（1:1/N:1/双向追溯/符合性分析完整率） |
| 第8步 | 完整的系统设计文档集（5 层）、需求映射矩阵、变更日志 |

## AI 能力边界和约束

### AI 可以做的

- ✅ 代码结构分析和模块划分识别
- ✅ 架构图纸解析和组件依赖关系提取
- ✅ 功能清单提取和 4 大组件分类（前端/后端/数据/基础设施）
- ✅ 从代码实现反推产品架构组件定义
- ✅ 从 PA 组件反推 SysReq 9 级场景活动
- ✅ 从 SysReq 反推 BA IPO（Input/Process/Output）
- ✅ 从 BA IPO 反推 SR 架构末级节点和详细定义
- ✅ 从 SR 架构末级节点聚合为 OR 条目和末级分解
- ✅ IPO 去重（复用→改进→新增三级策略）
- ✅ 按 0-9 级层级结构组织功能需求架构
- ✅ 按 0-6 级平行架构组织非功能需求架构
- ✅ 需求映射关系的建立和验证
- ✅ 符合性分析的编写
- ✅ 追溯矩阵的生成
- ✅ 从代码实际性能特征反推性能指标（标注"反推值"）

### AI 不可以做的（需要流水线开发者确认）

- ❌ 代码中缺失的业务逻辑判断 — 需原开发人员补充
- ❌ 架构图纸与代码实现不一致时的最终决策 — 需架构师确认
- ❌ 业务架构 IPO 的业务语义正确性 — 需业务人员审核
- ❌ 非功能需求的量化指标数值（从代码反推的仅是参考值）— 需架构师确认
- ❌ 原始需求的优先级判定 — 需产品经理确认
- ❌ 最终版本号的确定 — 需流水线开发者确认

## 角色分工

| 角色 | 职责 |
|------|------|
| **流水线开发者**（人） | 提供代码和图纸，审核 AI 每步的逆向推导结果，处理 AI 标记的待确认项，确定最终版本号 |
| **AI**（执行者） | 按本目录下的 workflow-prompts.md 依次执行 8 步逆向推导，生成文档初稿，标记待确认项，完成后运行 checklist.md 自检 |
| **AOS 开发者**（人） | 协助理解代码逻辑和业务含义，确认逆向推导结果的业务正确性 |
