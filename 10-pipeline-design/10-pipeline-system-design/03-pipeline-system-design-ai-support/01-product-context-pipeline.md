## 一、产品定位

### 1.1 产品定义

产品 B — **运营体系系统设计流水线**（System Design Pipeline），是为团队A（AOS 企业运营体系开发团队）的系统设计人员打造的一套"设计工具"，帮助他们高质量、高效率地完成产品 A 的系统设计。

> **一句话**：产品 B 是"设计工具"，产品 A 是"用这个工具来设计的目标系统"。

### 1.2 两层产品关系

| 产品 | 名称 | 负责团队 | 角色 |
|------|------|---------|------|
| **产品B** | 系统设计流水线 | 团队B（流水线开发者 + AI） | 产品 B 的生产者，同时是"设计工具"的生产者 |
| **产品A** | AOS 企业运营体系 | 团队A（系统设计人员） | 使用产品B产出的工具进行系统设计 |

产品 B 的**输出**（交付给产品 A 团队的资产）：

| 资产分类 | 内容 | 位置 |
|---------|------|------|
| 设计规范与准则 | 设计准则总纲、五层规范、术语表 | `50-aos-design/10-aos-system-design/01-aos-system-design-standards/` |
| 设计指南 | 四种场景的设计指南、快速参考卡 | `50-aos-design/10-aos-system-design/02-aos-system-design-guidelines/` |
| AI 辅助文档 | 四种场景的 AI 辅助文档集（各 4 份） | `50-aos-design/10-aos-system-design/03-aos-system-design-ai-support/` |
| 任务定义 | 四种场景的任务操作规程 | `50-aos-design/10-aos-system-design/04-aos-system-design-tasks/` |

### 1.3 产品类型

产品 B 属于**方法论/工具类产品**，而非传统的软件产品。其产品架构分类体系为：**规范与准则 / 设计指南 / AI辅助文档 / 任务定义**，而非软件产品的"前端/后端/数据/基础设施"。

### 1.4 产品阶段

产品 B 当前处于**阶段1（系统设计）**，严格限定在系统设计阶段的工作。
产品开发共四个阶段：系统设计 → 构件开发 → 系统集成与交付 → 系统运维。

> 产品 B 的范围限定在阶段1（系统设计），因为系统设计是产品开发中最重要也是难度最大的阶段。

## 二、用户画像

### 2.1 三个关键角色

| 角色 | 职责 | 在瀑布式中的参与 |
|------|------|-----------------|
| **流水线开发者** | 开发、维护和优化产品 B 本身 | 定义准则和指南，审核 AI 输出，关键决策 |
| **AI** | 按指令执行规范化、分解、映射、验证等自动化任务 | 所有步骤的自动生成和检查 |
| **AOS 开发者** | 使用产品 B 进行产品 A 的系统设计 | 提出需求、审核设计方案 |

### 2.2 角色交互方式

```
流水线开发者 ──(定义)──→ 准则/指南/AI辅助文档/任务定义
     │                            │
     │(审核/确认/决策)           │(指导)
     │                            ▼
     └───────── AI ──────────→ 执行系统设计任务
                                      │
                                      │(输出)
                                      ▼
                              产品设计文档集
```

## 三、产品全景

### 3.1 产品架构摘要（当前 PA）

产品 B 的产品架构由 60+ 个 PA 构件组成，分为六大层级：

| 层级 | 范围 | 构件数 |
|------|------|--------|
| 核心引擎层 | 需求解析、格式标准化、分类、检查、分解等 | ~10 |
| 设计映射层 | BA/SysReq/PA 设计组件、映射引擎 | ~22 |
| 验证追溯层 | 正反向追溯、符合性报告、质量检查 | ~12 |
| 标准资产层 | 准则、模式库、模板管理 | ~16 |
| 发布协作层 | 权限、检索、发布管理 | ~3 |
| AI 自动化层 | Skill 注册/编排/运行时 | ~6+ |

详细 PA 参见 [`10-pipeline-system-product-data/07-pipeline-product-architecture.md`](../10-pipeline-system-product-data/07-pipeline-product-architecture.md)。

## 四、设计约束

### 4.1 版本策略

- 版本号格式：`v[主版本].[次版本]`
- 语义化版本号（主版本破坏兼容、次版本兼容变更）
- 准则的版本变更会级联影响指南、AI 辅助文档和任务定义

### 4.2 文档规范

- 所有文档使用 **Markdown + Mermaid** 格式
- 文件命名规范：`NN-descriptive-name.md`
- 中文为主体语言，术语附英文对照
- 每个文档包含头部元数据（标题/版本/日期）和尾部版本记录

### 4.3 目录结构约束

仓库的目录结构按"通用方法论 / 产品B 开发空间 / 产品A 开发空间"三根目录组织：
- `00-general/` — 通用方法论
- `10-pipeline-design/` — 产品 B 开发空间
- `50-aos-design/` — 产品 A 开发空间

每个产品下按开发阶段组织为三个子目录：系统设计 / 构件开发 / 集成交付。

### 4.4 关键架构原则

- **产品同构性**：产品 B 和产品 A 共用同一套五层结构和设计方法论
- **同步设计**：上层详细定义与下层架构定义同步开展
- **架构定义与详细定义分离**：每层文档分为架构定义（树形结构）和详细定义（逐条细化）
- **分析活动贯穿**：每层在架构定义完成后、详细定义开始前，集中执行符合性分析

## 五、系统设计方法论（核心框架）

### 5.1 五层结构

```
第1层：原始需求及分析 (OR)
    ↓ N:1 分配
第2层：相关方需求 (SR)
    ├─ 功能部分 → 第3层：业务架构 (BA)
    └─ 非功能部分 → 第4层：系统需求-NFR
                    第4层：系统需求 (SysReq) ← BA IPO 去重后映射
                        ↓ N:1 分配
                    第5层：产品架构 (PA)
```

### 5.2 开发六步法

| 步骤 | 名称 | 分支 |
|------|------|------|
| 第1步 | 原始需求分析及相关方需求架构定义 | 统一处理 |
| 第2步 | 相关方需求功能部分详细定义及业务架构定义 | 功能分支 |
| 第3步 | 系统需求功能部分架构定义 | 功能分支 |
| 第4步 | 相关方需求非功能部分详细定义及系统需求非功能部分架构定义 | 非功能分支 |
| 第5步 | 系统需求详细定义及产品架构定义 | 分支合并 |
| 第6步 | 双向追溯验证 | 全链路验证 |

### 5.3 三条核心规则

| 规则 | 内容 |
|------|------|
| **1:1 分配约束** | 每条详细定义末级节点只能分配到下层方案中唯一一个架构末级节点 |
| **N:1 承接支持** | 一个架构末级节点可以承接多个上层详细定义末级节点（合理范围 2-5 条） |
| **双向追溯** | 支持从上层需求追溯到下层方案，也支持下层方案反向追溯到上层需求 |
