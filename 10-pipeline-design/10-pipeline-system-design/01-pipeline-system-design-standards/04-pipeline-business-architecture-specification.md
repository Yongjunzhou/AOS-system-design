## 一、核心概念

### 1.1 本层的定义

**业务架构（Business Architecture）** 是系统的运行概念（Operational Concept）——即**用户使用系统开展业务的过程说明**。

这意味着：
- 从**用户视角**描述如何使用系统，而非系统如何实现
- 体现**用户的工作价值**，而非技术实现细节
- 为**系统需求设计**提供功能和流程指导

### 1.2 在5层结构中的位置

```
相关方需求（第2层）
  ├─ 需求描述 + 性能指标
  └─ 功能部分 ↓ N:1承接支持

业务架构（第3层） ★ 本规范
  ├─ 业务组件（承接层）
  ├─ Input + Process + Output
  └─ 性能指标（继承自相关方需求）
      ↓ IPO去重后分配到系统需求（1:1分配约束）

系统需求（第4层）
  ├─ 系统功能需求架构（5级节点）
  └─ 性能指标（继承自业务架构）
```

| 维度 | 说明 |
|------|------|
| **文档类型** | 方案文档 — **仅架构定义**（无详细定义） |
| **承接对象** | 相关方需求详细定义末级（N:1承接支持） |
| **分配目标** | 系统需求架构5级节点（1:1分配约束，去重后） |
| **核心单位** | IPO（Input-Process-Output） |

### 1.3 本层特有的关键概念

**独立存在**：业务架构仅包含架构定义（IPO清单），不参与"详细定义的双重身份"递进链条。其架构末级节点自身承担"需求"角色，直接映射到系统需求架构末级节点。

**5级承接层**：业务架构的5级节点（业务组件）是承接相关方需求的关键层级，也是末级，不再向下分解。

**IPO模型**：每个业务组件用 Input（输入）、Process（处理）、Output（输出）三个维度来定义。

**去重机制**：同一IPO可能被多个业务流程引用。在映射到系统需求前，需完成IPO去重。

### 1.4 本层的分析活动

**需求说明符合性分析**：
- 检查每个业务组件（IPO）是否充分满足所承接的相关方需求详细定义末级
- 验证 1:1 分配约束：每条相关方需求详细定义末级只分配到一个业务组件
- 验证 N:1 承接支持的合理性：每个业务组件承接 2-5 条相关方需求详细定义末级为合理
- 验证 IPO 定义的完整性：Input、Process、Output 三维定义是否清晰

**需求性能符合性分析**：
- 验证性能指标是否从相关方需求正确继承
- 评估设计方案能否达到性能指标要求

分析活动贯穿本层设计工作流程（§4.1）的每个阶段，特别是在第三阶段集中执行。

## 二、架构定义规范

### 2.1 架构末级节点（5级承接层）

业务架构的末级节点是**5级业务组件**，是承接相关方需求的关键层级。

**关键特征**：
- **末级约束**：5级是最底层，不再向下分解
- **完整定义**：使用IPO模型完整定义每个业务组件
- **独立操作**：每个业务组件代表一个可独立进行的业务操作
- **可映射性**：每个业务组件可清晰映射到一条或多条相关方需求（N:1承接支持）
- **可实现性**：每个业务组件都有清晰的输入、处理、输出

**按实现方式的分类**：

| 类型 | 说明 | 示例 |
|------|------|------|
| **配置方式** | 通过流程/工单/计划/指标/表单引擎配置实现 | 需求收集流程、需求规范化流程、需求映射流程 |
| **开发方式** | 需要代码开发，提供配置引擎的基础功能 | 需求引擎、追溯分析系统、冲突检测引擎 |

### 2.2 层级结构

```
0级：架构根节点 — 整个系统的顶层根节点（唯一）
1级：业务域 — 按业务领域划分（3-10个）
2级：端到端项目 — 跨越多个业务域的项目或流程
3级：项目业务对象架构 — 业务对象和其架构
4级：工作包（WBS） — 业务对象的工作分解结构
5级：业务组件（承接层）— ★ 末级，不再分解
```

**设计要点**：
- 1-2 级定义业务结构和项目范围
- 3-4 级组织业务对象和工作单元
- **5 级是需求承接的关键层级**，使用 IPO 模型完整定义

### 2.3 架构定义的文档结构

```
BA-001: [IPO名称/业务组件名称]（5级·架构末级节点）
├─ 节点类型：[配置/开发]
├─ 定义：[业务组件的定义]
├─ 承接的相关方需求详细定义末级：
│  ├─ SR-F-001.1
│  ├─ SR-F-001.2
│  └─ SR-F-002.1
├─ 符合性分析：[如何满足所承接的需求]
├─ 性能指标：[继承自相关方需求]
├─ Input：[输入的具体形式和来源]
├─ Process：[处理的步骤和规则]
├─ Output：[输出的具体形式和去向]
├─ 参与角色：[执行此业务的角色]
└─ 为下层提供的设计指导：[对系统需求的指导]
```

**颗粒度对比**：

```
✅ 合适的粒度（应分为独立业务组件）：
  BA-001: 需求收集管理
  BA-002: 需求规范化
  BA-003: 需求映射验证

❌ 粒度过粗：
  BA-001: 需求管理（涵盖多个完全不同的业务，太宽泛）

❌ 粒度过细：
  BA-001: 需求字段修改（只是一个操作，不是独立业务组件）
```

## 三、IPO模型

### 3.1 定义

IPO模型是业务架构的基本设计单位，用三个维度完整定义一个业务组件：

| 维度 | 定义 | 内容 |
|------|------|------|
| **Input（输入）** | 业务组件接收的输入 | 形式、数据结构、数据来源、前置条件 |
| **Process（处理）** | 对输入进行的处理 | 处理规则、算法、步骤、决策点、约束 |
| **Output（输出）** | 业务组件生成的输出 | 形式、数据结构、验收标准、后续使用 |

### 3.2 示例

```
BA-001: 需求收集
├─ Input：原始需求文本、来源信息、优先级标注
├─ Process：
│  ├─ 验证原始需求完整性（标题、描述、来源）
│  ├─ 检查需求信息是否完整
│  ├─ 标记需求来源和优先级
│  └─ 生成需求条目并保存至需求池
└─ Output：结构化需求条目、需求列表（返回至需求池）
```

```
BA-002: 需求分解
├─ Input：结构化需求条目、分解规则配置
├─ Process：
│  ├─ 根据需求类型和范围确定分解策略
│  ├─ 将需求逐步分解至末级节点
│  └─ 验证末级节点满足原子性标准
└─ Output：分解后的需求末级节点清单、分解路径记录
```

### 3.3 颗粒度标准

IPO的体量取决于**期望的输出物颗粒度**：

```
期望输出是设计制品 → 一个IPO对应一份制品（或可由单人完成的部分）
期望输出是业务流程 → 一个IPO对应一个完整的业务流程或操作
期望输出是数据对象 → 一个IPO对应一个数据对象的生命周期
```

**判断标准**：能否分配到系统需求的唯一一个架构末级节点（5级节点）？

```
✅ 合适的IPO：
  IPO-001: 需求收集（Input: 原始需求 → Process: 验证+结构化+保存 → Output: 需求条目）
  → Process 步骤清晰可执行，输出物明确

❌ 过于抽象：
  IPO-001: Process = "处理需求"
  → 无法知道具体怎么处理，无法指导系统需求设计

❌ 过于细化：
  IPO-001: 输入需求标题
  → 这只是需求收集中的一个字段录入，不是一个独立业务操作
```

### 3.4 节点定义标准

每个业务架构末级节点应包含以下属性：

| 属性 | 说明 | 示例 |
|------|------|------|
| **节点ID** | 唯一标识 | BA-001 |
| **节点名称** | 业务组件名称 | 需求收集 |
| **节点类型** | 配置/开发 | 流程引擎配置 |
| **节点定义** | 业务功能的简要描述 | 处理需求的全生命周期管理 |
| **承接的相关方需求** | 承接了哪些详细定义末级 | SR-F-001.1, SR-F-001.2 |
| **符合性分析** | 如何满足所承接的需求 | 每个需求都有对应的业务操作说明 |
| **性能指标** | 继承自相关方需求的性能要求 | 需求保存 < 2s，支持 100+ 并发用户 |
| **Input** | 输入的具体形式和来源 | 原始需求文本、来源信息 |
| **Process** | 处理的步骤和规则 | 1. 验证完整性... 2. 格式化描述... |
| **Output** | 输出的具体形式和去向 | 结构化需求条目、保存至需求池 |
| **参与角色** | 执行此业务的角色 | 需求提出者、流水线开发者 |
| **设计指导** | 对系统实现的指导 | 系统应提供需求录入界面... |

## 四、设计方法

### 4.1 三阶段工作流程

> **与准则的关系**：本节描述的三阶段是准则 §1.5.3"两阶段工作流程"在业务架构层的具体应用——第一阶段对应准则的快速约定结构骨架，第二、三阶段对应准则的并行填充内容+验证。因业务架构是唯一只有架构定义、无详细定义的层，其工作流程额外强化了去重和冻结环节。

```
第一阶段：占位符创建（快速，1-2小时）
  1. 扫描相关方需求功能部分的详细定义末级
  2. 快速识别并创建业务组件占位符（5级节点名称）
  3. 建立相关方需求 → 业务组件的骨架映射
  4. 同时为系统需求创建结构骨架占位符
  输出：业务组件占位符清单 + 骨架映射关系

第二阶段：详细填充（并行，2-3天）
  1. 详细定义每个业务组件的IPO
  2. 进行业务组件去重（复用→改进→新增）
  3. 填充完整的符合性分析和设计指导
  4. 同步建立系统需求架构的完整结构
  输出：完整的业务架构定义文档

第三阶段：验证和冻结（最终确认）
  1. 验证1:1分配约束
  2. 验证N:1承接支持的合理性
  3. 检查IPO定义的完整性
  4. 冻结业务架构
  输出：已验证的业务架构文档
```

### 4.2 与相关方需求的映射

```
映射的源头：相关方需求详细定义末级节点（SR-F-XXX.X）
映射的目标：业务架构末级节点（5级业务组件，BA-XXX）
映射的关系：N:1承接支持 — 一个业务组件可承接多条相关方需求详细定义末级
```

**符合性分析要求**：

```
BA-001: 需求收集管理
├─ 承接的相关方需求：SR-F-001.1, SR-F-002.1, SR-F-003.1
├─ 符合性分析：
│  ├─ SR-F-001.1（需求录入）：通过需求录入界面满足
│  ├─ SR-F-002.1（需求规范化）：通过需求格式化功能满足
│  └─ SR-F-003.1（需求存储）：通过需求池管理功能满足
└─ 设计指导：系统应提供需求全生命周期管理功能
```

### 4.3 去重机制

**优先级**：复用 > 改进 > 新增

```
第1步：首先检查
  → 现有业务架构中是否已有相同或相似的IPO

第2步：复用优先
  → 如果存在完全匹配的IPO，直接复用（记录映射来源）

第3步：改进次之
  → 如果现有IPO改进后可满足需求，则改进后复用（标注变更记录）

第4步：新增最后
  → 确实需要新的IPO才新增
```

**去重对比**：

```
✅ 正确做法：
  流程A需要"需求收集"，流程B也需要"需求收集"
  → 复用同一个IPO（BA-001），两个流程都映射到BA-001
  → 好处：系统需求中只需实现一次"需求收集"

❌ 错误做法：
  流程A定义BA-001"需求收集"，流程B定义BA-101"需求收集"
  → 两个IPO功能相同但ID不同
  → 后果：系统需求中出现两个"需求收集"，导致重复开发
```

### 4.4 同步设计

在开展业务架构详细填充（第二阶段）的同时，应同步开展系统需求架构的骨架建设。

**原因**：
- 业务架构的IPO需要分配到系统需求架构的5级节点（去重后）
- 如果系统需求架构尚未建立，IPO就没有分配目标，容易造成返工

**实际操作**：

```
1. 在创建业务组件占位符时，同步创建系统需求5级节点占位符
2. 在填充IPO详细内容时，同步建立BA → SysReq的映射关系
3. 冻结映射关系后，并行进行内容深填
```

### 4.5 设计原则

**N:1承接支持**：一个业务组件可以承接多条相关方需求。承接 2-3 条最优，1 条需考虑合并，4-5 条可接受（需检查是否有过度聚合），6 条以上需考虑拆分。

**1:1分配约束**：每条相关方需求只分配到唯一的业务组件。如果一条需求需要分配到多个组件，说明分解不充分。

**设计指导原则**：每个业务组件应为系统需求提供清晰的设计指导，包括系统应提供什么功能、如何组织信息、如何验证业务规则。

### 5.2 IPO定义完整性验收

| 检查项 | 标准 | 说明 |

| **设计指导完整率** | 100% | 每个业务组件都为系统需求提供明确的指导 |

### 5.3 层级结构完整性验收

| 检查项 | 标准 | 说明 |
|--------|------|------|

## 六、检查清单

### 6.1 需求映射检查

- [ ] 所有相关方需求功能部分的详细定义末级都已映射到业务架构
- [ ] 没有遗漏任何相关方需求详细定义末级
- [ ] 没有重复映射（一条需求映射到多个业务组件）
- [ ] 映射覆盖率 = 100%

### 6.2 业务组件设计检查

- [ ] 每个业务组件都有清晰的ID、名称、定义
- [ ] 每个业务组件都有完整的Input/Process/Output定义
- [ ] 每个业务组件都标注了实现方式（配置/开发）
- [ ] 每个业务组件都有清晰的设计指导

### 6.3 业务组件去重检查

- [ ] 进行了跨流程的业务组件去重
- [ ] 相同或相似的业务操作已识别
- [ ] 去重结果已清晰标注
- [ ] 去重完成率 = 100%

### 6.4 映射关系验证

- [ ] 每条相关方需求详细定义末级分配到唯一的业务组件（1:1约束）
- [ ] 每个业务组件承接的详细定义末级之间有逻辑关联
- [ ] 映射关系支持双向追溯

### 6.5 符合性分析检查

- [ ] 每个业务组件都有符合性分析
- [ ] 符合性分析清晰说明了如何满足所承接的需求
- [ ] 符合性分析没有遗漏任何承接的需求

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误1：业务组件定义过于宽泛**

```
错误的做法：
BA-001: 需求管理系统
├─ Process: 管理所有需求
└─ 涵盖：需求收集、需求规范化、需求分解、需求映射、追溯验证

问题：一个组件涵盖多个完全不同的业务操作
后果：系统需求设计时无法确定该分配什么功能，IPO定义无法聚焦
```

```
✅ 正确做法：
BA-001: 需求收集管理    → 聚焦需求收集和结构化
BA-002: 需求规范化      → 聚焦需求格式化和检查
BA-003: 需求映射验证    → 聚焦映射关系管理

说明：每个组件聚焦一个独立的业务操作
```

**❌ 错误2：一条相关方需求分配到多个业务组件**

```
错误的做法：
SR-F-001.1（需求收集）→ BA-001（需求收集）+ BA-002（需求来源接口）

问题：一条需求需要两个组件承接，违反1:1分配约束
后果：系统需求设计时无法确定需求收集功能到底归谁
```

```
✅ 正确做法：
方案A：将SR-F-001.1继续分解
  SR-F-001.1.1（需求收集-信息录入）→ BA-001
  SR-F-001.1.2（需求收集-来源接入）→ BA-002

方案B：重新设计业务组件，使一个组件能完整承接
  BA-003: 需求收集（覆盖信息录入+来源接入）
```

**❌ 错误3：IPO定义过于抽象**

```
错误的做法：
BA-001: 需求处理
├─ Input: 需求数据
├─ Process: 处理需求
└─ Output: 处理结果

问题：Process = "处理需求" 没有说明任何处理逻辑
后果：系统需求设计者完全无法理解这个业务组件具体做什么
```

```
✅ 正确做法：
BA-001: 需求处理
├─ Input: 原始需求文本、来源信息、优先级
├─ Process:
│  1. 验证需求信息完整性
│  2. 检查需求冲突和重复
│  3. 格式化需求描述
│  4. 分配需求类别
│  5. 保存至需求池
└─ Output: 结构化需求条目、需求列表

说明：具体明确的操作步骤，系统需求设计者可直接据此设计功能
```

**❌ 错误4：忽视业务组件去重**

```
错误的做法：
流程A（常规需求）：BA-001"需求收集"
流程B（紧急需求）：BA-101"需求收集"（与BA-001功能相同）

问题：两个流程各自定义了相同的IPO
后果：系统需求中出现两个"需求收集"，造成重复开发
```

```
✅ 正确做法：
识别到BA-001和BA-101功能相同，合并为一个IPO
→ BA-001"需求收集"同时承接流程A和流程B的相关方需求
→ 系统需求中"需求收集"只需实现一次
```

### 7.2 最佳实践

**实践1：使用三阶段工作流程**：先占位符 → 再并行填充 → 最后验证冻结。避免一次性完成的瀑布式做法。

**实践2：充分利用去重机制**：在开始新增IPO前，先检查是否已有相同或相似的IPO可复用。优先复用，避免重复开发。

**实践3：设计指导具体化**：每个IPO的"设计指导"不应只是描述性文字，而应具体说明"系统应提供什么功能来支持这个业务过程"。

**实践4：保持双向追溯**：每条相关方需求都能追溯到业务组件，每个业务组件都能追溯到相关需求。

**实践5：与系统需求同步设计**：在填充IPO时同步创建系统需求架构的占位符，建立BA → SysReq的骨架映射。

## 八、FAQ

**Q1：业务架构为什么不需要详细定义？**

业务架构是从用户角度描述业务过程，IPO模型本身已充分定义了业务操作。系统需求的详细定义（如用例场景、活动步骤）是系统实现的细节，不是业务操作的一部分。

**Q2：如何判断业务组件的粒度是否合适？**

关键标准是1:1分配约束——这个业务组件是否能唯一对应系统需求的一个功能组件。如果一个业务组件需要分配到多个系统功能，说明粒度太大，需要进一步分解。

**Q3：业务组件去重是否必须完成？**

是的，完全必须。去重确保系统中不存在重复的业务操作。在映射到系统需求时，去重结果被作为系统功能需求的源头。去重不完整会直接导致系统功能重复定义。

**Q4：一个业务组件承接多少条相关方需求合适？**

没有硬性限制，但建议范围：
- 1 条：太少，考虑是否可与其他组件合并
- 2-3 条：最优，分组合理
- 4-5 条：可接受，但需检查是否有过度聚合
- 6 条以上：过多，强烈建议拆分

**Q5：业务架构和流程图的区别是什么？**

业务架构从系统设计角度描述用户使用系统的过程和组件，为系统需求提供指导。流程图描述具体的工作流程和操作步骤。业务架构的IPO可以理解为对流程的高度抽象——IPO关注"输入→处理→输出"的结构，而非每一步的细节。
