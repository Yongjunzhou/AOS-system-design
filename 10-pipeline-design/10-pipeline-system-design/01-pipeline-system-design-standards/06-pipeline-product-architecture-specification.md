## 一、核心概念

### 1.1 本层的定义

**产品架构（Product Architecture）** 是将系统需求转化为具体构件实现方案的设计结果。它回答的问题是：**"系统应该如何被构建？构件之间如何交互？"**

产品架构在系统设计流水线中扮演着**方案文档**的角色：
- **是系统需求（第4层）的方案**：将功能和非功能要求转化为构件的实现方案
- **是产品研发的蓝图**：为开发团队提供明确的实现指导

> **注**：构件的实现方案形态因产品类型而异。对于本流水线（方法论/工具产品），构件实现方案体现为文档、模板、工作流定义等设计制品。

### 1.2 在5层结构中的位置

```
系统需求（第4层）
  ├─ 功能：SysReq-F 9级（场景活动）
  └─ 非功能：SysReq-NFR（系统约束）
       ↓ N:1 映射

产品架构（第5层） ★ 本规范
  ├─ 架构定义：产品构件清单 ← 承接SysReq 9级
  └─ 详细定义：实现方案、文档结构、质量指标
       ↓
产品研发（构件实现）
```

| 维度 | 说明 |
|------|------|
| **文档类型** | 方案文档（架构定义 + 详细定义） |
| **承接对象** | SysReq 9级场景活动（N:1）+ SysReq-NFR |
| **输出对象** | 产品研发团队（开发实现） |

### 1.3 产品架构的组成

产品架构的组成因**产品类型**而异。本流水线（系统设计流水线）属于**知识/工具类产品**，其产品构件的组成如下：

| 组成 | 说明 | 示例 |
|------|------|------|
| **规范与准则** | 定义设计标准和方法论 | 系统设计准则、层规范文档 |
| **操作指南** | 指导不同场景下的操作流程 | 设计指南、快速参考卡 |
| **AI辅助工具** | AI参与设计的支持文档 | 工作流提示词、检查清单 |
| **任务定义** | 定义操作规程和步骤 | 六步法任务定义 |

### 1.4 本层的分析活动

**需求说明符合性分析**：
- 覆盖度检查 — 所有 SysReq 9 级节点是否都有对应的 PA 构件承接
- 1:1 分配验证 — 每个 SysReq 9 级节点是否分配到唯一的 PA 构件
- N:1 承接合理性 — 每个 PA 构件承接 2-5 个 SysReq 9 级节点为合理

**需求性能/质量符合性分析**：
- 指标继承验证 — PA 构件的质量指标不低于所承接 SysReq 和 NFR 的要求
- 可实现性评估 — 结合文档格式和内容结构评估质量指标的可实现性
- 权衡决策记录 — 当多个 NFR 冲突时，权衡决策是否已记录

## 二、架构定义规范

### 2.1 架构末级节点

产品架构的末级节点是**产品构件**，每个构件应有清晰的职责边界。

每个产品架构末级节点包含：

| 属性 | 说明 | 示例 |
|------|------|------|
| **构件ID** | 唯一标识 | PA-001 |
| **构件名称** | 构件名称 | 系统设计准则总纲 |
| **功能角色** | 规范/指南/AI辅助/任务定义 | 规范 |
| **定义** | 构件的职责和用途描述 | 定义系统设计的方法论框架包含五层结构和三条核心规则 |
| **承接的系统需求** | 承接了哪些SysReq 9级节点 | SysReq-F-001.1.1.1 |
| **符合性分析** | 如何满足所承接的系统需求 | 通过完整的章节结构覆盖全部方法论定义 |
| **实现方案** | 具体的文档实现方案 | Markdown + Mermaid，章节结构，模板约定 |
| **质量指标** | 该构件需要满足的质量目标 | 完整性100%，一致性0矛盾 |

### 2.2 层级结构

产品架构的层级相对灵活，一般遵循以下模式：

| 层级 | 名称 | 说明 |
|------|------|------|
| **0级** | 产品根节点 | 整个产品的顶层 |
| **1级** | 功能角色分类 | 规范/指南/AI辅助/任务定义 |
| **2级** | 子系统/模块 | 按功能领域划分 |
| **3级** | **构件（承接层）** | **★ 末级，承接SysReq 9级节点** |

> **注**：产品架构的层级数可根据产品复杂度适当调整。

### 2.3 构件分类

本流水线（系统设计流水线）作为知识/工具类产品，按功能角色分类：

| 类别 | 子类型 | 说明 | 构件示例 |
|------|--------|------|---------|
| **规范与准则** | 核心准则 | 定义系统设计方法论框架 | `01-pipeline-design-standards.md` |
| | 层规范 | 定义各层的设计规范和标准 | `02-` 至 `06-` 层规范文档 |
| **操作指南** | 场景指南 | 指导不同场景下的设计流程 | 瀑布式/敏捷式/逆向工程/DevOps 指南 |
| | 快速参考 | 提供快速查阅的摘要卡片 | `05-quick-reference-card.md` |
| **AI辅助文档** | 场景AI支持 | 各场景下的工作流提示词和检查清单 | 各场景的 AI 辅助文档目录 |
| | 通用AI支持 | AI文档理解要求 | `00-ai-document-requirements-understanding.md` |
| **任务定义** | 核心任务 | 开发六步法的任务定义 | 各任务定义目录 |

### 2.5 构件间关系定义

本流水线作为知识/工具类产品，构件间关系如下：

| 关系类型 | 说明 | 示例 |
|---------|------|------|
| **引用** | 构件A引用构件B的内容 | 层规范引用准则总纲的定义 |
| **实现** | 构件A是构件B的具体实现 | 任务定义实现设计指南规定的流程 |
| **组合** | 构件A由构件B和C组成 | AI辅助文档 = 工作流提示词 + 检查清单 + 示例 |
| **关联** | 构件A与构件B存在逻辑关联 | 层规范与对应场景指南存在交叉引用 |

### 2.6 构件划分原则

构件划分的核心原则是**高内聚、低耦合**——每个构件应有明确的职责边界。

```
✅ 合适的构件划分：
  PA-001: 系统设计准则总纲（一个完整的方法论定义，独立文件）
  PA-002: 原始需求规范（一个完整的层规范，独立文件）
  PA-003: 瀑布式设计指南（一个完整的场景指南，独立文件）
  → 每个构件是一个完整的设计制品，可独立阅读和使用

❌ 不合适的构件划分：
  PA-001: 所有层规范合并为一个文件（粒度过粗，难以维护）
  PA-002: 准则总纲中"1:1分配约束"的定义（粒度过细，应作为准则内的条目而非独立构件）
  → 粒度过粗导致文档臃肿，粒度过细导致碎片化
```

## 三、详细定义规范

### 3.1 详细定义的本质

产品架构详细定义是对每个产品构件进行深入的实现方案描述，使开发团队能够直接进入实现阶段。

对知识/工具类产品（本流水线）而言，每个构件的详细定义包含：

| 内容 | 说明 | 示例（以"系统设计准则总纲"构件为例） |
|------|------|------|
| **文档格式** | 使用的文档格式和图表语言 | Markdown + Mermaid |
| **内容结构** | 构件的章节和内容组成 | 总则→五层结构→核心规则→设计要点→产品构件 |
| **关键条目** | 构件中的核心定义和规则 | 五层结构定义、三条核心规则、术语定义 |
| **版本策略** | 版本管理和演进策略 | 语义化版本号（主版本·次版本） |
| **引用关系** | 对其他构件的引用和依赖 | 引用术语表；被所有层规范引用 |
| **质量指标** | 具体的质量标准 | 完整性100%（覆盖准则全部条目）、一致性0矛盾 |
| **模板约束** | 格式和风格要求 | 中文为主体语言，术语附英文对照；使用本规范的章节结构 |

### 3.2 详细定义模板

```yaml
PA-001: 系统设计准则总纲
├─ 文档格式
│   ├─ 标记语言: Markdown
│   └─ 图表语言: Mermaid
├─ 内容结构
│   ├─ 第1章: 总则（目的、范围、适用场景）
│   ├─ 第2章: 五层结构定义
│   ├─ 第3章: 核心规则（1:1分配约束、N:1承接支持、双向追溯）
│   ├─ 第4章: 各层设计要点（OR → SR → BA → SysReq → PA）
│   └─ 附录: 术语表
├─ 关键条目
│   ├─ 五层结构及其映射关系: 完整定义和图示
│   ├─ 三条核心规则: 每条规则带示例和边界条件
│   └─ 术语定义: 架构定义 vs 详细定义、需求 vs 方案 等
├─ 版本策略
│   ├─ 版本号: 语义化版本（如 v3.3）
│   ├─ 变更记录: 每次修订记录变更摘要
│   └─ 向下兼容: 次版本升级不破坏已有层级结构
├─ 引用关系
│   ├─ 被引用: 所有层规范（02-06）均引用本准则
│   └─ 发起引用: 术语表（07）
├─ 质量指标
│   ├─ 完整性: 100%覆盖准则全部定义条目
│   ├─ 一致性: 层规范间0矛盾
│   └─ 清晰性: 每条规则有明确的正例和反例
└─ 模板约束
    ├─ 语言: 中文为主体，术语附英文对照
    ├─ 文件命名: `NN-descriptive-name.md` 格式
    └─ 版本标注: 文件头部和尾部均标注版本号和最后更新日期
```

### 3.3 详细定义深度对比

```
✅ 充分的详细定义：
  文档格式: Markdown + Mermaid
  内容结构: 总则→五层结构→核心规则→设计要点→产品构件（5章+附录）
  关键条目: 五层结构含完整图示、三条规则含正例反例、术语表含20+术语
  → 文档编写者可直接按结构填充内容，无需额外沟通

❌ 不充分的详细定义：
  文档格式: Markdown
  内容结构: 待定
  关键条目: 五层结构
  → 信息太少，编写者需要反复确认结构和内容边界
```

## 四、设计方法

### 4.1 总体设计流程

```
步骤1：分析系统需求
  └─ 收集所有SysReq 9级节点和SysReq-NFR
     └─ 识别需求之间的关联关系
        └─ 输出：SysReq 9级清单 + NFR约束清单

步骤2：识别构件候选
  └─ 按功能相关性分组SysReq 9级节点
     └─ 按规范/指南/AI辅助/任务定义归类
        └─ 评估构件的粒度和边界

步骤3：建立架构骨架
  └─ 采用功能角色分类（规范/指南/AI辅助/任务定义）
     └─ 创建构件占位符，建立SysReq→PA初步映射
        └─ 输出：架构骨架 + 映射关系

步骤4：详细构件设计
  └─ 定义每个构件的职责、文档格式和内容结构
     └─ 定义构件间的引用关系
        └─ 输出：完整的架构定义 + 详细定义

步骤5：非功能约束验证
  └─ 检查每个NFR是否有对应的构件设计体现
     └─ 验证质量指标是否满足NFR要求
        └─ 输出：NFR符合性检查报告

步骤6：映射验证
  └─ SysReq→PA覆盖率 100%、1:1约束 100%
     └─ 输出：映射验证报告
```

### 4.2 两阶段工作流程

```
第一阶段：骨架建立（快速，1-2小时）
  1. 扫描SysReq 9级节点
  2. 创建PA构件占位符
  3. 建立SysReq→PA骨架映射
  4. 冻结骨架结构
  输出：构件占位符 + 映射关系

第二阶段：详细填充（深入，2-4天）
  1. 详细定义每个构件的实现方案
  2. 填充文档格式、内容结构、质量指标
  3. 验证非功能约束的可满足性
  4. 验证映射的完整性和一致性
  5. 冻结产品架构文档
  输出：完整的产品架构定义 + 详细定义
```

> **与准则的关系**：本节描述的是本层内部的两阶段工作流程。准则 §1.5.3 定义的"两阶段工作流程"是跨层级的：阶段一快速约定 PA 结构骨架并建立 SysReq→PA 映射关系（如 §4.3 所述），阶段二再并行填充 SysReq 详细定义和 PA 详细定义的详细内容。本节的"第一阶段：骨架建立"对应准则的阶段一在本层的操作，"第二阶段：详细填充"对应准则的阶段二中本层的操作。

### 4.3 同步设计

产品架构的初步结构应与系统需求详细定义**同步开展**。

**实际操作**：

```
1. SysReq 5级节点确定后 → 快速建立PA结构骨架（占位符）
2. SysReq 9级节点确定时 → 同步映射到PA构件
3. 冻结映射关系后 → 并行填充详细内容
4. 非功能约束 → 在构件设计阶段同步验证
```

**为什么同步**：SysReq 9级场景活动需要分配到PA构件。如果PA尚未建立，SysReq 9级分解就没有分配目标，可能导致分解粒度过粗或过细而需要返工。

### 4.4 非功能权衡决策

当多个非功能需求冲突时，需要进行权衡决策。本流水线常见权衡场景如下：

| 冲突场景 | 权衡决策 | 记录方式 |
|---------|---------|---------|
| 完整性 vs 可读性 | 详细定义增加文档长度 | 记录最低完整度要求和可读性标准 |
| 通用性 vs 针对性 | 通用化增加复杂度，专用化降低复用 | 记录适用范围边界和定制策略 |
| 稳定性 vs 演进速度 | 频繁修订影响团队使用连续性 | 记录版本策略和向下兼容承诺 |

### 4.5 设计原则

**N:1承接支持**：一个PA构件可承接多个SysReq 9级节点。2-5个合理，1个需考虑合并，6个以上需考虑拆分。

**1:1分配约束**：每个SysReq 9级节点分配到PA中唯一的构件。如果需要多个构件，说明9级活动还需继续分解。

**有序组织**：按功能角色分类（规范→指南→AI辅助→任务定义）。核心要求：职责清晰、独立演化、便于复用。

**非功能约束满足**：每个PA构件标注受哪些NFR约束，构件的质量指标不低于SysReq的要求。

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误1：构件粒度过大或过小**

```
错误的做法：
PA-001: 所有层规范合并为一个文件（粒度过粗，难以维护）
PA-002: 准则总纲中"1:1分配约束"的定义（粒度过细，应作为准则内条目）

问题：粒度过粗导致职责不清，粒度过细导致管理复杂
后果：开发团队无法明确分工，构件边界模糊
```

```
✅ 正确做法：
PA-001: 系统设计准则总纲（粒度适中——一个完整的方法论定义）
PA-002: 原始需求规范（粒度适中——一个完整的层规范）
→ 每个构件是一个完整设计制品，有清晰边界
```

**❌ 错误2：映射关系不完整**

```
错误的做法：
SysReq有20个9级节点，但PA只定义了15个构件承接
→ 5个SysReq 9级节点未映射

问题：部分功能需求没有对应的构件实现
后果：开发时遗漏功能或临时添加导致架构混乱
```

```
✅ 正确做法：
建立一个SysReq 9级 → PA构件的完整映射表
逐条验证：每个SysReq 9级都有对应的PA构件
→ 映射覆盖率 100%
```

**❌ 错误3：忽略非功能约束**

```
错误的做法：
PA构件定义中只写了功能描述，没有质量指标
→ "系统设计准则总纲：提供设计方法论定义"

问题：开发时不知道质量目标，交付后可能不达标
后果：质量标准不满足需要返工修订
```

```
✅ 正确做法：
PA构件定义中包含明确的质量指标
→ "系统设计准则总纲：完整性100%，层规范间0矛盾"
→ 编写时有明确的质量目标
```

### 7.2 最佳实践

**实践1：先骨架后填充**：先快速建立架构骨架（占位符），同步建立映射关系，再并行填充详细内容。

**实践2：构件以"独立交付单元"为粒度**：每个构件应是可独立交付和验证的最小单元——以"一个编写者可在1-2天内完成一个完整文档"为标准。

**实践3：将NFR纳入构件定义**：每个构件定义中都包含可测量、可验证的质量指标，在详细定义中说明NFR的满足方案。

**实践4：记录架构决策**：关键技术选择记录决策理由，NFR权衡决策记录在案。

## 八、FAQ

**Q1：产品架构的层级数是固定的吗？**

不是。产品架构的层级可根据产品复杂度适当调整。简单产品可能只有 2 级（根节点 → 构件），复杂产品可能需要 4 级或更多。关键是**保持层级的一致性**——同层节点应在同一抽象级别。

**Q2：一个PA构件承接多少个SysReq 9级节点合理？**

建议每个PA构件承接 2-5 个 SysReq 9 级节点（2-3 条最优，1 条需考虑合并，4-5 条可接受需检查过度聚合，6 条以上说明粒度过粗需考虑拆分）。承接 0 个说明构件可能多余。

**Q3：构件划分到多细才算合适？**

以"一个编写者可在 1-2 天内完成一个完整文档"和"可独立阅读和使用"为双重标准。如果构件太大（需要多天）、太依赖其他构件才能使用，说明还需拆分。

**Q4：当完整性和可读性冲突时怎么办？**

记录权衡决策。例如"准则总纲需要覆盖所有方法论条目（完整性要求），但全部展开会导致文档过长无法阅读（可读性要求）。决策：主体内容保持简洁，详细条目放入附录或引用到层规范。"
