# 敏捷式系统设计：产品数据范例
**Agile System Design: Product Data Example**

**用途**：展示一个敏捷式增量系统设计过程的输入和输出范例，AI 参照此范例生成自己的增量设计文档。

**范例项目**：设计线·系统设计流水线（System Design Pipeline）

**基线说明**：本范例基于瀑布式场景（`01-waterfall/product-example.md` v1.2）的完整设计输出作为基线。基线内容包含：
- 21 个 PA 构件（设计规范与准则 6 + 设计指南 5 + AI辅助文档 4 + 任务定义 6）
- 产品类型为**方法论/工具产品**，PA 分类为"规范与准则/设计指南/AI辅助文档/任务定义"

本次增量在基线基础上，新增"设计基线管理规范"能力，使流水线中准则、指南、AI辅助文档和任务定义支持设计基线的创建、差异对比和历史追溯。

## 一、第1步产出范例

### 1.1 新需求收集与分类

```
OR-006: 设计基线管理规范
├─ 需求描述：系统设计流水线应提供一套完整的设计基线管理规范和方法指引，
│           定义设计基线的概念、创建规则、差异对比方法以及历史追溯机制，
│           确保设计迭代过程中的版本可追溯和可回退
├─ 原始碎片清单：
│  ├─ "每次版本迭代完成后需要打基线标签，确保评审过的设计不被意外修改" — 流水线开发者 2026-05-12
│  ├─ "迭代多了之后，想比较当前设计和以前某个基线的差异" — EOS架构师 2026-05-13
│  └─ "基线作为设计过程的里程碑，应该在准则和指南中明确何时创建、怎么创建" — 系统设计者 2026-05-14
├─ 包含的性能指标：基线创建时间 < 5 分钟（编写者完成一次基线标记），差异对比 < 10 秒（AI辅助）
├─ 需求类型：功能需求
├─ 变更类型：新增
├─ 与现有需求的关系：与 OR-004（设计任务操作规程）相关，基线管理可作为任务定义中的质量检查点
└─ 依赖/冲突关系：依赖 OR-001（准则体系）和 OR-002（设计指南），基线规范需引用准则中的版本概念
```
