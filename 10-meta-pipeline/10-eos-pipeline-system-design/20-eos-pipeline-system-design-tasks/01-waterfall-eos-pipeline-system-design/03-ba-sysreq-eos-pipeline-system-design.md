# task-03：系统需求功能部分架构定义

**对应场景**：01-waterfall · 第3步
**对应 AI 模板**：`03-eos-pipeline-system-design-ai-support/01-waterfall-eos-pipeline-system-design/workflow-prompts.md` 第3步

---

## 目标

将 BA IPO 去重后按 0-9 级层级结构组织，形成系统需求功能部分架构定义。

## 上下文继承

```
从哪里来：
  - 第2步产出：业务架构定义（IPO 清单）
  - 第2步同步建立的系统需求架构骨架
  - 涉及规范：系统需求规范 §2.1、设计准则 §3.1 第3步

去哪里：
  - 本步产出 → task-05（SysReq 详细定义 + PA 定义）的 5 级节点输入
  - 同步产出 → 产品架构骨架（供 task-05 完善）

并行说明：
  - 本步（功能分支）与 task-04（非功能分支）完全独立并行
```

## 输入

| 输入 | 说明 |
|------|------|
| 业务架构定义（IPO 清单） | task-02 产出 |
| SysReq 架构骨架 | task-02 同步建立 |

## 操作流程

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | IPO 最终去重确认 | AI | 三级去重：检查→复用→新增（跨所有 BA 流程检查重复 IPO） |
| 2 | 分析活动：架构定义验证 | AI | 需求说明符合性分析（覆盖度/一致性/追溯）+ 需求性能符合性分析（性能继承/可行性/指标追溯） |
| 3 | 功能需求架构组织 | AI | 按 0-9 级层级结构组织（5 级承接 BA IPO） |
| 4 | 编写符合性分析和设计指导 | AI | 每个 5 级节点包含定义、承接来源、符合性分析、设计指导 |
| 5 | 同步建立 PA 骨架 | AI | 为每个 SysReq 5 级节点创建 PA 组件占位符 |
| 6 | 结果审核 | 人类 | 确认架构分层合理性，确认去重决策 |

## 人类审核要点

- [ ] 0-9 级分类是否合理
- [ ] IPO 去重后的语义与 BA 原始描述是否一致
- [ ] N:1 承接数量是否合理（每个 5 级节点承接 2-5 条 BA IPO）
- [ ] PA 骨架的组件划分粒度是否合适
- [ ] 分析活动报告中的覆盖度和追溯验证通过

## 质量门禁

- [ ] IPO 去重完成率 100%（三级策略执行完毕）
- [ ] 分析活动已完成（覆盖度和追溯验证通过）
- [ ] 去重后 IPO 映射完整，N:1 承接数量合理
- [ ] 功能需求架构定义完整（树形结构 + 映射关系表）
- [ ] 每个 5 级节点包含符合性分析和设计指导
- [ ] PA 骨架已同步建立
- [ ] 映射关系表已建立（BA→SysReq）
- [ ] 运行 rule-conformance-review 进行符合性审查，按审查修复流程处理问题
