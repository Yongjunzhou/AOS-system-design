# 瀑布式系统设计：提示词模板集
**Waterfall System Design: Workflow Prompt Templates**

**用途**：AI 按 6 步流程依次执行，每个步骤使用对应的提示词模板。AI 将提示词中的 `[占位符]` 替换为实际内容后执行。

**依据B说明**：本提示词模板是设计线（运营体系系统设计流水线）的"AI辅助文档"构件，是设计线产品架构（PA）的末级节点。模板中的提示词指令（分解方法、映射规则、验证标准）为方法论层面的通用操作，与目标产品类型无关。

AI使用本模板时，应根据目标产品类型选择对应的PA分类体系：
- **软件产品类型**（如EOS·EOS运营体系）：PA分类为`前端/后端/数据/基础设施`
- **方法论/工具产品类型**（如设计线·流水线）：PA分类为`规范与准则/设计指南/AI辅助文档/任务定义`

模板中的示例默认使用设计线的瀑布式场景进行演示，输出实例是设计线的另一个PA末级节点（即面向目标产品的AI辅助文档自定义实例）。为目标产品生成设计文档时，示例和内容应引用目标产品的实际领域。

## 跨步骤上下文管理说明

### 上下文传递方式

6步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策，包括：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

### 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**中，供后续步骤引用。

### 当需要回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见[AI决策框架 -> 跨步骤影响范围表](../02-ai-decision-framework-eos-pipeline.md#72-跨步骤影响范围表)）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

## 第1步：原始需求分析及相关方需求架构定义

### 目标
从原始需求（会议纪要、邮件、文档等原始表述）出发，产出原始需求详细定义和相关方需求架构定义。

### 上下文继承清单

```
从哪里来：
  - 直接输入：原始需求原始表述（作为本步骤的外部输入）
  - 参考文档：[产品上下文](../01-product-context-eos-pipeline.md)、[AI决策框架](../02-ai-decision-framework-eos-pipeline.md)
  - 涉及规范：原始需求规范 §4.1、相关方需求规范 §2、设计准则 §3.1 第1步

预期产出：
  1. 原始需求文档（含末级需求分解）
  2. 相关方需求架构定义文档（树形结构 + 映射关系表）
  3. 冲突和重复检测报告

本步决策对后续的影响：
  - 原始需求的末级分解直接影响第2步 SR 详细定义的范围
  - SR 架构的划分方式决定后续步骤的分支路径
  - 编号体系影响全链路的追溯能力
```

### 输入
- 原始需求原始表述（作为本步骤的输入提供）

### 涉及规范
- [原始需求规范 §4.1 完整处理流程](../../01-eos-pipeline-system-design-specification/02-eos-pipeline-original-requirements-specification.md#41-完整处理流程)
- [相关方需求规范 §2 架构定义规范](../../01-eos-pipeline-system-design-specification/03-eos-pipeline-stakeholder-requirements-specification.md#二架构定义规范)
- [系统设计准则 §3.1 第1步](../../01-eos-pipeline-system-design-specification/01-eos-pipeline-system-design-standards.md#第1步原始需求分析及相关方需求架构定义)

### 决策框架参考

本步涉及的决策类型及处理方法：

| 决策场景 | 处理方法 | 参考章节 |
|---------|---------|---------|

| 需求歧义处理 | 分级处理（L1补齐/L2选项/L3上报） | [需求歧义处理策略](../02-ai-decision-framework-eos-pipeline.md#四需求歧义处理策略) |

### 提示词模板

#### 1a. 原始需求收集和初步整理

```
请对以下原始需求进行初步整理。原始需求来自多个渠道，粒度参差不齐。

原始需求原始表述：
[在此处粘贴原始需求原文]

请执行：
1. 血缘归并 — 将描述同一功能不同切面的碎片合并，还原完整表述
2. 角色聚合 — 不同角色对同一功能的诉求合并为一条，标注角色差异
3. 流程串联 — 同一流程不同步骤的碎片按操作顺序串联
4. 父子补全 — 以主需求为载体，补充细节作为子约束

遵循五条规则：
- 不做过度合并（合并后能用一句话说清核心功能）
- 保留原始出处（记录提出方、原始表述、日期）
- 性能指标不合并（功能可合并，指标单独提取）
- 矛盾碎片标注不合并（标注"冲突待确认"）
- 无法判断的标记为"待补全"

输出格式：
原始需求 [OR-XXX]
├─ 需求标题：[一句话概括]
├─ 需求描述：[合并后的完整需求描述]
├─ 原始碎片清单：
│  ├─ [碎片1原始表述] — [提出方] [日期]
│  └─ [碎片2原始表述] — [提出方] [日期]
├─ 包含的性能指标：[如有，单独列出]
├─ 依赖/冲突关系：[如有，标注]
└─ 整理说明：[归并依据]
```

#### 1b. 原始需求规范化和四项检查

```
请对初步整理后的原始需求进行规范化处理。

原始需求清单：
[粘贴1a的输出]

请执行：
1. 完整性检查 — 每条需求是否有清晰的描述、类型、优先级？
2. 清晰性检查 — 模糊术语替换为精确表述（"实时"→"延迟<100ms"，"快速"→"<2s"）
3. 可测试性检查 — 功能需求：输入/处理/输出是否明确？非功能需求：指标/条件/方法是否明确？
4. 独立性检查 — 需求是否相对独立？是否与其他需求过度耦合？
5. 功能/非功能分离 — 将"做什么"（功能）和"有多好"（非功能）分开记录

⚠️ 遇到模糊术语时按[决策框架L1补齐规则](../02-ai-decision-framework-eos-pipeline.md#42-l1-级别可推测的补齐规则)处理：
  - 可推测（>80% confidence）→ 替换为精确表述，标注"AI推断"
  - 多个合理选项 → 列出选项，标注"待选择"

输出格式：
OR-XXX: [需求标题]
├─ 规范化描述：[经过四项检查后的描述]
├─ 类型：[功能需求/非功能需求]
├─ 优先级：[P0/P1/P2/P3] ⚠️ 待确认
├─ 验收标准：[可量化的验收条件]
└─ 性能指标：[如有，具体数值和条件]
```

#### 1c. 原始需求分解到末级

```
请将规范化后的原始需求分解到末级。

规范化原始需求清单：
[粘贴1b的输出]

分解原则：
1. 原子性原则 — 每条末级不可再分
2. 1:1映射原则 — 每条末级只需映射到一个SR架构末级节点
3. 语义完整性 — 分解后不丢失原始语义，所有末级之和=原始需求
4. 粒度一致性 — 同层末级粒度相当

触发分解的条件：
- 包含"和/并且/同时"连接多个功能 → 分解
- 涉及多个角色或相关方 → 分解
- 需要映射到多个SR末级 → 分解

⚠️ 粒度判断参考[决策框架 2.1 原始需求→末级的分解](../02-ai-decision-framework-eos-pipeline.md#21-原始需求--末级的分解)：
  - 先做1:1约束测试：能否用一个下层末级承接？
  - 再做粒度一致性检查：同层末级篇幅是否相似？

停止准则：问自己"这个条目能映射到相关方需求架构的一个末级节点吗？"
- 只需一个节点承接 → 停止
- 需要多个节点承接 → 继续分解

输出格式：
原始需求: OR-XXX（[需求标题]）
末级分解:
  - ID: OR-XXX.X
    描述: [原子性功能描述]
    分配到: SR-F/SR-NFR-[待定]
    验收标准: [可量化条件]
```

#### 1d. 冲突和重复检测

```
请对末级需求清单进行冲突和重复检测。

末级需求清单：
[粘贴1c的输出]

冲突检测：
- 直接冲突：语义直接矛盾，无法同时满足 → 标记"需确认"
- 间接冲突：同时实现可能行为不一致 → 标记"需调整"
- 可合并冲突：可通过调整合并解决 → 标记"建议合并"

重复检测：
- 完全重复：两条完全相同 → 去重
- 部分重复：功能范围部分重叠 → 建议合并
- 相似重复：表达不同但核心相同 → 确认是否为同一需求

⚠️ 遇到无法处理的冲突或重复时：
  - 如果是信息不足 → 标记"需澄清"，等待人工
  - 如果有多个合理处理方式 → 列出选项+推荐，标注"待选择"

输出格式：
| 需求A | 需求B | 问题类型 | 描述 | 建议 |
|-------|-------|---------|------|------|
| OR-XXX | OR-YYY | 直接冲突 | [描述] | 需确认 ⚠️ |
```

> **自治分析完成**：至此，需求侧的自治分析（可复用性检查：1d 重复检测 → 冲突性检查：1d 冲突检测 → 规范性检查：1b 规范化处理）已全部完成。以下 1e 开始进入下游方案资产审视，按照**复用→改进→新增**的优先级为每条需求末级寻找承接节点。此顺序确保需求定义的质量不被已有架构结构扭曲。参见[决策框架：工作流顺序](../02-ai-decision-framework-eos-pipeline.md#3-工作流顺序先自治分析再审视资产)。

#### 1e. 相关方需求架构定义

```
请基于末级需求清单，建立相关方需求架构定义。

末级需求清单： [粘贴1c的输出]
冲突/重复检测结果： [粘贴1d的输出]

请执行：
1. 分析末级需求的共性和关联性
2. 按功能需求（SR-F）和非功能需求（SR-NFR）分别分类
3. 为每组定义架构末级节点
4. 编写符合性分析（如何满足所承接的需求）
5. 编写设计指导（对业务架构的指导）
6. 建立映射关系表

⚠️ [决策框架 N:1承接判断](../02-ai-decision-framework-eos-pipeline.md#31-n1-承接数触发的判断)：
  每个SR节点承接 2-5 条 OR 末级为合理
  - 1条：检查可否合并
  - 6+条：建议拆分

架构定义文档包含两个部分：

第一部分：架构设计结果（树形结构）
相关方需求（0级·根）
├─ 功能需求（1级·分类）
│  └─ [分类名称]（2级·分类）
│     └─ SR-F-XXX: [名称]（架构末级节点）
│          ├─ 定义：[需求的定义]
│          ├─ 承接的原始需求末级：OR-XXX.X, OR-YYY.Y
│          ├─ 符合性分析：[如何满足]
│          ├─ 设计指导：[对业务架构的指导]
│          └─ 性能指标：[继承自原始需求]
│
└─ 非功能需求（1级·分类）
   └─ SR-NFR-XXX: [名称]（架构末级节点）
        ├─ 定义：[非功能需求的定义]
        ├─ 需求类型：[性能/安全/可用性/可维护性等]
        ├─ 需求类别：[质量特性/环境适应性/可实现性]
        ├─ 承接的原始需求末级：[OR-XXX]
        ├─ 符合性分析：[如何满足所承接的非功能需求]
        ├─ 量化指标：[具体可测量的指标，含数值+单位+条件]
        └─ 测量方法：[如何测试和验证]

第二部分：需求映射视图（映射关系表）
| 原始需求末级 | 承接的SR架构末级 | 关联说明 | 超链接 |
|-------------|-----------------|--------|--------|
| OR-001.1    | SR-F-XXX        | [说明]  | [→]    |
```

### 预期产出
- 原始需求文档（含末级需求分解）
- 相关方需求架构定义文档（树形结构 + 映射关系表）
- 冲突和重复检测报告

### 本步决策记录

```
本步关键决策记录：
├─ 合并/拆分决策：[哪些 OR 末级被合并/拆分，依据]
│  └─ SR节点承接数：[每个SR节点承接了几条OR末级，是否有合并/拆分]
├─ 编号体系：[OR-XXX ~ OR-YYY]
├─ ⚠️ 待确认项：[汇总需要人工确认的项]
├─ AI推断记录：[AI自行判断补齐的模糊术语/缺失内容]
├─ 待传递的约束：[后续步骤需要注意的决策/约束]
└─ 执行模式：[全自动/半自动]
```

### 提示词模板

#### 2a. 分析活动：架构定义验证（在详细定义开始前执行）

```
请在开始详细定义前，对第1步产出的相关方需求架构定义执行分析活动。

相关方需求架构定义（功能部分）：
[粘贴1e的输出中SR-F部分]

需求说明符合性分析：
1. 检查每个SR-F架构末级节点是否充分满足所承接的原始需求末级

3. 验证功能与非功能需求的分离正确性

需求性能符合性分析：
1. 验证性能指标是否从原始需求正确继承和量化

输出：分析活动报告
├─ 映射完整性：[通过/未通过，如有遗漏列出]
├─ 性能指标量化：[通过/未通过，如有问题列出]
└─ ⚠️ 待确认项：[如有]
```

## 并行说明

在瀑布式6步流程中，以下分支可以**并行执行**：

| 并行组 | 步骤 | 说明 |
|--------|------|------|
| **功能分支** | 第2步 + 第3步 | 功能分支连续执行（第2步定义SR详细定义+BA IPO，第3步定义SysReq功能架构） |
| **非功能分支** | 第4步 | 独立于功能分支，与第2-3步完全并行 |
| **合并** | 第5步 | 等待功能分支和非功能分支都完成后，合并执行 |

因此实际时间线为：
```
第1步 → 第2+3步（功能） + 第4步（非功能）→ 第5步 → 第6步
```

## 审查修复流程

### 审查输入
AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

### AI 修复流程
收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）
   → 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）
   → 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）
   → 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题**
   → 参考[AI决策框架 -> 跨步骤影响范围表](../02-ai-decision-framework-eos-pipeline.md#72-跨步骤影响范围表)进行影响范围评估，列出所有受影响的下游节点

### 影响范围评估方法
