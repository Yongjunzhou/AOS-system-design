## 一、EOS 流水线的产品语境

### 1.1 EOS 流水线产品概述

#### 产品类型

EOS 流水线是**知识/方法类产品**。其 PA 构件（交付物）的形态以文档和 AI Skill 定义为主，而非软件代码或实物组件。

#### 四个组成部分

EOS 流水线由四条子流水线组成：

| 子流水线 | 简称 | 说明 |
|---------|------|------|
| 系统设计流水线 | **设计线** | 定义如何从原始需求逐层推导至产品架构的方法和工具。当前已完成 |
| 组件开发流水线 | **开发线** | 定义如何根据产品架构对每个构件进行开发的方法和工具。待开发 |
| 系统集成流水线 | **集成线** | 定义如何沿产品架构的树形层级逐级集成形成完整产品的方法和工具。待开发 |
| 系统运维流水线 | **运维线** | 定义产品交付后的持续运行、监控、维护和优化的方法和工具。待开发 |

#### 产品开发四阶段

EOS 流水线本身作为一个产品，其开发也遵循四阶段框架：

| 阶段 | 名称 | 说明 |
|------|------|------|
| 阶段1 | **系统设计** | 从原始需求出发，经过五层结构（OR → SR → BA → SysReq → PA），开发出 EOS 流水线的产品架构，定义每个构件的需求和实现策略 |
| 阶段2 | **构件开发** | 对产品架构中所有需要新开发或改进的构件进行开发 |
| 阶段3 | **系统集成与交付** | 沿产品架构的树形层级逐级集成为完整的 EOS 流水线 |
| 阶段4 | **系统运维** | 交付后的持续运行、监控、维护和优化 |

**本准则的范围**：严格限定在**阶段1（系统设计）**。

#### 流水线工具组件

EOS 流水线的系统设计产出——即其产品架构的架构末级节点（构件）——包含以下四类交付物，它们构成 EOS 团队在系统设计阶段使用的工具集：

| 构件类型 | 交付物形态 | 说明 |
|---------|-----------|------|
| **设计准则** | 规范文档 | 定义系统设计的通用方法论和规则 |
| **设计指南** | 指南文档 | 为不同设计场景（瀑布式/敏捷式等）提供操作指导 |
| **AI 辅助文档** | 工作流/检查清单/示例 | 辅助 AI 参与系统设计过程 |
| **任务定义** | 操作规程文档 | 定义六步法各步骤的输入输出和操作流程 |

这四类构件的位置见 §2.1.2 的目录对应表。

### 1.3 设计场景

EOS 流水线的系统设计可按以下四种设计方法场景推进。设计方向取决于流水线产品当前所处的阶段：

| 方法 | 设计方向 | 起点 | 终点 |
|------|---------|------|------|
| **瀑布式** | 正向完整：OR → SR → BA → SysReq → PA | 原始需求 | 产品架构 |
| **敏捷式** | 正向增量：分析影响 → 增量更新 | 现有系统设计 | 更新的系统设计 |
| **逆向工程** | 反向推导：PA 及构件 → 反向推导至 OR | 已有构件 | 完整系统设计 |
| **DevOps** | 最小正向变更：定位 → 修复 → 验证 | 生产环境问题 | 快速修复/发布 |

**选择决策**：

```
EOS 流水线处于什么阶段？

┌─ 已开发完成（如设计线已存在），但文档不全？
│  └─ 方法：逆向工程
│     └─ 从已有设计线构件反向推导出完整系统设计文档
│
├─ 只有原始需求，需要从零开始（如开发线/集成线/运维线的新建）？
│  └─ 方法：瀑布式
│     └─ 逐层正向推导（OR → SR → BA → SysReq → PA）
│
├─ 已有完整系统设计，需要增加新构件或修改现有构件？
│  └─ 方法：敏捷式
│     └─ 分析影响范围 → 增量更新受影响的文档
│
└─ 流水线已上线运行（如 AI 辅助流程需要快速修复）？
   └─ 方法：DevOps
      └─ 快速评估影响 → 最小化变更 → 快速验证
```

**各方法风险特征**：

| 方法 | 主要风险 | 应对策略 |
|------|---------|---------|
| **瀑布式** | 需求不清晰导致后期返工 | 在 OR 层做充分的需求分析和冲突检测 |
| **敏捷式** | 增量范围扩散超出迭代边界 | 严格控制每次迭代的影响范围分析 |
| **逆向工程** | 反向推导中对原始意图的理解偏差 | 多方确认推导结果，保留推导依据 |
| **DevOps** | 快速修复引入副作用 | 最小化变更范围，变更前后做回归验证 |

## 二、EOS 流水线的特有设计框架

本章定义 EOS 流水线系统设计的核心框架——这些内容是通用准则中未覆盖的、EOS 流水线产品特有的设计规则。

### 2.1 多产品设计关系

当多个产品使用同一套五层设计方法论进行开发时，产品之间会产生结构性关系。本节定义 EOS 流水线与其他产品（通用方法论、EOS）之间的关系框架。

#### 2.1.1 三层工作层面（核心概念）

EOS 流水线和 EOS（企业运营体系）之间的关系由**三个工作层面**构成：

| 层面 | 内容 | 负责方 | 仓库位置 |
|------|------|--------|---------|
| **第1层：流水线设计** | EOS 流水线用通用方法论对其自身需求进行系统设计，推导出 EOS 流水线的产品架构 | 流水线开发者 | `10-meta-pipeline/` |
| **第2层：交付物构建** | 将 EOS 流水线的系统设计产物（准则、指南、AI 辅助、任务定义）适配为 EOS 设计工具 | 流水线开发者 | `20-eos-design/.../01~04-*` |
| **第3层：使用工具设计 EOS** | EOS 开发者使用第2层产出的工具对 EOS 进行系统设计，产出 EOS 的产品数据 | EOS 开发者 | `20-eos-design/.../10-product-data/` |

**内嵌关系**：运营体系涵盖市场、研发、生产、售后等业务，同时也包含"设计运营体系本身"这项业务。EOS 流水线就是支撑这项业务的方法和工具集。EOS 流水线的四类构件被放置在 `20-eos-design/10-eos-system-design/` 目录中，作为 EOS 开发者设计 EOS 时的操作规程。

```
运营体系
├── 市场
├── 研发
├── 生产
├── 售后
└── 设计运营体系本身 ← EOS 流水线提供方法支撑
      ├── 系统设计方法论
      └── 设计准则、指南、AI辅助、任务定义（EOS 流水线的 PA 构件）
```

> **EOS 流水线开发者视角**：你的工作对应第1层和第2层——先用通用方法论对 EOS 流水线进行系统设计（第1层），再将设计产物适配为 EOS 团队可用的工具（第2层）。

### 2.2 PA 分解三因素

**问题**：业务架构的多个 IPO 通过系统需求到达产品架构时——哪些 IPO 应合并到同一个 PA 构件（共享系统），哪些应分属不同的 PA 构件（专用系统）？

**这不是技术选择，而是业务信息属性的传递**。三个决定因素如下。

#### 2.2.1 因素一：信息模型内聚性

两个业务活动如果**操作同一组核心实体**（共享主数据、流程天然连贯），倾向合并为同一构件：

```
强内聚 → 可共享：
  客户信息 ←→ 销售订单 ←→ 售后服务
  → 共享客户主数据，流程天然连贯

弱内聚 → 倾向分离：
  设计 BOM ←→ 生产 BOM ←→ 维修 BOM
  → 虽都叫"BOM"，管理目的、变更节奏、精度要求不同
  → 强行合并导致数据模型被最低公分母拖累
```

**原则**：操作同一核心实体 → 合并；实体不同但有关联 → 通过接口集成。

#### 2.2.2 因素二：NFR Profile 相容性

不同业务活动对非功能需求的要求可能截然不同。当 NFR Profile 冲突时，不应共享同一构件：

| 维度 | 生产执行（MES） | 经营分析（BI） |
|------|---------------|---------------|
| 响应时间 | 毫秒级（产线节拍） | 分钟级可接受 |
| 可用性 | 99.99%（停机即停产） | 99.9% |
| 数据量 | 实时高频小数据 | 批量大吞吐 |
| 安全要求 | 操作权限精细 | 只读+脱敏 |

这两个活动的 NFR Profile 冲突——一个要求实时低延迟，一个要求批量大吞吐。强行合并会导致两头不靠。

> **NFR 走平行路径（SR-NFR → SysReq-NFR → PA）的根本意义在此**：NFR 差异本身就是 PA 分解的依据，而非仅在 SysReq 层做约束分析。

#### 2.2.3 因素三：信息处理模式一致性

不同业务活动消费和生产信息的方式存在本质差异（详见 §2.4 信息处理模式十三类）。以此作为 PA 分解的判断依据：

| IPO 组合 | 可否共享？ | 原因 |
|---------|-----------|------|
| 同模式（前9类） | ✅ 倾向共享 | 跨领域通用，NFR 通常相容 |
| 同模式（后4类）+ 同领域 | ✅ 倾向共享 | 同一专业领域 |
| 同模式（后4类）+ 不同领域 | ❌ 分离 | 领域不同则数据模型不同 |
| 不同模式 | ❌ 倾向分离 | 信息处理逻辑不同 |

### 2.3 三因素决策矩阵

| 信息模型 | NFR Profile | 信息模式 | 结论 |
|---------|------------|---------|------|
| 相同 | 相容 | 相同 | ✅ **合并**为同一构件 |
| 相同 | 冲突 | — | ❌ **分离**（NFR 优先） |
| 不同 | 相容 | 相同 | ⚠️ 看耦合度，接口集成 |
| 不同 | 冲突 | 不同 | ❌ **必然分离** |

> **核心认知**：PA 层的构件分解是对 BA 层业务边界的读取而非发明。架构设计的质量取决于能否读出这些自然边界，而非用人为的"统一平台"去抹平它们。

### 2.4 信息处理模式十三类

企业运营体系包含多种业务活动，它们的本质差异在于处理信息的方式不同。这些差异不是技术选择，而是业务自身的信息属性决定的。完整定义见[通用准则 §2.5.2](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#252-业务活动的信息处理模式)，以下为摘要：

**前9类——跨领域通用模式**：

| 模式 | 核心特征 | 典型系统 |
|------|---------|---------|
| **计划事务** | 基于约束条件生成资源/时间分配方案 | 计划系统 |
| **工单事务** | 创建→派发→执行→反馈→关闭的闭环处理 | 工单系统 |
| **审批流程** | 按规则链路由、汇集决策意见、形成审批结论 | 流程引擎 |
| **合规事务** | 对照规则库检查行为/数据、判定合规性、记录证据链 | 合规系统 |
| **规范知识** | 结构化编撰准则/规程/指南，规范人员操作或 AI 执行 | 文档/AI 辅助系统 |
| **指标看板** | 从多源数据聚合指标并以可视化形式呈现 | 看板/BI 系统 |
| **进展追踪** | 跟踪状态变更、比对计划与实际、标识偏差 | 追踪系统 |
| **信息台账** | 对信息进行登记、维护、检索和状态管理 | 台账系统 |
| **xBOM** | 管理产品结构及配置关系，追踪物料清单的版本和变更 | BOM 管理系统 |

**后4类——领域特有模式**（不同领域有各自的分析方法、建模方式、专用算法和设备 IO 模式）：

| 模式 | 核心特征 | 典型系统 |
|------|---------|---------|
| **设备 IO** | 与物理设备进行实时数据交换，采集和控制 IO 信号 | 设备控制/SCADA |
| **领域分析** | 特定领域公式/模型驱动的计算分析 | 专业分析工具 |
| **领域建模** | 构建领域模型、运行仿真场景、输出预测结果 | 建模仿真平台 |
| **领域算法** | 封装特定领域逻辑的算法执行 | 算法引擎 |

**核心规律**：判断两个业务活动是否共享系统——不同模式→分离；同模式→可共享。后4类即使模式相同，如果领域不同，仍然需要分离。

## 三、通用方法论精要

本章以极简篇幅摘要列出通用准则的核心方法论，并标注其在 EOS 流水线语境下的关注点。完整定义请直接查阅通用准则对应章节。

> **重要**：以下各节均非通用准则的替代品，而是帮助你快速定位到所需的通用方法内容。初次接触 EOS 流水线的系统设计前，**建议先通读通用准则 §一~§七**。

### 3.1 五层结构（通用 §四）

EOS 流水线的系统设计遵循标准的五层设计链路，各层在流水线语境下的关注点如下：

| 层级 | 在 EOS 流水线中的关注点 |
|------|----------------------|
| **OR → SR** | 流水线的原始需求从哪些渠道来（会议纪要、邮件、已有准则）、如何规范化为相关方需求 |
| **SR → BA** | 从流水线的功能需求中识别业务活动边界，定义 IPO |
| **BA → SysReq** | 将 IPO 去重后按系统功能分类重组——注意四类构件（准则/指南/AI辅助/任务定义）的功能差异 |
| **SysReq → PA** | 按 PA 分解三因素（§2.2）进行构件划分，得到四类构件的完整清单 |

> 完整五层结构定义见[通用准则 §四](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#四系统设计链路)，三层核心规则见[通用准则 §三](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#三文档结对设计)。

### 3.2 文档结对设计（通用 §三）

EOS 流水线的系统设计过程包含四个依次衔接的文档结对：

| 序号 | 结对 | 在 EOS 流水线中的映射 |
|------|------|---------------------|
| 1 | OR 条目级需求 ↔ SR 架构定义 | 流水线原始需求 → 相关方需求架构 |
| 2 | SR 详细定义末级 ↔ BA 架构（IPO） | 相关方需求 → 业务组件（IPO清单） |
| 3 | BA IPO（去重后）↔ SysReq 架构定义 | IPO → 系统功能模块 |
| 4 | SysReq 详细定义末级 ↔ PA 架构定义 | 系统功能 → 四类构件 |

**结对设计原则**：上层详细定义与下层架构定义应同步设计而非串行执行——先快速约定下层架构的结构骨架，再并行填充内容。

> 需求分解/分配/分析三个核心设计活动的完整操作规则见[通用准则 §三](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#三文档结对设计)。编号规则见[通用准则 §3.4](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#34-编号规则)。

### 3.3 组织资产优先（通用 §2.4）

EOS 流水线各层的架构节点不是凭空产生的，而是基于组织现有资产按 **复用 → 改进 → 新增** 优先级生成。

在 EOS 流水线语境下，各层的资产形态：

| 层级 | 资产形态 |
|------|---------|
| **SR 层** | 需求分类体系、领域知识结构、行业标准目录 |
| **BA 层** | 已有业务过程、岗位职责分工、跨系统交互流程 |
| **SysReq 层** | 系统边界划分、接口协议、技术域约定 |
| **PA 层** | 现有四类构件（准则/指南/AI辅助/任务定义）的定义和结构 |

> 资产优先级决策规则及操作细则见[通用准则 §2.4](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#24-基于资产定义架构)。

### 3.4 验收标准与检查清单（通用 §六、§七）

通用准则定义了六个维度的质量验收标准（§六）和对应的逐项检查方法（§七），是 EOS 流水线系统设计质量评估的通用框架。**本文档 §五 仅补充 EOS 流水线特有的验收项，通用项不再重复**。

> 完整验收标准见[通用准则 §六](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#六设计验收标准)，完整检查清单见[通用准则 §七](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#七设计检查清单)。

## 四、各层设计指引

本章为 EOS 流水线的五层设计提供场景化指导。每层聚焦"在 EOS 流水线语境下如何进行设计活动"，通用方法步骤不再复述。

### 4.1 第1层 OR：原始需求及分析

**EOS 流水线的 OR 特征**：EOS 流水线的原始需求主要来自以下渠道：
- 流水线开发者的设计目标和构想
- 对已有设计线构件的逆向分析需求
- EOS 开发者的使用反馈和改进建议
- 通用方法论版本更新触发的适配需求

**设计指引**：

1. **收集**阶段重点关注四条子流水线的需求覆盖面——当前仅设计线已完成，开发线、集成线、运维线处于待启动状态，OR 应覆盖所有四条线的需求
2. **初步整理**时区分"新建子线"和"改进现有构件"两类需求——前者需要完整五层设计，后者走敏捷或 DevOps 路径
3. **规范化**时关注流水线特有的术语（如"四类构件""设计场景""AI 辅助"等），确保术语一致
4. **分解到末级**时遵循 1:1 约束——每条子需求须能分配到 SR 唯一一个架构末级节点

**输出**：原始需求详细定义（末级条目清单）

> 通用方法：OR 预处理（碎片整理/规范化/冲突检测）见[通用准则 §5.2 阶段一](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#52-第1步or--sr)。

### 4.2 第2层 SR：相关方需求

**SR 架构分类**：EOS 流水线的相关方需求按功能和非功能两个分支组织。

**功能部分**的典型分类：

```
EOS 流水线相关方需求（功能部分）
├── 设计线功能
│   ├── 准则管理功能
│   ├── 指南管理功能
│   ├── AI 辅助功能
│   └── 任务定义功能
├── 开发线功能（待设计）
├── 集成线功能（待设计）
└── 运维线功能（待设计）
```

**非功能部分**按[通用准则附录B](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#附录b-非功能需求分类体系)的三大类（质量特性/环境适应性/可实现性）组织。

**设计指引**：

1. 功能需求和非功能需求分别标识（`SR-F-` / `SR-NFR-`）
2. 每条 SR 架构末级节点承接 2-5 条 OR 末级条目（N:1），检查合并的合理性
3. 详细定义分解到每条末级条目只需映射到 BA 的一个 IPO

> 通用方法：SR 详细定义分解和 IPO 映射见[通用准则 §5.3](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#53-第2步sr功能--ba)。

### 4.3 第3层 BA：业务架构

**EOS 流水线的 BA 层级结构**（0-5级）：

| 层级 | 名称 | 说明 |
|------|------|------|
| 0级 | 架构根节点 | EOS 流水线（唯一） |
| 1级 | 业务域 | 设计业务域、开发业务域、集成业务域、运维业务域 |
| 2级 | 端到端项目 | 各子线的端到端开发项目 |
| 3级 | 项目业务对象架构 | 由 EOS 流水线的 PA 构件类型（§1.1 四类构件）推导得出 |
| 4级 | 工作包（WBS） | 对各构件的开发工作分解 |
| **5级** | **业务组件（承接层）** | **末级——使用 IPO 模型完整定义** |

**IPO 识别方法**：从 SR 详细定义集合中识别业务活动边界。例如，从 SR 中的"准则开发"功能需求推导出 BA IPO：

```
BA-001: 设计准则开发（IPO名称）
├─ Input: 通用方法论核心内容 + EOS 流水线设计约束
├─ Process: 按准则结构编制 → 审查 → 修订 → 发布
└─ Output: 设计准则文档（v1.0）
```

**IPO 去重**：同一 IPO 可能被多条 SR 详细定义映射——将重复出现的 IPO 归并为唯一条目，形成去重后的 IPO 清单。

> 通用方法：BA 的独立存在特性和 IPO 映射规则见[通用准则 §5.3](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#53-第2步sr功能--ba)。

### 4.4 第4层 SysReq：系统需求

**双重来源**：

| 来源 | 说明 | 处理方式 |
|------|------|---------|
| **功能需求** | BA IPO 去重清单 | IPO 去重后按四类构件功能分类组织，形成 0-9 级架构 |
| **非功能需求** | SR-NFR（平行路径，不经过 BA） | 按质量特性/环境适应性/可实现性三大类组织，形成 0-6 级架构 |

**功能部分 0-9 级分解链**（以设计线为例）：

```
5级：组件实例 ←── 承接 BA 5级（如"设计准则开发"IPO）
6级：功能模块（如"准则版本管理"）
7级：系统用例（如"创建新准则版本"）
8级：用例场景（正常/异常/替代）
9级：场景活动（如"填写准则元数据"）──→ 分配到 PA 末级（1:1）
```

**设计指引**：

1. 功能部分从 BA IPO 去重清单出发，按工具类型（准则/指南/AI辅助/任务定义）组织系统功能分类
2. 非功能部分直接从 SR-NFR 承接，不经过 BA 层
3. 每条 9 级场景活动保证 1:1 分配到唯一的 PA 构件
4. NFR 在 SysReq 层的处理关注：可用性、可维护性、版本兼容性对工具类产品的影响

> 通用方法：SysReq 的完整分解过程和功能/非功能分配路径见[通用准则 §5.4~§5.5](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#54-第3步ba--sysreq功能架构)。

### 4.5 第5层 PA：产品架构

**EOS 流水线的四类构件**（知识/方法类产品的 PA 构件）：

| 类型 | 交付物形态 | 说明 |
|------|-----------|------|
| **设计准则** | 规范文档 | 定义 EOS 流水线的设计方法论和规则 |
| **设计指南** | 指南文档 | 为四种设计场景提供操作指导 |
| **AI 辅助文档** | 工作流/检查清单/示例 | 辅助 AI 参与 EOS 流水线的设计和开发 |
| **任务定义** | 操作规程文档 | 定义六步法各步骤的输入输出和操作流程 |

**构件划分原则**：每个构件应有明确的职责边界。对于 EOS 流水线（知识/方法类产品），粒度标准为"可由单人独立完成"且"可独立审查和验收"。

**PA 分解方法**：

1. 从 SysReq 9 级场景活动出发，按 §2.2 的 PA 分解三因素判断哪些活动应合并为同一构件
2. 按复用→改进→新增的资产优先顺序（[通用 §2.4](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#24-基于资产定义架构)）处理已有构件
3. 每个构件标注四要素：定义、所映射的 SysReq 末级、符合性分析、实现策略

**举例**：设计线 PA 构件的划分（四类各一个示例节点）：

```
PA-001: EOS 系统设计准则（设计准则构件）
├─ 定义：定义 EOS 系统设计的通用方法论和规则
├─ 交付物形态：规范文档
├─ 承接的 SysReq 末级：SysReq-001.01 ~ SysReq-001.05
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：参考通用准则结构，按 EOS 场景定制

PA-002: 瀑布式场景设计指南（设计指南构件）
├─ 定义：为瀑布式场景下的 EOS 设计提供操作指导
├─ 交付物形态：指南文档
├─ 承接的 SysReq 末级：SysReq-002.01 ~ SysReq-002.03
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：基于通用指南框架，填充 EOS 场景示例

PA-003: AI 决策框架（AI 辅助文档构件）
├─ 定义：定义 AI 在 EOS 设计中的决策规则和推荐逻辑
├─ 交付物形态：工作流/检查清单/示例
├─ 承接的 SysReq 末级：SysReq-003.01 ~ SysReq-003.04
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：参考 Pipeline AI 文档结构，内容按 EOS 定制

PA-004: SysReq 详细定义任务（任务定义构件）
├─ 定义：定义六步法中第5步的操作规程
├─ 交付物形态：操作规程文档
├─ 承接的 SysReq 末级：SysReq-004.01 ~ SysReq-004.02
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：参考 Pipeline 任务结构，操作对象改为 EOS
```

> 通用方法：PA 的完整设计和 SysReq→PA 的映射规则见[通用准则 §5.6](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#56-第5步sysreq--pa)。

## 六、常见错误、FAQ 与规范关系

### 6.1 常见错误

**❌ 错误1：混淆架构末级节点和详细定义末级节点的角色**

```
错误做法：
在相关方需求层，将"准则开发"直接作为 SR 架构末级节点，
但"准则开发"其实是 SR 的某个功能分类下的一个详细定义条目。

正确做法：
架构末级节点：SR-F-001（设计准则管理功能）—— 完整的业务能力
详细定义末级：SR-F-001.01（准则开发）—— 功能下的具体环节
```

**❌ 错误2：同步设计退化为串行设计**

```
错误做法：
先完成第2层全部详细定义，再开始第3层架构定义
→ 第2层详细定义完成后才发现 BA 没有对应的 IPO 可承接

正确做法：
第一轮：快速约定各层架构骨架（占位符 + 映射关系）
第二轮：并行填充各层详细内容
```

**❌ 错误3：验收标准和检查清单混为一谈**

```
错误做法：
验收标准和检查清单放在一起，列出的全是"是否已检查"类项，
没有量化指标衡量设计质量。

正确做法：
验收标准（量化指标）："映射覆盖率100%，1:1约束满足率100%"
检查清单（操作验证）："[ ] 检查所有需求是否已分解到末级"
```

**❌ 错误4：分析活动被遗漏或延后**

```
错误做法：
需求分解、分配和符合性分析在全部设计完成后才做，或者根本没做。

正确做法：
分析活动贯穿设计全过程——每完成一次分配立即执行对应的符合性分析。
```

**❌ 错误5：忽略资产优先原则，动辄新增架构节点**

```
错误做法：
新需求到达时，直接在架构中新增架构末级节点来承接，
而不检查现有节点是否能复用或改进。

正确做法：
按复用→改进→新增的优先级依次检查现有节点。
```

### 6.2 FAQ

**Q1：本文档与各层规范（02-06）是什么关系？**

本文档是 EOS 流水线系统设计的**总纲**，定义"为什么这样做"和"整体结构是什么"。各层规范定义"具体怎么做"。在阅读时，建议先读本文档了解全局，再查阅特定层规范。

**Q2：六步法中的步骤是否可以并行执行？**

可以。第2步（功能分支）和第4步（非功能分支）是独立并行的。第2步完成后可立即开始第3步和第5步的功能部分，无需等待第4步。两阶段工作流程是实现并行化的具体方法。

**Q3：通用准则、本准则（EOS 流水线）、EOS 准则三者的关系？**

三者构成"方法→实例→二次实例"的递进链：
- **通用准则**（`00-general/`）：产品无关的方法论核心
- **本准则**（`10-meta-pipeline/.../01-eos-pipeline-system-design-standards.md`）：通用方法论在 EOS 流水线产品上的实例化
- **EOS 准则**（`20-eos-design/.../01-eos-system-design-standards.md`）：通用方法论在 EOS 产品上的实例化（参考本准则的结构，内容按 EOS 定制）

三者共享同一套方法论核心（五层结构、核心规则），各有自己的产品特有规则。

**Q4：当详细定义无法 1:1 映射到下层架构时，应该调整哪一方？**

先检查详细定义是否还能进一步分解。如果详细定义已到原子级仍然无法 1:1，则说明下层架构设计不合理，需要补充或调整下层架构末级节点。经验法则：先怀疑分解不充分，再怀疑架构不完整。

**Q5：两阶段工作流程和同步设计是什么关系？**

两阶段工作流程是**实现方式**，同步设计是**设计原则**。同步设计要求上下游在设计时对齐，两阶段工作流程提供了具体的操作方法：第一阶段快速建立骨架和映射（对齐结构），第二阶段并行填充内容（对齐细节）。

**Q6：本准则的版本更新策略是什么？**

v3.x 系列定位为"内嵌通用方法论"的独立文档版本，v4.0 及以后定位为"基于通用方法论引用"的专有准则版本。版本号变动意味着准则的结构或定位发生了变化，使用者应关注版本文档头中的定位说明。
