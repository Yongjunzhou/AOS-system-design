## 一、EOS 流水线的产品语境

### 1.1 EOS 流水线产品概述

#### 产品类型

EOS 流水线是**知识/方法类产品**。其 PA 构件（交付物）的形态以文档和 AI Skill 定义为主，而非软件代码或实物组件。

#### 四个组成部分

EOS 流水线由四条子流水线组成：

| 子流水线 | 简称 | 说明 |
|---------|------|------|
| 系统设计流水线 | **设计线** | 定义如何从原始需求逐层推导至产品架构的方法和工具。当前已完成 |
| 组件开发流水线 | **开发线** | 定义如何根据产品架构对每个构件进行开发的方法和工具。待开发 |
| 系统集成流水线 | **集成线** | 定义如何沿产品架构的树形层级逐级集成形成完整产品的方法和工具。待开发 |
| 系统运维流水线 | **运维线** | 定义产品交付后的持续运行、监控、维护和优化的方法和工具。待开发 |

#### 产品开发四阶段

EOS 流水线本身作为一个产品，其开发也遵循四阶段框架：

| 阶段 | 名称 | 说明 |
|------|------|------|
| 阶段1 | **系统设计** | 从原始需求出发，经过五层结构（OR → SR → BA → SysReq → PA），开发出 EOS 流水线的产品架构，定义每个构件的需求和实现策略 |
| 阶段2 | **构件开发** | 对产品架构中所有需要新开发或改进的构件进行开发 |
| 阶段3 | **系统集成与交付** | 沿产品架构的树形层级逐级集成为完整的 EOS 流水线 |
| 阶段4 | **系统运维** | 交付后的持续运行、监控、维护和优化 |

**本准则的范围**：严格限定在**阶段1（系统设计）**。

#### 流水线工具组件

EOS 流水线的系统设计产出——即其产品架构的架构末级节点（构件）——包含以下四类交付物，它们构成 EOS 团队在系统设计阶段使用的工具集：

| 构件类型 | 交付物形态 | 说明 |
|---------|-----------|------|
| **设计准则** | 规范文档 | 定义系统设计的通用方法论和规则 |
| **设计指南** | 指南文档 | 为不同设计场景（瀑布式/敏捷式等）提供操作指导 |
| **AI 辅助文档** | 工作流/检查清单/示例 | 辅助 AI 参与系统设计过程 |
| **任务定义** | 操作规程文档 | 定义六步法各步骤的输入输出和操作流程 |

这四类构件的位置见 §2.2.2 的目录对应表。

### 1.2 三个参与者

EOS 流水线的系统设计活动涉及以下三个角色：

| 角色 | 职责 | 参与方式 | 关注的目录 |
|------|------|---------|-----------|
| **流水线开发者** | 开发、维护和优化 EOS 流水线本身。定义设计准则、设计指南、AI 辅助文档和任务定义 | 开发（构建流水线） | `01-*-specification/`、`02-*-guidelines/`<br>`03-*-ai-support/`、`04-*-tasks/` |
| **AI** | 按流水线开发者的指令执行自动化任务，生成文档初稿、检查一致性、验证映射关系 | 辅助（协作开发流水线） | `01-*-specification/`、`02-*-guidelines/`<br>`03-*-ai-support/`、`04-*-tasks/` |
| **EOS 开发者** | 使用 EOS 流水线对 EOS 进行系统设计（含需求提出者、业务流程开发者、信息系统开发者） | 使用（利用流水线做设计） | `01-*-specification/`、`02-*-guidelines/`<br>`04-*-tasks/` |

### 1.3 设计场景

EOS 流水线的系统设计可按以下四种设计方法场景推进。设计方向取决于流水线产品当前所处的阶段：

| 方法 | 设计方向 | 起点 | 终点 |
|------|---------|------|------|
| **瀑布式** | 正向完整：OR → SR → BA → SysReq → PA | 原始需求 | 产品架构 |
| **敏捷式** | 正向增量：分析影响 → 增量更新 | 现有系统设计 | 更新的系统设计 |
| **逆向工程** | 反向推导：PA 及构件 → 反向推导至 OR | 已有构件 | 完整系统设计 |
| **DevOps** | 最小正向变更：定位 → 修复 → 验证 | 生产环境问题 | 快速修复/发布 |

**选择决策**：

```
EOS 流水线处于什么阶段？

┌─ 已开发完成（如设计线已存在），但文档不全？
│  └─ 方法：逆向工程
│     └─ 从已有设计线构件反向推导出完整系统设计文档
│
├─ 只有原始需求，需要从零开始（如开发线/集成线/运维线的新建）？
│  └─ 方法：瀑布式
│     └─ 逐层正向推导（OR → SR → BA → SysReq → PA）
│
├─ 已有完整系统设计，需要增加新构件或修改现有构件？
│  └─ 方法：敏捷式
│     └─ 分析影响范围 → 增量更新受影响的文档
│
└─ 流水线已上线运行（如 AI 辅助流程需要快速修复）？
   └─ 方法：DevOps
      └─ 快速评估影响 → 最小化变更 → 快速验证
```

**各方法风险特征**：

| 方法 | 主要风险 | 应对策略 |
|------|---------|---------|
| **瀑布式** | 需求不清晰导致后期返工 | 在 OR 层做充分的需求分析和冲突检测 |
| **敏捷式** | 增量范围扩散超出迭代边界 | 严格控制每次迭代的影响范围分析 |
| **逆向工程** | 反向推导中对原始意图的理解偏差 | 多方确认推导结果，保留推导依据 |
| **DevOps** | 快速修复引入副作用 | 最小化变更范围，变更前后做回归验证 |

## 二、EOS 流水线的特有设计框架

本章定义 EOS 流水线系统设计的核心框架——这些内容是通用准则中未覆盖的、EOS 流水线产品特有的设计规则。

### 2.1 EOS 流水线的工程化开发逻辑

EOS 流水线的工程化开发遵循一条从产品架构识别到交付使用的完整链路。该链路以五层系统设计为基础（前置 OR → SR），从 BA 层开始聚焦，展示从 EOS 产品架构识别到流水线交付使用的完整工程化过程。

> **所属关系**：链中第①步产出归属于 **EOS 产品**；第②~⑦步的 BA、IPO、功能需求、SysReq、PA 构件全部归属于 **EOS 流水线**（其前置的 OR 和 SR 也归属于 EOS 流水线）；第⑧步的 EOS 开发团队是 **使用者**。

| 步 | 内容 | 核心要点 |
|----|------|---------|
| **①** | 识别 **EOS** 的产品架构及 **EOS 产品架构**的节点类型——整条链路的输入前提 | 明确流水线的服务对象和开发目标 |
| **②** | 对 EOS 产品架构所用每个类型的节点，定义使用 **EOS 流水线** 开发这类节点的业务过程——该过程即 **EOS 流水线的业务架构（EOS 流水线的 BA，即 EOS 流水线的作业指南）**，以多个 **EOS 流水线 BA 的 IPO** 表示 | BA 描述的不是 EOS 的内部运作，而是用户如何使用流水线来开发 EOS。BA 的输入来自前置的 OR 和 SR 工作 |
| **③** | 例如开发 EOS 中一个构件 A，其过程包含三个 **EOS 流水线 BA 的 IPO**：①概要设计 IPO——接收 EOS 构件 A 的需求，经 EOS 流水线生成概要设计方案；②详细设计 IPO——接收概要设计方案，经 EOS 流水线生成详细方案；③实现并发布 IPO——接收详细方案，经 EOS 流水线生成构件 A 并发布。这三个 IPO 都是 **EOS 流水线** 接收输入、处理后产生输出——它们就是 **EOS 流水线的功能需求** | BA IPO = 功能需求的来源 |
| **④** | **EOS 流水线的功能需求**（来自 BA IPO）经系统需求设计（在元流水线上执行），转化为 **EOS 流水线的系统需求（EOS 流水线的 SysReq）** | SysReq 以"接收→转换→生成"格式描述元流水线的信息处理功能 |
| **⑤** | **EOS 流水线的 SysReq** 经产品架构设计，分配给 **EOS 流水线的 PA 架构节点**，形成 **EOS 流水线各 PA 构件的构件需求**（即每个 PA 节点承接的 SysReq 条目集合） | SysReq → PA：功能需求决定了构件的内容 |
| **⑥** | **EOS 流水线的 PA 构件**（设计准则、设计指南、AI 辅助文档、任务定义）各自承接上述各自的构件需求 | 四类构件按职责分配 |
| **⑦** | 承接了构件需求的 **EOS 流水线 PA 构件**（如某个设计准则构件），即可基于其需求被设计开发出来 | 构件开发 = 实现功能需求 |
| **⑧** | **EOS 流水线** 的全部 PA 节点开发就绪后，EOS 开发团队即可使用它工程化地开发 EOS | 产出可用 |
| **⑨** | 传统方式非工程化，质量因人而异；此方法将人的经验显性化到流水线中，实现质量可控、效率提升，尤其有 AI 加持 | 方法论的工程化价值 |

上述流程中，每一对相邻层均构成"需求/方案相对"关系（详见[通用准则 §1.1](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#11-基本概念)）——各层的角色标签详见[通用准则 §4.2](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#42-五层结构)的五层结构图。

```
EOS 产品架构节点类型
  → (前置) EOS 流水线的 OR → EOS 流水线的 SR     ← 五层设计的前置阶段
    → EOS 流水线的 BA（作业指南：如何使用流水线开发该类节点）
      → BA 中的 IPO = EOS 流水线的功能需求
        → EOS 流水线的 SysReq
          → EOS 流水线的 PA 构件
            → PA 构件被开发出来
              → EOS 流水线可供使用
```

### 2.2 EOS 流水线的位置及用途

EOS 流水线的系统设计产物不是终点——它们最终要交付给 EOS 开发者使用。本节从静态层面结构阐述 EOS 流水线在整体产品结构中的位置、其构件交付去向，以及设计产物的复用策略。

#### 2.2.1 三层工作层面

EOS 流水线和 EOS（企业运营体系）之间的关系由**三个工作层面**构成：

| 层面 | 内容 | 负责方 | 仓库位置 |
|------|------|--------|---------|
| **第1层：流水线设计** | 基于通用方法论，使用元流水线对 EOS 流水线的原始需求进行系统设计，产出其产品架构（构件清单） | 流水线开发者 | `10-meta-pipeline/` |
| **第2层：构件开发** | 基于通用方法论，使用元流水线，根据 EOS 流水线 PA 的构件定义，开发各构件的完整内容（准则、指南、AI 辅助、任务定义），交付至 EOS 设计空间中供使用 | 流水线开发者 | `20-eos-design/.../01~04-*` |
| **第3层：使用工具设计 EOS** | EOS 开发者使用第2层产出的工具对 EOS 进行系统设计，产出 EOS 的产品数据 | EOS 开发者 | `20-eos-design/.../10-product-data/` |

**内嵌关系**：运营体系涵盖市场、研发、生产、售后等业务，同时也包含"设计运营体系本身"这项业务。EOS 流水线就是支撑这项业务的方法和工具集。EOS 流水线的四类构件被放置在 `20-eos-design/10-eos-system-design/` 目录中，作为 EOS 开发者设计 EOS 时的操作规程。

```
运营体系
├── 市场
├── 研发
├── 生产
├── 售后
└── 设计运营体系本身 ← EOS 流水线提供方法支撑
      ├── 系统设计方法论
      └── 设计准则、指南、AI辅助、任务定义（EOS 流水线的 PA 构件）
```

> **EOS 流水线开发者视角**：你的工作对应第1层和第2层——先用通用方法论对 EOS 流水线进行系统设计（第1层），再根据 PA 构件定义开发各构件的完整内容并交付至 EOS 设计空间（第2层）。

#### 2.2.2 构件交付目录映射

| EOS 流水线构件类型 | → 交付为 EOS 系统设计目录下的 | 说明 |

#### 2.2.3 设计产物的复用策略

设计产物在交付和引用时，遵循产品同构性原则（详见[通用准则 §1.8.2](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#182-产品同构性product-homomorphism)）。在此基础上，按复用程度分为三个层次：

| 参考类别 | 参考程度 | 适用产物 | 在 EOS 流水线中的实例 |
|---------|---------|---------|---------------------|
| **直接参考** | 参考内容 | 设计准则、设计指南 | 通用准则的方法论部分直接用于 EOS 流水线 |
| **结构化参考** | 结构参考，内容定制 | AI 辅助文档、任务定义 | 流程结构相同，但操作对象和业务场景按 EOS 流水线定制 |
| **模板参考** | 仅框架参考 | 产品数据（9 份文档） | 文件命名和文档框架相同，内容由 EOS 流水线独立填写 |

> 产品同构性的完整定义见[通用准则 §1.8.2](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#182-产品同构性product-homomorphism)，四个同构维度（设计链路、映射规则、文档体系、设计方法）此处不再重复。

### 2.3 PA 节点定义原则

**上下文与问题**：在工程化开发链路（§2.1 第⑤步）中，EOS 流水线的 SysReq 条目需要分配到 PA 架构节点，形成各构件的构件需求。分配决策不仅要确定每条 SysReq 属于四类构件（设计准则/设计指南/AI辅助文档/任务定义，见 §1.1）中的哪个类型，还需判断同类型内的多条 SysReq 是否应合并或拆分。

决策按以下次序执行：首先进行资产优先检查（§2.3.1），再基于三因素（§2.3.2~§2.3.4）判断。

---

#### 2.3.1 资产优先检查

**先检查组织已有组件资产**，遵循复用→改进→新增优先级（详见通用准则 §2.4.2）。四类构件的资产形态各异——准则的资产是已有规范文档和方法论标准，指南的资产是现有指南和场景模板，AI 辅助文档的资产是 Prompt 库和工作流定义，任务定义的资产是操作规程和任务模板。

| 优先级 | 策略 | 适用条件 | 操作 |
|--------|------|---------|------|
| 最高 | **复用** | 现有构件完全满足需要 | 直接使用，无需新分配 |
| 次之 | **改进** | 现有构件可扩展/拆分/合并后满足 | 调整现有构件后分配 |
| 最低 | **新增** | 无合适现有构件 | 用三因素判定边界 |

---

#### 2.3.2 因素一：知识领域内聚性

操作**同一知识领域**（共享同一概念体系）的 SysReq 倾向分配到同一构件：

- **强内聚可共享**：五层设计方法论准则和五层设计方法论指南共享"五层结构""核心规则"等知识基础，合并为同一构件组的两个子构件（分别属于设计准则和设计指南类型）
- **弱内聚应分离**：系统设计准则与系统运维准则虽同属 EOS 流水线，但设计阶段和运维阶段的知识领域不同，应分为两个独立构件（均属设计准则类型）

> **原则**：同领域→合并；不同领域但有关联→引用集成。同类型不同领域倾向分离；不同类型同领域倾向作为关联构件组而非合并。

---

#### 2.3.3 因素二：质量要求相容性

四类构件的质量要求不同——冲突时不应共享同一构件（例如极高精确性的需求应归设计准则，高可执行性的需求应归 AI 辅助文档）：

| 维度 | 设计准则 | 设计指南 | AI 辅助文档 | 任务定义 |
|------|---------|---------|------------|---------|
| **精确性** | 极高 | 高，允许灵活 | 高，允许推理 | 中高 |
| **可执行性** | 低（供人阅读） | 中（人执行） | 极高（AI 直接执行） | 高（人+AI） |

---

#### 2.3.4 因素三：交付形态相似性

知识制品的**消费方式**决定了构件边界。§2.4 定义了 EOS 流水线的信息处理模式：

- **同模式**→倾向合并（维护节奏一致）
- **不同模式**→倾向分离（如规范规则按版本管理，AI 指令需频繁调优）

---

#### 2.3.5 三因素决策矩阵

| 知识领域 | 质量要求 | 交付形态 | 结论 |
|---------|---------|---------|------|
| 相同 | 相容 | 相同 | ✅ **合并**为同一构件实例 |
| 相同 | 冲突 | — | ❌ **分离**（质量要求优先） |
| 不同 | 相容 | 相同 | ⚠️ 看耦合度，引用集成 |
| 不同 | 冲突 | 不同 | ❌ **必然分离** |

**决策输出**：分配结果进入工程化开发链路第⑥步（§2.1）——各 PA 构件承接构件需求，进入开发环节。

---

### 2.4 EOS 流水线的信息处理模式

知识/方法类产品的"信息处理"本质上是知识如何被结构化、交付和消费。EOS 流水线的构件遵循五种信息处理模式：

| 模式 | 核心特征 | 消费方式 | 典型构件 |
|------|---------|---------|---------|
| **规范规则** | 条文式约束，作为判断依据 | 人查阅、对照、引用 | 设计准则 |
| **操作指南** | 场景化指导，说明做法和理由 | 人阅读、理解、应用 | 设计指南 |
| **AI 指令** | 结构化 Prompt，包含步骤和约束 | AI 直接解析执行 | AI 辅助文档 |
| **操作规程** | 分步骤任务，明确角色分工 | 人+AI 协作执行 | 任务定义 |
| **数据模板** | 结构化数据框架，填充内容 | 人+工具填写 | 产品数据文件 |

**核心规律**：不同模式 → 倾向分离；同模式 → 可共享。两个系统需求即使知识领域接近，如果交付形态不同（如一个为 AI 指令、一个为规范规则），也倾向分配到不同构件。

---

## 三、通用方法论精要

本章以极简篇幅摘要列出通用准则的核心方法论，并标注其在 EOS 流水线语境下的关注点。完整定义请直接查阅通用准则对应章节。

> **重要**：以下各节均非通用准则的替代品，而是帮助你快速定位到所需的通用方法内容。初次接触 EOS 流水线的系统设计前，**建议先通读通用准则 §一~§七**。

### 3.1 五层结构（通用 §四）

EOS 流水线的系统设计遵循标准的五层设计链路，各层在流水线语境下的关注点如下：

| 层级 | 在 EOS 流水线中的关注点 |
|------|----------------------|
| **OR → SR** | 流水线的原始需求从哪些渠道来（会议纪要、邮件、已有准则）、如何规范化为相关方需求 |
| **SR → BA** | 从流水线的功能需求中识别业务活动边界，定义 IPO |
| **BA → SysReq** | 将 IPO 去重后按系统功能分类重组——注意四类构件（准则/指南/AI辅助/任务定义）的功能差异 |
| **SysReq → PA** | 按 PA 节点定义原则（§2.3）进行构件划分，得到四类构件的完整清单 |

> 完整五层结构定义见[通用准则 §四](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#四系统设计链路)，三层核心规则见[通用准则 §三](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#三文档结对设计)。

### 3.2 文档结对设计（通用 §三）

EOS 流水线的系统设计过程包含四个依次衔接的文档结对：

| 序号 | 结对 | 在 EOS 流水线中的映射 |
|------|------|---------------------|
| 1 | OR 条目级需求 ↔ SR 架构定义 | 流水线原始需求 → 相关方需求架构 |
| 2 | SR 详细定义末级 ↔ BA 架构（IPO） | 相关方需求 → 业务组件（IPO清单） |
| 3 | BA IPO（去重后）↔ SysReq 架构定义 | IPO → 系统功能模块 |
| 4 | SysReq 详细定义末级 ↔ PA 架构定义 | 系统功能 → 四类构件 |

**结对设计原则**：上层详细定义与下层架构定义应同步设计而非串行执行——先快速约定下层架构的结构骨架，再并行填充内容。

> 需求分解/分配/分析三个核心设计活动的完整操作规则见[通用准则 §三](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#三文档结对设计)。编号规则见[通用准则 §3.4](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#34-编号规则)。

### 3.3 组织资产优先（通用 §2.4）

EOS 流水线各层的架构节点不是凭空产生的，而是基于组织现有资产按 **复用 → 改进 → 新增** 优先级生成。

在 EOS 流水线语境下，各层的资产形态：

| 层级 | 资产形态 |
|------|---------|
| **SR 层** | 需求分类体系、领域知识结构、行业标准目录 |
| **BA 层** | 已有业务过程、岗位职责分工、跨系统交互流程 |
| **SysReq 层** | 系统边界划分、接口协议、技术域约定 |
| **PA 层** | 现有四类构件（准则/指南/AI辅助/任务定义）的定义和结构 |

> 资产优先级决策规则及操作细则见[通用准则 §2.4](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#24-基于资产定义架构)。

### 3.4 系统设计过程（通用 §五）

EOS 流水线的系统设计遵循通用准则定义的**六步法**，按以下步骤顺序执行：

| 步骤 | 名称 | 在 EOS 流水线中的映射 |
|------|------|---------------------|
| Step 1 | 原始需求分析及相关方需求架构定义 | 收集 EOS 流水线原始需求，建立 SR 架构 |
| Step 2 | 相关方需求详细定义及业务架构定义 | 展开 SR 详细定义，识别业务活动为 IPO |
| Step 3 | 系统需求功能部分架构定义 | BA IPO 去重后按四类构件功能组织 |
| Step 4 | 非功能需求详细定义及系统需求非功能架构 | SR-NFR 平行路径，不经过 BA |
| Step 5 | 系统需求详细定义及产品架构定义 | 按 PA 节点定义原则划分四类构件 |
| Step 6 | 双向追溯验证 | 验证 OR → SR → BA → SysReq → PA 完整追溯 |

> 六步法的完整定义、每步的检查要点和验收标准见[通用准则 §五](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#五系统设计过程)。

### 3.5 验收标准与检查清单（通用 §六、§七）

通用准则定义了六个维度的质量验收标准（§六）和对应的逐项检查方法（§七），是 EOS 流水线系统设计质量评估的通用框架。**本文档 §五 仅补充 EOS 流水线特有的验收项，通用项不再重复**。

> 完整验收标准见[通用准则 §六](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#六设计验收标准)，完整检查清单见[通用准则 §七](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#七设计检查清单)。

## 四、各层设计指引

本章为 EOS 流水线的五层设计提供场景化指导。每层聚焦"在 EOS 流水线语境下如何进行设计活动"，通用方法步骤不再复述。

### 4.1 第1层 OR：原始需求及分析

**EOS 流水线的 OR 特征**：EOS 流水线的原始需求主要来自以下渠道：
- 流水线开发者的设计目标和构想
- 对已有设计线构件的逆向分析需求
- EOS 开发者的使用反馈和改进建议
- 通用方法论版本更新触发的适配需求

**设计指引**：

1. **收集**阶段重点关注四条子流水线的需求覆盖面——当前仅设计线已完成，开发线、集成线、运维线处于待启动状态，OR 应覆盖所有四条线的需求
2. **初步整理**时区分"新建子线"和"改进现有构件"两类需求——前者需要完整五层设计，后者走敏捷或 DevOps 路径
3. **规范化**时关注流水线特有的术语（如"四类构件""设计场景""AI 辅助"等），确保术语一致
4. **分解到末级**时遵循 1:1 约束——每条子需求须能分配到 SR 唯一一个架构末级节点

**输出**：原始需求详细定义（末级条目清单）

> 通用方法：OR 预处理（碎片整理/规范化/冲突检测）见[通用准则 §5.2 阶段一](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#52-第1步or--sr)。

### 4.2 第2层 SR：相关方需求

**SR 架构分类**：EOS 流水线的相关方需求按功能和非功能两个分支组织。

**功能部分**的典型分类：

```
EOS 流水线相关方需求（功能部分）
├── 设计线功能
│   ├── 设计准则
│   ├── 设计指南
│   ├── AI 辅助设计
│   └── 任务定义
├── 开发线功能（待设计）
├── 集成线功能（待设计）
└── 运维线功能（待设计）
```

**非功能部分**按[通用准则附录B](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#附录b-非功能需求分类体系)的三大类（质量特性/环境适应性/可实现性）组织。

**设计指引**：

1. 功能需求和非功能需求分别标识（`SR-F-` / `SR-NFR-`）
2. 每条 SR 架构末级节点承接 2-5 条 OR 末级条目（N:1），检查合并的合理性
3. 详细定义分解到每条末级条目只需映射到 BA 的一个 IPO

> 通用方法：SR 详细定义分解和 IPO 映射见[通用准则 §5.3](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#53-第2步sr功能--ba)。

### 4.3 第3层 BA：业务架构

**EOS 流水线的 BA 层级结构**：

| 层级 | 名称 | 说明 |
|------|------|------|
| 0级 | 架构根节点 | 运营体系 |
| 1级 | 业务域 | 流程与IT |
| 2级 | 端到端项目 | EOS 流水线开发 |
| 3级 | 架构节点类型端到端业务 | 按四类构件类型（准则/指南/AI辅助/任务定义）组织的端到端开发业务 |
| 4级 | 工作包 | 某类型构件开发中的工作分解 |
| **5级** | **业务组件或引擎组件（承接层）** | **开发某类文档对应的 IPO（如设计准则开发 IPO：输入方法论→编制→审查→发布→输出准则文档）** |
| 6级 | 活动 | IPO 具体场景的活动 |

**BA 的本质**：EOS 流水线的业务架构本质上是 EOS 流水线的作业指南——描述用户（EOS 流水线开发者、AI）使用元流水线开展哪些业务活动，以及如何开展这些活动。BA 中的每个 IPO 描述的是元流水线上的一个知识加工步骤。

**IPO 识别方法**：从 SR 详细定义集合中识别业务活动边界，以"元流水线接收→转换→产出"为框架描述每个 IPO。例如，从 SR 中的"准则开发"功能需求推导出 BA IPO：

```
BA-001: EOS 系统设计准则开发（IPO名称）
├─ 元流水线接收：通用方法论核心内容 + EOS 流水线 SR 中的设计准则相关需求
├─ 元流水线转换：人工主导、AI 辅助，按准则结构进行裁剪、定制和审查
└─ 元流水线产出：EOS 系统设计准则文档 v1.0
```

**IPO 去重**：同一 IPO 可能被多条 SR 详细定义映射——将重复出现的 IPO 归并为唯一条目，形成去重后的 IPO 清单。

> 通用方法：BA 的独立存在特性和 IPO 映射规则见[通用准则 §5.3](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#53-第2步sr功能--ba)。

### 4.4 第4层 SysReq：系统需求

**SysReq 的性质**：EOS 流水线作为知识/方法类产品，其系统需求描述的是**元流水线的信息处理功能**——每项 SysReq 是分配给某个 PA 架构节点类型（构件类型）的一条功能需求，统一采用"接收→转换→生成"格式：

> **[节点类型] 应接收到 xx 信息，经过 xx 转换（人工参与或 AI 辅助），生成 xx 信息**

转换方式因构件类型而异：

| 构件类型 | 执行实体 | 转换方式 | 说明 |
|---------|---------|---------|------|
| **设计准则** | 人 + 准则文档 | 人按准则指引进行判断和操作 | 人与文档构成"虚拟业务组件" |
| **设计指南** | 人 + 指南文档 | 人按指南步骤进行场景化操作 | 同上 |
| **AI 辅助文档** | AI + 文档指令 | AI 按指令自动处理，人审查确认 | 文档本身是可执行的"业务组件" |
| **任务定义** | 人 + 任务规程 | 人按任务步骤操作，AI 辅助执行 | 人与文档构成"虚拟业务组件" |

**双重来源**：

| 来源 | 说明 | 处理方式 |
|------|------|---------|
| **功能需求** | BA IPO 去重清单 | IPO 去重后按四类构件类型组织，每项 IPO 对应一条 SysReq 功能条目 |
| **非功能需求** | SR-NFR（平行路径，不经过 BA） | 按质量特性/环境适应性/可实现性三大类组织，形成 0-6 级架构 |

**功能部分示例**（设计线四类构件的 SysReq 条目）：

| PA 架构节点类型 | 所承接的 SysReq 条目 |
|----------------|---------------------|
| EOS 系统设计准则 | 应接收到通用方法论核心内容及 EOS 流水线 SR 中的设计准则相关需求，经过人工主导的裁剪定制转换（辅以 AI 分析），生成 EOS 专用的系统设计准则文档 |
| EOS 系统设计指南 | 应接收到通用设计指南框架及 EOS 流水线的场景特征参数，经过人工主导的场景化定制转换（辅以 AI 推荐匹配），生成对应场景的设计指南文档 |
| EOS 系统设计 AI 辅助文档 | 应接收集成后的准则和指南结构化内容，经过 AI 指令化转换（AI 主导执行，人工审查确认），生成可执行的 AI 提示文档 |
| EOS 系统设计任务定义 | 应接收到端到端业务的 WBS 分解结果，经过人工主导的操作规程化转换（辅以 AI 模板），生成标准化的任务定义文档 |

> **关键认知**：SysReq 中的功能需求一旦定义完整，对应 PA 构件的内容（准则的规则体系、指南的流程步骤、AI 文档的提示指令、任务操作步骤）也就确定了——功能需求决定了内容需求。

**设计指引**：

1. 功能部分从 BA IPO 去重清单出发，每项 IPO 编写一条 SysReq，描述该 IPO 在元流水线上的"接收→转换→生成"功能
2. 每条 SysReq 对应于一个 PA 架构节点类型（构件类型），功能需求的完整定义即决定了该构件应包含的内容
3. 非功能部分直接从 SR-NFR 承接，不经过 BA 层
4. NFR 在 SysReq 层的处理关注：可用性、可维护性、版本兼容性对知识/方法类产品的影响

> 通用方法：SysReq 的分解过程和功能/非功能分配路径见[通用准则 §5.4~§5.5](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#54-第3步ba--sysreq功能架构)。

### 4.5 第5层 PA：产品架构

**EOS 流水线的四类构件**（知识/方法类产品的 PA 构件）：

| 类型 | 交付物形态 | 说明 |
|------|-----------|------|
| **设计准则** | 规范文档 | 定义 EOS 流水线的设计方法论和规则 |
| **设计指南** | 指南文档 | 为四种设计场景提供操作指导 |
| **AI 辅助文档** | 工作流/检查清单/示例 | 辅助 AI 参与 EOS 流水线的设计和开发 |
| **任务定义** | 操作规程文档 | 定义六步法各步骤的输入输出和操作流程 |

**构件划分原则**：每个构件应有明确的职责边界。对于 EOS 流水线（知识/方法类产品），粒度标准为"可由单人独立完成"且"可独立审查和验收"。

**SysReq → PA 的映射**：每个 PA 架构节点承接若干 SysReq 功能需求条目。对于 EOS 流水线，这些 SysReq 条目以"接收→转换→生成"格式描述元流水线的信息处理功能（见 §4.4）。SysReq 定义完整后，该构件的具体内容（准则的规则、指南的步骤、AI 指令、任务操作规程）即告确定——功能需求决定了内容需求。

**PA 节点定义方法**：

1. 从 SysReq 9 级场景活动出发，按 §2.3 的 PA 节点定义原则判断哪些活动应合并为同一构件
2. 按复用→改进→新增的资产优先顺序（[通用 §2.4](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#24-基于资产定义架构)）处理已有构件
3. 每个构件标注四要素：定义、所映射的 SysReq 末级、符合性分析、实现策略

**举例**：设计线 PA 构件的划分（四类各一个示例节点）：

```
PA-001: EOS 系统设计准则（设计准则构件）
├─ 定义：定义 EOS 系统设计的通用方法论和规则
├─ 交付物形态：规范文档
├─ 承接的 SysReq 末级：SysReq-001.01 ~ SysReq-001.05
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：参考通用准则结构，按 EOS 场景定制

PA-002: 瀑布式场景设计指南（设计指南构件）
├─ 定义：为瀑布式场景下的 EOS 设计提供操作指导
├─ 交付物形态：指南文档
├─ 承接的 SysReq 末级：SysReq-002.01 ~ SysReq-002.03
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：基于通用指南框架，填充 EOS 场景示例

PA-003: AI 决策框架（AI 辅助文档构件）
├─ 定义：定义 AI 在 EOS 设计中的决策规则和推荐逻辑
├─ 交付物形态：工作流/检查清单/示例
├─ 承接的 SysReq 末级：SysReq-003.01 ~ SysReq-003.04
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：参考 Pipeline AI 文档结构，内容按 EOS 定制

PA-004: SysReq 详细定义任务（任务定义构件）
├─ 定义：定义六步法中第5步的操作规程
├─ 交付物形态：操作规程文档
├─ 承接的 SysReq 末级：SysReq-004.01 ~ SysReq-004.02
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：参考 Pipeline 任务结构，操作对象改为 EOS
```

> 通用方法：PA 的完整设计和 SysReq→PA 的映射规则见[通用准则 §5.6](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#56-第5步sysreq--pa)。

## 五、EOS 流水线验收标准

以下仅列出 EOS 流水线特有的验收项。通用验收标准（架构定义完整率、详细定义完成率、1:1 约束满足率、映射覆盖率、追溯完整性等）见[通用准则 §六](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#六设计验收标准)。

### 5.1 四类构件的验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **构件分类完整率** | 100% | PA 构件按四类（准则/指南/AI辅助/任务定义）完整划分，无遗漏 |
| **构件交付物明确率** | 100% | 每个构件标注了交付物形态（规范文档/指南文档/工作流/操作规程） |
| **构件独立可验收率** | 100% | 每个构件可独立审查和验收，不依赖其他构件的交付物 |

### 5.2 四条子线的覆盖

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **设计线覆盖** | 100% | 系统设计输出完整覆盖设计线的四个构件类型 |
| **跨线扩展能力** | 记录在案 | 产品架构是否为开发线/集成线/运维线预留了扩展点（架构树中标注了待扩展的分类节点） |

### 5.3 多产品关系

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **构件-目录对齐率** | 100% | EOS 流水线的 PA 构件与 `20-eos-design/` 下目录的对应关系已在准则中明确标注 |
| **三层工作层面标记** | 100% | 每个 PA 构件标注服务于第几层工作层面（第1层/第2层） |

### 5.4 设计场景匹配

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **场景-方法一致性** | 100% | 当前所采用的设计方法场景与实际设计方向一致（参考 §1.3 的选择决策） |
| **风险记录完整率** | 100% | 所选场景的已知风险已在设计计划中记录应对策略 |

## 六、常见错误、FAQ 与规范关系

### 6.1 常见错误

**❌ 错误1：混淆架构末级节点和详细定义末级节点的角色**

```
错误做法：
在相关方需求层，将"准则开发"直接作为 SR 架构末级节点，
但"准则开发"其实是 SR 的某个功能分类下的一个详细定义条目。

正确做法：
架构末级节点：SR-F-001（设计准则管理功能）—— 完整的业务能力
详细定义末级：SR-F-001.01（准则开发）—— 功能下的具体环节
```

**❌ 错误2：同步设计退化为串行设计**

```
错误做法：
先完成第2层全部详细定义，再开始第3层架构定义
→ 第2层详细定义完成后才发现 BA 没有对应的 IPO 可承接

正确做法：
第一轮：快速约定各层架构骨架（占位符 + 映射关系）
第二轮：并行填充各层详细内容
```

**❌ 错误3：验收标准和检查清单混为一谈**

```
错误做法：
验收标准和检查清单放在一起，列出的全是"是否已检查"类项，
没有量化指标衡量设计质量。

正确做法：
验收标准（量化指标）："映射覆盖率100%，1:1约束满足率100%"
检查清单（操作验证）："[ ] 检查所有需求是否已分解到末级"
```

**❌ 错误4：分析活动被遗漏或延后**

```
错误做法：
需求分解、分配和符合性分析在全部设计完成后才做，或者根本没做。

正确做法：
分析活动贯穿设计全过程——每完成一次分配立即执行对应的符合性分析。
```

**❌ 错误5：忽略资产优先原则，动辄新增架构节点**

```
错误做法：
新需求到达时，直接在架构中新增架构末级节点来承接，
而不检查现有节点是否能复用或改进。

正确做法：
按复用→改进→新增的优先级依次检查现有节点。
```

### 6.2 FAQ

**Q1：本文档与各层规范（02-06）是什么关系？**

本文档是 EOS 流水线系统设计的**总纲**，定义"为什么这样做"和"整体结构是什么"。各层规范定义"具体怎么做"。在阅读时，建议先读本文档了解全局，再查阅特定层规范。

**Q2：六步法中的步骤是否可以并行执行？**

可以。第2步（功能分支）和第4步（非功能分支）是独立并行的。第2步完成后可立即开始第3步和第5步的功能部分，无需等待第4步。两阶段工作流程是实现并行化的具体方法。

**Q3：通用准则、本准则（EOS 流水线）、EOS 准则三者的关系？**

三者构成"方法→实例→二次实例"的递进链：
- **通用准则**（`00-general/`）：产品无关的方法论核心
- **本准则**（`10-meta-pipeline/.../01-eos-pipeline-system-design-standards.md`）：通用方法论在 EOS 流水线产品上的实例化
- **EOS 准则**（`20-eos-design/.../01-eos-system-design-standards.md`）：通用方法论在 EOS 产品上的实例化（参考本准则的结构，内容按 EOS 定制）

三者共享同一套方法论核心（五层结构、核心规则），各有自己的产品特有规则。

**Q4：当详细定义无法 1:1 映射到下层架构时，应该调整哪一方？**

先检查详细定义是否还能进一步分解。如果详细定义已到原子级仍然无法 1:1，则说明下层架构设计不合理，需要补充或调整下层架构末级节点。经验法则：先怀疑分解不充分，再怀疑架构不完整。

**Q5：两阶段工作流程和同步设计是什么关系？**

两阶段工作流程是**实现方式**，同步设计是**设计原则**。同步设计要求上下游在设计时对齐，两阶段工作流程提供了具体的操作方法：第一阶段快速建立骨架和映射（对齐结构），第二阶段并行填充内容（对齐细节）。

**Q6：本准则的版本更新策略是什么？**

v3.x 系列定位为"内嵌通用方法论"的独立文档版本，v4.0 及以后定位为"基于通用方法论引用"的专有准则版本。版本号变动意味着准则的结构或定位发生了变化，使用者应关注版本文档头中的定位说明。
