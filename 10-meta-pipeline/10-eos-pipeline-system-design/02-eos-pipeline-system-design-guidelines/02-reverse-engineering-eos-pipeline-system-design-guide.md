## 📌 指南范围

本指南针对**系统设计逆向工程**——从已有的产品架构（PA）及架构各节点（构件）的需求定义出发，沿五层结构反向追溯，生成完整的系统需求（SysReq）、业务架构（BA）、相关方需求（SR）乃至原始需求（OR）：

```
PA（产品架构 + 构件需求定义）→ SysReq → BA → SR → OR
```

**另一种逆向工程：组件开发逆向工程**（属于阶段2 构件开发）
从已完成组件的代码或图纸出发，逆向生成组件的详细设计方案、概要设计方案、组件需求定义。本指南不涉及此类型。

### 依据B：三层工作层面说明

本指南属于**第2层工作（交付物构建）**——设计线（系统设计流水线）的**设计指南构件**，对应的产品架构节点为 `02-eos-pipeline-system-design-guidelines/02-reverse-engineering/`。

三层工作层面：
- **第1层（流水线设计）**：设计线用方法论设计自身，产出设计线的产品数据
- **第2层（交付物构建）**：本指南属于此层——设计线的"设计指南"构件，由团队L开发、维护和优化，作为工具交付给团队M使用
- **第3层（使用工具设计EOS）**：团队M使用第2层工具设计运营体系，产出EOS的产品数据

设计线的架构节点对应四个目录——它们是设计线交付给团队M使用的工具：

| 设计线的构件目录 | 用途 |
|----------------|------|
| `01-*-system-design-specification/` | 设计准则（规范与标准） |
| `02-*-system-design-guidelines/` | 设计指南（操作方法） |
| `03-*-system-design-ai-support/` | AI 辅助文档（自动化支持） |
| `04-*-system-design-tasks/` | 任务定义（操作规程） |

以上四类构件在 `10-meta-pipeline/` 下开发，同时为 `20-eos-design/` 提供相同结构的模板。

团队M使用设计线的工具后产出的**EOS（企业运营体系）设计内容**，存放在 `20-eos-design/10-eos-system-design/10-eos-system-product-data/` 中。这些产品数据是设计线的**输出物**，而非设计线自身的构件。

### 逆向工程在设计线上的自应用

上述关系揭示了一个重要事实：`20-eos-design/10-eos-system-design/` 下的 `01/02/03/04` 目录中已存在的准则、指南、AI辅助文档、任务定义，就是设计线的产品架构（PA）节点。它们已有完整的内容，但从未被系统地反向推导过设计线上游的系统需求（SysReq）、业务架构（BA）、相关方需求（SR）和原始需求（OR）。

因此，本指南描述的逆向工程方法同样适用于**设计线自身**——以设计线的 PA（即 `01/02/03/04` 目录下的文档集）为起点，沿 `PA → SysReq → BA → SR → OR` 反向推导，生成设计线自身完整的系统设计文档链。这不仅能为设计线补齐缺失的上游需求基线，也是对这套方法论有效性的自验证。

## 📌 场景描述

### 适用条件

#### 核心条件（启动逆向推导的前提）
- ✅ 产品架构（PA）及架构各节点的需求定义已存在（即使质量不足或不完整）
- ✅ 技术文档缺失或不符合设计准则
- ✅ 缺少从需求到架构的完整追溯链路
- ✅ 需要生成符合准则的系统设计文档

#### 前置条件（仅在需要重建PA文档时需要）
- ✅ 有完整的产物目录或代码实现（当PA文档不存在或质量严重不足时）

### 不适用条件

- ❌ PA 文档极度缺失且无可用的产物目录或代码用于重建PA
- ❌ 原开发人员无法参与
- ❌ 需要快速完成（时间紧张）

### 参与角色

本指南涉及三个核心参与者，在逆向工程流程中各司其职：

| 角色 | 职责 | 参与环节 |
|------|------|---------|
| **流水线开发者** | 定义设计准则和指南，配置 AI 辅助工具，审核 AI 输出 | 所有步骤的审核确认 |
| **AI** | 按指令执行 PA 分析、需求反向推导、映射关系建立等自动化任务 | 核心逆向步骤的自动生成，前置准备中的代码分析 |
| **EOS开发者** | 提出需求、确认设计、审核方案（含需求提出者、业务流程开发者、信息系统开发者三类） | 文档审核与确认、前置准备中的代码分析 |

## 📊 关键概念

### 逆向工程的流向特点

与正向（瀑布式）设计不同，逆向工程的推导方向是 **自底向上**（第5层→第1层）。这意味着：

- 每步的分配及分析从"检查完整性"变为"验证推导一致性"——即衍生出的上层需求是否与下层实现一致
- **同步设计原则**在逆向中表现为：在推导上层时，需同步检查下层架构是否能合理承接
- **详细定义的双重身份**（既是本层细化又是下层需求）在逆向中帮助判断：从下层方案能否合理反推出上层需求

### 架构定义与详细定义在逆向中的关注点

每份方案文档由两个子文档构成（系统设计准则 §1.5），在逆向中需注意：

| 子文档 | 正向设计 | 逆向工程 |
|-------|---------|---------|
| **架构定义文档** | 先定义架构再分解 | 从下层的架构和实现信息归纳出架构树 |
| **详细定义文档** | 对架构末级节点分解 | 从下层的详细内容反推出分解条目 |

### 三个核心设计活动的逆向适配

系统设计准则要求每层执行三个核心设计活动。在逆向中适配为：

- **需求分解及分析**：从下层实现反推上层的需求分解，检查每层的分解是否完整覆盖已有实现
- **需求分配及分析**：验证下层方案到上层需求的映射关系完整，确保每个节点都有明确的分配/承接关系
- **架构承接需求分析**：从下层实现特征反向推导上层需求应满足的指标范围，确认下层实现充分满足上层要求

### 两阶段工作流程在逆向中的应用

由于逆向是自底向上推导，两阶段工作流程（§1.5.3）表现为：

- **第一阶段**：先快速从已有设计制品提取各层结构骨架（节点名称 + 映射关系），不做深度内容
- **第二阶段**：在各层骨架建立后，再并行填充详细内容
- 这与正向的区别在于：正向是先有需求再设计方案，逆向是先有实现再追溯需求

### 编号稳定性原则

在逆向工程中，各层节点的编号一经确定并冻结后，**应保持稳定**，不因后续修订而改变。这是因为：
- **已建立的映射关系依赖编号**：各层之间的映射关系基于编号建立，修改编号将导致追溯链路断裂
- **生成的文档是后续迭代的基线**：逆向产出的完整文档集将作为后续正向或敏捷设计的基线，基线编号应稳定
- **1:1 约束和 N:1 承接的引用基础**：所有分配关系和承接关系均以编号为锚点

编号变更仅限于：经评审确认编号冲突或架构调整时同步更新所有相关映射。

### 组织资产优先原则

在逆向工程的自底向上推导中，资产优先原则主要体现在反向聚合时的复用检查：

```
第1优先 ── 复用（Reuse）—— 反向聚合时检查上层是否已有可承接的节点
第2优先 ── 改进（Improve）—— 现有节点扩展后承接
第3优先 ── 新增（Add）—— 创建新节点
特殊情况 ── 废弃（Deprecate）—— 标注而非物理删除
```

**在逆向工程中的应用**：
- 第4步的 IPO 去重（复用→改进→新增）直接体现该原则：多个 SysReq 对应相同业务操作时，优先复用已识别的 IPO
- 合并建议机制：自底向上推导中发现重复节点时，标记合并建议而非立即删除
- 详见[AI决策框架中"组织资产优先"章节](../03-eos-pipeline-system-design-ai-support/02-ai-decision-framework-eos-pipeline.md#设计原则组织资产优先)

## 📋 完整操作流程

### 前置准备（当PA文档不存在或质量不足时执行）

#### 第1步：现有产物梳理与架构分析

**目标**：全面梳理项目现有的所有产物（文档、模板、配置、代码等），为完善 PA 定义和后续逆向推导准备基础素材

> **前置说明**：本步骤仅在 PA 架构及构件需求定义不存在或质量不足以支撑后续步骤时执行。如果 PA 文档已存在且可用，可直接从第3步开始。

**涉及规范**：[产品架构规范 §2 架构定义规范](../01-eos-pipeline-system-design-specification/06-eos-pipeline-product-architecture-specification.md#二架构定义规范)
> **阅读指引**：先读产品架构规范 §2 理解架构定义和构件划分原则，再按本步骤操作从现有产物目录中提取架构草图。

**具体操作**：

1. **现有产物目录梳理**
   - 扫描项目目录结构，列出所有产物文件
   - 按架构层级分类（OR/SR/BA/SysReq/PA）
   - 标注产物类型：规范定义/设计指南/操作模板/产品数据模板/工具脚本/其他
   - 识别覆盖盲区和缺口

2. **产物类型与层级分类**
   - 将已有产物映射到五层结构的对应层级
   - 识别哪些层级的文档已存在、哪些缺失
   - 评估现有文档的质量（完整性、一致性、准则符合性）

3. **功能模块提取**
   - 从现有产物（PA定义、设计指南、任务定义等）和代码（如有）中提取功能模块
   - 按照功能模块进行分类
   - 记录每个功能的实现方式和关键参数

4. **需求分解及分析：初版架构评估**
   - **架构完整性评估**：提取的功能模块是否构成一个完整的产品架构？是否有明显缺失的构件？
   - **性能特征反推**：从现有产物的性能特征反推产品应满足的性能指标范围；**指标追溯**（每个识别出的功能点或性能特征可关联到产品架构的特定构件）

**输出**：
- 产物总览清单（产物分类 + 架构层级映射）
- 覆盖盲区标记

> **分析要点**：产物梳理中识别的每个功能模块对应一个或多个产品架构构件。功能模块清单中的条目将作为第2步中产品架构构件的输入，建议先对功能模块按 **4大分类**（规范与准则、操作指南、AI辅助文档、任务定义）进行归类，再映射到架构末级节点。

#### 第2步：产品架构文档编写

**目标**：编写符合准则的产品架构文档

> **前置说明**：本步骤仅在 PA 文档不存在或质量不足时执行。如果 PA 架构及构件需求定义已存在且可用，可直接从第3步开始。

**涉及规范**：
- [产品架构规范 §2 架构定义规范](../01-eos-pipeline-system-design-specification/06-eos-pipeline-product-architecture-specification.md#二架构定义规范)
- [产品架构规范 §3 详细定义规范](../01-eos-pipeline-system-design-specification/06-eos-pipeline-product-architecture-specification.md#三详细定义规范)
- [系统设计准则 §1.5 架构定义与详细定义的分离](../01-eos-pipeline-system-design-specification/01-eos-pipeline-system-design-standards.md#15-架构定义与详细定义的分离)
> **阅读指引**：先读产品架构规范 §2-§3 理解架构定义和详细定义格式，再参考系统设计准则 §1.5 理解分离原则。

**具体操作**：

1. **产品构件识别和定义**
   - 从产物目录和功能清单中识别产品构件
   - 按**4大分类**（规范与准则、操作指南、AI辅助文档、任务定义）确定构件类型
   - 按**构件划分原则**判断粒度是否合适（以"可由单人独立完成"且"可独立审查和验收"为双重标准）
   ```
   PA-001: [产品构件名称]（3级·架构末级节点）
   ├─ 定义：[构件的职责和功能描述]
   ├─ 构件类型：[规范与准则/操作指南/AI辅助文档/任务定义]（因产品类型而异）
   ├─ 性能指标：[从下层制品反推的性能特征]
   └─ 承接的系统需求：（待第3步完成后补充）
   ```

2. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构）**
   ```
   产品架构：系统设计流水线（0级）
   ├─ 设计规范与准则（1级）
   │  └─ 系统设计准则体系（2级）
   │     ├─ 系统设计准则总纲（3级·构件/末级）
   │     ├─ 原始需求规范（3级·构件/末级）
   │     ├─ 相关方需求规范（3级·构件/末级）
   │     ├─ 业务架构规范（3级·构件/末级）
   │     ├─ 系统需求规范（3级·构件/末级）
   │     └─ 产品架构规范（3级·构件/末级）
   │
   ├─ 设计指南（1级）
   │  └─ 场景设计指南（2级）
   │     ├─ 瀑布式设计指南（3级·构件/末级）
   │     ├─ 敏捷式设计指南（3级·构件/末级）
   │     ├─ 逆向工程设计指南（3级·构件/末级）
   │     └─ DevOps设计指南（3级·构件/末级）
   │
   ├─ AI辅助文档（1级）
   │  └─ 设计场景AI支持（2级）
   │     ├─ 通用AI辅助（3级·构件/末级）
   │     ├─ 瀑布式AI辅助（3级·构件/末级）
   │     ├─ 敏捷式AI辅助（3级·构件/末级）
   │     ├─ 逆向工程AI辅助（3级·构件/末级）
   │     └─ DevOps AI辅助（3级·构件/末级）
   │
   └─ 任务定义（1级）
      └─ 六步法核心任务（2级）
         ├─ 原始需求分析及SR架构定义（3级·构件/末级）
         ├─ SR详细定义及BA定义（3级·构件/末级）
         ├─ SysReq功能部分架构定义（3级·构件/末级）
         ├─ SR-NFR详细定义及SysReq-NFR架构定义（3级·构件/末级）
         ├─ SysReq详细定义及PA定义（3级·构件/末级）
         └─ 双向追溯验证（3级·构件/末级）
   ```


   **第二部分：需求映射视图（预留）**
   - 映射关系表暂时预留，将在第3步系统需求反向推导完成后填充
   - 显示每个 PA 构件承接了哪些 SysReq 9 级节点
   - 产品的映射关系表以"对应交付文件"替代"构件类型"，详见[产品架构规范 §2.4](../01-eos-pipeline-system-design-specification/06-eos-pipeline-product-architecture-specification.md#24-架构定义的文档结构)

3. **详细定义同步编写**
   - 每个产品构件补充**详细定义**（文档格式、内容结构、版本策略、引用关系、质量指标等）
   - 详细定义是架构末级节点的实现细化，也是后续追溯的依据
   ```
   PA-001 详细定义示例：
   ├─ 文档格式: Markdown + Mermaid
   ├─ 内容结构: 总则→五层结构→核心规则→设计要点→产品构件
   ├─ 关键条目: 五层结构含完整图示、三条规则含正例反例
   ├─ 版本策略: 语义化版本号（主版本·次版本）
   └─ 质量指标: 完整性100%、一致性0矛盾
   ```
   > **注意**：详细定义中的质量指标**不低于**所承接系统需求的指标要求。待第3步完成非功能需求识别后，应回溯标注每个 PA 构件受哪些 NFR 约束。

**输出**：
- 产品架构文档（架构定义 + 详细定义）
- 待下游步骤填充的映射关系表骨架

### 核心逆向推导（PA → OR）

#### 第3步：系统需求反向推导

**目标**：从产品架构反向推导系统需求

**涉及规范**：
- [系统需求规范 §2 架构定义规范](../01-eos-pipeline-system-design-specification/05-eos-pipeline-system-requirements-specification.md#二架构定义规范)
- [系统需求规范 §3 详细定义规范](../01-eos-pipeline-system-design-specification/05-eos-pipeline-system-requirements-specification.md#三详细定义规范)
> **阅读指引**：先读系统需求规范 §2 理解功能/非功能双架构体系，再对照 §3 的详细定义规范执行反向推导。

**具体操作**：

1. **需求分配及分析：PA→SysReq 追溯验证**
   - **分配完整性检查**：每个 PA 构件实现的每个功能，能否反推出一条合理的 SysReq 9 级场景活动？是否有功能实现了但缺少对应的 SysReq？
   - **性能指标反推**：从 PA 构件的实现性能反推 SysReq 的性能指标范围；**指标追溯**（每个 SysReq 节点的性能指标可追溯到 PA 构件的实现特征）

2. **系统需求识别**
   - 分析每个产品构件实现了什么功能
   - 识别功能性需求（做什么）
   - 识别非功能性需求（性能、安全、可靠性等）——采用 **0-6 级平行架构**，按三大类组织
   - 记录每个系统需求的定义和验收标准

3. **系统需求条目定义**

   **功能需求部分（0-9级结构）**：
   - 从 PA 构件反推 SysReq 9 级场景活动（第9级是最细粒度的活动步骤）
   - 多条 9 级活动按场景聚合为 8 级 → 7 级 → 6 级 → 5 级节点
   - **5 级节点（组件实例）** 是架构末级节点，承接后续第4步的业务架构 IPO
   ```
   SysReq-F-001: [系统功能需求名称]（5级·架构末级节点）
   ├─ 定义：[系统需求的定义]
   ├─ 类型：功能需求
   ├─ 对应的PA构件：PA-001, PA-002
   ├─ 符合性分析：[如何通过产品架构实现]
   ├─ 设计指导：[对业务架构的指导]
   └─ 性能指标：[从PA反推的性能要求]

   （详细定义：6级→7级→8级→9级分解链）
   SysReq-F-001.1.1.1.1（9级·场景活动）
   ├─ 描述：[活动步骤说明]
   ├─ 分配到：PA-001（产品构件）
   └─ 验收标准：[验收条件]
   ```

   **非功能需求部分（0-6级平行架构）**：
   - 从 PA 构件的非功能特征反推系统非功能需求
   - 按三大类分类：质量特性 / 环境适应性 / 可实现性
   ```
   SysReq-NFR-001: [系统非功能需求名称]（5级·架构末级节点）
   ├─ 定义：[非功能需求的定义]
   ├─ 类型：[性能/安全/可用性等]
   ├─ 量化指标：[从PA构件反推的具体数值、单位、条件]
   ├─ 符合性分析：[如何通过产品架构实现]
   └─ 约束条件：[对产品架构的约束]
   ```

4. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构）**
   - 功能分支采用 **0-9 级层级结构**（5 级为架构末级，9 级为详细定义末级）
   - 非功能分支采用 **0-6 级平行架构**（独立分类组织）

   **第二部分：需求映射视图**
   - 显示每个 SysReq 末级节点对应的 PA 构件
   - 同时更新第2步 PA 文档中的映射关系表

5. **同步设计**
   - 完成 SysReq 定义后，更新第2步 PA 文档的"承接系统需求"字段
   - 同步在下层（业务架构）建立 5 级节点占位符（为第4步做准备）

**输出**：
- 系统需求文档（架构定义 + 详细定义）
- 需求映射关系表（PA ↔ SysReq）

#### 第4步：业务架构反向推导

**目标**：从系统需求反向推导业务架构

**涉及规范**：
- [业务架构规范 §2 架构定义规范](../01-eos-pipeline-system-design-specification/04-eos-pipeline-business-architecture-specification.md#二架构定义规范)
- [业务架构规范 §3 IPO模型](../01-eos-pipeline-system-design-specification/04-eos-pipeline-business-architecture-specification.md#三ipo模型)
- [业务架构规范 §4.3 去重机制](../01-eos-pipeline-system-design-specification/04-eos-pipeline-business-architecture-specification.md#43-去重机制)
> **阅读指引**：先读业务架构规范 §2 理解架构定义和 IPO 模型，再参照 §4.3 的去重机制执行 IPO 去重。

**具体操作**：

1. **需求分配及分析：SysReq→BA 追溯验证**
   - **分配完整性检查**：从 SysReq 5 级节点是否能推导出合理的业务操作过程（IPO）？是否存在 SysReq 实现了但 BA 中说不清业务语义的情况？
   - **性能覆盖检查**：业务层面的性能指标是否覆盖了 SysReq 的性能要求；**指标追溯**（每个 BA IPO 的性能指标可追溯到 SysReq 的性能指标）

2. **业务架构识别**
   - 分析系统需求的业务含义
   - 识别业务组件和业务流程
   - 每个业务组件使用 **IPO 模型**（Input-Process-Output）三维定义

3. **IPO 去重**
   - 逆向推导时，多个 SysReq 节点可能对应相同的业务操作
   - 采用**三阶段工作流程**（详见[业务架构规范 §4.1](../01-eos-pipeline-system-design-specification/04-eos-pipeline-business-architecture-specification.md#41-三阶段工作流程)）：先快速评估创建占位符和骨架映射，再填充/更新IPO内容并去重，最后验证和冻结
   - **去重三级策略**：复用→改进→新增（优先复用已识别的 IPO）—— 这是**组织资产优先原则**（详见[关键概念](#组织资产优先原则)）在 BA 层的直接应用
   - 去重后的 IPO 清单才是最终的业务架构输出

4. **业务架构条目定义**
   ```
   BA-001: [业务组件名称]（5级·架构末级节点）
   ├─ 定义：[业务组件的定义]
   ├─ Input（输入）：[输入的具体形式、数据结构、来源]
   ├─ Process（处理）：[规则、算法、步骤、约束条件]
   ├─ Output（输出）：[输出的具体形式、数据结构、验收标准]
   ├─ 对应的SysReq：SysReq-F-001, SysReq-F-002
   ├─ 符合性分析：[如何通过SysReq实现该业务操作]
   └─ 设计指导：[对系统需求的指导]
   ```

   > **业务架构仅包含架构定义（IPO清单），无详细定义**：业务架构的架构末级节点（IPO）自身承担"需求"角色，直接映射到系统需求。这与 SysReq 和 PA 不同——后两者既有架构定义也有详细定义。

5. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构，0-5级）**
   ```
   0级：架构根节点
   1级：业务域
   2级：端到端项目
   3级：项目业务对象架构
   4级：工作包（WBS）
   5级：业务组件（承接层）★ 末级
   └─ BA-001: [IPO名称]
        ├─ 定义：[业务组件定义]
        └─ Input / Process / Output
   ```

   **第二部分：需求映射视图**
   - 显示每个 BA IPO 对应的 SysReq 5 级节点
   - 同时更新第3步 SysReq 文档中的"承接 BA"字段

**输出**：
- 业务架构文档（架构定义 / IPO清单 + 映射关系表）
- 需求映射关系表（SysReq ↔ BA）

#### 第5步：相关方需求反向推导

**目标**：从业务架构反向推导相关方需求

**涉及规范**：
- [相关方需求规范 §2 架构定义规范](../01-eos-pipeline-system-design-specification/03-eos-pipeline-stakeholder-requirements-specification.md#二架构定义规范)
- [相关方需求规范 §3 详细定义规范](../01-eos-pipeline-system-design-specification/03-eos-pipeline-stakeholder-requirements-specification.md#三详细定义规范)
> **阅读指引**：先读相关方需求规范 §2 理解架构末级节点定义和分类体系，再对照 §3 的详细定义格式执行反向推导。

**具体操作**：

1. **需求分配及分析：BA→SR 追溯验证**
   - **分配完整性检查**：从 BA IPO 能否反推出清晰的相关方需求描述？业务操作是否都有对应的相关方诉求？
   - **性能覆盖检查**：SR 层面的性能指标是否覆盖了 BA 的性能要求；**指标追溯**（每个 SR 节点的性能指标可追溯到 BA IPO 的性能指标）

2. **相关方需求识别**
   - 分析业务架构的相关方需求
   - 功能需求（SR-F）和非功能需求（SR-NFR）**分别处理**
   - 非功能需求不经过 BA 层，从产品架构及下层设计制品的非功能特征反推

3. **相关方需求条目定义**

   **功能需求部分（SR-F）**：
   ```
   SR-F-001: [相关方需求名称]（架构末级节点）
   ├─ 定义：[需求的定义]
   ├─ 承接的原始需求末级：（待第6步完成后补充）
   ├─ 符合性分析：[如何通过业务架构实现]
   ├─ 设计指导：[对业务架构的指导]
   └─ 性能指标：[从BA继承的性能要求]

   （详细定义末级节点）
   SR-F-001.01: [需求的分解条目]
   ├─ 描述：[详细说明]
   ├─ 分配到：BA-001（业务架构IPO）[1:1映射]
   ├─ 验收标准：[验收条件]
   └─ 符合性：[如何满足上层架构末级节点 SR-F-001]
   ```

   **非功能需求部分（SR-NFR）**：
   ```
   SR-NFR-001: [相关方非功能需求名称]（架构末级节点）
   ├─ 定义：[非功能需求的定义]
   ├─ 量化指标：[具体数值、单位、条件]
   ├─ 测量方法：[如何验证]
   └─ 分配到：SysReq-NFR-001（系统非功能需求）
   ```

4. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构）**
   - 功能需求分类 + 非功能需求分类（明确分离）

   **第二部分：需求映射视图**
   - 显示每个 SR 架构末级节点对应的 BA IPO
   - 同时更新第4步 BA 文档中的"承接 SR"字段

5. **详细定义的双重身份**
   - SR 的详细定义末级节点既是本层架构的细化（向内），又是 BA 的需求输入（向下传递）
   - 确保每个 SR 详细定义末级都能分配到**唯一**的 BA IPO（1:1 约束）

**输出**：
- 相关方需求文档（架构定义 + 详细定义）
- 需求映射关系表（BA ↔ SR）

#### 第6步：原始需求反向推导

**目标**：从相关方需求反向推导原始需求

**涉及规范**：[原始需求规范 §3 分解规范](../01-eos-pipeline-system-design-specification/02-eos-pipeline-original-requirements-specification.md#三分解规范)
> **阅读指引**：先读原始需求规范 §3 理解分解原则和停止准则，再按本步骤从 SR 反向推导原始需求。

**具体操作**：

1. **需求分配及分析：SR→OR 追溯验证**
   - **分配完整性检查**：从 SR 架构末级节点能否合理反推出原始需求的表述？每条 SR 是否有真实的业务来源？
   - **指标一致性检查**：原始需求的性能指标是否与 SR 的量化指标一致；**指标追溯**（原始需求的指标可从 SR 的量化指标追溯）

2. **原始需求识别**
   - 分析相关方需求的来源
   - 将 SR 架构末级节点聚合为原始需求条目
   - 识别原始需求的末级需求并分配到 SR 架构末级节点

3. **原始需求条目定义**
   ```
   OR-001: [原始需求名称]
   ├─ 规范化描述：[经过规范化处理的描述]
   ├─ 末级需求分解：
   │  ├─ 末级需求 1.1 → SR-F-001
   │  ├─ 末级需求 1.2 → SR-F-002
   │  └─ 末级需求 1.3 → SR-NFR-001
   ├─ 类型：[功能需求/非功能需求]
   └─ 优先级：[P0/P1/P2/P3]
   ```

4. **原始需求文档结构**
   - 原始需求条目清单（规范化后的简洁格式）
   - 末级需求分解清单
   - 末级需求 → 相关方需求的分配关系

5. **同步设计**
   - 完成 OR 定义后，更新第5步 SR 文档中的"承接原始需求末级"字段
   - 至此，所有 5 层文档的映射关系均已闭环

**输出**：
- 原始需求文档（含末级需求分解）
- 完整的映射关系表（OR → SR）

### 验证与发布

#### 第7步：完整性和一致性验证

**目标**：验证逆向工程结果的完整性和一致性

**涉及规范**：
- [系统设计准则 §5 验收标准](../01-eos-pipeline-system-design-specification/01-eos-pipeline-system-design-standards.md#五验收标准)
- [系统设计准则 §6 检查清单](../01-eos-pipeline-system-design-specification/01-eos-pipeline-system-design-standards.md#六检查清单)
- [系统设计准则 §3.2 三条核心规则](../01-eos-pipeline-system-design-specification/01-eos-pipeline-system-design-standards.md#32-三条核心规则)
> **阅读指引**：完整阅读系统设计准则 §3.2（三条核心规则）、§5（验收标准）和 §6（检查清单），以验收标准为准绳执行全链路验证。

**具体操作**：

1. **全链路合并验证**
   - **分配覆盖及一致性汇总**：逐层验证映射覆盖率 100%，从 OR→SR→BA→SysReq→PA 每个节点都有明确的承接/分配关系
   - **性能指标追溯汇总**：性能指标从 OR 到 PA 逐层继承验证，一致性确认；**可实现性评估**（产品架构能否满足所有性能要求）；**指标追溯**（端到端性能指标可逐层追溯）

2. **1:1 分配约束验证**
   - 验证每条需求都分配到唯一的方案架构末级节点
   - 范围：OR 末级→SR、SR 详细定义末级→BA IPO、SysReq 9 级→PA
   - 发现遗漏或重复的映射

3. **N:1 承接支持验证**
   - 验证每个方案架构末级节点的承接数量合理（2-5 条/节点）
   - 发现过度分解或过度聚合的情况

4. **双向追溯验证**
   - **正向**：从原始需求末级 → 产品架构（自顶向下验证）
   - **反向**：从产品架构 → 原始需求末级（自底向上验证——这是逆向工程的核心产出）
   - 验证所有超链接的有效性

5. **符合性分析完整率验证**
   - 所有方案节点（SR/BA/SysReq/PA）是否都有符合性分析
   - 每个符合性分析是否清晰说明了如何满足所承接的需求

6. **同步设计一致性验证**
   - 每层的架构定义与详细定义之间的映射是否一致
   - 各层之间没有"推导了上层但漏了下层"的情况

**输出**：
- 验证报告（完整性和一致性分析）
- 问题清单和改进建议

## 🔗 跨步骤一致性检查

以下检查项覆盖所有步骤，确保逆向推导的整体一致性：

- [ ] **映射流向一致**：分配方向统一向下（↓），承接方向统一向上（↑）
- [ ] **术语一致性**：全文使用统一定义（架构末级节点/详细定义末级节点/1:1分配约束/N:1承接支持）
- [ ] **层级结构完整**：每层从根节点到末级节点的树形结构完整（BA 0-5级，SysReq 功能0-9级/非功能0-6级）
- [ ] **1:1约束满足率100%**：每条详细定义末级分配到唯一的架构末级节点
- [ ] **N:1承接合理性**：每个架构末级节点承接 2-5 条详细定义末级为合理（1条需考虑合并，6+条需拆分）

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| PA 文档质量不足 | 反向推导的上层需求不准确 | 先执行前置准备步骤（第1-2步）完善PA |

| 需求信息遗漏 | 反向推导的需求不完整 | 进行用户访谈，补充缺失的需求 |
| 映射关系复杂 | 难以建立清晰的映射 | 严格执行1:1约束和N:1承接规则 |
| 性能指标难以逆向 | 无法确定原始指标 | 从现有设计制品的性能特征反推，标注"反推值" |
