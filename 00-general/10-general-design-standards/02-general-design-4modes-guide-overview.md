# 四种设计方法的通用指南概述
**General Design Guide Overview**

**文档版本**：v1.0
**创建日期**：2026-05-18
**用途**：描述系统设计的四种基本方法及选择框架，不依赖具体产品业务领域。

---

## 一、四种设计方法

本方法论支持四种系统设计方法，适用于不同的产品开发生命周期模型或场景：

| 方法 | 适用场景 | 典型周期 | 设计方向 |
|------|---------|---------|---------|
| **瀑布式** | 原始需求全部齐备，从零开始的完整系统设计 | 完整周期 | 正向：OR → SR → BA → SysReq → PA |
| **敏捷式** | 已有系统设计文档，增量新增或修订需求 | 短周期迭代 | 正向增量：分析影响 → 增量更新 |
| **逆向工程** | PA设计已完成，但上游系统设计文档缺失需补全 | 完整周期 | 反向：PA → OR（系统级逆向） |
| **DevOps** | 在产产品需快速修复或小变更 | 极短 | 最小化变更：定位 → 修复 → 验证 |

### 选择决策

```
产品处于什么阶段？

┌─ 已开发完成，但文档不全？
│  ├─ PA已存在，上游文档缺失 → 系统设计逆向工程
│  │  └─ 从PA反向推导系统设计文档
│  └─ PA未建立，仅有代码/图纸 → 先重建PA再执行逆向
│     └─ 从代码/图纸提取PA架构，再反向推导
│
├─ 只有原始需求，需要从零开始？
│  └─ 方法：瀑布式/完整正向设计
│     └─ 逐层正向推导（OR → SR → BA → SysReq → PA）
│
├─ 已有完整系统设计，需要增加新需求？
│  └─ 方法：敏捷式/增量设计
│     └─ 分析影响范围 → 增量更新受影响的文档
│
└─ 产品已上线运行，需要快速修复？
   └─ 方法：DevOps/快速修复
      └─ 快速评估影响 → 最小化变更 → 快速验证
```

> **关于逆向工程的两种类型**：
> - **系统级逆向工程**（属于阶段1 系统设计）：从已有的产品架构（PA）及构件需求定义出发，反向推导完整的五层系统设计文档（PA → SysReq → BA → SR → OR）。这是本方法论及逆向工程详细指南所覆盖的范围。
> - **构件级逆向工程**（属于阶段2 构件开发）：从单个构件的代码或图纸出发，反向推导组件的详细设计、概要设计、组件需求。这属于构件开发阶段的方法，不在本指南范围内。

## 二、四种方法的特征对比

| 维度 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|------|--------|--------|---------|--------|
| **起点** | 原始需求 | 现有系统设计 | PA及构件需求定义 | 生产环境问题 |
| **终点** | 产品架构 | 更新的系统设计 | 完整系统设计 | 快速修复/发布 |
| **设计方向** | 正向完整 | 正向增量 | 反向推导 | 最小正向变更 |
| **需求状态** | 全部齐备 | 增量变化 | 需反向提取 | 明确且紧急 |
| **风险** | 需求不清晰 | 影响范围扩散 | 理解偏差 | 副作用 |

## 三、通用约束与质量指标

参见 [通用设计准则](./01-general-design-standards.md) 中定义的：
- **三条核心规则**：1:1分配约束、N:1承接支持、双向追溯
- **验收标准**：架构定义质量、详细定义质量、映射关系质量、追溯完整性
- **检查清单**：分解充分性、映射关系、符合性分析、双向追溯、文档规范化

## 四、各方法在产品中的使用

本指南仅提供方法框架。各产品的具体设计指南在此基础上填充：

- **场景适配**——根据产品类型确定适用的项目场景
- **操作流程**——各方法的具体步骤和操作细节
- **角色分工**——产品开发中不同角色的职责划分
- **示例说明**——结合产品业务领域的操作示例
