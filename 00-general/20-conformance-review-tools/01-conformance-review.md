## 核心方法论

> **核心思路**：一次只用一个语义段落去审视全文。每轮只带一个"判断透镜"，发现违规立即记录，完成后进入下一段。

### 依据条目即时生成

审查清单不是预设的，而是在审查启动时根据加载的依据文档即时生成：

1. 审查员加载依据文档（如准则）
2. 审查员将依据文档按语义构成划分为若干"语义段落"
3. 审查员逐一判断每个段落是否与目标文档相关：
   - 相关 → 列为一轮审查
   - 不相关 → 跳过
   - 本层专用（锚段落）→ 标记为最核心轮次
4. 生成审查轮次清单，逐轮执行

### 语义段落划分原则

| 原则 | 说明 |
|------|------|
| **先浏览再划分** | 先全文浏览依据文档，理解其语义构成，再基于理解划分段落——不拘泥于章或节 |
| **一段只带一个判断透镜** | 每段包含 1-3 条紧密相关的核心约束，不跨类型混用。同一章内若包含多种类型约束（如定义+格式），拆成多个段落；多节若讲同一类约束，合并为一个段落 |
| **独立可审查** | 读完整段即可判断目标文档中哪些内容违反本段规则，不依赖其他段落的结论 |
| **分类型标注** | 标注该段是定义规则(D)、流程规则(P)、格式规则(F)，还是其他类型 |

### 执行入口

审查的具体执行流程（加载依据→划分段落→策划审查任务→逐项执行→覆盖确认→出报告→策划修复任务→用户确认→执行修复→独立完整性审查[可选]）由 Skill 驱动。在 Claude Code 中运行：

```
/conformance-review
```

Skill 会按语义段落逐轮审查，并调用审查工具脚本执行机械检查。审查报告只输出发现，不执行修改；修复任务在审查完成后另行策划并经用户确认后执行。

### 独立完整性审查模板（修复后可选执行）

修复完成后，对目标文档进行独立完整性审查，关注文档自身的内部质量。**本轮不加载依据文档**。

> **与符合性审查的关系**：独立完整性审查只关注文档的内在质量，不验证对依据文档的符合性。以下两个常见维度因不适合独立完整性审查而被排除：
> - **完整性（已排除）**：判断"文档应该有哪些内容"需要对照依据文档中的需求定义，这本质上是一个符合性审查问题，与"不加载依据文档"的前提矛盾
> - **可操作性（已排除）**：可操作性是其他维度（语义清晰、逻辑自洽、前后一致）的**涌现属性**，没有独立的审查对照表可操作，且与多个维度大面积重叠

| 审查维度 | 检查要点 | 示例问题 |
|---------|---------|---------|
| **前后一致性** | 术语/编号/格式在全文是否统一，修复后是否存在漏改 | 编号是否混用（部分 .01 部分 .1） |
| **逻辑自洽性** | 章节间语义无矛盾 | 一处说"所有节点需编号"，另一处示例使用了无编号节点 |
| **语义清晰性** | 表述是否精确，是否使用了不可量化的模糊词 | "处理效率高"中的"高"无法验证 |
| **可理解性** | 概念表述是否清晰，是否需要补充说明或加注 | BA 节点使用名称而非编号，读者可能困惑 |
| **简洁性** | 内容是否精简，无冗余 | 同一内容是否在多处重复描述 |
| **可维护性** | 是否便于更新，修改时的联动影响范围 | 产品实例过时后规范失效，一处修改需同步更新多处 |

**操作方式**：
- 不加载依据文档，仅通过通读目标文档全文进行审查
- 发现记录格式同 Step 4 的 ❌/⚠️ 项
- 发现的独立审查问题按修复阶段流程执行（修复任务清单 → 用户确认 → 执行修复）

---

### 发现的处理方式

处理遵循"先输出发现，等确认后再修改"的总原则：

**审查阶段（仅发现，不修改）**：
- ❌ 项记入审查报告，不在审查轮次中直接修改
- ⚠️ 项记入审查报告，由审查者判断是否需要在修复阶段处理
- **禁止跨轮累积语义**——在本轮只查本轮的规则，不把上一轮的发现带到本轮

**修复阶段（审查报告后单独执行）**：
- 审查完成后，策划修复任务清单（将 ❌/⚠️ 项转化为具体修改任务）
- 输出 TodoWrite 修复清单，**用户确认后开始执行**
- 按清单逐项执行修改，每完成一项标记为 completed

**独立完整性审查阶段（修复后可选执行）**：
- 修复完成后，可选执行独立完整性审查，关注文档自身的内部质量
- 不加载依据文档，仅通过通读全文审查前后一致性、逻辑自洽性、可理解性、语义清晰性
- 发现的独立审查问题按修复阶段流程处理
