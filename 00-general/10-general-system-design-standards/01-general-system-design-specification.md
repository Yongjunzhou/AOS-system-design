## 第一部分：方法论与设计规则

### 〇、方法论逻辑总览

本方法论的每一层结构都从一条基础定义出发：

> **设计** = 将对客体的需求转换为对其更为详细的需求的一组过程（ISO 9000:2015, 3.4.8）

其中的"转换"在本方法论中被拆解为三个精确操作——分解、分配、符合性分析；"更为详细"则确定了持续的终止条件是"实现或可实现"：

> **设计 = 对需求的持续分解、分配和符合性分析，直至实现或可实现**

再特化为系统设计——"实现或可实现"在此语境中指"产出可实现的产品架构（PA）"，形成第一块基石：

> **系统设计 = 对需求的持续分解、分配和符合性分析，直至产出可实现的产品架构。**

完整推导链（先验约束 → ISO 9000定义 → 本方法论基石定义 → 概念落地 → 五层链）见附录A（方法论底层推导链）。


### 一、基本术语与概念

本章定义全文使用的核心术语体系，是后续所有方法论述的共同语言基础。术语按基本概念、需求详解、文档结构、文档结对、设计活动与设计规则六个主题展开，从最底层的元概念出发，经由需求解析、文档组织和结对协作，逐步深入到设计活动的操作逻辑与约束规则，覆盖从需求到方案的完整设计语汇。

#### 1.1 基本概念

定义质量、需求、方案、架构、需求/方案相对五个元概念的内涵及其相互关系，作为贯穿全文方法论所有原则和设计活动的前提和基础。"设计"的定义已置于§〇（方法论逻辑总览），此处不再重复。

| 术语 | 定义 | 详见 |
|------|------|------|
| **质量** (Quality) | 一组固有特性满足需求的程度（ISO 9000:2015, 3.6.2 / GB/T 19000-2016, 3.6.2）。本方法论中，质量特指系统设计质量——各层设计输出（方案）满足其所承接上层需求的程度，通过符合性分析（语义符合性 + 指标符合性）来度量和保证 | §3.3, §七 |
| **需求** (Requirement) | 明示的、通常隐含的或必须履行的需要或期望（ISO 9000:2015, 3.6.4）。本方法论中，需求指对系统应具备的功能或应满足的质量约束的描述，由需求描述和性能指标两部分构成——不可衡量则不成为需求。**类型维度**：方案的对立面——方案回答 How，需求回答 What。**角色维度**：设计链中需要被下层方案承接的设计输入。需求的两种基本类型（功能需求 FR / 非功能需求 NFR）见 §1.2 展开定义 | §1.2, ISO 9000:2015 |
| **方案** (Solution) | 在特定上下文中满足需要的一种具体方式（BABOK® v3 / BACCM）。本方法论中，方案指对上层需求的满足方式，承载于方案文档之中；方案文档由架构定义子文档和详细定义子文档组成。**类型维度**：需求的对立面——需求回答 What，方案回答 How。**角色维度**：设计链中对上层需求的满足输出；其详细定义末级条目在向下传递时承担需求角色，这是"文档的双重属性"的机制来源 | §2.2 |
| **架构** (Architecture) | 系统在其环境中的基本概念或属性，体现于其元素、关系以及设计和演化的原则之中（ISO/IEC/IEEE 42010:2011, 3.2）。本方法论中，架构特指各层设计活动的基本组织形态——以树形结构组织设计内容，其末级节点是所承接上层需求的**解决方案实体** | §2.2 |
| **需求/方案相对** (Requirement-Solution Relativity) | "需求"和"方案"不是文档的固有标签，而是文档在设计链中相对于相邻文档扮演的角色。同一文档在不同设计层位上兼具不同身份——向上是方案（满足上层需求），向下是需求（驱动下层设计）。本条确立了需求与方案作为相对角色而非固有标签的元概念关系，是对"需求"和"方案"两个独立概念的必要补充 | §2.1 |

#### 1.2 需求详解

定义需求构成要件——需求描述与性能指标不可分割——以及功能需求与非功能需求的分类标准，阐明两者在设计链中的不同分配路径。

需求与方案是相对的（详见 §1.1 "需求/方案相对"）。以下分类不受文档在设计链中扮演"需求"还是"方案"角色的影响。

| 术语 | 定义 | 详见 |
|------|------|------|
| **需求性能指标** (Requirement Performance Metric) | 每个需求由"需求描述"和"性能指标集合"两部分构成——不可衡量则不成为需求。指标定义了需求被满足的程度和测量方式，分为定量指标（响应时间 < 200ms、可用性 99.9% 等）和定性可验证指标（通过认证、满足标准等）两类。指标在分解阶段有三种处理场景：可拆分、下放、暂不分解 | §3.1 |
| **功能需求** (Functional Requirement, FR) | 系统必须执行的功能、行为或服务——系统对输入的反应和产生的输出（ISO/IEC/IEEE 29148:2018）。功能需求经业务架构（BA）到达系统需求（SysReq），在 BA 层以 IPO（输入→处理→输出）模型描述 | §五 |
| **非功能需求** (Non-Functional Requirement, NFR) | 系统运行的条件或系统必须拥有的品质——包括性能、接口约束、质量属性、标准合规等（ISO/IEC/IEEE 29148:2018）。非功能需求不经 BA，走 SR-NFR → SysReq-NFR → PA 的平行路径 | §1.2, §五 |
| **FR 与 NFR 的关系** (Relationship Between FR and NFR) | 两者是并列关系：NFR **不从 FR 派生**，是独立的需求条目；FR 自带的性能指标与 NFR 自身的性能指标并列存在；两者在 SR 层以后走不同的分配路径——FR 经 BA，NFR 不经 BA | §五 |

> **功能需求完整性保障机制**：FR 之所以必须经 BA 层（§五），是因为 BA 提供了功能需求完整性的系统性保障方法——BA 通过穷举各类用户角色使用目标产品的全部操作场景，确保由此推导出的系统功能需求（SysReq-F）集合是完整的。NFR 在当前方法框架中没有等价于 BA 的系统性保障方法，其完整性主要依赖开发方的专业能力和领域积累——因此 NFR 不经 BA，走 SR-NFR → SysReq-NFR → PA 的平行路径。

#### 1.3 文档结构

一份需求/方案文档由架构定义子文档和详细定义子文档两个独立子文档构成，两者职责分离、独立演化，是后续文档结对和需求分配的组织基础。架构定义子文档以树形结构组织内容并建立对上层需求的映射；详细定义子文档以平铺列表细化分解架构末级节点，其末级条目构成下一结对的需求。两者共同构成该层的完整文档。**特例**：OR 是纯需求文档（无架构定义、无详细定义）；BA 是纯方案文档（仅有架构定义，无详细定义）。

##### 1.3.1 架构定义子文档

架构定义子文档内包含四类概念——其自身定义、树中三种节点的分类与职责。

| 术语 | 定义 | 详见 |
|------|------|------|
| **架构定义子文档** (Architecture Definition Document) | 需求/方案文档的子文档之一。以树形结构组织文档层级，以架构末级节点为需求映射的落脚点，建立对上层需求的 N:1 承接关系 | §2.2 |
| **纯分类节点** (Pure Classification Node) | 架构定义子文档中仅用于对架构进行分类和组织的非末级节点，无实体含义，不承载需求映射。例如 SR 层"功能需求"分类分支 | §2.2.1 |
| **集成实体节点** (Integrated Entity Node) | 架构定义子文档中代表由其子节点功能集成的真实实体的非末级节点——节点本身在架构中有意义，但需求映射仍落在其下辖的末级节点上。例如 BA 层的业务域（如"生产管理"下有"生产计划""工单执行"）和 PA 层的服务组（如"订单服务"下有"创建订单""查询订单"） | §2.2.1 |
| **架构末级节点** (Architecture Leaf Node) | 架构定义子文档树形结构的末梢节点，是所承接上层需求的**解决方案实体**。与纯分类节点（仅用于组织层级）及集成实体节点（代表由子节点集成的整体）的本质区别在于：末级节点承载上层需求，非末级节点不直接承接需求映射。生成基于组织已有架构资产，遵循"复用 → 改进 → 新增"优先级（§2.4.1）。颗粒度标准：一个节点承接的条目数应合理（N:1），不过大、不过小、语义独立 | §2.2, §2.4.1；各层具体形态及识别方法详见§四 |

##### 1.3.2 详细定义子文档

详细定义子文档内包含两类概念——其自身定义，以及其条目单元的说明。

| 术语 | 定义 | 详见 |
|------|------|------|
| **详细定义子文档** (Detailed Definition Document) | 需求/方案文档的子文档之二。对架构末级节点进行详细分解，使每条末级条目可分配到后续下层方案架构定义子文档中唯一一个架构末级节点 | §2.2 |
| **详细定义末级节点** (Detailed Definition Leaf Node) | 对架构末级节点逐层分解后达到的原子粒度条目。分解终止标准：该条目已细化到可依 1:1 约束分配到下层方案架构定义子文档中唯一一个架构末级节点。它既是本层详细定义的输出末端，又是下一设计文档结对中需求方的源头 | §2.2 |

> **注**：本文档使用的"架构定义"与系统工程"架构定义"（Architecture Definition）含义一致——就是把**需求/方案的内容整理成树形结构，树上每个末级节点要标明承接了上层的哪条需求**。本文档使用的"详细定义"也对应系统工程中的"详细设计"（Detailed Design）概念——对树上每个末级节点，再**往下逐层细化拆分，拆到每条小条目都能唯一对应到下一层树的某个末级节点为止**。
> 
> 与系统工程的主要区别有两点：
> 1. **结对模式沿五层链路逐层重复应用**：架构定义 + 详细定义不是只在系统级做一次，而是在 SR、BA、SysReq、PA 每一层独立执行完整的设计文档结对。
> 2. **终止标准不同**：系统工程中详细设计走到具体的图纸、代码、工艺级别；本文档的详细定义仅做到**可分配粒度**——以详细定义末级节点能 1:1 分配到下一层架构末级节点为终止标准。

#### 1.4 文档结对

文档结对是系统设计的基本单元。文档在设计链中扮演两种角色——需求方与方案方——由上层详细定义末级与下层架构末级节点间的配对机制构成。

| 术语 | 定义 | 详见 |
|------|------|------|
| **需求文档** (Requirements Document) | 文档在结对中扮演需求方角色时即为需求文档，描述系统应该做什么（What），包含功能需求和非功能需求。本方法论中，需求文档由架构定义子文档和详细定义子文档两部分构成 | §2.2 |
| **方案文档** (Solution Document) | 文档在结对中扮演方案方角色时即为方案文档，描述如何满足需求（How），涵盖功能需求和非功能需求的实现方案。本方法论中，方案文档由架构定义子文档和详细定义子文档两部分构成 | §2.2 |
| **设计文档结对** (Design Document Pairing) | 系统设计的基本组织单元。配对发生在上层需求详细定义末级条目与下层方案架构定义子文档的架构末级节点之间——上层需求详细定义末级条目充当需求，下层方案架构定义架构末级节点充当方案。有需求必有方案，有方案必有需求，保证设计链的完整闭合 | §2.3 |
| **详细定义的双重身份** (Dual Identity of Detailed Definition) | 详细定义子文档中的每个末级节点具有双重身份：向内是对当前层架构末级节点的细化（定义具体职责和实现策略），向下是作为需求输入传递给下层方案（其架构末级节点需要承接这些条目）。这一身份使其成为上下层系统设计链路的衔接点——本层设计的输出末端，同时是下层设计的需求源头。双重身份也解释了 1:1 分配约束的必要性：若一条条目需分配到下层多个方案架构末级节点，说明本层分解尚未充分 | §2.2 |
| **结对设计** (Paired Design) | 设计文档结对的动态工作原则：在设计文档结对中，上层需求详细定义与下层方案架构定义应结对同步设计而非串行执行。操作方法：先快速约定下层方案架构的树形结构骨架（分类节点及末级节点标题）并建立初步映射（骨架阶段），再由上下层各自并行填充——上层完善需求详细定义，下层完善方案架构节点四要素（填充阶段）。目的：确保上层分解到末级时下层已有可承接的架构末级节点，避免返工 | §2.3, §1.4 |

#### 1.5 设计活动

定义系统设计的三个核心活动——需求分解、需求分配与符合性分析——以及组织这些活动的五层设计链路，三者依次执行构成完整的设计流程。

| 术语 | 定义 | 详见 |
|------|------|------|
| **系统设计链路** (System Design Chain) | 系统设计阶段中从原始需求到产品架构的完整设计链条，共五层：OR → SR → BA → SysReq → PA。链路由四个文档结对依次衔接而成：① OR 条目级需求 ↔ SR 架构定义；② SR 详细定义末级 ↔ BA 架构（IPO）；③ BA IPO（去重后）↔ SysReq 架构定义；④ SysReq 详细定义末级 ↔ PA 架构定义 | §五, §1.3-§1.4；各层节点类型详见§4.1总览表 |
| **需求分解** (Requirements Decomposition) | 三个核心设计活动之首（另两项为需求分配和符合性分析）。分解分为语义和性能指标两个维度同步进行：**语义分解**将父需求的功能描述无重叠无遗漏地切分为子需求，执行覆盖完整性、不超出性、逻辑一致性、约束继承四项自检；**指标分解**将父需求的性能指标分配到子需求，先识别父指标与子指标之间的运算关系或算法，据此将父需求指标分解至子需求指标 | §3.1 |
| **需求分配** (Requirements Allocation) | 三个核心设计活动的第二个（前序为需求分解，后续为符合性分析）。对需求分解产出的详细定义末级节点，按 1:1 约束逐一映射到下层方案架构定义子文档的架构末级节点，建立设计文档结对的映射关系。分配完成后执行三项**分配自检**：全覆盖（所有末级节点均已分配）、1:1（每节点分配到唯一目标）、N:1 合理性（目标节点承接条数合理） | §3.2 |
| **符合性分析** (Compliance Analysis) | 三个核心设计活动的第三个（前序为需求分解和需求分配）。验证架构定义子文档的架构末级节点如何充分满足所承接的上层需求集合。分两个维度：语义符合性（方案定义是否覆盖需求核心语义）和指标符合性（方案指标不降级、可验证） | §3.3 |

#### 1.6 文档体系跨层引用原则

定义本方法论文档体系中通用方法论层与各产品层之间的引用规则，确保依赖关系清晰可维护。

本体系的文档分布在通用方法论层和各产品层中：通用方法论层提供方法框架和格式定义，各产品层在此基础上定义各自的专有规则和产品数据。

层间引用遵循以下规则：

**规则一：方法论开放原则**

通用方法论层的方法论文档（本文档 §一~§十三 + 附录）是本体系的**基础设施**，而非上层产品的私有依赖。任何产品层的规范、指南、任务定义均可直接引用通用方法论，无需经过中间层转发。

- ✅ 产品层规范直接引用本规范（通用设计规范）
- ✅ 产品层任务定义引用本规范的操作规则

**规则二：产品专有内容链式引用原则**

产品专有内容（本层准则中的实例化规则、产品数据、场景指南、任务定义）遵循链式依赖——只引用通用方法论层或本层内容，不跨层引用其他产品层的专有内容。

- ✅ 产品A的场景指南引用产品A准则（本层）
- ❌ 产品A的场景指南不应引用产品B的产品数据（跨层专有内容）
- ❌ 产品B的准则不应引用产品A的任务定义（跨层专有内容）

**规则三：引用标注惯例**

跨层引用通用方法论时，应在引用处标注来源层级（如"通用方法论"或路径前缀），明确表明引用的是方法框架而非产品专有内容，防止读者混淆。

- ✅ 产品层准则行内引用："通用方法：OR 预处理见[通用准则 §6.3]……"
- ✅ 引用列表中明确标注："[通用指南 v2.1]（通用方法论框架）"

> **设计准则的构成**：各产品的设计准则由方法论通用层和产品特有层两部分构成——方法论通用层内容与本文档一致，可在产品间复用；产品特有层定义各自的产品约束，需要独立定义。

**规则四：本规范内部配套关系**

本文档由两部分组成——**第一部分：方法论与设计规则**（§一~§九 + 附录A~B）和**第二部分：四种设计模式**（§十~§十三 + 附录C~D）。

- 第一部分定义完整的方法论核心：五层结构、各层架构末级节点识别方法、业务架构设计方法、三个核心设计活动、验收标准与检查清单
- 第二部分是第一部分在四种设计场景（瀑布式/敏捷式/逆向工程/DevOps）上的方法框架展开，不重复准则已定义的内容
- 产品层在实例化时，应同时引用本文档的两个部分——第一部分提供方法规则，第二部分提供场景框架

两部分是分工关系而非替代关系，产品层不可只引用其一而忽略另一。

#### 1.7 产品类型与产品架构形态

定义产品类型分类及其对 PA 架构节点形态的影响，为各产品层准则实例化 PA 节点类型提供通用框架。

##### 1.7.1 产品类型的三种基本类别

| 类别 | 典型特征 | PA 节点形态 | 示例 |
|------|---------|------------|------|
| **知识/方法类** | 以文档和操作规程为核心交付物 | 规范准则/操作指南/AI辅助/任务定义 等 | 设计流水线、培训体系 |
| **软件类** | 以代码运行为核心交付物 | 前端层/后端层/数据层/基础设施 等 | ERP系统、电商平台 |
| **实物类** | 以物理实体为核心交付物 | 机械模块/电子单元/光学组件 等 | 数控机床、仪器设备 |

> 以上为基本类别，实际产品可能是混合型——如嵌入式系统同时含软件类和实物类特征；流水线类产品在软件/知识/实物基础上附加输出产品概念，见 §5.2.2。

PA 节点分类体系在各产品类型中的通用组织方式详见§4.7.2。

##### 1.7.2 产品类型对五层设计的影响

各层的关注点因产品类型而异，主要在以下方面：

| 影响维度 | 差异表现 | 详见 |
|---------|---------|------|
| **BA 开发方法** | 流水线类产品可选用输出产品架构锚定法 | §5.2.2 |
| **SysReq 条目格式** | 软件类描述"系统应执行什么功能"；知识类描述"应接收→转换→生成什么信息" | 各产品层准则实例化 |
| **PA 节点分类体系** | 不同类型的 PA 分类体系不同——软件按技术层、知识类按交付物形态 | 各产品层准则实例化 |
| **PA 判定维度** | 三独立原则（可独立分配/可独立替换/可独立验证）在各产品类型中具体化 | §4.7 |

##### 1.7.3 产品类型与 PA 架构节点分类

PA 架构末级节点的分类体系因产品类型而异，不存在跨产品类型的统一分类模板。各产品在实例化时应在本层准则中定义 PA 节点分类体系：

- **知识/方法类产品**：通常按交付物形态组织（准则/指南/AI文档/任务定义等）
- **软件类产品**：通常按技术架构层组织（前端/后端/数据/基础设施等）
- **实物类产品**：通常按物理架构域组织（机械/电子/光学/软件/系统等）

> 各产品层准则在定义 PA 节点分类体系时，应遵循通用准则 §6.7 的判定维度原则，并结合本产品类型特征实例化。


### 二、工程化设计内涵

本章将系统设计方法论核心转化为四条可操作的工程原则——需求相对、详概分离、文档结对、基于资产。四条原则依次回答设计链衔接、文档组织、跨层协作、节点生成四个核心问题，贯穿分解、分配与符合性分析三个设计活动，是日常设计工作的行动指南。

> **设计的本质**是对需求的持续**分解**（语义+指标切分）、**分配**（1:1 映射到下层架构末级节点）和**符合性分析**（语义覆盖+指标不降级）。这三个活动不是与四条原则并列的第五条，它们贯穿于每一条原则之中——在需求相对中识别上下游、在详概分离中组织分解内容、在文档结对中执行分配映射、在基于资产中指导架构决策。

工程化设计将这一本质组织为四条可操作的工程原则：

| 原则 | 要义 | 解决的问题 |
|------|------|-----------|
| **需求相对** | "需求"和"方案"不是文档的固有标签，而是文档在设计链中相对于相邻文档扮演的角色 | 设计链如何跨层衔接 |
| **详概分离** | 每份需求/方案文档由架构定义和详细定义两个子文档构成，职责分离、独立演化 | 单层文档如何组织 |
| **文档结对** | 结对发生在上层需求详细定义末级条目与下层方案架构定义末级节点之间，要求同步设计而非串行 | 跨层文档如何协同 |
| **基于资产** | 架构节点不是凭空产生的，而是基于组织已有资产进行识别、改进或新增，遵循复用→改进→新增优先级 | 架构节点如何生成 |



#### 2.1 需求相对

需求相对原则阐明：需求和方案不是文档固有标签，而是文档相对于相邻文档扮演的角色，中间层文档兼具方案和需求双重身份，是五层设计链连续传递的逻辑基础。

"需求"和"方案"不是文档的固有标签，而是文档在设计链中相对于相邻文档扮演的角色。同一份文档在不同位置上兼具不同身份：

| 层级 | 对上层的角色 | 对下层的角色 |
|------|-------------|-------------|
| OR（原始需求） | — | 纯需求（设计链起点） |
| SR（相关方需求） | 方案（对 OR） | 需求（对 BA / SysReq-NFR） |
| BA（业务架构） | 方案（对 SR） | 功能需求来源（对 SysReq） |
| SysReq（系统需求） | 方案（对 BA / SR-NFR） | 需求（对 PA） |
| PA（产品架构） | 方案（对 SysReq） | —（纯方案，设计链终点） |

SR 和 SysReq 是"中间层文档"——它们同时是方案（承接上层需求）和需求（向下层传递）。这种双重身份正是五层设计链能够连续传递的原因。

中间层文档的双重身份在操作层面落实为**详细定义的双重身份**：

- **向内**：是对当前层架构末级节点的细化，定义了该节点的具体职责和实现策略
- **向下**：作为需求传递给下一层方案，下层方案的架构末级节点需要承接这些详细定义条目

这一双重身份使详细定义成为上下层之间的衔接点——它是本层设计的输出末端，同时又是下层设计的需求源头。这也解释了 1:1 分配约束的必要性：如果一条详细定义需要分配到下层多个架构节点，说明本层的分解尚未充分。

**需求相对是设计链衔接的逻辑基础**：没有它，上层方案与下层需求之间就不存在可传递的接缝，详概分离和文档结对都无从建立。设计者应始终清楚当前文档在为谁提供方案、向谁提出需求。

#### 2.2 详概分离

详概分离原则规定：每份需求或方案文档由架构定义和详细定义两个独立子文档构成，前者以树形结构组织分类并建立映射，后者以平铺列表逐条细化，两者职责分离、独立演化。

无论文档当前扮演需求角色还是方案角色，一份完整的文档原则上由架构定义子文档和详细定义子文档两个子文档构成。两者职责分离、独立演化：

| 子文档 | 回答的问题 | 结构形态 | 核心任务 |
|-------|----------|---------|---------|
| **架构定义子文档** | "结构是什么、承接了谁" | 树形结构（根→末级） | 组织、分类、映射 |
| **详细定义子文档** | "每个模块具体做什么" | 末级条目平铺列表 | 细化、分解、分配 |

职责分离使两者可独立演化：调整架构分类层级无需重写详细条目，补充详细定义也无需改动架构树（新增或删除架构末级节点时除外）。

两类文档的核心任务不同——架构定义子文档采用树形结构对设计空间进行分类和组织，详细定义子文档采用平铺列表在已有分类框架下逐条细化。树形结构已完成分层，详细定义无需再嵌套。

| 维度 | 架构定义子文档 | 详细定义子文档 |
|------|-------------|-------------|
| 核心任务 | 组织、分类、映射 | 细化、分解、分配 |
| 信息视角 | "有哪些模块？谁承接谁？" | "每个模块具体做什么？" |

两者缺一则该层文档不完整。

**两项例外**：

- **OR（原始需求）**：处于最上游，只有需求无架构定义——作为设计链起点，其条目尚未结构化
- **BA（业务架构）**：只有架构定义无详细定义——其架构末级节点（IPO）自身承担"详细定义"的角色，直接映射到 SysReq

##### 2.2.1 架构定义子文档

架构定义子文档包含树形结构和映射关系表两部分。

**第一部分：架构设计结果（树形结构）**

从根节点直到末级节点的完整树形结构。节点按层级位置分为两类：

- **架构末级节点**：架构树的末梢节点，是所承接上层需求的**解决方案实体**。详见下文四要素
- **非末级节点**：末级节点之上的所有节点，不直接承接需求映射。细分为两种：
  - **纯分类节点**：仅用于对架构进行分类和组织，无实体含义（如 SR 层"功能需求"分类分支）
  - **集成实体节点**：代表由其子节点功能集成的真实实体——节点本身在架构中有意义，映射仍落在其下辖的末级节点上。BA 层的业务域（如"生产管理"下有"生产计划""工单执行"）和 PA 层的服务组（如"订单服务"下有"创建订单""查询订单"）典型属于此类

每个架构末级节点包含四要素：

| 要素 | 内容 |
|------|------|
| **节点定义** | 该架构末级节点的具体定义 |
| **上层需求映射** | 该架构末级节点承接了上层需求的哪些末级节点（N:1 映射） |
| **符合性分析说明** | 该架构末级节点如何定性和定量满足所承接的上层需求 |
| **下游设计指导** | 对下层方案设计的指导 |

**第二部分：需求映射视图（映射关系表）**

展示所有末级需求与架构末级节点的对应关系。以 SysReq→PA 结对为例：

```
需求末级节点      | 承接的架构末级节点 | 超链接
SysReq-001.01    | PA-001             | [跳转到第一部分]
SysReq-001.02    | PA-001             | [跳转到第一部分]
SysReq-002.01    | PA-002             | [跳转到第一部分]
```

两部分从不同视角服务同一目的：第一部分从架构视角展示分类和组织，第二部分从需求视角展示映射对应关系。架构末级节点在映射中充当核心枢纽——向上承接上层需求末级节点（N:1），向下为下层方案提供设计指引。

##### 2.2.2 详细定义子文档

详细定义子文档由一系列**详细定义末级节点**条目构成，采用平铺列表而非树形结构。每条条目包含以下信息：

| 字段 | 内容 | 说明 |
|------|------|------|
| **编号** | 遵循 §3.4.2 的编号规则 | 唯一标识，供下层架构映射引用 |
| **名称** | 简明描述该条目所定义的能力或约束 | 与架构末级节点的职责对齐 |
| **需求描述** | 功能或非功能需求的定性说明 | 不可衡量则不成为需求（§1.2） |
| **性能指标** | 需求被满足程度的可验证标准 | 分解时按三场景处理（§3.1） |
| **验收标准** | 判断该条目是否满足需求的判定条件 | 供符合性分析引用（§3.3） |
| **映射目标** | 所分配到的下层架构末级节点（1:1 映射） | 建立文档结对的链接 |

以 SysReq 详细定义分配到 PA 为例：

```
编号           | 需求描述                 | 性能指标         | 映射目标
SysReq-001.01 | 系统支持对输入数据的格式校验 | 单条校验 < 100ms | PA-001
SysReq-001.02 | 系统支持对校验结果的持久化存储 | 写入延迟 < 500ms  | PA-001
```

**分解方法**：每个架构末级节点展开为若干条详细定义末级节点，遵循 §3.1 的语义分解四检查（覆盖完整性、不超出性、逻辑一致性、约束继承）和指标分解三场景（可拆分、下放、暂不分解）。分解终止标准：条目已细化到可依 1:1 约束分配到下层方案中唯一一个架构末级节点。

**与文档结对的关系**：详细定义末级节点是文档结对中需求方的输出。结对设计要求上层需求详细定义与下层方案架构定义同步设计——先快速约定下层方案架构骨架，再由上层逐条填充需求详细定义。需求详细定义末级条目完成后，按 1:1 约束分配到下层方案架构末级节点，方案架构末级节点再对每条承接的条目执行符合性分析（§3.3）。

**详概分离是文档结对的前置条件**：只有将每层文档拆分为架构定义子文档和详细定义子文档，才谈得上"上层需求详细定义末级→下层方案架构定义末级"的精确结对。没有详概分离，文档结对就失去了操作对象。

#### 2.3 文档结对

文档结对原则定义系统设计的基本工作单元：上层需求详细定义末级条目与下层方案架构定义末级节点间的配对关系，要求结对同步设计而非串行执行。

系统设计的基本工作单元是"文档结对"——上层需求详细定义末级条目与下层方案架构定义末级节点之间的配对关系。

| 角色 | 内容 | 来源 |
|------|------|------|
| **需求方** | 上层需求详细定义末级条目，描述下层方案应该做什么（What） | 原始需求、相关方需求详细定义末级条目、系统需求详细定义末级条目 |
| **方案方** | 下层方案架构定义架构末级节点，描述如何满足上层需求（How） | 相关方需求架构定义、业务架构、系统需求架构定义、产品架构 |

需求方是对方案方的要求，方案方是对需求方的满足——有需求必有方案，有方案必有需求。OR（原始需求）是最上游层级，只有需求无方案，不参与文档结对。

##### 2.3.1 结对设计原则与两阶段工作法

**结对设计原则**：上层需求详细定义与下层方案架构定义应同步设计而非串行执行——在开展上层需求详细定义的同时（至少到结构骨架层级），同步启动下层方案架构定义。否则分解到末级后可能发现没有下层目标可承接，导致返工。

**两阶段工作法**：

1. **骨架阶段**：先快速约定下层架构的结构骨架（分类节点及末级节点标题）并建立初步映射。若下层方案已有**现有资产**，骨架阶段应优先**加载并理解**资产结构（替代从零创建骨架），以下层已有结构为分解参考方向
2. **填充阶段**：上下层各自并行填充——上层完善需求详细定义，下层完善方案架构节点四要素

> **现有资产场景下的同步设计**：当下层方案已有现有资产时，上层需求在分解前应先加载并理解资产结构（已有分类、节点职责边界、已承接的同类需求），做到心中有数后再开始分解。分解时以上层已有结构为参考方向，每条末级同步在下层方案中找承接节点（复用/改进/新增），而非全部分解完再统一映射。这样减少了不必要的扩展或新增，也使分解粒度自然匹配下层结构。

**结对关闭条件**：当上层全部需求详细定义末级条目都已明确分配至下层方案架构末级节点，且每个架构节点都完成了符合性分析说明时，该结对即视为闭合，进入双向追溯验证阶段。

**结对失败的典型信号**：

- **粒度不匹配**：上层末级条目经常需要分配到多个下层节点 → 分解不充分
- **映射模糊**：上层条目与下层节点之间无法建立清晰的语义对应关系 → 需求理解不一致
- **单方主导**：下层节点定义被上层需求完全决定（失去方案应有的设计空间），或上层需求被下层架构结构扭曲（失去需求的独立性）
- **闭环缺失**：映射关系建立后未执行符合性分析 → 无法确认承接是否充分

##### 2.3.2 结对责任边界

| 维度 | 需求方（上层） | 方案方（下层） |
|------|--------------|--------------|
| **主要输出** | 详细定义末级条目（编号、描述、验收标准、映射目标） | 架构末级节点四要素（定义、映射、分析、指导） |
| **核心责任** | 确保条目粒度满足 1:1 约束 | 确保节点定义充分承接所映射的全部需求 |
| **结对参与** | 骨架阶段指明需求分类方向；填充阶段逐条完善需求详细定义 | 骨架阶段提供可参考的现存架构资产；填充阶段补充方案架构节点四要素和映射关系 |
| **结对关闭条件** | 末级条目全部建立 1:1 映射 | 架构节点全部完成符合性分析 |

##### 2.3.3 多层结对的时序协调

一个完整系统设计中存在四个依次衔接的文档结对：

| 序号 | 结对 | 设计角色 |
|------|------|---------|
| 1 | OR 条目级需求 ↔ SR 架构定义 | 将原始需求组织为结构化相关方需求 |
| 2 | SR 详细定义末级 ↔ BA 架构（IPO） | 将相关方需求转化为业务过程说明 |
| 3 | BA IPO（去重后）↔ SysReq 架构定义 | 将业务过程映射为系统功能架构 |
| 4 | SysReq 详细定义末级 ↔ PA 架构定义 | 将系统需求转化为实现方案 |

这些结对不是串行执行的——上游结对进入骨架阶段后，下游结对即可启动骨架搭建：

```
结对1（OR↔SR）：骨架建立 → ────并行填充────→ 闭合
                              ↓ 骨架传递
结对2（SR↔BA）：           骨架建立 → ────并行填充────→ 闭合
                                          ↓ 骨架传递
结对3（BA↔SysReq）：                 骨架建立 → ────并行填充────→ 闭合
                                                      ↓ 骨架传递
结对4（SysReq↔PA）：                           骨架建立 → ────并行填充────→ 闭合
```

**核心规律**：骨架阶段可流水传递（上游骨架一建立，下游即可启动），填充阶段可全链并行（所有结对同时填充内容）。这正是结对设计原则能够跨层叠加的原因——四层结构不是四个串行阶段，而是四个部分重叠的设计窗口。

**文档结对是详概分离的跨层协作机制**：没有结对，详概分离产生的双文档结构就停留在单层内部，无法衔接为五层设计链。它将"上层需求详细定义末级"与"下层方案架构定义末级"配对，使需求相对的角色传递有了具体的操作载体。

#### 2.4 基于资产

基于资产原则规定：架构节点基于组织已有资产识别、选择、改造或新增而生成，遵循复用→改进→新增的优先级顺序，贯穿各设计层级（SR、BA、SysReq、PA）。每个架构末级节点都对应着组织中某个已有或需要构建的能力单元。

各层的资产是不同性质的东西，而非同一类资产在不同粒度上的呈现：

| 层级 | 资产形态 |
|------|---------|
| **SR 层** | 需求分类体系、领域知识结构、行业标准目录、法律法规框架 |
| **BA 层** | 已有业务过程、岗位职责分工、跨系统交互流程、业务规则集 |
| **SysReq 层** | 系统边界划分、接口协议、技术域约定、技术标准规范 |
| **PA 层** | 组件库、平台服务、可复用模块、中间件选型、框架约定 |

组织资产并非独立于开发过程预先存在——各产品的开发产出本身即构成组织资产，供后续设计和同类产品复用。资产的具体形态因产品类型和设计层级而异，需在各产品的设计准则中进一步定义。PA 层资产优先法的具体操作详见§4.7（资产优先法）。

##### 2.4.1 资产优先决策规则

新增上层需求需要分配到下层方案时，对现有架构末级节点的操作应遵循 **复用 → 改进 → 新增** 的优先级顺序：

```
第1优先 ── 复用（Reuse）
  检查：现有节点是否已完整覆盖此需求？→ 无需变更，直接映射

第2优先 ── 改进现有资产（Improve）
  检查：无现成节点，但通过调整可使现有节点承接？
  ├─ Extend（扩展）  — 扩展现有节点的定义和映射需求列表。**扩展后节点仍应保持单一职责**：扩展导致职责混杂时（扩展内容与原始定义描述的是不同性质的工作），不应扩展而应新增
  ├─ Split（拆分）   — 节点职责过宽，拆分为多个子节点
  └─ Merge（合并）   — 多个节点整合后共同承接

第3优先 ── 新增（Add）
  检查：复用和改进都不可行 → 在适当父节点下创建新末级节点

特殊情况 ── 废弃（Deprecate）
  上层需求取消，且无其他需求映射到此节点
  → 标记"无上游需求，已废弃"，保留节点（不物理删除）
```

**复用**：现有节点定义已完整覆盖新需求语义，无需变更即可直接映射——零成本、零风险。**改进**：语义基本对齐，但节点定义或映射关系需调整后才能完全承接，保留了已有设计资产的基本结构。**新增**：存在根本性语义鸿沟，应在复用和改进均不可行时才使用。

**资产决策的判断维度**（用以替代模糊的"语义相近"判断）：

| 维度 | 判断问题 | 结论 |
|------|---------|------|
| **职责方向** | 新需求的核心意图与哪个节点定义的职责方向一致？ | 方向匹配才值得进一步考虑复用或改进 |
| **边界偏差** | 新需求描述的具体内容在节点定义的边界之内还是之外？ | 在边界内→复用；略超→改进-扩展；远超→新增 |
| **同类承接** | 该节点是否已在承接同类型需求？ | 已有同类→倾向复用或扩展；无同类→重新评估职责方向 |

三个维度综合判断：**职责方向匹配 + 边界在范围内 + 已有同类 → 复用**；**方向匹配 + 边界略超 + 已有同类 → 改进-扩展**；**方向不匹配或超出过多 → 新增**。

上述决策基于已有架构节点作为组织资产的既有特征——每个节点已具备定义、映射和符合性分析记录，忽视这些特征将新增简化为"在树中添加条目"，导致架构快速膨胀。

##### 2.4.2 操作细则

**工作流程**：每分解产生一条新的需求，先在本层需求文档内部完成三项自治分析（可复用性检查——是否与本层已有需求重复；冲突性检查——是否与已有需求矛盾；规范性检查——格式/粒度/编号是否符合规范），再审视下游方案架构的所有资产，按复用→改进→新增的优先级找到承接节点。此顺序确保需求定义的质量不被已有架构结构扭曲。

**配套规则**：

- **废弃节点**：当某节点上没有上层需求映射到它时，在节点上标注"无上游需求，已废弃"，**不要从文档中删除该节点**。否则下游追溯时会出现"目标节点不存在"的断裂。
- **频繁扩展要拆分的信号**：同一节点累加 Extend 达到 3 次以上时，说明该节点的职责可能已超出合理范围，应审查是否需要拆分为多个职责单一的节点。
- **合并保留原节点**：当多个节点合并为一个时，原节点不删除，标注"已合并至XX节点"。原因同上——保留追溯路径。
- **信息处理模式的硬边界**：不同信息处理模式的需求不能共享同一个架构节点。例如实时处理和批量处理的工作方式完全不同，即使按资产优先"应优先复用"，也不能把实时需求和批量需求塞进同一个节点。这条规则的优先级高于资产优先——它是不可逾越的边界。
- **已有资产的只读隔离**：已完成的设计产出（如 v1 基线）作为只读资产使用，不得修改。对资产的任何扩展或修改在新版区（如 v2）进行，通过"基于 v1 的 XX 节点扩展"标注追溯变更来源。这确保资产的可追溯性——修改历史不会覆盖原有定义。

**基于资产是前三项原则的决策锚点**：需求相对明确了角色定位，详概分离提供了组织结构，文档结对建立了协作机制——但它们都依赖基于资产来回答"做成什么样"。没有资产优先原则，架构决策缺少优先级和方向，结构机制虽然可以运转，但设计者的选择没有依据。

### 三、文档结对设计

本章将文档结对设计机制展开为具体操作规则。以需求分解、需求分配和符合性分析三个核心设计活动为主线，定义每项活动的输入、输出、自检方法和终止条件，同时规定架构末级节点与详细定义末级节点的编号规则，将结对设计原理转化为各层设计工作中可执行的操作流程。本章中的"需求"与"方案"均指文档在设计链结对中的相对角色（§1.1 "需求/方案相对"）——上层详细定义在该结对中扮演需求角色，下层架构定义扮演方案角色。

---

#### 3.1 需求分解及分析

将父需求的功能描述和性能指标无重叠无遗漏地切分为子需求，包含语义分解四检查和指标分解三场景判定，是后续需求分配和符合性分析的前提。各层节点识别中的本层分解验证自检项见§4.2~§4.7各节末的"验证"步骤。

分解包含两个平行工作面，同步进行。

##### 3.1.1 需求语义分解

将父需求的功能内容无重叠无遗漏地划分到子需求，执行以下四项自检：

| 条件 | 说明 | 违规信号 |
|------|------|---------|
| **覆盖完整性** | 父需求中的每个功能点，子需求中都有对应——一条都不能丢 | 父需求的某些功能在子需求中无对应 → ❌ 遗漏 |
| **不超出性** | 子需求不能引入父需求范围外的新功能 | 子需求出现了父需求未提及的功能 → ❌ 超出 |
| **逻辑一致性** | 子需求之间不能相互矛盾 | 两条子需求对同一事物的要求互相冲突 → ❌ 冲突 |
| **约束继承** | 父需求的非功能约束（安全/合规/隐私等）被至少一条子需求继承 | 父需求的约束在子需求中消失 → ❌ 约束丢失 |

> **约束继承说明**：父需求的约束不要求出现在每一条子需求中，但必须至少有一条子需求承接它。例如父需求"用户管理"包含"需满足GDPR合规"，不应因分解为"注册/登录/注销"三条子需求而丢失GDPR要求——要么某条子需求显式继承，要么在分解记录中标注"作为父需求的附加约束传递至下层"。

##### 3.1.2 需求指标分解

将父需求的性能指标合理分配到子需求。不同专业有各自的算法，如机械结构的尺寸链（极值法/RSS）、电子系统的功耗预算、光学链路的MTF分配、软件系统的调用链延迟分配、可靠性工程的串联/并联模型等。指标分解前应**先识别父指标与子指标之间的运算关系或算法**，再选择对应的分解方式：

| 场景 | 典型运算关系 | 示例 | 处理方式 |
|------|------------|------|---------|
| **指标可拆分** | 加和/乘积/极值法等运算关系 | 父需求"齿轮箱传递误差<30角秒" → 子需求A"一级齿轮啮合误差<15角秒" + 子需求B"二级齿轮啮合误差<15角秒" | 拆分后各子指标组合运算满足父指标即可 |
| **指标下放** | 同值继承 | 父需求"工作温度范围-20°C~60°C" → 所有子需求都继承同一指标（各部件均需在此范围内正常工作，非和积关系） | 每子需求继承相同指标值 |
| **暂不分解** | 独立约束，无法运算分解 | 父需求"整机噪声≤45dB(A)" → 声学是源-路径-接收者的系统级耦合问题，在需求分解阶段尚无具体结构数据，无法按功能切分预先分配到各子需求 | 父指标的约束分解记录中注明"向下一层传递，不在本层子需求中展开" |

> 设计师需**明确记录所采用的分解方法和计算依据**。

##### 3.1.3 常见反模式

| 反模式 | 表现 | 后果 |
|--------|------|------|
| **过分解** | 把制造或实现步骤当作功能分解（"先加热→再保温→最后冷却"） | 需求中混入实现细节，限制了下层方案的设计空间 |
| **欠分解** | 一条需求包含多个独立功能（"设备应完成上料、加工、检测、分拣"） | 无法分配——一条需求需要多个下层架构节点承接 |
| **影子需求** | 子需求中凭空出现父需求没有的内容 | 引入未经上游确认的新需求，导致范围蔓延 |
| **约束丢失** | 父需求的非功能约束在子需求层次消失 | 下层方案无从知晓非功能要求，交付后才发现不满足 |

---

#### 3.2 需求分配及分析

将分解完成的末级需求条目按1:1约束逐一映射到下层方案架构末级节点，执行全覆盖、1:1和N:1合理性三项自检，确保无遗漏、无重复、无粒度失当。各层节点识别中的本层分配自检项见§4.2~§4.7各节末的"验证"步骤。

##### 3.2.1 分配的三项自检

| 检查项 | 问题 | 通过条件 | 违规后果 |
|--------|------|---------|---------|
| **全覆盖** | 是否所有末级需求都已被分配？ | 映射覆盖率达100% | 未分配的需求在下层方案中无对应设计，直接被遗漏 |
| **1:1约束** | 每条需求是否分配到唯一的架构末级节点？ | 无需求映射到多个节点 | 同一需求在多个节点中重复实现，浪费设计/开发资源 |
| **N:1合理性** | 每个架构末级节点承接的需求数量是否合理？ | 推荐条数合理，按语义相关性判定 | 过少→节点粒度过细；过多→节点职责过宽 |

##### 3.2.2 1:1约束的执行细则

"一条末级需求只能分配到一个方案架构末级节点"，例外和边界如下：

| 情况 | 是否违反1:1 | 说明 |
|------|-----------|------|
| 同一条末级需求被分配到两个架构节点 | ❌ 违反 | 需回到分解步骤进一步拆分该需求 |
| 一个架构节点承接了多条需求 | ✅ 不违反（N:1） | 正常的合并，检查合理性即可 |
| 同一条需求在两个架构节点中都被"引用"但实际实现只在一个节点中 | ⚠️ 需澄清 | 标注主承接节点和参考节点，避免混淆 |
| BA 层 IPO 去重后映射到 SysReq | ✅ 不违反 | 去重操作而非分配——同一IPO被多次引用，先去重再映射 |

##### 3.2.3 N:1合并的合理性判断

| 条件 | 判断 |
|------|------|
| 多条需求属于同一功能领域且信息处理模式相同 | ✅ 合理合并 |
| 多条需求操作同一组核心实体 | ✅ 合理合并 |
| 多条需求的 NFR Profile 相容 | ✅ 合理合并 |
| 合并后节点承接 >8 条需求 | ❌ 过度合并，需考虑拆分 |
| 节点承接 1 条需求 | ⚠️ 粒度偏细，评估是否可与其他节点合并 |

---

#### 3.3 符合性分析

在分解和分配完成的基础上，逐一验证每个架构末级节点是否充分满足所承接的全部需求集合，从语义符合性和指标符合性两个维度进行深度方案级验证。

**输入**：每个架构末级节点承接的需求集合
**输出**：符合性分析说明
**终止条件**：每个架构末级节点均已完成语义和指标符合性分析

两个维度并行。

##### 3.3.1 语义符合性

确认方案节点在语义上是否充分满足了所承接的需求：

| 检查维度 | 问题 | 充分分析的特征 | 不充分分析的特征 |
|---------|------|--------------|----------------|
| **覆盖度** | 方案节点的定义是否覆盖了所有承接需求的核心语义？ | 节点定义中提到了每条需求的对应处理 | 节点定义只覆盖部分需求 |
| **映射正确性** | 承接的需求与节点定义是否语义匹配？ | 需求的术语与节点的术语可对应 | 需求是"安全审计"而节点是"操作日志"（范围不同） |
| **分析具体性** | 符合性分析是否有实质内容而非套话？ | "本节点通过实现X机制满足Y需求，具体体现在Z" | "本节点满足上述需求" |

**编写符合性分析的基本句式**：

```
本节点（节点名称）通过以下方式满足所承接的需求（需求ID列表）：
- 针对需求A：在XX功能中实现YY处理，覆盖了需求的ZZ方面
- 针对需求B：以XX机制确保YY指标不低于ZZ
```

> **反面教材**："本节点满足上述所有需求。"——没有说明如何满足，等于没有分析。

##### 3.3.2 指标符合性

验证架构节点的性能指标是否满足其所承接需求的性能指标要求。与指标分解（§3.1.2）同理，分析前应根据方案节点定义与需求定义的语义**先识别方案节点指标与需求指标之间的运算关系或算法**。识别运算关系后，**基于该关系计算分析**，确定方案节点指标值是否满足承接需求的指标值。计算完成后执行以下检查验证分析是否到位：

| 检查项 | 问题 | 通过条件 |
|--------|------|---------|
| **指标对应完整性** | 所有承接需求的性能指标是否都在方案节点中有显式的对应指标条目？ | 逐指标比对，无遗漏 |
| **指标比较正确性** | 方案节点指标与需求指标的比较是否基于正确的运算关系，结论已明确记录？ | 运算关系有据可查，结论为"满足/降级/丢失"之一 |
| **指标可验证性** | 方案节点指标的测量条件和统计方式是否明确？ | 有测量方法描述，非空洞数字 |

#### 3.4 编号规则

规定架构末级节点与详细定义末级节点的编号格式、规则及编号稳定性约束。统一的编号体系确保所有设计条目可被唯一标识和跨层引用，是建立完整追溯链路的技术基础。

##### 3.4.1 架构末级节点编号

**格式**：`{层前缀}[-{类型后缀}]-{序号}`

```
SR-F-001      SR 功能需求     架构末级
SR-NFR-001    SR 非功能需求   架构末级
BA-001        BA IPO          架构末级（无详细定义）
SysReq-F-001  SysReq 功能需求      架构末级
SysReq-NFR-001 SysReq 非功能需求   架构末级
PA-001        PA 构件         架构末级
OR-001        OR 条目         条目级编号（OR 无架构末级节点）
```

**约定**：
- 序号使用 3 位数字，起始 001
- 不同功能/非功能分支通过类型后缀区分（SR 和 SysReq 需要）
- 各层的编号空间独立（`BA-001` 与 `SR-001` 无关联）
- 原始需求（OR）没有架构末级节点的概念，其 `OR-{SEQ}` 属于**条目级编号**，在系统中等同视为架构末级节点编号（为详细定义分解提供归属锚点）

##### 3.4.2 详细定义末级节点编号

**格式**：`{所归属的架构末级节点ID}.{层内序号}`

```
OR-001.01         ← 归属于 OR-001（OR 无架构，以其条目 ID 等同对待）
SR-F-001.01      ← 归属于 SR-F-001
SysReq-001.01    ← 归属于 SysReq-001
PA-001.01        ← 归属于 PA-001
```

**约定**：
- 层内序号**默认使用 2 位数字**，起始 01。当预计一个架构末级节点可能分解出超过 99 条详细定义末级节点时，可改用 **3 位数字**（起始 001）
- 同一产品的全部详细定义末级节点应统一采用相同的位数（全用 2 位或全用 3 位），避免混用
- 中间分解层级（如 SysReq 的 6/7/8 级）**不编号**，所有详细定义末级节点在所属架构末级节点之下平铺编号
- 业务架构（BA）无详细定义，不适用本规则
- 原始需求（OR）条目编号为 `OR-{SEQ}`（3 位序号），分解后的子条目编号为 `OR-{SEQ}.{SEQ}`。OR **不使用类型后缀**（如 `OR-F-001` 或 `OR-NFR-001` 均为非法模式），原因：单条原始需求可能同时包含功能内容和非功能内容，功能/非功能分类只有在分解至 SR 层后方可分离

##### 3.4.3 编号的稳定性

- 编号一旦分配即固定，不因树形结构的调整而改变
- 新增详细定义末级节点 → 追加新的序号（当前最大值 + 1）
- 中间层级重新组织 → 不改变详细定义末级节点 ID
- 追溯矩阵、验证报告、下层架构映射中引用的 ID 始终保持有效

### 四、各层架构末级节点

本章基于 §1.3.1"架构末级节点"的概念定义（**架构树末梢，作为承接上层需求的解决方案实体**），逐层回答两个问题：各层架构末级节点在现实中具体指代什么对象？如何系统地识别和定义这些节点？本章从 §4.2 至 §4.7 逐层展开，§4.8 提供跨层验证框架。

本章与§六（系统设计过程）的分工不同：§六按操作流程的先后顺序组织各步内容（"怎么做"的操作规程）；本章按节点类型组织（"各层节点分别指什么"的横向对照）。

#### 4.1 架构末级节点总览

| 层级 | 节点编号 | 具体指代对象 | 在五层链中的角色 |
|------|---------|------------|----------------|
| **OR** | OR-{SEQ}（条目级编号，等同处理） | 一条经过预处理的原始需求条目 | 设计链起点，纯需求 |
| **SR-F** | SR-F-{SEQ} | 某类用户角色的一个端到端业务闭环 | 对OR是方案，对BA是需求 |
| **SR-NFR** | SR-NFR-{SEQ} | 一个非功能需求关注点及其量化指标 | 对OR是方案，对SysReq-NFR是需求 |
| **BA** | BA-{SEQ}（IPO） | 产品接收一份输入单元→处理→产出一份输出单元的业务行为单元 | 独立方案层，无详细定义 |
| **SysReq-F** | SysReq-F-{SEQ} | 一个系统功能能力单元 | 对BA是方案，对PA是需求 |
| **SysReq-NFR** | SysReq-NFR-{SEQ} | 对一个 NFR 关注点的技术约束细化 | 对SR-NFR是方案，对PA是整体约束 |
| **PA** | PA-{SEQ} | 一个可独立分配/替换/验证的产品组件 | 设计链终点，纯方案 |

> **架构末级节点 ≠ 详细定义末级节点**（§1.3）：架构末级节点是树形结构的末梢——本层"解决方案实体"的组织单元；详细定义末级节点是对架构末级的进一步平铺细化，按 1:1 约束分配到下层架构末级。两者在同一层中共存但职责不同，不可混淆。

各层架构末级节点在五层链中的角色差异决定了它们指代对象的本质不同：

| 角色类型 | 包含的节点类型 | 指代特征 |
|---------|--------------|---------|
| **纯需求** | OR 条目 | 相关方的原始诉求，未经方案设计 |
| **中间层（方案+需求）** | SR-F、SR-NFR、SysReq-F、SysReq-NFR | 对上层是方案（组织/归类/设计），对下层是需求（传递约束） |
| **纯方案** | BA IPO、PA 构件 | 设计终点，不传递需求到下一层 |

> **BA 的分类说明**：BA 归为"纯方案"系因其仅有架构定义、无详细定义子文档（§1.3 特例），IPO 条目自身承担详细定义的角色，不同于 SR 和 SysReq 的架构定义+详细定义双层结构。BA IPO 去重后直接映射到 SysReq-F，功能需求的传递以 IPO 直接映射的方式实现而非通过详细定义子文档——这与"不传递需求"的分类不矛盾，而是 BA 作为独立方案层的结构特例（详见 §4.4.5）。

中间层节点同时具有双重身份的机制见§1.1"需求/方案相对"和§2.1。

架构末级节点进一步分为三种子类型（§1.3.1）：

| 子类型 | 说明 | 出现位置 |
|-------|------|---------|
| **纯分类节点** | 仅用于对架构进行分类和组织，无实体含义 | SR层的"功能需求""非功能需求"分类分支 |
| **集成实体节点** | 代表由子节点功能集成的真实实体，映射仍落在子节点上 | BA层的业务域、PA层的服务组 |
| **架构末级节点** | 树形结构的末梢，承载上层需求映射 | 各层末端 |

#### 4.2 SR-F 架构末级节点的识别与定义

##### 4.2.1 指代对象

**SR-F 架构末级节点指代：某类用户角色在使用目标产品时，从"接到一个业务需求"到"该需求被满足"的完整端到端业务闭环。**

| 用户角色 | 业务触发 | 端到端业务（起点→终点） | 对应 SR-F 节点 |
|---------|---------|---------------------|--------------|
| 采购员 | 收到审批通过的请购单 | 接收请购单 → 提交PO待审批 | 采购执行 |
| 采购员 | 收到供应商异常通知 | 收到异常通知 → 异常处置完成 | 跟单与异常处理 |
| 审批者 | 收到待审批事项 | 查阅详情 → 做出审批决策 | 审批任务处理 |

##### 4.2.2 架构分类结构

SR-F 架构按"**用户角色类型 × 触发类型**"二维结构组织：

| 维度 | 说明 |
|------|------|
| **用户角色类型** | 在组织中承担的业务职责（采购员、审批者、外部系统等） |
| **触发类型** | 角色"接到什么"作为一个独立的业务触发起点 |

同一用户角色下，业务行为本质相同的触发归并为同一触发类型（如"收到待审批事项"而非按单据类型拆为多个）；来源和目的不同的触发分属不同触发类型（如"收到请购单"与"收到异常通知"）。不同角色类型的端到端业务可能在业务流程上重叠，各自独立定义、在 SR-F 层共存。

##### 4.2.3 识别方法：三步法

**第一步：识别用户角色类型**

按"在组织中承担的业务职责"识别和分类，而非按"操作目标产品的行为是否相同"。用户角色类型应穷举：

- 操作用户（业务人员、管理人员等）
- 外部系统（与目标产品交互的其他系统）
- 保障者（运维、培训等）
- 其他相关方

同一业务职责的不同层级人员（如多级审批者）因职责不同，应归为不同的角色类型。

**第二步：按角色类型识别触发，定义端到端业务**

对每类角色，逐一识别其"接到什么"作为一个独立的业务触发，从触发到交付定义完整的端到端业务闭环。

触发的抽象级别判断原则：
- 同一用户角色下，业务行为本质相同的触发 → 归并为同一触发类型（如"收到待审批事项"而非按单据类型拆为多个）
- 同一用户角色下，来源和目的不同的触发 → 分属不同触发类型（如"收到请购单"与"收到异常通知"）

不同角色类型的端到端业务可能在业务流程上重叠，各自独立定义、在SR-F层共存。

**第三步：用需求分解分配分析验证候选节点**

候选节点须通过以下检查：
- **分解检查**：详细定义条目的语义精度是否足够支撑下层设计。如"核对依据材料"在不同审批场景中含义不同→节点边界需调整
- **分配检查（1:1约束）**：详细定义条目能否按1:1约束分配到唯一BA IPO。同一条目在不同场景需不同BA IPO→节点粒度不满足约束
- **N:1合理性检查**：节点承接的详细定义条目数超过8条→过度合并，需拆分

##### 4.2.4 与BA的边界划分

SR-F 与 BA 在"业务"层面不重叠：
- SR-F 架构末级 = **有什么业务**——用户角色的一个端到端业务闭环
- BA IPO = **怎么做这些业务**——该闭环中的原子操作步骤

两者的区分本质上是 What 与 How 的区分（§1.1）。

#### 4.3 SR-NFR 架构末级节点的识别与定义

##### 4.3.1 指代对象

**SR-NFR 架构末级节点指代：某类相关方对目标产品的一个非功能需求关注点，经量化后的表达。**

非功能需求不经 BA，走 SR-NFR → SysReq-NFR → PA 的平行路径（§1.2）。

##### 4.3.2 架构分类结构

SR-NFR 按"**相关方角色类型 → 附录B三级分类**"两级结构组织：

| 层级 | 组织方式 | 说明 |
|------|---------|------|
| **第一级（角色分类）** | 按 NFR 相关方角色类型分 | 每类角色可独立验证其 NFR 关切是否已覆盖 |
| **第二级（三级分类）** | 按附录B的三级分类分 | 质量特性 / 环境适应性 / 可实现性 |

NFR 的相关方不仅限于直接操作产品的用户：

| 角色类型 | 核心关切 | 典型 NFR 维度 |
|---------|---------|-------------|
| **操作用户** | 操作体验 | 性能、易用性、人机工效 |
| **运维保障者** | 稳定与可维护 | 可用性、可靠性、可维护性 |
| **合规/审计者** | 合规与安全 | 安全性、合规性、可追溯性 |
| **管理者/投资者** | 成本与风险 | 成本效益、风险控制 |
| **研发/扩展者** | 可演进性 | 可扩展性、可测试性、可移植性 |

##### 4.3.3 识别方法：角色→分类归并法

**第一步：按相关方角色类型识别 NFR 关注点**

从两个渠道获取 NFR 关注点：
- **显式提取**——从 OR 条目中提取非功能陈述（单条 OR 可能同时包含 FR 和 NFR 内容）
- **隐式推导**——按相关方角色类型的固有特征补充推导可能的关注点，不限于 OR 已明确陈述的内容。不同领域的 NFR 关注点可参考对应领域标准（如汽车参考车规、民机参考适航、工业系统参考 IEC 标准、信息系统参考等保标准）。

**第二步：按角色类型归并关注点到三级分类**

在每个角色类型下，将关注点按附录B的三级分类和二级分类归并。同分类的关注点合并为一个候选 SR-NFR 节点。

**第三步：处理不可归类的关注点**

当某关注点在现有分类体系中找不到对应末级时，执行新增评审：

| 评审准则 | 问题 |
|---------|------|
| **维度独立性** | 该关注点能否被已有分类末级覆盖？——如果只是同义表述或特例，不应新增 |
| **可量化性** | 该关注点能否被独立测量？有无明确的测量方法和判定标准？——无法量化的不构成独立节点 |
| **架构影响力** | 增加该关注点是否会导致 PA 产生不同的实现决策？——对架构无差异化影响的，已隐含在已有分类中 |

三准则全部通过方可新增为分类库节点，并记录到组织 NFR 分类资产中。NFR 分类体系是组织持续积累的设计资产，与 PA 组件资产同等重要。

**第四步：量化指标**

每个 SR-NFR 节点赋予量化指标（数值、单位、测量条件），参考领域标准中的指标定义。

##### 4.3.4 与 SR-F 的边界划分

NFR 相关方角色类型与 FR 的用户角色类型共享同一角色集——每个与目标产品交互的实体都可能同时产生功能需求和非功能需求。两者的区别在于：

| 维度 | SR-F（功能） | SR-NFR（非功能） |
|------|------------|----------------|
| **回答的问题** | 用户有什么业务 | 系统运行得怎么样 |
| **分类结构** | 按用户角色×触发组织 | 按相关方角色→三级分类组织 |
| **完备性保证** | BA 系统化推导（操作场景穷举） | 角色类型全面识别 + 专业领域积累 |
| **分配路径** | 经 BA 到 SysReq-F | 不经 BA，直通 SysReq-NFR |

#### 4.4 BA 架构末级节点（IPO）的识别与定义

##### 4.4.1 指代对象

BA 架构末级节点（IPO）指代：**目标产品接收一份具有完整表意能力的输入单元，经处理（自动化加工与人专业指引相结合），产出一份输出单元的业务行为单元。**

BA 的开发基于两个前提：其一，在企业中凡业务必有文档输出物——没有文档输出物的业务不可记录、不可追溯、不可考核，不构成企业工程活动，因此每个 IPO 必然以一份输入文档起始、以一份输出文档终止；其二，BA 将"用户有什么业务"翻译为"用户如何操作产品来完成业务"，为相关方提供可验证的业务操作界面，相关方能够用自己的业务语言确认操作场景是否完整，从而确保由此推导出的系统功能清单（SysReq-F）的完备性。

| 要素 | 含义 | 说明 |
|------|------|------|
| **I（输入）** | 一份具有完整表意能力的输入单元，由单一外部来源完整提供 | 企业业务场景中为业务单据或设计产物；系统交互场景中为 API 请求/消息/传感信号等 |
| **P（处理）** | 产品主责处理 + 人参与专业工作 | **产品自动化部分**：按规则/模板/计算/路由自动执行；**人参与部分**：专业判断、设计方案、分析诊断等**处理本身**，而非审批决策 |
| **O（输出）** | 一份具有完整表意能力的输出单元 | 企业业务场景中为业务文档或设计产物；系统交互场景中为 API 响应/通知/控制信号等 |

> **处理本身 vs 审批**：P 中的人参与是处理本身的组成部分——设计师设计方案、工程师诊断故障、分析师写结论，均属于"处理"范畴，不可省略且不可替代。审批是独立的 IPO，其 P 是"审批者行使决策权"，不属于处理范畴。

IPO 的输入/输出形态因用户角色类型而异——不同角色与目标产品的交互方式不同，I/O 的承载形态也相应不同：

| 角色类型 | I 的形态 | O 的形态 |
|---------|---------|---------|
| **操作用户（人）** | 表单/请求/指令 | 结果文档/界面展示/报告 |
| **外部系统** | API请求/消息/数据推送 | API响应/通知/数据同步 |
| **传感器/设备** | 传感信号/读数/事件 | 控制信号/驱动指令 |
| **自然环境** | 温度/湿度/振动/光照等环境条件 | 产品状态输出/物理作用 |

##### 4.4.2 架构分类结构

BA IPO 按**业务域**组织分类。每个业务域对应一类业务职能（采购、生产、售后、研发、运营等），包含一组业务操作步骤。同一业务域内的 IPO 共享相似的文档流转模式：

| 业务域 | 典型 SR-F | IPO 序列示例 |
|--------|----------|-------------|
| **采购** | 采购执行 | 采购需求 → 询价单 → 报价记录 → 采购订单 |
| **研发** | 产品变更处理 | 变更请求 → 影响评估报告 → 变更方案 → 变更确认单 |
| **生产** | 工单执行 | 工单/领料单 → 报工记录 → 质检报告 → 入库单 |
| **售后** | 客户报修处理 | 维修工单 → 派工单 → 维修记录 → 完工确认单 |
| **运营** | 经营分析报告编制 | 数据集 → 分析报告草稿 → 已发布的经营分析报告 |

##### 4.4.3 识别方法：三步法

**识别前置：输入/输出单元的定义与识别**

在 BA IPO 语境下，**输入/输出单元**指：具有完整表意能力的信息载体——一份自包含的、可独立标识和流转的业务记录、设计产物、系统消息或传感信号。

以企业业务场景为例，输入/输出单元常表现为以下形态：

| 类别 | 示例 |
|------|------|
| **业务单据** | 请购单、采购订单、质检报告、入库单 |
| **设计产物** | 需求定义、架构定义、变更方案、追溯矩阵 |
| **通知/记录** | 异常通知、审批提醒、公告 |
| **分析/报告** | 经营分析报告、报价对比表、影响评估报告 |

在其他场景中，输入/输出单元也可能是 API 请求/响应消息、传感信号读数、环境条件数据等（详见 §4.4.1 角色形态映射表）。

输入/输出单元的五个判断标准：

| 标准 | 含义 | 反例 |
|------|------|------|
| **表意完整性** | 承载一条完整的信息，独立表意 | 表单中的单个字段值（"单价100元"）；一次 API 调用中的单个参数 |
| **可独立标识** | 业务/接口中有明确的称谓或标识符 | "用户在输入框敲的内容"；未命名的临时变量 |
| **可独立流转** | 可在用户/产品/系统间传递 | 一次鼠标点击 |
| **由单一来源完整提供** | 可由一个用户或外部来源在合理时间内独立完成 | 需多人协作才能完成的报告（应拆分为多步 IPO） |
| **有交互阶段意义** | 输入/输出单元的产出标志一个交互回合或业务阶段的完成 | 用户页面上的每次键盘敲击；传感器每秒的原始读数（未聚合） |

> **判断技巧**：问"这份东西能独立地被存档、检索和追溯吗？"——能就是有效的输入/输出单元，不能就不是。

同一 IPO 中 I 和 O 的关系：

| 关系 | 说明 | 示例 |
|------|------|------|
| **不同种类** | I 和 O 是不同种类的单元 | I=请购单 → O=询价单 |
| **同种不同状态** | 同一单元被产品处理后状态变更 | I=待审批的采购订单 → O=已批准的采购订单 |

**第一步：从 SR-F 端到端业务导出输入输出流转链**

对每个 SR-F 端到端业务闭环，按业务时序识别产品全过程中接收的输入单元和产出的输出单元。每个单元应满足上述五个判断标准。

输入输出流转链的起点是"谁向产品提交了什么"，终点是"产品向谁输出了什么"，中间每个输入输出流转节点构成一个 IPO 的候选边界。

以采购执行业务为例的输入输出流转链：

```
采购员提交请购需求 → 产品生成询价单（发供应商）
                           ↓
                   供应商返回报价单
                           ↓
                   采购员确认选商 → 产品生成采购订单草稿
                                           ↓
                   采购员提交审批 → 产品流转至审批人
                                           ↓
                   审批人决策 → 产品返回已审批采购订单
```

**第二步：按输入输出流转节点定义 IPO 序列**

将输入输出流转链中每个"产品接收输入→处理→输出"的节点定义为一个 BA IPO。每个 IPO 需完整定义 I、P、O 三要素。

以采购执行业务为例：

| IPO | I（输入单元） | P（产品处理 + 人参与） | O（输出单元） |
|-----|-------------|----------------------|-------------|
| 1 | 采购员提交的采购需求（含物料、数量、交期） | 产品按品类匹配合格供应商 + **采购员基于供应商了解确认范围** | 待发送的询价单 |
| 2 | 供应商返回的报价单 | 产品记录报价并锁定 + **采购员核对报价完整性和商务条款** | 报价记录（含对比） |
| 3 | 采购员确认的选商结果 | 产品填充 PO 模板锁定价格条款 + **采购员确认价格/条款/交期** | 采购订单草稿 |
| 4 | 采购员提交的审批请求（附 PO） | 产品按规则路由至对应审批人 + **审批人行使审批决策权** | 已审批的采购订单 |

跨业务域 IPO 序列示例：

| 业务域 | SR-F | IPO 序列 |
|--------|------|---------|
| **研发** | 产品变更处理 | 变更请求 → 影响评估报告 → 变更方案 → 变更确认单 |
| **生产** | 工单执行 | 工单/领料单 → 报工记录 → 质检报告 → 入库单 |
| **售后** | 客户报修处理 | 维修工单 → 派工单 → 维修记录 → 完工确认单 |
| **运营** | 经营分析报告编制 | 数据集 → 分析报告草稿 → 已发布的经营分析报告 |

**第三步：用分解分配分析验证**

| 检查项 | 内容 |
|--------|------|
| **分解检查** | I 和 O 是否都是具有完整表意能力的输入/输出单元（满足§4.4.3 前置五标准）？P 是否覆盖了产品应做的全部处理以及人的专业工作？是否有独立的输入单元触发，还是将同一用户的连续操作拆分成了多个 IPO？ |
| **分配检查（1:1 约束）** | SR-F 详细定义中的每条条目能否分配到唯一一个 BA IPO？同一条目是否在不同场景下需要分配到不同 IPO——若是则说明 SR-F 节点边界需要重新审视 |
| **N:1 合理性检查** | 每个 BA IPO 承接的 SR-F 详细定义条目数是否合理（2-5 条最优）？是否存在过度合并或碎片化？ |

##### 4.4.4 与 SR-F 的边界划分

BA IPO 与 SR-F 末级节点在"业务"层面不重叠——前者描述产品行为，后者描述用户业务：

| 维度 | SR-F 架构末级 | BA IPO（架构末级） |
|------|-------------|-------------------|
| **指代** | 用户角色的一个端到端业务闭环 | 产品处理一个输入输出单元的行为单元 |
| **回答的问题** | 用户有什么业务 | 产品怎么支撑该业务 |
| **描述视角** | 用户视角（从触发到交付） | 产品视角（输入→处理→输出） |
| **颗粒度** | 一个完整业务闭环 | 闭环中的一个输入输出流转节点 |
| **人参与** | 用户是业务执行的主体 | 人在 P 中参与专业工作（处理本身） |

两者的区分本质上是 What 与 How 的区分（§1.1）：SR-F 回答"用户有什么业务"，BA IPO 回答"产品如何支撑该业务的每个环节"。

##### 4.4.5 BA 的特殊性与去重机制

BA 是五层中唯一**仅有架构定义、无详细定义**的层（§1.3）。IPO 条目自身承担"详细定义"角色，去重后直接映射到 SysReq-F 的架构末级节点。

**去重规则**：两个 IPO 的 I/O 单元类型、P 处理逻辑均一致时视为相同，去重后作为一个 BA 节点存在。例如"提交审批"这一 IPO 出现在采购审批、报销审批、变更审批等多个 SR-F 业务中——它们在 BA 层共享同一个 IPO 节点。

#### 4.5 SysReq-F 架构末级节点的识别与定义

##### 4.5.1 指代对象

**SysReq-F 架构末级节点指代：系统为实现一个唯一 BA IPO 而应具备的功能能力单元。**

每个唯一 BA IPO 经过视角转换（从业务视角到系统视角）后，一对一映射为一个 SysReq-F 架构末级节点。

| 维度 | BA IPO（业务视角） | SysReq-F（系统视角） |
|------|------------------|-------------------|
| **名称** | 业务行为名（"生成询价单"） | 系统功能名（"询价供应商匹配功能"） |
| **I** | 谁提交了什么文档 | 系统接收了什么输入 |
| **P** | 产品做了什么+人参与了什么 | 系统做什么处理+人操作什么（说清分工） |
| **O** | 产出了什么文档 | 系统输出了什么 |

SysReq-F 节点位于 0-9 级层级结构的 5 级（组件实例），是 SysReq 功能部分的架构末级节点。0-4 级是对功能清单的组织分类，对应系统功能菜单的分组结构。

##### 4.5.2 架构分类结构

SysReq-F 架构末级节点（功能清单）按系统功能菜单的组织结构归入 0-4 级分类树。分类层级数不固定——取决于系统的功能菜单深度：

| 层级 | 内容 | 对应 |
|------|------|------|
| 1 级 | 一级菜单/功能模块 | 采购管理、生产管理、财务管理 |
| 2 级 | 二级菜单/功能子模块 | 采购执行、供应商管理 |
| 3 级 | 三级模块 | 询价管理、订单管理 |
| 4 级 | （如有） | … |
| **5 级（架构末级）** | **功能清单（SysReq-F 节点）** | 询价供应商匹配功能 |

分类原则：
- **功能内聚**——同一父节点下的 SysReq-F 节点应属于同一业务领域
- **菜单对齐**——以系统实际功能菜单结构为组织参考
- **层级适度**——层级深度与菜单嵌套深度一致，不应额外增加或减少层级

##### 4.5.3 识别方法：视角转换法

**第一步：准备唯一 BA IPO 清单**

将 BA 层去重后的唯一 IPO 清单作为输入。对每个 IPO，识别 I/P/O 三要素中哪些属于系统能力、哪些属于人的专业工作。

**第二步：逐 IPO 转换为 SysReq-F 节点**

对每个唯一 BA IPO，执行四步转换：

| 转换 | 做法 | 示例 |
|------|------|------|
| **命名** | 业务行为名 → 系统功能名 | "生成询价单" → "询价供应商匹配功能" |
| **I 转换** | "谁提交了xx" → "系统接收xx" | "采购员提交的采购需求" → "系统接收采购需求（品类、数量、交期）" |
| **P 转换** | 拆分为"系统处理"和"人操作"，明确分工 | "产品匹配合格供应商+采购员确认" → "系统按品类自动匹配并展示供应商列表；采购员查看并调整确认" |
| **O 转换** | "产出了xx" → "系统输出xx" | "待发送的询价单" → "系统输出已确认供应商清单的询价单" |

完整转换示例：

```
BA IPO:
  I=采购员提交的采购需求
  P=产品按品类匹配合格供应商 + 采购员确认供应商范围
  O=待发送的询价单
  名称：生成询价单

  ↓ 视角转换

SysReq-F:
  I=系统接收采购需求（含物料品类、数量、交期）
  P=系统按品类自动匹配合格供应商列表并展示；采购员查看匹配结果，可调整供应商范围后确认
  O=系统输出已确认供应商清单的询价单
  名称：询价供应商匹配功能
```

```
BA IPO:
  I=供应商返回的报价单
  P=产品记录报价锁定 + 采购员核对报价完整性和商务条款
  O=报价记录
  名称：归集报价

  ↓ 视角转换

SysReq-F:
  I=系统接收供应商提交的报价单
  P=系统按供应商/物料维度归集并锁定报价；采购员查看归集结果，核对商务条款完整性后确认
  O=系统输出核验完成的报价记录
  名称：报价归集与核验功能
```

**第三步：验证**

| 检查项 | 内容 |
|--------|------|
| **一对一验证** | 每个唯一 BA IPO 是否都转换为一个 SysReq-F 节点？无遗漏、无多余 |
| **视角一致性** | 节点名称和 I/P/O 描述是否都使用了系统语言而非业务语言？ |
| **人机分工明确性** | P 中是否明确区分了"系统处理什么"和"人操作什么"？分工是否清晰无歧义？ |

##### 4.5.4 与 BA 的边界划分

| 维度 | BA IPO | SysReq-F |
|------|--------|---------|
| **描述视角** | 业务视角——产品做什么 | 系统视角——系统能做什么 |
| **描述语言** | 业务语言（"采购员提交需求"） | 系统语言（"系统接收需求"） |
| **人机分工** | P 中笼统描述（"人参与专业工作"） | P 中明确区分系统处理和人工操作 |
| **五层链角色** | 纯方案——独立方案层 | 中间层——对 BA 是方案，对 PA 是需求 |
| **详细定义** | 无（IPO 自身承担） | 有（6-9 级详细分解至场景活动） |

##### 4.5.5 SysReq-F 节点属性

每个 SysReq-F 架构末级节点包含以下属性：

| 属性 | 说明 |
|------|------|
| **节点ID** | SysReq-F-{SEQ} |
| **节点名称** | 系统功能名称，从 BA IPO 名称转换而来 |
| **视角转换定义** | 以系统语言定义的 I/P/O 描述，含明确的人机分工 |
| **承接的 BA IPO** | 所承接的唯一 BA IPO 编号（1:1 映射） |
| **符合性分析** | 该功能如何满足所承接的 BA IPO |
| **性能指标** | 从上层继承的功能性能指标 |
| **设计指导** | 对 PA 构件的方向提示 |

#### 4.6 SysReq-NFR 架构末级节点的识别与定义

##### 4.6.1 指代对象

**SysReq-NFR 架构末级节点指代：一个 NFR 关注点（SR-NFR）经分解细化后的技术约束集合。**

SysReq-NFR 从 SR-NFR 出发，将"关注点+量化指标"细化为具体的、可工程实现的技术要求，并作为整体约束条件作用于 PA 的所有组件。与功能路径不同，NFR 约束不分配到特定 PA 组件——它们是 PA 整体设计时须满足的品质要求。

##### 4.6.2 架构分类结构

SysReq-NFR 按附录B**三级分类**统一组织（不按角色分）：

| 一级类目 | 包含的二级分类 |
|---------|--------------|
| **质量特性** | 性能、安全性、可靠性、可用性、易用性、人机工效、韧性、功能正确性 |
| **环境适应性** | 可扩展性、弹性、国际化、可移植性、互操作性、兼容性、可替换性 |
| **可实现性** | 物料可采购性、可生产性、可测试性、可交付性、可部署性、可运维性、可退役性、经济可承受性 |

分类视图切换是 SR-NFR→SysReq-NFR 的关键特征：SR-NFR 按角色组织以便相关方验证完备性，SysReq-NFR 按三级分类重组织以便工程师查阅全量约束。来自不同角色的同一分类关注点在此层合并为一个 SysReq-NFR 节点。

##### 4.6.3 识别方法：去重归并法

**第一步：收集 SR-NFR 全部节点**

将各角色类型下的 SR-NFR 节点汇集为完整清单。

**第二步：按三级分类去重归并**

从"按角色组织"切换到"按分类组织"——同一三级分类下、来源于不同角色类型的 SR-NFR 节点，合并为一个 SysReq-NFR 节点。合并后应确保量化指标取各来源中最严格的要求。

**第三步：细化为技术约束**

将每个 SysReq-NFR 节点的关注点+量化指标展开为具体的技术要求，明确工程实现条件。

**第四步：验证**

| 检查项 | 内容 |
|--------|------|
| **全覆盖** | 所有 SR-NFR 节点是否都已归并到对应的 SysReq-NFR？ |
| **约束不降级** | 合并后的量化指标是否不低于各来源中最高要求？ |
| **技术可实现性** | 每个 SysReq-NFR 的技术约束是否有明确的技术实现路径或工程标准支撑？ |

##### 4.6.4 与 SR-NFR 的边界划分

SR-NFR → SysReq-NFR 在分类视图上发生一次切换：

```
SR-NFR（按角色组织——方便相关方验证完备性）
├── 操作用户
│   ├── 质量特性-性能：屏应面响应时间 < 1s
│   └── 质量特性-易用性：……
├── 运维保障者
│   ├── 质量特性-可用性：系统可用性 > 99.9%
│   └── 质量特性-可靠性：……

              ↓ 去重归并，视图切换

SysReq-NFR（按分类组织——方便工程实现）
├── 质量特性
│   ├── 性能：响应时间 < 200ms（承接用户）+ 吞吐量……
│   ├── 可用性：> 99.9%（承接运维）
│   └── 安全性：MFA + TLS 1.3 + 审计日志（承接合规+运维）
├── 环境适应性
└── 可实现性
```

| 维度 | SR-NFR | SysReq-NFR |
|------|--------|-----------|
| **组织方式** | 按相关方角色→三级分类 | 按三级分类统一组织（不按角色分） |
| **抽象级别** | 关注点层面 | 技术约束层面 |
| **受益者** | 相关方验证完备性 | 工程师看全量约束 |
| **承接关系** | — | N:1（多角色同一分类的关注点合并） |
| **对 PA 的作用** | 间接约束 | 整体约束条件（不分配到特定构件） |

#### 4.7 PA 架构末级节点（组件）的识别与定义

##### 4.7.1 指代对象

**PA 架构末级节点指代：目标产品的一个组件——可独立分配、可独立替换、可独立验证的产品组成部分。**

PA 是五层设计链的终点（纯方案层），同时承接 SysReq-F（功能）和 SysReq-NFR（非功能）两路输入。下游是组件开发阶段（非系统设计），因此 PA 节点必须是**可独立管理的最小交付单元**。

##### 4.7.2 架构分类结构

PA 组件按产品类型选择不同的分类体系：

| 产品类型 | PA 分类组织方式 | 示例 |
|---------|---------------|------|
| **软件类** | 按技术架构层或业务域 | 前端层、后端服务、数据库、中间件 |
| **实物类** | 按物理组件或子系统 | 控制面板、驱动组件、传感器 |
| **知识类** | 按交付物形态 | 规范准则、操作指南、AI提示词、任务定义 |

> **康威定律与三独立**：三独立判定原则源于康威定律（Conway's Law）——系统结构会镜像组织的沟通结构。可独立分配的组件适配独立团队，可独立替换的组件适配独立变更，可独立验证的组件适配独立验收。三独立是同一本质在开发、运维、验收三个阶段的体现。

##### 4.7.3 识别方法：资产优先法

**第一步：资产优先检查**

面对 SysReq 详细定义末级条目集合，按优先级依次检查：

1. **复用**——组织中已有组件的组件能否直接承接这些条目？
2. **改进（Extend/Split/Merge）**——扩展、拆分或合并现有组件后能否承接？
3. **外购**——市场上是否有成熟的标准产品或开源方案可作为组件？
4. **新制**——以上均不可行时，进入第二步按三独立原则新建组件

**第二步：新制时的三独立判定**

将 SysReq 详细定义末级条目分组，对每个候选组件执行"三独立"检查：

| 判断维度 | 问题 | 说明 |
|---------|------|------|
| **可独立分配** | 这一组条目能否作为一个独立工作包分配给一个团队（或外包商）开发？他们是否需要频繁与其他团队协调接口和数据？ | 如需要频繁协调，说明组件边界不当，倾向合并或调整分组 |
| **可独立替换** | 如果将来要更换技术方案或供应商，能否只更换这一组而不牵连其他组件？ | 如更换需连带修改其他组件，说明组件间耦合过强 |
| **可独立验证** | 这一组条目能否独立测试其是否符合 SysReq？还是必须依赖整个系统才能验证？ | 如无法独立验收，说明组件边界不够清晰 |

一个候选组件须同时满足三个"是"才能作为独立的 PA 组件。任何一项为"否"则需调整组件边界。

> 分类体系的选择见§4.7.2。

##### 4.7.4 与 SysReq 的边界划分

| 维度 | SysReq 详细定义末级（9级） | PA 组件（架构末级） |
|------|------------------------|-------------------|
| **描述内容** | 系统场景活动——系统在特定场景下的具体行为 | 产品组件——承载这些活动的实现载体 |
| **分配关系** | N 条详细定义条目 | 聚合为一个组件（N:1） |
| **角色** | 对 PA 是需求 | 纯方案——设计链终点 |
| **下游** | 无（已分配到 PA） | 组件开发阶段 |

#### 4.8 架构末级节点识别后的验证

> **注**：§4.2~§4.7 各节的识别方法最后一步均已包含节点验证检查（如 N:1 合理性检查、跨层对齐等）。本节从跨层视角补充全局验证维度，确保各层验证标准一致、无遗漏。

完成识别后，从以下维度验证：

| 检查项 | 说明 | 适用层 |
|--------|------|-------|
| **角色完整性** | 所有相关方均已识别为用户角色类型 | SR-F |
| **触发完整性** | 每类角色的所有业务触发均已识别 | SR-F |
| **NFR 角色覆盖** | 所有 NFR 相关方角色类型均已识别（操作用户/运维者/合规者/管理者/研发者等） | SR-NFR |
| **NFR 分类覆盖** | 各角色类型的 NFR 关注点是否已按附录B的分类体系完整覆盖？有无遗漏的 NFR 维度？ | SR-NFR |
| **NFR 约束传递** | SysReq-NFR 的约束条件是否明确且可验证？是否作为整体约束而非分配到特定 PA 构件？ | SysReq-NFR |
| **N:1合理性** | 每个节点承接条目数合理，无过度合并或碎片 | 全部 |
| **跨层对齐** | 相邻层的节点边界可对应追溯，无跨层跳跃 | 全部 |
| **三独立验证** | 每个 PA 组件是否同时满足可独立分配、可独立替换、可独立验证？ | PA |

### 五、业务架构设计

本章专门阐述业务架构（BA）——系统设计五层中承担功能需求完整性保障的关键层级。阐明业务架构是什么、为什么需要、如何开发，为后续系统设计过程中 SR→BA 结对设计提供理论基础。

#### 5.1 业务架构定义与作用

**业务架构**（Business Architecture, BA）是各类用户角色使用目标产品（产品 A）开展期望业务的**运行概念说明**（Operational Concept, OpsCon），即各类用户角色操作目标产品 A 的**操作规程**。该概念源于系统工程标准（ISO/IEC/IEEE 29148:2018, 15288:2015），以 IPO（输入→处理→输出）作为结构化描述单元，成为衔接相关方需求（SR）与系统需求（SysReq）的第3层。

在系统工程中，OpsCon 通常以场景叙事描述用户与系统的交互过程（"谁在什么情况下做了什么，系统如何响应"），帮助相关方在系统实现前确认需求是否完整。本方法论将其具体化为 IPO 这种结构化描述单元：每个 IPO 对应一个操作场景，以 I→P→O 三要素精确定义场景边界，使场景描述既具备业务可读性（相关方可评审），又具备工程可映射性（可直接推导 SysReq）。

BA 在五层设计链中的定位：

| 维度 | 说明 |
|------|------|
| **在上游（SR）担当的角色** | 方案方——承接 SR 功能部分详细定义末级条目，将"用户有什么业务"翻译为"产品如何支撑该业务" |
| **在下游（SysReq）担当的角色** | 功能需求输入源——BA IPO 去重后直接映射到 SysReq-F 架构定义，每个唯一 IPO 对应一个 SysReq 架构末级节点，SysReq-F 的功能清单由此派生 |
| **结构特征** | 五层中唯一仅有**架构定义**而无**详细定义**的层级——IPO 条目自身承担"详细定义"角色，不再另设详细定义子文档 |
| **分配机制** | BA→SysReq 是去重后的**直接映射**（非 N:1 分配），这是 BA 保障功能需求完整性的结构保证 |

BA 保障功能需求完整性的工作机制在于：BA 描述的是各类用户角色使用目标产品 A 的全部操作场景，目标产品 A 的功能需求正是为支撑这些场景而产生的。当 BA 覆盖了所有操作场景时，由此推导出的功能需求就是完整的。因此 BA 是功能需求完整性保障的系统性方法（非功能需求没有等价方法——路径差异的完整说明见 §1.2）。

> **注**：本文的"用户角色"泛指所有与目标产品发生交互的外部实体，包括操作用户（自然人）、输入/输出连接设备与系统、用户方保障者、研制方服务者、第三方运维者，以及自然环境（光照、温度、湿度等）——即产品边界外、向产品提供输入或期望从产品获得输出的所有角色。

#### 5.2 业务架构开发方法

BA 的开发方法因产品类型而异。所有产品类型均可采用**用户角色架构锚定法**——以目标产品的全量用户角色集为推导锚点。流水线类产品（目标产品的用户角色中包含明确的输出产品）还可采用**输出产品架构锚定法**，以输出产品的 PA 节点类型为推导锚点。输出产品架构锚定法是通用方法在流水线类产品上的特化：通用方法的第1步（识别用户角色集）在此简化为直接以输出产品的 PA 节点类型为锚点，后续步骤与通用方法一致。

##### 5.2.1 用户角色架构锚定法（通用方法）

**核心思想**：目标产品的 BA 由与产品交互的各类用户角色及其期望业务共同锚定——角色集完备则 BA 完备，由此推导出的 SysReq-F 集合也必然是完整的。

**开发思路**：

1. **识别用户角色集**——识别与目标产品的所有交互方。任何向目标产品输入和期望从目标产品输出的实体均可视为用户角色，包括操作用户（自然人）、输入/输出连接外部系统与设备、用户方保障者、研制方服务者、第三方运维者，以及自然环境（光照、温度、湿度等）。**关键**：角色集穷举是 BA 完备的前提——遗漏任何角色类将直接导致该角色的操作场景在 BA 中缺失，从而造成 SysReq 的功能缺口。

2. **按角色识别业务清单**——对每类角色，逐一识别该角色通过目标产品要实现的业务。用户角色类型不同，业务形态也不同：自然人类角色识别其要完成的工作任务（如"查看趋势图""提交采购申请"），外部系统角色识别与目标产品之间的交互功能和数据接口（如"上报读数""推送订单数据"），自然环境角色识别目标产品需要响应的环境条件及其范围（如"环境温度-20°C~60°C"）。**关键**：每个角色的全部期望业务都应纳入清单——这是后续推导功能需求完整性的基础。

3. **以 IPO 定义业务过程**——将业务清单转化为结构化业务过程描述，以 IPO（输入→处理→输出）作为描述单元。每个 IPO 定义业务过程的边界：接收什么输入、执行什么处理、产出什么输出。IPO 的 I 和 O 的形态因用户角色类型而异——操作用户为文档/指令，外部系统为消息/数据，传感器为信号/读数（详见 §4.4.1）。**关键**：IPO 的颗粒度以"一份输入单元→一份输出单元"为边界，确保每个业务过程的语义独立、边界清晰。

4. **归并去重**——识别不同角色、不同业务之间的复用关系。多项业务可能共享同一业务过程（如"提交审批"这个 IPO 出现在采购审批、报销审批、变更审批等多个业务中），归并去重后的唯一 IPO 即为 BA 架构末级节点。**关键**：去重是 BA 保证 SysReq 功能清单不冗余的机制——同类型的业务过程只需在 BA 中定义一次，后续在 SysReq 层映射时也自然归并为单一功能需求。

5. **确定架构树**——将去重后的唯一 IPO 清单组织为 0 级至 5 级的完整 BA 树形结构，按业务域分类（采购、生产、售后、研发、运营等）。**完备性判据**：用户角色集完全覆盖、每个角色的业务清单完全覆盖——BA 即视为完整。

> **反向回补机制**：上述分析过程中可能发现当前 SR 未覆盖的业务——某角色有某项期望业务但 SR 中无对应需求——此时视为发现 SR 缺口，应回补充实 SR 后方可进入下一设计步骤。

> **操作细节见 §4.4.3**（BA IPO 的具体识别与验证方法）和 **§6.4**（SR→BA 结对设计的完整操作流程），此处不再展开。

**开发示例**：以物联网环境监测平台为例，展示上述开发思路的具体应用。目标产品 A 是一个物联网环境监测平台，用户使用它采集、存储和分析环境数据。用户角色包括操作用户（环境分析师）和外部传感器（温度/湿度/PM2.5 传感器），以及外部告警系统。

| 步 | 内容 | 核心要点 |
|----|------|---------|
| **①** | 识别用户角色集 | 操作用户（环境分析师）+ 外部传感器（温度/湿度/PM2.5）+ 外部告警系统。传感器是非人类角色，其"业务"是周期性上报传感数据 |
| **②** | SR 功能部分详细定义 | 分析师的业务：查看趋势图、生成分析报告、配置告警规则；传感器的业务：上报温度/湿度/PM2.5 读数；告警系统的业务：接收告警通知 |
| **③** | 分配至 BA IPO——I/O 形态因角色而异 | 传感器 IPO：I=传感信号（温度读数）→ P=自动校验+格式化存储 → O=传感记录；分析师 IPO：I=查询条件（时间段、指标）→ P=数据聚合+图表生成 → O=趋势分析报告 |
| **④** | BA 仅架构定义 | IPO 自身承担详细定义，不展开详细定义子文档 |
| **⑤** | IPO 去重→SysReq | 同一"数据校验与存储"IPO 被多个传感器类型（温度/湿度/PM2.5）复用，去重后仅保留一个 SysReq 节点 |

##### 5.2.2 输出产品架构锚定法（流水线类产品的特化路径）

**关系**：本方法是用户角色架构锚定法在流水线类产品上的特化——当目标产品的用户角色中包含明确的输出产品（产品 B）时，跳过角色遍历步骤，直接以 B 的 PA 节点类型为推导锚点：输出产品有什么类型的构件，就需要有什么操作步骤来生产它。

**方法正确性**：本方法的有效性源于流水线类产品的本质——目标产品 A（流水线）的唯一存在意义就是生产输出产品 B。因此，A 的全部业务操作必然以生产 B 为导向，A 的 BA 也因此必然是"生产 B 的业务过程"的集合。当 B 的 PA 节点类型被穷举时（即 B 全量构件的类型均已识别），为每类构件定义的生产步骤就覆盖了 A 的全部操作场景——B 的 PA 完备性蕴含了 A 的 BA 完备性。这使原本需要遍历全部用户角色的通用方法第1步，简化为识别 B 的 PA 节点类型这一更可控的工作。

**核心思想**：目标产品 A（流水线）的期望业务即生产输出产品（产品 B）的生产过程。推导逻辑如下：

1. **识别 B 的产品架构及节点类型**——明确 A 需要支持生产哪些类型的构件。这是开发 A 的 BA 的前置条件，因为 A 的存在意义就是生产 B——B 由什么组成，A 就需要有什么对应的生产步骤。同时，分析"生产"这些节点类型所需的业务过程，也可能反向揭示当前 SR 中尚未覆盖的需求（如发现某类 B 构件需要"评审"步骤但 SR 中未提及），需回补充实 SR 后方可进入下一步。

2. **为每类节点定义操作步骤**——对 B 的每一类架构节点，定义使用 A 生产该类节点的业务过程。以 IPO（输入→处理→输出）描述每个操作步骤，所有 IPO 的集合构成 A 的 BA。**关键**：这些 IPO 描述的不仅是"生产 B 构件"本身，还包括触发生产（接收指令）、准备输入（加载前置文档）、验证产出（质量检查）等全生命周期操作。

3. **操作步骤的集合构成 A 的 BA**——所有操作步骤的 IPO 清单即为 A 的业务架构。与通用方法一样，BA 是五层中唯一仅有架构定义而无详细定义的层级，IPO 条目自身承担"详细定义"角色，去重后直接映射到 SysReq-F。

> **流水线类产品的 IPO 特点**：在流水线类产品中，每个 IPO 的输入（I）和输出（O）都是**广义的文档**——即具有完整表意能力的信息载体。这些文档包括正式设计产物（需求定义、架构定义、规范准则），也包括轻量信息载体（变更通知、审批请求、触发指令、核查清单等）。即使看似由"事件"触发（如定时计划、状态变更通知），实质仍是待处理的文档或信息到达——时间事件仅表示"到时间处理这批文档"，变更通知必然附有所属文档。因此，在流水线类产品中，IPO 的 I 和 O 始终以广义文档形态出现。

**完备性判据**：当 B 的所有节点类型都在 A 的 BA 中有对应的生产操作步骤时，A 的 BA 即视为完整——由此推导出的 A 的功能需求集合也必然是完整的。反之，若 B 的某类构件在 A 的 BA 中找不到对应的生产 IPO，则说明 BA 存在缺口。

> **操作细节见 §4.4.3**（BA 节点识别）和 **§6.4**（结对设计流程）。

**开发示例**：以流水线类产品为例，展示上述开发思路的具体应用。目标产品 A 是一条流水线，用户使用它"生产"输出产品 B。以下展示从 B 的 PA 锚点出发，经 SR→BA→SysReq 的完整推导过程。

| 步 | 内容 | 核心要点 |
|----|------|---------|
| **①** | 识别**输出产品（产品 B）的产品架构（PA）**及架构节点类型——明确 B 由哪些类型的构件组成，作为整条推导链的输入前提 | B 的 PA 节点类型是输出产品架构锚定法推导 BA 的锚点：有什么构件，就需要有什么操作步骤来生产它。同时，分析"生产"这些节点类型所需的业务过程，也可能发现原有 SR 中未覆盖的需求——需回补至 SR。第②步的 SR 详细定义应是**经此轮回补后**的完整版本 |
| **②** | 对**相关方需求（SR）**功能部分展开详细定义——将每个 SR-F 末级节点分解到可分配到唯一**BA IPO**的原子粒度 | SR 功能部分的详细定义是 BA 的需求输入，分解的充分性直接影响 BA 的完整性 |
| **③** | 结合②的分解结果与①的节点类型锚点，将每条 SR 详细定义末级条目分配到唯一**BA IPO**。每个 BA IPO 以**输入→处理→输出（IPO）**定义业务操作边界 | BA 将 SR 中的功能需求转化为各类用户角色的操作场景说明（OpsCon），IPO 三要素确保操作边界清晰 |
| **④** | BA 的架构树止于 IPO 末级节点——**不进一步展开详细定义子文档**，IPO 条目自身承担"详细定义"的角色 | BA 是五层设计链中唯一仅架构定义、无详细定义的层级（§1.3），IPO 清单即 BA 的完整内容 |
| **⑤** | 将 BA IPO **去重**后（同一 IPO 可能被多条 SR 条目映射），按系统功能分类重组，直接映射至**系统需求（SysReq）**功能部分架构定义。每个唯一 IPO 对应一个 SysReq 架构末级节点 | BA→SysReq 是去重后的直接映射（§6.5 第3步），而非 N:1 分配。功能需求经此路径进入 SysReq，BA 保障功能需求完整性的机制由此体现 |

功能需求的完整路径（本示例覆盖范围以 **加粗** 标记）：

```
B 的 PA 节点类型（推导锚点）
  → OR → SR（功能部分）                               ← 五层设计前置阶段
    → **BA（目标产品 A 的业务架构，IPO 清单）**            ← 本示例覆盖范围
      → **BA IPO 去重 → SysReq（功能架构）**           ← 本示例覆盖范围
        → PA（后续系统设计，见 §五）                   ← 超出本节范围
```

### 六、系统设计过程

本章综合应用§一~§四的定义、原则和设计活动方法，将五层设计链路转化为可执行的六步工作流程。先以结构图呈现五层设计架构全貌，再按需求文档与方案文档的结对关系（§1.1 "需求/方案相对"），依次执行需求分解、需求分配与符合性分析，从原始需求分析到双向追溯验证构成完整的设计操作规范。该流程以正向设计为默认表达，同样适用于逆向工程等其他设计方向。

#### 6.1 五层设计结构总览

以结构图和层级对照表直观呈现五层设计链结构——从原始需求经相关方需求、业务架构、系统需求到产品架构，包含功能与非功能需求的不同分配路径。图中 `[需求]` / `[需求/方案]` / `[方案]` 标签是"需求/方案相对"（§1.1）在结构层面的直接体现，`[方案]` 标签表示该层扮演方案角色不向下传递需求（设计链终点）。

```
第1层：原始需求及分析 [需求]
    ↓ N:1 分配
第2层：相关方需求 [需求/方案]
    ├─ 架构末级节点 ↔ 详细定义末级节点
    │
    ├─ 功能部分 ↓ N:1 分配
    │   第3层：业务架构 [方案]（独立存在）
    │   └─ 架构末级节点（IPO）—— 仅架构定义，无详细定义
    │        ↓ IPO直接映射（去重后）
    │   第4层：系统需求 [需求/方案]
    │   ├─ 架构末级节点 ↔ 详细定义末级节点
    │   │
    │   └─ ↓ N:1 分配
    │       第5层：产品架构 [方案]
    │       ├─ 架构末级节点 ↔ 详细定义末级节点
    │
    └─ 非功能部分 ↓ N:1 分配
        第4层：系统需求（非功能）[需求/方案]
        ├─ 架构末级节点 ↔ 详细定义末级节点
        │
        └─ ↓ N:1 分配
            第5层：产品架构 [方案]
```

> **图例说明**：`↔` 表示该层同时存在**架构末级节点**和**详细定义末级节点**两种类型的末级节点（共存关系，非双向映射）。

| 层级 | 文档名称 | 架构末级节点 | 详细定义末级节点 | 承接关系 |
|------|---------|------------|----------------|---------|
| 1 | 原始需求及分析 | - | 末级需求 | - |
| 2 | 相关方需求 | 相关方需求条目 | 对条目的分解 | 承接原始需求末级（N:1） |
| 3 | 业务架构 | IPO（业务组件） | —（无） | 承接相关方需求详细定义末级（N:1） |
| 4 | 系统需求 | 系统需求条目 | 对条目的分解 | 功能部分：承接业务架构末级（直接映射）<br>非功能部分：承接相关方需求非功能末级（N:1） |
| 5 | 产品架构 | 产品构件 | 实现方案 | 承接系统需求详细定义末级（N:1） |

以表格形式给出五个设计层级各自的统一定义，包含层级名称、本质说明、文档构成和承接关系，为各层设计提供快速查阅的参照。

| 层级 | 名称 | 定义 | 本质 | 文档构成 |
|------|------|------|------|---------|
| **1** | **原始需求** | 未经规范化的、来自各相关方的原始表述，是系统设计的起点 | "原材料"——只有需求，没有方案 | 原始需求清单 |
| **2** | **相关方需求** | 对原始需求的规范化整理和组织，同时其功能部分是业务架构的需求输入，非功能部分是系统需求架构的非功能输入 | 既是方案（对第1层），其功能部分和非功能部分都是需求（分别对第3层和第4层） | 架构定义 + 详细定义 |
| **3** | **业务架构** | 用户使用待开发系统及其互联系统开展业务的运行概念（Operational Concept），即业务过程说明 | 独立存在的方案——仅架构定义（IPO清单），无需详细定义，用于开发完整的功能需求 | 仅架构定义（IPO） |
| **4** | **系统需求** | 将业务操作过程转化为系统可实现的功能需求，并将相关方非功能需求转化为系统非功能需求 | 既是方案（功能部分对第3层，非功能部分对第1层非功能），又是需求（对第5层） | 架构定义 + 详细定义 |
| **5** | **产品架构** | 将系统需求转化为具体的实现方案，定义产品的构件组成及其职责和需求，为后续构件开发提供输入 | 实现方案——构件开发的直接输入 | 架构定义 + 详细定义 |

**关键要点**：
- 第1层是唯一**只有需求、没有方案**的层，不遵循"设计文档结对"模型
- 第3层是唯一**只有架构定义、没有详细定义**的层，其IPO自身承担"需求"角色
- 第2、5层是完整文档（架构定义 + 详细定义），其中第2层（SR）兼有方案和需求双重角色，第5层（PA）为纯方案
- 第4层（SysReq）兼有方案和需求双重角色，且有两套架构体系：功能需求和非功能需求（平行架构）

以流程图形式展示六步工作流程的整体结构及其功能/非功能两个分支的处理路径，标注各步的输入输出关系和先后顺序。

    │
    ▼
[第1步] 原始需求分析及相关方需求架构定义
    ├─ 输出：原始需求末级条目分解
    │       相关方需求架构定义（功能+非功能）
    │
    ├─ 功能分支 ↓
    │  [第2步] 相关方需求功能部分详细定义及业务架构定义
    │  ├─ 输出：相关方需求功能部分详细定义
    │  │       业务架构定义（IPO）
    │  │
    │  └─ [第3步] 系统需求功能部分架构定义

    │

       └─ 输出：系统需求非功能部分架构定义

    ├─ 功能部分：系统需求详细定义分解 → 分配到产品架构末级节点（1:1约束）

#### 6.3 第1步：OR → SR

对原始需求进行条目拆分、规范化和冲突检测后，执行三个核心设计活动建立相关方需求架构。

对应设计文档结对**1：OR 条目级需求 ↔ SR 架构定义**。

本步分两个阶段：先完成 OR 预处理（条目拆分、规范化、交叉检测），再执行三个核心设计活动。

**阶段一：OR 预处理**

**(a) 条目拆分**。从各相关方收集原始需求表述，每条原始需求可能混杂多角色、多语义或 FR/NFR 混杂的内容。按"四单一"原则逐一拆分至可分配粒度：

| 原则 | 问题 | 操作 |
|------|------|------|
| **单一角色** | 是否涉及多个用户角色？ | 拆到每条仅涉及一个角色 |
| **单一语义** | 是否描述了多个独立功能或关注点？ | 拆到每条仅含一个独立语义 |
| **单一路径** | 是否同时包含 FR 和 NFR 内容？ | 拆为 FR 条目和 NFR 条目，标注类型 |
| **可分配** | 拆分后能否直接对应到 SR 架构中一个末级节点？ | 如仍无法对应，继续拆分 |

拆分终止标准：每条 OR 条目满足"单一角色 + 单一语义 + 单一路径"，且可 1:1 映射到 SR 架构末级节点（SR-F 或 SR-NFR）。拆分后的条目分配 OR 编号（§3.4.2：OR-{SEQ}）。

> 超额拆分（条目过细）不影响设计链正确性——SR 层通过 N:1 映射自然合并；拆分不足（条目仍未达到可分配粒度）则阻塞分配链路。遇不确定时倾向于拆分。

**(b) 规范化**。对每条需求执行四项检查：**完整性**（描述、类型、优先级齐全）、**清晰性**（替换模糊术语，如"实时"→"延迟 < 100ms"）、**可测试性**（功能需求输入/处理/输出明确，非功能需求有具体指标）、**独立性**（需求间无过度耦合）。

**(c) 冲突与重复检测**。在拆分后的需求之间进行交叉检查：冲突类型包括**直接冲突**（语义矛盾）、**间接冲突**（同时实现可能导致行为不一致）、**可合并冲突**（可通过调整合并解决）；重复类型包括**完全重复**、**部分重复**、**相似重复**。

**阶段二：三个核心设计活动**

**(d) 需求分解**（§3.1）。执行需求分解到可分配到 SR 架构末级节点的原子粒度，完成语义分解四项自检和指标分解三场景判定。

**(e) 需求分配**（§3.2）。将分解后的原始需求末级条目按功能/非功能分类，按 1:1 约束逐一分配到 SR 架构末级节点（N:1），验证全覆盖/1:1/N:1 合理性。

**(f) 符合性分析**（§3.3）。确认每个 SR 架构末级节点的定义和性能指标充分承接所分配的原始需求末级条目。

SR 架构的构建应基于已有需求分类资产（§2.4），遵循复用→改进→新增的资产优先顺序（§2.4.1）。

结对设计原则（§2.3）：SR 架构的分类骨架应在 OR 条目分解前先行建立（至少到分类节点层级），在骨架基础上建立初步映射后，再并行填充 OR 详细定义和 SR 架构节点四要素。OR 预处理阶段（条目拆分、规范化、冲突检测）的结果直接影响 SR 骨架的合理性。

> **OR 条目格式**。经过预处理和规范化后的 OR 条目采用以下标准格式：

| 字段 | 说明 |
|------|------|
| **编号** | OR-{SEQ}（如 OR-001） |
| **标题** | 一句话概括需求 |
| **规范化描述** | 经过规范化的需求描述，功能/非功能未拆分但清晰可辨 |
| **类型** | 功能需求 / 非功能需求（OR 层标注类型，到 SR 层做正式拆分） |
| **性能指标** | 可量化的数值指标及测量条件（不可衡量则不成为需求） |
| **优先级** | P0/P1/P2/P3 |
| **验收标准** | 可量化的验收条件 |

OR 收集阶段应保留元数据字段（提出方、提出时间、来源渠道、原始表述等），用于追溯，但最终归档的条目格式以上表为准。

**输入**：从各相关方收集的原始需求表述
**输出**：原始需求末级条目分解 + 相关方需求架构定义（架构末级节点四要素）

> **相关指引**：SR-F 架构末级节点的识别方法详见§4.2（任务闭环法），SR-NFR 架构末级节点的识别方法详见§4.3（角色→分类归并法）。

#### 6.4 第2步：SR(功能) → BA

对相关方需求功能部分展开详细定义并分解分配，识别业务活动边界，建立业务架构IPO清单。

对应设计文档结对**2：SR 详细定义末级 ↔ BA 架构（IPO）**。

对 SR 功能部分执行三个核心设计活动，同步构建业务架构。

**(a) 需求分解**（§3.1）。对 SR 功能部分架构末级节点展开详细定义——执行需求分解到可分配到唯一 BA IPO 的原子粒度，完成语义分解四项自检和指标分解三场景判定。

**(b) 需求分配**（§3.2）。每一条 SR 详细定义末级条目按 1:1 约束分配到唯一 BA IPO，验证全覆盖/1:1/N:1 合理性。

**(c) 符合性分析**（§3.3）。确认每个 BA IPO 的定义和指标充分承接所分配的 SR 详细定义末级条目。

**(d) 业务架构识别**。此步骤同时也是 BA 的构建过程：从 SR 详细定义集合中识别业务活动边界，定义每个 IPO 的输入→处理→输出。BA 是唯一只有架构定义、无详细定义的层级（§1.3），IPO 自身承担"详细定义"的角色。

结对设计原则（§2.3）：采用骨架→填充两阶段工作——先快速约定 BA 的架构骨架（IPO 清单及分类）并建立初步映射，再并行填充 SR 详细定义和 BA IPO 四要素。

**输入**：相关方需求架构定义（功能部分）
**输出**：相关方需求功能部分详细定义 + 业务架构定义（IPO 清单）

BA 的识别应基于已有业务过程和组织能力资产，而非从零推导业务活动——脱离了真实业务依据的 BA 架构逻辑上自洽但现实中无法落地（§2.4）。

> **相关指引**：BA 架构末级节点（IPO）的识别方法详见§4.4（输入输出流转链法），BA 开发方法（用户角色架构锚定法 / 输出产品架构锚定法）详见§5.2。

> **SR 功能部分条目格式**。SR 功能部分架构末级节点（SR-F-{SEQ}）包含以下属性：

| 属性 | 说明 |
|------|------|
| **ID** | SR-F-{3位序号}（如 SR-F-001） |
| **标题** | 一个完整的功能或业务需求 |
| **定义** | 清晰的需求叙述，描述用户/系统应该能够做什么 |
| **承接的 OR** | 承接的原始需求末级条目列表（N:1） |
| **符合性分析** | 该节点如何满足所承接的 OR 条目 |
| **设计指导** | 对业务架构设计的方向提示 |
| **性能指标** | 功能相关的性能要求（处理效率、准确率等） |

SR 详细定义末级条目编号格式：`SR-F-{SEQ}.{子序号}`（如 SR-F-001.01、SR-F-001.02）。

> **BA IPO 格式**。BA 是唯一只有架构定义、无详细定义的层级（§1.3），其架构末级节点以 IPO（输入→处理→输出）三元组作为基本描述单元：

| 维度 | 定义 | 说明 |
|------|------|------|
| **Input（输入）** | 业务组件接收的输入 | 形式、数据结构、数据来源、前置条件 |
| **Process（处理）** | 对输入进行的处理 | 处理规则、算法、步骤、决策点、约束 |
| **Output（输出）** | 业务组件生成的输出 | 形式、数据结构、验收标准、后续使用 |

BA 一个 IPO 的描述体量取决于期望的输出物颗粒度——期望输出是设计制品时，一个 IPO 对应一份制品（或可由单人完成的部分）。编号格式：`BA-{SEQ}`（如 BA-001）。

#### 6.5 第3步：BA → SysReq(功能架构)

将BA层IPO去重后按系统功能分类重组，建立系统需求功能部分架构作为后续详细定义的分解起点。

对应设计文档结对**3 的前半：BA IPO（去重后）↔ SysReq 架构定义**。

将 BA IPO 去重后按系统功能分类重组为 SysReq 架构。

**(a) IPO 去重**。同一 BA IPO 可能被多条 SR 详细定义末级条目映射——将重复出现的 IPO 归并为唯一条目，形成去重后的 IPO 清单。

**(b) 系统功能分类重组**。将去重后的 IPO 按系统功能分类重组为 SysReq 架构末级节点，按业务领域和功能域组织分类层级。注意此步骤不遵循 N:1 分配模型——BA 到 SysReq 是去重后的**直接映射**，每个唯一 IPO 对应一个 SysReq 架构末级节点。本步骤的输出将为第 5 步中 SysReq 功能部分的详细定义提供分解起点。

**输入**：业务架构 IPO 清单
**输出**：系统需求功能部分架构定义

> **相关指引**：SysReq-F 架构末级节点的识别方法详见§4.5（视角转换法），BA IPO 到 SysReq-F 的转换示例详见§4.5.3。

SysReq 架构的重组应基于已有系统边界和技术域资产——模块划分、接口约定、领域界限决定了架构树的分支结构。忽略已有边界是 SysReq 层最常见的错误，直接导致 PA 层的职责重叠和冲突（§2.4）。

结对设计原则（§2.3）：SysReq 功能架构的分类骨架应在 IPO 映射前先行建立，再按直接映射关系将各 IPO 填入对应架构末级节点。骨架阶段不追求 IPO 映射的完整性，而是先确定系统功能分类的树形结构。

> **SysReq 功能部分条目格式**。SysReq 功能部分采用 0-9 级层级结构，其中 **5 级（组件实例）是架构末级**，**9 级（场景活动）是详细定义末级**：

| 层级 | 名称 | 说明 |
|------|------|------|
| 0级 | 根节点 | 整个系统的顶层根节点 |
| 1级 | 独立信息系统 | 一个独立的信息系统或子系统 |
| 2级 | 功能分支 | 功能需求分支入口 |
| 3级 | 业务组件类型 | 流程、工单、计划、指标等 |
| 4级 | 业务组件小类 | 组件类型下的小类分类 |
| **5级** | **组件实例（架构末级）** | **承接 BA IPO，含节点四要素** |
| 6级 | 功能模块 | 以页面/标签页为载体的功能模块 |
| 7级 | 系统用例 | 按钮/菜单项提供的功能 |
| 8级 | 用例场景 | 正常/异常/替代场景 |
| **9级** | **场景活动（详细定义末级）** | **分配到 PA 末级（1:1 约束）** |

5 级节点属性：

| 属性 | 说明 |
|------|------|
| **节点ID** | SysReq-F-{SEQ}（如 SysReq-F-001） |
| **节点名称** | 系统功能名称 |
| **定义** | 功能的简要描述 |
| **承接的 BA IPO** | 承接的 BA IPO 列表（直接映射） |
| **符合性分析** | 如何满足所承接的 IPO |
| **性能指标** | 从上层继承的功能性能约束 |
| **设计指导** | 对 PA 构件的方向提示 |

详细定义末级（9 级）编号格式：`SysReq-F-{SEQ}.{层级}.{序号}`（如 SysReq-F-001.1.1.1）。

#### 6.6 第4步：SR(非功能) → SysReq(非功能架构)

对相关方需求非功能部分执行三个核心设计活动，沿平行路径建立系统需求非功能部分架构。

对应设计文档结对**3 的后半：SR-NFR 详细定义末级 ↔ SysReq-NFR 架构定义**（NFR 平行路径）。

对 SR 非功能部分执行三个核心设计活动，组建 SysReq 非功能架构。

**(a) 需求分解**（§3.1）。对 SR 非功能部分架构末级节点展开详细定义——执行需求分解到原子粒度，完成语义分解四项自检和指标分解三场景判定。

**(b) 需求分配**（§3.2）。按 1:1 约束分配到 SysReq-NFR 架构末级节点（N:1），验证全覆盖/1:1/N:1 合理性。架构分类按 NFR 类型组织（性能、安全、可用性、可扩展性等，见附录B）。

**(c) 符合性分析**（§3.3）。确认每个 SysReq-NFR 架构末级节点的定义和指标充分承接所分配的 SR-NFR 末级条目。

此步骤与第 2-3 步在时间上可并行——功能部分和非功能部分独立展开（§1.2）。非功能需求不经过 BA，走 SR-NFR → SysReq-NFR 的平行路径。

**输入**：相关方需求架构定义（非功能部分）
**输出**：相关方需求非功能部分详细定义 + 系统需求非功能部分架构定义

> **相关指引**：SR-NFR 架构末级节点的识别方法详见§4.3（角色→分类归并法），SysReq-NFR 架构末级节点的识别方法详见§4.6（去重归并法），NFR 分类体系见附录B。

> **SR 非功能部分条目格式**。SR 非功能部分架构末级节点（SR-NFR-{SEQ}）包含以下属性：

| 属性 | 说明 |
|------|------|
| **ID** | SR-NFR-{3位序号}（如 SR-NFR-001） |
| **需求类型** | 性能/安全/可用性等明确的非功能需求类型 |
| **需求类别** | 质量特性 / 环境适应性 / 可实现性（三大类之一） |
| **定义** | 系统应满足的质量属性描述 |
| **承接的 OR** | 承接的原始非功能需求末级（N:1） |
| **符合性分析** | 如何满足所承接的需求 |
| **量化指标** | 具体的、可测量的指标（含数值、单位、条件） |
| **测量方法** | 如何测量/验证该指标（性能测试、安全审计等） |

> **SysReq 非功能部分条目格式**。SysReq 非功能部分采用 0-6 级平行架构：

| 层级 | 名称 | 说明 |
|------|------|------|
| 0级 | 非功能需求根节点 | 顶层根节点 |
| 1级 | 需求类别 | 三大类：质量特性 / 环境适应性 / 可实现性 |
| 2级 | 需求类型 | 具体类型（性能、安全、可测试性等） |
| 5级 | **非功能需求承接层（架构末级）** | **承接 SR-NFR** |
| 6级 | 操作序列/验证标准 | 验证步骤和标准 |

5 级节点属性：

| 属性 | 说明 |
|------|------|
| **节点ID** | SysReq-NFR-{SEQ}（如 SysReq-NFR-001） |
| **节点名称** | 非功能需求名称 |
| **需求描述** | 非功能需求的详细说明 |
| **类型** | 所属类别（性能、安全、可用性等） |
| **承接的 SR-NFR** | 承接的相关方非功能需求列表（N:1） |
| **量化指标** | 具体的、可测量的指标值 |
| **验证方法** | 如何验证指标 |
| **约束条件** | 对 PA 的约束 |

SysReq-NFR 架构的分类应基于已有技术标准和规范资产——质量特性、安全等级、合规要求等分类维度来源于行业标准和组织技术规范，而非临时定义（§2.4）。

结对设计原则（§2.3）：SysReq-NFR 架构的分类骨架应在 NFR 详细定义前先行建立，建立初步映射后，再并行填充 SR-NFR 详细定义和 SysReq-NFR 架构节点四要素。骨架阶段以 NFR 类型（性能、安全、可用性等）作为分类组织方式。

#### 6.7 第5步：SysReq → PA

对系统需求功能与非功能部分分别展开详细定义并分配到产品构件，同步完成产品架构定义。

对应设计文档结对**4：SysReq 详细定义末级 ↔ PA 架构定义**。

对 SysReq 执行三个核心设计活动，同步构建产品架构。

**(a) 需求分解**（§3.1）。对 SysReq 功能部分和非功能部分的架构末级节点分别展开详细定义——执行需求分解到可分配到唯一 PA 架构末级节点的原子粒度，完成语义分解四项自检和指标分解三场景判定。

**(b) 需求分配**（§3.2）。每一条 SysReq 详细定义末级条目按 1:1 约束分配到唯一的 PA 架构末级节点（产品构件），验证全覆盖/1:1/N:1 合理性。

**(c) 符合性分析**（§3.3）。确认每个 PA 构件的定义和性能指标充分承接所分配的 SysReq 末级条目。

**(d) PA 构件识别**。PA 构件的划分是对 BA 层业务边界的读取而非发明——构件不能跨越业务边界。判定遵循两步法：**首先**从组织已有组件资产出发（§2.4），遵循复用→改进→新增优先级（§2.4.1），优先复用现有构件；**其次**，当现有资产无法满足需新制时，基于**三独立原则**进行判定。三独立原则源于康威定律（Conway's Law）——系统结构镜像组织的沟通结构。PA 架构末级节点的完整识别方法（含资产优先检查、三独立判定、产品类型分类体系）详见§4.7，具体判定方法和验证步骤见§4.7.3。

**结对设计原则**（§2.3）：采用骨架→填充两阶段工作——先快速约定 PA 的架构骨架（构件清单及分类）并建立初步映射，再并行填充 SysReq 详细定义和 PA 构件四要素。

> **PA 架构末级节点格式**。PA 架构末级节点（产品构件）包含以下属性：

| 属性 | 说明 |
|------|------|
| **构件ID** | PA-{SEQ}（如 PA-001） |
| **构件名称** | 构件名称 |
| **功能角色** | 标注构件的类型/角色分类（因产品类型而异，在各产品准则中实例化） |
| **定义** | 构件的职责和用途描述 |
| **承接的 SysReq** | 承接的 SysReq 详细定义末级列表（N:1） |
| **符合性分析** | 如何满足所承接的 SysReq 条目 |
| **实现方案** | 构件的实现方式描述 |
| **质量指标** | 该构件需满足的质量目标 |

> **PA 详细定义格式**。每个 PA 构件的详细定义包含以下内容：

| 内容 | 说明 |
|------|------|
| **文档格式** | 使用的标记语言和图表语言 |
| **内容结构** | 构件的章节和内容组织 |
| **关键条目** | 构件中的核心定义和规则 |
| **版本策略** | 版本管理和演进策略 |
| **引用关系** | 对其他构件的引用和依赖 |
| **质量指标** | 具体的质量标准（可测量、可验证） |
| **模板约束** | 格式和风格要求 |

**输入**：系统需求功能部分架构定义 + 系统需求非功能部分架构定义
**输出**：系统需求详细定义 + 产品架构定义（架构末级节点四要素）

> **相关指引**：PA 架构末级节点的识别方法详见§4.7（资产优先法），产品类型分类体系见§4.7.2。

#### 6.8 第6步：双向追溯验证

对全部五层设计文档执行追溯完整性验证，检查1:1约束、N:1合理性和符合性分析完整度，生成追溯矩阵与验证报告。

对应设计文档结对**全量验证：OR → SR → BA → SysReq → PA**。

依次检查以下维度：

**(a) 1:1 约束**（§3.2.1）。无一条详细定义末级分配到多个架构末级节点。

**(b) N:1 合理性**（§3.2.1）。每个架构末级节点承接条数合理。

**(c) 双向追溯**。正向（OR→PA）和反向（PA→OR）均可追踪——每条上层末级条目标注分配到的下层节点，每个下层节点标注承接的上层条目。

**(d) 符合性分析完整**（§3.3）。每个方案节点均完成语义符合性和指标符合性分析。

**输入**：全部五层设计文档
**输出**：需求追溯矩阵 + 验证报告

> **相关指引**：跨层验证的完整维度（角色完整性、触发完整性、NFR 覆盖、跨层对齐、三独立验证等）详见§4.8。

### 七、设计验收标准

本章定义设计产出的质量验收标准，是判断各步骤产出能否进入下一步或交付的判定依据。验收标准按架构定义质量、详细定义质量、映射关系质量、符合性分析质量、追溯完整性和文档结对设计六个维度组织，每个维度标注适用的步骤范围。

#### 7.1 架构定义质量验收

检查架构定义子文档的完整率和一致性，确保每个架构末级节点具备四要素且架构树层级完整。

*适用步骤：§6.3（SR架构）、§6.4（BA架构）、§6.5（SysReq功能架构）、§6.6（SysReq-NFR架构）、§6.7（PA架构）*

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **架构末级节点定义完整率** | 100% | 每个架构末级节点（含 BA IPO）须具备四要素：定义、所映射的上层需求、符合性分析、下游指导（§2.2.1） |
| **架构树层级完整率** | 100% | 从根节点到末级节点的完整树形结构，无中断、无跨越、无循环 |
| **同层抽象级别一致性** | 通过评审 | 同层节点处于同一抽象级别，无粗细不均（§2.2.1） |
| **需求映射视图完整率** | 100% | 架构定义子文档的第二部分（映射关系表）覆盖全部末级需求与架构末级节点的对应关系（§2.2.1） |
| **增量变更合规率** | 100% | 架构变更遵循复用→改进→新增优先级（§2.4.1），新增/修改前确认无法通过复用或扩展解决 |
| **SR-F 角色完整性**（§4.8） | 通过评审 | 所有与目标产品交互的用户角色类型均已识别并分类，无遗漏 |
| **SR-F 触发完整性**（§4.8） | 通过评审 | 每类角色下所有业务触发均已识别，端到端业务闭环完整 |
| **SR-NFR 角色覆盖**（§4.8） | 通过评审 | 操作用户、运维保障者、合规/审计者、管理者/投资者、研发/扩展者等 NFR 相关方角色均已识别 |
| **SR-NFR 分类覆盖**（§4.8） | 通过评审 | 各角色类型的 NFR 关注点已按附录B分类体系完整覆盖，无遗漏的 NFR 维度 |

#### 7.2 详细定义质量验收

检查详细定义的分解质量和性能指标完整率，确保所有架构末级节点分解至原子粒度且指标可衡量。

*适用步骤：§6.4（SR功能详细定义）、§6.6（SR-NFR详细定义）、§6.7（SysReq详细定义）*

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **分解完成率** | 100% | 所有架构末级节点均已分解至详细定义末级，无遗漏 |
| **分解质量四项检查通过率** | 100% | 逐条检查：覆盖完整性（不漏）、不超出性（不增）、逻辑一致性（不矛盾）、约束继承（不失）（§3.1） |
| **末级需求原子性** | 100% | 每条详细定义末级不可再分（语义单一、不含复合能力），非指 1:1 分配约束 |
| **性能指标完整率** | 100% | 每条需求同时包含"需求描述"和"性能指标"（§1.2），指标含具体数值、单位和测量条件 |
| **指标分解正确率** | 100% | 父指标分解满足：可拆分指标的子项之和≤父项；同指标下放至全部子项；暂不分解的标注向下一层传递（§3.1） |

#### 7.3 映射关系质量验收

检查需求分配中1:1约束满足率、N:1承接合理性和映射覆盖率，确保分配无遗漏、无错位。

*适用步骤：§6.3~§6.7（所有含需求分配活动的步骤）*

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **1:1 约束满足率** | 100% | 每条详细定义末级只分配到下层唯一的架构末级节点（规则1，§3.2） |
| **N:1 承接合理性** | 通过评审 | 每个架构末级节点承接的详细定义末级条数合理——条数过少（每节点仅1条）说明架构粒度过细，条数过多说明架构粒度过粗；合理性由语义相关性判定，非硬性数字限制（规则2，§3.2） |
| **映射覆盖率** | 100% | 本层所有末级需求均已分配至下层架构末级节点，无遗漏（§3.2 全覆盖检查） |

#### 7.4 符合性分析质量验收

检查每个方案架构末级节点的语义符合性和指标符合性分析是否完整覆盖所承接的全部上层需求。

*适用步骤：§6.3~§6.7（所有含符合性分析的步骤）*

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **符合性分析覆盖完整率** | 100% | 每个方案架构末级节点均包含符合性分析，覆盖所承接的全部上层需求（§3.3） |
| **语义符合性满足率** | 100% | 方案节点定义覆盖承接需求的核心语义；分析有实质内容而非空洞声明或简单复述 |
| **指标符合性满足率** | 100% | 方案节点的性能指标不低于承接需求的指标要求；降级须在分析中说明理由并经评审确认 |
| **非功能约束承接率** | 100% | 非功能约束在方案节点中显式承接，未丢失（§1.2 NFR 平行路径） |
| **NFR 约束传递**（§4.8） | 100% | SysReq-NFR 的约束条件明确且可验证，作为整体约束作用于 PA 组件而非分配到特定构件 |

#### 7.5 追溯完整性验收

检查五层设计文档的正向和反向追溯链路完整率，确保从原始需求到产品架构全程可追溯。

*适用步骤：§6.8（第6步：双向追溯验证）*

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **正向追溯链路完整率** | 100% | 从 OR 末级→PA 末级正向可追溯，各层映射串接形成完整链路，无中断 |
| **反向追溯链路完整率** | 100% | 从 PA 末级→OR 末级反向可追溯，无断裂 |
| **追溯矩阵完整率** | 100% | 追溯矩阵（产品数据文件 08）覆盖全部五层映射关系 |
| **分配通路一致性** | 通过评审 | FR 沿 OR→SR→BA→SysReq→PA 通路分配；NFR 沿 OR→SR-NFR→SysReq-NFR→PA 通路分配，无跳层分配 |
| **跨层对齐**（§4.8） | 通过评审 | 相邻层的架构末级节点边界可对应追溯，无跨层跳跃或边界不匹配 |

#### 7.6 文档结对设计验收

检查需求-方案文档结对的对齐率、两阶段工作合规率和资产变更记录完整率。

*适用步骤：§6.3~§6.7（所有含文档结对设计的步骤）*

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **文档结对对齐率** | 100% | 每对需求-方案文档中，上层需求详细定义与下层方案架构定义的映射对齐，无遗漏或错位（§三） |
| **两阶段工作合规率** | 通过评审 | 阶段一（结构骨架约定+映射建立）先行完成，阶段二（内容填充）在其基础上展开（§2.3） |
| **资产变更记录完整率** | 100% | 所有架构变更（新增/修改/合并/废弃）标注变更类型和原因，可追溯至资产分析记录（§2.4.2） |
| **BA 完备性判据**（§4.8, §5.2） | 通过评审 | BA 的用户角色集穷举？每个角色的业务清单完整？输出产品架构锚定法下 B 的所有 PA 节点类型是否均有对应的生产操作 IPO？ |

### 八、设计检查清单

本章定义与设计验收标准配对使用的操作检查清单，按架构定义检查、详细定义检查、映射关系检查、符合性分析检查、追溯完整性检查和文档结对检查六个维度逐项列出验证方法，供日常设计工作中逐项验证。

#### 8.1 架构定义检查方法

逐项验证架构定义子文档的四要素完整率、层级完整率、同层一致性和资产变更合规性。

*适用步骤：§7.1*

- [ ] 遍历每层方案文档的架构树：每个架构末级节点（含 BA IPO）是否具备定义、所映射的上层需求、符合性分析、下游指导四要素
- [ ] 从根节点沿树形结构遍历至末级：层级是否完整？是否存在中间节点缺失或跨越？
- [ ] 同层节点的粒度是否一致？是否有的覆盖整个业务流程、有的细到单一操作？
- [ ] 检查架构定义子文档的第二部分（映射关系表）：所有末级需求是否均在表中有对应架构末级节点？每个架构末级节点是否列出了所承接的全部需求？
- [ ] 架构变更项逐一核查：修改前确认无法纯复用解决？新增前确认现有节点无法通过扩展（Extend）或拆分（Split）承接？
- [ ] SR-F 层：所有用户角色类型是否已穷举？每类角色的所有业务触发和端到端业务闭环是否已识别？（§4.2.2-§4.2.3）
- [ ] SR-NFR 层：操作用户、运维保障者、合规审计者等 NFR 相关方角色是否已全量识别？各角色的 NFR 关注点是否已按附录B分类体系覆盖？（§4.3.2-§4.3.3）

#### 8.2 详细定义检查方法

逐项验证详细定义的四项分解质量检查、末级原子性、指标完整性和指标分解正确性。

*适用步骤：§7.2*

> **BA 特例**：BA（业务架构）是唯一仅有架构定义、无详细定义的层级（§1.3），IPO 条目自身承担"详细定义"角色，因此以下针对详细定义子文档的检查不适用于 BA 层。BA 的完整性应通过§8.6 的完备性判据检查确认。

- [ ] 逐层检查（BA 除外）：每个架构末级节点是否都已完成分解——是否存在未分解的架构末级节点？
- [ ] 逐条检验分解质量四项检查：子需求是否完整覆盖父需求全部功能（不漏）、是否引入父需求未要求的内容（不增）、子需求之间是否有逻辑矛盾（不矛盾）、上层非功能约束是否在子需求中显式传递（不失）
- [ ] 每条详细定义末级是否不可再分——能否用一句话描述其单一语义？
- [ ] 每条需求是否同时包含"需求描述"和"性能指标"？指标是否有具体数值、单位和测量条件？
- [ ] 父需求指标分解后验算：可拆分指标的子项之和≤父项？同指标下放至全部子项？暂不分解的是否标注"向下一层传递"？

#### 8.3 映射关系检查方法

逐项验证需求分配的1:1约束、N:1承接合理性和末级需求全覆盖，确保映射关系正确完整。

*适用步骤：§7.3*

- [ ] 逐条确认：每条详细定义末级是否只分配到下层唯一的一个架构末级节点（1:1 约束）？
- [ ] 检查每个架构末级节点承接的条数：条数过少（每节点仅1条）说明架构粒度过细，条数过多说明架构粒度过粗？承接的需求之间是否有逻辑关联？
- [ ] 核对末级需求清单：是否存在未分配出去的末级需求？是否存在未承接任何需求的架构末级节点？

#### 8.4 符合性分析检查方法

逐项验证方案节点的语义符合性分析质量、指标不降级和非功能约束显式承接。

*适用步骤：§7.4*

- [ ] 每个方案架构末级节点（含 BA IPO）是否都包含符合性分析？分析是否覆盖了该节点承接的全部上层需求？
- [ ] 逐条读分析文本：是实质性分析（"通过XX机制实现XX能力"）还是空洞声明（"满足要求"）或简单复述需求？
- [ ] 核对方案节点的性能指标是否不低于承接需求的指标要求？如有降级，是否记录了理由和评审结论？
- [ ] 逐条追踪非功能约束：上层的可用性、响应时间等非功能约束在方案节点中是否有对应的承接语句？
- [ ] SysReq-NFR 的约束条件是否明确且可验证？是否作为整体约束作用于 PA 组件而非分配到特定构件？（§4.6.1）

#### 8.5 追溯完整性检查方法

逐项验证五层文档的正向和反向追溯链路、追溯矩阵完整性和分配通路一致性。

*适用步骤：§7.5*

- [ ] 从 OR 末级出发正向遍历至 PA 末级：每层分配关系是否串接成完整链路，无断裂、无跳跃？
- [ ] 从 PA 末级出发反向遍历至 OR 末级：是否存在无法追溯到原始需求的孤点节点？
- [ ] 追溯矩阵是否覆盖所有五层映射关系：OR→SR（功能）、SR→BA、BA→SysReq、SR-NFR→SysReq-NFR、SysReq→PA？
- [ ] 校验功能需求走 OR→SR→BA→SysReq→PA 通路，非功能需求走 OR→SR-NFR→SysReq-NFR→PA 通路——是否存在跳层分配？
- [ ] 检查跨层节点边界对齐：相邻层的架构末级节点边界是否可对应追溯？是否存在边界不匹配导致的跨层跳跃？（§4.8）

#### 8.6 文档结对与资产变更检查方法

逐项验证需求-方案文档结对对齐率、两阶段工作合规性和资产变更记录完整性。

*适用步骤：§7.6*

- [ ] 逐对检查需求-方案文档对：上层的每一条需求详细定义末级是否都在下层方案架构定义中有关联的架构末级节点？
- [ ] 检查两阶段工作记录：阶段一（结构骨架约定+映射建立）是否在阶段二（内容填充）之前完成？是否存在未建立映射先行填充的情况？
- [ ] 每个架构节点的新增/修改/合并/废弃是否标注了变更类型和原因？原因引用可追溯至资产分析记录？
- [ ] 同一架构节点经历 3 次以上的 Extend 后，是否已触发拆分（Split）评审？
- [ ] 用户角色架构锚定法下：BA 的用户角色集是否穷举？每个角色的业务清单是否完整？（§5.2.1 完备性判据）
- [ ] 输出产品架构锚定法下：输出产品（产品 B）的所有 PA 节点类型是否均有对应的生产操作 IPO？（§5.2.2 完备性判据）

#### 8.7 格式合规检查

逐项验证各层文档条目格式的完整性和一致性，确保格式正确是规则自动检查的前置条件。

- [ ] **OR 条目**：编号、规范化描述、类型（功能/非功能）、性能指标、优先级齐全？
- [ ] **SR 架构节点**：SR-F 节点是否包含 ID、标题、定义、承接 OR、符合性分析、设计指导、性能指标？SR-NFR 节点是否含量化指标和测量方法？
- [ ] **BA IPO**：每个 IPO 是否完整包含输入、处理、输出三维度？描述体量是否匹配期望的输出物颗粒度？
- [ ] **SysReq 功能节点**：5 层（架构末级）是否含节点 ID、定义、承接 BA、符合性分析、性能指标？9 级（详细定义末级）编号符合多级格式？
- [ ] **SysReq-NFR 节点**：分类归属三大类之一？含量化指标和验证方法？
- [ ] **PA 构件**：含构件 ID、名称、功能角色、定义、承接 SysReq、符合性分析、实现方案、质量指标八属性？详细定义含文档格式、内容结构、版本策略？

### 九、常见错误与FAQ

本章汇总系统设计各层中反复出现的典型错误和常见问题，帮助设计者预先识别并规避。所列错误在五个设计层（OR/SR/BA/SysReq/PA）中通用表现不同，本节按错误类型统一描述通用原则。

#### 9.1 常见错误

**错误：分解粒度不当**

分解粒度过粗或过细是各层最常见的错误。

| 问题 | 表现 | 后果 |
|------|------|------|
| **粒度过粗** | 一条末级条目需要分配到下层多个架构节点 | 违反1:1约束，追溯链路断裂 |
| **粒度过细** | 一个功能拆成过多条目，与同层其他条目粒度不一致 | 管理复杂，架构碎片化 |

✅ 正确做法：分解到每条末级只需下层一个架构末级节点承接（1:1约束），且同层条目粒度一致。上层详细定义条目数与下层架构末级节点数的比例通常在2-5:1。

**错误：功能与非功能需求混淆**

非功能需求被当作功能需求处理，或功能需求漏带了自有性能指标。

表现：非功能需求（如"系统应快速响应"）被混入功能需求经BA路径分配，导致BA IPO承载了不属于它的质量约束。

✅ 正确做法：在OR层完成FR/NFR关联识别（不做正式拆分），到SR层正式分离——FR走BA路径，NFR走平行路径直达SysReq-NFR。每条功能需求自带的性能指标作为该需求的固有属性一同传递。FR与NFR在SR层的边界划分详见§4.2.4（SR-F与SR-NFR的边界）。

**错误：映射不完整或违反1:1约束**

| 问题 | 表现 |
|------|------|
| **映射遗漏** | 部分上层条目未分配到下层架构节点 |
| **多目标映射** | 一条上层条目分配到多个下层节点（违反1:1约束） |
| **承接过量** | 一个下层节点承接过多上层条目（N值不合理，通常>5） |

✅ 正确做法：分配后执行三项自检——全覆盖（所有条目均已分配）、1:1（每条目分配到唯一目标）、N:1 合理性（目标节点承接条数合理，2-5条最优）。跨层对齐的全局验证维度详见§4.8。

**错误：非功能需求缺少量化指标**

表现：NFR使用"快速"、"稳定"、"安全"等定性词汇，没有给出具体测量指标和条件。

✅ 正确做法：每条NFR必须包含可量化的指标（含数值、单位、测量条件）。
- ❌ "系统应快速响应"
- ✅ "平均响应时间 < 200ms，P99 < 500ms（1000并发用户下）"

**错误：忽略资产优先原则，动辄新增架构节点**

表现：新的需求到达时，直接在架构中新增节点来承接，而不检查现有节点是否能复用或扩展。

✅ 正确做法：按 **复用→改进（Extend/Split/Merge）→新增** 的优先级依次检查。仅当复用和改进都不可行时，才选择新增。保护组织已有设计资产，控制架构整体复杂度。

#### 9.2 最佳实践

**实践1：先骨架后填充**

先快速约定下层架构的结构骨架（分类节点及末级节点标题）并建立初步映射，再并行填充上层需求详细定义和下层方案节点要素。避免上层分解到末级后下层无可承接目标的返工。

**实践2：以"独立单元"为粒度标准**

各层节点划分以"可独立定义、可独立验证"为粒度标准。具体到PA层，构件以"可独立交付和验收"为标准；到BA层，IPO以"可由不同角色独立执行"为标准。

**实践3：尽早开展符合性分析**

每层分配完成后立即进行符合性分析（语义覆盖+指标不降级），而非等所有层设计完成后再统一验证。尽早发现设计缺陷，降低返工成本。

**实践4：记录架构决策**

关键技术选择记录决策理由，NFR权衡决策记录在案（如"完整性vs可读性"、"通用性vs针对性"的取舍及理由）。

#### 9.3 FAQ

**问：系统设计是设计整个产品，还是只设计某个子系统或子模块？**

**答：** 系统设计的范围始终是**目标产品的完整系统**。在任何层级、任何开发空间，对"产品X"做系统设计，产出的就是X的完整产品架构——涵盖其所有组成部分。

不存在"只对X的某一部分做系统设计"的场景，原因：
- 五层设计链路（OR→SR→BA→SysReq→PA）以完整产品的原始需求为起点，以完整产品的PA为终点
- BA（业务架构）要求覆盖使用产品的全部用户角色和全部操作场景——这在局部范围内无法完成
- PA 的架构末级节点以产品的全部构件为分类范围，而非仅某个子集

如果需要设计的是某个构件的内部实现细节，这属于**构件开发阶段**而非系统设计阶段。系统设计聚焦于"产品由什么构件组成"，构件开发聚焦于"单个构件如何实现"。

**问：一个架构末级节点承接多少条上层条目合理？**

**答：** 建议每个架构末级节点承接2-5条（2-3条最优）。1条需考虑是否可合并；6条以上需检查是否粒度过粗需拆分。承接0条说明该节点可能多余。

**问：各层的层级数是固定的吗？**

**答：** 不是。各层的层级数可根据系统复杂度适当调整。关键约束是：同层节点应在同一抽象级别；架构末级节点（承接层）的位置固定不变。

**问：什么是"中间层文档"？哪些层属于中间层？**

**答：** "中间层文档"指同时承担方案和需求双重角色的文档（§1.1 需求/方案相对）。SR和SysReq属于中间层——它们既是方案（承接上层需求），又是需求（驱动下层设计）。OR只有需求无方案（设计链起点），PA只有方案无需求（设计链终点）。

**问：格式定义和设计规则冲突时以哪个为准？**

**答：** 格式定义（条目字段、编号规则）是"语法"层面的约定，确保各层文档结构一致；设计规则（分解原则、分配约束、分析方法）是"语义"层面的规范，确保设计内容的完整和正确。两者不冲突：格式是规则的载体，规则通过格式来执行。格式检查是规则检查的前置条件——格式不正确的内容无法被执行规则自动检查。

### 附录B 非功能需求分类体系

本附录定义非功能需求的完整分类体系——按质量特性、环境适应性和可实现性三个一级类目组织，每个类目下展开二级分类与定义，为系统设计中非功能需求的分类和分配提供标准框架。

| 分类 | 定义 | 评估视角 |
|------|------|---------|
| **质量特性** | 系统运行时表现出的内在品质 | 用户/运维视角 |
| **环境适应性** | 系统对外部变化的适应能力 | 外部环境视角 |
| **可实现性** | 系统在全生命周期各环节的实现便利性 | 开发/交付视角 |

#### B.1 质量特性

列出与系统运行时内在品质相关的非功能需求二级分类，涵盖性能、安全、可靠性、可用性、易用性等八个子类。

| 二级分类 | 定义 |
|---------|------|
| 性能（效率） | 响应时间、吞吐量、资源利用率 |
| 安全性 | 保密性、完整性、不可抵赖性、可追溯性、身份认证 |
| 可靠性 | 成熟度（缺陷率）、容错性、可恢复性 |
| 可用性 | 正常运行时间比例、故障转移能力 |
| 易用性 | 易学性、易操作性、可访问性 |
| 人机工效 | 用户操作体验、认知负荷、操作效率 |
| 韧性 | 面对故障的适应和恢复能力 |
| 功能正确性 | 功能实现的准确度、精度、完备性 |

#### B.2 环境适应性

列出与系统对外部变化适应能力相关的非功能需求二级分类，涵盖可扩展性、弹性、国际化、可移植性等七个子类。

| 二级分类 | 定义 |
|---------|------|
| 可扩展性 | 业务增长时平滑扩展的能力 |
| 弹性 | 负载波动时动态调整资源的能力 |
| 国际化 | 适应不同语言、地区、法规的能力 |
| 可移植性 | 跨平台/跨环境部署的能力 |
| 互操作性 | 与第三方系统对接交换数据的能力 |
| 兼容性 | 向后兼容、与现有系统共存的能力 |
| 可替换性 | 组件可被替代实现替换而不影响整体 |

> **注**：实物产品的环境适应性（气候、机械、电磁、防护、供电、海拔等）属于另一分类体系，在涉及实物产品层时按需引用。

#### B.3 可实现性

列出与系统全生命周期各环节实现便利性相关的非功能需求二级分类，涵盖可采购性、可生产性、可测试性等八个子类。

| 二级分类 | 定义 |
|---------|------|
| 物料可采购性 | 所需资源（广义物料：软件许可、云资源、API、组件库、工具链、人力技能等）采购的便捷程度和供应风险 |
| 可生产性 | 产出过程的复杂度高低，是否容易制造或构建 |
| 可测试性 | 功能和非功能指标是否容易验证，验证成本高低 |
| 可交付性 | 产品是否容易交付到用户手中，交付成本高低 |
| 可部署性 | 是否容易在目标环境中安装上线，部署耗时和复杂度 |
| 可运维性 | 运行中是否容易监控、维护和排障，运维投入大小 |
| 可退役性 | 下线时数据迁移和遗留清理是否容易操作 |
| 经济可承受性 | 以上所有活动的总成本是否在预算内 |

> **注**：上表中的"可"均理解为**方便/容易**之意，而非"能否"。可实现性的本质目标是降低产品全生命周期成本。

### 十、四种设计方法

本章定义四种系统设计方法及其适用场景。每种方法不是互斥的固定标签，而是在特定约束下对**需求分解、需求分配、符合性分析**三个核心设计活动的不同执行策略。

#### 10.1 四种模式概览

| 方法 | 适用场景 | 设计方向 | 典型周期 |
|------|---------|---------|---------|
| **瀑布式** | 原始需求全部齐备，从零开始的完整系统设计 | 正向：OR → SR → BA → SysReq → PA | 完整周期（30-48 天） |
| **敏捷式** | 已有完整系统设计，增量新增或修订需求 | 正向增量：分析影响 → 增量更新 | 短周期（13-21 天） |
| **逆向工程** | PA 架构及各节点需求定义已存在，需反向推导上游设计文档（SysReq → BA → SR → OR） | 反向：PA → ... → OR | 完整周期（16-26 天） |
| **DevOps** | 在产产品需快速修复或小变更 | 最小正向变更：定位 → 修复 → 验证 | 极短（小时级-3 天） |

> **逆向工程的前置条件**：系统设计逆向工程以**PA 架构及各节点需求定义已存在**为前提，从 PA 反向推导 SysReq → BA → SR → OR。若架构节点仅有代码/图纸而无明确的节点需求定义，则需先执行**构件级逆向工程**（属于构件开发阶段的方法，不属于系统设计阶段），从代码/图纸逆向生成各构件的需求定义，再以此为基础进入系统设计逆向工程。

##### 10.1.1 模式的 PACE 特征

PACE 四维度（Precision、Adaptability、Completeness、Efficiency）基于本规范§七（设计验收标准）和§八（设计检查清单）的质量维度提炼而来，用于刻画四种模式在精度、适应性、完备度和效率上的系统性差异——高精度通常意味着高耗时，高完备度通常意味着低适应性。模式选择的本质是在这些维度间做出符合场景约束的决策。

四模式在以下四个维度上呈现系统性差异，这些差异源于设计起点和约束条件的不同，而非设计质量的优劣：

| PACE 维度 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|-----------|--------|--------|---------|--------|
| **Precision（精度）**——产出的精细化程度 | 高：全量原子粒度分解 | 中：仅变更路径的精细分解 | 中-高：受限于可还原性 | 低：仅问题链路的必要精度 |
| **Adaptability（适应性）**——对需求变化的响应能力 | 低：完整周期固定，变更成本高 | 高：短周期迭代，变更路径已就绪 | 低：反向推导，变更将破坏还原链 | 高：极短反馈环，局部影响可控 |
| **Completeness（完备度）**——设计文档的完整程度 | 高：全五层、全双文档 | 中：仅更新受影响的文档层 | 高：目标为补齐全部缺失文档 | 低：仅变更必要的最小文档集 |
| **Efficiency（效率）**——设计执行的时效性 | 低：完整周期数周 | 中：短周期但需全局影响分析 | 中：需逐层恢复再验证 | 高：小时级，变更路径即执行路径 |

> 具体取舍对：Precision × Efficiency 此消彼长，Completeness × Adaptability 互为代价。

###### PACE 维度操作化定义

PACE 四维度的操作化定义如下，使模式维度从定性标签转变为可度量的选择依据。以下数值为通用参考边界，各产品在实例化时基于项目数据校准：

| PACE维度 | 操作化定义 | 高 | 中 | 低 |
|---------|-----------|----|----|-----|
| **Precision（精度）** | 需求分解的层次深度：从最顶层 OR 条目到末级需求经过的平均分解步数 | ≥6 层 | 3-5 层 | ≤2 层 |
| **Adaptability（适应性）** | 从需求变更到方案更新的平均周期 | ≤1 天 | 2-5 天 | >5 天 |
| **Completeness（完备度）** | 五层文档中完成双向追溯的末级条目占比 | ≥95% | 60-94% | <60% |
| **Efficiency（效率）** | 从模式启动到交付的平均时间效率 | 小时级产出 | 天级产出 | 周级产出 |

> PACE 维度在模式选择中的使用方式：选择适合**场景约束**的模式，而非追求所有维度同时最大化。模式选择的本质是在这些维度间做出符合场景约束的决策（如生产问题紧急修复时优先 Adaptability + Efficiency，接受较低的 Completeness）。

###### DevOps 三子模式

DevOps 模式在决策树（§10.2）中提及三个子模式，本节给出其方法论定义。三个子模式在涉及层、分解深度和符合性分析深度上逐级递进：

| 维度 | 模式1：快速修复 | 模式2：局部设计 | 模式3：完整设计检视 |
|------|---------------|---------------|-----------------|
| **涉及设计层** | 仅 PA | SysReq + PA | 多层（SR/BA/SysReq/PA） |
| **分解深度** | 问题链路的最小必要分解 | 局部范围内的充分分解 | 近似敏捷式的影响域分解 |
| **分配范围** | 定位到具体 PA 节点 | PA 节点 + SysReq 映射更新 | 变更影响的全链路映射更新 |
| **符合性分析** | L1（变更验证） | L1 + L2（语义） | L1+L2+条件性 L3 |
| **典型场景** | 配置错误/参数调整 | 接口变更/功能扩展 | 架构调整/非功能约束变化 |

**子模式间升级路径**：
- 模式1→模式2：修复方案涉及架构末级节点定义的修改，而非仅参数/配置调整
- 模式2→模式3：影响评估发现变更跨多个设计层，无法在单层内闭合
- 模式3→敏捷式：影响评估发现变更范围超过 50% 或需求变更涉及 3 个以上架构域，应升级到敏捷式而非继续 DevOps

> DevOps 三子模式在产品指南中实例化为具体的执行流程和任务定义。

##### 10.1.2 各模式与六步法映射

四种模式执行本规范§六（系统设计过程）六步法的范围和方式不同。下表展示各模式对六步法各步骤的执行情况：

| 六步法步骤（§六） | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|-----------------|--------|--------|---------|--------|
| **第1步**：OR → SR 原始需求分析及相关方需求架构定义 | ✅ | ❌ 基线已有 | ✅ 反向重建 | ❌ |
| **第2步**：SR(功能) → BA 相关方需求功能部分详细定义及业务架构定义 | ✅ | △ 仅变更影响 | ✅ 反向重建 | ❌ |
| **第3步**：BA → SysReq(功能架构) 系统需求功能部分架构定义 | ✅ | △ 仅变更影响 | ✅ 反向重建 | ❌ |
| **第4步**：SR(非功能) → SysReq(非功能架构) 系统需求非功能部分架构定义 | ✅ | △ 仅变更影响 | ✅ 反向重建 | ❌ |
| **第5步**：SysReq → PA 系统需求详细定义及产品架构定义 | ✅ | ✅ 增量变更 | ✅ 反向重建（PA 已存在） | ✅ 局部定点 |
| **第6步**：双向追溯验证 | ✅ | △ 变更链路追溯 | ✅ | △ 修复链路追溯 |

> ✅=完整执行 &nbsp;&nbsp; △=条件性执行（仅变更/受影响部分）&nbsp;&nbsp; ❌=不执行

**核心差异**：
- **瀑布式**全量执行六步，是唯一自顶向下完整覆盖的模式
- **敏捷式**仅执行第5步（增量变更）及受影响层的条件性步骤，其余复用已有基线
- **逆向工程**以第5步为起点逐层反向重建至第1步，最终仍需第6步验证
- **DevOps**仅在第5步做局部定点变更，第6步仅验证修复链路

#### 10.2 模式选择框架

模式选择应从产品当前状态、需求类型和约束条件三个维度综合判断。

##### 10.2.1 多因素选择矩阵

| 决策因素 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|---------|--------|--------|---------|--------|
| **需求确定性** | 需求完全确定 | 需求部分确定 | 需求需从产物中提取 | 需求明确且紧急 |
| **文档完备度** | 无任何设计文档 | 已有完整设计文档 | PA 已存在，上游缺失 | 设计文档完整 |
| **变更范围** | 全新系统 | 局部功能新增/修改 | 整个设计链补齐 | 单点问题/修复 |
| **时间约束** | 时间充足 | 中等节奏 | 中等节奏 | 极短时间内交付 |
| **设计风险容忍度** | 低（需完整验证） | 中（影响范围可控） | 中（理解偏差风险） | 高（快速验证后修复） |

##### 10.2.2 决策流程

```
产品当前处于什么状态？

┌─ 已开发完成，但设计文档缺失？
│  └─ PA 架构及各节点需求定义是否存在？
│     ├─ 是 → 系统设计逆向工程：从 PA 反向推导 SysReq → BA → SR → OR
│     └─ 否 → 先基于代码/图纸进行构件级逆向工程，生成各节点需求定义；
│              再以完整 PA（含节点需求定义）为起点执行系统设计逆向工程
│
├─ 从零开始，无任何设计文档？
│  └─ 原始需求是否已齐备？
│     ├─ 是 → 瀑布式：完整正向设计（OR → SR → BA → SysReq → PA）
│     └─ 否 → 先收集和确认原始需求，再进入瀑布式
│
├─ 已有完整系统设计，需新增或修改需求？
│  └─ 变更范围是否跨多个设计层？
│     ├─ 是 → 敏捷式：逐层分析影响 → 增量设计 → 验证
│     └─ 否 → 可在单层内直接执行局部设计变更
│
└─ 产品已上线，需快速修复生产问题？
   └─ 问题复杂度如何？
      ├─ 单点修复（PA 层可解决）→ DevOps 模式1：快速修复
      ├─ 需局部设计调整（涉及 SysReq+PA）→ DevOps 模式2：局部设计
      └─ 跨多层影响 → DevOps 模式3：完整设计检视
```

#### 10.3 模式间切换

产品在其生命周期中可能经历多次模式切换。典型切换路径包括：

| 切换路径 | 典型场景 | 注意事项 |
|---------|---------|---------|
| **逆向工程 → 瀑布式** | 完成文档补齐后，进入正向迭代开发 | 逆向产出的设计文档需通过双向追溯验证，方可作为瀑布式的可靠起点 |
| **瀑布式 → 敏捷式** | 首次完整设计完成后，后续需求变更为增量模式 | 瀑布式产出的完整设计文档是敏捷式的前提——无完整基线则无法做增量 |
| **瀑布式 → DevOps** | 产品上线后，生产问题需快速响应 | 从全量正向设计切换到最小变更模式，需建立快速影响评估机制 |
| **敏捷式 → 逆向工程** | 增量迭代中发现上游文档缺失严重，需回溯补齐 | 敏捷式的增量修改可能降低文档一致性，需阶段性插入逆向工程恢复文档质量 |
| **DevOps → 敏捷式** | 快速修复经生产验证后需纳入正式设计基线 | DevOps 的变更记录需要回填到设计文档中，避免设计文档与生产实现偏离 |

**切换的基本原则**：
- 每次切换应明确记录切换原因、时间点和基线状态
- 切换不改变已冻结的编号和映射关系，新增内容遵循编号稳定性原则
- 逆向工程的输出经双向追溯验证后方可作为正向设计的输入
- DevOps 的变更应在后续敏捷迭代或瀑布阶段中回填至完整的设计文档

##### 10.3.1 量化触发条件

各切换路径的量化触发条件如下，使切换决策可操作、可审计：

| 切换路径 | 量化触发条件 |
|---------|-------------|
| **逆向工程 → 瀑布式** | 重建的 OR/SR/BA/SysReq 文档通过自检，上层文档完整性恢复到 90% 以上 |
| **瀑布式 → 敏捷式** | 首次完整设计通过第 6 步双向追溯验证，追溯矩阵完整率 100% |
| **瀑布式 → DevOps** | 产品上线后，生产问题的紧急度评级 ≥ P1（影响生产运行） |
| **敏捷式 → 逆向工程** | 同一设计域经历 5 次以上敏捷迭代而未更新上游文档；或增量变更累积导致追溯链路断裂超过 20% |
| **DevOps → 敏捷式** | 同一产品或区域经历 3 次以上 DevOps 修复；或同一 PA 节点被修改 2 次以上 |
| **敏捷式 → 瀑布式（全量）** | 单次变更影响范围超过 50% 或涉及 3 个以上设计层 |
| **DevOps → 瀑布式（全量）** | 累积修复范围超过产品架构的 30%；或无可用基线（设计文档缺失严重） |

> **累积效应监控**：所有模式在切换前应评估累积变更效应，而非仅看单次变更范围。同一节点经历 3 次以上的 Extend 操作后应触发拆分评审（参见§2.4.2 扩展规则）。累积效应的具体判定标准在各产品指南中实例化。

### 十一、三种核心设计活动的跨模式差异

本章规定三种核心设计活动在四种模式中的执行差异。本规范§三定义了三种核心设计活动及其操作规则——需求分解及分析（§3.1）、需求分配及分析（§3.2）、符合性分析（§3.3）。以下按每个活动展开其在四种模式中的执行差异。

#### 11.1 结对设计原则（两阶段工作法）的跨模式差异

本规范§2.3.1 定义了结对设计的**两阶段工作法**——骨架阶段（先快速约定下层架构的分类节点及末级节点标题并建立初步映射）与填充阶段（上下层各自并行填充内容）。四种模式在骨架起始点、骨架范围、填充并行度和结对关闭条件上存在系统性差异：

| 维度 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|------|--------|--------|---------|--------|
| **骨架起始点** | 从 OR → SR 结对1开始，流水传递至结对4 | 从变更影响层开始，向上下游扩散 | 从 PA → SysReq 结对4开始，反向传递至结对1 | 不适用完整两阶段；以"定位→修复→验证"三步骤代替 |
| **骨架范围** | 全五层架构骨架 | 仅变更影响层的架构调整（新增/扩展/拆分节点） | 全重建链的架构骨架 | 仅变更节点的定位 |
| **填充并行度** | 全链并行——4个结对同时填充 | 仅变更层参与填充，不变层不重新填充 | 全链并行，内容为反向重建 | 仅问题链路的最小修复 |
| **结对关闭条件** | 全量验证通过 | 变更链路验证通过 | 还原的映射通过双向追溯验证 | 修复链路追溯完整 |

##### 11.1.1 多层结对时序协调的跨模式差异

本规范§2.3.3 描述了四个文档结对（OR↔SR、SR↔BA、BA↔SysReq、SysReq↔PA）的时序协调规律——骨架阶段可流水传递，填充阶段可全链并行。四种模式对此规律的偏离程度不同：

| 时序维度 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|---------|--------|--------|---------|--------|
| **结对覆盖范围** | 全部4个结对 | 仅变更影响层涉及的结对 | 全部4个结对（逆向执行） | 通常1个结对（受影响最大的层） |
| **骨架方向** | 正向流水：结对1→2→3→4 | 从影响源层向两侧扩散 | 反向流水：结对4→3→2→1 | 从问题层向 PA 层定位 |
| **填充并行度** | 全链并行（4个结对同时填充） | 变更层并行（仅受影响结对填充） | 全链并行但内容为反向重建 | 单点串行（定位→修复→验证） |
| **骨架传递规则** | 上游骨架完成即传递下游 | 增量骨架传递范围限于变更域 | 下游骨架先建立（PA已知），再反向传递上游 | 不涉及传递 |

> **敏捷式变更影响层的判定规则**：变更从哪层引入就影响哪些结对。单层内（如仅 SysReq 层修改）的变更不触发跨层结对；跨层变更（如 SR 新增需求→BA→SysReq→PA）需逐层评估影响范围。

#### 11.2 需求分解的跨模式差异

需求分解（§3.1）是将父需求切分为可分配的原子粒度条目的过程，包含语义分解和指标分解两个平行工作面。四种模式在分解起点、分解深度和自检重点上存在系统性差异。

##### 11.2.1 分解起点

| 模式 | 分解起点 | 含义 |
|------|---------|------|
| **瀑布式** | 全量原始需求条目 | 所有需求同时进入分解流程，从 OR 层开始逐层向下 |
| **敏捷式** | 变更需求条目集合 | 仅新增或修改的需求进入分解流程，已有条目保持不动 |
| **逆向工程** | PA 架构及节点需求定义 | 从 PA 节点需求定义出发，反向还原上游各层的需求定义（SysReq → BA → SR → OR）。分解方向与正向相反——从 PA 节点的需求定义逐层还原其上游需求来源 |
| **DevOps** | 生产问题/紧急需求 | 仅与问题解决直接相关的需求链路进入分解，其余不触及 |

##### 11.2.2 分解深度

| 模式 | 分解深度 | 终止条件 |
|------|---------|---------|
| **瀑布式** | 完整到原子粒度 | 每条子需求均可依 1:1 约束分配到下层唯一架构末级节点—完整执行三项自检 |
| **敏捷式** | 按需到 1:1 就绪 | 仅变更路径上的需求分解到可分配粒度，已有路径不复检；重点关注约束继承（变更项不能丢失原有约束） |
| **逆向工程** | 还原到正向等价 | 分解深度以"正向设计可接受的粒度"为目标，而非越细越好；过度分解（混入实现细节）是逆向工程的高频反模式。判据：当还原出的需求描述中出现了 PA 层实现术语（如组件名、协议名、库名）时，即为过度分解的信号——此时应退回到上一层，以"系统应具备什么能力"而非"系统应使用什么实现"的粒度描述需求 |
| **DevOps** | 最小必要分解 | 仅分解到问题定位和修复验证所需的粒度，不展开非相关分支 |

###### 指标分解的跨模式差异

本规范§3.1.2 定义了三类指标分解场景——**可拆分**（加和/乘积/极值法等运算关系）、**下放**（同值继承）、**暂不分解**（独立约束，无法运算分解）。以下为各模式的执行差异：

| 指标分解场景 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|------------|--------|--------|---------|--------|
| **可拆分** | 全量执行运算关系识别和子指标分配 | 仅变更涉及的新增需求执行指标拆分 | 拆分关系重组——从 PA 节点指标反向推导上游应分配的指标值 | 通常不涉及（指标调整不改变分解结构时） |
| **下放** | 检查所有子需求是否同值继承 | 增量检查新需求的指标继承是否与已有一致 | 还原时标识"约束信息残缺"——指标继承关系往往丢失，需注明来源 | 检查修复是否保持了约束继承的完整性 |
| **暂不分解** | 标注"向下一层传递"并记录分解依据 | 增量时检查暂不分解的前提是否仍然成立 | 逆向工程中应尽可能少用"暂不分解"——还原的目标是消除未分解状态 | 在最小修复路径中可接受暂不分解 |

> 指标分解的运算关系识别方法（加和/乘积/极值法等）在所有模式中一致，差异仅在执行范围和强制性上。具体运算关系的识别规则参见本规范§3.1.2。

##### 11.2.3 自检重点

四种模式在执行自检（覆盖完整性、不超出性、逻辑一致性、约束继承）时侧重不同：

| 自检项 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|--------|--------|--------|---------|--------|
| **覆盖完整性** | ★★★ 首要检查 | ★★ 仅检查变更域 | ★★★ 核心检查（没还原到完整功能就是失败） | ★ 仅检查修复链路是否覆盖 |
| **不超出性** | ★★★ 严格检查 | ★★★ 重点检查（增量不能超出变更范围） | ★ 容忍度较高（还原时可能引入合理的补充） | ★★ 检查修复是否引入了非必要的变更 |
| **逻辑一致性** | ★★ 标准检查 | ★★ 变更与已有间的冲突检查 | ★★★ 核心检查（还原出的需求不能自相矛盾） | ★★ 修复方案与现有设计的一致性 |
| **约束继承** | ★★ 标准检查 | ★★★ 重点检查（增量的约束不能低于原有基线） | ★ 弱检查（还原时约束信息往往不完整，需标注待确认） | ★★ 检查修复是否保持了原有约束 |

> ★ 数量表示该模式对该自检项的重视程度，非固定权重，而是设计活动中的实际投入优先级。

###### 敏捷式变更影响分析方法论

敏捷式以"仅变更影响"为核心概念，但需要系统的方法来确定影响范围。以下为通用的变更影响分析方法论四步，适用于敏捷式及 DevOps 模式3：

1. **变更源定位**：确定新增/修改需求的初始影响层（SR/BA/SysReq/PA）。变更源是增量设计的切入点，决定了后续影响扩散的起点。

2. **影响扩散分析**：从变更源层沿设计链向上下游扩散，逐层检查：
   - 当前层的架构定义是否需要调整（新增/扩展/拆分节点）？
   - 已有映射关系是否因变更而失效（需求重新分配到不同节点）？
   - 变更是否引入了新的非功能约束（需走 NFR 平行路径）？
   - 影响是否传递到相邻层（上游需求层需调整分解，下游方案层需调整节点定义）？

3. **影响范围分级**：基于扩散分析结果对变更分级：
   - **局部影响**（单层单节点）→ 在单层内直接执行局部设计变更，不触发跨层结对
   - **区域影响**（单层多节点，或相邻两层）→ 触发受影响层的增量结对设计
   - **全局影响**（三层以上，或影响范围 > 50%）→ 应退出敏捷式，进入瀑布式全量设计

4. **变更路径确定**：基于影响范围分级，确定需要增量更新的文档层列表和需要重新执行的结对设计范围。变更路径的记录是第 6 步追溯验证的输入。

**增量映射的三个手势**（基于资产优先原则，§2.4.1）：
- **Extend（扩展）**：将变更需求分配到已有节点，更新节点定义和映射列表。影响范围最小，默认手势
- **Split（拆分）**：已有节点职责过宽，拆分为多个子节点后重新分配。影响范围中等，需调整架构树
- **Add（新增）**：无合适现有节点承接，在适当父节点下创建新末级节点。影响范围取决于新增节点的层级位置

> 手势选择顺序：优先 Extend（复用）→ 其次 Split（改进）→ 最后 Add（新增）。同一节点累加 Extend 达到 3 次以上时应触发 Split 评审。

#### 11.3 需求分配的跨模式差异

需求分配（§3.2）是将分解后的末级需求条目按 1:1 约束映射到下层架构末级节点的过程。四种模式在分配范围、映射决策逻辑和合理性审查方式上各不相同。

##### 11.3.1 分配范围

| 模式 | 分配范围 | 含义 |
|------|---------|------|
| **瀑布式** | 全量全覆盖 | 所有末级需求条目全部映射到下层架构末级节点，一条不落 |
| **敏捷式** | 增量变更域 | 仅变更的末级需求条目参与分配，已有映射不动；需检查变更是否导致已有映射失效 |
| **逆向工程** | 反向重建全量 | 从 PA 层开始自底向上逐层建立正向映射关系——先确定"每个架构节点承接了哪些上层需求"，再将这些需求逐条归位到上层需求详细定义 |
| **DevOps** | 局部定点 | 仅变更的末级需求条目参与分配，且仅映射到与问题直接相关的架构末级节点 |

##### 11.3.2 映射决策逻辑

| 模式 | 分配决策逻辑 | 对资产优先原则的运用 |
|------|-------------|-------------------|
| **瀑布式** | 从零扫描：自上而下逐层建立新架构，逐条分配 | 资产优先原则（§2.4.1）用于指导架构节点的生成：复用→改进→新增；但分配本身是从零开始 |
| **敏捷式** | 资产优先扫描：遍历已有架构节点，逐条判断是否能承接变更需求 | 资产优先原则是敏捷分配的核心逻辑——首选复用已有节点，其次扩展（Extend），最后新增（Add） |
| **逆向工程** | 反向追溯重建：从 PA 节点需求定义出发，反向追溯其上游需求来源——某节点需求定义来自哪些 SysReq 条目，该条目又来自哪个 BA IPO，依此类推至 OR | 资产优先原则在此处表现为"先识别已有架构节点定义的隐含需求范围，再恢复上层需求条目"——节点已存在，恢复其"为何存在" |
| **DevOps** | 变更追踪定位：从问题出发追踪到具体的架构节点和详细定义条目 | 定位到节点后，按资产优先原则决定修复路径——复用（调整参数）→ 改进（扩展定义）→ 新增（补充节点） |

###### 资产优先原则的跨模式操作细则

资产优先原则（本规范§2.4）的完整操作细则在各模式中的运用逻辑不同。以下为"复用→改进→新增"优先级在每种模式中的典型手势和注意力焦点：

- **瀑布式**：资产优先作为"从零扫描"的辅助——先查询组织资产库确定已有可复用节点，再对无对应资产的领域执行新增。瀑布式中 Extend 操作相对罕见（因从零新建），更多依赖 Add。新建节点的颗粒度由信息处理模式硬边界（§2.4.2）约束——不同信息处理模式的需求不能共享同一架构节点
- **敏捷式**：资产优先是增量分配的核心逻辑。每次增量分配都必须先执行资产扫描——在所有已有节点中寻找可承接变更的节点。Extend 是敏捷式的默认手势；同一节点经历 3 次以上 Extend 应触发 Split 评审。**信息处理模式硬边界的风险**：Extend 操作容易无意识地将不同信息处理模式的需求合并到同一节点，需作为自检重点
- **逆向工程**：资产优先表现为"先识别已有架构节点定义的隐含需求范围，再恢复上层需求条目"——节点已存在，恢复其"为何存在"。注意检查已有映射是否跨信息处理模式——如发现同一节点承接了实时和批量两种需求，标记为设计缺陷，在重建中拆分
- **DevOps**：资产优先表现为修复路径优先级——复用（参数调整）→ 改进（扩展定义）→ 新增（补充节点）。**硬边界的不可逾越性**：修复路径不能因"最小变更"而跨越信息处理模式硬边界——实时和批量处理不能合并到同一节点，即使"只改一个参数"也不行

> **通用规则**："同一节点累加 Extend 达 3 次以上应触发拆分评审"适用于所有模式，不限于敏捷式——它是资产优先原则的通用操作细则（本规范§2.4.2）。**信息处理模式硬边界的优先级高于资产优先**——不能因为"优先复用"就把不同信息处理模式的需求塞进同一节点。

##### 11.3.3 N:1 合理性审查

| 审查维度 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|---------|--------|--------|---------|--------|
| **全覆盖** | 100% 强制 | 仅变更域（已有域不重复检查） | 100% 强制（否则重建不完整） | 不适用（局部变更） |
| **1:1 约束** | 100% 强制 | 变更条目强制，已有条目抽查 | 对重建的映射 100% 验证 | 变更条目强制 |
| **N:1 承接量** | 2-5 条/节点推荐，>8 条需拆分 | 1-6 条/节点（增量累加后不超过 8 条），仅 1 条时评估是否可合并 | 重建映射 2-6 条/节点，超过需标注合理性说明；多节点仅 1 条→过分解风险 | 不涉及新建合并，仅检查变更后是否超 8 条；超载需标注并规划拆分 |
| **合并合理性** | 语义相关性判定 | 需额外检查变更是否打破已有合并的合理性 | 需验证重建的合并是否与实现一致 | 最小变更，通常不涉及合并 |

#### 11.4 符合性分析的跨模式差异

符合性分析（§3.3）是在分配完成后验证方案节点是否充分满足所承接需求的深度验证。四种模式在分析深度和验证焦点上存在显著差异。

##### 11.4.1 分析深度

本规范定义了三个层次的验证，四模式在其中选取不同组合：

| 深度层次 | 内容 | 规范依据 |
|---------|------|---------|
| **L1：快速自检** | 分解四检 + 分配三检，在核心活动进行中即时执行 | §3.1.1（语义自检四项）、§3.2.1（分配自检三项） |
| **L2：语义符合性** | 覆盖度、映射正确性、分析具体性的实质验证 | §3.3.1（语义符合性三维度） |
| **L3：指标符合性** | 指标对应完整性、比较正确性、可验证性的量化验证 | §3.3.2（指标符合性三检查） |

| 模式 | 分析深度 | 说明 |
|------|---------|------|
| **瀑布式** | L1 + L2 + L3（全深度） | 每个方案节点完成语义符合性和指标符合性分析，这是完整设计的标准配置 |
| **敏捷式** | L1 + L2（条件性 L3） | 变更节点完成语义符合性分析；指标符合性在变更涉及性能/容量/安全等约束时需执行，纯功能变更可跳过 L3 |
| **逆向工程** | L1 + L2（恢复后执行） | 对恢复的映射执行 L1+L2；L3 指标符合性因原始指标信息可能缺失，需标注"待确认"而非跳过 |
| **DevOps** | L1（变更验证） | 以快速自检为主——验证分解覆盖了问题链路、分配未违反 1:1、修复未引入新矛盾；指标符合性仅在涉及性能/安全等质量属性时执行 |

###### 条件性升级规则

各模式的默认分析深度（上表）在以下条件下必须升级：

**DevOps 条件性升级**（L1→L2/L3）：
- L1→L2（语义符合性）：修复方案涉及架构末级节点定义的修改（不仅仅是参数/配置调整），或变更涉及多条需求的映射关系调整，或变更影响非功能约束的承接
- L2→L3（指标符合性）：变更了性能/容量/安全的量化指标，或修复引入了新的质量属性约束，或变更涉及资源消耗的重新分配

**敏捷式跳过 L3 的边界**：
- 纯功能变更（不涉及附录B中质量特性/环境适应性大类）可跳过 L3
- 功能变更附带最小指标影响（指标值调整不超过 10%）可跳过 L3
- 变更涉及安全/性能/可靠性的指标必须执行 L3

**逆向工程 L3 的"待确认"处理**：
- "待确认"标注依据附录B的 NFR 分类设定优先级——质量特性>环境适应性>可实现性
- "待确认"必须在 6 个月内或下次迭代前补充完成，不允许永久跳过

##### 11.4.2 验证焦点

| 验证维度 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|---------|--------|--------|---------|--------|
| **语义覆盖度** | 全面的定义-需求逐条覆盖 | 变更节点-变更需求间的覆盖 | 重建映射的语义对齐——"PA 节点实际做的事"与"还原出的需求"一致 | 修复方案是否覆盖了问题原因 |
| **映射正确性** | 架构节点与所承接需求的术语匹配 | 变更是否打破了已有映射的语义关系 | 还原出的需求是否真实反映了 PA 节点的设计意图 | 变更是否保持了映射链路的完整性 |
| **具体性** | 每节点有实质分析文本 | 变更节点的分析文本需更新 | 重建的分析文本是否基于实现证据而非推测 | 分析以"变更前/后对比"形式呈现 |
| **指标符合性** | 全量指标逐项比对 | 仅变更涉及的性能/约束指标 | 缺失的指标标注"待确认"并记录来源 | 仅验证受影响的质量属性指标 |

### 十二、通用约束与质量指标

本章定义各模式在切换或阶段交付时必须满足的最低质量标准。本规范§七（设计验收标准）定义了系统设计的六个质量验收维度和相应检查清单——各维度在各模式中的应用范围和执行严格度不同。

本文的**质量门禁**是本规范§六在具体模式中的选择性应用策略——每种模式因设计范围、深度和时间约束的差异，对§六六个质量维度的执行严格度不同。矩阵中的"全量"表示该维度的全部验收标准必须满足（§六 对应条目 100% 通过），"条件性"表示仅与变更/重建相关的部分需满足，"最小"表示仅直接相关的部分需满足，"不强制/不适用"表示该维度在当前模式中可不执行。

#### 12.1 质量门禁矩阵

| 质量维度（§七） | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|-------------------|--------|--------|---------|--------|
| **§7.1 架构定义质量** | 全量：四要素完整 | 条件性：仅变更/新增节点需完成四要素 | 全量：重建的架构节点需四要素完整 | 最小：仅变更节点维护四要素 |
| **§7.2 详细定义质量** | 全量：100% 分解完成 | 条件性：仅变更的架构末级节点需展开详细定义 | 全量：重建的详细定义需通过四项自检 | 最小：仅变更的详细定义条目需通过自检 |
| **§7.3 映射关系质量** | 全量：100% 全覆盖 | 增量：变更域 100% 覆盖 + 已有域抽查 | 全量：重建映射 100% 覆盖 + 1:1 验证 | 局部：变更链路 100% 覆盖 |
| **§7.4 符合性分析质量** | 全量：语义+指标全深度 | 条件性：变更节点语义分析 + 质量属性影响需指标分析 | 全量：语义分析必须完成，指标分析"待确认"不阻塞交付但需记录 | 最小：语义自检 + 受影响指标的确认 |
| **§7.5 追溯完整性** | 全量需验证通过 | 条件性：变更链路追溯需完整 | 全量需验证通过（这是逆向工程的核心交付物） | 不强制（但需确保修复链路可追溯） |
| **§7.6 文档结对设计** | 全量：结对对齐率 100% | 条件性：变更结对需对齐 | 全量：重建的结对关系需对齐 | 不适用（DevOps 不涉及完整结对设计） |

> **门禁说明**："全量"=该维度的全部验收标准必须满足；"条件性"=仅与变更/重建相关的部分需满足，已有且不受影响的部分可复用；"最小"=仅直接相关的部分需满足；"不强制/不适用"=该维度在当前模式中可不执行。

###### NFR 分类体系引用

本规范附录B定义了非功能需求的完整分类体系（三个一级类目——质量特性、环境适应性、可实现性——展开为十八个子类）。该分类体系在质量门禁中的应用方式如下：

- **条件性 L3 的执行范围判定**：敏捷式和 DevOps 的条件性指标分析应参考附录B——当变更影响**质量特性**大类（性能、安全、可靠性、可用性、韧性）或**环境适应性**大类（可扩展性、弹性、互操作性）的相关需求时，必须执行 L3 指标符合性分析；变更仅涉及**可实现性**大类（可测试性、可部署性、可运维性等）时可仅执行 L1+L2
- **逆向工程"待确认"的优先级**：缺失指标按附录B分类标注"待确认"，质量特性类别的指标缺失优先级最高（需优先补充），环境适应性次之，可实现性可在后续迭代中补充
- **产品指南实例化要求**：产品指南在继承本文档时，应引用并实例化 NFR 分类体系，将其作为条件性 L3 判定的分类依据

#### 12.2 模式-质量执行策略

**瀑布式**

所有六个维度的验收标准全部执行。这是最严格的配置，适合对设计质量有最高要求的场景（关键系统、安全攸关系统、合规强约束场景）。架构定义和详细定义的质量是整个设计链的基础。

**敏捷式**

核心策略是仅验证变更影响域内的质量。已有且不受变更影响的架构节点和映射关系可复用其已有的通过记录。但需注意：如果多次敏捷迭代导致累积影响超过阈值（如同一节点经历 3 次以上的 Extend），应触发全量质量复查。

**逆向工程**

核心策略是恢复的质量不能低于正向设计的标准。所有重建的文档和映射关系必须通过全量验收（§7.1、§7.3、§7.5）。详细定义（§7.2）和符合性分析（§7.4）在指标信息可能不完整时允许标注"待确认"，但语义部分必须完整。逆向工程的最终产出物必须是可通过双向追溯验证的完整设计文档集。

**DevOps**

核心策略是最小变更、最小验证。仅在变更链路上执行必要的质量检查。但 DevOps 模式不能无限期使用——同一个产品或同一区域经历 3 次以上 DevOps 修复后，应触发一次敏捷式或瀑布式设计检视，确保设计文档与生产实现的一致性。

| 维度 | 瀑布式 | 敏捷式 | 逆向工程 | DevOps |
|------|--------|--------|---------|--------|
| **AI 自动化程度** | 高：全量分解、映射、分析可 AI 批量生成，人工审核 | 中高：影响范围分析、增量分解可 AI 自动执行，人工确认变更路径 | 高：逐层反向推导可 AI 批量执行，但还原结果的正确性需人工重点审核 | 中：定位和修复方案可由 AI 辅助，但根因分析需人工确认 |

> 以上为方法论层面的 AI 角色差异描述。具体 AI 操作脚本、提示词模板、自动化程度在各产品规范中实例化。

### 附录A 方法论底层推导链

> **阅读说明**：本附录完整展现在§〇（方法论逻辑总览）中仅以骨架呈现的推导逻辑。附录B为配套的非功能需求分类体系。每一层都是对上一层的唯一合理的操作化展开。如果对前提没有异议，每一步都是必然。

#### A.1 第〇层：先验约束——方法论之所以必然存在的原因

##### A.1.1 范围限定：什么是"设计"？

本文谈论的"设计"是**工程化意义上的设计**（engineering design），而非"凭智力构思意义下的设计"（design as intellectual conception）。

两者的区别：

| 维度 | 工程化设计 | 凭智力构思的设计 |
|------|-----------|----------------|
| **输入** | 明确的、外部的需求 | 内在的意图、灵感或愿景 |
| **过程** | 可分解、可分配、可分析的连续转化 | 构思—表达的一步或多步创作 |
| **输出** | 可验证的、可实现的产品方案 | 作品、概念、表达 |
| **验证标准** | 是否满足需求（语义+指标） | 是否符合创作意图（主观判断） |
| **终止条件** | 达到"可实现"的粒度 | 创作完成（主观决定） |

前者遵循可分解、可追溯、可验证的约束——**系统设计是工程化设计的子集**，特指从原始需求到产品架构的完整设计链路。以下全部推导链均立足"工程化设计"的范畴。

##### A.1.2 约束一：复杂必须分解

**任何一个超出理性一次性把握能力的复杂体，都必须被分割为可独立理解的部分，否则无法被认知和操作。**

- 这不是经验观察，而是理性有限性的直接推论
- 否认这一点，等于否认理性有限性——这是不可否认的

##### A.1.3 约束二：分解必须可追溯

**任何一个子部分都必须保持与它所来源的整体的追踪关系，否则它的存在理由不可回答。**

- 分解产生的不只是"部分"，而是"从整体中分离出的部分"
- 如果切断了子部分与父需求的关联，就无法回答"为什么要做这个子部分"——这违反了充分理由原则
- 因此，可追溯性不是方法论的选项，而是分解合法的必要条件

##### A.1.4 约束三：转化必须可验证

**任何一个从输入到输出的转化过程，都必须有手段验证其保真度，否则无法判断转化是否正确。**

- 如果输入（需求）到输出（方案）的转换缺乏验证手段，质量无法被判断，迭代无法被终止
- 这是控制论的基本要求：没有反馈回路的系统无法收敛
- 否认这条，等于否认"设计可以判断好坏"

##### A.1.5 三条约束的相互关系

```
复杂必须分解 ───→ 分解必须可追溯 ───→ 转化必须可验证
       │                  │                     │
       ▼                  ▼                     ▼
  否则无法理解       否则无法解释          否则无法判断
```

三者不是独立的——第一条使分解成为必要，第二条使追溯成为必要，第三条使验证成为必要。缺少任何一条，理性的设计活动都无法成立。

##### A.1.6 先验约束的效力

这三条约束**不决定方法论的具体形态**。它们决定的是：**方法论的存在本身是必然的**。任何理性的设计活动都必须以某种形式满足这三条约束。不满足"复杂必须分解"的方法论不是一个可能的选择——它是一个逻辑矛盾。先验约束只能规定"必须做什么"，不能规定"具体怎么做"——具体的回答来自对工程实践的经验归纳。

#### A.2 第一层：原始定义——后验基石

> **设计** = 将对客体的需求转换为对其更为详细的需求的一组过程
> （ISO 9000:2015, 3.4.8）

这是第一个后验输入。它不是先验真理，而是人类工程实践的经验结晶——ISO 技术委员会观察了各行业的设计活动，找出共同特征，归纳为这条定义。

它与先验约束的关系：

| 先验约束 | ISO 9000 定义的对应 | 满足了什么 |
|---------|-------------------|-----------|
| 复杂必须分解 | "转换为更详细的需求"（连续细化本身就是分解） | 分解的操作化——通过分层细化来分解复杂度 |
| 分解必须可追溯 | "对客体的需求"（标注了源头的转换） | 追溯的锚点——每次转换都指明了是从哪个客体的需求出发的 |
| 转化必须可验证 | "一组过程"（可被重复检查的操作序列） | 验证的前提——过程必须是可复现的，才能被检查 |

**关键语义提取**：

| 原文要素 | 含义 | 不可替换的理由 |
|---------|------|--------------|
| **转换** | 输入→输出的变换操作 | 设计不是创造，而是转化——有源头的、可追溯的 |
| **对客体** | 设计有明确的作用对象 | 不是抽象思维，是对一个具体的"什么东西"在做设计 |
| **需求 → 更详细的需求** | 输出与输入是同一种东西，只是粒度不同 | 需求和方案不是两种不同的事物——方案就是对需求的细化 |
| **一组过程** | 不是一次动作，而是多次、多步 | 单次转换做不到"更详细"，必须通过一组操作序列 |

这条定义隐含一组前提，显式化后可以引出后续全部推论：

| 隐性问题 | 答案 | 推论 |
|---------|------|------|
| 谁对什么做转换？ | 设计者对上层的设计输入（需求）做转换 | 设计不是单层的，是多层的——一次细化不够 |
| 输出给谁？ | 更详细的需求交给下一层继续设计或直接实现 | 中间层必须兼具双重身份：对上是方案、对下是需求 |
| 什么时候停？ | 输出已达到"可直接用于实现"的粒度 | 需要一个确定的终止边界——即"可实现" |

---

#### A.3 第二层：对原始定义的转化——从"设计"到"系统设计"

从原始定义出发，分两步完成转变。

**第一步：转化——将"转换"拆解为三个操作，并确定终止条件。** "转换"太笼统，无法直接操作，必须拆解为三个不同性质的活动——切分（分解）、指派（分配）、验算（符合性分析）；"更为详细"确定了迭代的终止条件是"实现或可实现"：

> **设计 = 对需求的持续分解、分配和符合性分析，直至实现或可实现**

**第二步：特化——将"设计"特化为"系统设计"。** 系统设计中的"实现或可实现"具体化为"产出可实现的产品架构（PA）"——设计终止于可交给开发团队独立实现的粒度（PA组件满足三独立原则）：

> **系统设计 = 对需求的持续分解、分配和符合性分析，直至产出可实现的产品架构。**（此即方法论的基石定义，也是全文唯一需要记住的一句话）

与原始定义的差异：

| ISO 9000 | 本方法论的转化 | 转化的含义 |
|----------|--------------|----------|
| 转换（transform） | **分解 + 分配 + 分析** | "转换"太笼统，无法直接操作。必须拆解为三个不同性质的活动：切分、指派、验算 |
| 更详细的需求 | **可实现的产品架构** | 终止条件从文本性（更详细）变为交付性（可实现）。设计不是无限细化到"写不动为止"，而是终止于可交给开发团队独立实现的粒度 |
| 一组过程 | **持续**迭代至终止 | 不是单次线性过程，而是每一层重复执行同一组动作，直到终止条件达成 |

这条基石定义包含四个概念要素，每一个都需要操作落地：

```
分解 ─── 分配 ─── 分析 ─── 实现
```

#### A.4 第三层：概念落地——每个概念指向的真实对象

##### A.4.1 分解——对谁分解？

**分解，是对架构定义的末级节点做分解。**

- 每层先有架构树，架构末级节点是该层"解决方案实体"的容器
- 分解是对这些容器展开详细定义条目——不是对需求文本做切分
- 架构末级节点 = 分解的锚点。没有它，分解没有落处

分解的产物是详细定义末级条目。每个条目最终要能 1:1 分配到下层架构末级节点。

##### A.4.2 分配——将分解后的条目分配给谁？

**分配，是将分解后的条目分配到下层架构末级节点。**

- 方向是：**条目 → 下层节点**（1:1约束），不是节点→节点
- 下层的每个架构末级节点拿到条目后，作为自己的需求输入进行方案设计
- 这就是文档结对机制的本质：上层详细定义末级条目标定"要做什么"，下层架构末级节点回答"怎么做"
- 1:1约束不是人为规定，而是分解充分性的检验标准：只要一条条目需要分配到多个下层节点，说明本层分解尚未充分

##### A.4.3 分析——分析什么？

**分析，是对分解的分析、分配的分析、节点承接需求的分析。**

"分析"并非仅指最后一个环节的符合性分析——它贯穿活动的全过程，是对三个活动各自的自检：

| 分析对象 | 分析什么 | 对应方法 |
|---------|---------|---------|
| **对分解** | 分解是否完整覆盖父需求全部功能？有无超出？子需求之间是否矛盾？约束是否继承？ | §3.1 四项自检 |
| **对分配** | 是否全覆盖？是否1:1？N值是否合理？ | §3.2 三项自检 |
| **对节点承接** | 方案定义是否覆盖需求语义？指标未降级？ | §3.3 两维度分析 |

三个分析缺一不可。只做节点符合性分析不能保证分解和分配的准确性。

##### A.4.4 实现——什么是实现？

**实现，是基于资产的实现。**

"可实现"不意味着"画完图纸"——它意味着：**每一个 PA 组件都可以与组织的现有资产建立明确的复用/改进/替代关系：**

- **复用**已有组件——零成本，直接可实现
- **改进**已有组件——低成本，可实现
- **外购**成熟方案——外部资产，可实现
- **新制**——以上均不可用时，必须通过三独立判定（可独立分配、可独立替换、可独立验证），确认组件边界合理后方可启动

所以"实现"不是一个技术状态，而是一个**组织可交接的边界**：PA 组件的粒度由"能否交给一个独立团队在已有资产基础上完成"决定。这就是三独立原则与基于资产原则的内在联系——也是五层链恰好在此终止的原因。

---

#### A.5 第四层：方法论推导——原则与五层链

##### A.5.1 四条原则——保障以上操作准确执行的组织机制

四个概念落地的操作都有了，但还需要组织机制来保障它们被准确执行。四条原则由此产生：

| 原则 | 回答的问题 | 保障哪个活动的哪个方面 |
|------|-----------|----------------------|
| **需求相对** (§2.1) | 分解的方向怎么定？ | 保障始终对上层的需求做分解，而不是对着下层实现做设计 |
| **详概分离** (§2.2) | 分解有地方放吗？ | 保障架构树定框架、详细定义填内容——两者分离才能独立演化 |
| **文档结对** (§2.3) | 分配怎么验证？ | 保障详细定义条目→架构末级节点的1:1映射可追溯 |
| **基于资产** (§2.4) | 节点从哪里来？ | 保障架构节点不是凭空产生，遵循复用→改进→新增的优先级顺序 |

##### A.5.2 五层设计链——完整推导的最终产物

五层链是以上所有推导的必然结果，不是人为设计的：

```
                设计起点（纯需求，无方案层）
                      │
                    OR─── 原始需求
                      │
              ┌───────┴───────┐
              │               │
              ▼               ▼
          SR-F（功能）    SR-NFR（非功能）
              │               │
              ▼               │
          BA（IPO）───────→───┤
              │               │
              ▼               ▼
       SysReq-F（功能）  SysReq-NFR（非功能）
              │               │
              └───────┬───────┘
                      │
                      ▼
                   PA（产品架构）──── 设计终点（可实现，交接给组件开发）
```

每一层存在的原因：

| 层 | 为什么必须存在 |
|---|--------------|
| **OR** | 设计的起点。没有它，设计的源头上层不存在 |
| **SR** | 方案（对OR）+ 需求（对BA/SysReq-NFR）。第一个中间层——双重身份 |
| **BA** | 功能需求完整性的操作主义保障。只有穷举用户操作场景才能系统性地保证FR完整。NFR没有等价方法，因此走平行路径 |
| **SysReq** | 方案（对BA/SR-NFR）+ 需求（对PA）。第二个中间层——双重身份。FR和NFR在此层汇合 |
| **PA** | 设计终点——组件可独立开发。三独立判定保证这一层就是终止边界 |

#### A.6 总结：完整推导链

```
           ┌─────────────────────────────────┐
  第〇层   │ 先验约束:                       │
           │ ① 复杂必须分解                  │
           │ ② 分解必须可追溯                │
           │ ③ 转化必须可验证                │
           │ (使方法论文的存在成为必然)        │
           └─────────────┬───────────────────┘
                         │ 后验回答
                         ▼
           ┌─────────────────────────────────┐
  第一层   │ ISO 9000: "设计=需求的连续细化"    │
           └─────────────┬───────────────────┘
                         │ 转化
                         ▼
           ┌─────────────────────────────────┐
  第二层   │ 基石定义: "系统设计=对需求的持续   │
           │ 分解、分配和符合性分析，直至实现"   │
           └─────────────┬───────────────────┘
                         │ 概念拆解
                         ▼
           ┌─────────────────────────────────┐
  第三层   │ 概念落地:                        │
           │ 分解→对架构末级节点               │
           │ 分配→分配到下层架构末级节点         │
           │ 分析→三层分析贯穿                  │
           │ 实现→基于资产的实现                │
           └─────────────┬───────────────────┘
                         │ 方法论推导
                         ▼
           ┌─────────────────────────────────┐
  第四层   │ 四条原则→保障操作准确执行          │
           │ 五层设计链→完整推导的最终产物       │
           └─────────────────────────────────┘
```

每一步都是对上一步唯一合理的操作化展开。没有一步依赖直觉或经验判断。
