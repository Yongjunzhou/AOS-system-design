# EOS 敏捷式系统设计：提示词模板集
**EOS Agile System Design: Workflow Prompt Templates**

**用途**：AI 为EOS（企业运营体系）执行敏捷式增量系统设计时，按 8 步流程依次使用对应的提示词模板。AI 将提示词中的 `[占位符]` 替换为实际内容后执行。

**重要前提**：本场景假设 EOS 已有完整的系统设计文档（5 层 + 映射矩阵）。AI 在执行前应确保已加载现有基线文档。

**依据B说明**：本提示词模板是EOS（企业运营体系）系统设计的AI辅助文档，是设计线产品架构的末级节点。AI在使用本模板为 EOS 生成增量设计文档时，应注意：
- 所有生成内容应面向 EOS 的领域（企业运营体系的业务流程、组织管理、绩效指标、信息系统等）
- 本模板基于设计线的敏捷式场景模板定制，内容聚焦 EOS 的运营业务场景
- EOS 产品架构属于软件产品类型，PA 分类为"前端/后端/数据/基础设施"

---

## 跨步骤上下文管理说明

### 上下文传递方式

8步增量设计流程中，从第2步开始，每步都基于上一步的产出。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 新需求分析结果 | 需求分类、影响范围、优先级 | 第1步输出 |
| 需求末级分解 | 分解后的原子性需求，冲突检测结果 | 第2步输出 |
| 增量定义 | 各层的 +/~ 增量内容 | 各步增量输出 |
| 同步设计 | 上层更新后下层骨架的同步 | 各步的同步设计记录 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |

### 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**中，供后续步骤引用。

### 当需要回溯修正

如果在第 N 步发现问题需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见[AI决策框架 -> 跨步骤影响范围表](../02-ai-decision-framework-eos.md#72-跨步骤影响范围表)）
2. 由于增量设计的影响逐层向下传递，修改上层后需检查下游所有增量是否需要同步调整
3. 修改后重新运行受影响步骤的 self-consistency check
4. 标注变更记录

## 第1步：新需求分析和分类

### 上下文继承

**从哪里来**：
  - 直接输入：EOS 新需求原始表述
  - 参考基线：现有 EOS 完整系统设计文档集

**去哪里**：
  - 本步产出 → 第2步（需求分解）的输入

**特别注意**：
  - 本场景假设 EOS 已有完整的系统设计文档（5 层 + 映射矩阵）

### 目标
分析 EOS 新需求的性质和影响范围，确定优先级。

### 输入
- EOS 新需求原始表述（作为本步骤的输入提供）
- 现有 EOS 完整系统设计文档集（基线）

### 涉及规范
- [原始需求规范 §1 核心概念](../10-eos-system-design-specification/02-original-requirements-specification.md#一核心概念)

### 决策框架参考

| 决策场景 | 处理方法 | 参考章节 |
|---------|---------|---------|
| 需求分类 | 功能 vs 非功能、新增 vs 修改 vs 删除 | 按指南执行 |
| 需求歧义（L1/L2/L3） | 可推测则补齐，多选项则列出 | [§4 需求歧义处理策略](../02-ai-decision-framework-eos.md#四需求歧义处理策略) |
| 碎片归并 | 血缘归并/角色聚合/流程串联 | 按指南执行 |
| 矛盾碎片 | 标注"冲突待确认"不合并 | 按指南执行 |

### 提示词模板

#### 1a. 新需求收集与分类

```

[在此处粘贴新需求原文]

现有 EOS 系统设计文档集：
[说明基线的文档路径]

请执行：
1. 需求分类 — 每条需求的类型：
   - 功能需求 vs 非功能需求
   - 新增需求 vs 修改需求 vs 删除需求
   - 独立需求 vs 相关需求（与现有需求的关系）

2. 碎片归并（可选）：
   - 血缘归并：同一功能不同切面的碎片合并
   - 角色聚合：不同角色对同一功能的诉求合并
   - 流程串联：同一流程不同步骤按顺序串联
   - 父子补全：以主需求为载体，补充细节作为子约束

3. 保留原始出处：
   - 记录提出方（业务部门/管理者/IT部门）、原始表述、日期

├─ 变更类型：[新增/修改/删除]
├─ 与现有需求的关系：[相关/独立]
```

#### 1b. 影响范围评估与优先级

```

1. 是否影响现有的相关方需求（SR）？
2. 是否影响现有的业务架构（BA）？

| SR | [新增/修改/无] | SR-F-XXX | [说明] |
| BA | [新增/修改/无] | BA-XXX | [说明] |
| SysReq | [新增/修改/无] | SysReq-F-XXX | [说明] |
| PA | [新增/修改/无] | PA-XXX | [说明] |
```

## 第2步：新需求分解

### 上下文继承

**从哪里来**：
  - 第1步产出：EOS 新需求分析报告（含分类和影响范围评估）

**去哪里**：
  - 本步产出 → 第3步（SR 影响分析）的输入

### 目标
对 EOS 新需求进行规范化处理和分解，得到末级需求。

### 输入
- 第1步产出的新需求分析报告

### 涉及规范
- [原始需求规范 §3 分解规范](../10-eos-system-design-specification/02-original-requirements-specification.md#三分解规范)

### 决策框架参考

| 决策场景 | 处理方法 | 参考章节 |
|---------|---------|---------|
| 需求分解粒度 | 按原子性原则，1:1 约束测试 | [§2.1 原始需求→末级的分解](../02-ai-decision-framework-eos.md#21-原始需求--末级的分解) |
| 功能/NFR分离 | 将"做什么"和"有多好"分开 | 按指南执行 |
| 冲突检测 | 直接/间接/可合并三级分类 | 按指南执行 |
| 重复检测 | 完全/部分/相似三级分类 | 按指南执行 |

### 提示词模板

#### 2a. 需求规范化与末级分解

```
请对 EOS 新需求进行规范化处理和末级分解。

EOS 新需求清单：[粘贴1a或1b的输出]

请执行：
1. 四项检查：
   - 完整性：每条需求是否有清晰的描述、类型、优先级？
   - 清晰性：模糊术语替换为精确表述（如"高效"替换为具体指标）
   - 可测试性：输入/处理/输出是否明确？
   - 独立性：是否与其他需求过度耦合？

2. 功能/非功能分离：
   - 将运营功能描述（做什么）和性能约束（多好）分开记录

3. 需求分解到末级：
   分解原则：
   - 原子性原则：每条末级不可再分
   - 1:1 映射原则：每条末级只需映射到一个 SR 架构末级节点
   - 粒度一致性：末级粒度与现有 EOS 需求一致
   - 语义完整性：分解后不丢失原始语义

输出格式：
OR-XXX: [需求标题]
├─ 规范化描述：[经过四项检查后的描述]
├─ 类型：[功能需求/非功能需求]
├─ 优先级：[高/中/低] ⚠️ 待确认
├─ 末级分解：
│  ├─ OR-XXX.1: [原子性功能描述] → SR-F 承接
│  ├─ OR-XXX.2: [原子性功能描述] → SR-F 承接
│  └─ OR-XXX.3: [原子性描述] → SR-NFR 承接
└─ 验收标准：[可量化的验收条件]
```

#### 2b. 冲突与重复检测

```
请对 EOS 新末级需求与现有末级需求之间执行冲突和重复检测。

新末级需求清单：[粘贴2a的输出]
现有末级需求清单：[从 EOS 基线文档中提取]

冲突检测（新需求 vs 现有需求）：
- 直接冲突：语义直接矛盾，无法同时满足 → 标记"需确认"
- 间接冲突：同时实现可能行为不一致 → 标记"需调整"
- 可合并冲突：可通过调整合并解决 → 标记"建议合并"

重复检测：
- 完全重复：与现有需求完全相同 → 去重（合并到现有需求）
- 部分重复：功能范围部分重叠 → 建议合并
- 相似重复：表达不同但核心相同 → 确认是否为同一需求

输出格式：
| 新需求 | 现有需求 | 问题类型 | 描述 | 建议 |
|-------|---------|---------|------|------|
| OR-XXX.X | OR-YYY.Y | [直接/间接/重复] | [描述] | [建议] ⚠️ |
```

### 预期产出
- 更新后的 EOS 原始需求文档（添加新需求）
- 新末级需求清单
- 冲突/重复检测报告

### 本步决策记录

```
本步关键决策记录：
├─ 需求分解：
│  ├─ 需求总数：[N] 条 -> 末级 [M] 条
│  ├─ 功能需求末级：[N] 条
│  └─ 非功能需求末级：[N] 条
├─ 冲突检测：发现 [N] 项（直接 [N] / 间接 [N] / 可合并 [N]）
├─ 重复检测：发现 [N] 项（完全 [N] / 部分 [N] / 相似 [N]）
├─ 冲突/重复处理：合并 [N] / 标注待确认 [N] / 保持独立 [N]
├─ ⚠️ 待确认项：[列出]
└─ AI推断记录：[列出所有AI补齐/推断的内容]
```

