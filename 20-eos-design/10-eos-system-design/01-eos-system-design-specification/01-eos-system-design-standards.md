## 一、EOS 的产品语境

### 1.1 EOS 产品概述

#### 产品类型

EOS 为**统一可配置平台**，其 PA 构件分为两类形态：平台 PA 为**软件组件**（引擎的前端 UI 组件、后端服务、数据模型、API、基础设施）；业务 PA 为**结构化配置定义**（表单定义、标签页定义、台账定义、流程定义等），由引擎配置产生。

#### 什么是 EOS

EOS（Enterprise Operation System）是企业运营体系的简称。本准则关注的企业范围涵盖市场、研发、生产、售后和管理等业务。

#### 平台目标

EOS 的建设目标是构建一个**统一的可配置运营平台**，从根本上消除传统企业信息系统各自独立建设带来的信息孤岛和业务碎片化问题。所有业务域（产品研制、市场、生产、售后、经营管理）在同一个平台体系内贯通，实现数据共享和业务协同。

#### 平台定位

EOS 不是按功能线切分的传统 ISV（Independent Software Vendor，独立软件供应商）产品集（如独立的 ERP、PDM、OA），而是一个以**引擎为能力基础、以配置为使用方式**的统一平台：

| 传统 ISV 模式 | EOS 平台模式 |
|--------------|-------------|
| 各功能线独立产品，数据隔离 | 统一平台，共享主数据和实体模型 |
| 各产品接口集成，维护多套 | 引擎清单即 PA，天然集成 |
| 按产品采购和升级 | 按引擎扩展和演进 |
| 信息孤岛需额外治理 | 业务贯通是设计保证（统一业务链） |

#### 产品开发四阶段

产品开发一般分为四个阶段：

| 阶段 | 名称 | 说明 |
|------|------|------|
| 阶段1 | **系统设计** | 从原始需求出发，经过五层结构（OR → SR → BA → SysReq → PA），开发出产品架构，定义每个构件的需求和实现策略 |
| 阶段2 | **构件开发** | 对产品架构中所有需要新开发或改进的构件进行开发 |
| 阶段3 | **系统集成与交付** | 沿产品架构的树形层级逐级集成，形成满足系统需求的完整产品 |
| 阶段4 | **系统运维** | 产品交付后的持续运行、监控、维护和优化 |

**本准则的范围**：严格限定在**阶段1（系统设计）**。

#### EOS 开发者的三个角色

| 角色 | 职责 | 关注点 | 对应设计链 |
|------|------|--------|-----------|
| **引擎开发者** | 使用平台设计链（§2.3.2）对引擎进行分析、设计和实现 | 引擎功能完备性、性能、可扩展性 | 平台设计链 |
| **业务组件配置者** | 使用引擎配置业务组件（表单定义、台账定义、流程定义等） | 配置正确性、与引擎能力的匹配 | 业务设计链（§2.3.3） |
| **业务配置者** | 编排业务组件为业务信息系统模块 | 业务完整性、跨组件贯通性 | 业务设计链（§2.3.3） |

> 本准则同时服务于上述三类 EOS 开发者。

### 1.2 EOS 平台架构

#### 四层架构

EOS 平台按四层结构组织，各层对应不同的角色和产出：

| 层 | 角色 | 活动 | 产出 |
|----|------|------|------|
| **引擎层** | 引擎开发人员 | 开发底层引擎 | 表单引擎、流程引擎、工单引擎等基础能力 |
| **组件层** | 引擎应用用户 | 用引擎做配置 | 表单定义、标签页定义、台账定义等业务组件 |
| **模块层** | 业务配置人员 | 编排业务组件 | 各类业务信息系统模块 |
| **使用层** | 业务使用人员 | 使用模块开展工作 | 完成本职业务 |

#### 引擎清单（概览）

| 类别 | 引擎 |
|------|------|
| 基础 | 表单引擎、标签页引擎、菜单引擎、台账引擎、字典引擎、主数据引擎、关系引擎 |
| 流程 | 流程引擎、工单引擎、进展引擎 |
| 管理 | 项目引擎、WBS引擎、计划引擎、指标引擎、看板引擎 |
| 工程 | xBOM引擎、版本管理引擎、MRP运算引擎、MRO算法引擎、追踪引擎 |
| 集成 | CAX 引擎（桥接外部工具） |

> 完整引擎清单见产品数据文档。CAX 工具等外部系统的原始需求在 OR 阶段分流（§2.3.1），CAX 引擎仅作为集成桥接节点，其自身开发走平台设计链。

#### 架构图

```
┌─────────────────────────────────────────────────┐
│ 使用层：业务使用人员                              │
│ （市场人员、研发人员、生产人员、售后人员、管理人员） │
├─────────────────────────────────────────────────┤
│ 模块层：业务配置人员编排业务组件 → 信息系统模块     │
├─────────────────────────────────────────────────┤
│ 组件层：引擎应用用户配置引擎 → 业务组件             │
│ （表单定义、台账定义、流程定义、工单定义、指标定义…）│
├─────────────────────────────────────────────────┤
│ 引擎层：引擎开发人员开发 → 底层引擎                │
│ （表单/流程/工单/项目/台账/MRP/版本管理…等引擎）    │
└─────────────────────────────────────────────────┘
```

### 1.3 设计场景

EOS 的系统设计可按以下四种设计方法场景推进。设计方向取决于 EOS 产品当前所处的阶段：

| 方法 | 设计方向 | 起点 | 终点 |
|------|---------|------|------|
| **瀑布式** | 正向完整：OR → SR → BA → SysReq → PA | 原始需求 | 产品架构 |
| **敏捷式** | 正向增量：分析影响 → 增量更新 | 现有系统设计 | 更新的系统设计 |
| **逆向工程** | 反向推导：PA 及构件 → 反向推导至 OR | 已有构件 | 完整系统设计 |
| **DevOps** | 最小正向变更：定位 → 修复 → 验证 | 生产环境问题 | 快速修复/发布 |

**选择决策**：

```
EOS 产品处于什么阶段？

┌─ 已开发完成，但文档不全？
│  └─ 方法：逆向工程
│     └─ 从已有构件反向推导出完整系统设计文档
│
├─ 只有原始需求，需要从零开始？
│  └─ 方法：瀑布式
│     └─ 逐层正向推导（OR → SR → BA → SysReq → PA）
│
├─ 已有完整系统设计，需要增加新需求或修改现有构件？
│  └─ 方法：敏捷式
│     └─ 分析影响范围 → 增量更新受影响的文档
│
└─ 产品已上线运行，需要快速修复？
   └─ 方法：DevOps
      └─ 快速评估影响 → 最小化变更 → 快速验证
```

**各方法风险特征**：

| 方法 | 主要风险 | 应对策略 |
|------|---------|---------|
| **瀑布式** | 需求不清晰导致后期返工 | 在 OR 层做充分的需求分析和冲突检测 |
| **敏捷式** | 增量范围扩散超出迭代边界 | 严格控制每次迭代的影响范围分析 |
| **逆向工程** | 反向推导中对原始意图的理解偏差 | 多方确认推导结果，保留推导依据 |
| **DevOps** | 快速修复引入副作用 | 最小化变更范围，变更前后做回归验证 |

> 通用方法框架详见[通用设计指南](../../../00-general/10-general-system-design-standards/02-general-system-design-4modes-guide.md)。

### 2.1 三层工作层面

系统设计流水线（设计线）和 EOS 之间的关系由 **三个工作层面** 构成：

| 层面 | 内容 | 负责方 | 仓库位置 |

#### 内嵌关系

### 2.2 EOS 运营体系中的端到端业务框架

EOS 运营体系的顶层结构按业务领域分解为若干业务域（BA 1级），每个业务域包含一组端到端项目业务（E2E项目业务，BA 2级）：

| 层级 | 名称 | 说明 | 示例 |
|------|------|------|------|
| 1级 | 业务域 | 按业务领域划分 | 产品研制、市场、生产、售后、经营管理 |
| 2级 | 端到端项目业务（E2E项目业务） | 从需求提出到需求满足的完整闭环 | 各业务域内的端到端流程 |

每个 E2E项目业务从需求提出到需求满足构成完整闭环，跨域协作通过需求—响应契约实现（§2.2.3）。以下各节依次定义端到端的含义（§2.2.1）、客户与业务维度的分类（§2.2.2）、E2E项目业务间的接口规则（§2.2.3）、E2E项目业务的分解方法（§2.2.4）以及以流水线类为例的系统需求推导逻辑（§2.2.5）。

#### 2.2.1 端到端的定义

**从需求提出到需求满足**。

#### 2.2.2 三类客户与三个业务维度

企业有三类客户，每类客户的需求构成一个业务维度：

| 客户类型 | 提出的需求 | 对应的业务维度 |
|---------|-----------|--------------|
| 产品用户/客户 | 产品订货合同、产品研发需求、售后服务需求 | **主价值链** |
| 上级/监管方 | 经营管理目标、合规及资质取证、年度审核 | **管理监管** |
| 内部员工 | 对人、资金、物料、设备、设施、工具、数据、知识资产等需求 | **支持服务** |

三个维度与业务域（§2.2.4 BA层级1级）是不同性质的概念：业务域是 EOS 运营体系的结构分解（按业务领域划分），三个维度是对 E2E项目业务需求来源的分类（按客户类型划分）。同一业务域内的不同 E2E项目业务可能响应不同维度的需求。

同一 E2E项目业务同时受三个维度作用——归属于某个维度（以其主要服务的客户类型作为定位），被其他维度管理，向其他维度提出支持需求。

#### 2.2.3 E2E项目业务间接口原则

需求在端到端项目业务A的具体业务运行中产生，接收方是端到端项目业务B整体（BA 2级），B接收后自主决定内部如何分解响应，A不需要了解B的内部结构。不同维度的E2E项目业务之间的接口按以下三种关系处理：

| 关系 | 本质 | 适用范围 |
|-----|------|---------|
| 上下游 | 链式传递 | 同维度内（主价值链内部） |
| 管理/被管理 | 下达要求、监督考核 | 管理监管维度 → 其他维度 |
| 支持/被支持 | 提出需求、统筹满足 | 支持服务维度 → 其他维度 |

#### 2.2.4 E2E项目业务分解方法

端到端项目业务的分解从**被实现对象的系统架构**中工程化推导而来：

```
端到端项目业务（BA 2级）
├── 价值业务 ← 由E2E项目业务的"产品"决定
│     └── E2E项目业务产出什么（广义产品/服务）
│           └── 该产品的系统架构 → 架构节点类型
│                 └── 每个节点类型 → 一个从需求提出到需求满足的端到端业务
└── 管理业务 ← E2E项目业务本身的管控活动（目标、进度、成本、质量等）
```

##### BA 层级结构

| 层级 | 名称 | 说明 |
|------|------|------|
| 0级 | 架构根节点 | 整个系统的顶层根节点（唯一） |
| 1级 | 业务域 | 按业务领域划分 |
| 2级 | 端到端项目业务 | 从需求提出到需求满足的端到端项目业务 |
| 3级 | 项目业务对象的架构节点类型端到端业务 | 由产品架构节点类型推导得出 |
| 4级 | 工作包（WBS） | 业务对象的工作分解结构 |
| **5级** | **业务组件（承接层）** | **末级节点，使用 IPO 模型完整定义** |

##### 域间需求原则

端到端业务天然跨越多个业务域，但每个域只对自己域内的业务负责。跨域协作通过**需求—响应契约**来定义：

```
域A（端到端拥有者）
  ├── 定义：域A内部协调自己的端到端流程
  ├── 当需要域B支持时：向域B提出明确需求
  │     ↓
  └── 域B在自己的BA中响应该需求
        ├── 在自己的架构中安排能力（IPO）来满足域A的需求
        └── 定义服务范围、SLA和接口契约
```

#### 2.2.5 E2E项目业务流水线类的系统需求推导

本节以下先用通用推导逻辑说明流水线类的 5 步推导方法，再通过两个实例展示推导逻辑在不同目标产品形态下的应用。

流水线类的核心特征是其价值业务的推导锚点发生偏移：一般产品实现类的价值业务以该业务自身产出的"产品"的系统架构为推导锚点（§2.2.4）；流水线类的"产品"是开发/生产/构建**目标产品**的能力，其价值业务不能以自身能力架构为准，而应以**目标产品的产品架构**为推导锚点——目标产品有什么类型的构件，就需要有什么操作步骤来生产它，这些步骤就是流水线的 BA IPO。目标产品可以是实物、软件或方法/工具集等任何形态。

**通用推导逻辑**：

| 步 | 推导内容 | 在流水线类中的含义 |
|----|---------|----------------|
| **①** | 识别**目标产品（产品 B）**的产品架构及节点类型——明确 B 由哪些类型的构件组成 | 推导锚点（§2.2.4）：有什么构件，就需要有什么开发操作步骤来生产它 |
| **②** | 对**流水线**的 SR 功能部分展开详细定义，分解到可分配到 BA IPO 的粒度 | SR 功能详细定义是 BA 的需求输入，其分解充分性决定 BA 的完整性 |
| **③** | 将**流水线**的 SR 详细定义末级按 1:1 分配到**流水线**的 BA IPO，以 IPO 描述使用该流水线开发目标产品某类构件的操作步骤 | BA 描述的是"如何使用该流水线"——即流水线的操作规程（OpsCon） |
| **④** | **流水线**的 BA 止于 IPO，IPO 清单即 BA 的完整内容 | BA 仅架构定义、无详细定义（通用 §1.3） |
| **⑤** | **流水线**的 BA IPO 去重后，直接映射至**流水线**的 SysReq 功能架构 | BA→SysReq 是去重后的直接映射（通用 §5.5），而非 N:1 分配 |

**完整推导路径**：

```
目标产品 B 的 PA 节点类型（推导锚点）
  → OR → SR（功能部分）                               ← 前置
    → 流水线的 BA（IPO 清单）
      → BA IPO 去重 → 流水线的 SysReq（功能架构）
        → 流水线的 PA（自身产品架构）                   ← 超出本节范围
```

> 流水线的 PA 形态随目标产品类型而变：若目标产品为实物（如机载设备），PA 为准则/指南/AI辅助/任务定义等开发工具集；若目标产品为软件（如引擎），PA 为开发规范、构件框架、测试工具等不同类型的开发支持工具。推导逻辑不变。

**实例一：产品开发流水线**（目标产品为机载设备/实物）

产品开发流水线是 EOS 产品研制业务域中的一个流水线类端到端项目业务。

```
目标产品 B（机载设备）
  ↑ 开发产生
产品开发流水线（BA 2级端到端项目业务，属于产品研制业务域）
  ↑ 系统设计产生
EOS 运营体系
```

| 步 | 通用推导 | 实例化：机载设备产品开发流水线 |
|----|---------|------------------------------|
| **①** | 识别**目标产品（产品 B）**的 PA 节点类型 | 识别**目标产品（机载设备）**的产品架构——按产品分解结构识别各节点类型（电子组装件、结构部件、软件模块等） |
| **②** | 对**流水线**的 SR 功能部分展开详细定义 | 对**产品开发流水线**的 SR 功能部分展开详细定义 |
| **③** | **流水线**的 SR 详细定义末级 → **流水线**的 BA IPO（1:1） | **产品开发流水线**的每个 BA IPO 描述开发**目标产品（机载设备）**某类构件的一个步骤，如"需求分析 IPO：接收机载设备系统需求 → 需求分析与分配 → 输出系统规格说明" |
| **④** | **流水线**的 BA 止于 IPO | **产品开发流水线**的 BA 架构树止于 IPO 末级节点 |
| **⑤** | **流水线**的 BA IPO 去重 → **流水线**的 SysReq 功能架构 | 将**产品开发流水线**的 BA IPO 去重后按开发功能分类（需求分析、设计、仿真、验证等），重组为**产品开发流水线**的 SysReq 功能部分架构定义 |

**实例二：平台开发流水线**（目标产品为 EOS 引擎/软件）

| 步 | 通用推导 | 实例化：平台开发流水线 |
|----|---------|---------------------|
| **①** | 识别**目标产品（产品 B）**的 PA 节点类型 | 识别**目标产品（EOS 引擎）**的产品架构——按引擎清单（§1.2）识别各引擎类型（表单引擎、流程引擎、工单引擎、台账引擎、项目引擎等） |
| **②** | 对**流水线**的 SR 功能部分展开详细定义 | 对**平台开发流水线**的 SR 功能部分展开详细定义 |
| **③** | **流水线**的 SR 详细定义末级 → **流水线**的 BA IPO（1:1） | **平台开发流水线**的每个 BA IPO 描述开发**目标产品（某类引擎）**的一个步骤，如"流程引擎开发 IPO：接收流程引擎系统需求 → 流程引擎架构设计 → 流程执行引擎实现" |
| **④** | **流水线**的 BA 止于 IPO | **平台开发流水线**的 BA 架构树止于 IPO 末级节点 |
| **⑤** | **流水线**的 BA IPO 去重 → **流水线**的 SysReq 功能架构 | 将**平台开发流水线**的 BA IPO 去重后按引擎开发功能分类（架构设计、接口定义、数据模型设计等），重组为**平台开发流水线**的 SysReq 功能部分架构定义 |

> 两个实例的对比说明：无论目标产品是实物还是软件，流水线类的推导逻辑一致，仅①步的目标产品 PA 节点类型不同——**有什么构件，就需要有什么开发操作步骤来生产它**。

**与 §2.3 两条设计链的关系**：通过平台开发流水线推导出的引擎开发系统需求，在设计方法层面走 §2.3.2 平台设计链完成从 SysReq 到 PA 的后续推导。平台开发流水线（BA 层面的 E2E项目业务定义）和平台设计链（系统设计方法层面的工具）是同一组引擎开发活动的两个视角——前者回答"引擎开发需要哪些操作步骤"，后者回答"这些步骤如何遵循五层设计链路展开"。

### 2.3 EOS 的两条系统设计链

EOS 的四层平台架构（§1.2）决定了原始需求经过分析后，应分流至两条平行的系统设计链——平台设计链和业务设计链。

#### 2.3.1 需求的三种来源分类

原始需求到达后，首先按来源识别其类型：

| 需求类型 | 来源 | 发现时机 | 示例 | 设计链 |
|---------|------|---------|------|--------|
| **引擎开发需求** | 引擎能力有缺口，需新增或改进引擎 | ① **原始需求直接提出**；② **业务链推进中暴露**（§2.3.4）——后者是平台需求的主要来源之一 | 新增流程引擎、改进 MRP 算法 | 平台设计链（§2.3.2） |
| **业务配置需求** | 使用已有引擎配置新业务组件或模块 | 原始需求直接提出 | 用台账引擎配置设备台账 | 业务设计链（§2.3.3） |
| **CAX 工具需求** | 外部专业工具能力，平台不内置开发 | 原始需求直接提出 | CAD、CAE 工具集成 | OR 分流至外部 |

#### 2.3.2 平台设计链

平台设计链的目标产出是**引擎本身**——引擎及其构成软件构件（前端 UI 组件、后端服务、数据模型、API 等）。

```
平台 OR → 平台 SR → 平台 BA(IPO=引擎组件) → 平台 SysReq → 平台 PA(引擎)
```

| 层 | 在平台链中的含义 | 产出形态 |
|----|-----------------|---------|
| **OR** | 引擎开发的需求来源（如"需要支持业务流程自定义"） | 平台原始需求 |
| **SR** | 功能需求 + 非功能需求（性能、可扩展性、可靠性等） | 平台相关方需求 |
| **BA** | IPO 的处理实体为**引擎组件**，描述引擎内部处理逻辑 | 平台业务架构 |
| **SysReq** | 引擎功能分解与接口定义 | 平台系统需求 |
| **PA** | 引擎构件（后端服务、数据模型、API、基础设施） | 平台产品架构 |

> 平台设计链对应的 BA 层面 E2E项目业务定义为**平台开发流水线**（§2.2.5）。平台设计链回答"引擎开发需求如何走系统设计链路"，平台开发流水线回答"引擎开发需要哪些操作步骤"——两个视角互补。

#### 2.3.3 业务设计链

业务设计链的目标产出是**使用引擎配置出的业务组件**——表单定义、标签页定义、台账定义、流程定义等配置产物。

```
业务 OR → 业务 SR → 业务 BA(IPO=业务组件) → 业务 SysReq → 业务 PA(配置定义)
```

| 层 | 在业务链中的含义 | 产出形态 |
|----|-----------------|---------|
| **OR** | 业务配置需求（如"需要一套设备台账管理功能"） | 业务原始需求 |
| **SR** | 业务功能需求 | 业务相关方需求 |
| **BA** | IPO 的处理实体为**业务组件**，描述业务活动流程 | 业务链业务架构 |
| **SysReq** | 业务功能分解 | 业务系统需求 |
| **PA** | 配置产物：表单定义、标签页定义、台账定义、菜单定义、流程定义、工单定义、指标定义等 | 业务产品架构 |

> 所有业务配置需求共享同一条业务设计链，业务构件天然共用同一套实体模型和数据定义——这正是消除信息孤岛、实现业务贯通的设计保证。

#### 2.3.4 两条链的关系

**能力边界关系**：平台链产出的引擎是业务链的能力边界。业务链只能在已有引擎支持的范围内做配置，无法超越引擎能力。

**需求上升机制**：这是引擎开发需求的**第二来源**（与原始需求直接提出并列），也是平台需求持续演进的主要驱动力。当业务配置需求在推进过程中反复触及引擎能力边界——同一类需求需重复配置、或现有引擎功能无法满足——应上升为引擎开发需求，走平台设计链扩展引擎能力。新引擎交付后，业务链获得更强的能力底座，支撑更多业务配置。如此形成"业务发现缺口→平台补齐能力→业务再扩展"的持续循环。

**共享方法论**：两条链均遵循通用准则的五层设计链路和三条核心规则。关键区别在于 BA IPO 的处理实体不同——平台链是"引擎组件"，业务链是"业务组件"。

**架构一致性**：两条链的 PA 共同构成 EOS 的完整产品架构：

```
EOS 产品架构
├── 平台 PA：引擎（表单引擎、流程引擎、工单引擎、项目引擎……）
└── 业务 PA：配置定义（表单定义、台账定义、流程定义、工单定义……）
```

#### 2.3.5 需求分流判断逻辑

原始需求到达后的分流决策：

```
原始需求
  │
  ├─ CAX 工具需求？──→ 分流至外部，不纳入平台设计链
  │                     （CAX 引擎仅作集成桥接）
  │
  ├─ 引擎开发需求？──→ 走平台设计链（§2.3.2） ←───────┐
  │                      │                            │
  │                      └──→ 产出：引擎或引擎改进 ────┤
  │                                                    │ ② 需求上升
  └─ 业务配置需求？──→ 走业务设计链（§2.3.3） ─────────┤
                         │                            │
                         ├──→ 引擎能力足够？           │
                         │      └──→ 产出：配置定义     │
                         │                            │
                         └──→ 引擎能力不足？────────────┘
                               → 需求上升为引擎开发需求
                               → 返回走平台设计链（§2.3.2）
                               → 扩展引擎能力后再支撑业务配置
```

① 原始需求直接提出引擎开发需求——走前一路径。
② 业务配置需求在推进中暴露引擎缺口——走折返路径，需求上升为引擎开发需求。
两条设计链在各自工作层面上并行运行，互不阻塞。

### 2.4 PA 节点定义原则

**上下文与问题**：在系统设计链路中，SysReq 条目需要分配到 PA 架构节点，形成各构件的构件需求。分配决策不仅要确定每条 SysReq 属于哪类构件，还需判断同类内的多条 SysReq 是否应合并或拆分。

**适用范围**：本节原则适用于**平台设计链**（§2.3.2）——引擎作为软件组件，其 PA 分解按软件模块边界划分。业务设计链（§2.3.3）的 PA 为配置定义，其节点边界由引擎的配置机制直接决定（每个表单定义、台账定义等天然是一个独立配置单元），无需经三因素判断；三因素在业务链中仅作为"同类型配置定义是否合并"的参考依据。

决策按以下次序执行：首先进行资产优先检查（§2.4.1），再基于三因素（§2.4.2~§2.4.4）判断。

#### 2.4.1 资产优先检查

**先检查组织已有组件资产**，遵循复用→改进→新增优先级（详见通用准则 §2.4.2）。EOS 的资产形态以已有软件模块、平台服务、可复用组件为主：

| 优先级 | 策略 | 适用条件 | 操作 |
|--------|------|---------|------|
| 最高 | **复用** | 现有构件完全满足需要 | 直接使用，无需新分配 |
| 次之 | **改进** | 现有构件可扩展/拆分/合并后满足 | 调整现有构件后分配 |
| 最低 | **新增** | 无合适现有构件 | 创建新构件，用三因素判定边界 |

#### 2.4.2 因素一：信息模型内聚性

两个业务活动如果**操作同一组核心实体**（共享主数据、流程天然连贯），倾向合并为同一构件：

```
强内聚 → 可共享：
  客户信息 ←→ 销售订单 ←→ 售后服务
  → 共享客户主数据，流程天然连贯

弱内聚 → 倾向分离：
  设计 BOM ←→ 生产 BOM ←→ 维修 BOM
  → 虽都叫"BOM"，管理目的、变更节奏、精度要求不同
  → 强行合并导致数据模型被最低公分母拖累
```

**原则**：操作同一核心实体 → 合并；实体不同但有关联 → 通过接口集成。

#### 2.4.3 因素二：NFR Profile 相容性

不同业务活动对非功能需求的要求可能截然不同。当 NFR Profile 冲突时，它们**不应共享同一构件**：

| 维度 | 生产执行（MES） | 经营分析（BI） |
|------|---------------|---------------|
| 响应时间 | 毫秒级（产线节拍） | 分钟级可接受 |
| 可用性 | 99.99%（停机即停产） | 99.9% |
| 数据量 | 实时高频小数据 | 批量大吞吐 |
| 安全要求 | 操作权限精细 | 只读+脱敏 |

这两个活动的 NFR Profile 相互冲突——一个要求实时低延迟，一个要求批量大吞吐。强行合并会导致两头不靠。

> **NFR 走平行路径（SR-NFR → SysReq-NFR → PA）的根本意义在此**：NFR 差异本身就是 PA 分解的依据，而非仅在 SysReq 层做约束分析。

#### 2.4.4 因素三：信息处理模式一致性

不同业务活动消费和生产信息的方式存在本质差异（详见 §2.5）。在此作为 PA 分解的判断依据：

| IPO 组合 | 可否共享？ | 原因 |
|---------|-----------|------|
| 同模式（前9类） | 倾向共享 | 跨领域通用，NFR 通常相容 |
| 同模式（后4类）+ 同领域 | 倾向共享 | 同一专业领域 |
| 同模式（后4类）+ 不同领域 | 分离 | 领域不同则数据模型不同 |
| 不同模式 | 倾向分离 | 信息处理逻辑不同 |

#### 2.4.5 三因素决策矩阵

| 信息模型 | NFR Profile | 信息模式 | 结论 |
|---------|------------|---------|------|
| 相同 | 相容 | 相同 | **合并**为同一构件 |
| 相同 | 冲突 | — | **分离**（NFR 优先） |
| 不同 | 相容 | 相同 | 看耦合度，接口集成 |
| 不同 | 冲突 | 不同 | **必然分离** |

> **核心认知**：PA 层的构件分解是对 BA 层业务边界的读取而非发明。架构设计的质量取决于能否读出这些自然边界。

### 2.5 参考：业务信息属性

#### 2.5.1 产品类型与产品架构形态

同一套五层方法论可应用于不同产品类型，但 PA 层的产物形态取决于**产品类型**：

| 产品类型 | 特点 | PA 构件的本质 | 示例 |
|---------|------|-------------|------|
| **知识/方法类** | 用户是人+AI，消费方式是阅读和理解 | 知识制品（准则、指南、指令） | 元流水线 |
| **软件类** | 用户是人+机器，消费方式是操作和运行 | 软件模块（前端、后端、数据） | 通用软件产品 |
| **可配置平台类** | 平台 PA 为引擎（软件组件），业务 PA 为配置定义 | 引擎构件 + 结构化配置定义 | EOS |
| **实物类** | 用户是使用者，消费方式是物理交互 | 物理组件（零件、装配件） | 硬件产品 |

**EOS 为可配置平台类产品**，其 PA 构件分为两类：平台 PA 为软件模块（引擎），业务 PA 为结构化配置定义。详见 §1.1 产品类型和 §1.2 平台架构。

#### 2.5.2 业务活动的信息处理模式

企业运营体系包含多种业务活动，其本质差异在于处理信息的方式不同。这些差异不是技术选择，而是业务自身的信息属性决定的。

以下十三类模式是理解业务活动边界的参考工具，在两条设计链中各有用途：

- **引擎开发者**：不同模式的业务活动应由不同引擎承载，同模式可共享引擎——作为引擎边界划分的参考
- **业务配置者**：识别业务需求所属的模式，选择对应的引擎进行配置

**信息处理模式十三类**：

前9类为**跨领域通用**的处理模式：

| 模式 | 核心特征 | 典型系统 / 对应 EOS 引擎 |
|------|---------|------------------------|
| **计划事务** | 基于约束条件生成资源/时间分配方案 | 计划系统 / **计划引擎** |
| **工单事务** | 创建→派发→执行→反馈→关闭的闭环处理 | 工单系统 / **工单引擎** |
| **审批流程** | 按规则链路由、汇集决策意见、形成审批结论 | **流程引擎** |
| **合规事务** | 对照规则库检查行为/数据、判定合规性、记录证据链 | 合规系统 / 多引擎组合 |
| **规范知识** | 结构化编撰准则/规程/指南，规范人员操作或AI执行 | 文档系统 / — |
| **指标看板** | 从多源数据聚合指标并以可视化形式呈现 | 看板/BI系统 / **指标引擎+看板引擎** |
| **进展追踪** | 跟踪状态变更、比对计划与实际、标识偏差 | 追踪系统 / **进展引擎** |
| **信息台账** | 对信息进行登记、维护、检索和状态管理 | 台账系统 / **台账引擎** |
| **xBOM** | 管理产品结构及配置关系，追踪物料清单的版本和变更 | BOM管理系统 / **xBOM引擎+版本管理引擎** |

后4类为**领域特有**的处理模式：

| 模式 | 核心特征 | 典型系统 / 对应 EOS 引擎 |
|------|---------|------------------------|
| **设备IO** | 与物理设备进行实时数据交换，采集和控制IO信号 | 设备控制/SCADA / **CAX引擎（桥接）** |
| **领域分析** | 特定领域公式/模型驱动的计算分析 | 专业分析工具 / **CAX引擎（桥接）** |
| **领域建模** | 构建领域模型、运行仿真场景、输出预测结果 | 建模仿真平台 / **CAX引擎（桥接）** |
| **领域算法** | 封装特定领域逻辑的算法执行 | **MRP运算引擎、MRO算法引擎** 等 |

**核心规律**：判断两个业务活动是否共享系统——**不同模式 → 分离；同模式 → 可共享**。后4类即使模式相同，如果领域不同，仍然需要分离。

## 三、通用方法论精要

本章以极简篇幅摘要列出通用准则的核心方法论，并标注其在 EOS 语境下的关注点。完整定义请直接查阅通用准则对应章节。

> **重要**：以下各节均非通用准则的替代品，而是帮助你快速定位到所需的通用方法内容。初次接触 EOS 系统设计前，**建议先阅读本文档 §一 产品语境和 §二 EOS 的特有设计框架**，建立对平台架构和两条设计链的理解后再进入通用准则的方法论。

### 3.1 五层结构（通用 §四）

EOS 的系统设计遵循标准的五层设计链路，各层在 EOS 语境下的关注点如下：

| 层级 | 适用链 | 在 EOS 中的关注点 |
|------|--------|----------------|
| **OR → SR** | 双链 | 原始需求到来自何渠道，按三种来源分类（§2.3.1）分流至对应设计链后规范化为相关方需求 |
| **SR → BA** | 双链 | 注意区分两条链的 BA 实体差异：平台链的 IPO 处理实体为引擎组件，业务链的 IPO 处理实体为业务组件（§2.3.2~§2.3.3） |
| **BA → SysReq** | 双链，各自独立展开 | 将 BA IPO 去重后按系统功能分类重组——平台链按引擎功能模块划分，业务链按业务功能模块划分 |
| **SysReq → PA** | 双链，各自独立展开 | 按 PA 节点定义原则（§2.4）进行构件划分：平台链产出引擎构件，业务链产出配置定义构件 |

> 完整五层结构定义见[通用准则 §四](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#四系统设计链路)，三层核心规则见[通用准则 §三](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#三文档结对设计)。

### 3.2 文档结对设计（通用 §三）

EOS 的系统设计过程包含四个依次衔接的文档结对：

| 序号 | 结对 | 在 EOS 中的映射 |
|------|------|---------------|
| 1 | OR 条目级需求 ↔ SR 架构定义 | EOS 原始需求 → 相关方需求架构 |
| 2 | SR 详细定义末级 ↔ BA 架构（IPO） | 相关方需求 → 业务组件（IPO清单） |
| 3 | BA IPO（去重后）↔ SysReq 架构定义 | IPO → 系统功能模块 |
| 4 | SysReq 详细定义末级 ↔ PA 架构定义 | 系统功能 → 软件构件 |

**结对设计原则**：上层详细定义与下层架构定义应同步设计而非串行执行——先快速约定下层架构的结构骨架，再并行填充内容。

> 上述 4 对文档结对适用于**每一条设计链**——平台设计链和业务设计链各有一套 OR/SR/BA/SysReq/PA 文档，共 8 份核心产品数据文档。两条链的文档结构相同，内容实体不同。

> 需求分解/分配/分析三个核心设计活动的完整操作规则见[通用准则 §三](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#三文档结对设计)。编号规则见[通用准则 §3.4](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#34-编号规则)。

### 3.3 组织资产优先（通用 §2.4）

EOS 各层的架构节点不是凭空产生的，而是基于组织现有资产按 **复用 → 改进 → 新增** 优先级生成。

在 EOS 语境下，各层的资产形态：

| 层级 | 资产形态 |
|------|---------|
| **SR 层** | 需求分类体系、领域知识结构、行业标准目录、法律法规框架 |
| **BA 层** | 已有业务过程、岗位职责分工、跨系统交互流程 |
| **SysReq 层** | 系统边界划分、接口协议、技术域约定、技术标准规范 |
| **PA 层** | 已有软件模块、平台服务、可复用组件、中间件选型 |

> 资产优先级决策规则及操作细则见[通用准则 §2.4](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#24-基于资产定义架构)。

### 3.4 系统设计过程（通用 §五）

EOS 的系统设计遵循通用准则定义的**六步法**：

| 步骤 | 名称 | 在 EOS 中的映射 |
|------|------|---------------|
| Step 1 | 原始需求分析及相关方需求架构定义 | 收集 EOS 原始需求 → 按三种来源分流（§2.3.1）至对应设计链 → 建立该链的 SR 架构 |
| Step 2 | 相关方需求详细定义及业务架构定义 | 展开 SR 详细定义，识别 EOS 业务活动为 IPO |
| Step 3 | 系统需求功能部分架构定义 | BA IPO 去重后按 EOS 功能模块组织 |
| Step 4 | 非功能需求详细定义及系统需求非功能架构 | SR-NFR 平行路径，不经过 BA；两条链各自的 NFR 定义在各设计链内独立展开 |
| Step 5 | 系统需求详细定义及产品架构定义 | 按 PA 节点定义原则划分 EOS 软件构件 |
| Step 6 | 双向追溯验证 | 验证 OR → SR → BA → SysReq → PA 完整追溯 |

> 六步法的完整定义、每步的检查要点和验收标准见[通用准则 §五](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#五系统设计过程)。

### 3.5 验收标准与检查清单（通用 §六、§七）

通用准则定义了六个维度的质量验收标准（§六）和对应的逐项检查方法（§七），是 EOS 系统设计质量评估的通用框架。**本文档 §五 仅补充 EOS 特有的验收项，通用项不再重复**。

> 完整验收标准见[通用准则 §六](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#六设计验收标准)，完整检查清单见[通用准则 §七](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#七设计检查清单)。

## 四、各层设计指引

本章为 EOS 的五层设计提供场景化指导。每层聚焦"在 EOS 语境下如何进行设计活动"，并区分**平台设计链**和**业务设计链**的不同关注点。通用方法步骤不再复述。

### 4.1 第1层 OR：原始需求及分析

**EOS 的 OR 特征**：EOS 的原始需求主要来自以下渠道：
- 运营管理部门提出的运营体系建设需求
- 业务部门提出的流程优化和改进需求
- 信息系统部门提出的系统升级和集成需求
- 合规和监管要求触发的适配需求

**需求分流**：原始需求到达后，首先按 §2.3.1 的三种来源分类，分流至对应设计链：

| 需求类型 | 分流目标 | 后续设计链 |
|---------|---------|-----------|
| 引擎开发需求 | 平台设计链 | 按 §2.3.2 的五层结构展开 |
| 业务配置需求 | 业务设计链 | 按 §2.3.3 的五层结构展开 |
| CAX 工具需求 | 分流至外部 | 不纳入平台设计链，CAX 引擎仅作集成桥接 |

**设计指引**：

1. **收集**阶段关注企业运营体系的全覆盖——市场、研发、生产、售后、管理等各业务域的需求覆盖面
2. **分类分流**是 EOS 特有的 OR 处理步骤——收集到的原始需求先按上述三种类型分类，分流至对应设计链后各自展开
3. **初步整理**时使用四种方法（血缘归并、角色聚合、流程串联、父子补全）对同一条设计链内的需求进行归并
4. **规范化**时确认每条需求都包含"需求描述 + 性能指标"（不可衡量则不成为需求）
5. **分解到末级**时遵循 1:1 约束——每条子需求须能分配到该设计链 SR 的唯一一个架构末级节点

**输出**：原始需求详细定义（末级条目清单，按设计链分别组织）

> 通用方法：OR 预处理（碎片整理/规范化/冲突检测）见[通用准则 §5.2 阶段一](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#52-第1步or--sr)。

### 4.2 第2层 SR：相关方需求

**SR 架构分类**：EOS 的两条设计链各有独立的 SR 架构。

**业务链 SR（功能部分）**按业务域组织：

```
业务链 SR（功能部分）
├── 市场业务域
├── 研发业务域
├── 生产业务域
├── 售后业务域
└── 管理业务域
```

**平台链 SR（功能部分）**按引擎类型组织：

```
平台链 SR（功能部分）
├── 基础引擎类（表单引擎、标签页引擎、菜单引擎等）
├── 流程引擎类（流程引擎、工单引擎、进展引擎等）
├── 管理引擎类（项目引擎、计划引擎、指标引擎等）
├── 工程引擎类（xBOM引擎、版本管理引擎、MRP运算引擎等）
└── 集成引擎类（CAX 引擎）
```

**非功能部分**两条链共用相同的分类体系，按[通用准则附录B](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#附录b-非功能需求分类体系)的三大类（质量特性/环境适应性/可实现性）组织，但各链的 NFR 指标独立定义。

**设计指引**：

1. 功能需求和非功能需求分别标识（`SR-F-` / `SR-NFR-`）
2. 每条 SR 架构末级节点承接 2-5 条 OR 末级条目（N:1），检查合并的合理性
3. 详细定义分解到每条末级条目只需映射到 BA 的一个 IPO
4. 两条链的 SR 属于不同编号空间，建议分别管理（如 `SR-PF-` 标识平台链功能需求、`SR-BF-` 标识业务链功能需求）

> 通用方法：SR 详细定义分解和 IPO 映射见[通用准则 §5.3](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#53-第2步sr功能--ba)。

### 4.3 第3层 BA：业务架构

**EOS 的 BA 层级结构**：

| 层级 | 名称 | 说明 |
|------|------|------|
| 0级 | 架构根节点 | 运营体系 |
| 1级 | 业务域 | 市场、研发、生产、售后、管理等 |
| 2级 | 端到端项目业务 | 从需求提出到需求满足的端到端项目业务 |
| 3级 | 项目业务对象的架构节点类型端到端业务 | 由产品架构节点类型推导得出 |
| 4级 | 工作包（WBS） | 业务对象的工作分解结构 |
| **5级** | **业务组件（承接层）** | **使用 IPO 模型完整定义** |

**BA 的本质**：EOS 的业务架构是 EOS 的运行概念（Operational Concept）——描述用户如何使用 EOS 产品开展业务。BA 中的每个 IPO 描述的是 EOS 上的一个业务处理步骤。

**两条链的 BA 实体差异**：

| 设计链 | BA 实体含义 | 示例 |
|--------|-----------|------|
| **平台链** | IPO 的处理实体为**引擎组件**，描述引擎内部处理逻辑 | 流程引擎 IPO：接收流程定义 → 解析流程模型 → 执行流程实例 |
| **业务链** | IPO 的处理实体为**业务组件**，描述业务活动流程 | 销售订单 IPO：接收客户订单 → 校验/分配/计算 → 确认订单 |

**IPO 识别方法**：从 SR 详细定义集合中识别业务活动边界，以"系统接收→转换→产出"为框架描述每个 IPO。例如，业务链中从 SR 的"销售订单处理"需求推导出 BA IPO：

```
BA-001: 销售订单处理（IPO名称）
├─ 系统接收：客户订单信息（含产品、数量、交期、价格）
├─ 系统转换：订单校验 → 库存分配 → 价格计算 → 订单确认
└─ 系统产出：已确认的销售订单（含交期承诺）
```

**IPO 去重**：同一 IPO 可能被多条 SR 详细定义映射——将重复出现的 IPO 归并为唯一条目，形成去重后的 IPO 清单。

> 通用方法：BA 的独立存在特性和 IPO 映射规则见[通用准则 §5.3](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#53-第2步sr功能--ba)。

### 4.4 第4层 SysReq：系统需求

**SysReq 的性质**：EOS 作为软件类产品，其系统需求描述的是**系统的信息处理功能**——每项 SysReq 是分配给某个 PA 构件的一条功能需求。

**SysReq 架构层级**（功能需求 0-9 级）：

```
5级：组件实例 ←── 承接 BA 5级（N:1，架构末级）
6级：功能模块（Tab/页面）
7级：系统用例（按钮/菜单）
8级：用例场景（正常/异常/替代）
9级：场景活动 ──→ 分配到 PA 末级（1:1，详细定义末级）
```

**非功能需求**使用**0-6级**平行架构：根节点 → 需求类别（三大类）→ 需求类型 → 需求子类型 → 需求细分 → **5级承接层（架构末级）** → 6级验证标准。

**双重来源**：

| 来源 | 说明 | 处理方式 |
|------|------|---------|
| **功能需求** | BA IPO 去重清单 | IPO 去重后按功能模块组织，每项 IPO 对应一条 SysReq 功能条目 |
| **非功能需求** | SR-NFR（平行路径，不经过 BA） | 按质量特性/环境适应性/可实现性三大类组织，形成 0-6 级架构 |

**业务链 SysReq 的特殊性**：业务链的 SysReq 不描述软件功能，而是描述**配置项的处理逻辑**——每条 SysReq 对应一个配置产出物（表单定义、台账定义、流程定义等）的功能要求。平台链的 SysReq 描述引擎的软件处理功能，二者本质不同。

**平台链 SysReq 功能部分示例**：

| PA 构件类型 | 所承接的 SysReq 条目 |
|------------|---------------------|
| 前端组件 | 应接收用户输入的操作指令，经过界面交互逻辑处理（表单校验/状态更新/路由跳转），生成对应的界面展示或请求数据 |
| 后端组件 | 应接收 API 请求和业务数据，经过业务规则引擎处理（校验/计算/编排/持久化），生成处理结果或响应数据 |
| 数据层 | 应接收数据读写请求，经过数据存取处理（查询/写入/事务/缓存），生成查询结果或写入确认 |
| 基础设施 | 应接收系统运行监控请求，经过监控指标处理（采集/聚合/告警/日志），生成监控视图或告警通知 |

**设计指引**：

1. 功能部分从 BA IPO 去重清单出发，每项 IPO 编写一条 SysReq
2. 平台链的每条 SysReq 对应于一个 PA 构件类型；业务链的每条 SysReq 对应于一个配置产出物
3. 非功能部分直接从 SR-NFR 承接，不经过 BA 层

> 通用方法：SysReq 的分解过程和功能/非功能分配路径见[通用准则 §5.4~§5.5](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#54-第3步ba--sysreq功能架构)。

### 4.5 第5层 PA：产品架构

**EOS 的构件分类体系**（软件类产品的 PA 构件）：

| 类型 | 子类型 | 说明 |
|------|--------|------|
| **前端组件** | 页面组件、功能组件、交互组件 | 用户界面 |
| **后端组件** | API服务、业务引擎、数据处理、消息处理 | 业务逻辑 |
| **数据层** | 关系型数据库、NoSQL、文件存储、消息队列 | 数据存储 |
| **基础设施** | API网关、认证授权、监控告警、日志管理、CI/CD | 支撑能力 |

**构件划分原则**：每个构件应有明确的职责边界。对 EOS（软件产品），粒度标准为"一个团队可在 1-2 周内完成开发"且"可独立部署和测试"。

**SysReq → PA 的映射**：每个 PA 架构节点承接若干 SysReq 条目。对于 EOS，这些 SysReq 条目描述系统的信息处理功能（见 §4.4）。

**PA 节点定义方法**：

1. 从 SysReq 9 级场景活动出发，按 §2.4 的 PA 节点定义原则判断哪些活动应合并为同一构件
2. 按复用→改进→新增的资产优先顺序（通用准则 §2.4）处理已有构件
3. 每个构件标注四要素：定义、所映射的 SysReq 末级、符合性分析、实现策略

**举例**：EOS 设计线 PA 构件的划分（四类各一个示例节点）：

```
PA-001: 客户管理页面（前端组件）
├─ 定义：客户信息的增删改查界面
├─ 交付物形态：前端页面组件
├─ 承接的 SysReq 末级：SysReq-001.01 ~ SysReq-001.03
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：基于 React 实现，复用现有 UI 组件库

PA-002: 订单服务（后端组件）
├─ 定义：订单创建、修改、查询、状态管理的业务逻辑
├─ 交付物形态：API 服务
├─ 承接的 SysReq 末级：SysReq-002.01 ~ SysReq-002.04
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：微服务架构，与库存服务通过消息队列集成

PA-003: 生产数据库（数据层）
├─ 定义：生产订单、BOM、工艺路线等核心数据的持久化存储
├─ 交付物形态：关系型数据库实例
├─ 承接的 SysReq 末级：SysReq-003.01 ~ SysReq-003.02
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：PostgreSQL，按业务域分库设计

PA-004: API 网关（基础设施）
├─ 定义：统一 API 入口，提供路由、认证、限流等功能
├─ 交付物形态：网关服务
├─ 承接的 SysReq 末级：SysReq-004.01 ~ SysReq-004.02
├─ 符合性分析：[如何满足所承接的需求]
└─ 实现策略：基于 Kong 实现，与认证服务集成
```

> 通用方法：PA 的完整设计和 SysReq→PA 的映射规则见[通用准则 §5.6](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#56-第5步sysreq--pa)。

**业务链 PA 构件类型**：业务链的 PA 不是软件组件，而是**使用引擎配置出的结构化配置定义**。

| 类型 | 说明 | 示例 |
|------|------|------|
| **表单定义** | 数据录入和展示的界面配置 | 设备台账表单、订单表单 |
| **标签页定义** | 多视图切换配置 | 设备详情标签页 |
| **台账定义** | 数据列表和查询配置 | 设备台账、物料台账 |
| **菜单定义** | 导航结构配置 | 主菜单、子菜单 |
| **流程定义** | 业务流程步骤和规则配置 | 审批流程、工单流程 |
| **指标定义** | 绩效指标计算配置 | KPI 看板指标 |

> 两条链 PA 的关系：平台链 PA（引擎）是业务链 PA（配置定义）的能力底座——每个配置定义运行在对应引擎之上（表单定义→表单引擎，台账定义→台账引擎）。两者共同构成 EOS 的完整产品架构（§2.3.4）。

## 五、EOS 验收标准

以下仅列出 EOS 特有的验收项。通用验收标准（架构定义完整率、详细定义完成率、1:1 约束满足率、映射覆盖率、追溯完整性等）见[通用准则 §六](../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#六设计验收标准)。

### 5.1 平台链 PA 验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **构件分类完整率** | 100% | PA 构件按四类（前端/后端/数据/基础设施）完整划分，无遗漏 |
| **构件交付物明确率** | 100% | 每个构件标注了交付物形态（页面组件/API服务/数据库实例/网关服务等） |
| **构件独立可验收率** | 100% | 每个构件可独立部署和测试，不依赖其他构件的交付物 |

### 5.2 业务链 PA 验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **配置定义类型覆盖** | 100% | 业务链 PA 覆盖业务所需的所有配置类型（表单/台账/流程/菜单/标签页/指标等），无遗漏 |
| **配置与引擎匹配率** | 100% | 每个配置定义使用对应引擎的能力实现，不超出引擎能力边界 |
| **配置定义可执行率** | 100% | 配置定义在对应引擎上可正确运行 |
| **配置项独立性** | 100% | 每个配置定义为独立单元，不依赖其他配置定义的内部实现 |

### 5.3 层级结构验收

> 以下层级验收适用于两条设计链各自独立展开——平台链和业务链各有自己的 BA/SysReq 层级结构，须分别验收。

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **BA 层级完整率** | 100% | BA 从 0 级到 5 级（业务组件/承接层）的完整树形结构 |
| **SysReq 功能层级完整率** | 100% | SysReq 功能部分从 5 级到 9 级（场景活动）的完整分解链路 |
| **SysReq NFR 层级完整率** | 100% | SysReq 非功能部分从 0 级到 6 级（验证标准）的完整分解链路 |

### 5.4 多产品关系验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **三层工作层面标记** | 100% | 每个 PA 构件标注服务于第几层工作层面 |
| **端到端框架应用** | 通过评审 | BA 2级端到端项目业务按三类客户维度（主价值链/管理监管/支持服务）正确划分 |
| **域间接口记录率** | 100% | 跨域协作关系的接口契约已记录在案 |

## 六、常见错误、FAQ 与规范关系

### 6.1 常见错误

**错误1：混淆架构末级节点和详细定义末级节点的角色**

```
错误做法：
在相关方需求层，将"需求收集"直接作为 SR 架构末级节点，
但"需求收集"其实是 SR 某个功能分类下的一个详细定义条目。

正确做法：
架构末级节点：SR-F-001（需求规范化功能）—— 完整的业务能力
详细定义末级：SR-F-001.01（需求收集）—— 功能下的具体环节
```

**错误2：同步设计退化为串行设计**

```
错误做法：
先完成第2层全部详细定义，再开始第3层架构定义
→ 第2层详细定义完成后才发现 BA 没有对应的 IPO 可承接

正确做法：
第一轮：快速约定各层架构骨架（占位符 + 映射关系）
第二轮：并行填充各层详细内容
```

**错误3：验收标准和检查清单混为一谈**

```
错误做法：
验收标准和检查清单放在一起，列出的全是"是否已检查"类项，
没有量化指标衡量设计质量。

正确做法：
验收标准（量化指标）："映射覆盖率100%，1:1约束满足率100%"
检查清单（操作验证）："[ ] 检查所有需求是否已分解到末级"
```

**错误4：分析活动被遗漏或延后**

```
错误做法：
需求分解、分配和符合性分析在全部设计完成后才做，或者根本没做。

正确做法：
分析活动贯穿设计全过程——每完成一次分配立即执行对应的符合性分析。
```

**错误5：忽略资产优先原则，动辄新增架构节点**

```
错误做法：
新需求到达时，直接在架构中新增架构末级节点来承接，
而不检查现有节点是否能复用或改进。

正确做法：
按复用→改进→新增的优先级依次检查现有节点。
```

**错误6：PA 分解跨越业务边界**

```
错误做法：
将信息处理模式不同的两个业务活动强行合并到同一构件，
或将 NFR Profile 冲突的需求分配到同一构件。

正确做法：
按三因素决策矩阵（§2.4.5）判断构件边界——信息模型不同或 NFR 冲突的需求应分离。
```

### 6.2 FAQ

**Q1：本文档与各层规范（02-06）是什么关系？**

本文档是 EOS 系统设计的**总纲**，定义"为什么这样做"和"整体结构是什么"。各层规范定义"具体怎么做"。在阅读时，建议先读本文档了解全局，再查阅特定层规范。

**Q2：六步法中的步骤是否可以并行执行？**

可以。第2步（功能分支）和第4步（非功能分支）是独立并行的。第2步完成后可立即开始第3步和第5步的功能部分，无需等待第4步。两阶段工作流程是实现并行化的具体方法。

**Q3：通用准则、本准则（EOS 系统设计）、EOS 流水线准则三者的关系？**

三者构成"方法→二次实例→一次实例"的递进链：
- **通用准则**（`00-general/`）：产品无关的方法论核心
- **EOS 流水线准则**（`10-meta-pipeline/.../01-eos-pipeline-system-design-standards.md`）：通用方法论在 EOS 流水线上的实例化
- **本准则**（`20-eos-design/.../01-eos-system-design-standards.md`）：通用方法论在 EOS 产品上的实例化（参考 EOS 流水线准则的结构，内容按 EOS 软件产品定制）

三者共享同一套方法论核心（五层结构、核心规则），各有自己的产品特有规则。

> **理解要点**：本准则与 EOS 流水线准则在方法论层面引用的是同一份通用准则（§一~§七）。之所以参考后者而不是前者的结构，是因为**产品同构性**——本准则和 EOS 流水线准则都是"将通用方法论实例化到具体产品"的产物，只不过实例化目标不同（EOS vs EOS 流水线）。

**Q4：当详细定义无法 1:1 映射到下层架构时，应该调整哪一方？**

先检查详细定义是否还能进一步分解。如果详细定义已到原子级仍然无法 1:1，则说明下层架构设计不合理，需要补充或调整下层架构末级节点。经验法则：先怀疑分解不充分，再怀疑架构不完整。

**Q5：两阶段工作流程和同步设计是什么关系？**

两阶段工作流程是**实现方式**，同步设计是**设计原则**。同步设计要求上下游在设计时对齐，两阶段工作流程提供了具体的操作方法：第一阶段快速建立骨架和映射（对齐结构），第二阶段并行填充内容（对齐细节）。

**Q6：本准则的版本更新策略是什么？**

v2.x 系列定位为"内嵌通用方法论"的独立文档版本，v3.0 及以后定位为"基于通用方法论引用"的专有准则版本。版本号变动意味着准则的结构或定位发生了变化，使用者应关注版本文档头中的定位说明。

**Q7：EOS 的端到端业务框架如何在 BA 层落地？**

端到端项目业务（BA 2级）按三类客户维度（主价值链/管理监管/支持服务）划分业务边界。每个端到端项目业务下，价值业务由被实现对象的系统架构节点类型推导（§2.2.4），管理业务独立于价值业务之外单独补充。跨域协作通过需求—响应契约实现，而非跨域吞并业务。
