## 一、核心概念

### 1.1 本层的定义

**业务架构（Business Architecture）** 是系统的运行概念（Operational Concept）——即**用户使用系统开展业务的过程说明**。

这意味着：
- 从**用户视角**描述如何使用系统，而非系统如何实现
- 体现**用户的工作价值**，而非技术实现细节
- 为**系统需求设计**提供功能和流程指导

### 1.2 在5层结构中的位置

```
相关方需求（第2层）
  ├─ 需求描述 + 性能指标
  └─ 功能部分 ↓ N:1承接支持

业务架构（第3层） ★ 本规范
  ├─ 业务组件（承接层）
  ├─ Input + Process + Output
  └─ 性能指标（继承自相关方需求）
      ↓ IPO去重后分配到系统需求（1:1分配约束）

系统需求（第4层）
  ├─ 系统功能需求架构（5级节点）
  └─ 性能指标（继承自业务架构）
```

| 维度 | 说明 |
|------|------|
| **文档类型** | 方案文档 — **仅架构定义**（无详细定义） |
| **承接对象** | 相关方需求详细定义末级（N:1承接支持） |
| **分配目标** | 系统需求架构5级节点（1:1分配约束，去重后） |
| **核心单位** | IPO（Input-Process-Output） |

### 1.3 本层特有的关键概念

**独立存在**：业务架构仅包含架构定义（IPO清单），不参与"详细定义的双重身份"递进链条。其架构末级节点自身承担"需求"角色，直接映射到系统需求架构末级节点。

**5级承接层**：业务架构的5级节点（业务组件）是承接相关方需求的关键层级，也是末级，不再向下分解。

**IPO模型**：每个业务组件用 Input（输入）、Process（处理）、Output（输出）三个维度来定义。

**去重机制**：同一IPO可能被多个业务流程引用。在映射到系统需求前，需完成IPO去重。

### 1.4 本层的设计活动

三个活动遵循准则 [§1.5.1 需求的分解、分配与分析](./01-eos-system-design-standards.md#151-需求的分解分配与分析)（含语义分解四条件、指标分解规则、1:1约束细则等完整规范），本层的具体要点如下。

> **层特有说明**：业务架构无详细定义（IPO 自身承担"需求"角色），**需求分解及分析不适用**。

**需求分解及分析**：不适用——业务架构只有架构定义（IPO 清单），IPO 本身即末级，无需进一步分解。

**需求分配及分析**——本层要点：
- 将 BA 架构末级节点（IPO）去重后分配到 SysReq 架构末级节点，检查 1:1 约束、全覆盖、N:1 合理性
- 去重是分配的前置步骤——同一 IPO 被多次引用时先去重再映射，不违反 1:1 约束
- 每个业务组件承接 2-5 条 SR 详细定义末级为合理范围

**架构承接需求分析**——本层要点：
- 每个 IPO 自检是否充分满足所承接的 SR 详细定义末级——输入/处理/输出三维定义是否完整覆盖 SR 语义
- 性能指标不低于 SR 的要求，IPO 层面已明确指标的测量条件

## 二、架构定义规范

### 2.1 架构末级节点（5级承接层）

业务架构的末级节点是**5级业务组件**，是承接相关方需求的关键层级。

**关键特征**：
- **末级约束**：5级是最底层，不再向下分解
- **完整定义**：使用IPO模型完整定义每个业务组件
- **独立操作**：每个业务组件代表一个可独立进行的业务操作
- **可映射性**：每个业务组件可清晰映射到一条或多条相关方需求（N:1承接支持）
- **可实现性**：每个业务组件都有清晰的输入、处理、输出

**按实现方式的分类**：

| 类型 | 说明 | 示例 |
|------|------|------|
| **配置方式** | 通过流程/工单/计划/指标/表单引擎配置实现 | 考核计划审批流程、评分表配置、KPI指标设定 |
| **开发方式** | 需要代码开发，提供配置引擎的基础功能 | 考核引擎、评分计算服务、报表系统 |

### 2.2 层级结构

```
0级：架构根节点 — 整个系统的顶层根节点（唯一）
1级：业务域 — 按业务领域划分（3-10个）
2级：端到端项目 — 跨越多个业务域的项目或流程
3级：项目业务对象架构 — 业务对象和其架构
4级：工作包（WBS） — 业务对象的工作分解结构
5级：业务组件（承接层）— ★ 末级，不再分解
```

**设计要点**：
- 1-2 级定义业务结构和项目范围
- 3-4 级组织业务对象和工作单元
- **5 级是需求承接的关键层级**，使用 IPO 模型完整定义

### 2.3 架构定义的文档结构

```
BA-001: [IPO名称/业务组件名称]（5级·架构末级节点）
├─ 节点类型：[配置/开发]
├─ 定义：[业务组件的定义]
├─ 承接的相关方需求详细定义末级：
│  ├─ SR-F-003.01
│  ├─ SR-F-003.02
│  └─ SR-F-004.01
├─ 符合性分析：[如何满足所承接的需求]
├─ 性能指标：[继承自相关方需求]
├─ Input：[输入的具体形式和来源]
├─ Process：[处理的步骤和规则]
├─ Output：[输出的具体形式和去向]
├─ 参与角色：[执行此业务的角色]
└─ 为下层提供的设计指导：[对系统需求的指导]
```

**颗粒度对比**：

```
✅ 合适的粒度（应分为独立业务组件）：
  BA-001: 考核计划管理
  BA-002: 评分表配置
  BA-003: 绩效报告生成

❌ 粒度过粗：
  BA-001: 考核管理（涵盖多个完全不同的业务，太宽泛）

❌ 粒度过细：
  BA-001: 考核周期选择（只是一个操作，不是独立业务组件）
```

### 2.4 第二部分：需求映射视图（映射关系表）

业务架构的架构定义除树形结构外，还应包含需求映射视图——展示所承接的相关方需求详细定义末级与业务架构末级节点的对应关系。

**结构**：
```
相关方需求详细定义末级 | 承接的业务组件(BA) | 超链接
SR-F-001.01           | BA-001             | [跳转到第一部分]
SR-F-001.02           | BA-001             | [跳转到第一部分]
SR-F-002.01           | BA-002             | [跳转到第一部分]
```

**作用**：
- 清晰地展示相关方需求与业务组件的对应关系
- 支持从需求视角快速定位到架构设计
- 便于验证 1:1 分配约束和 N:1 承接支持的完整性

**导航方式**：
- 单击此处业务组件 ID，跳转到第一部分的对应架构末级节点
- 支持双向追溯：相关方需求 ↔ 业务组件

## 三、IPO模型

### 3.1 定义

IPO模型是业务架构的基本设计单位，用三个维度完整定义一个业务组件：

| 维度 | 定义 | 内容 |
|------|------|------|
| **Input（输入）** | 业务组件接收的输入 | 形式、数据结构、数据来源、前置条件 |
| **Process（处理）** | 对输入进行的处理 | 处理规则、算法、步骤、决策点、约束 |
| **Output（输出）** | 业务组件生成的输出 | 形式、数据结构、验收标准、后续使用 |

### 3.2 示例（EOS 运营考核场景）

```
BA-001: 考核计划创建
├─ Input：考核周期、参与部门、考核模板
├─ Process：
│  ├─ 选择考核周期（月度/季度/年度）
│  ├─ 确定参与考核的部门和人员范围
│  ├─ 选择适用的评分表模板
│  └─ 发布考核计划并通知相关人员
└─ Output：已发布的考核计划、系统通知

BA-002: 评分录入
├─ Input：考核计划、评分表、员工名单
├─ Process：
│  ├─ 部门主管登录评分入口
│  ├─ 系统按考核计划展示待评分员工列表
│  ├─ 主管逐项选择评分并填写评语
│  └─ 提交评分，系统校验完整性后保存
└─ Output：已提交的评分记录、确认状态

BA-003: 绩效报告生成
├─ Input：已提交的评分数据、考核计划
├─ Process：
│  ├─ 汇总各部门评分数据
│  ├─ 按维度计算 KPI 得分
│  ├─ 生成绩效报告（含排名、分布、趋势）
│  └─ 推送给指定人员审阅
└─ Output：绩效报告（PDF/在线预览）
```

### 3.3 颗粒度标准

IPO的体量取决于**期望的输出物颗粒度**：

```
期望输出是考核报告 → 一个IPO对应一份报告（或可由单人完成的报告部分）
期望输出是评分流程 → 一个IPO对应一个完整的评分操作
期望输出是数据对象 → 一个IPO对应一个考核对象的生命周期
```

**判断标准**：能否分配到系统需求的唯一一个架构末级节点（5级节点）？

```
✅ 合适的IPO：
  IPO-001: 评分录入（Input: 评分表+员工名单 → Process: 展示+录入+提交 → Output: 评分记录）
  → Process 步骤清晰可执行，输出物明确

❌ 过于抽象：
  IPO-001: Process = "进行考核"
  → 无法知道具体怎么处理，无法指导系统需求设计

❌ 过于细化：
  IPO-001: 选择评分等级
  → 这只是评分录入中的一个步骤，不是一个独立业务操作
```

### 3.4 节点定义标准

每个业务架构末级节点应包含以下属性：

| 属性 | 说明 | 示例 |
|------|------|------|
| **节点ID** | 唯一标识 | BA-001 |
| **节点名称** | 业务组件名称 | 考核计划创建 |
| **节点类型** | 配置/开发 | 流程引擎配置 |
| **节点定义** | 业务功能的简要描述 | 处理考核计划的全生命周期管理 |
| **承接的相关方需求** | 承接了哪些详细定义末级 | SR-F-001.01, SR-F-001.02 |
| **符合性分析** | 如何满足所承接的需求 | 每个需求都有对应的业务操作说明 |
| **性能指标** | 继承自相关方需求的性能要求 | 考核计划发布 < 3s，支持 1000+ 部门 |
| **Input** | 输入的具体形式和来源 | 考核周期、部门范围 |
| **Process** | 处理的步骤和规则 | 1. 选择周期... 2. 确定范围... |
| **Output** | 输出的具体形式和去向 | 已发布的考核计划、通知相关人 |
| **参与角色** | 执行此业务的角色 | 运营管理员、部门主管 |
| **设计指导** | 对系统实现的指导 | 系统应提供考核计划配置界面... |

## 四、设计方法

### 4.1 三阶段工作流程

> **与准则的关系**：本节描述的三阶段是准则 §1.5.3"两阶段工作流程"在业务架构层的具体应用——第一阶段对应准则的快速约定结构骨架，第二、三阶段对应准则的并行填充内容+验证。因业务架构是唯一只有架构定义、无详细定义的层，其工作流程额外强化了去重和冻结环节。

```
第一阶段：占位符创建（快速，1-2小时）
  1. 扫描相关方需求功能部分的详细定义末级
  2. 快速识别并创建业务组件占位符（5级节点名称）
  3. 建立相关方需求 → 业务组件的骨架映射
  4. 同时为系统需求创建结构骨架占位符
  输出：业务组件占位符清单 + 骨架映射关系

第二阶段：详细填充（并行，2-3天）
  1. 详细定义每个业务组件的IPO
  2. 进行业务组件去重（复用→改进→新增）
  3. 填充完整的符合性分析和设计指导
  4. 同步建立系统需求架构的完整结构
  输出：完整的业务架构定义文档

第三阶段：验证和冻结（最终确认）
  1. 验证1:1分配约束
  2. 验证N:1承接支持的合理性
  3. 检查IPO定义的完整性
  4. 冻结业务架构
  输出：已验证的业务架构文档
```

### 4.2 与相关方需求的映射

```
映射的源头：相关方需求详细定义末级节点（SR-F-XXX.X）
映射的目标：业务架构末级节点（5级业务组件，BA-XXX）
映射的关系：N:1承接支持 — 一个业务组件可承接多条相关方需求详细定义末级
```

**符合性分析要求**（以 EOS 考核场景为例）：

```
BA-001: 考核计划管理
├─ 承接的相关方需求：SR-F-001.01, SR-F-001.02, SR-F-002.01
├─ 符合性分析：
│  ├─ SR-F-001.01（考核计划创建）：通过考核计划创建流程满足
│  ├─ SR-F-001.02（考核计划发布）：通过考核计划发布功能满足
│  └─ SR-F-002.01（考核对象选择）：通过考核范围配置功能满足
└─ 设计指导：系统应提供考核计划全生命周期管理功能
```

### 4.3 去重机制

**优先级**：复用 > 改进 > 新增

```
第1步：首先检查
  → 现有业务架构中是否已有相同或相似的IPO

第2步：复用优先
  → 如果存在完全匹配的IPO，直接复用（记录映射来源）

第3步：改进次之
  → 如果现有IPO改进后可满足需求，则改进后复用（标注变更记录）

第4步：新增最后
  → 确实需要新的IPO才新增
```

**去重对比**（以 EOS 考核场景为例）：

```
✅ 正确做法：
  流程A需要"绩效评分录入"，流程B也需要"绩效评分录入"
  → 复用同一个IPO（BA-002），两个流程都映射到BA-002
  → 好处：系统需求中只需实现一次"评分录入"

❌ 错误做法：
  流程A定义BA-002"评分录入"，流程B定义BA-102"评分录入"
  → 两个IPO功能相同但ID不同
  → 后果：系统需求中出现两个"评分录入"，导致重复开发
```

### 4.4 同步设计

在开展业务架构详细填充（第二阶段）的同时，应同步开展系统需求架构的骨架建设。

**原因**：
- 业务架构的IPO需要分配到系统需求架构的5级节点（去重后）
- 如果系统需求架构尚未建立，IPO就没有分配目标，容易造成返工

**实际操作**：

```
1. 在创建业务组件占位符时，同步创建系统需求5级节点占位符
2. 在填充IPO详细内容时，同步建立BA → SysReq的映射关系
3. 冻结映射关系后，并行进行内容深填
```

### 4.5 设计原则

**N:1承接支持**：一个业务组件可以承接多条相关方需求。承接 2-3 条最优，1 条需考虑合并，4-5 条可接受（需检查是否有过度聚合），6 条以上需考虑拆分。

**1:1分配约束**：每条相关方需求只分配到唯一的业务组件。如果一条需求需要分配到多个组件，说明分解不充分。

**设计指导原则**：每个业务组件应为系统需求提供清晰的设计指导，包括系统应提供什么功能、如何组织信息、如何验证业务规则。

**组织资产优先原则**：新增IPO前优先复用已有IPO，其次扩展/拆分/合并，最后新增。去重机制本身就是资产优先原则在BA层的直接应用——先检查是否已有相同或相似的IPO可复用，避免重复定义。详见[EOS系统设计准则 §3.3](01-eos-system-design-standards.md#33-组织资产优先原则)。

## 五、验收标准

### 5.1 需求映射完整性验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **映射覆盖率** | 100% | 所有相关方需求功能部分详细定义末级都映射到业务组件 |
| **1:1约束满足率** | 100% | 每条相关方需求只分配到一个业务组件 |
| **去重完成率** | 100% | 所有重复的业务组件都已识别和处理 |

### 5.2 IPO定义完整性验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **IPO定义清晰率** | 100% | 每个业务组件都有清晰的Input/Process/Output定义 |
| **符合性分析完整率** | 100% | 每个业务组件都说明如何满足所承接需求 |
| **设计指导完整率** | 100% | 每个业务组件都为系统需求提供明确的指导 |

### 5.3 层级结构完整性验收

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误1：业务组件定义过于宽泛**

```
错误的做法：
BA-001: 考核管理
├─ Process: 管理考核全流程
└─ 涵盖：计划管理、评分录入、报告生成、结果确认

问题：一个组件涵盖多个完全不同的业务操作
后果：系统需求设计时无法确定该分配什么功能，IPO定义无法聚焦
```

```
✅ 正确做法：
BA-001: 考核计划管理    → 聚焦考核计划创建和发布
BA-002: 评分录入        → 聚焦评分录入和提交
BA-003: 绩效报告生成    → 聚焦报告汇总和分析

说明：每个组件聚焦一个独立的业务操作
```

**❌ 错误2：一条相关方需求分配到多个业务组件**

```
错误的做法：
SR-F-003.01（评分录入）→ BA-002（评分录入）+ BA-003（报告生成）

问题：一条需求需要两个组件承接，违反1:1分配约束
后果：系统需求设计时无法确定评分录入功能到底归谁
```

```
✅ 正确做法：
方案A：将SR-F-003.01继续分解
  SR-F-003.01.01（评分录入-数据填写）→ BA-002
  SR-F-003.01.02（评分录入-报告生成）→ BA-003

方案B：重新设计业务组件，使一个组件能完整承接
  BA-004: 评分与报告（覆盖评分录入+报告生成）
```

**❌ 错误3：IPO定义过于抽象**

```
错误的做法：
BA-001: 考核计划管理
├─ Input: 考核数据
├─ Process: 管理考核计划
└─ Output: 管理结果

问题：Process = "管理考核计划" 没有说明任何处理逻辑
后果：系统需求设计者完全无法理解这个业务组件具体做什么
```

```
✅ 正确做法：
BA-001: 考核计划管理
├─ Input: 考核周期、部门范围、考核模板
├─ Process:
│  1. 选择考核周期类型（月度/季度/年度）
│  2. 确定参评部门和人员范围
│  3. 选择适用的评分表模板
│  4. 设定考核起止时间
│  5. 发布计划并系统通知
└─ Output: 已发布的考核计划、待办通知

说明：具体明确的操作步骤，系统需求设计者可直接据此设计功能
```

**❌ 错误4：忽视业务组件去重**

```
错误的做法：
流程A（月度考核）：BA-001"评分录入"
流程B（年度考核）：BA-101"评分录入"（与BA-001功能相同）

问题：两个流程各自定义了相同的IPO
后果：系统需求中出现两个"评分录入"，造成重复开发
```

```
✅ 正确做法：
识别到BA-001和BA-101功能相同，合并为一个IPO
→ BA-001"评分录入"同时承接流程A和流程B的相关方需求
→ 系统需求中"评分录入"只需实现一次
```

**❌ 错误5：忽略资产优先原则，重复定义相同IPO**

```
错误的做法：
在新增业务流中直接定义新的IPO，而不检查现有IPO清单中
是否已有相同或功能相似的IPO可复用。

问题：忽视已有IPO资产，导致BA层出现功能重复的组件
后果：SysReq层出现重复功能定义，后续产品架构冗余
```

```
✅ 正确做法：
在定义新IPO前，先全面扫描现有IPO清单。
按复用→扩展→新增的优先级选择处理方式。
说明：去重机制是资产优先的具体应用，减少后续系统需求的重复定义
```

### 7.2 最佳实践

**实践1：使用三阶段工作流程**：先占位符 → 再并行填充 → 最后验证冻结。避免一次性完成的瀑布式做法。

**实践2：充分利用去重机制**：在开始新增IPO前，先检查是否已有相同或相似的IPO可复用。优先复用，避免重复开发。

**实践3：设计指导具体化**：每个IPO的"设计指导"不应只是描述性文字，而应具体说明"系统应提供什么功能来支持这个业务过程"。

**实践4：保持双向追溯**：每条相关方需求都能追溯到业务组件，每个业务组件都能追溯到相关需求。

**实践5：与系统需求同步设计**：在填充IPO时同步创建系统需求架构的占位符，建立BA → SysReq的骨架映射。

## 八、FAQ

**Q1：业务架构为什么不需要详细定义？**

业务架构是从用户角度描述业务过程，IPO模型本身已充分定义了业务操作。系统需求的详细定义（如用例场景、活动步骤）是系统实现的细节，不是业务操作的一部分。

**Q2：如何判断业务组件的粒度是否合适？**

关键标准是1:1分配约束——这个业务组件是否能唯一对应系统需求的一个功能组件。如果一个业务组件需要分配到多个系统功能，说明粒度太大，需要进一步分解。

**Q3：业务组件去重是否必须完成？**

是的，完全必须。去重确保系统中不存在重复的业务操作。在映射到系统需求时，去重结果被作为系统功能需求的源头。去重不完整会直接导致系统功能重复定义。

**Q4：一个业务组件承接多少条相关方需求合适？**

没有硬性限制，但建议范围：
- 1 条：太少，考虑是否可与其他组件合并
- 2-3 条：最优，分组合理
- 4-5 条：可接受，但需检查是否有过度聚合
- 6 条以上：过多，强烈建议拆分

**Q5：业务架构和流程图的区别是什么？**

业务架构从系统设计角度描述用户使用系统的过程和组件，为系统需求提供指导。流程图描述具体的工作流程和操作步骤。业务架构的IPO可以理解为对流程的高度抽象——IPO关注"输入→处理→输出"的结构，而非每一步的细节。
