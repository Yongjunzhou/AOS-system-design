## 目标

从 BA 反向推导相关方需求，建立 SR 架构定义和详细定义。

## 上下文继承

```
来源：第4步产出
产出：SR 架构定义 + 详细定义（含 BA→SR 追溯映射表）
```

## 输入

| 输入 | 说明 |
|------|------|
| BA IPO 定义 | task-04 产出 |
| 现有相关方需求描述 | 仓库中已有的需求类文档 |
| 制品非功能特征 | 从PA及制品中识别的非功能特征（性能、安全、可用性） |

## 操作流程

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 分析活动：需求说明符合性分析 + 需求性能符合性分析 | 人类+AI | 两类分析：(1) 需求说明符合性——BA→SR 映射完整性分析，验证每个 BA IPO 是否对应到至少一个 SR-F 末级节点；(2) 需求性能符合性——相关方角色覆盖度分析，评估反向推导的 SR 是否覆盖所有相关方角色 |
| 2 | SR 功能部分详细定义推导 | AI | 为每个 SR 架构末级节点推导详细分解条目 |
| 3 | 非功能部分推导（SR-NFR） | AI | 从PA及制品中识别非功能特征，反推 SR-NFR，按 0-6 级独立体系组织。非功能需求不经过 BA 层，直接从PA及制品反推 |
| 4 | SR→OR 反向推导 | AI | 从 SR 末级反向还原原始需求描述，标注反向推导有效性评级（AAA/AA/A/B/C） |
| 5 | 结果审核 | 人类 | 确认推导的合理性 |

> **三条核心规则**：1:1分配约束（每个SR详细定义末级只能分配到唯一的 BA IPO）、N:1承接支持（一个 BA 架构节点承接 2-5 条 SR 需求）、双向追溯。在每一步中保持对这些规则的关注。

## 人类审核要点

- [ ] SR 推导是否准确反映了用户意图
- [ ] SR-NFR 是否有遗漏
- [ ] OR 还原是否合理
- [ ] BA→SR 映射完整性分析中发现的断链是否已修正
- [ ] 相关方角色是否完整——是否有缺失的角色尚未被 SR 覆盖
- [ ] 反向推导有效性评级（AAA/AA/A/B/C）是否合理

## 质量门禁

- [ ] SR 架构定义和详细定义完成
- [ ] BA → SR → OR 映射关系建立且完整
- [ ] 每个 SR 详细定义末级分配到唯一的 BA IPO（1:1 约束）
- [ ] SR-NFR 已从PA及制品反推完成，按 0-6 级组织，无遗漏
- [ ] SR 功能与非功能部分无交叉冲突
- [ ] 性能指标反推值需标注置信度分级（D-R021）
- [ ] 制品-文档差异已按四类型处理（D-R020）
- [ ] 运行 rule-conformance-review 进行符合性审查
- [ ] 已按审查修复流程处理发现的问题
