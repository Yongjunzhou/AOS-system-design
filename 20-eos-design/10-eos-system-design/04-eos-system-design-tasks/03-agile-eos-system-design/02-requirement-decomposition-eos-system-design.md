## 目标

将新需求分解到末级，验证 1:1 约束，输出末级需求清单。

## 上下文继承

```
从哪里来：
  - task-01 产出：新需求分析报告（分类 + 影响范围 + 优先级）
  - 参考基线：现有原始需求文档

去哪里：
  - 本步产出 → task-03（SR 影响分析）的新末级需求清单
```

## 输入

| 输入 | 说明 |
|------|------|
| 新需求分析报告 | task-01 产出 |
| 现有 OR 文档（基线） | 基线中的 OR 层 |

## 操作流程

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 规范化 + 分解到末级 | AI | 四项检查（完整性/清晰性/可测试性/独立性）、模糊术语替换为精确表述（按歧义分级处理：L1 可推测→补齐标注/L2 多选项→列出推荐/L3 不可解→暂停上报）、功能/非功能分离、原子性分解 |
| 2 | 1:1 约束验证 | AI | 检查每条末级是否只需一个 SR 节点承接 |
| 3 | 分析活动：冲突和重复检测 | AI | 需求说明符合性分析（新需求之间、新需求与基线需求的冲突/重复检测）+ 需求性能符合性分析（新需求性能指标是否明确量化） |
| 4 | 增量标注 | AI | 使用增量标记（`+`/`~`）标注新增/修改项 |
| 5 | 结果审核 | 人类 | 确认末级清单 |

## 人类审核要点

- [ ] 冲突检测的最终决策 — 需相关方确认
- [ ] 需求分解粒度是否与现有需求一致
- [ ] 增量标记是否准确
- [ ] 功能/非功能分离是否正确
- [ ] 分解停止准则是否满足（每条末级只需映射到 SR 架构的一个末级节点）

## 质量门禁

- [ ] 分析活动：冲突和重复检测已执行
- [ ] 新原始需求末级清单完整
- [ ] 功能/非功能分离完成
- [ ] 冲突/重复检测完成
- [ ] 运行 rule-conformance-review 进行符合性审查
- [ ] 按审查修复流程处理问题
