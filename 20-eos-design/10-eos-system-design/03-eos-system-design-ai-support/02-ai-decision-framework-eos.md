# AI 决策框架
**AI Decision Framework**

**文档版本**：v1.1
**创建日期**：2026-05-20
**用途**：为 AI 在 EOS 系统设计过程中进行设计判断提供决策规则和判断方法，支持 AI 深度参与而非仅机械执行

---

## 一、决策类型总览

系统设计过程中 AI 会遇到的决策可分为四类，对应不同的处理策略：

| 决策类型 | AI 权限 | 适用场景 |
|---------|---------|---------|
| **自动执行** | AI 直接执行，标记"AI推断" | 规则明确的机械性操作 |
| **提出建议** | AI 提出方案+依据，人工确认 | 有多个合理选项的设计决策 |
| **标记上报** | AI 记录事实，暂停等待 | 信息不足、有矛盾、需领域知识 |
| **仅作记录** | AI 不做判断，仅记录待确认 | 纯人工决策范围（优先级、技术选型） |

## 设计原则：组织资产优先

### 1. 核心优先级

新增上层需求需要分配到下层方案时，架构末级节点的变更遵循以下优先级（从高到低）：

```
第1优先 ── 复用（Reuse）
  检查：下游方案现有架构末级节点是否已有完整覆盖此需求的节点？
  → 是：直接映射到该节点，无需任何变更

第2优先 ── 改进（Improve）
  检查：没有现成节点完整覆盖，但通过调整可使现有节点承接？
  ├─ Extend（扩展）  — 扩展现有节点的定义和所映射需求列表
  ├─ Split（拆分）   — 现有节点职责过宽，拆分为多个子节点后分别承接
  └─ Merge（合并）   — 多个现有节点整合后共同承接

第3优先 ── 新增（Add）
  检查：复用和改进都不可行
  → 在适当父节点下创建新末级节点，建立映射

特殊情况 ── 废弃（Deprecate）
  上层需求取消，且无其他需求映射到此节点
  → 标记"无上游需求，已废弃"，保留节点（不物理删除）
```

**认知基础**：每个架构末级节点都是组织资产——承载了已投入的设计工作（四要素定义）、已建立的下游映射关系和已通过的验证记录。新增节点意味着后续的设计、维护和验证成本。优先复用和改进已有资产，是控制设计成本的关键。

### 2. 五种变更操作指南

#### 2.1 新增节点（Add）

| 项 | 说明 |
|---|------|
| **触发条件** | 新增需求与下游方案所有节点的覆盖范围无重叠，且模式/领域不同 |
| **AI 操作** | 自动执行：在适当父节点下创建末级节点，填写四要素（定义/映射/符合性/下游指导） |
| **标注** | 标注"新增节点 — 无现有节点可承接" |
| **人工确认** | 位点选择 → 确认父节点归属是否正确 |

#### 2.2 扩展节点（Extend）

| 项 | 说明 |
|---|------|
| **触发条件** | 新增需求落在某现有节点的合理职责范围内 |
| **AI 操作** | 提出建议：扩展现有节点的定义描述 + 追加映射的上层需求条目，每次扩展后做内聚性检查 |
| **内聚性检查** | 扩展后节点定义中是否存在"以及/同时/另外"连接不相关内容？→ 是则触发送代告警 |
| **扩展次数阈值** | 同一节点经历 3 次以上 Extend 后，AI 自动告警："该节点已扩展 N 次，建议审查是否需要 Split" |
| **人工确认** | 扩展后的边界是否仍然合理 |

#### 2.3 拆分节点（Split）

| 项 | 说明 |
|---|------|
| **触发条件** | 节点已多次扩展职责不清；或承接数超过合理范围（6+条）|
| **AI 操作** | 提出建议：按功能域/角色/流程阶段给出拆分方案，标注原节点"已废弃，拆分为A1、A2" |
| **标注** | 原节点标注"已废弃，拆分为[新节点ID列表]"，不物理删除 |
| **人工确认** | 拆分粒度是否合适 |

#### 2.4 合并节点（Merge）

| 项 | 说明 |
|---|------|
| **触发条件** | 新增需求要求跨多个现有节点的横向整合；或承接数过少（1条）且相邻节点功能相似 |
| **AI 操作** | 提出建议：新建节点承接合并范围，标注原节点"已合并至[新节点ID]" |
| **标注** | 原节点标注"已合并至[新节点ID]"，不物理删除 |
| **人工确认** | 合并范围是否合适，是否能保持内聚 |

#### 2.5 废弃节点（Deprecate）

| 项 | 说明 |
|---|------|
| **触发条件** | 上层需求取消，且节点无其他上游需求映射 |
| **AI 操作** | 全自动：标注"无上游需求，已废弃"，检查下游是否有节点映射到此节点 → 若有则同步告警 |
| **标注** | 标注废弃状态，保留节点结构（不物理删除）以维护追溯链完整性 |

**重要规则**：废弃/合并的节点**不物理删除**。保留节点标注有以下作用：
- 追溯链不因删除而断裂（规则3：双向追溯）
- 历史决策可回溯——后期加入的人员能理解"当时为什么这里有一个节点"
- 如果未来类似需求再次出现，避免重复设计

### 3. 工作流顺序：先自治分析，再审视资产

**核心顺序**：每分解产生一条新的需求 x，先在本需求文档内完成自治分析，再去下游方案中按资产优先原则找承接节点。

不得跳过自治分析直接审视下游资产——此顺序防止方案结构扭曲需求定义。

**自治分析的三项检查**：

```
需求 x 分解完成后 → 
  ① 可复用性检查：x 与本层已有需求是否存在重复？
     ├─ 完全重复 → 去重，不再分配
     ├─ 部分重叠 → 标注重叠范围，建议合并
     └─ 独立     → 通过

  ② 冲突性检查：x 与已有需求是否存在矛盾？
     ├─ 直接冲突 → 标记"需确认"，暂停分配
     ├─ 间接冲突 → 标记"需调整"
     └─ 无冲突   → 通过

  ③ 规范性检查：x 的格式/粒度/编号是否符合本层规范？
     ├─ 格式/编号 → AI 直接修复
     ├─ 粒度异常  → 标记"粒度不一致"
     └─ 规范      → 通过

三项检查通过后 → 带着需求 x 去审视下游方案资产
```

### 4. AI 参与度总览

| 阶段 | 操作 | AI 角色 | 人工介入点 |
|------|------|---------|-----------|
| 需求侧自治 | 可复用性检查 | 自动执行（边界情况标记上报） | 意图差异的判断 |
| | 冲突性检查 | 自动执行 | 冲突场景下哪方正确的判断 |
| | 规范性检查 | **全自动** | 无需介入 |
| 方案侧资产审视 | 复用检查 | 自动执行（边界存疑时标记上报） | 语义匹配置信度60-80%的判断 |
| | 扩展建议（Extend） | 提出建议 | 扩展后边界的确认 |
| | 拆分建议（Split） | 提出建议 | 拆分粒度的确认 |
| | 合并建议（Merge） | 提出建议 | 合并范围的确认 |
| | 新增必要性判断（Add） | **标记上报** | 确认是否确实没有其他选择 |
| | 废弃标注（Deprecate） | 全自动 | 无需介入 |

### 5. 与信息处理模式的关系

资产优先原则受信息处理模式的约束：
- 不同模式 → 不可共用同一节点，直接走 Add
- 同模式但不同领域 → 后4类领域特有模式，即使模式相同但领域不同，仍不可共用
- 同模式同领域 → 可以扩展（Extend），但需检查内聚性

模式/领域差异是**不可逾越的边界**，不能因为想复用资产而将不同模式的需求塞入同一节点。

## 二、分解粒度判断规则

### 2.1 原始需求 → 末级的分解

**判断问题**：这条 EOS 原始需求需要分解到几个末级？

**判断步骤**：
```
第1步：语义分析
  如果需求中包含"和/并且/同时"连接多个功能  → 分解
  如果涉及多个角色或相关方（如管理者与执行者）  → 分解
  如果描述的是"系统需要做A和B"  → 检查A和B是否可独立实现

第2步：1:1 约束测试（核心判断方法）
  问自己：这个条目能用一个下层末级节点承接吗？
  如果"是" → 停止分解 ✅
  如果"否" → 继续分解（说明还有原子性单元未拆开）

第3步：粒度一致性检查
  同一次分解得到的末级，篇幅不应差异过大
  - 一条 1 行 vs 另一条 20 行 → 前者过粗，需进一步审视
  - 所有末级在 3-8 行描述范围内 → 粒度一致 ✅
```

**EOS 示例对比**：

| 场景 | ❌ 错误分解 | ✅ 正确分解 |
|------|-----------|-----------|
| "绩效指标录入和查询" | OR-001: 绩效指标录入和查询（未分解） | OR-001.1: 绩效指标录入功能 → SR-F-XXX |
| | | OR-001.2: 绩效指标查询功能 → SR-F-YYY |
| "运营流程管理（建模、审批、监控）" | 直接映射到 SR | OR-001.1: 流程建模 |
| | | OR-001.2: 流程审批 |
| | | OR-001.3: 流程监控 |

### 2.2 SR 详细定义 → BA IPO 的分解

**判断方法**：按业务操作的自然边界分解

**分解策略**（按优先级）：
1. **按功能流程分解**：步骤1 → 步骤2 → 步骤3（适合流程性功能，如"指标采集→计算→发布"）
2. **按用户角色分解**：角色A的操作 → 角色B的操作（适合多角色功能，如"管理者审批→执行者填报"）
3. **按业务规则分解**：条件1的处理 → 条件2的处理（适合规则性功能）

**停止准则**：
```
问自己：这个条目能映射到业务架构的一个 IPO 吗？
- 需要多个 IPO 承接 → 继续分解
- 只需一个 IPO 承接 → 停止 ✅
```

**BA 命名规则**：
```
✅ 正确：绩效指标录入、流程审批处理、运营报表生成
❌ 错误：需求处理、系统管理、数据维护（过于模糊）
规则：使用"动词+名词"结构，准确反映运营业务活动的核心操作
```

### 2.3 SysReq 5级 → 9级的分解

**分解链**：
```
5级（组件实例）→ 6级（功能模块/Tab/页面）
  → 7级（系统用例/按钮/菜单）
    → 8级（用例场景：正常/异常/替代）
      → 9级（场景活动：可操作步骤）
```

**分解停止判断**：
```
每条 9 级场景活动只需分配到产品架构的一个组件（1:1约束）
- 如果一条 9 级活动需要分配到多个 PA 组件 → 继续分解
- 如果一条 9 级活动只需要一个 PA 组件承接 → 停止 ✅
```

**场景覆盖规则**：
```
每个 7 级系统用例至少包含：
- 1 个 8 级正常场景（Happy Path）
- 1 个 8 级异常场景（出错/边界条件）
- 可选 1 个 8 级替代场景（可选路径）
```

## 三、合并/拆分决策树

> **前置原则**：以下合并/拆分决策受「设计原则：组织资产优先」约束。优先尝试 Reuse，其次考虑改进（Extend/Split/Merge），最后才 Add。拆分和合并都是"改进"操作——在现有资产基础上调整，而非新增。详见本章之前的设计原则一节。

### 3.1 N:1 承接数触发的判断

每个架构末级节点承接 2-5 条详细定义末级为合理范围：

```
承接数 → 操作
  1条 ─→ 检查可否合并：
         ├─ 同层相邻、功能相似 → 建议合并并标注依据
         └─ 天然独立 → 保持，标注"独立节点"

  2-3条 ─→ ✅ 最优，保持

  4-5条 ─→ 检查上下文关联：
         ├─ 强关联 → 可接受
         └─ 弱关联/无关联 → 评估拆分

  6+条 ─→ 建议拆分：
         ├─ 按功能域拆分
         ├─ 按用户角色拆分
         └─ 按流程阶段拆分
```

### 3.2 IPO 去重的三级决策

```
第1步：检查 — 跨所有 BA 流程，检查是否有相同或相似的 IPO
第2步：复用 — 完全相同的 IPO 合并为一个，记录所有承接来源
第3步：新增 — 确实唯一的 IPO 保留独立

去重判断标准：
"完全重复" → 功能描述、输入输出完全相同 → 合并
"部分重复" → 核心功能相同但部分不同 → 合并为通用+特化
"独立"    → 功能定位完全不同 → 保持独立 ✅
```

**EOS 示例**：
```
BA-001: 绩效指标录入（I=指标数据, P=校验+存储, O=确认结果）
BA-002: KPI 数据录入（I=KPI数据, P=校验+存储, O=确认结果）
→ 判断：部分重复（都是录入指标数据，但绩效指标和 KPI 可能不同）
→ 建议：合并为 BA-通用指标录入 + 特化标记
```

### 3.3 PA 构件粒度判断

```
问：一个 SysReq 9 级活动应该映射到怎样的 PA 构件？

粒度标准（EOS 软件产品）：
- 一个可独立部署的服务或模块 = 一个 PA 构件
- 一个可独立审查的功能单元 = 一个 PA 构件
- 一个团队可在 1-2 周内完成的单元 = 一个 PA 构件

⚠️ 常见陷阱：
- 将整个"绩效管理"作为一个 PA 构件 → 评估是否应该拆分为"绩效指标管理"、"绩效评价"、"绩效分析"等独立构件
- 将"流程引擎"拆分为多个组件 → 视引擎复杂度而定，已封装的服务可保持一个构件
```

## 四、需求歧义处理策略

### 4.1 歧义程度分级

| 级别 | 判断 | AI 操作 |
|------|------|--------|
| **L1: 可推测** | 语义清晰度 > 80%，有上下文佐证 | AI 自行补齐，标注"AI推断" |
| **L2: 多选项** | 2-3 个合理理解，各有支持依据 | 列出选项+推荐，标注"待选择" |
| **L3: 不可解** | 语义矛盾或完全无法理解 | 标注"需澄清"，暂停该步骤 |

### 4.2 L1 级别（可推测）的补齐规则

```
规则：
1. 模糊术语替换：将模糊词汇替换为精确表述
   "实时" → "延迟 < 100ms"（标注 AI 推断）
   "快速" → "响应时间 < 2s"（标注 AI 推断）
   "大量" → "支持同时处理 > 1000 条"（标注 AI 推断）

2. 残缺语义补齐：根据上下文推测缺失的信息
   残缺："系统应该支持绩效查询"
   补齐："系统应该支持管理者按部门、时间段查询员工业绩考核结果，支持分页显示"
   标注：AI 推断的依据（"绩效"+"查询"上下文中推测为业绩考核查询）

3. 边界条件：
   - 补齐内容不改变原有需求的核心意图
   - 补齐内容不增加新的功能需求
   - 补齐内容标注为 AI 推断，供人工审核
```

### 4.3 L2 级别（多选项）的处理方法

```
格式：
⚠️ 待选择：[决策点名称]

选项：
├─ 选项A：[描述] — 优点：[...] 缺点：[...]
├─ 选项B：[描述] — 优点：[...] 缺点：[...]
└─ AI 推荐：选项A — 理由：[简要说明]

EOS 示例：
⚠️ 待选择：运营流程分类方式
├─ 选项A：按流程类型分类（核心流程/管理流程/支持流程）— 优点：与管理理论一致
├─ 选项B：按组织层级分类（公司级/部门级/岗位级）— 优点：与组织架构对齐
└─ AI 推荐：选项A — 与流程管理领域标准分类一致
```

## 五、同步设计操作指导

### 5.1 "三个及时"原则

```
1. 及时建骨架
   上层分解出第 3 个末级时，立即创建下层骨架占位符
   示例：第2步中，当 BA IPO 已有 3 个时 → 创建 SysReq 5 级骨架

2. 及时建映射
   每完成一个上层末级完整定义，立即建立到下层节点的映射
   示例：BA-001（绩效指标录入）定义完成后 → 立即建立 BA-001 → SysReq-F-XXX 映射

3. 及时冻结
   映射关系一旦建立，同步冻结编号，后续不随意更改
   如需修改 → 评估影响范围，标注变更记录
```

### 5.2 同步设计的冲突处理

```
场景：BA IPO 已映射到 SysReq 骨架后发现需要调整 BA

处理步骤：
1. 检查调整是否影响已建立的 BA → SysReq 映射
2. 如果影响：
   a. 评估需要修改的 SysReq 节点数
   b. 调整映射关系
   c. 标注变更记录（原节点ID → 新节点ID）
3. 如果不影响：
   a. 直接修改 BA IPO
   b. 确保修改后仍能满足 1:1 约束
```

## 六、适应性判断

### 6.1 执行模式选择

AI 根据当前步骤的执行特征自动选择执行模式：

| 工作特征 | 推荐模式 | 说明 |
|---------|---------|------|
| 纯机械性操作（格式化、编号） | 全自动 | 规则明确，无需人工干预 |
| 有创造性的设计工作（架构定义） | 半自动 | 完成一步后摘要报告，等待确认 |
| 跨步骤验证（追溯检查） | 全自动 | 规则明确，批量处理 |
| 涉及业务判断（IPO 语义） | 半自动 | 提出方案后等待业务审核 |

### 6.2 遇到异常的处置

```
发现异常时，AI 按以下优先级处理：

P0 — 立即暂停
  - 发现 EOS 原始需求中有无法解析的语义矛盾
  - 发现设计路径有多个显著不同的方向，无明确选择依据
  - 发现关键依赖信息缺失

P1 — 标记并继续
  - 非关键的歧义项（不影响当前步骤核心产出的完整性）
  - 可留待后续步骤验证的暂定项

P2 — 仅记录
  - 轻微格式不一致
  - 后续可统一修改的非关键问题
```

## 七、质量自检触发规则

### 7.1 何时需要回溯修正

```
第 N 步发现问题需要修改第 M 步（M < N）的产出时：

差异范围判断：
├─ 局部差异（影响 < 3 个节点）→ 直接修改，不影响后续
├─ 中等差异（影响 3-10 个节点）→ 修改后检查下游映射是否需要调整
└─ 全局差异（影响 > 10 个节点）→ 暂停，评估是否系统性问题，标记待人工处理

自检发现的问题分类：
├─ 格式问题 → 直接修复，无需确认
├─ 内容缺失 → 补充，标注修复说明
├─ 决策冲突 → 提出方案，标注待确认
└─ 映射断裂 → 评估后修复，重新验证
```

### 7.2 跨步骤影响范围表

| 第 N 步的修改 | 影响下游 | 需要重新执行的步骤 |
|--------------|---------|-----------------|
| 第1步 OR 末级修改 | SR、BA、SysReq、PA | 第2-6步全部或部分 |
| 第2步 SR 详细修改 | BA、SysReq、PA | 第3步开始 |
| 第2步 BA IPO 修改 | SysReq、PA | 第3步开始 |
| 第3步 SysReq 架构修改 | PA | 第5步开始 |
| 第4步 NFR 修改 | PA（NFR约束） | 第5步NFR检查 |
| 第5步 SysReq 详细修改 | PA 映射 | 第5步PA映射+第6步 |
| 第6步发现问题 | 取决于问题定位 | 定位到对应步骤 |

## 八、逆向工程决策补充

逆向工程（自底向上推导）的决策逻辑与正向设计（自顶向下分解）有本质差异。本节补充逆向工程特有的决策规则。

### 8.1 代码分析深度判断

**判断问题**：代码分析到什么程度算"足够"？

```
分析深度分级：

L1 — 目录结构级（第1步最低要求）
├─ 分析：目录结构、模块划分、关键文件列表
├─ 适用：获取组件划分的基本依据
└─ AI输出：组件识别清单（不含详细接口分析）

L2 — API/接口级（推荐深度）
├─ 分析：L1 + Controller/API 注解、前端路由、数据库 Schema
├─ 适用：反向推导 SysReq 9 级场景活动
└─ AI输出：功能模块清单 + 候选 SysReq 活动

L3 — 实现逻辑级（深入分析）
├─ 分析：L2 + 业务逻辑代码、缓存策略、异步处理、异常处理
├─ 适用：需要精细的性能反推或复杂的业务逻辑恢复
└─ AI输出：完整的性能指标反推 + 复杂运营场景恢复

停止判断：
- 已达到 L2 深度 → 可开始第1步架构评估 ✅
- 特殊需要（复杂运营业务/关键性能要求）→ 推进至 L3
- 代码量太大 → 先 L1 快速扫描再按组件优先级深入
```

### 8.2 反向推导的有效性判定

**判断问题**：从代码反推的需求是否合理有效？

```
判断维度：

1. 对应性（必须满足）
   - 代码中能找到对应的实现（类/方法/配置）
   - 若找不到 → 标记"推导无代码对应"，供人工确认

2. 语义合理性（应满足）
   - 反推出的需求描述符合运营业务常识
   - 若不符合 → 标记"语义异常"，需人工解释

3. 完整性（应满足）
   - 从前端→后端→数据层形成完整链路
   - 若缺失中间环节 → 标记"链路断点"

4. 一致性与边界（建议满足）
   - 反推的需求与 EOS 产品整体定位一致 / 若不一致 → 标记"边界存疑"

推导有效性评级：
├─ AAA：代码对应 + 语义合理 + 链路完整 + 产品一致 → 直接使用 ✅
├─ AA：代码对应 + 语义合理 — 链路断点 → 标注"待补齐链路"
├─ A：代码对应 — 语义模糊 → 标注"语义待确认"
├─ B：代码对应 — 不合常识 → 标注"语义异常需人工判断"
└─ C：无代码对应 → 标注"推导无代码对应"，等待人工
```

### 8.3 反向聚合规则

逆向工程中，下层多个节点聚合为上层一个节点（与正向的分解相反）。

```
聚合方法（按优先级）：

1. 功能域聚合
   同一功能域（如"绩效管理"的多个操作）→ 聚合为一条 OR
   SysReq-F-录入绩效指标 + SysReq-F-计算考核得分 + SysReq-F-生成绩效报告
   → OR: 绩效全流程管理（聚合后自然语言描述）

2. 流程阶段聚合
   同一流程的不同阶段 → 聚合为一条 OR
   BA-流程建模 + BA-流程审批 + BA-流程发布
   → OR: 业务流程全生命周期管理

3. 质量属性聚合
   同一质量维度的多个 NFR 指标 → 聚合为一条 OR-NFR

聚合粒度控制：
├─ 聚合后 OR 能用一句话说清核心意图 ✅
├─ 聚合后 OR 应使用自然语言（非术语/非技术词汇）
├─ 聚合过多（OR 包含 4+ 个不同域的 SR）→ 拆分
└─ 聚合过少（OR 只有 1 个 SR 末级）→ 检查是否可合并
```

### 8.4 代码-文档差异处理

**判断问题**：当代码实现与预期文档描述不一致时如何处理？

```
差异类型 → 处理方式：

1. 代码有，文档应有但没有
   → 补充文档，标注"从代码恢复"

2. 文档描述了，代码中不存在
   → 标注"代码中未实现"，等待人工确认是否遗漏

3. 代码与文档描述不一致
   → 以代码为准生成文档，记录差异项
   → 格式："[差异项]: 代码[实际行为] vs 文档[原描述]"

4. 代码有废弃/兼容遗留代码
   → 尝试判断是否活跃代码
   → 无法判断 → 标注"待确认是否活跃"
```

### 8.5 性能指标反推的置信度判断

```
置信度分级 → 标注方式：

高（从代码配置直接提取）
  ├─ 示例：连接池 maxTotal=100 → 并发能力 >= 100
  └─ 标注："取自配置"

中（从代码实现特征反推）
  ├─ 示例：有 Redis 缓存 → 响应时间预期 < 200ms
  └─ 标注："反推值"

低（从架构或常识推断）
  ├─ 示例：单体应用无特殊优化 → 性能要求一般
  └─ 标注："推断值 ⚠️ 待确认"

无法推断
  └─ 标注："待补充"
```
