## 一、跨步骤上下文管理

### 1.1 上下文传递方式

6 步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

### 1.2 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**（见本章末尾）中，供后续步骤引用。

### 1.3 回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见 AI决策框架 §7.2 跨步骤影响范围表）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

## 二、AI 经济原则

本步操作以下列顺序执行，AI 需注意：

1. **执行顺序不可逆**：分析活动 → 加载 BA 资产 → SR 分解与 BA 承接（同步）→ BA 节点完善 → 闭环检查 → 同步建骨架
   核心驱动力：需求从 SR 架构出发，逐条向 BA 找宿主。BA 架构的调整（扩展/新增）由 SR 需求驱动，不是预设的
2. **基于资产优先**：BA 架构调整决策应先检查已有资产（现有 BA IPO），确实需要修改时再改进，最后才新增
3. **资产路径已知**：本步的 BA 资产是 `90-eos-system-product-data/04-eos-business-architecture.md`，其中可能存在已有 BA IPO（v1 基线区），是 BA 架构决策的首要资产来源。`02-eos-stakeholder-requirements-architecture.md` 是本步的输入数据（待分解的 SR 架构），不是"资产"

## 三、本步提示词模板

### 3.1 分析活动：架构定义验证

```
请在开始详细定义前，对第1步产出的相关方需求架构定义执行分析活动。

相关方需求架构定义（功能部分）：
[粘贴上一步SR-F部分的输出]

需求说明符合性分析：
1. 映射完整性 — 检查每个SR-F架构末级节点是否充分满足所承接的原始需求末级，N:1映射有无遗漏
2. 一致性校验 — SR架构节点的命名和定义是否与原始需求的意图一致
3. 功能/非功能分离正确性 — 功能需求和非功能需求是否已在SR层正确分离

需求性能符合性分析：
1. 性能继承验证 — SR架构节点是否继承了OR的性能指标？
2. 可测量性检查 — 指标是否可测试、可验证？

输出：分析活动报告
├─ 映射完整性：[通过/未通过，如有遗漏列出]
├─ 性能指标量化：[通过/未通过，如有问题列出]
└─ ⚠️ 待确认项：[如有]
```

### 3.2 SR 功能部分详细定义 → BA 承接（同步）

> 先了解资产再动手——分解前加载现有BA架构，理解每个现有BA IPO的职责边界。然后面对每条SR架构末级时，已经知道"哪个方向有节点、哪个方向还空着"，有方向地去分解。

```
请先加载现有BA架构资产，理解其分类结构和各BA IPO的职责边界后，再逐条处理SR架构末级。

SR架构末级节点清单：[粘贴上一步SR-F部分的架构末级节点]

—— 第一步：加载并理解 BA 架构资产（先做到心中有数）——

资产文件：90-eos-system-product-data/04-eos-business-architecture.md

加载所有现有BA IPO节点（v1 基线区·只读），对每个节点理解：
├─ 节点名称和定义 — 这个IPO"管什么业务活动"？
├─ 职责边界 — 它的覆盖范围到哪里为止？相邻IPO之间的分界在哪里？
├─ 已承接的SR — 这个IPO目前已经承接了哪些类别的SR详细末级？

整体理解：
├─ BA IPO按什么逻辑分类？（按功能域？按产品构件？按操作流程？）
├─ 哪个分类下节点密集（已覆盖充分），哪个分类下节点稀疏（可能需要补充）
└─ 现有BA IPO的粒度是否一致（合理范围2-5条SR详细末级/节点）

—— 第二步：逐条 SR 架构末级处理（分解 + 同步找 BA 宿主）——

对每条 SR 架构末级（SR-F-XXX），执行以下操作：

0. 【前置判断】这条SR架构末级涉及多个BA职责范围吗？
├─ 例如：SR-F-001同时涉及"建立准则体系"和"准则维护" → 这两个可能分属不同BA IPO
├─ 如果是 → 先按涉及的职责范围分拆，再分别执行下面的分解步骤
└─ 如果不是 → 直接进入步骤1

1. 【分解】原子化分解到详细末级：
├─ 原则：原子性、语义完整性、粒度一致性
├─ 触发分解条件：
│  ├─ 包含"和/并且/同时"连接多个功能 → 分解
│  ├─ 涉及多个业务活动或操作步骤 → 分解
│  └─ 涉及多个BA职责范围 → 分解（由前置判断的结果决定）
├─ 停止准则："在不了解BA的情况下，这也是一条不可再分的单一业务诉求吗？"
│  （先独立判断原子性，再用BA知识验证）
│  - 是 → 得到详细末级 SR-F-XXX.X
│  - 否 → 继续分解
└─ 输出：SR-F-XXX.X [具体功能名称]

2. 【找宿主】对每个 SR-F-XXX.X，在已理解的 BA 知识中找承接节点：

   判断维度（用三个问题来决策）：
   ├─ Q1 业务职责方向匹配：此SR详细末级的核心意图，与哪个BA IPO的职责方向一致？
   │  例："建立设计准则体系" — BA-001（准则开发）? BA-002（模板管理）? BA-003（质量检查）?
   │  判断依据：BA IPO定义中描述的主语和核心业务动作
   │
   ├─ Q2 覆盖范围偏差：此SR详细末级描述的具体内容，是否在BA IPO定义的边界之内？
   │  例：BA-001定义是"开发系统设计准则文档" → 
   │  "准则的版本管理和变更控制" > "开发准则文档"，超出了BA-001的边界
   │
   └─ Q3 同类承接情况：此BA IPO是否已经在承接同类SR详细末级？
      例：BA-001 已承接 SR-F-001.1(准则架构), SR-F-001.2(准则规则), SR-F-001.3(准则模板)
      → 它已经在承接各类"准则定义"类SR详细末级，覆盖范围较广

   根据三个问题的答案做决策：
   ├─ 条件A — 直接复用：
   │  Q1命中 + Q2在边界内 + Q3同类
   │  → SR-F-XXX.X → BA-YYY（复用）
   │  例："建立设计准则体系" → BA-001（已在准则开发范围内）
   │
   ├─ 条件B — 扩展后承接：
   │  Q1命中 + Q2略超边界 + Q3同类
   │  → 标记 BA-YYY（改进-扩展），记录扩展理由
   │  例："准则的版本管理规则"略超BA-001的"准则开发"范围
   │       → 扩展BA-001定义：从"准则开发"扩展为"准则生命周期管理"
   │
   ├─ 条件C — 新增：
   │  Q1无命中 + Q2不在任何BA IPO边界内 + Q3无同类
   │  → 记录需新增BA IPO，写明新增原因
   │
   └─ 条件D — ❓ 无法判断：
       Q1命中了多个方向 / Q2边界完全模糊 / Q3信息不足以对比
       → 标记❓，说明卡在哪个维度上，**不要强行推测**，留给人类

3. 【记录】同步记录本步决策：
├─ 承接的BA IPO：[BA-YYY / 新增]
├─ 资产决策：[复用 / 改进-扩展 / 新增]
└─ 决策理由：引用以上三个判断维度的结果

⚠️ 关键判断原则：
- "业务职责方向一致"的含义：SR详细末级的核心意图与BA IPO的定义动词匹配
  （如"建立准则"→开发类IPO、"检查质量"→审核类IPO）
- "超边界"的含义：SR详细末级的具体内容超出了BA IPO定义中列举的范围
- 不是"感觉上接近"，而是用三个维度可验证地判断
- 同一个BA IPO可承接多条SR详细末级（N:1）
- ⚠️ **扩展和新增的强内聚性检查**：
  ├─ 扩展前问"扩展后这个IPO还描述一个单一业务活动吗？"
  │  例：BA-001负责"准则开发"，扩展为"准则生命周期管理"
  │  → 包括开发、维护、版本管理等 → 扩展后会变"杂"，应新增节点而非扩展
  ├─ 新增前问"这个新节点和同级兄弟节点的颗粒度一致吗？"
  │  例：相邻IPO都是"某个具体操作步骤"的粒度，新节点不能是一个"大类"
  └─ 颗粒度标准：该IPO应可自然对应一个SysReq 5级节点（t03可操作）

—— 输出 ——

SR→BA映射清单：
| SR详细末级 | 描述 | 承接的BA IPO | 资产决策 | 决策理由（Q1/Q2/Q3） |
|-----------|------|-------------|---------|--------------------|
| SR-F-XXX.X | [描述] | BA-YYY | 复用 ✅ | Q1命中[职责]，Q2在边界内，Q3已有同类 |
| SR-F-YYY.Y | [描述] | BA-ZZZ | 改进-扩展 ⚠️ | Q1命中，Q2略超[范围]，需扩展定义 |
| SR-F-ZZZ.Z | [描述] | [新增] | 新增 ⚠️ | Q1无命中，无BA IPO职责方向匹配 |
| SR-F-WWW.W | [描述] | — | ❓ 无法判断 | Q1命中多个[BA-AAA][BA-BBB]，无法确定主方向 |

资产决策汇总：
├─ 复用节点 ✅：[N个] — BA-XXX, BA-YYY, ...
├─ 改进（扩展）节点 ⚠️：[N个] — BA-ZZZ（扩展[内容]）
├─ 新增节点 ⚠️：[N个] — 新增用于承接[SR详细末级清单]
├─ ❓ 待人类裁定：[N个] — [SR详细末级编号] + [卡在哪个维度]
└─ 未承接的SR详细末级：[清单]（理论上应为空）
```

SR→BA映射结果：[粘贴3.2的输出（SR→BA映射清单）]

## 五、审查修复流程

### 5.1 审查输入

AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

### 5.2 AI 修复流程

收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）→ 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题** → 参考AI决策框架的跨步骤影响范围表进行影响范围评估，列出所有受影响的下游节点

### 5.3 影响范围评估

| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 某步的架构定义 | 本层 + 下游已建立映射的节点 | 修改后同步更新映射 |
| 详细定义 | 下层方案 | 修改后检查下层是否受影响 |
| 映射关系 | 所有关联节点 | 需重新运行追溯验证 |
| 编号变更 | 全链路 | 需逐层更新引用，重新验证 |
