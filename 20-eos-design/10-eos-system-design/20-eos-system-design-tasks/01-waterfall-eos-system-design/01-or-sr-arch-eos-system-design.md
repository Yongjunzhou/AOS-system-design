# task-01：原始需求分析及相关方需求架构定义

## 目标

从原始需求出发，经过收集整理、规范化、分解到末级、冲突检测，建立相关方需求架构定义。

## 上下文继承

```
从哪里来：
  - 直接输入：原始需求原始表述（会议纪要、邮件、文档等）
  - 参考文档：AI决策框架（分解粒度判断、冲突/重复处理策略）
  - 涉及规范：原始需求规范 §4.1、相关方需求规范 §2、设计准则 §3.1 第1步

预期产出：
  1. 原始需求文档（含末级需求分解）
  2. 相关方需求架构定义子文档（树形结构 + 映射关系表）
  3. 冲突和重复检测报告

本步决策对后续的影响：
  - 原始需求的末级分解直接影响 task-02 SR 详细定义的范围
  - SR 架构的划分方式（功能/非功能分离）决定后续任务的分支路径
  - 编号体系影响全链路的追溯能力
```

## 输入

| 输入 | 说明 |
|------|------|
| 原始需求原始表述 | 来自EOS各相关方（需求提出者）的需求描述（会议纪要、邮件、文档等） |

## 操作流程

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 原始需求收集和初步整理 | AI | 血缘归并、角色聚合、流程串联、父子补全 |
| 2 | 四项检查和规范化 | AI | 完整性、清晰性、可测试性、独立性检查 |
| 3 | 需求分解到末级 | AI | 原子性原则，1:1 映射约束（参考 AI 决策框架 §2.1 分解粒度判断规则） |
| 4 | 分析活动：冲突和重复检测 | AI | 需求说明符合性分析（冲突/重复检测）+ 需求性能符合性分析（指标量化检查） |
| 5 | 相关方需求架构定义 | AI | 按 SR-F 和 SR-NFR 分别建立架构树，编写符合性分析和设计指导 |
| 6 | 结果审核 | 人类 | 确认优先级、处理冲突、确认 SR 架构划分 |

## 人类审核要点

- [ ] 优先级判定（P0-P3）— 需与需求提出者确认
- [ ] 冲突/重复检测的最终决策（保留/合并/删除）
- [ ] 相关方需求架构的完整性——是否覆盖所有末级需求
- [ ] 功能需求和非功能需求的分离正确性
- [ ] 每个 SR 架构末级节点包含符合性分析和设计指导

## 质量门禁

- [ ] 原始需求已初步整理（碎片归并、补全语境）
- [ ] 原始需求已分解到末级（原子性 + 1:1 约束满足）
- [ ] 原始需求末级清单完整，四项检查通过
- [ ] 冲突和重复检测完成，结果已记录
- [ ] 所有原始需求末级已映射到 SR 架构节点（覆盖率 100%）
- [ ] SR 架构定义完整（树形结构 + 映射关系表）
- [ ] 每个 SR 架构末级节点包含符合性分析和设计指导
- [ ] 编号在架构定义阶段确定后保持稳定，变更需经评审
- [ ] 运行 rule-conformance-review 进行符合性审查，按审查修复流程处理问题
