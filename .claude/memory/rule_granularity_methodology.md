# 规则提取颗粒度方法论

规则颗粒度的判定不是基于"能否独立违反"，而是基于：**审查时两个子句是否在同一次文档遍历中、用同一个判断焦点检查？**

## 三种边界案例类型

| 类型 | 特征 | 例子 | 处理 |
|------|------|------|------|
| **复合属性** | 多个子句共同定义一个完整概念或规范 | "序号使用3位数字，起始001"、"构件1-2周可完成且可独立部署和测试" | **不应拆分**——拆开丢失语义完整性 |
| **独立维度** | 多个子句定义不同维度的验收项，检查对象不同 | "架构末级节点定义完整率100% + 架构层级结构完整率100%" | **应当拆分**——各自有独立的检查对象 |
| **条件-动作对** | "如果X，则Y" | "同步设计场景下，开展上层详细定义的同时同步建立下层架构骨架" | **不应拆分**——条件和动作是一体的 |

## 核心判据（比"是否可独立违反"更精确）

> 审查时，你是在同一个地方、用同一个视角检查这两个子句，还是需要在文档的不同部分、用不同的视角检查？

## 三个粒度异常信号

1. **"且"/"并"/"以及"出现在规则中**——但不是复合属性（如"无遗漏"和"无冗余"是同一遍历过程的正反两面）
2. **两个条件需要跨章节才能验证**——拆分信号
3. **审查员需要切换判断焦点才能完成检查**——但需区分"不同维度"（应拆）和"同一维度的不同侧面"（不拆，如"四大分类"不拆为四条规则）

## 实际提取的决策树

```
一条候选规则，包含子句A和子句B
├─ A和B是否在文档中不同的位置/不同的文档中检查？
│  ├─ 是 → 拆分
│  └─ 否 → 继续
├─ A和B是否属于同一个判断焦点（如"这个构件的定义是否完整"）？
│  ├─ 否 → 考虑拆分
│  └─ 是 → 继续
├─ 分开A和B审查时，A单独审查是否缺乏独立意义？
│  ├─ 是 → 不拆
│  └─ 否 → 继续
└─ A和B拆开后，是否会增加不必要的审查轮次而不会提升审查质量？
   ├─ 是 → 不拆
   └─ 否 → 可拆
```

## 与"标准"的关系

标准（准则原文）的条款边界是**提取的起点，不是终点**：
- **条款包含多个独立判断维度** → 向下拆分
- **条款的一个子句本身不构成独立判断** → 向上合并（保持复合属性完整）

正确的颗粒度在"一个完整的规范单元"和"一个独立的判断焦点"的交汇处。

**Why:** 通过实际审查产物分析发现，约12%的规则存在多段落共享，但规则本身可能包含可拆分的子句。颗粒度不当会导致审查时无法给出清晰的 pass/fail 判定，或增加不必要的审查轮次而不提升审查质量。

**How to apply:** 在 conformance-review 的 Step 0a 第二遍（规则清单提取）中，使用上述决策树判断每条候选规则的颗粒度。将拆分的规则分配独立的 R 编号，保持复合属性的规则保留为一个条目。在语义段落组装（第五遍）中，根据判断焦点而非规则边界来组织段落。
