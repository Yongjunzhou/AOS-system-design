## 端到端业务的"域间需求"原则

### 问题

端到端业务天然跨越多个业务域，如果试图将所有涉及的域都纳入一个端到端流程中，会导致该端到端流程吞并企业大部分甚至全部业务，失去架构意义。

### 核心原则

端到端业务的拥有者不将其他域的业务纳入自己的范围，而是：

```
域A（端到端拥有者）
  ├── 定义：域A内部协调自己的端到端流程
  ├── 当需要域B支持时：向域B提出明确需求
  │     ↓
  └── 域B在自己的BA中响应该需求
        ├── 在自己的架构中安排能力（IPO）来满足域A的需求
        └── 定义服务范围、SLA和接口契约
```

### 本质

每个域只对自己域内的业务负责。跨域协作通过 **需求—响应契约** 来定义，而不是通过把一个域的逻辑纳入另一个域的端到端流程。

### 与现有方法论的对接

| 方法论概念 | 在此原则中的作用 |
|-----------|----------------|
| 需求—方案配对 | 域A的需求 = 域B的方案输入 |
| IPO模型的Input/Output | 标明跨域数据交换的边界 |
| 13种信息处理模式 | 域间交互必定表现为某类模式（工单、审批、台账、计划等），是接口的"协议语言" |

### 待解决的后续问题

域间的需求关系应该在架构的哪个层级定义——SR层面（业务方跨域诉求）还是BA层面（域间设计协商）？
