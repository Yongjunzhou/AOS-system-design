# BA 存在的本质原因

## 核心命题

BA 存在的根本原因：**实现产品功能需求捕获的完备性。**

## 为什么不能直接从 SR-F 跳到 SysReq-F？

SR-F 描述"用户有什么业务"，SysReq-F 描述"系统有什么功能"。如果跳过 BA，直接让相关方验证 SysReq-F 功能清单是否完整，相关方没有能力系统化判断——因为系统语言（"系统接收→处理→输出"）与相关方的业务认知之间存在鸿沟。

## BA 作为可验证界面

BA IPO 使用业务语言（"采购员接收请购单→发起询价→归集报价→比价选商→生成PO→提交审批"），相关方看到后能直接判断"对，我就是这么干的"或"你漏了步骤"。

验证链：
```
相关方验证 BA（业务操作序列）→ 确信业务操作场景完整
    ↓ 基于已验证的 BA 系统化推导
SysReq-F（系统功能）必然完整
```

## 一句话

没有 BA，相关方无法显性化地确认系统功能是否完整性地支持他所期望的业务。BA 将"系统功能"翻译回"业务操作"，让相关方用自己的业务语言来验证功能完备性。
