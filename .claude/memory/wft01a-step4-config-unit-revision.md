wft01a Step 4（新增引擎需求设计）重构完成，版本 v1.27。

**核心变更**：
- 引擎场景的信息载体从 I/P/O 三维扁平清单改为**配置单元**——(配置要素 + 处理说明 + 业务说明) 三栏一体的原子单位
- 一个引擎场景 = 多个配置单元
- 粒度判据：I→P→O 连贯链完整性——一条完整的"配置了什么→怎么处理→交付了什么业务能力"链就是一个配置单元
- 25 资产中引擎模型的"引擎配置 IPO 类型"直接映射为配置单元（Input→配置要素 / Process→处理说明 / Output→业务说明）

**Step 4 新流程**：
- Phase A（触发与场景边界确定）：A.1 触发识别 → A.2 引擎模型匹配 → A.3 场景定义名称
- Phase B（配置单元展开）：B.1 候选配置单元加载 → B.2 逐配置单元处理 → B.3 配置单元设计 → B.4 排序与标识
- Phase C（验证与输出）：C.1 跨单元检查 → C.2 引擎能力边界陈述 → C.3 推理依据 → C.4 STR-E 要素 → C.5 写入规则

**25 资产同步变更**：
- 新增规则 8 明确 "配置 IPO 类型 → wft01a 配置单元 / wft02a 配置 IPO" 的术语映射
- Skill 读写契约补充 wft01a 为读角色

**与 Step 3 的区别**：
- Step 3 有 Phase A Q1/Q2/Q3 聚合 + Phase B 四关系检测 + Phase C 新建 STR（三步判断+十链覆盖）
- Step 4 走配置单元展开路径（引擎模型驱动 + 逐单元 I→P→O 连贯链），不需要十链覆盖

**Why**：引擎场景不是业务流场景，没有"信息在角色间流转"的概念；I/P/O 三遍独立扫描会导致配置单元事后配对困难。配置单元法直接在每组内锁定 I→P→O 连贯性。
**How to apply**：后续章节如需引用引擎场景定义，应按配置单元输出 STR-E 节点；wft02a 仍沿用"配置 IPO"术语，无影响。

**讨论起点**：对话从 "wft01a Step 4 的信息载体应该是什么" 开始。先明确了引擎场景是"引擎的可配置能力的完整定义"（使能场景而非业务流场景），再推导出配置单元概念。粒度判据（I→P→O 连贯链）来自跨引擎示例推演（审批引擎/表单引擎/台账引擎/报表引擎/规则引擎的配置单元分解对比）。
