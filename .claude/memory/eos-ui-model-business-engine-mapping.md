# EOS UI 交互模型与业务-引擎映射（2026-06-27 讨论结论）

## 一、UI 交互模型

### 初始态（单窗全宽）

单击场景菜单（L4 场景业务）→ 浏览器窗口为**单窗全宽**，顶部多 TAB 对应场景业务下的多个场景部件。

```
┌─────────────────────────────────────────────────────────┐
│ [TAB: 场景部件A] [TAB: 场景部件B] [TAB: 场景部件C]      │
│                                                         │
│  ┌───────────────────────────────────────────────┐      │
│  │ 主表单（列表/树表）                              │      │
│  │ ┌────┬────┬────┬────┐                         │      │
│  │ │行1 │    │    │    │                         │      │
│  │ │行2*│ ← 双击                                   │      │
│  │ │行3 │    │    │    │                         │      │
│  │ └────┴────┴────┴────┘                         │      │
│  └───────────────────────────────────────────────┘      │
│                                                         │
│  底部导航面包屑：场景业务 > 场景部件A                     │
└─────────────────────────────────────────────────────────┘
```

### 分屏态（下钻触发）

双击行数据 → 窗口分裂为左右双窗：

```
┌───────────────────────┬─────────────────────────────────┐
│ 左窗（原窗变窄）        │ 右窗（新开）                     │
│                       │                                 │
│ [TAB: 部件A]          │ [TAB: 子业务X] [TAB: 子业务Y]    │
│ ┌──────────────┐      │ ┌─────────────────────────┐     │
│ │行1           │      │ │ 行2的子业务数据           │     │
│ │行2*          │      │ │ ┌────┬────┬────┬────┐   │     │
│ │行3           │      │ │ │行a │    │    │    │   │     │
│ └──────────────┘      │ │ │行b*│                     │     │
│                       │ │ └────┴────┴────┴────┘   │     │
│                       │ └─────────────────────────┘     │
│                       │                                 │
│ 导航：场景业务 > 部件A │ 导航：场景业务 > 部件A           │
│                       │       > 子业务X                 │
└───────────────────────┴─────────────────────────────────┘
```

### 持续下钻

双击右窗行 → 右窗变为左窗，新子业务 TAB 出现在右窗位置：

```
┌───────────────────────┬─────────────────────────────────┐
│ 左窗：子业务X          │ 右窗：子子业务Z                   │
│ ...                   │ ...                             │
└───────────────────────┴─────────────────────────────────┘
```

### 关键交互规则

1. **初始单窗**：进入场景业务时为单窗全宽，不下钻不分屏
2. **首次下钻**：双击数据行触发分屏，左窗窄、右窗新开
3. **无限下钻**：右窗→左窗，新右窗出现，直到系统下钻层级数限制
4. **面包屑导航**：页面底部常驻导航条，形如资源管理器目录路径，显示当前表单的层级路径
5. **面包屑跳转**：双击导航条中任意节点，该节点对应表单成为左窗，右窗为其子业务
6. **多 TAB**：每个场景部件窗口包含多个 TAB，对应不同的场景部件或子业务

## 二、业务层级 L0~L6

| 层级 | 名称 | EOS 平台体现 | 生成引擎 |
|------|------|-------------|---------|
| L0 | EOS 平台 | 平台自身 | — |
| L1 | 业务系统 | 一级菜单 | 菜单引擎（横切） |
| L2 | 项目业务（业务模块） | 二级菜单 | 菜单引擎（横切） |
| L3 | 构件业务 | 三级菜单 | 菜单引擎（横切） |
| L4 | 场景业务 | **四级菜单（末级）**，STR-F 节点锚点，打开后多窗口 | 场景引擎（组件级） |
| L5 | 场景部件 | **一个窗口 = 一个场景部件 = 一份输出文档** | 表单/流程/指标/工单引擎（部件级） |
| L6 | 功能元件 | 原子构件（实体对象、事件、定时任务） | 实体引擎（元件级） |

## 三、场景部件的两种产生方式

| 维度 | 类型1 · 引擎直产 | 类型2 · 人工外设 |
|------|-----------------|-----------------|
| **处理主体** | 引擎算法 | 人类 + CAX 工具 |
| **人类角色** | 操作引擎界面 | 使用专业工具完成实质转换 |
| **场景部件职责** | 提供操作界面 + 处理算法 | 提供规则/知识/合规/资源，收纳输出附件 |
| **输入文档** | 引擎读取并处理 | 人类取出，导入 CAX 工具 |
| **输出文档** | 引擎生成 | 人类在 CAX 中产出，作为附件回传 |
| **CU 选配剖面** | 完整选配（含校验/行为联动） | 轻量选配（不选校验/行为） |

两种方式均由现有部件级引擎支撑，区别在**配置单元的选配剖面**不同，不需要新引擎类型。

## 四、表单引擎配置模型

### 主表单 + 子业务 TAB

表单引擎的一次配置实例 = **一个主表单 + N 个子业务 TAB**。主表单每一行数据可下钻，子业务数据每个占一个 TAB。

### 实体引擎约束

- **实体引擎**：配置实体 + 实体间关系，定义"哪些实体可以互相关联"
- **表单引擎**：消费实体关系图谱，配置主表单和子业务 TAB 群——子业务 TAB 只能从「主实体关联的实体清单」中选取
- 实体 A 关联了实体 B/C/D → 表单引擎可将 B/C/D 配置为 A 的主表单的子业务 TAB
- 实体 A 未关联实体 E → E 不可作为 A 的子业务 TAB

## 五、引擎分工总图

```
实体引擎（元件级）
  │  配置实体 + 实体间关系
  │  产出：L6 功能元件 + 实体关系图谱
  │        实体A → 子业务实体清单 [B, C, D, ...]
  │        该清单是表单引擎配置子业务TAB的约束范围
  │
  ↓  实体关系图谱作为约束输入
  │
表单/流程/指标/工单引擎（部件级）
  │  配置：主表单 + 子业务TAB群（TAB只能从实体关系清单中选取）
  │  产出：L5 场景部件 = 1个窗口 = 1份输出文档（含主表单+TAB群）
  │
  ↓  多个场景部件被编排
  │
场景引擎（组件级）
  │  编排场景部件的执行序列
  │  产出：L4 场景业务（端到端业务闭环 = STR-F 节点锚点）
  │
  ↓  菜单映射
  │
菜单引擎（横切，不参与业务生成）
  │  L1~L4 菜单树 CRUD + 权限 + 末级节点映射到 L4 场景业务
```

### 场景部件 : 输出文档 = 1 : 1

一个场景部件就是一个部件级引擎的一次配置实例，产生一份输出文档。输出文档不是扁平表单，而是以主实体为根的表单树（主表单 + 子业务 TAB 群）。子业务 TAB 不是独立的场景部件，而是同一场景部件内输出文档的子结构。

### 场景业务 = STR-F 锚点

- STR-F 节点 = L4 场景业务，端到端业务闭环
- STR-F 下是一组场景部件（L5）的有序列表
- 场景引擎负责编排这些场景部件的执行序列
- 每个场景部件的 TAB 由场景引擎编排为场景菜单中的可见 TAB

## 六、对 wft01-biz 的影响

1. **§3.1 STR-F 三层结构需修正**：当前"场景→场景部件→输出文档"的嵌套模型改为"场景业务→场景部件"两层——场景部件与输出文档是 1:1 关系，不再有独立的"输出文档"子层
2. **§3.2 场景部件识别规则需重写**：当前四条规则基于"场景部件包含多个文档"的旧假设，新模型下不再适用
3. **§3.2 标注字段合并**："场景部件级"和"文档级"标注字段在 1:1 关系下需合并
4. **输出文档的产生方式**：需区分"引擎直产"/"人工外设"，影响 wft02-biz 的 CU 选配推荐

**Why:** 2026-06-27 讨论澄清——此前 L5 场景部件与"输出文档"的关系模糊（是否 1:N），且场景部件在 UI 上的多 TAB 形态和实体引擎的约束角色未显性化。本记忆记录讨论达成的完整共识。

**How to apply:** wft01-biz §3.1/§3.2 改写时以此为权威参考；wft02-biz 展开时引擎分配以本记忆的引擎分工总图为依据；24/25 资产文件如有不一致以本记忆的澄清为准。
