# wft01 三路径自闭环模型

## 核心决策

### 1. 从 A/B 两阶段到三路径自闭环

wft01 的 biz/eng/nfr 三条路径各自闭环——设计+资产对准+写回在同一 Skill 内完成，不再拆分为 Skill A（设计）+ Skill B（资产维护）。人类一次"整体确认"覆盖设计决策和资产对准（Q1/Q2/Q3 判定），资产写回不设二次确认门禁。

### 2. 生命周期：5+1 态

```
待确认方案 → 可以分解分配 → 在分解分配 → 已分解分配
                                  ↑ 上游修订 → 待补充分解分配
```

- `可以分解分配` 由人类确认触发（Step 2），不由资产写回完成触发
- `在分解分配` 和 `已分解分配` 由下游写入
- 上游修订时仅 `已分解分配` 回退为 `待补充分解分配`；`在分解分配` 通过版本号感知变更

### 3. 资产与下游解耦

人类确认后两条线并行：
- 资产线（Step 4）——写回 23/26/22/25
- 下游线（wft02a/wft04a）——读 `可以分解分配` 即可进入

下游不依赖资产文件中的写回结果，它读的是 STR 节点中的对准结论。

### 4. 变更传播协议

上游 STR 因新 OR 修订后，通过变更影响声明通知下游（wft02a/wft04a）。三类变更：
- 增量型（仅新增原子单元）→ 下游自动增量展开
- 修订型（已有单元修改）→ 下游修订对应节点
- 结构型（关键边界改变）→ 停止等人类裁决

各路径的关键边界：biz=场景职责边界、eng=引擎模型匹配+抽象业务定义、nfr=分类维度+一级类目。

### 5. Step 结构收敛

统一的 5 步结构：Start → 1(加载) → 2(反馈处理) → 3(新增设计) → 4(资产写回与落账) → Step End。旧 Step 4/5/6 合并为 Step 4，资产建议反馈（原 Step 5）消除。

### 6. 91 规范修订

91 规范 v2.0 以自闭环为主线，A/B 模型降格为历史参考（wft02~05 仍使用）。新增 §六 变更传播协议。

## 三条路径对照

| | biz | eng | nfr |
|------|-----|------|------|
| STR 类型 | STR-F | STR-E | STR-NFR |
| 原子单元 | IC（信息载体） | CU（配置单元） | NFRI（量化指标） |
| 资产 | 23+26 | 23+25 | 22 |
| 下游 | wft02a | wft02a | wft04a |

## 设计原则

- **无决策点的步骤不应独立存在**：合并 Step 4/5/6 的依据——写回与落账之间无人类交互、无路由分叉
- **状态名表达真实下一步动作**：`可以分解分配` 而非"待进入下一步"
- **入口条件必须是可观察状态**：来自状态表/反馈区，不来自运行提示
- **一次确认覆盖全部**：人类说"同意"即确认了方案设计+资产对准，不应再追问

**Why:** wft01 是 OR→STR 的第一步，三种 OR 类型的处理逻辑差异大（场景定义/引擎配置/NFR 量化），但设计→资产→写回的闭环结构相同。自闭环消除 A/B 交接的人机交互开销，同时保留下游的并行入口。

**How to apply:** wft02~05 若后续考虑从 A/B 迁移到自闭环，直接复用三条 wft01 路径的 Step 结构和生命周期模型。新增 Skill 路径时优先采用自闭环，除非确实需要 A/B 的阶段分离（如 wft05 的详细定义下半）。
