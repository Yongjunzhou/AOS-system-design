# SR-F 架构末级节点识别方法

## 核心命题

SR-F 架构末级节点 = 用户角色类型在使用目标产品时，从"接到一个业务需求"到"该需求被满足"的端到端业务闭环。

## 三步分析法

### 第一步：识别用户角色类型

以"在组织中的业务职责"为分类标准，而非"操作系统的行为是否相同"。三级审批者业务职责不同 → 三个角色类型。

**判断标准**：两个个体在组织中的角色职责是否相同？他们使用目标产品的业务目标是否一致？

### 第二步：按角色类型识别触发，定义端到端业务

对每类角色，逐一识别其"接到什么"作为一个独立的业务触发，从触发到交付定义端到端业务闭环。

**触发需要合适的抽象级别**：
- 太细（按单据类型分四个"审批"）→ 业务行为本质相同，不应拆
- 刚好（"接到待审批事项"作为统一触发）→ 一个端到端业务
- 太粗（把"审批需求"和"异常处理"合在一起）→ 两张皮

**不同角色的端到端业务可能在业务流程上重叠**——采购员的"采购执行"涵盖审批环节，审批者的"采购审批"也把它作为全过程。各自独立、各自完整，在SR-F层共存而非冲突。

### 第三步：用分解分配分析验证候选节点

候选节点的合理性必须经过三个核心设计活动的检查：

**分解检查**：详细定义条目的语义精度是否足够支撑下层设计（如"核对依据材料"太模糊 → 暴露节点内包含不同业务场景）

**分配检查（1:1约束）**：详细定义条目能否按1:1约束分配到唯一BA IPO（同一条目在不同场景需要不同BA IPO → 节点边界不对）

**N:1合理性检查**：节点承接的详细定义条目数 > 8 → 过度合并，需考虑拆分

## 两个关键认识

### SR-F节点与用户角色类型的关系

不总是"一个角色独占一个节点"——一个SR-F节点可能服务于多个角色类型。节点的成立性由分解分配分析验证，而不是从"属于谁"来认定。

### 触发抽象级别

"触发"有不同抽象层级，需要选择对应该角色类型端到端业务的合适级别——以"审批者收到待审批事项"为统一触发，而非按单据类型拆为四个。
