EOS 的产品形态是四层平台架构：

| 层 | 角色 | 活动 | 产出 |
|----|------|------|------|
| 引擎层 | 引擎开发人员 | 开发底层引擎 | 表单引擎、流程引擎、工单引擎、台账引擎等基础能力 |
| 组件层 | 引擎应用用户 | 用引擎做配置 | 表单定义、流程定义、工单定义等业务组件 |
| 模块层 | 业务配置人员 | 编排业务组件 | 各类业务信息系统模块 |
| 使用层 | 业务使用人员 | 使用模块工作 | 完成本职业务 |

**两类需求 → 两条设计链**：

- **引擎开发需求** → 平台设计链（§2.3.2）：要求新增或改进引擎。走完整 OR → SR → BA → SysReq → PA 设计链路，PA 产出引擎构件
- **业务配置需求** → 业务设计链（§2.3.3）：要求使用已有引擎配置业务组件。走五层设计链路，PA 产出配置定义（表单/台账/流程定义等）
- CAX 工具需求 → OR 阶段分流至外部，不纳入平台设计链

**引擎开发需求的两条来源路径**：
1. **原始需求直接提出**——客户/市场直接要求新引擎
2. **业务链推进中折返（需求上升机制）**——业务配置需求在推进中发现引擎能力不足，上升为引擎开发需求，**回到平台 OR**（携带业务链分析上下文），走平台设计链扩展引擎能力。这是平台需求的主要来源之一。

**两条链的关系**：
- 平台链产出的引擎是业务链的能力边界
- 业务链暴露缺口 → 平台补齐能力 → 业务再扩展（持续循环）
- 两条链在各自工作层面上并行运行，互不阻塞

**EOS PA = 引擎清单 + 配置定义清单**。引擎是平台的能力边界，配置是平台的使用方式。

**引擎清单**：

| 类别 | 引擎 | 说明 |
|------|------|------|
| 基础 | 表单引擎、标签页引擎、菜单引擎、台账引擎、字典引擎、主数据引擎、关系引擎 | 数据定义与展现 |
| 流程 | 流程引擎、工单引擎、进展引擎 | 业务流程驱动 |
| 管理 | 项目引擎、WBS引擎、计划引擎、指标引擎、看板引擎 | 项目与绩效管理 |
| 工程 | xBOM引擎、版本管理引擎、MRP运算引擎、MRO算法引擎、追踪引擎 | 工程数据与追溯 |
| 集成 | CAX引擎 | 桥接外部 CAX 工具 |
