# 需求分解与符合性审查中的指标/语义关系

## 核心模式：承载体 ≥ 来源方

两个审查场景遵循同一模式——**承载体（手段）能否满足来源方（目标）的要求**。定性的走语义/逻辑符合性，定量的走指标计算符合性。

所有"父/子"命名均遵循：**父 = 被分解/被满足的源头，子 = 用于实现父的分解产物或承接方**。子（承载体）需要满足父（来源方）。

## 场景一：需求文档中的分解审查

**审查子需求能否满足父需求**——父需求被分解为若干子需求（详细定义末级），子需求们（承载体）需完整覆盖父需求（来源方）的要求。

```
父需求（被分解的源头，来源方）
  │ 分解
  ├─ 子需求 D₁ ── 语义 S₁ + 指标 I_D₁（分解产物，承载体）
  ├─ 子需求 D₂ ── 语义 S₂ + 指标 I_D₂（分解产物，承载体）
  └─ 子需求 D₃ ── 语义 S₃ + 指标 I_D₃（分解产物，承载体）

★ 审查判据：
   语义符合性：S₁ ∪ S₂ ∪ S₃ 覆盖父需求的完整语义（无遗漏、不超出、不矛盾）
   指标符合性：基于子项间的多维语义关系（口径量纲、时序协作、互补约束等），综合判定 I_D₁~I_D₃ 能否满足父指标
```

→ 适用场景：需求规范文档端自身分解质量的验证。

## 场景二：架构文档中的符合性审查（需求符合性审查）

**审查架构节点能否满足其所承接的需求**——架构节点（方案端，承载体）承接了若干需求条目（需求端，来源方），需证明自身能力满足所有来源方的要求。

```
架构节点 Q ──── 功能定义 S_Q + 非功能定义 + 指标 I_Q（承载体）
  ↑ N:1 承接
  所承接的需求们（来源方）
  ├─ 需求 R₁ ── 定义 D₁ + 指标 I_D₁
  ├─ 需求 R₂ ── 定义 D₂ + 指标 I_D₂
  └─ 需求 R₃ ── 定义 D₃ + 指标 I_D₃

★ 审查判据：
   语义符合性：S_Q 覆盖 D₁~D₃ 的所有功能/非功能语义
   指标符合性：基于指标间的多维语义关系（口径量纲、互补约束等），逐组判断 I_Q 是否满足所有 I_Dₙ
```

→ 适用场景：需求符合性审查（审查方案是否满足需求）。

## 两个场景的统一框架

| | 场景一：需求文档分解审查 | 场景二：架构文档符合性审查 |
|---|---|---|
| **承载体（子）** | 子需求（详细定义末级，分解产物） | 架构节点（方案端的承接方） |
| **来源方（父）** | 父需求（被分解的源头） | 所承接的需求条目（需求端） |
| **审查方向** | 子→父：子能否满足父 | 承载体→来源方：节点能否满足需求 |
| **定性审查** | 子语义之和覆盖父语义 | 节点功能定义覆盖需求语义 |
| **定量审查** | 识别子项间/与父指标的多维语义关系（口径量纲、协作模式、互补约束等），综合判定子指标能否满足父指标 | 识别需求间指标的多维语义关系（口径量纲、互补约束等），综合判定节点指标能否满足所有需求 |

## 承接关系的审查维度（需求符合性审查）

在需求符合性审查中，架构节点（父）需满足其所承接的所有需求条目（子）。审查包含三个维度：

### 功能覆盖

架构节点的功能定义是否覆盖了所有承接需求的功能语义？

| 状态 | 示例 |
|------|------|
| ❌ **有遗漏** | 节点承接了R₁"需求录入"和R₂"需求查询"，但节点功能定义只描述了"录入"，未描述"查询" |
| ✅ **完整** | 节点定义中涵盖了R₁和R₂各自的功能语义，无遗漏 |

### 非功能覆盖

架构节点的非功能定义是否不低于所有承接需求的非功能要求？

典型情况：多个需求对同一节点有不同的NFR要求（如R₁要求"可用性≥99.9%"，R₂要求"响应时间<2s"），节点需同时满足全部要求。

### 指标满足

架构节点的性能指标审查，核心方法是通过语义理解识别多个需求指标之间的关系类型，再根据关系类型确定审查方式。**同类指标取最严格值**仅是"同一口径同一维度"这一种关系类型下的审查方式，不能泛化到所有情况。

| 需求间指标关系 | 说明 | 审查方式 | 示例 |
|---------------|------|---------|------|
| **同一口径同一维度** | 多个需求对同一指标的数值要求完全可比 | 取最严格值后验证 | R₁≤2s, R₂≤1s → 节点必须≤1s |
| **同一维度不同口径** | 如95分位 vs 99分位响应时间 | 分别验证，都满足才算通过 | 节点95分位≤2s 且 99分位≤5s |
| **等价表达** | 同一物理量不同单位或表述 | AI 识别等价性后归并验证 | 可用性≥99.9% ↔ 年停机≤8.76h |
| **互补约束** | 不同维度指标共同定义同一能力 | 同时验证，缺一不可 | R₁"1000用户并发" + R₂"日处理10000笔"→节点需同时满足 |
| **独立无关** | 指标分属不同维度，无相互影响 | 逐条独立验证 | R₁响应时间≤2s + R₂存储≥1TB → 各自检查 |

**说明**：场景二与场景一的指标审查遵循统一的范式——AI 通过语义理解识别指标间的关系类型，根据关系类型确定审查方法，而非套用任何单一公式。场景二的"取最严格值"仅是其中一种子情况。

## 场景一补充：需求文档分解的指标符合性

## 需求分解的两个符合性维度

需求分解的符合性审查包含两个**独立**维度：

**维度一：语义/逻辑符合性**——子需求是否正确承载了父需求"做什么"和"不做什么"。

**维度二：指标符合性**——子需求的性能指标是否满足父需求的指标要求（审查方法见"指标关系语义分析与审查方法"一章）。

两者互为正交，但存在交集：当父需求包含语义上的约束条件（如"在XX条件下不应XX"）时，该约束影响子需求的分解边界。

## 语义/逻辑符合性的审查维度

### 语义覆盖完整性

子需求之和应完整覆盖父需求的语义，无遗漏。

| 状态 | 示例 |
|------|------|
| ❌ **有遗漏** | 父"支持需求全生命周期管理"，子只分解为"录入+保存+查询"，遗漏了"关闭/归档" |
| ✅ **完整** | 父"支持需求全生命周期管理" → 子分解为"录入+规范化+分配+确认+关闭+归档" |

### 语义不超出

子需求不应引入父需求未要求的额外功能或约束。

| 状态 | 示例 |
|------|------|
| ❌ **超出范围** | 父"支持需求录入"，子增加"录入后自动发送邮件通知全体成员"（隐含了通知功能） |
| ✅ **边界内** | 父"支持需求录入"，子"录入时验证必填项完整性"（录入的附属验证，合理） |

区分"合理细化"和"范围超出"的判据：子需求是否引入了父需求中未隐含的**新功能类别或新外部接口**。

### 逻辑一致性

子需求之间不应存在逻辑矛盾。

| 状态 | 示例 |
|------|------|
| ❌ **矛盾** | 子1"需求提交后自动生效" vs 子2"需求提交后需审核通过才生效" |
| ✅ **一致** | 子1"需求提交→进入待审核状态" vs 子2"审核通过→需求生效" |

### 负需求继承

父需求中隐含的"不应做"（约束/限制）在子需求分解中必须被正确继承。

| 状态 | 示例 |
|------|------|
| ❌ **负需求丢失** | 父"确保需求数据安全"→子"支持外部系统通过API直接读取需求库"（违反了父的隐含约束"不应未授权访问"） |
| ✅ **负需求继承** | 父"确保需求数据安全"→子"提供基于角色的访问控制，未授权请求应拒绝" |

**负需求的典型来源**：合规要求、安全策略、隐私保护、业务约束（如"不超过现有预算"）等。这些在父需求中往往是隐含的前提条件，在分解时容易被忽略。

## 指标关系语义分析与审查方法

两个场景的指标审查遵循**统一的范式**——AI 通过语义理解识别指标间的多维关系类型，根据关系类型确定审查方法，而非套用任何单一公式。子项间的协作模式、口径一致性、互补约束等都是这个统一框架下的特定关系子类型。

### 指标间语义关系类型（通用框架）

两个场景共享以下指标关系类型：

| 关系类型 | 说明 | 场景一（子↔子、子↔父） | 场景二（需求↔需求） |
|---------|------|----------------------|-------------------|
| **口径/量纲关系** | 指标是否同一口径、是否需要单位转换 | 各子项与父项指标的口径一致性 | 各需求间指标在口径和单位上是否可比 |
| **数值关系** | 同口径指标的可比较关系 | 子项与父项的数值约束（上限/下限/区间） | 需求间同维指标的数值叠加或严格化 |
| **互补/叠加关系** | 多个指标共同构成完整能力要求 | 子维度和父维度的覆盖关系 | 各需求指标共同定义的完整能力边界 |

### 场景一特有的协作关系（原"运算关系 f"表）

场景一在通用关系类型之上，还需考虑子项间的**时序/协作关系**——这是由需求分解的业务语义决定的：

#### 响应时间 / 延迟

| 子项关系 | 运算规则 | 示例 |
|---------|---------|------|
| **顺序执行** | Σ(I_Dₙ) ≤ I_P | 父≤2s，3步顺序→每步≤0.67s 或动态分配 |
| **并行执行** | max(I_Dₙ) ≤ I_P | 父≤2s，3路并行→每路≤2s |
| **选择分支** | max(I_Dₙ) ≤ I_P | 父≤2s，任一分支→每分支≤2s |
| **带概率分支** | Σ(pₙ·I_Dₙ) ≤ I_P | 父≤2s，80%走A(1.5s)+20%走B(3s)=1.8s≤2s |

#### 可用性 / 成功率

| 子项关系 | 运算规则 | 示例 |
|---------|---------|------|
| **串联** | Π(I_Dₙ) ≥ I_P | 父≥99.9%，3组件串联→每组件≥99.97% |
| **冗余（热备）** | 1-Π(1-I_Dₙ) ≥ I_P | 父≥99.9%，双机热备→每台≥99% |
| **复杂路径** | 根据架构计算 | 混合串并联需简化或仿真 |

#### 吞吐量 / 容量

| 子项关系 | 运算规则 | 示例 |
|---------|---------|------|
| **流水线（瓶颈）** | min(I_Dₙ) ≥ I_P | 父≥100tps，瓶颈环节→各环节≥100tps |
| **水平扩展** | Σ(I_Dₙ) ≥ I_P | 父≥100tps，3实例→每实例≥34tps |
| **存储容量** | Σ(I_Dₙ) ≥ I_P | 父≥1TB，3模块→各模块之和≥1TB |

#### 准确率

| 子项关系 | 运算规则 | 示例 |
|---------|---------|------|
| **串联处理** | Π(I_Dₙ) ≥ I_P | 父≥95%，3步识别→每步≥98.3% |
| **投票机制** | 多数正确的概率≥I_P | 父≥99%，5节点投票→少数节点可出错 |

**场景二不涉及协作关系**：各需求指标之间不存在子项间的时序/协作关系，但其他通用关系类型（口径、量纲、互补）仍然适用。

### AI 审查方法

审查时 AI 读取需求内容自动执行以下步骤：

1. **提取所有指标** — 收集来源方的全部性能指标及承载体的声明指标，包括指标值、单位、统计口径
2. **分析指标间语义关系** — 识别各指标间的口径/量纲关系（同一口径还是不同口径）、数值关系（可严格化还是分别验证）、互补/叠加关系（各维度共同定义的能力），以及场景一中的时序协作关系
3. **按关系类型分组** — 同一口径可归一化的归为一组，不同口径的单独列出，等价表达（如 99.9% ↔ 年停机 8.76h）转换后归并
4. **逐组验算** — 对每组指标按对应的审查方法计算验证
5. **输出结果** — 清晰标识每个指标组的验证结果及不通过项的原因

如果指标间关系无法从文本中明确推断，文档应标注关系说明，或审查时标注为"需人工判定"。

### 领域/专业性补充

实物产品（如光学系统、机械结构、电气系统）中的指标关系分析还需要领域知识：
- **领域特有的维度关系**：光学系统的分辨率/亮度均匀性/色域覆盖率涉及光路计算，需领域知识判断这些维度是独立、互补还是冲突
- **领域特有的量纲转换**：机械公差链中的 RSS（平方和根法）是领域特有的累积计算方法
- **领域特有的协作模式**：热设计中多个热源的散热路径协同关系

这些关系类型在上述通用框架中仍适用，但其具体的计算方法和判定标准需要按领域另行补充。**两个场景都可能涉及此类领域知识**——不仅是场景一的分解运算，场景二的需求间指标关系分析同样需要领域知识来判断口径和比较方法。

## 适用层级

**场景一（需求文档分解审查）**——父子指标分解发生在以下层级（同层内部）：

| 层级 | 父指标位置 | 子指标位置 | 指标关系的核心依据 |
|------|-----------|-----------|-----------------|
| OR → OR末级 | OR条目 | OR详细定义末级 | 末级间的业务时序关系 + 口径一致性 |
| SR架构末级→详细定义末级 | SR架构末级节点 | SR详细定义末级 | 分解后需求间的协作关系 + 指标语义关系 |
| SysReq 5级→6-9级 | SysReq 5级(架构末级) | SysReq 9级(详细定义末级) | 场景活动间的执行流程 + 指标语义关系 |
| PA构件→PA详细定义 | PA构件(架构末级) | PA详细定义末级 | 构件内部的实现步骤关系 + 指标语义关系 |

**场景二（架构符合性审查）**——跨层承接关系中的指标审查：架构节点（承载体）承接多个需求条目（来源方），审查核心是指标间的口径/量纲关系分析和互补约束验证，不涉及时序协作关系。

## 范围说明

上述指标类型与关系表覆盖的是**信息系统/知识类产品**通用的指标类型（响应时间、可用性、吞吐量、准确率等）。实物产品（如光学系统、机械结构、电气系统）存在大量领域专用的指标关系——例如光学系统的分辨率/亮度均匀性/色域覆盖率涉及光路计算，机械结构的公差链分析涉及RSS（平方和根法），这些规则不在当前表格范围内，需按领域另行补充。两个场景均需考虑此类领域知识。
