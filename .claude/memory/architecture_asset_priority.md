当新增上层需求需要分配给下层方案时，架构末级节点的变更遵循以下优先级：

1. **Reuse（复用）** — 现有节点已完整覆盖需求，无需变更，直接建立映射
2. **Improve（改进现有资产）** — 通过 Extend（扩展）、Split（拆分）、Merge（合并）使现有节点承接
3. **Add（新增）** — 仅当复用和改进都不可行时才新增节点
4. **Deprecate（废弃）** — 标记废弃，不物理删除，维护追溯链完整性

**Why:** 每个架构末级节点都是组织资产——承载了已投入的设计工作（四要素定义）、已建立的下游映射关系（N:1承接）和已通过的验证记录。新增节点意味着后续的设计、维护和验证成本。

**How to apply:** 在架构增量变更决策中，优先检查现有架构能否复用。同一节点经历3次以上Extend后应审查是否需要Split。模式/领域差异是不可逾越的边界，此时直接走Add。废弃节点不物理删除。

**关键工作流顺序**：每分解产生一条新需求 x，先在本需求文档内自治分析（可复用性/冲突性/规范性），再去下游方案架构中按资产优先原则审查。此顺序防止方案结构扭曲需求定义。
