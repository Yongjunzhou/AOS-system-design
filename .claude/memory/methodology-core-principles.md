# 方法论核心内核（5条，自 2026-05-24）

以下五条是本方法论的核心内核，适用于所有设计活动和文档工作。

### 1. 文档的相对角色

"需求"和"方案"不是贴在文档上的固定标签，而是文档在**设计链中相对于相邻文档扮演的角色**。同一份文档——比如 SR——在对 OR 的结对中扮演方案（承接 OR 的需求），而在对 BA 的结对中扮演需求（被 BA 承接）。不存在"纯需求文档"或"纯方案文档"这种固有分类，只有"此刻在这段结对关系中充当哪一方"。

中间层的文档（SR、SysReq）同时兼有两种角色。也因此，说"需求文档也需要架构定义"才是自然的——因为它同时也要作为方案被下层承接。

### 2. 文档原则上由架构定义和详细定义两个子文档构成

不论是扮演需求角色还是方案角色，一份完整的文档原则上包含两个子文档：架构定义文档（树形结构，建立映射）和详细定义文档（对末级节点逐条细化）。两者职责分离——一个回答"结构是什么、承接了谁"，一个回答"具体怎么做、要分解到什么粒度"。

特例：OR 在最上游，只有需求无架构定义；BA 只有架构定义无详细定义（独立存在的分析层）。

### 3. 结对设计

结对不是发生在两层文档整体之间，而是发生在**上层需求的详细定义末级条目**与**下层方案的架构定义末级节点**之间。这是设计链中每一次"需求→方案"传递的基本单元。

结对设计要求同步设计，不能先完成上层全部再开始下层——否则上层分解到末级后可能发现下层没有合适的架构节点来承接，导致返工。操作上两阶段：先骨架（约定结构+建立映射），再填充（各自并行细化内容）。

### 4. 设计 = 需求的持续分解、分配与分析

系统设计的实质不是"画架构图"，而是对需求进行**分解**（语义切分 + 指标切分，直到每条末级需求可被一个下层架构节点承接）、**分配**（按 1:1 约束将末级需求逐一映射到下层架构末级节点）、**符合性分析**（验证每个架构节点是否充分满足所承接的需求集合——语义上覆盖、指标上不降级）。

三个活动顺序执行，每完成一个立即自检，而非全做完再回头检查。这个"分解→分配→分析"的循环贯穿五层设计链路的每一段结对。

### 5. 基于资产定义架构

架构节点不是凭空产生的，而是基于组织现有资产进行识别、选择、改进或新增而生成。各层（SR、BA、SysReq、PA）的架构末级节点都应当对应组织已有或需要构建的能力单元，遵循**复用 → 改进 → 新增**的优先顺序。
