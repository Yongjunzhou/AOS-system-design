## 背景

2026-06-25，在完成 v2.3 标注简化后，对 wft02-eng §3.2（设计各配置单元）做了深入分析。这是本 Skill 的核心设计步骤。讨论未达结论，需后续继续。

## 数据流澄清

```
wft01-eng Phase C·3 (CU 识别)
  → 产出：CU 类型清单（名称级 + 一句话能力需求说明）+ 顺序 1/2/3 判定
  → 写入 STR-E 节点
       ↓
wft02-eng Step 3.2 (设计各配置单元)
  → 消费：上述 CU 类型清单 + 25 资产 CU 类型定义
  → 产出：每个 CU 的完整 I/P/O + 25-Q 判定
  → 写入 BA 节点
```

wft01-eng 做「识别」——触及了哪些 CU 类型。wft02-eng 做「设计」——基于 25 类型模板 + 具体能力需求，完整展开 I/P/O。两者是不同性质的工作。

## 当前 A/B/C 三段式结构评审

| 段 | 职责 | 评价 |
|----|------|------|
| A. 多 STR-E 收敛检测 | 新旧 CU 清单比对，五关系判定 | ✓ 结构合理 |
| B. 引擎内置 CU 处理 | 25 中预定义但未触发的 CU 占位保留 | ✓ 填补缺口 |
| C. 三要素设计与 25 对齐 | 对需设计的 CU 完整设计 I/P/O + 25-Q 判定 | ⚠ 核心步骤，需深入 |

## 6 个待讨论问题

### 问题 1：设计颗粒度——"完整三要素"到底多完整？

25 类型定义已给出类型级 I/P/O 模板。wft02-eng 的实例级设计应增加三样东西：
1. **业务上下文具体化**：不列全字段，但刻画字段模式/类别
2. **能力方向聚焦**：沿 STR-E 能力需求方向展开细节
3. **I→P→O 实例级因果链**：这个具体场景下的完整链路

待讨论：应更粗（仅确认类型模板适用）还是更细（逐字段罗列）？

### 问题 2：设计方法论——认知步骤拆解

隐含的 6 个推理步骤：
C1. 读 25 CU 类型定义 → 理解 I/P/O 模板
C2. 读 STR-E 能力需求说明 → 理解具体场景需求
C3. 交叉比对 → 类型模板的适用性（→ 25-Q 判定）
C4. 实例化 I/P/O：模板维度 × 具体场景
C5. 记录依赖配置单元（从 25 类型定义继承）
C6. 标注来源类型

待讨论：是否需要在 3.2 中显性化这些步骤？C3 和 C4 是先后还是交织？

### 问题 3：25-Q 判定粒度标准

建议以 I/P/O 三维度的"结构匹配度"为准：
- 三维度结构均匹配 → Q1
- 至少一个维度匹配，至少一个需扩展 → Q2
- 三维度均不匹配 → Q3

wft01-eng 的顺序 1/2/3 是更粗粒度的资产覆盖判定，不等同于 25-Q。

待讨论：是否应更明确判定标准，还是保持 AI 自行判断灵活性？

### 问题 4：A 收敛检测 → C 设计的信息传递

当 A 判定"补充"→ 进入 C 重新设计时，原有 I/P/O 的处理方式：
- 选项 A：全量重新设计（推荐——保证内部一致性）
- 选项 B：增量追加
- 选项 C：比对后合并

### 问题 5：CU 依赖影响设计内容

依赖关系不仅影响排序，还应影响设计内容——被依赖 CU 的 Output 应回引到依赖 CU 的 Input/Process 中。待讨论是否在 C 中显性加入"依赖 CU Output 回引"步骤。

### 问题 6：引擎内置 CU 占位的生命周期

- 占位 CU 在 BA 方案中以什么形态呈现（仅名称+`[待触发]`标记的空条目？）
- 后续触发时来源升级路径（`引擎内置`→`OR明示` 还是 `引擎内置`→`OR隐含`？取决于新 STR-E 的触及方式）
- 长期不触发时是否在 Step End 提示

## 初步判断

问题 1（设计颗粒度）和问题 2（设计方法论）是最核心的——它们决定"设计"到底是什么、怎么做。其他 4 个问题可在核心问题有结论后自然推导。

另外，当前 3.2 缺少**具体设计示例**——一个贯穿 A→B→C 的完整示例会极大提升可操作性。
