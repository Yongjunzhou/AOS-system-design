## Step 3 · SR 详细定义分解与 BA 承接（核心改造）

### 六步分解流程

对每条 STR 架构末级（SR-F-XXX）的功能说明，执行：

```
┌─ ① 确定业务域（1级）
│   输出：业务域 = {采购/生产/售后/...}
│
├─ ② 匹配项目类型 = 产品类型（2级）
│   从 Step 1 加载的 23 路径索引中匹配
│   输出：@prod-{id} · {产品类型名称}
│
├─ ③ 匹配架构节点类型端到端业务（3级）
│   从 Step 1 加载的 23 路径索引中匹配
│   输出：@node-{id} · {架构节点类型名称}
│
├─ ④ 前置判断 + {链×模式×域} 原子化分解
│   ├─ 前置判断：涉及几个 BA 职责范围？
│   └─ 按设计链归属 × 信息处理模式 × 业务域 分解到原子单元
│   输出：SR-F-XXX.1 ~ SR-F-XXX.N（各带链标签）
│
├─ ⑤ 归入 WBS 包（4级，可选）
│   同级 IPO > 7 → 按过程阶段分组
│   IPO ≤ 7 → 省略 WBS 层
│
└─ ⑥ 找 BA IPO 宿主（5级）
    在已确定的 {1级→2级→3级→4级} 路径内执行 Q1/Q2/Q3：
    ├─ Q1：该步骤与路径内哪个组件的职责方向匹配？
    ├─ Q2：具体内容是否在组件边界内？
    └─ Q3：该组件是否已有同类承接记录？

    决策：
    ├─ ✅ 复用 → 关联到现有 24/25 条目
    ├─ ⚠️ 扩展 → 更新 24/25 条目定义
    ├─ ⚠️ 新增 → 按链路写入 24-*.md 或 25-*.md
    └─ ❓ 待裁定 → 记录路径位置，标记待人类确认
```

### 层级路径记录格式

```
{业务域} / {产品类型@id} / {架构节点类型@id} / {WBS 包} → {组件@id}
```

## v3.0 最终实施结构

实施中决定采用 **三个独立资产管理 Step** 模式（对齐 wft01）：

```
Step Start · 前置校验
Step 1 · 资产加载（23 路径索引 + 24/25 组件清单 + 04 BA 资产）
Step 2 · 分析活动
Step 3 · 五步层级导航分解（不含资产管理——Q1/Q2/Q3 已移出）
Step 4 · BA IPO 四要素完善（定义、I/P/O、符合性分析、设计指导，不写资产）
Step 5 · 24 业务组件资产管理（业务链 BA IPO → Q1/Q2/Q3 路径内匹配 → 直接写入 24-*.md）
Step 6 · 25 引擎组件资产管理（平台链 BA IPO → Q1/Q2/Q3 路径内匹配 → 直接写入 25-*.md）
Step 7 · 23 输出产品架构资产联动（全量 I/O 载体 → 匹配 23 → 直接写入 23-*.md）
Step 8 · 04 BA 架构写回（聚合五级架构树 + WBS + 流转清单 + 状态表）
Step End · 输出交互
```

三个资产管理 Step 各自读源→匹配→写资产自成闭环，无独立审查 Step（审查归入 Step End）。
