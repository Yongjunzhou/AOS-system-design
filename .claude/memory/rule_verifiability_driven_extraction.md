# 以可验证性驱动的规则提取

## 原则

规则提取不是在做"文档阅读理解"（总结文档在说什么），而是在做"可验证性分析"（找出文档中可独立给出 pass/fail 判断的约束单元）。提取的单位不是句子、不是段落，而是**约束判断单元**。

## 三条底层逻辑

1. **审查单元应当等于判断单元，而非段落单元**——可独立判断的约束语句 → 按同一判断焦点聚类成段落 → 逐个判断约束语句 → 汇总为段落级结论。不应反过来：段落 → 段落包含规则 → 批量判断。

2. **颗粒度偏差来自流程结构，而非 AI 能力不足**——不是要求"AI 更努力"，而是在流程层面增加**必须执行的拆解检查点**，让不拆变成需要理由的决策。

3. **颗粒度失控点在下游发现已太晚**——颗粒度检查应尽量前移（左移），在提取阶段就设阀门，而不是等语义文档生成后再重构。

## 应用位置

已在 SKILL.md 总原则中追加第一条：
> 以可验证性驱动提取——提取的单位是约束判断单元。

后续如需修改具体步骤（Step 0a-1 拆句、Step 0a-3 聚类准则收缩、内嵌规则独立编号、决策树调整等），以此总则为判断依据。

## 与原有方法论的关系

原有[[rule_granularity_methodology]]提供的是颗粒度的**判定工具**（决策树、异常信号），本原则提供的是提取的**认知范式**（为什么而提取）。两者互补——前者回答"拆不拆"，后者回答"找什么"。

**Why**: 通过分析规则符合性审查的语义提取产物发现，提取出的规则颗粒度整体偏粗，根因在于流程设计默认"不拆"，且提取范式是阅读理解式的段落总结而非可验证性驱动的约束单元分析。

**How to apply**: 每次执行规则提取时，先记住"我要找的是能独立给出 pass/fail 的约束单元"，然后再用决策树判断颗粒度。决策树从"这个句子是不是规则"转变为"这个约束单元是否可独立判断"。
