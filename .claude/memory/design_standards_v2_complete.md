## 核心改进

### 1. 架构末级节点定义明确化

**定义**：被分类和组织的最大颗粒度的需求/方案条目

**特点**：
- 是实际的条目，不是分类结构本身
- 是某一层级中最大的、不再向上聚合的单位
- 是下层映射的关键层级

### 2. 详细设计末级节点定义明确化

**定义**：对架构末级节点的进一步分解，直到只需一个下层方案架构末级节点承接

**特点**：
- 是对架构末级节点的细化和展开
- 分解标准：一直分解到任意末级条目都只需找到后续文档架构的一个末级节点来承接即可
- 是1:1映射的源头

### 3. 5层结构确立

```
第1层：原始需求及分析
    ├─ 架构末级节点：末级需求
    └─ 详细设计末级节点：（无，原始需求本身就是末级）

第2层：相关方需求
    ├─ 架构末级节点：相关方需求架构末级（被分类和组织的最大颗粒度的相关方需求条目）
    └─ 详细设计末级节点：相关方需求详细设计末级（对架构末级的进一步分解）

第3层：业务架构
    ├─ 架构末级节点：IPO（业务组件）
    └─ 详细设计末级节点：I、P、O详细说明

第4层：系统需求
    ├─ 架构末级节点：系统需求架构末级（功能/非功能需求分类）
    └─ 详细设计末级节点：系统需求详细设计末级（对架构末级的进一步分解）

第5层：产品架构
    ├─ 架构末级节点：产品组件
    └─ 详细设计末级节点：实现细节
```

### 4. 映射关系完整化

**N:1映射**（多条需求映射到一个方案条目）：
- 原始需求末级 → 相关方需求架构末级（N:1）
- 相关方需求详细设计末级 → 业务架构末级（N:1）
- 业务架构详细设计末级 → 系统需求架构末级（N:1）
- 系统需求详细设计末级 → 产品架构末级（N:1）

**1:1约束**（每条需求只映射到一个方案条目）：
- 相关方需求详细设计末级 → 业务架构末级（1:1）
- 业务架构详细设计末级 → 系统需求架构末级（1:1）
- 系统需求详细设计末级 → 产品架构末级（1:1）

**1:N分解**（一个架构末级分解为多个详细设计末级）：
- 相关方需求架构末级 → 相关方需求详细设计末级（1:N）
- 业务架构末级 → 业务架构详细设计末级（1:N）
- 系统需求架构末级 → 系统需求详细设计末级（1:N）
- 产品架构末级 → 产品架构详细设计末级（1:N）
