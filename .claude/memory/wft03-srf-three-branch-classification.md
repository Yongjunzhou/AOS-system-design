# wft03 SR-F 分类重组：三分支模式

**日期**：2026-06-16  
**上下文**：推进 eos-wft03，讨论从 BA IPO 生成 SysReq（SR-F）时 0-4 级分类骨架如何组织。

## 背景

wft02 v5.3 产出三类 BA IPO（Bn 业务事务 / A1 业务引擎 / 角色场景·编排定义），经去重+视角转换后，1:1 映射为 SR-F 5 级节点。但 BA 的层级路径（业务域→产品类型→端到端业务→WBS）是业务视角，不能直接作为 SR-F 分类骨架。需要解决"已知 5 级节点清单怎么组织成树"的问题。

## 决策：三分支模式

参考现代 SaaS（Salesforce/Workday 等）的"业务功能+平台服务"模式，结合 EOS 三类 BA IPO 的天然分型，SR-F 采用三个 1 级分支：

```
EOS 系统需求（功能部分）
│
├── 业务功能          ← 来自 Bn 业务事务 BA IPO
│   ├── 采购管理      ← 2 级 = 业务域
│   │   ├── 询价至下单 ← 3 级 = BA 的端到端业务
│   │   │   ├── [5] 询价供应商匹配功能
│   │   │   └── [5] 报价归集与核验功能
│   │   └── 供应商管理
│   │       └── [5] …
│   ├── 生产管理
│   └── 财务管理
│
├── 平台服务          ← 来自 A1 业务引擎 BA IPO
│   ├── 流程与审批服务 ← 2 级 = 引擎类别（扁平化）
│   │   └── [5] 审批流转执行功能
│   ├── 规则引擎服务
│   ├── 报表与分析服务
│   └── 通知与通信服务
│
└── 工作空间          ← 来自 角色场景·编排定义 BA IPO
    ├── 角色工作台    ← 2 级 = 场景类别
    │   ├── [5] 采购员工作台功能
    │   └── [5] 车间主任工作台功能
    ├── 待办与任务中心
    └── 系统导航与个性化
```

## 分支归属判定规则

SR-F 5 级节点的 1 级分支归属按下述次序判定：

```
BA IPO（附带产品类型 + 跨域引用信息）
    │
    ├─ 产品类型 = A1 业务引擎 ──────────────→ 平台服务
    │
    ├─ 产品类型 = 角色场景·编排定义 ─────────→ 工作空间
    │
    └─ 产品类型 = Bn 业务事务
            │
            ├─ 被多个业务事务引用（跨域）？──是──→ 平台服务
            │
            └─ 否（单一域特有） ──────────→ 业务功能（按业务域归入）
```

**跨域引用规则**：尽管 wft02 的 {产品类型×模式×域} 原子化分解保证了每个 BA IPO 的唯一产品类型，但去重后的 BA IPO 本质上可能被多个业务域引用（如审批流自动化）。此时应将其提升到"平台服务"分支，而非强留在单一业务域下。

**产品类型继承**：三类 BA IPO 的产品类型在 wft02 分解时确定，wft03 视角转换后直接继承，无需额外仲裁。因此不存在"同一个 BA IPO 同时属于两个分支"的边界模糊——产品类型已经唯一决定了归属。

**唯一例外**：Bn 业务事务如果跨域引用，则提升到平台服务分支。这是解决"每个域自建一套审批"工程反模式的机制。

## 设计原则

1. **产品类型自动分流**——BA IPO 的产品类型（Bn/A1/角色场景）决定流入哪个 1 级分支
2. **跨域引用提升**——被多域引用的 Bn 类 IPO 提升至平台服务
3. **内部层级复用资产**——业务功能分支的 2-4 级与 BA 路径 / 23 号资产层级对齐；平台服务和工作空间扁平化
4. **独立演进**——三个分支变更互不干扰，平台能力可独立升级

## SR-F 5 级节点属性定义

架构末级节点格式三个分支统一，不按分支差异化：

| 属性 | 说明 | 来源 |
|---|---|---|
| **节点ID** | SR-F-{SEQ} | wft03 分配 |
| **节点名称** | 系统功能名称 | BA IPO 名称经视角转换 |
| **所属分支** | 业务功能/平台服务/工作空间 | 按产品类型+跨域规则判定 |
| **视角转换定义** | 系统语言的 I/P/O，人机分工明确 | BA IPO 经四步转换 |
| **承接的 BA IPO** | 所承接的唯一 BA IPO 编号 | 1:1 映射 |
| **符合性分析** | 如何满足所承接的 BA IPO | BA IPO 承接分析经视角转换 |
| **性能指标** | 从上层继承的性能指标 | STR → BA → SR 继承 |
| **设计指导** | 对 PA 构件的方向提示 | BA IPO 设计指导经视角转换 |
| **详细定义初步方案** | 系统功能内部的更细结构，作为 t05 详细定义的起点 | 从 BA 的"业务定义初步方案"迁移并转换 |

**关键设计**：wft02 的 BA IPO 已包含"业务定义初步方案"（流程定义含节点/流转，工单定义含生命周期状态，规则定义含规则项等）。wft03 不应丢弃此信息，而应将其迁移为 SR-F 节点的"详细定义初步方案"——即 6-9 级详细定义的骨架。这样 t05 拿到 SR-F 节点时不是从零开始展开详细定义，而是在初步方案的基础上细化。

## 下午转向：按业务定义类型分类

**关键转折**：下午讨论中，三分支被质疑是否还应该作为 SR-F 输出的结构标签。用户提出全新的分类方向：

> **系统功能需求架构 = 对业务定义资产的分类方案 + 业务定义作为末级节点**

这意味着：
- SR-F 的 1 级节点不是"业务功能/平台服务/工作空间"，而是**业务定义类型**（流程定义、工单定义、计划定义、台账定义、规则定义、报表定义、角色编排等）
- 每个 BA IPO 本来就有业务定义类型属性（wft02 标记的），视角转换后自然归属
- 三分支作为内部路由规则仍然用来判断 BA IPO 去向，但**不输出到 05 文档作为标签**

### 层级初步设想

```
SR-F:
1 级 = 业务定义类型（流程定义、工单定义、计划定义……）
2 级 = 业务域
3 级 = 产品类型 / 角色类型
4 级 = ? （待讨论）
5 级 = SR-F 功能节点（视角转换后的 IPO）
```

**待解决的问题**：
- 各业务定义类型的完整清单
- 每种类型内部的 2-4 级分类维度是否统一，还是各从其类
- 4 级的指代是什么
- SR-F 的非末级节点指代"系统能力聚合"，非末级节点不承诺业务流程完整性

## 已确认的剩余事项

- **引用关系**：编排 SR-F 节点不需要在 SR-F 层维护跨分支引用，那是 t05 详细定义阶段的事
- **派生 STR**：节点级迭代（通用规范 §2.3.4），已达 `待架构映射` 的 BA IPO 可独立进入 wft03，无需等待派生 STR 的 wft02 循环完成
