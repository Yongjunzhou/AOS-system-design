# 文档符合性审查 — 规则符合性审查（Rule Conformance Review）

> 与需求符合性审查（Requirements Conformance Review，审查方案是否满足需求）相区别，本技能聚焦于**规则符合性审查**——审查目标文档是否符合规则文档中定义的规则（rule）。

## 总原则

- **以可验证性驱动提取**——提取规则不是在做"文档阅读理解"（总结文档在说什么），而是在做"可验证性分析"（找出文档中可独立给出 pass/fail 判断的约束单元）。提取的单位不是句子、不是段落，而是**约束判断单元**。
- **依据条目即时生成**——审查清单不是预设的，而是在审查启动时根据加载的规则文档即时生成的
- **一次只用一个语义段落**去审视全文，每轮只带一个"判断透镜"（rule lens）
- **禁止跨轮累积语义**——每轮只查本轮对应的规则（rule）
- **先输出发现，等确认后再修改**

## 审查流程

### Step 0a：加载规则文档，执行三遍语义提取

规则文档加载后，通过三遍递进提取（替代原五遍），从不同维度完整捕获文档语义。每遍聚焦单一认知活动类型——阅读提取、推理分析、编排组装，避免在同一遍中混合不同类型的认知负载。

**多规则文档场景**：若存在多份规则文档，每份文档独立执行三遍提取，各自生成独立的实体字典和语义段落（内嵌规则）。语义段落编号按文档区分前缀，例如规则文档 A 用 `A-P01`、规则文档 B 用 `B-P01`。报告引用时才加方括号（如 `[A-P01]`、`[A-P01-R01]`）。

#### Step 0a-1：宏观骨架 + 规则清单提取（一次通读）

目标：一次通读完成两项提取任务——宏观骨架（文档结构/概念图谱/范围）和规则清单（关键词模式驱动的逐条提取）。

**任务一：宏观骨架提取**

| 维度 | 提取内容 |
|------|---------|
| 文档结构 | 章节目录及每节目的 |
| 概念图谱 | 文档定义了哪些主要概念/实体 |
| 范围声明 | 文档适用范围、适用场景 |
| 外部引用 | 引用了哪些其他文档 |

**输出**：概念清单 + 结构层级图（记录在工作笔记中）

**任务二：规则清单提取（模式驱动）**

用关键词模式扫描全文档，**不分段落，逐条提取**所有约束性语句。

关键词模式与约束强度：

| 模式 | 约束强度 |
|------|---------|
| `必须` / `应` | **强制** |
| `不得` / `禁止` / `不应` | **禁止** |
| `建议` / `宜` / `推荐` | **建议** |
| `可` / `可以` / `允许` | **可选** |
| `如果...则...` / `当...时` | **条件约束**（叠加在以上强度上） |

提取结果示例：

```
R001 [强制] 每条详细定义末级节点只能分配到下层方案中唯一一个架构末级节点
R002 [条件·强制] 当 BA 包含多个 IPO 时，必须建立 IPO 之间的交互关系
R003 [建议] 架构末级节点的命名应反映其承担的职责
```

**补充：候选规则复合性检查**

关键词扫描提取的是约束性语句，但一条语句可能包含多个可独立判断的约束子句。提取完成后，对每条候选规则执行复合性检查：

1. 扫描规则文本中是否包含"、"、"；"、"以及"、"并"、"和"等并列连词
   - 若包含，将连词处断开为若干子句
2. 对各子句执行"独立判断焦点"检查：
   - 两个子句的审查检查位置是否相同？
   - 审查时是否需要在文档的不同部分、用不同的判断角度验证？
3. 若各子句属于不同判断焦点 → 拆分为多条独立规则，各自分配独立 R 编号
4. 若各子句共同定义一个完整的单一规范（如"编号格式=前缀+序号+位数"），拆分后单独审查缺乏独立意义 → 保持为复合属性，不拆

**输出**：结构化规则清单（编号 + 强度 + 条件 + 内容 + 来源章节）

**补充：规则颗粒度判定指南（Rule Granularity Guide）**

提取规则时，颗粒度不当会导致审查时无法给出清晰的 pass/fail 判定。以下判据和信号用于判断颗粒度是否合适。

**核心判据**：不是"子句能否独立违反"，而是"审查时两个子句是否在同一次文档遍历中、用同一个判断焦点检查"：

| 类型 | 特征 | 例子 | 处理 |
|------|------|------|------|
| **复合属性** | 多个子句共同定义一个完整概念或规范 | "序号使用3位数字，起始001"、"构件1-2周可完成且可独立部署和测试" | **不应拆分**——拆开丢失语义完整性 |
| **独立维度** | 不同子句定义不同维度的验收项，检查对象不同 | "架构末级节点定义完整率100% + 架构层级结构完整率100%" | **应当拆分**——各自有独立的检查对象和遍历路径 |
| **条件-动作对** | "如果X，则Y" | "同步设计场景下，开展上层详细定义的同时同步建立下层架构骨架" | **不应拆分**——条件和动作是一体的 |

**三个颗粒度异常信号**：

1. **规则中包含"且"、"并"、"以及"**——判断它是否是复合属性（如"无遗漏、无冗余"是同一遍历的正反两面，不拆）；如果是不同检查对象（如"架构定义完整且详细定义分解完成"涉及两份文档），应拆
2. **两个子句出自不同章节、涉及不同层级的文档**——拆分信号
3. **审查时两个子句需要不同的判断焦点**——需要考虑拆分，但先确认是否属于同一分类体系（如"四大分类"拆为四条规则反而荒谬）

**实际提取决策树**：

```text
一条候选规则，包含子句A和子句B
├─ A和B是否在文档中不同的位置/不同的文档中检查？
│  ├─ 是 → 拆分
│  └─ 否 → 继续
├─ A和B是否属于同一个判断焦点？
│  ├─ 否 → 考虑拆分
│  └─ 是 → 继续
├─ 分开后A单独审查是否缺乏独立意义？
│  ├─ 是 → 不拆
│  └─ 否 → 继续
└─ 拆分为两条规则后，审查时是否仍需要在同一次文档遍历中、用同一个判断焦点检查？
   ├─ 是 → 不拆（两条规则的检查路径完全重叠，拆分只会增加轮次而无收益）
   └─ 否 → 可拆（各自需要独立的检查路径，不应因"怕增加轮次"而合并）
```

#### Step 0a-2：实体字典 + 约束关系映射（推理）

目标：基于 0a-1 提取的清单，进行推理密集型分析——提取概念定义形成实体字典，识别规则-实体关联和隐含约束。此步是纯推理活动，不涉及文档阅读。

**实体定义提取**

提取文档中所有概念性定义，形成实体字典。

```
实体：架构末级节点
定义：在树形结构中位置最底层、且下属无子节点的节点
属性：名称、父节点、所映射的上层需求、符合性分析、下游指导
关联段落：P03, P07, P11
```

**输出**：实体字典（实体名 + 定义 + 关键属性 + 关联段落）

**约束关系映射**

将规则和实体关联起来，识别以下关系：

1. **规则-实体关联**：每条规则约束的是哪个概念
2. **规则间依赖**：规则 A 是否引用规则 B 中定义的概念或方法
3. **隐含约束**：从多个定义和规则的关系中可以推导出的未显式陈述的约束

示例——若多处出现变量命名规范但没有单独定义"变量名格式"，则从"追溯矩阵通过编号关联"可推导"需求必须编号"虽未直接陈述但属于隐含约束。

**多规则文档场景**：增加一层跨文档关系识别：

4. **跨文档引用**：规则文档之间的交叉引用（如设计指南引用了准则中的某条定义）
5. **冲突检测**：不同规则文档之间是否存在相互矛盾的规则。检测到的冲突标记为 **⚠️ 跨文档冲突**，在审查报告中豁免目标文档（目标文档无法同时满足两份矛盾文档的要求）

**输出**：规则-实体关联矩阵 + 隐含约束清单 +（多文档场景）跨文档引用清单 + 冲突清单

#### Step 0a-3：语义段落组装（编排）

目标：基于 0a-1 的规则清单和 0a-2 的实体字典/关联矩阵，将规则按语义聚类为"一个判断透镜"——即语义段落。此时的分段不再依赖原文章节边界，而是基于规则内容的自然聚合。此步是编排密集型活动（组织/聚类/标注），不涉及新的信息提取。

原则：
- **聚类层级与优先级**：
  - **第一优先：通读路径**——先问"这些规则在审查时能否在同一次文档遍历中检查？"（即读文档同一个位置、同一遍就能覆盖全部）。通读路径不同则必须分属不同段落
  - **第二优先：判断焦点**——再问"它们是否用同一审查问句来判断？"通读路径相同但判断焦点不同的规则，可合并在同一段落，但必须通过独立编号（`P0X-RXX`）保证逐条判断；段落标题中 `——` 后的描述（焦点问句）仅作聚焦之用，不可替代规则级判据
  - **第三参考（最低优先）**：原文章节位置，仅供参考，不作聚类依据
  - 段落组装完成后自检：当前段落是否可以用一个"焦点问句"概括？若无法用一个焦点问句覆盖所有内嵌规则的检查位置，则说明颗粒度过粗，需拆分段落
  - ⚠️ 以下场景虽然"关于同一实体"，但判断焦点不同，不应合并：
    - 规则 X 约束需求描述的格式（检查：描述是否符合格式要求）
    - 规则 Y 约束需求的分配路径（检查：分配路径是否正确）
    → 判断焦点不同，通读路径也不同，应分属不同段落
- 不同类型的约束（定义 vs 格式 vs 原则）→ 即使在同一章，也要拆分
- **内嵌规则分配独立编号**（格式：`P0X-RXX`），每条内嵌规则在审查时逐条聚焦判断，段落标题中的焦点问句不可替代规则级的 pass/fail
- **内嵌规则的颗粒度遵循〈规则颗粒度判定指南〉（Step 0a-1）和 Step 0a-1 的候选规则复合性检查结果**——复合属性合并为**一条**内嵌规则，独立维度才分别列出

段落类型标注：

| 标签 | 含义 | 用途 |
|------|------|------|
| **D** | 定义 | 概念、术语、规则的定义 |
| **P** | 流程 | 操作步骤、方法流程 |
| **F** | 格式 | 格式规范、编号规则 |
| **G** | 原则 | 设计方向、权衡决策、指导性思想 |
| **S** | 范围 | 范围界定、各层定位 |
| **Q** | 质量 | 验收标准、质量指标 |
| **A** | 反模式 | 常见错误、禁忌 |
| **E** | 示例 | 规则难以文字化时，以示例作为规范标准（仅规则模糊或不明确时使用） |
| **N** | 导航 | 交叉引用、关系矩阵 |
| **R** | 参考 | 参考信息、FAQ |

**输出**：语义段落清单——按**结构化块**格式组织，每段独立列出（段落编号 + 类型 + 焦点问句 + 内嵌规则列表 + 完整原文段落 + 来源章节）。规则内嵌在所属段落中，完整原文段落按来源独立分段，不再保留独立的规则清单节。

### Step 0b：生成临时语义提取文档

将前三步的三项核心输出（实体字典 + 语义段落（内嵌规则） + 宏观骨架信息）合并为一份语义提取文档，写入临时目录。

**路径格式**：

```
.claude/skills/rule-conformance-review/temp/{源文档文件名前缀}-semantics.md
```

例如依据 `01-general-system-design-specification.md` 则生成 `temp/general-design-specification-semantics.md`。

**多规则文档场景**：每份规则文档生成独立的临时语义提取文档，例如：

```
temp/
├── general-design-standards-semantics.md     # 规则文档A
├── pipeline-design-standards-semantics.md    # 规则文档B
└── waterfall-design-guide-semantics.md       # 规则文档C
```

段落编号按文档分配独立序号，审查时通过文档前缀区分（A-P01、B-P01、C-P01）。

**文档结构**：

---
source: 01-general-system-design-specification.md
source-version: v3.5
extracted-at: 2026-05-20

>
> 输出前自检以下颗粒度异常信号，将结果标注在语义提取文档开头供用户参考：
> 1. 是否有段落的内嵌规则超过 5 条？→ 可能颗粒度太粗，检查是否需要拆分段落（按判断焦点拆分）
> 2. 是否有单条内嵌规则包含了 3 个以上以"、"或"、"连接的并列项？→ 可能是并列列举而非复合属性
> 3. 是否多个段落可以用同一焦点问句概括？→ 可能颗粒度过细，考虑合并段落
>
> **自检信号与聚类优先级的关系**：三条自检信号是经验性启发（heuristic），不应机械套用。第 1 条"内嵌规则超过 5 条"需结合通读路径判断——若所有规则共享同一通读路径（如同属一节的概念体系），超过 5 条仍可合理共处一段；若来自不同通读路径，即使只有 3 条也应拆分。第 3 条"多个段落可用同一焦点问句"直接对应第二优先（判断焦点），是段落聚合的正面信号。
>
> 用户确认时不仅确认内容的正确性，也确认颗粒度是否合适。

### Step 1：策划审查任务清单

将各子步骤合并为 TodoWrite 任务清单，遵循 Step 2 的子步骤划分：

- **2-A** 列为一个任务（格式/机械类规则一次通读）
- **2-B～2-D** 各列为一个任务（语义类规则按组通读，每个聚类一轮）
- **2-E** 列为一个任务（机械检查脚本执行）
- **2-F** 列为一个任务（总体一致性检查）
- 最后两轮为：Step 3 覆盖确认 + Step 4 报告汇总
- 任务标题格式：
  - `2-A：格式/机械类规则一次通读`
  - `2-B：定义类（D）+ 范围类（S）`
  - `2-C：原则类（G）+ 流程类（P）`
  - `2-D：反模式类（A）+ 参考类（R）`
  - `2-E：机械检查脚本执行`
  - `2-F：总体一致性检查`
  - `3：语义段落覆盖确认`
  - `4：出审查报告`
- 输出完整的 TodoWrite 清单，**等用户确认后执行**

### Step 2：逐项执行（分轮次聚焦）

将审查拆分为多个聚焦单一任务的子步骤，避免在同一轮次中混合不同类型的判据。每步完成后标记 TodoWrite 对应任务为 completed。

#### Step 2-A：格式/机械类规则一次通读

聚焦语义提取文档中标签为 **F**（格式）、**Q**（质量）的段落。这些规则的判定通常不依赖深度语义理解，可一次通读完成：

1. 从语义提取文档中加载所有 **F**/**Q** 类型的语义段落
2. 以这些段落的完整原文为判据，通读目标文档，查找违反格式/质量规则的地方
3. **逐条判断内嵌规则**——每条规则独立给出 pass/fail
4. 记录发现时引用段落编号和规则编号，以及完整原文段落

#### Step 2-B～2-D：语义类规则按组通读（2-3轮）

聚焦语义提取文档中标签为 **D**（定义）、**G**（原则）、**P**（流程）、**A**（反模式）、**R**（参考）等需要深度语义理解的段落。每轮聚焦一个语义聚类（段落自然分组），一次只带一个"判断透镜"：

1. 从语义提取文档中加载本轮对应的语义段落（同一聚类）
2. 读取其类型标签、段落标题中的焦点问句（用于快速理解判断透镜）、内嵌规则及完整原文段落，获取每条规则的完整措辞、限定条件和约束强度
3. 以完整原文段落为判据，通读目标文档全文，只查找违反这些约束的地方
4. **逐条判断内嵌规则**——每条规则独立给出 pass/fail，不可用段落标题中的焦点问句替代规则级判断
5. 记录发现时引用段落编号和规则编号（如 `P05-R03`），以及完整原文段落

**语义聚类建议分组**（可根据实际语义段落调整）：
- **2-B**：定义类（D）+ 范围类（S）——概念一致性和范围边界
- **2-C**：原则类（G）+ 流程类（P）——设计方向和操作方法
- **2-D**：反模式类（A）+ 参考类（R）——常见错误和参考信息

> **注意**：段落类型 **E（示例）** 和 **N（导航）** 不单独成轮。E 段落依附于其所说明的规则段落（如 P01 [D] 中的示例随 P01 一起审查），N 段落用于辅助理解，不产生独立的审查判据。

#### Step 2-E：机械检查脚本执行

按检查内容选择辅助工具执行机械化检查：

- 版本号格式 → `bash .claude/skills/rule-conformance-review/scripts/04-check-version.sh <目标文档>`
- 超链接有效性 → `bash .claude/skills/rule-conformance-review/scripts/03-check-links.sh <目标文档>`
- 编号格式一致性 → `bash .claude/skills/rule-conformance-review/scripts/02-check-numbering.sh <目标文档>`
- 其他格式类检查由 AI 直接验证

#### Step 2-F：总体一致性检查

通读目标文档全文，进行跨规则的总体一致性检查——检查不同规则之间的综合影响是否被满足（例如：多条规则共同定义的格式要求是否同时被满足，而非单独检查时的遗漏）。

### Step 3：语义段落覆盖确认

确认所有语义提取文档中相关的语义段落及其内嵌规则都已覆盖，无遗漏。

### Step 4：出报告（Conformance Report）

汇总所有发现，输出审查报告。发现项中以规则编号（`P0X-RXX`）为主要引用，辅以段落编号（`P0X`）定位上下文。多文档场景中包含文档前缀（如 `[A-P01-R01]`、`[B-P03-R02]`）。

### Step 5：独立完整性审查

> **位置说明**：本步骤位于报告之后、修复之前，与修复无关。无论是否有发现，每次审查都必须执行。**本步骤强制执行，不得跳过。**

不加载规则文档，而是以文档自身的质量为目标，关注文档的**内在质量**而非与规则文档的符合性。

**审查维度**：
- **前后一致性**：术语、编号、格式在全文中是否统一
- **逻辑自洽性**：章节间语义无矛盾
- **语义清晰性和明确性**：表述是否精确，避免模糊词（如"快"、"高"、"好"等不可量化表述）
- **可理解性**：概念表述是否清晰易懂，是否需要补充说明或加注
- **简洁性**：内容是否精简，无冗余
- **可维护性**：是否便于更新，示例是否依赖特定产品实例（过期后规范失效），一处修改是否需同步更新多处

**操作方式**：
1. 不加载规则文档，仅通过通读目标文档全文进行审查
2. 重点关注示例中的表述是否与规范自身的质量标准一致

**输出与修复**：
- 发现的独立审查问题记录为发现清单（格式同 Step 4 报告中的 ❌/⚠️ 项）
- 按 Step 6 流程执行修复（策划修复任务清单 → 用户确认 → 执行修复）

### Step 6：策划修复任务清单并执行修改

总原则："先输出发现，等确认后再修改"——Step 4 的报告仅输出发现，不执行任何修改。

**Step 6a：策划修复任务清单**

将审查报告中的发现转化为修复任务，合并为 TodoWrite 任务清单：

- 每个 ❌ 发现（含严重/中等）单独列为一个修复任务，⚠️ 发现根据情况合并或单列
- 每个任务包含：
  - 任务标题：明确描述修改内容
  - 关联的发现编号（如发现3.1）
  - 修改位置（相关章节和行号）
  - 预期修改方式
- 输出完整的 TodoWrite 修复清单，**等用户确认后执行**

**Step 6b：按清单修复**

- 按确认后的修复清单逐项执行修改
- 关键约定：
  - 每完成一项修复，标记 TodoWrite 对应任务为 completed
  - 关联发现（如编号体系→需同步修改所有涉及的实例）应在同一轮次或连续轮次中完成
  - Base 修改完成后，用户确认所有变更

### Step 7：修复后独立完整性审查（回检）与终验

> **关键实践**：Step 5 在修复前执行过一次，但修复操作本身会修改文档，可能引入新的质量问题（编号不一致、格式断裂、遗漏同步修改等）。**修复完成后必须对修改后的文档重新执行独立完整性审查。**

**Step 7a：修复后独立完整性审查（回检）**

按 Step 5 的六维审查标准，对修复后的文档重新执行独立完整性审查：

| 维度 | 关注点（回检时特别关注） |
|------|------------------------|
| 前后一致性 | 修复用的编号/术语是否在全文中**所有**实例都已同步更新，无遗漏 |
| 语义清晰性 | 修复引入的新表述是否精确，修复后上下文是否仍连贯 |
| 简洁性 | 修复是否引入了不必要的冗余或重复 |

**输出**：
- 回检发现的遗留问题记录为发现清单（格式同 Step 4 报告中的 ❌/⚠️ 项）
- 按 Step 6 流程补充修复（策划修复任务 → 用户确认 → 执行）
- 回检通过后方可进入 Step 7b

**Step 7b：语义段落覆盖确认（终验）**

快速确认所有语义提取文档中相关的语义段落及其内嵌规则均已覆盖，无遗漏。
