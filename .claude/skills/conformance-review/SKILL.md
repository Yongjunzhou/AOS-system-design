# 文档符合性审查

## 总原则

- **依据条目即时生成**——审查清单不是预设的，而是在审查启动时根据加载的依据文档即时生成的
- **一次只用一个语义段落**去审视全文，每轮只带一个"判断透镜"
- **禁止跨轮累积语义**——每轮只查本轮的规则
- **先输出发现，等确认后再修改**

## 审查流程

### Step 0a：加载依据文档，执行多遍语义提取

依据文档加载后，不急于划分段落，而是通过五遍递进提取，从不同维度完整捕获文档语义。

**多依据文档场景**：若存在多份依据文档，每份文档独立执行五遍提取，各自生成独立的规则清单和语义段落。语义段落编号按文档区分前缀，例如依据文档 A 用 `[A-P01]`、依据文档 B 用 `[B-P01]`。

#### 第一遍：宏观骨架提取

目标：建立文档的整体结构认知。

| 维度 | 提取内容 |
|------|---------|
| 文档结构 | 章节目录及每节目的 |
| 概念图谱 | 文档定义了哪些主要概念/实体 |
| 范围声明 | 文档适用范围、适用场景 |
| 外部引用 | 引用了哪些其他文档 |

**输出**：概念清单 + 结构层级图（记录在工作笔记中）

#### 第二遍：规则清单提取（模式驱动）

目标：用关键词模式扫描全文档，**不分段落，逐条提取**所有约束性语句。

关键词模式与约束强度：

| 模式 | 约束强度 |
|------|---------|
| `必须` / `应` | **强制** |
| `不得` / `禁止` / `不应` | **禁止** |
| `建议` / `宜` / `推荐` | **建议** |
| `可` / `可以` / `允许` | **可选** |
| `如果...则...` / `当...时` | **条件约束**（叠加在以上强度上） |

提取结果示例：

```
R001 [强制] 每条详细定义末级节点只能分配到下层方案中唯一一个架构末级节点
R002 [条件·强制] 当 BA 包含多个 IPO 时，必须建立 IPO 之间的交互关系
R003 [建议] 架构末级节点的命名应反映其承担的职责
```

**输出**：结构化规则清单（编号 + 强度 + 条件 + 内容 + 出处章节）

#### 第三遍：实体定义提取

目标：提取文档中所有概念性定义，形成实体字典。

```
实体：架构末级节点
定义：在树形结构中位置最底层、且下属无子节点的节点
属性：名称、父节点、所映射的上层需求、符合性分析、下游指导
出处：§1.5.1
```

**输出**：实体字典（实体名 + 定义 + 关键属性 + 出处章节）

#### 第四遍：约束关系映射

目标：将规则和实体关联起来，识别以下关系：

1. **规则-实体关联**：每条规则约束的是哪个概念
2. **规则间依赖**：规则 A 是否引用规则 B 中定义的概念或方法
3. **隐含约束**：从多个定义和规则的关系中可以推导出的未显式陈述的约束

示例——若多处出现变量命名规范但没有单独定义"变量名格式"，则从"追溯矩阵通过编号关联"可推导"需求必须编号"虽未直接陈述但属于隐含约束。

**多依据文档场景**：增加一层跨文档关系识别：

4. **跨文档引用**：依据文档之间的交叉引用（如设计指南引用了准则中的某条定义）
5. **冲突检测**：不同依据文档之间是否存在相互矛盾的规则。检测到的冲突标记为 **⚠️ 跨文档冲突**，在审查报告中豁免目标文档（目标文档无法同时满足两份矛盾文档的要求）

**输出**：规则-实体关联矩阵 + 隐含约束清单 +（多文档场景）跨文档引用清单 + 冲突清单

#### 第五遍：语义段落组装

目标：基于前四遍的输出，将规则清单中的条目按语义聚类为"一个判断透镜"——即语义段落。此时的分段不再依赖原文章节边界，而是基于规则内容的自然聚合。

原则：
- 关于同一个实体的同类规则 → 合并为一个段落
- 不同类型的约束（定义 vs 格式 vs 原则）→ 即使在同一章，也要拆分
- 每段包含 1-3 条紧密相关的核心约束

段落类型标注（沿用已有体系）：

| 标签 | 含义 | 用途 |
|------|------|------|
| **D** | 定义 | 概念、术语、规则的定义 |
| **P** | 流程 | 操作步骤、方法流程 |
| **F** | 格式 | 格式规范、编号规则 |
| **G** | 原则 | 设计方向、权衡决策、指导性思想 |
| **S** | 范围 | 范围界定、各层定位 |
| **Q** | 质量 | 验收标准、质量指标 |
| **A** | 反模式 | 常见错误、禁忌 |
| **E** | 示例 | 规则难以文字化时，以示例作为规范标准（仅规则模糊或不明确时使用） |
| **N** | 导航 | 交叉引用、关系矩阵 |
| **R** | 参考 | 参考信息、FAQ |

**输出**：语义段落清单（段落编号 + 类型 + 核心约束 + 关联规则编号 + 来源章节）

---

### Step 0b：生成临时语义提取文档

将前五遍的四项核心输出合并为一份语义提取文档，写入临时目录。

**路径格式**：

```
.claude/skills/conformance-review/temp/{源文档文件名前缀}-semantics.md
```

例如依据 `01-general-design-standards.md` 则生成 `temp/general-design-standards-semantics.md`。

**多依据文档场景**：每份依据文档生成独立的临时语义提取文档，例如：

```
temp/
├── general-design-standards-semantics.md     # 依据文档A
├── pipeline-design-standards-semantics.md    # 依据文档B
└── waterfall-design-guide-semantics.md       # 依据文档C
```

段落编号按文档分配独立序号，审查时通过文档前缀区分（A-P01、B-P01、C-P01）。

**文档结构**：

```markdown
---
source: 01-general-design-standards.md
source-version: v3.5
extracted-at: 2026-05-20
---

# 语义提取：通用设计准则

## 1. 实体字典

| 实体 | 定义 | 关键属性 | 出处 |
|------|------|---------|------|
| 架构末级节点 | 在树形结构中位置最底层且无子节点的节点 | 名称、父节点、所映射的上层需求、符合性分析、下游指导 | §1.5.1 |

## 2. 规则清单

| 编号 | 强度 | 条件 | 内容 | 出处 |
|------|------|------|------|------|
| R001 | 强制 | — | 每条详细定义末级节点只能分配到下层方案中唯一一个架构末级节点 | §3.2 |
| R002 | 强制 | BA 场景 | BA 架构末级节点必须以 IPO 为基本设计单位 | §2.3 |

## 3. 语义段落

| 段落 | 类型 | 核心约束 | 关联规则 | 来源 |
|------|------|---------|---------|------|
| P01 | D | 定义架构末级节点的基本概念 | R001 | §1.5.1 |
| P02 | G | 同步设计原则：上层详细定义与下层方案架构同步开展 | — | §1.5.3 |
```

**说明**：
- 此文件仅在本次审查期间存在，审查结束后不保留
- `temp` 目录已加入 `.gitignore`，不会被版本控制追踪
- 后续步骤直接引用此文件中的段落编号（`P01`、`P02`...）和规则编号（`R001`、`R002`...）作**导航和聚焦**之用
- **审查的最终依据始终是依据文档原文**，语义文档的摘要不可替代原文的完整措辞

> **Step 0b 输出关卡**：将语义提取文档（单文档场景：1 份；多文档场景：全部文档）的完整内容呈现给用户确认。确认后方可进入 Step 1。

---

### Step 1：策划审查任务清单

将所有语义审查轮次和机械检查项合并为 TodoWrite 任务清单：

- 每个相关的语义段落列为一个语义审查轮次
- 目标文档对应的锚段落（依据文档中定义该层具体要求的语义段落）标记为最核心轮次
- 机械检查项单独列轮次
- 最后两轮为：总体一致性检查 + 汇总审查发现
- 任务标题格式（单文档）：
  - `第[n]轮：P0X [类型标签]——[核心约束]`
- 任务标题格式（多文档）：
  - `第[n]轮：[文档前缀-P0X] [类型标签]——[核心约束]`
- 输出完整的 TodoWrite 清单，**等用户确认后执行**

### Step 2：逐项执行

语义审查轮次：
1. 从对应的临时语义提取文档中加载本轮的语义段落（多文档场景按文档前缀匹配）
2. 读取其类型标签和 1-3 条核心约束，识别关联规则编号（如 R001、R002）
3. **回查依据文档原文**：加载语义段落所对应的依据文档原文章节，确认规则的完整措辞、限定条件和适用范围。**审查的最终依据是依据文档原文，而非语义文档的摘要**
4. 通读目标文档全文，以原文措辞为准绳，只查找违反这些约束的地方
5. 记录发现时引用依据文档原文和规则编号（如"§4.1 规定'只有需求，没有方案'，但目标文档中出现了...违反 [A-R001]"），标记 TodoWrite 为 completed

机械检查轮次：按检查内容选择辅助工具执行
- 版本号格式 → `bash .claude/skills/conformance-review/scripts/04-check-version.sh <目标文档>`
- 超链接有效性 → `bash .claude/skills/conformance-review/scripts/03-check-links.sh <目标文档>`
- 编号格式一致性 → `bash .claude/skills/conformance-review/scripts/02-check-numbering.sh <目标文档>`
- 其他格式类检查由 AI 直接验证

### Step 3：语义段落覆盖确认

确认所有依据文档中相关的语义段落都已覆盖，无遗漏。

### Step 4：出报告

汇总所有发现，输出审查报告。发现项中应引用语义段落编号（P0X）和规则编号（R00X）实现精确定位。多文档场景中包含文档前缀（如 `[A-P01]`、`[B-R001]`）。

### Step 7：独立完整性审查

> **位置说明**：本步骤位于报告之后、修复之前，与修复无关。无论是否有发现，每次审查都必须执行。**本步骤强制执行，不得跳过。**

不加载依据文档，而是以文档自身的质量为目标，关注文档的**内在质量**而非与依据文档的符合性。

**审查维度**：
- **前后一致性**：术语、编号、格式在全文中是否统一
- **逻辑自洽性**：章节间语义无矛盾
- **语义清晰性和明确性**：表述是否精确，避免模糊词（如"快"、"高"、"好"等不可量化表述）
- **可理解性**：概念表述是否清晰易懂，是否需要补充说明或加注
- **简洁性**：内容是否精简，无冗余
- **可维护性**：是否便于更新，示例是否依赖特定产品实例（过期后规范失效），一处修改是否需同步更新多处

**操作方式**：
1. 不加载依据文档，仅通过通读目标文档全文进行审查
2. 重点关注示例中的表述是否与规范自身的质量标准一致

**输出与修复**：
- 发现的独立审查问题记录为发现清单（格式同 Step 4 报告中的 ❌/⚠️ 项）
- 按 Step 5 流程执行修复（策划修复任务清单 → 用户确认 → 执行修复）

### Step 5：策划修复任务清单并执行修改

总原则："先输出发现，等确认后再修改"——Step 4 的报告仅输出发现，不执行任何修改。

**Step 5a：策划修复任务清单**

将审查报告中的发现转化为修复任务，合并为 TodoWrite 任务清单：

- 每个 ❌ 发现（含严重/中等）单独列为一个修复任务，⚠️ 发现根据情况合并或单列
- 每个任务包含：
  - 任务标题：明确描述修改内容
  - 关联的发现编号（如发现3.1）
  - 修改位置（相关章节和行号）
  - 预期修改方式
- 输出完整的 TodoWrite 修复清单，**等用户确认后执行**

**Step 5b：按清单修复**

- 按确认后的修复清单逐项执行修改
- 关键约定：
  - 每完成一项修复，标记 TodoWrite 对应任务为 completed
  - 关联发现（如编号体系→需同步修改所有涉及的实例）应在同一轮次或连续轮次中完成
  - Base 修改完成后，用户确认所有变更

### Step 6：语义段落覆盖确认（终验）

修复完成后，快速确认所有依据文档中相关的语义段落均已覆盖，无遗漏。
