# 文档符合性审查 — 需求符合性审查（Requirements Conformance Review）

> 与规则符合性审查（Rule Conformance Review，审查目标文档是否符合规则）相区别，本技能聚焦于**需求符合性审查**——审查方案文档（目标文档）是否满足需求文档中定义的需求。

## 总原则

- **依据条目即时生成**——审查清单不是预设的，而是在审查启动时根据加载的需求文档即时生成的
- **两个独立维度**——语义符合性（需求说明符合性）和性能指标符合性（需求性能符合性）独立提取、独立审查、在报告中分别呈现
- **一次只用一个审查单元**去审视全文，每轮只带一组需求或一条性能指标
- **禁止跨轮累积**——每轮只查本轮对应的需求或指标
- **先输出发现，等确认后再修改**
- **自适应双模式**——自动检测文档结构化程度（规范模式/自由模式），选择对应的提取和审查策略

### 需求符合性审查的两个维度

| 维度 | 方法论依据 | 审查内容 | 发现类型 |
|------|-----------|---------|---------|
| **语义符合性**（需求说明符合性） | 检查每个架构末级节点是否充分满足所承接的上层需求，验证映射关系是否完整、正确、无矛盾 | 覆盖度、映射正确性、符合性分析质量 | ❌ Gap/Mis-Map、⚠️ Partial/No-Compliance/Weak-Compliance |
| **性能指标符合性**（需求性能符合性） | 验证设计方案能否达到所承接的性能指标要求，确保性能需求在架构层面可实现 | 指标继承、目标值一致性、测量条件、验证方法 | ❌ PI-Gap、⚠️ PI-Degraded/PI-Partial/PI-NoVerify |

### 与规则符合性审查的区别

| 维度 | 规则符合性审查 | 需求符合性审查 |
|------|---------------|---------------|
| 源文档类型 | 规则/标准（约束性语句） | 需求文档（需求条目 + 性能指标） |
| 提取方法 | 五遍语义提取（情态动词驱动） | 结构化/语义提取（ID 模式驱动） + 性能指标提取 |
| 审查单元 | 语义段落（1-3 条规则） | 语义需求簇 + 性能指标清单 |
| 审查维度 | 单维度：是否违反规则 | **双维度**：语义符合 + 性能指标符合 |
| 跨层校验 | 无 | 1:1、N:1、双向追溯（仅规范模式） |

## 审查流程

### Step 0a：加载需求文档，执行文档结构评估与需求提取

#### 文档结构评估

加载需求文档后，首先判定文档的结构化程度。两条文档独立评估。

| 信号 | 规范模式（模式A） | 自由模式（模式B） |
|------|-----------------|-----------------|
| 是否有正式 ID 体系（`XXX-001`、`XXX-001.01`） | 是 | 否 |
| 是否有架构节点定义表格 | 有 | 无 |
| 是否有"架构定义/详细定义"的分离结构 | 有 | 无 |
| 是否有明确的分类（功能/非功能） | 有 | 无 |
| 是否有性能指标列或验收标准列 | 有 | 无 |

**判定输出**：`需求文档: 模式A/模式B` + `目标文档: 模式A/模式B`

---

#### 模式A：规范模式提取（结构化 ID 驱动）

适用于文档已有正式 ID 体系（`{Prefix}[-{Type}]-{seq}`、`{Parent}.{seq}`）。

**第一遍：宏观结构分析**

| 维度 | 提取内容 |
|------|---------|
| 文档结构 | 章节目录及每节目的 |
| ID 模式 | 识别前缀（OR/SR/BA/SysReq/PA）、类型后缀（F/NFR）、序号位数、编号层级 |
| 层识别 | 属于五层的哪一层（1=OR, 2=SR, 3=BA, 4=SysReq, 5=PA） |
| 需求分类 | 功能/非功能分离方式、架构节点 vs 详细定义 |
| 外部引用 | 引用了哪些其他文档 |

**第二遍：结构化需求条目提取**

扫描所有 ID 模式，逐条提取。提取格式：

```
RQI-{seq}
- 需求ID: {文档中的正式编号}
- 名称/描述: {需求名称及定性说明}
- 类型: {功能需求 / 非功能需求 / 架构节点 / 详细定义}
- 性能指标: {列举指标摘要或"N/A"}
- 验收标准: {标准描述或"N/A"}
- 优先级: {P0/P1/P2 或 "N/A"}
- 来源章节: {§x.x}
```

**第三遍：性能指标提取与分析准备**

提取所有性能指标，建立从"原始表述"到"量化指标"的映射。搜寻范围不仅限于"性能指标"显式标题，还包括验收标准中的量化要求、需求描述中内嵌的数字表述、约束条件中的量化限定、表格中的指标列。

每条性能指标提取结构：

```
PI-{seq}
- 关联需求: {RQI-xxx} / {需求ID}
- 原始表述: {需求文档中的原文}
- 指标分解:
  ├─ 指标名称: {响应时间}
  ├─ 目标值: {1}
  ├─ 单位: {秒}
  ├─ 测量条件: {正常负载，1000并发用户}
  ├─ 统计方式: {TP95 / 平均值 / 最大值}
  └─ 指标类型: {功能性能 / 质量特性 / 环境适应性 / 可实现性}
- 可验证性评估: {明确可测 / 需补充条件 / 不可验证}
- 来源章节: {§x.x}
```

**第四遍：需求关系映射**

识别需求之间的层级关系（父-子）、依赖关系（需求A依赖需求B）、交叉引用、冲突检测（矛盾的需求描述或冲突的性能目标）。

**第五遍：审查单元组装**

- **语义需求簇**：按目标架构节点或功能领域将需求条目组织为审查轮次单元
- **性能指标清单**：独立于需求簇的完整清单（审查时走专用轮次）

---

#### 模式B：自由模式提取（语义主题驱动）

适用于文档无正式 ID 体系，需求以章节、列表、自然段落组织。

**第一遍：宏观结构分析** — 识别文档的章节结构、主题领域、核心概念

**第二遍：语义需求识别** — 以语义方式识别需求陈述：

- 每个标题/子标题下的内容块 → 候选需求条目
- 列表项（`-` 或 `1.`）→ 候选需求条目
- 含有"需/应/要/必须/支持"等需求关键词的段落 → 候选需求条目
- 表格行（需求描述列）→ 候选需求条目

**第三遍：需求陈述规范化 + 性能指标提取**

- 为每个识别出的需求分配临时编号 RQI-{seq}，提取语义描述、涉及的概念实体
- **同步提取性能指标**：扫描所有已识别需求中的量化表述
  - 数字+单位组合（"200ms"、"99.9%"、"1000并发"）
  - 比较级表述（"不超过"、"不低于"、"小于"）
  - 频率/容量表述（"每天"、"每小时可处理"）
  - 提取格式同模式A（PI-{seq} + 分解字段），但标注"自由模式提取，需确认"

**第四遍：主题聚类** — 按文档章节结构和语义相似度，将需求聚类为主题域

**第五遍：审查单元组装**

- **语义需求簇**：按主题域组织
- **性能指标清单**：独立于需求簇的完整清单

---

### Step 1：策划审查任务清单

审查任务分为两个系列，合并为 TodoWrite 任务清单。

**语义审查系列**：
- 每个需求簇为一个语义审查轮次
- 任务标题格式：`第[n]轮：[CL-{seq}]——{审查焦点}`

**性能指标审查系列**：
- **模式A**：`第[n]轮：性能指标审查——逐指标继承比对（共 N 条 PI）`
- **模式B**：`第[n]轮：性能指标审查——定性评估（共 N 条 PI）`

**其他轮次**：
- **模式A**：跨层校验轮次
- **两种模式**：机械检查轮次（版本号、超链接、编号格式）+ 总体一致性检查轮次

输出完整的 TodoWrite 清单，等用户确认后执行。

---

### Step 2：逐项执行审查

#### 语义审查轮次

每轮加载一个需求簇，通读目标文档全文，只查找与本簇需求相关的问题。

**模式A 判据**（规范文档，有映射关系）：

| 判据 | 问题 | 通过条件 | 发现类型 |
|------|------|---------|---------|
| **覆盖度** | 需求是否映射到至少一个架构末级节点？ | 在目标文档中找到映射记录 | ❌ Gap |
| **映射正确性** | 映射语义是否适当？ | 节点定义与需求意图一致 | ❌ Mis-Map |
| **符合性分析存在性** | 节点是否包含符合性分析？ | 符合性分析段落非空 | ⚠️ No-Compliance |
| **符合性分析质量** | 分析是否充分说明如何满足需求？ | 具体、可验证、直接针对需求 | ⚠️ Weak-Compliance |
| **1:1 唯一性** | 是否每条需求映射到唯一方案节点？ | 无重复映射 | ⚠️ 记录至跨层校验 |
| **性能继承** | 性能指标是否从需求继承到方案？ | 方案节点包含相同或更严格的指标 | 转至性能指标审查轮次 |

**模式B 判据**（自由文档，无映射关系）：

| 判据 | 问题 | 通过条件 | 发现类型 |
|------|------|---------|---------|
| **覆盖度** | 目标文档中是否有对应内容处理此需求？ | 找到语义相关的内容段落 | ❌ Gap |
| **语义一致性** | 目标文档的处理方向与需求意图一致？ | 无方向性偏差或理解错误 | ❌ Mis-Map |
| **完整性** | 目标文档是否覆盖了需求的所有关键方面？ | 无关键遗漏 | ⚠️ Partial |
| **性能/质量** | 隐含的性能要求是否被提及或满足？ | 有对应处理或明确说明不适用 | 转至性能指标审查轮次 |

**模式B 附加**：若发现目标文档中出现了可规范化的结构（如可识别的 ID 模式或映射表雏形），可产生 ⚠️ **No-Structure** 提示："本文档的部分内容具备规范化潜力，建议按方法论进行分解分配以提升可审查性。"

---

#### 性能指标审查轮次

语义审查完成后，加载 Step 0b 提取的 PI 清单，执行专用审查。

**模式A 性能指标判据**（逐指标比对）：

| 判据 | 问题 | 通过条件 | 发现类型 |
|------|------|---------|---------|
| **继承存在性** | 该指标是否在目标文档中被显式承接？ | 在对应方案节点中找到 PI 引用或等同指标 | ❌ PI-Gap |
| **目标值一致性** | 目标值是否不低于需求文档的要求？ | 方案中的指标值 ≥ 需求值（或更严格） | ⚠️ PI-Degraded |
| **测量条件完整** | 测量条件/统计方式是否被继承？ | 条件被明确记录 | ⚠️ PI-Partial |
| **可验证性** | 方案中是否有明确的验证方法？ | 有验证方法描述或指向验证文档 | ⚠️ PI-NoVerify |
| **覆盖全面性** | 所有 PI 都被覆盖，无遗漏？ | PI 清单逐条比对完成 | ❌ PI-Gap |

**模式B 性能指标判据**（定性评估）：

| 判据 | 问题 | 通过条件 | 发现类型 |
|------|------|---------|---------|
| **提及度** | 目标文档是否提及该性能关注点？ | 找到相关讨论 | ❌ PI-Gap |
| **方向合理性** | 数值方向是否合理（不倒退）？ | 无明确倒退信号 | ⚠️ PI-Degraded |
| **完整性** | 所有隐含性能点都有对应处理？ | 无显著缺失 | ⚠️ PI-Partial |

**性能指标发现引用格式**：`PI-{seq}（关联需求 {RQI-xxx}）`

**模式B 附加建议**：审查完成后，附加一段结构化建议——提示文档按方法论规范分解分配后，可提升审查的精确度并支持跨层校验。此举是信息性提示，非审查发现。

---

### Step 3：跨层校验（仅模式A）

仅当需求文档和目标文档都处于规范模式时执行。

1. **1:1 分配约束检查**
   - 逐条检查上层需求详细定义末级节点 → 是否分配到**唯一一个**下层方案架构末级节点
   - 违规：同一需求映射到多个方案节点
   - 记录：需求ID + 涉及的多余方案节点

2. **N:1 承接合理性检查**
   - 统计每个方案架构末级节点承接的上层需求数量
   - 理想范围：2-5 条/节点
   - 警告阈值：=1（考虑合并）或 >5（考虑拆分）
   - 异常阈值：>8（需重点关注）
   - 记录：节点ID + 承接数 + 评估（合理/偏多/偏少/异常）

3. **双向追溯完整性**
   - 正向追溯：所有上层需求能否追踪到下层方案节点？
   - 反向追溯：所有下层方案节点能否回溯到上层需求？
   - 检查方案节点是否有显式的"承接的需求末级节点"字段
   - 检查是否有需求映射视图（如映射关系表）
   - 记录：正反向完整率、断链清单

模式B 跳过此步，在报告中注明"跨层校验：需求-目标均为自由模式，未执行形式化跨层校验"。

---

### Step 4：覆盖确认

确认所有语义需求簇和性能指标清单都已审查，无遗漏。
