# Task 3：系统需求最佳实践指南

**目的**：帮助团队设计高质量的系统需求，避免常见错误

**版本**：v1.0  
**最后更新**：2026-05-13

---

## 一、SysReq 层级结构详解

### 1.1 SysReq 0 级：根节点

**定义**：整个系统的顶层根节点

**特点**：
- 唯一的根节点
- 代表整个系统
- 所有其他节点都是其子节点

---

### 1.2 SysReq 1 级：独立信息系统

**定义**：一个独立的信息系统

**特点**：
- 代表一个完整的系统
- 通常对应一个业务系统
- 一个项目可以有多个系统

**示例**：
```
SysReq-1-ECOMMERCE：电商平台系统
SysReq-1-LOGISTICS：物流系统
SysReq-1-FINANCE：财务系统
```

---

### 1.3 SysReq 2 级：功能/非功能需求分支

**定义**：功能需求分支或非功能需求分支

**特点**：
- 分支点，将功能需求和非功能需求分开
- 功能需求分支继续向下分解
- 非功能需求分支进入非功能需求架构

**示例**：
```
SysReq-2-FUNC：功能需求分支
SysReq-2-NFR：非功能需求分支
```

---

### 1.4 SysReq 3 级：业务组件类型

**定义**：业务组件的类型分类

**特点**：
- 代表不同的业务组件类型
- 通常对应不同的业务流程类型
- 一个系统可以有多个业务组件类型

**示例**：
```
SysReq-3-ORDER-FLOW：订单流程类型
SysReq-3-PAYMENT-FLOW：支付流程类型
SysReq-3-NOTIFICATION：通知类型
```

---

### 1.5 SysReq 4 级：业务组件小类

**定义**：业务组件类型下的小类分类

**特点**：
- 代表业务组件类型下的具体小类
- 通常对应具体的业务流程
- 一个业务组件类型可以有多个小类

**示例**：
```
SysReq-4-ORDER-CREATION：订单创建小类
SysReq-4-PAYMENT：支付小类
SysReq-4-ORDER-NOTIFICATION：订单通知小类
```

---

### 1.6 SysReq 5 级：组件实例（承接层）

**定义**：承接 BA 5 级节点的关键层级

**特点**：
- 是 BA → SysReq 映射的承接点
- 代表具体的组件实例
- 与 BA 5 级节点是 N:1 映射关系

**示例**：
```
SysReq-5-001：订单创建组件实例
SysReq-5-002：支付处理组件实例
SysReq-5-004：通知发送组件实例
```

**最佳实践**：
- 每个 SysReq 5 级节点应该对应一个 BA 5 级节点
- 每个 SysReq 5 级节点应该有明确的系统能力
- 每个 SysReq 5 级节点应该分解到 SysReq 6-9 级

---

### 1.7 SysReq 6 级：功能模块

**定义**：以标签页为载体的功能模块

**特点**：
- 代表系统用户界面上的一个标签页
- 一个组件实例可能有多个功能模块
- 功能模块是用户交互的基本单位

**示例**：
```
SysReq-6-001：订单信息输入模块
SysReq-6-002：订单确认模块
SysReq-6-003：支付方式选择模块
```

**最佳实践**：
- 功能模块应该对应一个标签页或页面
- 功能模块应该有明确的功能描述
- 功能模块应该包含相关的系统用例

---

### 1.8 SysReq 7 级：系统用例

**定义**：在标签页上提供的功能

**特点**：
- 代表系统提供的一个功能
- 通常以功能按钮或右键菜单为载体
- 一个功能模块可能有多个系统用例

**示例**：
```
SysReq-7-001：选择商品用例
SysReq-7-002：填写收货地址用例
SysReq-7-003：确认订单用例
```

**最佳实践**：
- 系统用例应该代表一个完整的用户操作
- 系统用例应该有明确的输入和输出
- 系统用例应该包含相关的用例场景

---

### 1.9 SysReq 8 级：用例场景

**定义**：系统用例的具体场景

**特点**：
- 代表系统用例的一个具体场景
- 通常包括正常场景和异常场景
- 一个系统用例可能有多个用例场景

**示例**：
```
SysReq-8-001：正常选择场景
SysReq-8-002：商品缺货场景
SysReq-8-003：新增地址场景
```

**最佳实践**：
- 用例场景应该包括正常场景和异常场景
- 用例场景应该有明确的前置条件和后置条件
- 用例场景应该包含相关的场景活动

---

### 1.10 SysReq 9 级：场景活动

**定义**：用例场景的具体活动步骤

**特点**：
- 代表用例场景的一个具体活动步骤
- 是系统需求的最细粒度
- 映射到产品架构末级节点

**示例**：
```
SysReq-9-001：浏览商品列表
SysReq-9-002：搜索商品
SysReq-9-003：查看商品详情
SysReq-9-004：添加到购物车
```

**最佳实践**：
- 场景活动应该是原子操作
- 场景活动应该有明确的输入和输出
- 场景活动应该映射到产品架构末级节点

## 二、常见错误

### 错误 1：SysReq 5 级节点定义不清

**❌ 错误示例**
```
SysReq-5-001：订单组件实例
SysReq-5-002：用户组件实例
```

**问题**：
- 定义太宽泛
- 无法明确系统能力
- 无法进行有效的产品设计

**✅ 正确做法**
```
SysReq-5-001：订单创建组件实例
SysReq-5-002：支付处理组件实例
SysReq-5-003：订单修改组件实例
```

---

### 错误 2：忽视 SysReq 6-9 级的分解

**❌ 错误示例**
```
SysReq-5-001：订单创建组件实例
（直接跳到产品架构，没有分解到 SysReq 6-9 级）
```

**问题**：
- 无法清晰地定义系统功能
- 无法进行有效的产品设计
- 无法进行有效的测试

**✅ 正确做法**
```
SysReq-5-001：订单创建组件实例
├── SysReq-6-001：订单信息输入模块
│   ├── SysReq-7-001：选择商品用例
│   │   ├── SysReq-8-001：正常选择场景
│   │   │   ├── SysReq-9-001：浏览商品列表
│   │   │   ├── SysReq-9-002：搜索商品
│   │   │   └── SysReq-9-003：查看商品详情
│   │   └── SysReq-8-002：商品缺货场景
│   │       └── SysReq-9-005：显示缺货提示
│   └── SysReq-7-002：填写收货地址用例
│       ├── SysReq-8-003：新增地址场景
│       └── SysReq-8-004：选择已有地址场景
└── SysReq-6-002：订单确认模块
    └── SysReq-7-003：确认订单用例
        └── SysReq-8-005：正常确认场景
```

---

### 错误 3：缺少异常场景

**❌ 错误示例**
```
SysReq-8-001：正常选择场景
（只定义正常场景，没有异常场景）
```

**问题**：
- 无法处理异常情况
- 系统容错能力不足
- 用户体验不好

**✅ 正确做法**
```
SysReq-7-001：选择商品用例
├── SysReq-8-001：正常选择场景
│   ├── SysReq-9-001：浏览商品列表
│   ├── SysReq-9-002：搜索商品
│   ├── SysReq-9-003：查看商品详情
│   └── SysReq-9-004：添加到购物车
└── SysReq-8-002：商品缺货场景
    └── SysReq-9-005：显示缺货提示
```

---

### 错误 4：场景活动定义不清

**❌ 错误示例**
```
SysReq-9-001：处理商品选择
```

**问题**：
- 定义太宽泛
- 无法明确具体操作
- 无法进行有效的产品设计

**✅ 正确做法**
```
SysReq-9-001：浏览商品列表
SysReq-9-002：搜索商品
SysReq-9-003：查看商品详情
SysReq-9-004：添加到购物车
```

---

### 错误 5：忽视系统能力定义

**❌ 错误示例**
```
SysReq-5-001：订单创建组件实例
（没有说明系统能力）
```

**问题**：
- 无法明确系统应该提供什么功能
- 无法进行有效的产品设计
- 无法进行有效的测试

**✅ 正确做法**
```
SysReq-5-001：订单创建组件实例
系统能力:
  1. 商品选择能力
  2. 地址管理能力
  3. 订单生成能力
  4. 库存检查能力
```

## 三、最佳实践

### 最佳实践 1：使用结构化格式

**推荐格式**：YAML 或表格

```yaml
组件实例ID: SysReq-5-001
组件实例名称: 订单创建组件实例
业务组件类型: 流程
业务组件小类: 订单创建
承接的 BA: BA-5-001

业务功能: |
  支持用户创建订单，包括选择商品、数量、收货地址等信息。

系统能力: |
  1. 商品选择能力
  2. 地址管理能力
  3. 订单生成能力
  4. 库存检查能力

功能模块（6级）:
  - SysReq-6-001：订单信息输入模块
  - SysReq-6-002：订单确认模块

系统用例（7级）:
  - SysReq-7-001：选择商品用例
  - SysReq-7-002：填写收货地址用例
  - SysReq-7-003：确认订单用例
```

**优势**：
- 信息完整
- 易于解析
- 易于版本控制

---

### 最佳实践 2：完整分解到 SysReq 9 级

**每个 SysReq 5 级节点都应该分解到 SysReq 9 级**

```
SysReq-5-001：订单创建组件实例
├── SysReq-6：功能模块（2-3 个）
│   ├── SysReq-7：系统用例（3-5 个）
│   │   ├── SysReq-8：用例场景（2-3 个）
│   │   │   └── SysReq-9：场景活动（3-5 个）
```

**优势**：
- 清晰的功能分解
- 便于产品设计
- 便于测试设计

---

### 最佳实践 3：包括异常场景

**每个系统用例都应该包括正常场景和异常场景**

```
SysReq-7-001：选择商品用例
├── SysReq-8-001：正常选择场景
│   ├── SysReq-9-001：浏览商品列表
│   ├── SysReq-9-002：搜索商品
│   ├── SysReq-9-003：查看商品详情
│   └── SysReq-9-004：添加到购物车
└── SysReq-8-002：商品缺货场景
    └── SysReq-9-005：显示缺货提示
```

**优势**：
- 提高系统容错能力
- 提高用户体验
- 便于测试设计

---

### 最佳实践 4：明确系统能力

**每个 SysReq 5 级节点都应该明确说明系统能力**

```
SysReq-5-001：订单创建组件实例
系统能力:
  1. 商品选择能力
  2. 地址管理能力
  3. 订单生成能力
  4. 库存检查能力
```

**优势**：
- 便于产品设计
- 便于工作量估算
- 便于技术选型

---

### 最佳实践 5：记录关联关系

**为 SysReq 节点记录相关的 BA 节点和产品架构节点**

```
SysReq-5-001：订单创建组件实例
承接的 BA: BA-5-001
相关的 SysReq 9级: SysReq-9-001 ~ SysReq-9-016
映射的 PA 末级: PA-001（订单创建前端组件）、PA-002（订单创建后端组件）
```

**优势**：
- 便于追踪需求链路
- 便于产品设计
- 便于测试设计

## 五、常见问题解答

**Q：SysReq 5 级节点和 BA 5 级节点的关系是什么？**

A：
- BA 5 级节点是业务组件
- SysReq 5 级节点是系统组件实例
- 一个 BA 5 级节点映射到一个 SysReq 5 级节点（1:1 关系）
- 一个 SysReq 5 级节点可以承接多个 BA 5 级节点（N:1 关系）

**Q：SysReq 9 级场景活动应该有多详细？**

A：
- 场景活动应该是原子操作
- 场景活动应该能够映射到产品架构末级节点
- 通常一个系统用例有 3-5 个场景活动

**Q：如何处理跨越多个 SysReq 5 级节点的流程？**

A：
1. 识别流程涉及的 SysReq 5 级节点
2. 为每个 SysReq 5 级节点定义相应的场景活动
3. 记录场景活动之间的关系
4. 在产品设计阶段处理跨组件集成

**Q：SysReq 8 级用例场景应该如何定义？**

A：
- 用例场景应该包括前置条件和后置条件
- 用例场景应该包括正常场景和异常场景
- 用例场景应该有明确的输入和输出
- 用例场景应该包含相关的场景活动
