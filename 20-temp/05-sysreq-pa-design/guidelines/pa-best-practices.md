# Task 5：产品架构最佳实践指南

**目的**：帮助团队设计高质量的产品架构，避免常见错误

**版本**：v1.0  
**最后更新**：2026-05-13

---

## 一、产品架构末级节点分类

### 1.1 前端组件

**定义**：用户界面层的组件

**特点**：
- 与用户直接交互
- 负责数据展示和用户输入
- 通常是页面或页面中的模块

**示例**：
```
PA-FE-001：商品列表页面
PA-FE-002：商品详情页面
PA-FE-003：购物车页面
PA-FE-004：收货地址页面
PA-FE-005：订单确认页面
```

**最佳实践**：
- 前端组件应该对应一个页面或页面中的主要模块
- 前端组件应该有明确的职责
- 前端组件应该与后端 API 对应

---

### 1.2 后端组件

**定义**：业务逻辑层的组件

**特点**：
- 处理业务逻辑
- 与数据库交互
- 通常是 API 或服务

**示例**：
```
PA-BE-001：商品查询 API
PA-BE-002：购物车 API
PA-BE-003：地址 API
PA-BE-004：订单 API
PA-BE-005：支付 API
```

**最佳实践**：
- 后端组件应该对应一个业务功能
- 后端组件应该有明确的职责
- 后端组件应该与数据库对应

---

### 1.3 数据库组件

**定义**：数据存储层的组件

**特点**：
- 存储数据
- 提供数据查询和修改
- 通常是数据库或数据表

**示例**：
```
PA-DB-001：商品数据库
PA-DB-002：购物车数据库（Redis）
PA-DB-003：地址数据库
PA-DB-004：订单数据库
PA-DB-005：支付数据库
```

**最佳实践**：
- 数据库组件应该对应一个业务实体
- 数据库组件应该有明确的数据结构
- 数据库组件应该与后端 API 对应

---

### 1.4 其他组件

**定义**：其他类型的组件

**特点**：
- 缓存、消息队列、搜索引擎等
- 支持系统的非功能需求
- 通常是基础设施组件

**示例**：
```
PA-CACHE-001：Redis 缓存
PA-MQ-001：消息队列
PA-SEARCH-001：搜索引擎
PA-CDN-001：CDN 加速
```

**最佳实践**：
- 其他组件应该支持系统的非功能需求
- 其他组件应该有明确的职责
- 其他组件应该与其他组件集成

---

## 二、常见错误

### 错误 1：PA 末级节点定义不清

**❌ 错误示例**
```
PA-001：订单组件
PA-002：用户组件
PA-003：商品组件
```

**问题**：
- 定义太宽泛
- 无法明确组件职责
- 无法进行有效的开发

**✅ 正确做法**
```
PA-FE-001：商品列表页面
PA-FE-002：商品详情页面
PA-BE-001：商品查询 API
PA-DB-001：商品数据库
```

---

### 错误 2：忽视非功能权衡

**❌ 错误示例**
```
需求 1：系统响应时间 < 100ms
需求 2：支持 10000+ 并发用户
需求 3：采用 AES-256 加密
需求 4：系统成本 < 10 万元
（没有权衡决策）
```

**问题**：
- 这些需求之间可能存在冲突
- 无法进行有效的架构设计
- 容易导致项目失败

**✅ 正确做法**
```
需求 1：系统响应时间 < 200ms（在 1000 并发下）
需求 2：支持 1000+ 并发用户
需求 3：采用 TLS 1.2 + AES-128 加密（平衡性能和安全性）
需求 4：系统成本 < 50 万元

权衡决策：
- 选择性能优先，因为用户体验是核心竞争力
- 选择 AES-128 而不是 AES-256，以平衡性能和安全性
- 使用云服务的自动扩展功能，以降低成本
```

---

### 错误 3：一个 SysReq 9 级节点映射到多个 PA 末级节点

**❌ 错误示例**
```
SysReq-9-001：浏览商品列表
├── PA-FE-001：商品列表页面
├── PA-BE-001：商品查询 API
└── PA-DB-001：商品数据库
```

**问题**：
- 一个 SysReq 9 级节点应该只映射到一个 PA 末级节点
- 多个映射会导致需求追踪困难

**✅ 正确做法**
```
SysReq-9-001：浏览商品列表 → PA-FE-001：商品列表页面
SysReq-9-002：搜索商品 → PA-FE-001：商品列表页面 + PA-BE-001：商品查询 API
SysReq-9-003：查看商品详情 → PA-FE-002：商品详情页面
```

---

### 错误 4：忽视组件之间的关系

**❌ 错误示例**
```
PA-FE-001：商品列表页面
PA-BE-001：商品查询 API
（没有记录它们之间的关系）
```

**问题**：
- 无法理解系统架构
- 无法进行有效的开发
- 容易遗漏关键的集成点

**✅ 正确做法**
```
PA-FE-001：商品列表页面
  相关组件：PA-BE-001（商品查询 API）、PA-DB-001（商品数据库）
  调用关系：PA-FE-001 → PA-BE-001 → PA-DB-001

PA-BE-001：商品查询 API
  相关组件：PA-FE-001（商品列表页面）、PA-DB-001（商品数据库）
  调用关系：PA-FE-001 → PA-BE-001 → PA-DB-001
```

---

### 错误 5：缺少非功能需求的实现方案

**❌ 错误示例**
```
非功能需求：系统响应时间 < 200ms
（没有说明如何实现）
```

**问题**：
- 无法进行有效的架构设计
- 无法进行有效的开发
- 容易导致需求无法满足

**✅ 正确做法**
```
非功能需求：系统响应时间 < 200ms

实现方案：
1. 使用 Redis 缓存购物车数据（PA-CACHE-001）
2. 使用 CDN 加速静态资源（PA-CDN-001）
3. 使用数据库连接池优化数据库访问
4. 使用消息队列异步处理订单（PA-MQ-001）
5. 使用数据库索引优化查询性能

预期效果：
- 平均响应时间 < 200ms
- P99 响应时间 < 500ms
```

## 三、最佳实践

### 最佳实践 1：使用结构化格式

**推荐格式**：YAML 或表格

```yaml
组件ID: PA-FE-001
组件名称: 商品列表页面
组件类型: 前端页面
技术栈: React + TypeScript
职责: |
  显示商品列表，支持搜索、筛选、排序等功能

承接的 SysReq 9级: SysReq-9-001, SysReq-9-002
相关组件: PA-FE-002（商品详情页面）、PA-BE-001（商品查询API）

关键功能:
  - 商品列表展示
  - 搜索功能
  - 筛选功能
  - 排序功能

关键数据:
  - 商品列表
  - 搜索关键词
  - 筛选条件
```

**优势**：
- 信息完整
- 易于解析
- 易于版本控制

---

### 最佳实践 2：明确组件职责

**每个 PA 末级节点都应该有明确的职责**

```
❌ 错误：订单组件
✅ 正确：订单 API（明确的组件类型和职责）

❌ 错误：用户组件
✅ 正确：用户认证 API（明确的组件类型和职责）

❌ 错误：商品组件
✅ 正确：商品查询 API（明确的组件类型和职责）
```

**优势**：
- 便于理解
- 便于沟通
- 便于开发

---

### 最佳实践 3：记录组件关系

**为 PA 末级节点记录相关的组件和调用关系**

```
PA-FE-001：商品列表页面
  相关组件：PA-BE-001（商品查询 API）、PA-DB-001（商品数据库）
  调用关系：PA-FE-001 → PA-BE-001 → PA-DB-001
  
PA-BE-001：商品查询 API
  相关组件：PA-FE-001（商品列表页面）、PA-DB-001（商品数据库）
  调用关系：PA-FE-001 → PA-BE-001 → PA-DB-001
```

**优势**：
- 便于理解系统架构
- 便于开发和测试
- 便于维护和优化

---

### 最佳实践 4：明确非功能权衡

**每个非功能需求都应该有明确的实现方案和权衡决策**

```
非功能需求：系统响应时间 < 200ms

权衡决策：
- 选择性能优先，因为用户体验是核心竞争力
- 使用缓存优化性能，增加缓存成本
- 使用异步处理优化性能，增加系统复杂度

实现方案：
1. 使用 Redis 缓存购物车数据
2. 使用消息队列异步处理订单
3. 使用数据库索引优化查询性能
4. 使用 CDN 加速静态资源
```

**优势**：
- 便于架构设计
- 便于工作量估算
- 便于技术选型

---

### 最佳实践 5：使用分层架构

**使用分层架构组织 PA 末级节点**

```
表现层（Presentation Layer）
├── PA-FE-001：商品列表页面
├── PA-FE-002：商品详情页面
├── PA-FE-003：购物车页面
└── ...

业务逻辑层（Business Logic Layer）
├── PA-BE-001：商品查询 API
├── PA-BE-002：购物车 API
├── PA-BE-003：地址 API
└── ...

数据访问层（Data Access Layer）
├── PA-DB-001：商品数据库
├── PA-DB-002：购物车数据库
├── PA-DB-003：地址数据库
└── ...

基础设施层（Infrastructure Layer）
├── PA-CACHE-001：Redis 缓存
├── PA-MQ-001：消息队列
├── PA-CDN-001：CDN 加速
└── ...
```

**优势**：
- 清晰的架构结构
- 便于开发和维护
- 便于扩展和优化

## 五、常见问题解答

**Q：一个 SysReq 9 级节点可以映射到多个 PA 末级节点吗？**

A：可以。例如：
- SysReq-9-002：搜索商品 → PA-FE-001（前端搜索框）+ PA-BE-001（后端搜索 API）
- 但每个 SysReq 9 级节点应该只映射到一个主要的 PA 末级节点

**Q：如何处理跨越多个 PA 末级节点的功能？**

A：
1. 识别功能涉及的 PA 末级节点
2. 为每个 PA 末级节点定义相应的职责
3. 记录 PA 末级节点之间的调用关系
4. 在开发阶段处理跨组件集成

**Q：如何确定 PA 末级节点的粒度？**

A：
- 前端组件：通常对应一个页面或页面中的主要模块
- 后端组件：通常对应一个业务功能或 API
- 数据库组件：通常对应一个业务实体或数据表
- 其他组件：通常对应一个基础设施服务

**Q：如何处理非功能需求的实现？**

A：
1. 识别非功能需求
2. 分析非功能需求与功能需求的关系
3. 设计实现方案（如缓存、异步处理等）
4. 在 PA 末级节点中体现实现方案
5. 记录权衡决策
