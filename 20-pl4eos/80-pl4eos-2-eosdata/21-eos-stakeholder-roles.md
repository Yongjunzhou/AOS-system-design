## 正文区

> 正文区按角色 ID 组织，每个角色一个条目。每个条目标注 **状态**（已确认 / 疑似）标记置信度。
>
> - **已确认**：经过人类确认，可用于 STR 树骨架建立、FR/NFR 角色分配
> - **疑似**：AI 从 OR 文本提取的候选角色，暂不应用于结构约束
>
> **角色类型**区分：**用户角色**（直接操作目标系统，可提 FR+NFR）vs **相关方角色**（不直接操作但关心质量属性，仅 NFR）。详见通用设计规范 §4.3.4。

### 结构模板

```markdown
## {角色名称} `@{role-id}`

**状态**：已确认 / 疑似
**角色类型**：用户角色 / 相关方角色
**关注需求类型**：FR / NFR / FR+NFR
**定义**：[角色的详细定义——在 EOS 中的职责和交互范围]
**典型 NFR 维度**：[该角色天然关注的 NFR 维度，引用 22 号文档中的分类条目 ID]
**首次识别**：YYYY-MM-DD · {skill}
**最后确认**：YYYY-MM-DD · {skill}
**操作反馈**：[确认] / [修改]→{新名称} / [删除]

**承担的 OR 末级**：

| OR 编号 | 摘要 | 识别 Skill |
|---------|------|-----------|

**相关的 STR 节点**：

| STR 节点 | 名称 | 状态 |
|---------|------|------|
```

## 各 Skill 读写契约

| Skill | 操作 | 说明 |
|-------|------|------|
| **ort03** | Step 4 写 + Step 3 修订 | Step 4（角色资产管理）汇总全部分析建议块中的角色字段，写入/更新本文档正文区。Step 3 检测操作反馈列并执行修订（确认→已确认；修改→更新；删除→移除）。同时将标准名写回分析建议块 |
| **wft01** | Step 1 读 | Step 1 加载本文档正文区，获取已确认角色清单作为 STR 树一级骨架来源 + OR→角色映射依据 |
| **wft02~wft06** | 引用 | 全链路引用角色信息作为各层分解的角色归属依据 |

### 状态迁移

```
ort03                       wft01 · wft02~wft06
 Step 4 汇总写入 ───→  人类确认 ───→  已确认
  ├─ 新增疑似角色             （在正文区条目设操作反馈=确认）
  ├─ 归并同义词角色
  └─ 检测操作反馈列并执行修订
      ├─ [确认] → 已确认
      ├─ [修改] → 更新角色名
      └─ [删除] → 移除条目
```
