## AI读取文档说明

本文件是角色资产文档，角色数量会随 OR 分析和后续系统设计持续增长。AI 读取本文件时应按需读取，不应默认读取全部角色正文。

默认读取顺序：

1. 读取文档元信息和本说明，确认本文档是角色资产的权威来源。
2. 读取 `0. 角色资产全貌`，了解角色类型、状态口径和分片方式。
3. 读取 `1. 角色资产状态与处理队列`，确认本轮是否存在待确认、待归并、待废弃或待转交角色。
4. 读取 `2. 角色索引 / 分片索引`，按角色 ID、角色类型、状态或来源 OR 定位目标角色块。
5. 仅读取目标角色正文块、相关追溯关系，以及需要写回的反馈区。

扩大读取范围仅在以下情况允许：

- 角色索引缺失或角色 ID 无法定位。
- 多个候选角色存在同义、上下位或边界冲突。
- 需要为 STR 树建立一级骨架并进行全局角色去重。
- 人类明确要求通读角色资产。

禁止事项：

- 不得在常规 Skill 运行中默认读取全部角色正文块。
- 不得将 `疑似` 角色直接作为 STR 树骨架的稳定来源。
- 不得只凭对话上下文修改角色状态；人类反馈必须写入队列、索引或目标角色反馈区。

## 3. 规则区 / 口径区

### 3.1 角色识别规则

1. 角色不是被设计出来的，而是在原始需求、业务材料、人类反馈和运行实践中被发现并确认。
2. 用户角色可承载功能需求和非功能需求；相关方角色默认只承载非功能需求。
3. 角色名称应表达业务身份，不以单个系统页面、岗位编制或个人姓名命名。
4. 同一角色在不同业务域职责差异明显时，可拆分为领域化角色；差异不明显时优先归并。
5. `疑似` 角色不得直接进入 STR 稳定骨架，必须经确认或转化为待处理项。

### 3.2 Skill 读写契约

| Skill | 操作 | 说明 |
|-------|------|------|
| `ort03` | 读 + 写 | 汇总 OR 分析建议块中的角色字段，写入或更新角色资产，并处理人类操作反馈 |
| `wft01` | 读 | 读取已确认用户角色作为 STR 功能树一级骨架来源，读取相关方角色支持 NFR 识别 |
| `wft02~wft06` | 引用 | 在 BA、SysReq、PA 和验证链路中引用角色归属 |
