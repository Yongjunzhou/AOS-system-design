### 1.2 引擎层次关系

```mermaid
graph BT
  subgraph 实体级
    ENTITY[实体引擎<br/>物理表]
  end
  subgraph 功能级
    FUNC[功能引擎<br/>功能表单]
    FLOW[流程引擎]
    IND[指标引擎]
    WO[工单引擎]
  end
  subgraph 部件级
    COMP[部件引擎<br/>PL5 场景部件]
  end
  subgraph 场景级
    SCENE[场景引擎<br/>SL5 场景业务]
  end
  subgraph 菜单级
    MENU[菜单引擎<br/>ML4 末级菜单及高层菜单]
  end
  ENTITY -->|提供物理表元数据| FUNC
  FUNC --> FLOW
  FUNC --> IND
  FUNC --> WO
  FLOW -->|生成功能表单| COMP
  IND -->|生成功能表单| COMP
  WO -->|生成功能表单| COMP
  COMP -->|组装场景部件| SCENE
  SCENE -->|生成末级菜单输入| MENU
```

**五层装配链文本图**：

```text
实体引擎 ──→ 物理表  ← 基于数据库生成物理表，并为该物理表配置访问权限
  │
  └── 功能引擎 ──→ 功能表单  ← 将物理表转换成功能表单，功能表单以一定样式展现，并使得物理表中的数据具有一定的运算、流转和被操作的能力
        │
        └── 部件引擎 ──→ 场景部件（PL5）  ← 将一组 1+N 个功能表单组装成一父多子的场景部件，父表单数据与子表单数据是父子业务关系，用户在父表单选中一行数据双击下钻可看到其所有子表单数据并进行处理；若子表单也被部件引擎配置为场景部件，可继续下钻
              │
              └── 场景引擎 ──→ 场景业务（SL5）  ← 将多个场景部件编排成一个场景业务，用户在该场景业务下可完成多个场景部件提供的业务
                    │
                    └── 菜单引擎 ──→ 末级菜单（ML4）  ← 把场景业务配置并发布为 EOS 平台末级菜单，生成高层级菜单节点（末级菜单节点的分类分组），为菜单节点配置可见权限
```

**关键编排关系**：

- EOS 平台按五层引擎逐级装配：实体引擎生成物理表 → 功能引擎将物理表转换为功能表单 → 部件引擎将 1+N 个功能表单组装为场景部件（PL5）→ 场景引擎将多个场景部件编排为场景业务（SL5）→ 菜单引擎生成末级菜单（ML4）并分组分配高层菜单。
- 功能引擎负责将物理表转换成功能表单：功能表单以一定样式展现，并使得物理表中的数据具有一定的运算、流转和被操作的能力。功能引擎包括但不限于：`@engine-kanban` 看板引擎、`@engine-indicator` 指标引擎、`@engine-wbs` WBS 引擎、`@engine-task` 计划引擎、`@engine-project` 项目引擎、`@engine-workorder` 工单引擎、`@engine-ledger` 台账引擎、`@engine-flow` 流程引擎、`@engine-scheduler` 定时任务引擎、`@engine-progress` 进展引擎。
- 部件引擎负责把一组（1+N 个，N≥0）功能表单组装成一父多子模式的场景部件。父表单与子表单的数据是父子业务关系（如父表单为员工信息，子表单可能是培训记录、考勤记录、考评记录、任务清单、资产清单等）；用户在父表单选中一行数据双击下钻，可看到其所有子表单数据并进行处理；若子表单也被部件引擎配置为场景部件，可继续下钻。
- 场景引擎负责把端到端业务的多个场景部件编排在一起，使用户在一个场景中完成端到端业务。
- 菜单引擎负责把场景业务配置并发布为 EOS 平台末级菜单，生成高层级菜单节点（末级菜单节点的分类分组），为菜单节点配置可见权限。
- 权限配置能力不设独立权限引擎：菜单节点对角色 / 人员的可见或不可见规则由菜单引擎承载；物理表数据行或列对角色 / 人员的权限由实体引擎承载。
- 下钻层级由实体关系穿越次数决定，达到系统层级数限制时停止

**配置单元选配模型**：每个引擎包含若干配置单元（可独立选配的 IPO），部分配置单元即可让业务运行——引擎提供能力池，配置人员按业务需要从中挑选组合，非全量必配。

| 来源节点 | — |

## 附录A 规则区与口径区

1. 引擎描述平台能力边界，不替代产品架构组件或代码实现设计。
2. 引擎应能追溯到 `23` 构件类型、`24` 业务或 `04` BA A1 方案。
3. 同一引擎可被多个业务复用；复用关系记录在索引和追溯区。
4. 外购、自研、自研+外购是架构级实现策略，不代表采购或开发已经完成。
5. 对外接口只记录架构级能力和输入输出摘要，详细 API 规格应在后续产品本体或开发文档中定义。
6. 引擎配置单元类型描述某个引擎可独立配置、校验、发布、复用或版本管理的元数据对象类型；具体业务的配置单元实例不写入本文档，应由 `04-eos-business-architecture.md` 的 A1 BA 方案或后续 PA / 配置资产承载。
7. `wft02` 生成 A1 引擎配置单元建议时，应优先查询本文档的配置单元类型库，再按当前 Bn 业务、A2 场景业务或公共业务实例化；若本文档不存在可承接的配置单元类型，应形成 25 资产新增 / 扩展建议或引擎能力缺口反馈。
8. **术语统一**：本文档与 wft01 统一使用"配置单元"术语。配置单元的三个核心字段为 Input（配置要素）/ Process（处理说明）/ Output（业务说明）。
9. **功能表单口径**：功能表单是功能引擎将物理表转换后的产物，以一定样式展现，并使得物理表中的数据具有一定的运算、流转和被操作的能力；实体引擎的产物称为物理表。
10. **五层装配链**：物理表 → 功能表单 → 场景部件（PL5）→ 场景业务（SL5）→ 末级菜单（ML4）是 EOS 平台引擎装配的主链路。
11. **权限配置归属**：不设置独立权限引擎。菜单节点可见 / 不可见规则由菜单引擎承载；物理表数据行 / 列权限由实体引擎承载。

- 多个引擎存在职责边界冲突或替代关系。

## 附录E 业务层级与引擎装配链

EOS 平台的用户可见业务由五层引擎逐级装配生成。物理表是底座，功能表单是功能引擎产物，场景部件 / 场景业务 / 菜单是面向业务使用的组织层级：

| 装配层 | 产物 | 定义 | EOS 平台体现 | 生成引擎 |
|--------|------|------|-------------|---------|
| A1 | 物理表 | 业务对象的数据结构、物理存储结构和关系约束 | 物理表及其元数据、字段名称 / 类型 / 格式、主键、外键、索引字段、字段业务逻辑、实体关系、生命周期 | 实体引擎 |
| A2 | 功能表单 | 物理表转换后的功能表单，以一定样式展现，并使得物理表中的数据具有一定的运算、流转和被操作的能力 | 功能表单承载样式展现、字段间取值规则、显示规则、校验、行为、发布，以及流程、工单、指标、台账、看板、WBS、项目、定时任务、进展等功能能力；通常通过表单标签上的功能按钮、右键操作项或快捷键触发 | 功能引擎 |
| A3 | 场景部件 | 具有父子业务关系的一组功能表单，支持父表单行数据双击下钻到子表单 | PL5 场景部件 | 部件引擎 |
| A4 | 场景业务 | 多个场景部件组成的业务场景 | SL5 场景业务 | 场景引擎 |
| A5 | 末级菜单 | 用户访问单个场景业务的入口 | ML4 末级菜单；可分组分配为 ML1~ML3 高层菜单 | 菜单引擎 |

**与设计视角层级的映射**：`PL5` 对应场景部件，由部件引擎承接；`SL5` 对应场景业务，由场景引擎承接；`ML4` 对应末级菜单，由菜单引擎承接并指向单个 `SL5` 场景业务。`ML1~ML3` 是菜单分组路径，不直接生成业务内容。

页面底部常驻导航条，形如资源管理器目录路径，显示当前表单的层级路径：

#### 场景部件实现形态（内置 CAX / 外置 CAX）

每个输出文档对应一个场景部件，但场景部件可以有两种实现形态。

**规则定义**：

> 每个输出文档必须对应一个场景部件。若输出文档由平台内功能引擎生成，则该场景部件的父功能表单为对应功能表单；若输出文档由平台外部 CAX 生成，则该场景部件的父功能表单为计划功能表单，输出文档作为计划功能表单的附件 / 交付物 / 版本化成果管理。

**内置 CAX 型场景部件**

平台内有对应功能引擎，能够生成真正的父功能表单。例如：流程、工单、指标、台账、WBS、计划等。此时装配链为：

```
输出文档 → 场景部件 → 父功能表单 → 子功能表单
```

**外置 CAX 型场景部件**

输出文档由平台外部工具生成，平台内不直接生成该文档内容。但平台仍然需要一个"承载型父功能表单"，用于管理该输出文档的任务、状态、责任人、版本、附件、评审、归档。最自然的承载者是计划功能表单。此时装配链为：

```
输出文档 → 场景部件 → 计划功能表单（父表单 / 承载表单）→ 附件
```

也就是说，外部 CAX 的输出文档不是"无对应场景部件"，而是：有场景部件，但该场景部件的父功能表单不是"文档生成表单"，而是"计划 / 任务承载表单"；输出文档作为该父功能表单的附件、交付物或版本化成果存在。

**需求链路全景**：

```
SL5 场景业务
  │
  ├── 输出文档1 → 场景部件1 → 内置功能表单
  ├── 输出文档2 → 场景部件2 → 计划功能表单 + 外部 CAX 附件
  ├── 输出文档3 → 场景部件3 → 内置功能表单
  └── 输出文档N → 场景部件N → 计划功能表单 + 外部 CAX 附件
```

**两种形态对比**：

| 维度 | 内置 CAX | 外置 CAX |
|------|-----------------|-----------------|
| **处理主体** | 引擎算法 | 人类 + CAX 工具 |
| **人类角色** | 操作引擎界面 | 使用专业工具完成实质转换 |
| **场景部件职责** | 提供操作界面 + 处理算法 | 提供规则/知识/合规/资源，收纳输出附件 |
| **父功能表单** | 对应功能引擎生成的功能表单（如流程功能表单） | 计划功能表单（承载型，管理任务/状态/责任人/版本/附件/评审/归档） |
| **输入文档** | 引擎读取并处理 | 人类取出，导入 CAX 工具 |
| **输出文档** | 引擎生成 | 人类在 CAX 中产出，作为附件回传 |
| **CU 选配剖面** | 完整选配（含校验/行为联动） | 轻量选配（不选校验/行为） |

两种方式均由现有功能引擎和部件引擎支撑，区别在**配置单元的选配剖面**不同，不需要新引擎类型。wft02-biz 展开时按开发方式推荐 CU 选配剖面。

> **外置 CAX 场景部件与部件引擎的关系**：外置 CAX 场景部件仍符合部件引擎"1+N 个功能表单组装成一父多子模式的场景部件"的模型（N≥0）。外置 CAX 下 N=0（无非计划功能表单），输出文档不作为子表单挂接，而是作为父表单的附件 / 交付物 / 版本化成果管理。附件能力由实体引擎的内置子业务配置单元（`@entity-cu-builtin-subbiz`）提供，不改变部件引擎的装配模型。
