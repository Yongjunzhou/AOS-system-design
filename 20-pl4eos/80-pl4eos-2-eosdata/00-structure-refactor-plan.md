### 一、改造目标

本次改造以 [文档编写约定 §十四](../../00-generalspec/00-doc-conventions.md#十四树形资产文档布局规范) 为统一约束，将 `80-pl4eos-2-eosdata/` 下的系统设计过程文档和资产文档升级为可按需读取的树形资产文档。

核心目标如下：

1. 支持 Skill 先读取树画像 / 资产全貌、状态队列和节点索引，再按目标节点读取正文块。
2. 支持 `wft01a/wft01b/wft02a/wft02b` 的活动锁、单 ID 待办、反馈转写和跨 Skill 转交。
3. 避免在节点规模达到几千条时默认全量读取正文。
4. 将人类审阅入口、AI 写回入口、状态队列、追溯关系和资产引用关系显式化。
5. 将历史内容交由 Git 管理，文档内仅保留当前正文与轻量变更记录。

### 三、目标文档结构

系统设计过程文档和资产文档统一采用“前置信息区 + 树形资产区”结构。前置信息区用于人类和 AI 快速理解文档用途与读取方式；树形资产区严格沿用 [文档编写约定 §14.1](../../00-generalspec/00-doc-conventions.md#141-统一布局) 的 `0~9` 编号。

```text
前置信息区
├─ 文档元信息
└─ AI读取文档说明

树形资产区
0. 树画像 / 资产全貌
1. 状态与处理队列
2. 节点索引 / 分片索引
3. 规则区 / 口径区
4. 正文节点块
5. 追溯关系
6. 反馈区 / 建议区
9. 文档版本与变更记录
```

各区职责如下：

| 区域 | 编号 | 职责 | Skill 读取方式 |
|------|------|------|---------------|
| 文档元信息 | 前置 | 标识文档版本、状态、用途和适用范围 | 首次读取 |
| AI读取文档说明 | 前置 | 面向人类和 AI 说明本文件的读取入口、禁止全量读取场景、局部读取路径和扩大读取触发条件 | 首次读取 |
| 树画像 / 资产全貌 | 0 | 提供节点规模、层级、分片、设计风格摘要 | Step Start 必读 |
| 状态与处理队列 | 1 | 承载当前可处理对象、活动锁、当前待办 ID、人类决策 | Step Start 必读 |
| 节点索引 / 分片索引 | 2 | 将节点 ID、父子关系、块 ID、状态和追溯关系建立定位入口 | 按目标对象读取 |
| 规则区 / 口径区 | 3 | 记录本文件专属识别规则、边界口径、命名与归并原则 | 按 Skill 需要读取 |
| 正文节点块 | 4 | 存放节点定义、承接关系、设计指导、局部反馈区 | 只读目标块和必要相关块 |
| 追溯关系 | 5 | 记录跨文档、跨层级、跨资产引用关系 | 按目标对象读取 |
| 反馈区 / 建议区 | 6 | 承载全局反馈、资产建议、待确认项 | 按目标对象读取和写回 |
| 文档版本与变更记录 | 9 | 记录轻量变更历史 | 版本检查时读取 |

### 四、AI读取文档说明标准

每份被改造文档均应在文档元信息之后增加 `### AI读取文档说明`。该说明既约束 AI 的读取行为，也让人类理解该文档为什么不要求一次性读完整正文。

该区面向人类可读，应避免写成内部提示词。说明内容应清晰解释：AI 默认读哪些区、何时读正文块、何时允许扩大读取范围、哪些内容不应作为当前资产读取。

#### 4.1 标准模板

```markdown
### AI读取文档说明

本文件是树形资产文档，节点数量可能持续增长。AI 读取本文件时应按需读取，不应默认读取全部正文。

默认读取顺序：

1. 读取文档元信息和本说明，确认文档用途、状态和读取规则。
2. 读取树画像 / 资产全貌，建立整体结构感知。
3. 读取状态与处理队列，确定本轮目标对象、活动锁和当前待办 ID。
4. 读取节点索引 / 分片索引，定位目标正文块和必要相关块。
5. 仅读取目标正文块、追溯关系指向的相关块，以及本轮需要写回的反馈区或建议区。

扩大读取范围仅在以下情况允许：

- 节点索引缺失或块 ID 无法定位。
- 追溯关系断裂，无法判断上游或下游关系。
- 候选节点或候选资产存在边界冲突。
- 需要执行全局一致性检查或批量迁移。
- 人类明确要求通读全文。

禁止事项：

- 不得在常规 Skill 运行中默认全量读取正文节点块。
- 不得将历史区、旧版正文或废弃节点作为当前资产参与判断。
- 不得只凭对话上下文推进状态；人类反馈必须先写入反馈区、状态队列或转交记录。
```

#### 4.2 文档类型差异

不同文档的 AI 读取说明应保留相同骨架，但允许按文档类型补充重点：

| 文档类型 | 重点补充 |
|----------|---------|
| `01` OR 文档 | 说明先读 OR 状态队列和 OR 节点索引，再读目标 OR 正文块 |
| `02` STR 架构文档 | 说明先读 STR 状态队列、活动锁、十链归属状态和 STR 节点索引 |
| `03` STR 详细定义文档 | 说明先读 STR 详细定义状态队列、节点索引和待写回 / 待审阅详细定义块 |
| `04` BA 架构文档 | 说明先读 BA 状态队列、待资产写回队列和 BA 节点索引 |
| `05` SysReq 架构文档 | 说明先读 SysReq 状态队列、节点索引和目标 SysReq 架构块 |
| `06` SysReq 详细定义文档 | 说明先读 SysReq 详细定义状态队列、节点索引和待写回 / 待审阅详细定义块 |
| `07` PA 架构文档 | 说明先读 PA 组件状态队列、组件索引和目标组件块 |
| `21` 角色资产 | 说明先读角色资产全貌和角色索引，再按角色 ID 读取正文块 |
| `22` NFR 资产 | 说明先读 NFR 资产全貌和分类索引，再按分类维度或具体需求 ID 读取正文块 |
| `23` 输出产品架构资产 | 说明先读资产全貌和分片索引，再按业务域 / 产品类型 / 构件类型读取候选块 |
| `24~27` 配套资产 | 说明先读资产索引，再按业务定义、引擎模型、文档资产或资源资产 ID 读取正文块 |
| `08/09` 验证类文档 | 说明先读验证任务队列、问题索引和追溯关系，再读目标问题块 |

### 五、正文块与索引标准

#### 5.1 正文块格式

每个可被 Skill 独立读取或写回的节点均应使用稳定块标识：

```markdown
<!-- BLOCK: STR-F-001 -->
#### STR-F-001 采购员采购申请处理场景

**当前状态**：待文档归属
**父节点**：ROLE-PURCHASER
**块ID**：STR-F-001

| 要素 | 内容 |
|------|------|
| 节点定义 | ... |
| 承接关系 | ... |
| 符合性分析 | ... |
| 为下层提供的设计指导 | ... |

##### 反馈区
<!-- 人类对话反馈由 AI 转写到这里 -->

<!-- /BLOCK: STR-F-001 -->
```

块标识规则：

1. `BLOCK` ID 应稳定，不随节点名称微调而变化。
2. 节点索引中的 `块ID` 必须能定位到正文块。
3. 一个节点的状态、定义、承接关系和局部反馈区应尽量放在同一正文块内。
4. 跨文档关系只记录引用，不复制对方正文。

#### 5.2 节点索引字段

系统设计过程文档的节点索引建议包含：

| 字段 | 说明 |
|------|------|
| 节点ID | 稳定节点编号，如 `STR-F-001`、`BA-IPO-001` |
| 节点名称 | 当前名称 |
| 节点类型 | 场景定义 / 业务定义 / 系统需求 / PA组件等 |
| 父节点ID | 上级节点编号 |
| 层级 | 架构层级或文档内层级 |
| 是否末级 | 是 / 否 |
| 当前状态 | 进入下游或等待反馈的状态 |
| AI建议 | 当前 AI 建议 |
| 人类决策 | 人类选材或反馈决策 |
| 追溯来源 | 上游节点 ID |
| 下游承接 | 下游节点 ID |
| 块ID | 正文块定位 ID |
| 反馈入口 | 节点反馈区位置 |
| 最后更新 | 日期 |

资产文档的节点索引建议包含：

| 字段 | 说明 |
|------|------|
| 资产ID | 如 `@node-form`、`@def-xxx`、`@doc-xxx` |
| 名称 | 资产名称 |
| 资产类型 | 业务域 / 产品类型 / 构件类型 / 业务定义 / 文档定义 / 资源资产等 |
| 父级ID | 上级资产 |
| 分片ID | 所属分片 |
| 生命周期状态 | 已确认 / 疑似 / 待确认 / 已废弃 |
| AI建议 | 当前建议 |
| 人类决策 | 当前决策 |
| 引用来源 | STR / BA / SysReq / 人类规则等 |
| 块ID | 正文块定位 ID |
| 最后更新 | 日期 |

### 六、状态队列与活动锁标准

每个由 Skill 操作的过程文档应提供状态与处理队列。队列至少包含以下字段：

| 字段 | 说明 |
|------|------|
| 待办ID | 本轮人类或 AI 需要处理的单一对象 ID |
| 待办类型 | 方案审阅 / 文档归属 / 资产写回 / 详细定义审阅 / 下游选择 |
| 关联节点ID | 被处理的节点 |
| 当前状态 | 节点当前生命周期状态 |
| 活动锁 | 无 / 已锁定 / 已释放 / 已转交 |
| 锁定Skill | 当前占用该节点的 Skill |
| 下一步Skill | 推荐进入的下游 Skill |
| 人类决策 | 当前人类输入 |
| AI建议 | 当前 AI 建议 |
| 剩余待办数量 | 同类剩余待办数量 |
| 转交记录 | `选择进入wft01b` / `选择进入wft02a` / `选择进入wft02b` 等 |
| 反馈入口 | 文档内反馈区或节点块 |
| 最后更新 | 日期 |

状态队列规则：

1. 每个 Skill 同一时间只处理一个活动节点。
2. Step End 每次只输出一个当前待办 ID。
3. 对话反馈必须先转写到文档反馈区或状态队列，再推进状态。
4. 状态到位不等于下游可自动处理；必须有选择记录或活动锁释放 / 转交记录。
5. 若同一 Skill 检测到多个活动节点，应停止新增处理并输出冲突说明。

资产文档应提供资产处理队列或资产建议队列，不强制套用过程文档的状态队列表。资产队列至少记录：资产建议 ID、资产 ID、建议类型、生命周期状态、AI建议、人类决策、关联来源、目标块 ID、反馈入口和最后更新。`23~27` 的新增、扩展、归并、废弃、待确认引用均通过资产队列或资产索引行留痕。

### 七、追溯关系标准

追溯关系区应记录节点间关系，不复制对方正文。

建议关系类型如下：

| 关系类型 | 含义 | 示例 |
|----------|------|------|
| `derived_from` | 从上游材料或节点推导而来 | STR 节点 derived_from OR 条目 |
| `refines` | 对上游架构节点做详细化 | STR 详细条目 refines STR 架构节点 |
| `allocated_to` | 上游需求分配到下游方案节点 | OR allocated_to STR |
| `satisfies` | 下游节点满足上游需求 | BA satisfies STR 详细条目 |
| `uses_asset` | 节点引用资产条目 | STR uses_asset @nfr-001 |
| `produces_doc` | 节点产生文档资产 | BA produces_doc @doc-xxx |
| `consumes_doc` | 节点消费文档资产 | BA consumes_doc @doc-yyy |
| `implemented_by` | 需求或定义由组件承接 | SysReq implemented_by PA |

追溯关系表建议字段：

| 源节点 | 关系类型 | 目标节点 | 目标文档 | 状态 | 说明 |
|--------|----------|----------|----------|------|------|

### 八、分片策略

当节点数量达到数百条以上时，应采用分片读取策略。分片层级如下：

```text
L0 全局画像：节点总数、层级结构、分片目录、设计风格摘要
L1 分片目录：按业务域、产品类型、角色、状态或架构层级列出分片
L2 分片摘要：描述目标分片内节点类型、边界、命名和典型对象
L3 节点正文块：具体节点定义
```

分片建议：

| 文档 | 推荐分片维度 |
|------|-------------|
| `02-eos-stakeholder-requirements-architecture.md` | 角色类型 / 需求类型 / 状态 |

| `04-eos-business-architecture.md` | 业务域 / 产品类型 / A1-A2-Bn 路径 / 状态 |
| `05-eos-system-requirements-architecture.md` | 业务定义类型 / NFR 分类 / 状态 |
| `06-eos-system-requirements-detailed.md` | SysReq 节点 / 业务定义类型 / 细节审阅状态 |
| `07-eos-product-architecture.md` | 组件层 / 产品类型 / 业务链与平台链 |
| `21-eos-stakeholder-roles.md` | 角色类型 / 状态 / 来源 OR |
| `22-eos-nfr-taxonomy.md` | 一级类目 / 分类维度 / 具体需求状态 |

### 九、状态权威源规则

为避免状态冲突，统一采用以下规则：

1. 对象所在文档的状态队列是该对象的权威状态源。
2. 集中状态文档只作为派生仪表盘，不直接作为下游 Skill 的唯一入口。
3. OR 预处理阶段的材料级状态可由 `00-origin-requirement-materials/01-eos-sysdev-status.md` 管理。
4. OR 条目写入 `01-eos-original-requirements.md` 后，OR 条目级状态应逐步迁入 01 文档状态队列。
5. STR 节点状态以 `02-eos-stakeholder-requirements-architecture.md` 的状态队列为准。
6. BA 节点状态以 `04-eos-business-architecture.md` 的状态队列为准。
7. SysReq 节点状态以 `05-eos-system-requirements-architecture.md` 的状态队列为准。
8. PA 组件状态以 `07-eos-product-architecture.md` 的状态队列为准。
9. 资产生命周期状态以各资产文档自身的资产全貌 / 状态字段为准。

迁移期规则：

1. 目标文档状态队列尚未建成前，沿用现有状态文档、内嵌状态表或资产表字段作为兼容入口。
2. 新状态队列建成后，应先与旧状态源做一次同步校验，再切换为权威源。
3. 切换完成后，旧状态源应降级为派生视图或历史兼容说明，不再作为下游 Skill 的唯一入口。
4. 若迁移期检测到旧状态源与新状态队列冲突，应停止状态推进，先输出冲突说明和人工确认项。

### 十一、迁移原则

1. 先建索引和队列，再迁移正文。
2. 先改空模板和骨架文档，再改已有大量正文的资产文档。
3. 每批只处理一组语义相近文档，避免跨批状态口径混杂。
4. 不在改造中重写业务内容，除非原内容是明显占位或链接错误。
5. 所有正文迁移应保持节点 ID、资产 ID、追溯引用稳定。
6. 新增字段优先使用空值或占位说明，不伪造状态。
7. 若现有规则与目标结构冲突，应以上层规范和 `91-doc-paired-skill-spec.md` 为准，并记录冲突。
8. 每份被改造文档必须新增 AI读取文档说明，并根据文档类型说明默认读取入口和扩大读取触发条件。
