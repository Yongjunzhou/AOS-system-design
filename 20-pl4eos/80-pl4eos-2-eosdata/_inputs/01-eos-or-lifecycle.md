## 作为 Skill 间接口

登记表是 EOS 预处理链（ort00-chunk-propose → ort01-chunk-execute → ort02-annotate → ort03-clarify → ort04-norm → t01 ...）中各顺序 Skill 间的唯一共享接口。每个 Skill 从登记表读取符合输入条件的记录，处理后将结果状态写回登记表，下游 Skill 通过生命周期状态条件感知上游产出。Skill 之间不直接共享文件路径或会话上下文。

典型状态流转（非切片材料）：
```
ort02-annotate → ort03-clarify: 已登记 → 已澄清（+ 人类决策=已确认）
  → ort04-norm: 已澄清 → 已规范化
    → t01: 已规范化 → ...
```

典型状态流转（需切片材料）：
```
ort00-chunk-propose: 原始材料（切片方案区域）→ 建议/方案/理由
  → ort01-chunk-execute: 原始材料 → 各切片（登记表主表，生命周期=已登记）
    → ort02-annotate（对每个切片）: 已登记 → 澄清中
      → ort03-clarify: 澄清中 → 已澄清
        → ort04-norm: 已澄清 → 已规范化
          → t01: 已规范化 → ...
```

每个下游 Skill 的输入条件是"生命周期状态 = 上游写入的特定状态"，具体条件由该 Skill 的定义文档约定。此模式确保整条链的 Skill 可独立执行、互不耦合。

## 生命周期状态总览

各条目类型的生命周期状态流转路径：

### 原始需求（完整路径）

**非切片材料**（普通体量，直接走标注→澄清→规范化）：

```
已登记 ──→ 澄清中 ──→ 已澄清 ──→ 规范化中 ──→ 已规范化 ──→ 分配中 ──→ 已基线化
  │          │          │           │             │            │
  └────────────────← 已废弃 →────────────────────────────────────┘
```

**需切片材料**（体量大，先切片再逐片走标注→澄清→规范化）：

```
原始材料（切片方案区域）
  │
  ├─ ort00: 分析提议 → 切片方案区域（待人类确认）
  ├─ ort01: 执行切片 → 各切片在登记表主表注册为"已登记"
  │
  ↓ 各切片独立走以下路径
已登记 ──→ 澄清中 ──→ 已澄清 ──→ 规范化中 ──→ 已规范化 ──→ 分配中 ──→ 已基线化
  │          │          │           │             │            │
  └────────────────← 已废弃 →────────────────────────────────────┘
```

| 状态 | 含义 | 转迁方式 |
|------|------|---------|
| 已登记 | 录入登记表，未开始处理 | AI 自动（初始状态） |
| 澄清中 | ort02-annotate 初处理后 / ort03-clarify 迭代中 | AI 自动（首次标注后） |
| 已澄清 | ort03-clarify 完成，人类确认通过 | 决策门B 人类确认 |
| 规范化中 | ort04-norm 规范化处理中 | AI 自动 |
| 已规范化 | ort04-norm 完成，OR 条目就绪 | AI 自动 |
| 分配中 | t01 映射分配中 | AI 自动 |
| 已基线化 | 全流程完成 | 人类确认（终端） |
| 已废弃 | 被放弃 | 人类确认（终端） |

### 缺陷 / 架构变更（简化路径）

```
已登记 ──→ 处理中 ──→ 待确认 ──→ 已基线化
  │          │                    │
  └────→ 已废弃
```

| 状态 | 含义 | 转迁方式 |
|------|------|---------|
| 已登记 | 录入登记表 | AI 自动（初始状态） |
| 处理中 | AI 执行修复/处理 | AI 自动 |
| 待确认 | AI 完成，等待人类确认 | AI 自动 |
| 已基线化 | 处理完成，人类确认 | 人类确认（终端） |
| 已废弃 | 被放弃 | 人类确认（终端） |

### 三个正交维度的协作方式

```
AI 完成阶段工作
  → 更新生命周期状态（材料走到哪一步了）
  → 输出 AI 建议（"需澄清 / 可确认"或"需澄清——仍有模糊项未解决"）
      ↓
人类看到 AI 建议 + 支撑材料
  → AI 根据文档中的保留操作选项同步更新人类决策（[确认通过]→生命周期→已澄清；[确认终止]→生命周期→已废弃）
```

## 操作流程

### 原始需求

1. **AI 登记**：人类将原始材料放入 `_inputs/10-source-materials/`，告知 AI 登记。AI 创建独立文件并追加到此表（生命周期状态="已登记"）
2. **ort00 切片提议（大文件）**：体量大的材料，AI 执行 ort00-chunk-propose（文本化→语义结构分析→切片方案），结果写入切片方案区域。人类在文本化文档中调整切片标记后进入下一步
3. **ort01 切片执行**：AI 执行 ort01-chunk-execute，读取文本化文档中的切片标记，输出切片文件并在登记表中注册各切片为"已登记"。人类确认切片边界后可进入下一步，或调整标记后重跑
4. **ort02-annotate — 首轮标注**：AI 对每个切片（或非切片材料）执行文本化（如尚未文本化）+ 首轮模糊标注，将生命周期状态更新为"澄清中"，AI 建议="需澄清 / 可确认"，输出行动清单供人类决定下一步
5. **人类沟通与澄清**：人类审查标注后与相关方沟通，将澄清结果记录到文本化文档中
6. **ort03-clarify — 迭代管理**：启动 ort03-clarify。AI 检测文档变更，扫描保留选项：
   - 读到 [确认通过] → 校验三项条件 → 追加确认块 → 生命周期→已澄清
   - 读到 [确认终止] → 追加废弃说明 → 生命周期→已废弃
   - 读到 [继续迭代] 或无保留标记 → 通读文档 → 更新标注状态 → 输出迭代报告
   - 仍有模糊 → 保持"澄清中"，AI 建议="需澄清"
   - 三项标准全满足 → 输出"建议已澄清"，AI 建议="可确认"，追加操作选项
7. **人类确认（决策门B）**：人类审查 AI 的已澄清建议，确认通过后：
   - AI 追加人类确认块到文档，更新登记表中人类决策="已确认"
   - AI 将生命周期状态更新为"已澄清"
8. **ort03-clarify — 批次选材**：AI 扫描"已澄清+人类决策为空"的材料，标记为"候选纳入规范化"
9. **人类选材确认**：人类到登记表中将 人类决策 设为"已确认"（纳入）或"已终止"（废弃）
10. **ort04-norm**：AI 筛选"生命周期状态=已澄清"且"人类决策=已确认"的记录执行原始需求规范化。对切片材料，各片的 OR 条目在此步合并后统一编号
11. **完成**：AI 将生命周期状态更新为"已规范化"→"分配中"→"已基线化"（按实际推进）

**废弃路径**：人类可在任意步骤要求废弃某条原始需求记录。AI 将生命周期状态更新为"已废弃"，并在文档中追加废弃说明。

**督促机制**：AI 每次执行 ort03-clarify 时，在摘要中列出"澄清中 ≥ 3 轮"的记录，提示人类注意长期未解决的项。

### 缺陷/问题

1. **AI 登记**：AI 发现缺陷或收到反馈后，创建 `_inputs/20-issues/ISSUE-YYYYMMDD-NNN.md`（含问题描述+影响评估摘要），追加到此表（生命周期状态="已登记"）
2. **AI 建议**：AI 评估缺陷影响范围和紧急程度，输出 AI 建议（"需澄清 / 可确认"/"需澄清"/"建议终止"）
3. **人类决策**：人类浏览 AI 评估摘要，设置人类决策（"已确认"→继续；"已终止"→废弃）
4. **AI 处理**：AI 筛选"人类决策=已确认"且条目类型=缺陷的记录，更新生命周期状态="处理中"，按分类和定位自动选择处理模式执行 t02/t03/t04
5. **完成**：AI 完成修复后更新生命周期状态="待确认"，AI 建议="需澄清 / 可确认"；人类确认后更新生命周期状态="已基线化"，将 issue 文件移入 `_inputs/30-archived/`

### 架构变更

1. **AI 检测或人类报告**：Source 1 构件（`20-eos-design/10-eos-system-design/`）发生变化后，AI 评估变更影响，创建变更详情文件到 `_inputs/20-issues/CHANGE-YYYYMMDD-NNN.md`，追加到此表（条目类型="架构变更"，生命周期状态="已登记"）
2. **AI 建议**：AI 评估变更影响范围和紧急程度，输出 AI 建议（"需澄清 / 可确认"/"需澄清"/"建议终止"）
3. **人类决策**：浏览 AI 的影响评估，设置人类决策（"已确认"→继续；"已终止"→废弃）
4. **AI 处理**：筛选"人类决策=已确认"的记录，更新生命周期状态="处理中"，按变更类型执行对应操作（更新 PA → 评估上游影响 → 补充推导）
5. **完成**：AI 完成处理后更新生命周期状态="待确认"，AI 建议="需澄清 / 可确认"；人类确认后更新生命周期状态="已基线化"，将变更详情文件移入 `_inputs/30-archived/`
