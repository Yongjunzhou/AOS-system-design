> **产品 B 范围说明**：本台账中的 E2E 项目（产品类型）分为两类——**输出产品**（流水线视角，如采购管理、生产管理）和**角色场景**（工作台视角，如采购员、车间主任）。两者都是 EOS 运行输出的产品。输出产品的 5 级 BA IPO 为原子业务操作，角色场景的 5 级 BA IPO 为编排定义。两类产品分别在各业务域下作为独立 E2E 项目行记录。

## AI读取文档说明

本文件是输出产品架构资产文档，资产行数量会随 STR、BA 和产品数据沉淀持续增长。AI 读取本文件时应按需读取，不应默认读取全部大表。

默认读取顺序：

1. 读取文档元信息和本说明，确认本文档是输出产品架构资产的权威台账。
2. 读取 `0. 输出产品资产全貌`，了解业务域、E2E 项目、构件类型和分片方式。
3. 读取 `1. 输出产品资产状态与处理队列`，确认待确认、待归并、待废弃、待补引用的资产行。
4. 读取 `2. 输出产品资产索引 / 分片索引`，按业务域、产品类型、构件类型或资产 ID 定位候选分片。
5. 仅读取目标分片、目标资产行、相关 `24/25/26/27` 资产和需要写回的反馈区。

扩大读取范围仅在以下情况允许：

- 业务域、产品类型或构件类型边界冲突。
- 目标资产 ID 在索引中无法定位。
- 需要执行全局归并、状态同步、资产引用一致性检查。
- 人类明确要求通读输出产品架构台账。

禁止事项：

- 不得在常规 Skill 运行中默认读取完整资产大表。
- 不得将 `疑似` 或 `待明确` 条目作为下游强依赖。
- 不得在本文档复制 `24/25/26/27` 的资产正文，只记录 ID 引用和关系。

## 3. 规则区 / 口径区

### 3.1 输出产品类型与构件类型识别规则

### 3.2 规则定位

本文件不是脱离需求预先枚举的业务全集，而是在 EOS 系统设计过程中持续沉淀的**输出产品架构资产**。输出产品类型、构件类型及其产品数据清单，应主要来自对 `02-eos-stakeholder-requirements-architecture.md` 中相关方需求架构末级节点的分析，并在多轮设计中持续复用、修订和确认。

因此，23 号资产同时承担两类职责：

1. **资产台账**：记录已识别、待确认或废弃的业务域、产品类型、构件类型及其文档 / 数据载体关系
2. **识别规则库**：记录分析 STR 架构末级节点时用于判断业务域、产品类型、构件类型和产品数据的规则与经验

### 3.3 基本生成路径

对每个被选入设计的 STR 架构末级节点，应按以下路径识别输出产品架构资产：

1. **理解角色场景**
   明确该 STR 末级节点描述的角色、业务目标、触发条件、期望结果和边界。

2. **补齐文档序列**
   基于该角色场景，识别完成该场景所需的输入文档、过程文档、输出文档和状态记录，形成文档 / 信息载体序列。

3. **逐文档识别归属**
   对文档序列中的每个文档或信息载体，识别其所属业务域、产品类型、构件类型和产品数据类型。

4. **匹配既有 23 资产**
   优先复用已确认条目；若语义接近但边界不足，标记为扩展或改进；若无合适承接对象，生成候选新条目。

5. **人类反馈与确认**
   AI 只能提出识别建议和候选归属。业务边界、产品类型命名、构件类型归并和废弃判断必须经过人类反馈或确认，可多轮迭代。

6. **沉淀为资产**
   人类确认后，将产品类型、构件类型、产品数据 / 文档载体关系写入本文件，并维护 STR 引用、状态、AI 建议和人类决策。

### 3.4 识别对象边界

| 对象 | 定义口径 | 识别依据 | 写入位置 |
|------|---------|---------|---------|
| 业务域 | 企业业务职责的一级归属范围 | 三维业务结构、组织职责、场景目标 | 业务层级=`业务域` |
| 产品类型 | EOS 运行输出的一类产品或服务 | STR 末级节点所表达的业务结果、角色场景产出 | 业务层级=`E2E项目` |
| 构件类型 | 构成某类产品类型的稳定组成类型 | 文档序列中的核心业务对象、配置定义或引擎能力 | 业务层级=`构件类型` |
| 产品数据 / 信息载体 | 构件实现、运行或交付所需的数据和文档 | 输入文档、输出文档、过程记录、配置载体 | 输入文档 / 输出文档列，并引用 26 号资产 |

### 3.5 三维与子域规则

EOS 业务域识别应优先参照以下业务结构：

| 维度 | 主要目的 | 典型业务域 |
|------|---------|-----------|
| 管理维 | 满足监管相关方要求 | 规划与经营、质量管理、合规管理、企业党建、部门运行 |
| 价值维 | 满足产品相关方需求 | 市场销售、产品研发、生产制造、售后服务 |
| 支持维 | 满足资源申请方需要 | 采供管理、人力资源管理、财经业务管理、能力建设管理、流程与IT、行政服务 |

子域判断规则：

1. 每个业务域可包含管理子域和运行子域
2. 管理维业务域以管理子域为主
3. 价值维、支持维业务域通常同时具有管理子域和运行子域
4. 规划与经营业务域代表企业全部业务，其他业务域默认只负责本业务域
5. 各业务域的管理子域通常复用通用管理类产品类型业务，如五年规划开发、业务变革管理、年度运营管理、年度业务改进与运行
6. 各业务域的运行子域产品类型业务按业务域特性分别识别，不强制统一模板

### 3.7 资产确认原则

1. **需求驱动**：新增或修订产品类型、构件类型时，应能追溯到一个或多个 STR 架构末级节点、BA IPO 或人类确认的业务架构规则
2. **资产优先**：识别时优先复用已确认资产，再考虑扩展、归并或新增
3. **人类裁定**：全企业级 / 本域级边界、产品类型命名、构件归并、废弃判断由人类最终确认
4. **候选隔离**：AI 推断但未确认的条目应标记为疑似、待确认或待裁定，不得直接作为已确认资产供下游强依赖
5. **规则沉淀**：反复出现的识别判断、边界口径和归并原则，应沉淀到本规则区或相关权威规范中
