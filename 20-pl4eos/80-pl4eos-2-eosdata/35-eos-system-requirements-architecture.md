## 功能部分架构（SysReq-F）

> 将业务架构 IPO 去重后，按 EOS 系统功能分类组织。功能部分采用 0-9 级层级结构，5 级为承接层。

### 架构树形结构

本批次覆盖**不合格品审理**业务域。SysReq-F 节点采用 `SysReq-F-NCR-{seq}` 编号，NCR 为不合格品审理（Non-Conformance Report）缩写。

```
EOS 企业运营体系 — 系统需求（功能部分）
└── 质量管理
    └── 不合格品审理
        ├── SysReq-F-NCR-001: 不合格品审理申请管理
        ├── SysReq-F-NCR-002: 不合格品多级审理流转管理
        ├── SysReq-F-NCR-003: 不合格品分级评审管理
        ├── SysReq-F-NCR-004: 不合格品审理审批管理
        ├── SysReq-F-NCR-005: 不合格品复检与放行管理
        └── SysReq-F-NCR-006: 不合格品审理档案管理
```

### SysReq-F 架构末级节点定义

#### SysReq-F-NCR-002：不合格品多级审理流转管理
<a id="sysreq-f-ncr-002"></a>

| 要素 | 内容 |
|------|------|
| **节点定义** | 系统应提供按固定路由顺序（审理小组→使用方→主管副总师→质量安全部）的多级审理流转功能。审理申请提交后，系统自动为各节点创建审理待办任务并顺序推送。各节点审理人员查看申请信息，录入审理意见，从预定义处置方案（返工/返修/报废/让步使用）中选择。若任一节点选择"报废"，则流程终止直接进入归档；支持上级节点驳回回退至指定下级节点重新审理，原审理记录保留作为补充。 |
| **承接的 BA IPO** | BA-NCR-002（多级审理）：审理任务自动推送 + 审理意见录入 + 报废终止流程 + 审理驳回回退 |
| **承接的 SR 详细定义末级节点** | <!-- SR-F-XXX.X，待 03-*.md 详设完成后补充 --> |
| **符合性分析** | 覆盖 BA-NCR-002 全部 L4 场景业务 IPO，包含固定路由四节点顺序流转、处置方案选择、报废分支终止、驳回回退机制。支持异常路径：节点超时催办、审理任务重分配、驳回后原审理人不可用时指定他人代为审理。 |
| **为下层提供的设计指导** | ① 审理路由为固定顺序流转，路由配置需支持未来调整；② 审理权限需与组织角色绑定（审理小组/使用方/副总师/质量安全部为四类角色）；③ 报废分支为硬终止逻辑，需在流程设计中固化；④ 驳回回退需保留原审理意见链，新意见作为追加而非覆盖。组部件归属：不合格品审理（@node-ncr） |

#### SysReq-F-NCR-003：不合格品分级评审管理
<a id="sysreq-f-ncr-003"></a>

| 要素 | 内容 |
|------|------|
| **节点定义** | 系统应提供按 A/B/C 三级自动分配评审任务的功能。系统根据不合格严重程度自动判定级别（A类→MRB评审/8h时限，B类→质量主管评审/24h时限，C类→授权检验员即时判定），按分级结果自动分配评审任务并启动时效管控倒计时。评审人员查看案情后从预定义处置方案（返工/返修/让步/降级/报废/退货）中选择，处置方案经审批后生效。MRB 评审全过程记录保存。 |
| **承接的 BA IPO** | BA-NCR-006（分级评审）：不合格品自动分级 + 评审任务分配与时效管控 + 处置方案评审与审批 |
| **承接的 SR 详细定义末级节点** | <!-- SR-F-XXX.X，待 03-*.md 详设完成后补充 --> |
| **符合性分析** | 覆盖 BA-NCR-006 全部 L4 场景业务 IPO。分级标准（A/B/C 判定规则）需支持可配置，时效管控需支持倒计时与超时催办。处置方案的审批权限按分级级别不同而不同。 |
| **为下层提供的设计指导** | ① 分级标准需通过规则引擎实现可配置（支持判定规则的条件表达式）；② MRB 成员列表需与组织管理联动；③ 时效管控需支持配置（级别→时限映射表）；④ 处置方案选项池需支持按级别配置可用方案子集。组部件归属：不合格品审理（@node-ncr） |

#### SysReq-F-NCR-004：不合格品审理审批管理
<a id="sysreq-f-ncr-004"></a>

| 要素 | 内容 |
|------|------|
| **节点定义** | 系统应提供审理结论的审批裁决功能。审理流程完成后自动推送至不合格品审理委员会主任，主任审阅全流程审理记录后做出批准或驳回决策。若驳回需指定回退节点（回退至指定审理节点重新审理）。若批准且涉及顾客利益，继续推送至顾客代表审批。两级审批均通过后进入复检放行流程。 |
| **承接的 BA IPO** | BA-NCR-003（审批管理）：审批任务推送 + 审批裁决 + 驳回回退 + 顾客代表审批 |
| **承接的 SR 详细定义末级节点** | <!-- SR-F-XXX.X，待 03-*.md 详设完成后补充 --> |
| **符合性分析** | 覆盖 BA-NCR-003 全部 L4 场景业务 IPO。两级审批流程：委员会主任→（如适用）顾客代表。支持批准/驳回双分支，驳回可指定回退节点。顾客代表审批为条件分支（仅当合同要求或涉及顾客利益时执行）。 |

#### SysReq-F-NCR-005：不合格品复检与放行管理
<a id="sysreq-f-ncr-005"></a>

| 要素 | 内容 |
|------|------|
| **节点定义** | 系统应提供复检确认与放行管理功能。两级审批通过后（且审理结论非"报废"），系统自动触发复检确认任务。复检人员录入实际处置结果，系统将复检结果与审批结论中的处置方案做一致性校验（如审批"返工"则确认返工完成并合格）。校验通过后系统自动生成放行凭证（含完整审理链条摘要），检验员确认放行，不合格品处置闭环完成。审理结论为"报废"时跳过本步骤直接进入归档。 |
| **承接的 BA IPO** | BA-NCR-004（复检放行）：复检任务触发 + 复检结果录入与一致性校验 + 放行凭证生成 + 检验员放行确认 |
| **承接的 SR 详细定义末级节点** | <!-- SR-F-XXX.X，待 03-*.md 详设完成后补充 --> |
| **符合性分析** | 覆盖 BA-NCR-004 全部 L4 场景业务 IPO。前置条件"审理结论非报废"需在流程设计中固化。一致性校验需根据处置方案类型定义校验规则（返工→确认返工合格，返修→确认返修合格，让步使用→确认让步条件满足等）。 |
| **为下层提供的设计指导** | ① "报废"跳过复检的逻辑需在流程引擎中固化为条件分支；② 一致性校验规则因处置方案类型而异，需通过规则引擎实现可配置；③ 放行凭证（@doc-ncr-voucher）应自动汇总完整审理链信息，格式为标准化模板。组部件归属：不合格品审理（@node-ncr） |
| **关联 Skill** | <!-- 待后续补充 --> |

#### SysReq-F-NCR-006：不合格品审理档案管理
<a id="sysreq-f-ncr-006"></a>

| 要素 | 内容 |
|------|------|
| **节点定义** | 系统应提供审理档案的归档与查询追溯功能。不合格品审理流程结束后（已放行或已报废），系统自动汇集全流程记录（审理申请、各节点审理意见、评审记录、审批结论、复检结果、放行凭证等）。交付人员确认档案完整性后归档为只读状态。归档后的审理档案支持按产品名称/编号/批次/日期/不合格类型等条件组合查询和追溯。 |
| **承接的 BA IPO** | BA-NCR-005（档案归档）：审理档案自动汇集 + 档案完整性确认 + 档案归档（只读） + 档案查询追溯 |
| **承接的 SR 详细定义末级节点** | <!-- SR-F-XXX.X，待 03-*.md 详设完成后补充 --> |
| **符合性分析** | 覆盖 BA-NCR-005 全部 L4 场景业务 IPO。档案自动汇集覆盖审理全链路数据，归档后为只读不可修改。查询支持多维度组合条件检索，满足追溯审计需求。 |
| **为下层提供的设计指导** | ① 档案数据结构应涵盖审理全链路信息模型，各节点数据自动归集；② 只读归档需在数据访问层强制实施，归档后不允许任何修改；③ 查询索引应覆盖产品/批次/日期/类型等主要维度，支持组合条件检索性能需求；④ 档案文档资产：不合格品审理档案（@doc-ncr-archive）。组部件归属：不合格品审理（@node-ncr） |
| **关联 Skill** | <!-- 待后续补充 --> |
