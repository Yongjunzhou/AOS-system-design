# 原始需求澄清

**Skill ID**: eos-ort02-clarify · **Step ort02-clarify: Requirement Clarification**

> **概述**：对已切分的原始需求材料执行澄清迭代。每次运行时检测正文区状态——无模糊标注时执行首轮全量标注；有 `[✓已澄清]` 注释时按注释生成新正文并归档旧版，对新正文执行三项条件评估；有未解决标注但无新注释时输出迭代报告等待人类操作。本步是人类-in-the-loop 步骤，人类通过在正文区追加 `[✓已澄清]` 注释驱动流程。
>
> **输入 → 输出**：切分文件（`-sub-{seq}.md`，status=raw）+ 状态文档 → 已澄清切分文件（同文件，正文区为最新版本，status=clarified）+ 状态文档状态更新（生命周期→已澄清 / 保持澄清中 / 已废弃）
>
> **接口设计**：本步靠正文区状态驱动——无标注时走首轮标注路径；有 `[✓已澄清]` 时走迭代路径；有 `[⨯]` 但无 `[✓已澄清]` 时等待。上游 eos-ort01-chunk 产出切分文件（status=raw）。下游 eos-ort03-norm 读取"生命周期=已澄清"的记录。
>
> **版本**：v0.2 | **修订**：2026-06-08

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：`../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 目录中的切分文件（`-sub-{seq}.md`）+ `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/01-eos-sysdev-status.md`
- **产出**：已澄清的切分文件（同文件，正文区为最新纯净版本，status=clarified）+ 状态文档（生命周期→已澄清 / 保持澄清中 / 已废弃）

### 与四条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **需求相对** | 本步不涉及需求/方案角色切换，需求澄清仍属原始需求预处理阶段，无需求相对性可言，不适用 |
| **详概分离** | 本步仅做需求澄清，不涉及任何分解或设计决策 |
| **文档结对** | 本步不产生文档结对关系 |
| **基于资产** | 所有迭代历史和旧版正文保留在历史区，可追溯每轮处理 |

### 三区结构说明

本步操作的文件（切分文件）的三区：

| 区域 | 内容 | 本步操作 |
|------|------|---------|
| **管理区** | 状态摘要、澄清决策记录 | 更新 OR切分文档状态表（首轮标注完成/已澄清/澄清中/已废弃）；追加决策记录 |
| **正文区** | 原文 + 模糊标注 `[⨯/✓]`（临时）+ 模糊标注报告（临时） | 首轮标注：追加 `[⨯]` 标记+报告；迭代：生成新正文（纯原文），旧版归档至历史区；已澄清：正文区为纯原文（无标注） |
| **历史区** | 旧版正文 + 标注标记（迭代归档时写入） | 每次迭代归档旧版正文+全部标注标记 |

## 三、操作流程

核心驱动逻辑：读取 状态文档中生命周期=已登记/澄清中的记录，检测正文区状态——无模糊标注时执行首轮全量标注（五种子类型），写入 `[⨯]` 标记和模糊标注报告；检测到 `[✓已澄清]` 注释时根据注释生成新正文，将旧正文+全部标注标记转入历史区，对新正文执行三项条件评估；无 `[✓已澄清]` 但仍有 `[⨯]` 时输出迭代报告等待人类操作。

本步按"扫描分流 → 正文区检测 → 首轮标注/迭代生成+评估 → 写入 → 行动清单"的线性序列执行。

**批处理说明**：每轮扫描 状态文档中所有生命周期=已登记/澄清中的切分文件，按各自状态独立执行对应分支（首轮标注/迭代评估/等待）。同一原始文档的不同切分文件互不依赖，各自在批次选材中分组呈现。

操作流程概览：

```
[前置校验] 检查上游前置条件
  ├─ `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/01-eos-sysdev-status.md` 不存在 → 输出"请先运行 eos-ort00-textualize" → 结束
  ├─ `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 中无 `-sub-*.md` 切分文件 → 输出"请先运行 eos-ort01-chunk" → 结束
  └─ 全部就绪 → [扫描分流] ↓

[扫描分流] 读取 状态文档 + 切分文件
  ├─ 人类决策=已终止 → 更新生命周期=已废弃，追加废弃说明，跳过
  ├─ 生命周期=已登记（status=raw）→ [首轮标注] ↓
  ├─ 生命周期=澄清中 → [正文区检测] ↓
  ├─ 生命周期=已澄清 → 跳过正文处理，进入 [批次选材] ↓
  └─ 其他 → 跳过

[正文区检测] 扫描正文区标注标记
  │
  ├─ 发现 [✓已澄清] 注释 → [迭代生成+评估] ↓
  │   ① [归档] 旧正文 + 所有标注标记 → 历史区（版本 #N）
  │   ② [生成] 按注释更新正文为新版本（纯原文）
  │   ③ [评估] 三项条件判定 ↓
  │        ├─ 清晰 → 状态→已澄清，status→clarified
  │        └─ 仍有模糊 → 追加新 [⨯] 标记 → 保持澄清中
  │
  ├─ 有 [⨯] 未解决，无 [✓已澄清] → 输出迭代报告，保持澄清中
  │
  └─ 无任何标记（异常）→ 检查 status，如=clarified 跳过

[写入] 更新管理区 + 状态文档
  按 §3.3 顺序：状态文档（权威源）→ frontmatter status → OR切分文档状态表（派生）

[批次选材] 扫描已澄清材料 → 输出摘要供人类确认（§3.4）

[行动清单] 按状态分组输出行动项
    ↓

[人机交互] 询问人类是否打开状态文档
  ├─ 回答"是"→ 输出路径
  └─ 回答"否"→ 跳过

→ 本轮结束
```

### 3.1 首轮标注

AI 通读切分文件正文区，全量标注模糊项。

| # | 操作 | 说明 |
|---|------|------|
| 1 | 通读完整文档 | 从头到尾通读，在完整语境中判断哪些部分需要澄清 |
| 2 | 全量模糊标注 | 按五种模糊子类型标注全部发现。每种标注包含：子类型、具体问题描述、建议澄清方向 |
| 3 | 输出模糊标注报告 | 按模板输出，列出本轮全部新增模糊项、三项条件判定结果 |
| 4 | 更新 frontmatter status | status=raw → status=annotated |

**模糊标注报告写入位置**：附加在正文区末尾（所有原文 + 行内 `[⨯]` 标注之后），与正文内容以分隔线隔开。迭代路径中报告替换为迭代报告，格式见 §5.2。

**五种模糊子类型**：

| 子类型 | 适用场景 | 标注示例 |
|--------|---------|---------|
| **术语模糊** | 非量化表述需明确度量方式 | `[⨯术语模糊: 高效→需量化处理能力指标]` |
| **主体/客体不明** | 角色或对象不可识别 | `[⨯主体不明]提出该需求` |
| **边界模糊** | 条件或指标缺少边界 | `[⨯边界模糊: 实时→需明确具体延迟上限]` |
| **信息缺失** | 缺少元信息 | `[⨯信息缺失→缺来源日期和提出方]` |
| **多重模糊** | 同处存在两种以上类型，按优先级分层 | `[⨯多重模糊: 术语+边界→非量化+缺时限]` |

**标注格式**：

```
[⨯子类型: 原文片段→问题说明]
```

- 状态：`⨯` 未解决 / `✓` 已解决
- 子类型：必填
- 原文片段：推荐保留
- 问题说明：推荐保留，类型自明时可省

**首轮标注后状态摘要**（更新 OR切分文档状态表）：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 首轮标注完成 | 需澄清 | N 项模糊待解决（见行内 [⨯] 标记） | 追加 [✓已澄清: ...] 注释后重启 eos-ort02-clarify |
```

无模糊项时（AI建议=可确认）：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 首轮标注完成 | 可确认 | 无 | 状态文档 → 1.1 OR材料进展状态及处理建议一览表 → 该记录 人类决策 字段设为 已确认 → 运行 eos-ort03-norm |
```

### 3.2 迭代生成+评估（检测到 `[✓已澄清]` 时）

| # | 操作 | 说明 |
|---|------|------|
| 1 | 读取所有 `[✓已澄清]` 注释 | 扫描正文区，收集所有 `[✓已澄清: ...]` 注释内容 |
| 2 | 生成新正文 | 根据注释内容将模糊表述替换为精确值，生成新版本正文（纯原文，无标注） |
| 3 | 归档旧版本 | 旧正文 + 全部标注标记（`[⨯]`/`[✓]`）→ 历史区（版本 #N） |
| 4 | 更新正文区 | 替换为新版本（纯原文） |
| 5 | 三项条件评估 | 通读新正文，判定三项条件 |

**三项条件**（引用通用规范 §7.3 阶段一(d)）：

1. 正文内容清晰完整（所有非量化表述已替换为精确值）
2. 所有历史标注的模糊项已在归档前的正文中有对应的 `[✓已澄清]` 注释
3. 无新增的模糊标注产生

**评估分支**：

| 条件 | 操作 |
|------|------|
| 三项全满足 | 状态→已澄清，frontmatter status→clarified，输出完成报告 |
| 仍有未清晰项 | 在新正文对应位置追加 `[⨯...]` 标注，生命周期保持澄清中，输出迭代报告 |

**归档格式**（历史区，倒序，最新在最上）：

```
## 历史版本

### 版本 #2 · 2026-06-07 · 第2轮澄清
- 当次正文：（全文）
- 当次标注：[⨯...] / [✓...] 列表
- 人类注释：[✓已澄清: ...] 内容

### 版本 #1 · 2026-06-06 · 首轮标注
- 当次正文：（全文）
- 当次标注：[⨯...] 列表
```

### 3.3 写入：更新管理区 + 状态文档

**同步契约**：状态文档是材料级状态追踪的权威源，OR切分文档状态表由状态文档派生。写入时必须**先更新状态文档，再从状态文档推导写入 OR切分文档状态表**。若检测到两者不一致，以状态文档为准重建 OR切分文档状态表。

| # | 操作 | 说明 |
|---|------|------|
| 1 | 更新 状态文档 | 生命周期转迁 + AI 建议 + last_size + iteration_count + unresolved_count |
| 2 | 更新 frontmatter status | 已澄清→status=clarified；已废弃→status=abandoned；其余保持当前值 |
| 3 | 更新 OR切分文档状态表 | 从状态文档推导写入：已澄清/已废弃/保持澄清中 |
| 4 | 追加决策记录 | 文档达到已澄清或已废弃时追加 |

**已澄清状态摘要**（更新 OR切分文档状态表）：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 已澄清 | 可确认 | 无 | 状态文档 → 1.1 OR材料进展状态及处理建议一览表 → 该记录 人类决策 字段设为 已确认 → 运行 eos-ort03-norm |
```

**保持澄清中（需澄清）状态摘要**：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 澄清中 | 需澄清 | N 项模糊未解决（见行内 [⨯] 标记） | 追加 [✓已澄清: ...] 注释后重启 eos-ort02-clarify |
```

**已废弃状态摘要**：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 已废弃 | 建议终止 | 无 | 该文件不再进入下游处理 |
```

```

══════════════════════════════════════════

已澄清材料（可送入规范化）：

| # | 源文件 | 摘要 | 状态 |
|---|-------|------|------|
| 1 | [文件] | [核心内容一句话] | 候选纳入 |

```

## 四、AI 执行指南

### 4.1 正文区检测规则

AI 在入口分流中检测正文区状态：

| 正文区状态 | 判定 |
|-----------|------|
| 无任何 `[⨯]` 或 `[✓]` 标注标记 | 异常处理：检查 frontmatter.status，如=clarified 跳过；否则标记数据不一致，等待人类处理 |
| 存在 `[✓已澄清: ...]` 注释 | 迭代路径 → 归档→生成→评估 |
| 存在 `[⨯]` 标记，无 `[✓已澄清]` 注释 | 输出迭代报告，等待人类 |
| 无任何标注标记且 status=clarified | 跳过（已澄清终态） |

### 4.2 标注规范

**标注纪律**：

1. **标"哪里不清楚"，不标"应该怎么做"**：即使发现明显内部矛盾，也只标注矛盾位置和双方表述
2. **先通读后标注**：通读全文后再逐处标注，避免局部视角导致重复或遗漏
3. **一条只标一个点**：同一处不同维度的模糊合并为一条多重模糊
4. **报告与内联一致**：标注报告中列出的每项对应文档中的一处内联标注

**省略规则**：

| 省略内容 | 条件 | 示例 |
|---------|------|------|
| 原文片段 | 模糊位置自明（如独立成行的信息缺失） | `[⨯信息缺失→缺提出方]` |
| 问题说明 | 子类型+原文片段已足够自明 | `[⨯术语模糊: 智能化]` |
| 不可省略状态和子类型 | — | — |

### 4.3 启动方式

#### 人类启动指令

> 开始原始需求澄清（或 eos-ort02-clarify）

#### AI 动作序列

0. **前置校验**：检查上游前置条件。**逐级报错，首次即停**：
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/01-eos-sysdev-status.md` 不存在 → 输出"请先运行 **eos-ort00-textualize**" → **结束**
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 中无 `-sub-*.md` 切分文件 → 输出"请先运行 **eos-ort01-chunk**" → **结束**
   - 全部就绪 → 进入步骤 1
1. **扫描分流**：读取 状态文档，按生命周期分流
   - **人类决策=已终止** → 废弃处理（更新生命周期=已废弃，追加废弃说明）
   - **生命周期=已登记（status=raw）** → 进入步骤 2（首轮标注）
   - **生命周期=澄清中** → 进入步骤 3（正文区检测）
   - **生命周期=已澄清** → 跳至步骤 6（批次选材）
   - **其他** → 跳过
2. **首轮标注**：通读全文 → 全量模糊标注 → 写入 `[⨯]` 标记 → 输出模糊标注报告 → 更新 status=annotated → 跳至步骤 5
3. **正文区检测**：扫描行内标注标记（仅生命周期=澄清中时执行）
   - **发现 `[✓已澄清]` 注释** → 进入步骤 4（迭代生成+评估）
   - **有 `[⨯]` 未解决，无 `[✓已澄清]`** → 输出迭代报告，跳至步骤 7
   - **无任何标记（异常）** → 检查 frontmatter.status，如=clarified 跳过，否则标记数据不一致并等待人类处理
4. **迭代生成+评估**：读取 `[✓已澄清]` 注释 → 生成新正文 → 归档旧版+标注到历史区 → 更新正文区 → 三项条件评估 → 清晰→已澄清；模糊→追加新 `[⨯]`
5. **写入**：更新 状态文档（生命周期/AI 建议/last_size/iteration_count/unresolved_count，权威源）→ 推导更新 OR切分文档状态表 → 更新 frontmatter status
6. **批次选材**：扫描已澄清材料，输出摘要
7. **输出行动清单**
8. **人机交互**：询问人类是否打开状态文档
