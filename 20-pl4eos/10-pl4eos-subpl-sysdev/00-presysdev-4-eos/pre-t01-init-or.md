# 原始需求初处理
**Skill ID**: pre-t01-init · **Step 00-init: Original Requirement Initial Processing**

> **概述**：将原始需求材料通过文本化（格式转换）和 AI 首轮模糊标注，转化为已标注的原始需求记录文档，供下游 pre-t02-clarify 进行多轮澄清迭代。本步是 AI 独立执行段，一次性运行完成，无需人类介入。
>
> **输入 → 输出**：原始材料文件（PDF/图片/邮件/纪要等）→ 已标注的原始需求记录文档（含首轮模糊标注）+ registry-log 状态更新（生命周期→澄清中）
>
> **接口设计**：登记表（registry-log）是预处理链中各顺序 Skill 间的共享接口。pre-t01-init 负责将文本化+首轮标注后的材料状态写入登记表（生命周期状态=澄清中），下游 pre-t02-clarify 读取此条件进入澄清迭代。
>
> **适用场景**：所有四种设计场景的公共前置步骤的初处理段。本步完成后，人类启动 pre-t02-clarify 进入澄清迭代。
>
> **版本**：v0.1 | **修订**：2026-06-05

## 三、操作流程

AI 每次启动执行一轮完整的单次处理——初始化目录与登记表（如首次运行）、扫描输入目录、文本化、首轮标注、输出报告、更新 registry-log，然后结束。本步为一次性执行，无迭代、无人工介入。

操作流程概览：

```
[初始化] 检查目录与登记表
  ├─ 10-source-materials/ 不存在 → 新建目录 → ⛔ 目录为空，退出（提示人类放入材料）
  ├─ 00-registry-log.md 不存在（目录已存在） → 新建登记表，继续 ↓
  └─ 就绪 → 进入扫描 ↓

[扫描] 扫描源文件目录 + registry-log
  ├─ 无 registry-log 记录的新文件 → 自动登记（生命周期=已登记）
  ├─ 生命周期=已登记，无 *-clarified.md → [阶段一] 文本化 ↓
  ├─ 生命周期=已登记，有 *-clarified.md → 跳过（异常→提示）
  └─ 生命周期≠已登记 → 跳过

[阶段一] 文本化
  将非文本格式原始材料转换为 .md 文本化文档
  一个源文件 → 一个文本化副本
  原始文件保持只读不动
    ↓

[阶段二] 首轮模糊标注
  AI 通读完整文档
  标注本轮发现的所有模糊项（完整的初次标注）
  输出模糊标注报告 + AI 建议 + 操作选项
  更新 registry-log（生命周期→澄清中，AI 建议，last_size）
    ↓

[行动清单] 输出行动清单
  按材料状态分组列出人类需执行的行动项

────────────────────────────────────────────────────
▲ AI 独立执行段结束 —— 以下为人类执行段
────────────────────────────────────────────────────

  → 人类后续操作：
    ├─ 追加澄清记录到文档 → 启动 pre-t02-clarify
    └─ 在文档中保留选项 → 启动 pre-t02-clarify
```

> **执行说明**：本步是 AI 独立执行段。从扫描到输出行动清单一气呵成，不等待人类介入。人类收到清单后根据行动指引决定下一步——追加记录或选择选项后启动 pre-t02-clarify，或废弃材料。

### 阶段一：文本化

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 扫描输入源 | AI | 扫描 `10-source-materials/` 目录中的原始材料，按文件类型分类。非文本格式（PDF/图片/扫描件/录音转写等）需执行操作2（文本化转换）；文本格式（.md/.txt）直接从原文复制到工作副本 |
| 2 | 文本化转换 | AI | 将非文本格式原始材料转换为 .md 文档。转换规则：保留原文结构（章节/段落/列表）、原文表述不做任何修改或摘要、无法识别的内容标注占位符、保留原始文件名作为元数据 |
| 3 | 写入工作副本 | AI | 文本化后的 .md 文档写入与源文件相同的 `10-source-materials/` 目录，文件命名规则：`{原始文件名}-clarified.md`。原始文件保持只读状态 |

> **文本化原则**：AI 在此阶段是"转录者"而非"理解者"。文本化只改变载体格式，不改变内容。任何超出格式转换的推断或补充均不允许——应在阶段二中标注为模糊项。

### 阶段二：首轮模糊标注

AI 通读完整的文本化文档，全量标注模糊项。本步不涉及历史记录检查或多轮迭代。

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 通读完整文档 | AI | 从头到尾通读文本化文档的全部内容，在完整语境中判断哪些部分需要澄清 |
| 2 | 全量标注模糊项 | AI | 在文档中按五种模糊子类型（§四.2）标注全部发现。每种标注均需包含：模糊子类型、具体问题描述、建议的澄清方向 |
| 3 | 输出模糊标注报告 | AI | 按 §五.2 模板输出报告，列出本轮全部新增模糊项、三项条件判定结果 |
| 4 | 追加操作选项 | AI | 根据报告结论在文档末尾追加操作选项（§五.3）：可确认→追加三选项；建议终止→追加两选项；需澄清→不加选项 |
| 5 | 更新 registry-log | AI | 生命周期→澄清中，AI 建议=需澄清/可确认/建议终止，记录 `last_size`，`iteration_count`=1 |

> **操作1~5是线性执行**，不需要分支和条件判断——每一项都是前一项执行后的自然结果。

> **已澄清判定标准**（引用通用规范 §7.3 阶段一(d)）：
> 三项条件同时满足时，AI 在报告中输出"三项条件已满足"，AI 建议=可确认：
> 1. 可读的文本化工作副本（`*-clarified.md`）已就绪
> 2. 所有历史 AI 标注的模糊项均已在文档中有对应的澄清记录
> 3. 无新增的模糊标注产生

> **首轮标注的特点**：条件2在首轮通读时**不适用**（尚无历史标注），因此条件3是首轮判定关键——通读后如果文档无任何模糊项，AI 建议=可确认；否则 AI 建议=需澄清。

### 启动方式

本步为一次性执行，无需断点续接。每次启动处理所有"生命周期=已登记"的材料后正常结束。

## 四、AI 执行指南

### 4.1 文本化注意事项

以下适用于非文本格式的文本化转换。文本格式（.md/.txt）直接从原文复制到工作副本，不执行转换操作。

| 场景 | 处理方式 |
|------|---------|
| PDF 中的表格 | 转为 Markdown 表格，保留行列结构和数值 |
| 图片/扫描件 | 识别文字内容，标注"[AI 识别，待人工确认]" |
| 录音转写 | 标注发言人和时间戳，无法确认的内容标注"[听写不清]" |
| 手写内容 | 可辨识的写入文档，标注"[AI 推测]"，不可辨识的标注"[原文模糊]" |
| 多种语言混写 | 保留原文种，必要时标注"[中文/英文/...]" |
| 完全无法识别的图片 | 写入占位符 `[图片：无法识别文字内容]` |
| 手绘图+少量文字 | 文字照录，图描述为 `[手绘图：XXX主题，AI 推测]` |
| 多人对话录音转写 | 标注发言人，重叠说话处标注"[多人讲话，内容不定]" |
| 多语混写且无法全部识别 | 可识别语种照录，无法识别处标注"[原文: 非中文/英文，待翻译]" |
| 文档明显不完整 | 在文档开头标注 `[文档疑似不完整：缺页/截断]` |
| 原文存在明显内部矛盾 | 标注为"边界模糊"，指出矛盾位置和双方表述，不尝试调和 |

> **文本化自检**：AI 完成文本化后须确认：转换忠实于原文（无摘要、推断或补充）、无法识别的内容已标注占位符、文件名符合命名规则。不满足时须修正后重检，禁止带瑕疵进入阶段二。

### 4.2 模糊标注规范

本规范定义 `*-clarified.md` 中模糊标注的完整规则。标注分两层——文档正文中的**内联标注**（标记模糊位置）和 AI 输出的**标注报告**（汇总清单），两者配合使用。

#### 4.2.1 格式定义

内联标注的完整语法：

```
[状态 子类型[: 原文片段][ → 问题说明]]
```

| 字段 | 必填 | 取值 | 说明 |
|------|------|------|------|
| `状态` | 是 | `⨯` 未解决 / `✓` 已解决 | AI 在迭代中自动更新。首轮标注全部为 `⨯` |
| `子类型` | 是 | `术语模糊` / `主体不明` / `客体不明` / `边界模糊` / `信息缺失` / `多重模糊` | 五类之一。主体/客体按实际选一个 |
| `原文片段` | 推荐 | 被判定模糊的原文词或短语 | 便于行内定位。模糊位置自明时可省略（如独立成行的信息缺失标注） |
| `→ 问题说明` | 推荐 | 简述为什么模糊、需要澄清什么方向 | 类型自明时可省略（如 `[⨯术语模糊]` 单独出现无法理解，需补充 "`→需量化指标`"） |

**放置规则**：

| 场景 | 位置 | 示例 |
|------|------|------|
| 单处模糊（词/短语） | 紧跟在模糊词/短语**之后** | `实现[⨯术语模糊: 高效→需量化指标]处理` |
| 整句或段落实质模糊 | 句首**或**段首 | `[⨯边界模糊: 实时→缺少数值+单位+条件]系统需实时响应` |
| 信息缺失（无具体位置） | 缺失信息所在行单独成行 | `[⨯信息缺失→缺提出方和日期]` |
| 多处模糊在同一句 | 各自在对应位置嵌入 | `[⨯术语模糊: 高效]...[⨯边界模糊: 实时]` |

> **标注粒度**：一条标注覆盖一个不可再分的模糊点。同一段落中不同位置的模糊各自独立标注，不合并为一条。

#### 4.2.2 各子类型应用示例

**术语模糊** —— 非量化表述需明确度量方式：

```
原文：系统需具备[⨯术语模糊: 高效→需量化处理能力指标]的数据处理能力
     优化后应达到[⨯术语模糊: 显著提升→需基线对比值]的效果
```

**主体/客体不明** —— 角色或对象不可识别：

```
原文：[⨯主体不明]提出该需求
     [⨯客体不明]的数据应作为唯一来源
     系统向[⨯客体不明: 相关部门→缺具体部门名称]推送报告
```

**边界模糊** —— 条件或指标缺少边界：

```
原文：[⨯边界模糊: 实时→需明确具体延迟上限]系统需实时响应
     系统可用性应达到[⨯边界模糊: 高可用→缺少数值+单位+条件]
```

**信息缺失** —— 缺少元信息：

```
原文：[⨯信息缺失→缺来源日期和提出方]
     根据统计数据显示[⨯信息缺失→缺统计口径和周期]
```

**多重模糊** —— 同一处存在两种以上类型，取最高优先级为主标记：

```
原文：实现[⨯多重模糊: 术语+边界→"严格"非量化+缺少具体时限]的权限控制
     按 术语→主体/客体→边界→信息 优先级，"严格"为术语模糊→主标记为术语模糊
```

> **多重模糊的主标记选取**：按 术语模糊→主体/客体不明→边界模糊→信息缺失 的优先级，取其中最高优先级类型作为内联标注的 `子类型`，其余层次在 `→ 问题说明` 中依次注明。

#### 4.2.3 使用指导

**标注纪律**（必须遵守）：

1. **标"哪里不清楚"，不标"应该怎么做"**：即使发现明显内部矛盾，也只标注矛盾位置和双方表述，不做方案性修复或偏向性判断
2. **先通读后标注**：通读全文后再逐处标注，避免局部视角导致重复或遗漏
3. **一条只标一个点**：同一处不同维度的模糊合并为一条多重模糊，不用多条并列标注
4. **报告与内联一致**：标注报告中列出的每项必须对应文档中的一处内联标注，数量一致

**省略规则**——什么情况下可以省略字段：

| 省略内容 | 条件 | 示例 |
|---------|------|------|
| 省略 `原文片段` | 模糊位置自明（如独立成行的信息缺失） | `[⨯信息缺失→缺提出方]` |
| 省略 `→ 问题说明` | 子类型 + 原文片段已足够自明 | `[⨯术语模糊: 智能化]`（知是术语模糊，但"智能化"需量化） |
| 不可省略 `状态` 和 `子类型` | — | 这两项是状态管理和分类的基础 |

**省略建议**：
- `→ 问题说明` 推荐保留，除非确实类型+片段就足够理解
- `原文片段` 只在独立成行时省略

**什么不该标注**：

- ❌ 文本化阶段已处理的占位符（`[原文模糊]`、`[AI 推测]`）——这些是格式转换标记，不是模糊标注
- ❌ 方案性判断——"这个功能应该用消息队列实现"不应作为模糊标注出现
- ❌ 主观偏好——"我觉得用词不当"不是模糊，原文表述清晰仅风格不合偏好时不标注

#### 4.2.4 状态更新规则

AI 在 pre-t02-clarify 的每次迭代中自动管理内联标注的状态：

| 转迁 | 触发条件 | AI 操作 | 示例 |
|------|---------|--------|------|
| `⨯`（保持） | 本轮通读后该模糊项仍没有对应澄清记录 | 标注保持 `⨯`，迭代报告中列为"未解决" | `[⨯术语模糊: ...]` |
| `⨯ → ✓` | 文档中存在对应澄清记录，且记录状态=已澄清 | 将 `⨯` 替换为 `✓`，报告中列为"已解决" | `[⨯...]` → `[✓...]` |
| `⨯ → 删除` | 对应澄清记录状态=已废弃（该模糊项被放弃） | 删除整条内联标注，报告中列為"已废弃" | — |
| 新增 `⨯` | 本轮通读发现新的模糊项 | 在对应位置插入新标注，报告中列为"新增" | `[⨯新增的模糊标注]` |

> **状态更新在通读时统一执行**：AI 完成通读后，根据文档中所有澄清记录的状态，一次性更新所有内联标注的 `⨯/✓`，不做逐条实时修改。

**已解决示例**：
```
原文：系统需具备[✓术语模糊: 高效→≤200ms]处理能力  ← 人类已追加澄清记录
```

### 4.3 经济原则

1. **不动原文语义**：文本化只转换格式，不做摘要、推断或补充
2. **先完整阅读后标注**：通读全文后再标注，避免局部视角导致重复或遗漏
3. **一标明确定**：首轮标注力求完整覆盖所有模糊项，减少后续迭代轮次
