# 原始需求材料纯文本化

**Skill ID**: eos-ort00-textualize · **Step ort00-textualize: Source Material Textualization**

> **概述**：对原始需求材料执行纯格式转换（非文本→Markdown），输出文本化文档到 `10-raw-files/` 目录。本步不涉及语义分析或切分建议——那是 eos-ort01-chunk 的职责。同一文件可重新执行（覆盖旧版）。文本化后自动执行查重检测——正文区内容与已有文本化文档完全重复的材料标记为"已废弃（重复）"，输出软链接文档替代全量副本。
>
> **输入 → 输出**：原始材料文件（PDF/图片/邮件/纪要等）→ 文本化 `.md` 文档（管理区：状态摘要；正文区：纯原文）+ EOS系统设计状态文档 → 表A — OR原料处理建议及进展更新

> **接口设计**：本步与后续步骤通过正文区状态衔接——eos-ort01-chunk 检测正文区原文的存在性决定执行路径。本步是预处理链的入口，每轮扫描所有未文本化的原始文件执行批量格式转换，人类可随时放入新文件后重启本步。

> **版本**：v0.3 | **修订**：2026-06-09

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：`../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 目录中的原始材料文件（PDF/图片/邮件/纪要/已有的 `.md` 等）
- **产出**：文本化 `.md` 文档（管理区：状态摘要；正文区：纯原文；历史区：空）+ EOS系统设计状态文档 → 表A — OR原料处理建议及进展更新

### 与四条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **需求相对** | 本步不涉及需求/方案角色切换，纯格式转换步骤无需求相对性可言，不适用 |
| **详概分离** | 本步仅做格式转换（非文本→Markdown），不涉及语义理解或切分分析 |
| **文档结对** | 本步不产生文档结对关系。本步产出是 eos-ort01-chunk 的输入 |
| **基于资产** | 原始出处保留——文本化文档是原始文件的处理副本，不覆盖原始文件 |

### 三区结构说明

文本化文档采用三区结构：

| 区域 | 位置 | 内容 | 写入者 |
|------|------|------|--------|
| **管理区** | 文档顶部（正文前） | 状态摘要（阶段/AI建议/待处理/操作指引）、迭代记录 | eos-ort00-textualize 写入状态摘要；下游追加迭代记录 |
| **正文区** | 管理区之后 | 纯原文内容（仅文本化后原文，无标注、无方案块） | eos-ort00-textualize |
| **历史区** | 正文区之后 | 空（本步无历史归档） | — |

**正文区纯净原则**：正文区仅包含原文。所有下游分析产物（切分方案块、模糊标注、分析建议块）均由对应步骤写入并适时清理。本步不产生任何分析产物。

**历史区累积原则**：历史区跨步骤累积，各下游步骤（eos-ort01-chunk / eos-ort02-clarify / eos-ort03-norm）在执行过程中将归档内容（旧版方案块、旧版正文+标注、旧版分析建议块等）追加写入同一历史区，彼此不清理。可通过版本标记和时间戳区分各步骤的归档条目。

## 三、操作流程

核心驱动逻辑：扫描原始材料目录，对尚未文本化的文件执行格式转换（非文本格式 → `.md`），输出文本化文档并更新 状态文档 → 表A — OR原料处理建议及进展。

本步操作模式为"检测即执行"的三步简化变体（见 EOS系统设计规范 §15.8），AI 每轮按 Step 1 前置校验 → Step 2 文本化处理 → Step 3 写入输出 循环执行。本步为一次性执行，AI 独立完成所有处理段；末端的[人机交互]归入 Step 3 作为可选对话交接。

操作流程概览：

```
[Step 1·前置校验] 检查前置条件（首次运行自动初始化）
  ├─ ../../80-pl4eos-2-eosdata/ 不存在 → 新建目录（含 00-origin-requirement-materials/ 等子目录）
  ├─ 10-raw-files/ 不存在 → 新建目录
  ├─ 01-eos-sysdev-status.md 不存在 → 新建（含表A/表B/表C + 同步契约 + 字段说明）
  ├─ 10-raw-files/ 为空 → 输出"目录已就绪，请将原始材料放入后重新启动本步" → 结束
  └─ 全部就绪 → [Step 2·文本化处理] ↓

[Step 2·文本化处理] 扫描源文件 → 格式转换 → 查重 → 记录登记
  扫描 10-raw-files/ + 对照 表A：
  ├─ 对每个原始文件：
  │   ├─ 表A中无此文件记录 → 自动登记 → [记录] 生命周期=已登记 | AI建议=可以切分
  │   ├─ 已有文本化 .md → 跳过
  │   ├─ 生命周期=已废弃（重复）→ 跳过（如需重新启用，清除生命周期值后重启本步）
  │   └─ 无文本化 .md →
  │       ① 执行格式转换（非文本→.md / 文本格式直接复制）
  │       ② 查重——将新文本化的正文区与已有文本化文档的正文区逐一比对：
  │         ├─ 发现重复 → [记录] 更新 表A（内容摘要="与[原始源文件名]内容完全重复"
  │         │               + 生命周期=已废弃（重复）+ AI建议=已废弃（重复））
  │         │               → 输出软链接文档
  │         └─ 无重复   → [记录] 更新 表A（内容摘要 + 生命周期=已文本化 + AI建议=可以切分）
  │
  ├─ 全部文件已处理 → [Step 3·写入输出] ↓
  └─ 待处理清单为空 → 输出"所有材料已文本化，无需本步操作"→ 结束

[Step 3·写入输出]
  ├─ [文档] 无重复→输出文本化文档（管理区+正文区原文）；发现重复→输出软链接文档（管理区+正文区定向引用）
  ├─ [行动清单] 输出行动清单：
  │   ├─ 待选入切分的文本化文档（表A → 生命周期=已文本化 + AI建议=可以切分 + 人类决策=空）
  │   ├─ 重复文档通知（生命周期=已废弃（重复）→ 确认后删除源文件）
  │   └─ 下一步操作指引（选入后运行 eos-ort01-chunk）
  │
  └─ [人机交互] 询问人类是否打开 EOS系统设计状态文档
      ├─ 回答"是"→ 输出路径 + 关注行（待选入切分/重复文档/下一步指引）
      └─ 回答"否"→ 跳过

→ 本轮结束
```

> 图中"OR材料进展表"为"EOS系统设计状态文档 → 表A — OR原料处理建议及进展"的流程内简称。

### 文本化执行

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 读取原始文件 | AI | 读取已在扫描分流阶段确定的原始文件。非文本格式（PDF/图片/扫描件/录音转写等）执行操作 2（文本化转换）；文本格式（.md/.txt）直接复制到工作副本 |
| 2 | 文本化转换 | AI | 将非文本格式原始材料转换为 .md 文档。转换规则：保留原文结构（章节/段落/列表）、原文表述不做任何修改或摘要、无法识别的内容标注占位符、保留原始文件名作为元数据 |
| 3 | 查重检测 | AI | 将新文本化后的正文区全文与所有已有文本化文档（生命周期≠已废弃（重复））的正文区逐一比对。匹配标准：全文逐字精确匹配（忽略行尾空白和连续空行差异）。比较范围不含管理区状态表 |
| 4 | 输出处理结果 | AI | 无重复 → 输出文本化 `.md` 文档（三区结构）；发现重复 → 输出软链接 `.md` 文档（管理区 + 正文区定向引用）。文件命名规则：`{原始文件名}.md`（原始文件保持只读不动）。当源文件已是 .md 格式时，输出 `{原始文件名}.textualized.md` |

> **文本化原则**：AI 在此阶段是"转录者"而非"理解者"。文本化只改变载体格式，不改变内容。任何超出格式转换的推断或补充均不允许。

## 四、AI 执行指南

### 4.1 文本化注意事项

同通用系统设计规范 §7.3 阶段一(a)。以下为常见场景：

| 场景 | 处理方式 |
|------|---------|
| PDF 中的表格 | 转为 Markdown 表格，保留行列结构和数值 |
| 图片/扫描件 | 识别文字内容，标注"[AI 识别，待人工确认]" |
| 录音转写 | 标注发言人和时间戳，无法确认的内容标注"[听写不清]" |
| 手写内容 | 可辨识的写入文档，标注"[AI 推测]"，不可辨识的标注"[原文模糊]" |
| 多种语言混写 | 保留原文种，必要时标注"[中文/英文/...]" |
| 完全无法识别的图片 | 写入占位符 `[图片：无法识别文字内容]` |
| 文档明显不完整 | 在文档开头标注 `[文档疑似不完整：缺页/截断]` |

### 4.2 跳过检测逻辑

AI 扫描 `10-raw-files/` 目录后，按以下规则判断每份原始文件是否需处理：

| 条件 | 处理 |
|------|------|
| 文件名为 `.md` 格式且 状态文档 → 表A — OR原料处理建议及进展中已有对应记录（生命周期=已登记/已文本化/已切分/澄清中/已澄清/已规范化） | 跳过（已在处理流程中。如需重文本化，先清除状态文档 → 表A — OR原料处理建议及进展中该记录的生命周期值 → 删除旧 `.textualized.md` 文件 → 重启本步） |
| 文件名为 `.md` 格式且 状态文档 → 表A — OR原料处理建议及进展中已有对应记录（生命周期=已废弃（重复）） | 跳过（标记为重复的文档已通过软链接文档记录，如需重新启用，清除生命周期值后重启本步） |
| 文件名为 `.md` 格式且 状态文档 → 表A — OR原料处理建议及进展中已有对应记录（生命周期=已废弃） | 跳过（如需重新启用，请先清除 状态文档 → 表A — OR原料处理建议及进展中的生命周期值后重启本步） |
| 文件名为 `.md` 格式且 状态文档 → 表A — OR原料处理建议及进展中无对应记录 | 执行文本化（源文件为 `.md` 格式，输出 `{文件名}.textualized.md` 避免同名覆盖） |
| 文件名为非 `.md` 格式且 状态文档 → 表A — OR原料处理建议及进展中无对应文本化记录 | 执行文本化 |
| 文件名为非 `.md` 格式且 状态文档 → 表A — OR原料处理建议及进展中已有对应文本化记录 | 跳过（如需重新文本化，先清除状态文档 → 表A — OR原料处理建议及进展中该记录的生命周期值，再删除旧 `.md` 文件，然后重启本步。如有原始文件变更，AI 可在行动清单中提醒人类确认） |
| 所有文件均跳过 | 输出"所有材料已文本化，无需本步操作"→ 结束 |

### 4.3 启动方式

本步为一次性执行，无需断点续接。每次启动处理所有未文本化的原始材料后正常结束。

#### 人类启动指令

> 开始原始需求材料纯文本化（或 eos-ort00-textualize）

#### AI 动作序列

AI 按 EOS系统设计规范 §15.8（检测即执行四步模式）执行，各步骤操作如下：

1. **前置校验**：检查 `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/` 目录结构。**逐级报错，首次即停**——只报告最早的未满足条件：
   - `../../80-pl4eos-2-eosdata/`（含 `00-origin-requirement-materials/` 子目录）不存在 → 新建目录结构
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 不存在 → 新建目录
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/01-eos-sysdev-status.md` 不存在 → 新建（含 OR 阶段 表A/表B/表C + 同步契约 + 字段说明）
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 为空 → 输出"目录已就绪，请将原始材料放入 `10-raw-files/` 后重新启动本步" → **结束**
   - 全部就绪 → 进入步骤 2
2. **自动发现**：扫描 `10-raw-files/` 中的文件 + 状态文档 → 表A — OR原料处理建议及进展登记记录，按 §4.2 规则判断需处理的文件
3. **格式转换**：对需要处理的文件执行格式转换
4. **查重检测**：将新文本化的正文区与已有文本化文档（生命周期≠已废弃（重复））的正文区逐一比对：
   - 发现重复 → 表A 标记为已废弃（重复）+ 输出软链接文档
   - 无重复 → 继续正常输出
5. **写入输出**：
   - 无重复 → 输出文本化 `.md` 文档（管理区状态摘要 + 正文区原文）+ 更新 表A（内容摘要 + 生命周期=已文本化 + AI建议=可以切分）
   - 发现重复 → 输出软链接文档（管理区+正文区定向引用）+ 更新 表A（内容摘要="与[源文件]重复"+ 生命周期=已废弃（重复）+ AI建议=已废弃（重复））
   → 输出行动清单（含重复通知）→ 询问人类是否打开状态文档

### 4.4 重进入说明

本步可在以下场景重进入：

- **新增原始材料**：新放入的非文本格式文件在 状态文档 → 表A — OR原料处理建议及进展中无对应文本化记录，进入文本化流程
- **重新文本化**：先清除状态文档 → 表A — OR原料处理建议及进展中该记录的生命周期值，再删除旧文本化 `.md` 文件，然后重启本步，AI 重新生成
- **重复文档误判处理**：如某文件被误判为重复（生命周期=已废弃（重复）），清除 表A 中该记录的生命周期值，删除软链接文档，再重启本步；AI 将重新文本化并再次查重

重进入不影响已存在的切分方案和下游已处理的结果。

## 五、模板与格式

### 5.1 OR文本化文档状态表模板

该单行表写在 OR文本化文档顶部，供人类打开文档时快速了解当前处理状态。每次写入完全覆盖旧表。

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 文本化完成 | — | 在 状态文档 → 表A — OR原料处理建议及进展中确认入选本批材料 → 运行 eos-ort01-chunk | 状态文档 → 表A — OR原料处理建议及进展 → 该记录 人类决策 字段设为"同意切分"（选入切分）→ 运行 eos-ort01-chunk |
```

**字段说明：**

| 字段 | 说明 |
|------|------|
| 阶段 | 当前处理环节名称 |
| AI建议 | AI 对当前阶段的判断结论。无建议值时填"—" |
| 待处理 | 人类待办事项概述 |
| 操作指引 | 人类下一步应执行的操作 |

> **注**：OR文本化文档状态表的 AI建议与本步写入 表A — OR原料处理建议及进展的 AI建议（可以切分）取值不同——文档内表给人类查阅当前阶段（文本化为纯机械步骤，AI 无需判断，故填"—"），表A 的 AI建议是面向选材决策的**下一步建议**，让人类在状态文档中一目了然每份材料的后续走向。用途不同，不要求一致。

### 5.2 EOS系统设计状态文档（01-eos-sysdev-status.md）模板

状态文档是设计链全流程状态追踪的共享接口，各阶段子表覆盖 OR→SR→BA→SysReq→PA 全链的材料级和条目级状态追踪。仅在首次运行时由 eos-ort00 自动创建（含 OR 阶段 表A/表B/表C），下游 SR/BA/SysReq/PA 阶段子表（2.1~5.1）在其对应步骤首次运行时自动追加。本表是材料级和条目级状态追踪的**权威源**——各产品文档（OR 基线、SR 架构定义等）头部的内嵌状态表由本表对应子表派生。

**同步契约**：AI 每次执行写入时，必须**先更新本表（权威源），再从本表推导写入各产品文档的内嵌状态表**。若检测到两者不一致，以本表为准重建内嵌状态表。

**标准表头（本步创建：OR 阶段 表A/表B/表C）：**

# EOS 系统设计状态文档

## 七、与下游步骤的关系

```
eos-ort00-textualize（本步：纯文本化）
  └─ 正文区：纯原文
      ↓ 检测到正文区有原文（无切分方案块）
eos-ort01-chunk（语义切分）
  ├─ 正文区：原文 + 切分方案块 + 反馈提示区
  └─ 执行后清理方案块，正文区回归纯原文 + 管理区状态摘要指向 eos-ort02-clarify
      ↓
eos-ort02-clarify（澄清）
  ├─ 正文区：原文 + 模糊标注 [⨯/✓] + 模糊标注报告
  └─ 达成已澄清后清理标注，正文区回归纯原文
      ↓
eos-ort03-norm（规范化）
  ├─ 正文区：原文 + 分析建议块（临时编号 OR 条目 + 冲突检测结果）+ 反馈提示区
  └─ 执行后清理分析建议块，正文区回归纯原文
```
