# 原始需求切片执行

**Skill ID**: ort01-chunk-execute · **Step ort01: Original Requirement Chunk Execution**

> **概述**：读取文本化原始需求文档（含 AI 提议的切片标记和/或人类调整后的标记），执行语义切片，输出切片文件并注册 OR 状态表。本步可迭代——人类可在文本化文档中调整切片标记后重启本步，AI 将按更新后的标记重新切片。
>
> **输入 → 输出**：文本化原始需求文档（`{原始文件名}.md`，含切片标记）+ OR 状态表（含切片方案区域）→ 切片文件（`{原始文件名}-chunk-{seq}.md`）+ OR 状态表更新（注册各切片记录）
>
> **接口设计**：原始需求生命周期状态表（简称 OR 状态表）是预处理链中各顺序 Skill 间的共享接口。ort01 读取 OR 状态表中的切片方案区域和文本化文档中的切片标记执行切片。每个切片在 OR 状态表中注册为独立记录（含 parent_file 字段关联回原始文件），生命周期初始化为"已登记"。本步可多次执行——每次读取文本化文档中的当前切片标记重新切片，旧切片标记为"已废弃"。
>
> **适用场景**：接续 ort00 输出的文本化原始需求文档（含切片标记）或人类自行编写的切片标记文档。切片文件产出后供下游 ort02-annotate 进行首轮模糊标注。
>
> **版本**：v0.1 | **修订**：2026-06-06

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：`../../80-pl4eos-2-eosdata/_inputs/10-source-materials/` 目录中的文本化原始需求文档（`{原始文件名}.md`，含切片标记）+ `../../80-pl4eos-2-eosdata/_inputs/01-eos-or-lifecycle.md`
- **产出**：`10-source-materials/` 目录中的切片文件（每片一个 `{原始文件名}-chunk-{seq}.md`）+ OR 状态表 更新（各切片独立记录）

### 与三条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **详概分离** | 本步仅执行语义切片，不涉及需求内容的理解或分析。切片是纯粹的文档结构重组 |
| **文档结对** | 本步不产生文档结对关系。本步产出的切片文件是 ort02-annotate（首轮模糊标注）的输入 |
| **基于资产** | 切片文件是原始文档的派生资产，通过文件名关联（`-chunk-{seq}`）保持追溯。OR 状态表中通过 parent_file 字段维护切片与原始文件的关系 |

## 三、操作流程

本步按"扫描 → 读取标记 → 执行切片 → 输出与注册 → 行动清单"的线性序列执行。支持迭代——人类可在文本化文档中调整切片标记后重启本步。

操作流程概览：

```
[入口] 扫描文本化文档 + OR 状态表
    │
    ├─ 无符合条件（含切片标记且状态=待执行）的文本化文档 → 结束
    │
    └─ 有符合条件的文本化文档
         │
         ├─ 首次执行
         │    ├─ [阶段一] 读取切片标记
         │    ├─ [阶段二] 按标记边界执行切片
         │    ├─ [阶段三] 输出切片文件 + 注册 OR 状态表
         │    └─ [行动清单] 提示人类确认切片结果或反馈建议
         │
         └─ 重切执行（切片标记已变更）
              ├─ [阶段一] 读取更新后的切片标记
              ├─ [阶段二] 对比新旧标记，识别变更
              ├─ [阶段三] 更新切片文件 + 废弃旧切片 + 注册新切片
              └─ [行动清单] 提示人类确认或反馈建议
```

### 阶段一：读取切片标记

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 扫描文本化文档 | AI | 读取 `{原始文件名}.md`，扫描文档中的 `[切片建议]` 和 `[切片结束]` 标记 |
| 2 | 解析标记 | AI | 提取每对标记之间的内容范围。标记格式：`[切片建议: {序号} · {语义标签}]` ... `[切片结束: {序号}]` |
| 3 | 校验标记完整性 | AI | 检查所有标记是否成对、范围是否重叠、是否有未覆盖的文档区域。发现异常时在行动清单中标注 |

**标记解析规则**：

| 情况 | 处理 |
|------|------|
| 标记成对且边界清晰 | 按标记范围切片 |
| 标记不成对（有开始无结束/反过来） | 标注异常，提示人类修正 |
| 标记范围有重叠 | 按先出现的标记为准，提示重叠范围 |
| 文档中有未覆盖的段落 | 标注为"未分配内容"，由人类决定归属哪些片 |
| 整份文档标记为 `[切片建议: 不切片]` | 跳过切片流程，生命周期保持"已登记"，提示人类直接启动 ort02-annotate |
| 无任何切片标记 | 输出"未发现切片标记"提示，结束（返回 ort00 或人工处理） |

### 阶段二：执行切片

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 按标记边界分割内容 | AI | 将每对 `[切片建议]` ~ `[切片结束]` 之间的文本提取为一个独立切片 |
| 2 | 为每片附加元数据 | AI | 每片开头插入 YAML frontmatter，记录来源文件、切片序号、语义标签、范围描述 |
| 3 | 移除标记行 | AI | 切出的文档内容中移除 `[切片建议]` 和 `[切片结束]` 标记行（元数据已在 frontmatter 中保留） |

**切片文件内容结构**：

```markdown
---
chunk:
  source: 01-市场调研报告.md       # 源文件（文本化文档）
  original_source: 01-市场调研报告.pdf  # 原始文件
  seq: 001
  total: 3                          # 本次切片总数
  label: 市场部业务需求
  range: §2.1~§2.5
---

（切片正文内容，无切片标记）
```

**切片文件的目录与命名**：

与原始文件同目录，命名格式：`{原始文件名}-chunk-{seq}.md`

```
10-source-materials/
├── 01-市场调研报告.pdf              ← 原始文件（只读）
├── 01-市场调研报告.md                ← 文本化文档（含切片标记）
├── 01-市场调研报告-chunk-001.md      ← 切片输出
├── 01-市场调研报告-chunk-002.md      ← 切片输出
└── 01-市场调研报告-chunk-003.md      ← 切片输出
```

### 阶段三：输出与注册

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 输出切片文件 | AI | 将每片写入独立的 `{原始文件名}-chunk-{seq}.md` 文件 |
| 2 | 更新 OR 状态表 | AI | 为每个切片创建独立记录行。首次切片→注册为"已登记"；重切→旧切片标记为"已废弃"，新切片注册为"已登记" |
| 3 | 更新切片方案区域状态 | AI | 将 OR 状态表中该文件的切片方案区域状态更新为"切片已执行" |

**OR 状态表记录格式（切片行）**：

| 文件 | parent_file | chunk_label | 生命周期状态 | 人类决策 | ... |
|------|------------|-------------|------------|---------|-----|
| 01-市场调研报告-chunk-001.md | 01-市场调研报告.pdf | 市场部业务需求(§2.1~§2.5) | 已登记 | | ... |
| 01-市场调研报告-chunk-002.md | 01-市场调研报告.pdf | 研发部开发流程(§3.1~§3.4) | 已登记 | | ... |

> **重切时**：旧切片记录的生命周期状态标记为"已废弃"（AI 建议=已废弃，人类决策不修改），新切片注册为"已登记"。已废弃的切片不再被下游步骤拾取。

### 行动清单输出

```
────────────────────────────────────────────────────
▲ ort01-chunk-execute 执行完成 —— 以下为人类执行段
────────────────────────────────────────────────────

  → 对已切片的文件（如 01-市场调研报告）：
    已输出 3 个切片文件：
    - 01-市场调研报告-chunk-001.md（市场部业务需求）
    - 01-市场调研报告-chunk-002.md（研发部开发流程）
    - 01-市场调研报告-chunk-003.md（售后部工单管理）

    请确认切片结果：
    ├─ 切片边界正确 → 启动 ort02-annotate（进入首轮模糊标注）
    ├─ 需调整边界   → 打开 01-市场调研报告.md 移动 [切片建议] 标记后重启 ort01-chunk-execute
    └─ 完全不同意   → 打开 01-市场调研报告.md 重写切片标记后重启 ort01-chunk-execute

  → 对标记为"不切片"的文件：
    └─ 确认后启动 ort02-annotate（直接进入首轮模糊标注）
```

## 四、AI 执行指南

### 4.1 切片边界处理

| 边界情况 | 处理方式 |
|---------|---------|
| 标记边界在段落中间 | 按标记边界切，被截断的段落完整归入起始片。在行动清单中标注"段落被边界截断" |
| 连续两片之间有空白行 | 空白行归入前一片 |
| 切片内容为空（标记之间无内容） | 不输出空文件，视为无效标记，在行动清单中报告 |
| 切片文件与已有文件重名（重切场景） | 覆盖旧文件。旧切片在 OR 状态表中标记为"已废弃" |

### 4.2 重切时的变更检测

AI 通过比对切片文件的 **内容签名** 与 OR 状态表中记录的上次切片结果来判断是否需要重切：

| 条件 | 处理 |
|------|------|
| 文本化文档中的切片标记未变化 | 跳过，复用已有切片文件 |
| 文本化文档中的切片标记已变化 | 按新标记重新切片。新注册的切片文件覆盖旧文件，旧切片标记为"已废弃" |
| 新增了切片（原 3 片变 5 片） | 新增片注册为"已登记" |
| 减少了切片（原 5 片变 3 片） | 减少的片标记为"已废弃" |
| 切片序号重新排列 | 按新序号输出，旧序号标记为"已废弃" |

### 4.3 人类指令与标记

#### 4.3.1 人类对切片标记的调整方式

人类在文本化文档中可直接：

| 操作 | 方式 | AI 处理 |
|------|------|---------|
| 确认边界 | 保留 `[切片建议]` 标记不变 | 按标记执行 |
| 移动边界 | 将 `[切片建议]` 或 `[切片结束]` 标记移到其他位置 | 按新边界切片 |
| 新增切片 | 插入新的 `[切片建议: {新序号} · {标签}]` ... `[切片结束: {新序号}]` | 识别新标记，增加切片 |
| 合并切片 | 删除两片之间的边界标记 | 两片合并为一片 |
| 取消切片 | 删除所有切片标记，添加 `[切片建议: 不切片]` | 跳过切片 |

#### 4.3.2 标记约定

切片标记使用与 ort00 相同的格式，人类可在此基础上调整：

```
[切片建议: 001 · 市场部业务需求]    ← 人类可移动此行位置
...内容...
[切片结束: 001]                      ← 人类可移动此行位置
```

### 4.4 启动方式

#### 人类启动指令

> 开始原始需求切片执行（或 ort01-chunk-execute）

#### AI 动作序列

0. **校验前置条件**：检查 OR 状态表中是否存在"切片方案区域状态=待人类确认 或 人类已确认"的记录。不存在则输出"未找到待切片材料"→ **结束**
1. **扫描**：扫描 `10-source-materials/` 目录中的文本化 `.md` 文件，提取切片标记
2. **分流**：
   - 有切片标记且 OR 状态表无对应切片记录 → 首次切片
   - 有切片标记且 OR 状态表有对应切片记录但标记已变更 → 重切
   - 有切片标记但 OR 状态表中现有切片记录与标记一致 → 跳过
   - 标记为"不切片"→ 跳过切片流程
   - 无切片标记 → 报告异常
3. **首次切片**：阶段一（读取标记）→ 阶段二（执行切片）→ 阶段三（输出+注册）
4. **重切**：阶段一（读取更新后标记）→ 阶段二（对比变更+执行切片）→ 阶段三（更新+注册）
5. **输出行动清单**：按 §三.行动清单输出 格式输出

### 4.5 与 OR 状态表的字段联动

| 字段 | 操作 | 说明 |
|------|------|------|
| 文件 | 写入 | 切片文件名称（如 `01-市场调研报告-chunk-001.md`） |
| parent_file | 写入 | 原始文件名称 |
| chunk_label | 写入 | 语义标签+范围 |
| 生命周期状态 | 写入（首次）→ 已登记；写入（重切·新片）→ 已登记；更新（重切·旧片）→ 已废弃 |
| 切片方案区域·状态 | 更新 | 待人类确认 → 切片已执行 |
| AI 建议 | 写入 | 切片完成 / 需确认 |
| 人类决策 | 不操作 | 保留，由人类在确认后修改 |

## 五、模板与输出格式

### 5.1 切片文件模板

```markdown
---
chunk:
  source: {原始文件名}.md
  original_source: {原始文件名}.{ext}
  seq: {序号}
  total: {总片数}
  label: {语义标签}
  range: {原文范围}
---

（切片正文内容）
```

| 字段 | 说明 |
|------|------|
| source | 文本化文档文件名 |
| original_source | 原始文件文件名（只读档案） |
| seq | 本片序号（三位数字） |
| total | 本次切片总片数 |
| label | 语义标签，概括本片主题 |
| range | 原文中的范围描述（章节/段落/行号范围等） |

### 5.2 OR 状态表切片记录格式

OR 状态表在现有字段基础上扩展以下字段（仅切片文件的行使用）：

| 字段 | 示例 | 说明 |
|------|------|------|
| 文件 | 01-市场调研报告-chunk-001.md | 切片文件名称 |
| parent_file | 01-市场调研报告.pdf | 原始文件名称 |
| chunk_label | 市场部业务需求(§2.1~§2.5) | 语义标签+范围 |

非切片文件（普通原始材料）的 parent_file 和 chunk_label 字段为空。

### 5.3 自检清单

- [ ] 切片文件是否与原始文件命名前缀一致（`{原始文件名}-chunk-{seq}`）
- [ ] 每片文件是否包含完整的 YAML frontmatter 元数据
- [ ] 切片文件中是否已移除 `[切片建议]` / `[切片结束]` 标记行
- [ ] 各切片内容是否无遗漏（全部标记范围内的内容均已输出）
- [ ] OR 状态表中各切片行是否已正确注册（含 parent_file + chunk_label）
- [ ] 重切场景下旧切片是否已标记为"已废弃"
- [ ] 标记为"不切片"的文件是否已跳过切片流程
