# 原始需求切分执行

**Skill ID**: ort01-chunk-execute · **Step ort01: Original Requirement Chunk Execution**

> **概述**：读取文本化原始需求文档（含 AI 提议的切分标记和/或人类调整后的标记），执行语义切分，输出切分文件并注册 OR 状态表。本步可迭代——人类可在文本化文档中调整切分标记后重启本步，AI 将按更新后的标记重新切分。
>
> **输入 → 输出**：文本化原始需求文档（`{原始文件名}.md`，含切分标记）+ OR 状态表（含切分状态表）→ 切分文件（`{原始文件名}-chunk-{seq}.md`）+ OR 状态表更新（注册各切分记录 + 切分状态表更新）
>
> **接口设计**：原始需求生命周期状态表（简称 OR 状态表）是预处理链中各顺序 Skill 间的共享接口。ort01 读取切分状态表中状态=切分已确认 的记录确定待切分材料，从文本化文档中读取切分标记执行切分。每个切分在 OR 状态表中注册为独立记录（含 parent_file 字段关联回原始文件），生命周期初始化为"已登记"。本步可多次执行——每次读取文本化文档中的当前切分标记重新切分，旧切分标记为"已废弃"。
>
> **适用场景**：接续 ort00 输出的文本化原始需求文档（含切分标记）或人类自行编写的切分标记文档。切分文件产出后供下游 ort02-annotate 进行首轮模糊标注。
>
> **版本**：v0.4 | **修订**：2026-06-06

- **输入**：`../../80-pl4eos-2-eosdata/_inputs/10-source-materials/` 目录中的文本化原始需求文档（`{原始文件名}.md`，含切分标记。.md 格式源材料对应 `{原始文件名}.textualized.md`）+ `../../80-pl4eos-2-eosdata/_inputs/01-eos-or-lifecycle.md`

| 原则 | 在本步中的体现 |
|------|--------------|
| **详概分离** | 本步仅执行语义切分，不涉及需求内容的理解或分析。切分是纯粹的文档结构重组 |
| **文档结对** | 本步不产生文档结对关系。本步产出的切分文件是 ort02-annotate（首轮模糊标注）的输入 |
| **基于资产** | 切分文件是原始文档的派生资产，通过文件名关联（`-chunk-{seq}`）保持追溯。OR 状态表中通过 parent_file 字段维护切分与原始文件的关系 |

## 三、操作流程

本步按"扫描确认 → 读取标记 → 执行切分 → 写入输出 → 行动清单 → 人机交互"的线性序列执行。支持迭代——人类可在文本化文档中调整切分标记后重启本步。

操作流程概览：

```
[入口] 扫描 `10-source-materials/` 中的文本化文档
    │
    ├─ 检查文档末尾的 `【切分确认】` 区
    │
    ├─ 读到 [确认切分] ← 保留
    │    ├─ 自动更新切分状态表 → "切分已确认"
    │    └─ 进入切分分流 ↓
    │
    ├─ 读到 [确认不切分] ← 保留
    │    ├─ 自动更新切分状态表 → "不切分已确认"
    │    └─ 跳过该文件，提示启动 ort02（无需切分）
    │
    ├─ 读到 [返回分析] ← 保留
    │    ├─ 更新切分状态表 → "待重切"
    │    └─ 跳过该文件，提示人类运行 ort00
    │
    ├─ 读到 [仍需切分] ← 保留
    │    ├─ 更新切分状态表 → "待重切"
    │    └─ 跳过该文件，提示人类运行 ort00
    │
    └─ 无有效确认标记
         └─ 跳过该文件（不处理）
              │
         ┌────┘
         ▼
    [切分分流] 检查已有切分记录
         │
         ├─ OR 状态表无对应切分记录 → 首次切分 ↓
         │
         ├─ 有对应记录但标记已变更 → 重切 ↓
         │
         └─ 有对应记录且标记未变 → 跳过（复用已有切分文件）
              │
         ┌────┘
         ▼
    [阶段一] 读取切分标记
         │
    [阶段二] 按标记边界执行切分
         ↓

    [写入] 输出切分文件 + 更新 OR 状态表
      切分文件写入 10-source-materials/（{原始文件名}-chunk-{seq}.md）
      切分状态表：更新为"切分已执行"+ 写入操作指引
      登记表：各切分注册为独立记录（生命周期=已登记，含 parent_file + chunk_label）
         ↓

    [行动清单] 提示人类确认切分结果或反馈建议
         ↓

    [人机交互] 询问人类是否打开登记表
      ├─ 回答"是"→ 输出登记表路径 + 建议关注行
      └─ 回答"否"→ 跳过

    → 本轮结束
```

### 入口：确认标记检测

AI 在读取切分标记之前，先检测文档末尾的切分确认区：

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 0 | 扫描确认区 | AI | 读取文档末尾的 `【切分确认】` 区，检测人类保留的操作选项 |
| 1 | 分流判断 | AI | 读到 `[确认切分]` 保留→自动更新为"切分已确认"→进入阶段一；读到 `[确认不切分]` 保留→自动更新为"不切分已确认"→跳过后提示启动 ort02；读到 `[返回分析]` 或 `[仍需切分]` 保留→更新为"待重切"→跳过后提示运行 ort00；无有效确认标记→跳过该文件 |

> **文档即状态**：确认标记是文档中的人类操作记录，AI 据此自动维护切分状态表，人类不需要直接编辑状态表字段。此机制与三字段正交模型的"人类操作文档、AI 同步状态"原则一致。

### 阶段一：读取切分标记

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 扫描文本化文档 | AI | 读取 `{原始文件名}.md`，扫描文档中的 `[切分建议]` 和 `[切分结束]` 标记 |
| 2 | 解析标记 | AI | 提取每对标记之间的内容范围。标记格式：`[切分建议: {序号} · {语义标签}]` ... `[切分结束: {序号}]` |
| 3 | 校验标记完整性 | AI | 检查所有标记是否成对、范围是否重叠、是否有未覆盖的文档区域。发现异常时在行动清单中标注 |

**标记解析规则**：

| 情况 | 处理 |
|------|------|
| 标记成对且边界清晰 | 按标记范围切分 |
| 标记不成对（有开始无结束/反过来） | 标注异常，提示人类修正 |
| 标记范围有重叠 | 按先出现的标记为准，提示重叠范围 |
| 文档中有未覆盖的段落 | 标注为"未分配内容"，由人类决定归属哪些片 |
| 整份文档标记为 `[切分建议: 不切分]` | 跳过切分流程，生命周期保持"已登记"，提示人类直接启动 ort02-annotate |
| 无任何切分标记 | 输出"未发现切分标记"提示，结束（返回 ort00 或人工处理） |

### 阶段二：执行切分

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 按标记边界分割内容 | AI | 将每对 `[切分建议]` ~ `[切分结束]` 之间的文本提取为一个独立切分 |
| 2 | 为每片附加元数据 | AI | 每片开头插入 YAML frontmatter，记录来源文件、切分序号、语义标签、范围描述 |
| 3 | 移除标记行 | AI | 切出的文档内容中移除 `[切分建议]` 和 `[切分结束]` 标记行（元数据已在 frontmatter 中保留） |

**切分文件内容结构**：

```markdown
---
chunk:
  source: 01-市场调研报告.md       # 源文件（文本化文档）
  original_source: 01-市场调研报告.pdf  # 原始文件
  seq: 001
  total: 3                          # 本次切分总数
  label: 市场部业务需求
  range: §2.1~§2.5
---

（切分正文内容，无切分标记）
```

**切分文件的目录与命名**：

与原始文件同目录，命名格式：`{原始文件名}-chunk-{seq}.md`

```
10-source-materials/
├── 01-市场调研报告.pdf              ← 原始文件（只读）
├── 01-市场调研报告.md                ← 文本化文档（含切分标记）
├── 01-市场调研报告-chunk-001.md      ← 切分输出
├── 01-市场调研报告-chunk-002.md      ← 切分输出
└── 01-市场调研报告-chunk-003.md      ← 切分输出
```

### 写入：输出切分文件 + 更新 OR 状态表

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 输出切分文件 | AI | 将每片写入独立的 `{原始文件名}-chunk-{seq}.md` 文件 |
| 2 | 更新 OR 状态表 | AI | 为每个切分创建独立记录行。首次切分→注册为"已登记"；重切→旧切分标记为"已废弃"，新切分注册为"已登记" |
| 3 | 更新切分状态表 | AI | 将切分状态表中该文件的状态更新为"切分已执行"，同步写入操作指引 |

**OR 状态表记录格式（切分行）**：

| 文件 | parent_file | chunk_label | 生命周期状态 | 人类决策 | ... |
|------|------------|-------------|------------|---------|-----|
| 01-市场调研报告-chunk-001.md | 01-市场调研报告.pdf | 市场部业务需求(§2.1~§2.5) | 已登记 | | ... |
| 01-市场调研报告-chunk-002.md | 01-市场调研报告.pdf | 研发部开发流程(§3.1~§3.4) | 已登记 | | ... |

> **重切时**：旧切分记录的生命周期状态标记为"已废弃"（AI 建议=已废弃，人类决策不修改），新切分注册为"已登记"。已废弃的切分不再被下游步骤拾取。

AI 随后发起对话交互：

人类回答"是"后，AI 直接提供可打开的文件路径：

建议关注行：

## 四、AI 执行指南

### 4.1 切分边界处理

| 边界情况 | 处理方式 |
|---------|---------|
| 标记边界在段落中间 | 按标记边界切，被截断的段落完整归入起始片。在行动清单中标注"段落被边界截断" |
| 连续两片之间有空白行 | 空白行归入前一片 |
| 切分内容为空（标记之间无内容） | 不输出空文件，视为无效标记，在行动清单中报告 |
| 切分文件与已有文件重名（重切场景） | 覆盖旧文件。旧切分在 OR 状态表中标记为"已废弃" |

### 4.2 重切时的变更检测

AI 通过比对切分文件的 **内容签名** 与 OR 状态表中记录的上次切分结果来判断是否需要重切：

| 条件 | 处理 |
|------|------|
| 文本化文档中的切分标记未变化 | 跳过，复用已有切分文件 |
| 文本化文档中的切分标记已变化 | 按新标记重新切分。新注册的切分文件覆盖旧文件，旧切分标记为"已废弃" |
| 新增了切分（原 3 片变 5 片） | 新增片注册为"已登记" |
| 减少了切分（原 5 片变 3 片） | 减少的片标记为"已废弃" |
| 切分序号重新排列 | 按新序号输出，旧序号标记为"已废弃" |

### 4.3 人类指令与标记

#### 4.3.1 人类对切分标记的调整方式

人类在文本化文档中可直接：

| 操作 | 方式 | AI 处理 |
|------|------|---------|
| 确认边界 | 保留 `[切分建议]` 标记不变 | 按标记执行 |
| 移动边界 | 将 `[切分建议]` 或 `[切分结束]` 标记移到其他位置 | 按新边界切分 |
| 新增切分 | 插入新的 `[切分建议: {新序号} · {标签}]` ... `[切分结束: {新序号}]` | 识别新标记，增加切分 |
| 合并切分 | 删除两片之间的边界标记 | 两片合并为一片 |
| 取消切分 | 删除所有切分标记，添加 `[切分建议: 不切分]` | 跳过切分 |

#### 4.3.2 标记约定

切分标记使用与 ort00 相同的格式，人类可在此基础上调整：

```
[切分建议: 001 · 市场部业务需求]    ← 人类可移动此行位置
...内容...
[切分结束: 001]                      ← 人类可移动此行位置
```

#### 4.3.3 切分确认区的操作方式

ort00 在文本化文档末尾追加了 `【切分确认】` 区，嵌入两个操作选项。确认区的格式根据 ort00 的切分建议动态生成——两种变体：

**变体 A — 切分建议=切分(N片) 时：**

```
---

【切分确认】请确认切分方案。删除不适用的选项，保留你的选择（保留一整行）：

1️⃣ [确认切分] → 切分边界正确（含任何已做的调整），启动 ort01-chunk-execute
2️⃣ [返回分析] → 不同意切分方案或标记有较大调整，返回 ort00-chunk-propose 重新分析
```

**变体 B — 切分建议=不切分 时：**

```
---

【切分确认】请确认切分建议。删除不适用的选项，保留你的选择（保留一整行）：

1️⃣ [确认不切分] → 同意无需切分，启动 ort01
2️⃣ [仍需切分] → 文档应切分，返回 ort00 重新分析
```

| 变体 | 保留选项 | AI 反应 |
|------|---------|---------|
| A（切分建议=切分） | `[确认切分]` | 自动提升状态为"切分已确认"→ 执行切分 |
| A（切分建议=切分） | `[返回分析]` | 更新状态为"待重切"→ 提示运行 ort00 |
| B（切分建议=不切分） | `[确认不切分]` | 自动提升状态为"不切分已确认"→ 提示启动 ort02 |
| B（切分建议=不切分） | `[仍需切分]` | 更新状态为"待重切"→ 提示运行 ort00 |

> **注意**：人类无需直接编辑切分状态表。确认标记是文档中的人类操作记录，AI 在 ort01 入口自动检测并同步状态表。

### 4.4 启动方式

#### 人类启动指令

> 开始原始需求切分执行（或 ort01-chunk-execute）

#### AI 动作序列

0. **扫描确认**：扫描 `../../80-pl4eos-2-eosdata/_inputs/10-source-materials/` 中的文本化 `.md` 文件，读取文档末尾的 `【切分确认】` 区：
   - 读到 `[确认切分]` 保留 → 自动更新切分状态表为"切分已确认"，进入步骤 1
   - 读到 `[确认不切分]` 保留 → 自动更新切分状态表为"不切分已确认"，跳过后提示启动 ort02
   - 读到 `[返回分析]` 或 `[仍需切分]` 保留 → 更新切分状态表为"待重切"，提示人类运行 ort00-chunk-propose，跳过该文件
   - 无有效确认标记 → 跳过该文件（不处理）
1. **扫描标记**：读取步骤 0 中已确认的文本化文档，提取文档中的 `[切分建议]` 和 `[切分结束]` 标记
2. **分流**：
   - 有切分标记且 OR 状态表无对应切分记录 → 首次切分
   - 有切分标记且 OR 状态表有对应切分记录但标记已变更 → 重切
   - 有切分标记但 OR 状态表中现有切分记录与标记一致 → 跳过（复用已有切分文件）
   - 标记为"不切分"→ 跳过切分流程
   - 无切分标记 → 报告异常
3. **首次切分**：阶段一（读取标记）→ 阶段二（执行切分）→ 写入（输出+注册）
4. **重切**：阶段一（读取更新后标记）→ 阶段二（对比变更+执行切分）→ 写入（更新+注册）
5. **输出行动清单**：按 §三.行动清单输出 格式输出
6. **人机交互**：向人类提供登记表路径和关注指引，询问是否需要打开查看

### 4.5 与 OR 状态表（含切分状态表）的字段联动

| 字段 | 操作 | 说明 |
|------|------|------|
| 文件 | 写入 | 切分文件名称（如 `01-市场调研报告-chunk-001.md`）；不切分场景无需写入 |
| parent_file | 写入 | 原始文件名称；不切分场景无需写入 |
| chunk_label | 写入 | 语义标签+范围；不切分场景无需写入 |
| 生命周期状态 | 写入（首次）→ 已登记；写入（重切·新片）→ 已登记；更新（重切·旧片）→ 已废弃；不切分场景→保持"已登记"不变 |
| 切分状态表·状态 | 更新 | `[确认切分]` → 切分已确认 → 切分已执行；`[确认不切分]` → 不切分已确认（终态） |
| 切分状态表·操作指引 | 更新 | 切分执行后写入"切分已完成，进入切分各自的登记表流程"；不切分确认后写入"无需切分，启动 ort02 进入标注处理" |
| AI 建议 | 写入 | 切分完成 / 不切分已确认 |
| 人类决策 | 不操作 | 保留，由人类在确认后修改 |

## 五、模板与输出格式

### 5.1 切分文件模板

```markdown
---
chunk:
  source: {原始文件名}.md
  original_source: {原始文件名}.{ext}
  seq: {序号}
  total: {总片数}
  label: {语义标签}
  range: {原文范围}
---

（切分正文内容）
```

| 字段 | 说明 |
|------|------|
| source | 文本化文档文件名 |
| original_source | 原始文件文件名（只读档案） |
| seq | 本片序号（三位数字） |
| total | 本次切分总片数 |
| label | 语义标签，概括本片主题 |
| range | 原文中的范围描述（章节/段落/行号范围等） |

### 5.2 OR 状态表切分记录格式

OR 状态表在现有字段基础上扩展以下字段（仅切分文件的行使用）：

| 字段 | 示例 | 说明 |
|------|------|------|
| 文件 | 01-市场调研报告-chunk-001.md | 切分文件名称 |
| parent_file | 01-市场调研报告.pdf | 原始文件名称 |
| chunk_label | 市场部业务需求(§2.1~§2.5) | 语义标签+范围 |

非切分文件（普通原始材料）的 parent_file 和 chunk_label 字段为空。

### 5.3 自检清单

- [ ] 切分文件是否与原始文件命名前缀一致（`{原始文件名}-chunk-{seq}`）
- [ ] 每片文件是否包含完整的 YAML frontmatter 元数据
- [ ] 切分文件中是否已移除 `[切分建议]` / `[切分结束]` 标记行
- [ ] 各切分内容是否无遗漏（全部标记范围内的内容均已输出）
- [ ] OR 状态表中各切分行是否已正确注册（含 parent_file + chunk_label）
- [ ] 重切场景下旧切分是否已标记为"已废弃"
- [ ] 标记为"不切分"的文件是否已跳过切分流程
- [ ] 变体 B（确认不切分/仍需切分）的确认标记是否已正确处理
- [ ] `[确认不切分]` 状态是否已自动更新为"不切分已确认"（终态）
