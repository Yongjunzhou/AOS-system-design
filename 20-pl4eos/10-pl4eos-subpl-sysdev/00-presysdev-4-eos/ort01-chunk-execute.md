# 文本化的原始需求材料切分执行与切分优化

**Skill ID**: ort01-chunk-execute · **Step ort01: Textualized Original Requirement Chunk Execution and Optimization**

> **概述**：读取文本化原始需求文档（含 AI 提议的切分标记和/或人类调整后的标记），执行语义切分，输出切分文件并注册 OR 状态表。本步可迭代——人类可在文本化文档中调整切分标记后重启本步，AI 将按更新后的标记重新切分。
>
> **输入 → 输出**：文本化原始需求文档（`{原始文件名}.md`，含切分标记）+ OR 状态表（含切分状态表）→ 切分文件（`{原始文件名}-chunk-{seq}.md`）+ OR 状态表更新（注册各切分记录 + 切分状态表更新）
>
> **接口设计**：原始需求生命周期状态表（简称 OR 状态表）是预处理链中各顺序 Skill 间的共享接口。ort01 读取切分状态表中状态=切分已确认 的记录确定待切分材料，从文本化文档中读取切分标记执行切分。每个切分在 OR 状态表中注册为独立记录（含 parent_file 字段关联回原始文件），生命周期初始化为"已登记"。本步可多次执行——每次读取文本化文档中的当前切分标记重新切分，旧切分标记为"已废弃"。
>
> **适用场景**：接续 ort00 输出的文本化原始需求文档（含切分标记）或人类自行编写的切分标记文档。切分文件产出后供下游 ort02-annotate 进行首轮模糊标注。
>
> **版本**：v0.6 | **修订**：2026-06-06

## 三、操作流程

核心驱动逻辑：读取 OR 状态表中切分状态表的当前状态和文本化文档中的确认标记，按状态分流——"待确认切分方案"时检测首次确认标记后执行切分；"切分已执行"时通过 last_size 比对检测标记是否变更，确认后执行重切。每轮执行后刷新确认区，使旧标记不再可被误读，支持当前状态→确认→执行的显式迭代。

本步按"扫描分流 → 阶段一 → 阶段二 → 写入 → 行动清单 → 人机交互"的线性序列执行。扫描分流→阶段一→阶段二→写入为 AI 独立执行段；末端的[行动清单]→[人机交互]为人类交接点（提供切分确认指引和登记表查阅），不影响本轮 AI 执行结果。

操作流程概览：

```
[扫描分流] 扫描文本化文档 + OR 状态表（切分状态表）
  │
  ├─ "待确认切分方案"（首次进入）
  │    └─ 读确认区
  │         ├─ [确认切分] → 自动更新状态="切分已确认" → [阶段一] ↓
  │         └─ [确认不切分] → 更新状态="不切分已确认" → 提示启动 ort02
  │
  ├─ "切分已执行"（迭代重入）
  │    ├─ last_size 未变 → 跳过（标记未调整）
  │    └─ last_size 已变 → 读确认区
  │         ├─ [重新切分] → [阶段一] ↓（保留"切分已执行"状态）
  │         └─ 无有效选项 → 跳过（标记变了但未确认）
  │
  ├─ "不切分已确认"
  │    └─ 提示启动 ort02（如需改主意，加切分标记后手动改状态）
  │
  └─ 其他状态 → 跳过

[阶段一] 读取切分标记
    │
[阶段二] 按标记边界执行切分
    ↓

[写入] 输出切分文件 + 更新 OR 状态表 + 刷新确认区
  切分文件写入 10-source-materials/（{原始文件名}-chunk-{seq}.md）
  切分状态表：更新为"切分已执行"+ 写入操作指引 + 更新 last_size
  登记表：各切分注册为独立记录（生命周期=已登记，含 parent_file + chunk_label）
  文本化文档：追加新确认区（含[重新切分]选项）
    ↓

[行动清单] 输出线下行动指南
    ↓

[人机交互] 询问人类是否打开登记表
  ├─ 回答"是"→ 输出登记表路径 + 建议关注行
  └─ 回答"否"→ 跳过

→ 本轮结束
```

### 入口：扫描分流

AI 在读取切分标记之前，先读取 OR 状态表中的切分状态表，按当前状态分流：

| 当前切分状态表状态 | AI 操作 | 说明 |
|-------------------|---------|------|
| **待确认切分方案** | 检测文档末尾的 `【切分确认】` 区。读到 `[确认切分]` → 自动更新状态为"切分已确认" → 进入阶段一；读到 `[确认不切分]` → 更新状态为"不切分已确认" → 跳过该文件，提示启动 ort02；无有效选项 → 跳过该文件 | 首次进入 |
| **切分已执行** | 对比 `last_size`：文件大小 ≠ last_size → 标记已变更，检测确认区是否保留 `[重新切分]`。有效→进入阶段一执行重切（**切分状态表保持"切分已执行"，不降级**）；无有效选项→跳过。文件大小 = last_size → 标记未变，跳过 | 迭代重入 |
| **不切分已确认** | 提示启动 ort02（如需改主意，人类加切分标记后手动改状态表） | 终态 |
| **其他状态**（待分析/待重切/切分已确认等） | 跳过 | 非本步职责 |

> **last_size 变更检测**：切分状态表记录上次执行时文本化文档的文件大小。人类调整标记位置后文件大小变化，AI 据此感知"可能标记已变"，再通过确认区确认。未确认时不执行重切，防止文件保存过程中的临时变更触发意外操作。
>
> **状态不降级原则**："切分已执行"是执行态，迭代重切时不降级回"待确认切分方案"或"切分已确认"。重切完成后仍保持"切分已执行"，仅更新 last_size。此原则确保切分状态表的状态单调递进——"已执行"之后不再回退。人类操作文档中的标记（移动边界），AI 根据标记变更检测自动触发重切——无需重新经过 ort00。

### 阶段一：读取切分标记

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 扫描文本化文档 | AI | 读取 `{原始文件名}.md`，扫描文档中的 `[切分建议]` 和 `[切分结束]` 标记 |
| 2 | 解析标记 | AI | 提取每对标记之间的内容范围。标记格式：`[切分建议: {序号} · {语义标签}]` ... `[切分结束: {序号}]` |
| 3 | 校验标记完整性 | AI | 检查所有标记是否成对、范围是否重叠、是否有未覆盖的文档区域。发现异常时在行动清单中标注 |

**标记解析规则**：

| 情况 | 处理 |
|------|------|
| 标记成对且边界清晰 | 按标记范围切分 |
| 标记不成对（有开始无结束/反过来） | 标注异常，提示人类修正 |
| 标记范围有重叠 | 按先出现的标记为准，提示重叠范围 |
| 文档中有未覆盖的段落 | 标注为"未分配内容"，由人类决定归属哪些片 |
| 整份文档标记为 `[切分建议: 不切分]` | 跳过切分流程，生命周期保持"已登记"，提示人类直接启动 ort02-annotate |
| 无任何切分标记 | 输出"未发现切分标记"提示，结束（返回 ort00 或人工处理） |

### 阶段二：执行切分

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 按标记边界分割内容 | AI | 将每对 `[切分建议]` ~ `[切分结束]` 之间的文本提取为一个独立切分 |
| 2 | 为每片附加元数据 | AI | 每片开头插入 YAML frontmatter，记录来源文件、切分序号、语义标签、范围描述 |
| 3 | 移除标记行 | AI | 切出的文档内容中移除 `[切分建议]` 和 `[切分结束]` 标记行（元数据已在 frontmatter 中保留） |

**切分文件内容结构**：

```markdown
---
chunk:
  source: 01-市场调研报告.md       # 源文件（文本化文档）
  original_source: 01-市场调研报告.pdf  # 原始文件
  seq: 001
  total: 3                          # 本次切分总数
  label: 市场部业务需求
  range: §2.1~§2.5
---

（切分正文内容，无切分标记）
```

**切分文件的目录与命名**：

与原始文件同目录，命名格式：`{原始文件名}-chunk-{seq}.md`

```
10-source-materials/
├── 01-市场调研报告.pdf              ← 原始文件（只读）
├── 01-市场调研报告.md                ← 文本化文档（含切分标记）
├── 01-市场调研报告-chunk-001.md      ← 切分输出
├── 01-市场调研报告-chunk-002.md      ← 切分输出
└── 01-市场调研报告-chunk-003.md      ← 切分输出
```

### 写入：输出切分文件 + 更新 OR 状态表 + 刷新确认区

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 输出切分文件 | AI | 将每片写入独立的 `{原始文件名}-chunk-{seq}.md` 文件 |
| 2 | 更新 OR 状态表 | AI | 为每个切分创建独立记录行。首次切分→注册为"已登记"；重切→旧切分标记为"已废弃"，新切分注册为"已登记" |
| 3 | 更新切分状态表 | AI | 将切分状态表中该文件的状态更新为"切分已执行"，写入操作指引，更新 last_size（记录当前文本化文档的文件大小） |
| 4 | 刷新确认区 | AI | 将文本化文档末尾的确认区替换为形态②【切分确认】（形态②：报告切分已执行 + 提供 `[重新切分]` 选项），使旧确认标记不再可被误读 |

**OR 状态表记录格式（切分行）**：

| 文件 | parent_file | chunk_label | 生命周期状态 | 人类决策 | ... |
|------|------------|-------------|------------|---------|-----|
| 01-市场调研报告-chunk-001.md | 01-市场调研报告.pdf | 市场部业务需求(§2.1~§2.5) | 已登记 | | ... |
| 01-市场调研报告-chunk-002.md | 01-市场调研报告.pdf | 研发部开发流程(§3.1~§3.4) | 已登记 | | ... |

> **重切时**：旧切分记录的生命周期状态标记为"已废弃"（AI 建议=已废弃，人类决策不修改），新切分注册为"已登记"。已废弃的切分不再被下游步骤拾取。

## 四、AI 执行指南

### 4.1 切分边界处理

| 边界情况 | 处理方式 |
|---------|---------|
| 标记边界在段落中间 | 按标记边界切，被截断的段落完整归入起始片。在行动清单中标注"段落被边界截断" |
| 连续两片之间有空白行 | 空白行归入前一片 |
| 切分内容为空（标记之间无内容） | 不输出空文件，视为无效标记，在行动清单中报告 |
| 切分文件与已有文件重名（重切场景） | 覆盖旧文件。旧切分在 OR 状态表中标记为"已废弃" |

### 4.2 重切时的变更检测

AI 通过比对文本化文档的 **last_size** 与切分状态表中记录的上次执行时的文件大小来判断是否需要重切（由入口扫描分流触发，详见 §三·入口：扫描分流）。last_size 在每次切分/重切执行后更新：

| 条件 | 处理 |
|------|------|
| 文本化文档中的切分标记未变化 | 跳过，复用已有切分文件 |
| 文本化文档中的切分标记已变化 | 按新标记重新切分。新注册的切分文件覆盖旧文件，旧切分标记为"已废弃" |
| 新增了切分（原 3 片变 5 片） | 新增片注册为"已登记" |
| 减少了切分（原 5 片变 3 片） | 减少的片标记为"已废弃" |
| 切分序号重新排列 | 按新序号输出，旧序号标记为"已废弃" |

### 4.3 人类指令与标记

#### 4.3.1 人类对切分标记的调整方式

人类在文本化文档中可直接：

| 操作 | 方式 | AI 处理 |
|------|------|---------|
| 确认边界 | 保留 `[切分建议]` 标记不变 | 按标记执行 |
| 移动边界 | 将 `[切分建议]` 或 `[切分结束]` 标记移到其他位置 | 按新边界切分 |
| 新增切分 | 插入新的 `[切分建议: {新序号} · {标签}]` ... `[切分结束: {新序号}]` | 识别新标记，增加切分 |
| 合并切分 | 删除两片之间的边界标记 | 两片合并为一片 |
| 取消切分 | 删除所有切分标记，添加 `[切分建议: 不切分]` | 跳过切分 |

#### 4.3.2 标记约定

切分标记使用与 ort00 相同的格式，人类可在此基础上调整：

```
[切分建议: 001 · 市场部业务需求]    ← 人类可移动此行位置
...内容...
[切分结束: 001]                      ← 人类可移动此行位置
```

#### 4.3.3 切分确认区的操作方式

确认区有三种形态，分别对应首次进入时（由 ort00 写入）和执行后（由 ort01 刷新）。

**形态① — 由 ort00 写入（首次进入，建议切分）**：

```
---

【切分确认】请确认切分方案。删除不适用的选项，保留你的选择：

1️⃣ [确认切分] → 按当前切分标记执行切分
2️⃣ [确认不切分] → 文档不宜切分，跳过切分流程

（提示：可在本文件中直接移动 [切分建议] 标记调整边界）
```

**形态①' — 由 ort00 写入（首次进入，建议不切分）**：

```
---

【切分确认】请确认切分方案。删除不适用的选项，保留你的选择：

1️⃣ [确认不切分] → 同意无需切分
2️⃣ [确认切分] → 文档应切分，按标记执行切分
```

**形态② — 由 ort01 执行后刷新（迭代入口）**：

```
---

【切分确认】✅ 切分已执行（共 N 片），切分状态表="切分已执行"。

如需调整切分标记后重新切分，请先移动 `[切分建议]/[切分结束]` 标记位置，
然后保留下方选项并重新启动 ort01：

1️⃣ [重新切分] → 标记已调整，按新边界重新切分

如结果满意，删除此行直接启动下游步骤（ort02-annotate）。
```

| 确认区形态 | 写入者 | 保留选项 | AI 反应 |
|-----------|-------|---------|---------|
| 形态①（建议切分） | ort00 | `[确认切分]` | 自动更新状态="切分已确认"→ 执行切分 |
| 形态①（建议切分） | ort00 | `[确认不切分]` | 更新状态="不切分已确认"→ 提示启动 ort02 |
| 形态①'（建议不切分） | ort00 | `[确认不切分]` | 更新状态="不切分已确认"→ 提示启动 ort02 |
| 形态①'（建议不切分） | ort00 | `[确认切分]` | 自动更新状态="切分已确认"→ 执行切分 |
| 形态②（执行后刷新） | ort01 | `[重新切分]` | 入口扫描分流确认 last_size 变更后执行重切（保留"切分已执行"状态） |
| 形态②（执行后刷新） | ort01 | 删除选项（暗示满意） | 无操作，跳过该文件（结果满意，直接启动下游） |

> **注意**：人类无需直接编辑切分状态表。确认标记是文档中的人类操作记录，AI 在 ort01 入口分流中自动检测并同步状态表。形态②是关键新增——它消除了旧标记被误读的风险，将迭代变为显式操作。

### 4.4 启动方式

#### 人类启动指令

> 开始文本化的原始需求材料切分执行与切分优化（或 ort01-chunk-execute）

#### AI 动作序列

0. **扫描分流**：读取 OR 状态表中切分状态表的当前状态：
   - **状态="待确认切分方案"**：检测文本化文档末尾的确认区。读到 `[确认切分]` → 更新切分状态表为"切分已确认"，进入步骤 1；读到 `[确认不切分]` → 更新为"不切分已确认"，跳过后提示启动 ort02；无有效选项 → 跳过
   - **状态="切分已执行"**：对比 last_size。文件大小 = last_size → 跳过；文件大小 ≠ last_size → 检测确认区，读到 `[重新切分]` 保留 → 进入步骤 1（**切分状态表保持"切分已执行"不降级**）；无有效选项 → 跳过
   - **状态="不切分已确认"** → 提示启动 ort02
   - **其他状态** → 跳过
1. **扫描标记**：读取步骤 0 中已确认的文本化文档，提取文档中的 `[切分建议]` 和 `[切分结束]` 标记
2. **分流**：
   - 有切分标记且 OR 状态表无对应切分记录 → 首次切分
   - 有切分标记且标记已变更（last_size 已确认变更） → 重切
   - 有切分标记且标记未变 → 跳过（复用已有切分文件）
   - 标记为"不切分"→ 跳过切分流程
   - 无切分标记 → 报告异常
3. **首次切分**：阶段一（读取标记）→ 阶段二（执行切分）→ 写入（输出+注册+刷新确认区）
4. **重切**：阶段一（读取更新后标记）→ 阶段二（对比变更+执行切分）→ 写入（更新+注册+刷新确认区）
5. **输出行动清单**：按 §三.行动清单输出 格式输出
6. **人机交互**：向人类提供登记表路径和关注指引，询问是否需要打开查看

### 4.5 与 OR 状态表（含切分状态表）的字段联动

| 字段 | 操作 | 说明 |
|------|------|------|
| 文件 | 写入 | 切分文件名称（如 `01-市场调研报告-chunk-001.md`）；不切分场景无需写入 |
| parent_file | 写入 | 原始文件名称；不切分场景无需写入 |
| chunk_label | 写入 | 语义标签+范围；不切分场景无需写入 |
| 生命周期状态 | 写入（首次）→ 已登记；写入（重切·新片）→ 已登记；更新（重切·旧片）→ 已废弃；不切分场景→保持"已登记"不变 |
| 切分状态表·状态 | 更新 | 首次：`[确认切分]` → 切分已确认 → 切分已执行；`[确认不切分]` → 不切分已确认（终态）；迭代重切：保持"切分已执行"不变（不降级） |
| 切分状态表·操作指引 | 更新 | 切分执行后写入"切分已完成，进入切分各自的登记表流程"；不切分确认后写入"无需切分，启动 ort02 进入标注处理" |
| 切分状态表·last_size | 更新 | 每次切分/重切执行后更新为当前文本化文档的文件大小。入口扫描分流据此检测标记是否变更 |
| AI 建议 | 写入 | 切分完成 / 不切分已确认 |
| 人类决策 | 不操作 | 保留，由人类在确认后修改 |

## 五、切分文件模板与格式

### 5.1 切分文件模板

```markdown
---
chunk:
  source: {原始文件名}.md
  original_source: {原始文件名}.{ext}
  seq: {序号}
  total: {总片数}
  label: {语义标签}
  range: {原文范围}
---

（切分正文内容）
```

| 字段 | 说明 |
|------|------|
| source | 文本化文档文件名 |
| original_source | 原始文件文件名（只读档案） |
| seq | 本片序号（三位数字） |
| total | 本次切分总片数 |
| label | 语义标签，概括本片主题 |
| range | 原文中的范围描述（章节/段落/行号范围等） |

### 5.2 OR 状态表切分记录格式

OR 状态表在现有字段基础上扩展以下字段（仅切分文件的行使用）：

| 字段 | 示例 | 说明 |
|------|------|------|
| 文件 | 01-市场调研报告-chunk-001.md | 切分文件名称 |
| parent_file | 01-市场调研报告.pdf | 原始文件名称 |
| chunk_label | 市场部业务需求(§2.1~§2.5) | 语义标签+范围 |

非切分文件（普通原始材料）的 parent_file 和 chunk_label 字段为空。

## 七、线下审查指引

本指引供人类在 ort01 执行完成后审查切分产出，确保切分质量符合下游处理要求。

### 7.1 审查内容

| 审查项 | 说明 | 通过标准 |
|--------|------|---------|
| 边界合理性 | 每片是否为语义完整的独立段落 | 单片内容围绕单一主题，无生硬截断 |
| 覆盖完整性 | 全部标记范围均已输出为切分文件 | 无遗漏段落，无空切分文件 |
| 标签准确性 | YAML frontmatter 中的 label 与内容匹配 | 标签概括了该片核心主题 |
| 命名一致性 | 文件名遵循 `-chunk-{seq}` 格式 | 序号连续无跳号，与标记对应 |
| 非切分材料 | `[确认不切分]` 的材料是否确实无需切分 | 无需切分的材料可跳过 ort01 |

### 7.2 调整方式

审查发现问题时，打开文本化文档调整切分标记后重启本步：

| 问题 | 调整方式 |
|------|---------|
| 边界偏前/偏后 | 移动 `[切分建议]` 或 `[切分结束]` 标记位置 |
| 漏切了一段 | 插入一对新标记包围遗漏内容 |
| 某片语义不连贯 | 在该片中间插入切分标记拆分为两片 |
| 两片应合并 | 删除两片之间的边界标记 |
| 整份不应切分 | 删除所有切分标记，添加 `[切分建议: 不切分]` |

## 八、质量门禁

### 8.1 过程正确性

- [ ] 入口扫描分流已按切分状态表当前状态正确分流（待确认切分方案/切分已执行/不切分已确认）
- [ ] 首次进入时确认标记已正确检测（确认切分/确认不切分）
- [ ] 迭代重入时 last_size 变更检测已执行（文件大小比对）
- [ ] 重切确认标记已正确检测（仅 `[重新切分]` 有效）
- [ ] 状态不降级原则已遵守（迭代重切不降级"切分已执行"回前态）
- [ ] 切分状态表已同步更新（切分已确认/不切分已确认/切分已执行）
- [ ] 切分标记完整性已校验（成对、无重叠）
- [ ] 旧切分记录已标记为"已废弃"（仅重切场景）
- [ ] 确认区已刷新为形态②（含 `[重新切分]` 选项，仅切分执行场景）
- [ ] 不切分场景已跳过处理流程

### 8.2 产出验证

- [ ] 所有切分文件命名前缀与原始文件一致（`{原始文件名}-chunk-{seq}`）
- [ ] 每个切分文件包含完整的 YAML frontmatter（source/original_source/seq/total/label/range）
- [ ] 切分文件中已移除 `[切分建议]` / `[切分结束]` 标记行
- [ ] 各切分内容覆盖全部标记范围，无遗漏、无空文件
- [ ] 切分序号连续，无跳号
- [ ] OR 状态表中各切分行已正确注册（含 parent_file + chunk_label）
- [ ] 切分状态表状态已更新为"切分已执行"并写入操作指引
- [ ] 切分状态表 last_size 已更新为当前文本化文档的文件大小
- [ ] 文本化文档末尾的确认区已刷新为形态②（报告切分已执行 + 提供 `[重新切分]` 选项）
- [ ] 已废弃的旧切分不再被下游步骤拾取
