# 文本化的原始需求材料切分执行与迭代

**Skill ID**: ort01-chunk-execute · **Step ort01: Textualized Original Requirement Chunk Execution and Iteration**

> **概述**：读取文本化原始需求文档（含 AI 提议的切分标记和/或人类调整后的标记），执行语义切分，输出切分文件并注册 OR 状态表。本步可迭代——人类可在文本化文档中调整切分标记后重启本步，AI 将按更新后的标记重新切分。
>
> **输入 → 输出**：文本化原始需求文档（`{原始文件名}.md`，含切分标记）+ OR 状态表（含切分状态表）→ 切分文件（`{原始文件名}-sub-{seq}.md`）+ OR 状态表更新（注册各切分记录 + 切分状态表更新）
>
> **接口设计**：原始需求生命周期状态表（简称 OR 状态表）是预处理链中各顺序 Skill 间的共享接口。ort01 读取切分状态表中状态=切分已确认 的记录确定待切分材料，从文本化文档中读取切分标记执行切分。每个切分在 OR 状态表中注册为独立记录（含 parent_file 字段关联回原始文件，含 chunk_batch 标记执行批次），生命周期初始化为"已登记"。本步可多次执行——每次读取文本化文档中的当前切分标记重新切分，旧切分标记为"已废弃"。切分迭代后，文本化文档中追加切分迭代记录（含批次号和各片摘要）供人类追溯。
>
> **适用场景**：接续 ort00 输出的文本化原始需求文档（含切分标记）或人类自行编写的切分标记文档。切分文件产出后供下游 ort02-clarify-propose 进行首轮模糊标注。
>

## 三、操作流程

核心驱动逻辑：读取 OR 状态表中切分状态表的当前状态和文本化文档中的变更记录表，按状态分流——"待确认切分方案"时检测变更记录表（情形A/B）后执行切分；"切分已执行"时通过 last_size 比对检测标记是否变更，确认后执行重切。每轮执行后刷新变更记录表，使旧确认行不再可被误读，支持当前状态→确认→执行的显式迭代。

本步按"扫描分流 → 阶段一 → 阶段二 → 写入 → 行动清单 → 人机交互"的线性序列执行。扫描分流→阶段一→阶段二→写入为 AI 独立执行段；末端的[行动清单]→[人机交互]为人类交接点（提供切分确认指引和登记表查阅），不影响本轮 AI 执行结果。

操作流程概览：

```
[前置校验] 检查上游前置条件
  ├─ 10-source-materials/ 不存在 → 输出"请先运行 ort00-chunk-propose 初始化目录" → 结束
  ├─ 01-eos-or-lifecycle.md 不存在 → 输出"请先运行 ort00-chunk-propose 初始化 OR 状态表" → 结束
  ├─ 无文本化 .md 文档（不含 .pdf 等非文本格式） → 输出"请先运行 ort00-chunk-propose 完成文本化和切分建议" → 结束
  └─ 全部就绪 → [扫描分流] ↓

[扫描分流] 扫描文本化文档 + OR 状态表（切分状态表）
  │
  ├─ "待确认切分方案"（首次进入）
  │    └─ 读变更记录表
  │         ├─ 情形A（1行待确认剩余）→ 自动更新="已确认"→ [阶段一] ↓
  │         ├─ 情形B（追加"已确认"行）→ 校验后执行对应操作
  │         └─ 情形D（≥2行待确认/0行）→ 跳过（等待人类清理）
  │
  ├─ "切分已执行"（迭代重入）
  │    ├─ last_size 未变 → 跳过（标记未调整）
  │    └─ last_size 已变 → 读变更记录表
  │         ├─ 情形C（重切行）→ [阶段一] ↓（保留"切分已执行"状态）
  │         └─ 无"待确认"行 → 跳过（标记变了但未确认）
  │
  └─ 其他状态 → 跳过

[阶段一] 读取切分标记
    │
[阶段二] 按标记边界执行切分
    ↓

[写入]
  ├─ 产出：切分文件（10-source-materials/，{原始文件名}-sub-{seq}.md）
  │   · 常规切分→按标记边界输出多片；不切分→输出单片（total=1）
  ├─ 更新：OR 状态表（切分行注册+原始文档生命周期→已切分+切分状态表+last_size+chunk_batch）
  ├─ 追加：切分迭代记录到文本化文档（含批次号+各片摘要）
  ├─ 刷新：变更记录表为形态②（执行行→已完成+重切行[待确认]）
  └─ 追加：决策记录到文本化文档末尾（§六）
    ↓

[行动清单] 输出线下行动指南
    ↓

[人机交互] 询问人类是否打开登记表
  ├─ 回答"是"→ 输出登记表路径 + 建议关注行
  └─ 回答"否"→ 跳过

→ 本轮结束
```

### 入口：扫描分流

AI 在读取切分标记之前，先读取 OR 状态表中的切分状态表，按当前状态分流：

| 当前切分状态表状态 | AI 操作 | 说明 |
|-------------------|---------|------|
| **待确认切分方案** | 检测文档末尾的 `## 变更记录` 表：**情形A**（仅剩 1 行"待确认"）→ 将幸存行更新为"已确认"，按操作列值分流——"确认切分"→ 状态="切分已确认"→ 阶段一；"确认不切分"→ 状态="切分已执行"→ 直接写入。**情形B**（追加 1 行"已确认"）→ 校验后执行。**情形D**（≥2 行"待确认"或 0 行"待确认"）→ 跳过，等待人类清理 | 首次进入 |
| **切分已执行** | 对比 `last_size`：有变更 → 检测变更记录表。**情形C**（出现 1 行"重切"且状态=待确认）→ 进入阶段一执行重切（**状态保持"切分已执行"，不降级**）。**情形D**→ 跳过。无变更→跳过 | 迭代重入 |
| **切分已确认** | 直接进入阶段一（上一轮入口分流已完成变更记录表检测，无需重复检测） | 中断恢复 |
| **其他状态**（待分析/待重切等） | 跳过 | 非本步职责 |

> **last_size 变更检测**：通过比对文本化文档文件大小感知标记是否变更——人类调整标记位置后文件大小随之变化。未确认时不执行重切，防止文件保存中的临时变更触发意外操作。
>
> **状态不降级原则**：迭代重切不降级"切分已执行"回前态，确保状态单调递进——"已执行"之后不再回退。人类在文档中调整标记后重启本步即可，无需重新经过 ort00。
>
> **统一输出原则**：所有文档在 ort01 后均输出 `-sub-*.md` 文件（切分文档输出多片，不切分文档输出单片 total=1），ort02 统一处理全部切分文件，不再区分来源。

### 阶段一：读取切分标记

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 扫描文本化文档 | AI | 读取 `{原始文件名}.md`，扫描文档中的 `[切分建议]` 和 `[切分结束]` 标记 |
| 2 | 解析标记 | AI | 提取每对标记之间的内容范围。标记格式：`[切分建议: {序号} · {语义标签}]` ... `[切分结束: {序号}]` |
| 3 | 校验标记完整性 | AI | 检查所有标记是否成对、范围是否重叠、是否有未覆盖的文档区域。发现异常时在行动清单中标注 |

**校验结果对应处理**：

| 情况 | 处理 |
|------|------|
| 标记成对且边界清晰 | 校验通过，标记可执行（进入阶段二） |
| 标记不成对（有开始无结束/反过来） | 标注异常，提示人类修正 |
| 标记范围有重叠 | 按先出现的标记为准，提示重叠范围 |
| 文档中有未覆盖的段落 | 标注为"未分配内容"，由人类决定归属哪些片 |
| 无任何切分标记 | 标注异常，输出"未发现切分标记"提示，结束（返回 ort00 或人工处理） |

### 阶段二：执行切分

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 按标记边界分割内容 | AI | 将每对 `[切分建议]` ~ `[切分结束]` 之间的文本提取为一个独立切分 |
| 2 | 为每片附加元数据 | AI | 每片开头插入 YAML frontmatter，记录来源（source/original_source）、序号（seq/total）、语义标签（label）、范围（range），状态初始化为 `status: raw` |
| 3 | 移除标记行 | AI | 切出的文档内容中移除 `[切分建议]` 和 `[切分结束]` 标记行（元数据已在 frontmatter 中保留） |

**切分文件内容结构**（YAML frontmatter 模板见 §五·5.1，切分文件中已移除 `[切分建议]/[切分结束]` 标记行）。

**切分文件的目录与命名**：

与原始文件同目录，命名格式：`{原始文件名}-sub-{seq}.md`

```
├── 01-市场调研报告.pdf              ← 原始文件（只读）
├── 01-市场调研报告.md                ← 文本化文档（含切分标记）
├── 01-市场调研报告-sub-001.md      ← 切分输出
├── 01-市场调研报告-sub-002.md      ← 切分输出

```

### 写入：输出切分文件 + 生成执行批次 → 写入状态摘要 → 更新状态表 + 追加执行记录 + 刷新变更记录表 + 追加决策记录

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 输出切分文件 | AI | 将每片写入独立的 `{原始文件名}-sub-{seq}.md` 文件。常规切分→按标记边界分片输出；不切分场景→输出单片（整份文档，seq=001，total=1，label="不切分（整份文档）"） |
| 2 | 写入/更新状态摘要 | AI | 在切分文件顶部（文档属性后、正文前）写入状态摘要块（格式见 §五·5.2）。同时在文本化文档顶部覆盖更新状态摘要。切分完成后内容：阶段=切分完成，待处理=无，操作指引=运行 ort02-clarify-propose 进行首轮模糊标注 |
| 3 | 生成执行批次号 | AI | 生成本次执行的唯一批次号，格式 `YYYYMMDD-v{序号}`（序号=已有执行记录最大序号+1，从 1 起始）。本次执行的所有切分共享同一批次号 |
| 4 | 更新 OR 状态表 | AI | 为每个切分创建独立记录行，含 `chunk_batch`。首次切分→注册为"已登记"；重切→旧切分标记为"已废弃"，新切分注册为"已登记"（含新批次号）。**原始文档生命周期更新为"已切分"**（不切分场景同样处理，原始文档不再被下游步骤拾取） |
| 5 | 更新切分状态表 | AI | 将切分状态表中该文件的状态更新为"切分已执行"，写入操作指引，更新 last_size 和 chunk_batch |
| 6 | 追加切分迭代记录 | AI | 在文本化文档末尾（变更记录表之前）追加切分迭代记录块，含批次号、执行日期、切分类型、各片摘要。重切时在已有记录之前插入新记录（最新记录最上） |
| 7 | 刷新变更记录表 | AI | 将文本化文档末尾的变更记录表刷新为形态②——原确认行更新为"已完成"，追加执行记录行（"已完成"）和重切行（"待确认"）。使旧确认行不再可被误读。详见§4.3.3 形态② 示例 |
| 8 | 追加决策记录 | AI | 根据本次处理结果填充决策记录模板（§六），追加到文本化文档末尾（变更记录表之后），供历史追溯 |

OR 状态表记录格式定义见 §五·5.2（切分行含 parent_file / chunk_label / chunk_batch 等字段）。

切分迭代记录的格式定义见 §5.4 切分迭代记录格式。

## 四、AI 执行指南

### 4.1 切分边界处理

| 边界情况 | 处理方式 |
|---------|---------|
| 标记边界在段落中间 | 按标记边界切，被截断的段落完整归入起始片。在行动清单中标注"段落被边界截断" |
| 连续两片之间有空白行 | 空白行归入前一片 |
| 切分内容为空（标记之间无内容） | 不输出空文件，视为无效标记，在行动清单中报告 |
| 切分文件与已有文件重名（重切场景） | 覆盖旧文件。旧切分在 OR 状态表中标记为"已废弃" |

### 4.2 重切时的变更检测

AI 通过比对文本化文档的 **last_size** 与切分状态表中记录的上次执行时的文件大小来判断是否需要重切（由入口扫描分流触发，详见 §三·入口：扫描分流）。last_size 在每次切分/重切执行后更新：

| 条件 | 处理 |
|------|------|
| 文本化文档中的切分标记未变化 | 跳过，复用已有切分文件 |
| 文本化文档中的切分标记已变化 | 按新标记重新切分。新注册的切分文件覆盖旧文件，旧切分标记为"已废弃" |
| 新增了切分（原 3 片变 5 片） | 新增片注册为"已登记" |
| 减少了切分（原 5 片变 3 片） | 减少的片标记为"已废弃" |
| 切分序号重新排列 | 按新序号输出，旧序号标记为"已废弃" |

### 4.3 人类指令与标记

#### 4.3.1 人类对切分标记的调整方式

人类在文本化文档中可直接：

| 操作 | 方式 | AI 处理 |
|------|------|---------|
| 确认边界 | 保留 `[切分建议]` 标记不变 | 按标记执行 |
| 移动边界 | 将 `[切分建议]` 或 `[切分结束]` 标记移到其他位置 | 按新边界切分 |
| 新增切分 | 插入新的 `[切分建议: {新序号} · {标签}]` ... `[切分结束: {新序号}]` | 识别新标记，增加切分 |
| 合并切分 | 删除两片之间的边界标记 | 两片合并为一片 |
| 取消切分 | 删除所有切分标记，添加 `[切分建议: 不切分]` | 输出单一切分文件（整份为一片） |

#### 4.3.2 标记约定

切分标记使用与 ort00 相同的格式，人类可在此基础上调整：

```
[切分建议: 001 · 市场部业务需求]    ← 人类可移动此行位置
...内容...
[切分结束: 001]                      ← 人类可移动此行位置
```

#### 4.3.3 变更记录表的操作方式

本步使用统一变更记录表替代传统确认区。变更记录表有两种形态，分别对应首次进入时（由 ort00 写入）和执行后（由 ort01 刷新）。

**形态① — 由 ort00 写入（首次进入）：**

首次进入时，文本化文档末尾由 ort00 写入变更记录表，包含两行建议行（状态=待确认）。人类删除不选择的行，保留一行，然后启动本步。AI 检测到仅剩一行"待确认"（情形A）后自动执行。

切分建议下的变更记录表示例：

```
---

## 变更记录

| 日期 | 操作者 | 操作 | 说明 | 状态 |
|------|--------|------|------|------|
| {日期} | AI | 切分建议 | 确认切分 → 按当前切分标记执行切分（N片：{标签}） | 待确认 |
| {日期} | AI | 切分建议 | 确认不切分 → 文档不宜切分，输出单一切分文件 | 待确认 |

⚠️ **操作提示**：删除你不选择的行，保留下你想要的行。只保留一行后启动本步。
```

不切分建议下的变更记录表示例：

```
---

## 变更记录

| 日期 | 操作者 | 操作 | 说明 | 状态 |
|------|--------|------|------|------|
| {日期} | AI | 切分建议 | 确认不切分 → 无需切分，输出单一切分文件 | 待确认 |
| {日期} | AI | 切分建议 | 确认切分 → 文档应切分，按标记执行切分 | 待确认 |

⚠️ **操作提示**：删除你不选择的行，保留下你想要的行。只保留一行后启动本步。
```

**形态② — 由 ort01 执行后刷新（迭代入口）：**

本步执行完成后，将变更记录表刷新为执行记录 + 重切选项行（状态=待确认）。人类可编辑后重启本步触发重切。

```
---

## 变更记录

| 日期 | 操作者 | 操作 | 说明 | 状态 |
|------|--------|------|------|------|
| {日期} | AI | 切分建议 | 建议切分 {N} 片（{标签}） | 已完成 |
| {日期} | AI | 切分执行 | 已按切分标记执行切分，共 {N} 片，批次 {批次号} | 已完成 |
| ... | ... | ... | ... | ... |
| {日期} | AI | 重切 | 如已调整切分标记，保留此行使 AI 按新边界重新切分 | 待确认 |

⚠️ **操作提示**：如需重切，调整切分标记后保留"重切"行；如结果满意，删除"重切"行。
```

**入口检测规则**（AI 在入口分流中的判断逻辑）：

| 场景 | 变更记录表行状态 | AI 检测 | AI 操作 |
|------|----------------|--------|---------|
| 人类同意切分建议 | 仅剩 1 行"待确认"（结果行） | 情形A（删行） | 将此行更新为"已确认"→ 执行切分 |
| 人类否决切分建议 | 仅剩 1 行"待确认"（反建议行） | 情形A（删行） | 将此行更新为"已确认"→ 输出单一切分文件 |
| 人类追加确认行 | 出现 1 行"已确认" | 情形B（追加） | 校验后执行对应操作 |
| 人类追加重切行 | 出现 1 行"重新切分" | 情形C（重做） | last_size 变更→ 执行重切 |
| 人类满意切分结果 | 0 行"待确认" | 情形D（无法判定） | 跳过，文件就绪 |
| 无法判定 | ≥2 行"待确认" | 情形D（无法判定） | 跳过，等待人类清理 |

> **注意**：所有文档在 ort01 后均输出 `-sub-*.md` 切分文件（切分文档输出多片，不切分文档输出单片 total=1），ort02 统一处理全部切分文件，不再区分来源。检测规则由 AI 在入口分流中自动执行，无需人类手动编辑状态表。

### 4.4 启动方式

#### 人类启动指令

> 开始文本化的原始需求材料切分迭代（或 ort01-chunk-execute）

#### AI 动作序列

0. **前置校验**：检查上游前置条件。**逐级报错，首次即停**——只报告最早的未满足条件。
   - `../../80-pl4eos-2-eosdata/_inputs/10-source-materials/` 不存在 → 输出"请先运行 **ort00-chunk-propose** 初始化目录结构" → **结束**
   - `../../80-pl4eos-2-eosdata/_inputs/01-eos-or-lifecycle.md` 不存在 → 输出"请先运行 **ort00-chunk-propose** 初始化 OR 状态表" → **结束**
   - `10-source-materials/` 中无文本化 `.md` 文档（不含 .pdf 等非文本格式） → 输出"请先运行 **ort00-chunk-propose** 完成文本化和切分建议" → **结束**
   - 全部就绪 → 进入步骤 1
1. **扫描分流**：读取 OR 状态表中切分状态表的当前状态：
   - **状态="待确认切分方案"**：检测文本化文档末尾的变更记录表。**情形A**（1 行"待确认"剩余）→ 将该行状态更新为"已确认"，按操作列分流——"确认切分"→ 切分状态表="切分已确认"，进入步骤 2；"确认不切分"→ 切分状态表="切分已执行"，直接进入写入。**情形B**（追加"已确认"行）→ 校验后按操作列分流。**情形D**（≥2 行或 0 行待确认）→ 跳过
   - **状态="切分已执行"**：对比 last_size。文件大小 = last_size → 跳过；文件大小 ≠ last_size → 检测变更记录表。**情形C**（出现"重切"行且状态=待确认）→ 进入步骤 2（**切分状态表保持"切分已执行"不降级**）。**情形D**→ 跳过
   - **状态="切分已确认"** → 直接进入步骤 2（上一轮入口分流已完成变更记录表检测，无需重复，属中断恢复）
   - **其他状态** → 跳过
2. **阶段一：读取切分标记**：读取已确认的文本化文档，提取文档中的 `[切分建议]` 和 `[切分结束]` 标记
3. **执行分流**：按标记情况决定执行路径：
   - 有切分标记且 OR 状态表无对应切分记录 → 首次切分 → 进入步骤 4
   - 有切分标记且标记已变更（last_size 已确认变更） → 重切 → 进入步骤 4
   - 有切分标记且标记未变 → 跳过（复用已有切分文件）
   - 无切分标记 → 报告异常
4. **阶段二：执行切分**：按标记边界分割内容 → 为每片附加 YAML frontmatter → 移除标记行
5. **写入**：
   - 输出切分文件到 `10-source-materials/`（`{原始文件名}-sub-{seq}.md`）
   - 生成执行批次号（`YYYYMMDD-v{序号}`，本次执行所有切分共享）
   - 更新 OR 状态表：首次→各切分注册为"已登记"；重切→旧切分标记"已废弃"+新切分注册"已登记"
   - 更新切分状态表："切分已执行"+ 操作指引 + last_size + chunk_batch
   - 追加切分迭代记录到文本化文档（格式见 §5.3；重切时前插，最新最上）
   - 刷新变更记录表为形态②（更新执行行为"已完成"+追加重切行[待确认]）
   - 追加决策记录到文本化文档末尾（模板见 §六）
6. **输出行动清单**：按 §三.行动清单输出 格式输出（含孤儿文件提醒）
7. **人机交互**：向人类提供登记表路径和关注指引，询问是否需要打开查看

### 4.5 与 OR 状态表（含切分状态表）的字段联动

| 字段 | 操作 | 说明 |
|------|------|------|
| 文件 | 写入 | 切分文件名称（如 `01-市场调研报告-sub-001.md`）；不切分场景同样写入（单片 total=1） |
| parent_file | 写入 | 原始文件名称；不切分场景同样写入 |
| chunk_label | 写入 | 语义标签+范围；不切分场景写入"不切分（整份文档）" |
| 生命周期状态 | 写入（切分片）→ 已登记；更新（原始文档行）→ **已切分**（所有文档在 ort01 后原始行标记为"已切分"，不再被下游拾取）；更新（重切·旧片）→ 已废弃 |
| 切分状态表·状态 | 更新 | 常规切分（变更记录表·情形A→"确认切分"行幸存）：切分已确认 → 切分已执行；不切分（情形A→"确认不切分"行幸存）：直接→切分已执行（跳过阶段一/二，输出单片）；迭代重切：保持"切分已执行"不变（不降级） |
| 切分状态表·操作指引 | 更新 | 切分迭代后写入"切分已完成，进入切分各自的登记表流程"；不切分场景同样写入（输出单片，启动 ort02 进入标注处理） |
| 切分状态表·last_size | 更新 | 每次切分/重切执行后更新为当前文本化文档的文件大小。入口扫描分流据此检测标记是否变更 |
| 切分状态表·chunk_batch | 写入 | 每次切分/重切执行后更新为本次生成的批次号（格式 YYYYMMDD-v{序号}），本次执行的所有切分共享。序号=已有执行记录最大序号+1，从 1 起始 |
| AI 建议 | 写入 | 切分完成 / 整份文档未切分 |
| 人类决策 | 不操作 | 保留，由人类在确认后修改 |

## 五、切分文件模板与格式

### 5.1 切分文件模板

```markdown
---
chunk:
  source: {原始文件名}.md
  original_source: {原始文件名}.{ext}
  seq: {序号}
  total: {总片数}
  label: {语义标签}
  range: {原文范围}
status: raw
---

（切分正文内容）
```

| 字段 | 说明 |
|------|------|
| source | 文本化文档文件名 |
| original_source | 原始文件文件名（只读档案） |
| seq | 本片序号（三位数字） |
| total | 本次切分总片数 |
| label | 语义标签，概括本片主题 |
| range | 原文中的范围描述（章节/段落/行号范围等） |
| status | 处理状态，初始化为 `raw`；ort02 更新为 `annotated`，ort03 更新为 `clarified` |

### 5.2 状态摘要模板

状态摘要写在切分文件顶部（文档属性后、正文前），同步更新文本化文档顶部的状态摘要。AI 在写入阶段写入，每次完全覆盖旧摘要。切分完成后统一写入以下内容：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
文件状态
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段：切分完成
AI建议：—
待处理：无
操作指引：运行 ort02-clarify-propose 进行首轮模糊标注
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> **文本化文档同步更新**：同一轮执行中，文本化文档顶部的状态摘要也同步覆盖更新为上述内容（替换 ort00 写入的"切分建议"阶段状态）。

### 5.3 OR 状态表切分记录格式

OR 状态表在现有字段基础上扩展以下字段（仅切分文件的行使用）：

| 字段 | 示例 | 说明 |
|------|------|------|
| 文件 | 01-市场调研报告-sub-001.md | 切分文件名称 |

### 5.4 切分迭代记录格式

## 七、线下审查指引

本指引供人类在 ort01 执行完成后审查切分产出，确保切分质量符合下游处理要求。

### 7.1 审查内容

| 审查项 | 说明 | 通过标准 |
|--------|------|---------|
| 边界合理性 | 每片是否为语义完整的独立段落 | 单片内容围绕单一主题，无生硬截断 |
| 覆盖完整性 | 全部标记范围均已输出为切分文件 | 无遗漏段落，无空切分文件 |
| 标签准确性 | YAML frontmatter 中的 label 与内容匹配 | 标签概括了该片核心主题 |
| 命名一致性 | 文件名遵循 `-sub-{seq}` 格式 | 序号连续无跳号，与标记对应 |
| 不切分标记 | `[确认不切分]` 的材料是否确为整份文档无需细分 | 单一切分文件内容完整，label 标注合理 |

### 7.2 调整方式

审查发现问题时，打开文本化文档调整切分标记后重启本步：

> **标记调整后注意事项**：调整切分标记会触发 `last_size` 变更检测。请确保文本化文档末尾的变更记录表中保留"重切"行（形态②已自带），或将切分状态表手动改为"切分已确认"以走中断恢复路径。否则入口分流将因无有效重切行而跳过该文件。

| 问题 | 调整方式 |
|------|---------|
| 边界偏前/偏后 | 移动 `[切分建议]` 或 `[切分结束]` 标记位置 |
| 漏切了一段 | 插入一对新标记包围遗漏内容 |
| 某片语义不连贯 | 在该片中间插入切分标记拆分为两片 |
| 两片应合并 | 删除两片之间的边界标记 |
| 整份不应切分 | 删除所有切分标记，添加 `[切分建议: 不切分]` |

## 八、质量门禁

### 8.1 过程正确性

- [ ] 入口扫描分流已按切分状态表当前状态正确分流（§三·入口：扫描分流）
- [ ] 变更记录表检测已正确执行——情形A（1行待确认剩余→更新为已确认并分流）、情形B（追加行→校验）、情形C（重切行→执行重切）、情形D（跳过）（§三·入口分流表）
