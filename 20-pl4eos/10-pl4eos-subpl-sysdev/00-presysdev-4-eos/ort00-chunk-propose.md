# 原始需求切片提议

**Skill ID**: ort00-chunk-propose · **Step ort00: Original Requirement Chunk Proposal**

> **概述**：对原始需求材料执行文本化和语义结构分析，输出摘要、切片建议、切片方案和切片理由。本步是 AI 独立执行段，一次性运行完成。人类在输出的文本化原始需求中标注切片反馈后进入 ort01（切片执行）。
>
> **输入 → 输出**：原始材料文件（PDF/图片/邮件/纪要等）→ 文本化原始需求文档（含切片标记）+ OR 状态表（摘要/切片建议/切片方案/切片理由/行动清单）
>
> **接口设计**：原始需求生命周期状态表（简称 OR 状态表）是预处理链中各顺序 Skill 间的共享接口。ort00 将切片分析结果写入 OR 状态表的**切片方案区域**（扩展字段），同时同步更新**切片状态表**（轻量状态，供 AI 再进入时快速分流）。人类在输出的文本化原始需求文档中直接编辑切片标记作为反馈。下游 ort01 读取切片状态表中的状态和文本化文档中的标记后执行切片。本步可对同一原始文档多次执行——每次重新分析并更新切片方案。
>
> **适用场景**：适用于体量较大、超过 AI 单次处理能力或语义边界不清晰的原始需求材料。体量适中的文件经本步分析后也可能输出"无需切片"的建议。
>
> **版本**：v0.1 | **修订**：2026-06-06

## 三、操作流程

AI 每次启动执行一轮完整的单次处理——初始化目录与 OR 状态表（如首次运行）、扫描输入目录、文本化、语义结构分析、输出切片方案、更新切片方案区域与切片状态表，然后结束。本步为一次性执行，无迭代、无人工介入。

操作流程概览：

```
[初始化] 检查目录与 OR 状态表
  ├─ 10-source-materials/ 不存在 → 新建目录
  ├─ 01-eos-or-lifecycle.md 不存在 → 新建（含登记表/切片状态表/切片方案区域/生命周期状态总览）
  ├─ 任一新建 → 目录为空 → 退出（提示人类放入材料后重新启动）
  └─ 就绪 → 进入扫描 ↓

[扫描] 扫描源文件目录 + 切片状态表
  ├─ 无 OR 状态表 记录的新原始文件 → 自动登记 + 切片状态表写入"待分析"
  ├─ 切片状态=待分析 → [阶段一] ↓
  ├─ 切片状态=待重切 → [阶段二~三]（重新分析，跳过文本化）
  └─ 其它状态（待确认/已确认/已执行）→ 跳过

[阶段一] 文本化
  将非文本格式原始材料转换为 .md 文档
  文本格式（.md/.txt）直接从原文复制
  原始文件保持只读不动
    ↓

[阶段二] 语义结构分析
  分析文档结构（章节/表格分组/业务域边界）
  识别自然语义段落和主题转换点
    ↓

[阶段三] 切片方案生成 + 优化建议
  按语义准则输出切片建议（切/不切）
  如需切片：输出切片方案（每片范围、语义标签、切片理由）
  如不切片：输出理由，标记为"无需切片"
  输出文档质量优化建议（结构完整性/模糊项/缺失信息）
    ↓

[写入] 输出文本化文档（含切片标记）+ 更新 OR 状态表
  文本化文档行内嵌入 [切片建议] / [切片结束] 标记
  切片方案区域：写入摘要、切片建议、切片方案、切片理由
  切片状态表：同步更新状态为"待确认切片方案"+ 按切片建议写入操作指引
  登记表：更新内容摘要（文档概要）+ AI 建议（切片建议+理由）
    ↓

[行动清单] 输出行动清单 + 优化建议
  按材料状态分组列出人类需执行的操作项
  附带本轮优化建议摘要
    ↓

[人机交互] 询问人类是否打开登记表
  ├─ 回答"是"→ 输出登记表路径 + 建议关注行
  └─ 回答"否"→ 结束本轮
```

### 阶段一：文本化

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 读取原始文件 | AI | 读取已在扫描分流阶段确定的原始文件。非文本格式（PDF/图片/扫描件/录音转写等）执行操作2（文本化转换）；文本格式（.md/.txt）直接复制到工作副本 |
| 2 | 文本化转换 | AI | 将非文本格式原始材料转换为 .md 文档。转换规则：保留原文结构（章节/段落/列表）、原文表述不做任何修改或摘要、无法识别的内容标注占位符、保留原始文件名作为元数据 |
| 3 | 写入工作副本 | AI | 文本化后（或复制后）的 .md 文档写入与源文件相同的 `10-source-materials/` 目录，文件命名规则：`{原始文件名}.md`。当原始文件已是 `.md` 格式时，直接在原文中嵌入切片标记（不创建同名副本），原始文件保持只读状态 |

> **文本化原则**：AI 在此阶段是"转录者"而非"理解者"。文本化只改变载体格式，不改变内容。任何超出格式转换的推断或补充均不允许。

### 阶段二：语义结构分析

AI 对文本化文档进行结构分析，识别语义边界：

| # | 操作 | 说明 |
|---|------|------|
| 1 | 文档结构识别 | 识别文档的层次结构（章节/标题/子标题层级）。有目录结构的按结构展开；无显式结构的按段落语义聚类 |
| 2 | 语义段落识别 | 识别自然语义段落的起止边界。判断依据：主题转换点（如业务域切换、角色切换）、章节边界、段落主题一致性 |
| 3 | 体量评估 | 评估每段的篇幅，判断是否超过正常单次处理范围 |
| 4 | 关联性分析 | 分析各段之间的语义依赖关系——是否存在跨段的概念定义引用 |

> **结构识别优先**：有章节/目录结构的文档优先按结构分析；表格类文档按行分组维度（如部门/业务域）分析；连续长文本按语义段落聚类分析。

### 阶段三：切片方案生成

基于语义结构分析的结果生成切片方案：

**判断规则**：

| 条件 | 结论 | 输出 |
|------|------|------|
| 文档总篇幅适中且语义单一 | **不切片** | 切片建议=不切片，理由=篇幅适中/语义单一 |
| 文档篇幅大但语义单一（如长篇幅的制度规范） | **不切片** | 切片建议=不切片，理由=语义不可分，提示下游注意体量 |
| 文档篇幅大且含多个语义单元 | **切片** | 切片建议=切片，输出切片方案 |
| 文档篇幅小但含多个语义独立的业务域 | **切片** | 切片建议=切片，输出切片方案 |

**切片方案格式**（写入 OR 状态表切片方案区域）：

| 片序号 | 范围 | 语义标签 | 预估行数 | 切片理由 |
|--------|------|---------|---------|---------|
| 001 | §2.1~§2.5 | 市场部业务需求 | 120 | §2.1~§2.5 同属市场部业务域 |
| 002 | §3.1~§3.4 | 研发部开发流程 | 85 | §3.1~§3.4 同属研发部业务域 |
| 003 | §4.1~§4.2 | 售后部工单管理 | 60 | §4.1~§4.2 同属售后部业务域 |

**切片准则**（语义驱动）：

| 准则 | 说明 |
|------|------|
| **高内聚** | 每片内容围绕单一语义主题 |
| **相对完整** | 每片内容可在不阅读其他片的前提下被理解和澄清 |
| **自然边界优先** | 沿文档已有结构边界（章节/表格分组/业务域）切片，不做人为等分 |

> **不追加机械约束**：不要求各切片体量可比、不设最小粒度门禁、切片后不合并。

**切片标记嵌入**：

AI 在输出的文本化文档中行内嵌入切片标记，供人类审阅和调整：

```markdown
# 01-市场调研报告

## 一、市场部需求

[切片建议: 001 · 市场部业务需求]
从 2.1 线索管理...
...
到 2.5 报表
[切片结束: 001]

## 二、研发部需求

[切片建议: 002 · 研发部开发流程]
...内容...
[切片结束: 002]
```

标记格式：

| 标记 | 格式 | 说明 |
|------|------|------|
| 切片开始 | `[切片建议: {序号} · {语义标签}]` | 标记每片的起始位置 |
| 切片结束 | `[切片结束: {序号}]` | 标记每片的终止位置 |
| 不切片 | `[切片建议: 不切片]` | 整份文档不做切片 |
| 全局前置片 | `[切片建议: 000 · 全局约束]` | 文档开头概述/全局约束独立成片 |

### 写入：输出文本化文档 + 更新 OR 状态表

**输出文本化文档**：含切片标记的 `.md` 文件写入 `10-source-materials/` 目录。

**更新 OR 状态表**：
- **切片方案区域**：写入切片分析结果（摘要/切片建议/切片方案）
- **切片状态表**：同步更新为"待确认切片方案"，按切片建议写入操作指引

**优化建议**：AI 在对文本化文档做结构分析的同时，输出轻量的内容质量建议，作为人类查阅文本化文档时的参考。这些建议随行动清单直接输出（不写入 登记表）：

| 建议类型 | 说明 | 示例 |
|---------|------|------|
| 结构完整性 | 文档是否缺章少节、是否有明显断点 | "§2.3 内容明显截断，疑似缺页" |
| 模糊项提示 | 非量化表述、主体缺失等 | "多处出现'高效''实时'等非量化词，下游澄清时需关注" |
| 缺失信息 | 缺版本号/日期/提出方等元信息 | "文档顶部缺少版本号和日期" |
| 体量提示 | 篇幅较大但语义不可分时提示 | "全文 800 行且语义紧密，建议下游预留充足处理资源" |

## 四、AI 执行指南

### 4.1 文本化注意事项

同通用系统设计规范 §7.3 阶段一(a) 及原 ort02-annotate 阶段一规范。以下为常见场景：

| 场景 | 处理方式 |
|------|---------|
| PDF 中的表格 | 转为 Markdown 表格，保留行列结构和数值 |
| 图片/扫描件 | 识别文字内容，标注"[AI 识别，待人工确认]" |
| 录音转写 | 标注发言人和时间戳，无法确认的内容标注"[听写不清]" |
| 手写内容 | 可辨识的写入文档，标注"[AI 推测]"，不可辨识的标注"[原文模糊]" |
| 多种语言混写 | 保留原文种，必要时标注"[中文/英文/...]" |
| 完全无法识别的图片 | 写入占位符 `[图片：无法识别文字内容]` |
| 文档明显不完整 | 在文档开头标注 `[文档疑似不完整：缺页/截断]` |

### 4.2 结构分析要点

| 场景 | 分析要点 |
|------|---------|
| 有章节结构的文档 | 识别章节层级，以自然章节为切片候选边界；检查各节是否语义独立 |
| 表格类文档 | 按业务维度分组（部门/产品线/区域等），以分组为切片候选边界 |
| 无结构连续文本 | 按段落主题聚类，标记主题转换点，以主题段群为切片候选边界 |
| 混合型文档 | 优先按章节结构切片，表格部分按行分组归入所属章节 |

### 4.3 切片标记在文本化文档中的使用规范

#### 4.3.1 标记格式

切片的行内标记：

```
[切片建议: {序号} · {语义标签}]
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `序号` | 是 | 三位数字（001/002/...）。全局约束类用 000 |
| `语义标签` | 是 | 对该片内容的主题概括（如"市场部业务需求"） |

对应的结束标记：

```
[切片结束: {序号}]
```

不分片的文档在文档开头标注：

```
[切片建议: 不切片 · 文档体量适中或语义单一，无需切片]
```

#### 4.3.2 切片的语义准则

切片时遵循以下准则（同 §三·阶段三）：

1. **高内聚**：每片内容围绕单一语义主题
2. **相对完整**：每片内容可在不阅读其他片的前提下被理解和澄清
3. **自然边界优先**：沿文档已有结构边界切片，不做人为等分

### 4.4 启动方式

本步为一次性执行，无需断点续接。每次启动处理所有未分析的原始材料后正常结束。

#### 人类启动指令

> 开始原始需求切片提议（或 ort00-chunk-propose）

#### AI 动作序列

0. **初始化**：检查 `../../80-pl4eos-2-eosdata/_inputs/` 目录结构。以下两项均不存在时依次新建，一次性完成：
   - `10-source-materials/` 不存在 → 新建目录
   - `01-eos-or-lifecycle.md` 不存在 → 新建（含标准表头 + 登记表表格 + 切片状态表 + 切片方案区域 + 字段说明 + 生命周期状态总览）
   - 任一新建 → 输出"目录已就绪，请将原始材料放入 `10-source-materials/` 后重新启动本步" → **结束**
   - 两者均就绪 → 继续执行步骤 1
1. **扫描**：扫描 `10-source-materials/` 中的文件（原始文件 + 已有文本化 `.md`），读取 OR 状态表和切片状态表。对 OR 状态表中无对应记录的新原始文件，自动创建数据行并初始化：生命周期状态=「已登记」，AI 建议=（空），人类决策=（空）；同时在切片状态表中写入记录，切片状态=「待分析」
2. **分流**：读取切片状态表，筛选需处理的材料：
   - 切片状态=待分析 → 进入阶段一
   - 切片状态=待重切 → 重新分析（阶段二~三），更新切片方案
   - 其他状态（待确认切片方案/不切片已确认/切片已确认/切片已执行）→ 跳过
3. **阶段一执行**：文本化 → 写入工作副本
4. **阶段二执行**：语义结构分析
5. **阶段三执行**：生成切片方案 → 在文本化文档中嵌入切片标记 → 生成优化建议（文档质量分析）→ 更新 OR 状态表（写入切片方案区域 + 登记表主表 内容摘要+AI 建议）→ 同步更新切片状态表（状态=待确认切片方案 + 按切片建议写入操作指引）
6. **输出行动清单**：按 §三.行动清单输出 格式输出行动清单，按材料状态分组列出人类需执行的操作项，附带优化建议摘要

> **执行节奏**：本步一次性完成所有待处理材料的文本化+结构分析+切片方案生成。所有处理完成后即结束。若因初始化时目录为空提前退出，由人类放入材料后重新启动本步。

### 4.5 重进入说明

本步可在以下场景重进入：

- **新增原始材料**：新放入的文件在 OR 状态表中无记录，AI 自动登记并在切片状态表中写入"待分析"，进入处理流程
- **已有记录的重新分析**：人类修改了文本化文档后重进入，AI 检测到变更后将切片状态表更新为"待重切"，进入重新分析（阶段二~三）

重进入不影响已存在的切片方案和下游已处理的结果。AI 通过切片状态表快速判断需处理的材料，无需读取切片方案区域。
