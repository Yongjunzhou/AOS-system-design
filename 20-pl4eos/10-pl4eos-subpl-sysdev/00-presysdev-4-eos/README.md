# ORT 预处理链 · 共享运行模式

**目录导航**：本文档描述 `eos-ort01-chunk` / `eos-ort02-clarify` / `eos-ort03-norm` 三份 Skill 共用的四步循环运行模式，供链级概览查阅。每份 Skill 文档自包含，不强制依赖本文档。

---

## 一、ORT 预处理链概览

```
原始材料 (PDF/图片/邮件/纪要)
    │
    ▼
eos-ort00-textualize ── 纯格式转换（非文本 → Markdown）
    │                      产出：文本化 .md 文档
    ▼
eos-ort01-chunk ──────── 语义切分
    │                      产出：-sub-{seq}.md 切分文件（status=raw）
    ▼
eos-ort02-clarify ────── 需求澄清（人类-in-the-loop）
    │                      产出：已澄清切分文件（status=clarified）
    ▼
eos-ort03-norm ───────── 需求规范化
    │                      产出：01-eos-original-requirements.md（OR 基线）
    ▼
t01 (OR→SR 分解分配)
```

## 二、共享四步循环

每步 Skill 启动时，AI 按以下四步循环执行：

```
┌─────────────────────────────────────────────────┐
│  Step 1 · 前置校验                               │
│    ├─ 检查上游前置条件（文件存在、状态文档就绪）   │
│    └─ 不通过 → 提示人类 → 结束                    │
├─────────────────────────────────────────────────┤
│  Step 2 · 新增处理（发现 + 处理 + 记录）          │
│    ├─ 扫描 EOS系统设计状态文档 对应子表           │
│    │   ├─ 人类决策=已确认 → 直接生成方案          │
│    │   │   （语义分析 → 写入方案块+反馈区）        │
│    │   │   [记录]更新权威源表→[同步]派生管理区    │
│    │   ├─ 人类决策=已终止 → [记录]废弃处理        │
│    │   └─ 人类决策=空 → 记入选材提醒              │
│    └─ 全部处理完毕 → 进入 Step 3                  │
├─────────────────────────────────────────────────┤
│  Step 3 · 反馈处理（发现 + 处理 + 记录）          │
│    ├─ 扫描处理结果文档正文区                      │
│    │   ├─ 有方案块 + 反馈文本 → 直接执行           │
│    │   │   （同意执行→输出 / 调整→重生成）         │
│    │   │   每分支末尾:[记录]更新权威源表+追加决策   │
│    │   │   记录 → [同步]派生管理区                 │
│    │   ├─ 有方案块 + 无反馈 → 跳过（等待人类）     │
│    │   └─ 无方案块 → 跳过（异常）                 │
│    └─ 全部处理完毕 → 进入 Step 4                  │
├─────────────────────────────────────────────────┤
│  Step 4 · 输出交互                               │
│    ├─ 输出行动清单（按状态分组）                   │
│    └─ 人机交互：询问是否打开状态文档               │
└─────────────────────────────────────────────────┘
```

## 三、两个交互点

| 交互点 | 位置 | 人类操作 | AI 操作 |
|--------|------|---------|--------|
| **选材**（确定哪些文件进入本步处理） | EOS系统设计状态文档 → 对应子表的 `人类决策` 字段 | 设为 `已确认` 或 `已终止` | Step 2 新增处理时读取并直接生成方案 |
| **审阅**（对 AI 方案给出反馈意见） | 处理结果文档正文区的反馈区/标注 | 输入 `[同意执行]` / `[修改建议]` / `[✓已澄清: ...]` 等 | Step 3 反馈处理时检测并直接执行 |

---

## 四、同步契约通用规则

```
先更新 EOS系统设计状态文档 对应子表（权威源）
  → 再推导写入各处理结果文档头部的派生状态表
若检测到不一致 → 以权威源为准重建派生表
```

## 五、各步对照表

| 方面 | ort01-chunk | ort02-clarify | ort03-norm |
|------|------------|---------------|------------|
| 读取的状态表 | 表A（选材+方案追踪） | 表B（选材入澄清） | 表B（选材入规范化）+ 表C（OR条目选材入t01） |
| 权威源表 | 表A（文档级）+ 表B（切分文件级） | 表B | 表B + 表C |
| 处理对象文件 | 文本化 `.md` 文档 | 切分文件 `-sub-*.md` | 已澄清切分文件（status=clarified） |
| 输出文件 | `-sub-{seq}.md` 切分文件 | 原地更新同文件 | `01-eos-original-requirements.md` |
| 方案/标注载体 | `╔═ 切分方案 ═╗` 方案块 | `[⨯]` / `[✓已澄清]` 行内标注 | `╔═ 分析建议块 ═╗` 方案块 |
| 反馈格式 | `[同意执行]` / `[调整]` / `[不切分]` / `[废弃]` | `[✓已澄清: ...]` 注释 | `[同意执行]` / `[修改建议]` / `[废弃]` |
| 分支 1（无方案） | 语义结构分析 → 切分方案块 | 全量模糊标注（5 子类型） | 四单一拆分 → 五项检查 → 冲突检测 |
| 分支 2（执行） | 按行号范围分割出 sub 文件 | 按注释生成新正文 → 三项条件评估 | 统一编号 OR-XXX → 写入 表C |
| 分支 3（重生成） | 调边界 → 归档旧版+反馈 | 归档旧版+标注 → 新版+再评估 | 归档到历史区 → 调整 → 重检 |
