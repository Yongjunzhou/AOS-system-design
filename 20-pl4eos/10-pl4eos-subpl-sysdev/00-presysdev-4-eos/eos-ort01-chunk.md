# 原始需求语义切分

**Skill ID**: eos-ort01-chunk · **Step ort01-chunk: Semantic Chunking**

> **概述**：对已文本化的原始需求文档执行语义结构分析和切分。每次运行时检测正文区状态——无切分方案块时生成方案块并等待反馈；有方案块且人类已输入反馈时按反馈执行切分或重新生成方案。本步可反复进入，人类在反馈区输入意见，再次运行本步时 AI 自动检测反馈并迭代。
>
> **输入 → 输出**：文本化 `.textualized.md` 文档（正文区：纯原文）+ EOS系统设计状态文档 → 切分文件（`.chunk-{seq}.md`）+ 文本化文档正文区（切分方案块 / 清理后纯原文）+ 状态文档更新（表B — OR（原始需求）切分处理建议及进展 + 表A — OR（原始需求）原料处理建议及进展）
>
> **接口设计**：本步接口载体有两个——文本化文档正文区的**方案块**（写入=待确认，清理=已执行）和状态文档的**表A/表B**（文档级+切分文件级进度追踪）。入口靠检测方案块存在性分流——无块生成，有块读反馈执行。无须独立变更记录表或行内标记。上游 eos-ort00-textualize 提供纯原文正文区和表A，下游输出切分文件和表B 供 eos-ort02-clarify 使用。
>
> **版本**：v0.2 | **修订**：2026-06-08

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：`../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 目录中的文本化 `.textualized.md` 文档（正文区：纯原文）+ `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/01-eos-sysdev-status.md`
- **产出**：切分文件（每片一个 `{原始文件名}.chunk-{seq}.md`）+ 文本化文档正文区更新（方案块/清理后纯原文）+ EOS系统设计状态文档更新（状态文档 → 表B — OR切分处理建议及进展 + 状态文档 → 表A — OR原料处理建议及进展）

### 与四条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **需求相对** | 本步不涉及需求/方案角色切换，语义切分仍属原始需求预处理阶段，无需求相对性可言，不适用 |
| **详概分离** | 本步仅执行语义结构分析和切分边界识别，内容理解仅限于语义聚类和段落边界定位所需的最低程度。不做需求内容的评价、分解或设计决策 |
| **文档结对** | 本步不产生文档结对关系。本步产出的切分文件是 eos-ort02-clarify 的输入 |
| **基于资产** | 切分文件是原始文档的派生资产，通过文件名关联保持追溯。状态文档 → 表A/表B 中通过**源文件**和**文本化文档**字段维护切分与原始文件的关系 |

### 三区结构说明

本步操作的文本化文档（`{原始文件名}.textualized.md`，位于 `10-raw-files/`）的三区：

| 区域 | 内容 | 本步操作 |
|------|------|---------|
| **管理区** | 状态摘要、切分迭代记录、决策记录 | 写入/更新状态摘要；追加迭代记录和决策记录 |
| **正文区** | 原文 + 切分方案块 + 反馈提示区（临时） | 生成阶段：写入方案块+反馈区；执行阶段：清理方案块+反馈区，正文区回归纯原文 |
| **历史区** | 旧版方案块 + 人类反馈原文（重新生成时归档） | 重新生成分支：归档旧版+反馈 |

## 三、操作流程

### Step 布局总览

| Step | 名称 | 职责 | 执行模式 | 产出 | 写资产 |
|------|------|------|---------|------|-------|
| Start | 前置校验 | 检查上游前置条件就绪 | 统一 | 就绪/结束 | — |
| 1 | 新增处理 | 语义结构分析 → 生成切分方案块 | 统一 | 方案块 | 表A |
| 2 | 反馈处理 | 读反馈 → 执行切分/不切分/调整/废弃 | 统一 | 切分文件 | 表B + 切分文件 |
| End | 输出交互 | 行动清单 + 人机交互 | 人类 | 行动清单 | — |

本步遵循 ORT 预处理链四步循环（详见 README §二）：

| Step | 本步对应 |
|------|---------|
| **Step Start · 前置校验** | 检查上游前置条件（文本化文档、状态文档就绪） |
| **Step 1 · 新增处理** | 读 状态文档 → 表A — OR原料处理建议及进展，筛出新增文档（人类决策=同意切分 + 生命周期=已文本化），**直接执行语义结构分析并生成切分方案块**写入正文区，记录到状态文档→推导写入管理区 |
| **Step 2 · 反馈处理** | 扫描状态文档 → 表A — OR原料处理建议及进展中生命周期=切分方案待确认的文档，检测正文区方案块→读反馈文本**直接执行**切分/不切分确认/重新生成/废弃，记录到状态文档→推导写入管理区 |
| **Step End · 输出交互** | 输出行动清单 + 人机交互 |

核心驱动逻辑：每次运行时先处理**新增**（表A 中人类决策=同意切分 + 生命周期=已文本化的文档），直接生成方案块写入正文区，更新状态文档并追加记录到管理区；再处理**反馈**（表A 中生命周期=切分方案待确认且已有方案块的文档），读取正文区反馈文本按内容执行切分或不切分确认或重新生成，更新状态文档并追加记录到管理区。

> **Step 1"新增处理"说明**：本步的"新增"指**人类从 ort00 文本化文档中选入本批、待 ort01 处理的 OR 原料**。其**入口硬条件**是两字段联合——`生命周期=已文本化`（已完成上游文本化）且 `人类决策=同意切分`（人类从候选中选定本批）。数据源为**表A（原料级）**，而非表B（切分文件级）。其中 `AI建议=可以切分` 仅作为面向人类的候选提示，**不参与 Step 入口硬判断**。`已登记` 仅是 ort00 内部中间状态，不构成 ort01 的入口条件。

### 3.1 生成方案（正文区无方案块时）

AI 对文本化文档执行语义结构分析，生成切分方案块写入正文区（紧接原文之后），末尾附反馈提示区。

**语义结构分析**：

| # | 操作 | 说明 |
|---|------|------|
| 1 | 文档结构识别 | 识别文档的层次结构（章节/标题/子标题层级）。有目录结构的按结构展开；无显式结构的按段落语义聚类 |
| 2 | 语义段落识别 | 识别自然语义段落的起止边界。判断依据：主题转换点、章节边界、段落主题一致性 |
| 3 | 体量评估 | 评估每段的篇幅，判断是否超过正常单次处理范围 |
| 4 | 关联性分析 | 分析各段之间的语义依赖关系——是否存在跨段的概念定义引用 |

**切分判断规则**：

| 条件 | 结论 | 输出 |
|------|------|------|
| 文档总篇幅适中且语义单一 | **不切分** | 切分建议=不切分，理由=篇幅适中/语义单一 |
| 文档篇幅大但语义单一 | **不切分** | 切分建议=不切分，理由=语义不可分，提示下游注意体量 |
| 文档篇幅大且含多个语义单元 | **切分** | 切分建议=切分，输出切分方案 |
| 文档篇幅小但含多个语义独立的业务域 | **切分** | 切分建议=切分，输出切分方案 |

**切分方案块格式**（写入正文区，紧接原文之后）：

```markdown
╔═ 切分方案 ═══════════════════════╗
║                                    ║
║ 建议：切分为 3 片                    ║
║                                    ║
║ 001 · 市场部业务需求                 ║
║   行号：L120~L340                   ║
║   对应：§2.1~§2.5                  ║
║   理由：同属市场部业务域，语义独立     ║
║                                    ║
║ 002 · 研发部开发流程                 ║
║   行号：L341~L480                   ║
║   对应：§3.1~§3.4                  ║
║   理由：同属研发部业务域，语义独立     ║
╚═════════════════════════════════════╝
```

不切分场景：

```markdown
╔═ 切分方案 ═══════════════════════╗
║                                    ║
║ 建议：不切分                        ║
║ 理由：文档语义单一，无需切分         ║
╚═════════════════════════════════════╝
```

**反馈提示区**（紧接方案块之后）：

```
──────────────────────────────────────────
📋 请在此处输入反馈意见：

  [同意执行] — 按当前方案执行切分
  [调整]    — 描述需要调整的内容（如"002 范围扩到 L500"）
  [不切分]   — 不应切分，输出单片
  [废弃]     — 该材料应废弃

您的反馈：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**写入状态摘要**（AI 直接写入 OR切分文档状态表到文本化文档管理区；同步更新 状态文档 → 表A — OR原料处理建议及进展，见 §3.4）：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 待确认切分方案 | 审核切分方案 | 审阅切分方案块后输入反馈 | 打开文档审阅方案块→在反馈区输入意见 |
```

### 3.2 执行/确认切分方案（反馈=同意执行或不切分时）

本分支处理两种场景：**同意执行**（按方案块行号范围切分为多片）和**不切分**（输出单片 total=1）。不切分场景跳过行号校验和分割步骤（2-3），直接输出单片。

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 读取切分方案块 | AI | 从正文区读取语义标签；不切分场景仅读取总体理由 |
| 2 | 校验方案完整性 | AI | 仅切分场景：检查行号范围无重叠、无遗漏覆盖区域、语义标签完整 |
| 3 | 按行号范围分割 | AI | 仅切分场景：根据方案块中的行号范围，将原文按行分割为独立切分 |
| 4 | 附加 YAML frontmatter | AI | 每片插入 YAML frontmatter（source/original_source/seq/total/label/range），status=raw |
| 5 | 输出切分文件 | AI | 写入 `{原始文件名}.chunk-{seq}.md`；不切分场景输出单片（total=1） |
| 6 | 清理正文区方案块 | AI | 删除切分方案块 + 反馈区，正文区回归纯原文 |
| 7 | 更新 状态文档（权威源）→ 推导写入 OR切分文档状态表 | AI | 先更新 状态文档（状态文档 → 表B — OR切分处理建议及进展 / 状态文档 → 表A — OR原料处理建议及进展），再据此推导写入 OR切分文档状态表（切分文件顶部 + 文本化文档顶部同步更新） |

**切分文件命名**：

```
10-raw-files/
├── 01-市场调研报告.pdf              ← 原始文件（只读）
├── 01-市场调研报告.textualized.md      ← 文本化文档
├── 01-市场调研报告.chunk-001.md        ← 切分输出
├── 01-市场调研报告.chunk-002.md        ← 切分输出
└── 01-市场调研报告.chunk-003.md        ← 切分输出
```

**切分文件 YAML frontmatter**：

```markdown
---
chunk:
  source: {原始文件名}.textualized.md
  original_source: {原始文件名}.{ext}
  seq: {序号}
  total: {总片数}
  label: {语义标签}
  range: {原文范围}
status: raw
---
（切分正文内容）
```

**执行后状态摘要**（同步更新文本化文档 + 各切分文件的 OR切分文档状态表）：

切分已执行场景：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 切分已执行 | 可以澄清 | 无 | 运行 eos-ort02-clarify 进行澄清标注 |
```

不切分已确认场景：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 不切分已确认 | 可以澄清 | 无 | 运行 eos-ort02-clarify 进行澄清标注 |
```

### 3.3 重新生成方案（反馈=调整意图时）

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 解析修改建议 | AI | 读取人类反馈文本中的具体要求（如"002 范围扩到 L500""新增一片"） |
| 2 | 调整方案 | AI | 按反馈调整方案块中的边界、标签、片数 |

## 四、AI 执行指南

### 4.1 正文区检测规则

AI 在入口分流中检测正文区状态：

| 检测结果 | 判定 |
|---------|------|
| 正文区以 `╔═ 切分方案` 开头的内容块存在 | 有方案块 → 读取末尾反馈文本 |
| 正文区无 `╔═ 切分方案` 开头的内容块 | 无方案块 → 进入生成方案路径 |
| 方案块存在但无反馈文本（反馈区为空） | 跳过，等待人类操作 |

> **检测范围**：仅检测正文区（管理区之后的内容）。历史区中的历史版本块不影响检测——正文区中当前是否有方案块是关键。

### 4.2 人类决策语义等价映射

AI 在 Step 1 读取 人类决策 字段时，除精确匹配标准值外，按以下映射理解人类的自然表达：

| 标准值 | 等价表达 |
|--------|---------|
| `同意切分` | 同意、可以、确认、入切分、是、好、开始切分 |
| `暂搁置` | 暂不、以后、先不做、非本轮、下次、再说 |
| `已终止` | 废弃、取消、终止、不做、删除 |
| `空` | 未填、—、/、无 |

AI 优先尝试精确匹配，不匹配时按语义等价映射判断。匹配结果写入行动清单的摘要中供人类确认。无法映射时提示人类使用标准值。

### 4.3 切分边界的处理

| 边界情况 | 处理方式 |
|---------|---------|
| 方案块行号边界落在段落中间 | 按边界切，被截断的段落完整归入起始片。在行动清单中标注"段落被边界截断" |
| 连续两片之间有空白行 | 空白行归入前一片 |
| 切分内容为空（标记之间无内容） | 不输出空文件，视为无效标记 |

#### AI 动作序列（标注 Step Start~4 对应四步循环）

**Step Start · 前置校验**（检查上游前置条件，逐级报错，首次即停）：
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 不存在 → 输出"请先运行 **eos-ort00-textualize**" → **结束**
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/01-eos-sysdev-status.md` 不存在 → 输出"请先运行 **eos-ort00-textualize**" → **结束**
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 中无文本化 `.textualized.md` 文档 → 输出"请先运行 **eos-ort00-textualize**" → **结束**
   - 全部就绪 → 进入 Step 1

**Step 1 · 新增处理**：读取 状态文档 → 表A — OR原料处理建议及进展，筛选新增并直接处理
   - **人类决策=同意切分 + 生命周期=已文本化** → 语义结构分析（文档结构→段落→体量→关联性）→ 切分判断 → 写入方案块+反馈提示区 → [记录] 更新 表A（生命周期→切分方案待确认 | AI建议→审核切分方案）

   - **人类决策=暂搁置** → 跳过
   - **人类决策=空** → 跳过（记入行动清单选材提醒）

**Step 2 · 反馈处理**：读取 状态文档 → 表A — OR原料处理建议及进展，筛选有方案块的文档并直接执行
   - **生命周期=切分方案待确认 + 有方案块 + 反馈=同意执行** → 校验行号 → 按行号分割 → 输出切分文件（YAML frontmatter）→ 清理方案块 → [记录] 每一切分文件追加一行到 表B（→已切分）+ [记录] 更新 表A（→已切分）→ [同步] 从表B 推导各切分文件的 OR切分文档状态表 + 追加切分迭代记录+决策记录到管理区
   - **生命周期=切分方案待确认 + 有方案块 + 反馈=不切分** → 输出单片切分文件（total=1）→ 清理方案块 → [记录] 追加一行到 表B（→已切分）+ [记录] 更新 表A（→已切分）→ [同步] 从表B 推导 OR切分文档状态表 + 追加切分迭代记录+决策记录到管理区
   - **生命周期=切分方案待确认 + 有方案块 + 反馈=调整意图** → 解析反馈 → 调整方案块 → 归档旧版+反馈到历史区 → 写入新版方案块+新反馈区 → 表A 保持"切分方案待确认" → [同步] 追加重生成记录到管理区
   - **生命周期=切分方案待确认 + 有方案块 + 反馈=废弃** → [记录] 废弃处理（更新表A 生命周期→已废弃）→ 追加废弃记录到管理区
   - **生命周期=切分方案待确认 + 有方案块 + 无反馈文本** → 跳过（等待人类操作）

**Step End · 输出交互**：
   - **输出行动清单**：按状态分组输出（待选材/待澄清/待反馈）
   - **人机交互**：询问人类是否打开 EOS系统设计状态文档

### 5.3 状态文档 → 表A / 表B — 权限分工

本步使用两张表协同管理切分处理进度：

- **表A — OR原料处理建议及进展**：文档级生命周期（已登记→已文本化→切分方案待确认→已切分/已废弃），由 Step 1 生成方案时更新、Step 2 读取方案状态
- **表B — OR切分处理建议及进展**：切分文件级生命周期（已切分→下游处理），由 Step 2 执行切分时为每一切分文件追加一行，是各切分文件头部 OR切分文档状态表（§5.1）的权威源

**字段职责约定**：

- **生命周期**：流程状态字段，供 AI 和流程判断使用，决定材料当前所处客观阶段
- **人类决策**：批次控制字段，决定材料是否进入本轮处理
- **AI建议**：提示字段，仅供人类快速理解下一步建议，**不参与 Step 入口硬判断**

**同步契约**（按四步循环通用规则）：先更新权威源表 → 再推导写入派生表。不一致时以权威源为准重建。

```

| 源文件 | 内容摘要 | 切分文件 | 生命周期 | AI建议 | 操作指引 | 最后更新 |
|--------|---------|---------|---------|-------|---------|---------|
| 01-市场调研报告.pdf | 市场部业务需求汇总 | 01-市场调研报告.chunk-001.md | 已切分 | 可以澄清 | 可送 ort02 进入澄清处理 | 2026-06-07 |
```

**表A 生命周期取值**：已登记（ort00 自动登记）→ 已文本化（ort00 文本化完成）→ 切分方案待确认（Step 1）→ 已切分 / 已废弃（Step 2）
**表B 生命周期取值**：已切分（Step 2 创建时初始值）
**AI建议取值**：审核切分方案（表A）/ 可以澄清（表B），仅供人类参考
