# 原始需求语义切分

**Skill ID**: eos-ort01-chunk · **Step ort01-chunk: Semantic Chunking**

> **概述**：对已文本化的原始需求文档执行语义结构分析和切分。每次运行时检测正文区状态——无切分方案块时生成方案块并等待反馈；有方案块且人类已输入反馈时按反馈执行切分或重新生成方案。本步可反复进入，人类在反馈区输入意见后重启即可迭代。
>
> **输入 → 输出**：文本化 `.md` 文档（正文区：纯原文）+ 状态文档 → 切分文件（`-sub-{seq}.md`）+ 文本化文档正文区（切分方案块 / 清理后纯原文）+ 状态文档更新
>
> **接口设计**：本步靠正文区状态驱动——检测正文区有无切分方案块决定走生成路径还是执行/重切路径。无变更记录表、无行内切分标记。方案块写入正文区（紧接原文之后），执行后清理。上游 eos-ort00-textualize 提供纯原文正文区。
>
> **版本**：v0.2 | **修订**：2026-06-08

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：`../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 目录中的文本化 `.md` 文档（正文区：纯原文）+ `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/01-eos-sysdev-status.md`
- **产出**：切分文件（每片一个 `{原始文件名}-sub-{seq}.md`）+ 文本化文档正文区更新（方案块/清理后纯原文）+ 状态文档更新（状态文档 → 1.2 OR切分进展状态及处理建议一览表 + 状态文档 → 1.1 OR材料进展状态及处理建议一览表）

### 与四条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **需求相对** | 本步不涉及需求/方案角色切换，语义切分仍属原始需求预处理阶段，无需求相对性可言，不适用 |
| **详概分离** | 本步仅执行语义结构分析和切分边界识别，内容理解仅限于语义聚类和段落边界定位所需的最低程度。不做需求内容的评价、分解或设计决策 |
| **文档结对** | 本步不产生文档结对关系。本步产出的切分文件是 eos-ort02-clarify 的输入 |
| **基于资产** | 切分文件是原始文档的派生资产，通过文件名关联保持追溯。状态文档中通过**源文件**和**文本化文档**字段维护切分与原始文件的关系 |

### 三区结构说明

本步操作的文件（文本化 `.md` 文档）的三区：

| 区域 | 内容 | 本步操作 |
|------|------|---------|
| **管理区** | 状态摘要、切分迭代记录、决策记录 | 写入/更新状态摘要；追加迭代记录和决策记录 |
| **正文区** | 原文 + 切分方案块 + 反馈提示区（临时） | 生成阶段：写入方案块+反馈区；执行阶段：清理方案块+反馈区，正文区回归纯原文 |
| **历史区** | 旧版方案块 + 人类反馈原文（重新生成时归档） | 重新生成分支：归档旧版+反馈 |

## 三、操作流程

核心驱动逻辑：读取 状态文档 → 1.2 OR切分进展状态及处理建议一览表的当前状态，检测正文区有无切分方案块——无方案块时执行语义结构分析并生成方案块（附反馈提示区），写入正文区后结束，等待人类反馈；已有方案块时读取末尾反馈文本，按内容执行切分或重新生成方案。

本步按"扫描分流 → 正文区检测 → 生成/执行/重新生成 → 写入/归档 → 行动清单"的线性序列执行。

**批处理说明**：每轮扫描状态文档 → 1.2 OR切分进展状态及处理建议一览表中所有处于可处理状态的文件，按正文区状态分组执行对应分支（生成/执行/重切/跳过）。同轮中多个文件的处理互不依赖，各自在行动清单中按状态独立输出。人类可为多个文件同时输入反馈后集中重启本步。

操作流程概览：

```

  │   │    ├─ [同意执行] → [执行/确认切分方案] ↓
```

### 3.1 生成方案（正文区无方案块时）

AI 对文本化文档执行语义结构分析，生成切分方案块写入正文区（紧接原文之后），末尾附反馈提示区。

**语义结构分析**：

| # | 操作 | 说明 |
|---|------|------|
| 1 | 文档结构识别 | 识别文档的层次结构（章节/标题/子标题层级）。有目录结构的按结构展开；无显式结构的按段落语义聚类 |
| 2 | 语义段落识别 | 识别自然语义段落的起止边界。判断依据：主题转换点、章节边界、段落主题一致性 |
| 3 | 体量评估 | 评估每段的篇幅，判断是否超过正常单次处理范围 |
| 4 | 关联性分析 | 分析各段之间的语义依赖关系——是否存在跨段的概念定义引用 |

**切分判断规则**：

| 条件 | 结论 | 输出 |
|------|------|------|
| 文档总篇幅适中且语义单一 | **不切分** | 切分建议=不切分，理由=篇幅适中/语义单一 |
| 文档篇幅大但语义单一 | **不切分** | 切分建议=不切分，理由=语义不可分，提示下游注意体量 |
| 文档篇幅大且含多个语义单元 | **切分** | 切分建议=切分，输出切分方案 |
| 文档篇幅小但含多个语义独立的业务域 | **切分** | 切分建议=切分，输出切分方案 |

**切分方案块格式**（写入正文区，紧接原文之后）：

```markdown
╔═ 切分方案 ═══════════════════════╗
║                                    ║
║ 建议：切分为 3 片                    ║
║                                    ║
║ 001 · 市场部业务需求                 ║
║   行号：L120~L340                   ║
║   对应：§2.1~§2.5                  ║
║   理由：同属市场部业务域，语义独立     ║
║                                    ║
║ 002 · 研发部开发流程                 ║
║   行号：L341~L480                   ║
║   对应：§3.1~§3.4                  ║
║   理由：同属研发部业务域，语义独立     ║
╚═════════════════════════════════════╝
```

不切分场景：

```markdown
╔═ 切分方案 ═══════════════════════╗
║                                    ║
║ 建议：不切分                        ║
║ 理由：文档语义单一，无需切分         ║
╚═════════════════════════════════════╝
```

**反馈提示区**（紧接方案块之后）：

```
──────────────────────────────────────────
📋 请在此处输入反馈意见：

  [同意执行] — 按当前方案执行切分
  [调整]    — 描述需要调整的内容（如"002 范围扩到 L500"）
  [不切分]   — 不应切分，输出单片
  [废弃]     — 该材料应废弃

您的反馈：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**写入状态摘要**（由状态文档权威源推导写入 OR切分文档状态表，见 §3.4）：

切分建议=切分(N片) 时：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 切分建议（切分） | 切分（N片：{标签1}/{标签2}/...） | 审阅切分方案块后输入反馈 | 打开文档审阅方案块→在反馈区输入意见→重启 eos-ort01-chunk |
```

切分建议=不切分 时：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 切分建议（不切分） | 不切分（整份文档） | 在反馈区输入意见 | 审阅方案块→在反馈区输入"同意执行"或"不切分"→重启 eos-ort01-chunk |
```

### 3.2 执行/确认切分方案（反馈=同意执行或不切分时）

本分支处理两种场景：**同意执行**（按方案块行号范围切分为多片）和**不切分**（输出单片 total=1）。不切分场景跳过行号校验和分割步骤（2-3），直接输出单片。

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 读取切分方案块 | AI | 从正文区读取语义标签；不切分场景仅读取总体理由 |
| 2 | 校验方案完整性 | AI | 仅切分场景：检查行号范围无重叠、无遗漏覆盖区域、语义标签完整 |
| 3 | 按行号范围分割 | AI | 仅切分场景：根据方案块中的行号范围，将原文按行分割为独立切分 |
| 4 | 附加 YAML frontmatter | AI | 每片插入 YAML frontmatter（source/original_source/seq/total/label/range），status=raw |
| 5 | 输出切分文件 | AI | 写入 `{原始文件名}-sub-{seq}.md`；不切分场景输出单片（total=1） |
| 6 | 清理正文区方案块 | AI | 删除切分方案块 + 反馈区，正文区回归纯原文 |
| 7 | 更新 状态文档（权威源）→ 推导写入 OR切分文档状态表 | AI | 先更新 状态文档（状态文档 → 1.2 OR切分进展状态及处理建议一览表 / 状态文档 → 1.1 OR材料进展状态及处理建议一览表），再据此推导写入 OR切分文档状态表（切分文件顶部 + 文本化文档顶部同步更新） |

**切分文件命名**：

```
10-raw-files/
├── 01-市场调研报告.pdf              ← 原始文件（只读）
├── 01-市场调研报告.md                ← 文本化文档
├── 01-市场调研报告-sub-001.md      ← 切分输出
├── 01-市场调研报告-sub-002.md      ← 切分输出
└── 01-市场调研报告-sub-003.md      ← 切分输出
```

**切分文件 YAML frontmatter**：

```markdown
---
chunk:
  source: {原始文件名}.md
  original_source: {原始文件名}.{ext}
  seq: {序号}
  total: {总片数}
  label: {语义标签}
  range: {原文范围}
status: raw
---
（切分正文内容）
```

**执行后状态摘要**（同步更新文本化文档 + 各切分文件的 OR切分文档状态表）：

```
| 阶段 | AI建议 | 待处理 | 操作指引 |
|------|--------|--------|---------|
| 切分已执行 | — | 无 | 运行 eos-ort02-clarify 进行澄清标注 |
```

### 3.3 重新生成方案（反馈=调整意图时）

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 解析修改建议 | AI | 读取人类反馈文本中的具体要求（如"002 范围扩到 L500""新增一片"） |
| 2 | 调整方案 | AI | 按反馈调整方案块中的边界、标签、片数 |
| 3 | 归档 | AI | 旧版方案块 + 人类反馈原文 → 历史区（版本 #N） |
| 4 | 写入新版 | AI | 更新后的方案块 + 新反馈提示区 → 正文区，等待人类下一轮反馈 |

**归档格式**（历史区）：

```
## 历史版本

### 方案 #2 · 2026-06-07 · 重新生成
- 旧方案摘要：[切分 N 片 / 不切分]
- 人类反馈：[反馈原文]

### 方案 #1 · 2026-06-07 · 首次生成
- ...
```

### 3.4 写入：更新状态表 + 追加记录

| # | 操作 | 说明 |
|---|------|------|
| — | **生成方案分支** | 更新状态文档 → 1.2 OR切分进展状态及处理建议一览表为"待确认切分方案" + 操作指引；状态文档 → 1.1 OR材料进展状态及处理建议一览表更新 AI 建议（切分建议+理由） |
| — | **执行切分分支** | 各切分文件注册为独立记录（生命周期=已登记）；原始文档生命周期→已切分；状态文档 → 1.2 OR切分进展状态及处理建议一览表→切分已执行（不切分→不切分已确认）；状态文档 → 1.1 OR材料进展状态及处理建议一览表更新 AI 建议=切分完成 |
| — | **重新生成分支** | 状态文档 → 1.2 OR切分进展状态及处理建议一览表保持"待确认切分方案"不变 |
| — | 追加切分迭代记录 | 在管理区追加执行记录块，含批次号、各片摘要 |
| — | 追加决策记录 | 根据本次处理结果填充决策记录模板，追加到管理区 |

**决策记录格式**（追加到管理区）：

```
本步关键决策记录：
├─ 源文件：[文件名]
├─ 执行模式：[首次切分 / 重切 / 不切分 / 废弃]
├─ 切分总览：
│  ├─ 总片数：[N 片]
│  ├─ 执行日期：[YYYY-MM-DD]
│  └─ 执行批次：[批次号]
├─ 重切记录（仅重切场景）：
│  ├─ 前次总片数：[N]
│  └─ 变更类型：[新增片 / 减少片 / 边界调整]
├─ 边界异常记录：[如有段落被边界截断等问题在此记录]
└─ 状态文档 → 1.2 OR切分进展状态及处理建议一览表更新：[待确认切分方案 → 切分已执行 / 不切分已确认]
```

```
|--------|---------|---------|------|
```

## 四、AI 执行指南

### 4.1 正文区检测规则

AI 在入口分流中检测正文区状态：

| 检测结果 | 判定 |
|---------|------|
| 正文区以 `╔═ 切分方案` 开头的内容块存在 | 有方案块 → 读取末尾反馈文本 |
| 正文区无 `╔═ 切分方案` 开头的内容块 | 无方案块 → 进入生成方案路径 |
| 方案块存在但无反馈文本（反馈区为空） | 跳过，等待人类操作 |

> **检测范围**：仅检测正文区（管理区之后的内容）。管理区中的历史版本块不影响检测——正文区中当前是否有方案块是关键。

### 4.2 切分边界的处理

| 边界情况 | 处理方式 |
|---------|---------|
| 方案块行号边界落在段落中间 | 按边界切，被截断的段落完整归入起始片。在行动清单中标注"段落被边界截断" |
| 连续两片之间有空白行 | 空白行归入前一片 |
| 切分内容为空（标记之间无内容） | 不输出空文件，视为无效标记 |

### 4.3 启动方式

#### 人类启动指令

> 开始原始需求语义切分（或 eos-ort01-chunk）

#### AI 动作序列

0. **前置校验**：检查上游前置条件。**逐级报错，首次即停**：
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 不存在 → 输出"请先运行 **eos-ort00-textualize**" → **结束**
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/01-eos-sysdev-status.md` 不存在 → 输出"请先运行 **eos-ort00-textualize**" → **结束**
   - `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 中无文本化 `.md` 文档 → 输出"请先运行 **eos-ort00-textualize**" → **结束**
   - 全部就绪 → 进入步骤 1
1. **扫描分流**：读取 状态文档 → 1.2 OR切分进展状态及处理建议一览表 + 读取文件正文区
   - **人类决策=已终止** → 废弃处理
   - **人类决策=空** → 跳过（输出"已在 状态文档 → 1.1 OR材料进展状态及处理建议一览表中列出待处理材料，请在对应记录的 人类决策 字段设 已确认 后重启本步"）
   - **人类决策=已确认** → 继续检测：
     - **状态文档 → 1.2 OR切分进展状态及处理建议一览表=切分已执行/不切分已确认** → 跳过（已在终态，无需重复处理）
     - **正文区无方案块（不在上述终态）** → 进入步骤 2（语义结构分析 + 生成方案）
     - **正文区有方案块 + 反馈=同意执行** → 进入步骤 3（执行切分）
     - **正文区有方案块 + 反馈=调整意图** → 进入步骤 4（重新生成）
     - **正文区有方案块 + 反馈=不切分** → 输出单片（total=1），跳至步骤 5
     - **正文区有方案块 + 无反馈文本** → 跳过
2. **生成方案**：语义结构分析 → 生成切分方案块 + 反馈提示区 → 写入正文区 → 更新状态文档 → 1.2 OR切分进展状态及处理建议一览表 = 待确认切分方案 → 推导更新 OR切分文档状态表 → 跳至步骤 6
3. **执行切分**：读取方案块 → 校验完整性 → 按行号范围分割 → 输出切分文件 → 清理正文区方案块 → 跳至步骤 5
4. **重新生成**：解析反馈 → 调整方案 → 归档旧版+反馈到历史区 → 写入新版方案块 + 新反馈提示 → 跳至步骤 5
5. **写入**：更新 状态文档（权威源：1.2 OR切分进展状态及处理建议一览表 / 1.1 OR材料进展状态及处理建议一览表）→ 推导更新 OR切分文档状态表 → 追加切分迭代记录 → 追加决策记录
6. **输出行动清单**
7. **人机交互**：询问人类是否打开状态文档

## 五、模板与格式

### 5.1 切分文件模板

```markdown
---
chunk:
  source: {原始文件名}.md
  original_source: {原始文件名}.{ext}
  seq: {序号}
  total: {总片数}
  label: {语义标签}
  range: {原文范围}
status: raw
---
（切分正文内容）
```

### 5.2 OR切分文档状态表

| 阶段 | AI 建议 | 待处理 | 操作指引 |
|------|---------|--------|---------|
| 切分建议（切分） | 切分（N片） | 审阅方案块后输入反馈 | 审阅方案块→反馈区输入意见→重启 |
| 切分建议（不切分） | 不切分 | 在反馈区输入意见 | 审阅方案块→反馈区输入意见→重启 |
| 切分已执行 | — | 无 | 运行 eos-ort02-clarify |
| 已废弃 | 建议终止 | 无 | 该文件不再处理 |

### 5.3 状态文档 → 1.2 OR切分进展状态及处理建议一览表

状态文档 → 1.2 OR切分进展状态及处理建议一览表是 `01-eos-sysdev-status.md` 文件中的二级子表（完整文件结构见 eos-ort00-textualize §5.2）。本表是切分状态的权威源，各文件头部的 OR切分文档状态表（§5.1）由本表派生。

**同步契约**：AI 每次执行写入时，必须**先更新本表（权威源），再推导写入各文件头部的 OR切分文档状态表**。若检测到两者不一致，以本表为准重建 OR切分文档状态表。

```
状态文档 → 1.2 OR切分进展状态及处理建议一览表：

| 源文件 | 内容摘要 | 文本化文档 | 切分状态 | 操作指引 | 最后更新 |
|--------|---------|-----------|---------|---------|---------|
| 01-市场调研报告.pdf | 市场部业务需求汇总，含线索管理、客户分级、销售报表 | 01-市场调研报告.md | 待确认切分方案 | 审阅正文区切分方案块 → 在反馈区输入意见 → 重启 eos-ort01-chunk | 2026-06-07 |
```

**切分状态取值**：待分析 / 待确认切分方案 / 切分已执行 / 不切分已确认 / 待重切
