# 原始需求规范化

**Skill ID**: eos-ort03-norm · **Step ort03-norm: Requirement Normalization**

> **概述**：对已澄清的原始需求材料执行条目拆分、规范化检查和冲突检测，输出分析建议块到 OR 基线文档正文区。同步从 OR 文本中推理并管理三类 OR 层跨步骤资产——**输出产品构成**（直接匹配 23 资产，未匹配条目以"疑似"状态追加写入 23）、**非功能需求资产**（直接匹配 22 三层资产，未匹配候选以"候选"状态追加写入 22）及**用户角色**（汇总到 OR 基线正文区顶部 `#role-list`）。每次运行时检测 OR 基线文档正文区状态——无分析建议块时执行拆分→规范化→冲突检测，写入方案块等待反馈；有分析建议块且人类已输入反馈时按反馈写入 OR 基线管理区或重新生成方案。本步可反复进入，人类在反馈区输入意见后重启即可迭代。
>
> **输入 → 输出**：已澄清切分文件（`-sub-{seq}.md`，status=clarified，人类决策=同意规范化）+ EOS系统设计状态文档 → OR 基线文档（`01-eos-original-requirements.md`，三区：管理区 OR基线条目状态表 + 正文区 OR 层资产清单+分析建议块 + 历史区归档）+ `23-eos-output-architecture.md`（输出产品构成疑似条目追加）+ `22-eos-nfr-taxonomy.md`（NFR 候选条目追加）+ EOS系统设计状态文档更新（状态文档 → 表B — OR切分处理建议及进展 + 生命周期→已规范化）
>
> **接口设计**：上游 eos-ort02-clarify 在状态文档 → 表B — OR切分处理建议及进展中产出"生命周期=已澄清 + 人类决策=同意规范化"的记录作为新增待处理材料。本步从 `10-raw-files/` 读取已澄清切分文件，输出规范化 OR 条目写入 OR 基线文档（管理区/正文区/历史区三区）并更新状态文档表B/表C。本步同时产出 OR 层资产——输出产品构成（产品类型/组部件/文档）直接追加写入 23-eos-output-architecture.md，角色汇总到 OR 基线正文区 `#role-list`。下游 t01（OR→SR 映射）通过读取表C中"生命周期=已规范化"的记录获取 OR 基线条目，并通过正文区角色清单和 23 资产获取角色和产品构成上下文。
>

### 与四条工程化设计原则的关系

| 原则 | 在本步中的体现 |

### OR 切分文档结构

本步读取的切分文件（`{原始文件名}-sub-{seq}.md`，位于 `10-raw-files/`）沿用上游 ort02 的三区结构。本步对切分文件的操作**限于管理区**（更新状态），不再操作正文区和历史区——正文区的规范化工作迁移到 OR 基线文档正文区：

| 区域 | 内容 | 本步操作 |
|------|------|---------|
| **管理区** | OR切分文档状态表、规范化迭代记录、决策记录 | 写入/更新 OR切分文档状态表；追加迭代记录和决策记录 |
| **正文区** | 原文（已澄清） | **不再操作**——正文区内容在规范化时被复制到 OR 基线文档正文区 |
| **历史区** | ort02 遗留的澄清迭代记录 | **不再操作**——规范化阶段的归档迁移到 OR 基线文档历史区 |

### OR 基线文档结构

本步产出的文档 `01-eos-original-requirements.md` 采用三区结构：

#### 管理区

管理区的职责是为人类提供 OR 的全貌。本步执行后从状态文档 → 表C — OR基线处理建议及进展推导写入本表。

1. **文档元信息**：版本号、创建日期、状态
2. **OR基线条目进展状态及处理建议一览表**：父子行结构——父行为**源文件（规范状态）**，子行为该源文件规范化产出的各 OR 条目。本表是 `表C — OR基线处理建议及进展` 的派生视图（由表C推导写入）。列定义见 §5.2。`分批` 用于批次选材——人类在 表C中设为 `本批` 选入下游，`—` 暂不处理

> 管理区的状态表是本步最终产物，写入后不再修改（除非被废弃或迭代覆盖）。本步的规范化工作（拆分建议、冲突检测、人机交互）发生在正文区，不直接写入管理区。OR 条目的详细内容（描述、指标、出处等）见正文区分析建议块。

#### 正文区

正文区是**规范化结果的永久存放区**。每次 ort03 运行时，将已澄清切分文档的内容和 AI 分析建议块写入正文区；人类在正文区阅读并输入反馈。人类确认（同意执行）后内容保留在正文区作为当前确认版本，不再迁移。当同一源文件重新规范化时，旧版正文区内容归档到历史区，新版本覆盖正文区。

正文区包含两类内容（按自上而下顺序）：

1. **角色预汇总**（位于正文区 `#role-list`）：由 Step 4 角色资产管理汇总写入，含操作反馈列供人类标注角色修改意见
2. **各切分文件条目**：按切分文件组织的原文、分析建议块和反馈区

```
### 角色预汇总 `#role-list`

### {切分文件名} `#body-{seq}`
```

各切分文件条目含：
- **源文件标识**：原始文件名、生命周期状态
- **正文原文**：已澄清切分文件的原文内容
- **分析建议块**：AI 的拆分方案、冲突检测结果（`╔═ 分析建议块 ═══╗` 格式框）
- **反馈区**：供人类输入意见（确认后保留作为决策记录）
- **历史导航**：跳转到历史区旧版本的超链接（`→ [归档版本 #N](#hist-{seq}-vN)`）。尚无归档时为灰色提示"尚无，归档时创建"；归档后变为活跃链接
- **状态**：正文区条目的审阅状态，取值 `待确认`（AI 刚生成建议，等待人类反馈）或 `已确认`（人类已同意执行，内容已定稿）

#### 历史区

历史区的职责是**存放因重新规范化而被替换的旧版正文区内容**。当同一源文件重新运行 ort03 并确认执行后，正文区的旧版本整体归档到历史区，新版本占用正文区。

历史区每个条目对应一份切分文件的一次规范化迭代：

```
### {切分文件名} — 归档 #{N} · {日期} `#hist-{seq}-v{N}`
```

每条含：
- **原文内容**：旧的已澄清切分文件原文
- **分析建议块**：旧的 AI 拆分方案和冲突检测
- **人类反馈**：旧的人类输入反馈
- **返回导航**：跳回正文区当前版本的超链接（`← [返回当前版本](#body-{seq})`）

#### 三区协作关系

```
管理区（永久）        正文区（当前确认版本）          历史区（替换旧版）
┌────────────────┐      ┌───────────────────┐      ┌────────────────┐
│ OR基线条目      │      │ ★ #role-list      │      │ 切分文件A v1    │
│ 状态表（派生）    │      │  （Step 4 汇总）    │      │ + 建议+反馈     │
│                │      │                     │      │ 切分文件B v1    │
│                │      │                     │      │（被v2替换时归档）│
│                │      │ 切分文件A 原文       │      │                │
│                │      │ + AI建议+反馈       │      │                │
│                │      │ （确认后保留）       │      │                │
│                │      │                     │      │                │
│                │      │ 切分文件B 原文       │      │                │
│                │      │ + AI建议+反馈       │      │                │
└────────────────┘      └───────────────────┘      └────────────────┘
      ↑    推导写入           ↑  重新规范化时     ←超链接→
      └── 从 表C ──────── 旧版→新版覆盖     ↕ 双向导航
```

OR 条目生命周期（7 状态，由本步和下游步骤协同管理）：

| 状态 | 含义 | 管理步骤 |
|------|------|---------|
| **待分解映射** | 初写入（本步设定初始值） | 本步写入 → t01 启动时自动扫描 |
| **待确认方案** | t01 已给出分解映射方案，等人类确认 | t01（人类审阅） |
| **迭代中** | 人类已反馈，AI 需修改方案 | t01（AI 按反馈调整） |
| **已确认** | 分解映射方案已通过 | t06（等待全链走通） |
| **验证中** | 至少一条子路径已在下游流转 | 自动进入（无需操作） |
| **已验证冻结** | 全链验证通过 | t06 批量推进 |
| **已废弃** | 需求被废弃 | 本步/下游均可触发 |

## 三、操作流程

本步遵循 ORT 预处理链四步循环（详见 README §二）：

| Step | 本步对应 |
|------|---------|
| **Step Start · 前置校验** | 检查上游前置条件（已澄清切分文件、状态文档就绪） |
| **Step 1 · OR 层资产加载** | 加载 23（已确认+疑似）+ 22（三层资产：一级类目→分类维度→具体需求）+ 现有 `#role-list` → 标记资产状态供 Step 4/5/6 引用 |
| **Step 2 · 新增处理** | 读 状态文档（表B），筛出新增切分文件，引用 Step 1 资产做标注基准→逐文件生成分析建议块（条目拆分→角色/产品类型/组部件/文档逐条匹配资产→规范化→冲突检测→优先级），[记录]更新表B→[同步]推导派生表 |
| **Step 3 · 方案反馈处理** | 扫描 OR 基线文档正文区有分析建议块的条目，读反馈文本**直接执行**——[同意执行]→统一编号+替换分析建议块 C→OR 编号并写入表C，[修改建议]→归档调整，[废弃]→废弃+移除分析建议块角色行；分支末尾[记录]更新表B/表C→[同步]推导派生表+追加决策记录。**不处理角色反馈** |
| **Step 4 · 角色资产管理** | 读取全部分析建议块角色字段 → 汇总去重 → 写入/更新 `#role-list`；标准名写回分析建议块；检测操作反馈列执行修订 |
| **Step 5 · 输出产品构成管理** | 匹配 Step 1 已加载的 23 资产 → 未匹配条目直接追加写入 23（状态=疑似）→ 检测 23 操作反馈列执行修订 |
| **Step 6 · 非功能需求资产管理** | 匹配 Step 1 已加载的 22 三层资产 → 逐级匹配（具体需求→分类维度→候选）→ 未匹配追加候选（状态=候选）→ 检测 22 操作反馈列执行修订 |
| **Step End · 输出交互** | 行动清单（含三类资产状态）+ 人机交互 |

执行序列：OR 层资产加载 → 新增处理 → 方案反馈处理 → 角色资产管理 → 输出产品构成管理 → 非功能需求资产管理 → 输出交互。各 Step 内部逻辑见下方流程图和对应 §3.x 子节。

> **批处理与"新增"判定说明**：本步每轮扫描 表B，将满足 `AI建议=可以规范化`（ort02 通过后标记）**且** `人类决策=同意规范化`（人类选定本批）**且** `生命周期=已澄清`（尚未被 ort03 处理过）的切分文件判定为"新增"，进入 Step 2 生成方案。三个条件缺一不可——缺少任一条件对应其他分支（见下方流程图）。数据源为表B（切分文件级）。不同源文件的 OR 条目在统一编号时跨源排序，在管理区中连续编号。

#### 阶段一：条目拆分

按"四单一原则"从已澄清切分文件中逐条提取并拆分原始需求表述：

| 原则 | 问题 | 操作 |
|------|------|------|
| **单一角色** | 是否涉及多个用户角色？ | 拆到每条仅涉及一个角色 |
| **单一语义** | 是否描述了多个独立功能或关注点？ | 拆到每条仅含一个独立语义 |
| **单一路径** | 是否同时包含 FR 和 NFR 内容？ | 拆为 FR 条目和 NFR 条目，标注类型 |
| **单一可确认性** | 不依赖 SR 知识，这也是一条可被相关方独立确认的单一诉求吗？ | 如不是，继续拆分 |

**拆分纪律**：

1. **不做合并**：仅拆分不做归并——不同角色的诉求各自独立
2. **保留原始出处**：每条 OR 条目记录提出方、原始表述、日期
3. **超额拆分优于不足**：遇不确定时倾向于拆得更细
4. **无法拆分标记"待补全"**：原文不完整无法判断语义边界时标记
5. **角色轻量标注**：拆分后为每条 OR 条目标注角色类型（用户角色或相关方角色）。FR 条目从规约描述的主体中提取执行主体 → 标注为用户角色；NFR 条目从原文中提取关注方 → 按是否直接操作目标系统判定（见通用规范 §4.3.4）。**优先采用 Step 1 已加载的 `#role-list` 中的标准角色名**——匹配到已有角色时使用同一名称避免同义词，未匹配到时按规则推断新角色名。本步仅做轻量标注（写入分析建议块的角色字段），**不做汇总也不写入 `#role-list`**——汇总由 Step 4 角色资产管理统一处理
6. **产品类型轻量标注**：从 OR 条目的业务域特征词推理产品类型（如 OR 描述航电业务 → 产品类型=民机航电设备）。**优先匹配 Step 1 已加载的 23 产品类型**——同名/同义则用标准名；无匹配则推断新名。**逐条独立推理**——同批不同 OR 条目可标注不同产品类型（由 Step 5 汇聚到 23）。写入分析建议块的`产品`字段，由 Step 5 统一汇总
7. **架构节点类型轻量标注**：从 OR 文本提取组部件类型名称（如"电子模块""软件模块""机箱"）→ 写入分析建议块的`组部件`字段。**优先匹配 Step 1 已加载的 23 已有组部件名**——匹配到已知名称时使用同一名称，避免同一组件在分析建议块中出现不同叫法。如 OR 中提到采办策略线索（"采购""供应商"→外购；"自研"→自研），一并写在括号内。由 Step 5 统一汇总写入 23（匹配已有资产；未匹配条目以"疑似"状态追加）
8. **文档轻量标注**：从 OR 文本提取该条目涉及的文档名称（如"合格证""入厂测试报告"）→ 写入分析建议块的`文档`字段。**优先匹配 Step 1 已加载的 23 关联文档**——匹配到已知文档名时使用同一名称。本步仅做轻量标注（写入分析建议块的文档字段），**不做汇总**——汇总由 Step 5 统一处理（在对应组部件下查重，未匹配的追加到 23 的关联文档表）
9. **NFR 资产轻量标注**：仅对 NFR 类型 OR 条目执行。从 OR 原文提取 NFR 语义（关注方向或具体指标约束），**逐级匹配 Step 1 已加载的 22 三层资产**——① 优先匹配具体需求（@nfr-xxx-REQ-yy）：OR 原文的指标约束（如"保留≥365 天"）匹配到已确认具体需求时引用其 ID；② 无具体需求匹配→匹配分类维度（@nfr-xxx）：OR 的关注方向匹配到已确认分类时引用其 ID；③ 均无匹配→检查候选区是否有同义候选（有则引用候选 ID，无则标记"待写候选"）。匹配成功时**必须使用 22 资产中的标准化 ID**。本步仅做轻量标注（写入分析建议块的新增字段），**不做汇总也不写入 22 正文区**——汇总由 Step 6 统一处理

**关于 Step 1 资产引用的总则**：Step 2 的所有字段标注均以 Step 1 加载结果为基准，优先使用已有名称避免同义词噪音。五个字段都是**逐条级**标注——每条 OR 条目独立提取，不做批次级统一。各字段的引用方式：

| 字段 | 参考源 | 操作模式 |
|------|--------|---------|
| **角色** | Step 1 加载的 `#role-list` | 从 OR 原文提取角色候选名 → 查清单：同名/同义则用清单标准名；无匹配→按 FR/NFR 规则推断新名 |
| **产品类型** | Step 1 加载的 23 产品类型 | 从 OR 业务域特征词推理 → 匹配 23 产品类型（同名/同义则用标准名）；无匹配则推断新名。**逐条独立**，同批不同 OR 条目可标注不同产品类型 |
| **组部件** | Step 1 加载的 23 组部件清单 | 从 OR 原文提取组部件候选名 → 查清单：同名/同义则用清单标准名；无匹配则用提取名 |
| **文档** | Step 1 加载的 23 关联文档（各节点下属） | 从 OR 原文提取文档候选名 → 查所属组部件在 23 中的关联文档：同名则用标准名；无匹配则用提取名 |
| **NFR 资产** | Step 1 加载的 22 三层资产（一级类目/分类维度 @nfr-xxx/具体需求 @nfr-xxx-REQ-yy）+ 候选清单 | NFR 条目从 OR 原文提取 NFR 语义 → 逐级匹配：优先匹配具体需求 ID → 次优匹配分类维度 ID → 无匹配→检查候选区；真无则标记"待写候选"（由 Step 6 写入 22） |

#### 阶段二：规范化（五项检查）

> 输入材料已通过 eos-ort02-clarify 的全量澄清，五项标准已在源文档层面经过检查。本步的五项检查是规范化阶段的**最终质量门禁**。如发现残留的清晰性问题，标注为"待确认"并转人类裁定是否退回 eos-ort02-clarify 重新澄清。

#### 阶段三：冲突检测

检测两个范围内的语义冲突：

| 层 | 范围 | 处理 |
|----|------|------|
| **层 A** | 同一切分内拆出的 OR 条目间 | 检测并标记 |
| **层 B** | 跨切分（同原始文件）的 OR 条目间 | 检测并标记 |
| **层 C** | 跨原始文件的 OR 条目冲突 | **不在本步范围**，留给下游 OR→SR 映射阶段按职责范围分组时自然捕获 |

**冲突类型**：

| 类型 | 说明 | 处理 |
|------|------|------|
| 直接冲突 | 语义直接矛盾，无法同时满足 | 标记"需确认"，列出双方原文 |
| 间接冲突 | 各自可实现，但同时实现导致行为不一致 | 标记"需调整"，列出可能的影响 |
| 部分内容冲突 | 一条 OR 条目仅部分内容与其他条目冲突 | 先分解再检；无法分解时标记"内部冲突" |

> **AI 不做冲突裁决**：仅做结构化呈现，不做裁决。

## 四、AI 执行指南

### 4.1 OR 基线正文区 + 23 资产检测规则

| 状态 | 判定 |
|------------------|------|
| 无 `╔═ 分析建议块` 开头的内容块 | 无方案 → 生成路径（复制切分原文→条目拆分→角色+产品+组部件轻量标注→规范化→冲突检测→写入方案块）。Step 2 执行 |
| 存在分析建议块 + 反馈文本（`[同意执行]`/`[修改建议]`/`[废弃]`） | 方案反馈路径 → Step 3 按反馈内容执行 |
| 存在分析建议块 + 无反馈文本 | 跳过，等待人类操作 |
| `#role-list` 中的操作反馈列有标注 | 角色反馈路径 → Step 4 执行角色修订 |
| 23 正文区条目上有操作反馈标注 | 输出产品构成反馈路径 → Step 5 检测并执行修订（确认/修改/新增/删除） |
| 22 正文区条目上有操作反馈标注 | NFR 资产反馈路径 → Step 6 检测并执行修订（确认/修改/删除） |
| 各资产清单已存在 + 分析建议块有变更（由 Step 3 导致） | 对应资产的 Step 各自检测变更后更新。Step 5 检测 23 上的操作反馈并直接执行修订。Step 6 检测 22 上的操作反馈并直接执行修订 |

### 4.2 执行顺序不可逆

条目拆分在前 → 规范化在后 → 冲突检测最后。拆分未完成时做的规范化可能因后续拆分而需重做。

### 4.3 重入变更检测（正文区）

当状态文档 → 表B — OR切分处理建议及进展=已规范化时（重入场景），AI 按以下规则检测正文区是否有变更：

| 检测方法 | 说明 |
|---------|------|
| **last_size 对比** | 读取当前正文区字节数，与 表B 中记录的 last_size 比较。不一致→判定有变更，进入重新生成流程 |
| **人类反馈读取** | 无论大小有无变化，检查末尾是否有人类输入的修改反馈文本。有反馈→按反馈执行；无反馈→跳过 |

> last_size 一致不保证内容无冲突，此时以人类反馈为唯一驱动信号。检测优先级：有反馈文本 > 有 last_size 变化 > 无变化跳过。

### 4.4 启动方式

#### 人类启动指令

> 开始原始需求规范化（或 eos-ort03-norm）

#### AI 动作序列

按 §三 操作流程图依次执行 Step Start→1→2→3→4→End。AI 执行时注意以下细则：
- **前置校验路径**：`01-eos-sysdev-status.md`、`10-raw-files/` 目录
- **编号规则**：临时 `C{seq}-{序号}` → 全局 `OR-{全局SEQ}`
- **方案反馈格式**：`[同意执行]` / `[修改建议]（具体内容）` / `[废弃]`
- **角色反馈位置**：标注在 `#role-list` 操作反馈列（`[确认]` / `[修改]→...` / `[新增]` / `[删除]`），不混入 `#body-{seq}` 方案反馈区
- **输出产品构成反馈位置**：标注在 23 正文区条目上的操作反馈列，不混入 OR 基线方案反馈区
- **NFR 资产反馈位置**：标注在 22 正文区条目上的操作反馈列，不混入 OR 基线方案反馈区
- **各资产管理 Step 独立重入**：仅修改输出产品构成反馈 → 检测到 23 正文区有操作反馈标注 → 仅执行 Step 5，无需重跑 Step 4。仅修改 NFR 资产反馈 → 检测到 22 正文区有操作反馈标注 → 仅执行 Step 6
