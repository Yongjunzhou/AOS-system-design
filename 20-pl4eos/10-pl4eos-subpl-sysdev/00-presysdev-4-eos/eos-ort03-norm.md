# 原始需求规范化

**Skill ID**: eos-ort03-norm · **Step ort03-norm: Requirement Normalization**

> **概述**：对已澄清的原始需求材料执行条目拆分、规范化检查和冲突检测，输出分析建议块到 OR 基线文档正文区。每次运行时检测 OR 基线文档正文区状态——无分析建议块时执行拆分→规范化→冲突检测，写入方案块等待反馈；有分析建议块且人类已输入反馈时按反馈写入 OR 基线管理区或重新生成方案。本步可反复进入，人类在反馈区输入意见后重启即可迭代。
>
> **输入 → 输出**：已澄清切分文件（`-sub-{seq}.md`，status=clarified，人类决策=已确认）+ 状态文档 → OR 基线文档（`01-eos-original-requirements.md`，三区：管理区 OR基线条目状态表 + 正文区分析建议块 + 历史区归档）+ 状态文档更新（状态文档 → 1.3 OR基线进展状态及处理建议一览表 + 生命周期→已规范化）
>
> **接口设计**：本步靠 OR 基线文档正文区状态驱动——检测正文区有无分析建议块决定走生成路径还是执行/重切路径。无变更记录表。分析建议块写入 OR 基线文档正文区；人类确认执行后写入管理区，正文区已确认内容原地保留（仅在重新规范化时替换归档）。上游 eos-ort02-clarify 产出"生命周期=已澄清 + 人类决策=已确认"的记录。下游 t01（OR→SR 映射）通过读取"生命周期=已规范化"的记录继续处理。
>
> **版本**：v0.5 | **修订**：2026-06-09

### OR 切分文档结构

本步读取的切分文件沿用上游 ort02 的三区结构。本步对切分文件的操作**限于管理区**（更新状态），不再操作正文区和历史区——正文区的规范化工作迁移到 OR 基线文档正文区：

| 区域 | 内容 | 本步操作 |
|------|------|---------|
| **管理区** | OR切分文档状态表、规范化迭代记录、决策记录 | 写入/更新 OR切分文档状态表；追加迭代记录和决策记录 |
| **正文区** | 原文（已澄清） | **不再操作**——正文区内容在规范化时被复制到 OR 基线文档正文区 |
| **历史区** | ort02 遗留的澄清迭代记录 | **不再操作**——规范化阶段的归档迁移到 OR 基线文档历史区 |

### OR 基线文档结构

本步产出的文档 `01-eos-original-requirements.md` 采用三区结构：

#### 管理区

管理区的职责是为人类提供 OR 的全貌。本步执行后从状态文档 → 1.4 OR条目进展状态及处理建议一览表推导写入本表。

1. **文档元信息**：版本号、创建日期、状态
2. **OR基线条目进展状态及处理建议一览表**：父子行结构——父行为**源文件（规范状态）**，子行为该源文件规范化产出的各 OR 条目。本表是 `1.4 OR条目进展状态及处理建议一览表` 的派生视图（由 1.4 表推导写入）。列定义见 §5.2。`分批` 用于批次选材——人类在 1.4 子表中设为 `本批` 选入下游，`—` 暂不处理

> 管理区的状态表是本步最终产物，写入后不再修改（除非被废弃或迭代覆盖）。本步的规范化工作（拆分建议、冲突检测、人机交互）发生在正文区，不直接写入管理区。OR 条目的详细内容（描述、指标、出处等）见正文区分析建议块。

#### 正文区

正文区是**规范化结果的永久存放区**。每次 ort03 运行时，将已澄清切分文档的内容和 AI 分析建议块写入正文区；人类在正文区阅读并输入反馈。人类确认（同意执行）后内容保留在正文区作为当前确认版本，不再迁移。当同一源文件重新规范化时，旧版正文区内容归档到历史区，新版本覆盖正文区。

正文区每个条目对应一份已澄清切分文件，按切分文件组织：

```
### {切分文件名} `#body-{seq}`
```

每条含：
- **源文件标识**：原始文件名、生命周期状态
- **正文原文**：已澄清切分文件的原文内容
- **分析建议块**：AI 的拆分方案、冲突检测结果（`╔═ 分析建议块 ═══╗` 格式框）
- **反馈区**：供人类输入意见（确认后保留作为决策记录）
- **历史导航**：跳转到历史区旧版本的超链接（`→ [归档版本 #N](#hist-{seq}-vN)`）。尚无归档时为灰色提示"尚无，归档时创建"；归档后变为活跃链接
- **状态**：正文区条目的审阅状态，取值 `待确认`（AI 刚生成建议，等待人类反馈）或 `已确认`（人类已同意执行，内容已定稿）

#### 历史区

历史区的职责是**存放因重新规范化而被替换的旧版正文区内容**。当同一源文件重新运行 ort03 并确认执行后，正文区的旧版本整体归档到历史区，新版本占用正文区。

历史区每个条目对应一份切分文件的一次规范化迭代：

```
### {切分文件名} — 归档 #{N} · {日期} `#hist-{seq}-v{N}`
```

每条含：
- **原文内容**：旧的已澄清切分文件原文
- **分析建议块**：旧的 AI 拆分方案和冲突检测
- **人类反馈**：旧的人类输入反馈
- **返回导航**：跳回正文区当前版本的超链接（`← [返回当前版本](#body-{seq})`）

#### 三区协作关系

```
管理区（永久）        正文区（当前确认版本）        历史区（替换旧版）
┌────────────────┐      ┌─────────────────┐      ┌────────────────┐
│ OR基线条目      │      │ 切分文件A 原文    │      │ 切分文件A v1    │
│ 状态表（派生）    │      │ + AI建议+反馈    │      │ + 建议+反馈     │
│                │      │ （确认后保留）    │      │（被v2替换时归档）│
│                │      │                  │      │                │
│                │      │ 切分文件B 原文    │      │ 切分文件B v1    │
│                │      │ + AI建议+反馈    │      │（被v2替换时归档）│
└────────────────┘      └─────────────────┘      └────────────────┘
      ↑    推导写入           ↑  重新规范化时     ←超链接→
      └── 从 1.4 子表 ──────── 旧版→新版覆盖     ↕ 双向导航
```

OR 条目生命周期（7 状态，由本步和下游步骤协同管理）：

| 状态 | 含义 | 管理步骤 |
|------|------|---------|
| **待分解映射** | 初写入（本步设定初始值） | 本步写入 → t01 启动时自动扫描 |
| **待确认方案** | t01 已给出分解映射方案，等人类确认 | t01（人类审阅） |
| **迭代中** | 人类已反馈，AI 需修改方案 | t01（AI 按反馈调整） |
| **已确认** | 分解映射方案已通过 | t06（等待全链走通） |
| **验证中** | 至少一条子路径已在下游流转 | 自动进入（无需操作） |
| **已验证冻结** | 全链验证通过 | t06 批量推进 |
| **已废弃** | 需求被废弃 | 本步/下游均可触发 |

## 三、操作流程

核心驱动逻辑：读取 状态文档中"生命周期=已澄清 + 人类决策=已确认"的记录，检测 OR 基线文档正文区有无分析建议块——无时**将已澄清切分文件原文复制到正文区**，执行条目拆分（四单一原则）、规范化（五项检查）、冲突检测（层 A+层 B）、优先级初值，写入分析建议块+反馈提示区后结束，等待人类反馈；已有分析建议块时读取末尾反馈文本，按内容执行 OR 基线管理区写入（正文区已确认内容原地保留）或重新生成方案（归档旧版到历史区）。

本步按"扫描分流 → OR 基线正文区检测 → 生成/执行/重新生成 → 写入/归档 → 行动清单"的线性序列执行。

**批处理说明**：每轮扫描状态文档 → 1.3 OR基线进展状态及处理建议一览表中所有处于"生命周期=已澄清 + 人类决策=已确认"的切分文件，按 OR 基线正文区中各文件的独立条目状态各自执行对应分支（生成方案/执行/重新生成）。不同源文件的 OR 条目在统一编号时跨源排序，在 OR 基线管理区中连续编号。

```

  ├─ `../../80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 中无切分文件 → 输出"请先运行 eos-ort01-chunk" → 结束
```

### 3.1 生成方案（OR 基线正文区无分析建议块时）

#### 阶段零：复制已澄清切分文件原文到 OR 基线正文区

将已澄清切分文件的正文区原文复制到 `01-eos-original-requirements.md` 的正文区。每个切分文件独立一个条目。

> **重新规范化检测**：如果正文区已存在该切分文件的 `#body-{seq}` 条目（即重新规范化），先将该旧条目整体归档到历史区（添加 `#hist-{seq}-v{N+1}` 锚点和 ← 返回链接），再写入新内容。正文区中保留的旧版本仅在重新规范化时被替换归档。

```markdown
---

### 01-市场调研报告-sub-001.md `#body-sub-001`

**源文件**：01-市场调研报告.pdf · **状态**：待确认

→ [归档版本 #1](#hist-sub-001-v1)（尚无，归档时创建）

原文内容：
```
[复制切分文件正文区原文]
```

```

> **超链接规则**：正文区每个条目标题的 `#body-{seq}` 锚点是历史区超链接的目标。当尚无归档版本时，链接文字为"尚无，归档时创建"（灰色提示）；归档后自动更新为指向历史区的活跃链接。
>
> **状态初始值**：正文区条目的状态初始设为`待确认`，表示 AI 分析建议块已就位，等待人类审阅反馈。人类确认执行后（§3.2）状态更新为`已确认`。

#### 阶段一：条目拆分

按"四单一原则"从已澄清切分文件中逐条提取并拆分原始需求表述：

| 原则 | 问题 | 操作 |
|------|------|------|
| **单一角色** | 是否涉及多个用户角色？ | 拆到每条仅涉及一个角色 |
| **单一语义** | 是否描述了多个独立功能或关注点？ | 拆到每条仅含一个独立语义 |
| **单一路径** | 是否同时包含 FR 和 NFR 内容？ | 拆为 FR 条目和 NFR 条目，标注类型 |
| **单一可确认性** | 不依赖 SR 知识，这也是一条可被相关方独立确认的单一诉求吗？ | 如不是，继续拆分 |

**拆分纪律**：

1. **不做合并**：仅拆分不做归并——不同角色的诉求各自独立
2. **保留原始出处**：每条 OR 条目记录提出方、原始表述、日期
3. **超额拆分优于不足**：遇不确定时倾向于拆得更细
4. **无法拆分标记"待补全"**：原文不完整无法判断语义边界时标记

#### 阶段二：规范化（五项检查）

引用通用规范 §7.3 阶段二(c) 清晰性五标准——完整性、主体明确、客体可识别、边界可判定、可测试性。

> 输入材料已通过 eos-ort02-clarify 的全量澄清，五项标准已在源文档层面经过检查。本步的五项检查是规范化阶段的**最终质量门禁**。如发现残留的清晰性问题，标注为"待确认"并转人类裁定是否退回 eos-ort02-clarify 重新澄清。

#### 阶段三：冲突检测

检测两个范围内的语义冲突：

| 层 | 范围 | 处理 |
|----|------|------|
| **层 A** | 同一切分内拆出的 OR 条目间 | 检测并标记 |
| **层 B** | 跨切分（同原始文件）的 OR 条目间 | 检测并标记 |
| **层 C** | 跨原始文件的 OR 条目冲突 | **不在本步范围**，留给下游 OR→SR 映射阶段按职责范围分组时自然捕获 |

**冲突类型**：

| 类型 | 说明 | 处理 |
|------|------|------|
| 直接冲突 | 语义直接矛盾，无法同时满足 | 标记"需确认"，列出双方原文 |
| 间接冲突 | 各自可实现，但同时实现导致行为不一致 | 标记"需调整"，列出可能的影响 |
| 部分内容冲突 | 一条 OR 条目仅部分内容与其他条目冲突 | 先分解再检；无法分解时标记"内部冲突" |

> **AI 不做冲突裁决**：仅做结构化呈现，不做裁决。

### 3.2 执行流程（反馈=同意执行时）

> **批处理时序**：当批次中有多份文件同时进入执行分支时，AI 先全部读取 OR 基线正文区的分析建议块（步骤 1），按源文件（原始文件）分组收集（步骤 2），再对所有文件的 OR 条目统一执行编号（步骤 3）和状态表写入（步骤 4）。同一批次的条目共占一段连续 OR 编号区间。

| # | 操作 | 说明 |
|---|------|------|
| 1 | 读取分析建议块 | 读取 OR 基线正文区分析建议块中的临时编号 OR 条目。批处理时先聚齐所有文件的分析建议块，再进入下一步 |

## 四、AI 执行指南

### 4.1 OR 基线正文区检测规则

| OR 基线正文区状态 | 判定 |
|------------------|------|
| 无 `╔═ 分析建议块` 开头的内容块 | 无方案 → 生成路径（复制切分原文→条目拆分→规范化→冲突检测→写入方案块） |
| 存在分析建议块 + 反馈文本 | 执行/重新生成路径（按反馈内容） |
| 存在分析建议块 + 无反馈文本 | 跳过，等待人类操作 |

### 4.2 执行顺序不可逆

条目拆分在前 → 规范化在后 → 冲突检测最后。拆分未完成时做的规范化可能因后续拆分而需重做。

### 4.3 写入规则

| 场景 | 操作 |
|------|------|
| 新条目写入管理区 | 先追加到 状态文档 → 1.4 OR条目进展状态及处理建议一览表（权威源），再推导写入 OR 基线管理区（派生），初始状态="待分解映射"。正文区已确认内容原地保留 |
| 重规范化（迭代覆盖） | 旧 OR 条目在 1.4 子表中状态更新为"已废弃"；新版追加到 1.4 子表（权威源），再推导更新管理区状态表，状态="待分解映射" |
| 正文区内容更新 | 生成/重新生成分支：写入或更新 OR 基线正文区，正文区和管理区互不影响 |

### 4.4 重入变更检测（正文区）

当状态文档 → 1.3 OR基线进展状态及处理建议一览表=已规范化时（重入场景），AI 按以下规则检测正文区是否有变更：

| 检测方法 | 说明 |
|---------|------|
| **last_size 对比** | 读取当前正文区字节数，与 1.3 表中记录的 last_size 比较。不一致→判定有变更，进入重新生成流程 |
| **人类反馈读取** | 无论大小有无变化，检查末尾是否有人类输入的修改反馈文本。有反馈→按反馈执行；无反馈→跳过 |

> last_size 一致不保证内容无冲突，此时以人类反馈为唯一驱动信号。检测优先级：有反馈文本 > 有 last_size 变化 > 无变化跳过。
