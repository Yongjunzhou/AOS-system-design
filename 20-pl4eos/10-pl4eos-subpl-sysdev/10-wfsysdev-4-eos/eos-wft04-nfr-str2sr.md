# STR-NFR → NFR 量化约束 · 非功能需求量化设计与资产化
**Skill ID**: wft04-nfr

> **概述**：每次执行处理 ≥1 个 STR-NFR（非功能相关方需求）架构末级节点（来自 wft01-nfr，状态=`可以NFR设计` / `待补充NFR设计`），SR-NFR（非功能系统需求）反馈优先于新 STR-NFR 处理。将 STR-NFR 的模糊期望转化为 SR-NFR 的量化约束——逐条量化定义、按三大类归类找宿主、标注分层约束表（全局/业务域/SR-F（功能性系统需求）/CU（配置单元）/PA（产品架构）组件 + 适用链路 + 影响级别/影响点）、写回 22（非功能需求分类资产）NFR 分类资产、展开 03（详细定义文档）详细定义。完成跨域匹配、资产写回与状态推进。
>
> **输入 → 输出**：STR-NFR 架构末级节点（≥1 个，`可以NFR设计` / `待补充NFR设计`）/ 有反馈的 SysReq-NFR 节点 → SysReq-NFR 量化定义 + 分层约束表 + 22 资产写回 + 03 详细定义 + 状态推进
>
> **适用场景**：瀑布式系统设计第4步 · 非功能需求路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v2.2 | **修订**：2026-06-29

## 一、职责定位

### 本 Skill 目的与目标

> **主要目的**：取信于架构师——「指标合理，分层正确」。把 STR-NFR 的模糊期望转化为 SysReq-NFR 的量化约束——每个 NFR 有可验证的量化指标，有明确的约束层级。
>
> **目标**：量化不拍脑袋，分层不张冠李戴。全局约束不误标为局部，局部约束不误标为全局；业务侧配置约束不误下沉为引擎组件约束，引擎组件约束不误停留在业务配置需求。SysReq-NFR 的分层约束表是 wft05-biz / wft05-eng 消费 NFR 约束包的唯一输入。

本 Skill 负责：

1. **接收**——接收人类选中的 ≥1 个 STR-NFR 架构末级节点（`可以NFR设计` / `待补充NFR设计`），校验状态/类型/数量，不合格则说明原因并退出
2. **量化**——逐条 STR-NFR 量化定义（引用 22 具体需求模板作指标参考），模糊指标用 L1 补齐规则处理
3. **归类**——按非功能三大类（质量特性 / 环境适应性 / 可实现性）归类，经 Q1/Q2/Q3（Q1=直接引用 / Q2=扩展 / Q3=新建）找 SR-NFR 宿主
4. **标注**——标注分层约束表（全局/业务域/SR-F/CU/PA组件 + 适用链路 + 影响级别/影响点），写入 SR-NFR 节点供 wft05-biz / wft05-eng 消费
5. **落账**——执行候选区匹配、正式写回 22 NFR 分类资产、更新反向引用
6. **展开**——将量化的 STR-NFR 条目展开至 `03-*.md` 详细定义
7. **推进**——人类确认后 STR-NFR 节点推进为 `已NFR设计`

本 Skill 不负责：

1. 修改 wft01-nfr 已确认的 STR-NFR 架构设计
2. 修改 22 资产的已确认分类维度（@nfr-xxx 分类维度本身）
3. 功能需求的展开（→ wft03-biz）
4. PA 组件设计（→ wft05-biz/eng）

### 定位说明

wft04 是 NFR 路径的唯一 Skill——从 STR-NFR 量化到 SR-NFR 落账，自闭环完成。NFR 路径没有 BA（业务架构）展开步骤（与 biz/eng 路径不同），STR-NFR 直接进入量化约束设计。

### 链身份

每个 STR-NFR 节点携带链身份（平台链 / 业务链 / 共享），影响：
1. 分层约束表中 `约束对象` 字段的取值
2. 约束资产的作用域加载
3. Step End 分链统计摘要

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（STR-NFR 架构末级节点 ≥1 个 / 有反馈的 SysReq-NFR 节点）
2. 上游追溯链（`02-*.md`（相关方需求架构文档）STR-NFR 节点正文块 / 链身份 / 需求描述 / 初始指标 / 所属非功能大类 / 变更影响声明）
3. 已有 SR-NFR 架构资产（`05-*.md`（系统需求架构文档）树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`22-eos-nfr-taxonomy.md` 三层资产：已确认分类 / 具体需求模板 / 候选条目）
5. `05-*.md`（系统需求架构文档）中功能部分的 SR-F（功能性系统需求）节点 ID 清单——用于分层约束表 `约束对象=SR-F` / `约束对象=CU` 时填写具体功能、配置单元或引擎能力线索
6. 重入反馈

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 STR-NFR 节点选中 + 无待处理反馈的 SysReq-NFR 节点 | 输出"无待处理对象"→ **退出** |
| STR-NFR 节点状态不是 `可以NFR设计` / `待补充NFR设计` | 输出状态分布，列出不合格节点 → **退出** |
| 选中 STR-NFR 节点 >5 个 | 输出"本轮 N 个节点，建议分批" → 人类确认后继续，不强制退出 |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 SysReq-NFR 节点 | 本轮先处理全部有反馈的 SysReq-NFR 节点 |
| 存在 `待确认方案` + 反馈区为空的 SysReq-NFR 节点 | 不阻塞新 STR-NFR；Step End 时提示 |

#### 第三层：新 STR-NFR 节点路由

SysReq-NFR 反馈处理完后，逐节点处理新设计 → Step 3。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 SysReq-NFR 节点的量化定义或约束表，因此 Step 3 必须基于 Step 2 修改后的当前内容，不得使用 Step 1 加载时的缓存旧状态。

### Step 2 · 方案反馈处理

#### 反馈模型

人类对 SysReq-NFR 方案做两层反馈——条目级为权威留痕，整体确认为快捷补全：

1. **条目级反馈**（权威）：对 SysReq-NFR 节点的独立决策单元（各条量化定义、宿主决策、分层约束表行）逐项给出 `[同意]` / `[修改]` / `[驳回]`。已 `[同意]` 的条目在后续轮次中默认不重做（写保护）；人类可明确再次反馈覆盖。
2. **整体确认**（快捷）：人类说「整体确认」= 对所有**未逐条反馈的条目**自动补 `[同意]`。

**冲突消解**：条目级反馈 > 整体确认。两者并存时，条目级反馈保留原结论不受影响，整体确认仅补未逐条反馈的剩余条目。

#### 处理流程

若本轮存在有反馈的 SysReq-NFR 节点（`待确认方案` + 反馈区有效），逐个处理：

1. **收集条目级反馈**——扫描全部条目级 `[同意]` / `[修改]` / `[驳回]`，记录到条目列表
2. **收集整体确认**——若有「整体确认」，对**无条目级反馈的剩余条目**自动补 `[同意]`（标注来源=`整体确认`）
3. **执行**——逐条目按结论处理：

| 条目结论 | 动作 |
|---------|------|
| `[同意]`（明示或整体确认补全） | 条目维持不变，标记保护 |
| `[修改]`（局部） | 按反馈调整该条目 → 仅重对齐受影响部分 → 标记保护清除 |
| `[驳回]`（局部） | 移除被驳回的量化定义和约束表 → 其余对象保持原结论 → 标记保护清除 |
| `[驳回]`（整体） | 重做 Step 3 → 全部保护标记清除 |

4. **判定 SysReq-NFR 状态**：
   - 全部条目 `[同意]` → STR-NFR → `已NFR设计`；SysReq-NFR → `已验证冻结`
   - 任一条目 `[修改]` 或 `[驳回]` → SysReq-NFR 保持 `待确认方案`
5. 写入 `### 反馈处理记录`（含本轮各条目结论+来源），重置反馈区

**多轮迭代**：上一轮已 `[同意]` 的条目本轮默认不重做。但上游 STR-NFR 修订可能改变已有 SysReq-NFR 对象——AI 在处理修订时先清除受影响条目的保护标记（上游变更优先于人类确认）。

#### 反馈区位置与格式

反馈区位于 `05-*.md` 中每个 SysReq-NFR 节点块的末尾。格式同 wft03-biz 模式。

### Step 3 · NFR 量化约束设计

将 STR-NFR 的模糊期望转化为 SysReq-NFR 的量化约束。

> **设计姿态**：wft01-nfr 已将 NFR 从定性推到架构末级；本步将架构末级推到量化约束——从"期望高性能"到"P99 ≤ 200ms"，从"需要安全"到"OWASP Top 10 全覆盖"。量化约束表是 wft05 消费 NFR 的唯一入口。

#### 3a · 逐条量化定义

消费 STR-NFR 节点，逐条量化定义：

1. 从 22 具体需求模板（@nfr-xxx-REQ-yy）中提取指标集作为量化参考
2. 逐条 STR-NFR 完善量化定义：

   | 条目 | 需求描述 | 量化指标 | 测量方法 | 应用范围 | 优先级 | 模板来源 | 备注 |
   |------|---------|---------|---------|---------|--------|---------|------|
   | STR-NFR-001 | ... | [指标名]: [数值] [单位]（[条件]） | JMeter/安全审计等 | 全局/业务域/SR-F/CU/PA组件 | 高/中/低 | @nfr-xxx-REQ-yy | AI推断/待确认 |

3. 处理模糊指标（如"高性能"）→ L1 补齐规则：可推测则补齐并标注 `AI推断`；无法确定则标注 `待确认 ⚠️`

#### 3b · 三大类归类与 Q1/Q2/Q3 宿主匹配

**归类**：每条 STR-NFR 归入非功能三大类之一：

| 大类 | 覆盖范围 |
|------|---------|
| **质量特性** | 性能、安全、可用性、可靠性等 |
| **环境适应性** | 可扩展性、兼容性、可移植性等 |
| **可实现性** | 可维护性、可测试性、可部署性等 |

不可归类 → 新增评审（维度独立性 / 可量化性 / 架构影响力）→ 全部通过则新增分类，否则留待人类裁定。

**Q1/Q2/Q3 宿主匹配**：

| 判定 | 条件 | 动作 |
|------|------|------|
| **Q1 直接匹配** | SysReq-NFR 资产中已有同类型节点且边界内 | 复用已有宿主 → 追加引用 |
| **Q2 扩展匹配** | 已有同类型节点但边界略超 | 改进-扩展 → 全量重写（查强内聚性） |
| **Q3 新建** | 无同类型宿主 | 新增 SysReq-NFR 节点（查颗粒度） |

#### 3c · 分层约束表标注

每个 SysReq-NFR 5 级节点必须带一张分层约束表（供 wft05-biz / wft05-eng 消费）：

| 约束范围 | 适用链路 | 约束对象 | wft05 消费目标 | 影响级别 | 影响点 | 说明 |
|----------|----------|----------|----------------|---------|-------|------|

字段取值规则：

1. `约束范围`：`全局` / `业务域` / `SR-F` / `CU` / `PA组件`
2. `适用链路`：`biz` / `eng` / `共享`
   - `biz`：约束业务配置单元需求、配置项、配置规则、配置可达性
   - `eng`：约束引擎前后端组件类型、组件边界、接口机制、运行机制
   - `共享`：同时约束 wft05-biz 和 wft05-eng，wft05 两侧均需装配
3. `约束对象`：
   - `全局` → `EOS`
   - `业务域` → 业务域名或能力类名（链身份影响取值）
   - `SR-F` → 具体 SR-F 节点 ID
   - `CU` → 目标引擎 + CU 类型 / 主 CU 能力类型 / 配置单元需求候选
   - `PA组件` → 目标组件类型（如前端配置支持组件、后端配置定义处理组件、后端运行执行组件、前端业务能力支持组件、后端业务能力服务组件）；若具体组件尚未由 wft05-eng 生成，可先标注组件类型
4. `wft05 消费目标`：
   - `wft05-biz:配置单元需求`：进入配置项、配置规则、配置可达性、验收标准
   - `wft05-eng:组件需求`：进入组件类型、组件边界、接口机制、运行约束机制、共享归属
   - `wft05-biz+eng`：两侧均需消费，并在 Step 4 检查一致性
5. `影响级别`：`阻断型` / `非阻断型` / `待判定`
6. `影响点`：`配置项` / `配置规则` / `配置可达性` / `验收标准` / `组件类型` / `组件边界` / `接口机制` / `运行约束机制` / `共享归属`
7. `说明`：一句话解释该 NFR 为什么对该对象产生该级别影响

填写约束：

1. 一行只表达一个约束对象和一个消费目标的关系；一个 NFR 同时约束 biz 和 eng 时写两行或使用 `wft05-biz+eng`
2. 若 `影响级别=阻断型`，则 `影响点` 不可为空
3. 若 `影响级别=待判定`，则 `说明` 必须写明待判原因
4. 当一个 NFR 影响面很大时，优先使用 `全局` 或 `业务域` 行，不强制展开全部 SR-F/CU/PA组件
5. 当 NFR 明确影响引擎实现（性能、安全、可靠性、可观测性、可部署性等）时，必须至少补一行 `适用链路=eng` 的组件约束；若暂不能确定具体组件，`约束对象=PA组件` 可先写组件类型
6. 当 NFR 明确影响业务配置或使用方式（权限、审计、字段脱敏、配置校验、可访问性等）时，必须至少补一行 `适用链路=biz` 的配置单元需求约束
