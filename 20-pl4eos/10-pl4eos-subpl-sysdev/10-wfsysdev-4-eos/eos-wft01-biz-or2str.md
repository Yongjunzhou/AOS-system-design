# 业务需求 → STR-F · 场景定义设计与资产化
**Skill ID**: wft01-biz-or2str

> **概述**：每次执行处理一个角色的 FR-BIZ 类型 OR 条目（≤10 条），STR 反馈优先于新 OR 处理。完成 OR→STR-F 场景定义设计、23/26 资产对准与写回，形成可进入 wft02a 的 STR-F 节点。三步递进：确定场景定义（含场景指标）→ 确定所需信息载体 → 确定每个载体所属产品类型/构件类型并写回 23（产品架构）+ 26（文档定义）。
>
> **输入 → 输出**：一个角色的 FR-BIZ OR 条目（≤10 条）/ 有反馈的 STR-F 节点 → STR-F 节点方案 + 23/26 资产对准与写回（状态=`待分解分配`）+ OR 条目推进为 `已分配`
>
> **适用场景**：瀑布式系统设计第1步 · 业务需求路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.5 | **修订**：2026-06-23

---

## 一、职责定位

本 Skill 负责：

1. 接收人类选中的一批同角色 FR-BIZ OR 条目（≤10 条）
2. 将 OR 分配到已有 STR-F 节点（Q1/Q2）或聚合为新的端到端场景（Q3）
3. 对已有 STR-F 节点的修订，执行多 OR 收敛检测——四关系判定 + 来源升级 + 矛盾记录
4. 形成 STR-F 场景定义方案（含信息载体清单 + 场景指标 + 推理依据）
5. 对信息载体执行 23-Q1/Q2/Q3 资产对准（十类节点归属规则）和 26-Q1/Q2/Q3 文档定义对准
6. 全 Q1 时自动写回 23/26 资产；Q2/Q3 时形成资产建议等人类确认后写回
7. 写回完成后推进 STR-F 节点到 `待分解分配`，对应 OR 到 `已分配`

本 Skill 不负责：

1. 引擎需求（FR-ENG → `wft01-eng-or2str`）
2. 非功能需求（NFR → `wft01-nfr-or2str`）
3. 下游 wft02a 的详细分解展开
4. 业务人员确认——这是人类线下活动

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（同角色 FR-BIZ OR 条目 ≤10 条 / 有反馈的 STR-F 节点）
2. 上游追溯链（`01-*.md` 树画像 / 状态队列 / 节点索引）
3. 已有 STR-F 架构资产（`02-*.md` 树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`23` 产品架构 + `26` 文档定义）
5. 重入反馈

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 人类选中的 OR 分属多个角色 | 输出角色分布，要求只选一个角色 → **退出** |
| 同角色 OR >10 条 | 输出超限提示，要求分批 → **退出** |
| OR 类型含 FR-ENG 或 NFR | 输出类型分布，要求分路径 → **退出** |
| 无 OR 选中 + 无待处理反馈的 STR-F 节点 | 输出"无待处理对象"→ **退出** |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 STR-F 节点 | 本轮先处理全部有反馈的 STR-F 节点 |
| 存在 `待确认方案` + 反馈区为空的 STR-F 节点 | 不阻塞新 OR；Run End 时提示 |

#### 第三层：新 OR 路由

STR-F 反馈处理完后，逐条处理新 OR → Phase A 宿主匹配。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 STR-F 的场景职责边界陈述或信息载体清单，因此 Step 3 Phase A 宿主匹配时**必须基于 Step 2 修改后的 STR-F 节点当前内容**，不得使用 Step 1 加载时的缓存旧状态。

### Step 3 · 新增场景定义设计

分三个阶段。

#### Phase A：OR → 目标 STR-F 分组

1. 沿用 `#role-list` 中已确认角色归属
2. 对每条 OR 独立执行 **Q1/Q2/Q3 宿主匹配**：
   - Q1（直接匹配）：OR 的功能描述与已有场景定义语义一致
   - Q2（扩展匹配）：OR 是已有场景的子集或超集边缘
   - Q3（新增）：无可用宿主
3. 分组结果：Q1/Q2 → 归入已有 STR-F；Q3 → 进入待聚合池
4. 待聚合池按**业务域 + 触发相似性**分组

#### Phase B：修订已有 STR-F（四关系检测）

**边界匹配检查**：先检查 OR 是否在 STR-F 节点边界内。

| 结果 | 判定 | 处理 |
|------|------|------|
| 边界内 | OR 描述的职责未超出当前场景职责边界陈述的范围 | 正常进入逐载体四关系检测 |
| 边缘 | OR 处于边界边缘，部分职责与当前终点相邻但不完全超出 | 增量标记 `[边界扩展]`，更新场景职责边界陈述，重新执行 2c 边界检查（完整性/一致性/紧凑性/粒度匹配） |
| 超出 | OR 描述的职责明确超出当前终点 | 增量标记 `[边界扩展]`，更新场景职责边界陈述 + 按 Phase C 2b 追踪自然终点，重新执行 2c 边界检查；若边界扩展导致与原 STR-F 粒度不一致，标记 `[推断]` 供人类裁决是否拆分 |

**Phase A Q2 与 Phase B 边界结果不一致时的处理**：Phase A 的 Q2 是候选筛选（"大概率归这个 STR-F"），Phase B 的边界检查是精确匹配。当 Phase B 检出"超出"比 Q2 预期的更严重时，AI 不得自行回退 Phase A 重选宿主——继续完成 Phase B 全部处理（四关系检测 + 边界陈述更新 + 2c 重检），标记 `[推断]`，将边界冲突显性化提交人类方案审阅。原因：拆分场景是业务语义决策，不应由 AI 单方面判断。

**逐载体四关系检测**：比对锚点"角色+业务对象+职责"三元组。

| 关系 | 判定 | 处理 |
|------|------|------|
| 确认 | 新 OR 与既有载体一致 | 来源升级（`OR隐含`→`OR明示`），增量标记 `[确认]` |
| 补充 | 新 OR 提到未识别载体 | 新增到清单，来源=`OR明示`，增量标记 `[新增]` |
| 矛盾 | 载体形态/范围与既有不同 | 追加矛盾记录，增量标记 `[矛盾]`，等人类裁决 |
| 无新增 | 完全被覆盖 | 仅追加 OR 引用 |

来源类型升级规则（只升不降）：`OR隐含`→`OR明示`、`同类场景推断`→`OR明示`、`资产线索推断`→`OR明示`。

矛盾记录追加式不覆盖，存在未解决矛盾时不得推进到写回。

#### Phase C：新建 STR-F

**1. 确定业务域**：从角色类型映射 23 资产业务域。

**2. 三步判断法定义端到端业务闭环**：

2a. **识别触发**：提取"角色 + 动作 + 业务对象"。组内触发不匹配时移除或标注为内部补充。

2b. **追踪自然终点**：端到端约束——OR 语义覆盖的终点即追踪终点，不得在终点之后扩展。跨角色/跨域截断并标注交接点。

2c. **边界检查**：完整性（触发到终点完成一件完整的事）/ 一致性（同角色同业务域）/ 紧凑性（不混多个不连续职责段→拆分）/ 粒度匹配（与同层已有一致）。

输出**场景职责边界陈述**：`「<角色>」在「<业务域>」中「从触发到终点的职责概括」`。

**3. 推理依据**：业务常识推理 / 系统设计推理 / 企业治理推理，编号引用。

**4. 派生场景定义名称**：与同层已有节点粒度和命名风格一致。

**5. 完善 STR-F 要素与资产对准**：

信息载体标注字段：
- 载体名称 / 归属链 / 文档性质 / 产生方式 / 一句话职责 / 来源 OR / 来源类型 / 确认状态
- **26 资产方案级判断**：`26-Q1` / `26-Q2` / `26-Q3` / `未检查`
- **23 产品类型** / **23 构件类型** / **23 判定**（23-Q1/Q2/Q3）/ **23 资产建议**（Q2/Q3时）
- **关系对象**（按信息载体类型记录）

**十类信息载体归属对准规则**（AI 判定 23-Q 的依据）：

| 节点类型 | 归属锚点 | 归属规则 |
|------|---------|---------|
| 执行业务链节点 | 被执行/创建/处理/流转的产品数据 | 归属到该产品数据所在产品类型和构件类型；归属锚点是业务对象 |
| 管理业务链节点 | 被管理的业务对象/过程/控制对象 | 归属到主产品类型下的项目管理构件，归入管理业务 WBS |
| 个人绩效链节点 | 个人对业务/项目的贡献/达成/质量/效率 | 归属到项目管理构件，作为个人绩效 WBS |
| 组织绩效链节点 | 组织/团队/部门/项目的贡献/达成/质量/效率 | 归属到项目管理构件，作为组织绩效 WBS |
| 规则知识链节点 | 所附属的执行业务链节点 | 继承所附属执行节点的产品类型和构件类型 |
| 资源需求链节点 | 支持维资源提供方 | 按资源类型归属到资源提供类产品类型和构件类型 |
| 风险合规链节点 | 所附属的执行业务链节点 | 继承所附属执行节点，内容为事先/事中/事后规则 |
| 状态事件链节点 | 所记录的执行业务链节点 | 继承所附属执行节点；跨多业务对象的状态汇总转入管理业务链 |
| 协同交互链节点 | 公共协同或目标承接方 | 归属到 `流程与IT → 公共业务构件 → 协同交互构件`（23 资产预设路径）；若 23 中不存在则 23-Q3 形成新增建议 |
| 场景指标链节点 | 被度量对象 | 按被度量对象确定产品类型和构件类型 |

**信息载体关系记录要求**：

| 节点类型 | 必须记录的关系 |
|------|---------------|
| 执行业务链节点 | 被执行产品数据 / 业务对象 |
| 管理业务链节点 | 被管理对象、项目管理构件、管理业务 WBS |
| 个人绩效链节点 | 评价的个人/角色、项目管理构件、个人绩效 WBS |
| 组织绩效链节点 | 评价的组织对象、项目管理构件、组织绩效 WBS |
| 规则知识链节点 | 所附属执行业务链节点、附属文档类型 |
| 资源需求链节点 | 需求发起执行节点、资源类型、资源提供产品类型/构件类型 |
| 风险合规链节点 | 所附属执行业务链节点、事先/事中/事后规则类型 |
| 状态事件链节点 | 所记录执行业务链节点、状态/事件/轨迹/日志类型 |
| 协同交互链节点 | 来源执行节点、目标承接方（归属 `流程与IT → 公共业务构件 → 协同交互构件`）、请求/通知/待办/回执/接口消息类型 |
| 场景指标链节点 | 被度量对象、指标口径、数据来源 |

**跨链归属冲突消解**：当一个信息载体可归入多条链时，按两步消解：

**Step 1 — 判定冲突是否真实**：比对候选链在归属规则表中的 23 归属落点。若落点相同（如规则知识链+状态事件链均继承附属执行节点；管理业务链+组织绩效链均归属项目管理构件），则不是真实冲突——以主要职责链标注归属，次级链在关系记录中注明（如"同时记录状态流转""同时作为组织绩效评价依据"）。

**Step 2 — 真实冲突时按主要职责判定**：当候选链指向不同 23 归属落点时，AI 自问"如果没有这个载体，哪种业务能力会最先受损？"答案指向的链即为归属链。无法判定时（两条链同等重要），标记 `[推断]` + 写入推理依据，供人类裁决。常见冲突对：

| 冲突对 | 判定 |
|--------|------|
| 协同交互链 vs 状态事件链 | 通知/待办/回执等以协同为第一目的 → 协同交互链；状态记录是副产品 → 关系记录中注明 |
| 执行业务链 vs 管理/绩效链 | 业务流程直接依赖的产出 → 执行业务链；管理评价是后续行为 → 关系记录中注明 |

场景指标标注字段（场景指标是功能需求的一部分，定义随 STR-F 节点写入 `02-*.md`，不涉及 22 资产）：
- 指标名称 / 指标类别 / 指标含义 / 主要数据来源 / 来源 OR / 来源类型 / 确认状态

STR-F 完整要素：
- 定义 / 业务域 / 产品类型 / 用户角色 / 视角类型=`场景` / 功能说明
- 信息载体清单（含 26/23 对准） / 场景指标
- 推理依据与补全说明 / 十链覆盖自查 / 承接的 OR 末级
- 符合性分析（`待 wft02 生成`）/ 设计指导（`待 wft02 生成`）

**6. 十链覆盖自查**：逐链确认每条链是否至少有一个信息载体或标注合理空置原因。自查结果写入 STR-F 节点。

### Step 5 · 资产建议反馈

存在 Q2 或 Q3 时，在 `### 资产建议反馈区` 形成正式建议：

```text
[操作] 同意 / 修改 / 驳回

[23 资产建议] / [26 资产建议]
```

| 操作 | 行为 |
|------|------|
| `[同意]` | 执行写回 → 进入 Step 6 |
| `[修改]` | 调整建议 → 重新等待 |
| `[驳回]` | 归属另有目标 → 重新定位；原子单元有问题 → 回退 Phase B/C 重做 |

- `02` STR-F 节点推进为 `待分解分配`
- `01` 对应 OR 条目推进为 `已分配`
- OR→STR-F→23/26 引用链记录
- 清空反馈区待办信号
