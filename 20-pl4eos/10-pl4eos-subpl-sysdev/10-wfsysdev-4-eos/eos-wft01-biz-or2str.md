# 业务需求 → STR-F · 场景业务设计与资产化
**Skill ID**: wft01-biz-or2str

> **概述**：每次执行处理一个角色的 FR-BIZ 类型 OR 条目（≤10 条），STR 反馈优先于新 OR 处理。完成 OR→STR-F 场景方案设计、23/26 资产对齐与写回，形成可进入 wft02-biz 的 STR-F 节点。三阶段递进（Step 3 内）：OR 轻量提取与路由 → 修订已有 STR-F（含要素提取与比对）或新建 STR-F（含要素提取与组装）→ Step 4 资产写回与落账。
>
> **输入 → 输出**：一个角色的 FR-BIZ OR 条目（≤10 条）/ 有反馈的 STR-F 节点 → STR-F 节点方案 + 23/26 资产对齐与写回（状态=`可以分解分配`）+ OR 条目推进为 `已分配`
>
> **适用场景**：瀑布式系统设计第1步 · 业务需求路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.26 | **修订**：2026-06-26

## 一、职责定位

### 本 Skill 目的与目标

> **主要目的**：取信于相关方——「你听懂了我的需求」。把非结构化的 FR-BIZ OR 转化为结构化的 STR-F 声明——识别场景职责边界、按场景部件分组的输出文档集、每个文档需要什么支撑引擎。
>
> **目标**：名称级识别，不脑补。AI 推断标注 `[推断]`，架构决策（拆分场景/更换宿主 STR）提交人类。STR-F 的场景职责边界陈述和场景部件分组是 wft02-biz 展开的输入——wft01-biz 不做 IPO 展开和指标定义。

本 Skill 负责：

1. **接收**——接收人类选中的一批同角色 FR-BIZ OR 条目（≤10 条），校验类型/数量/内容适用性，不合格则说明原因并退出
2. **匹配**——按用户角色搜索已有 STR-F 节点——同角色的 OR 归入匹配的已有 STR-F，无已有节点则新建 STR-F
3. **收敛**——将本轮新 OR 的要素与 STR-F 已有要素逐项比对（含场景部件级与文档级），判定关系并执行对应动作——确认则升级来源、补充则进入定义（新文档归入已有场景部件或触发新场景部件）、矛盾则记录待人类裁决、无新增则仅加引用
4. **定义**——基于 OR 语义还原场景职责边界，形成 STR-F 场景业务——场景职责陈述 + 场景部件分组（每个场景部件含名称/职责+有序输出文档清单，含文档性质/产生方式/支撑引擎）+ 场景级 QTC 指标 + 推理依据
5. **对齐**——对要素逐项判定与 23 资产（产品路径归属）和 26 资产（文档定义）的 Q1/Q2/Q3 对齐关系
6. **写回**——人类确认方案后统一写回 23/26 资产（Q1/Q2/Q3 均直接写入，不需二次确认）
7. **推进**——人类确认时 STR-F 节点推进为 `可以分解分配`（wft02-biz 可即时进入）；资产落账后对应 OR 推进为 `已分配`

本 Skill 不负责：

1. 引擎需求（FR-ENG → `wft01-eng-or2str`）
2. 非功能需求（NFR → `wft01-nfr-or2str`）
3. 下游 wft02-biz 的详细分解展开
4. 业务人员确认——这是人类线下活动

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（同角色 FR-BIZ OR 条目 ≤10 条 / 有反馈的 STR-F 节点）
2. 上游追溯链（`01-*.md` 树画像 / 状态队列 / 节点索引）
3. 已有 STR-F 架构资产（`02-*.md` 树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`23` 产品架构 + `26` 文档定义）
5. 重入反馈

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 人类选中的 OR 分属多个角色 | 输出角色分布，要求只选一个角色 → **退出** |
| 同角色 OR >10 条 | 输出超限提示，要求分批 → **退出** |
| OR 类型含 FR-ENG 或 NFR | 输出类型分布，要求分路径 → **退出** |
| 无 OR 选中 + 无待处理反馈的 STR-F 节点 | 输出"无待处理对象"→ **退出** |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 STR-F 节点 | 本轮先处理全部有反馈的 STR-F 节点 |
| 存在 `待确认方案` + 反馈区为空的 STR-F 节点 | 不阻塞新 OR；Step End 时提示 |

#### 第三层：新 OR 路由

STR-F 反馈处理完后，逐条处理新 OR → Phase A 宿主匹配。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 STR-F 的场景职责边界陈述、场景部件分组或要素清单，因此 Step 3 Phase A 宿主匹配时**必须基于 Step 2 修改后的 STR-F 节点当前内容**，不得使用 Step 1 加载时的缓存旧状态。

3. `23-eos-output-architecture.md` 资产全貌层

### 按需加载线索

- **对象线索**：按 OR 条目 ID 或 STR-F 节点 ID 在索引中定位必须读取的正文块
- **追溯线索**：按追溯关系读取上下游相关块，不复制对方正文
- **结构线索**：按同角色/同业务域/同层级筛选候选 STR-F 块，用于宿主匹配（匹配已有 / 需新建）
- **语义线索**：按 OR 中的角色/动作/业务对象/流程阶段/约束条件/异常条件/指标口径匹配资产候选分片

- **扩大读取触发**：索引缺失/候选冲突/资产全貌不足以判断边界/追溯关系断裂时，说明原因后扩大

## 四、操作流程

### Step 布局总览

| Step | 名称 | 产出 | 写资产 |
|------|------|------|--------|
| Start | 前置校验与路由 | 就绪 / 结束 | — |
| 1 | 设计上下文加载 | OR / STR / 资产全景 | — |
| 2 | 方案反馈处理 | 调整后的 STR-F 方案 | `01/02` |
| 3 | 场景方案设计 | STR-F 节点方案（含 23/26 对齐） | `01/02` |
| 4 | 资产写回与落账 | 23/26 引用写回 + OR→已分配 + 资产写回记录 | `23/26/01/02` |
| Step End | 运行结束输出 | 人类下一步操作提示 | — |

### STR-F 节点生命周期

#### 推进态（按顺序）

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `待确认方案` | 方案已生成或修订，等待人类反馈 | Step 3 完成 |
| `可以分解分配` | 方案已确认，wft02-biz 可启动 BA 展开 | 人类反馈 `[同意]` 或整体确认 |
| `在分解分配` | wft02-biz 已锁定此 STR-F，正在展开 BA | wft02-biz 写入活动锁 |
| `已分解分配` | BA 展开完成，STR-F 消费版本与当前版本一致 | wft02-biz 处理完成且未检测到上游新版本 |

人类确认后，两条线并行推进、互不阻塞：
- **资产线**（本 Skill）：Step 4，将要素引用写回 23/26 资产文件。资产写回完成与否不改变 STR-F 状态。
- **BA 线**（wft02-biz）：读 `可以分解分配`，展开 BA 节点。不依赖 23/26 资产文件中的写回结果。

`在分解分配` 和 `已分解分配` 由 wft02-biz 写入和推进。本 Skill（wft01-biz）写入的范围为 `待确认方案` / `可以分解分配` / `待补充分解分配`，并在修订时更新版本号。

#### 回退态（上游修订触发）

上游修订 = 本 Skill 因新 OR 对已有 STR-F 执行 Step 3 Phase B。修订后版本号递增，状态按原状态分流：

| 原状态 | 修订后状态 | 含义 |
|--------|----------|------|
| `可以分解分配` | `可以分解分配`（版本递增） | 未被下游消费过，版本升级不影响任何 BA |
| `在分解分配` | `在分解分配`（版本递增） | wft02-biz 锁仍有效，重入时检测版本差，按变更类型增量处理 |
| `已分解分配` | **`待补充分解分配`** | BA 已不完整，需 wft02-biz 重新介入补全 |

- **`待补充分解分配`**：上游 STR-F 修订后，已消费的 BA 可能不完整。wft02-biz 进入后按变更影响声明处理，处理完成后若版本一致 → 推进回 `已分解分配`

### Step 3 · 场景方案设计

先建立 STR-F 架构认知和要素提取规则，再进入三阶段操作流程。

#### §3.1 STR-F 架构说明

STR-F 节点以场景为组织单元，三层结构，每层有确定的支撑引擎和 QTC 指标：

```
场景名称（节点名称）        ← @engine-场景引擎 + 场景级 QTC
  └── 场景部件                ← 部件级引擎（表单/流程/指标/工单） + 场景部件级 QTC
        └── 输出文档         ← 具体业务引擎 + 文档级 QTC
```

**场景业务**（L4）：由 `@engine-场景引擎` 支撑，定义端到端业务闭环。场景级 QTC 指标为该场景的整体指标——独立的子业务，由 `@engine-指标引擎` 支撑，度量维度 = 各场景部件 QTC 汇总 + 场景特有指标。本 Skill 标注度量维度，具体指标定义由 wft02-biz 展开。

**场景部件**（L5）：由部件级引擎（表单/流程/指标/工单）支撑，按产品数据对象分组输出文档。场景部件级 QTC 指标为该场景部件的整体指标——独立的子业务，由 `@engine-指标引擎` 支撑，度量维度 = 场景部件下各文档 QTC 汇总 + 场景部件特有指标。本 Skill 标注度量维度，具体指标定义由 wft02-biz 展开。

**输出文档**：由具体业务引擎支撑（`@engine-表单引擎` / `@engine-流程引擎` / `@engine-台账引擎` 等），由 AI 基于文档职责语义判定。将在 wft02-biz Phase B 中展开为 Bn IPO，归属到 23 产品类型和构件类型。每个输出文档包含以下要素：

- **支撑引擎**：25 资产中的引擎模型引用
- **生命周期状态**：每个文档有其状态流转路径。本 Skill 识别该文档存在生命周期，标注待设计，具体状态定义由 wft02-biz 展开
- **支持信息**：该文档生成所需输入——均不独立成要素，作为文档属性标注，继承所附属输出文档的 23 归属：
  - 规则知识——按什么规则生成？（填写模板、校验规则、审批规则、判断标准、知识库引用）
  - 风险合规——有什么合规约束？（审批留痕、授权校验、异常处置方式、审计证据要求）
  - 资源工具——需要什么工具？（CAX 工具、AI 工具、模板、专业能力、作业指南）
- **交互信息**：该文档生成后触发的后续交互，类型包括但不限于：通知 / 待办 / 回执 / 状态 / 告警 / 异步消息(MQ) / 同步调用(API)。交互不作为当前业务域的独立要素，具体交互由 wft02-biz 展开
- **文档级 QTC 指标**：该文档的质量/及时性/完整性等指标——独立的子业务，由 `@engine-指标引擎` 支撑。本 Skill 标注度量维度和责任人/组织，具体指标定义由 wft02-biz 展开

**STR-F 节点完整要素**：

- 定义 / 业务域 / 产品类型 / 用户角色 / 视角类型=`场景` / 功能说明
- **场景部件分组**（每个场景部件含：场景部件名称/场景部件职责描述/支撑引擎（部件级引擎，AI 判定）/场景部件级 QTC + 场景部件下输出文档清单，每个文档含：支撑引擎 + 文档级 QTC + 生命周期 + 支持信息 + 交互信息（交互类型/内容/判定依据） + 26/23 对齐）
- 场景级 QTC 指标（待 wft02 展开为子业务）
- 推理依据与补全说明 / 覆盖自查 / 承接的 OR 末级
- 符合性分析（`待 wft02 生成`）/ 设计指导（`待 wft02 生成`）
- STR-F 版本号（初始 v1，每次因 OR 修订递增）/ 变更影响声明

#### §3.2 要素提取规则

§3.4 Phase B（修订已有 STR-F）与 §3.5 Phase C（新建 STR-F）均需从 OR 文本中提取要素。以下规则为二者共用。

**1. 场景部件识别**：AI 基于 OR 语义 + 业务常识，按以下规则将输出文档归入场景部件：

| 规则 | 说明 |
|------|------|
| **同产品数据对象** | 围绕同一业务对象的文档归入同一场景部件——采购申请单 + 采购申请审批 + 采购申请查询 → "采购申请管理" |
| **同生命周期阶段** | 同一对象的创建/审批/查询/归档在一个场景部件下闭环，不跨阶段混杂 |
| **不同对象分属不同场景部件** | 采购申请 vs 供应商评估 → 分属两个场景部件 |
| **单文档场景部件** | 当前仅识别出一个文档的对象，仍独立建场景部件——后续 OR 可能补充同对象文档 |

每个场景部件节点标注：场景部件名称 / 场景部件职责描述 / 支撑引擎（部件级引擎，由 AI 基于该场景部件内输出文档的职责语义判定，如 `@engine-表单引擎` / `@engine-流程引擎` 等）。

**2. 输出文档标注**：对识别出的输出文档，逐项判定生命周期、支持信息、交互信息、文档级 QTC 和 26/23 对齐，标注来源类型、推理依据和确认状态。

**要素标注字段**：

场景部件级：
- 场景部件名称 / 场景部件职责描述 / 支撑引擎（部件级引擎，AI 判定）/ 来源类型 / 确认状态
- 场景部件级 QTC 指标：度量维度 → 待 wft02 展开为子业务（引擎=`@engine-指标引擎`）

文档级（每个场景部件下文档）：
- 要素名称 / 文档性质 / 产生方式 / 一句话职责 / 来源 OR / 来源类型 / 确认状态
- **所属场景部件**：引用场景部件名称
- **支撑引擎**：25 资产中的引擎模型引用（如 `@engine-表单引擎` / `@engine-流程引擎`），由 AI 基于文档职责语义判定
- **判定依据**：OR 语义线索 + 业务常识推理
- 文档级 QTC 指标：度量维度（质量/及时性/完整性）/ 责任人 / 责任组织 → 待 wft02 展开为子业务（引擎=`@engine-指标引擎`）
- 生命周期状态（待设计，具体状态由 wft02-biz 展开）
- 支持信息：规则知识内容 / 风险合规内容 / 资源工具内容
- 交互信息（每个交互标注）：交互类型（通知/待办/回执/状态/告警/异步消息(MQ)/同步调用(API)）/ 交互内容 / 判定依据。判定依据=OR 语义线索 + 业务常识推理
- **26 资产方案级判断**：`26-Q1` / `26-Q2` / `26-Q3` / `未检查`
- **23 产品类型** / **23 构件类型** / **23 判定**（23-Q1/Q2/Q3）/ **23 资产建议**（Q2/Q3时）

**3. 23/26 对齐判定规则**

Phase B 四关系检测和 Phase C 组装 STR-F 时，对每个输出文档要素逐项判定 23 和 26 对齐关系。判定基于 Step 1 已加载的 23/26 资产全貌层进行语义匹配，不需重新读取。

**23 资产对齐判定**（产品架构归属）：

| 判定 | 条件 | 动作 |
|------|------|------|
| **23-Q1** | 产品类型 + 构件类型均在 23 资产中存在，且当前文档的职责范围未超出已有定义 | 直接引用已有产品类型/构件类型路径 |
| **23-Q2** | 产品类型存在，但构件类型需扩展范围；或构件类型存在但产品类型归属需调整 | 引用已有路径 + 输出扩展建议（`[23-Q2]` 标注 + 建议内容） |
| **23-Q3** | 产品类型或构件类型在 23 资产中不存在 | 输出新建建议（产品类型名称/构件类型名称/归属路径 + `[23-Q3]` 标注） |

判定锚点：输出文档的**业务域 + 数据对象 + 文档性质**，与 23 资产中已有产品类型/构件类型的语义匹配。

**26 资产对齐判定**（文档定义）：

| 判定 | 条件 | 动作 |
|------|------|------|
| **26-Q1** | 26 资产中已存在同名/同义文档定义，且业务域+产品类型一致 | 直接引用已有文档定义块 |
| **26-Q2** | 26 资产中存在相似文档定义（同业务域/相似职责），但名称/字段/范围需扩展 | 引用已有块 + 输出扩展建议（`[26-Q2]` 标注 + 建议内容） |
| **26-Q3** | 26 资产中无匹配文档定义 | 输出新建文档定义块的建议（名称/职责/归属业务域 + `[26-Q3]` 标注） |

判定锚点：输出文档的**名称 + 一句话职责 + 业务域**，与 26 资产中已有文档定义块的语义匹配。

**Q3 的新建建议内容**（23 和 26 通用）：建议名称 / 建议归属路径（23）或归属业务域（26）/ 建议定义（一句话职责）/ 与当前要素的关系说明。Q3 建议跟随要素写入 02 资产中 STR-F 节点，待人类确认后由 Step 4 统一写入目标资产文件。

#### §3.3 Phase A：OR 轻量提取与路由

1. **逐条轻量提取**：对每条新 OR，提取角色/业务域/触发动作（角色+动作+业务对象），够路由用即可，不做完整要素提取
2. 沿用 `#role-list` 中已确认角色归属
3. 对每条 OR 独立执行宿主匹配：能否归入已有 STR-F？
   - **匹配**：OR 的角色+业务域与已有 STR-F 节点语义相关 → 归入该 STR-F，进入 Phase B
   - **新增**：无可用宿主 → 进入待聚合池，进入 Phase C
4. 待聚合池按**业务域 + 触发相似性**分组，输出分组清单（`{业务域, 触发描述, OR列表}`）供 Phase C 使用

#### §3.4 Phase B：修订已有 STR-F

按 §3.2 从本轮 OR 中提取要素，然后：

**1. 边界匹配检查**：检查 OR 是否在 STR-F 节点边界内。

| 结果 | 判定 | 处理 |
|------|------|------|
| 边界内 | OR 描述的职责未超出当前场景职责边界陈述的范围 | 正常进入逐要素四关系检测 |
| 边界扩展 | OR 描述的职责部分或全部超出当前场景职责边界陈述的范围 | 增量标记 `[边界扩展]` → 更新场景职责边界陈述 → 按 §3.5 步骤 2b 追踪自然终点 → 重新执行边界检查（完整性/一致性/紧凑性/粒度匹配）。若边界扩展导致与原 STR-F 粒度不一致，标记 `[推断]` 供人类裁决是否拆分 |

**Phase A 匹配与 Phase B 边界结果不一致时的处理**：Phase A 的匹配是候选筛选（"大概率归这个 STR-F"），Phase B 的边界检查是精确匹配。当 Phase B 检出边界扩展时，AI 不得自行回退 Phase A 重选宿主——继续完成 Phase B 全部处理（四关系检测 + 边界陈述更新 + 边界重检），标记 `[推断]`，将边界冲突显性化提交人类方案审阅。原因：拆分场景是业务语义决策，不应由 AI 单方面判断。

**2. 逐要素四关系检测**：将提取的要素与 STR-F 已有要素清单逐项比对，比对锚点"角色+业务对象+职责"三元组。对文档级要素按以下四关系判定；对场景部件级要素，新增/调整场景部件需按 §3.2 §1 场景部件识别规则判定归属。

| 关系 | 判定 | 处理 |
|------|------|------|
| 确认 | 新 OR 与既有要素一致 | 来源升级（`OR隐含`→`OR明示`），增量标记 `[确认]` |
| 补充 | 新 OR 提到未识别要素 | 新增到清单（文档归入已有场景部件或触发新场景部件），来源=`OR明示`，增量标记 `[新增]` |
| 矛盾 | 要素形态/范围与既有不同 | 追加矛盾记录，增量标记 `[矛盾]`，等人类裁决 |
| 无新增 | 完全被覆盖 | 仅追加 OR 引用 |

来源类型升级规则（只升不降）：`OR隐含`→`OR明示`、`同类场景推断`→`OR明示`、`资产线索推断`→`OR明示`。

矛盾记录追加式不覆盖，存在未解决矛盾时不得推进到写回。

**3. 下游变更影响声明**：四关系检测完成后，按本轮 OR 对 STR-F 的实际变更范围判定变更类型——仅新增要素（含场景部件内文档，不含新场景部件或场景部件重组）且边界未变 → `增量型`；已有要素定义/范围/归属/场景部件分配修订但边界未变 → `修订型`；边界检出扩展、要素被移除、场景部件新增/合并/拆分、或场景粒度需拆分 → `结构型`。将变更影响声明写入 STR-F 节点末尾（格式与完整规则见 §四·附 变更传播协议）。

#### §3.5 Phase C：新建 STR-F

按 §3.2 从本轮 OR 中提取要素，然后：

**1. 确定业务域**：从角色类型映射 23 资产业务域。

**2. 三步判断法定义端到端业务闭环**：

2a. **识别触发**：提取"角色 + 动作 + 业务对象"。组内触发不匹配时移除或标注为内部补充。

2b. **追踪自然终点**：端到端约束——OR 语义覆盖的终点即追踪终点，不得在终点之后扩展。跨角色/跨域截断并标注交接点。

2c. **边界检查**：完整性（触发到终点完成一件完整的事）/ 一致性（同角色同业务域）/ 紧凑性（不混多个不连续职责段→拆分）/ 粒度匹配（与同层已有一致）。

输出**场景职责边界陈述**：`「<角色>」在「<业务域>」中「从触发到终点的职责概括」`。

**3. 推理依据**：业务常识推理 / 系统设计推理 / 企业治理推理，编号引用。

**4. 派生场景业务名称**：与同层已有节点粒度和命名风格一致。

**5. 组装 STR-F**：将 §3.2 提取的要素按场景部件分组填入 STR-F 节点，补全定义/业务域/产品类型/用户角色/视角类型/功能说明等，形成完整 STR-F 节点（完整要素见 §3.1 STR-F 节点完整要素）。

**6. 覆盖自查**：逐场景部件确认场景部件名称/职责/支撑引擎和场景部件级 QTC 已标注；逐场景部件下逐文档确认所属场景部件、支撑引擎、文档级 QTC 度量维度、生命周期状态、支持信息和交互信息（交互类型，或合理空置原因）已标注。确认场景级 QTC 指标已标注。自查结果写入 STR-F 节点。

### 四·附、STR-F 变更传播协议

下游 wft02-biz 消费 STR-F 后生成 BA 节点。当 STR-F 因新 OR 修订而变更时，已生成的 BA 节点面临同步问题。本协议定义 STR-F → BA 的变更分类、通知格式和消费追溯，使下游可增量响应而非全量重做。

#### 变更类型分类

| 变更类型 | 判定标准 | 实例 | 对下游影响 |
|---------|---------|------|----------|
| **增量型** | 仅新增要素（含场景部件内文档，不含新场景部件或场景部件重组），已有要素的定义/范围/归属/场景部件分配均未改动；场景边界未变 | 新 OR 补充了一个要素 | 已有 BA 节点不受影响，仅需展开新要素 |
| **修订型** | 已有要素的定义/范围/归属/场景部件分配被修改；场景边界未变 | IC-02 适用范围从"部门内"改为"跨部门"；或文档在场景部件间迁移 | 对应 BA 节点需局部调整 |
| **结构型** | 场景边界检出扩展、要素被从清单移除、场景部件新增/合并/拆分、或场景粒度不一致需拆分 | 边界扩展导致原 STR-F 涵盖了两个不连续职责段；新 OR 触发新场景部件 | BA 结构可能重塑，需人类裁决后方可处理 |

判定优先级：结构型 > 修订型 > 增量型。即：只要边界有变化或场景部件结构重组，不论要素如何，一律判定为结构型。

#### 变更影响声明格式

每次修订已有 STR-F 后，在 STR-F 节点末尾写入（`02-*.md` 中 STR-F 节点块内）：

```text
### 变更影响声明

STR-F 版本：v{N}（本次修订前为 v{N-1}）
本轮变更类型：增量型 / 修订型 / 结构型

变更项：
- [新增] / [改进] / [矛盾] / [边界扩展] / [场景部件新增] / [场景部件调整] / [文档迁移] ...

下游已消费此 STR-F 的 BA 节点：
  <BA节点ID列表>（如无则写"尚未被下游消费"）

对已消费 BA 节点的影响：
  受影响节点：<BA节点ID + 受影响原因>
  不受影响节点：<BA节点ID>
  待新增 BA 节点：<新要素/新场景部件对应的待展开 BA 说明>

建议 wft02-biz 动作：
  - 增量型：增量展开新增要素即可（按场景部件分组），已有 BA 节点不动
  - 修订型：修订受影响的 BA 节点 → 标记 [来自上游变更]；含场景部件分配变更时同步调整 Phase D 编排序列
  - 结构型：输出影响范围，等人类裁决后再处理；含场景部件新增/合并/拆分时 wft02-biz 需新建或重组 Phase C 场景部件
```

#### 版本标记与消费追溯

- STR-F 节点初次生成（Phase C 新建）时版本 = `v1`，变更影响声明写"首版，无变更"
- 每次因新 OR 触发 Step 3 Phase B 修订，版本号递增（v1 → v2 → v3 ...）
- 版本号写入 STR-F 节点头部（与节点 ID 同级）
- wft02-biz 首次消费 STR-F 时，在 `04-*.md` 对应 BA 节点中记录 `STR-F 消费版本：v{N}`
- wft02-biz 重入同一 STR-F 时比对版本：若 STR-F 当前版本 > 已消费版本 → 触发变更感知，读取变更影响声明，按变更类型处理

#### 增量清单与变更影响声明的关系

| 维度 | 增量清单（§五） | 变更影响声明 |
|------|----------------|------------|
| 面向读者 | 人类 | 下游 Skill（wft02-biz） |
| 目的 | 告知"本轮我做了什么，请逐项审阅" | 告知"STR-F 变了什么，你需要做什么" |
| 内容 | 本轮新增/改进/确认/矛盾的逐项列表 | 变更类型 + 对已消费 BA 的影响 + 建议动作 |
| 写入位置 | STR-F 节点末尾（与人类反馈区配对） | STR-F 节点末尾（独立小节） |
| 生成时机 | 每次 Step 3 产出方案时 | 仅修订已有 STR-F 时；新建 STR-F 写"首版，无变更" |
