# STR-F → BA · 业务架构设计与资产化
**Skill ID**: wft02-biz-str2ba

> **概述**：每次执行处理 ≥1 个 STR-F 节点（来自 wft01-biz，状态=`可以分解分配` / `待补充分解分配`），BA 反馈优先于新 STR-F 处理。基于输出产品架构锚定法，将 STR-F 场景定义中的十链信息载体转换为 EOS 两类产品（A2 xx业务运行系统 + Bn 平台输出产品）的 BA 主结构方案，完成 23/24/26/27 资产对准与写回，形成可进入 wft03 的 BA 节点。三步递进：十链消费分组 → 确定 A2+Bn 产品路径与业务定义 → 对准并写回 23（A2+Bn）+ 24（业务定义）+ 26（文档定义）+ 27（资源资产）。业务定义需要改进或新增引擎支撑时，将引擎需求写回原始需求目录，供人类经 OR 预处理链进入 wft01-eng 全链路。
>
> **输入 → 输出**：STR-F 节点（≥1 个，`可以分解分配` / `待补充分解分配`）/ 有反馈的 BA 节点 → BA 节点方案 + 23/24/26/27 资产对准与写回（状态=`可以SR设计`）+ STR-F 推进为 `已分解分配` + 引出引擎需求时写回原始需求目录
>
> **适用场景**：瀑布式系统设计第2步 · 业务架构路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.7 | **修订**：2026-06-24

## 一、职责定位

本 Skill 负责：

1. 接收人类选中的 ≥1 个 STR-F 节点（`可以分解分配` / `待补充分解分配`）
2. 将 STR-F 节点分配到已有 BA 节点（Q1/Q2）或聚合为新的 BA 节点（Q3）
3. 对已有 BA 节点的修订，执行多 STR-F 收敛检测——四关系判定 + 来源升级 + 矛盾记录
4. 对 STR-F 的场景输出文档清单执行消费分析，每个输出文档展开为 Bn 业务定义 IPO（含 QTC/生命周期/支持信息/交互信息），交互信息归入公共业务构件
5. 形成 A2（业务运行系统）和 Bn（平台输出产品）的 BA 主结构方案
6. 对 BA 方案执行 23-Q1/Q2/Q3（A2+Bn 产品路径）、24-Q1/Q2/Q3（业务定义）、26-Q1/Q2/Q3（文档定义）、27 资源引用对准
7. 基于 BA 架构语境记录构件产品数据完备性缺口观察（默认不升级为正式 BA 节点）
8. 校验每个业务定义是否能被已有 25 引擎支撑——顺序 4（需改进引擎）或顺序 5（需新增引擎）时，将引擎需求写回原始需求目录，供人类经 OR 预处理链（ort00~ort03）文本化/切分/澄清/规范化后进入 wft01-eng → STR-E → wft02-eng
9. 人类确认方案后统一写回 23/24/26/27 资产（Q1/Q2/Q3 均直接写入，不需二次确认）
10. 人类确认时 BA 节点推进为 `可以SR设计`（wft03 可即时进入）；资产落账后对应 STR-F 推进为 `已分解分配`

本 Skill 不负责：

1. 引擎配置需求（STR-E → `wft02-eng-str2ba`）
2. 非功能需求（STR-NFR → `wft04a`）
3. A1 业务配置平台的 BA 设计（→ `wft02-eng-str2ba`）
4. 下游 wft03 的 SR 架构设计展开
5. 将未升级的 BA 完备性缺口观察写入资产文件

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（STR-F 节点 ≥1 个 / 有反馈的 BA 节点）
2. 上游追溯链（`02-*.md` STR-F 正文块 / 信息载体清单 / 场景指标 / wft01-biz 文档归属方案）
3. 已有 BA 架构资产（`04-*.md` 树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`23` 产品架构 + `24` 业务定义 + `26` 文档定义 + `27` 资源资产）
5. `25` 引擎模型资产（仅用于业务定义支撑链校验，不写回）
6. 重入反馈

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 STR-F 选中 + 无待处理反馈的 BA 节点 | 输出"无待处理对象"→ **退出** |
| STR-F 节点状态不是 `可以分解分配` / `待补充分解分配` | 输出状态分布，列出不合格节点及其当前状态 → **退出** |
| 选中节点含 STR-E / STR-NFR 类型 | 输出类型分布（STR-E → `wft02-eng`，STR-NFR → `wft04a`），要求分路径 → **退出** |
| 选中的 STR-F 分属多个业务域且无已确认关联 | 输出业务域分布，提示可能存在跨域设计 → 人类确认后继续或分批 |
| 选中 STR-F >5 个 | 输出"本轮 N 个 STR-F，BA 设计工作量大，增量清单长，建议分批"，提示节点 ID 列表 → 人类确认后继续，不强制退出 |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 BA 节点 | 本轮先处理全部有反馈的 BA 节点 |
| 存在 `待确认方案` + 反馈区为空的 BA 节点 | 不阻塞新 STR-F；Step End 时提示 |

#### 第三层：新 STR-F 路由

BA 反馈处理完后，逐节点处理新 STR-F → Phase A 场景定义处理。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 BA 节点的产品路径或业务定义清单，因此 Step 3 Phase A 场景定位时**必须基于 Step 2 修改后的 BA 节点当前内容**，不得使用 Step 1 加载时的缓存旧状态。

### 按需加载线索

- **对象线索**：按 STR-F 节点 ID 或 BA 节点 ID 在索引中定位必须读取的正文块
- **追溯线索**：按追溯关系读取上下游相关块，不复制对方正文

- **扩大读取触发**：索引缺失/候选冲突/资产全貌不足以判断边界/追溯关系断裂时，说明原因后扩大

### Step 布局总览

| Step | 名称 | 产出 | 写资产 |

| Start | 前置校验与路由 | 就绪 / 结束 | — |

| Step End | 运行结束输出 | 人类下一步操作提示 | — |

### BA 节点生命周期

#### 推进态（按顺序）

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `待确认方案` | 方案已生成或修订，等待人类反馈 | Step 3 完成 |
| `可以SR设计` | 方案已确认，wft03 可启动 SR 架构设计 | 人类反馈 `[同意]` 或整体确认 |
| `在SR设计` | wft03 已锁定此 BA，正在展开 SR-F | wft03 写入活动锁 |
| `已SR设计` | SR-F 展开完成，BA 消费版本与当前版本一致 | wft03 处理完成且未检测到上游新版本 |

人类确认后，两条线并行推进、互不阻塞：
- **资产线**（本 Skill）：Step 4，将 BA 引用写回 23/24/26/27 资产文件。资产写回完成与否不改变 BA 状态。
- **SR设计线**（wft03）：读 `可以SR设计`，展开 SR-F 节点。不依赖 23/24/26/27 资产文件中的写回结果。

`在SR设计` 和 `已SR设计` 由 wft03 写入和推进。本 Skill（wft02-biz）写入的范围为 `待确认方案` / `可以SR设计` / `待补充SR设计`，并在修订时更新版本号。

#### 回退态（上游修订触发）

上游修订 = 本 Skill 因上游 STR-F 变更对已有 BA 执行 Step 3 Phase B。修订后版本号递增，状态按原状态分流：

| 原状态 | 修订后状态 | 含义 |
|--------|----------|------|
| `可以SR设计` | `可以SR设计`（版本递增） | 未被下游消费过，版本升级不影响任何 SR-F |
| `在SR设计` | `在SR设计`（版本递增） | wft03 锁仍有效，重入时检测版本差，按变更类型增量处理 |
| `已SR设计` | **`待补充SR设计`** | SR-F 已不完整，需 wft03 重新介入补全 |

- **`待补充SR设计`**：上游 STR-F 修订后，已消费的 SR-F 可能不完整。wft03 进入后按变更影响声明处理，处理完成后若版本一致 → 推进回 `已SR设计`

### Step 2 · 方案反馈处理

若本轮存在有反馈的 BA 节点（`待确认方案` + 反馈区有效），逐个处理。

#### 基本操作

| 操作 | 行为 |
|------|------|
| `[同意]` 或整体确认 | 归档反馈 → 生成空反馈区 → BA → `可以SR设计`（wft03 此时即可进入）→ 进入 Step 4 写回（Q1/Q2/Q3 统一写入，不需二次确认） |
| `[修改]`（局部） | 按反馈调整被指定的单元 → 仅重对准受影响部分 → 更新增量清单 → 回写 `待确认方案` |
| `[驳回]`（局部） | 移除被驳回的对象和关联对准结论 → 其他对象 → `已确认` → 回写 `待确认方案` |
| `[驳回]`（整体） | 重做 Step 3 → 所有对象确认状态重置 → 回写 `待确认方案` |

#### 分层反馈

BA 节点包含多个可独立决策的单元（A2 场景定义、Bn 计划定义、各业务定义 IPO、缺口观察、资产对准）。人类可以**局部修改**——整体确认一个 BA 节点的同时，要求调整其中某个业务定义或某个 23-Q 判定，不需要因一个局部问题而驳回整个 BA 节点。

AI 读取反馈时按以下规则处理，同步更新每个对象的确认状态：

- **局部 `[修改]`**：被指定的对象确认状态重置为 `待确认`，其涉及的资产对准重新执行。未提及的对象确认状态推进为 `已确认`，保持原结论不动，不触发全量重新对准。
- **局部修改导致级联**：产品路径变更 → 级联更新其下所有业务定义的 23 对准（确认状态均重置为 `待确认`）；业务定义拆分/合并 → 级联更新 23/24/26 对准；构件类型归属变更 → 级联更新 23/26 对准。级联完成后重新执行 Phase C·9 覆盖自查。
- **BA 节点推进条件**：所有内部对象的确认状态均为 `已确认` → BA 节点推进为 `可以SR设计`。存在任何一个 `待确认` → BA 节点保持 `待确认方案`。
- **`[驳回]`（局部）**：移除被驳回的对象和关联对准结论。剩余对象的确认状态推进为 `已确认`。
- **`[驳回]`（整体）**：BA 节点主结构被推翻（"A2 角色职责划分根本不对"或"整个 Bn 构件归属错误"）→ 重做 Step 3，所有对象的确认状态重置。

#### 反馈区位置

反馈区位于 `04-*.md` 中**每个 BA 节点块的末尾**，作为 BA 节点正文的最后一个子节。新增 BA 节点时，AI 在节点块末写入空反馈区；修订已有 BA 节点时，在既有反馈区后面追加或按规则覆盖。

#### 反馈区格式

**AI 写入（Step 3 完成时）** —— 空反馈区，等待人类填写：

```text
### 反馈区

[操作] 
```

**人类填写（或 AI 从对话转写）** —— 三种操作之一：

```text
### 反馈区

[操作] 同意

```
或
```text
### 反馈区

[操作] 修改

- BIZDEF-采购审批：Output 应包含审批留痕记录
- 23-Q2 供应商评估表归属 → 改到 @comp-供应商管理:供应商评估

```
或
```text
### 反馈区

[操作] 驳回

- OBS-001：已有 STR-F-YYY 覆盖，缺口不成立。不移除 BA 节点其他内容。
- 整体驳回：A2 角色职责划分不合理，采购执行和采购审批不应放在同一职责下。

```

- 整体同意 → `[操作] 同意`，无需逐项列举
- 局部修改 → `[操作] 修改`，列出被修改的单元 ID + 修改内容。未列出的单元视为同意
- 驳回 → 逐项列出被驳回的单元 ID + 原因。未列出的单元视为同意。如果驳回的是 BA 主结构，标注"整体驳回"
- AI 读取时按单元 ID 定位对象，未提及的单元保持原结论不动

**AI 处理完成后** —— 归档旧反馈，重置反馈区：

AI 将本轮的人类反馈原文和处理动作写入 `### 反馈处理记录`（覆盖上轮内容，仅保留本轮），然后将反馈区重置为初始空状态 `[操作] `。完整历史追溯通过 BA 版本号 + §八 变更记录实现。

BA 节点块末尾结构：
```
### 反馈区              ← 当前反馈（AI 每次处理完重置为 [操作] ）
### 反馈处理记录         ← 仅保留本轮（如本轮无反馈则写"无"）
   修改 BIZDEF-XXX Output → 已扩展；23-Q2 已重新对准
### 变更影响声明         ← 见 §四·附
```

### Step 3 · 新增 BA 设计

分三个阶段：场景定义处理 → 业务定义处理 → 要素集成处理。

#### Phase A：场景定义处理

STR-F 场景定义的大多数 A2 路径要素已由 wft01-biz 确定。Phase A 沿已知路径逐级检查 23/04 资产，缺则补、有则复用。

**A2 侧——场景定义自身路径**：

| 层级 | 来源 | 动作 |
|------|------|------|
| 业务域 | STR-F | ✓ |
| xx业务运行系统 | 从业务域映射 23 | ✓ 已有则复用，无则新建 |
| 用户角色 | STR-F | ✓ |
| 产品类型 | wft01-biz 23 对准结论 | ✓ 已有则复用，无则新建 |
| 构件类型 | wft01-biz 23 对准结论 | ✓ 已有则复用，无则新建 |
| 角色职责（四级节点） | **待定** | 查 23/04——该构件类型下是否有匹配该角色+业务域的职责。有则复用；无则 AI 基于 STR-F 场景职责边界陈述推断一个，标记 `[推断]`，人类审 |
| 场景定义（五级节点） | STR-F 本身 | 写入 BA 节点 |

**QTC 指标路径**——场景定义的指标归属到同一产品类型下的项目管理构件：

| 层级 | 动作 |
|------|------|
| 项目管理构件 | 已有则复用，无则新建 |
| QTC 指标 | 按指标引擎承载，写入 |

**Bn 侧——每个输出文档**：产品类型和构件类型来自 wft01-biz，本 Skill 生成 WBS 包（见 Phase C）。输出文档的主业务定义、交互业务定义、QTC/进展各自对应引擎（已在 wft01-biz 标注）。

匹配完成后进入 Phase B（业务定义处理）和 Phase C（要素集成处理）。场景定义定位完毕 → 展开输出文档为业务定义 → 全局集成对准。三个阶段线性推进，无路由分叉。

#### Phase B：业务定义处理

Phase A 匹配到已有 BA 后，将 STR-F 的输出文档增量合并进 BA 的 Bn 侧。

**1. Bn 路径逐级检查**——对每个输出文档，沿 Bn 路径逐级查 23/04 资产：

| 层级 | 动作 |
|------|------|
| 产品类型 | 已有 BA Bn 侧是否已涵盖？是→复用；否→新建 |
| 构件类型 | 同上 |
| WBS 包 | 已有则复用；无则本 Skill 生成 |
| 业务定义 IPO | 与已有 IPO 逐项比对（四关系检测） |

**2. 逐输出文档四关系检测**——比对锚点为输出文档本身（文档名称 + 业务定义类型 + 支撑引擎）：

| 关系 | 判定 | 处理 |
|------|------|------|
| 确认 | 新输出文档与既有 BA 业务定义一致（同文档+同类型+同引擎） | 来源升级（`BA完备性观察`→`STR推导`），增量标记 `[确认]` |
| 补充 | 新输出文档在既有 BA 中无对应 | 新增 Bn IPO 到 BA 方案，来源=`STR推导`，增量标记 `[新增]`；对应的交互业务定义（通知/待办/回执…）同步新增到公共业务构件 |
| 矛盾 | 输出文档与既有业务定义同名但类型/归属冲突 | 追加矛盾记录，增量标记 `[矛盾]`，等人类裁决 |
| 无新增 | 完全被既有 BA 覆盖 | 仅追加 STR-F 引用 |

来源类型升级规则（只升不降）：`BA完备性观察`→`STR推导`、`人类指定补充`→`STR推导`、`资产已确认补充`→`STR推导`。

矛盾记录追加式不覆盖，存在未解决矛盾时不得推进到写回。

**3. 构件管理 WBS 与项目管理构件**：逐文档展开完成后，同步生成：

- **构件管理 WBS**：该构件类型下若有多个输出文档，生成构件管理 WBS 包，内含构件级管理类业务定义（输出文档=构件看板/报表，由指标引擎支撑）。构件级 QTC = 该构件下各文档 QTC 的加权汇总。
- **项目管理构件**：该产品类型下生成项目管理构件（若不存在），内含产品级管理类业务定义（输出文档=产品看板/计分卡，由指标引擎支撑）。产品级 QTC = 各构件 QTC 的汇总。
- **角色 QTC**：嵌入 A2 角色职责节点属性字段，聚合口径 = 该角色下各场景 QTC 之和。本 Skill 定义度量口径，值留待后续场景积累填充。
- **业务域 QTC**（=组织 QTC）：嵌入该业务域对应组织节点属性字段，聚合口径 = 该业务域下各角色 QTC 之和。本 Skill 定义度量口径，值留待后续场景积累填充。

**4. BA 完备性缺口观察复查**：Phase B 修订时，复查与该 BA 所在业务域/产品类型/构件类型相关的未关闭缺口观察——若新 STR-F 的输出文档恰好覆盖了观察项指向的缺口，标记为 `已被当前STR吸收` 并关闭观察；若新 STR-F 提供了升级证据，按成熟度重新判定。

**4. 下游变更影响声明**：四关系检测完成后，按本轮 STR-F 对 BA 的实际变更范围判定变更类型——仅新增输出文档/业务定义且产品路径无新增构件类型 → `增量型`；已有业务定义/初步方案修订但产品路径未变 → `修订型`；产品路径涉及新增构件类型或跨产品类型 → `结构型`。将变更影响声明写入 BA 节点末尾（格式与完整规则见 §四·附 变更传播协议）。

#### Phase C：要素集成处理

**1. 确定业务域与产品路径**：从 STR-F 读取业务域和用户角色，在 23 资产中匹配 A2（xx业务运行系统）和 Bn（产品类型/构件类型）候选路径。

**2. 输出文档消费分析**——将 STR-F 的场景输出文档清单映射为 BA 节点：

STR-F 的场景定义 = 一条场景输出文档链。每个输出文档在 wft01-biz 中已标注其 QTC 指标、生命周期状态、支持信息（规则知识/风险合规/资源工具）和交互信息（协同交互/状态事件）。wft02-biz 消费这些文档：

- **每个输出文档 → 一个 Bn 业务定义 IPO**：该 IPO 接收输入文档 + 支持信息，通过 CAX 工具等处理，生成输出文档本身。QTC 指标、生命周期状态作为 IPO 属性携带。产品类型和构件类型来自 wft01-biz 已识别的归属结论，本 Skill 负责生成 WBS 包。
- **每个输出文档的交互信息 → 公共业务构件下的五级业务定义**：路径为 `流程与IT域 → 公共业务构件 → [三级] → [四级] → IPO`。初步方案=细化活动序列集（如通知的发送/接收/确认、事件的状态推进/告警等）。若已被 wft01-biz 判定为已有公共业务定义，则只维护引用关系，不重复生成。

**3. A2 场景定义设计**：

A2 路径：业务域 → xx业务运行系统 → 用户角色 → 角色职责 → 场景定义 + 初步方案。

- STR-F 场景定义直接映射为 A2 五级节点。用户角色、角色职责、业务域均来自 STR-F 已确认信息，本 Skill 做 1:1 映射。
- A2 场景定义的初步方案 = 业务定义序列集——将步 2 产生的 Bn 业务定义和公共业务定义，按角色履职顺序编排为调用序列。
- 场景进展 = 各输出文档生命周期状态按前后依赖关系加权计算。
- 场景 QTC 指标：嵌入 A2 场景定义作为属性字段，由指标引擎支撑计算。场景级 QTC = 各输出文档 QTC 的汇总 + 场景级指标（如该场景整体及时率、完成率）。
- 角色 QTC 指标：嵌入 A2 角色职责节点属性字段。聚合口径 = 该角色下各场景 QTC 之和。值留待后续场景积累填充，本 Skill 定义度量口径。
- 业务域 QTC 指标（=组织 QTC）：嵌入该业务域对应组织节点属性字段。聚合口径 = 该业务域下各角色 QTC 之和。值留待后续场景积累填充，本 Skill 定义度量口径。
- 标明每个业务定义在场景中的使用方式：创建 / 处理 / 查看 / 审批 / 协同 / 监控 / 评价 / 触发。

**4. Bn 业务定义设计**：

每个输出文档展开为一个 Bn 业务定义 IPO（计划定义）。

Bn 路径：业务域 → 产品类型 → 构件类型 → WBS 包 → IPO。

- 产品类型、构件类型：来自 wft01-biz 已识别的归属结论。本 Skill 直接使用，不重新判定。
- WBS 包：本 Skill 生成——按业务对象/生命周期阶段/构件职责/产品输出结构聚合（见步 5）。
- IPO 要素：
  - **Input**：前序文档 + 三种支持信息（规则知识/风险合规/资源工具）。在 EOS 平台上，支持信息以右侧栏功能按钮的形式呈现该文档的三类信息列表。
  - **Process**：通过 CAX 工具/AI 工具/模板/专业能力/作业指南等处理。
  - **Output**：该输出文档本身。
- QTC 指标：携带该文档的质量/及时性/完整性指标。若文档有责任人和责任组织，指标天然赋予他们。
- 生命周期状态：按该文档的业务性质确定具体状态流转路径（如采购申请单：草稿→提交→审核→批准→下达；验收单：待验收→验收中→已验收→已入库）。不同文档生命周期不同，本 Skill 为每个文档设计具体状态。
- 交互信息：每个输出文档的协同交互和状态事件 → 公共业务构件下的五级业务定义（见步 2）。本 Skill 在 Bn IPO 中记录对这些公共业务定义的调用关系。
- 匹配 24 业务定义资产，校准业务定义命名和定义。
- 引用 27 资源资产（CAX/AI 工具/模板/专业能力/作业指南）；不存在时形成资源资产建议。

所有输出文档展开完成后，同步生成：
- **构件管理 WBS**：该构件类型下若有多个输出文档，生成构件管理 WBS 包，内含构件级管理类业务定义（输出文档=构件看板/报表，由指标引擎支撑）。构件级 QTC = 该构件下各文档 QTC 的加权汇总。
- **项目管理构件**：该产品类型下生成项目管理构件（若不存在），内含产品级管理类业务定义（输出文档=产品看板/计分卡，由指标引擎支撑）。产品级 QTC = 各构件 QTC 的汇总。
- **角色 QTC**：嵌入 A2 角色职责节点属性字段，聚合口径 = 该角色下各场景 QTC 之和。本 Skill 定义度量口径，值留待后续场景积累填充。
- **业务域 QTC**（=组织 QTC）：嵌入该业务域对应组织节点属性字段，聚合口径 = 该业务域下各角色 QTC 之和。本 Skill 定义度量口径，值留待后续场景积累填充。

### 四·附、BA 变更传播协议

下游 wft03 消费 BA 后生成 SR-F 节点。当 BA 因上游 STR-F 修订而变更时，已生成的 SR-F 节点面临同步问题。本协议定义 BA → SR-F 的变更分类、通知格式和消费追溯。

#### 变更类型分类

| 变更类型 | 判定标准 | 实例 | 对下游影响 |
|---------|---------|------|----------|
| **增量型** | 仅新增业务定义/场景要素，已有业务定义/初步方案/产品路径均未改动 | 新 STR-F 补充了一个业务定义 | 已有 SR-F 节点不受影响，仅需映射新业务定义 |
| **修订型** | 已有业务定义/初步方案/场景要素被修改；产品路径未变 | 业务定义 I/P/O 从 X 改为 Y | 对应 SR-F 节点需局部调整 |
| **结构型** | 产品路径边界检出边缘/超出、BA 结构需重塑 | A2 场景定义拆分或 Bn 构件类型新增 | SR-F 结构可能重塑，需人类裁决后方可处理 |

判定优先级：结构型 > 修订型 > 增量型。

#### 变更影响声明格式

每次修订已有 BA 后，在 BA 节点末尾写入（`04-*.md` 中 BA 节点块内）：

```text
### 变更影响声明

BA 版本：v{N}（本次修订前为 v{N-1}）
本轮变更类型：增量型 / 修订型 / 结构型

变更项：
- [新增] / [改进] / [矛盾] / [边界扩展] ...

下游已消费此 BA 的 SR-F 节点：
  <SR-F节点ID列表>（如无则写"尚未被下游消费"）

对已消费 SR-F 节点的影响：
  受影响节点：<SR-F节点ID + 受影响原因>
  不受影响节点：<SR-F节点ID>
  待新增 SR-F 映射：<新业务定义对应的待映射说明>

建议 wft03 动作：
  - 增量型：增量映射新增业务定义即可，已有 SR-F 节点不动
  - 修订型：修订受影响的 SR-F 节点 → 标记 [来自上游变更]
  - 结构型：输出影响范围，等人类裁决后再处理
```

#### 版本标记与消费追溯

- BA 节点初次生成时版本 = `v1`，变更影响声明写"首版，无变更"
- 每次因新 STR-F 触发 Step 3 Phase B 修订，版本号递增
- 版本号写入 BA 节点头部（与节点 ID 同级）
- wft03 首次消费 BA 时，在对应 SR-F 节点中记录 `BA 消费版本：v{N}`
- wft03 重入同一 BA 时比对版本：若 BA 当前版本 > 已消费版本 → 触发变更感知

#### 增量清单与变更影响声明的关系

| 维度 | 增量清单（§五） | 变更影响声明 |
|------|----------------|------------|
| 面向读者 | 人类 | 下游 Skill（wft03） |
| 目的 | 告知"本轮我做了什么，请逐项审阅" | 告知"BA 变了什么，你需要做什么" |
| 内容 | 本轮新增/改进/确认/矛盾的逐项列表 | 变更类型 + 对已消费 SR-F 的影响 + 建议动作 |
| 写入位置 | BA 节点末尾（与人类反馈区配对） | BA 节点末尾（独立小节） |
| 生成时机 | 每次 Step 3 产出方案时 | 仅修订已有 BA 时；新建 BA 写"首版，无变更" |
