# STR-F → BA · 业务架构设计与资产化
**Skill ID**: wft02-biz-str2ba

> **概述**：每次执行处理 ≥1 个 STR-F 节点（来自 wft01-biz，状态=`可以分解分配` / `待补充分解分配`），BA 反馈优先于新 STR-F 处理。将 STR-F 场景定义沿四个 Phase 展开为 BA 节点——A 交互业务定义 + B 文档业务定义 + C 场景定义 + D 上层指标定义，完成 23/24/26/27 资产对准与写回，形成可进入 wft03 的 BA 节点。四个 Phase 统一遵循三步递进（生成初步方案 → 资产匹配 → 定位）。业务定义需要改进或新增引擎支撑时，Step 4 将引擎需求写回原始需求目录，供人类经 OR 预处理链进入 wft01-eng 全链路。
>
> **输入 → 输出**：STR-F 节点（≥1 个，`可以分解分配` / `待补充分解分配`）/ 有反馈的 BA 节点 → BA 节点方案 + 23/24/26/27 资产对准与写回（状态=`可以SR设计`）+ STR-F 推进为 `已分解分配` + 引出引擎需求时写回原始需求目录
>
> **适用场景**：瀑布式系统设计第2步 · 业务架构路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v2.3 | **修订**：2026-06-24

## 一、职责定位

本 Skill 负责：

1. 接收人类选中的 ≥1 个 STR-F 节点（`可以分解分配` / `待补充分解分配`）
2. 将 STR-F 节点分配到已有 BA 节点（Q1/Q2）或聚合为新的 BA 节点（Q3）
3. 对已有 BA 节点的修订，执行多 STR-F 收敛检测——四关系判定 + 来源升级 + 矛盾记录
4. 对 STR-F 的场景输出文档清单执行消费分析，每个输出文档展开为 Bn 业务定义 IPO（含 QTC/生命周期/支持信息/交互信息），交互信息归入公共业务构件
5. 形成 A2（业务运行系统）和 Bn（平台输出产品）的 BA 主结构方案
6. 对 BA 方案执行 23-Q1/Q2/Q3（A2+Bn 产品路径）、24-Q1/Q2/Q3（业务定义）、26-Q1/Q2/Q3（文档定义）、27 资源引用对准
7. 基于 BA 架构语境记录构件产品数据完备性缺口观察（默认不升级为正式 BA 节点）
8. 校验每个业务定义是否能被已有 25 引擎支撑——顺序 4（需改进引擎）或顺序 5（需新增引擎）时，将引擎需求写回原始需求目录，供人类经 OR 预处理链（ort00~ort03）文本化/切分/澄清/规范化后进入 wft01-eng → STR-E → wft02-eng
9. 人类确认方案后统一写回 23/24/26/27 资产（Q1/Q2/Q3 均直接写入，不需二次确认）
10. 人类确认时 BA 节点推进为 `可以SR设计`（wft03 可即时进入）；资产落账后对应 STR-F 推进为 `已分解分配`

本 Skill 不负责：

1. 引擎配置需求（STR-E → `wft02-eng-str2ba`）
2. 非功能需求（STR-NFR → `wft04a`）
3. A1 业务配置平台的 BA 设计（→ `wft02-eng-str2ba`）
4. 下游 wft03 的 SR 架构设计展开
5. 将未升级的 BA 完备性缺口观察写入资产文件

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（STR-F 节点 ≥1 个 / 有反馈的 BA 节点）
2. 上游追溯链（`02-*.md` STR-F 正文块 / 信息载体清单 / 场景指标 / wft01-biz 文档归属方案）
3. 已有 BA 架构资产（`04-*.md` 树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`23` 产品架构 + `24` 业务定义 + `26` 文档定义 + `27` 资源资产）
5. `25` 引擎模型资产（仅用于业务定义支撑链校验，不写回）
6. 重入反馈

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 STR-F 选中 + 无待处理反馈的 BA 节点 | 输出"无待处理对象"→ **退出** |
| STR-F 节点状态不是 `可以分解分配` / `待补充分解分配` | 输出状态分布，列出不合格节点及其当前状态 → **退出** |
| 选中节点含 STR-E / STR-NFR 类型 | 输出类型分布（STR-E → `wft02-eng`，STR-NFR → `wft04a`），要求分路径 → **退出** |
| 选中的 STR-F 分属多个业务域且无已确认关联 | 输出业务域分布，提示可能存在跨域设计 → 人类确认后继续或分批 |
| 选中 STR-F >5 个 | 输出"本轮 N 个 STR-F，BA 设计工作量大，增量清单长，建议分批"，提示节点 ID 列表 → 人类确认后继续，不强制退出 |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 BA 节点 | 本轮先处理全部有反馈的 BA 节点 |
| 存在 `待确认方案` + 反馈区为空的 BA 节点 | 不阻塞新 STR-F；Step End 时提示 |

#### 第三层：新 STR-F 路由

BA 反馈处理完后，逐节点处理新 STR-F → Phase A 场景定义处理。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 BA 节点的产品路径或业务定义清单，因此 Step 3 Phase A 场景定位时**必须基于 Step 2 修改后的 BA 节点当前内容**，不得使用 Step 1 加载时的缓存旧状态。

### BA 节点生命周期

#### 推进态（按顺序）

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `待确认方案` | 方案已生成或修订，等待人类反馈 | Step 3 完成 |
| `可以SR设计` | 方案已确认，wft03 可启动 SR 架构设计 | 人类反馈 `[同意]` 或整体确认 |
| `在SR设计` | wft03 已锁定此 BA，正在展开 SR-F | wft03 写入活动锁 |
| `已SR设计` | SR-F 展开完成，BA 消费版本与当前版本一致 | wft03 处理完成且未检测到上游新版本 |

人类确认后，两条线并行推进、互不阻塞：
- **资产线**（本 Skill）：Step 4，将 BA 引用写回 23/24/26/27 资产文件。资产写回完成与否不改变 BA 状态。
- **SR设计线**（wft03）：读 `可以SR设计`，展开 SR-F 节点。不依赖 23/24/26/27 资产文件中的写回结果。

`在SR设计` 和 `已SR设计` 由 wft03 写入和推进。本 Skill（wft02-biz）写入的范围为 `待确认方案` / `可以SR设计` / `待补充SR设计`，并在修订时更新版本号。

#### 回退态（上游修订触发）

上游修订 = 本 Skill 因上游 STR-F 变更对已有 BA 执行 Step 3 Phase B。修订后版本号递增，状态按原状态分流：

| 原状态 | 修订后状态 | 含义 |
|--------|----------|------|
| `可以SR设计` | `可以SR设计`（版本递增） | 未被下游消费过，版本升级不影响任何 SR-F |
| `在SR设计` | `在SR设计`（版本递增） | wft03 锁仍有效，重入时检测版本差，按变更类型增量处理 |
| `已SR设计` | **`待补充SR设计`** | SR-F 已不完整，需 wft03 重新介入补全 |

- **`待补充SR设计`**：上游 STR-F 修订后，已消费的 SR-F 可能不完整。wft03 进入后按变更影响声明处理，处理完成后若版本一致 → 推进回 `已SR设计`

### Step 2 · 方案反馈处理

若本轮存在有反馈的 BA 节点（`待确认方案` + 反馈区有效），逐个处理。

#### 基本操作

| 操作 | 行为 |
|------|------|
| `[同意]` 或整体确认 | 归档反馈 → 生成空反馈区 → BA → `可以SR设计`（wft03 此时即可进入）→ 进入 Step 7 写回（Q1/Q2/Q3 统一写入，不需二次确认） |
| `[修改]`（局部） | 按反馈调整被指定的单元 → 仅重对准受影响部分 → 更新增量清单 → 回写 `待确认方案` |
| `[驳回]`（局部） | 移除被驳回的对象和关联对准结论 → 其他对象 → `已确认` → 回写 `待确认方案` |
| `[驳回]`（整体） | 重做 Step 3 → 所有对象确认状态重置 → 回写 `待确认方案` |

#### 分层反馈

BA 节点包含多个可独立决策的单元（A2 场景定义、Bn 计划定义、各业务定义 IPO、缺口观察、资产对准）。人类可以**局部修改**——整体确认一个 BA 节点的同时，要求调整其中某个业务定义或某个 23-Q 判定，不需要因一个局部问题而驳回整个 BA 节点。

AI 读取反馈时按以下规则处理，同步更新每个对象的确认状态：

- **局部 `[修改]`**：被指定的对象确认状态重置为 `待确认`，其涉及的资产对准重新执行。未提及的对象确认状态推进为 `已确认`，保持原结论不动，不触发全量重新对准。
- **局部修改导致级联**：产品路径变更 → 级联更新其下所有业务定义的 23 对准（确认状态均重置为 `待确认`）；业务定义拆分/合并 → 级联更新 23/24/26 对准；构件类型归属变更 → 级联更新 23/26 对准。级联完成后，Step 5/6 按顺序自然覆盖受影响对象的对准检查和语义验证。
- **BA 节点推进条件**：所有内部对象的确认状态均为 `已确认` → BA 节点推进为 `可以SR设计`。存在任何一个 `待确认` → BA 节点保持 `待确认方案`。
- **`[驳回]`（局部）**：移除被驳回的对象和关联对准结论。剩余对象的确认状态推进为 `已确认`。
- **`[驳回]`（整体）**：BA 节点主结构被推翻（"A2 角色职责划分根本不对"或"整个 Bn 构件归属错误"）→ 重做 Step 3，所有对象的确认状态重置。

#### 反馈区位置

反馈区位于 `04-*.md` 中**每个 BA 节点块的末尾**，作为 BA 节点正文的最后一个子节。新增 BA 节点时，AI 在节点块末写入空反馈区；修订已有 BA 节点时，在既有反馈区后面追加或按规则覆盖。

#### 反馈区格式

**AI 写入（Step 3 完成时）** —— 空反馈区，等待人类填写：

```text
### 反馈区

[操作] 
```

**人类填写（或 AI 从对话转写）** —— 三种操作之一：

```text
### 反馈区

[操作] 同意

```
或
```text
### 反馈区

[操作] 修改

- BIZDEF-采购审批：Output 应包含审批留痕记录
- 23-Q2 供应商评估表归属 → 改到 @comp-供应商管理:供应商评估

```
或
```text
### 反馈区

[操作] 驳回

- OBS-001：已有 STR-F-YYY 覆盖，缺口不成立。不移除 BA 节点其他内容。
- 整体驳回：A2 角色职责划分不合理，采购执行和采购审批不应放在同一职责下。

```

- 整体同意 → `[操作] 同意`，无需逐项列举
- 局部修改 → `[操作] 修改`，列出被修改的单元 ID + 修改内容。未列出的单元视为同意
- 驳回 → 逐项列出被驳回的单元 ID + 原因。未列出的单元视为同意。如果驳回的是 BA 主结构，标注"整体驳回"
- AI 读取时按单元 ID 定位对象，未提及的单元保持原结论不动

**AI 处理完成后** —— 归档旧反馈，重置反馈区：

AI 将本轮的人类反馈原文和处理动作写入 `### 反馈处理记录`（覆盖上轮内容，仅保留本轮），然后将反馈区重置为初始空状态 `[操作] `。完整历史追溯通过 BA 版本号 + §八 变更记录实现。

BA 节点块末尾结构：
```
### 反馈区              ← 当前反馈（AI 每次处理完重置为 [操作] ）
### 反馈处理记录         ← 仅保留本轮（如本轮无反馈则写"无"）
   修改 BIZDEF-XXX Output → 已扩展；23-Q2 已重新对准
### 变更影响声明         ← 见 §四·附
```

### Step 3 · BA 方案设计

四个 Phase 按依赖顺序执行：Phase A 交互业务定义处理 → Phase B 文档业务定义处理 → Phase C 场景定义处理 → Phase D 上层指标定义处理。四个 Phase 统一遵循三步递进（1.生成初步方案 → 2.资产匹配 → 3.定位），各自处理一类业务定义的完整闭环——差异仅在输入和业务定义类型。资产对准标注和全局覆盖检查由后续 Step 5/6 执行。

#### Phase A：交互业务定义处理

基于 wft01-biz 标注的交互业务定义类型和支撑引擎，对每个交互信息生成具体交互业务定义的初步方案，匹配已有资产或确定新建位置。三步递进：语义分析与初步方案生成 → 资产匹配 → 定位。

**1. 语义分析与初步方案生成**——基于支撑引擎的配置能力 + 交互内容语义，生成该交互业务定义的 IPO 初步方案。

输入来自 wft01-biz 对每个交互信息的标注（见 [wft01-biz §四·Phase C·5](eos-wft01-biz-or2str.md)）：
- 交互业务定义类型（通知类/待办类/回执类/接口类/状态事件类/告警类）
- 支撑引擎（`@engine-通知引擎` / `@engine-待办引擎` / ...）
- 交互内容（"审批通过后通知申请人和采购主管"等）
- 判定依据（OR 语义线索 + 业务常识推理）

**1. 聚合路径分析与初步方案生成**——对每个上层对象确定其聚合源（聚合哪些下层指标）、聚合口径（按什么规则聚合）、IPO（I/P/O 结构）。

四个上层对象均基于 `@engine-指标引擎`，IPO 结构一致——I=下层指标运行数据（从指标引擎已采集的下层业务定义数据中读取），P=指标引擎按聚合口径计算并按维度分组展示，O=该层指标看板/报表。差异仅在聚合源和聚合口径。

| 上层对象 | 聚合源 | 聚合口径 | 输出文档 |
|---------|--------|---------|---------|
| **角色职责** | 该角色职责下所有场景级指标子业务定义 | 各场景 QTC 按维度（Q/T/C）分别求和 | 角色指标看板 |

| **构件** | 该构件下所有文档级指标子业务定义 | 各文档 QTC 按维度加权汇总——权重=文档在该构件中的业务量占比（暂无数据时等权处理） | 构件看板/报表 |
| **产品** | 所有构件 QTC | 各构件 QTC 按维度汇总 | 产品看板/计分卡 |

聚合维度统一为 Q（质量）、T（及时性）、C（完整性），与 Phase B 文档级 QTC 度量维度对齐。首次生成时各维度值为空（标注 `待后续场景/文档积累填充`）；后续每次 Phase B/C 新增文档/场景级指标子业务定义时，聚合源自动扩展，实际值由指标引擎在运行时按聚合口径计算。

- 角色职责下仅有一个场景 → 角色级指标仍生成（聚合源=1，后续可能增加）
- 业务域下仅有一个角色职责 → 业务域级指标仍生成（同上）
- 构件下暂无文档级指标子业务定义 → 构件级指标仍生成，聚合源为空，标注 `待文档级指标积累`

### Step 4 · 业务引擎支撑链校验

Phase A/B/D·1 在生成初步方案时做**正向**分析（"25 引擎能配置生成什么能力 → 业务定义可做什么"），Step 4 做**反向**校验（"Step 3 产出的每个业务定义能否被已有 25 引擎模型和配置 IPO 类型实际支撑"）。校验范围覆盖 Phase A/B/C/D 生成的全部业务定义。

> **与 Phase B·2 的分工**：Phase B·2 做的是 24 业务定义**资产**匹配（已有业务定义→Q1复用/Q2扩展/Q3新建），Step 4 做的是 25 引擎**能力**支撑校验（已有引擎→能支撑/需改进/需新增）。两者互补，先后执行。

每个业务定义按以下顺序判定：

| 顺序 | 判定 | 本 Skill 动作 |
|------|------|--------------|
| 1 | 已有 25 引擎 + 已有配置 IPO 类型 → 可直接支撑 | 引用已有业务定义或扩展方案，记录支撑引擎和配置 IPO 类型 |
| 2 | 已有 25 引擎 + 已有配置 IPO 类型 → 可扩展已有业务定义 | 提出业务定义扩展方案，列出将实例化的配置 IPO 类型 |
| 3 | 已有 25 引擎 + 已有配置 IPO 类型 → 可新增业务定义 | 提出业务定义新增方案，列出将实例化的配置 IPO 类型 |
| 4 | 已有 25 引擎能力不足 → 需改进引擎后扩展或新增业务定义 | 提出引擎改进建议 + 业务定义方案；**将引擎需求写回原始需求目录** |
| 5 | 无可用引擎 → 需新增引擎后新增业务定义 | 提出引擎新增建议 + 业务定义方案；**同上——写回原始需求目录** |

按业务定义类型分别校验：

| 业务定义类型 | 来源 | 校验重点 |
|------------|------|---------|
| 交互业务定义 | Phase A | Phase A 已绑定支撑引擎类型（如 `@engine-通知引擎`），Step 4 仅校验该引擎是否具备交互内容所需的特定能力（如是否支持"按角色推送""已读回执"等）。校验通过 → 顺序 1~3；特定能力缺失 → 顺序 4 |
| 主业务定义 | Phase B | 校验支撑引擎的配置 IPO 类型是否覆盖初步方案的全部要素——I（三种支持信息接入能力）、P（配置单元是否覆盖所需行为/结构/权限）、O（文档结构是否可生成）。全部覆盖 → 顺序 1~3；局部缺失 → 顺序 4；完全不可支撑 → 顺序 5 |
| 指标类业务定义（六层） | Phase B/C/D | 统一校验 `@engine-指标引擎` 是否支持所需的度量维度和聚合计算口径（求和/加权汇总/按维度分组）。六层指标的引擎校验结论可合并——若指标引擎通过，则所有指标类业务定义均为顺序 1~3；若指标引擎需改进，统一记录一条引擎改进需求 |
| 进展类业务定义（生命周期） | Phase B | 校验 `@engine-进展引擎` 是否支持该文档类型的状态流转路径。完全支持 → 顺序 1~3；需扩展状态类型 → 顺序 4 |

顺序 4~5 写回原始需求目录的要求：
- 输出引擎需求描述（引擎名称/类型/需要具备的能力/支撑的业务定义/推理依据）
- 写入 `01-original-requirements.md` 或等价的原始需求文件，标记来源=`wft02-biz Step 4 引擎支撑链校验`，来源 BA 节点 ID
- 该引擎需求将作为新的原始需求条目，经 OR 预处理链独立走通 wft01-eng → STR-E → wft02-eng → A1 BA 全链路
- BA 方案中记录该引擎需求已引出（引出的原始需求条目 ID / 引出日期），当前 BA 节点对该引擎能力的依赖标记为 `[推断]` + `引擎待开发`
- 顺序 4~5 不阻断当前 BA 方案——BA 节点可推进到 `待确认方案` 和 `可以SR设计`，但依赖的引擎能力在正式落地前保持 `[推断]` 标记

同类型引擎需求合并：多个业务定义触发同一引擎的改进需求 → 合并为一条引擎需求，在需求描述中列出所有支撑的业务定义。

字段定义：

### Step 6 · 覆盖自查

对 Step 3~5 的产出执行语义覆盖完整性检查（"设计是否自闭环"），并统一管理跨轮次的 BA 完备性缺口观察。分两步：缺口观察管理 → 五维覆盖自查。

**6a. 缺口观察管理**——统一管理跨轮次的 BA 完备性缺口观察的完整生命周期。分两步：6a-1 复查已有观察（回收）→ 6a-2 创建新观察（播种）。观察记录均写入 `04-*.md` 的对应 BA 节点块中的"### 缺口观察区"小节。

##### 6a-1. 复查已有观察（回收）

将 Phase A/B/C/D 本轮生成的所有业务定义（交互业务定义/主业务定义/场景定义/上层指标定义），与 `04-*.md` 中**本 BA 节点及同业务域/同产品类型/同构件类型的其他 BA 节点**内的未关闭缺口观察进行比对。

**搜索范围与条件**：

| 本轮产出类型 | 搜索范围 | 筛选条件 |
|------------|---------|---------|
| 交互业务定义（Phase A） | 公共业务构件下的交互 BA 节点 | 同支撑引擎类型（如均属 `@engine-通知引擎`） |
| 主业务定义（Phase B） | 同业务域下的 Bn BA 节点 | 同产品类型 + 同构件类型 |
| 场景定义（Phase C） | 同业务域下的 A2 BA 节点 | 同用户角色 |
| 上层指标定义（Phase D） | 同业务域下的指标 BA 节点 | 同上层对象类型（角色职责/业务域/构件/产品） |

搜索方式：按 §三 全景层加载时已拉取的 BA 节点索引定位候选节点，读取其缺口观察区。无需维护独立的全局缺口观察索引。

**比对结果**：

| 结果 | 条件 | 动作 |
|------|------|------|
| **覆盖** | 本轮生成的业务定义恰好就是观察指向的缺口 | 标记 `已被当前STR吸收`，关闭观察。若此前成熟度为 `候选观察` 或 `待后续STR验证`，且来自 STR-F 直接推导 → 来源类型升级为 `STR推导` |
| **部分相关** | 本轮生成的内容跟观察有关联，但不足以完全覆盖 | 更新观察描述，记录本轮进展，观察保持开放 |
| **无关** | 本轮生成的内容跟观察无关联 | 观察保持原状态 |

##### 6a-2. 创建新观察（播种）

基于 BA 架构语境，对 A2/Bn 路径和公共业务构件路径执行产品数据完备性检查——识别本轮 STR-F 未推导出但可能需要的候选缺口。默认只记录观察，不升级为正式 BA 五级节点。

**范围**：覆盖 Phase A（交互）、Phase B（主业务定义）、Phase C（场景定义）、Phase D（上层指标定义）四个维度。

缺口观察标注字段：
- 观察 ID / 来源构件/WBS/IPO（或来源引擎类型/场景/上层对象） / 缺口类型（交互/文档/场景/指标） / 来源类型 / 成熟度 / 缺口描述 / 观察依据 / 当前处理建议 / 是否进入本轮 BA 主结构

成熟度分级：

| 成熟度 | 含义 | 默认动作 |
|--------|------|----------|
| `当前STR必需补充` | 没有该节点，当前场景的业务定义序列无法闭合 | 可升级为 BA 补充建议或 STR 回补建议 |
| `资产已确认补充` | 23/24/26/27 已有明确资产，当前只是建立引用 | 可升级为 BA 补充建议 |
| `人类指定补充` | 人类明确要求本轮补充 | 可升级为 BA 补充建议 |
| `候选观察` | AI 基于经验发现可能缺口，但证据不足 | 仅记录观察 |
| `待后续STR验证` | 需要后续 STR-F 节点进一步验证 | 仅记录观察 |
| `待构件级收敛` | 需要多个 STR-F 处理后做构件级产品数据收敛 | 仅记录观察 |

升级条件（满足任一）：
- `当前STR必需补充`：不补则业务定义序列无法闭合
- `资产已确认补充`：23/24/26/27 已有明确资产
- `人类指定补充`：人类明确要求

升级后的回补判定：

| 判定 | 适用情况 | 动作 |
|------|----------|------|
| `只补BA` | 只是在满足升级条件下补齐 BA 路径/资产引用 | 作为 BA 补充建议进入人类审阅和 Step 7 写回 |
| `回补当前STR` | 缺口意味着当前 STR-F 遗漏了信息载体或场景指标 | 形成 STR 回补建议，供人类确认后回到 wft01-biz |
| `新增STR场景建议` | 缺口属于另一个角色/触发/业务闭环 | 形成新增 STR 场景建议 |
| `业务引擎能力缺口` | 现有引擎无法配置生成所需业务定义 | 将引擎需求写回原始需求目录（同 Step 4 顺序 4~5），由人类经 OR 预处理链进入 wft01-eng → STR-E → wft02-eng |

**6b. 五维覆盖自查**——验证设计内容的语义覆盖完整性。未覆盖项回补或标注为缺口观察。

**6b-i. STR-F → BA 覆盖**

- 每个 STR-F 输出文档是否已展开为 Bn IPO（Phase B）+ 文档级指标子业务定义 + 生命周期状态流转 + 支持信息/交互信息标注
- 每个 STR-F 交互信息是否已展开为交互业务定义（Phase A）+ 在对应输出文档的"关联的交互业务定义引用"中正确列出
- STR-F 场景输出文档清单中的全部文档是否均已覆盖，无遗漏
- STR-F 场景指标度量维度是否已在场景级指标子业务定义中全部承接

**6b-ii. Phase 间交叉覆盖**

- Phase B 各文档的"关联的交互业务定义引用"是否与 Phase A 生成的交互业务定义一致——文档引用的每个交互条目在 Phase A 中均已生成；Phase A 生成的每个交互条目至少被一个文档引用（无孤立交互）
- Phase C 场景的业务定义序列是否完整覆盖 Phase B 全部文档 + Phase A 公共业务定义——从触发到完成的逻辑链无断点
- Phase C 场景级指标子业务定义是否汇总了该场景下全部文档级指标子业务定义的度量维度（Q/T/C），无遗漏维度

**6b-iii. QTC 指标链覆盖**

- 六层 QTC 指标是否逐层完整——文档级（Phase B）→ 场景级（Phase C）→ 角色级 → 业务域级（Phase D）；构件级 → 产品级（Phase D）
- 每层聚合源的维度（Q/T/C）与下层输出的维度是否对齐——下层有什么维度，上层就聚合什么维度
- 聚合路径是否无跳跃——无文档级直接跳到产品级（跳过构件级和场景级），无场景级直接跳到业务域级（跳过角色级）

**6b-iv. 资产引用覆盖**

- 每个业务定义（含六层指标子业务定义）是否已有 24 资产匹配结论（Q1/Q2/Q3）
- 每个业务定义是否已有 25 引擎支撑校验结论（顺序 1~5）
- 公共业务定义引用是否正确指向 `流程与IT域 → 公共业务构件` 路径
- 各 Phase·3 定位的四级节点是否均已定位到 23 路径并标注 Q1/Q2/Q3

**6b-v. 缺口观察覆盖**

- 6a-1 是否已对本轮全部业务定义（A/B/C/D 四类产出）执行已有观察回收比对
- 6a-2 是否已对 A2/Bn/公共业务构件/上层指标四个维度执行新观察播种
- 本轮标记为 `当前STR必需补充` 的缺口是否已升级并进入 BA 补充建议或 STR 回补建议

Step 6 完成后，BA 节点推进为 `待确认方案`。

### 四·附、BA 变更传播协议

下游 wft03 消费 BA 后生成 SR-F 节点。当 BA 因上游 STR-F 修订而变更时，已生成的 SR-F 节点面临同步问题。本协议定义 BA → SR-F 的变更分类、通知格式和消费追溯。

#### 变更类型分类

| 变更类型 | 判定标准 | 实例 | 对下游影响 |
|---------|---------|------|----------|
| **增量型** | 仅新增业务定义/场景要素，已有业务定义/初步方案/产品路径均未改动 | 新 STR-F 补充了一个业务定义 | 已有 SR-F 节点不受影响，仅需映射新业务定义 |
| **修订型** | 已有业务定义/初步方案/场景要素被修改；产品路径未变 | 业务定义 I/P/O 从 X 改为 Y | 对应 SR-F 节点需局部调整 |
| **结构型** | 产品路径边界检出边缘/超出、BA 结构需重塑 | A2 场景定义拆分或 Bn 构件类型新增 | SR-F 结构可能重塑，需人类裁决后方可处理 |

判定优先级：结构型 > 修订型 > 增量型。

#### 变更影响声明格式

每次修订已有 BA 后，在 BA 节点末尾写入（`04-*.md` 中 BA 节点块内）：

```text
### 变更影响声明

BA 版本：v{N}（本次修订前为 v{N-1}）
本轮变更类型：增量型 / 修订型 / 结构型

变更项：
- [新增] / [改进] / [矛盾] / [边界扩展] ...

下游已消费此 BA 的 SR-F 节点：
  <SR-F节点ID列表>（如无则写"尚未被下游消费"）

对已消费 SR-F 节点的影响：
  受影响节点：<SR-F节点ID + 受影响原因>
  不受影响节点：<SR-F节点ID>
  待新增 SR-F 映射：<新业务定义对应的待映射说明>

建议 wft03 动作：
  - 增量型：增量映射新增业务定义即可，已有 SR-F 节点不动
  - 修订型：修订受影响的 SR-F 节点 → 标记 [来自上游变更]
  - 结构型：输出影响范围，等人类裁决后再处理
```

#### 版本标记与消费追溯

- BA 节点初次生成时版本 = `v1`，变更影响声明写"首版，无变更"
- 每次因新 STR-F 触发 Step 3 Phase B 修订，版本号递增
- 版本号写入 BA 节点头部（与节点 ID 同级）
- wft03 首次消费 BA 时，在对应 SR-F 节点中记录 `BA 消费版本：v{N}`
- wft03 重入同一 BA 时比对版本：若 BA 当前版本 > 已消费版本 → 触发变更感知

#### 增量清单与变更影响声明的关系

| 维度 | 增量清单（§五） | 变更影响声明 |
|------|----------------|------------|
| 面向读者 | 人类 | 下游 Skill（wft03） |
| 目的 | 告知"本轮我做了什么，请逐项审阅" | 告知"BA 变了什么，你需要做什么" |
| 内容 | 本轮新增/改进/确认/矛盾的逐项列表 | 变更类型 + 对已消费 SR-F 的影响 + 建议动作 |
| 写入位置 | BA 节点末尾（与人类反馈区配对） | BA 节点末尾（独立小节） |
| 生成时机 | 每次 Step 3 产出方案时 | 仅修订已有 BA 时；新建 BA 写"首版，无变更" |

| 标记 | 含义 | 使用场景 |
