# BA → SysReq 架构 · Skill B · 功能定义落账与状态推进
**Skill ID**: wft03b · **Phase 2: BA → SysReq Functional Definition Settlement**

> **概述**：对 Skill A 已确认的 SysReq 功能定义方案执行正式资产落账与状态推进。`wft03` 无 Phase 2 下半；因此本 Skill 只包含上半：功能定义确认后按 IPO 类型分流落账——A1 引擎配置类功能定义写回 `25-*.md`，Bn 产品数据类与 A2 场景定义类仅推进 `05-*.md` 状态，以及 `04/05` 状态推进。
>
> **输入 → 输出**：已通过门禁 1 的 SysReq 功能定义方案 → 按类型分流落账 + `04/05` 状态推进 + 活动锁释放
>
> **适用场景**：瀑布式系统设计第3步 · Phase 2。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v2.0 | **修订**：2026-06-21

## 一、职责定位

Skill B 负责：

1. 接收已确认的 SysReq 功能定义方案
2. 按 IPO 类型分流落账：
   - A1 引擎配置类 → 功能定义写回 `25-*.md`
   - Bn 产品数据类 → 仅推进 `05-*.md` 状态
   - A2 场景定义类 → 仅推进 `05-*.md` 状态
3. 推进 BA / SysReq 状态，使节点进入 `wft05` 准备态
4. 记录活动 SysReq 锁待交接 / 释放

Skill B 不负责：

1. 重做 BA→SysReq 功能定义设计
2. 修改 Skill A 已确认的功能定义（四要素）、分类边界与主承接关系
3. 生成 `06-*.md` / `07-*.md` 详细定义与 PA 组件设计（由 wft05 承担）
4. 修改 wft02b 已落账的初步方案活动描述

## 二、输入与路由

### 标准输入包

1. 已确认的功能定义方案（四要素 + 分类映射结果）
2. IPO 类型判定（业务定义类 / 引擎配置类）
3. 当前资产环境（23/24/25/26/27）
4. `05-*.md` 树画像 / 状态队列 / 节点索引
5. 重入反馈（如发生回退后再次进入）

### 进入条件

目标 SysReq 节点满足以下条件时可进入本 Skill：

1. 当前状态为 `待功能定义确认`（由 wft03a Step 6 的 `[同意]` 分支在 `可确认方案` 通过后设置）
2. 已有 `选择进入wft03b` 记录，或已有 `wft03a` 活动锁释放 / 转交记录

### 活动 SysReq 节点锁定

`wft03b` 同一时间只能允许一个 SysReq 节点处于资产落账处理状态。一个 SysReq 节点进入 `wft03b` 后，在其落账和状态推进未完成前，不得再让新的节点进入 `wft03b`。

活动 SysReq 节点判定：

1. `05-*.md` 中存在 SysReq 节点状态为 `待功能定义确认`、`落账中` 或其他未完成状态，且该节点已被选择进入 `wft03b`。
2. 状态队列中存在该节点的待办锁定记录，且当前待办 ID 尚未完成。

活动 SysReq 节点完成判定：

1. 该 SysReq 节点已完成落账和状态推进，到达 `待产品架构`。
2. 该节点被人类明确撤销、废弃或回退 `wft03a`。

锁定规则（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §四）：

1. Step Start 必须先检查是否存在活动 SysReq 节点，再检查新的待进入节点。
2. 若存在活动 SysReq 节点，AI 只能处理该节点的反馈、落账、状态推进或回退；不得选择新的节点。
3. 若存在多个活动 SysReq 节点，说明状态队列不一致，AI 必须停止新增处理，输出状态冲突说明，并要求人类先选择保留一个活动 SysReq 节点。
4. 只有不存在活动 SysReq 节点时，AI 才能从已具备入口许可的节点中选择一个新的节点进入本 Skill。
5. 新节点一旦进入本 Skill，AI 必须记录活动锁定关系：`活动节点ID`、`当前待办ID`、`锁定状态`、`释放条件`。

### 冻结原则

进入 Skill B 后，下列内容视为冻结输入；如需修改，必须回退 `wft03a`：

1. SysReq 节点清单
2. 功能定义（四要素）
3. 分类宿主与系统职责桶归属
4. 节点边界
5. BA→SysReq 主承接关系
6. 复用 / 改进 / 新增 决策

## 三、操作流程

### Step 布局总览

| Step | 名称 | 产出 | 写资产 |
|------|------|------|--------|
| Start | 前置校验与路由 | 就绪 / 结束 | — |
| A | 资产环境加载 | 资产基线 | — |
| B | 实体提取 | 落账清单 | — |
| C | 正式匹配与落账判断 | 匹配决策 + IPO 类型分流 | — |
| D | 正式写回 | 功能定义写回 + 状态更新 | `25-*.md`（仅 A1）/ `05-*.md` |
| E | 状态推进 | BA / SysReq 状态更新 + 活动锁待交接 / 释放 | `04-*.md` / `05-*.md` |
| End | 运行结束输出 | 本步完成或回退说明 | 独立成章，见 §五 |

### Step Start · 前置校验与路由

通过 `05-*.md` 的树画像 / 状态队列 / 节点索引定位本轮 SysReq 节点正文块，读取该节点状态、功能定义方案、反馈区和活动锁状态，按以下规则进入后续步骤：

入口许可 SysReq 节点包括两类：

1. `05-*.md` 状态队列中 `当前状态=待功能定义确认`，且已有 `选择进入wft03b` 记录或 `wft03a` 活动锁释放 / 转交记录。
2. 本 Skill 回退记录中已被标记为 `待重入` 的 SysReq 节点。

| 条件 | 路由 |
|------|------|
| SysReq 节点 = `待产品架构`，人类已回复 `选择进入wft05`，但活动锁尚未释放或转交 | 只记录活动 SysReq 锁释放 / 转交到 `wft05`，不再重复落账 |
| SysReq 节点 = `待产品架构`，且活动锁尚未释放或转交 | 输出 `等待选择进入wft05`，当前待办 ID 为该 SysReq 节点 ID |
| 存在一个活动 SysReq 节点 | 只处理该活动节点的反馈、落账、状态推进或回退；不得进入新节点 |
| 存在多个活动 SysReq 节点 | 停止执行，输出状态冲突说明，要求人类选择保留一个活动 SysReq 节点 |
| 存在多个入口许可 SysReq 节点，且不存在活动 SysReq 节点 | 输出 `等待选择进入wft03b`，当前待办 ID 为 AI 推荐的一个建议方案包 ID |
| SysReq 节点 = `待功能定义确认`，已有 `选择进入wft03b` 记录或 `wft03a` 锁释放 / 转交记录，且不存在活动 SysReq 节点 | 执行 Step A → Step B → Step C，并写入活动 SysReq 节点锁定关系 |
| 仅存在一个 SysReq 节点 = `待功能定义确认`，但既无 `选择进入wft03b` 也无 `wft03a` 锁释放 / 转交记录 | 输出 `等待选择进入wft03b`，当前待办 ID 为该节点 ID |
| SysReq 节点状态不是 `待功能定义确认` / `待产品架构` | 输出不可进入说明后结束 |

### Step A · 资产环境加载

1. 加载 `05-*.md` 当前 SysReq 节点功能定义方案与状态表
2. 加载 `04-*.md` 中对应的 BA 节点
3. 加载 `23-eos-output-architecture.md`，定位目标 BA 节点所属产品路径
4. 加载 `24-eos-business-definitions.md`，读取目标 BA 节点匹配的业务定义
5. 加载 `25-eos-engine-models.md`，读取引擎模型与配置 IPO 类型（仅 A1 引擎配置类落账时需要）

### Step B · 实体提取

1. 提取 SysReq 节点的功能定义（四要素）与分类映射结果
2. 提取对应 BA IPO 的承接关系
3. 判断 IPO 类型——业务定义类 / 引擎配置类（决定落账分流方向）
4. 形成本轮资产落账清单

### Step C · 正式匹配与落账判断

1. 检查依赖的上游 BA 节点与相关资产是否均已确认
2. 检查功能定义四要素是否完整
3. 检查分类合规性——分类宿主是否有效、系统职责桶是否匹配、原则是否遵守
4. 按 IPO 类型分流，确定落账目标和内容：

   | IPO 类型 | 落账目标 | 落账内容 |
   |---------|---------|---------|
   | **A1 引擎配置类** | `25-*.md` | 功能定义写回（引擎模型 / 配置 IPO 类型定义） |
   | **Bn 产品数据类** | 仅推进状态 | 不写回资产 |
   | **A2 场景定义类** | 仅推进状态 | 不写回资产 |

**判定表：**

| 判定 | 含义 | Skill B 动作 |
|------|------|--------------|
| `可直接落账` | 上游资产均已确认、功能定义完整、分类合规 | 标记为可落账，进入 Step D |
| `回退 Skill A` | 功能定义不完整、分类不合规、关键资产缺失 | 执行 §四回退动作，停止当前 Skill B |
| `暂不推进` | 上游关键资产仍为候选 / 疑似 / 待确认 | 停止写回，输出 `等待建议方案反馈`，等待上游确认后再进入 |

说明：

1. 若直接依赖的关键资产仍为候选 / 疑似 / 待确认，标记为 `暂不推进`，不得视为正式落账完成
2. 若功能定义不完整或分类不合规，标记为 `回退 Skill A`，按 §四 回退动作执行完整回退（写反馈区、改状态、释放锁、输出唯一路由）

### Step D · 正式写回

1. **A1 引擎配置类**：写回 `25-*.md`
   - 引擎模型 / 配置 IPO 类型的功能定义（四要素）
   - 功能确定性标注（PA层收敛项）
   - 承接的 BA IPO 追溯
   - 更新 `05-*.md` 状态
2. **Bn 产品数据类 / A2 场景定义类**：不写回资产
   - 仅更新 `05-*.md` 对应节点状态

### Step E · 状态推进

1. 在 `04-*.md` 中将目标 BA 节点推进为 `已架构映射`
2. 在 `05-*.md` 中将目标 SysReq 节点推进为 `待产品架构`
3. 将活动 SysReq 锁状态记为 `待进入wft05`；只有人类回复 `选择进入wft05` 或存在等效转交记录后，才记录活动 SysReq 锁释放 / 转交到 `wft05`
4. 清空已通过节点的反馈区待办信号

说明：

1. 对 `wft03` 而言，Skill B 完成后不再进入 Phase 2 下半
2. `待产品架构` 是本步完成后的下游准备态，必须通过 `等待选择进入wft05` / `选择进入wft05` 完成交接留痕后，才供 `wft05` 选入
3. A1 引擎配置类的功能定义写回后，25 资产由 wft05 消费做 PA 设计

## 四、回退规则

### 结构问题触发条件

发现以下问题时，必须回退 `wft03a`：

1. SysReq 功能定义（四要素）不完整或不准确
2. 分类映射违反分类原则（按业务对象分类、主职责不唯一等）
3. 节点分类宿主需要改
4. 节点边界需要重切
5. BA→SysReq 主承接关系需要重做
6. 复用 / 改进 / 新增 决策需要推翻

### 回退动作（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §9）

1. 立即停止当前 Skill B
2. 不继续执行后续写回
3. 在目标 SysReq 节点反馈区写入回退记录（含 `[修改]`，供 wft03a Step 6 按 [修改] 分支识别）
4. 节点状态回退为 `迭代中`
5. 记录活动 SysReq 锁转交或释放
6. 输出唯一下一步路由：`重新运行 wft03a`

若结构问题在正式写回后才被发现，则必须先撤销本轮已写入的：

1. `25-*.md` 功能定义更新（仅 A1 引擎配置类）
2. `05-*.md` 状态推进
3. `04/05` 状态推进

## 五、运行结束输出（Step End）

Step End 的作用是把本轮落账结果翻译成人类下一次运行前的最小待办。由于 `wft03` 无 Phase 2 下半，本 Skill 的尾部输出只有两类结果。

推荐交互方式（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §四）：

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应节点的 `### 反馈区`、状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

### 单 ID 最小待办原则

1. 每次 Step End 只提示人类处理一个 ID。
2. 该 ID 可以是一个回退的 SysReq 方案 ID、一个待进入 `wft03b` 的建议方案包 ID、一个待选 BA 节点 ID。
3. AI 必须同时给出剩余待办 ID 数量。
4. 若上一次 Step End 已提示某个当前待办 ID，且该 ID 尚未被人类确认完成、明确切换或变为不可操作，下次运行必须继续提示同一个 ID。
5. 人类对当前 ID 的回复可以是 `同意`、`修改：……`、`驳回：……`，或 `选择：……`。
6. 只有当前 ID 已完成、被人类明确切换，或因状态变化不再可操作时，AI 才能切换当前 ID，并说明原因。

### 输出结果

本 Skill 结束时，只允许输出以下四类结果之一，每类结果必须带一个当前待办 ID：

| 输出结果 | 对应待办类型 | 当前 ID | 人类下一步 |
|---------|--------------|---------|------------|
| `已回退 wft03a` | `选择进入wft03a` | 一个 SysReq 方案 ID | 对该 ID 回复 `选择进入wft03a` 或补充结构反馈 |
| `等待选择进入wft03b` | `选择进入wft03b` | 一个建议方案包 ID | 对该 ID 回复 `选择进入wft03b` |
| `等待选择进入wft05` | `选择进入wft05` | 一个 SysReq 节点 ID | 对该 ID 回复 `选择进入wft05` |
| `当前节点完成` | `选择进入wft03b` / `选择进入wft03a` / 完成记录 | 按 §五子情况判定 | 按当前待办 ID 的待办类型回复 |

### 最小待办输出格式

```text
当前结果：已回退 wft03a / 等待选择进入wft03b / 等待选择进入wft05 / 当前节点完成
当前待办ID：
是否延续上次待办ID：
剩余待办ID数量：
待办类型：
为什么先处理这个ID：
人类只需回复：
回复会写入：
下次运行：
其他待办摘要：
```

要求：

1. `当前待办ID` 必须是单个 ID。
2. `是否延续上次待办ID` 必须写 `是 / 否`；若为 `否`，必须说明原因。
3. `剩余待办ID数量` 必须是数字。
4. `人类只需回复` 必须是一句话。
5. `其他待办摘要` 只能给数量和类型。.

### 当前节点完成后的动作

当前节点完成后的下一步由当前待办 ID 指示，按以下优先级判断：

1. **还有待进入 wft03b 的节点**（`待功能定义确认` 状态且有入口许可）→ 当前待办 ID 为一个建议方案包 ID，输出 `等待选择进入wft03b`，人类回复 `选择进入wft03b` 后进入下一节点
2. **没有待进入 wft03b 的节点，但有待 wft03a 处理的 BA 节点** → 提示一个 BA 节点 ID，并输出 `等待选择进入wft03a`
3. **确实没有 wft03a/wft03b 待办，但本轮完成节点尚未完成 wft05 交接** → 当前 ID 为本轮完成 SysReq 节点 ID，输出 `等待选择进入wft05`，人类回复 `选择进入wft05` 后记录活动锁释放 / 转交
4. **确实没有 wft03a/wft03b 待办，且本轮完成节点已完成 wft05 交接** → 当前 ID 可为本轮完成节点 ID，剩余待办 ID 为 `0`，说明该节点已完成 wft03b 交接
