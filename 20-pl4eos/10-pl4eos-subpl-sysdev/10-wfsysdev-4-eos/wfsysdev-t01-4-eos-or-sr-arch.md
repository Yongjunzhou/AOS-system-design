# OR → SR 架构定义
**Skill ID**: waterfall-t01 · **Step 1: Original Requirements → Stakeholder Requirements Architecture**

> **概述**：将预处理产出的 OR 条目级需求按三类来源分流至平台设计链和业务设计链，分别建立两链的 SR 架构定义，为 t02 详细分解提供架构侧。
>
> **输入 → 输出**：OR 条目级需求 → OR 末级分解 + SR 架构定义（两链）
>
> **适用场景**：瀑布式系统设计第1步。所有设计场景（瀑布式/逆向工程/敏捷/DevOps）中涉及 OR→SR 映射时均可参考。
>
> **版本**：v2.1 | **修订**：2026-06-04

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：00-or-preprocessing 产出的 OR 条目级需求，为设计链起点
- **关键活动**：**需求分流**——将 OR 条目按 §2.3.1 三种来源（引擎开发需求/业务配置需求/CAX工具需求）分类，分流至对应设计链
- **产出1**：OR→SR 映射导向分解——基于 t00 的单一诉求 OR 条目，按 SR 职责范围拆分为多个 OR 子条目。引用通用规范 §7.3 两阶段粒度约定：t00 止于"可被相关方独立确认的单一诉求"，t01 基于 SR 职责范围做映射导向拆分
- **产出2**：**平台链 SR 架构定义**（`02-eos-stakeholder-requirements-architecture.md` 平台链部分）——承接平台链 OR 条目，按引擎类型组织
- **产出3**：**业务链 SR 架构定义**（同一文件业务链部分）——承接业务链 OR 条目，按业务域组织

### 与三条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **详概分离** | 本步仅产出 SR 层的**架构定义子文档**（`02-*.md`），不涉及详细定义子文档（t02 产出）。两条设计链的 SR 架构在同一文件内分节组织，编号空间独立（平台链 `SR-PF-`/`SR-PNF-`，业务链 `SR-BF-`/`SR-BNF-`） |
| **文档结对** | 本步执行结对1（OR 条目级需求 ↔ SR 架构定义），两条设计链独立执行。OR 驱动：先分流→再逐链分别加载现有 SR 资产→逐条 OR 分解→三维度找宿主（复用/扩展/新增）→完善 SR 节点 |
| **基于资产** | SR 架构节点不是从零创建的，基于已有需求分类资产（领域知识结构、行业标准、已有分类体系等），遵循复用→改进→新增的资产优先决策规则。两条链各有独立的资产基线 |

## 三、操作流程

本步按五个阶段的顺序执行：需求分流（双重设计链分拆）→ 加载 OR 资产 → OR 分解与 SR 承接（两链并行）→ 闭环检查 → 审查。

核心驱动逻辑：需求从源头（OR）出发，先按三类来源分流至平台链和业务链，再逐条向对应设计链的 SR 架构找宿主。每条 OR 条目先判断涉及几个 SR 职责范围 → 分解到末级 → 用三个维度（职责方向/边界偏差/同类承接）逐条找宿主。SR 架构的调整由 OR 需求驱动，不是预设的。AI 无法判断时标记 ❓ 留给人类，不是强行推测。

操作流程概览：

```
[阶段零] 需求分流                 [阶段一] 加载 OR 资产          [阶段二] OR分解与SR承接（两链并行）
 OR按三类来源分类                  按设计链分别加载                 两条链各自独立执行：
 ├─ 引擎开发需求→平台链             ├─ 加载平台链OR                 ├─ 先加载链内SR资产→心中有数
 ├─ 业务配置需求→业务链             └─ 加载业务链OR                 ├─ 逐条OR→分解+三维度找宿主
 └─ CAX工具需求→平台链                                                   ├─ 复用✅
                                                                         ├─ 扩展/新增⚠️+强内聚性
                                                                         └─ 无法判断❓不猜
                                                                       └─ SR节点四要素完善
[阶段三] 闭环检查                 [阶段四] 审查
 两条链各自独立执行                两条链合并审查
 ├─ 分配三项自检                  ├─ 重点审 ⚠️ 和 ❓
 └─ 粒度与结对预留                └─ 确认分流正确性
```

## 四、AI 执行指南

### 4.1 跨步骤上下文管理

#### 4.1.1 上下文传递方式

6 步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

#### 4.1.2 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**（见 §六）中，供后续步骤引用。

#### 4.1.3 回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见 §7.4.3 影响范围评估）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

### 4.2 AI 经济原则

本步操作以下列顺序执行，AI 需注意：

1. **执行顺序不可逆**：OR 资产加载 → OR 分解与 SR 承接（同步）→ SR 节点完善 → 闭环检查
   核心驱动力：需求从源头（OR）出发，逐条向 SR 架构找宿主。SR 架构的调整（扩展/新增）由 OR 需求驱动，不是预设的
2. **基于资产优先**：SR 架构调整决策应先检查已有资产（复用），确实需要修改时再改进，最后才新增
3. **资产路径已知**：本步的资产是 `90-eos-system-product-data/02-eos-stakeholder-requirements-architecture.md`，其中已存在 v1 基线内容（只读状态），是 SR 架构决策的首要资产来源。`01-eos-original-requirements.md` 是本步的输入数据（00-or-preprocessing 产出的 OR 条目），不是"资产"

### 5.2 OR 分解到末级 + 同步找 SR 承接节点

> 先了解资产再动手——分解前加载现有SR架构，理解每个节点的职责边界。然后面对每条OR条目时，已经知道"哪个方向有节点、哪个方向还空着"，有方向地去分解。

```
请先加载现有SR架构资产，理解其分类结构和各节点职责边界后，再逐条处理OR条目。

OR 条目清单：[粘贴5.1的输出]

—— 第一步：加载并理解 SR 架构资产（先做到心中有数）——

资产文件：90-eos-system-product-data/02-eos-stakeholder-requirements-architecture.md

加载所有架构末级节点，对每个节点理解：
├─ 节点名称和定义 — 这个节点"管什么"？
├─ 职责边界 — 它的覆盖范围到哪里为止？相邻节点之间的分界在哪里？
├─ 已承接的OR — 这个节点目前已经承接了哪些类别的OR？

整体理解：
├─ 功能需求（SR-F）按什么逻辑分类？（按相关方？按功能域？）
├─ 非功能需求（SR-NF）分了哪几类？（质量特性/环境适应性/可实现性）
└─ 哪个分类下节点密集（已覆盖充分），哪个分类下节点稀疏（可能需要补充）

—— 第二步：逐条 OR 条目分解 + 同步找 SR 宿主 ——

对每条 OR 条目（OR-XXX），执行以下操作：

0. 【前置判断】这条OR条目是否涉及多个SR职责范围？
├─ 例如：OR 描述中同时涉及"规范化"和"架构定义" → 这两个分属不同SR节点
├─ 如果是 → 先按涉及的职责范围分拆，再分别执行下面的分解步骤
└─ 如果不是 → 直接进入步骤1

> **注**：t00 已通过清晰性五标准检查（完整性/主体明确/客体可识别/边界可判定/可测试性），本步不再重复验证。前置判断仅判断能否分配到唯一的 SR 职责范围，不需要审查 OR 条目本身的清晰性

1. 【分解】原子化分解：
├─ 原则：原子性、语义完整性、粒度一致性
├─ 触发分解条件：
│  ├─ 包含"和/并且/同时"连接多个功能 → 分解
│  ├─ 涉及多个角色或相关方 → 分解
│  └─ 涉及多个SR职责范围 → 分解（由前置判断的结果决定）
├─ 停止准则："这条 OR 条目是否可以分配到唯一的 SR 职责范围？"
│  （先独立判断职责范围唯一性，再用 SR 知识验证）——这是与 t00 停止准则的两阶段区分标志
│  - 是 → 得到末级 OR-XXX.X
│  - 否 → 继续分解
└─ 输出：OR-XXX.X [原子功能描述]

2. 【找宿主】对每个 OR-XXX.X，在已理解的 SR 知识中找承接节点：

   判断维度（用三个问题来决策）：
   ├─ Q1 职责方向匹配：此OR的核心意图，与哪个SR节点的职责方向一致？
   │  例："定义第1步流程" — SR-F-006（规范化）? SR-F-007（分解）? SR-F-001（准则体系）?
   │  判断依据：SR节点定义中描述的主语和核心动作
   │
   ├─ Q2 覆盖范围偏差：此OR描述的具体内容，是否在SR节点定义的边界之内？
   │  例：SR-F-006定义是"需求规范化规则和分类方法" → 
   │  "第1步流程" > "规范化规则"，超出了SR-F-006的边界
   │
   └─ Q3 同类型承接情况：此SR节点是否已经在承接同类OR？
      例：SR-F-001 已承接 1.1(链路结构), 10.1(SysReq层级), 14.1(占位符), 15.1(版本管理), 16.1(质量标准)
      → 它已经在承接各类"准则标准"类OR，覆盖范围较广

   根据三个问题的答案做决策：
   ├─ 条件A — 直接复用：
   │  Q1命中 + Q2在边界内 + Q3同类
   │  → OR-XXX.X → SR-F-YYY（复用）
   │  例："定义需求规范化规则" → SR-F-006（已在定义范围内）
   │
   ├─ 条件B — 扩展后承接：
   │  Q1命中 + Q2略超边界 + Q3同类
   │  → 标记 SR-F-YYY（改进-扩展），记录扩展理由
   │  例："定义第1步全流程"略超SR-F-006的"规范化"范围
   │       → 扩展SR-F-006定义：从"规范化规则"扩展为"第1步操作规程"
   │
   ├─ 条件C — 新增：
   │  Q1无命中 + Q2不在任何边界内 + Q3无同类
   │  → 记录需在哪个父节点下新增，写明新增原因
   │
   └─ 条件D — ❓ 无法判断：
       Q1命中了多个方向 / Q2边界完全模糊 / Q3信息不足以对比
       → 标记❓，说明卡在哪个维度上，**不要强行推测**，留给人类

3. 【记录】同步记录本步决策：
├─ 承接的SR节点：[SR-F-YYY / 新增]
├─ 资产决策：[复用 / 改进-扩展 / 新增]
└─ 决策理由：引用以上三个判断维度的结果

⚠️ 关键判断原则：
- "职责方向一致"的含义：OR的核心意图与SR节点的定义动词匹配
  （如"定义...规则"→准则类节点、"提供...工具"→工具类节点）
- "超边界"的含义：OR具体内容超出了SR节点定义中列举的范围
  （如SR节点定义写"规范化"但OR描述"流程定义"）
- 不是"感觉上接近"，而是用三个维度可验证地判断
- 同一个SR节点可承接多条OR末级（N:1）
- ⚠️ **扩展和新增的强内聚性检查**：
  ├─ 扩展前问"扩展后这个节点还描述一个单一职责吗？"
  │  例：SR-F-006负责"规范化规则"，扩展为"第1步操作规程"
  │  → 操作规程 ≠ 规范化规则，扩展后会变"杂"，应新增节点而非扩展
  ├─ 新增前问"这个新节点和同级兄弟节点的颗粒度一致吗？"
  │  例：相邻节点都是"某个具体设计能力"的粒度，新节点不能是一个"大类"
  └─ 颗粒度标准：该节点应可自然分解为2-4条详细末级（t02可操作）
```

## 七、审查流程

### 7.1 审核要点

- [ ] 需求分流正确性：OR 条目是否按三类来源（引擎开发/业务配置/CAX工具）正确分流至对应设计链
- [ ] 两条设计链 SR 编号空间是否正确：平台链 `SR-PF-`/`SR-PNF-`，业务链 `SR-BF-`/`SR-BNF-`
- [ ] 优先级判定（P0-P3）— 需人工确认
- [ ] 冲突/重复检测的最终决策（保留/合并/删除）
- [ ] 资产决策记录：逐条OR末级检查了哪些SR节点，复用/扩展/新增决策是否合理
- [ ] ✅/⚠️/❓ 三层标记完整：AI 无法判断的条目已标记 ❓ 未强猜
- [ ] 扩展/新增节点通过了强内聚性检查和颗粒度检查
- [ ] 功能需求和非功能需求的分离正确性
- [ ] 每个 SR 架构末级节点的粒度适合 t02 详细分解（可展开 2-4 条详细末级）
- [ ] 每个 SR 架构末级节点包含符合性分析和设计指导，设计指导需能为 t02 的详细分解提供明确指引
- [ ] 两条设计链的审查结论均已覆盖

### 7.2 审查修复流程

#### 7.2.1 审查输入

AI 完成本步后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

#### 7.2.2 AI 修复流程

收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）→ 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题** → 参考AI决策框架的跨步骤影响范围表进行影响范围评估，列出所有受影响的下游节点

#### 7.2.3 影响范围评估

| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 某步的架构定义 | 本层 + 下游已建立映射的节点 | 修改后同步更新映射 |
| 详细定义 | 下层方案 | 修改后检查下层是否受影响 |
| 映射关系 | 所有关联节点 | 需重新运行追溯验证 |
| 编号变更 | 全链路 | 需逐层更新引用，重新验证 |
