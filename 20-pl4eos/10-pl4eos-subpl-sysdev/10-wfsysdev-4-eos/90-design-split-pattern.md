# 设计开发 + 资产维护 两阶段分离模式

> **概述**：将每个瀑布式系统设计 Skill 拆分为"设计开发"和"资产维护"两个阶段，中间以人类审阅为门禁，使"基于资产的设计"原则从指导性条款变为流程强制。
>
> **版本**：v0.2 | **修订**：2026-06-17

---

## 一、模式总览

### 核心流程

```
┌──────────────────────────────────────────────────┐
│ Phase 1 · 设计开发（创造性/分析性）                │
│                                                    │
│  入口：上游就绪节点（状态=待设计开发 + 人类同意）    │
│  核心：逐条做映射、分解、定义、完善                │
│  产出：设计方案 → 写入目标资产文档                 │
│        （节点状态=待确认方案）                     │
│                                                    │
│  ── 人类审阅门禁 1 ──                              │
│  [同意] → 状态→待资产落账                          │
│  [修改] → 状态→迭代中（重入 Phase 1）              │
│  [驳回] → 重做方案                                │
│                                                    │
├──────────────────────────────────────────────────┤
│ Phase 2 上半 · 资产维护（匹配/写回/记录）          │
│                                                    │
│  步骤A 资产环境加载          ← 加载共享资产当前状态 │
│  步骤B 实体提取              ← 从设计方案中提取引用 │
│  步骤C 跨域资产匹配          ← Q1/Q2/Q3 找宿主     │
│  步骤D 写回主资产            ← 写入目标资产文档     │
│  步骤E 反向引用更新          ← 写回上游资产引用列表 │
│  步骤F 状态推进              ← 状态→已资产落账     │
│                                                    │
│  ──（机械性操作，不需要人类审阅）──                  │
│                                                    │
├──────────────────────────────────────────────────┤
│ Phase 2 下半 · 详细定义展开                        │
│                                                    │
│  步骤G 详细条目展开         ← 架构末级→详细条目    │
│  产出：详细定义子文档草案（状态=待确认细节）         │
│                                                    │
│  ── 人类审阅门禁 2 ──                              │
│  [同意] → 状态→已确认细节 → 可进入下游             │
│  [修改]/[驳回] → 重入 Phase 2 下半                 │
└──────────────────────────────────────────────────┘
```

### 关键设计决策

| 决策 | 结论 | 理由 |
|------|------|------|
| Phase 1 方案是否写入目标资产文档？ | **写入**，状态=待确认方案 | 人类审阅需要架构树上下文，状态机足以区分提案和已落账资产 |
| Phase 2 Q1/Q2/Q3 的目的？ | **跨域资产归并**，非设计重审 | 补偿分区加载导致的跨域盲区，是机械匹配而非创造性审核 |
| 详细定义谁产出？ | **消费方**产出，非生产方 | wft02 消费 STR 架构→产出 STR 详细定义；wft05 消费 SR 架构→产出 SR 详细定义 |

### 详细定义"消费方产出"规律

```
wft01 生产 STR 架构           ← 不产生 STR 详细定义（不知道下游粒度）
                             ↓
wft02 消费 STR → 设计 BA      ← 知道 STR 多细才能映射到 BA
  Phase 2 下半：写回 STR 详细定义（03-*.md）

wft03 生产 SR 架构            ← 不产生 SR 详细定义（不知道下游粒度）
                             ↓
wft05 消费 SR → 设计 PA       ← 知道 SR 多细才能映射到 PA
  Phase 2 下半：写回 SR 详细定义（06-*.md）+ PA 组件需求规格
```

## 二、统一状态机

### 节点状态

| 统一状态 | 含义 | 管理阶段 |
|---------|------|---------|
| **待设计开发** | 上游就绪，等待进入 Phase 1 | Phase 1 入口 |
| **待确认方案** | Phase 1 已完成，等待人类审阅 | 人类审阅门禁 1 |
| **迭代中** | 人类反馈修改意见，等待重新处理 | Phase 1 重入 |
| **待资产落账** | 方案已确认，等待进入 Phase 2 | Phase 2 入口 |
| **已资产落账** | Phase 2 上半完成，进入 Phase 2 下半 | Phase 2 下半入口 |
| **待确认细节** | Phase 2 下半已完成，等待人类审阅详细定义 | 人类审阅门禁 2 |
| **确认中** | 人类反馈修改意见，等待重新处理详细定义 | Phase 2 下半重入 |
| **已验证冻结** | 全链验证通过 | 终态 |
| **已废弃** | 节点停止使用 | 终态 |

### 人类决策字段

| 统一决策 | 作用 | 触发的操作 |
|---------|------|-----------|
| 同意设计开发 | 选入 Phase 1 | AI 执行设计开发 |
| 同意资产落账 | 选入 Phase 2 | AI 执行资产维护 |
| 同意详细细节 | 选入 Phase 2 下半 | AI 展开详细条目 |
| 同意验证 | 选入下游验证 | 下游 skill 执行 |

## 四、详细定义"消费方产出"模式

### 规律说明

在任何相邻两层（X → Y）中，**X 层的详细定义由消费 X 的 Y 层 Skill 产出，而非生产 X 层的 Skill 产出**。

```
生产方（X 层 Skill）            消费方（Y 层 Skill）
                      ← 知道多细才能映射
  X 架构定义（Phase 1）          Y 架构设计（Phase 1）
                                 Phase 2 下半：写回 X 详细定义
```

### 原因

生产方不知道下游消费粒度。如果 wft01 提前把 STR 展开为详细条目，可能太粗（不足以映射到 BA IPO）或太细（超出当前可知的范围）。直到 wft02 实际做 BA 设计时，才知道每个 STR 节点需要多细才能分配到唯一 IPO。

### 对应关系

| 层 | 架构由谁产出 | 详细定义由谁产出 | 写入位置 |
|----|------------|----------------|---------|
| STR | wft01 | wft02 Phase 2 下半 | 03-*.md |
| SR | wft03 | wft05 Phase 2 下半 | 06-*.md |
| PA | wft05 Phase 1 | wft05 Phase 2 下半（组件需求规格） | 07-*.md（组件定义内） |

例外：
- OR 无详细定义结构（设计链起点）
- BA 无详细定义（通用规范 §1.3 特例）
- NFR 详细定义内嵌于架构末级节点（通用规范 §7.6）

## 五、大文档内部架构设计

### 适用对象

07-eos-product-architecture.md（约 1,000 条 PA 组件 / 600KB+）等超大数据文档，需在文档内部设计分区结构以支撑 AI 按需加载。Skill 文档（~15KB/个）不适用此模式——Skill 选择层已天然实现按需加载。

### 原则

见通用设计规范 §3.9（大文档内部架构设计原则）。核心概念：

| 概念 | 说明 |
|------|------|
| **分区** | 按稳定维度（对 EOS PA 文档为业务域）划分文档节区 |
| **导航索引** | §0 列出所有分区的锚点标题和条目概览，AI 先读 §0 决定加载哪些分区 |
| **锚点定位** | `## §N` 标题锚点即分区边界，AI 通过两个锚点间的范围加载分区，不依赖行号 |
| **按需加载** | AI 在一次执行中只读导航索引 + 本次涉及的目标域分区，不加载全量 |

### EOS 07-*.md 的具体结构

见 EOS 流水线设计规范 §4.6（产品架构数据文档结构）。节区划分：

```
§0  导航索引
§1  全局管理区（§1.1 状态表 / §1.2 跨域引用 / §1.3 变更日志）
§2  经营规划域 PA（与以下各域同构）
§3  质量管理域 PA
...
§N  跨域共享定义
```

每个业务域分区内部：§X.1 架构树 / §X.2 配置方案组件定义 / §X.3 前后端组件定义 / §X.4 NFR 映射 / §X.5 SysReq→PA 映射 / §X.6 反馈区。

## 六、人机交互模型

### 各阶段的人类操作

| 阶段 | 人类操作 | 在哪个文档操作 | 操作内容 |
|------|---------|--------------|---------|
| **Phase 1 前** | 选材 | 上游状态表（如 05-*.md SysReq 状态表） | 设 人类决策=同意产品架构 |
| **Phase 1 后** | 审阅方案 | 目标资产文档（如 07-*.md §X.6 反馈区） | 写 [同意]/[修改]/[驳回] |
| **Phase 2 下半前** | — | AI 自动执行 Phase 2 上半（不需要人类介入） | — |
| **Phase 2 下半后** | 审阅详细定义 | 详细定义文档（如 06-*.md / 03-*.md） | 写 [同意]/[修改]/[驳回] |
| **全链路完成后** | 选入下一站 | 目标资产文档 §1.1 状态表 | 设 人类决策=同意进入t06 |

### Phase 1 [修改]/[驳回] 后

人类需回到上游状态表重新选材，然后重新运行 Phase 1。

### 关键约定

- 人类反馈按**组件/节点级别**标注，同一批中不同节点可以有不同反馈
- AI 在 Step Start 中检测反馈区内容，自动同步状态表，人类不直接编辑状态表
- Phase 2 上半不需要人类审阅（机械性资产归并操作）

## 七、对当前 wft05 v3.0 的调整点

根据上述模式，wft05 v3.0 需做以下调整：

| 调整 | 说明 |
|------|------|
| Step Start 改为三路分发 | 检测 05 入口（Phase 1）/ 07 入口（Phase 2）/ 迭代中（重入） |
| Step 3 移入 Phase 2 下半 | SysReq 详细定义展开不应在方案设计阶段，应在资产确认后按"消费方产出"规律执行 |
| Step 5 拆分为 Phase 1 版和 Phase 2 版 | Phase 1 只做交互总表 + NFR 映射决策（不写回）；Phase 2 做资产写回（目前的 5b/5c/5d/5f/5g） |
| Step 5g（反馈处理）自然消失 | 三种分支被两阶段门禁的自然分流替代 |
| 补充 Phase 2 下半步骤 | 组件需求规格整理，对应第二次人类审阅门禁 |
| §四/§五 补充 Phase 2 内容 | Q1/Q2/Q3 规则、跨资产写回顺序、Phase 2 提示词模板 |

## 八、试点建议

wft05（SysReq → PA）作为试点：

1. **资产维护空白最大**——当前无 Q1/Q2/Q3 框架，无跨轮引用，无反向引用
2. **PA 是设计链终点**——PA 资产的质量直接影响后续的开发、集成、运维
3. **体量适中**——能体现分离收益，又不会因拆分造成过大开销

试点目标：
- 在 wft05 中完整实施"两阶段 + 详细定义消费方产出"模式
- 补充 PA 组件 Q1/Q2/Q3 跨域匹配框架
- 建立跨资产反向引用机制
- 验证模式可复用性，产出迁移指南用于其他 skill
