# 设计开发 + 资产维护 两阶段分离模式

> **概述**：将每个瀑布式系统设计 Skill 拆分为"设计开发"和"资产维护"两个阶段，中间以人类审阅为门禁，使"基于资产的设计"原则从指导性条款变为流程强制。
>
> **版本**：v0.1 | **修订**：2026-06-17

---

## 一、模式总览

```
┌──────────────────────────────────────────────────┐
│ Phase 1 · 设计开发（创造性/分析性）                │
│                                                    │
│  入口：上游就绪节点（状态=待设计开发 + 人类同意）    │
│  核心：逐条做映射、分解、定义、完善                │
│  产出：设计方案节点（状态→待确认方案）             │
│                                                    │
│  ── 人类审阅门禁 ──                                │
│  [同意] → 状态→待资产落账                          │
│  [修改] → 状态→迭代中（重入 Phase 1）              │
│  [驳回] → 重做方案                                │
│                                                    │
├──────────────────────────────────────────────────┤
│ Phase 2 · 资产维护（匹配/写回/记录）               │
│                                                    │
│  步骤A 资产环境加载          ← 加载共享资产当前状态 │
│  步骤B 实体提取              ← 从设计方案中提取引用 │
│  步骤C 资产匹配决策          ← Q1/Q2/Q3 找宿主     │
│  步骤D 写回主资产            ← 写入目标资产文档     │
│  步骤E 反向引用更新          ← 写回上游资产引用列表 │
│  步骤F 状态推进 + 缺口检测   ← 状态→已资产落账     │
│                                                    │
│  产出：已落账的资产节点（状态→已资产落账）          │
└──────────────────────────────────────────────────┘
```

**核心思想**：两阶段之间有强制人类审阅门禁。"基于资产"不再是一个可选参考的原则，而是流程上的强制约束——不走完资产匹配，方案卡在 `待资产落账`，流不到下一站。

## 二、统一状态机

| 统一状态 | 含义 | 管理阶段 |
|---------|------|---------|
| **待设计开发** | 上游就绪，等待进入 Phase 1 | Phase 1 入口 |
| **待确认方案** | Phase 1 已完成，等待人类审阅 | 人类审阅门禁 |
| **迭代中** | 人类反馈修改意见，等待重新处理 | Phase 1 重入 |
| **待资产落账** | 方案已确认，等待进入 Phase 2 | Phase 2 入口 |
| **已资产落账** | Phase 2 完成，等待进入下游 | 下游入口 |
| **验证中** | 下游已选入验证 | 下游 |
| **已验证冻结** | 全链验证通过 | 终态 |
| **已废弃** | 节点停止使用 | 终态 |

**人类决策字段统一**：

| 统一决策 | 作用 | 触发的操作 |
|---------|------|-----------|
| 同意设计开发 | 选入 Phase 1 | AI 执行设计开发 |
| 同意资产落账 | 选入 Phase 2 | AI 执行资产维护 |
| 同意验证 | 选入下游验证 | 下游 skill 执行 |

## 五、试点建议

wft05（SysReq → PA）作为试点，原因是：

1. **资产维护空白最大**——当前无 Q1/Q2/Q3 框架，无跨轮引用，无反向引用
2. **PA 是设计链终点**——PA 资产的质量直接影响后续的开发、集成、运维
3. **体量适中**（594 行）——能体现分离收益，又不会因拆分造成过大开销

试点目标：
- 在 wft05 中完整实施"设计开发 + 资产维护"两阶段
- 补充 PA 组件 Q1/Q2/Q3 匹配框架
- 建立跨资产反向引用机制
- 验证模式可复用性，产出迁移指南用于其他 skill
