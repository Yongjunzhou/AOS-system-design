# 设计开发 + 资产维护 两阶段分离模式

> **概述**：将每个瀑布式系统设计 Skill 拆分为"设计开发"和"资产维护"两个阶段，中间以人类审阅为门禁，使"基于资产的设计"原则从指导性条款变为流程强制。
>
> **版本**：v0.5 | **修订**：2026-06-17

---

## 一、模式总览

### 核心流程

```
┌──────────────────────────────────────────────────┐
│ Phase 1 · 设计开发（创造性/分析性）                │
│                                                    │
│  入口：上游就绪节点（状态=待设计开发 + 人类同意）    │
│  核心：逐条做映射、分解、定义、完善                │
│  产出：设计方案 → 写入目标资产文档                 │
│        （节点状态=待确认方案）                     │
│                                                    │
│  ── 人类审阅门禁 1 ──                              │
│  [同意] → 状态→待资产落账                          │
│  [修改] → 状态→迭代中（重入 Phase 1）              │
│  [驳回] → 重做方案                                │
│                                                    │
├──────────────────────────────────────────────────┤
│ Phase 2 上半 · 资产维护（匹配/写回/记录）          │
│                                                    │
│  步骤A 资产环境加载          ← 加载共享资产当前状态 │
│  步骤B 实体提取              ← 从设计方案中提取引用 │
│  步骤C 跨域资产匹配          ← Q1/Q2/Q3 找宿主     │
│  步骤D 写回主资产            ← 写入目标资产文档     │
│  步骤E 反向引用更新          ← 写回上游资产引用列表 │
│  步骤F 状态推进              ← 状态→已资产落账     │
│                                                    │
│  ──（机械性操作，不需要人类审阅）──                  │
│                                                    │
├──────────────────────────────────────────────────┤
│ Phase 2 下半 · 详细定义展开                        │
│                                                    │
│  步骤G 详细条目展开         ← 架构末级→详细条目    │
│  产出：详细定义子文档草案（状态=待确认细节）         │
│                                                    │
│  ── 人类审阅门禁 2 ──                              │
│  [同意] → 状态→已确认细节 → 可进入下游             │
│  [修改]/[驳回] → 重入 Phase 2 下半                 │
└──────────────────────────────────────────────────┘
```

### 关键设计决策

| 决策 | 结论 | 理由 |
|------|------|------|
| Phase 1 方案是否写入目标资产文档？ | **写入**，状态=待确认方案 | 人类审阅需要架构树上下文，状态机足以区分提案和已落账资产 |
| Phase 2 Q1/Q2/Q3 的目的？ | **跨域资产归并**，非设计重审 | 补偿分区加载导致的跨域盲区，是机械匹配而非创造性审核 |
| 详细定义谁产出？ | **消费方**产出，非生产方 | wft02 消费 STR 架构→产出 STR 详细定义；wft05 消费 SR 架构→产出 SR 详细定义 |
| Skill A 每次处理多少节点？ | **单节点** | 基于资产的设计原则的自然推论（见下） |
| 节点处理有顺序约束吗？ | **无**，任意序 | 每个架构末级节点的四要素自描述，设计不依赖其他节点的设计结果；承接关系一致性由闭环检查事后验证 |

### Skill A 单节点处理模式

Skill A 每次执行只处理**一个**上游架构末级节点。这是全链路（wft01~wft05）的统一原则。

#### 适用性

| Skill | 单节点含义 | 说明 |
|-------|-----------|------|
| wft01 | 一个 OR 末级需求 → STR 节点 | 每条需求独立映射 |
| wft02 | 一个 STR 架构末级节点 → BA IPO | 后续节点可看到已有 IPO 并复用/合并 |
| wft03 | 一个 BA 末级 IPO → SysReq 节点 | 每个 IPO 独立分类 |
| wft04 | 一个 NFR 末级条目 → SysReq-NFR | 独立量化和组织 |
| wft05 | 一个 SysReq 末级 IPO → PA 组件 | 每个 IPO 自描述，独立设计 |

#### 成立的根本原因

基于资产的设计原则使得序列处理天然可行：

```
执行第1次：读已有资产 → 对节点A做设计 → 写回资产（状态=待确认方案）
执行第2次：读已有资产（含节点A结果）→ 对节点B做设计 → 写回
执行第3次：读已有资产（含A+B结果）→ 对节点C做设计 → 写回
...
```

每次写回后资产文档即为最新状态，后续执行天然看到前序结果。"复用 > 改进 > 新增"原则在此模式下自然生效——后处理的节点总能看到已有资产并做出正确的资产决策。

#### 无顺序约束

人类在上游状态表中选材时，不需要考虑节点类型或处理顺序。理由：

1. **每个节点自描述**：架构末级节点的四要素（I/P/O + 边界）已包含设计所需的全部信息
2. **不依赖其他节点的设计结果**：设计只依赖自身四要素 + 已有资产状态
3. **承接关系是事后验证**：节点间的输入输出一致性由 Step 5 闭环检查保障，不匹配时进入迭代修复

#### 质量风险与应对

| 风险 | 场景 | 应对 |
|------|------|------|
| 应合并未合并 | 两个节点本应归入同一宿主，但分别处理时各建一个 | Step 1 加载已有资产时 AI 应检测重叠；未检出由闭环检查或人类审阅发现；发现后迭代合并 |
| 承接链不匹配 | 配置方案组件的输出 ≠ 前后端组件期望的输入 | Step 5 闭环检查发现 → 迭代修复（不是阻断性问题，是正常收敛过程） |
| 非末级结构演化 | 新增末级节点后原分类结构不再稳定 | 非末级本来就是从末级反推的；需重组时作为"改进"类资产决策处理 |

#### 对重入效率的贡献

单节点模式使重入问题自然消失：

- 无需区分"哪些节点动、哪些不动"——每次只有一个节点
- 人类反馈后重跑 = 对同一节点再次执行 Skill A
- 上下文可控：只加载与该节点相关的资产分区
- 人类审阅聚焦：每次只审一个方案，反馈质量高

### 详细定义"消费方产出"规律

```
wft01 生产 STR 架构           ← 不产生 STR 详细定义（不知道下游粒度）
                             ↓
wft02 消费 STR → 设计 BA      ← 知道 STR 多细才能映射到 BA
  Phase 2 下半：写回 STR 详细定义（03-*.md）

wft03 生产 SR 架构            ← 不产生 SR 详细定义（不知道下游粒度）
                             ↓
wft05 消费 SR → 设计 PA       ← 知道 SR 多细才能映射到 PA
  Phase 2 下半：写回 SR 详细定义（06-*.md）+ PA 组件需求规格
```

## 二、统一状态机

### 节点状态

| 统一状态 | 含义 | 管理阶段 |
|---------|------|---------|
| **待设计开发** | 上游就绪，等待进入 Phase 1 | Phase 1 入口 |
| **待确认方案** | Phase 1 已完成，等待人类审阅 | 人类审阅门禁 1 |
| **迭代中** | 人类反馈修改意见，等待重新处理 | Phase 1 重入 |
| **待资产落账** | 方案已确认，等待进入 Phase 2 | Phase 2 入口 |
| **已资产落账** | Phase 2 上半完成，进入 Phase 2 下半 | Phase 2 下半入口 |
| **待确认细节** | Phase 2 下半已完成，等待人类审阅详细定义 | 人类审阅门禁 2 |
| **确认中** | 人类反馈修改意见，等待重新处理详细定义 | Phase 2 下半重入 |
| **已验证冻结** | 全链验证通过 | 终态 |
| **已废弃** | 节点停止使用 | 终态 |

### 人类决策字段

| 统一决策 | 作用 | 触发的操作 |
|---------|------|-----------|
| 同意设计开发 | 选入 Phase 1 | AI 执行设计开发 |
| 同意资产落账 | 选入 Phase 2 | AI 执行资产维护 |
| 同意详细细节 | 选入 Phase 2 下半 | AI 展开详细条目 |
| 同意验证 | 选入下游验证 | 下游 skill 执行 |

1. **wft01/wft03/wft04 无 Phase 2 下半**——它们不产出下游的详细定义（由消费方 wft02/wft05 负责）
2. **wft02 和 wft05 有 Phase 2 下半**——它们是 STR 和 SR 的消费方，负责展开并写回详细定义
3. **Skill A 的设计活动**在每个 wft 中性质不同，但门禁和迭代模型统一
4. **Skill B 的 Phase 2 上半高度同构**——加载 → 提取 → Q1/Q2/Q3 匹配 → 写回 → 反向引用 → 状态推进（可抽象为通用模板）
5. **实施节奏**：wft05 为试点，验证后推广到其他 Skill

## 四、详细定义"消费方产出"模式

### 规律说明

在任何相邻两层（X → Y）中，**X 层的详细定义由消费 X 的 Y 层 Skill 产出，而非生产 X 层的 Skill 产出**。

```
生产方（X 层 Skill）            消费方（Y 层 Skill）
                      ← 知道多细才能映射
  X 架构定义（Phase 1）          Y 架构设计（Phase 1）
                                 Phase 2 下半：写回 X 详细定义
```

### 原因

生产方不知道下游消费粒度。如果 wft01 提前把 STR 展开为详细条目，可能太粗（不足以映射到 BA IPO）或太细（超出当前可知的范围）。直到 wft02 实际做 BA 设计时，才知道每个 STR 节点需要多细才能分配到唯一 IPO。

### 对应关系

| 层 | 架构由谁产出 | 详细定义由谁产出 | 写入位置 |
|----|------------|----------------|---------|
| STR | wft01 | wft02 Phase 2 下半 | 03-*.md |
| SR | wft03 | wft05 Phase 2 下半 | 06-*.md |
| PA | wft05 Phase 1 | wft05 Phase 2 下半（组件需求规格） | 07-*.md（组件定义内） |

例外：
- OR 无详细定义结构（设计链起点）
- BA 无详细定义（通用规范 §1.3 特例）
- NFR 详细定义内嵌于架构末级节点（通用规范 §7.6）

## 五、大文档内部架构设计

### 适用对象

07-eos-product-architecture.md（约 1,000 条 PA 组件 / 600KB+）等超大数据文档，需在文档内部设计分区结构以支撑 AI 按需加载。Skill 文档（~15KB/个）不适用此模式——Skill 选择层已天然实现按需加载。

### 原则

见通用设计规范 §3.9（大文档内部架构设计原则）。核心概念：

| 概念 | 说明 |
|------|------|
| **分区** | 按稳定维度（对 EOS PA 文档为业务域）划分文档节区 |
| **导航索引** | §0 列出所有分区的锚点标题和条目概览，AI 先读 §0 决定加载哪些分区 |

### EOS 07-*.md 的具体结构

```

§2  经营规划域 PA（与以下各域同构）
§3  质量管理域 PA
...
§N  跨域共享定义
```

每个业务域分区内部：§X.1 架构树 / §X.2 配置方案组件定义 / §X.3 前后端组件定义 / §X.4 NFR 映射 / §X.5 SysReq→PA 映射 / §X.6 反馈区。

## 七、wft05 拆分设计详述

### 7.1 文件拆分

| 文件 | Skill | 职责 |
|------|-------|------|
| `eos-wft05a-sr2pa-design.md` | Skill A · 设计开发 | 创造性/分析性设计——结对4 的设计活动 |
| `eos-wft05b-sr2pa-settle.md` | Skill B · 资产落账 | 机械性落账——资产匹配、写回、详细定义格式化 |

### 7.2 原 Step 3（SysReq 详细定义展开）的归属

原 v3.0 中 Step 3 作为独立步骤消失。原因：两类 IPO 的处理逻辑不对称。

| IPO 类型 | Step 3 内容的去向 | 理由 |
|---------|-----------------|------|
| **业务定义类** | 整体后置到 Skill B Step G | 配置方案组件的内容从配对引擎 IPO 的 Input 可直接推得，展开只是格式化写入，不是设计发现 |
| **业务引擎能力类** | 融入 Skill A Step 3 设计活动 | 分解（产出需求条目）和分配（映射到前后端组件）是不可分的迭代活动；正式写入 06 推迟到 Skill B Step G |

### 7.3 两类 IPO 的不对称设计逻辑

这是 wft05 拆分的核心设计认知：

```
业务定义类 IPO → 配置方案组件
  信息来源：配对引擎 IPO 的 I（已详细描述所需配置信息）
  生成方式：信息重组（引擎 IPO 的 I → 7 类子组件结构）
  性质：接近机械化，模式匹配

业务引擎能力类 IPO → 前后端组件
  信息来源：EOS 前后端组件资产 + IPO 四要素 + 系统需求
  生成方式：分解分配分析（创造性设计）
  性质：依赖经验、直觉和设计智慧
```

#### 为什么业务定义类 IPO 不需要"设计"

以"工单定义 IPO"为例——其配对的"工单引擎 IPO"的 Input 已详细说明需要接收什么配置信息（字段结构、流转规则、审批权限、展示布局等）。这些信息直接对应配置方案组件的 7 类子组件。Skill A 只需确认组件类型和结构，Skill B 负责格式化写入 06。

#### 为什么业务引擎能力类 IPO 必须"设计"

前后端组件不能从 IPO 四要素直接推出——需要：
1. 将 IPO 功能分解为细粒度需求条目
2. 对每个条目寻找承接的前后端组件（已有资产中）
3. 找不到宿主时新增或改进组件
4. 分解粒度由分配需要决定，分配结果反过来驱动分解调整

这就是经典的"分解分配分析"——分解与分配交织迭代，不可分离。

### 7.4 Skill A Step 3 设计活动的内部结构

> 单节点模式下，每次只处理一个 IPO，按其类型走 3a 或 3b 路径。

```
3a · 业务定义类 IPO → 配置方案组件
    ├─ 从配对引擎 IPO 的 I 推出组件结构
    ├─ 确定主定义组件 + 子组件骨架
    └─ 资产决策（复用 > 改进 > 新增）

3b · 业务引擎能力类 IPO → 前后端组件（分解分配分析）
    ├─ 将 IPO 功能分解为细分需求条目
    ├─ 同步分配到前后端组件
    ├─ 条目无宿主 → 新增/改进组件
    └─ 每条目有唯一承接时收敛

3c · 生成非末级宿主（如需要）
    └─ 从当前末级组件集合反推稳定分类结构
```

注：原"建立输入承接关系"（配置方案信息 → 业务引擎核心输入）不再作为设计时步骤。承接关系的一致性由 Step 5 闭环检查事后验证——每个 IPO 的 I/O 已自描述其输入输出，无需等待其他 IPO 的设计结果。

### 7.5 创造性设计活动的 Skill 文档定位

#### 问题

Skill A 的核心步骤（如 wft05 Step 3b 分解分配分析、wft02 STR→BA 分解分配）是**创造性设计行为**，不是信息提取或算法推导。设计行为的本质是：

```
看到已有资产的全景
感觉到某个节点应该怎么拆
判断某个条目放在哪个组件"对"
发现放不下时决定改还是加
```

这四件事交织在一起，靠的是设计直觉（AI 的模式识别能力 / 人类的经验判断），不是步骤规程。它们不是顺序执行的 while 循环，而是同时发生的设计认知。

#### Skill 文档能写什么、不能写什么

| 能写的 | 不能写的 |
|--------|---------|
| **看**：要看什么输入（已有资产全景、节点四要素、参考资产） | 怎么想出分解方案 |
| **守**：要守什么约束（禁止跨类、复用优先、承接链闭合） | 怎么判断"该新增而非改进" |
| **交**：要交什么产出（映射关系、资产决策、承接关系） | 怎么感觉到两个节点该共享组件 |
| **完**：怎么判断做完了（每条目唯一宿主、粒度合理、边界清晰） | 怎么确定"粒度合理" |

#### Skill 文档应该写的

**设计框架声明**（看/守/交/完）+ **设计姿态声明**：

> 基于已加载的资产全景，提出你认为最合理的方案。这是创造性设计，不存在唯一正确答案；方案将经人类审阅后迭代收敛。

这句话的作用：
1. 将 AI 从"信息提取模式"切换到"方案生成模式"
2. 明确不需要推导唯一正确答案
3. 降低首版完美度预期——敢于提出"足够好"的方案

#### Skill 文档不应该写的

**伪步骤序列**——如"① 分解 → ② 分配 → ③ 处理无宿主 → ④ 收敛"。这种写法暗示线性流程，但实际上分解与分配交织发生、不可分离。写成步骤会让 AI 以 checklist 心态执行，降低设计质量。

**启发式问题清单**——如"问自己：边界在哪？该复用哪个？"。AI 在生成方案时自然会权衡这些因素，显式列出反而把创造性活动降格为 checklist。

#### 质量兜底机制

Skill 文档不规定"怎么想"，但通过以下机制保障质量：

1. **设计框架的约束层**（守）——划定不可越的边界
2. **设计框架的完成标准**（完）——定义收敛条件
3. **门禁 1**——人类审阅方案
4. **迭代重入**——人类反馈 → Skill A 再次执行 → 多轮收敛

#### 对 AI 真正有用的三件事

1. **充分的上下文**（Step 1 资产加载）——AI 的设计判断质量正比于看到的资产全景质量
2. **明确是创造性任务**（设计姿态声明）——从推导模式切换到生成模式
3. **知道会被迭代**（门禁 + 重入）——敢于给出"足够好"的首版

#### 适用范围

本定位适用于全链路 Skill A 中的创造性设计步骤：

| Skill | 创造性设计步骤 | 设计行为 |
|-------|--------------|---------|
| wft01 Skill A | OR→STR 映射 | 需求分类与架构组织 |
| wft02 Skill A | STR→BA 分解分配 | 功能分解、视角分离、IPO 组织 |
| wft03 Skill A | BA→SysReq 分类映射 | IPO 类型判定与分类宿主选择 |
| wft05 Skill A | SysReq→PA 分解分配分析 | 需求条目分解 + 组件分配 |

> wft04 Skill A（NFR 量化）偏分析性而非创造性——量化标准有客观依据，不完全适用本定位。

### 7.6 迭代执行模型

单节点模式下，迭代自然简化为对同一节点的反复执行：

```
人类选入 IPO-X（05 状态表设 人类决策=同意产品架构）
    ↓
Skill A 执行 → 对 IPO-X 给出 PA 组件方案（状态=待确认方案）
    ↓ 门禁1
人类审阅 → [同意] → 状态→待资产落账 → 进入 Skill B
         → [修改] "组件边界不对 / 该复用不该新增"
              ↓ 状态→迭代中
Skill A 再次执行 → 读取 IPO-X 的反馈 → 调整方案
    ↓ 门禁1
... 直到 [同意] ...
    ↓
人类选入 IPO-Y → Skill A 执行（此时能看到 IPO-X 的已确认结果）
```

无需"三路分发"的复杂逻辑——Step Start 只需判断：目标节点当前是"待设计开发"（首次）还是"迭代中"（重入）。

### 7.7 文档结对设计原则的体现

wft05 的 Step 3 设计活动正是结对4（SysReq 详细定义 ↔ PA 架构）的具体实现：

- 分解 IPO 为需求条目 → 产出 06 的内容（需求侧）
- 分配条目到组件 → 产出 07 的内容（方案侧）
- **一次思考、两个记录**

Skill A 在设计活动中同时产出 06 和 07 的**内容**。Skill B 将这些内容**正式落入**资产文档。分离的不是设计思考，而是"设计"与"入库"两种性质不同的操作。

### 7.8 接口契约

| 字段 | 说明 |
|------|------|
| 门禁信号 | `07` PA 状态表：待确认方案 → 待资产落账 |
| 传递内容 | Skill A 写入 07 的方案（组件树、定义、映射、NFR 决策） |
| Skill B 不重做设计 | 只做匹配/写回/格式化，不改变方案的设计内容 |
