# STR-NFR → SysReq-NFR 架构 · Skill B · 非功能定义落账与详细定义
**Skill ID**: wft04b · **Phase 2: STR-NFR → SysReq-NFR Settlement**

> **概述**：对 Skill A 已确认的 SysReq-NFR 方案执行正式资产落账与详细定义展开。上半完成指标匹配与写回 22 资产，下半完成 STR-NFR 详细定义展开至 03-*.md。`wft04` 有完整上下半。
>
> **输入 → 输出**：已通过门禁 1 的 SysReq-NFR 方案 → 上半：22 资产写回（候选升级/新增已确认条目）+ 状态推进 → 下半：03-*.md STR-NFR 详细定义展开 + 人类细节审阅
>
> **适用场景**：瀑布式系统设计第4步 · Phase 2。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.0 | **修订**：2026-06-21

## 一、职责定位

Skill B 负责：

**上半：**
1. 接收已确认的 SysReq-NFR 方案（含量化定义、宿主决策、分层约束表、候选写回标记）
2. 加载 `22-eos-nfr-taxonomy.md` 当前状态（候选区索引）
3. 对每个候选需求执行候选区匹配——有同义候选则升级，无则新增已确认条目
4. 更新 22 管理区（资产概览表 + 最近变更记录）
5. 推进 SysReq-NFR 节点状态至 `已资产落账`

**下半：**
6. 将量化的 STR-NFR 详细条目展开至 `03-*.md`（补充业务级解读、测量方法说明、约束依据）
7. 维护 03-*.md 树画像、节点索引和状态队列
8. 提交人类进行门禁 2 细节审阅

Skill B 不负责：

1. 重做 NFR 量化定义或宿主分配
2. 修改 Skill A 已确认的分层约束表
3. 修改 22 资产的分类维度（@nfr-xxx 分类维度本身）
4. 生成 `05-*.md` / `06-*.md` 的功能定义或 PA 设计（由 wft05 承担）
5. 评估指标集是否合理（已在 wft04a 门禁 1 中确认）

## 二、输入与路由

### 标准输入包

1. 已确认的 SysReq-NFR 方案（量化定义 + 分类映射 + 分层约束表 + 候选写回标记）
2. 当前 `22-eos-nfr-taxonomy.md` 资产状态
3. `03-*.md` 树画像 / 状态队列 / 节点索引（下半需要）
4. `05-*.md` 树画像 / 状态队列 / 节点索引
5. 重入反馈（如发生回退后再次进入）

### 进入条件

**上半入口：**
目标 SysReq-NFR 节点满足以下条件时可进入本 Skill 上半：
1. `当前状态 = 待资产写回`（由 wft04a Step 6 的 `[同意]` 分支在 `可确认方案` 通过后设置）
2. Skill A 已明确输出唯一路由为 `运行 wft04b`
3. 存在 `选择进入wft04b` 记录或 wft04a 活动锁释放 / 转交记录

**下半入口：**
满足以下条件时可自动进入下半：
1. 上半已完成（SysReq-NFR 节点状态 = `已资产落账`）
2. `03-*.md` 中该节点尚未生成详细定义
3. 无待处理的结构问题或回退标记

### 活动落账节点锁定

`wft04b` 同一时间只能允许一个 SysReq-NFR 节点处于落账处理状态。一个节点进入 `wft04b` 后，在其上半落账未完成前，不得再让新的节点进入 `wft04b`。

活动节点判定：

1. `05-*.md` 中存在 SysReq-NFR 节点状态为 `待资产写回`、`已资产落账` 或其他未完成状态，且该节点已被选择进入 `wft04b`。
2. 状态队列中存在该节点的待办锁定记录，且当前待办 ID 尚未完成。

活动节点完成判定：

1. 该 SysReq-NFR 节点已完成落账和状态推进，到达 `已验证冻结`。
2. 该节点被人类明确撤销、废弃或回退 `wft04a`。

锁定规则（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §四）：

1. Step Start 必须先检查是否存在活动落账节点，再检查新的待进入节点。
2. 若存在活动落账节点，AI 只能处理该节点的反馈、落账、状态推进或回退；不得选择新的节点。
3. 若存在多个活动落账节点，说明状态队列不一致，AI 必须停止新增处理，输出状态冲突说明，并要求人类先选择保留一个活动落账节点。
4. 只有不存在活动落账节点时，AI 才能从已具备入口许可的节点中选择一个新的节点进入本 Skill。
5. 新节点一旦进入本 Skill，AI 必须记录活动锁定关系：`活动节点ID`、`当前待办ID`、`锁定状态`、`释放条件`。

### 冻结原则

进入 Skill B 后，下列内容视为冻结输入；如需修改，必须回退 `wft04a`：

1. SysReq-NFR 节点清单
2. NFR 量化定义（指标名 / 数值 / 单位 / 条件 / 测量方法）
3. 分类宿主与 SysReq-NFR 分类归属
4. 分层约束表
5. 候选写回标记列表（哪些需求写回 22 资产）
6. 复用 / 改进 / 新增 决策

### Step Start 路由

| 条件 | 路由 |
|------|------|
| 存在一个活动落账节点 | 只处理该活动节点的反馈、落账或回退 |
| 存在多个活动落账节点 | 停止执行，输出状态冲突说明，要求人类选择保留一个 |
| SysReq-NFR 节点 = `待资产写回`，且不存在活动落账节点 | 执行上半（Step A → B → C → D → E），写入活动节点锁定 |
| SysReq-NFR 节点 = `已资产落账`，且 `03-*.md` 未生成详细定义 | 自动进入下半（Step F） |
| 存在多个入口许可节点，且不存在活动落账节点 | 输出 `等待选择进入wft04b`，当前待办 ID 为 AI 推荐的一个建议方案包 ID |
| 仅存在一个入口许可节点，但无选择/转交记录 | 输出 `等待选择进入wft04b`，当前待办 ID 为该节点 ID |
| `03-*.md` 详细定义反馈区有 `[同意] / [修改] / [驳回]` | 下半反馈路径（Step G） |
| 节点状态不是 `待资产写回` 或 `已资产落账` | 输出不可进入说明后结束 |

### Step 布局总览

| Step | 名称 | 职责 | 写资产 |
|------|------|------|--------|
| Start | 前置校验与路由 | 进入条件检查 + 活动节点锁定 | — |

| **上半 B** | 实体提取 | 从已确认方案提取写回实体 | — |
| **上半 C** | 候选匹配与落账判断 | 检查候选区 + 匹配决策 | — |

### Step Start · 前置校验与路由

入口许可节点包括两类：

1. `05-*.md` 状态队列中 `当前状态=待资产写回`，且已有 `选择进入wft04b` 记录或 `wft04a` 活动锁释放 / 转交记录。
2. 本 Skill 回退记录中已被标记为 `待重入` 的 SysReq-NFR 节点。

路由规则见 §二 Step Start 路由表。

### 上半：资产写回

#### Step A · 资产环境加载

1. 加载 `22-eos-nfr-taxonomy.md` 当前状态：
   - 树画像 / 资产全貌（了解当前具体需求条目索引和候选区规模）
   - 候选区索引（@cand-nfr-xxx 列表）
   - 目标分类维度的正文块（确定写回目标位置）
2. 加载 `05-*.md` 中目标 SysReq-NFR 节点的已确认方案（含量化定义、宿主决策、分层约束表、候选写回标记）
3. 加载 `05-*.md` 状态表

#### Step B · 实体提取

1. 从已确认方案中提取候选写回列表（来自 wft04a Step 5 的标记表）
2. 提取每个候选需求的四要素：
   - 指标名 / 数值 / 单位 / 条件
   - 测量方法
   - 归属分类维度 ID（@nfr-xxx）
   - 来源 STR-NFR 追溯
3. 形成本轮资产落账清单

#### Step C · 候选匹配与落账判断

对每个候选需求执行匹配：

| 条件 | 判定 | 操作 |
|------|------|------|
| 候选区存在同义候选（@cand-nfr-xxx） | **升级** | 保留原始 ID，状态→已确认，补充指标集，更新来源 |
| 候选区无同义候选 | **新增** | 创建新 ID（@nfr-xxx-REQ-{seq}），状态=已确认，写入完整指标集 |
| 候选指标集不完整或口径不明确 | **结构问题** | 标记为 `回退 wft04a` |
| 目标分类维度不存在或已废弃 | **结构问题** | 标记为 `回退 wft04a`（wft04a 的归类基准可能遗漏此维度） |

**判定表：**

| 判定 | 含义 | Skill B 动作 |
|------|------|--------------|
| `可直接落账` | 候选匹配完成、指标集完整、分类维度有效 | 标记为可落账，进入 Step D |
| `回退 wft04a` | 指标集不完整、口径不明确、分类维度缺失 | 执行 §四回退动作，停止当前 Skill B |

#### Step D · 正式写回 22 资产

1. **升级操作（若匹配到候选）：**
   - 定位目标候选条目正文块
   - 将状态从 `候选` 改为 `已确认`
   - 补充指标集字段（指标名 / 数值 / 单位 / 条件 / 测量方法）
   - 追加 SysReq-NFR 引用
   - 更新最后确认日期

2. **新增操作（若无候选可匹配）：**
   - 定位目标分类维度 @nfr-xxx 下
   - 按编号规则追加新具体需求条目：`### @nfr-xxx-REQ-{seq}`
   - 写入完整信息：状态=已确认、定义、指标口径、引用关系
   - 格式遵循 `22-eos-nfr-taxonomy.md` 的 NFR-TEMPLATE 模板

3. **更新管理区：**
   - 更新 22 资产概览表（具体需求数、最近更新日期）
   - 追加最近变更记录（日期、变更类型=新增/确认升级、变更摘要、操作Skill=wft04b）

#### Step E · 状态推进

1. 在 `05-*.md` 状态表中将目标 SysReq-NFR 节点推进为 `已资产落账`
2. 更新 `05-*.md` 树画像 / 状态队列 / 节点索引
3. 记录上半已完成的运行摘要

**说明：** `已资产落账` 是上半完成的标识状态，也是下半的入口条件。因 wft04 有 Phase 2 下半，这一步不释放活动落账节点锁——锁保持到下半完成才释放。

### 下半：详细定义展开

#### Step F · 详细条目展开

将已确认并落账的 STR-NFR 量化条目展开为 `03-*.md` 详细定义。每个 STR-NFR 条目补全以下字段：

1. **条目名称**：STR-NFR 节点名称
2. **承接的 STR 架构节点**：原始 STR-NFR 节点 ID
3. **来源需求**：OR-NFR 追溯（如有）
4. **需求描述**：STR-NFR 需求描述原文
5. **非功能类别**：质量特性 / 环境适应性 / 可实现性
6. **量化指标**：指标名 + 数值 + 单位 + 条件
7. **测量方法**：如何验证该指标（JMeter/安全审计/可用性监控等）
8. **SysReq-NFR 宿主**：映射到的 SysReq-NFR 节点 ID
9. **分层约束表**：从 wft04a 方案中引用
10. **应用范围**：全局 / 业务域 / IPO
11. **优先级**：高 / 中 / 低
12. **22 资产引用**：对应的 22 资产条目 ID（@nfr-xxx-REQ-yy）
13. **链身份**：平台链 / 业务链 / 共享
14. **匹配的 22 资产条目**：Step D 写回的条目 ID
15. **备注**：AI 推断标注 / 待确认项等

写入要求：

1. 在 `03-*.md` 中新增或更新稳定 `BLOCK`（NFR 部分节区）
2. 同步更新树画像、状态队列、节点索引和追溯引用
3. 将目标 STR-NFR 详细定义条目状态写为 `待确认细节`
4. 在 `05-*.md` 中记录详细定义写回摘要
5. 不得把 `05-*.md` 的量化方案全文复制到 `03-*.md`；`03` 写业务级可验证细节

#### Step G · 反馈重入处理

处理人类对下半详细定义的反馈。分支：

1. **`[同意]`**
   - 在 `03-*.md` 中将 STR-NFR 详细定义条目推进为 `已确认`
   - 在 `05-*.md` 中将 SysReq-NFR 节点推进为 `已验证冻结`
   - 记录活动落账节点锁释放
   - 清空已通过节点的反馈区待办信号
   - 输出 `当前节点完成`

2. **`[修改]`**
   - 按反馈调整 `03-*.md` 详细定义正文（仅修改业务级描述、测量方法说明、约束依据解读等非结构文字）
   - 保持 SysReq-NFR 主结构（量化定义 / 宿主 / 约束表）不变——**约束表内容（影响级别/影响点/说明字段）属冻结输入，不得在 Step G 修改**
   - 若人类修改请求涉及约束表字段取值，必须回退 `wft04a`，走 §四 回退动作
   - 详细定义状态保持 `待确认细节`
   - 输出 `等待建议方案反馈`

3. **`[驳回]`**
   - 仅详细定义表述问题 → 重做 Step F
   - NFR 主结构问题 → 回退 `wft04a`（需先撤销 Step D 的 22 资产写回）

## 四、回退规则

### 结构问题触发条件

发现以下问题时，必须回退 `wft04a`：

1. NFR 量化指标（指标名/数值/单位/条件）不准确
2. 宿主分配需要改
3. 分层约束表需要重做
4. 复用 / 改进 / 新增 决策需要推翻
5. 候选写回标记列表需要改（哪些需求写回 22）

### 回退动作（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §9）

1. 立即停止当前 Skill B
2. 不继续执行后续写回或下半展开
3. 在目标 SysReq-NFR 节点反馈区写入回退记录（含 `[修改]`，供 wft04a Step 6 按 [修改] 分支识别）
4. 节点状态回退为 `迭代中`
5. 记录活动落账节点锁转交或释放
6. 输出唯一下一步路由：`重新运行 wft04a`

若结构问题在正式写回后才被发现，则必须先撤销本轮已写入的：

1. `22-*.md` 具体需求条目更新（升级或新增）
2. `05-*.md` 状态推进
3. `03-*.md` 详细定义写回

## 五、运行结束输出（Step End）

Step End 的作用是把本轮落账结果翻译成人类下一次运行前的最小待办。

推荐交互方式（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §四）：

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应节点的 `### 反馈区`、状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

### 单 ID 最小待办原则

1. 每次 Step End 只提示人类处理一个 ID。
2. 该 ID 可以是一个详细定义方案 ID、一个待进入 `wft04b` 的建议方案包 ID、一个待选 STR-NFR 节点 ID。
3. AI 必须同时给出剩余待办 ID 数量。
4. 若上一次 Step End 已提示某个当前待办 ID，且该 ID 尚未被人类确认完成、明确切换或变为不可操作，下次运行必须继续提示同一个 ID。
5. 人类对当前 ID 的回复可以是 `同意`、`修改：……`、`驳回：……`，或 `选择：……`。
6. 只有当前 ID 已完成、被人类明确切换，或因状态变化不再可操作时，AI 才能切换当前 ID，并说明原因。

### 输出结果

本 Skill 结束时，只允许输出以下三类结果之一，每类结果必须带一个当前待办 ID：

| 输出结果 | 对应待办类型 | 当前 ID | 人类下一步 |
|---------|--------------|---------|------------|
| `已回退 wft04a` | `选择进入wft04a` | 一个 SysReq-NFR 方案 ID | 对该 ID 回复 `选择进入wft04a` 或补充结构反馈 |
| `待确认细节` | `确认详细定义` | 一个 STR-NFR 详细定义 ID | 对该 ID 回复 `同意 / 修改 / 驳回` |
| `当前节点完成` | `选择进入wft04b` / `选择进入wft04a` / 完成记录 | 按子情况判定 | 按当前待办 ID 的待办类型回复 |

### 当前节点完成后的可选动作

当前节点完成后的下一步由当前待办 ID 指示，按以下优先级判断：

1. **还有待进入 wft04b 的节点**（`待资产写回` 状态且有入口许可）→ 当前待办 ID 为一个建议方案包 ID，输出 `等待选择进入wft04b`
2. **没有待进入 wft04b 的节点，但有待 wft04a 处理的 STR-NFR 节点** → 提示一个 STR-NFR 节点 ID，输出 `等待选择进入wft04a`
3. **确实没有 wft04a/wft04b 待办** → 当前 ID 可为本轮完成节点 ID，剩余待办 ID 为 `0`，说明该节点可进入 `wft05`

要求：

1. `当前待办ID` 必须是单个 ID。
2. `是否延续上次待办ID` 必须写 `是 / 否`；若为 `否`，必须说明原因。
3. `剩余待办ID数量` 必须是数字。
4. `人类只需回复` 必须是一句话。
5. `其他待办摘要` 只能给数量和类型。
