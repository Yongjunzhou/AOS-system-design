# SysReq → PA 架构 · Skill B · 资产落账
**Skill ID**: wft05b · **Phase 2: SysReq → PA Settlement**

> **概述**：对 Skill A 已确认的 PA 组件方案执行资产落账（Phase 2 上半）和详细定义展开（Phase 2 下半）。上半为机械性操作（跨域匹配、写回、反向引用），无需人类审阅；下半产出 SysReq 详细定义和 PA 组件需求规格，经人类审阅后冻结。
>
> **输入 → 输出**：已确认 PA 方案（`07-*.md`，状态=待资产落账）→ 资产落账 + SysReq 详细定义（`06-*.md`）+ PA 组件需求规格（`07-*.md`）
>
> **适用场景**：瀑布式系统设计第5步 · Phase 2。Skill A（`eos-wft05a-sr2pa-design.md`）方案通过门禁 1 后运行本 Skill。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v5.0 | **修订**：2026-06-21

## 一、设计角色

- **输入**：`07-*.md` PA 状态表中**当前状态=待资产落账**（进入上半）或**已资产落账/确认中**（进入下半）的目标节点
- **产出1**：资产落账——跨域匹配结果写回 `07-*.md`，反向引用更新 `05/22/23`
- **产出2**：SysReq 详细定义（`06-*.md`）——消费方产出规律：wft05 作为 SR 的消费方产出 SR 详细定义
- **产出3**：PA 组件需求规格——功能清单、接口、数据、验收标准写入 `07-*.md`

### "消费方产出"规律

```
wft03 生产 SR 架构    ← 不知道下游粒度
                    ↓
wft05 消费 SR → 设计 PA  ← 知道 SR 多细才能映射到 PA
  Phase 2 下半：写回 SR 详细定义（06-*.md）+ PA 组件需求规格
```

### Skill B 的本质

Skill B **不做设计决策**——方案在 Skill A 中已确认。Skill B 的操作全部是：
- 匹配（Q1/Q2/Q3 跨域资产归并）
- 写回（将方案正式落入资产文档体系）
- 格式化（将设计思考结果组织为标准化详细定义和规格文档）

### Skill B 的主边界

`wft05b` 负责把 `wft05a` 已确认的 PA 主结构正式落账并展开成可开发规格，但**不得重做主结构设计**。

1. **上半负责落账，不负责改结构**：可以决定正式写到哪里、补哪些引用，不能重新决定谁来承接、怎么切边界
2. **下半负责展开，不负责改结构**：可以把详细定义和组件规格写细写全，不能借细化之名推翻 `wft05a` 已冻结的主结构
3. **凡触及主结构的问题，一律回退 `wft05a`**：组件是否存在、组件类型、组件边界、宿主分配、复用/改进/新增 决策一旦需要改动，必须停止 `wft05b` 并回退

### 活动落账节点锁定

`wft05b` 同一时间只能允许一个 SysReq 节点处于落账处理状态。一个节点进入 `wft05b` 后，在其验证冻结未完成前，不得再让新的节点进入 `wft05b`。

**活动节点判定：**

1. `07-*.md` 中存在 SysReq 节点状态为 `待资产落账`、`已资产落账`、`确认中` 或其他未完成状态，且该节点已被选择进入 `wft05b`。
2. 状态队列中存在该节点的待办锁定记录，且当前待办 ID 尚未完成。

**活动节点完成判定：**

1. 该 SysReq 节点已完成落账和状态推进，到达 `已验证冻结`。
2. 该节点被人类明确撤销、废弃或回退 `wft05a`。

**锁定规则：**

1. Step Start 必须先检查是否存在活动落账节点，再检查新的待进入节点。
2. 若存在活动落账节点，AI 只能处理该节点的反馈、落账、状态推进或回退；不得选择新的节点。
3. 若存在多个活动落账节点，说明状态队列不一致，AI 必须停止新增处理，输出状态冲突说明，并要求人类先选择保留一个活动落账节点。
4. 只有不存在活动落账节点时，AI 才能从已具备入口许可的节点中选择一个新的节点进入本 Skill。
5. 新节点一旦进入本 Skill，AI 必须记录活动锁定关系：`活动节点ID`、`当前待办ID`、`锁定状态`、`释放条件`。

### 标准输入包

1. 已确认的 PA 组件方案（组件清单、资产决策、边界摘要、NFR 决策）
2. 当前 `07-*.md` PA 相关分区资产
3. `05-*.md` / `06-*.md` 树画像 / 状态队列 / 节点索引
4. 重入反馈（如发生回退后再次进入）

## 二、前置条件与状态机

### Step Start 路由逻辑

| 条件 | 路由 |
|------|------|
| 存在一个活动落账节点 | 只处理该活动节点的反馈、落账、状态推进或回退 |
| 存在多个活动落账节点 | 停止执行，输出状态冲突说明，要求人类选择保留一个 |
| SysReq 节点 = `待资产落账`，且不存在活动落账节点 | 执行 Phase 2 上半（Step A→F），写入活动节点锁定 |
| SysReq 节点 = `已资产落账`，且 `06/07` 未生成详细定义 | 自动进入 Phase 2 下半（Step G→H→End） |
| SysReq 节点 = `确认中`，且不存在活动落账节点 | Phase 2 下半重入（加载反馈→Step G→H→End） |
| 存在多个入口许可节点，且不存在活动落账节点 | 输出 `等待选择进入wft05b`，当前待办 ID 为 AI 推荐的一个建议方案包 ID |
| 仅存在一个入口许可节点，但无选择/转交记录 | 输出 `等待选择进入wft05b`，当前待办 ID 为该节点 ID |
| SysReq 节点 = `迭代中` | 提示"已回退 Skill A，请运行 wft05a" |
| 其他状态 | 输出不可操作说明后结束 |

### 节点状态（Skill B 管理的范围）

| 状态 | 含义 | 转迁 |
|------|------|------|
| **待资产落账** | Skill A 方案已通过门禁 1 | → 已资产落账（Phase 2 上半完成后） |
| **已资产落账** | 上半完成，等待下半 | → 待确认细节（Phase 2 下半完成后） |
| **待确认细节** | 下半完成，等待人类审阅 | 人类 [同意] → 已验证冻结；人类 [修改] → 确认中 |
| **确认中** | 人类反馈修改 | → 待确认细节（重入下半后） |
| **迭代中** | Phase 2 发现结构问题，已回退 Skill A | → 返回 `wft05a` 重入 |
| **已验证冻结** | 终态 | — |

#### Phase 2 上半回退规则

若上半任一时点发现的是**结构问题**而非**规格问题**，则必须立即停止 `wft05b`，不得继续下半。

结构问题包括但不限于：

1. 组件是否存在需要改
2. 组件类型需要改
3. 组件边界需要重切
4. SysReq → PA 宿主分配需要重做
5. 复用 / 改进 / 新增 决策需要推翻
6. 业务定义类 IPO 与引擎配置类 IPO 的依附关系不成立
7. 阻断型 NFR 导致已确认主结构失效

发生回退时执行：

1. **立即停止上半继续推进**
2. **不进入 Phase 2 下半**
3. **状态回退为 `迭代中`**
4. **在 `07-*.md` 对应节点反馈区写入回退记录**
5. **输出唯一下一步路由：重新运行 `wft05a`**

回退时允许保留：

- 回退原因记录
- 受影响内容说明
- 建议重审点

回退时不得正式提交：

- `07` 的正式落账位置变更
- `05` 的状态成功推进
- `22/23` 的正式引用更新
- `06` 的详细定义展开
- `07` 的组件需求规格展开

若结构问题是在 Step D / Step E 已执行后才被识别，则必须先**撤销本轮已写入的正式落账结果**，再执行回退。撤销范围至少包括：

1. 本轮写入 `07` 的落账位置与组件定义变更
2. 本轮写入 `05` 的状态推进
3. 本轮写入 `22/23` 的引用更新

撤销完成后，节点方可正式回退为 `迭代中`，避免留下半落账资产污染

#### PA 末级节点内容要求

**配置方案组件**应达到"**可开发的定义宿主规格**"粒度，至少包含：组件定义、承接上游条目、配置方案结构（元数据/结构/规则/权限/展示/引用/校验发布）、输入输出、依附引擎、约束与校验、符合性分析、开发/配置指导。

**前后端组件**应达到"**开发可执行定义**"粒度，至少包含：组件定义（前端配置/前端运行/后端定义处理/后端运行执行/接口与集成）、承接上游条目、职责边界、输入（配置方案信息/运行上下文/外部事件）、输出、内部处理逻辑、对外接口、NFR 约束、开发指导。

#### Step End · 输出交互（门禁 2）

Step End 的作用是把本轮落账结果翻译成人类下一次运行前的最小待办。

推荐交互方式（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §四）：

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应节点的 `### 反馈区`、状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

#### 单 ID 最小待办原则

1. 每次 Step End 只提示人类处理一个 ID。
2. 该 ID 可以是一个详细定义方案 ID、一个待进入 `wft05b` 的建议方案包 ID、一个待选 SysReq 节点 ID。
3. AI 必须同时给出剩余待办 ID 数量。
4. 若上一次 Step End 已提示某个当前待办 ID，且该 ID 尚未被人类确认完成、明确切换或变为不可操作，下次运行必须继续提示同一个 ID。
5. 人类对当前 ID 的回复可以是 `同意`、`修改：……`、`驳回：……`，或 `选择：……`。
6. 只有当前 ID 已完成、被人类明确切换，或因状态变化不再可操作时，AI 才能切换当前 ID，并说明原因。

#### 输出结果

本 Skill 结束时，只允许输出以下三类结果之一，每类结果必须带一个当前待办 ID：

| 输出结果 | 对应待办类型 | 当前 ID | 人类下一步 |
|---------|--------------|---------|------------|
| `已回退 wft05a` | `选择进入wft05a` | 一个 PA 方案 ID | 对该 ID 回复 `选择进入wft05a` 或补充结构反馈 |
| `待确认细节` | `确认详细定义` | 一个详细定义 ID | 对该 ID 回复 `同意 / 修改 / 驳回` |
| `当前节点完成` | `选择进入wft05b` / `选择进入wft05a` / 完成记录 | 按子情况判定 | 按当前待办 ID 的待办类型回复 |

#### 当前节点完成后的可选动作

当前节点完成后的下一步由当前待办 ID 指示，按以下优先级判断：

1. **还有待进入 wft05b 的节点**（`待资产落账` 状态且有入口许可）→ 当前待办 ID 为一个建议方案包 ID，输出 `等待选择进入wft05b`
2. **没有待进入 wft05b 的节点，但有待 wft05a 处理的 SysReq 节点** → 提示一个 SysReq 节点 ID，输出 `等待选择进入wft05a`
3. **确实没有 wft05a/wft05b 待办** → 当前 ID 可为本轮完成节点 ID，剩余待办 ID 为 `0`，说明该节点可进入 `t06`

#### 最小待办输出格式

```text
当前结果：已回退 wft05a / 待确认细节 / 当前节点完成
当前待办ID：
是否延续上次待办ID：
剩余待办ID数量：
待办类型：
为什么先处理这个ID：
人类只需回复：
回复会写入：
下次运行：
其他待办摘要：
[写回摘要] 已落账 PA 组件至 07-*.md，更新反向引用至 05/22/23
[详细定义] 已展开 SR 详细定义至 06-*.md，PA 组件规格至 07-*.md
```

要求：

1. `当前待办ID` 必须是单个 ID。
2. `是否延续上次待办ID` 必须写 `是 / 否`；若为 `否`，必须说明原因。
3. `剩余待办ID数量` 必须是数字。
4. `人类只需回复` 必须是一句话。
5. `其他待办摘要` 只能给数量和类型。

#### 回退记录模板

当 `wft05b` 回退 `wft05a` 时，回退原因的权威记录写在 `07-*.md` 对应节点的反馈区。

固定模板：

```md
### [wft05b 回退记录]

- 回退结论：回退至 `wft05a`
- 回退原因类型：`组件类型问题` / `组件边界问题` / `宿主分配问题` / `复用决策问题` / `依附关系问题` / `阻断型NFR问题` / `其他结构问题`
- 回退原因说明：...
- 受影响内容：...
- 建议重审点：...
- 下一步路由：`重新运行 wft05a`
```
