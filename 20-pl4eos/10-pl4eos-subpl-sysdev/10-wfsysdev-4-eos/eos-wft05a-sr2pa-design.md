# SysReq → PA 架构 · Skill A · 设计开发
**Skill ID**: wft05a · **Phase 1: SysReq → PA Design**

> **概述**：每次执行处理**一个** SysReq 功能架构末级节点（IPO），执行创造性/分析性设计活动，产出 PA 组件方案。按 IPO 类型走不对称设计路径：业务定义类 IPO 为模式匹配（机械性），引擎配置类 IPO 为分解分配分析（创造性）。
>
> **输入 → 输出**：一个 SysReq 末级 IPO（`05-*.md`）→ PA 组件方案（`07-*.md`，状态=待确认方案）
>
> **适用场景**：瀑布式系统设计第5步 · Phase 1。设计方案经人类审阅通过后，由 Skill B（`eos-wft05b-sr2pa-settle.md`）执行资产落账和详细定义展开。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v5.0 | **修订**：2026-06-21

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：`05-*.md` SysReq 状态表中人类选入的**一个** SysReq 功能架构末级节点（IPO）
- **产出**：PA 组件方案写入 `07-*.md`（状态=待确认方案），供人类审阅

本步按 IPO 类型生成两类 PA 末级组件：
- **业务定义类 IPO → 配置方案组件**
- **引擎配置类 IPO → 前后端组件**

### 与三条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **详概分离** | 本步产出 PA 层的**架构方案**（`07-*.md`）；详细定义（`06-*.md`）由 Skill B Phase 2 下半产出 |
| **文档结对** | 本步执行结对4（SysReq 详细定义末级 ↔ PA 架构）的设计思考——一次思考同时形成 06 和 07 的内容，正式落入由 Skill B 完成 |
| **基于资产** | 每次执行时读取已有 PA 资产，优先复用 > 改进 > 新增。单节点模式使后处理的节点总能看到前序已确认结果 |

### 单节点处理模式

每次执行只处理**一个** SysReq 架构末级节点（IPO）。这是全链路（wft01~wft05）的统一原则。

**成立的根本原因**——基于资产的设计原则使序列处理天然可行：

```
执行第1次：读已有资产 → 对 IPO-A 做设计 → 写回（状态=待确认方案）
执行第2次：读已有资产（含 A 结果）→ 对 IPO-B 做设计 → 写回
```

**无顺序约束**——人类选材时不需考虑 IPO 处理顺序：

1. **每个节点自描述**：四要素（I/P/O + 边界）已包含设计所需全部信息
2. **不依赖其他节点的设计结果**：只依赖自身四要素 + 已有资产状态
3. **承接关系是事后验证**：节点间 I/O 一致性由 Step 5 闭环检查保障

### 活动输入节点锁定协议

`wft05a` 同一时间只能允许一个上游输入节点处于活动处理状态。活动输入节点未处理结束前，不得让新的上游输入节点进入本 Skill。

**活动输入节点判定：**

以下任一情况成立时，应视为存在活动输入节点：

1. `07-*.md` 中已存在引用某个 SysReq 节点的 PA 组件方案节点，且该方案节点仍处于 `待确认方案`、`迭代中` 或其他未完成状态。
2. 状态队列、反馈区或运行摘要中存在该输入节点的待办锁定记录。
3. SysReq 状态队列中该输入节点已被标记为进入 `wft05a`，但尚未记录 `wft05a` 处理完成。

**活动输入节点完成判定：**

以下任一情况成立时，活动输入节点才可视为完成或释放：

1. 对应 PA 方案已达到 `可确认方案`，且人类已确认进入 `wft05b`。
2. 该输入节点被人类明确撤销、废弃、跳过或退回上游。

**锁定规则：**

1. Step Start 必须先检查是否存在活动输入节点，再检查新的待进入节点。
2. 若存在一个活动输入节点，AI 只能重入该节点、处理该节点反馈、输出该节点当前待办，或提示进入正确的下游 Skill；不得选择新的输入节点。
3. 若存在多个活动输入节点，说明状态队列不一致，AI 必须停止新增处理，输出状态冲突说明，并要求人类先选择保留一个活动输入节点。
4. 只有不存在活动输入节点时，AI 才能从待进入队列中选择一个新的输入节点。
5. 新输入节点一旦进入本 Skill，AI 必须在状态队列、方案节点或运行摘要中记录活动锁定关系：`活动输入节点ID -> 方案节点ID / 当前待办ID`。

### 标准输入包

1. 目标 SysReq 末级 IPO（含四要素、边界描述、IPO 类型判定）
2. 当前 `07-*.md` PA 相关分区资产（PA 架构树、已有组件定义）
3. `05-*.md` 树画像 / 状态队列 / 节点索引与 SysReq 节点正文块
4. `07-*.md` 树画像 / 状态队列 / 节点索引（工作目标文档）
5. NFR 约束包（从 wft04 分层约束表装配）
6. 重入反馈（若为重入路径）

### 跨 Skill 入口选择与活动锁转交

`wft05a` 在节点完成门禁 1 并确认进入 `wft05b` 时，必须写入入口选择记录，供 `wft05b` Step Start 识别。

**转交记录至少包含：**

1. 活动输入节点 ID
2. 下游待处理节点或建议方案包 ID
3. 当前状态到位说明
4. 人类选择动作：`选择进入wft05b`
5. 活动锁状态：`已释放` 或 `已转交wft05b`
6. 下一步 Skill：`wft05b`

### 两类 IPO 的不对称设计逻辑

```
业务定义类 IPO → 配置方案组件
  信息来源：配对引擎 IPO 的 I（已详细描述所需配置信息）
  生成方式：信息重组（模式匹配）

引擎配置类 IPO → 前后端组件
  信息来源：EOS 前后端组件资产 + IPO 四要素 + 系统需求
  生成方式：分解分配分析（创造性设计）
```

### Step Start 路由

| 条件 | 路由 |
|------|------|
| 存在一个活动输入节点 | 只处理该活动节点对应的反馈、重入或当前待办，不得进入新节点 |
| 存在多个活动输入节点 | 停止执行，输出状态冲突说明，要求人类选择保留一个活动输入节点 |
| SysReq 节点 = `待设计开发`，且不存在活动输入节点 | 新增路径，写入活动输入节点锁定关系 |
| SysReq 节点 = `迭代中` | 重入路径（Step 3 或 Step 6，按反馈区内容路由） |
| SysReq 节点 = `待确认方案` 且反馈区为空 | 输出 `等待PA方案反馈`，不得覆盖既有方案 |
| SysReq 节点 = `待确认方案` 且反馈区有 `[同意] / [修改-*] / [驳回]` | 方案反馈路径（Step 6） |
| SysReq 节点 = `待资产落账` | 提示运行 `wft05b` |
| SysReq 节点 = `已资产落账` / `确认中` / `待确认细节` | 提示运行 `wft05b` |
| SysReq 节点 = `已验证冻结` / `已废弃` | 输出不可操作说明后结束 |
| 其他状态 | 输出不可操作说明后结束 |

| 4 | 检查资产存在性 | 确认 `05-*.md` / `07-*.md` 存在可读 |
| 5 | 路由决策 | 按 Step Start 路由表分发；若为重入路径，加载反馈区 |

| 5 | 加载反馈（重入时） | 若状态=迭代中，读取 `07-*.md` 反馈区人类修改意见 |

### Step 2 · 分析活动

| # | 操作 | 说明 |
|---|------|------|
| 1 | 输入完整性验证 | IPO 四要素完整可用（I/P/O + 边界均非空） |

### Step 3 · 设计活动

> 按目标 IPO 类型分流。这是结对4 的核心设计活动——一次思考同时产出 06 和 07 的**内容**（正式落入资产由 Skill B 完成）。

**3a · 业务定义类 IPO → 配置方案组件**

| # | 操作 | 说明 |
|---|------|------|
| 1 | 定位配对引擎 IPO | 找到与目标业务定义类 IPO 配对的引擎配置类 IPO |
| 2 | 提取配置信息需求 | 从配对引擎 IPO 的 Input 提取所需配置信息 |
| 3 | 组织组件结构 | 组织为主定义组件 + 7 类子组件结构 |
| 4 | 资产决策 | 对每个组件执行：复用已有 > 改进已有 > 新增 |

**3b · 引擎配置类 IPO → 前后端组件（分解分配分析）**

> **设计姿态**：基于已加载的资产全景，提出你认为最合理的分解分配方案。这是创造性设计，不存在唯一正确答案；方案将经人类审阅后迭代收敛。
>
> **选择依据**：引擎配置类 IPO 满足 91-doc-paired-skill-spec §4.1.1 的以下条件，归入直觉驱动模式（分解分配分析）——输出空间不可穷举（条件一）、无固定映射路径（条件二）、质量依赖人类判断边界合理性（条件三）、方案强依赖当前资产语境（条件四）。

| | 内容 |
|--|------|
| **看** | 已有 PA 资产全景、目标 IPO 四要素、引擎能力全景、EOS 组件参考 |
| **守** | 禁止跨类生成、复用 > 改进 > 新增、承接链闭合、新增组件须与现有资产风格一致（粒度/命名/边界模式/层级位置/依赖关系——参见 91-doc-paired-skill-spec §5.2） |
| **交** | 条目→组件映射、资产决策（复用/改进/新增）、承接关系 |
| **完** | 每条目有唯一宿主、粒度合理、边界清晰 |

**收敛指南**：

| 情况 | 合理倾向 |
|------|---------|
| IPO 的 I 字段直接对应已有引擎能力 | 优先分配，不做多余组件拆分 |
| IPO 的 I 字段覆盖多个引擎能力 | 按能力领域拆分为多个承接组件 |
| 前后端职责可明确分离 | 拆为前端组件 + 后端组件 |
| 前后端职责紧密耦合 | 合为一个能力主组件，不强行拆分 |
| 已有同类组件且可扩展 | 优先扩展，不新增 |
| 无同类组件可参考 | 按 IPO 语义分解，保持与现有资产近似的粒度 |

**迭代预期**：预期 2~3 轮收敛。收敛信号——组件边界稳定、无未闭合承接关系、NFR 约束已映射。

**人类在门禁 1 的审查焦点**：
1. 组件边界是否合理——各组件职责是否不重叠、不遗漏
2. 资产决策是否正确——复用/改进/新增的判断是否合理
3. 组件风格是否与现有资产协调——新增组件的粒度、命名、层级是否与同类一致
4. 承接关系是否闭合——所有需求条目有唯一宿主

质量兜底：门禁 1（人类审阅上述焦点）+ 迭代重入。

**3c · 非末级宿主生成（如需要）**

从当前末级组件集合反推稳定分类结构。总原则：先有末级组件，再生成非末级宿主。

### Step End · 输出交互（门禁 1）

Step End 的作用是把本轮设计结果翻译成人类下一次运行 `wft05a` 前的最小待办。

推荐交互方式（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §四）：

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应节点的 `### 反馈区`、状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

#### 单 ID 最小待办原则

1. 每次 Step End 只提示人类处理一个 ID。
2. 该 ID 可以是一个 PA 组件方案 ID、一个待选 SysReq 节点 ID、一个待进入 `wft05b` 的建议方案包 ID。
3. AI 必须同时给出剩余待办 ID 数量，帮助人类知道当前队列规模。
4. 若上一次 Step End 已提示某个当前待办 ID，且该 ID 尚未被人类确认完成、明确切换或变为不可操作，下次运行必须继续提示同一个 ID。
5. 人类对当前 ID 的回复可以是 `同意`、`修改：……`、`驳回：……`，或 `选择：……`。
6. 只有当前 ID 已完成、被人类明确切换，或因状态变化不再可操作时，AI 才能按阻断优先级、状态时间和推荐顺序自动选择下一个当前 ID，并说明切换原因。
7. 当前 ID 处理完成后，下一次运行 `wft05a` 再提示下一个 ID。
8. AI 可以附带其他待办摘要，但不得要求人类同时审阅或选择多个 ID。

#### 门禁 1 审查口径

审查的是：

1. PA 主结构方案是否成立——组件类型、组件边界、复用/改进/新增 决策是否合理
2. 承接关系是否闭合——所有 IPO 条目有唯一 PA 组件宿主
3. NFR 约束映射是否完整
4. IPO 类型判定是否正确

不审：

1. `07-*.md` 的正式落账位置（由 wft05b 负责）
2. `06-*.md` / `07-*.md` 的组件详细规格（由 wft05b 下半负责）

#### 方案状态

| 方案状态 | 含义 | 是否可放行至 wft05b |
|---------|------|-------------------|
| `阶段性方案` | 存在阻断项 / 关键缺口 / 阻断型 NFR 未明确 | 否 |
| `可确认方案` | 组件结构合理、承接关系闭合、NFR 已映射 | 是 |

#### Step End 待办类型

| 待办类型 | 何时出现 | 当前 ID | 人类只需做什么 | 下次运行 |
|---------|----------|---------|----------------|----------|
| `反馈PA方案` | 本轮已生成或修订 PA 方案，状态为 `待确认方案` | 一个 PA 组件方案 ID | 对该 ID 回复 `同意 / 修改 / 驳回` | `wft05a` |
| `选择进入wft05a` | 存在多个已获入口许可的 SysReq 节点，且当前没有活动输入节点 | 一个 SysReq 节点 ID | 对该 ID 回复 `选择` 或指定另一个 SysReq ID | `wft05a` |
| `选择进入wft05b` | 方案已被 `[同意]` 并推进到可进入 wft05b | 一个建议方案包 ID | 对该 ID 回复 `选择进入wft05b` | `wft05b` |
| `需要补充后重跑wft05a` | 当前方案为 `阶段性方案`，仍有阻断项 | 一个 PA 方案 ID | 对该 ID 回复 `修改：……` 并补充缺失信息 | `wft05a` |

#### 输出结果

本 Skill 结束时，只允许输出以下四类结果之一，每类结果必须带一个当前待办 ID：

| 输出结果 | 对应待办类型 | 当前 ID | 人类下一步 |
|---------|--------------|---------|------------|
| `等待PA方案反馈` | `反馈PA方案` | 一个 PA 组件方案 ID | 对该 ID 回复 `同意 / 修改 / 驳回` |

| `等待选择进入wft05a` | `选择进入wft05a` | 一个 SysReq 节点 ID | 对该 ID 回复 `选择` 或 `选择进入wft05a` |
| `等待选择进入wft05b` | `选择进入wft05b` | 一个建议方案包 ID | 对该 ID 回复 `选择进入wft05b` |

```text

### 固定审查包（补充信息）
- 方案状态：可确认方案 / 阶段性方案
- IPO 类型：业务定义类 / 引擎配置类
- 组件清单：[组件名称，标注复用/改进/新增]
- 边界摘要：[每个组件一句话职责边界]
- 关键映射：[IPO 条目 → PA 组件宿主]
- 阻断项清单：[列出阻断原因，无则写无]
- 预检结果：是否缺依附引擎配置类 IPO、阻断型 NFR、追溯链
```

要求：

1. `当前待办ID` 必须是单个 ID，不得写成多个 ID 列表。
2. `是否延续上次待办ID` 必须写 `是 / 否`；若为 `否`，必须说明当前 ID 已完成、被人类切换或不再可操作的原因。
3. `剩余待办ID数量` 必须是数字；若无法精确统计，应写 `至少 N 个` 并说明统计口径。
4. `人类只需回复` 必须是一句话，且只包含当前 ID 的动作。
5. `其他待办摘要` 只能给数量和类型，不得要求人类同时处理。

#### 唯一路由规则

Step End 必须输出**唯一下一步路由**。路由规则：

1. 若当前方案状态为 `阶段性方案` → `下一步路由 = 重新运行 wft05a`
2. 若当前方案状态为 `可确认方案` → `下一步路由 = 运行 wft05b`
3. 若人类在 `阶段性方案` 下给出 `[同意]`，AI 必须拒绝放行，说明阻断项仍未解除
4. 当输出结果为 `等待选择进入wft05b` 时，必须写入活动锁转交记录

## 四、PA 组件生成规则

### 4.1 类型判定

| IPO 类型 | 判定标准 | PA 末级组件类型 |
|---------|---------|----------------|
| **业务定义类 IPO** | 目标是产出具体业务定义；需求落在"定义内容怎么配" | **配置方案组件** |
| **引擎配置类 IPO** | 目标是提供/增强引擎配置承接能力；需求落在"引擎如何接收、解析、运行配置" | **前后端组件** |

### 4.2 业务定义类 IPO → 配置方案组件

处理步骤：识别依附引擎 → 识别业务定义对象 → 展开配置方案信息 → 组织组件。

与引擎配置类 IPO 的关系：业务定义类 IPO 所生成的配置方案信息构成引擎配置类 IPO 的核心输入。承接关系一致性由闭环检查事后验证，不构成设计时顺序约束。

建议"**1 个主定义组件 + 7 类子组件**"结构：

| 组件层 | 组件名称 | 职责 |
|------|---------|------|
| 主组件 | **主定义组件** | 该业务定义的核心承载单元 |
| 子组件1 | **元数据组件** | 编号、名称、版本、状态、适用范围、归属域 |
| 子组件2 | **结构组件** | 字段结构、流程节点结构、指标维度结构等 |
| 子组件3 | **规则组件** | 业务规则、流转规则、计算规则、校验规则 |
| 子组件4 | **权限组件** | 查看、配置、发起、审批、维护等权限 |
| 子组件5 | **展示组件** | 页面布局、列表视图、字段展示、菜单挂接 |
| 子组件6 | **引用集成组件** | 依赖的其他业务定义、引擎能力、外部数据源 |
| 子组件7 | **校验发布组件** | 完整性检查、冲突检查、发布/生效条件 |

### 4.3 引擎配置类 IPO → 前后端组件

处理步骤：识别引擎配置承接类型 → 识别核心输入（含配置方案信息）→ 判断前端/后端/两者 → 拆为组件。

建议"**1 个能力主组件 + 5 类承接组件**"结构：

| 组件层 | 组件名称 | 职责 |
|------|---------|------|
| 主组件 | **能力主组件** | 该引擎配置类 IPO 的核心承接单元 |
| 子组件1 | **前端配置组件** | 配置页面、设计器、参数编辑器 |
| 子组件2 | **前端运行组件** | 运行期输入/操作/结果/查看页面 |
| 子组件3 | **后端定义处理组件** | 配置信息解析、校验、组装、版本化 |
| 子组件4 | **后端运行执行组件** | 定义解释、调度、执行、状态流转 |
| 子组件5 | **后端接口与集成组件** | API、外部集成、消息、存储、事件 |

### 4.4 禁止跨类生成

1. 业务定义类 IPO 不直接落前后端组件
2. 引擎配置类 IPO 不直接落配置方案组件
3. 存在引擎缺口时回 `wft02` 处理

### 4.5 PA 非末级节点生成规则

**总原则**：先有末级组件集合，再生成非末级分类宿主。

1. 先按组件对象类型分大类（配置方案/前端/后端）
2. 再按职责稳定性分中类
3. 仅在边界稳定时才抽象非末级宿主
4. 非末级同样遵循复用 > 改进 > 新增

### 4.6 `wft05a` 的到位粒度边界

`wft05a` 只负责形成 **PA 主结构方案**，不负责把主结构展开成开发级详细规格。判断边界的总原则是：

- 影响"谁来承接、怎么切边界、是否复用"的内容，属于 `wft05a`
- 不影响上述结构判断、只是把内容写细写全的，属于 `wft05b`

#### 业务定义类 IPO 的到位粒度

业务定义类 IPO 在 `wft05a` 中应细到 **组件类别级 + 职责边界级**，以便人类判断该业务定义应由哪些配置方案组件承接。

必须达到：

1. 已判定是否需要独立的主定义组件
2. 已判定 7 类子组件中哪些需要、哪些可合并、哪些暂不需要
3. 已给出各组件的一句话职责边界
4. 已完成上游需求条目到主组件/子组件宿主的分配

不在本步展开：

1. 结构组件中的字段明细
2. 规则组件中的规则项全集
3. 权限组件中的角色操作项明细
4. 展示组件中的页面与交互细节
5. 校验发布组件中的逐条校验规则

#### 引擎配置类 IPO 的到位粒度

引擎配置类 IPO 在 `wft05a` 中应细到 **承接链级 + 组件边界级**，以便人类判断配置如何被接收、处理、执行以及这些职责应落在哪些前后端组件。

必须达到：

1. 已判定需要哪些承接组件（前端配置/前端运行/后端定义处理/后端运行执行/后端接口与集成）
2. 已判定这些组件是否都需要，哪些可合并，哪些暂不需要
3. 已给出各组件的一句话职责边界
4. 已说明配置输入如何进入引擎、结果大致从哪里输出
5. 已完成关键需求条目到承接组件的分配

不在本步展开：

1. API 或消息的字段级接口规格
2. 配置解析流程的详细步骤
3. 执行引擎内部算法或时序细节
4. 存储结构、表结构、DTO 全量定义
5. 重试、补偿、异常处理等开发级机制细节

#### 门禁 1 的审查焦点

人类在门禁 1 审查的不是"方案是否已经足够详细"，而是"当前 PA 主结构方案是否成立，是否允许进入 `wft05b` 做资产落账和详细展开"。

因此门禁 1 重点审查：

1. IPO 类型判定是否正确
2. 组件类型和组件清单是否正确
3. 组件边界是否合理
4. 复用 / 改进 / 新增 决策是否合理
5. SysReq → PA 承接关系是否闭合
6. 当前方案状态是阶段性方案还是可确认方案

## 五、AI 执行指南

1. **单节点处理**：每次只处理一个 IPO，不跨节点批量操作
2. **门禁 1 前全局串行**：若存在其他 IPO 处于 `待确认方案` 或 `迭代中`，不得接收新 IPO
3. **先装配 NFR 约束包，再做预检**：从 `wft04` 分层约束表中按 `全局 + 业务域 + IPO` 读取约束，不要求人类手工指定
4. **先做快速预检，再进入设计**：缺少依附引擎配置类 IPO、阻断型 NFR、追溯链时，应先报缺口；允许形成阶段性方案，但不得误报为稳定确认
5. **先判 IPO 类型，再设计**：进入 Step 3 前必须明确 IPO 类型
6. **基于资产优先**：复用已有配置宿主/引擎构件
7. **分解分配分析 = 创造性设计**：基于资产全景提出方案，人类审核反馈，多轮迭代收敛。不是从输入推导唯一正确答案
8. **配置方案信息是引擎配置类 IPO 的核心输入**：验证在闭环检查中事后执行
9. **Skill A 不做落账**：只写方案区（状态=待确认方案），不写反向引用/详细定义
10. **输出必须短而可审**：优先给固定审查包，不要求人类重复阅读完整推理过程

### NFR 约束包读取规则

目标 IPO 的 NFR 约束包按以下公式装配：

`约束包 = 全局约束 + 所属业务域约束 + 目标 IPO 专属约束`

读取规则：

1. `全局约束`：读取分层约束表中 `约束范围=全局` 且 `约束对象=EOS` 的行
2. `业务域约束`：读取分层约束表中 `约束范围=业务域` 且 `约束对象=目标 IPO 所属业务域/配置能力类` 的行
3. `IPO 专属约束`：读取分层约束表中 `约束范围=IPO` 且 `约束对象=目标 IPO ID` 的行
4. 若同一 NFR 在三层同时命中，按 `IPO > 业务域 > 全局` 解释具体说明，但三层约束都保留在约束包中
5. 若约束包中存在 `影响级别=阻断型` 且该约束未明确或未完成，则本轮只允许形成阶段性方案，不得作为稳定方案推进门禁 1

## 七、审查流程

### 审核要点

- [ ] IPO 类型判断正确
- [ ] 组件类型与 IPO 类型匹配（业务定义类→配置方案，引擎配置类→前后端）
- [ ] 每条需求条目分配到唯一 PA 组件
- [ ] 组件粒度合理、边界清晰
- [ ] NFR 约束映射完整
- [ ] 资产决策遵循"复用 > 改进 > 新增"

### 审查修复

人类在 `07-*.md` 反馈区按固定格式标注 `[修改-边界]` / `[修改-复用决策]` / `[修改-组件粒度]` / `[修改-NFR映射]` / `[修改-其他]` + 意见 → 状态变为迭代中 → 下次执行 Skill A 加载反馈并调整方案。

### 审查节奏建议

1. 人类优先审查 `组件边界` 与 `复用/改进/新增` 决策，不逐段重审设计推理
2. AI 每次只暴露最少必要信息，避免将串行等待时间浪费在人类重新理解上下文上
3. 同一节点重入时按"差异审查"进行，只审本轮修改点
