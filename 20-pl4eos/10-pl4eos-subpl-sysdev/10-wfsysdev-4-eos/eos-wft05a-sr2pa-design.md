# SysReq → PA 架构 · Skill A · 设计开发
**Skill ID**: wft05a · **Phase 1: SysReq → PA Design**

> **概述**：每次执行处理**一个** SysReq 功能架构末级节点（IPO），执行创造性/分析性设计活动，产出 PA 组件方案。按 IPO 类型走不对称设计路径：业务定义类 IPO 为模式匹配（机械性），业务引擎能力类 IPO 为分解分配分析（创造性）。
>
> **输入 → 输出**：一个 SysReq 末级 IPO（`05-*.md`）→ PA 组件方案（`07-*.md`，状态=待确认方案）
>
> **适用场景**：瀑布式系统设计第5步 · Phase 1。设计方案经人类审阅通过后，由 Skill B（`eos-wft05b-sr2pa-settle.md`）执行资产落账和详细定义展开。
>
> **设计模式参考**：[90-design-split-pattern.md](90-design-split-pattern.md)
>
> **版本**：v4.0 | **修订**：2026-06-17

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：`05-*.md` SysReq 状态表中人类选入的**一个** SysReq 功能架构末级节点（IPO）
- **产出**：PA 组件方案写入 `07-*.md`（状态=待确认方案），供人类审阅

本步按 IPO 类型生成两类 PA 末级组件：
- **业务定义类 IPO → 配置方案组件**
- **业务引擎能力类 IPO → 前后端组件**

### 与三条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **详概分离** | 本步产出 PA 层的**架构方案**（`07-*.md`）；详细定义（`06-*.md`）由 Skill B Phase 2 下半产出 |
| **文档结对** | 本步执行结对4（SysReq 详细定义末级 ↔ PA 架构）的设计思考——一次思考同时形成 06 和 07 的内容，正式落入由 Skill B 完成 |
| **基于资产** | 每次执行时读取已有 PA 资产，优先复用 > 改进 > 新增。单节点模式使后处理的节点总能看到前序已确认结果 |

### 单节点处理模式

每次执行只处理**一个** SysReq 架构末级节点（IPO）。这是全链路（wft01~wft05）的统一原则。

**成立的根本原因**——基于资产的设计原则使序列处理天然可行：

```
执行第1次：读已有资产 → 对 IPO-A 做设计 → 写回（状态=待确认方案）
执行第2次：读已有资产（含 A 结果）→ 对 IPO-B 做设计 → 写回
```

**无顺序约束**——人类选材时不需考虑 IPO 处理顺序：

1. **每个节点自描述**：四要素（I/P/O + 边界）已包含设计所需全部信息
2. **不依赖其他节点的设计结果**：只依赖自身四要素 + 已有资产状态
3. **承接关系是事后验证**：节点间 I/O 一致性由 Step 5 闭环检查保障

### 两类 IPO 的不对称设计逻辑

```
业务定义类 IPO → 配置方案组件
  信息来源：配对引擎 IPO 的 I（已详细描述所需配置信息）
  生成方式：信息重组（模式匹配）

业务引擎能力类 IPO → 前后端组件
  信息来源：EOS 前后端组件资产 + IPO 四要素 + 系统需求
  生成方式：分解分配分析（创造性设计）
```

## 二、前置条件与状态机

### 文档结对关系

| 结对 | 需求方 | 方案方 | 本步角色 |
|------|--------|--------|---------|
| **结对4** | SysReq 详细定义条目 | PA 架构定义 | 设计思考在此完成，正式落入由 Skill B 执行 |

### 节点状态定义

| 状态 | 含义 | 所处阶段 |
|------|------|---------|
| **待设计开发** | 上游就绪 + 人类同意 | Skill A 入口（首次） |
| **迭代中** | 人类反馈修改意见 | Skill A 入口（重入） |
| **待确认方案** | Skill A 已完成，等待人类审阅 | 人类审阅门禁 1 |
| **待资产落账** | 方案已确认 | Skill B 入口 |
| **已资产落账** | Skill B 上半完成 | Skill B 下半入口 |
| **待确认细节** | Skill B 下半完成，等待人类审阅 | 人类审阅门禁 2 |
| **确认中** | 人类反馈修改详细定义 | Skill B 下半重入 |
| **已验证冻结** | 全链验证通过 | 终态 |
| **已废弃** | 停止使用 | 终态 |

### Step Start 路由逻辑

```
读取目标节点状态：
  ├─ 待设计开发 → 执行 Skill A（首次）
  ├─ 迭代中 → 执行 Skill A（重入：加载反馈）
  ├─ 待资产落账 / 已资产落账 / 确认中 → 提示"请运行 Skill B"
  ├─ 待确认方案 / 待确认细节 → 提示"等待人类审阅"
  └─ 已验证冻结 / 已废弃 → 不可操作
```

| # | 操作 | 说明 |
|---|------|------|
| 1 | 加载目标 IPO 四要素 | I/P/O + 边界描述 |

### Step 3 · 设计活动

> 按目标 IPO 类型分流。这是结对4 的核心设计活动——一次思考同时产出 06 和 07 的**内容**（正式落入资产由 Skill B 完成）。

**3a · 业务定义类 IPO → 配置方案组件**

| # | 操作 | 说明 |
|---|------|------|
| 1 | 定位配对引擎 IPO | 找到与目标业务定义类 IPO 配对的业务引擎能力类 IPO |
| 2 | 提取配置信息需求 | 从配对引擎 IPO 的 Input 提取所需配置信息 |
| 3 | 组织组件结构 | 组织为主定义组件 + 7 类子组件结构 |
| 4 | 资产决策 | 对每个组件执行：复用已有 > 改进已有 > 新增 |

**3b · 业务引擎能力类 IPO → 前后端组件（分解分配分析）**

> **设计姿态**：基于已加载的资产全景，提出你认为最合理的分解分配方案。这是创造性设计，不存在唯一正确答案；方案将经人类审阅后迭代收敛。

| | 内容 |
|--|------|
| **看** | 已有 PA 资产全景、目标 IPO 四要素、引擎能力全景、EOS 组件参考 |
| **守** | 禁止跨类生成、复用 > 改进 > 新增、承接链闭合 |
| **交** | 条目→组件映射、资产决策（复用/改进/新增）、承接关系 |
| **完** | 每条目有唯一宿主、粒度合理、边界清晰 |

质量兜底：门禁 1（人类审阅方案）+ 迭代重入。

**3c · 非末级宿主生成（如需要）**

从当前末级组件集合反推稳定分类结构。总原则：先有末级组件，再生成非末级宿主。

### Step 4 · 方案写入

| # | 操作 | 说明 |
|---|------|------|
| 1 | 写入 PA 组件方案 | 写入 `07-*.md` 目标业务域分区（状态=待确认方案） |

| 3 | 记录 SysReq→PA 映射 | 记录本次 IPO 到 PA 组件的映射关系 |

### Step 5 · 闭环检查

| # | 操作 | 说明 |
|---|------|------|
| 1 | 覆盖率 | 目标 IPO 的全部功能已有 PA 组件承接 |
| 2 | 1:1 承接 | 每条需求条目有唯一 PA 组件宿主 |
| 3 | 粒度与边界 | 组件边界清晰、可独立维护与验收 |
| 4 | NFR 完整性 | 相关 NFR 约束已映射 |
| 5 | 承接链一致性 | 与已有组件 I/O 关系无冲突（事后验证——发现不一致标记为迭代建议，不阻断） |

## 四、PA 组件生成规则

### 4.1 类型判定

| IPO 类型 | 判定标准 | PA 末级组件类型 |
|---------|---------|----------------|
| **业务定义类 IPO** | 目标是产出具体业务定义；需求落在"定义内容怎么配" | **配置方案组件** |
| **业务引擎能力类 IPO** | 目标是提供/增强引擎承接能力；需求落在"系统能力怎么实现" | **前后端组件** |

### 4.2 业务定义类 IPO → 配置方案组件

处理步骤：识别依附引擎 → 识别业务定义对象 → 展开配置方案信息 → 组织组件。

与业务引擎能力类 IPO 的关系：业务定义类 IPO 所生成的配置方案信息构成业务引擎能力类 IPO 的核心输入。承接关系一致性由闭环检查事后验证，不构成设计时顺序约束。

建议"**1 个主定义组件 + 7 类子组件**"结构：

| 组件层 | 组件名称 | 职责 |
|------|---------|------|
| 主组件 | **主定义组件** | 该业务定义的核心承载单元 |
| 子组件1 | **元数据组件** | 编号、名称、版本、状态、适用范围、归属域 |
| 子组件2 | **结构组件** | 字段结构、流程节点结构、指标维度结构等 |
| 子组件3 | **规则组件** | 业务规则、流转规则、计算规则、校验规则 |
| 子组件4 | **权限组件** | 查看、配置、发起、审批、维护等权限 |
| 子组件5 | **展示组件** | 页面布局、列表视图、字段展示、菜单挂接 |
| 子组件6 | **引用集成组件** | 依赖的其他业务定义、引擎能力、外部数据源 |
| 子组件7 | **校验发布组件** | 完整性检查、冲突检查、发布/生效条件 |

### 4.3 业务引擎能力类 IPO → 前后端组件

处理步骤：识别引擎能力类型 → 识别核心输入（含配置方案信息）→ 判断前端/后端/两者 → 拆为组件。

建议"**1 个能力主组件 + 5 类承接组件**"结构：

| 组件层 | 组件名称 | 职责 |
|------|---------|------|
| 主组件 | **能力主组件** | 该业务引擎 IPO 的核心承接单元 |
| 子组件1 | **前端配置组件** | 配置页面、设计器、参数编辑器 |
| 子组件2 | **前端运行组件** | 运行期输入/操作/结果/查看页面 |
| 子组件3 | **后端定义处理组件** | 配置信息解析、校验、组装、版本化 |
| 子组件4 | **后端运行执行组件** | 定义解释、调度、执行、状态流转 |
| 子组件5 | **后端接口与集成组件** | API、外部集成、消息、存储、事件 |

### 4.4 禁止跨类生成

1. 业务定义类 IPO 不直接落前后端组件
2. 业务引擎能力类 IPO 不直接落配置方案组件
3. 存在引擎缺口时回 `wft02` 处理

### 4.5 PA 非末级节点生成规则

**总原则**：先有末级组件集合，再生成非末级分类宿主。

1. 先按组件对象类型分大类（配置方案/前端/后端）
2. 再按职责稳定性分中类
3. 仅在边界稳定时才抽象非末级宿主
4. 非末级同样遵循复用 > 改进 > 新增

## 五、AI 执行指南

1. **单节点处理**：每次只处理一个 IPO，不跨节点批量操作
2. **先判 IPO 类型，再设计**：进入 Step 3 前必须明确 IPO 类型
3. **基于资产优先**：复用已有配置宿主/引擎构件
4. **分解分配分析 = 创造性设计**：基于资产全景提出方案，人类审核反馈，多轮迭代收敛。不是从输入推导唯一正确答案
5. **配置方案信息是业务引擎核心输入**：验证在闭环检查中事后执行
6. **Skill A 不做落账**：只写方案区（状态=待确认方案），不写反向引用/详细定义
