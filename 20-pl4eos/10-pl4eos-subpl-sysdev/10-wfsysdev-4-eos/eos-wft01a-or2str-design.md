# OR → STR 架构 · Skill A · 设计开发
**Skill ID**: wft01a · **Phase 1: OR → STR Design**

> **概述**：每次执行只处理一个对象：一个待处理反馈的 STR 节点，或一个待分配 OR 末级条目（无待处理 STR 时）。STR 节点优先于新 OR 条目。Skill A 负责完成 OR→STR 的主结构方案形成，明确 STR-F 末级节点为”场景定义”，并在场景定义中识别十类信息链、场景指标、推理依据与补全说明；不负责 22/23/26 资产正式写回。AI 对新增/改进内容以 `[新增]` / `[改进]` 标记提示人类阅读，待人类整体确认同意后将 STR 节点置为「可以资产化」。
>
> **输入 → 输出**：一个待处理反馈的 STR 节点 / 一个待分配 OR 末级条目 → STR 场景定义方案局部写入 `02-*.md`（状态=`待确认方案` 或 `可以资产化`）+ OR 条目在 `01-*.md` 中同步更新分配、状态与追溯引用
>
> **适用场景**：瀑布式系统设计第1步 · Phase 1。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.21 | **修订**：2026-06-22

## 一、职责定位

Skill A 负责：

1. 识别本轮处理对象（STR 节点优先于新 OR 条目）
2. 加载上游追溯链与相关约束资产
3. 形成 STR-F / STR-NFR 主结构方案
4. 对 STR-F 场景定义识别信息载体清单、场景指标、推理依据与补全说明
5. 以 `[新增]` / `[改进]` 标记输出本轮增量内容清单，供人类逐项阅读
6. 判断当前方案状态
7. 提交人类审阅 STR 场景定义方案
8. 人类"整体确认同意"后，将 STR 节点生命周期状态置为「可以资产化」

Skill A 不负责：

1. 23 输出产品架构资产的正式引用写回
2. 22 非功能需求资产的正式引用写回
3. 26 广义文档资产的正式归属写回
4. 下游 `wft02` 的详细分解展开

## 二、输入与路由

### 本轮处理对象

每次执行只处理一个对象，按以下优先级：

1. **STR 优先路径**：`02-*.md` 状态队列中 `当前状态=待确认方案` 且反馈区已有 `[同意] / [修改] / [驳回]` 的 STR 节点；或 `当前状态=迭代中` 的 STR 节点（重入路径）
2. **新增路径**（仅当 STR 优先路径为空时）：`01-*.md` 状态队列中 `当前状态=待分配 + 人类决策=同意分配` 的一个 OR 末级条目

### 标准输入包

1. 本轮处理对象
2. 上游追溯链
3. 相关约束资产
4. 已有 STR 架构资产
5. 重入反馈
6. 树画像 / 资产全貌、状态队列、节点索引与本轮正文块定位信息

### Step Start 路由

第一层：STR 节点优先处理

| 条件 | 路由 |
|------|------|
| STR 节点 = `待确认方案` + 反馈区有 `[同意] / [修改] / [驳回]` | 方案反馈路径，执行 Step 3 |
| STR 节点 = `迭代中` | 重入路径 |
| 存在多个 STR 节点 = `待确认方案` + 有反馈 | 选择"正在处理优先"的 STR 节点（本轮刚处理过的优先）；均为第一次进入时选最早进入的状态的节点；同一次运行只处理一个 STR 节点 |
| STR 节点 = `待确认方案` + 反馈区为空 | 输出 `等待建议方案反馈`，当前待办 ID 为该 STR 节点 ID，不执行新增设计 |

第二层：STR 优先路径为空时，检查新 OR 条目

| 条件 | 路由 |
|------|------|
| 人类已通过对话选择了一个 OR 条目（如 "选择OR-002"），且该选择已写入 `01-*.md` 状态队列 | 新增路径，执行 Step 2 |
| 多条 OR 条目 = `待分配 + 同意分配`，且人类尚未选择 | 输出 `等待选择进入wft01a`，当前待办 ID 为 AI 推荐的一个 OR 条目 ID，退出 |
| 仅一条 OR 条目 = `待分配 + 同意分配` | 新增路径，执行 Step 2 |
| 零条 OR 条目 = `待分配 + 同意分配` | 输出 `无可处理对象`，退出 |

运行保护规则：

1. STR 节点优先于新 OR 条目：只要存在 `待确认方案` + 有反馈的 STR 节点，AI 必须先处理该 STR 节点，不得进入新 OR 条目。
2. 若存在待处理 STR 且同时存在新 OR 条目，AI 在处理 STR 反馈时输出提醒"还有新 OR 条目排队中，等待当前 STR 处理完后进入 wft01a"，继续处理 STR，不影响本次运行。
3. 若 STR 节点已处于 `待确认方案` 且反馈区为空，AI 不得重新生成或覆盖该节点方案，只输出 `等待建议方案反馈`。
4. 若多个 OR 条目同时满足 `待分配 + 同意分配`，AI 不得自动选择其中一个进入新增路径，必须先输出 `等待选择进入wft01a`。
5. 若需要重做 STR 场景定义，只能由 Step 3 的 `[修改] / [驳回]` 分支触发，并必须在反馈区或变更记录中保留旧方案处理摘要与人类反馈记录。
6. 若本轮处理对象为 STR 节点，Run End 时仍有其他 `待确认方案`（含反馈为空）的 STR 节点，AI 选一个该 STR 节点提示人类反馈，并结束本轮运行。

### Step 1 · OR 层已确认资产加载

读取并建立本轮设计上下文：

1. 加载 `01-*.md` 树画像 / 状态队列 / 节点索引，定位本轮 OR 条目正文块与 `#role-list`
2. 加载 `02-*.md` STR 树画像 / 状态队列 / 节点索引，定位本轮 STR 节点块；若为方案反馈或重入路径，读取该节点反馈区
3. 加载 `23-eos-output-architecture.md` 资产全貌层，按角色、业务域、产品类型和典型文档筛选候选业务域 / 产品类型 / 构件类型分片
4. 加载 `22-eos-nfr-taxonomy.md` 资产全貌层和与本轮 OR / STR 相关的 NFR 分片
5. 加载 `26-eos-document-definitions.md` 资产全貌层和与候选信息载体相关的文档定义分片
6. 按追溯关系读取必要的上游 / 下游相关块，不默认全量读取大文档正文

加载要求：

1. 仅已确认资产可直接依赖
2. 候选 / 疑似 / 待确认资产仅可作为参考
3. 若关键输入缺失，本轮最多只能形成 `阶段性方案`
4. 读取系统设计过程文档和资产文档时，必须遵守 [文档编写约定 §十四](../../../../00-generalspec/00-doc-conventions.md)：先读树画像 / 资产全貌、状态队列和节点索引，再读本轮正文块和必要相关块
5. 只有在索引缺失、关系断裂或执行全局一致性检查时，才允许说明原因后扩大读取范围

局部细节选择规则：

1. **对象线索**：根据本轮 OR 条目 ID 或 STR 节点 ID，在节点索引中定位必须读取的本轮正文块；这是本轮最小必读范围。
2. **结构线索**：根据本轮处理对象的父节点、角色、需求类型、业务域和状态，选择同角色、同业务域、同层级或同状态的候选 STR 块，用于判断复用、改进或新增。
3. **语义线索**：根据 OR 中的角色、动作、业务对象、流程阶段、约束条件、异常条件和指标口径，从资产全貌中筛选候选业务域、产品类型、构件类型、NFR 分类和文档定义分片。
4. **信息链线索**：按十链扫描法识别潜在信息载体和场景指标，再用候选载体名称、产生方式、状态事件、审批留痕、报表看板等关键词定位 `26` 文档定义分片和相关资产块。
5. **追溯线索**：若本轮处理对象已有上游、下游或相邻承接关系，只读取追溯关系指向的必要块；不得复制对方正文到本节点。
6. **反馈线索**：若本轮为方案反馈路径或重入路径，只读取该 STR 节点反馈区及反馈指向的相关块；不得重新扩大为首次新增设计的全量候选读取。
7. **扩大读取触发**：只有当索引缺失、候选冲突、资产全貌不足以判断边界、追溯关系断裂或需要做全局一致性检查时，才可扩大读取范围，并在输出中说明扩大读取的原因。

### Step 2 · 新增设计处理

#### A. FR 路径

若本轮处理对象为 FR 类型 OR 条目，执行：

1. 沿用 `#role-list` 中已确认角色归属；无记录时再补做角色分配
2. **确定业务域**：从角色类型映射 23 资产业务域（如采购员→采购管理），显式写入 STR-F 节点
3. 按触发定义端到端业务闭环（该场景定义的职责范围）
4. 判断视角类型：`场景`（STR 末级节点固定为场景定义）
5. **信息载体与场景指标识别（十链扫描法）**：

   wft01a 从 STR 场景定义承接的 OR 语义中识别该场景需要哪些广义文档 / 信息载体，并识别用于评价场景运行成效的场景指标。信息链是 AI 的分析框架，不是场景定义的内容结构；场景定义正文始终输出扁平的信息载体清单和场景指标清单。

   本步采用“十链扫描法”：

   | 链 | 类型 | 分析引导 |
   |----|------|---------|
   | 执行业务链 | 信息载体 | 角色要创建、处理、提交、确认哪些业务单据、表单、记录或台账 |
   | 管理业务链 | 信息载体 | 管理者需要查看哪些进度、异常、统计、报表或看板 |
   | 个人绩效链 | 信息载体 | 场景是否沉淀个人工作量、效率、质量、达成率等评价依据 |
   | 组织绩效链 | 信息载体 | 场景是否沉淀部门、团队、项目、供应商等组织对象绩效依据 |
   | 资源需求链 | 信息载体 | 场景是否涉及资源申请、占用、消耗、查看、调度或约束 |
   | 协同交互链 | 信息载体 | 谁与谁之间需要通知、确认、反馈、回执、协同任务或接口交互 |
   | 风险合规链 | 信息载体 | 哪些动作需要审批、授权、留痕、审计、证据或异常处置 |
   | 规则知识链 | 信息载体 | 场景依赖哪些规则、标准、模板、SOP、知识库或策略配置 |
   | 状态事件链 | 信息载体 | 关键业务对象有哪些状态变化、事件触发、过程轨迹或日志 |
   | 场景指标链 | 指标 | 如何评价该场景运行得好不好，包含效率、质量、成本、风险、协同、体验等口径 |

   每个信息载体必须标注：

   - 载体名称
   - 归属链，可标注主链和必要的辅助链
   - 文档性质：业务显性文档 / 系统衍生文档 / 规则文档 / 知识文档 / 视图报表 / 其他
   - 产生方式：人类填写 / 系统生成 / 人机协同 / 外部输入
   - 一句话职责
   - 来源 OR
   - 来源类型：`OR明示` / `OR隐含` / `同类场景推断` / `资产线索推断`
   - 确认状态：`可确认` / `待确认` / `阶段性`
   - 26 资产方案级判断：`26-Q1 已有` / `26-Q2 可扩展` / `26-Q3 待新增` / `未检查`
   - 建议归属批次：按十链处理顺序标注，供 `wft01b` 逐链归属

   每个场景指标必须标注：

   - 指标名称
   - 指标类别：效率 / 质量 / 成本 / 风险 / 协同 / 体验 / 其他
   - 指标含义
   - 主要数据来源
   - 来源 OR
   - 来源类型
   - 确认状态
   - 建议归属批次：`场景指标链`

   十链建议处理顺序：

   | 顺序 | 信息链 |
   |------|--------|
   | 1 | 执行业务链 |
   | 2 | 状态事件链 |
   | 3 | 协同交互链 |
   | 4 | 风险合规链 |
   | 5 | 规则知识链 |
   | 6 | 资源需求链 |
   | 7 | 管理业务链 |
   | 8 | 个人绩效链 |
   | 9 | 组织绩效链 |
   | 10 | 场景指标链 |

   STR 场景定义中应保留 `### 信息链归属处理状态`，初始状态如下，供 `wft01b` 按链逐批处理：

   ```text
   | 顺序 | 信息链 | 处理状态 | 说明 |
   |------|--------|----------|------|
   | 1 | 执行业务链 | 待归属 | — |
   | 2 | 状态事件链 | 待归属 | — |
   | 3 | 协同交互链 | 待归属 | — |
   | 4 | 风险合规链 | 待归属 | — |
   | 5 | 规则知识链 | 待归属 | — |
   | 6 | 资源需求链 | 待归属 | — |
   | 7 | 管理业务链 | 待归属 | — |
   | 8 | 个人绩效链 | 待归属 | — |
   | 9 | 组织绩效链 | 待归属 | — |
   | 10 | 场景指标链 | 待归属 | — |
   ```

   若 OR 中没有直接触发词，AI 可以依据业务对象、角色职责、流程阶段、约束条件、异常条件、指标口径、上游资产或同类场景模式做弱推断，但必须标注为 `OR隐含`、`同类场景推断` 或 `资产线索推断`，并将确认状态置为 `待确认`。若缺少足够语义线索，AI 不得补全具体载体，只能输出缺口和澄清建议，方案状态置为 `阶段性方案`。

6. **推理依据与补全说明**：

   对所有超出 OR 直接明示的信息载体和场景指标，必须显性化 AI 推理依据。推理依据至少分为三类：

   | 推理类型 | 使用场景 | 输出要求 |
   |---------|----------|---------|
   | 业务常识推理 | 从业务动作、对象、角色职责推断常见业务单据、表单、记录 | 说明 OR 线索、业务规则、补全结果 |
   | 系统设计推理 | 从进度跟踪、状态变化、指标计算、自动触发推断系统衍生文档 | 说明实现该能力所需的状态、事件、日志或数据来源 |
   | 企业治理推理 | 从预算、供应商、质量、安全、审批、异常推断合规、规则、审计与证据载体 | 说明治理风险、控制点、补全结果 |

   推理依据采用编号引用，如 `R1`、`R2`。信息载体和场景指标条目应引用对应 `R#`，使人类可追溯每项补全的依据。

7. 派生 STR-F 节点名称（即场景定义名称）
8. 执行 Q1/Q2/Q3 找宿主与资产决策：复用 / 改进 / 新增

   - **Q1（直接匹配）**：OR 的功能描述与已有场景定义在语义上一致，则复用该宿主。判定依据：OR 中的"动词 + 业务对象"能直接对应场景定义的职责范围。宿主候选池包含所有 STR 节点，不论其当前生命周期状态（`待确认方案`、`可以资产化`、`待分解分配` 等均可匹配）
   - **Q2（扩展匹配）**：OR 的功能描述是已有场景定义的子集或超集边缘，可扩展后承接。边界判断：扩展不改变场景定义的核心职责边界——若扩展后变成大杂烩，应改为新增
   - **Q3（新增）**：OR 无任何可用宿主，或扩展后导致边界模糊。要求：新增场景定义的命名和粒度应与同层已有节点保持一致

	   若命中的宿主 STR 节点当前状态不是 `待确认方案`（如为 `可以资产化` 或 `待分解分配`），AI 应在写入该节点新内容后，将其状态回退为 `待确认方案`，等待人类对新内容的反馈。

9. 完善 STR-F 要素：

   - 定义
   - 业务域（从角色映射）
   - 视角类型：`场景`（固定值，STR 末级节点均为场景定义）
   - 功能说明
   - 信息载体清单（十链扫描结果，含 26 资产方案级判断）
   - 场景指标
   - 信息链归属处理状态
   - 推理依据与补全说明
   - 十链覆盖自查
   - 承接的 OR 末级
   - 符合性分析
   - 设计指导

#### B. NFR 路径

若本轮处理对象为 NFR 类型 OR 条目，执行：

1. 沿用 `#role-list` 中已确认角色归属
2. 使用 22 资产逐级匹配：具体需求 → 分类维度 → 候选
3. 无匹配时执行候选新增评审
4. 完善 STR-NFR 节点要素：
   - 定义
   - 需求类型
   - 承接的 OR 末级
   - 符合性分析
   - 量化指标
   - 测量方法

#### C. 写入规则

Skill A 写文档时必须遵守 [文档编写约定 §十四](../../../../00-generalspec/00-doc-conventions.md) 的树形资产文档布局。写入范围按本轮相关块局部更新，不得重写大文档全文。

1. 写入 `02-*.md` 正文节点块：
   - 复用 / 改进时，定位既有 STR 节点块，只更新该块内的节点定义、方案状态、信息载体清单、场景指标、推理依据、承接 OR 清单、反馈区和必要的变更摘要
   - 新增时，在合适父节点下新增稳定 `BLOCK`，写入完整 STR-F / STR-NFR 要素、方案状态，并生成空 `### 反馈区`
   - 方案状态必须以可观察字段写入 STR 节点正文块，取值只能为 `阶段性方案` 或 `可确认方案`；若为 `阶段性方案`，必须同步写入阻断项或关键缺口说明
2. 更新 `02-*.md` 树画像 / 节点索引 / 状态队列：
   - 若新增节点，或既有节点的一句话边界、父节点、层级、名称、状态发生变化，同步更新树画像行
   - 复用 / 改进时，更新既有 STR 节点索引行的状态、来源 OR、块 ID 和一句话边界
   - 新增时，新增 STR 节点索引行，记录节点 ID、父节点 ID、节点类型、层级、状态、块 ID 和一句话边界
   - 将本轮 STR 节点状态写为 `待确认方案`
3. 更新 `01-*.md` 树画像 / OR 正文块 / 状态队列：
   - 若 OR 条目状态或一句话边界在树画像中有摘要记录，同步更新树画像行
   - 在本轮 OR 正文块的当前分配记录中写入承接 STR 节点编号、承接方式和方案状态
   - 在 `01-*.md` 状态队列中将该 OR 条目状态写为 `待确认方案`
4. 更新追溯关系：
   - 在 `01` 与 `02` 的追溯区或本轮相关块承接关系中记录 OR→STR 引用
   - 只写引用关系，不复制对方正文
5. 写入反馈区：
   - 新方案必须保留 `### 反馈区`
   - 若已有未处理反馈，不得覆盖或清空
   - 若本轮消费了人类反馈，必须先归档反馈处理摘要，再生成新的空反馈区

写入分支：

1. **复用已有 STR 节点**：不新增节点；只在既有节点块内追加本 OR 到承接 OR 清单，并重新检查功能说明、信息载体清单、场景指标和推理依据是否需要补充；同步更新树画像、索引行和状态队列。
2. **改进已有 STR 节点**：在原节点块内更新节点定义与要素；保留旧版摘要和变更说明；更新该节点承接 OR 清单、树画像、索引行和状态队列。
3. **新增 STR 节点**：在合适角色分类下新增稳定节点块；写入完整 STR-F / STR-NFR 要素、反馈区、树画像行、节点索引行和状态队列记录。
4. 任一分支都不得覆盖人类尚未处理的反馈区内容。

状态回退规则：若命中的宿主 STR 节点当前状态为 `可以资产化` 或 `待分解分配`（即非 `待确认方案` 状态），AI 在写入新内容后，必须将 `02-*.md` 中该节点状态回退为 `待确认方案`，并在 `01-*.md` 状态队列中对应 OR 条目的状态同步为 `待确认方案`。回退后该节点需重新经历人类确认流程。如果 wft01b 已开始处理该节点（信息链中有非 `待归属` 的状态），则不应回退，应输出冲突提示由人类判断。

信息链状态同步：回退节点状态时，AI 必须同步将本次新增/改进信息载体所归属的信息链在 `### 信息链归属处理状态` 中重置为 `待归属`；其他未涉及的信息链保持原有状态（如 `已写回`）不变。这样 wft01b 重新处理时只处理有改动的链。

### Step 3 · 方案反馈处理

若本轮处理对象是一个 STR 节点，且满足以下任一条件，则按该节点反馈区执行：

1. 节点状态为 `待确认方案`，且反馈区已有 `[同意] / [修改] / [驳回]` 人类审阅反馈
2. 节点状态为 `迭代中`

反馈分支：

1. **整体确认同意**（人类整体确认本轮所有新增/改进内容）

   适用：人类在对话中回复"整体确认同意"或"全部同意"。

   - **先检查方案状态**
   - 若方案状态为 `阶段性方案`：不得推进到 `可以资产化`，应输出当前方案仍有阻断项/关键缺口，提示人类补充缺失信息或改写为 `[修改]`
   - 若方案状态为 `可确认方案`：归档反馈处理摘要并生成新的空反馈区，在 `02-*.md` 树画像、状态队列和节点索引中将 STR 节点生命周期状态推进为 `可以资产化`
   - 在 `01-*.md` 树画像、本轮 OR 块和状态队列中同步方案已确认状态
   - 该 STR 节点不再由 `wft01a` 处理，后续由 `wft01b` 接手资产化

2. `[同意]`（逐项确认）

   - 先检查方案状态
   - 若方案状态为 `可确认方案`：确认当前项已通过，继续等待其他项或全部确认完成；全部确认完成后执行上述"整体确认同意"的推进动作
   - 若方案状态为 `阶段性方案`：不得推进到 `可以资产化`，应输出当前方案仍有阻断项，并提示人类补充缺失信息或改写为 `[修改]`
3. `[修改]`
   - 按反馈类型调整节点定义与要素。反馈类型包括结构反馈、信息链反馈、指标反馈、推理依据反馈、其他说明
   - 必要时重做 OR→STR 承接关系
   - 重新生成推理依据与补全说明
   - 重新执行十链覆盖自查
   - 在当前 STR 节点块内归档本次人类反馈和修改前方案摘要，并生成新空反馈区
   - 更新 `02-*.md` 树画像 / 节点索引 / 状态队列，节点状态回写为 `待确认方案`
   - 若承接关系变化，同步更新 `01-*.md` 树画像、本轮 OR 块、状态队列和追溯引用
4. `[驳回]`
   - 重新执行该节点对应的 OR→STR 设计
   - 在当前 STR 节点块内归档原方案和本次人类反馈，写入新方案或按新宿主更新承接关系
   - 更新 `02-*.md` 树画像 / 节点索引 / 状态队列，节点状态回写为 `待确认方案`
   - 若改为其他 STR 宿主或新增节点，同步更新 `01-*.md` 树画像、本轮 OR 块、状态队列和追溯引用

### 增量标记输出

Step 2 新增方案时，以及 Step 3 修订方案后，AI 必须在本轮输出首部附加以 `===` 包裹的增量清单，明确标识本次新增/改进的内容。增量清单供人类阅读和反馈，不写入资产文档。

增量清单格式：

```text
=== STR-F-XXXX 本轮更新清单 ===

[新增] 承接 OR-XXXX "OR 标题摘要"
[新增] 信息载体 IC-XXX "载体名称"
[改进] 场景指标 KP-XXX 口径调整：旧→新
[改进] 功能说明补充：……
[新增] 推理依据 R3：……

=== 以上为本次全部新内容 ===

请逐项确认，或回复"整体确认同意"。
```

三类标记：

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| `[新增]` | 本次新增的内容 | 新 OR 进入、新增信息载体、新增指标、新增推理依据 |
| `[改进]` | 对既有内容的修改 | 已有节点的功能说明补充、口径调整、边界澄清 |
| `[删除]` | 移除既有内容 | 少见，一般需在反馈区说明原因 |

要求：

1. 增量清单作为 AI 本轮输出的一部分呈现给人类，不写入 `02-*.md` 或 `01-*.md`
2. 增量清单必须标注每项的来源（OR 条目 ID、资产线索等）
3. 人类确认全部内容后，STR 节点状态推进为 `可以资产化`

## 四、方案审阅

### 审查口径

人类审阅 STR 场景定义方案时，重点审查本轮增量清单中的内容：

1. STR 主结构方案是否成立
2. OR→STR 承接是否正确
3. 业务域归属是否合理
4. 信息载体识别是否覆盖全面，尤其是十链中是否有明显遗漏或过度推断
5. 场景指标是否属于本场景，口径是否合理
6. 推理依据是否透明，`OR隐含` / `同类场景推断` / `资产线索推断` 是否可接受
7. 本次新增/改进内容是否需要调整

**人类审查焦点**：
1. `[新增]` 和 `[改进]` 标记的内容是否符合预期
2. OR→STR 的语义匹配是否正确——是否命中正确的宿主
3. 信息载体识别是否有明显缺口或过度补全
4. 场景指标是否需要保留、删除、改名或调整数据来源
5. AI 推理依据是否可信，待确认项是否接受
6. 新增 STR-F 节点的业务域、命名和粒度是否与现有资产风格一致

方案审阅不审：

1. 23 / 22 / 26 资产的正式写回结果
2. 下游 `wft02` 的详细展开

### 方案状态

| 方案状态 | 含义 | 是否可放行 |
|------|------|------|
| `阶段性方案` | 仍存在阻断项 / 关键缺口 | 否 |
| `可确认方案` | 主结构已稳定 | 是 |

### STR 节点生命周期状态

| 状态 | 含义 | 谁处理 |
|------|------|--------|
| `待确认方案` | 方案已生成或修订，等待人类反馈 | wft01a |
| `可以资产化` | 人类已整体确认同意，等待 wft01b 进行文档归属 | wft01b |

## 五、运行结束输出（Run End）

Run End 的作用是把本轮运行结果翻译成人类下一步动作的最小提示。AI 不只输出”方案已生成”或”处理完成”，还必须根据当前运行结果，为人类提供一个当前可动作的提示。

### 推荐交互方式

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应 STR 节点的 `### 反馈区`、`01/02` 状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

### Run End 三类尾部输出

处理完本轮 STR 反馈或新 OR 设计后，按以下优先级输出：

| 输出类型 | 条件 | AI 动作 |
|---------|------|---------|
| **继续处理 STR** | 存在其他 `待确认方案` 的 STR 节点（含反馈为空或已有反馈） | 选一个该 STR 节点，提示人类对其反馈，当前待办 ID 为该 STR 节点 ID |
| **选择下一步** | 无待处理的 STR 节点 | 提示人类为 wft01a 选择新 OR 条目，或为 wft01b 选择「可以资产化」的 STR 节点 |
| **无可处理对象** | 无待处理 STR 节点，且无可分配的 OR 条目，且无可资产化的 STR 节点 | 输出当前无可处理对象 |

### 输出一：继续处理 STR

适用：当前运行已处理完一个 STR 节点或新 OR，状态队列中仍存在其他 `待确认方案` 的 STR 节点。

当前待办 ID 选择规则：

1. 本轮刚处理过的 STR 节点优先：如果该节点方案状态仍为 `待确认方案`（人类尚未全部确认），继续提示该节点 ID。
2. 若本轮刚处理过的 STR 节点已完成（已推进到 `可以资产化`），选择最早进入 `待确认方案` 的其他 STR 节点。
3. 若存在多个 `待确认方案` 的 STR 节点，且本轮未处理任何一个，选择最早进入该状态的一个。

人类只需做什么：

1. 查看提示的 STR 节点 ID 及其增量清单（如有新增内容）。
2. 对该 STR 节点回复 `整体确认同意`、`同意`、`修改：……` 或 `驳回：……`。
3. AI 收到回复后，先转写为该 STR 节点 `### 反馈区` 的规范记录，再在下一次 `wft01a` 中处理。

输出格式：

```text
当前状态：继续处理 STR
当前待办ID：STR-F-XXX
剩余待办STR节点数量：N
人类只需回复：「整体确认同意」/「同意」/「修改：……」/「驳回：……」

其他待办摘要：
- 待反馈 STR 节点：N 个
- 待分配 OR 条目：M 个
- 可资产化 STR 节点：K 个（可运行 wft01b 处理）
```

### 输出二：选择下一步

适用：无待处理 STR 节点，可进入下一步选择。

当前提示：

1. 若存在 `待分配 + 同意分配` 的 OR 条目，提示人类”选择一条 OR 进入 wft01a”。
2. 若存在「可以资产化」的 STR 节点，提示人类”选择一个 STR 进入 wft01b 做资产化”。
3. 若两者都存在，AI 推荐优先处理哪一个，但最终由人类决定。

人类只需做什么：

1. 根据提示回复 `选择：<OR 条目 ID>` 或 `选择：<STR 节点 ID>`。
2. 或回复 `运行 wft01a` / `运行 wft01b` 进入对应 Skill。

输出格式：

```text
当前状态：选择下一步
可选择的动作：
  1. 为 wft01a 选择新 OR → 候选：OR-XXX、OR-YYY
  2. 为 wft01b 选择资产化 STR → 候选：STR-F-ZZZ（可以资产化）
请回复「选择：<ID>」或「运行 wft01a / wft01b」
```

### 输出三：无可处理对象

适用：无待处理 STR 节点，无可分配 OR 条目，无可资产化的 STR 节点。

输出：

```text
当前状态：无可处理对象
wft01a 当前无可处理的 STR 节点或 OR 条目。
```
