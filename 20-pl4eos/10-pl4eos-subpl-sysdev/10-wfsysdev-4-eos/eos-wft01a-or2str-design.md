# OR → STR 架构 · Skill A · 设计开发
**Skill ID**: wft01a · **Phase 1: OR → STR Design**

> **概述**：每次执行只处理一个对象：一个待分配 OR 末级条目，或一个待处理反馈的 STR 节点。Skill A 负责完成 OR→STR 的主结构方案形成，明确 STR-F 末级节点为“场景定义”，并在场景定义中识别十类信息链、场景指标、推理依据与补全说明；不负责 22/23/26 资产正式写回。
>
> **输入 → 输出**：一个待分配 OR 末级条目 / 一个待处理反馈的 STR 节点 → STR 场景定义方案局部写入 `02-*.md`（状态=`待确认方案` 或 `待文档归属`）+ OR 条目在 `01-*.md` 中同步更新分配、状态与追溯引用
>
> **适用场景**：瀑布式系统设计第1步 · Phase 1。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.20 | **修订**：2026-06-20

---

## 一、职责定位

Skill A 负责：

1. 识别本轮处理对象
2. 加载上游追溯链与相关约束资产
3. 形成 STR-F / STR-NFR 主结构方案
4. 对 STR-F 场景定义识别信息载体清单、场景指标、推理依据与补全说明
5. 判断当前方案状态
6. 提交人类审阅 STR 场景定义方案

Skill A 不负责：

1. 23 输出产品架构资产的正式引用写回
2. 22 非功能需求资产的正式引用写回
3. 26 广义文档资产的正式归属写回
4. 下游 `wft02` 的详细分解展开

## 二、输入与路由

### 本轮处理对象

每次执行只处理一个对象：

1. **新增路径**：`01-*.md` 状态队列中 `当前状态=待分配 + 人类决策=同意分配` 的一个 OR 末级条目
2. **方案反馈路径**：`02-*.md` 状态队列中 `当前状态=待确认方案` 且反馈区已有 `[同意] / [修改] / [驳回]` 的一个 STR 节点
3. **重入路径**：`02-*.md` 状态队列中 `当前状态=迭代中` 的一个 STR 节点

### 活动 OR 条目锁定

`wft01a` 同一时间只能允许一个 OR 末级条目处于活动处理状态。一个 OR 条目进入 `wft01a` 后，在其对应 STR 场景定义方案未完成本 Skill 处理前，不得再让新的 OR 条目进入 `wft01a`。

活动 OR 条目判定：

1. `02-*.md` 中存在引用某个 OR 条目的 STR 节点，且状态为 `待确认方案`、`迭代中`、`确认中` 或其他未完成状态。
2. `02-*.md` 中存在该 OR 条目的待办锁定记录，且当前待办 ID 尚未完成。
3. `01-*.md` 状态队列中该 OR 条目已被标记为进入 `wft01a`，但尚未记录 `wft01a` 处理完成。

活动 OR 条目完成判定：

1. 对应 STR 节点已达到 `待文档归属`，且人类已明确选择该 STR 节点进入 `wft01b`。
2. 该 OR 条目被人类明确撤销、废弃或退回上游，不再由当前 `wft01a` 继续处理。

说明：STR 节点达到 `待文档归属` 只表示场景定义方案已确认，不等于活动 OR 条目已释放。只要人类尚未完成 `选择进入wft01b`，该 OR 条目仍视为活动 OR 条目，AI 不得进入新的 OR 条目。

锁定规则：

1. Step Start 必须先检查是否存在活动 OR 条目，再检查新的 `待分配` OR 条目。
2. 若存在活动 OR 条目，AI 只能重入该 OR 条目对应的 STR 节点，或输出该节点的当前待办；不得从 `待分配` 队列中选择新 OR 条目。
3. 若存在多个活动 OR 条目，说明状态队列不一致，AI 必须停止新增处理，输出状态冲突说明，并要求人类先选择保留一个活动 OR 条目。
4. 只有不存在活动 OR 条目时，AI 才能从 `待分配 + 同意分配` 队列中选择一个新 OR 条目进入新增路径。
5. 新 OR 条目一旦进入新增路径，AI 必须在 `01-*.md` 状态队列、`02-*.md` STR 节点块或本轮运行摘要中记录活动锁定关系。
6. 活动锁定关系至少包含：`活动OR条目ID`、`关联STR节点ID`、`当前待办ID`、`锁定状态`、`锁定原因`、`锁定开始时间或轮次`、`释放条件`。
7. `锁定状态` 建议取值为：`锁定中`、`待进入wft01b`、`已释放`、`已撤销`、`冲突待处理`。

活动锁定记录建议格式：

```text
活动OR条目ID：
关联STR节点ID：
当前待办ID：
锁定状态：锁定中 / 待进入wft01b / 已释放 / 已撤销 / 冲突待处理
锁定原因：
锁定开始：
释放条件：
```

### 标准输入包

1. 本轮处理对象
2. 上游追溯链
3. 相关约束资产
4. 已有 STR 架构资产
5. 重入反馈
6. 树画像 / 资产全貌、状态队列、节点索引与本轮正文块定位信息

### Step Start 路由

| 条件 | 路由 |
|------|------|
| STR 节点 = `待文档归属` 且人类已回复 `选择进入wft01b`，但活动锁尚未释放或转交 | 只执行活动锁释放 / 转交记录，不进入新增设计或反馈处理 |
| 存在一个活动 OR 条目 | 只处理该活动 OR 条目对应的 STR 反馈、重入或当前待办，不得进入新 OR 条目 |
| 存在多个活动 OR 条目 | 停止执行，输出状态冲突说明，要求人类选择保留一个活动 OR 条目 |
| OR 条目 = `待分配 + 同意分配` 且人类已回复 `选择`、`选择进入wft01a` 或 `选择：<OR条目ID>`，且不存在活动 OR 条目 | 新增路径，并写入活动 OR 条目锁定关系 |
| 存在多个 OR 条目 = `待分配 + 同意分配`，且不存在活动 OR 条目 | 输出 `等待选择进入wft01a`，当前待办 ID 为 AI 推荐的一个 OR 条目 ID |
| 仅存在一个 OR 条目 = `待分配 + 同意分配`，且不存在活动 OR 条目 | 新增路径 |
| STR 节点 = `待确认方案` + 反馈区有 `[同意] / [修改] / [驳回]` | 方案反馈路径 |
| STR 节点 = `迭代中` | 重入路径 |
| STR 节点 = `待确认方案` | 输出 `等待建议方案反馈`，当前待办 ID 为该 STR 节点 ID，不执行新增设计 |
| STR 节点 = `待文档归属` 且活动锁定状态不是 `已释放` | 输出 `等待选择进入wft01b`，当前待办 ID 为该 STR 节点 ID |
| STR 节点 = `待文档归属` 且活动锁定状态为 `已释放` | 提示运行 `wft01b`，不得继续修改该节点 |
| 其他状态 | 输出不可操作说明后结束 |

运行保护规则：

1. 若 STR 节点已处于 `待确认方案` 且反馈区为空，AI 不得重新生成或覆盖该节点方案，只输出 `等待建议方案反馈`，当前待办 ID 为该 STR 节点 ID。
2. 若 STR 节点已处于 `待确认方案` 且反馈区有 `[同意] / [修改] / [驳回]`，AI 只执行 Step 3，不得重新进入新增设计处理。
3. 若 STR 节点已处于 `待文档归属`，说明方案已确认，AI 不得继续修改该节点方案；若活动锁未释放，应先输出 `等待选择进入wft01b`，若活动锁已释放，才提示运行 `wft01b`。
4. 若需要重做 STR 场景定义，只能由 Step 3 的 `[修改] / [驳回]` 分支触发，并必须在反馈区或变更记录中保留旧方案处理摘要与人类反馈记录。
5. 只要存在活动 OR 条目未完成，AI 不得响应其他 OR 条目的 `同意分配`，即使这些条目也满足新增路径条件。
6. 若多个 OR 条目同时满足 `待分配 + 同意分配`，AI 不得自动选择其中一个进入新增路径，必须先输出 `等待选择进入wft01a`。
7. 若人类已对某个待分配 OR 条目回复 `选择`、`选择进入wft01a` 或 `选择：<OR条目ID>`，下一次 `wft01a` 应先将选择记录转写到 `01-*.md` 状态队列或本轮运行摘要，再让该 OR 条目进入新增路径；不得再次输出同一个选择提示。
8. 若人类已对待文档归属 STR 节点回复 `选择进入wft01b`，下一次 `wft01a` 只允许记录活动锁释放或转交，不得重新设计、修改或覆盖该 STR 节点。

### Step 1 · OR 层已确认资产加载

读取并建立本轮设计上下文：

1. 加载 `01-*.md` 树画像 / 状态队列 / 节点索引，定位本轮 OR 条目正文块与 `#role-list`
2. 加载 `02-*.md` STR 树画像 / 状态队列 / 节点索引，定位本轮 STR 节点块；若为方案反馈或重入路径，读取该节点反馈区
3. 加载 `23-eos-output-architecture.md` 资产全貌层，按角色、业务域、产品类型和典型文档筛选候选业务域 / 产品类型 / 构件类型分片
4. 加载 `22-eos-nfr-taxonomy.md` 资产全貌层和与本轮 OR / STR 相关的 NFR 分片
5. 加载 `26-eos-document-definitions.md` 资产全貌层和与候选信息载体相关的文档定义分片
6. 按追溯关系读取必要的上游 / 下游相关块，不默认全量读取大文档正文

加载要求：

1. 仅已确认资产可直接依赖
2. 候选 / 疑似 / 待确认资产仅可作为参考
3. 若关键输入缺失，本轮最多只能形成 `阶段性方案`
4. 读取系统设计过程文档和资产文档时，必须遵守 [文档编写约定 §十四](../../../../00-generalspec/00-doc-conventions.md)：先读树画像 / 资产全貌、状态队列和节点索引，再读本轮正文块和必要相关块
5. 只有在索引缺失、关系断裂或执行全局一致性检查时，才允许说明原因后扩大读取范围

局部细节选择规则：

1. **对象线索**：根据本轮 OR 条目 ID 或 STR 节点 ID，在节点索引中定位必须读取的本轮正文块；这是本轮最小必读范围。
2. **结构线索**：根据本轮处理对象的父节点、角色、需求类型、业务域和状态，选择同角色、同业务域、同层级或同状态的候选 STR 块，用于判断复用、改进或新增。
3. **语义线索**：根据 OR 中的角色、动作、业务对象、流程阶段、约束条件、异常条件和指标口径，从资产全貌中筛选候选业务域、产品类型、构件类型、NFR 分类和文档定义分片。
4. **信息链线索**：按十链扫描法识别潜在信息载体和场景指标，再用候选载体名称、产生方式、状态事件、审批留痕、报表看板等关键词定位 `26` 文档定义分片和相关资产块。
5. **追溯线索**：若本轮处理对象已有上游、下游或相邻承接关系，只读取追溯关系指向的必要块；不得复制对方正文到本节点。
6. **反馈线索**：若本轮为方案反馈路径或重入路径，只读取该 STR 节点反馈区及反馈指向的相关块；不得重新扩大为首次新增设计的全量候选读取。
7. **扩大读取触发**：只有当索引缺失、候选冲突、资产全貌不足以判断边界、追溯关系断裂或需要做全局一致性检查时，才可扩大读取范围，并在输出中说明扩大读取的原因。

### Step 2 · 新增设计处理

#### A. FR 路径

若本轮处理对象为 FR 类型 OR 条目，执行：

1. 沿用 `#role-list` 中已确认角色归属；无记录时再补做角色分配
2. **确定业务域**：从角色类型映射 23 资产业务域（如采购员→采购管理），显式写入 STR-F 节点
3. 按触发定义端到端业务闭环（该场景定义的职责范围）
4. 判断视角类型：`场景`（STR 末级节点固定为场景定义）
5. **信息载体与场景指标识别（十链扫描法）**：

   wft01a 从 STR 场景定义承接的 OR 语义中识别该场景需要哪些广义文档 / 信息载体，并识别用于评价场景运行成效的场景指标。信息链是 AI 的分析框架，不是场景定义的内容结构；场景定义正文始终输出扁平的信息载体清单和场景指标清单。

   本步采用“十链扫描法”：

   | 链 | 类型 | 分析引导 |
   |----|------|---------|
   | 执行业务链 | 信息载体 | 角色要创建、处理、提交、确认哪些业务单据、表单、记录或台账 |
   | 管理业务链 | 信息载体 | 管理者需要查看哪些进度、异常、统计、报表或看板 |
   | 个人绩效链 | 信息载体 | 场景是否沉淀个人工作量、效率、质量、达成率等评价依据 |
   | 组织绩效链 | 信息载体 | 场景是否沉淀部门、团队、项目、供应商等组织对象绩效依据 |
   | 资源需求链 | 信息载体 | 场景是否涉及资源申请、占用、消耗、查看、调度或约束 |
   | 协同交互链 | 信息载体 | 谁与谁之间需要通知、确认、反馈、回执、协同任务或接口交互 |
   | 风险合规链 | 信息载体 | 哪些动作需要审批、授权、留痕、审计、证据或异常处置 |
   | 规则知识链 | 信息载体 | 场景依赖哪些规则、标准、模板、SOP、知识库或策略配置 |
   | 状态事件链 | 信息载体 | 关键业务对象有哪些状态变化、事件触发、过程轨迹或日志 |
   | 场景指标链 | 指标 | 如何评价该场景运行得好不好，包含效率、质量、成本、风险、协同、体验等口径 |

   每个信息载体必须标注：

   - 载体名称
   - 归属链，可标注主链和必要的辅助链
   - 文档性质：业务显性文档 / 系统衍生文档 / 规则文档 / 知识文档 / 视图报表 / 其他
   - 产生方式：人类填写 / 系统生成 / 人机协同 / 外部输入
   - 一句话职责
   - 来源 OR
   - 来源类型：`OR明示` / `OR隐含` / `同类场景推断` / `资产线索推断`
   - 确认状态：`可确认` / `待确认` / `阶段性`
   - 26 资产方案级判断：`26-Q1 已有` / `26-Q2 可扩展` / `26-Q3 待新增` / `未检查`
   - 建议归属批次：按十链处理顺序标注，供 `wft01b` 逐链归属

   每个场景指标必须标注：

   - 指标名称
   - 指标类别：效率 / 质量 / 成本 / 风险 / 协同 / 体验 / 其他
   - 指标含义
   - 主要数据来源
   - 来源 OR
   - 来源类型
   - 确认状态
   - 建议归属批次：`场景指标链`

   十链建议处理顺序：

   | 顺序 | 信息链 |
   |------|--------|
   | 1 | 执行业务链 |
   | 2 | 状态事件链 |
   | 3 | 协同交互链 |
   | 4 | 风险合规链 |
   | 5 | 规则知识链 |
   | 6 | 资源需求链 |
   | 7 | 管理业务链 |
   | 8 | 个人绩效链 |
   | 9 | 组织绩效链 |
   | 10 | 场景指标链 |

   STR 场景定义中应保留 `### 信息链归属处理状态`，初始状态如下，供 `wft01b` 按链逐批处理：

   ```text
   | 顺序 | 信息链 | 处理状态 | 说明 |
   |------|--------|----------|------|
   | 1 | 执行业务链 | 待归属 | — |
   | 2 | 状态事件链 | 待归属 | — |
   | 3 | 协同交互链 | 待归属 | — |
   | 4 | 风险合规链 | 待归属 | — |
   | 5 | 规则知识链 | 待归属 | — |
   | 6 | 资源需求链 | 待归属 | — |
   | 7 | 管理业务链 | 待归属 | — |
   | 8 | 个人绩效链 | 待归属 | — |
   | 9 | 组织绩效链 | 待归属 | — |
   | 10 | 场景指标链 | 待归属 | — |
   ```

   若 OR 中没有直接触发词，AI 可以依据业务对象、角色职责、流程阶段、约束条件、异常条件、指标口径、上游资产或同类场景模式做弱推断，但必须标注为 `OR隐含`、`同类场景推断` 或 `资产线索推断`，并将确认状态置为 `待确认`。若缺少足够语义线索，AI 不得补全具体载体，只能输出缺口和澄清建议，方案状态置为 `阶段性方案`。

6. **推理依据与补全说明**：

   对所有超出 OR 直接明示的信息载体和场景指标，必须显性化 AI 推理依据。推理依据至少分为三类：

   | 推理类型 | 使用场景 | 输出要求 |
   |---------|----------|---------|
   | 业务常识推理 | 从业务动作、对象、角色职责推断常见业务单据、表单、记录 | 说明 OR 线索、业务规则、补全结果 |
   | 系统设计推理 | 从进度跟踪、状态变化、指标计算、自动触发推断系统衍生文档 | 说明实现该能力所需的状态、事件、日志或数据来源 |
   | 企业治理推理 | 从预算、供应商、质量、安全、审批、异常推断合规、规则、审计与证据载体 | 说明治理风险、控制点、补全结果 |

   推理依据采用编号引用，如 `R1`、`R2`。信息载体和场景指标条目应引用对应 `R#`，使人类可追溯每项补全的依据。

7. 派生 STR-F 节点名称（即场景定义名称）
8. 执行 Q1/Q2/Q3 找宿主与资产决策：复用 / 改进 / 新增

   - **Q1（直接匹配）**：OR 的功能描述与已有场景定义在语义上一致，则复用该宿主。判定依据：OR 中的"动词 + 业务对象"能直接对应场景定义的职责范围
   - **Q2（扩展匹配）**：OR 的功能描述是已有场景定义的子集或超集边缘，可扩展后承接。边界判断：扩展不改变场景定义的核心职责边界——若扩展后变成大杂烩，应改为新增
   - **Q3（新增）**：OR 无任何可用宿主，或扩展后导致边界模糊。要求：新增场景定义的命名和粒度应与同层已有节点保持一致

9. 完善 STR-F 要素：

   - 定义
   - 业务域（从角色映射）
   - 视角类型：`场景`（固定值，STR 末级节点均为场景定义）
   - 功能说明
   - 信息载体清单（十链扫描结果，含 26 资产方案级判断）
   - 场景指标
   - 信息链归属处理状态
   - 推理依据与补全说明
   - 十链覆盖自查
   - 承接的 OR 末级
   - 符合性分析
   - 设计指导

#### B. NFR 路径

若本轮处理对象为 NFR 类型 OR 条目，执行：

1. 沿用 `#role-list` 中已确认角色归属
2. 使用 22 资产逐级匹配：具体需求 → 分类维度 → 候选
3. 无匹配时执行候选新增评审
4. 完善 STR-NFR 节点要素：
   - 定义
   - 需求类型
   - 承接的 OR 末级
   - 符合性分析
   - 量化指标
   - 测量方法

#### C. 写入规则

Skill A 写文档时必须遵守 [文档编写约定 §十四](../../../../00-generalspec/00-doc-conventions.md) 的树形资产文档布局。写入范围按本轮相关块局部更新，不得重写大文档全文。

1. 写入 `02-*.md` 正文节点块：
   - 复用 / 改进时，定位既有 STR 节点块，只更新该块内的节点定义、信息载体清单、场景指标、推理依据、承接 OR 清单、反馈区和必要的变更摘要
   - 新增时，在合适父节点下新增稳定 `BLOCK`，写入完整 STR-F / STR-NFR 要素，并生成空 `### 反馈区`
2. 更新 `02-*.md` 树画像 / 节点索引 / 状态队列：
   - 若新增节点，或既有节点的一句话边界、父节点、层级、名称、状态发生变化，同步更新树画像行
   - 复用 / 改进时，更新既有 STR 节点索引行的状态、来源 OR、块 ID 和一句话边界
   - 新增时，新增 STR 节点索引行，记录节点 ID、父节点 ID、节点类型、层级、状态、块 ID 和一句话边界
   - 将本轮 STR 节点状态写为 `待确认方案`
3. 更新 `01-*.md` 树画像 / OR 正文块 / 状态队列：
   - 若 OR 条目状态或一句话边界在树画像中有摘要记录，同步更新树画像行
   - 在本轮 OR 正文块的当前分配记录中写入承接 STR 节点编号、承接方式和方案状态
   - 在 `01-*.md` 状态队列中将该 OR 条目状态写为 `待确认方案`
4. 更新追溯关系：
   - 在 `01` 与 `02` 的追溯区或本轮相关块承接关系中记录 OR→STR 引用
   - 只写引用关系，不复制对方正文
5. 写入反馈区：
   - 新方案必须保留 `### 反馈区`
   - 若已有未处理反馈，不得覆盖或清空
   - 若本轮消费了人类反馈，必须先归档反馈处理摘要，再生成新的空反馈区

写入分支：

1. **复用已有 STR 节点**：不新增节点；只在既有节点块内追加本 OR 到承接 OR 清单，并重新检查功能说明、信息载体清单、场景指标和推理依据是否需要补充；同步更新树画像、索引行和状态队列。
2. **改进已有 STR 节点**：在原节点块内更新节点定义与要素；保留旧版摘要和变更说明；更新该节点承接 OR 清单、树画像、索引行和状态队列。
3. **新增 STR 节点**：在合适角色分类下新增稳定节点块；写入完整 STR-F / STR-NFR 要素、反馈区、树画像行、节点索引行和状态队列记录。
4. 任一分支都不得覆盖人类尚未处理的反馈区内容。

### Step 3 · 方案反馈处理

若本轮处理对象是一个 STR 节点，且满足以下任一条件，则按该节点反馈区执行：

1. 节点状态为 `待确认方案`，且反馈区已有 `[同意] / [修改] / [驳回]` 人类审阅反馈
2. 节点状态为 `迭代中`

反馈分支：

1. `[同意]`
   - 先检查方案状态
   - 若方案状态为 `可确认方案`：不修改正文方案，归档反馈处理摘要并生成新的空反馈区，在 `02-*.md` 树画像、状态队列和节点索引中将节点状态推进为 `待文档归属`，在 `01-*.md` 树画像、本轮 OR 块和状态队列中同步方案已确认 / 待文档归属状态，输出下一步处理方式：运行 `wft01b`
   - 若方案状态为 `阶段性方案`：不得推进到 `待文档归属`，应输出 `需要补充后重跑wft01a`，并提示人类补充缺失信息或改写为 `[修改]`
2. `[修改]`
   - 按反馈类型调整节点定义与要素。反馈类型包括结构反馈、信息链反馈、指标反馈、推理依据反馈、其他说明
   - 必要时重做 OR→STR 承接关系
   - 重新生成推理依据与补全说明
   - 重新执行十链覆盖自查
   - 在当前 STR 节点块内归档本次人类反馈和修改前方案摘要，并生成新空反馈区
   - 更新 `02-*.md` 树画像 / 节点索引 / 状态队列，节点状态回写为 `待确认方案`
   - 若承接关系变化，同步更新 `01-*.md` 树画像、本轮 OR 块、状态队列和追溯引用
3. `[驳回]`
   - 重新执行该节点对应的 OR→STR 设计
   - 在当前 STR 节点块内归档原方案和本次人类反馈，写入新方案或按新宿主更新承接关系
   - 更新 `02-*.md` 树画像 / 节点索引 / 状态队列，节点状态回写为 `待确认方案`
   - 若改为其他 STR 宿主或新增节点，同步更新 `01-*.md` 树画像、本轮 OR 块、状态队列和追溯引用

## 四、方案审阅

### 审查口径

人类审阅 STR 场景定义方案时，重点审查：

1. STR 主结构方案是否成立
2. OR→STR 承接是否正确
3. 业务域归属是否合理
4. 信息载体识别是否覆盖全面，尤其是十链中是否有明显遗漏或过度推断
5. 场景指标是否属于本场景，口径是否合理
6. 推理依据是否透明，`OR隐含` / `同类场景推断` / `资产线索推断` 是否可接受
7. 是否允许进入文档归属

**人类审查焦点**：
1. OR→STR 的语义匹配是否正确——是否命中正确的宿主
2. 信息载体识别是否有明显缺口或过度补全
3. 场景指标是否需要保留、删除、改名或调整数据来源
4. AI 推理依据是否可信，待确认项是否接受
5. 新增 STR-F 节点的业务域、命名和粒度是否与现有资产风格一致

方案审阅不审：

1. 23 / 22 / 26 资产的正式写回结果
2. 下游 `wft02` 的详细展开

### 方案状态

| 方案状态 | 含义 | 是否可放行 |
|------|------|------|
| `阶段性方案` | 仍存在阻断项 / 关键缺口 | 否 |
| `可确认方案` | 主结构已稳定 | 是 |

## 五、运行结束输出（Step End）

Step End 的作用是把本轮设计结果翻译成人类下一次运行 `wft01a` 前的最小待办。AI 不只输出“方案已生成”，还必须根据当前运行结果，为人类确定一个且仅一个当前动作。

推荐交互方式：

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应 STR 节点的 `### 反馈区`、`01/02` 状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

### 单 ID 最小待办原则

1. 每次 Step End 只提示人类处理一个 ID。
2. 该 ID 可以是一个 STR 场景定义建议方案 ID、一个待选 OR 条目 ID、一个待进入 `wft01b` 的 STR 节点 ID。
3. AI 必须同时给出剩余待办 ID 数量，帮助人类知道当前队列规模。
4. 若上一次 Step End 已提示某个当前待办 ID，且该 ID 尚未被人类确认完成、明确切换或变为不可操作，下次运行必须继续提示同一个 ID。
5. 人类对当前 ID 的回复可以是 `同意`、`修改：……`、`驳回：……`，或 `选择：……`。
6. 只有当前 ID 已完成、被人类明确切换，或因状态变化不再可操作时，AI 才能按阻断优先级、状态时间和推荐顺序自动选择下一个当前 ID，并说明切换原因。
7. 当前 ID 处理完成后，下一次运行 `wft01a` 再提示下一个 ID。
8. AI 可以附带其他待办摘要，但不得要求人类同时审阅或选择多个 ID。

### 当前待办 ID 连续性规则

1. Step End 输出的 `当前待办ID` 默认形成“待办锁定”。
2. AI 应在 `02-*.md` 对应 STR 节点反馈区、`01/02` 状态队列、待进入 `wft01b` 选择记录或本轮运行摘要中保留该 ID 的待办状态。
3. 若下一次运行时该 ID 仍处于待反馈、待确认、待补充或待选择状态，AI 必须继续提示该 ID，不得因为出现新的待办而切换。
4. 若人类回复 `换一个 / 跳过 / 选择：<另一个ID>`，AI 可以切换当前待办 ID，但必须先记录切换原因。
5. 若当前 ID 已经完成，AI 才从剩余待办中选择下一个 ID。
6. 若当前 ID 因上游状态变化不再可操作，AI 必须说明原 ID 不再可操作的原因，再选择新的当前待办 ID。

### Step End 待办类型

| 待办类型 | 何时出现 | 当前 ID | 人类只需做什么 | 下次运行 |
|---------|----------|---------|----------------|----------|
| `反馈建议方案` | 本轮已生成或修订 STR 场景定义建议方案，状态为 `待确认方案` | 一个 STR 节点 ID | 对该 ID 回复 `同意 / 修改 / 驳回` | `wft01a` |
| `选择进入wft01a` | 当前没有占用中的 STR 方案审阅对象，且存在多个 `待分配 + 同意分配` OR 条目 | 一个 OR 条目 ID | 对该 ID 回复 `选择` 或指定另一个 OR ID | `wft01a` |
| `选择进入wft01b` | 一个 STR 场景定义方案已被 `[同意]` 并推进到 `待文档归属`，或存在多个待归属 STR 节点 | 一个 STR 节点 ID | 对该 ID 回复 `选择进入wft01b` | `wft01b` |
| `补充后重跑wft01a` | 当前方案为 `阶段性方案`，仍有阻断项 | 一个 STR 节点 ID | 对该 ID 回复 `修改：……` 并补充缺失信息 | `wft01a` |

### 输出结果

本 Skill 结束时，只允许输出以下四类结果之一，且每类结果必须带一个当前待办 ID：

| 输出结果 | 对应待办类型 | 当前 ID | 人类下一步 |
|---------|--------------|---------|------------|
| `等待建议方案反馈` | `反馈建议方案` | 一个 STR 节点 ID | 对该 ID 回复 `同意 / 修改 / 驳回` |
| `需要补充后重跑wft01a` | `补充后重跑wft01a` | 一个 STR 节点 ID | 对该 ID 回复 `修改：……` |
| `等待选择进入wft01a` | `选择进入wft01a` | 一个 OR 条目 ID | 对该 ID 回复 `选择`，或指定另一个 OR ID |
| `等待选择进入wft01b` | `选择进入wft01b` | 一个 STR 节点 ID | 对该 ID 回复 `选择进入wft01b` |

### 最小待办输出格式

AI 在 Step End 应按以下格式输出：

```text
当前结果：等待建议方案反馈 / 需要补充后重跑wft01a / 等待选择进入wft01a / 等待选择进入wft01b
当前待办ID：
是否延续上次待办ID：
剩余待办ID数量：
待办类型：
为什么先处理这个ID：
人类只需回复：
回复会写入：
下次运行：
其他待办摘要：
```

要求：

1. `当前待办ID` 必须是单个 ID，不得写成多个 ID 列表。
2. `是否延续上次待办ID` 必须写 `是 / 否`；若为 `否`，必须说明当前 ID 已完成、被人类切换或不再可操作的原因。
3. `剩余待办ID数量` 必须是数字；若无法精确统计，应写 `至少 N 个` 并说明统计口径。
4. `人类只需回复` 必须是一句话，且只包含当前 ID 的动作。
5. `其他待办摘要` 只能给数量和类型，不得要求人类同时处理。
6. 若人类直接在对话中回复，AI 应先转写到 `02-*.md` 对应 STR 节点反馈区、`01/02` 状态队列或待进入 `wft01b` 选择记录，再执行下一步。

### 结果一：等待建议方案反馈

适用：AI 已生成 STR 场景定义建议方案，状态为 `待确认方案`。

当前待办 ID：

1. 优先选择本轮刚生成或刚修订的 STR 节点 ID。
2. 若有多个 `待确认方案`，选择活动 OR 条目对应且最早进入该状态的一个。

人类只需做什么：

1. 对当前 STR 节点 ID 回复 `同意`、`修改：……` 或 `驳回：……`。
2. AI 收到回复后，先转写为该 STR 节点 `### 反馈区` 的规范记录，再在下一次 `wft01a` 中处理。

对话反馈示例：

```text
修改：同意整体方案，但 IC-02 名称改为“采购申请状态变更记录”；新增“采购预算占用信息”，归入资源需求链。
```

文档反馈区备选格式：

```text
### 反馈区

[操作] 修改 / 同意 / 驳回

[结构反馈]
-

[信息链反馈]
-

[指标反馈]
-

[推理依据反馈]
-

[其他说明]
-
```

重点查看位置：

1. `02-*.md` 对应 STR 节点正文
2. 该节点的 `信息载体清单`
3. 该节点的 `场景指标`
4. 该节点的 `推理依据与补全说明`
5. 该节点的 `十链覆盖自查`

### 结果二：需要补充后重跑wft01a

适用：当前 STR 场景定义建议方案仍有阻断项，或只能形成 `阶段性方案`。

当前待办 ID：

1. 必须是当前阶段性 STR 节点 ID。
2. 不得同时要求人类选择其他 OR 条目或进入 `wft01b`。

人类当前应做什么：

1. 查看 AI 输出的阻断项或缺口说明
2. 优先在对话中回复 `修改：……`
3. 补充缺失角色、业务对象、流程阶段、约束条件、异常条件、指标口径或其他必要信息
4. 下一次继续运行 `wft01a`

AI 收到对话反馈后，应先转写到 STR 节点反馈区，再执行重入处理。

重点查看位置：

1. AI 输出的阻断项清单
2. `02-*.md` 对应 STR 节点反馈区
3. 原始 OR 条目和其当前分配记录

### 结果三：等待选择进入wft01a

适用：当前没有待审阅 STR 建议方案，且存在可继续处理的 `待分配 + 同意分配` OR 条目。

口径说明：

1. `同意分配` 是上游状态队列中的正式入口许可，表示该 OR 条目允许进入 `wft01a`。
2. `选择进入wft01a` 是当多个 OR 条目均已获得入口许可时，本轮处理对象的选择动作。
3. 只有不存在活动 OR 条目时，AI 才能提示 `等待选择进入wft01a`。
4. 若已经存在活动 OR 条目，即使其他 OR 条目已有 `同意分配`，也不得提示选择新 OR 条目。
5. 若多个 OR 条目同时具备入口许可，AI 必须先输出本结果，不能自动进入新增路径。
6. 若只有一个 OR 条目具备入口许可且不存在活动 OR 条目，AI 可以直接进入新增路径。

当前待办 ID：

1. 必须是一个 OR 条目 ID。
2. 若存在活动 OR 条目未完成，不得输出本结果。

人类只需做什么：

1. 对当前 OR 条目 ID 回复 `选择`，或回复 `选择：<另一个OR ID>`。
2. AI 收到回复后，先转写到 `01-*.md` 状态队列或本轮选择记录，再在下一次 `wft01a` 中处理。

重点查看位置：

1. `01-*.md` 状态队列中 `待分配 + 同意分配` 的 OR 条目
2. 当前推荐 OR 条目的正文块
3. 相关角色列表和上游追溯关系

### 结果四：等待选择进入wft01b

适用：STR 场景定义方案已确认并推进到 `待文档归属`。

锁定说明：

1. 当 STR 节点已推进到 `待文档归属`，但人类尚未选择进入 `wft01b` 时，活动 OR 条目仍保持锁定。
2. 人类对当前 STR 节点 ID 回复 `选择进入wft01b` 后，AI 应先记录活动锁释放或转交状态，再由 `wft01b` 接手处理。
3. 活动锁释放前，`wft01a` 不得进入新的 OR 条目。
4. 若下一次运行 `wft01a` 时检测到人类已回复 `选择进入wft01b`，AI 只执行锁释放 / 转交记录，不得重新进入新增设计、方案反馈或方案改写。

释放 / 转交记录至少包含：

1. 活动 OR 条目 ID
2. 关联 STR 节点 ID
3. 被选择进入 `wft01b` 的 STR 节点 ID
4. 锁定状态：`已释放` 或 `已转交wft01b`
5. 释放 / 转交时间或轮次
6. 下一步 Skill：`wft01b`

当前待办 ID：

1. 必须是一个 `待文档归属` STR 节点 ID。
2. 若存在多个待归属 STR 节点，优先提示活动 OR 条目刚完成形成的 STR 节点。

人类当前应做什么：

1. 确认该 STR 节点状态已为 `待文档归属`
2. 对当前 STR 节点 ID 回复 `选择进入wft01b`
3. 在 Skill B 中按信息链顺序逐批处理该场景定义下广义文档的业务域、产品类型和构件类型归属

重点查看位置：

1. `02-*.md` 状态队列和节点索引中该 STR 节点状态
2. 该 STR 节点已确认的信息载体清单
3. 该 STR 节点已确认的场景指标
