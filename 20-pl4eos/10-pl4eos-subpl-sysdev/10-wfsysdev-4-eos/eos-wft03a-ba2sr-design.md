# BA → SysReq 架构 · Skill A · 功能定义设计
**Skill ID**: wft03a · **Phase 1: BA → SysReq Functional Definition Design**

> **概述**：每次执行只处理一个目标对象：一个 BA IPO 节点或一个迭代中的 SysReq 节点。Skill A 负责从 04-*.md 获取 BA IPO 节点及其初步方案，复制到 05-*.md 中作为工作草案，在此基础上产出 SR 层功能定义（四要素：功能描述、功能流程、功能规则、功能接口）并完成 SR-F 分类映射，供人类在 05-*.md 审阅确认。
>
> **输入 → 输出**：04-*.md 中一个 BA 五级 IPO 节点（含初步方案）→ 复制到 05-*.md → AI 产出功能定义建议（保留初步方案）→ 人类在 05 审阅确认 → 进入 wft03b 落账
>
> **适用场景**：瀑布式系统设计第3步 · Phase 1。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v2.1 | **修订**：2026-06-26

## 一、职责定位

### 本 Skill 目的与目标

> **主要目的**：取信于业务开发人员——「功能覆盖业务」。把 BA 的业务方案转化为 SysReq 的功能规格——逐操作划定人机边界：系统承接的写成功能，人保留的不写，系统辅助的写成交互规格。
>
> **目标**：功能规格独立于实现——不绑定技术栈或组件方案，一步不落，一步不多余。SysReq 是 wft05 映射 PA 组件的输入。

Skill A 负责：

1. 识别本轮唯一目标对象
2. 加载 wft02b 已确认的 BA IPO（含初步方案）、23/24/25/26/27 资产引用
3. 将 BA IPO 与初步方案从 04-*.md 复制到 05-*.md 作为工作草案，在此基础上收口为 SR 层功能定义（四要素：功能描述、功能流程、功能规则、功能接口）
4. 基于功能定义，按 SR-F 分类规则（参考 01-pl4eos-spec-sysdev.md §4.4）执行分类映射（确定所属系统职责桶）
5. 判断当前方案状态（阶段性方案 / 可确认方案）
6. 提交人类进行门禁 1 审查

Skill A 不负责：

1. `05-*.md` 的正式资产落账与状态推进（由 wft03b 承担）
2. `07-*.md` / `06-*.md` 的 PA 组件设计与详细定义（由 wft05 承担）
3. 引擎配置 IPO 的功能定义落账到 25（由 wft03b 承担）
4. `wft02` 已确认的 BA 初步方案活动描述改写

## 二、输入与路由

### 目标对象

每次执行只处理一个目标对象：

1. **新增路径**：`04-*.md` BA 状态表中 `当前状态=待架构映射 + 人类决策=同意架构映射` 的一个 BA 末级节点（含 wft02b 已确认的初步方案）
2. **重入路径**：`05-*.md` 中 `当前状态=迭代中` 的一个 SysReq 节点

### 活动 SysReq 节点锁定

`wft03a` 同一时间只能允许一个上游输入节点处于活动处理状态。活动输入节点未处理结束前，不得让新的上游输入节点进入本 Skill。

**活动输入节点判定：**

以下任一情况成立时，应视为存在活动输入节点：

1. `05-*.md` 中已存在引用某个 BA 节点的 SysReq 方案节点，且该方案节点仍处于 `待确认方案`、`迭代中` 或其他未完成状态。
2. 状态队列、反馈区或运行摘要中存在该输入节点的待办锁定记录。
3. BA 状态队列中该输入节点已被标记为进入 `wft03a`，但尚未记录 `wft03a` 处理完成。

**活动输入节点完成判定：**

以下任一情况成立时，活动输入节点才可视为完成或释放：

1. 对应功能定义方案已达到 `可确认方案`，且人类已确认进入 `wft03b`。
2. 该输入节点被人类明确撤销、废弃、跳过或退回上游。

**锁定规则：**

1. Step Start 必须先检查是否存在活动输入节点，再检查新的待进入节点。
2. 若存在一个活动输入节点，AI 只能重入该节点、处理该节点反馈、输出该节点当前待办，或提示进入正确的下游 Skill；不得选择新的输入节点。
3. 若存在多个活动输入节点，说明状态队列不一致，AI 必须停止新增处理，输出状态冲突说明，并要求人类先选择保留一个活动输入节点。
4. 只有不存在活动输入节点时，AI 才能从待进入队列中选择一个新的输入节点。
5. 新输入节点一旦进入本 Skill，AI 必须在状态队列、方案节点或运行摘要中记录活动锁定关系：`活动输入节点ID -> 方案节点ID / 当前待办ID`。

### 标准输入包

1. 目标 BA 节点（含 wft02b 已确认的 A1/A2/Bn 路径、业务定义匹配、引擎支撑判定、初步方案）
2. 当前 23/24/25/26/27 资产引用
3. `04-*.md` 树画像 / 状态队列 / 节点索引与 BA 节点正文块（含初步方案）
4. `05-*.md` 树画像 / 状态队列 / 节点索引（工作目标文档）
5. 重入反馈（若为重入路径）

### Step Start 路由

| 条件 | 路由 |
|------|------|
| 存在一个活动输入节点 | 只处理该活动节点对应的反馈、重入或当前待办，不得进入新节点 |
| 存在多个活动输入节点 | 停止执行，输出状态冲突说明，要求人类选择保留一个活动输入节点 |
| 存在多个 BA 节点 = `待架构映射 + 同意架构映射`，且不存在活动输入节点 | 输出 `等待选择进入wft03a`，当前待办 ID 为 AI 推荐的一个 BA 节点 ID |
| BA 节点 = `待架构映射 + 同意架构映射`，人类已回复 `选择进入wft03a` 或 `选择：<BA节点ID>`，且不存在活动输入节点 | 新增路径，写入活动输入节点锁定关系 |
| 仅存在一个 BA 节点 = `待架构映射 + 同意架构映射`，且不存在活动输入节点 | 新增路径，写入活动输入节点锁定关系 |
| SysReq 节点 = `迭代中` | 重入路径（Step 6） |
| SysReq 节点 = `待确认方案` 且反馈区为空 | 输出 `等待建议方案反馈`，不得覆盖既有方案 |
| SysReq 节点 = `待确认方案` 且反馈区有 `[同意] / [修改] / [驳回]` | 方案反馈路径（Step 6） |
| SysReq 节点 = `待功能定义确认`，人类已回复 `选择进入wft03b`，但活动锁尚未释放或转交 | 只记录活动锁释放 / 转交到 `wft03b`，不再进入新增设计或反馈处理 |
| SysReq 节点 = `待功能定义确认`，且活动锁尚未释放或转交 | 输出 `等待选择进入wft03b`，当前待办 ID 为该 SysReq 方案 ID |
| SysReq 节点 = `待功能定义确认`，且活动锁已释放或已转交 `wft03b` | 输出 `已转交wft03b`，不得继续修改该节点 |
| SysReq 节点 = `待架构映射完成` | 输出当前节点已完成，提示选择下一个 BA 节点 |
| 其他状态 | 输出不可操作说明后结束 |

## 三、操作流程

### Step 布局总览

| Step | 名称 | 产出 | 写资产 |
|------|------|------|--------|
| Start | 前置校验与路由 | 就绪 / 结束 | — |
| 1 | 资产加载与上下文建立 | 设计上下文 | — |
| 2 | 输入预检与方案可行性判断 | 预检结论 | — |
| 3 | 功能定义生成（核心设计） | 四要素功能定义 | `05-*.md` |
| 4 | 分类映射 | 分类归属方案 | `05-*.md` |
| 5 | 方案整理与写入 05 | 完整方案（初步方案+功能定义+分类） | `05-*.md` |
| 6 | 反馈重入处理 | 调整后的方案 | `05-*.md` |
| End | 运行结束输出 | 唯一路由 + 审查指引 | 独立成章，见 §五 |

### Step 1 · 资产加载与上下文建立

读取并建立本轮设计上下文：

1. 加载 `04-*.md` 中目标 BA 节点正文块，读取 BA IPO 五级节点信息、初步方案（活动描述、资源引用、场景定义、业务定义序列、异常路径等）、BA 路径、业务定义匹配、引擎支撑判定
2. 将目标 BA 节点信息与初步方案复制到 `05-*.md` 正文区，作为 `### 初步方案（来自 wft02b）` 小节（保留原始内容供人类对照）
3. 加载 `05-*.md` 当前 SysReq 状态表与已有方案
4. 加载 `23-eos-output-architecture.md`，定位目标 BA 节点所属产品路径
5. 加载 `24-eos-business-definitions.md`，读取目标 BA 节点匹配的业务定义
6. 加载 `25-eos-engine-models.md`，读取目标 BA 节点对应的引擎模型与配置 IPO 类型
7. 若为重入，加载目标 SysReq 节点反馈区的人类意见

加载要求（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §三）：

1. 读取系统设计过程文档和资产文档时，先读树画像 / 资产全貌、状态队列和节点索引，再读本轮正文块和必要相关块
2. 仅已确认资产可直接依赖；候选 / 疑似 / 待确认资产仅可作为参考
3. 若关键输入缺失，本轮最多只能形成 `阶段性方案`

### Step 2 · 输入预检与方案可行性判断

对目标 BA 节点执行统一分析：

1. 校验 BA→STR→OR 追溯链是否成立
2. 校验初步方案是否完整可消费——活动描述、资源引用、引擎匹配判定是否齐全
3. 判断 IPO 类型——业务定义类 IPO / 业务引擎能力类 IPO（决定功能定义的侧重方向）
4. 识别已有的标准职责桶宿主
5. 识别关键缺口：
   - 初步方案缺口（活动描述不足以支撑功能定义生成）
   - 功能定义缺口（四要素中哪些无法从初步方案确定）
   - 分类宿主缺口（是否需要新增职责桶）
   - 追溯链缺口

说明：

1. `wft03a` 不重复做 IPO 去重，直接消费 `wft02` 已确认的 `首次出现` 结果
2. 若关键缺口影响功能定义完整性，本轮应标记为 `阶段性方案`

### Step 3 · 功能定义生成（核心设计活动）

消费 BA 初步方案，收口为 SR 层功能定义。按 IPO 类型分流，两类 IPO 的功能定义四要素侧重不同。

**功能定义四要素：**

| 要素 | 回答的问题 | 业务定义类侧重 | 引擎配置类侧重 |
|------|-----------|---------------|---------------|
| 功能描述 | 系统做什么？ | "系统提供XX配置方案" | "系统提供XX引擎能力" |
| 功能流程 | 系统几步做完？ | 配置项处理管线（定结构→配规则→发布→生效） | 配置信息接收→解析→执行→产出 |
| 功能规则 | 系统怎么约束？ | 字段校验、业务规则、权限约束（可写完整、确定） | 引擎行为约束（部分在 PA 层才能收敛） |
| 功能接口 | 系统跟谁交互？ | 指向对应引擎（表单引擎、流程引擎等） | 对外 API、数据源、事件通知（内部接口在 PA 层确定） |

**3a · 业务定义类 IPO：**

将初步方案中的计划活动序列收口为功能定义：

1. 从初步方案的活动序列中提取 **功能流程**（系统视角的处理步骤，5~10 步）
2. 从初步方案的规则/约束中提取 **功能规则**（关键约束，每条一句话）
3. 从初步方案的引擎匹配/资源引用中提取 **功能接口**（指向哪个引擎、外部服务）
4. 综合形成 **功能描述**（一段话收口该 IPO 的系统功能使命）

**3b · 引擎配置类 IPO：**

将初步方案的引擎配置活动描述收口为功能定义：

1. 从配置 IPO 的 I/P/O 中提取 **功能流程**（配置信息接收→系统处理→运行期表现管线）
2. 从引擎模型定义中提取 **功能规则**（引擎行为的约束条件）
3. 从配置 IPO 的依赖关系中提取 **功能接口**（外部 API、集成点）
4. 综合形成 **功能描述**（该引擎能力提供什么系统功能）

**两类 IPO 的确定性差异：** 业务定义类 IPO 的功能定义可写到基本确定；引擎配置类 IPO 的功能规则和接口部分留"到 PA 阶段收敛"的空间，在功能定义中标注 `PA层收敛`

3. 将生成的功能定义四要素写入 `05-*.md` 正文区，作为 `### 功能定义建议（wft03a）` 小节，追加在初步方案之后，保留初步方案原文不动

### Step 4 · 分类映射

基于 Step 3 已生成的功能定义，按 SR-F 分类规则执行分类映射。

1. **确定第 1 层（业务定义类型 / 引擎能力类型）**：按 IPO 类型分流。业务定义类→按业务定义类型归入（流程定义、表单定义、指标定义等）；引擎配置类→按引擎能力类型归入
2. **确定第 2 层（系统职责分类）**：查标准职责桶清单，匹配已定义的功能描述与流程。按系统职责聚合，不按业务对象
3. **确定第 3 层（SysReq 末级节点）**：功能定义本身即 SysReq 末级节点的核心内容
4. 引擎能力类 IPO 默认只有第 1 层 + 第 3 层，不设中间层
5. 执行 `复用 > 改进 > 新增` 分类宿主决策

**分类原则遵守检查：**

- [ ] 先有末级（功能定义），再做聚合（原则 1）
- [ ] 按系统职责聚合，不按业务对象（原则 2）
- [ ] 一节点只归一个主职责（原则 3）
- [ ] 非末级节点能写出一句统一定义（原则 4）
- [ ] 粒度适中（原则 5）
- [ ] 业务域只做上层分区，不替代系统职责（原则 6）
- [ ] 跨域可复用同一职责分类（原则 7）
- [ ] 引擎能力类不分中间层（原则 8）

6. 将分类映射结果写入 `05-*.md` 正文区，作为 `### 分类映射` 小节，追加在功能定义建议之后

### Step 5 · 方案整理与写入 05

1. 将初步方案（来自 wft02b）、功能定义建议（四要素）、分类映射结果整理为一份完整的 05 工作草案
2. 在 `05-*.md` 中形成以下结构：

   ```
   ### 初步方案（来自 wft02b）      ← Step 1 已复制，保留原文不动
   ### 功能定义建议（wft03a）       ← Step 3 已写入
   ### 分类映射                    ← Step 4 已写入
   ### 完整方案摘要                 ← 本轮整理
   ### 反馈区                     ← 人类审阅入口
   ```

3. 节点状态设为 `待确认方案`
4. 在 SysReq 正文块写入可观察的方案状态字段，取值只能为 `阶段性方案` / `可确认方案`；若为 `阶段性方案`，必须写明阻断项 / 关键缺口
5. 更新 `05-*.md` 树画像 / 状态队列 / 节点索引

说明：初步方案原文保留不动，供人类对照 AI 的功能定义建议。人类审阅确认后，由 Step 6 的 `[同意]` 分支推进状态。

### Step 6 · 反馈重入处理

若目标为 `迭代中` SysReq 节点，按反馈区执行：

1. `[同意]`
   - 先检查方案状态是否为 `可确认方案`
   - 是 → 方案内容已在 05-*.md 中，无需重新写入；在 `05-*.md` 状态表将节点状态推进为 `待功能定义确认`，清空反馈区，将活动锁状态记为 `待进入wft03b` → 进入 End，输出 `等待选择进入wft03b`；只有人类回复 `选择进入wft03b` 或存在等效转交记录后，才记录活动 SysReq 节点锁释放 / 转交。转交记录至少包含：上游活动 BA 节点 ID、下游待处理 SysReq 方案 ID、状态到位说明（`待功能定义确认`）、人类选择动作（`选择进入wft03b`）、活动锁状态（`已释放` 或 `已转交wft03b`）、释放 / 转交时间或轮次、下游 Skill 名称（`wft03b`）
   - 否（阶段性方案）→ 不得放行。归档本次 [同意] 反馈，清空反馈区（留白）。节点状态保持 `待确认方案` → 进入 End，输出 `需要补充后重跑wft03a`，在阻断项清单中说明「当前方案为阶段性方案，[同意] 不构成有效放行；人类需回复 `修改：……` 补充缺失信息」
2. `[修改]`
   - 按反馈调整功能定义（四要素）、分类宿主、节点边界、承接关系
   - 若涉及分类宿主变更，重新验证分类原则合规性
   - 归档本次人类反馈和修改前方案摘要，清空旧反馈区并生成新空反馈区
   - 节点状态回写为 `待确认方案`
   - → 进入 End，重新评估方案状态并输出对应结果
3. `[驳回]`
   - 重新执行该节点对应的功能定义与分类设计（可回到 Step 2→Step 3→Step 4 完整链路，也可按驳回指明的范围定位到具体步骤）
   - 原方案归档，写入新方案
   - 清空旧反馈区并生成新空反馈区
   - 节点状态回写为 `待确认方案`
   - → 进入 End，重新评估方案状态并输出对应结果

## 四、方案审阅（门禁 1）

### 审查口径

门禁 1 审查的是：

1. **功能定义四要素是否完整**——功能描述、功能流程、功能规则、功能接口是否覆盖本 IPO 的核心功能需求
2. **功能定义是否可被 wft05 消费**——内容的粒度是否到组件设计级别（不要求开发级细节）
3. **分类映射是否符合 SR-F 分类规则**——按系统职责分类、非末级节点可写一句统一定义、粒度适中、主职责唯一
4. **两类 IPO 的确定性标注是否正确**——业务定义类是否已基本确定，引擎配置类是否已标注需 PA 层收敛的项
5. **承接关系是否闭合**——功能定义与 BA IPO 之间的追溯链成立
6. **是否允许进入正式功能定义落账**

门禁 1 不审：

1. `05-*.md` 的正式写回结果（由 wft03b 负责）
2. `06-*.md` / `07-*.md` 的 PA 组件详细设计（由 wft05 负责）
3. 引擎配置 IPO 的功能定义落账到 25 的格式正确性

### 方案状态

| 方案状态 | 含义 | 是否可放行 |
|------|------|------|
| `阶段性方案` | 存在阻断项 / 关键缺口 | 否 |
| `可确认方案` | 功能定义完整、分类合理、承接关系闭合 | 是 |

## 五、运行结束输出（Step End）

Step End 的作用是把本轮设计结果翻译成人类下一次运行 `wft03a` 前的最小待办。AI 不只输出"方案已生成"，还必须根据当前运行结果，为人类确定一个且仅一个当前动作。

推荐交互方式（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §四）：

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应节点的 `### 反馈区`、状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

### 单 ID 最小待办原则

1. 每次 Step End 只提示人类处理一个 ID。
2. 该 ID 可以是一个 SysReq 功能定义方案 ID、一个待选 BA 节点 ID、一个待进入 `wft03b` 的建议方案包 ID。
3. AI 必须同时给出剩余待办 ID 数量，帮助人类知道当前队列规模。
4. 若上一次 Step End 已提示某个当前待办 ID，且该 ID 尚未被人类确认完成、明确切换或变为不可操作，下次运行必须继续提示同一个 ID。
5. 人类对当前 ID 的回复可以是 `同意`、`修改：……`、`驳回：……`，或 `选择：……`。
6. 只有当前 ID 已完成、被人类明确切换，或因状态变化不再可操作时，AI 才能按阻断优先级、状态时间和推荐顺序自动选择下一个当前 ID，并说明切换原因。
7. 当前 ID 处理完成后，下一次运行 `wft03a` 再提示下一个 ID。
8. AI 可以附带其他待办摘要，但不得要求人类同时审阅或选择多个 ID。

### 方案状态与结果对照

End 不直接决定路由，而是输出方案状态和当前待办 ID。路由在下一轮 Step Start 或 Step 6 处理人类反馈后确定：

| 方案状态 | End 输出结果 | 人类操作 | 下一轮动作 |
|---------|-------------|---------|-----------|
| 可确认方案 | `等待功能定义方案反馈` | [同意] → Step 6 推进到待功能定义确认 → 输出 `等待选择进入wft03b`；人类回复 `选择进入wft03b` 后完成锁释放 / 转交 | wft03a 或 wft03b |
| 阶段性方案 | `等待功能定义方案反馈` 或 `需要补充后重跑wft03a` | [修改] → Step 6 调整方案 → 回到待确认方案；[同意] 无效 | wft03a |

### 当前待办 ID 连续性规则

1. Step End 输出的 `当前待办ID` 默认形成"待办锁定"。
2. AI 应在 `05-*.md` 对应 SysReq 节点反馈区、BA 状态队列或本轮运行摘要中保留该 ID 的待办状态。
3. 若下一次运行时该 ID 仍处于待反馈、待确认、待补充或待选择状态，AI 必须继续提示该 ID，不得因为出现新的待办而切换。
4. 若人类回复 `换一个 / 跳过 / 选择：<另一个ID>`，AI 可以切换当前待办 ID，但必须先记录切换原因。
5. 若当前 ID 已经完成，AI 才从剩余待办中选择下一个 ID。
6. 若当前 ID 因上游状态变化不再可操作，AI 必须说明原 ID 不再可操作的原因，再选择新的当前待办 ID。

### Step End 待办类型

| 待办类型 | 何时出现 | 当前 ID | 人类只需做什么 | 下次运行 |
|---------|----------|---------|----------------|----------|
| `反馈功能定义方案` | 本轮已生成或修订功能定义方案，状态为 `待确认方案` | 一个 SysReq 功能定义方案 ID | 对该 ID 回复 `同意 / 修改 / 驳回` | `wft03a` |
| `选择进入wft03a` | 存在多个已获入口许可的 BA 节点，且当前没有活动输入节点 | 一个 BA 节点 ID | 对该 ID 回复 `选择` 或指定另一个 BA ID | `wft03a` |
| `选择进入wft03b` | 方案已被 `[同意]` 并推进到可进入 wft03b | 一个建议方案包 ID | 对该 ID 回复 `选择进入wft03b` | `wft03b` |
| `需要补充后重跑wft03a` | 当前方案为 `阶段性方案`，仍有阻断项 | 一个 SysReq 方案 ID | 对该 ID 回复 `修改：……` 并补充缺失信息 | `wft03a` |

### 输出结果

本 Skill 结束时，只允许输出以下四类结果之一，每类结果必须带一个当前待办 ID：

| 输出结果 | 对应待办类型 | 当前 ID | 人类下一步 |
|---------|--------------|---------|------------|
| `等待功能定义方案反馈` | `反馈功能定义方案` | 一个 SysReq 方案 ID | 对该 ID 回复 `同意 / 修改 / 驳回` |
| `需要补充后重跑wft03a` | `需要补充后重跑wft03a` | 一个 SysReq 方案 ID | 对该 ID 回复 `修改：……` |
| `等待选择进入wft03a` | `选择进入wft03a` | 一个 BA 节点 ID | 对该 ID 回复 `选择` 或 `选择进入wft03a` |
| `等待选择进入wft03b` | `选择进入wft03b` | 一个建议方案包 ID | 对该 ID 回复 `选择进入wft03b` |

### 最小待办输出格式

AI 在 Step End 应按以下格式输出：

```text
当前结果：等待功能定义方案反馈 / 需要补充后重跑wft03a / 等待选择进入wft03a / 等待选择进入wft03b
当前待办ID：
是否延续上次待办ID：
剩余待办ID数量：
待办类型：
为什么先处理这个ID：
人类只需回复：
回复会写入：
下次运行：
其他待办摘要：
```

要求：

1. `当前待办ID` 必须是单个 ID，不得写成多个 ID 列表。
2. `是否延续上次待办ID` 必须写 `是 / 否`；若为 `否`，必须说明当前 ID 已完成、被人类切换或不再可操作的原因。
3. `剩余待办ID数量` 必须是数字；若无法精确统计，应写 `至少 N 个` 并说明统计口径。
4. `人类只需回复` 必须是一句话，且只包含当前 ID 的动作。
5. `其他待办摘要` 只能给数量和类型，不得要求人类同时处理。
