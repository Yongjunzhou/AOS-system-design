# 非功能需求 → STR-NFR · NFR 定义设计与资产化
**Skill ID**: wft01-nfr-or2str

> **概述**：每次执行处理 NFR 类型 OR 条目（≤10 条），STR-NFR 反馈优先于新 OR 处理。完成 OR→STR-NFR NFR 定义设计、22 资产对准与写回，形成可进入 wft04a 的 STR-NFR 节点。核心动作是**定性→定量**转化——将 OR 中的定性描述转化为可测量的量化指标，并逐指标对准 22 资产分类维度。
>
> **输入 → 输出**：NFR OR 条目（≤10 条）/ 有反馈的 STR-NFR 节点 → STR-NFR 节点方案 + 22 资产对准与写回（状态=`待分解分配`）+ OR 条目推进为 `已分配`
>
> **适用场景**：瀑布式系统设计第1步 · 非功能需求路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.4 | **修订**：2026-06-23

---

## 一、职责定位

本 Skill 负责：

1. 接收人类选中的 NFR OR 条目（≤10 条）
2. 按分类维度匹配 OR，确定宿主 STR-NFR 节点（Q1/Q2/Q3）
3. 将 OR 中的定性描述转化为量化指标（定性→定量）
4. 逐指标执行 22-Q1/Q2/Q3/Q3+ 资产对准（一级类目/分类维度）
5. 全 Q1 时自动写回 22 资产；Q2/Q3 时形成资产建议等人类确认后写回
6. 写回完成后推进 STR-NFR 节点到 `待分解分配`，对应 OR 到 `已分配`

本 Skill 不负责：

1. 业务需求（FR-BIZ → `wft01-biz-or2str`）
2. 引擎需求（FR-ENG → `wft01-eng-or2str`）
3. 下游 wft04a 的 NFR 详细展开

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（NFR OR 条目 ≤10 条 / 有反馈的 STR-NFR 节点）
2. 上游追溯链（`01-*.md`）
3. 已有 STR-NFR 架构资产（`02-*.md`）
4. `22-eos-nfr-taxonomy.md`（三层资产：已确认分类/具体需求模板/候选条目）
5. 重入反馈

### Step Start 路由

| 条件 | 路由 |
|------|------|
| OR 类型非 NFR | 输出"本 Skill 仅处理 NFR 类型"→ **退出** |
| OR >10 条 | 要求分批 → **退出** |
| 无 OR 选中 + 无待处理反馈 | 输出"无待处理对象"→ **退出** |
| 存在待处理反馈的 STR-NFR 节点 | **先处理反馈** |
| 存在 `待确认方案` + 反馈区为空的 STR-NFR 节点 | 不阻塞新 OR；Run End 时提示 |

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变分类匹配结果或指标量化方案，因此 Step 3 设计时**必须基于 Step 2 修改后的 STR-NFR 节点当前内容**，不得使用 Step 1 加载时的缓存旧状态。

按需加载线索：

- **反馈线索**：重入路径只读取反馈区及反馈指向的相关块，不重新做全量候选读取

### Step 3 · 新增 NFR 需求设计

#### Phase A：NFR 分类匹配与分组

**A.1 22 资产加载**：加载分类体系全貌（一级类目 → 分类维度 → 具体需求）。

**A.2 逐 OR 分类匹配**：

| 判定 | 条件 | 动作 |
|------|------|------|
| Q1（直接匹配） | OR 语义明确对应 22 中某分类维度 | 归入该维度 |
| Q2（扩展匹配） | OR 是某维度的子类或细化 | 归入该维度，扩展指标范围 |
| Q3（新增维度） | 无匹配分类维度 | 提出新分类维度建议，标记 `[推断]` + `分类体系扩展` |
| Q3+（新增一级类目） | 连一级类目都不匹配 | 提出新一级类目+分类维度建议 |

状态敏感匹配：`已确认`→正常；`待确认`→标注 `[推断]`；`疑似`→标注 `[推断]` + `归属待定`；`废弃`→不使用。

**A.3 STR-NFR 宿主匹配**：以分类维度 ID 搜索已有 STR-NFR 节点。Q1（复用）/ Q2（扩展）/ Q3（新建）。

**A.4 分组聚合**：按分类维度分组——同维度 OR 归入同一 STR-NFR。

#### Phase B：NFR 量化设计

**B.1 多 OR 收敛检测**（复用路径时）：比对锚点为 NFR 分类维度+量化指标口径。

| 关系 | 判定 | 处理 |
|------|------|------|
| 确认 | 新 OR 与既有指标一致 | 来源升级（`OR隐含`→`OR明示`），增量标记 `[确认]` |
| 补充 | 新 OR 补充既有指标的细节（测量方法/适用范围） | 追加内容，来源=`OR明示`，增量标记 `[新增]` |
| 矛盾 | 指标口径/目标值与既有不同 | 追加矛盾记录，增量标记 `[矛盾]`，等人类裁决 |
| 无新增 | 完全被既有指标覆盖 | 仅追加 OR 引用 |
| 未触及 | OR 未涉及该指标 | 保持既有内容不变，来源保持原类型 |
| 口径扩展 | 新 OR 要求既有指标扩展适用范围/调整目标值 | 更新目标值+适用范围，增量标记 `[改进]` + `[推断]` |

来源类型升级规则（只升不降）：`OR隐含`→`OR明示`、`推断`→`OR明示`。矛盾记录追加式不覆盖，存在未解决矛盾时不得推进到写回。

**B.2 量化指标提取**（定性→定量转化）：

| OR 描述类型 | 量化动作 | 来源标注 |
|-----------|---------|---------|
| 含具体数值 | 直接采信 | `OR明示` |
| 含比较词（快/稳定/安全） | 基于 22 资产+行业常识+同类经验推断 | `OR隐含` + `[推断]` |
| 含定性约束（必须/禁止/支持） | 转化为布尔型指标 | `OR明示` |
| 无任何量化线索 | 标注 `[阶段性]` + 缺口 | `无来源`（不得推进到写回） |

**B.3 测量方法**：压测 / 审计检查 / 日志统计 / 模拟 / 外部认证 / 人工检查 / 其他。

**B.4 适用范围**：标注该 NFR 约束哪些 STR-F/STR-E 节点（通过节点 ID 引用）。从全景加载的 `02-*.md` 树画像中筛选与本 NFR 分类维度相关的候选 STR-F/STR-E 节点；OR 明确指定适用范围时优先采信 OR，OR 未指定时基于 NFR 分类语义推断候选节点，标记 `[推断]`。

#### Phase C：验证与输出

**C.1 跨场景一致性检查**：口径一致 / 目标值不冲突 / 分类归属完整。

**C.2 22 资产扩展建议**：Q3/Q3+ 时在 STR-NFR 节点中追加扩展建议：

```text
[22 资产扩展建议]
建议类型：新增分类维度 / 新增一级类目
建议 ID：AS-XXX
22 资产路径：@nfr-{一级类目}:{分类维度}
建议名称：
建议定义：
建议依据：OR-XXXX 的 NFR 语义
关联指标：NFRI-XXX, NFRI-YYY
```

**C.3 STR-NFR 完整要素**：

```
STR-NFR 节点：
├── 定义
├── NFR 分类路径（一级类目/分类维度，含 22 资产 ID 引用）
├── 承接的 OR 末级
├── 量化指标清单：
│   ├── 指标 1：[名称]
│   │   ├── 目标值 / 单位 / 来源类型 / 测量方法
│   │   ├── 适用范围（约束的 STR-F/STR-E 节点 ID 列表）
│   │   ├── 22 一级类目 / 22 分类维度
│   │   ├── 22 判定（22-Q1/Q2/Q3/Q3+）
│   │   └── 22 资产建议（Q2/Q3时）
│   └── ...
├── 推理依据 / 适用范围汇总
├── 符合性分析 / 设计指导（待下游生成）
└── 量化指标清单变更摘要
```

### Step 4 · 资产写回

#### 写回 22（NFR 分类体系资产）

每个量化指标：
- Q1：定位一级类目→分类维度块 → 追加指标引用（含名称/目标值/单位/测量方法/适用范围）→ 记录追溯引用
- Q2/Q3/Q3+ → 进入 Step 5

写回要求：幂等去重、局部写回、全貌同步。

---

### Step 5 · 资产建议反馈

存在 Q2/Q3 时，在 `### 资产建议反馈区` 形成正式建议：

```text
[操作] 同意 / 修改 / 驳回

[22 资产建议]
- 建议 ID / 建议类型（新增分类维度 / 新增一级类目 / 扩展指标范围）
- 22 资产路径 / 建议名称 / 建议依据
```

| 操作 | 行为 |
|------|------|
| `[同意]` | 执行写回 → 进入 Step 6 |
| `[修改]` | 调整建议 → 重新等待 |
| `[驳回]` | 分类归属另有目标 → 重新对准；指标口径有问题 → 回退 Phase B 重做 |

### Step 6 · 状态推进

- `02` 记录 `### 资产写回记录`
- STR-NFR → `待分解分配`，OR → `已分配`
- OR→STR-NFR→22 引用链记录
- 清空反馈区待办信号

## 五、增量标记输出

NFR 路径的标记：

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| `[新增]` | 本次新增的内容 | 新 OR、新增量化指标 |
| `[改进]` | 对既有内容的修改 | 人类反馈驱动的指标/分类修订 |
| `[确认]` | 推断内容被新 OR 证实 | 来源类型升级（`OR隐含`→`OR明示`等） |
| `[矛盾]` | 新 OR 与既有指标存在分歧 | 需人类裁决 |
| `[推断]` | 匹配基于待确认边界或信息不足 | 候选分类维度/指标量化推断 |
| `[分类体系扩展]` | Q3 新增分类维度建议 | 22 资产无匹配分类维度 |
| `[阶段性]` | 指标量化线索不足，即使推断也无法完成完整设计 | 不得推进到写回，待后续 OR 补全 |
| `[22-Q1/22-Q2/22-Q3/22-Q3+]` | NFR 指标对 22 资产的对准结果 | Phase A/B 对准 |

增量清单格式（新建 STR-NFR）：

```text
=== STR-NFR-XXXX 本轮更新清单 ===

[新增] 承接 OR-XXXX, OR-YYYY（共2条）
[新增] NFRI-XXX "指标名称" 目标值=≤2s [22-Q1] → @nfr-性能:响应时间
[推断] NFRI-YYY "指标名称" 目标值=≥99.9% [22-Q2] → 扩展建议：@nfr-可靠性:可用性

=== 以上为本次全部新内容 ===

请逐项确认，或回复"整体确认同意"。
对有 [22-Q3/22-Q3+] 的项，请确认资产新增建议。
```

## 六、Run End 输出

### 单 ID 最小待办原则

1. 每次 Run End 只提示人类处理一个 ID
2. 当前 ID 未完成时，下次继续提示同一 ID
3. 只有当前 ID 完成/切换后，才选择下一个

### 最小待办输出格式

```text
当前结果：
当前待办ID：
是否延续上次待办ID：
剩余待办ID数量：
待办类型：
为什么先处理这个ID：
人类只需回复：
回复会写入：
下次运行：
其他待办摘要：
```

### Run End 提示

```text
当前状态：完成 NFR OR 处理
  修订 STR-NFR-XXX（+N 条 OR）/ 新建 STR-NFR-YYY（聚合 K 条 OR）

你可以：
  A. 增加新 OR → 选择 NFR OR 条目重新运行
  B. 确认方案 → 回复「整体确认同意」/「修改：…」/「驳回：…」
     - STR-NFR-XXX（待确认方案）
  C. 确认资产建议 → 回复「同意」/「修改」/「驳回」
     - 资产建议 ID：AS-XXX
  D. 进入 wft04a → 可用的待分解分配 STR-NFR 节点：
     - STR-NFR-WWW
```
