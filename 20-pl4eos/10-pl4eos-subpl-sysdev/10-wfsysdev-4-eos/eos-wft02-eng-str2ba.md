# STR-E → BA · 引擎配置架构设计与资产化
**Skill ID**: wft02-eng-str2ba

> **概述**：每次执行处理 ≥1 个 STR-E 节点（来自 wft01-eng，状态=`可以分解分配` / `待补充分解分配`），BA 反馈优先于新 STR-E 处理。基于配置单元法，将 STR-E 引擎定义中的配置单元清单转换为 A1 业务配置平台的 BA 主结构方案（引擎配置 IPO），完成 23/25 资产对准与写回，形成可进入 wft03 的 BA 节点。三步递进：配置单元分组→确定 A1 引擎配置 IPO →对准并写回 23（A1 引擎注册）+ 25（配置单元类型引用）。
>
> **输入 → 输出**：STR-E 节点（≥1 个，`可以分解分配` / `待补充分解分配`）/ 有反馈的 BA 节点 → BA 节点方案 + 23/25 资产对准与写回（状态=`可以SR设计`）+ STR-E 推进为 `已分解分配`
>
> **适用场景**：瀑布式系统设计第2步 · 引擎配置路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v2.0 | **修订**：2026-06-24

## 一、职责定位

本 Skill 负责：

1. 接收人类选中的 ≥1 个 STR-E 节点（`可以分解分配` / `待补充分解分配`）
2. 匹配候选引擎模型，确定 A1 场景定义名称和抽象业务定义类型
3. 以配置单元法将 STR-E 的配置单元清单展开为 A1 引擎配置 IPO
4. 引擎模型 → 23 产品类型/构件类型对准（23-Q1/Q2/Q3 —— A1 引擎注册）
5. 每个配置 IPO → 25 配置单元类型对准（25-Q1/Q2/Q3）
6. 人类确认方案后统一写回 23/25 资产（Q1/Q2/Q3 均直接写入，不需二次确认）
7. 人类确认时 BA 节点推进为 `可以SR设计`（wft03 可即时进入）；资产落账后对应 STR-E 推进为 `已分解分配`

本 Skill 不负责：

1. 业务架构需求（STR-F → `wft02-biz-str2ba`）
2. 非功能需求（STR-NFR → `wft04a`）
3. A2/Bn 的 BA 设计（→ `wft02-biz-str2ba`）
4. 引擎能力的功能定义正式写入 25（由 wft03b 在 SR 层功能定义确认后负责）
5. 下游 wft03 的 SR 架构设计展开

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（STR-E 节点 ≥1 个 / 有反馈的 BA 节点）
2. 上游追溯链（`02-*.md` STR-E 正文块 / 配置单元清单 / 引擎能力边界陈述）
3. 已有 BA 架构资产（`04-*.md` 树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`25` 引擎模型 + `23` 产品架构 + `24` 业务定义）
5. 重入反馈

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 STR-E 选中 + 无待处理反馈的 BA 节点 | 输出"无待处理对象"→ **退出** |
| STR-E 节点状态不是 `可以分解分配` / `待补充分解分配` | 输出状态分布，列出不合格节点及其当前状态 → **退出** |
| 选中节点含 STR-F / STR-NFR 类型 | 输出类型分布（STR-F → `wft02-biz`，STR-NFR → `wft04a`），要求分路径 → **退出** |
| STR-E 分属不同引擎模型且无依赖关联 | 输出引擎分布，提示可能需分批处理 → 人类确认后继续或分批 |
| 选中 STR-E >5 个 | 输出"本轮 N 个 STR-E，A1 配置 IPO 设计工作量大，建议分批"，提示节点 ID 列表 → 人类确认后继续，不强制退出 |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 BA 节点 | 本轮先处理全部有反馈的 BA 节点 |
| 存在 `待确认方案` + 反馈区为空的 BA 节点 | 不阻塞新 STR-E；Step End 时提示 |

#### 第三层：新 STR-E 路由

BA 反馈处理完后，逐节点处理新 STR-E → Phase A 宿主匹配。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变引擎模型匹配结果或配置 IPO 清单，因此 Step 3 设计时**必须基于 Step 2 修改后的 BA 节点当前内容**，不得使用 Step 1 加载时的缓存旧状态。

| Step | 名称 | 产出 | 写资产 |

### BA 节点生命周期

#### 推进态（按顺序）

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `待确认方案` | 方案已生成或修订，等待人类反馈 | Step 3 完成 |
| `可以SR设计` | 方案已确认，wft03 可启动 SR 架构设计 | 人类反馈 `[同意]` 或整体确认 |
| `在SR设计` | wft03 已锁定此 BA，正在展开 SR-F | wft03 写入活动锁 |
| `已SR设计` | SR-F 展开完成，BA 消费版本与当前版本一致 | wft03 处理完成且未检测到上游新版本 |

人类确认后，两条线并行推进、互不阻塞：
- **资产线**（本 Skill）：Step 4，将引擎模型/配置 IPO 引用写回 23/25 资产文件。资产写回完成与否不改变 BA 状态。
- **SR设计线**（wft03）：读 `可以SR设计`，展开 A1 引擎配置 SR-F 节点。不依赖 23/25 资产文件中的写回结果。

`在SR设计` 和 `已SR设计` 由 wft03 写入和推进。本 Skill（wft02-eng）写入的范围为 `待确认方案` / `可以SR设计` / `待补充SR设计`，并在修订时更新版本号。

#### 回退态（上游修订触发）

上游修订 = 本 Skill 因上游 STR-E 变更对已有 BA 执行 Step 3 Phase B。修订后版本号递增，状态按原状态分流：

| 原状态 | 修订后状态 | 含义 |
|--------|----------|------|
| `可以SR设计` | `可以SR设计`（版本递增） | 未被下游消费过，版本升级不影响任何 SR-F |
| `在SR设计` | `在SR设计`（版本递增） | wft03 锁仍有效，重入时检测版本差，按变更类型增量处理 |
| `已SR设计` | **`待补充SR设计`** | SR-F 已不完整，需 wft03 重新介入补全 |

- **`待补充SR设计`**：上游 STR-E 修订后，已消费的 SR-F 可能不完整。wft03 进入后按变更影响声明处理，处理完成后若版本一致 → 推进回 `已SR设计`

### Step 2 · 方案反馈处理

若本轮存在有反馈的 BA 节点（`待确认方案` + 反馈区有效），逐个处理。

#### 基本操作

| 操作 | 行为 |
|------|------|
| `[同意]` 或整体确认 | 归档反馈 → 生成空反馈区 → BA → `可以SR设计`（wft03 此时即可进入）→ 进入 Step 4 写回（Q1/Q2/Q3 统一写入，不需二次确认） |
| `[修改]`（局部） | 按反馈调整被指定的单元 → 仅重对准受影响部分 → 更新增量清单 → 回写 `待确认方案` |
| `[驳回]`（局部） | 移除被驳回的配置 IPO 和关联对准结论 → 其他对象 → `已确认` → 回写 `待确认方案` |
| `[驳回]`（整体） | 引擎模型匹配被推翻或抽象业务定义类型不对 → 重做 Step 3 → 所有对象确认状态重置 → 回写 `待确认方案` |

#### 分层反馈

BA 节点包含多个可独立决策的单元（引擎模型匹配结论、各配置 IPO 的四要素、23/25 资产对准）。人类可以**局部修改**——整体确认一个 BA 节点的同时，要求调整其中某个配置 IPO 或某个 25-Q 判定，不需要因一个局部问题而驳回整个 BA 节点。

AI 读取反馈时按以下规则处理，同步更新每个对象的确认状态：

- **局部 `[修改]`**：被指定的对象确认状态重置为 `可确认`，其涉及的资产对准重新执行。未提及的对象确认状态推进为 `已确认`，保持原结论不动，不触发全量重新对准。
- **局部修改导致级联**：引擎模型匹配变更 → 级联更新所有配置 IPO 的 23/25 对准（确认状态均重置为 `可确认`）；配置 IPO 拆分/合并 → 级联更新 25 对准。级联完成后重新执行 Phase C·1 跨 IPO 检查。
- **BA 节点推进条件**：所有内部对象的确认状态均为 `已确认` → BA 节点推进为 `可以SR设计`。存在任何一个 `可确认` 或 `待确认` → BA 节点保持 `待确认方案`。
- **`[驳回]`（局部）**：移除被驳回的配置 IPO 和关联对准结论。剩余对象的确认状态推进为 `已确认`。
- **`[驳回]`（整体）**：引擎模型匹配被推翻或抽象业务定义类型根本不对 → 重做 Step 3，所有对象的确认状态重置。

#### 反馈区位置

反馈区位于 `04-*.md` 中**每个 BA 节点块的末尾**，作为 BA 节点正文的最后一个子节。

#### 反馈区格式

**AI 写入（Step 3 完成时）** —— 空反馈区：

```text
### 反馈区

[操作] 
```

**人类填写（或 AI 从对话转写）** —— 三种操作之一：

- 整体同意 → `[操作] 同意`
- 局部修改 → `[操作] 修改`，列出被修改的单元 ID + 修改内容。未列出的视为同意
- 驳回 → 逐项列出被驳回的单元 ID + 原因。未列出的视为同意。驳回主结构时标注"整体驳回"

**AI 处理完成后** —— 归档旧反馈，重置反馈区：

AI 将本轮的人类反馈原文和处理动作写入 `### 反馈处理记录`（覆盖上轮内容，仅保留本轮），然后将反馈区重置为初始空状态 `[操作] `。完整历史追溯通过 BA 版本号 + §八 变更记录实现。

BA 节点块末尾结构：
```
### 反馈区              ← 当前反馈（AI 每次处理完重置为 [操作] ）
### 反馈处理记录         ← 仅保留本轮（如本轮无反馈则写"无"）
   修改 配置IPO-XXX Input → 已调整；25-Q2 已重新对准
### 变更影响声明         ← 见 §四·附
```

### Step 3 · 新增引擎配置设计

#### Phase A：触发与引擎模型匹配

**A.1 触发识别**

从 STR-E 读取驱动源——"哪个业务定义的配置需要启动了这项引擎定义需求"。触发 = "业务定义配置人员 + 需要让XX引擎具备YY能力 + 为了支撑ZZ业务定义"。

**A.2 引擎模型匹配与 23 产品架构对准**

从 `25-eos-engine-models.md` 匹配候选引擎模型：

| 情形 | 处理 |
|------|------|
| 匹配到单个已知引擎模型 | 加载配置单元类型清单 |
| 匹配到多个引擎模型 | 列出候选，人类选择 |
| 匹配到但配置单元类型清单为空或全为 `模板` | 基于 STR-E 配置单元推断候选配置 IPO，标记 `[推断]` + `模型内容待填充` |
| 无匹配 | 提出新引擎模型方案建议（引擎名称/类型/定义/候选配置 IPO），全部标记 `[推断]` + `归属待定` |

状态敏感匹配：`已确认`→正常；`待确认`→标注 `[推断]`；`疑似`→标注 `[推断]` + `归属待定`；`废弃`→不使用。

引擎模型匹配完成后，加载 `23` 资产全貌层，执行 **23 产品架构对准**：

| 情形 | 处理 |
|------|------|
| 23 中存在该引擎模型对应的产品类型/构件类型（`流程与IT域 → 产品A1 → <引擎模型>`） | 23-Q1 + 产品类型 ID + 构件类型 ID |
| 23 中存在相近的可扩展 | 23-Q2 + 扩展建议 |
| 23 中无匹配 | 23-Q3 + 新增建议 |

引擎模型是 EOS 产品架构的一部分，应在 23 中注册为 A1 平台引擎类构件类型。

**A.3 抽象业务定义确定**

定位于业务定义层——`xx业务配置运行`，xx 为抽象业务定义类型。以引擎模型的支撑能力边界为锚点确定粒度——通用引擎可较粗（如"表单类业务配置运行"），领域引擎应较细（如"计划类业务配置运行""BOM类业务配置运行"）。

**A.4 STR-E 宿主匹配**

以双键搜索已有 BA 节点：Q1（复用）/ Q2（扩展，同引擎不同业务定义）/ Q3（新建）。

#### Phase B：配置 IPO 展开

**B.1 候选配置 IPO 加载**：从 25 资产获取该引擎模型的配置单元类型清单作为候选集。每个配置单元类型对应一个 A1 引擎配置 IPO，字段映射：

| 25 资产字段 | → 配置 IPO 对应 |
|------------|--------------|
| 配置单元类型名称 | 配置 IPO 名称 |
| Input | 配置页面及配置信息（配置项、字段、规则、布局等） |
| Process | 系统处理（校验、解析、存储、编译、发布、运行期装载） |
| Output | 业务能力（业务用户可见、可操作的操作页面及运行机制） |
| 对应系统实体 | 运行时实体引用 |
| 依赖配置单元 | 前置配置 IPO 依赖 |

**B.2 多 STR-E 收敛检测**（复用路径时）：比对锚点为配置 IPO 的职责范围。

| 关系 | 判定 | 处理 |
|------|------|------|
| 确认 | 新 STR-E 触及的配置能力与既有配置 IPO 一致 | 来源升级（`OR隐含`→`OR明示`），增量标记 `[确认]` |
| 补充 | 新 STR-E 补充既有配置 IPO 的细节 | 逐栏追加内容，来源=`OR明示`，增量标记 `[新增]` |
| 矛盾 | 配置要素/处理方式与既有不同 | 追加矛盾记录，增量标记 `[矛盾]`，等人类裁决 |
| 无新增 | 完全被既有配置 IPO 覆盖 | 仅追加 STR-E 引用 |
| 未触及 | STR-E 未涉及该配置 IPO | 保持既有内容不变，来源保持原类型 |
| 占位→实体 | 模型标准单元被新 STR-E 首次触及 | 来源升级（`模型内置`→`OR明示`），增量标记 `[确认]` |

支持 IPO 内部逐栏标注和粒度调整建议。来源类型升级规则（只升不降）：`OR隐含`→`OR明示`、`模型内置`→`OR明示`、`推断`→`OR明示`。矛盾记录追加式不覆盖，存在未解决矛盾时不得推进到写回。

**下游变更影响声明**：多 STR-E 收敛检测完成后，按本轮 STR-E 对 BA 的实际变更范围判定变更类型——仅新增配置 IPO 且引擎模型匹配未变 → `增量型`；已有配置 IPO 修订但引擎模型匹配未变 → `修订型`；引擎模型匹配改变或引擎能力边界变化 → `结构型`。将变更影响声明写入 BA 节点末尾（格式与完整规则见 §四·附 变更传播协议）。

**B.3 逐配置 IPO 处理**：

| 触发判定 | 处理 | 来源类型 | 增量标记 |
|---------|------|---------|---------|
| STR-E 明确触及 | 完整设计四要素（Input/Process/Output/承接关系） | `OR明示` | `[新增]` |
| STR-E 隐含触及 | 完整设计四要素，标注推断 | `OR隐含` | `[新增]` + `[推断]` |
| 模型标准单元，STR-E 未触及 | 占位 | `模型内置` | `[跨场景]` |
| STR-E 超出已知类型 | 完整设计四要素 + 25 资产扩展建议 | `推断` | `[新增]` + `[推断]` + `模型需扩展` |

**B.4 配置 IPO 设计**（四要素）：

- **Input**：配置页面及配置信息——配置人员需要设置哪些参数/字段/规则/流程节点/状态/权限/指标/资源/布局等
- **Process**：系统处理——配置数据如何校验、解析、存储、编译、发布、版本管理和运行期装载，如何绑定到对应业务定义
- **Output**：业务能力——业务用户获得什么操作能力、以什么形态呈现（操作页面/列表台账/流程动作/状态流转/权限控制/指标看板等）
- **承接关系**：支撑的业务定义/场景定义 + 来源 STR-E 信息载体 + 25 引擎模型/配置单元类型引用

I→P→O 连贯性当场验证，四要素缺一不可。含 `[阶段性]` 的不得推进到写回。

**B.5 配置 IPO 排序与标识**：按 25 资产顺序/依赖拓扑/自然时序/STR-E 配置单元顺序四级优先级排列。

每个配置 IPO 标识：
- 序号/名称/来源类型/确认状态
- **25 判定**（25-Q1/Q2/Q3）/ **25 引擎模型 ID** / **25 配置单元类型 ID** / **25 资产建议**（Q2/Q3时）

#### Phase C：验证与输出

**C.1 跨 IPO 检查**：覆盖完整性 / 粒度自洽 / IPO 间无重叠 / 集中性（同引擎模型同抽象业务定义）。

**C.2 引擎能力边界陈述**：`「业务定义配置人员」需要为「xx引擎」配置「yy抽象业务定义」的从配置到运行的完整能力，包含 N 个配置 IPO（…）`。

**C.3 推理依据**：配置常识推理 / 引擎机制推理 / 运行体验推理，按主导角度归类，编号引用。

**C.4 BA 节点完整要素**：

BA 节点及内部对象均标注状态。**BA 节点状态**（写入 `04-*.md` 树画像/状态队列/节点索引，与 §四 生命周期表一致）：`待确认方案` / `可以SR设计` / `在SR设计` / `已SR设计` / `待补充SR设计`。

**内部对象确认状态**（写入 BA 节点正文块内，每个对象独立标注）：`可确认` / `待确认` / `已确认`。

**来源类型**（每个配置 IPO 标注，记录其从何而来，只升不降）：`OR明示` / `OR隐含` / `模型内置` / `推断`。来源类型升级规则见 Phase B·B.2（只升不降：`OR隐含`→`OR明示`、`模型内置`→`OR明示`、`推断`→`OR明示`）。

**增量标记**（每个对象标注，记录本轮对该对象做了什么）：按 §五 标记表。

### 四·附、BA 变更传播协议

下游 wft03 消费 BA 后生成 A1 引擎配置 SR-F 节点。当 BA 因上游 STR-E 修订而变更时，已生成的 SR-F 节点面临同步问题。本协议定义 BA → SR-F 的变更分类、通知格式和消费追溯。

#### 变更类型分类

| 变更类型 | 判定标准 | 实例 | 对下游影响 |
|---------|---------|------|----------|
| **增量型** | 仅新增配置 IPO，已有 IPO 的定义/四要素/归属均未改动；引擎模型匹配未变 | 新 STR-E 补充了一个配置 IPO | 已有 SR-F 节点不受影响，仅需映射新配置 IPO |
| **修订型** | 已有配置 IPO 的定义/四要素/25-Q归属被修改；引擎模型匹配未变 | 配置 IPO Input 从 X 改为 Y | 对应 SR-F 节点需局部调整 |
| **结构型** | 引擎模型匹配改变、引擎能力边界变化、抽象业务定义类型调整 | 新 STR-E 要求更换引擎模型或拆分抽象业务定义 | SR-F 结构可能重塑，需人类裁决后方可处理 |

判定优先级：结构型 > 修订型 > 增量型。

#### 变更影响声明格式

每次修订已有 BA 后，在 BA 节点末尾写入（`04-*.md` 中 BA 节点块内）：

```text
### 变更影响声明

BA 版本：v{N}（本次修订前为 v{N-1}）
本轮变更类型：增量型 / 修订型 / 结构型

变更项：
- [新增] / [改进] / [矛盾] / [模型需扩展] ...

下游已消费此 BA 的 SR-F 节点：
  <SR-F节点ID列表>（如无则写"尚未被下游消费"）

对已消费 SR-F 节点的影响：
  受影响节点：<SR-F节点ID + 受影响原因>
  不受影响节点：<SR-F节点ID>
  待新增 SR-F 映射：<新配置 IPO 对应的待映射说明>

建议 wft03 动作：
  - 增量型：增量映射新增配置 IPO 即可，已有 SR-F 节点不动
  - 修订型：修订受影响的 SR-F 节点 → 标记 [来自上游变更]
  - 结构型：输出影响范围，等人类裁决后再处理
```

#### 版本标记与消费追溯

- BA 节点初次生成时版本 = `v1`，变更影响声明写"首版，无变更"
- 每次因新 STR-E 触发 Step 3 Phase B 修订，版本号递增
- 版本号写入 BA 节点头部（与节点 ID 同级）
- wft03 首次消费 BA 时，在对应 SR-F 节点中记录 `BA 消费版本：v{N}`
- wft03 重入同一 BA 时比对版本：若 BA 当前版本 > 已消费版本 → 触发变更感知

#### 增量清单与变更影响声明的关系

| 维度 | 增量清单（§五） | 变更影响声明 |
|------|----------------|------------|
| 面向读者 | 人类 | 下游 Skill（wft03） |
| 目的 | 告知"本轮我做了什么，请逐项审阅" | 告知"BA 变了什么，你需要做什么" |
| 内容 | 本轮新增/改进/确认/矛盾的逐项列表 | 变更类型 + 对已消费 SR-F 的影响 + 建议动作 |
| 写入位置 | BA 节点末尾（与人类反馈区配对） | BA 节点末尾（独立小节） |
| 生成时机 | 每次 Step 3 产出方案时 | 仅修订已有 BA 时；新建 BA 写"首版，无变更" |

## 五、增量标记输出

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| `[新增]` | 本次新增的内容 | 新 STR-E、新增配置 IPO |
| `[改进]` | 对既有内容的修改 | 人类反馈驱动的配置 IPO 修订 |
| `[确认]` | 推断内容被新 STR-E 证实 | 来源类型升级（`OR隐含`→`OR明示`等） |
| `[矛盾]` | 新 STR-E 与既有配置 IPO 存在分歧 | 需人类裁决 |
| `[推断]` | 匹配基于待确认边界或信息不足 | 候选引擎模型/配置 IPO 推断 |
| `[跨场景]` | 模型标准单元，当前 STR-E 未触及 | 占位标记，待后续 STR-E 填充 |
| `[粒度调整]` | 配置 IPO 建议拆分或合并 | 跨 IPO 检查发现粒度不一致 |
| `[边界模糊]` | 两个配置 IPO 职责重叠 | 跨 IPO 检查发现边界不清 |
| `[模型需扩展]` | 超出已知配置单元类型 | STR-E 触及 25 未覆盖的能力 |
| `[阶段性]` | 配置 IPO 信息不足，即使推断也无法完成完整设计 | 不得推进到写回 |
| `[23-Q1/23-Q2/23-Q3]` | 引擎模型对 23 资产的对准结果 | Phase A 对准 |
| `[25-Q1/25-Q2/25-Q3]` | 配置 IPO 对 25 资产的对准结果 | Phase B 对准 |

增量清单是 **AI 在对话中的输出**，供人类阅读和反馈，不写入 `04-*.md` 或其他资产文档。格式如下（新建 BA）：

```text
=== BA-XXXX 本轮更新清单 ===

[新增] 承接 STR-E-XXXX, STR-E-YYYY（共2个）
[新增] 配置IPO-XXX "审批节点定义" [25-Q1] → @engine-流程引擎:审批节点定义
[推断] 配置IPO-YYY "超时规则定义" [25-Q2] → 扩展建议：@engine-流程引擎:超时规则定义

=== 以上为本次全部新内容 ===

请逐项确认，或回复"整体确认同意"。
```
