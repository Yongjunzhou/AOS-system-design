# STR-E → BA · 引擎配置架构设计与资产化
**Skill ID**: wft02-eng-str2ba

> **概述**：每次执行处理 ≥1 个 STR-E 节点（来自 wft01-eng，状态=`可以分解分配` / `待补充分解分配`），BA 反馈优先于新 STR-E 处理。基于 STR-E 的 CU 类型清单和引擎能力边界陈述，首次完整设计每个配置单元的三要素（Input/Process/Output），逐 CU 执行 25-Q 对准，完成 23/25 资产对准与写回，形成可进入 wft03 的 BA 节点。BA 节点级别声明承接关系（本引擎含 N 个配置单元，支撑 XX 业务定义/场景）。
>
> **输入 → 输出**：STR-E 节点（≥1 个，`可以分解分配` / `待补充分解分配`）/ 有反馈的 BA 节点 → BA 节点方案 + 23/25 资产对准与写回（状态=`可以SR设计`）+ STR-E 推进为 `已分解分配`
>
> **适用场景**：瀑布式系统设计第2步 · 引擎配置路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v2.1 | **修订**：2026-06-25

---

## 一、职责定位

本 Skill 负责：

1. 接收人类选中的 ≥1 个 STR-E 节点（`可以分解分配` / `待补充分解分配`）
2. 从 STR-E 读取业务引擎、23 对准结论、CU 类型清单和引擎能力边界陈述（不再重新匹配业务引擎和判定 23 对准）
3. **首次完整设计**每个配置单元的三要素（Input/Process/Output）——基于 25 资产的 CU 类型定义 + STR-E 的能力需求说明
4. 逐配置单元执行 25 资产对准（25-Q1/Q2/Q3）
5. 读取并确认 STR-E 的 23 对准结论（不重新判定；仅在 Q2 宿主匹配 + 23 路径不一致时标记 `[矛盾]`）
6. BA 节点级别声明承接关系（本引擎含 N 个配置单元，支撑 XX 业务定义/场景），不逐 CU 单独声明
7. 人类确认方案后统一写回 23/25 资产（Q1/Q2/Q3 均直接写入，不需二次确认）
8. 人类确认时 BA 节点推进为 `可以SR设计`（wft03 可即时进入）；资产落账后对应 STR-E 推进为 `已分解分配`

本 Skill 不负责：

1. 业务架构需求（STR-F → `wft02-biz-str2ba`）
2. 非功能需求（STR-NFR → `wft04a`）
3. A2/Bn 的 BA 设计（→ `wft02-biz-str2ba`）
4. 引擎定位、CU 类型识别和引擎能力边界判定——由 `wft01-eng-or2str` 负责
5. 下游 wft03 的 SR 架构设计展开

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（STR-E 节点 ≥1 个 / 有反馈的 BA 节点）
2. 上游追溯链（`02-*.md` STR-E 正文块 / CU 类型清单 / 引擎能力边界陈述）
3. 已有 BA 架构资产（`04-*.md` 树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`25` 业务引擎 + `23` 产品架构 + `24` 业务定义）
5. 重入反馈

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 STR-E 选中 + 无待处理反馈的 BA 节点 | 输出"无待处理对象"→ **退出** |
| STR-E 节点状态不是 `可以分解分配` / `待补充分解分配` | 输出状态分布，列出不合格节点及其当前状态 → **退出** |
| 选中节点含 STR-F / STR-NFR 类型 | 输出类型分布（STR-F → `wft02-biz`，STR-NFR → `wft04a`），要求分路径 → **退出** |
| STR-E 分属不同业务引擎且无依赖关联 | 输出引擎分布，提示可能需分批处理 → 人类确认后继续或分批 |
| 选中 STR-E >5 个 | 输出"本轮 N 个 STR-E，BA 方案设计工作量大，建议分批"，提示节点 ID 列表 → 人类确认后继续，不强制退出 |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 BA 节点 | 本轮先处理全部有反馈的 BA 节点 |
| 存在 `待确认方案` + 反馈区为空的 BA 节点 | 不阻塞新 STR-E；Step End 时提示 |

#### 第三层：新 STR-E 路由

BA 反馈处理完后，逐 STR-E 节点处理新设计 → Step 3。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 BA 宿主匹配或配置单元清单，因此 Step 3 设计时**必须基于 Step 2 修改后的 BA 节点当前内容**，不得使用 Step 1 加载时的缓存旧状态。

### BA 节点生命周期

#### 推进态（按顺序）

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `待确认方案` | 方案已生成或修订，等待人类反馈 | Step 3 完成 |
| `可以SR设计` | 方案已确认，wft03 可启动 SR 架构设计 | 人类反馈 `[同意]` 或整体确认 |
| `在SR设计` | wft03 已锁定此 BA，正在展开 SR-F | wft03 写入活动锁 |
| `已SR设计` | SR-F 展开完成，BA 消费版本与当前版本一致 | wft03 处理完成且未检测到上游新版本 |

人类确认后，两条线并行推进、互不阻塞：
- **资产线**（本 Skill）：Step 5，将业务引擎/配置单元引用写回 23/25 资产文件。资产写回完成与否不改变 BA 状态。
- **SR设计线**（wft03）：读 `可以SR设计`，展开 A1 配置单元 SR-F 节点。不依赖 23/25 资产文件中的写回结果。

`在SR设计` 和 `已SR设计` 由 wft03 写入和推进。本 Skill（wft02-eng）写入的范围为 `待确认方案` / `可以SR设计` / `待补充SR设计`，并在修订时更新版本号。

#### 回退态（上游修订触发）

上游修订 = 本 Skill 因上游 STR-E 变更对已有 BA 执行 Step 3。修订后版本号递增，状态按原状态分流：

| 原状态 | 修订后状态 | 含义 |
|--------|----------|------|
| `可以SR设计` | `可以SR设计`（版本递增） | 未被下游消费过，版本升级不影响任何 SR-F |
| `在SR设计` | `在SR设计`（版本递增） | wft03 锁仍有效，重入时检测版本差，按变更类型增量处理 |
| `已SR设计` | **`待补充SR设计`** | SR-F 已不完整，需 wft03 重新介入补全 |

- **`待补充SR设计`**：上游 STR-E 修订后，已消费的 SR-F 可能不完整。wft03 进入后按变更影响声明处理，处理完成后若版本一致 → 推进回 `已SR设计`

### Step 2 · 方案反馈处理

若本轮存在有反馈的 BA 节点（`待确认方案` + 反馈区有效），逐个处理。

#### 基本操作

| 操作 | 行为 |
|------|------|
| `[同意]` 或整体确认 | 归档反馈 → 生成空反馈区 → BA → `可以SR设计`（wft03 此时即可进入）。若本轮无新 STR-E 待处理则立即进入 Step 5 写回（Q1/Q2/Q3 统一写入，不需二次确认）；若本轮尚有新 STR-E 待处理则先执行 Step 3~4，待该批全部内容确认后由 Step 5 统一写回 |
| `[修改]`（局部） | 按反馈调整被指定的单元 → 仅重对准受影响部分 → 更新增量清单 → 回写 `待确认方案` |
| `[驳回]`（局部） | 移除被驳回的配置单元和关联对准结论 → 其他对象 → `已确认` → 回写 `待确认方案` |
| `[驳回]`（整体） | 业务引擎匹配被推翻 → 重做 Step 3 → 所有对象确认状态重置 → 回写 `待确认方案` |

#### 分层反馈

BA 节点包含多个可独立决策的单元（业务引擎匹配结论、各配置单元的三要素、23/25 资产对准）。人类可以**局部修改**——整体确认一个 BA 节点的同时，要求调整其中某个配置单元或某个 25-Q 判定，不需要因一个局部问题而驳回整个 BA 节点。

AI 读取反馈时按以下规则处理，同步更新每个对象的确认状态：

- **局部 `[修改]`**：被指定的对象确认状态重置为 `可确认`，其涉及的资产对准重新执行。未提及的对象确认状态推进为 `已确认`，保持原结论不动，不触发全量重新对准。
- **局部修改导致级联**：业务引擎匹配变更 → 级联更新所有配置单元的 23/25 对准（确认状态均重置为 `可确认`）；配置单元拆分/合并 → 级联更新 25 对准。级联完成后重新执行 3.3 跨配置单元验证。
- **BA 节点推进条件**：所有内部对象的确认状态均为 `已确认` → BA 节点推进为 `可以SR设计`。存在任何一个 `可确认` 或 `待确认` → BA 节点保持 `待确认方案`。
- **`[驳回]`（局部）**：移除被驳回的配置单元和关联对准结论。剩余对象的确认状态推进为 `已确认`。
- **`[驳回]`（整体）**：业务引擎匹配被推翻 → 重做 Step 3，所有对象的确认状态重置。

#### 反馈区位置

反馈区位于 `04-*.md` 中**每个 BA 节点块的末尾**，作为 BA 节点正文的最后一个子节。

#### 反馈区格式

**AI 写入（Step 3 完成时）** —— 空反馈区：

```text
### 反馈区

[操作] 
```

**人类填写（或 AI 从对话转写）** —— 三种操作之一：

- 整体同意 → `[操作] 同意`
- 局部修改 → `[操作] 修改`，列出被修改的单元 ID + 修改内容。未列出的视为同意
- 驳回 → 逐项列出被驳回的单元 ID + 原因。未列出的视为同意。驳回主结构时标注"整体驳回"

**AI 处理完成后** —— 归档旧反馈，重置反馈区：

AI 将本轮的人类反馈原文和处理动作写入 `### 反馈处理记录`（覆盖上轮内容，仅保留本轮），然后将反馈区重置为初始空状态 `[操作] `。完整历史追溯通过 BA 版本号 + §七 变更记录实现。

BA 节点块末尾结构：
```
### 反馈区              ← 当前反馈（AI 每次处理完重置为 [操作] ）
### 反馈处理记录         ← 仅保留本轮（如本轮无反馈则写"无"）
   修改 配置IPO-XXX Input → 已调整；25-Q2 已重新对准
### 变更影响声明         ← 见 §四·附
```

### Step 3 · BA 方案设计

#### 三字段标注规则

对 BA 节点内每个配置单元标注三个字段——**来源类型** / **增量标记** / **确认状态**，在 3.2 逐 CU 设计时内联标注。

| 字段 | 取值 | 含义 |
|------|------|------|
| 来源类型 | `OR明示` / `OR隐含` / `模型内置` / `推断` | 该对象从何而来，只升不降 |
| 增量标记 | `[新增]` / `[改进]` / `[确认]` / `[矛盾]` / `[推断]` / `[跨场景]` / `[模型需扩展]` / `[阶段性]` / `[粒度调整]` / `[边界模糊]` | 本轮对该对象做了什么 |
| 确认状态 | `可确认` / `待确认` / `已确认` | `可确认`=AI 设计完成等人类审阅；`待确认`=信息不足等人补充；`已确认`=人类审阅通过 |

标注范围覆盖 BA 节点内全部对象——业务引擎匹配结论、各配置单元的三要素、23/25 资产对准。

**初始标注规则**（对象创建时）：

| 时机 | 来源类型 | 增量标记 | 确认状态 |
|------|---------|---------|---------|
| 3.2 顺序 1 新建（STR-E 明确触及 + 25 类型匹配） | `OR明示` | `[新增]` | `可确认` |
| 3.2 顺序 2 新建（STR-E 触及 + 25 需扩展） | `推断` | `[新增]` + `[推断]` | `可确认` |
| 3.2 模型标准 CU 类型占位（STR-E 未触及） | `模型内置` | `[跨场景]` | `待确认` |
| 3.2 新建（信息不足，无法完成完整设计） | — | `[阶段性]` | `待确认` |
| 3.1 Q1 直接匹配（复用已有 BA） | 继承原来源类型 | `[确认]` | 保持原有状态 |
| 3.1 Q2 扩展匹配（改进已有 BA） | 不变 | `[改进]` | `可确认` |

**反馈处理后的流转规则**：已确认对象的确认状态通过 Step 2 反馈推进。`[整体同意]`→ 所有 → `已确认`；局部 `[修改]` → 被修改对象重置为 `可确认`（其余 → `已确认`）；局部 `[驳回]` → 移除对象（剩余 → `已确认`）。

#### 3.1 BA 宿主匹配与 23 对准

以**业务引擎为单键**（从 STR-E 读取）搜索已有 A1 BA 节点：

| 判定 | 条件 | 动作 |
|------|------|------|
| **Q1 直接匹配** | 已有 BA 节点，业务引擎相同 | 复用已有 BA → 追加 STR-E 引用 → 执行 3.2 时基于已有 CU 做收敛（确认/补充/矛盾/无新增/未触及） |
| **Q2 扩展匹配** | 已有 BA 节点可容纳但需扩展（如同引擎新增 CU） | 扩展已有 BA → 追加引用 |
| **Q3 新建** | 无可用已有 BA 节点 | 标记待新建 |

**23 对准确认**：读取 STR-E 中的 23 产品类型/构件类型/23-Q 判定，直接继承。仅在一种情况下标记 `[矛盾]`：BA 宿主匹配为 Q2（扩展已有 BA），且已有 BA 节点的 23 路径与 STR-E 的 23 判定不一致 → 标记 `[矛盾]`，等人类裁决。

> 注：此步骤不重新做业务引擎匹配——业务引擎和 23 对准结论来自 STR-E（wft01-eng 已判定）。wft02-eng 仅做 BA 宿主匹配，确定本轮 STR-E 分配到哪个 BA 节点。

#### 3.2 各配置单元设计

**输入**：STR-E 的 CU 类型清单（名称级，含能力需求说明）+ 引擎能力边界陈述（顺序 1/4/5 判定）。

对 STR-E 中顺序 1 和顺序 2 的 CU 类型执行设计；顺序 3 的 CU 类型**跳过**（留待 25 资产补齐后由后续轮次处理，Step End 提示）。

**加载**：对每个待设计的 CU 类型，从 25 资产获取其完整定义（Input 模板/Process 模板/Output 模板/依赖配置单元）。

**按能力判定分流**：

| 顺序 | 设计方式 | 来源类型 | 增量标记 |
|------|---------|---------|---------|
| **顺序 1**（25 已有匹配类型） | 基于 25 类型定义 + STR-E 能力需求说明，设计三要素完整方案 + 25-Q1 引用 | `OR明示` | `[新增]` |
| **顺序 2**（25 有近似类型需扩展） | 基于近似类型推断设计 + 25-Q2 引用 + 扩展建议 | `推断` | `[新增]` + `[推断]` |
| **顺序 3**（25 无匹配） | **跳过**，不进入设计 | — | — |

**三要素设计**（每个配置单元）：

```
配置单元：{CU名称}
  Input：配置页面及配置信息——配置人员设置哪些参数/字段/规则/选项
  Process：系统处理——校验→存储→编译→发布→运行期装载，如何绑定到引擎
  Output：业务能力——业务用户获得什么操作能力、以什么形态呈现
  
  依赖配置单元：<前置 CU 类型 ID 列表>（从 25 CU 类型定义读取，引用式标注，不决定设计顺序）
  来源类型 / 增量标记 / 确认状态（三字段见 Step 3 顶部）
  25 判定（25-Q1/Q2/Q3）/ 25 业务引擎 ID / 25 CU 类型 ID
  25 资产建议（顺序 2 时，含扩展方向说明）
```

**25-Q 对准**（每个 CU 设计完成后内联执行）：

| 判定 | 条件 | 动作 |
|------|------|------|
| 25-Q1 | CU 的 Input/Process/Output 与已有 CU 类型一致 | 引用已有 CU 类型 ID |
| 25-Q2 | 可映射到已有类型但需扩展 | 引用已有 CU 类型 ID + 扩展建议 |
| 25-Q3 | 无可用 CU 类型 | 新建 CU 类型建议 |

> 注：本步骤执行的是**首次完整设计**——wft01-eng 仅做了 CU 类型识别（名称级别），wft02-eng 基于 25 类型定义和 STR-E 的能力需求说明，设计每个 CU 的完整三要素。此处不包含承接关系——承接关系提升到 BA 节点级别（见 3.4）。

**CU 排序**：按 25 资产顺序/依赖拓扑/STR-E CU 类型顺序三级优先级排列。CU 类型之间的依赖关系从 25 资产"依赖配置单元"字段读取，作为排序参考，不决定设计顺序。

#### 3.3 跨配置单元验证

| 检查项 | 内容 |
|--------|------|
| **覆盖完整性** | STR-E 的顺序 1/4 CU 类型是否全部完成设计 |
| **粒度自洽** | CU 之间粒度是否统一（无不合理的拆分或聚合） |
| **无重叠** | 两个 CU 的 Input/Output 有无交叉，职责边界是否清晰 |
| **集中性** | 同业务引擎，全部 CU 归入同一 BA 节点 |
| **依赖完整性** | CU 间的依赖链是否闭环（被依赖的 CU 均在清单中） |

#### 3.4 BA 节点要素汇总

```
A1 BA 节点（每个 CU 标注三字段）：
├── 定义
├── 驱动源（从 STR-E 读取——"哪个业务定义需要该引擎具备什么能力"）
├── 业务引擎（25 资产引用，含业务引擎 ID + 来源类型/增量标记/确认状态，从 STR-E 读取）
├── 23 产品架构对准（继承 STR-E 的 23 判定）
│   ├── 产品类型（流程与IT域 → 产品A1）
│   ├── 构件类型（业务引擎）
│   ├── 23 判定（23-Q1/23-Q2/23-Q3）
│   └── 23 资产建议（Q2/Q3 时）
├── 引擎能力边界陈述（引用自 STR-E）
├── 配置单元清单：
│   ├── CU-01：[名称]
│   │   ├── Input（配置页面及配置信息）
│   │   ├── Process（系统处理）
│   │   ├── Output（业务能力）
│   │   ├── 依赖配置单元
│   │   ├── 来源类型 / 增量标记 / 确认状态
│   │   ├── 25 判定 / 25 业务引擎 ID / 25 CU 类型 ID
│   │   └── 25 资产建议（顺序 2 时）
│   └── ...
├── 承接关系：本引擎（含 N 个配置单元）支撑 XX 业务定义/场景，来源 STR-E-XXX
├── 推理依据 / 承接的 STR-E 末级
├── 符合性分析（`待 wft03 生成`）/ 设计指导（`待 wft03 生成`）
├── BA 版本号（初始 v1，每次因 STR-E 修订递增）
└── 配置单元清单变更摘要
```

**承接关系**为 BA 节点级别声明——整个引擎的全部配置单元协同工作，才能配置出可运行的业务定义。不逐 CU 单独声明承接。

**BA 节点推进到 `可以SR设计` 的前置条件**：所有内部对象的确认状态均为 `已确认`。

### Step 5 · 资产写回与落账

人类已在 Step 2 中整体确认方案（含业务引擎的 23-Q 判定及每个配置单元的 25-Q 判定），Step 5 统一执行写回与落账，不区分 Q 级别——Q1/Q2/Q3 均直接写入，不需二次确认。

#### 资产写回

**写回 23（产品架构资产——A1 引擎注册）**，按业务引擎的 23 判定：
- 23-Q1：定位`流程与IT域 → 产品A1` 下对应业务引擎的产品类型/构件类型 → 追加业务引擎引用（含引擎名称/定义/25业务引擎ID）→ 记录追溯引用
- 23-Q2：扩展已有产品类型/构件类型 → 追加业务引擎引用 + 扩展说明 → 记录追溯引用
- 23-Q3：新建产品类型/构件类型块 → 写入定义 + 引用 → 记录追溯引用

**写回 25（业务引擎资产——配置单元引用）**，每个配置单元按 25 判定：
- 25-Q1：定位业务引擎→配置单元类型块 → 追加配置单元引用（含 Input/Process/Output 摘要）
- 25-Q2：扩展已有配置单元类型 → 追加引用 + 扩展说明
- 25-Q3：新建配置单元类型块 → 写入定义 + 引用

写回 23 时写入 BA 节点级别的**承接关系**（本引擎含 N 个配置单元，支撑 XX 业务定义/场景）。

写回要求：幂等去重、局部写回、全貌同步（树画像/资产全貌/分片索引/状态队列/追溯引用）。

#### 资产落账

写回完成后：
- `04` 记录 `### 资产写回记录`（日期/路径/原子单元/目标资产块/动作/判定）
- `02` 对应 STR-E 节点推进为 `已分解分配`（记录 `STR-E 消费版本：v{N}`）
- STR-E→BA→23/25 引用链记录
- 清空反馈区待办信号

说明：Step 5 不改变 BA 节点状态——状态已在 Step 2 设为 `可以SR设计`（或已在 Step 3 修订时设为 `待补充SR设计`）。资产写回是独立的收尾工作，不构成 wft03 的前置条件。

### 四·附、BA 变更传播协议

下游 wft03 消费 BA 后生成 A1 引擎配置 SR-F 节点。当 BA 因上游 STR-E 修订而变更时，已生成的 SR-F 节点面临同步问题。本协议定义 BA → SR-F 的变更分类、通知格式和消费追溯。

#### 变更类型分类

| 变更类型 | 判定标准 | 实例 | 对下游影响 |
|---------|---------|------|----------|
| **增量型** | 仅新增配置单元，已有 CU 的定义/三要素/归属均未改动；业务引擎匹配未变 | 新 STR-E 补充了一个配置单元 | 已有 SR-F 节点不受影响，仅需映射新配置单元 |
| **修订型** | 已有配置单元的定义/三要素/25-Q归属被修改；业务引擎匹配未变 | 配置单元 Input 从 X 改为 Y | 对应 SR-F 节点需局部调整 |
| **结构型** | 业务引擎匹配改变、引擎能力边界变化 | 新 STR-E 要求更换业务引擎 | SR-F 结构可能重塑，需人类裁决后方可处理 |

判定优先级：结构型 > 修订型 > 增量型。

#### 变更影响声明格式

每次修订已有 BA 后，在 BA 节点末尾写入（`04-*.md` 中 BA 节点块内）：

```text
### 变更影响声明

BA 版本：v{N}（本次修订前为 v{N-1}）
本轮变更类型：增量型 / 修订型 / 结构型

变更项：
- [新增] / [改进] / [矛盾] / [模型需扩展] ...

下游已消费此 BA 的 SR-F 节点：
  <SR-F节点ID列表>（如无则写"尚未被下游消费"）

对已消费 SR-F 节点的影响：
  受影响节点：<SR-F节点ID + 受影响原因>
  不受影响节点：<SR-F节点ID>
  待新增 SR-F 映射：<新配置单元 对应的待映射说明>

建议 wft03 动作：
  - 增量型：增量映射新增配置单元 即可，已有 SR-F 节点不动
  - 修订型：修订受影响的 SR-F 节点 → 标记 [来自上游变更]
  - 结构型：输出影响范围，等人类裁决后再处理
```

#### 版本标记与消费追溯

- BA 节点初次生成时版本 = `v1`，变更影响声明写"首版，无变更"
- 每次因新 STR-E 触发 Step 3 修订，版本号递增
- 版本号写入 BA 节点头部（与节点 ID 同级）
- wft03 首次消费 BA 时，在对应 SR-F 节点中记录 `BA 消费版本：v{N}`
- wft03 重入同一 BA 时比对版本：若 BA 当前版本 > 已消费版本 → 触发变更感知

#### 增量清单与变更影响声明的关系

| 维度 | 增量清单（§四·附二） | 变更影响声明 |
|------|----------------|------------|
| 面向读者 | 人类 | 下游 Skill（wft03） |
| 目的 | 告知"本轮我做了什么，请逐项审阅" | 告知"BA 变了什么，你需要做什么" |
| 内容 | 本轮新增/改进/确认/矛盾的逐项列表 | 变更类型 + 对已消费 SR-F 的影响 + 建议动作 |
| 写入位置 | BA 节点末尾（与人类反馈区配对） | BA 节点末尾（独立小节） |
| 生成时机 | 每次 Step 3 产出方案时 | 仅修订已有 BA 时；新建 BA 写"首版，无变更" |

---

### 四·附二、增量标记输出

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| `[新增]` | 本次新增的内容 | 新 STR-E、新增配置单元 |
| `[改进]` | 对既有内容的修改 | 人类反馈驱动的配置单元 修订 |
| `[确认]` | 推断内容被新 STR-E 证实 | 来源类型升级（`OR隐含`→`OR明示`等） |
| `[矛盾]` | 新 STR-E 与既有配置单元 存在分歧 | 需人类裁决 |
| `[推断]` | 匹配基于待确认边界或信息不足 | 候选业务引擎/配置单元 推断 |
| `[跨场景]` | 模型标准单元，当前 STR-E 未触及 | 占位标记，待后续 STR-E 填充 |
| `[粒度调整]` | 配置单元 建议拆分或合并 | 跨配置单元检查发现粒度不一致 |
| `[边界模糊]` | 两个配置单元 职责重叠 | 跨配置单元检查发现边界不清 |
| `[模型需扩展]` | 超出已知配置单元类型 | STR-E 触及 25 未覆盖的能力 |
| `[阶段性]` | 配置单元 信息不足，即使推断也无法完成完整设计 | 不得推进到写回 |
| `[23-Q1/23-Q2/23-Q3]` | 业务引擎对 23 资产的对准结果 | 3.1 对准确认 |
| `[25-Q1/25-Q2/25-Q3]` | 配置单元 对 25 资产的对准结果 | 3.2 逐 CU 设计 |

增量清单是 **AI 在对话中的输出**，供人类阅读和反馈，不写入 `04-*.md` 或其他资产文档。格式如下（新建 BA）：

```text
=== BA-XXXX 本轮更新清单 ===

[新增] 承接 STR-E-XXXX, STR-E-YYYY（共2个）
[新增] 配置单元 "审批行为配置" [25-Q1] → @engine-form:@form-cu-behavior
[推断] 配置单元 "超时规则配置" [25-Q2] → 扩展建议：@engine-flow:超时规则定义

=== 以上为本次全部新内容 ===

请逐项确认，或回复"整体确认同意"。
```
