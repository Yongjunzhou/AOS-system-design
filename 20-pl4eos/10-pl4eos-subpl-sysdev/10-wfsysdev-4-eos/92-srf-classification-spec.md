## 一、分类层级结构

SR-F 功能架构默认采用以下三层结构：

```
第1层：业务定义类型            ← "由某类业务引擎通过配置生成的一组同形态业务定义集合"
第2层：系统职责分类            ← "按系统功能职责、实现机制与承接边界聚合"（中间层）
第3层：SysReq 末级节点         ← 视角转换后的具体业务定义或具体引擎能力 IPO
```

图示：

```
SR-F
├── 业务定义类                                 ← IPO 大类（顶部分流，非 SR-F 层节点）
│   ├── 流程定义                               ← 第1层：业务定义类型
│   │   ├── 流程定义配置                       ← 第2层：系统职责分类（中间层）
│   │   │   └── SR-F-xxx 采购申请流程定义      ← 第3层：SysReq 末级节点
│   │   ├── 审批规则配置
│   │   │   └── SR-F-xxx 采购审批规则定义
│   │   └── 状态跟踪
│   │       └── SR-F-xxx 采购申请处理状态跟踪
│   ├── 表单定义
│   │   ├── 结构定义
│   │   │   └── SR-F-xxx 采购申请表单结构定义
│   │   └── 录入提交
│   │       └── SR-F-xxx 采购申请表单录入提交
│   └── 指标定义
│       ├── 指标定义配置
│       │   └── SR-F-xxx 采购及时率指标定义
│       └── 指标计算
│           └── SR-F-xxx 采购及时率指标计算
│
└── 业务引擎能力类                             ← IPO 大类（顶部分流）
    ├── 流程引擎配置能力                        ← 第1层：引擎能力类型
    │   └── 流程定义配置页面                     ← 第3层（通常无中间层）
    └── 运行时交互能力
        └── 流程运行输入页面
```

说明：

1. **第1层（业务定义类型）**：不是"具体业务定义"（如请假流程），而是由某类业务引擎生成的一组同形态业务定义集合（如流程定义、表单定义、指标定义）。参见 §四 的标准清单。
2. **第2层（系统职责分类）**：本规范的核心定义对象。按系统功能职责、实现机制与承接边界对末级节点进行聚合。参见 §二~§五。
3. **第3层（SysReq 末级节点）**：视角转换后的具体业务定义或具体引擎能力 IPO，与 BA IPO 同体。
4. 引擎能力类 IPO 默认只有第1层+第3层，通常不设中间层，因为其分类已经由引擎能力类型足够表达。

分类与 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §三 全链路映射表的关系：

| 91 规范映射 | 本规范补充 |
|-------------|-----------|
| `wft03`：BA 末级 IPO → SysReq 节点 | 明确了 SysReq 节点之上的**非末级分类结构**：业务定义类型 → 系统职责分类 → SysReq 末级节点 |
| 分类宿主匹配与写回 | 明确了分类依据是**系统职责**（非业务域、非业务对象） |

## 二、系统职责的定义

### 2.1 定义

**系统职责**是 SR-F 非末级节点的分类维度，指：

> 对去重后的功能清单按**系统功能职责、实现机制与承接边界**进行聚合后形成的分类节点。

它回答的问题是：

> "系统用什么功能职责来承接这个业务定义？"

而不是：

> "这个业务定义属于哪个业务域？"

### 2.2 与 BA 分类的区分

| 维度 | BA 分类（业务视角） | SR-F 分类（系统职责视角） |
|------|-------------------|------------------------|
| 分类依据 | 业务域、业务对象、角色场景 | 系统功能职责、实现机制、PA 承接边界 |
| 聚合方式 | 按"做什么业务"聚合 | 按"系统怎么处理"聚合 |
| 分类稳定性 | 随业务组织方式变动 | 相对稳定，跨业务域复用 |
| 对 PA 的意义 | 间接 | 直接——为 PA 组件划分边界 |

### 2.3 系统职责的判断句式

可用以下句式判断一个 SR-F 末级节点应归入哪个系统职责：

> "这个功能在系统里主要负责什么？"

典型答案（标准职责桶）包括：采集/录入、校验/计算、计划/调度、执行/驱动、监控/预警、分析/评价、规则/配置、协同/通知 等。详见 §四。

## 三、分类原则

### 原则 1：先有末级，再做聚合

分类只能建立在已完成视角转换的 SysReq 末级节点之上。不能先假设上层分类，再倒推末级节点去适配。

### 原则 2：按系统职责聚合，不按业务对象

两个末级节点能否归入同一系统职责分类，首先看它们是否承担同一种系统职责。判断句式见 §2.3。

✅ 按系统职责分：
```
流程定义 → 规则配置 → 采购审批规则定义
                       生产审批规则定义     ← 跨业务域，但同一种系统职责
```

❌ 不按业务域分：
```
流程定义 → 采购域 → 采购审批规则定义       ← 这不是系统职责
                       采购状态跟踪定义     ← 这里两个职责不同，不该硬聚
```

### 原则 3：一节点只归一个主职责

一个末级节点可能同时具有执行、校验、通知等多重特征。分类时只按其**主职责**归类，不做一节点多父类。次要职责写入节点定义或设计指导，不用于主导分类。

### 原则 4：非末级节点必须能写出一句统一定义

如果一个候选分类节点无法写出一句能覆盖其所有子节点的统一定义，说明这个分类不成立。

判断模板：

> "本类功能用于支持系统完成 **\_\_\_** 类职责，为下层末级功能提供统一的能力边界。"

写不顺，通常就是聚合错了。

### 原则 5：粒度适中

一个系统职责分类下的末级节点数量应适中：
- **太少**：只有 1 个末级节点，通常说明这层分类没有价值
- **太多**：超过 7~9 个末级节点，通常说明职责混杂，需要再拆
- **合理**：2~5 个最稳，6~8 个可接受

### 原则 6：业务域只做上层分区，不替代系统职责

SR-F 可以**先按业务域大分区**，但**每个分区内仍须按系统职责再分类**。不能直接用业务域代替系统职责。

```
✅ 正确：
采购管理（业务域分区）
├── 规则配置
├── 业务执行
├── 过程监控
└── 统计分析

❌ 错误：
采购管理（业务域分区）
└── 采购相关功能（非系统职责）
```

### 原则 7：允许跨域复用同一职责分类

不同业务域下可以重复出现相同的系统职责分类名称。例如采购域、生产域都可有：

- `规则配置`
- `业务执行`
- `过程监控`
- `统计分析`

这样做的好处是跨域保持系统功能架构的一致性。

### 原则 8：引擎能力类默认不分中间层

业务引擎能力类 IPO 的 SR-F 结构通常只有：

```
引擎能力类型 → 具体引擎能力 IPO
```

不设中间层，因为引擎能力类型本身已经足够表达分类。仅在引擎能力类型下节点过多（>7个）或 PA 承接边界混杂时才考虑添加中间层。

## 四、标准系统职责桶清单

### 4.1 跨类型通用职责桶

以下系统职责桶在多个业务定义类型下可复用：

| 职责桶 | 系统职责描述 | 典型适用类型 |
|--------|-------------|-------------|
| 定义配置 | 业务定义的结构、属性、规则的配置与管理 | 流程定义、表单定义、指标定义、工单定义、计划定义 |
| 规则判定 | 条件判断、审批路由、自动决策 | 流程定义 |
| 流程编排 | 多步骤流程定义、条件分支、并行/串行编排 | 流程定义 |
| 状态跟踪 | 状态推进、进度查看、历史记录 | 流程定义、工单定义、计划定义 |
| 通知触发 | 消息生成、待办推送、通知分发 | 流程定义、工单定义 |
| 结构定义 | 数据字段、表格结构、关系定义 | 表单定义、台账定义 |
| 字段校验 | 数据格式验证、必填检查、业务规则校验 | 表单定义 |
| 录入提交 | 数据创建、编辑、保存、提交 | 表单定义、工单定义 |
| 展示查询 | 列表展示、筛选、排序、详情查看 | 表单定义、台账定义 |
| 数据采集 | 外部数据接入、批量导入、接口同步 | 指标定义、台账定义 |
| 指标计算 | 公式计算、聚合运算、周期计算 | 指标定义 |
| 分析预警 | 阈值判断、异常检测、预警生成 | 指标定义 |
| 计划编制 | 计划创建、任务分解、资源分配 | 计划定义 |
| 进度追踪 | 执行反馈、进度汇总、偏差分析 | 计划定义 |
| 授权审批 | 权限控制、审批节点、签名确认 | 定义通用 |
| 协同交互 | 跨角色通知、待办转办、共享协作 | 定义通用 |

### 4.2 按业务定义类型的推荐职责桶

#### 流程定义

```
流程定义
├── 定义配置         ← 流程定义的结构配置（节点、流向、角色）
├── 规则判定         ← 审批条件、自动路由、决策规则
├── 流程编排         ← 多步骤编排、条件分支、并行处理
├── 状态跟踪         ← 流程实例状态、当前节点、处理记录
└── 通知触发         ← 待办通知、超时提醒、事件通知
```

#### 表单定义

```
表单定义
├── 结构定义         ← 表单字段定义、布局结构、关联关系
├── 字段校验         ← 必填检查、数据格式、业务规则
├── 录入提交         ← 数据创建、编辑、保存、提交流程
└── 展示查询         ← 表单列表、详情查看、搜索筛选
```

#### 指标定义

```
指标定义
├── 指标定义配置     ← 指标名称、口径、维度、周期
├── 数据采集         ← 数据源接入、数据加载、接口获取
├── 指标计算         ← 公式计算、聚合运算、趋势计算
└── 分析预警         ← 阈值判断、异常检测、预警推送
```

#### 工单定义

```
工单定义
├── 定义配置         ← 工单模板、字段配置、流转规则
├── 任务分派         ← 自动派工、抢单、指定责任人
├── 状态跟踪         ← 工单状态推进、处理记录、回退
└── 通知触发         ← 新工单通知、超时催办、完成通知
```

#### 计划定义

```
计划定义
├── 计划编制         ← 计划创建、任务分解、资源分配
├── 进度追踪         ← 执行反馈、进度上报、偏差分析
├── 监控预警         ← 进度延迟提醒、资源冲突预警
└── 统计报表         ← 计划完成率、资源利用率统计
```

### 4.3 桶清单的使用方式

1. **优先选择**：对一个 SysReq 末级节点，先看 §4.1 的通用职责桶中是否有匹配项。
2. **按业务定义类型特化**：若通用桶不够精确，按 §4.2 查找对应业务定义类型的推荐桶。
3. **可新增职责桶**：当通用桶和推荐桶均无法覆盖时，允许新增职责桶，但必须：
   - 写明桶名和系统职责描述
   - 确保该桶名在同类业务定义类型下可复用
   - 经人类审阅确认后再纳入

## 五、冲突裁定规则

### 5.1 多归属冲突

当一个末级节点同时匹配多个系统职责桶时：

| 冲突类型 | 裁定规则 | 示例 |
|---------|---------|------|
| 执行类 vs 管理类 | 取**执行类**为主职责 | "采购审批"既像"审批授权"又像"流程编排"——以执行流程编排为主 |
| 配置类 vs 运行类 | 取**运行类**为主职责 | "表单录入"既涉及"结构定义"又涉及"录入提交"——以录入提交为主 |
| 计算类 vs 展示类 | 取**计算类**为主职责 | "指标计算"同时需要"数据采集"和"计算结果展示"——以数据采集和计算为主 |
| 业务类 vs 支撑类 | 取**业务类**为主职责 | "通知触发"既是"协同交互"又是流程定义中的"通知触发"——按业务定义类型的推荐桶为主 |

### 5.2 通用裁定流程

按以下优先级裁定：

1. **看主功能**——该末级节点在系统中**主要**做什么处理？
2. **看对应业务定义类型的推荐桶**——§4.2 的推荐桶优先于 §4.1 的通用桶
3. **看 PA 承接方向**——该节点未来大概率进入哪个 PA 组件？按 PA 归属裁定
4. **仍不确定时**——以"该职责桶下是否已有一句统一定义"为最终标准：能写通统一节点定义的桶才是正确归属

### 5.3 命名冲突

不同业务定义类型下可能出现同名职责桶（例如"定义配置"在流程定义、表单定义、指标定义中都出现）。这是允许的——因为它们属于不同的第1层父节点，在树形结构中路径不同。

但如果同一第1层父节点下出现两个同名职责桶，说明分类粒度冲突，应合并或重新命名词义更精确的桶名。
