# SysReq → PA 架构（单节点 · 状态机驱动）
**Skill ID**: wft05 · **Step 5: SysReq → Product Architecture**

> **概述**：每次执行处理**一个** SysReq 功能架构末级节点（IPO），根据其当前状态路由到对应阶段：Phase 1（设计开发）→ 人类审阅门禁 1 → Phase 2 上半（资产落账）→ Phase 2 下半（详细定义展开）→ 人类审阅门禁 2。单节点处理模式使迭代自然简化为对同一节点的反复执行，无需区分"动/不动"的复杂路由逻辑。
>
> **核心变化**：本版采用**状态机驱动路由**替代线性 Step 序列——Step Start 读取目标节点当前状态，决定进入 Phase 1（首次或重入）、Phase 2 上半（落账）或 Phase 2 下半（详细定义）。两类 IPO 在 Phase 1 中走不对称设计路径：业务定义类 IPO 为模式匹配（机械性），业务引擎能力类 IPO 为分解分配分析（创造性）。
>
> **输入 → 输出**：SysReq 架构定义（`05-*.md`）+ SysReq-NFR 约束 → SysReq 详细定义（`06-*.md`）+ PA 架构定义（`07-*.md`）
>
> **适用场景**：瀑布式系统设计第5步。本步合并功能分支（`wft03`）与非功能分支（`wft04`），完成 SysReq→PA 的桥接重组与组件生成。
>
> **设计模式参考**：[90-design-split-pattern.md](90-design-split-pattern.md)（两阶段分离模式 v0.4）
>
> **版本**：v4.0 | **修订**：2026-06-17

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：`05-*.md` SysReq 状态表中人类选入的**一个** SysReq 功能架构末级节点（IPO）
- **产出1**：PA 架构定义（`07-*.md`）——按 IPO 类型生成 PA 末级组件
- **产出2**：SysReq 详细定义（`06-*.md`）——围绕该 IPO 展开为可分配条目（Phase 2 下半）
- **产出3**：PA 组件状态更新——供人类确认与后续 `t06` 追溯验证

本步按 IPO 类型生成两类 PA 末级组件：
- **业务定义类 IPO → 配置方案组件**：流程定义组件、表单定义组件、工单定义组件、计划定义组件、指标定义组件等
- **业务引擎能力类 IPO → 前后端组件**：前端配置组件、前端运行组件、后端定义处理组件、后端运行执行组件、后端接口与集成组件

### 与三条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **详概分离** | 本步产出 SysReq 层的**详细定义子文档**（`06-*.md`）和 PA 层的**架构定义子文档**（`07-*.md`） |
| **文档结对** | 本步执行结对4（SysReq 详细定义末级 ↔ PA 架构）。一次思考、两个记录——分解 IPO 为需求条目（06 内容）+ 分配条目到组件（07 内容） |
| **基于资产** | PA 节点不是从零创建的；每次执行时读取已有 PA 资产，优先复用 > 改进 > 新增。单节点模式使后处理的节点总能看到前序已确认结果 |

### 单节点处理模式

每次执行只处理**一个**上游 SysReq 架构末级节点（IPO）。这是全链路（wft01~wft05）的统一原则。

**成立的根本原因**——基于资产的设计原则使得序列处理天然可行：

```
执行第1次：读已有资产 → 对 IPO-A 做设计 → 写回资产（状态=待确认方案）
执行第2次：读已有资产（含 IPO-A 结果）→ 对 IPO-B 做设计 → 写回
执行第3次：读已有资产（含 A+B 结果）→ 对 IPO-C 做设计 → 写回
```

每次写回后资产文档即为最新状态，后续执行天然看到前序结果。"复用 > 改进 > 新增"原则在此模式下自然生效。

**无顺序约束**——人类选材时不需要考虑 IPO 处理顺序：

1. **每个节点自描述**：架构末级节点的四要素（I/P/O + 边界）已包含设计所需的全部信息
2. **不依赖其他节点的设计结果**：设计只依赖自身四要素 + 已有资产状态
3. **承接关系是事后验证**：节点间的输入输出一致性由 Phase 1 Step 5 闭环检查保障，不匹配时进入迭代修复

### 两类 IPO 的不对称设计逻辑

这是 wft05 的核心设计认知：

```
业务定义类 IPO → 配置方案组件
  信息来源：配对引擎 IPO 的 I（已详细描述所需配置信息）
  生成方式：信息重组（引擎 IPO 的 I → 7 类子组件结构）
  性质：接近机械化，模式匹配

业务引擎能力类 IPO → 前后端组件
  信息来源：EOS 前后端组件资产 + IPO 四要素 + 系统需求
  生成方式：分解分配分析（创造性设计）
  性质：依赖经验、直觉和设计智慧
```

**为什么业务定义类 IPO 不需要"设计"**：以"工单定义 IPO"为例——其配对的"工单引擎 IPO"的 Input 已详细说明需要接收什么配置信息（字段结构、流转规则、审批权限、展示布局等）。这些信息直接对应配置方案组件的 7 类子组件。

**为什么业务引擎能力类 IPO 必须"设计"**：前后端组件不能从 IPO 四要素直接推出——需要将 IPO 功能分解为细粒度需求条目，对每个条目寻找承接的前后端组件，找不到宿主时新增或改进组件。分解粒度由分配需要决定，分配结果反过来驱动分解调整——这就是经典的"分解分配分析"。

## 三、状态机与路由

### 节点状态定义

| 状态 | 含义 | 所处阶段 |
|------|------|---------|
| **待设计开发** | 上游就绪（`wft03` 已完成 + 人类同意），等待进入 Phase 1 | Phase 1 入口 |
| **待确认方案** | Phase 1 已完成，等待人类审阅 | 人类审阅门禁 1 |
| **迭代中** | 人类反馈修改意见，等待重新处理 | Phase 1 重入 |
| **待资产落账** | 方案已确认，等待进入 Phase 2 上半 | Phase 2 上半入口 |
| **已资产落账** | Phase 2 上半完成，等待进入 Phase 2 下半 | Phase 2 下半入口 |
| **待确认细节** | Phase 2 下半已完成，等待人类审阅详细定义 | 人类审阅门禁 2 |
| **确认中** | 人类反馈修改详细定义意见，等待重新处理 | Phase 2 下半重入 |
| **已验证冻结** | 全链验证通过 | 终态 |
| **已废弃** | 节点停止使用 | 终态 |

### 状态路由逻辑

Step Start 读取目标节点的当前状态，决定进入哪个阶段：

```
Step Start 读取目标节点状态
  │
  ├─ 待设计开发 ──→ Phase 1（首次执行）
  │
  ├─ 迭代中 ──→ Phase 1（重入：加载人类反馈 → 调整方案）
  │
  ├─ 待资产落账 ──→ Phase 2 上半（机械性资产匹配/写回）
  │
  ├─ 已资产落账 ──→ Phase 2 下半（详细定义展开 + 组件需求规格）
  │
  ├─ 确认中 ──→ Phase 2 下半（重入：按反馈调整详细定义）
  │
  └─ 其他（待确认方案/待确认细节/已验证冻结/已废弃）
       → 不可操作，输出状态提示后结束
```

### 人类决策字段

| 决策 | 作用 | 触发阶段 |
|------|------|---------|
| **同意产品架构** | 选入 Phase 1 | 从 `05-*.md` SysReq 状态表选材 |
| **同意方案**（门禁 1） | Phase 1 方案通过 → 状态推进到待资产落账 | `07-*.md` 反馈区 |
| **修改方案**（门禁 1） | 状态→迭代中，人类标注修改意见 | `07-*.md` 反馈区 |
| **同意细节**（门禁 2） | 详细定义通过 → 状态推进到已验证冻结 | `06-*.md` 反馈区 |
| **修改细节**（门禁 2） | 状态→确认中，人类标注修改意见 | `06-*.md` 反馈区 |

#### Step 3 · 设计活动

> 按目标 IPO 类型分流执行。这是结对4（SysReq 详细定义 ↔ PA 架构）的核心设计活动——一次思考同时产出 06 和 07 的**内容**（正式落入资产文档在 Phase 2 下半完成）。

**3a · 业务定义类 IPO → 配置方案组件**

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 定位配对引擎 IPO | AI | 找到与目标业务定义类 IPO 配对的业务引擎能力类 IPO |
| 2 | 提取配置信息需求 | AI | 从配对引擎 IPO 的 Input 字段提取所需配置信息（字段结构、流转规则、审批权限、展示布局等） |
| 3 | 组织组件结构 | AI | 将配置信息组织为主定义组件 + 7 类子组件结构 |
| 4 | 资产决策 | AI | 对每个组件执行：复用已有 > 改进已有 > 新增 |

**3b · 业务引擎能力类 IPO → 前后端组件（分解分配分析）**

> 分解分配分析不是可规程化的算法，而是依赖设计经验和直觉的创造性活动。本步只定义设计框架：

| 能规定的 | 不能规定的 |
|---------|-----------|
| 输入：已有 PA 资产、IPO 四要素、引擎能力全景、EOS 组件参考 | 怎么想出分解方案 |
| 约束：禁止跨类生成、复用 > 改进 > 新增、承接链闭合 | 怎么判断"该新增而非改进" |
| 产出：条目→组件映射、资产决策、承接关系 | 怎么感觉到两个 IPO 该共享组件 |
| 质量标准：每条目唯一宿主、粒度合理、边界清晰 | 怎么确定"粒度合理" |

设计活动的基本动作：

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 将 IPO 功能分解为细分需求条目 | AI | 基于 IPO 的 P（处理逻辑）和 O（输出）分解 |
| 2 | 同步分配到前后端组件 | AI | 每个条目寻找已有 PA 资产中的承接宿主 |
| 3 | 处理无宿主条目 | AI | 新增或改进组件以承接 |
| 4 | 收敛判定 | AI | 每条目有唯一承接时收敛 |

质量兜底靠人类审阅门禁和多轮迭代。

**3c · 非末级宿主生成（如需要）**

| # | 操作 | 执行者 | 说明 |
|---|------|--------|------|
| 1 | 评估分类需求 | AI | 当前末级组件集合是否需要新的非末级分类宿主 |
| 2 | 反推分类结构 | AI | 从已形成的末级组件集合反推稳定的非末级节点（先有末级再有非末级） |

| # | 操作 | 执行者 | 说明 |

## 五、PA 组件生成规则

### 5.1 类型判定

对目标 SysReq 末级 IPO，先判定其对象类型：

| IPO 类型 | 判定标准 | PA 末级组件类型 |
|---------|---------|----------------|
| **业务定义类 IPO** | 目标是产出一个具体业务定义；需求主要落在"定义内容怎么配" | **配置方案组件** |
| **业务引擎能力类 IPO** | 目标是提供或增强一类业务引擎承接能力；需求主要落在"系统能力怎么实现" | **前后端组件** |

### 5.2 业务定义类 IPO → 配置方案组件

处理步骤：

1. 识别其依附的业务引擎；
2. 识别要产出的业务定义对象；
3. 将 IPO 需求展开为配置方案信息；
4. 组织为一个或多个配置方案组件。

与业务引擎能力类 IPO 的关系：

1. 业务定义类 IPO 所生成的配置方案信息，构成后续业务引擎能力类 IPO 的核心输入；
2. 因此，本类组件不是孤立产物，而是"供业务引擎消费的定义信息载体"；
3. 承接关系的一致性由闭环检查事后验证，不构成设计时的顺序约束。

建议优先采用"**1 个主定义组件 + 7 类子组件**"的结构：

| 组件层 | 组件名称 | 职责 |
|------|---------|------|
| 主组件 | **主定义组件** | 作为该业务定义的核心承载单元 |
| 子组件1 | **元数据组件** | 管理编号、名称、版本、状态、适用范围、归属域 |
| 子组件2 | **结构组件** | 管理字段结构、流程节点结构、指标维度结构等静态骨架 |
| 子组件3 | **规则组件** | 管理业务规则、流转规则、计算规则、校验规则 |
| 子组件4 | **权限组件** | 管理查看、配置、发起、审批、维护等权限范围 |
| 子组件5 | **展示组件** | 管理页面布局、列表视图、字段展示、菜单挂接、场景挂接 |
| 子组件6 | **引用集成组件** | 管理依赖的其他业务定义、引擎能力、外部数据源 |
| 子组件7 | **校验发布组件** | 管理完整性检查、冲突检查、发布条件、生效条件 |

### 5.3 业务引擎能力类 IPO → 前后端组件

处理步骤：

1. 识别新增或增强的是哪类引擎能力；
2. 识别其核心输入，尤其是由业务定义类 IPO 提供的配置方案信息；
3. 判断该能力落在前端、后端，还是前后端共同承接；
4. 按技术职责拆成 PA 末级组件。

建议优先采用"**1 个能力主组件 + 5 类承接组件**"的结构：

| 组件层 | 组件名称 | 职责 |
|------|---------|------|
| 主组件 | **能力主组件** | 作为该业务引擎 IPO 的核心承接单元 |
| 子组件1 | **前端配置组件** | 承接配置页面、设计器、配置器、参数编辑器、规则编辑器 |
| 子组件2 | **前端运行组件** | 承接运行期输入页面、操作页面、结果页面、查看页面 |
| 子组件3 | **后端定义处理组件** | 承接配置信息解析、校验、组装、版本化、业务定义生成 |
| 子组件4 | **后端运行执行组件** | 承接业务定义解释、调度、执行、状态流转、结果产出 |
| 子组件5 | **后端接口与集成组件** | 承接前后端 API、外部集成、消息、存储访问、事件通知 |

### 5.4 禁止跨类生成

1. 业务定义类 IPO 不直接落前后端组件；
2. 业务引擎能力类 IPO 不直接落配置方案组件；
3. 若某业务定义类 IPO 在上游仍存在引擎缺口，应按 `wft02` 的引擎缺口机制先回到 A1 路径处理，而不是在本步混写 PA 类型。

### 5.5 PA 末级节点内容要求

PA 末级节点下的内容，不应只停留在"节点命名"和"一句定义"，而必须达到可直接支撑后续开发的粒度。

#### 5.5.1 配置方案组件末级节点的内容要求

至少应包含：

1. **组件定义**：说明该组件是什么、承接哪个业务定义类 IPO；
2. **承接的上游条目**：明确承接哪些 SysReq 详细定义条目；
3. **配置方案结构**：列出元数据、结构、规则、权限、展示、引用、校验发布等配置块；
4. **输入输出**：输入什么配置数据，输出什么业务定义结果或可加载定义；
5. **依附引擎**：明确依附哪个业务引擎运行；
6. **约束与校验**：明确完整性校验、边界约束、版本/发布条件；
7. **符合性分析**：说明为何足以承接上游需求；
8. **开发/配置指导**：给配置方案开发者可直接执行的指导。

一句话要求：配置方案组件末级节点应达到"**可开发的定义宿主规格**"粒度。

#### 5.5.2 前后端组件末级节点的内容要求

至少应包含：

1. **组件定义**：明确这是前端配置组件、前端运行组件、后端定义处理组件、后端运行执行组件，还是接口与集成组件；
2. **承接的上游条目**：明确承接哪些 SysReq 详细定义条目；
3. **职责边界**：说明负责什么、不负责什么；
4. **输入**：尤其说明是否接收配置方案信息、运行上下文、外部事件或接口输入；
5. **输出**：说明输出页面能力、定义生成结果、执行结果、接口响应、事件等；
6. **内部处理逻辑**：说明解析、校验、组装、执行、调度、渲染、交互等关键逻辑；
7. **对外接口**：说明 API、消息、回调、存储关系、依赖服务；
8. **NFR 约束**：说明性能、安全、可用性、并发、审计等要求；
9. **开发指导**：给前端/后端开发者可直接展开的指导。

一句话要求：前后端组件末级节点应达到"**开发可执行定义**"粒度。

### 5.6 PA 非末级节点生成规则

PA 的非末级节点不是预先想好一棵树再硬塞末级节点，而是从已形成的末级组件集合中反推生成的稳定宿主结构。

**总原则**：`先有末级组件集合，再生成非末级分类宿主。`

**生成规则**：

1. **先按组件对象类型分大类**：如配置方案组件类、前端组件类、后端组件类；
2. **再按职责稳定性分中类**：如业务定义配置组件、引擎配置前端组件、引擎运行前端组件、引擎定义处理后端组件、引擎运行执行后端组件、接口与集成后端组件；
3. **仅在边界稳定时生成非末级节点**：只有当一组末级节点已经具备稳定命名、稳定边界、稳定复用意义时，才抽象出非末级宿主；
4. **非末级节点同样遵循复用 > 改进 > 新增**：优先复用现有宿主，不够再改，最后才新增。

## 六、AI 执行指南

### 6.1 跨步骤上下文管理

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | IPO 类型、组件边界、资产决策 | 07-*.md 方案区 |

| 反馈记录 | 人类标注的修改意见 | 07-*.md / 06-*.md 反馈区 |

### 6.2 AI 经济原则

1. **单节点处理**：每次只处理一个 SysReq 末级 IPO，不跨节点批量操作
2. **状态机驱动**：严格按目标节点当前状态路由到对应 Phase，不跨 Phase 操作
3. **先判 IPO 类型，再设计**：进入 Phase 1 Step 3 前必须明确 IPO 类型
4. **基于资产优先**：配置方案组件优先复用已有配置宿主；前后端组件优先复用已有引擎构件
5. **配置方案信息是业务引擎核心输入**：这是承接关系的核心规则，但验证在闭环检查中事后执行
6. **Phase 2 上半不重做设计**：只做匹配/写回/格式化，不改变 Phase 1 的设计决策

### 6.3 分解分配分析的设计框架定位

Phase 1 Step 3b 的分解分配分析是**设计框架声明**，不是可执行算法：

- AI 基于直觉给出方案建议（分解方案 + 分配方案 + 资产决策）
- 人类根据设计直觉审核和反馈
- 通过门禁 1 的多轮迭代收敛到满意方案
- Skill 文档不规定"怎么想"，只规定输入条件、约束边界、产出格式和质量标准
