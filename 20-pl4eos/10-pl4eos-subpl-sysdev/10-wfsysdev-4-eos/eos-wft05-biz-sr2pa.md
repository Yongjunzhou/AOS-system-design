# SR-F → PA · 业务侧引擎配置单元需求方案设计
**Skill ID**: wft05-biz-sr2pa

> **概述**：每次执行处理 ≥1 个 SR-F（功能性系统需求）业务功能需求节点（来自 wft03-biz，状态=`待产品架构` / `待补充PA设计`），PA（产品架构）反馈优先于新 SR-F 处理。wft03-biz 已按支撑引擎的 CU（配置单元）能力结构将 SR-F 分解为末级条目；本 Skill 按 `支撑引擎 + 主 CU 能力类型 + 业务配置对象` 汇聚、去重并规范化这些条目，形成既有 EOS 引擎配置单元的配置需求规格。引擎能力缺口不进入本路径；若无法在已有引擎及配置单元中找到承接对象，本 Skill 只形成配置单元映射失败说明和阶段性结论，不生成缺口材料、不触发 eng（引擎）链。完成跨域匹配、资产写回、SR 详细定义展开、PA 配置单元需求规格写入与状态推进。
>
> **输入 → 输出**：已按 CU 能力结构分解的 SR-F 业务功能需求节点（≥1 个，`待产品架构` / `待补充PA设计`）/ 有反馈的 PA 方案节点 → PA 引擎配置单元需求方案或映射失败说明 + 跨域匹配写回 + SR 详细定义（`06-*.md`）+ PA 配置单元需求规格（`07-*.md`）；仅当形成可确认 PA 配置单元需求时，SR-F 推进为 `已产品架构`
>
> **适用场景**：瀑布式系统设计第5步 · 业务配置路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v3.4 | **修订**：2026-06-29

## 一、职责定位

### 本 Skill 目的与目标

> **主要目的**：证明 SR-F 可以被已有 EOS 引擎配置体系承接。
>
> **目标**：把 wft03-biz 已分解的 SR-F 末级条目，按支撑引擎、主 CU 能力类型和业务配置对象汇聚为确定的引擎配置单元需求，并形成可交给业务配置/实施人员使用的配置需求规格。
>
> **一句话**：biz 链到 05 不是"开发组件"，而是"定义要配置哪些引擎配置单元"；eng 链才是"开发这些引擎和配置单元"。

### 三步边界

| Skill | 回答的问题 |
|-------|-----------|
| wft02-biz | 业务架构对不对 |
| wft03-biz | 系统功能是否完整承接业务 |
| wft05-biz | 业务功能能否被引擎配置体系承接——分解后的 SR-F 如何汇聚、去重并规范化为哪个引擎配置单元的配置需求 |

### 前提与边界

**前提**：wft05-biz 默认目标引擎及其配置单元已经存在，或者已由 eng 路径另行开发。

**biz 线与 eng 线的分工**：

| | biz 线（本 Skill） | eng 线（wft05-eng） |
|--|-------------------|-------------------|
| 做什么 | 定义要配置哪些引擎配置单元 | 开发这些引擎和配置单元 |
| 产出 | 引擎配置单元需求方案 | 引擎前后端组件实现 |
| 对引擎的态度 | 消费者：配置需求方 | 生产者：开发供应方 |

本 Skill 负责：

1. **接收**——接收人类选中的 ≥1 个 SR-F 业务功能需求节点（`待产品架构` / `待补充PA设计`），校验状态/类型/数量，不合格则说明原因并退出
2. **汇聚**——按 `支撑引擎 + 主 CU 能力类型 + 业务配置对象` 汇聚 SR-F 末级条目，形成配置需求候选组
3. **规范化**——读取 25（平台引擎资产）中既有引擎及配置单元定义，将候选组规范化为确定的配置单元需求规格
4. **判定**——引擎或配置单元不存在/能力不足时，形成映射失败说明和阶段性结论，不试图在本路径开发或处理缺口
5. **落账**——执行 Q1/Q2/Q3（Q1=直接引用 / Q2=扩展 / Q3=新建）跨域匹配、正式写回 07（产品架构文档）资产、更新反向引用（05/23/24）
6. **展开**——写入 SR 详细定义（`06-*.md`（系统需求详细定义文档））和 PA 配置单元需求规格（`07-*.md`（产品架构文档）），达到业务配置/实施人员可直接使用的粒度
7. **推进**——人类确认且 PA 方案为 `可确认方案` 后，SR-F 节点推进为 `已产品架构`；阶段性方案不得放行

本 Skill 不负责：

1. 引擎本体的开发与前后端组件设计（→ `wft05-eng-sr2pa`）
2. 判断或开发引擎是否存在——本 Skill 只读取已存在的 25 引擎模型用于映射，不负责追问、补建或登记缺失引擎
3. 非功能需求的量化与分类（由 wft04 承担，本 Skill 只消费 NFR（非功能需求）约束包）
4. 引擎能力缺口的所有处理——引擎能力缺口不进入 wft03-biz / wft05-biz

### 定位说明

wft05-biz 形成的 PA 组件，不是开发引擎本体的前后端组件，而是**业务配置侧需要使用的配置单元需求**。PA 组件在 biz 线上 = 对应引擎中的配置单元需求 / 配置方案组件。本 Skill 不设计引擎本体，不拆前后端实现组件，只把业务侧 SR-F 映射为对既有引擎配置单元的配置需求。

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（SR-F 业务功能需求节点 ≥1 个 / 有反馈的 PA 方案节点）
2. 上游追溯链（`05-*.md`（系统需求架构文档）SR-F 业务功能需求节点正文块 / 四要素 / SR-F 分类类型 / BA 业务对象 / 支撑引擎 / 主 CU 能力类型 / 辅助能力类型 / 分解依据 / 变更影响声明）
3. 已有 PA 架构资产（`07-*.md`（产品架构文档）树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`23`（产品架构资产） + `24`（场景组件资产） + `25`（平台引擎资产，读取已有引擎及配置单元定义） + `26`（业务文档资产） + `27`（资源工具资产） + `22`（非功能需求分类资产））
5. NFR 约束包（从 wft04 分层约束表 / wft05-biz 装配视图装配：全局 + 所属业务域 + SR-F + 目标配置单元；仅消费 `适用链路=biz/共享` 且 `wft05 消费目标=wft05-biz:配置单元需求` / `wft05-biz+eng` 的约束）
6. 重入反馈
7. 上次 PA 映射状态（重入时读取，区分 `待映射` / `已映射` / `映射失败` / `待重试` 条目——仅处理 `映射失败` 和 `待重试`，跳过 `已映射`）

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 SR-F 业务功能需求节点选中 + 无待处理反馈的 PA 方案节点 | 输出"无待处理对象"→ **退出** |
| SR-F 节点状态不是 `待产品架构` / `待补充PA设计` | 输出状态分布，列出不合格节点 → **退出** |
| 选中 SR-F 节点含引擎能力类 SR-F（来自 wft03-eng） | 输出类型分布（引擎能力类 → `wft05-eng`），要求分路径 → **退出** |
| 选中 SR-F 节点分属多个业务域且无已确认关联 | 输出业务域分布，提示可能需分批 → 人类确认后继续或分批 |
| 选中 SR-F 节点 >5 个 | 输出"本轮 N 个节点，建议分批" → 人类确认后继续，不强制退出 |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 条目 `确认状态` 字段中有人类标注且无 `[已处理]` 的 PA 方案节点 | 本轮先处理全部有反馈的 PA 方案节点 |
| 存在 `待确认方案` + 所有条目均已 `[同意]` + `[已处理]` 的 PA 方案节点 | 不阻塞新 SR-F；Step End 时提示 |

#### 第三层：新 SR-F 业务功能需求节点路由

PA 反馈处理完后，逐节点处理新设计 → Step 3。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 PA 配置需求方案，因此 Step 3 必须基于 Step 2 修改后的 PA 当前内容，不得使用 Step 1 加载时的缓存旧状态。

### SR-F 节点生命周期

#### 推进态

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `待产品架构` | SR-F 已确认，wft05 可启动 PA 设计 | wft03-biz 完成确认 |
| `在PA设计` | 本 Skill 正在设计 PA | 本 Skill 写入 |
| `已产品架构` | PA 设计完成，消费版本与当前一致 | 本 Skill 处理完成且未检测到上游新版本 |

#### 回退态（上游修订触发）

当 wft03-biz 修订了 SR-F（版本递增），按本 Skill 的消费进度分流：

| 修订前状态 | 修订后 | 说明 |
|-----------|--------|------|
| `待产品架构` | 保持，版本递增 | 本 Skill 尚未消费，不受影响 |
| `在PA设计` | 保持，版本递增 | 本 Skill 正在处理中，重入时检测版本差异，按变更影响声明增量处理 |
| `已产品架构` | 降为 **`待补充PA设计`** | PA 基于旧版本 SR-F，需补全新增部分 |

SR-F 节点状态分工：wft03-biz 写入 `待产品架构` / `待补充PA设计`，并在修订时更新版本号；本 Skill 写入 `在PA设计` / `已产品架构`。

#### 版本标记与消费追溯

- 本 Skill 首次消费 SR-F 时，在对应 PA 方案节点中记录 `SR-F 消费版本：v{N}`
- 本 Skill 重入同一 SR-F 时比对版本：若 SR-F 当前版本 > 已消费版本 → 触发变更感知

#### PA 映射状态

本 Skill 在每个 SR-F 末级条目上标注 `PA映射状态`，用于追踪条目是否已被成功映射为配置单元需求。当本 Skill 形成阶段性方案重入时，只处理映射失败的条目，已成功映射的条目不重复处理。

| 映射状态 | 含义 | 触发条件 |
|---------|------|---------|
| `待映射` | 尚未处理该条目 | 首次进入或新增 SR-F 末级条目 |
| `已映射` | 已成功找到目标引擎 CU 并形成配置需求 | Step 3 汇聚并规范化成功 |
| `映射失败` | 引擎不存在 / CU 不存在 / CU 能力不足，本步无法映射 | Step 3 判定无承接对象 + 映射失败说明已写入 |
| `待重试` | 上次 `映射失败`，下游 eng 路径产出后在重入中重试 | 本 Skill 重入 + 检测到 25 中有新 CU 定义 |

`已映射` 条目在 Step 5 正式写回时随方案整体写入 07 资产。`映射失败` 条目不进入 Step 5 写回——PA 方案标记为阶段性方案，待重入重试。`待重试` 条目仅在重入路径中生效（Step 1 加载时检测 25 是否有新增 CU 定义）。

### Step 2 · 方案反馈处理

#### 人类审阅规则

人类对 PA 方案做两层审阅——条目级为权威留痕，整体确认为快捷补全。人类直接在条目 `确认状态` 字段标注意见，AI 处理完后追加 `[已处理]`。

1. **条目级反馈**（权威）：对 PA 方案的独立决策单元（各引擎配置单元需求、引擎映射关系）逐条标注。已 `[同意]` + `[已处理]` 的条目在后续轮次中默认跳过——人类若改变主意，直接写入新的 `[修改]` / `[驳回]` 即覆盖旧标注，覆盖后 `[已处理]` 自然消失，条目回到待处理状态
2. **整体确认**（快捷）：人类在对话中说「整体确认」→ AI 对**无条目级标注的条目**写入 `[同意]`（标注来源=`整体确认`），随后对这批条目追加 `[已处理]`——与条目级 `[同意]` 等效

**冲突消解**：条目级 > 整体确认。两者并存时，条目级标注保留原结论不受影响，整体确认仅补未标注的剩余条目。

#### AI处理规则

若本轮存在 `待确认方案` 的 PA 方案节点，AI 扫描各条目的 `确认状态` 字段——有人类标注但无 `[已处理]` 的条目为本轮待处理对象：

1. **扫描待处理条目**——遍历 PA 方案各条目，筛选 `确认状态` 中有人类标记（`[同意]` / `[修改]` / `[驳回]`）但无 `[已处理]` 的条目
2. **收集整体确认**——若人类说了「整体确认」，对**无条目级标注的剩余条目**写入 `[同意]`（标注来源=`整体确认`），随后为该批条目追加 `[已处理]`
3. **执行**——逐条目按人类意见处理：

| 人类意见 | AI处理 |
|---------|------|
| `[同意]`（明示或整体确认补全） | 条目维持不变 → 在 `确认状态` 字段末尾追加 `[已处理]` |
| `[修改]`：修改内容 | 按反馈调整该条目 → 仅重对齐受影响部分 → 追加 `[已处理]` |
| `[驳回]`：驳回理由 | 移除被驳回的配置单元需求 → 其余对象保持原结论 → 追加 `[已处理]`。若人类标注"整体驳回"，重做 Step 3 → 追加 `[已处理]` |

4. **判定 PA 方案推进**（全部条目 `[同意]` + `[已处理]` 且方案为 `可确认方案`）：
   - 满足 → SR-F → `已产品架构`，PA → `已验证冻结`
   - 不满足（任一条目含 `[修改]`/`[驳回]` 或方案为 `阶段性方案`）→ PA 保持 `待确认方案` 或转为 `待补充PA设计`
5. 写入 `### 反馈处理记录`（含本轮各条目人类意见+来源+处理动作+时间戳）

**多轮迭代**：`[同意]` + `[已处理]` 双标记完整的条目本轮默认跳过（已确认且已执行）。但上游 SR-F 修订可能改变已有 PA 对象——AI 在处理修订时先清除受影响条目的 `[已处理]` 标记（保留 `[同意]`），使条目回到待处理状态，供人类重新审阅。上游变更优先于人类确认。

#### 确认状态字段

各条目的 `确认状态` 字段位于 `07-*.md` PA 方案节点各条目正文中，不再设独立的反馈区节。人类在字段中直接标注，AI 处理后追加 `[已处理]`。格式同 [wft02-biz Step 2](eos-wft02-biz-str2ba.md)。

### Step 3 · 配置单元需求汇聚与规范化

wft03-biz 已按支撑引擎的 CU 能力结构把业务功能分解为 SR-F 末级条目。本步不再从粗粒度 SR-F 中猜测 CU，而是把这些末级条目按 `支撑引擎 + 主 CU 能力类型 + 业务配置对象` 汇聚为配置需求候选组，再读取 25 中既有引擎及配置单元定义，规范化为 PA 配置单元需求规格。

> **设计姿态**：wft05-biz 默认目标引擎及其配置单元已经存在，或者已由 eng 路径另行开发。本步不设计引擎本体，不拆前后端实现组件，只把业务侧 SR-F 汇聚并规范化为对既有引擎配置单元的配置需求。引擎或配置单元不存在/能力不足时，形成映射失败说明和阶段性结论，不试图在本路径开发，也不把失败原因登记为引擎能力缺口。

#### 到位粒度

必须达到**引擎级 + 主 CU 能力类型级 + 配置单元需求规格级**：

1. 已按支撑引擎、主 CU 能力类型和业务配置对象完成 SR-F 汇聚
2. 已定位目标引擎中承接该主 CU 能力类型的配置单元定义
3. 已给出每个配置单元需求的核心配置内容、规则、接口和验收口径
4. 已完成上游 SR-F 末级条目到配置单元需求的分配
5. 若存在映射失败，已给出失败说明并标记为阶段性方案

不在本步展开：开发实现方案、前后端组件设计、引擎内部机制设计。

#### 汇聚键

| 汇聚维度 | 来源 | 用途 |
|----------|------|------|
| 支撑引擎 | wft03-biz SR-F 末级条目 | 定位 25 中的目标引擎 |
| 主 CU 能力类型 | wft03-biz SR-F 末级条目 | 定位候选配置单元类型 |
| 业务配置对象 | 来源 BA 对象 / 主业务对象 / 输出文档 / 数据对象 | 判断哪些 SR-F 应合并为同一配置实例需求 |
| 辅助能力类型 | wft03-biz SR-F 末级条目 | 形成依赖关系，不改变主配置单元宿主 |
| SR-F 分类桶 | wft03-biz 分类结果 | 辅助跨域复用、归并和去重 |

#### 汇聚与规范化逻辑

1. 按汇聚键把 SR-F 末级条目聚成候选组。
2. 从 25 引擎模型中定位目标引擎和与主 CU 能力类型匹配的配置单元定义。
3. 将候选组中的功能描述、流程、规则、接口和验收口径整理为配置单元需求规格。
4. 对辅助能力类型建立依赖关系，如表单 CU 依赖实体、权限、流程或指标能力。
5. 装配 NFR 约束包：将 `配置项` / `配置规则` / `配置可达性` / `验收标准` 类影响点写入配置单元需求规格；`阻断型` 约束必须影响配置项、规则或验收口径，不能只作为备注。
6. 对每个配置单元需求执行资产决策：**复用已有配置需求 > 改进已有配置需求 > 新增配置需求**。
7. 新增配置需求须与已有 CU 的配置模式一致（配置项结构 / 粒度 / 命名），但不展开开发实现。

#### NFR 约束装配规则

| wft04 影响点 | 写入位置 | 处理规则 |
|-------------|---------|---------|
| 配置项 | 配置项清单 | 转化为必须配置/可选配置/禁止配置项；阻断型缺失时方案不得进入可确认 |
| 配置规则 | 配置规则 | 转化为校验规则、权限规则、脱敏规则、发布规则等配置要求 |
| 配置可达性 | 配置路径 / 配置入口 | 要求配置人员能通过明确页面、角色和流程完成配置 |
| 验收标准 | 配置单元需求验收标准 | 写入可验证指标、检查方法和通过条件 |
| 组件类型/组件边界/接口机制/运行约束机制/共享归属 | 不在 biz 侧展开 | 标注为 `eng 约束`，仅保留引用；具体承接由 wft05-eng 处理 |

同一 NFR 同时进入 biz 与 eng 时，wft05-biz 只消费配置单元需求相关部分，并保留 NFR 指标 ID，供 wft05-eng 检查口径一致。

#### SR-F 类型 → 支撑引擎兜底映射表

当 SR-F 缺少支撑引擎或主 CU 能力类型时，才使用下表作为兜底推断，并将方案标记为 `阶段性方案`，要求回到 wft03-biz 补充分解依据。

| SR-F 类型 | 兜底目标引擎 | 说明 |
|-----------|--------------|------|
| 菜单 / 入口 / 可见性 | 菜单引擎 | 菜单导航、场景入口、角色可见性控制 |
| L4 场景业务 | 场景引擎 | 场景编排、上下文传递、状态衔接、工作台 |
| L5 表单部件 | 表单引擎 | 表单设计、字段定义、校验规则 |
| L5 流程部件 | 流程引擎 | 流程定义、流转规则、审批配置 |
| L5 工单部件 | 工单引擎 | 工单定义、分配规则、状态流转 |
| L6 功能元件 / 实体对象 | 实体引擎 | 实体定义、字段结构、数据模型、版本状态、引用关系 |
| 指标 / 看板 / 聚合 | 指标引擎 | 指标定义、计算规则、展示配置、异常下钻 |
| 资产目录 / 数据目录 / 血缘 / 质量 | 治理类引擎或通用配置单元 | 元数据管理、数据地图、血缘追踪、质量规则、分类分级、脱敏 |

#### Q1/Q2/Q3 PA 配置需求匹配

汇聚后，按 `支撑引擎 + 主 CU 能力类型 + 业务配置对象` 匹配已有 PA 配置需求节点：

| 判定 | 条件 | 动作 |
|------|------|------|
| **Q1 直接匹配** | 已有 PA 配置需求对应同一支撑引擎、同一主 CU 能力类型、同一业务配置对象且需求一致 | 复用已有 PA → 追加 SR-F 引用 → 不重新生成 |
| **Q2 扩展匹配** | 已有 PA 配置需求对应同一支撑引擎、同一主 CU 能力类型、同一业务配置对象但需扩展（新增配置项/规则） | 扩展已有 PA → 全量重写配置需求（非增量追加） |
| **Q3 新建** | 无可用已有 PA 配置需求 | 标记待新建 → 按映射表生成配置需求 |

#### 配置单元映射失败说明

当 SR-F 无法在 25 已有引擎模型中定位到可承接的配置单元，或已有 CU 无法承接该 SR-F 时，本 Skill 只记录映射失败说明：

| 说明项 | 内容 |
|--------|------|
| 失败类型 | `无目标引擎` / `无可用 CU` / `CU 承接不足` |
| 失败说明 | 哪条 SR-F 无法映射到已有引擎配置单元，缺少哪类承接对象 |
| 本步结论 | `阶段性方案`，不得推进为可确认 PA 配置单元需求 |
| 后续归属 | 不在本 Skill 处理；如确属引擎能力缺口，应由 wft02-biz 的原始需求材料链路独立处理 |

映射失败说明写入 PA 方案的"配置单元映射失败说明"小节。存在映射失败时，PA 方案只能标记为 `阶段性方案`，不得进入可确认方案或 `已产品架构` 状态；Step End 必须提示该 SR-F 当前无法由已有引擎配置体系承接。

### Step 4 · 闭环检查

| 检查项 | 内容 |
|--------|------|
| **承接完整** | 每个 SR-F 汇聚候选组的全部功能需求已有引擎配置单元需求承接 |
| **引擎覆盖** | 支撑引擎在 25 中有定义，且可按主 CU 能力类型定位到配置单元定义 |
| **1:1 承接** | 每条 SR-F 末级条目有唯一配置单元需求宿主 |
| **配置可达** | 配置需求可交给业务配置/实施人员直接使用——配置路径清晰、配置项可操作 |
| **NFR 完整性** | 相关 `biz/共享` NFR 约束已映射到配置单元需求，且阻断型配置约束已进入配置项、规则或验收标准 |
| **NFR 边界正确** | `eng` 组件类约束未在 biz 侧展开为配置需求，仅保留引用并提示由 wft05-eng 承接 |
| **分类一致性** | 同一分类桶、同一主 CU 能力类型的 SR-F 被归并到一致的配置单元需求 |
| **映射失败处理** | 所有无法映射的 SR-F 已形成失败说明，且方案状态保持 `阶段性方案` |

### Step 5 · 资产落账与写回

人类已在 Step 2 中确认方案（全部条目 `[同意]` + `[已处理]`）后，Step 5 统一执行落账与写回。若方案仍为 `阶段性方案`，不得执行正式落账，只写入阶段性说明并等待上游或 eng 路径补齐后重入。

#### 跨域匹配（Q1/Q2/Q3）

| 匹配 | 内容 |
|------|------|
| **Q1: 域内复用** | 同一业务域内是否已有可复用配置需求 |
| **Q2: 跨域归并** | 其他业务域是否有同引擎同 CU 的配置需求需归并 |
| **Q3: 共享决策** | 是否需要抽取为跨域共享配置需求 |

**权限边界**：Q1/Q2/Q3 只做落账级匹配（写到哪里、补什么引用），不做结构级改判（哪个引擎承接、怎么配 CU）。若匹配结果意味着主结构需要重定，必须回退 Step 3。

#### 写回操作

1. **写入 07**：PA 配置单元需求方案 + 跨域匹配结果 + 配置单元需求规格（引擎映射、配置项清单、配置规则、验收标准）；若存在映射失败，仅写入阶段性方案和映射失败说明
2. **写入 06**：SR 详细定义条目（将 SR-F 功能需求格式化为详细条目）
3. **更新 05**：反向引用更新（可确认方案：SR-F 节点 → `已产品架构`；阶段性方案：SR-F 节点保持 `待补充PA设计` 或原状态）
4. **更新 23**：PA 配置需求索引更新
5. **更新 24**：业务定义引用更新（PA 配置单元需求承接关系写入业务定义引用块）

#### 状态推进

- 可确认方案：`05` SR-F 节点推进为 `已产品架构`，`07` PA 方案推进为 `已验证冻结`
- 阶段性方案：不得推进为 `已产品架构` / `已验证冻结`，只保留阶段性说明和重入条件

写回要求：幂等去重、局部写回、全貌同步。
