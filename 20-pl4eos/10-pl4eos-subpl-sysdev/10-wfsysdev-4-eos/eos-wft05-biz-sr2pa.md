# SR-F → PA · 业务侧引擎配置单元需求方案设计
**Skill ID**: wft05-biz-sr2pa

> **概述**：每次执行处理 ≥1 个 SR-F 业务功能需求节点（来自 wft03-biz，状态=`待产品架构` / `待补充PA设计`），PA 反馈优先于新 SR-F 处理。将业务侧 SR-F 映射为对既有 EOS 引擎配置单元的配置需求——按 SR-F 类型对应目标引擎，形成引擎配置单元需求方案。引擎能力缺口不进入本路径，走 raw material → ort00~03 → wft01-eng。完成跨域匹配、资产写回、SR 详细定义展开、PA 配置单元需求规格写入与状态推进。
>
> **输入 → 输出**：SR-F 业务功能需求节点（≥1 个，`待产品架构` / `待补充PA设计`）/ 有反馈的 PA 方案节点 → PA 引擎配置单元需求方案 + 引擎能力缺口标注 + 跨域匹配写回 + SR 详细定义（`06-*.md`）+ PA 配置单元需求规格（`07-*.md`）+ SR-F 推进为 `已产品架构`
>
> **适用场景**：瀑布式系统设计第5步 · 业务配置路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v3.0 | **修订**：2026-06-28

## 一、职责定位

### 本 Skill 目的与目标

> **主要目的**：证明 SR-F 可以被已有 EOS 引擎配置体系承接。
>
> **目标**：把每条 SR-F 分配到确定的引擎配置单元，并形成可交给业务配置/实施人员使用的配置需求。
>
> **一句话**：biz 链到 05 不是"开发组件"，而是"定义要配置哪些引擎配置单元"；eng 链才是"开发这些引擎和配置单元"。

### 三步边界

| Skill | 回答的问题 |
|-------|-----------|
| wft02-biz | 业务架构对不对 |
| wft03-biz | 系统功能是否完整承接业务 |
| wft05-biz | 业务功能能否被引擎配置体系承接——每条 SR-F 由哪个引擎的哪个配置单元承接 |

### 前提与边界

**前提**：wft05-biz 默认目标引擎及其配置单元已经存在，或者已由 eng 路径另行开发。

**biz 线与 eng 线的分工**：

| | biz 线（本 Skill） | eng 线（wft05-eng） |
|--|-------------------|-------------------|
| 做什么 | 定义要配置哪些引擎配置单元 | 开发这些引擎和配置单元 |
| 产出 | 引擎配置单元需求方案 | 引擎前后端组件实现 |
| 对引擎的态度 | 消费者：配置需求方 | 生产者：开发供应方 |

本 Skill 负责：

1. **接收**——接收人类选中的 ≥1 个 SR-F 业务功能需求节点（`待产品架构` / `待补充PA设计`），校验状态/类型/数量，不合格则说明原因并退出
2. **映射**——将每条 SR-F 映射到确定的目标引擎及其配置单元，形成配置需求方案
3. **标注**——引擎或配置单元不存在/能力不足时，标注为缺口，不试图在本路径开发
4. **落账**——执行 Q1/Q2/Q3 跨域匹配、正式写回 07 资产、更新反向引用（05/23/24）
5. **展开**——写入 SR 详细定义（`06-*.md`）和 PA 配置单元需求规格（`07-*.md`），达到业务配置/实施人员可直接使用的粒度
6. **推进**——人类确认后 SR-F 节点推进为 `已产品架构`

本 Skill 不负责：

1. 引擎本体的开发与前后端组件设计（→ `wft05-eng-sr2pa`）
2. 判断或开发引擎是否存在——如果不存在，那是 eng 链的工作（raw material → ort00~03 → wft01-eng → ...）
3. 非功能需求的量化与分类（由 wft04 承担，本 Skill 只消费 NFR 约束包）
4. 引擎能力缺口的所有处理——引擎能力缺口不进入 wft03-biz / wft05-biz

### 定位说明

wft05-biz 形成的 PA 组件，不是开发引擎本体的前后端组件，而是**业务配置侧需要使用的配置单元需求**。PA 组件在 biz 线上 = 对应引擎中的配置单元需求 / 配置方案组件。本 Skill 不设计引擎本体，不拆前后端实现组件，只把业务侧 SR-F 映射为对既有引擎配置单元的配置需求。

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（SR-F 业务功能需求节点 ≥1 个 / 有反馈的 PA 方案节点）
2. 上游追溯链（`05-*.md` SR-F 业务功能需求节点正文块 / 四要素 / SR-F 分类类型 / BA 业务对象 / 变更影响声明）

6. 引擎能力缺口线索（SR-F 功能接口指向的引擎在 25 中无对应 CU 定义时，标注缺口）
7. 重入反馈

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 SR-F 业务功能需求节点选中 + 无待处理反馈的 PA 方案节点 | 输出"无待处理对象"→ **退出** |
| SR-F 节点状态不是 `待产品架构` / `待补充PA设计` | 输出状态分布，列出不合格节点 → **退出** |
| 选中 SR-F 节点含引擎能力类 SR-F（来自 wft03-eng） | 输出类型分布（引擎能力类 → `wft05-eng`），要求分路径 → **退出** |
| 选中 SR-F 节点分属多个业务域且无已确认关联 | 输出业务域分布，提示可能需分批 → 人类确认后继续或分批 |
| 选中 SR-F 节点 >5 个 | 输出"本轮 N 个节点，建议分批" → 人类确认后继续，不强制退出 |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 PA 方案节点 | 本轮先处理全部有反馈的 PA 方案节点 |
| 存在 `待确认方案` + 反馈区为空的 PA 方案节点 | 不阻塞新 SR-F；Step End 时提示 |

#### 第三层：新 SR-F 业务功能需求节点路由

PA 反馈处理完后，逐节点处理新设计 → Step 3。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 PA 配置需求方案，因此 Step 3 必须基于 Step 2 修改后的 PA 当前内容，不得使用 Step 1 加载时的缓存旧状态。

### Step 2 · 方案反馈处理

若本轮存在有反馈的 PA 方案节点（`待确认方案` + 反馈区有效），逐个处理。

#### 基本操作

| 操作 | 行为 |
|------|------|
| `[同意]` 或整体确认 | 归档反馈 → SR-F → `已产品架构`；PA → `已验证冻结` |
| `[修改]`（局部） | 按反馈调整配置单元需求/引擎映射 → 更新增量清单 → 回写 `待确认方案` |
| `[驳回]`（局部） | 移除被驳回的配置单元需求 → 其他需求 → `已确认` → 回写 `待确认方案` |
| `[驳回]`（整体） | 重做 Step 3 → 回写 `待确认方案` |

#### 分层反馈

PA 方案包含多个可独立决策的单元（各引擎配置单元需求、引擎映射关系）。人类可以**局部修改**——整体确认一个 PA 方案的同时，要求调整其中某个配置单元的映射或需求内容。

- **局部 `[修改]`**：按反馈调整被指定的对象，其涉及的承接关系重新检查。未提及的对象保持原结论不动。
- **PA 方案推进条件**：全部反馈处理完成 → SR-F 推进为 `已产品架构`，PA 推进为 `已验证冻结`。存在未处理项 → PA 保持 `待确认方案`。
- **`[驳回]`（整体）**：PA 主结构被推翻（"引擎映射根本不对"或"配置单元分配错误"）→ 重做 Step 3。

#### 反馈区位置与格式

反馈区位于 `07-*.md` 中每个 PA 方案节点的末尾。格式同 wft03-biz 模式。

### Step 3 · 引擎配置单元需求方案设计

将每条 SR-F 映射到确定的目标引擎及其配置单元，形成配置需求方案。

> **设计姿态**：wft05-biz 默认目标引擎及其配置单元已经存在，或者已由 eng 路径另行开发。本步不设计引擎本体，不拆前后端实现组件，只把业务侧 SR-F 映射为对既有引擎配置单元的配置需求。引擎或配置单元不存在/能力不足时，标注为缺口，不试图在本路径开发。

#### SR-F 类型 → 目标引擎配置单元映射表

| SR-F 类型 | PA 组件 = 目标引擎配置单元需求 | 说明 |
|-----------|-------------------------------|------|
| 菜单 / 入口 / 可见性 | **菜单引擎**配置单元需求 | 菜单导航、场景入口、角色可见性控制 |
| L4 场景业务 | **场景引擎**配置单元需求 | 场景编排、上下文传递、状态衔接、工作台 |
| L5 表单部件 | **表单引擎**配置单元需求 | 表单设计、字段定义、校验规则 |
| L5 流程部件 | **流程引擎**配置单元需求 | 流程定义、流转规则、审批配置 |
| L5 工单部件 | **工单引擎**配置单元需求 | 工单定义、分配规则、状态流转 |
| L6 功能元件 / 实体对象 | **实体引擎**配置单元需求 | 实体定义、字段结构、数据模型、版本状态、引用关系 |
| 指标 / 看板 / 聚合 | **指标引擎**配置单元需求 | 指标定义、计算规则、展示配置、异常下钻 |
| 资产目录 / 数据目录 / 血缘 / 质量 | **治理类引擎**或通用配置单元需求 | 元数据管理、数据地图、血缘追踪、质量规则、分类分级、脱敏 |

#### 映射逻辑

1. 从 SR-F 分类类型确定**目标引擎**（上表）
2. 从 25 引擎模型中定位目标引擎的**配置单元定义**——引擎提供哪些 CU、每个 CU 接受哪些配置信息
3. 将 SR-F 功能需求映射到具体的**配置单元**——SR-F 的四要素（功能描述/流程/规则/接口）转化为配置单元的配置项
4. 对每个配置单元需求执行资产决策：**复用已有配置 > 改进已有配置 > 新增配置需求**
5. 新增配置需求须与已有 CU 的配置模式一致（配置项结构 / 粒度 / 命名）

#### Q1/Q2/Q3 PA 配置需求匹配

映射前，按 SR-F 分类结果 + 引擎 CU 维度匹配已有 PA 配置需求节点：

| 判定 | 条件 | 动作 |
|------|------|------|
| **Q1 直接匹配** | 已有 PA 配置需求对应同一引擎 CU 且需求一致 | 复用已有 PA → 追加 SR-F 引用 → 不重新生成 |
| **Q2 扩展匹配** | 已有 PA 配置需求但需扩展（新增配置项/规则） | 扩展已有 PA → 全量重写配置需求（非增量追加） |
| **Q3 新建** | 无可用已有 PA 配置需求 | 标记待新建 → 按映射表生成配置需求 |

#### 引擎能力缺口标注

当 SR-F 功能接口指向的引擎在 25 中无对应 CU 定义，或已有 CU 能力不足时，标注为引擎能力缺口：

| 标注项 | 内容 |
|--------|------|
| 缺口类型 | `引擎不存在` / `CU 不存在` / `CU 能力不足` |
| 缺口描述 | 需要什么引擎能力来承接本 SR-F |
| 处理链路 | raw material → ort00~03 → wft01-eng → wft02-eng → wft03-eng → wft05-eng |

标注结果写入 PA 配置单元需求规格的"引擎能力缺口"小节。缺口不阻断 PA 方案推进——PA 方案标注缺口后仍可进入 `待确认方案`，但缺口须在 Step End 中显式提示。

#### 到位粒度

必须达到**引擎级 + 配置单元级 + 配置项级**：

1. 已判定每条 SR-F 的目标引擎
2. 已判定目标引擎中承接的配置单元
3. 已给出每个配置单元的核心配置项清单
4. 已完成上游需求条目到配置单元的分配
5. 已标注引擎能力缺口

不在本步展开：配置项的字段明细、校验规则逐条定义、UI 交互细节、前后端实现方案。

### Step 4 · 闭环检查

| 检查项 | 内容 |
|--------|------|
| **承接完整** | 每条 SR-F 的全部功能需求已有引擎配置单元承接 |
| **引擎覆盖** | 目标引擎在 25 中有定义（或已标注缺口） |
| **1:1 承接** | 每条需求条目有唯一配置单元宿主 |
| **配置可达** | 配置需求可交给业务配置/实施人员直接使用——配置路径清晰、配置项可操作 |
| **NFR 完整性** | 相关 NFR 约束已映射到配置单元 |
| **分类一致性** | 同一分类桶的 SR-F 对应同一引擎的同类 CU |
| **缺口标注** | 所有引擎能力缺口已标注处理链路 |

### Step 5 · 资产落账与写回

人类已在 Step 2 中整体确认方案，Step 5 统一执行落账与写回。

#### 跨域匹配（Q1/Q2/Q3）
