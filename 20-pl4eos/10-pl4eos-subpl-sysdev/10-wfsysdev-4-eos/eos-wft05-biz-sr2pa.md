# SR-F → PA · 业务侧引擎配置单元需求设计与资产化
**Skill ID**: wft05-biz-sr2pa

> **概述**：处理 ≥1 个 SR-F 业务功能需求节点（来自 `wft03-biz`，状态=`待产品架构` / `待补充PA设计`），PA 反馈优先。沿三步设计形成业务侧 PA：Step 3 SR-F 末级条目汇聚 + Step 4 配置单元需求规范化 + Step 5 NFR 约束装配与闭环检查；随后 Step 6 执行资产写回与状态推进。wft05-biz 的 PA 不是引擎本体组件，而是业务配置侧需要使用的“引擎配置单元需求规格”。本 Skill 读取 25 中已存在的引擎 / CU 定义作为承接边界；若找不到承接对象，只形成映射失败说明和阶段性方案，不生成引擎缺口材料，不触发 eng 路径。
>
> **输入 → 输出**：已按支撑引擎 / 主能力类型分解的 SR-F 业务功能需求节点 / 有反馈的 PA 方案节点 → PA 配置单元需求方案或映射失败说明 + SR 详细定义（`06`）+ PA 配置单元需求规格（`07`）+ SR-F→`已产品架构`（仅可确认方案）
>
> **适用场景**：瀑布式系统设计第5步 · 业务配置路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v4.0 | **修订**：2026-07-01

## 一、职责定位

### 1.1 本 Skill 目的与目标

> **主要目的**：证明业务功能 SR-F 可以被既有 EOS 引擎配置体系承接。
>
> **目标**：把 wft03-biz 已分解的 SR-F 末级条目，按支撑引擎、主能力类型和业务配置对象汇聚、去重并规范化为可确认的引擎配置单元需求规格，供业务配置 / 实施人员使用。
>
> **一句话**：biz 链到 wft05 不是“开发组件”，而是“定义要配置哪些既有引擎配置单元”；eng 链才是“开发这些引擎和配置单元”。

本 Skill 负责：

1. **接收**——接收人类选中的 ≥1 个 SR-F 业务功能需求节点（`待产品架构` / `待补充PA设计`）或有反馈的 PA 方案节点，校验状态、类型、数量和分解依据
2. **汇聚**（Step 3）——按 `支撑引擎 + 主能力类型 + 业务配置对象` 汇聚 SR-F 末级条目，形成配置需求候选组
3. **规范化**（Step 4）——读取 25 中既有引擎和 CU 定义，将候选组规范化为确定的配置单元需求规格
4. **装配 NFR**（Step 5）——消费 wft04 提供的 `biz/共享` 且目标为配置单元需求的 NFR 约束，将配置项、配置规则、配置可达性和验收标准写入 PA
5. **判定映射状态**——成功映射则形成可确认方案；引擎不存在、CU 不存在或 CU 承接不足时，只形成映射失败说明和阶段性方案
6. **落账**（Step 6）——写入 `07` PA 配置单元需求规格、`06` SR 详细定义、`05` 反向引用和状态，按需更新 `23/24` 引用
7. **推进**——人类确认且方案为 `可确认方案` 后，SR-F → `已产品架构`；阶段性方案不得放行
8. **回修**——接收下游或反馈中的 PA 缺失说明并修订；若根因在 SR-F 分解依据缺失，可将 SR-F → `需wft03修订`

本 Skill 不负责：

1. 引擎本体开发、前后端组件设计、CU 类型开发（→ `wft05-eng-sr2pa` 或 eng 路径）
2. 判断是否应新增引擎能力；本 Skill 只读取已存在的 25 引擎模型
3. 非功能需求的提出、量化和分类（由 wft04 承担）
4. 引擎能力缺口处理——不生成缺口材料，不触发 OR 预处理链，不转交 wft01-eng
5. 从粗粒度业务语义重新拆 SR-F；SR-F 分解依据缺失时退回 wft03-biz

### 1.2 三步边界

| Skill | 回答的问题 | 核心产物 |
|-------|------------|----------|
| wft02-biz | 业务架构对不对 | BA 双视角方案 + 引擎支撑信息 |
| wft03-biz | 系统功能是否完整承接业务 | SR-F 末级功能需求 + 主能力类型 |
| wft05-biz | 业务功能能否被配置体系承接 | PA 配置单元需求规格 |

### 1.3 与 wft05-eng 的边界

| 维度 | wft05-biz | wft05-eng |
|------|-----------|-----------|
| 输入 | 业务功能类 SR-F | 引擎能力类 SR-F |
| PA 本质 | 业务侧配置单元需求 | 引擎本体前后端组件需求 |
| 25 使用 | 只读，定位既有引擎/CU 承接边界 | 延续 25-Q，可形成引擎组件方案 |
| 失败处理 | 映射失败说明 + 阶段性方案 | 组件缺失或模型扩展按 eng 路径处理 |
| 下游对象 | 业务配置 / 实施人员 | 开发人员 |

## 二、输入与路由

### 2.1 标准输入包

1. 本轮处理对象（SR-F 业务功能需求节点 ≥1 个 / 有反馈的 PA 方案节点 / 映射失败后待重试条目）
2. 上游追溯链（`05-*.md` SR-F 正文块、`06-*.md` SR 详细定义、来源 BA、支撑引擎、主能力类型、辅助能力类型、功能四要素、验收口径、变更影响声明）
3. 已有 PA 架构资产（`07-*.md` 树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`23` 输出产品架构、`24` 业务资产、`25` 引擎模型、`26` 文档定义、`27` 资源资产、`22` NFR 分类资产）
5. NFR 约束包（从 wft04 分层约束表 / wft05-biz 装配视图装配，只消费 `适用链路=biz/共享` 且 `wft05 消费目标=wft05-biz:配置单元需求` / `wft05-biz+eng` 的约束）
6. 上次 PA 映射状态（`待映射` / `已映射` / `映射失败` / `待重试`）
7. 重入反馈

### 2.2 Step Start 路由

#### 2.2.1 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 SR-F 业务功能需求节点选中 + 无待处理反馈的 PA 方案节点 | 输出“无待处理对象”→ **退出** |
| SR-F 节点状态为 `需wft03修订` | 输出缺失说明，提示回到 wft03-biz 修订 SR-F → **退出** |
| SR-F 节点状态不是 `待产品架构` / `待补充PA设计` | 输出状态分布，列出不合格节点 → **退出** |
| 选中 SR-F 节点含引擎能力类 SR-F（来自 wft03-eng） | 输出类型分布（引擎能力类 → `wft05-eng`），要求分路径 → **退出** |
| SR-F 缺少支撑引擎、主能力类型、业务配置对象或验收口径 | 输出缺项清单 → SR-F → `需wft03修订` 候选，附缺失说明 |
| 选中 SR-F 节点分属多个业务域且无已确认关联 | 输出业务域分布，提示可能需分批 → 人类确认后继续或分批 |
| 选中 SR-F 节点 >5 个 | 输出“本轮 N 个节点，建议分批”→ 人类确认后继续，不强制退出 |

#### 2.2.2 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 条目 `确认状态` 字段中有人类标注且无 `[已处理]` 的 PA 方案节点 | 本轮先处理全部有反馈的 PA 方案节点 |
| 存在 `待确认方案` + 所有条目均已 `[同意]` + `[已处理]` 的 PA 方案节点 | 不阻塞新 SR-F；Step End 时提示 |
| 存在 `映射失败` 且人类指定重试 | 将对应条目标记为 `待重试`，本轮仅重试这些条目 |

#### 2.2.3 第三层：新 SR-F 路由

PA 反馈处理完后，逐 SR-F 节点处理新设计 → Step 3 SR-F 末级条目汇聚。

#### 2.2.4 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 PA 配置需求方案、映射状态或 NFR 装配结论，因此 Step 3 必须基于 Step 2 修改后的 PA 当前内容，不得使用 Step 1 加载时的缓存旧状态。

## 四、PA 设计基础

### 4.1 SR-F 与 PA 节点生命周期

#### 4.1.1 SR-F 推进态

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `待产品架构` | SR-F 已确认，本 Skill 可启动 PA 设计 | wft03-biz 完成确认 |
| `在PA设计` | 本 Skill 正在设计 PA | 本 Skill 写入 |
| `已产品架构` | PA 设计完成，SR-F 消费版本与当前一致 | 可确认方案完成落账 |

#### 4.1.2 回退态

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `待补充PA设计` | SR-F 已被旧版 PA 消费，上游 SR-F 修订后需补全 | wft03-biz 修订 SR-F |
| `需wft03修订` | SR-F 分解依据缺失或错误，PA 无法继续 | 本 Skill 或 wft05-biz 发现源头缺陷 |

SR-F 节点状态分工：wft03-biz 写入 `待产品架构` / `待补充PA设计` / `需wft03修订`；本 Skill 写入 `在PA设计` / `已产品架构`。

#### 4.1.3 PA 方案状态

| 状态 | 含义 |
|------|------|
| `待确认方案` | PA 方案已生成，等待人类反馈 |
| `可确认方案` | 所有 SR-F 均已成功映射且无阻断型 NFR 未处理 |
| `阶段性方案` | 存在映射失败、SR-F 分解依据缺失或阻断型 NFR 未落入配置规格 |
| `已验证冻结` | 人类确认后完成落账，可供配置 / 实施使用 |

阶段性方案不得推进 SR-F 为 `已产品架构`。

### 4.2 PA 架构说明

**PA 本质**：wft05-biz 的 PA 是业务侧配置单元需求规格，不是软件组件、不是引擎开发任务、不是业务运行页面设计。

**汇聚主键**：

```text
支撑引擎 + 主能力类型 + 业务配置对象
```

| 维度 | 来源 | 用途 |
|------|------|------|
| 支撑引擎 | SR-F 末级条目 | 定位 25 目标引擎 |
| 主能力类型 | SR-F 末级条目 | 定位目标 CU 类型 |
| 业务配置对象 | 来源 BA / SR-F / 输出文档 / 数据对象 | 判断哪些 SR-F 应合并为同一配置需求 |
| 辅助能力类型 | SR-F 末级条目 | 形成依赖关系，不改变主宿主 |
| NFR 约束 | wft04 装配视图 | 补充配置项、规则和验收口径 |

**PA 配置单元需求完整要素**：

| 要素 | 内容 |
|------|------|
| PA 需求 ID / 名称 | 稳定标识和业务可读名称 |
| 来源 SR-F | SR-F 节点、SR-F 消费版本、末级条目清单 |
| 来源 BA / 业务配置对象 | BA 对象、输出文档、实体、菜单、指标等 |
| 目标引擎 | 25 中既有 `@engine-*` |
| 目标 CU / 主能力类型 | 25 中既有 CU 类型或能力结构 |
| 配置项清单 | 业务配置人员需要填写、选择或维护的配置项 |
| 配置规则 | 校验、权限、状态、发布、引用、口径规则 |
| 依赖配置 | 辅助能力、实体、权限、指标、资源等依赖 |
| NFR 约束 | 进入配置项、规则、可达性或验收的约束 |
| 配置路径 / 可达性 | 配置入口、角色、流程或页面路径 |
| 验收标准 | 配置完成后可验证的通过条件 |
| 映射状态 | `待映射` / `已映射` / `映射失败` / `待重试` |
| 确认状态 | `[同意]` / `[修改]` / `[驳回]` + `[已处理]` |

### 4.3 映射状态

| 映射状态 | 含义 | 触发条件 |
|----------|------|----------|
| `待映射` | 尚未处理该 SR-F 末级条目 | 首次进入或新增 SR-F |
| `已映射` | 已成功定位目标引擎 / CU 并形成配置需求 | Step 4 规范化成功 |
| `映射失败` | 引擎不存在、CU 不存在或 CU 承接不足 | Step 4 无法规范化 |
| `待重试` | 上次失败后，人类要求重试或 25 已补充新 CU | Step Start 重入 |

`已映射` 条目可进入 Step 6 正式落账。`映射失败` 条目只写阶段性方案，不进入正式配置需求规格落账。

### 4.4 设计规则

1. **只消费既有引擎/CU**：25 中不存在的引擎或 CU 不在本 Skill 中新建。
2. **先汇聚后规范化**：不得逐条 SR-F 机械生成一堆碎 PA；先按主键汇聚，再形成配置单元需求。
3. **复用优先**：已有 PA 配置需求可承接时优先追加引用；需扩展时全量重写配置需求；无承接时新建。
4. **NFR 必须落位**：阻断型 NFR 不能只做备注，必须改变配置项、配置规则、配置可达性或验收标准。
5. **映射失败不造缺口**：失败说明只描述“本 SR-F 无法被现有配置体系承接”，不生成 FR-ENG 材料。
6. **阶段性方案不放行**：存在映射失败或阻断型 NFR 未落位时，SR-F 不得推进 `已产品架构`。
7. **SR-F 源头缺陷退回**：缺少支撑引擎、主能力类型、业务配置对象、验收口径时，退回 `需wft03修订`。

### 5.2 Step 1 · 设计材料加载

按 §三加载全景层和本轮正文块。重点确认：

1. SR-F 是否来自 wft03-biz 的业务功能类节点
2. 每条 SR-F 末级是否具备支撑引擎、主能力类型、业务配置对象、验收口径
3. 25 是否存在目标引擎和候选 CU
4. NFR 约束包是否存在 `biz/共享` 且目标为配置单元需求的约束
5. 上次映射失败条目是否具备重试条件

### 5.3 Step 2 · 方案反馈处理

#### 5.3.1 人类审阅规则

人类在 PA 方案条目的 `确认状态` 字段标注 `[同意]` / `[修改]：修改内容` / `[驳回]：驳回理由`。AI 处理完后追加 `[已处理]`。

**条目级反馈**（权威）：逐条标注。已 `[同意] + [已处理]` 的条目后续默认跳过。

**整体确认**（快捷）：人类说“整体确认”时，AI 对无条目级标注的条目写入 `[同意] + [已处理]`。冲突消解：条目级 > 整体确认。

#### 5.3.2 AI 处理规则

| 人类意见 | AI处理 |
|---------|------|
| `[同意]` | 条目维持不变，追加 `[已处理]` |
| `[修改]` | 按反馈调整配置项、规则、依赖、NFR 落位或验收标准；重做受影响闭环检查；追加 `[已处理]` |
| `[驳回]` | 移除或重做被驳回配置需求；若整体驳回则重做 Step 3~5；追加 `[已处理]` |

**状态判定**：

- 全部条目 `[同意] + [已处理]` 且 PA 为 `可确认方案` → 进入 Step 6，SR-F → `已产品架构`
- 任一条目含 `[修改]` / `[驳回]` → PA 保持 `待确认方案`
- PA 为 `阶段性方案` → 即使人类同意，也不得推进 SR-F 为 `已产品架构`，只能保留阶段性结论或重试条件

### 5.4 Step 3 · SR-F 末级条目汇聚

**设计线第一步**。将 wft03-biz 输出的 SR-F 末级条目按汇聚主键聚成配置需求候选组。

#### 5.4.1 汇聚过程

1. 读取每条 SR-F 的支撑引擎、主能力类型、辅助能力类型、业务配置对象、功能四要素和验收口径
2. 以 `支撑引擎 + 主能力类型 + 业务配置对象` 分组
3. 同组内合并功能描述、流程、规则、接口和验收口径
4. 辅助能力类型转为依赖配置，不改变主配置单元宿主
5. 标记来源 SR-F 列表和 SR-F 消费版本

#### 5.4.2 汇聚候选组写入结构

| 候选组 | 支撑引擎 | 主能力类型 | 业务配置对象 | 来源 SR-F | 辅助能力 | 初步结论 |
|--------|----------|------------|--------------|-----------|----------|----------|
| {组ID} | @engine-form | 表单结构 / 字段 / 校验 | {输出文档/实体} | SR-F-... | 权限 / 实体 | 待规范化 |
| {组ID} | @engine-flow | 流程定义 / 节点配置 | {流程对象} | SR-F-... | 表单 / 实体 | 待规范化 |
| {组ID} | @engine-indicator | 指标定义 / 视图 | {指标对象} | SR-F-... | 实体 / 权限 | 待规范化 |

### 5.5 Step 4 · 配置单元需求规范化

**设计线第二步**。读取 25 中既有引擎和 CU 定义，把候选组规范化为 PA 配置单元需求规格。

#### 5.5.1 25 映射规则

| 判定 | 条件 | 动作 |
|------|------|------|
| PA-Q1 | 25 中存在目标引擎和匹配 CU，07 中已有同业务配置对象需求且内容一致 | 复用已有 PA，追加 SR-F 引用 |
| PA-Q2 | 25 中存在目标引擎和匹配 CU，07 中已有同类需求但需扩展配置项/规则 | 扩展已有 PA，全量重写配置需求 |
| PA-Q3 | 25 中存在目标引擎和匹配 CU，07 中无可用 PA | 新建 PA 配置单元需求 |
| 映射失败 | 25 中无目标引擎 / 无匹配 CU / CU 承接不足 | 写映射失败说明，方案标记为阶段性 |

#### 5.5.2 配置需求规格写入结构

| 字段 | 内容 |
|------|------|
| PA 需求 ID | {PA-CU-XXX} |
| 需求名称 | {业务对象 + CU 能力名称} |
| 目标引擎 | `@engine-*` |
| 目标 CU / 主能力类型 | `{CU ID / 能力类型}` |
| 来源 SR-F | {SR-F 列表 + 消费版本} |
| 来源业务对象 | {BA / 输出文档 / 实体 / 菜单 / 指标} |
| 配置项清单 | {配置人员需要维护的配置项} |
| 配置规则 | {校验、权限、状态、发布、引用、口径规则} |
| 依赖配置 | {辅助能力类型和目标依赖} |
| 配置路径 / 可达性 | {配置入口、角色、流程} |
| 验收标准 | {可验证通过条件} |
| 映射状态 | 已映射 / 映射失败 / 待重试 |
| 确认状态 | — |

#### 5.5.3 映射失败说明

| 说明项 | 内容 |
|--------|------|
| 失败类型 | `无目标引擎` / `无可用 CU` / `CU 承接不足` / `SR-F 分解依据缺失` |
| 失败对象 | SR-F 条目 / 候选组 / 业务配置对象 |
| 失败说明 | 哪类既有配置承接对象缺失或不足 |
| 本步结论 | `阶段性方案`，不得推进为可确认 PA |
| 后续归属 | 回 wft03-biz 修订分解依据，或等待 eng 路径补齐 25 后重试；本 Skill 不生成缺口材料 |

### 5.6 Step 5 · NFR 装配与闭环检查

**设计线第三步**。将 wft04 输出的适用 NFR 约束装配到配置单元需求，并完成可确认 / 阶段性判定。

#### 5.6.1 NFR 装配规则

| wft04 影响点 | 写入位置 | 处理规则 |
|--------------|----------|----------|
| 配置项 | 配置项清单 | 转化为必须配置 / 可选配置 / 禁止配置项 |
| 配置规则 | 配置规则 | 转化为校验、权限、脱敏、发布、审计规则 |
| 配置可达性 | 配置路径 / 可达性 | 明确配置角色、入口、流程和访问控制 |
| 验收标准 | 验收标准 | 写入可验证指标、检查方法和通过条件 |
| 组件类型 / 组件边界 / 接口机制 / 运行约束机制 / 共享归属 | 不在 biz 侧展开 | 标注为 `eng 约束`，仅保留引用；由 wft05-eng 承接 |

阻断型 NFR 必须进入配置项、配置规则、配置可达性或验收标准；否则 PA 保持 `阶段性方案`。

#### 5.6.2 闭环检查

| 检查项 | 内容 |
|--------|------|
| SR-F 承接完整 | 每条 SR-F 末级均有唯一 PA 配置需求宿主或映射失败说明 |
| 25 承接完整 | 每个可确认 PA 均能定位到既有目标引擎和 CU |
| 配置可达 | 配置路径、角色、入口和操作流程清晰 |
| NFR 落位 | `biz/共享` NFR 已进入配置项、规则、可达性或验收 |
| NFR 边界正确 | `eng` 组件类约束未在 biz 侧展开 |
| 映射失败隔离 | 映射失败条目未进入可确认 PA |
| 下游可用 | 配置 / 实施人员可据此执行配置 |

#### 5.6.3 方案判定

| 判定 | 条件 | 后续 |
|------|------|------|
| `可确认方案` | 所有 SR-F 均已映射，NFR 阻断项已落位，闭环检查通过 | 等待人类确认后 Step 6 落账 |
| `阶段性方案` | 存在映射失败、SR-F 源头缺失、NFR 阻断项未落位 | 不推进 SR-F；输出失败说明和重入条件 |

### 5.7 Step 6 · 资产写回与落账

人类确认且方案为 `可确认方案` 后执行正式落账。

1. **写入 07**：PA 配置单元需求规格、PA-Q 判定、NFR 装配结果、确认状态、资产写回记录
2. **写入 06**：SR 详细定义补全，记录 PA 承接对象和配置需求摘要
3. **更新 05**：SR-F 反向引用、PA 消费版本、状态推进为 `已产品架构`
4. **更新 23**：PA 配置需求索引或输出产品资产引用（如本仓库约定需要）
5. **更新 24**：业务资产引用和 PA 承接关系（如本仓库约定需要）

阶段性方案只写入 07 的阶段性说明和映射失败说明，不推进 `05` 状态，不写正式 PA 配置需求规格。

## 六、SR-F→PA 变更传播协议

### 6.1 变更类型

| 类型 | 判定条件 | PA 处理 |
|------|----------|---------|
| 增量型 | 新增 SR-F 末级条目，既有主键不变 | 新增候选组或追加到既有 PA |
| 修订型 | SR-F 四要素、主能力类型、业务配置对象或 NFR 约束变更 | 修订受影响 PA，全量重写配置需求 |
| 结构型 | 支撑引擎、主能力类型、业务配置对象主键变化 | 重做汇聚和规范化，必要时废弃旧 PA |
| 缺失型 | SR-F 缺少支撑引擎、主能力类型、验收口径 | SR-F → `需wft03修订` 候选 |

### 6.2 受影响范围判断

| SR-F 变化 | 可能影响 |
|-----------|----------|
| 支撑引擎变更 | 目标 25 引擎、PA 宿主、配置路径 |
| 主能力类型变更 | 目标 CU、配置项清单、配置规则 |
| 业务配置对象变更 | 汇聚组、复用/新增判定、业务资产引用 |
| 功能规则变更 | 配置规则、NFR 装配、验收标准 |
| 功能接口变更 | 依赖配置、配置路径、下游实施边界 |
| NFR 约束变更 | 配置项、规则、可达性、验收口径 |

## 九、与历史 wft05a/05b 的关系

本文件继承原 wft05a / wft05b 的自闭环思路，但将 PA 业务侧产物稳定收敛为“引擎配置单元需求规格”：

- 不再使用“1+7 配置方案组件”或“四源组件”作为主模型
- 不从粗粒度 SR-F 猜测 CU，而是消费 wft03-biz 已写明的支撑引擎 / 主能力类型
- 25 从“参考资料”升格为“既有承接边界”
- 映射失败不进入缺口链路，只形成阶段性方案和重入条件
- NFR 只消费配置单元需求相关部分，eng 组件类约束交给 wft05-eng
