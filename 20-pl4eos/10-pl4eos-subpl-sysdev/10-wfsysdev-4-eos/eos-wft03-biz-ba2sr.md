# BA(A2+Bn双视角) → SR-F · 业务功能需求设计与资产化
**Skill ID**: wft03-biz-ba2sr

> **概述**：每次执行处理 ≥1 个 BA（业务架构）业务节点（来自 wft02-biz，状态=`可以SR设计` / `待补充SR设计`），SR-F 反馈优先于新 BA 处理。将 wft02-biz 形成的 A2（角色场景产品）/Bn（平台输出产品）双视角业务架构翻译成系统必须提供的功能需求，并按 BA 末级对象的支撑引擎读取 25（平台引擎资产）中对应引擎的 CU（配置单元）能力结构，将功能需求分解为可被 wft05-biz 汇聚的 SR-F（功能性系统需求）末级条目。三条输入流仍然保留：流 1：A2 角色场景视角 BA → 用户可操作 SR-F；流 2：Bn 产品视角 BA → 业务/数据/配置资产 SR-F；流 3：A2-Bn 双视角映射 → 一致性/追溯/可达性 SR-F。完成 SR-F 末级分解、分类映射、完备性校验与状态推进。业务功能类 SR-F 只读取 25 作为 CU 能力结构参考，不写回 25 资产，仅推进 04/05 状态。
>
> **输入 → 输出**：BA 业务节点（≥1 个，`可以SR设计` / `待补充SR设计`）/ 有反馈的 SR-F 节点 → 按支撑引擎 CU 能力结构分解后的 SR-F 末级业务功能需求 + 分类映射 + BA 推进为 `已SR设计` + SR-F 推进为 `待产品架构`
>
> **适用场景**：瀑布式系统设计第3步 · 业务功能需求路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.2 | **修订**：2026-06-28

## 一、职责定位

### 本 Skill 目的与目标

> **主要目的**：取信于业务开发人员——「业务架构已经能被系统功能完整承接」。把 wft02-biz 形成的 A2/Bn 双视角业务架构，转成清晰、可审查、可继续设计的 SR-F 系统功能需求。
>
> **目标**：功能需求覆盖业务、边界清楚、粒度适中，并分解到可由单一主引擎 CU 能力类型承接的末级条目。具体说：
>
> - A2 角色场景视角 BA → 用户可操作的系统功能需求
> - Bn 产品视角 BA → 业务资产、数据资产、配置资产的系统功能需求
> - A2-Bn 映射 → 可达性、权限、数据对象、场景部件之间的一致性需求
> - L4/L5/L6 业务对象 → 读取其支撑引擎的 CU 能力结构，形成对应的 SR-F 末级条目、功能描述、功能流程、功能规则、功能接口
>
> 一句话：wft03-biz 把"业务怎么运转"翻译成"系统必须提供哪些功能"。

本 Skill 负责：

1. **接收**——接收人类选中的 ≥1 个 BA 业务节点（`可以SR设计` / `待补充SR设计`），校验状态/类型/数量/业务域分布，不合格则说明原因并退出
2. **匹配**——按 BA 节点 + 业务对象维度匹配已有 SR-F 节点——同业务对象的 SR-F 直接归入已有节点（Q1=直接引用）、扩展归入（Q2=扩展），无已有则新建（Q3=新建）
3. **展开**——将 BA 双视角业务方案沿三条输入流转化为 SR-F 业务功能需求，并读取 25 中支撑引擎的 CU 能力结构，把功能需求分解为可由单一主 CU 能力类型承接的末级条目；沿三步推进：3a 流 1 A2→用户可操作 SR-F → 3b 流 2 Bn→资产管理 SR-F → 3c 流 3 A2-Bn 映射→一致性 SR-F
4. **分类**——基于功能需求，按 SR-F 分类规则执行分类映射（Layer 1 业务定义类型 → Layer 2 系统职责分类 → Layer 3 SysReq（系统需求）末级节点），三流产出统一进入同一 SR-F 分类树
5. **校验**——校验三流 SR-F 末级条目覆盖 BA 节点中所有 L4/L5/L6 + 映射关系（消费完备性），校验分类合规性和 CU 能力结构分解合规性
6. **推进**——人类确认方案后统一推进 BA/SysReq 状态。业务功能类 SR-F 不写回 25 资产，仅推进 04/05 状态

本 Skill 不负责：

1. 引擎配置需求（A1（平台引擎产品）BA → `wft03-eng-ba2sr`）
2. 非功能需求（STR-NFR → `wft04a`）
3. 引擎能力功能定义与 25 资产写回（→ `wft03-eng-ba2sr`）；本 Skill 可读取 25 的既有 CU 能力结构用于分解 SR-F，但不得修改 25 或生成引擎需求
4. PA（产品架构）组件设计与详细定义（由 wft05 承担）
5. 引擎能力缺口的所有处理——不读取引擎缺口原始需求材料、不判断引擎能力是否足够、不把引擎缺口作为 SR-F 阻断项、不记录"待引擎增强依赖"、不触发 OR 预处理链或 wft01-eng、不把引擎缺口转化为 SR-F（引擎缺口由 wft02-biz 的原始需求材料链路独立处理：wft02-biz → 10-raw-files → ort00~03 → FR-ENG OR → wft01-eng → eng 路径）

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 BA 业务节点选中 + 无待处理反馈的 SR-F 节点 | 输出"无待处理对象"→ **退出** |
| BA 节点状态不是 `可以SR设计` / `待补充SR设计` | 输出状态分布，列出不合格节点及其当前状态 → **退出** |
| 选中 BA 节点含 A1 产品类型 | 输出类型分布（A1 → `wft03-eng`），要求分路径 → **退出** |
| 选中的 BA 节点分属多个业务域且无已确认关联 | 输出业务域分布，提示可能存在跨域设计 → 人类确认后继续或分批 |
| 选中 BA 节点 >5 个 | 输出"本轮 N 个 BA 节点，SR-F 设计工作量大，增量清单长，建议分批"，提示节点 ID 列表 → 人类确认后继续，不强制退出 |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 SR-F 节点 | 本轮先处理全部有反馈的 SR-F 节点 |
| 存在 `待确认方案` + 反馈区为空的 SR-F 节点 | 不阻塞新 BA；Step End 时提示 |

#### 第三层：新 BA 业务节点路由

SR-F 反馈处理完后，逐 BA 业务节点处理新设计 → Step 3。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 SR-F 节点的功能定义或分类归属，因此 Step 3 必须基于 Step 2 修改后的 SR-F 节点当前内容，不得使用 Step 1 加载时的缓存旧状态。

### BA 节点生命周期

#### 推进态（按顺序）

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `可以SR设计` | BA 方案已确认，本 Skill 可启动 SR-F 展开 | wft02-biz 人类确认方案 |
| `在SR设计` | 本 Skill 正在展开 SR-F | 本 Skill 写入 |
| `已SR设计` | SR-F 展开完成，BA 消费版本与当前版本一致 | 本 Skill 处理完成且未检测到上游新版本 |

#### 回退态（上游修订触发）

当新 STR-F 汇入已有 BA 触发 wft02-biz 修订时，BA 内容变更、版本号递增——但本 Skill 可能已基于旧版本生成了 SR-F。修订后 BA 状态按本 Skill 的消费进度分流：

| 修订前状态 | 修订后 | 说明 |
|-----------|--------|------|
| `可以SR设计` | 保持，版本递增 | 本 Skill 尚未消费，不受影响——下次进入时直接读最新版本 |
| `在SR设计` | 保持，版本递增 | 本 Skill 正在处理中，重入时检测版本差异，按变更影响声明增量处理 |
| `已SR设计` | 降为 **`待补充SR设计`** | 本 Skill 之前生成的 SR-F 基于旧版本 BA，本轮新增的业务/场景要素缺少对应 SR-F 节点，需重新介入补全 |

- **`待补充SR设计`**：已消费的 SR-F 节点不动（它们仍然有效），但 BA 新增了业务、场景部件结构或场景要素需要生成新的 SR-F。本 Skill 进入后按 BA 的变更影响声明确定增量映射范围，处理完成后若消费版本追上 BA 当前版本 → 推进回 `已SR设计`

BA 节点状态分工：wft02-biz 写入 `待确认方案` / `可以SR设计` / `待补充SR设计`，并在修订时更新版本号；本 Skill 写入 `在SR设计` / `已SR设计`。

#### 版本标记与消费追溯

- BA 节点初次生成时版本 = `v1`（由 wft02-biz 设置）
- 每次因新 STR-F 触发 wft02-biz 修订，版本号递增
- 本 Skill 首次消费 BA 时，在对应 SR-F 节点中记录 `BA 消费版本：v{N}`
- 本 Skill 重入同一 BA 时比对版本：若 BA 当前版本 > 已消费版本 → 触发变更感知，按变更影响声明增量处理

### Step 2 · 方案反馈处理

若本轮存在有反馈的 SR-F 节点（`待确认方案` + 反馈区有效），逐个处理。

#### 基本操作

| 操作 | 行为 |
|------|------|
| `[同意]` 或整体确认 | 归档反馈 → SR-F → `待产品架构`（wft05 此时即可进入）；BA → `已SR设计` |
| `[修改]`（局部） | 按反馈调整被指定的功能需求/分类归属 → 更新增量清单 → 回写 `待确认方案` |
| `[驳回]`（局部） | 移除被驳回的功能需求和分类归属 → 其他功能需求 → `已确认` → 回写 `待确认方案` |
| `[驳回]`（整体） | 重做 Step 3 → 回写 `待确认方案` |

> **写回时机**：`[同意]` 后的资产写回分两种情况——若本轮仅有反馈处理（无新 BA），直接进入 Step 5 推进状态；若本轮同时有新 BA 待处理，先执行 Step 3~4 完成新设计，待全部内容经人类确认后由 Step 5 统一推进——避免分批次推进造成状态不一致。

#### 分层反馈

SR-F 节点包含多个可独立决策的单元（三流各自的功能需求、分类归属、A2-Bn 映射关系）。人类可以**局部修改**——整体确认一个 SR-F 节点的同时，要求调整其中某条流的功能需求或某个分类归属，不需要因一个局部问题而驳回整个 SR-F 节点。

AI 读取反馈时按以下规则处理：

- **局部 `[修改]`**：按反馈调整被指定的对象，其涉及的分类归属重新执行。未提及的对象保持原结论不动，不触发全量重新分类。
- **局部修改导致级联**：流 1 功能需求变更 → 级联更新流 3 映射关系；流 2 功能需求变更 → 级联更新流 3 映射关系；映射关系变更 → 级联更新分类归属。级联完成后重新执行受影响对象的一致性验证。
- **SR-F 节点推进条件**：全部反馈处理完成（无未处理的 `[修改]`/`[驳回]` 项）→ SR-F 推进为 `待产品架构`，BA 推进为 `已SR设计`。存在未处理项 → SR-F 保持 `待确认方案`。
- **`[驳回]`（局部）**：移除被驳回的功能定义和分类归属。剩余对象保持原结论。
- **`[驳回]`（整体）**：SR-F 主结构被推翻（"三流划分根本不对"或"A2/Bn 双视角不一致"）→ 重做 Step 3。

#### 反馈区位置

反馈区位于 `05-*.md` 中**每个 SR-F 节点块的末尾**，作为 SR-F 节点正文的最后一个子节。

#### 反馈区格式

**AI 写入（Step 3 完成时）** —— 空反馈区：

```text
### 反馈区

[操作] 
```

**人类填写（或 AI 从对话转写）** —— 三种操作之一：

```text
### 反馈区

[操作] 同意

```
或
```text
### 反馈区

[操作] 修改

- 流1-场景入口：功能流程应增加"异常回退"步骤
- 流2-主数据维护：功能规则缺少"数据权限"约束
- 分类归属：流程定义→审批流转 归属桶不合适，应归入"审批配置"

```
或
```text
### 反馈区

[操作] 驳回

- 流1-场景入口：功能定义与 A2 视角不匹配。不移除 SR-F 节点其他内容。
- 整体驳回：三流划分不合理，A2 和 Bn 的边界不清晰。

```

**AI 处理完成后** —— 归档旧反馈，重置反馈区（同 wft02-biz 模式）。

### Step 3 · 三流功能需求分解设计

将 BA 双视角业务方案沿三条输入流转化为 SR-F 业务功能需求，并进一步按支撑引擎的 CU 能力结构分解为 SR-F 末级条目。顺序推进：流 1（A2 → 用户可操作 SR-F）→ 流 2（Bn → 资产管理 SR-F）→ 流 3（A2-Bn 映射 → 一致性/追溯 SR-F）。

> **三流顺序逻辑**：流 1 和流 2 可并行设计（各自从 BA 的 A2/Bn 视角独立提取），流 3 必须在流 1 和流 2 完成后执行（需要两端产出才能校验映射关系）。

#### SR-F 末级分解规则

SR-F 分解以 wft02-biz 写入的 **BA 末级对象引擎支撑信息** 为入口，以 25 中对应支撑引擎的 **CU 能力结构** 为分解边界。分解目标是形成 wft05-biz 可直接汇聚和规范化的 SR-F 末级条目，而不是在本步生成 PA 配置单元需求。

分解步骤：

1. **定位 BA 末级对象**：读取 L4 场景业务、L5 场景部件、L6 功能元件、业务菜单方案、上层指标方案、治理对象等 BA 末级对象。
2. **读取支撑引擎**：从 BA 末级对象引擎支撑信息中读取支撑引擎和引擎支撑状态。
3. **读取 CU 能力结构**：在 25 中读取该支撑引擎的 CU 能力结构，只作为 SR-F 分解依据，不写回 25。
4. **按 CU 能力边界分解**：把 BA 对象的业务功能拆为若干 SR-F 末级条目，每条条目只允许一个主支撑引擎和一个主 CU 能力类型，可列辅助能力类型。
5. **保留业务语义**：每条 SR-F 仍写业务功能描述、功能流程、功能规则、功能接口和验收口径，不写配置项明细和 PA 规格。
6. **处理支撑异常**：若 BA 对象的引擎支撑状态不是 `通过`，本 Skill 不处理引擎缺口；仅生成业务侧可表达的 SR-F 条目并标注 `分解依据不足`，不得把缺口转成 SR-F。

每条 SR-F 末级条目至少包含：

| 字段 | 填写要求 |
|------|----------|
| 来源 BA 对象 | L4/L5/L6/菜单/指标/治理对象 ID |
| 输入流 | 流 1 / 流 2 / 流 3 |
| 支撑引擎 | 来自 wft02-biz 的 BA 末级对象引擎支撑信息 |
| 主 CU 能力类型 | 来自 25 中该支撑引擎的 CU 能力结构；只写能力类型，不写 PA 配置规格 |
| 辅助能力类型 | 可选，如权限、实体、通知、指标等辅助能力 |
| 功能描述 / 流程 / 规则 / 接口 | SR-F 四要素 |
| 验收口径 | 能独立验证该末级条目是否被系统承接 |
| 分解依据 | BA 对象 + 支撑引擎 + 25 CU 能力结构引用 |

分解粒度约束：

- **单一主职责**：每条末级 SR-F 只表达一个主系统职责，如字段定义、校验规则、流程流转、实体状态、菜单可达、指标计算。
- **单一主能力类型**：每条末级 SR-F 只对应一个主 CU 能力类型；需要多个能力协同时，用辅助能力类型表达，不合并为一个大条目。
- **可独立验收**：每条末级 SR-F 必须能写出一条验收口径。
- **不生成 PA 规格**：不写字段明细、配置参数、CU 输入格式、配置发布规则等 PA 层内容。

#### 三流分流总表

| 输入流 | wft02-biz 产物 | 转化方式 | SR-F 产出 |
|--------|---------------|---------|-----------|
| 流 1 | A2 角色场景视角 BA | 先识别用户可操作功能，再按支撑引擎 CU 能力结构分解 | 菜单、页面、场景流、权限、交互、异常处理、场景指标 |
| 流 2 | Bn 产品视角 BA | 先识别业务/数据/配置资产功能，再按支撑引擎 CU 能力结构分解 | 业务产品资产、主数据、业务数据、配置资产的维护、查询、引用、版本、状态 |
| 流 3 | A2-Bn 双视角映射 | 仅当需要系统提供校验/查询/告警/修复能力时生成 SR-F，再按支撑引擎 CU 能力结构分解 | 场景部件与 Bn 原子业务的映射、菜单到 L4、L5 到产品数据、权限到数据对象 |

#### 3a · 流 1：A2 角色场景视角 → 用户可操作 SR-F

**A2 SR-F 生成规则**：从"谁通过什么入口完成什么职责闭环"生成系统功能。A2 的核心锚点是用户履职——每个 SR-F 必须回答"哪个角色通过什么入口完成什么职责闭环"，不生成与用户操作无关的系统内部功能。交互信息（系统辅助、消息通知等）不生成独立 SR-F，作为 L5 Process / L4 编排 / 权限或集成能力线索吸收到 A2 SR-F 中。

从 BA 的 A2 角色场景视角中提取用户可操作功能，逐 L4 场景业务 / L5 场景部件生成 SR-F。

**生成逻辑**：

1. 从 A2 的 L4 场景业务 IPO 中提取**场景入口功能**——用户从哪里进入（菜单/工作台/消息通知/快捷入口）
2. 从 A2 的 L5 场景部件编排中提取**场景流功能**——用户在场景内完成什么操作序列（打开→填写→提交→流转→查看结果）
3. 从 A2 的上下文传递和状态衔接中提取**交互功能**——场景间跳转、状态提示、操作反馈
4. 从 A2 的异常路径中提取**异常处理功能**——操作失败回退、数据冲突处理、超时处理
5. 从 A2 的权限规则（Phase D 权限推导结果）中提取**权限功能**——角色可见菜单、字段可编辑/只读、操作按钮可见/隐藏
6. 从 A2 的场景级指标中提取**指标展示功能**——KPI（关键绩效指标）卡片、趋势图、钻取导航
7. 综合形成四要素：功能描述（一句话收口该场景的用户操作使命）、功能流程（用户操作步骤序列）、功能规则（交互约束 + 权限规则 + 场景守卫）、功能接口（指向表单引擎/流程引擎/权限引擎）

**A2 SR-F 产出清单模板**：

```
│   ├── [支撑引擎: 菜单引擎] 菜单可见性（角色→菜单可见/隐藏）
```

#### 3b · 流 2：Bn 产品视角 → 资产管理 SR-F

**Bn SR-F 生成规则**：从"平台沉淀哪些业务产品资产/数据资产/配置资产"生成资产管理功能。Bn 的核心锚点是资产沉淀与复用——每个 SR-F 必须回答"平台沉淀了什么可复用的业务产品能力/数据资产/配置资产，如何维护和使用它们"，不生成与资产管理无关的用户界面或场景流功能（那些归流 1）。Bn 高层节点体现为业务产品资产树 / 数据资产目录 / 配置资产目录，不以用户菜单为主结构。

从 BA 的 Bn 产品视角中提取资产管理功能，逐 Bn 原子业务 / L6 功能元件生成 SR-F。

**生成逻辑**：

1. 从 Bn 的原子业务操作中提取**资产维护功能**——创建、编辑、复制、删除
2. 从 Bn 的产品数据（输出文档）中提取**资产查询功能**——列表查询、详情查看、全文检索、按条件筛选
3. 从 Bn 的引用关系中提取**资产引用功能**——跨业务域引用、主数据引用、上下游关联
4. 从 Bn 的版本策略中提取**版本管理功能**——版本创建、版本对比、版本回滚
5. 从 Bn 的状态流转中提取**状态管理功能**——发布、归档、作废、恢复
6. 从 Bn 的 L6 功能元件需求中提取**细粒度资产操作功能**——字段级编辑、规则配置、权限设置
7. 综合形成四要素：功能描述（一句话收口该资产管理的系统使命）、功能流程（资产生命周期步骤序列）、功能规则（业务规则 + 数据规则 + 配置约束）、功能接口（指向数据层与引擎）

#### 3c · 流 3：A2-Bn 双视角映射 → 一致性/追溯/可达性 SR-F

**双视角校验 SR-F 生成规则**：确保 A2 可访问的东西在 Bn 有业务资产承接，Bn 沉淀的能力能被 A2 场景使用或标注为后台公共能力。流 3 的核心锚点是双视角一致性——每个 SR-F 必须回答"用户能访问的场景/操作是否都有对应的业务资产承接"和"平台沉淀的资产是否都能被场景使用或标注为公共能力"，不生成仅属于单视角的功能（那些归流 1 或流 2）。未被任何 A2 场景使用的 Bn 资产标注为 `[后台公共能力]`，不视为一致性缺口。架构映射关系本身不自动生成 SR-F，只有需要系统提供可操作、可查询、可校验、可告警或可修复能力时，才生成流 3 SR-F。

基于流 1 和流 2 的产出，从 BA 的 A2-Bn 双视角映射关系中提取一致性和追溯性功能。

**生成逻辑**：

1. 从 A2→Bn 映射中提取**可达性校验功能**——菜单入口能否到达目标产品数据、场景部件是否引用了有效的 Bn 原子业务
2. 从 Bn→A2 映射中提取**使用追溯功能**——产品数据被哪些场景使用、Bn 原子业务支撑了哪些用户操作
3. 从 A2 L5 与 Bn 原子业务的对应关系中提取**场景-数据映射功能**——场景操作同步更新对应产品数据、产品数据变更通知相关场景
4. 从菜单到 L4 的可达性中提取**菜单-场景映射功能**——菜单配置变更校验场景可达性
5. 从权限到数据对象的关系中提取**权限-数据映射功能**——权限变更校验数据对象可达性
6. 综合形成四要素：功能描述（一句话收口该一致性/可达性的系统保证）、功能流程（触发→比对→告警→修复步骤序列）、功能规则（映射规则 + 追溯规则 + 可达性规则）、功能接口（指向双端：A2 侧 + Bn 侧）

**流 3 SR-F 产出清单模板**：

```

├── 可达性校验功能
│   ├── [支撑引擎: 菜单引擎] 菜单→场景可达性（菜单配置变更后校验场景是否可达）
│   │   └── 主 CU 能力类型：菜单可达性校验
│   └── [支撑引擎: 实体引擎] 场景→数据可达性（场景操作后校验产品数据是否可访问）
│       └── 主 CU 能力类型：实体访问校验
├── 使用追溯功能
│   ├── [支撑引擎: 场景引擎+实体引擎] 产品数据被谁使用（Bn 原子业务→A2 场景的反向追溯）
│   │   └── 主 CU 能力类型：实体引用追溯 | 辅助能力类型：场景引用关系
│   └── [支撑引擎: 实体引擎] 场景使用了什么数据（A2 场景→Bn 原子业务的正向追溯）
│       └── 主 CU 能力类型：实体引用查询
├── 映射同步功能
│   ├── [支撑引擎: 场景引擎+实体引擎] 场景操作→数据更新（A2 操作同步更新 Bn 产品数据）
│   │   └── 主 CU 能力类型：实体变更同步
│   └── [支撑引擎: 场景引擎+实体引擎] 数据变更→场景通知（Bn 数据变更通知相关 A2 场景）
│       └── 主 CU 能力类型：实体事件通知 | 辅助能力类型：场景事件监听
├── 权限-数据映射功能
│   └── [支撑引擎: 菜单引擎] 权限变更→数据可达性校验（角色权限变更后校验数据对象访问权限一致性）
│       └── 主 CU 能力类型：权限级联校验
└── 映射完整性校验功能
    └── [支撑引擎: 场景引擎] 双视角映射完整性检查（A2 L4/L5 与 Bn 原子业务/产品数据的映射无遗漏、无冲突）
        └── 主 CU 能力类型：场景映射校验
```

#### 引擎能力缺口——本 Skill 明确不处理

BA 节点中可能包含 wft02-biz Step 4 标注的引擎能力缺口线索。**本 Skill 对引擎能力缺口完全不处理**，具体规则：

1. **不读取**：不读取 `80-pl4eos-2-eosdata/00-origin-requirement-materials/10-raw-files/` 中的引擎能力缺口原始需求材料作为输入
2. **不判断**：不判断引擎能力是否足够支撑业务功能
3. **不阻断**：不把引擎缺口作为 SR-F 的阻断项或阶段性方案的依据
4. **不依赖**：不在 SR-F 功能接口中记录"待引擎增强依赖"
5. **不触发**：不触发 OR 预处理链（ort00/ort01/ort02/ort03）或 wft01-eng
6. **不转化**：不把引擎缺口转化为 SR-F

引擎能力缺口的正确处理链路（由 wft02-biz 负责，与本 Skill 无关）：

```
wft02-biz 引出引擎能力缺口
  → 生成引擎能力缺口原始需求材料
  → 放入 80/.../00-origin-requirement-materials/10-raw-files
  → 人类运行 ort00 / ort01 / ort02 / ort03
  → 转成正式 FR-ENG OR
  → 人类运行 wft01-eng
  → 后续进入 eng 路径
```

本 Skill 只消费 wft02-biz 产出的业务架构 BA：A2 角色场景视角 BA、Bn 产品视角 BA、A2-Bn 双视角映射、L4/L5/L6 业务对象、菜单/权限/指标/治理业务方案。

#### Q1/Q2/Q3 SR-F 匹配

在生成 SR-F 末级条目前，先按 BA 节点 + 业务对象 + 支撑引擎 + 主 CU 能力类型维度匹配已有 SR-F 节点：

| 判定 | 条件 | 动作 |
|------|------|------|
| **Q1 直接匹配** | 已有 SR-F 节点对应同一业务对象、同一支撑引擎和同一主 CU 能力类型，且功能需求一致 | 复用已有 SR-F → 追加 BA 引用 → 不重新生成 |
| **Q2 扩展匹配** | 已有 SR-F 节点对应同一业务对象、同一支撑引擎和同一主 CU 能力类型，但需扩展（新增功能项/规则） | 扩展已有 SR-F → 全量重写功能需求（非增量追加） |
| **Q3 新建** | 无可用已有 SR-F 节点 | 标记待新建 → 生成 SR-F 末级条目 |

#### SR-F 完备性校验

三流生成完毕后，校验 SR-F 产出是否覆盖 BA 节点中所有要素：

| 校验项 | 内容 |
|--------|------|
| **L4 覆盖** | BA 节点中每个 L4 场景业务是否有对应的流 1 SR-F |
| **L5 覆盖** | BA 节点中每个 L5 场景部件是否有对应的流 1 SR-F |
| **L6 覆盖** | BA 节点中每个 L6 功能元件需求是否有对应的流 2 SR-F |
| **Bn 原子业务覆盖** | BA 节点中每个 Bn 原子业务操作是否有对应的流 2 SR-F |
| **A2-Bn 映射覆盖** | BA 节点中每条 A2-Bn 映射关系是否有对应的流 3 SR-F |
| **CU 能力分解覆盖** | BA 末级对象中需系统承接的能力，是否已按支撑引擎的 CU 能力结构拆成末级 SR-F 条目 |

### Step 4 · 分类映射

基于 Step 3 已生成的功能需求，按 SR-F 分类规则执行分类映射。引用 [92-srf-classification-spec.md](92-srf-classification-spec.md)。

1. **确定第 1 层（业务定义类型）**：按业务定义类型归入（流程定义、表单定义、指标定义、工单定义、计划定义、角色编排等）。三流的产出统一进入同一 SR-F 分类树，不按流编号分区
2. **确定第 2 层（系统职责分类）**：查标准职责桶清单，匹配已定义的功能描述与流程。按系统职责聚合，不按业务对象
3. **确定第 3 层（SR-F 末级条目）**：功能需求必须落到可由单一主 CU 能力类型承接、可独立验收的末级条目
4. 执行 `复用 > 改进 > 新增` 分类宿主决策

**分类原则遵守检查：**

- [ ] 先有末级（功能定义），再做聚合（原则 1）
- [ ] 按系统职责聚合，不按业务对象（原则 2）
- [ ] 一节点只归一个主职责（原则 3）
- [ ] 非末级节点能写出一句统一定义（原则 4）
- [ ] 粒度适中（原则 5）
- [ ] 业务域只做上层分区，不替代系统职责（原则 6）
- [ ] 跨域可复用同一职责分类（原则 7）
