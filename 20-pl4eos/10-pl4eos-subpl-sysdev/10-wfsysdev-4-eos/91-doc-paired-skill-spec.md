# 文档结对设计 Skill 设计规范

> **概述**：规定文档结对设计场景下，如何将系统设计 Skill 编制为 `design`（设计开发）与 `settle`（资产维护）两个相互配合的子 Skill，并以人类审阅反馈串联状态机、回退边界和人机协作规则。
>
> **定位**：本文件是 EOS 设计线中文档结对设计 Skill 编制方法的权威规范；各具体 Skill（如 `wft01a/01b`、`wft05a/05b`）是本规范的实例化实现。
>
> **版本**：v1.15 | **修订**：2026-06-21

---

## 一、总体流程

### 核心流程

```
┌──────────────────────────────────────────────────┐
│ Phase 1 · 设计开发（创造性/分析性）                │
│                                                    │
│  入口：上游就绪节点（状态=待设计开发 + 人类同意）    │
│  核心：逐条做映射、分解、定义、完善                │
│  产出：设计方案 → 写入目标资产文档                 │
│        （节点状态=待确认方案）                     │
│                                                    │
│  ── 人类审阅反馈 ──                                │
│  [同意]（仅可确认方案）→ 状态→待进入 Skill B        │
│  [修改] → 状态→迭代中（重入 Phase 1）              │
│  [驳回] → 重做方案                                │
│                                                    │
├──────────────────────────────────────────────────┤
│ Phase 2 上半 · 资产维护（匹配/确认/写回/记录）      │
│                                                    │
│  步骤A 资产环境加载          ← 加载共享资产当前状态 │
│  步骤B 实体提取              ← 从设计方案中提取引用 │
│  步骤C 跨域资产匹配          ← Q1/Q2/Q3 找宿主     │
│  步骤D 写回主资产            ← 写入目标资产文档     │
│  步骤E 反向引用更新          ← 写回上游资产引用列表 │
│  步骤F 状态推进              ← 状态→已完成上半     │
│                                                    │
│  ── 可直接写回；若资产缺口/归属争议则进入反馈迭代 ── │
│                                                    │
├──────────────────────────────────────────────────┤
│ Phase 2 下半 · 详细定义展开                        │
│                                                    │
│  步骤G 详细条目展开         ← 架构末级→详细条目    │
│  产出：详细定义子文档草案（状态=待确认细节）         │
│                                                    │
│  ── 人类审阅反馈 ──                                │
│  [同意] → 状态→已验证冻结 → 可进入下游             │
│  [修改]/[驳回] → 重入 Phase 2 下半                 │
└──────────────────────────────────────────────────┘
```

### 核心原则

1. Skill A 每次仅处理一个上游架构末级节点。
2. Skill A 负责主结构方案形成；Skill B 负责已确认方案的正式资产维护与详细展开。
3. 人类 `[同意]` 仅放行 `可确认方案`；`阶段性方案` 不得进入 Skill B。
4. Skill B 仅处理规格问题，不得重做主结构设计；凡属结构问题，必须回退 Skill A。
5. 详细定义应遵循”消费方产出”规律，由消费该层架构的下游 Skill 在 Phase 2 下半写回。
6. **STR 架构末级节点（STR-F）指代的是场景定义**——每个 STR-F 节点对应一个角色的端到端职责闭环，与 wft02 中该角色的场景定义形成 1:1 映射。场景定义集合是 EOS 系统功能菜单的唯一设计来源。
7. **十类信息链是设计分析框架，不是场景定义的内容结构**：执行业务链、管理业务链、个人绩效链、组织绩效链、资源需求链、协同交互链、风险合规链、规则知识链、状态事件链、场景指标链，是 AI 在 wft01a 中识别信息载体与场景指标时的分析维度。场景定义的内容仍是扁平的信息载体清单与指标清单，每个载体标注归属链。
8. **业务定义与场景定义**：**业务定义**（不变）指单次业务操作，有业务产出，归属输出产品路径的各构件，分为直接产出类和管理控制类。**场景定义**（新增）指业务定义的有序序列集，不产生产出，归属角色场景产品类型。场景定义的子节点是业务定义，不能是其他场景定义。
9. **Skill 读取树形资产文档时必须遵守文档布局规范**：系统设计过程文档和资产文档应按 [文档编写约定 §十四](../../../../00-generalspec/00-doc-conventions.md) 的树形资产文档布局规范组织。Skill 默认读取树画像 / 资产全貌、状态队列、节点索引和目标正文块，不得默认全量读取大文档正文。
10. **观察类中间项必须有生命周期**：当 Skill A 因单节点上下文不足而记录缺口观察、风险观察、资产一致性观察等中间项时，不得把观察项等同于正式设计节点或待写回项。后续上游节点进入时，AI 必须复查相关未关闭观察，并将其吸收、升级、转收敛、证伪关闭或重复关闭，避免观察项长期悬空。
11. **产品数据业务定义口径**：产品数据类 BA 五级 IPO 对应的业务定义是产品数据开发管理定义，即计划定义；产品数据输出文档是该计划定义的 Output。产品数据内容如何实现不再另建“开发实现业务定义”，而是在计划定义初步方案中通过资源引用表达，如 CAX 工具、AI 工具、模板、专业能力和作业指南。CAX / AI 等资源资产由 `27` 号资源资产文档承接。
12. **业务定义与引擎配置数据关系**：每个具体业务定义背后应对应一份业务引擎配置数据；多个同类业务定义可共享同一个业务引擎和配置模板，但不能共享同一份配置数据，除非它们本来就是同一个业务定义。计划定义共享计划引擎和计划定义配置模板，但每个产品数据计划定义对应独立的计划引擎配置数据。

## 二、统一状态机

### 节点状态

| 统一状态 | 含义 | 管理阶段 |
|---------|------|---------|
| **待设计开发** | 上游就绪，等待进入 Phase 1 | Phase 1 入口 |
| **待确认方案** | Phase 1 已完成，等待人类审阅 | 人类审阅反馈（仅可确认方案可放行） |
| **迭代中** | 人类反馈修改意见，等待重新处理 | Phase 1 重入 |
| **待进入 Skill B** | 方案已确认，等待进入 Phase 2；具体 Skill 可按真实下一步命名，如 `待文档归属` | Phase 2 入口 |
| **已完成上半** | Phase 2 上半完成，进入 Phase 2 下半；具体 Skill 可命名为 `已完成资产写回` 等 | Phase 2 下半入口 |
| **待确认细节** | Phase 2 下半已完成，等待人类审阅详细定义 | 人类审阅反馈 |
| **确认中** | 人类反馈修改意见，等待重新处理详细定义 | Phase 2 下半重入 |
| **已验证冻结** | 全链验证通过 | 终态 |
| **已废弃** | 节点停止使用 | 终态 |

### 人类操作类型

| 人类操作 | 作用 | 发生位置 |
|---------|------|-----------|
| 同意设计开发 | 将节点选入 Phase 1 | 上游状态表 |
| `[同意] / [修改] / [驳回]`（方案审阅） | 决定当前节点进入 Skill B，或留在 Skill A 重入 | 目标资产文档的反馈区 |
| `[同意] / [修改] / [驳回]`（细节审阅） | 决定当前节点冻结完成，或留在 Skill B 重入 | 详细定义/规格文档的反馈区 |
| 同意验证 | 将节点选入下游验证 | 下游状态表或入口文档 |

## 三、全链路映射

| 步骤 | Skill A 处理对象 | Skill B 上半 | Skill B 下半 | 详细定义写入 |
|------|------|------|------|------|
| `wft01` | OR 末级需求 → STR 节点（即场景定义），用十类信息链识别广义文档/信息载体/指标，形成扁平清单 | 文档归属、资产建议与写回 | 不适用 | 由 `wft02` 产出 |
| `wft02` | STR 架构末级（场景定义及信息载体清单）→ 展开为完整业务定义 + 场景定义编排 | 匹配、写回、派生回写 | STR 详细条目展开 | `03-*.md` |
| `wft03` | BA 末级 IPO → SysReq 节点 | 分类宿主匹配与写回 | 不适用 | 由 `wft05` 产出 |
| `wft04` | STR-NFR 末级条目 → SysReq-NFR | 指标匹配与写回（22-*.md） | STR-NFR 详细条目展开 | `03-*.md`（NFR 部分） |
| `wft05` | SysReq 末级 IPO → PA 组件 | 跨域匹配、写回、反向引用 | SR 详细定义 + PA 组件规格 | `06-*.md` / `07-*.md` |

规范说明：

1. `wft01 / wft03` 无 Phase 2 下半；`wft04` 有 Phase 2 下半（wft04b 下半写回 STR-NFR 详细定义至 `03-*.md`）。
2. `wft02 / wft05` 是消费方，因此负责在 Phase 2 下半写回详细定义。
3. 当前完整实例是 `wft05a/05b`，可作为 `wft01~04` 的直接参照。

推导依据：

1. `STR` 架构由 `wft01` 产出，但 `STR` 详细定义只有在 `wft02` 消费 `STR` 去设计 `BA` 时，才能确定应细化到什么粒度，因此由 `wft02` 写回。
2. `SR` 架构由 `wft03` 产出，但 `SR` 详细定义只有在 `wft05` 消费 `SR` 去设计 `PA` 时，才能确定应细化到什么粒度，因此由 `wft05` 写回。
3. `wft04` 对应的 NFR 详细内容走三条路径：（a）量化指标与约束表写入 `05-*.md` 架构末级节点；（b）确认的具体需求条目写回 `22-eos-nfr-taxonomy.md`（wft04b 上半）；（c）STR-NFR 详细定义展开至 `03-*.md` 的 NFR 部分（wft04b 下半）。与 `wft02` 写回 STR-F 详细定义对称。

**STR 架构与业务定义体系的说明**：

1. **STR-F 末级节点 = 场景定义**。每个 STR-F 节点指代一个角色的端到端职责闭环，与 wft02 中该角色的场景定义形成 1:1 映射。STR 树的角色组织方式直接对应系统功能菜单的结构。

2. **wft01a 的职责范围**：
   - 从 OR 语义中识别场景定义需要哪些广义文档、信息系统记录、指标报表、绩效数据、资源视图、规则知识、状态事件等信息载体
   - 以十类信息链为分析框架，引导全面扫描，确保不遗漏
   - 只形成方案级的信息载体清单、指标清单、推理依据与待确认项；不正式写回 `22/23/26` 资产
   - 每个载体标注归属链，形成扁平的信息载体清单
   - wft01a 不展开业务定义的 I/P/O——那是 wft02 的工作

3. **wft02 的两条路径分工不同**：
   - **3A 输出产品路径（直觉驱动）**：接收 wft01a 的信息载体清单（含归属链标签），匹配 24 资产业务定义，展开完整 I/P/O，定位到业务域→产品类型→构件。归属链标签指引 3A 路由：执行业务链→对应构件、管理业务链→项目管理构件、绩效链→绩效管理构件
   - **3B 角色场景路径（规则驱动）**：基于 STR-F 节点的功能说明，将 3A 已产出的业务定义按自然流向编排为场景定义的四条链。场景定义是业务定义的有序序列集，子节点不能是其他场景定义

4. **十类信息链作为设计分析框架**：
   - **执行业务链**：引导 AI 扫描 OR 中与业务操作直接相关的文档/表单——这些最终会展开为直接产出类业务定义，归各业务构件
   - **管理业务链**：引导 AI 扫描 OR 中与指标/绩效/进展监控相关的报表——展开为管理控制类业务定义，归项目管理构件
   - **个人绩效链**：引导 AI 扫描与个人绩效评估相关的信息载体
   - **组织绩效链**：引导 AI 扫描与部门/组织绩效评估相关的信息载体
   - **资源需求链**：引导 AI 扫描与资源申请、占用、释放、查看相关的信息载体
   - **协同交互链**：引导 AI 扫描跨角色、跨组织、跨系统协同过程中产生的信息载体
   - **风险合规链**：引导 AI 扫描审批、授权、审计、异常、内控、法规合规相关的信息载体
   - **规则知识链**：引导 AI 扫描制度、作业规则、判断标准、知识库等信息载体
   - **状态事件链**：引导 AI 扫描由信息系统产生的状态流转、事件日志、操作记录、消息通知等载体
   - **场景指标链**：引导 AI 扫描本场景定义自身需要管理和验证的指标
   - 以上十链是 wft01a 的分析维度，不是场景定义的内容结构。场景定义的内容是扁平的，每个信息载体标注其归属链标签。

## 四、人机交互

### 各阶段的人类操作

| 阶段 | 人类操作 | 在哪个文档操作 | 操作内容 |
|------|---------|--------------|---------|
| **Phase 1 前** | 选材 | 上游状态表（如 05-*.md SysReq 状态表） | 设 人类决策=同意设计开发 |
| **Phase 1 后** | 审阅方案 | 目标资产文档（如 07-*.md §X.6 反馈区） | 写 [同意]/[修改]/[驳回] |
| **Phase 2 上半中** | 视情况审阅资产建议 | 目标资产文档的反馈区 | 若存在资产缺口、归属争议或新增建议，写 [同意]/[修改]/[驳回]；若全部可直接匹配，可由 AI 自动写回 |
| **Phase 2 下半后** | 审阅详细定义 | 详细定义文档（如 06-*.md / 03-*.md） | 写 [同意]/[修改]/[驳回] |
| **全链路完成后** | 选入下一站 | 目标资产文档 §1.1 状态表 | 按下游 Skill 的入口规则设置对应人类决策 |

### Phase 1 [修改]/[驳回] 后

若当前节点收到 `[修改]` 或 `[驳回]`，该节点应进入 Skill A 重入，不得改为重新选入其他节点。仅当当前节点完成方案审阅放行后，后续节点方可进入 Skill A。

### 交互约定

- 人类反馈应按**组件/节点级别**标注；同一批次中不同节点可具有不同反馈。
- AI 应在 Step Start 中检测反馈区内容，并自动同步状态表；人类不直接编辑状态表。
- Phase 2 上半不是必然无须审阅：直接匹配时可自动写回；出现资产缺口、归属争议、产品类型/构件类型新增建议时，必须进入人类反馈迭代。

### Step Start 活动输入节点锁定协议

所有文档结对 Skill A 都应保证同一时间只有一个上游输入节点处于活动处理状态。活动输入节点未处理结束前，不得让新的上游输入节点进入同一 Skill A。

#### 活动输入节点判定

以下任一情况成立时，应视为存在活动输入节点：

1. 目标资产文档中已存在引用某个上游输入节点的方案节点，且该方案节点仍处于 `待确认方案`、`迭代中`、`确认中`、`待补充` 或其他未完成状态。
2. 状态队列、反馈区、人类决策字段、待处理选择记录或运行摘要中存在该输入节点的待办锁定记录。
3. 上游状态队列中该输入节点已被标记为进入当前 Skill A，但尚未记录当前 Skill A 处理完成。

#### 活动输入节点完成判定

以下任一情况成立时，活动输入节点才可视为完成或释放：

1. 对应方案已达到当前 Skill A 的完成态，并已被人类确认进入 Skill B。
2. 该输入节点被人类明确撤销、废弃、跳过或退回上游。
3. 当前 Skill A 已写入可被 Step Start 识别的完成记录或释放记录。

#### 锁定规则

1. Step Start 必须先检查是否存在活动输入节点，再检查新的待进入节点。
2. 若存在一个活动输入节点，AI 只能重入该节点、处理该节点反馈、输出该节点当前待办，或提示进入正确的下游 Skill；不得选择新的输入节点。
3. 若存在多个活动输入节点，说明状态队列不一致，AI 必须停止新增处理，输出状态冲突说明，并要求人类先选择保留一个活动输入节点。
4. 只有不存在活动输入节点时，AI 才能从待进入队列中选择一个新的输入节点。
5. 新输入节点一旦进入 Skill A，AI 必须在状态队列、目标方案节点、待办记录或运行摘要中记录活动锁定关系：`活动输入节点ID -> 目标方案节点ID / 当前待办ID`。

### Step End 单 ID 待办协议

所有文档结对 Skill 的 Step End 都应把本轮运行结果转换为人类下一次运行前的最小待办。最小待办不是“列出所有事情”，而是为人类确定一个当前应处理的 ID，并说明剩余队列规模。

#### 单 ID 原则

1. 每次 Step End 只提示人类处理一个 `当前待办ID`。
2. `当前待办ID` 可以是一个方案 ID、资产建议 ID、详细定义 ID、待进入 Skill A/B 的节点 ID、回退反馈 ID，具体类型由实例化 Skill 定义。
3. Step End 必须同时输出 `剩余待办ID数量`，让人类知道队列规模；若无法精确统计，应写 `至少 N 个` 并说明统计口径。
4. AI 可以附带其他待办摘要，但摘要只能给数量和类型，不得要求人类同时审阅、反馈或选择多个 ID。
5. 人类对当前 ID 的回复应是一句话完成一个动作，例如 `同意`、`修改：……`、`驳回：……`、`选择：……`。

#### 当前待办 ID 连续性

1. Step End 输出的 `当前待办ID` 默认形成“待办锁定”。
2. 若下一次运行时该 ID 仍处于待反馈、待确认、待补充或待选择状态，AI 必须继续提示该 ID，不得因为出现新的待办而切换。
3. 只有当前 ID 已完成、被人类明确切换，或因状态变化不再可操作时，AI 才能选择下一个 ID。
4. 若人类回复 `换一个 / 跳过 / 选择：<另一个ID>`，AI 可以切换当前待办 ID，但必须先记录切换原因。
5. 若当前 ID 已经完成，AI 才从剩余待办中选择下一个 ID。
6. 若当前 ID 因上游状态变化不再可操作，AI 必须说明原 ID 不再可操作的原因，再选择新的当前待办 ID。

#### 待办状态留痕

1. AI 应在反馈区、状态队列、人类决策字段、待处理选择记录或本轮运行摘要中保留当前待办 ID 的状态。
2. 对话是反馈输入通道，文档中的反馈区 / 状态队列 / 决策字段是权威留痕。
3. AI 不得只凭对话上下文推进状态；必须先将人类回复转写为规范记录，再继续执行方案修改、状态推进、资产写回、回退或下游提示。

#### 最小待办输出格式

Step End 推荐采用以下最小格式；具体 Skill 可增加必要字段，但不得删除核心字段：

```text
当前结果：
当前待办ID：
是否延续上次待办ID：
剩余待办ID数量：
待办类型：
为什么先处理这个ID：
人类只需回复：
回复会写入：
下次运行：
其他待办摘要：
```

字段约束：

1. `当前待办ID` 必须是单个 ID，不得写成多个 ID 列表。
2. `是否延续上次待办ID` 必须写 `是 / 否`；若为 `否`，必须说明当前 ID 已完成、被人类切换或不再可操作的原因。
3. `剩余待办ID数量` 必须是数字；若无法精确统计，应写 `至少 N 个` 并说明统计口径。
4. `人类只需回复` 必须是一句话，且只包含当前 ID 的动作。
5. `其他待办摘要` 只能给数量和类型，不得要求人类同时处理。

### 跨 Skill 入口选择与活动锁转交协议

所有文档结对 Skill 在从当前 Skill 交接到下游 Skill 时，必须区分“状态到位”和“交接完成”。状态到位只表示节点具备进入下游的业务前提；交接完成必须有可观察的选择记录或活动锁释放 / 转交记录。

#### 基本原则

1. 节点状态达到下游入口状态，不等于下游 Skill 可以立即自动处理。
2. 当前 Skill 的活动锁在节点状态到位后仍保持，直到人类对当前待办 ID 明确回复 `选择进入<下游Skill>`，或已有等价的活动锁释放 / 转交记录。
3. 下游 Skill 的 Step Start 必须能识别上游写入的 `选择进入<下游Skill>` 记录或活动锁释放 / 转交记录，并将其视为入口选择记录。
4. 下游 Skill 不得要求人类对同一 ID 重复确认；若上游已经完成选择 / 转交，下游应直接承接该 ID。
5. 若节点状态已到位但没有选择 / 转交记录，即使只有一个候选节点，下游 Skill 也不得自动进入处理，应输出 `等待选择进入<下游Skill>`，当前待办 ID 为该节点或建议方案包 ID。
6. 若存在多个入口许可节点且不存在活动锁，下游 Skill 必须输出 `等待选择进入<下游Skill>`，每次只提示一个推荐 ID。
7. 若存在一个活动节点未完成，下游 Skill 不得响应其他入口许可节点，即使其他节点也已状态到位并具备选择记录。

#### 入口许可定义

下游 Skill 的入口许可节点至少包括以下两类：

1. 状态队列中达到下游入口状态，且具有人类入口许可或选择记录的节点。
2. 上游 Skill 已写入 `选择进入<下游Skill>`、活动锁释放记录或活动锁转交记录的节点。

示例：

1. `wft01b → wft02a`：`待分解分配` 只是 STR 节点完成文档归属后的下游准备态；`选择进入wft02a` 或 `wft01b` 活动锁释放 / 转交记录才是 `wft02a` 可识别的入口选择记录。
2. `wft02a → wft02b`：`待资产写回` 只是 BA 主结构方案确认后的下游准备态；`选择进入wft02b` 或 `wft02a` 活动锁释放 / 转交记录才是 `wft02b` 可识别的入口选择记录。

#### 转交记录要求

上游 Skill 在收到人类 `选择进入<下游Skill>` 后，必须先转写到反馈区、状态队列、待处理选择记录或本轮运行摘要，再提示运行下游 Skill。

转交记录至少包含：

1. 上游活动节点 ID
2. 下游待处理节点或建议方案包 ID
3. 当前状态到位说明
4. 人类选择动作：`选择进入<下游Skill>`
5. 活动锁状态：`已释放` 或 `已转交<下游Skill>`
6. 释放 / 转交时间或轮次
7. 下游 Skill 名称

#### 下游 Step Start 要求

下游 Skill 的 Step Start 必须按以下顺序处理入口：

1. 先检查是否存在本 Skill 的活动节点。
2. 若存在活动节点，只处理该活动节点，不得进入新节点。
3. 若不存在活动节点，检查是否存在上游选择 / 转交记录。
4. 若有且只有一个入口许可节点，直接进入本 Skill，并写入本 Skill 的活动锁。
5. 若存在多个入口许可节点，输出 `等待选择进入<本Skill>`，每次只提示一个当前待办 ID。
6. 若只有一个状态到位节点但没有选择 / 转交记录，输出 `等待选择进入<本Skill>`，不得自动进入。

## 五、文档结对设计 Skill 编制规范

本节规定 `wft01~05` 拆分为 `design / settle` 后的统一交互与状态处理规则。凡后续 `wft01a/b ~ wft05a/b` 的实现、修订与评审，均应优先遵守本节。

### 1. 适用范围

本规范适用于瀑布式系统设计各步骤在拆分为两个 Skill 后的统一交互与状态处理方式：

- `Skill A = design`
- `Skill B = settle`

其中：

- `Skill A` 负责方案形成、设计判断、人类方案审阅
- `Skill B` 负责资产维护、详细展开、人类细节审阅

### 2. 统一职责边界

#### 2.1 Skill A（design）

Skill A 负责当前步骤目标节点的主结构方案形成，不负责正式资产维护及开发级详细展开。

其核心职责是：

1. 识别目标节点
2. 加载上游承接链与相关约束资产
3. 形成方案
4. 判断方案状态
5. 提交人类进行方案审阅

#### 2.2 Skill B（settle）

Skill B 负责将 Skill A 已确认的主结构方案正式写回资产体系，并在需要时展开为详细定义或需求规格。

其核心职责是：

1. 接收已确认方案
2. 做资产匹配、写回、反向引用更新
3. 在需要时展开详细定义和规格
4. 提交人类进行细节审阅

#### 2.3 总边界

1. Skill A 负责决定“谁来承接、如何切分边界、是否复用”。
2. Skill B 负责将已确认的承接结构正式写回并展开细化。
3. Skill B 不得重做 Skill A 已确认的主结构设计。
4. Skill B 发现结构问题时，必须回退 Skill A。

#### 2.4 编制规则沉淀（wft01a/b）

基于 `wft01a-or2str-design` 与 `wft01b-or2str-settle` 的编制经验，后续 Skill 编制应遵守以下规则：

1. **状态名要表达真实下一步动作**：不要用过宽泛的状态名掩盖业务动作。例如 wft01 中 Skill A 放行后的下一步是“文档归属”，因此可命名为 `待文档归属`，而不是只写成泛泛的“待进入下一步”。统一状态只定义语义骨架，实例状态应按真实动作具体化。
2. **入口条件必须是可观察状态，不使用伪条件**：不得把“上一步已提示运行某 Skill”“已完成某说明”作为入口条件。入口应来自状态表、反馈区、方案区是否存在等可检查事实。
3. **反馈格式必须可解析**：人类反馈区应使用 `[同意] / [修改] / [驳回]`，或等价的结构化字段如 `[操作] 修改`。跨 Skill 回退时，回退方必须写入能被上游 Step Start 识别的反馈，而不是只写自然语言说明。
4. **重复运行必须有保护**：若方案已存在且反馈区为空，Skill 不得覆盖既有方案，应输出“等待人类反馈”；若反馈区已有意见，只处理反馈；若状态已进入下游，应停止本 Skill 并提示正确下一步。
5. **Step 布局总览必须列出 End**：即使运行结束输出独立成章，Step 布局总览也应保留 `End` 行，并注明“独立成章，见运行结束输出章节”。这样人类能在总览中看到完整运行闭环。
6. **Step End 必须给出人类可执行动作**：尾部输出不能只说“完成/等待”。必须明确本轮结果、人类现在能做什么、在哪个文档/章节做、用什么格式写反馈、下一步可选择哪个 STR 节点或下游 Skill。
7. **A/B 边界要按决策性质划分**：Skill A 负责场景边界、信息载体、指标、推理依据和待确认项；Skill B 负责基于已确认方案做文档归属、资产匹配、资产新增建议和写回。B 若发现 A 的场景边界或信息载体清单需要改，应回退 A，而不是在 B 内重做。
8. **AI 推理要显性化**：当 OR 没有触发词时，AI 仍应基于业务常识、系统设计、企业治理、同类场景经验进行补全，但必须标注来源类型、推理依据和确认状态，避免把推断伪装成原始需求明示内容。
9. **分析框架不得误写成资产结构**：信息链、判断维度、检查清单是 AI 的工作框架，不等同于最终节点层级。最终写入资产时，应转换为该资产要求的扁平清单、状态表、定义表或追溯关系。
10. **Skill B 可以有人类迭代**：如果资产中没有合适的业务域、产品类型、构件类型或文档定义，Skill B 应提出建议并等待人类反馈。不能默认 settle 阶段永远是无审阅的机械写回。

### 3. Skill A 的统一输入处理

#### 3.1 输入处理步骤

每个 Skill A 应按以下顺序处理输入：

1. 接收一个目标节点
2. 检查该节点状态是否允许进入 Skill A
3. 检查是否存在其他节点占用当前方案审阅前工位
4. 加载相关文档的树画像 / 资产全貌和状态队列
5. 通过节点索引 / 分片索引定位目标正文块
6. 加载目标节点自身内容
7. 加载上游追溯链
8. 加载与本步相关的约束资产全貌、候选分片摘要和必要正文块
9. 加载已有目标资产
10. 若为重入，加载反馈区的人类意见
11. 执行输入预检并形成处理前结论

#### 3.2 输入包定义

Skill A 的标准输入包为：

1. 目标节点
2. 上游追溯链
3. 相关约束资产
4. 已有目标资产
5. 重入反馈
6. 树画像 / 资产全貌、节点索引和目标正文块定位信息

#### 3.3 输入预检要求

Skill A 在进入正式设计前，必须先判断：

1. 输入是否完整
2. 上游追溯链是否成立
3. 是否存在阻断项
4. 本轮是否只能形成阶段性方案
5. 本轮是否具备进入 Skill B 的前提

补充要求：

1. 输入预检发现缺口时，应先报告缺口，再进入正式设计。
2. 常见缺口至少包括：依附关系缺口、关键约束缺口、追溯链缺口、必要资产缺口。

### 4. 设计判断

> 本节描述 Skill A 在形成方案时的核心思维过程——它回答的是"输入处理之后、方案写入之前，设计判断如何运作"的问题。这个过程不是线性步骤，而是同时在多个维度上进行的判断循环。
>
> 本节与 §2（职责边界）配合：§2 规定 Skill A 负责"形成方案"和"判断方案状态"，本节展开这个职责内部的思维过程。与 §11（审查口径）配合：方案审阅所审的"主结构是否成立"可引用本节 4.4 和 §5 的内容作为判断支撑。

#### 4.1 设计的两种工作方式

Skill A 进入设计活动后，存在两种本质上不同的工作方式：

| 方面 | 规则驱动（模式匹配） | 直觉驱动（分解分配分析） |
|------|---------------------|------------------------|
| 决定关系 | 输入 → 规则 → 输出 | 输入 + 资产 + 理解 → 方案 |
| 确定性 | 确定性：不同执行者结果趋于一致 | 非确定性：存在多种合理方案 |
| 可自动化程度 | 高 | 低 |
| 人类参与模式 | **审核确认**：人类审方案是否按规则执行完整、映射是否正确 | **深度参与**：人类判断边界是否合理、风格是否一致、方案是否收敛，需多轮迭代 |
| 迭代次数预期 | **单轮**：AI 产出 → 人类审核 → 确认/微调 → 放行 | **多轮**：AI 初案 → 人类反馈 → AI 调整 → 人类再判断 → 直到收敛 |
| 典型实例 | 业务定义类 IPO → 1+7 组件结构 | 引擎配置类 IPO → 前后端组件划分 |

说明：§3 的输入预检完成后，Skill A 进入上述两种方式之一。规则驱动部分按既有模式执行即可；直觉驱动部分遵循 4.2~4.4 的描述。Skill 编制者在设计 Step 3 时，应先按 4.1.1 的条件判断当前设计活动应归入哪种方式，再按相应方式编写。

##### 4.1.1 工作方式的选择条件

> **适用对象**：Skill 编制者。在编写某个 Skill 的 Step 3（设计活动）时，用以下条件判断"这个设计活动应该写成规则驱动的模式匹配，还是需要写成直觉驱动的分解分配分析框架"。

**条件一：输出空间是否已知**

| 规则倾向 | 直觉倾向 |
|---------|---------|
| 你能提前列出所有可能的输出类型/结构 | 你无法穷举输出可能长什么样 |

判断方法：问自己"如果做 10 次，输出结构是否大致相同？"——是则规则驱动，否则直觉驱动。

**条件二：从输入到输出是否存在固定映射路径**

| 规则倾向 | 直觉倾向 |
|---------|---------|
| 存在确定的翻译/转换规则：输入的 X 字段对应输出的 Y 结构 | 不存在逐字段映射，需要理解问题域后重新组织 |

判断方法：问自己"能否用一张映射表说明输入字段和输出结构的关系？"——能则规则驱动，不能则直觉驱动。

**条件三：方案质量是否可规则验证**

| 规则倾向 | 直觉倾向 |
|---------|---------|
| 可以用检查清单验证"做得对不对"（字段齐全、格式正确） | 质量依赖于人类判断"合不合理"（边界恰当、风格一致） |

判断方法：问自己"能否编写一份完整的通过/不通过检查清单来验证方案正确性？"——能则规则驱动，不能则直觉驱动。

**条件四：方案是否强依赖资产语境**

| 规则倾向 | 直觉倾向 |
|---------|---------|
| 方案内容主要由输入决定，资产仅作为引用参考 | 方案内容必须结合当前资产状态来决定——已有组件长什么样直接影响本次设计 |

判断方法：问自己"如果换一套资产，同样的输入是否会产出不同的方案？"——不会则规则驱动，会则直觉驱动。

**综合判断方法**：采取简单多数原则——四个条件中 ≥2 条指向直觉驱动，则该设计活动应按直觉驱动方式编制；≤1 条指向直觉驱动，可按规则驱动方式编制。最终决定因素是条件一——如果输出空间根本不可穷举，则必然为直觉驱动。

**选择结果对 Skill 结构的影响**：

| 差异项 | 规则驱动 | 直觉驱动 |
|--------|---------|---------|
| Step 3 的写法 | 写为确定的步骤 + 映射规则 | 写为判断框架 + 收敛方向指引 |
| 判断框架模板 | 不适用 | **四维框架：看/守/交/完**。各 Skill 填充各自的"看什么、守什么、交什么、完标准" |
| 人类交互设计 | 人类审"是否按规则执行完整" | 人类审"边界、风格、方案是否合理" |
| 迭代设计 | 不单独设计迭代路径，走统一方案审阅重入即可 | 需主动设计迭代循环的预期次数和收敛条件 |
| Step End 重点 | "映射完整，可进入 B" | "方案已形成，人类请判断以下关键点……" |
| 编制风险 | 模式误用：把直觉题当规则题做，导致方案僵硬 | 过度发散：没有收敛锚点，迭代无法结束 |

#### 4.2 分解分配分析的判断结构

分解分配分析的核心特征是**并行判断循环**——分解方案和分配方案同时形成、互相影响，不存在固定的先后顺序：

```
上游末级节点
     │
     ├── 分解判断：按什么维度拆？拆成几个部分？
     │       ↕  同时进行，互相约束
     ├── 分配/匹配判断：已有宿主？可扩展宿主？需新增？
     │
     └── 资产感知判断：新增组件是否与现有资产风格一致？
```

三个判断维度的具体含义和牵制关系：

| 判断维度 | 核心问题 | 牵制关系 |
|---------|---------|---------|
| **分解判断** | 拆成哪些条目？划分逻辑是什么？ | 受限于宿主是否存在、可扩展性——无宿主的条目需要调整分解方案 |
| **分配/匹配判断** | 每条需求由哪个组件承接？复用 / 改进 / 新增？ | 受限于分解方案——不同分解方式产生不同的分配路径 |
| **资产感知判断** | 新增/扩展的组件是否与现有资产协调？ | 影响分解方式和分配方案——避免产生与现有资产脱节的组件 |

三者不是先 A 后 B 再 C 的关系。设计者实际上是在三者间快速跳跃：试探一个分解方案 → 检查是否有宿主 → 发现没有合适宿主或宿主风格不匹配 → 调整分解方案 → 再匹配 → 直到三者收敛为一个可接受的平衡点。

#### 4.3 收敛方向判断

分解分配分析的终点不是"唯一正确答案"——流程设计不存在唯一解，而是在**多个可行方案中选取一个合理的收敛点**。

典型情况下的判断倾向：

| 情况 | 合理倾向 |
|------|---------|
| 有明确宿主可承接 | 优先分配，不做多余分解 |
| 有可扩展的已有宿主 | 检查扩展边界是否合理，合理则扩展后承接；不合理则回到分解判断 |
| 需要新增组件 | 参考同层同类组件的风格，保持资产一致性 |
| 多组件可承接同一条目 | 检查边界是否重叠，调整分配方向 |
| 条目无任何宿主 | 回到分解判断，重新考虑条目划分逻辑 |

#### 4.4 方案状态的形成逻辑

§2 和 §6 规定了"方案状态"（阶段性方案 / 可确认方案）影响路由和人类放行，但方案状态不是方案的自有属性——它是**设计者对当前判断信心的综合表达**。

**判定为"阶段性方案"的典型信号：**

1. 存在未闭合的承接关系（部分条目无宿主或扩展路径不明确）
2. 存在阻断型 NFR 约束未满足或未完成映射
3. 缺少关键约束资产，无法做出准确判断
4. 新增组件与现有资产风格差异较大，无法确认是否合理
5. 方案刚形成，尚未经过任何迭代检视

**判定为"可确认方案"的典型信号：**

1. 所有需求条目有明确宿主或可验证的扩展路径
2. NFR 约束已映射完成或已明确标记为"可接受风险"
3. 必要约束资产已加载并纳入设计考量
4. 组件风格与现有资产协调，无不合理处
5. 方案经过了至少一轮内部检视或迭代验证

**信号不一致时的处理规则**：当上述信号不一致时（例如承接关系闭合但 NFR 存疑），设计者应取保守判断——标记为阶段性方案，将缺口作为阻断项一并提交人类审阅。

### 5. 资产感知与风格一致性

> 本节与 §2.3（总边界第3项"Skill A 负责决定是否复用"）配合。是否复用的判断不仅取决于功能匹配度，还取决于新增组件是否能与现有资产自然融合。这种融合判断的能力称为资产感知。

#### 5.1 资产感知的定义

**资产感知**是指设计者对已有资产体系的理解程度，体现在新增或扩展节点时能自然融入现有结构，而不是以孤立或不协调的方式存在。

#### 5.2 风格一致性的检查维度

资产感知无法用规则穷举，但可以转化为以下检查维度作为判断框架：

| 维度 | 判断问题 |
|------|---------|
| **粒度** | 新增组件的粒度是否与同层同类组件一致？（不过粗也不过细） |
| **命名** | 组件名称的命名模式是否与同类组件一致？ |
| **边界模式** | 职责边界的划分逻辑是否与同域组件一致？ |
| **层级位置** | 新增组件是否放在正确的架构层级？ |
| **依赖关系** | 新增组件对上下游的依赖方式是否与同类一致？ |

#### 5.3 资产感知在设计判断中的贯穿作用

资产感知不是独立的设计步骤，而是贯穿 §4.2 中三个判断维度的底层能力：

- **在分解判断中**：了解资产中同类节点的分解深度和惯用方式，使新节点的分解方案与现有资产自然对齐
- **在分配/匹配判断中**：感知资产的组件风格偏好，新增节点时自然模仿同层同类组件的形式和规格
- **在方案审阅中**：作为"主结构是否成立"的软性判断维度——结构成立不仅是逻辑正确，还包括资产体系的内在协调一致性

### 6. Skill A 的运行结束输出（Step End）

#### 6.1 输出目标

Skill A 的 Step End 必须明确告知人类：

1. 当前方案是什么状态
2. 为什么是这个状态
3. 下一步预定进入哪个 Skill
4. 人类现在该做什么
5. 人类可以通过对话反馈，还是需要直接编辑文档

#### 6.2 固定输出项

每个 Skill A 在 Step End 必须输出：

1. 方案状态
2. 下一步路由
3. 路由原因
4. 关键方案摘要
5. 阻断项清单
6. 当前应执行动作
7. 重点查看位置
8. 反馈入口说明：优先提示人类可在预览模式阅读文档后，通过对话给出 `[同意] / [修改] / [驳回]` 及具体意见；同时保留文档反馈区作为备选方式。
9. 对话转写说明：若人类通过对话反馈，AI 必须先将对话内容整理为目标文档反馈区的规范记录，再执行状态推进、方案修改、回退或下游运行。
10. 单 ID 待办说明：必须遵守 §四“Step End 单 ID 待办协议”，输出当前待办 ID、是否延续上次待办 ID、剩余待办 ID 数量和人类只需回复的一句话动作。

#### 6.3 路由规则

Skill A 的 Step End 必须输出唯一预定路由：

1. `重新运行 wftXa`
2. `运行 wftXb`

其中：

1. 若方案仍存在阻断项，或当前仅为阶段性方案，则预定路由为 `重新运行 wftXa`。
2. 若方案已达到可确认状态，则预定路由为 `运行 wftXb`。

#### 6.4 放行限制

仅当满足以下条件时，Skill A 才允许进入 Skill B：

1. 当前方案状态为 `可确认方案`
2. 人类给出 `[同意]`

若当前方案状态为 `阶段性方案`，则 `[同意]` 不构成有效放行。

#### 6.5 对话反馈处理规则

Step End 应把“对话反馈”作为推荐入口。人类可以不直接编辑 Markdown，而是在预览模式阅读方案后，于对话中说明操作和意见。

AI 收到对话反馈后必须按以下顺序处理：

1. 识别反馈对象、操作类型和具体意见。
2. 将对话反馈转写为目标文档 `### 反馈区` 或等价反馈区中的规范记录。
3. 再按反馈区规则执行方案修改、状态推进、回退或下游运行。

约束：

1. 对话是输入通道，文档反馈区是权威留痕。
2. AI 不得只凭对话上下文推进状态，必须先完成反馈区转写。
3. Step End 中仍应给出文档反馈区位置，供人类需要时手工填写。

### 7. Skill B 的统一输入处理

#### 7.1 输入处理步骤

每个 Skill B 应按以下顺序处理输入：

1. 接收一个目标节点
2. 检查该节点状态是否允许进入 Skill B
3. 加载相关文档的树画像 / 资产全貌和状态队列
4. 通过节点索引 / 分片索引定位已确认方案正文块
5. 加载 Skill A 已确认的主结构方案
6. 将该方案视为冻结输入
7. 加载当前资产环境的全貌层、候选分片摘要和必要正文块
8. 按上半 / 下半进行路由
9. 若为重入，加载反馈区的人类意见

#### 7.2 输入包定义

Skill B 的标准输入包为：

1. 已确认主结构方案
2. 当前资产环境
3. 重入反馈
4. 树画像 / 资产全貌、节点索引和目标正文块定位信息

#### 7.3 冻结原则

Skill B 输入中的主结构方案视为冻结内容，至少包括：

1. 组件清单
2. 组件类型
3. 组件边界
4. 宿主分配
5. 复用 / 改进 / 新增 决策
6. 主承接关系

### 8. Skill B 的运行结束输出（Step End）

#### 8.1 输出目标

Skill B 的 Step End 必须明确告知人类：

1. 本轮结果是什么
2. 当前节点该回 A、留在 B，还是已完成
3. 人类现在该做什么
4. 应重点看哪份文档的哪一部分
5. 人类是否可以通过对话反馈，以及 AI 如何转写留痕

#### 8.2 三类标准结果

每个 Skill B 的 Step End 都应归入以下三类之一：

1. `已回退 Skill A`
2. `待确认细节`
3. `已验证冻结`

#### 8.3 固定输出项

每个 Skill B 在 Step End 必须输出：

1. 当前处理结论
2. 当前节点路由
3. 路由原因
4. 当前应执行动作
5. 重点查看位置
6. 反馈入口说明：优先提示人类可在预览模式阅读文档后，通过对话给出 `[同意] / [修改] / [驳回]` 及具体意见；同时保留文档反馈区作为备选方式。
7. 对话转写说明：若人类通过对话反馈，AI 必须先将对话内容整理为目标文档反馈区的规范记录，再执行资产写回、回退、状态推进或下游提示。
8. 单 ID 待办说明：必须遵守 §四“Step End 单 ID 待办协议”，输出当前待办 ID、是否延续上次待办 ID、剩余待办 ID 数量和人类只需回复的一句话动作。

#### 8.4 路由规则

1. 若发现结构问题，应回退 Skill A。
2. 若主结构成立但细节仍待确认，应重入 Skill B。
3. 若详细定义与规格均通过确认，则当前节点完成。

#### 8.5 节点完成后的可选动作

仅当节点已 `已验证冻结` 时，才允许在流程层面提示多个后续可选动作，例如：

1. 进入下一步
2. 回到 Skill A 选择下一个节点

#### 8.6 对话反馈处理规则

Skill B 的 Step End 应与 Skill A 保持一致：人类可通过对话提交资产归属、细节确认、修改或驳回意见；AI 必须先转写到对应反馈区，再继续执行。

典型对话反馈包括：

1. `同意`
2. `修改：将 IC-02 的产品类型改为……`
3. `驳回：这个文档不属于当前场景，请回退 Skill A`

AI 处理约束与 §6.5 相同：对话是输入通道，文档反馈区是权威留痕。

### 9. 统一回退规则

#### 9.1 回退触发条件

Skill B 一旦发现以下问题触及主结构，必须回退 Skill A：

1. 组件是否存在需要改
2. 组件类型需要改
3. 组件边界需要重切
4. 宿主分配需要重做
5. 复用 / 改进 / 新增 决策需要推翻
6. 主依附关系不成立
7. 阻断型约束导致主结构失效

#### 9.2 回退动作

回退时必须执行：

1. 停止当前 Skill B
2. 不继续进入下半或后续审阅
3. 状态回退到 Skill A 重入状态
4. 写入回退记录
5. 输出唯一回退路由

补充要求：

1. 若结构问题在 Skill B 已正式写入主资产后才被发现，必须先撤销本轮正式写回结果，再执行回退。
2. 撤销范围至少包括：主资产正式写入、状态成功推进、反向引用正式更新。

#### 9.3 回退记录位置

回退原因的权威记录位置统一为：

目标主资产文档中对应节点的反馈区 / 审查区。

状态变化写入状态表，详细原因写入反馈区。

### 10. 统一状态原则

#### 10.1 Skill A 状态原则

Skill A 至少应覆盖：

1. 首次进入状态
2. 重入状态
3. 待方案审阅状态
4. 已放行进入 Skill B 状态

#### 10.2 Skill B 状态原则

Skill B 至少应覆盖：

1. 上半入口状态
2. 上半完成状态
3. 待细节审阅状态
4. 下半重入状态
5. 已验证冻结状态
6. 回退 Skill A 状态

#### 10.3 状态与路由一致性

任何新增交互规则都必须满足：

1. 状态表中有对应状态
2. 路由逻辑中有对应入口
3. 人类操作后有明确状态转迁
4. 不允许出现“文案里有动作，但状态机里无承接状态”的情况

### 11. 统一审查口径

#### 11.1 方案审阅

方案审阅关注的是：

1. 主结构方案是否成立
2. 是否允许进入资产维护与详细展开
3. 不审开发级细节

#### 11.2 细节审阅

细节审阅关注的是：

1. 详细定义和规格是否达到可开发、可验证、可冻结粒度
2. 不重审主结构

### 12. 统一复用建议

在 `wft01~04` 拆分时，仅替换以下内容：

1. 本步主处理节点对象
2. 本步依赖的约束资产
3. 本步是否存在下半详细展开
4. 本步的具体写入资产位置

其余输入处理、运行结束输出、状态原则、回退机制与审查口径，均应优先复用本统一规范。
