# STR → BA 架构 · Skill A · 设计开发
**Skill ID**: wft02a · **Phase 1: STR → BA Design**

> **概述**：每次执行只处理一个对象：一个已完成文档归属、可以分解分配的 STR 架构末级节点（即场景业务），或一个待处理反馈的 BA 节点。Skill A 负责基于输出产品架构锚定法，将 STR 场景业务中的十链信息节点转换为 EOS 三类产品（A1 业务配置平台、A2 xx业务运行系统、Bn 平台输出产品）的 BA 主结构方案；同时可基于 BA 架构语境记录构件产品数据完备性缺口观察，但默认不把经验型观察升级为正式 BA 五级节点，只有当前 STR 必需、资产已确认或人类指定时，才形成 BA 补充建议、STR 回补建议或业务引擎能力缺口反馈建议；不负责 23/24/25/26/27 资产正式写回，也不负责 `03-*.md` STR 详细定义正式写回。
>
> **输入 → 输出**：一个已确认并完成文档归属的 STR 场景业务 → BA 主结构方案局部写入 `04-*.md`（状态=`待确认方案` 或 `待资产写回`）+ BA 完备性缺口观察 + 例外升级的 BA 补充建议 + 业务定义 / 业务引擎缺口建议方案 + STR 回补或能力缺口反馈建议
>
> **适用场景**：瀑布式系统设计第2步 · Phase 1。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v3.18 | **修订**：2026-06-23

## 一、职责定位

Skill A 负责：

1. 识别本轮处理对象
2. 加载 STR 承接链、wft01b 文档归属方案、23/24/25/26/27 相关资产与现有 BA 资产
3. 基于输出产品架构锚定法，形成 EOS 三类产品的 BA 主结构方案
4. 将十链信息节点按规则消费为业务定义、业务定义初步方案活动或 STR 缺口反馈建议
5. 校验业务定义是否来自 24 业务定义资产，并识别 25 引擎模型、配置 IPO 类型、扩展、改进或新增需求
6. 基于 BA 架构语境记录构件产品数据完备性缺口观察
7. 判断缺口观察是否满足升级条件，是否需要形成 BA 补充建议、回补当前 STR、新增 STR 场景业务或形成业务引擎能力缺口反馈
8. 判断当前方案状态
9. 提交人类审阅 BA 主结构方案

Skill A 不负责：

1. 23/24/25/26/27 资产的正式写回
2. `03-*.md` STR 详细定义的正式写回
3. 直接修改 wft01a 已确认的 STR 场景边界、信息载体清单或场景指标
4. 直接确认业务引擎扩展 / 新增需求
5. 直接把 BA 完备性缺口观察或 BA 补充建议正式写回 23/24/25/26/27
6. 将 BA 节点放行到 `wft03`

## 二、输入与路由

### 本轮处理对象

每次执行只处理一个对象：

1. **新增路径**：`02-*.md` 状态队列中 `当前状态=可以分解分配 + 人类决策=同意分解分配` 的一个 STR 架构末级节点；或 `wft01b` 已完成活动锁释放 / 转交，且该 STR 节点已有 `选择进入wft02a` 记录；或 STR 节点当前状态 = `待补充分解分配`（上游已修订，需 wft02a 补全）

   说明：`wft01b` 产生的 `选择进入wft02a` 记录**等效于** `人类决策=同意分解分配`，二者满足其一即可进入新增路径。`待补充分解分配` 不需要额外的人类决策——上游修订本身即构成 wft02a 需要介入的充分条件。AI 不得要求人类对同一 STR 节点既记录 `选择进入wft02a` 又补写 `同意分解分配`。

2. **方案反馈路径**：`04-*.md` 状态队列中 `当前状态=待确认方案` 且反馈区已有 `[同意] / [修改] / [驳回]` 的一个 BA 节点
3. **重入路径**：`04-*.md` 状态队列中 `当前状态=迭代中` 的一个 BA 节点

### 活动 STR 节点锁定

`wft02a` 同一时间只能允许一个 STR 架构末级节点处于活动处理状态。一个 STR 节点进入 `wft02a` 后，在该节点的 BA 主结构方案未完成本 Skill 处理前，不得再让新的 STR 节点进入 `wft02a`。

活动 STR 节点判定：

1. `04-*.md` 中存在引用某个 STR 节点的 BA 节点，且状态为 `待确认方案`、`迭代中`、`确认中` 或其他未完成状态。
2. `04-*.md` 中存在该 STR 节点的待办锁定记录，且当前待办 ID 尚未完成。
3. `02-*.md` 状态队列中该 STR 节点已被标记为进入 `wft02a`，但尚未记录 `wft02a` 处理完成。

活动 STR 节点完成判定：

1. 对应 BA 节点已达到 `待资产写回`，且人类已明确选择该 BA 建议方案包进入 `wft02b`。
2. 该 STR 节点被人类明确撤销、废弃或退回上游，不再由当前 `wft02a` 继续处理。

说明：BA 节点达到 `待资产写回` 只表示主结构方案已确认，不等于活动 STR 节点已释放。只要人类尚未完成 `选择进入wft02b`，该 STR 节点仍视为活动 STR 节点，AI 不得进入新的 STR 节点。

锁定规则：

1. Step Start 必须先检查是否存在活动 STR 节点，再检查新的 `可以分解分配` STR 节点。
2. 若存在活动 STR 节点，AI 只能重入该节点对应的 BA 节点，或输出该节点的当前待办；不得从 `可以分解分配` 队列中选择新 STR 节点。
3. 若存在多个活动 STR 节点，说明状态队列不一致，AI 必须停止新增处理，输出状态冲突说明，并要求人类先选择保留一个活动 STR 节点。
4. 只有不存在活动 STR 节点时，AI 才能从 `可以分解分配 + 同意分解分配` 队列，或已由 `wft01b` 转交并记录 `选择进入wft02a` 的 STR 节点中，选择一个新 STR 节点进入新增路径。
5. 新 STR 节点一旦进入新增路径，AI 必须在 `02-*.md` 状态队列中将 STR-F 节点状态推进为 `在分解分配`，并在 `04-*.md` BA 节点块或本轮运行摘要中记录活动锁定关系。
6. 活动锁定关系至少包含：`活动STR节点ID`、`关联BA节点ID`、`当前待办ID`、`锁定状态`、`锁定原因`、`锁定开始时间或轮次`、`释放条件`。
7. `锁定状态` 建议取值为：`锁定中`、`待进入wft02b`、`已释放`、`已撤销`、`冲突待处理`。
8. 本 Skill 处理完成（活动锁释放）时，在 `02-*.md` 状态队列中将 STR-F 节点状态推进为 `已分解分配`，并在 BA 节点中记录 `STR-F 消费版本：v{N}`（等于当前 STR-F 版本）。
9. 若处理过程中检测到 STR-F 版本再次递增（上游再次修订），且本次处理为增量型或修订型，AI 不得自行推进到 `已分解分配`——处理完本轮变更后保持 `在分解分配`，等人类裁决是否还有未处理的上游变更。

活动锁定记录建议格式：

```text
活动STR节点ID：
关联BA节点ID：
当前待办ID：
锁定状态：锁定中 / 待进入wft02b / 已释放 / 已撤销 / 冲突待处理
锁定原因：
锁定开始：
释放条件：
```

### Step Start 路由

入口许可 STR 节点包括三类：

1. `02-*.md` 状态队列中 `当前状态=可以分解分配 + 人类决策=同意分解分配` 的 STR 节点。
2. `02-*.md` 状态队列、本轮运行摘要或 `wft01b` 锁释放 / 转交记录中已有 `选择进入wft02a` 的 STR 节点。
3. `02-*.md` 状态队列中 `当前状态=待补充分解分配` 的 STR 节点（上游修订后自动具备入口许可，不需额外人类决策）。

| 条件 | 路由 |
|------|------|
| BA 节点 = `待资产写回` 且人类已回复 `选择进入wft02b`，但活动锁尚未释放或转交 | 只执行活动锁释放 / 转交记录，不进入新增设计或反馈处理 |
| 存在一个活动 STR 节点 | 只处理该活动 STR 节点对应的 BA 反馈、重入或当前待办，不得进入新 STR 节点 |
| 存在多个活动 STR 节点 | 停止执行，输出状态冲突说明，要求人类选择保留一个活动 STR 节点 |
| STR 节点 = `可以分解分配` 且人类已回复 `选择`、`选择进入wft02a` 或 `选择：<STR节点ID>`，且不存在活动 STR 节点 | 新增路径，并写入活动 STR 节点锁定关系（状态 → `在分解分配`） |
| STR 节点 = `待补充分解分配`，且不存在活动 STR 节点 | 新增路径（上游变更补全），读取变更影响声明，按变更类型处理 |
| 存在多个入口许可 STR 节点，且不存在活动 STR 节点 | 输出 `等待选择进入wft02a`，当前待办 ID 为 AI 推荐的一个 STR 节点 ID |
| 仅存在一个入口许可 STR 节点，且不存在活动 STR 节点 | 新增路径 |
| BA 节点 = `待确认方案` + 反馈区有 `[同意] / [修改] / [驳回]` | 方案反馈路径 |
| BA 节点 = `迭代中` | 重入路径 |
| BA 节点 = `待确认方案` 且反馈区为空 | 输出 `等待建议方案反馈`，当前待办 ID 为该 BA 建议方案 ID，不执行新增设计 |
| BA 节点 = `待资产写回` 且活动锁定状态不是 `已释放` | 输出 `等待选择进入wft02b`，当前待办 ID 为该 BA 建议方案包 ID |
| BA 节点 = `待资产写回` 且活动锁定状态为 `已释放` 或 `已转交wft02b` | 输出 `已转交wft02b`，不得继续修改该节点 |
| STR 节点未完成文档归属 | 提示先运行 `wft01b` |
| 存在活动 STR 节点，且 BA 节点记录的 `STR-F 消费版本` < STR-F 当前版本 | 上游变更感知路径（增量型/修订型自动处理，结构型等人类裁决） |
| 其他状态 | 输出不可操作说明后结束 |

运行保护规则：

1. 若 BA 节点已处于 `待确认方案` 且反馈区为空，AI 不得覆盖既有方案，只输出 `等待建议方案反馈`，当前待办 ID 为该 BA 建议方案 ID。
2. 若 BA 节点已处于 `待确认方案` 且反馈区有 `[同意] / [修改] / [驳回]`，AI 只执行 Step 5，不得重新进入新增设计处理。
3. 若 BA 节点已处于 `待资产写回`，说明主结构方案已确认，AI 不得继续修改该节点方案；若活动锁未释放，应先输出 `等待选择进入wft02b`，若活动锁已释放或已转交，应输出 `已转交wft02b`。
4. 若需要重做 BA 主结构，只能由 Step 5 的 `[修改] / [驳回]` 分支触发，并必须保留旧方案处理摘要与人类反馈记录。
5. 只要存在活动 STR 节点未完成，AI 不得响应其他入口许可 STR 节点，即使这些节点也满足新增路径条件。
6. 若多个 STR 节点同时满足入口许可，AI 不得自动选择其中一个进入新增路径，必须先输出 `等待选择进入wft02a`。
7. 若 STR 节点已由 `wft01b` 完成 `选择进入wft02a` 的转交记录，下一次 `wft02a` 可将该记录视为本 Skill 的进入选择；AI 应先将选择记录转写到 `02-*.md` 状态队列或本轮运行摘要，再让该 STR 节点进入新增路径，不得再次输出同一个选择提示。
8. 若多个 STR 节点同时满足入口条件，且其中一个是上次 Step End 已提示的当前待办 ID，AI 必须延续该 ID；只有该 ID 已完成、被人类切换或不再可操作时，才可推荐其他 STR 节点。
9. 若人类已对待写回建议方案包回复 `选择进入wft02b`，下一次 `wft02a` 只允许记录活动锁释放或转交，不得重新设计、修改或覆盖该 BA 节点。

#### 上游变更感知与处理

当 Step Start 检出上游变更（BA 节点 `STR-F 消费版本` < STR-F 当前版本）时，AI 读取 STR-F 节点末尾的变更影响声明，按变更类型处理：

| 变更类型 | Step Start 行为 | 设计行为 |
|---------|----------------|---------|
| **增量型** | 正常进入，加载上下文时仅加载新增 IC | 仅展开新增 IC 的 BA 节点；已有 BA 节点不动；写回时不覆盖已有 |
| **修订型** | 正常进入，加载受影响的 IC 及其变更项 | 修订受影响的 BA 节点，局部调整 I/P/O 或归属；标记 `[来自上游变更]`；修订后 BA 节点状态 = `待确认方案` |
| **结构型** | 进入但**阻断设计** | 输出影响范围评估（哪些 BA 节点受波及、可能的 BA 结构调整方向）；等人类裁决——人类可能选择"接受变更，允许 BA 重构""回退 STR-F 变更"或"分拆 STR-F" |

处理约束：

1. **增量型**和**修订型**：AI 可自动处理，但处理完成后必须将本轮变更写入 BA 节点的 `### 上游变更处理记录`，供人类审阅。
2. **结构型**：AI 不得自动修改 BA 节点。只输出影响范围评估。人类裁决后，AI 按裁决结果执行。
3. 无论哪种类型，处理完成后 BA 节点中 `STR-F 消费版本` 必须更新为当前 STR-F 版本。
4. 若上游变更与本轮 BA 节点自身的待处理反馈（`待确认方案` + 反馈区有效）同时存在，上游变更优先——先处理上游变更，再处理人类反馈。原因：人类反馈可能针对的是旧版本 STR-F 的 BA，上游变更可能已使部分反馈过时。

### Step 1 · 已确认资产加载

读取并建立本轮设计上下文：

1. 加载 `02-*.md` STR 树画像 / 状态队列 / 节点索引，定位本轮 STR 场景业务正文块
2. 读取本轮 STR 节点的已确认场景业务、信息载体清单、场景指标、推理依据、信息链归属处理状态和 wft01b 文档归属方案
2a. 若已存在对应 BA 节点（非首次消费此 STR-F），比对 BA 节点记录的 `STR-F 消费版本` 与 STR-F 当前版本——若当前版本 > 消费版本，读取 STR-F 节点末尾的 `### 变更影响声明`，按变更类型确定本轮设计范围（增量型仅处理新增 IC、修订型修订受影响 BA 节点、结构型停止设计输出影响评估）
3. 加载 `03-*.md` 当前 STR 详细定义文档的树画像 / 状态队列 / 节点索引，用于识别是否已有相关详细定义或待确认细节
4. 加载 `04-*.md` BA 树画像 / 状态队列 / 节点索引，定位候选 BA 宿主与本轮 BA 节点块
5. 加载 `23-eos-output-architecture.md` 资产全貌层，按业务域、产品 A1、产品 A2、产品 Bn、产品类型、构件类型和 WBS 筛选候选分片
6. 加载 `24-eos-business-definitions.md` 资产全貌层，按文档节点、业务对象、业务动作、管理对象、指标对象和候选业务定义名称筛选候选分片
7. 加载 `25-eos-engine-models.md` 资产全貌层，按业务定义生成机制、编排机制、规则机制、状态机制、协同机制、指标机制和资源机制筛选候选引擎模型与引擎配置 IPO 类型分片
8. 加载 `26-eos-document-definitions.md` 资产全貌层，按 wft01b 已确认文档归属读取候选文档定义块
9. 加载 `27` 资源资产全貌层，按 CAX 工具、AI 工具、模板、专业能力、作业指南和所需资源筛选候选资源块
10. 若为方案反馈或重入路径，加载目标 BA 节点反馈区及反馈指向的相关资产块
11. 按追溯关系读取必要的上游 / 下游相关块，不默认全量读取大文档正文

### Step 2 · STR 场景与十链消费分析

对本轮 STR 场景业务执行统一分析：

1. 校验 OR→STR 承接、角色、业务域、约束链是否成立
2. 校验 wft01b 是否已完成十条信息链的文档归属与 23/26/22 写回
3. 读取每个信息节点的文档名称、归属链、文档性质、产生方式、一句话职责、关系对象、业务域、产品类型、构件类型和确认状态
4. 将 EOS 输出产品架构拆分为三类产品路径：
   - **产品 A1：业务配置平台**。全局唯一，提供给 IT 人员配置并生成业务定义。
   - **产品 A2：xx业务运行系统**。每个业务域一个，为该业务域所有角色提供业务执行或操作。
   - **产品 Bn：平台输出产品**。一个业务域可有多个，提供给客户、外部用户或内部用户使用，如实物产品、规划经营类产品、人财物资源等。
5. 按十链消费规则判断每个信息节点的 BA 落点
6. 识别关键缺口：
   - A1 / A2 / Bn 产品路径缺口
   - 24 业务定义资产缺口
   - 25 引擎模型、引擎配置 IPO 类型、扩展、改进或新增缺口
   - 26 文档定义与 BA 业务定义关系缺口
   - 需要反馈回 STR 的业务引擎能力缺口

处理边界：

1. 十条信息链中每个信息节点的业务域、产品类型和构件类型由 `wft01b` 在文档归属阶段确定并写回。
2. `wft02a` 只读取、校验和使用这些已确认归属，不重新识别或重新定义信息节点所属产品类型、构件类型。
3. 若 `wft02a` 发现业务域、产品类型或构件类型归属明显冲突，只能形成反馈或回退上游，不得在本 Skill 中直接改写归属结论。

#### 2.1 三类产品 BA 层级

| 产品 | 二级节点 | 三级节点 | 四级节点 | 五级节点 |
|------|----------|----------|----------|----------|
| A1 业务配置平台 | 流程与IT域 / 产品A1 | 业务引擎 | 引擎配置包 | 引擎配置 IPO + 初步方案 |
| A2 xx业务运行系统 | xx业务运行系统 | 用户角色 | 角色职责 | 场景业务 + 初步方案 |
| Bn 平台输出产品 | xx产品类型 | 构件类型 | WBS 包 | 产品数据开发管理 IPO（计划定义）+ 初步方案 |

说明：

1. A1 是业务定义生产机制的产品化表达，所属业务域固定为流程与IT域，不按其他业务域重复设置；采购、生产、研发等业务域只作为 A1 配置支撑对象的业务上下文记录。
2. A2 是业务域运行入口和多角色场景编排系统，不吞并 Bn 的产品数据开发管理定义 / 计划定义。
3. Bn 是 EOS 运行输出产品的业务架构承载对象，其分类可来自 STR 推导、既有资产、人类确认或架构预置。
4. STR 场景业务是 A2 五级节点的来源；wft02a 将其转换为 A2 的“场景业务 + 初步方案”节点，其中初步方案是完成该场景所需的业务定义序列集合。
5. Bn 五级节点的 IPO 是产品数据开发管理业务定义，即计划定义；产品数据输出文档是该计划定义的 Output。其初步方案描述计划生成、任务下发、资源使用、成果提交、审核发布和归档等活动，其中部分活动可能引用公共业务定义或资源资产。

#### 2.2 四级节点生成规则

四级节点不是由 STR 直接机械生成，而是 wft02a 在优先复用 23 已确认资产的基础上，根据三类产品的三级节点语义和五级 IPO 聚合关系提出建议方案。

| 产品 | 四级节点 | 生成来源 | 生成规则 |
|------|----------|----------|----------|
| A1 业务配置平台 | 引擎配置包 | 业务引擎能力与配置目的 | 围绕同一业务引擎中可复用的一组配置项组织，如规则配置包、编排配置包、状态配置包、指标配置包、资源配置包；应能支撑一个或多个业务定义、场景业务或业务定义生成机制 |
| A2 xx业务运行系统 | 角色职责 | STR 场景业务的用户角色与职责边界 | 同一用户角色下，把职责相近、业务闭环相同的一组场景业务归入同一角色职责；不得按页面、菜单或文档形态分组 |
| Bn 平台输出产品 | WBS 包 | 产品数据开发管理定义 / 计划定义的业务对象、生命周期阶段、构件职责和产品输出结构 | 同一构件类型下，把围绕同一产品数据对象、同一生命周期阶段或同一交付结构的一组 IPO 归入一个 WBS 包；不得按信息链或单个场景机械分组 |

处理要求：

1. 若 23 中已有合适四级节点，应优先复用。
2. 若已有四级节点语义相近但边界不足，应提出扩展建议方案。
3. 若无合适四级节点，应提出新增建议方案，并说明生成依据。
4. 四级节点建议方案进入人类方案审阅，不在 wft02a 中正式写回 23。
5. 四级节点必须能解释其下五级节点为什么聚合在一起；无法解释聚合依据时，不得新增四级节点。

#### 2.3 十链消费规则

十链节点必须被 BA 消费，但不必全部成为同一类五级 IPO。默认规则如下：

| 信息链 | BA 消费方式 | 说明 |
|--------|-------------|------|
| 执行业务链 | 形成主业务定义 | 默认成为 Bn 产品数据开发管理定义 / 计划定义，或成为 A2 场景初步方案调用的核心业务定义 |
| 规则知识链 | 并入对应主业务定义 | 默认作为执行业务链业务定义的规则、模板、知识、判断标准或配置约束 |
| 风险合规链 | 并入对应主业务定义 | 默认作为执行业务链业务定义的审批、授权、校验、留痕、异常处置或证据要求 |
| 资源需求链 | 并入对应主业务定义 | 默认作为执行业务链业务定义的资源查看、申请、占用、释放、约束或核销要求 |
| 协同交互链 | 进入主业务定义初步方案活动 | 默认成为执行业务链业务定义内部的通知、待办、回执、转办、催办或接口交换活动 |
| 状态事件链 | 进入主业务定义初步方案活动 | 默认成为执行业务链业务定义内部的状态推进、事件记录、日志生成、轨迹沉淀或告警活动 |
| 管理业务链 | 形成管理类业务定义 | 默认成为面向管理对象、进度、异常、统计、干预、纠偏的业务定义 |
| 个人绩效链 | 形成个人绩效类业务定义 | 默认成为面向个人贡献、效率、质量、达成率和归因的业务定义 |
| 组织绩效链 | 形成组织绩效类业务定义 | 默认成为面向团队、部门、项目、供应商等组织对象绩效的业务定义 |
| 场景指标链 | 形成指标类业务定义 | 默认成为面向场景成效、质量、风险、成本、协同或体验的度量业务定义 |

例外规则：

1. 规则知识、风险合规、资源需求节点若具有独立业务对象、独立生命周期、独立维护责任或跨场景复用价值，可以升级为独立业务定义。
2. 协同交互、状态事件节点若已由 `wft01b` 判定为公共业务定义，应作为初步方案中的公共业务定义资产引用处理，优先引用 `流程与IT域 → 公共业务构件 → 协同交互构件 / 状态事件构件` 下的已确认资产，不在当前业务域 Bn 路径下重复生成 BA 节点。
3. 管理业务、绩效、指标节点若只是某个主业务定义的简单派生展示，且没有独立管理对象或评价口径，可进入该主业务定义的初步方案活动。
4. 任一例外必须写明判定依据，并进入人类方案审阅。

公共业务定义引用规则：

1. 只有协同交互链、状态事件链中的节点可以作为公共业务定义引用进入初步方案。
2. 公共业务定义的产品归属应来自 `wft01b` 已确认的产品类型和构件类型；典型路径为 `流程与IT域 → 公共业务构件 → 协同交互构件 / 状态事件构件`。
3. 当前主业务定义或 A2 场景业务的初步方案只记录对公共业务定义的调用顺序、调用条件、输入输出衔接和资产引用。
4. `wft02a` 不因引用公共业务定义而在当前业务域产品类型、构件类型或 WBS 下新增公共业务定义节点。
5. 若公共业务定义资产不存在或未确认，应形成 24/23 资产建议或待反馈项，不应把公共业务定义临时归入当前业务域产品。

#### 2.4 业务定义资产与业务引擎支撑链

业务定义必须来源于 24 业务定义资产，新增或扩展业务定义必须能被 25 引擎模型及其引擎配置 IPO 类型支撑；产品数据开发所需的 CAX/AI/模板/专业能力等作为资源引用，来源于 27 资源资产。

判定顺序：

| 顺序 | 判定 | Skill A 动作 |
|------|------|--------------|
| 1 | 24 已有业务定义可直接复用 | 引用已有业务定义 |
| 2 | 可基于已有引擎模型和配置 IPO 类型扩展已有业务定义 | 提出业务定义扩展方案，并列出将实例化的 A1 配置 IPO 类型 |
| 3 | 可基于已有引擎模型和配置 IPO 类型新增业务定义 | 提出业务定义新增方案，并列出将实例化的 A1 配置 IPO 类型 |
| 4 | 需改进已有业务引擎后扩展或新增业务定义 | 提出业务引擎改进 IPO 建议（含 I/P/O、初步方案、相关方需求），同时形成业务定义方案 |
| 5 | 需新增业务引擎后新增业务定义 | 提出业务引擎新增 IPO 建议（含 I/P/O、初步方案、相关方需求），同时形成业务定义方案 |

处理要求：

1. 顺序 1~3 可作为 BA 主结构建议方案进入人类审阅。
2. 顺序 4~5 产生业务引擎能力边界变化，AI 仍须提出业务引擎 IPO 建议（含 I/P/O、初步方案及相关方需求），供人类审阅迭代；同时形成 STR 缺口反馈建议说明需要回补的能力描述。引擎 IPO 建议方案进入人类审阅，通过确认后作为 25 候选 / 缺口建议留痕并移交后续 `wft03b` 在 SR 层功能定义确认后写回，不作为当前 BA 主结构直接写入 04，也不得在 `wft02a/wft02b` 中补入 25 功能定义。
3. 若业务定义缺口仅涉及命名、别名、输入输出补全或配置项扩展，优先按顺序 2~3 处理。
4. 若缺口涉及编排机制、状态机制、规则执行机制、协同机制、指标计算机制或资源机制改变，必须识别为可能的 STR 级反馈。
5. 每个具体业务定义背后应对应一组实例化后的 A1 引擎配置 IPO；多个同类业务定义可共享 25 中的引擎模型和配置 IPO 类型，但不同具体业务定义不得共享同一份配置实例数据，除非它们本来就是同一个业务定义。
6. 产品数据开发管理定义是计划定义，默认由计划引擎和计划定义配置 IPO 类型支撑；每个产品数据计划定义对应独立的计划配置实例数据。
7. 产品数据内容开发不单独生成“开发实现业务定义”；CAX 工具、AI 工具、模板、专业能力和作业指南作为计划定义初步方案中的资源引用。

#### 2.5 BA 完备性缺口观察与升级判定

`wft02a` 不只是把 STR 十链节点机械转换为 BA 五级节点，也可以在 BA 架构语境中观察构件产品数据是否可能存在缺口。但由于本 Skill 每次只处理一个上游 STR 节点，AI 不具备全局补全构件产品数据的上下文，因此默认只能记录“完备性缺口观察”，不得基于经验直接补全整个构件类型的产品数据。

##### 来源类型

所有 BA 五级节点建议方案和完备性缺口观察必须标注来源类型：

| 来源类型 | 含义 | 处理口径 |
|----------|------|----------|
| `STR推导` | 直接来源于当前 STR 场景业务及十链信息节点 | 正常进入 BA 主结构建议方案 |
| `BA完备性观察` | 由构件产品数据完整性、生命周期完整性或 WBS/IPO 完整性检查发现的经验型缺口 | 默认只记录观察，不进入 BA 主结构建议方案 |
| `人类指定补充` | 人类在审阅或对话中明确要求补充的 BA 五级节点 | 可升级为 BA 补充建议，并保留人类来源和依据 |
| `资产已确认补充` | 23/24/25/26/27 已有明确资产，只是当前 BA 路径尚未引用 | 可升级为 BA 补充建议，并引用已确认资产 |
| `资产一致性观察` | 由同类构件惯例或架构预置推导出的可能缺口 | 默认只记录观察，进入待后续 STR 验证或待构件级收敛 |

##### 缺口观察对象

完备性缺口观察优先面向 Bn 平台输出产品路径：

1. 构件类型下产品数据类 IPO 是否覆盖该构件的关键业务对象。
2. WBS 包下是否存在输入、处理、输出、状态、规则、资源、协同、管理评价等必要产品数据。
3. 产品数据生命周期是否存在明显断点，如只有申请无审批、只有计划无执行反馈、只有输出无状态记录。
4. 同类构件或既有 23/24/26 资产中常见的产品数据，在当前构件下是否缺失。
5. A2 场景初步方案调用的业务定义序列是否缺少必要的支撑性业务定义。
6. A1 引擎配置是否能配置生成缺口观察项或已升级建议所需的业务定义或场景编排。

##### 观察触发条件

满足以下任一情况时，可形成 BA 完备性缺口观察：

1. 某构件下缺少必要的产品数据输出文档对应 IPO。
2. 某 WBS 包下业务对象生命周期不完整。
3. 主业务定义的输入、处理、输出链断裂。
4. 规则、风险、资源、状态、协同、管理、绩效或指标支撑数据缺失，导致业务定义不可运行、不可管理或不可验证。
5. 既有资产或同类构件显示通常应存在某类产品数据，但当前 STR 未推导出来。
6. 人类指出某构件产品数据不完备，要求补充五级节点。

##### 缺口观察成熟度

BA 完备性缺口观察必须标注成熟度：

| 成熟度 | 含义 | 默认动作 |
|--------|------|----------|
| `当前STR必需补充` | 没有该节点，当前 STR 场景业务对应的业务定义序列无法闭合 | 可升级为 BA 补充建议或 STR 回补建议 |
| `资产已确认补充` | 已确认资产中已存在对应产品数据、业务定义或文档关系 | 可升级为 BA 补充建议 |
| `人类指定补充` | 人类明确要求本轮补充该五级节点 | 可升级为 BA 补充建议 |
| `候选观察` | AI 基于经验发现可能缺口，但证据不足 | 只记录观察，不进入 BA 主结构 |
| `待后续STR验证` | 需要后续 STR 节点进一步验证、细化或纠偏 | 只记录观察，不进入 BA 主结构 |
| `待构件级收敛` | 需要在多个 STR 处理后做构件级产品数据收敛 | 只记录观察，不进入 BA 主结构 |

##### 缺口观察生命周期

BA 完备性缺口观察不是静态备忘录，而是随后续 STR 节点进入逐步被处理的临时观察项。每次 `wft02a` 处理新的 STR 节点时，必须读取与当前业务域、产品类型、构件类型、WBS 或 IPO 相关的未关闭缺口观察，并重新判定其状态。

缺口观察状态建议取值：

| 状态 | 含义 | 处理动作 |
|------|------|----------|
| `待验证` | 尚未有足够 STR 或资产证据 | 保留观察，等待后续 STR 节点 |
| `已被当前STR吸收` | 新进入的 STR 节点已直接推导出该产品数据或业务定义 | 将观察关闭，并在新 BA 节点中标注“吸收自缺口观察” |
| `升级为BA建议` | 新 STR、资产或人类反馈使观察满足升级条件 | 按升级后的 BA 建议、STR 回补或业务引擎能力缺口处理 |
| `转构件级收敛` | 单个 STR 节点仍不足以处理，但多个观察指向同一构件级缺口 | 保留到构件级收敛队列，不进入本轮 BA 主结构 |
| `关闭-证伪` | 后续 STR 或资产证明该缺口不存在、不适用或已由其他路径覆盖 | 关闭观察并记录证伪依据 |
| `关闭-重复` | 与已有 BA 节点、观察项或资产引用重复 | 关闭观察并记录被合并对象 |

处理规则：

1. 新 STR 节点进入时，AI 必须优先检查是否存在与当前处理范围相关的未关闭缺口观察。
2. 若新 STR 节点能够直接推导出观察项所指向的产品数据、业务定义或场景编排，优先按 `STR推导` 生成 BA 主结构，并把原观察标记为 `已被当前STR吸收`。
3. 若新 STR 节点提供了足够证据但仍不是直接推导结果，按成熟度重新判定是否可升级为 BA 建议、STR 回补建议或业务引擎能力缺口反馈。
4. 若新 STR 节点证明原观察不成立、范围错误或已被其他 BA 节点覆盖，必须关闭观察，不得继续保留悬空项。
5. 若多个后续 STR 节点持续指向同一构件级缺口，但没有单个 STR 节点足以承接，应转入 `待构件级收敛`，等待构件级产品数据收敛处理。
6. 关闭或升级缺口观察时，必须记录处理依据、处理轮次、关联 STR 节点 ID 和关联 BA 节点 ID。

##### 升级条件

只有满足以下任一条件时，BA 完备性缺口观察才可升级为正式建议：

1. `当前STR必需补充`：该缺口不补，当前 STR 场景无法形成可运行的业务定义序列。
2. `资产已确认补充`：23/24/25/26/27 已有明确资产，当前只是建立引用、补齐关系或纳入当前 BA 路径。
3. `人类指定补充`：人类明确要求本轮补充该节点。

除此之外，经验型缺口不得升级为 BA 五级节点建议、不得进入待写回包、不得触发正式资产写回，只能作为观察项保留。

##### 升级后的回补判定

已升级的 BA 补充建议必须判断是否需要回补 STR：

| 判定 | 适用情况 | 动作 |
|------|----------|------|
| `回补当前STR` | 缺口意味着当前 STR 场景业务遗漏了信息载体、场景指标、规则、状态、协同或资源需求 | 在 `04` 中形成 `STR回补建议`，经人类确认后由回退链路写入 `02` 反馈区，供 `wft01a` 重跑 |
| `新增STR场景建议` | 缺口不属于当前角色场景职责，而属于另一个角色、触发或业务闭环 | 在 `04` 中形成新增 STR 场景建议，供人类确认后回到上游处理 |
| `业务引擎能力缺口` | 缺口源于现有业务引擎无法配置生成所需业务定义或场景编排 | AI 须提出业务引擎 IPO 建议（含 I/P/O、初步方案、相关方需求）写入 `04` 供人类审阅迭代，同时在 `04` 中形成缺口反馈建议供 `wft02b` 转写到上游 |
| `只补BA` | 缺口只是在已满足升级条件下补齐 BA 路径、资产引用或当前 STR 必需的产品数据，不改变当前 STR | 作为 BA 补充建议进入人类审阅和 `wft02b` 写回 |
| `人类判断` | 是否应回补 STR 仍不确定 | 作为审阅问题提交人类判断，不得直接写回 |

处理要求：

1. `wft02a` 可以记录 BA 完备性缺口观察，但默认不得据此新增 BA 五级节点。
2. `wft02a` 可以提出 STR 回补建议、业务引擎能力缺口反馈建议或新增 STR 场景建议，但不得直接修改 `02-*.md` 或 `03-*.md`。
3. 所有观察、升级建议和回补建议先写入 `04-*.md` 本轮 BA 节点块，作为 BA 方案的一部分供人类审阅。
4. 人类确认后，由 `wft02b` 或回退流程将确认后的内容转写为上游可识别的反馈记录。
5. 若回补建议会改变当前 STR 场景边界、信息载体清单或场景指标，当前 BA 方案应标记为阶段性方案或带阻断项，不得直接进入资产写回。
6. 若缺口观察只属于 `候选观察`、`待后续STR验证` 或 `待构件级收敛`，当前 BA 方案可继续形成，但不得把这些观察项计入待写回 BA 五级节点。

2. 先查询 `25-eos-engine-models.md` 中已确认的引擎模型和引擎配置 IPO 类型库，判断当前业务定义 / 场景业务需要实例化哪些配置 IPO 类型。
3. 若已有配置 IPO 类型可支撑，按 25 的类型定义实例化为本轮 A1 引擎配置 IPO 建议，并记录类型 ID、所属引擎模型、对应系统实体、配置页面、I/P/O、依赖 IPO 和支撑对象。
4. 若已有配置 IPO 类型边界不足但不改变引擎能力边界，提出 25 配置 IPO 类型扩展建议和本轮 A1 实例化建议。
5. 若需新增配置 IPO 类型、改进引擎模型或新增业务引擎，AI 须提出业务引擎 IPO 建议（含 I/P/O、初步方案、相关方需求），同时形成 25 资产建议和 STR 缺口反馈建议——引擎 IPO 建议供人类审阅迭代，缺口反馈说明需要回补的能力描述。改变编排机制、状态机制、规则机制、协同机制、指标机制、资源机制或核心元模型的建议，必须标记为带阻断项的阶段性方案。
6. 若无法说明支撑对象、引擎模型、配置 IPO 类型或实例化依据，不得新增 A1 IPO，只能形成业务引擎能力缺口反馈建议或阶段性方案。
7. 若业务引擎缺口来自 BA 完备性缺口观察或升级建议，必须保留来源类型、成熟度、影响范围和是否回补 STR 的判定。

A1 引擎配置 IPO 的四要素口径：

| 要素 | 内容 |
|------|------|
| Input | 配置页面及配置信息。配置页面是 IT 人员或配置人员录入、维护、校验配置的交互入口；配置信息包括字段、规则、流程、状态、权限、指标、资源、页面布局等引擎可处理的配置项。 |
| Process | 系统对配置信息执行校验、解析、存储、编译、发布、版本管理和运行期装载等处理，使配置项转化为对应引擎可执行、可调用的业务能力。 |
| Output | 该引擎生成的业务定义对应配置信息的业务能力；其外在表现是用户使用该业务定义时可见、可操作的业务操作页面，以及页面背后的规则、流程、状态、权限、指标等运行能力。 |
| 承接关系 | 承接需要业务引擎配置支撑的 Bn 业务定义、A2 场景业务或公共业务定义；来源业务域仅作为支撑对象上下文记录，A1 自身归属流程与IT域。 |

A1 引擎配置 IPO 在 STR 场景业务中的显化规则：

1. STR-F 场景业务的主语义是业务角色端到端履职场景，通常直接显化 A2 场景业务和 Bn 产品数据开发管理定义；A1 引擎配置 IPO 是支撑这些业务定义可运行的配置机制，在 STR 中可以是弱显化。
2. A1 弱显化线索包括：字段、表单、页面、流程、状态、规则、权限、指标、资源、模板、校验、发布、版本、菜单、台账、主数据、字典、关系等需要由业务引擎配置生成或支撑的内容。
3. `wft02a` 不要求 STR 场景业务中存在独立的“A1 配置场景”或“配置人员操作场景”后才生成 A1 引擎配置 IPO；只要 Bn 业务定义、A2 场景业务或公共业务定义需要业务引擎配置支撑，就应按 A1 路径派生对应的引擎配置 IPO 建议。
4. A1 派生必须说明支撑对象和派生依据：来源 STR 信息载体 / Bn 业务定义 / A2 场景业务 / 24 业务定义资产 / 25 引擎资产。若无法说明支撑对象或引擎机制，不得新增 A1 IPO，只能形成业务引擎能力缺口反馈建议。
5. A1 派生不改变 STR 场景边界；只有当缺少配置线索会导致业务定义、场景业务或业务能力边界无法判断时，才形成 STR 回补建议。

业务引擎新需求到 A1 配置 IPO 的识别链：

1. **识别运行能力缺口**：先判断新需求希望业务用户获得什么运行能力，如页面录入、列表查询、流程审批、状态推进、规则校验、指标计算、对象关系维护、主数据 / 字典维护、外部集成等。
2. **判定候选引擎模型**：根据运行能力缺口，在 25 中匹配候选引擎模型，如表单、台账、流程、状态、规则、指标、关系、主数据、字典、集成等；若无候选引擎模型，形成业务引擎能力缺口反馈。
3. **识别元数据对象**：分析该运行能力需要哪些引擎内部可配置元数据对象，如 `FieldDefinition`、`LayoutDefinition`、`ProcessNode`、`TransitionRule`、`IndicatorFormula`、`RelationDefinition` 等。
4. **匹配配置 IPO 类型**：用元数据对象和运行能力在 25 的配置 IPO 类型库中匹配。已存在则实例化；边界不足但不改变能力边界则提出类型扩展；不存在且需新增元数据对象时提出配置 IPO 类型新增建议。
5. **判断是否能力边界变化**：若新增 / 扩展会改变引擎核心元模型、算法、规则执行机制、编排机制、状态机制、协同机制、指标机制、资源机制或对外接口边界，必须形成 STR 缺口反馈建议，不得直接作为已确认 A1 IPO。
6. **生成 A1 配置 IPO 建议**：对每个已匹配或待确认的配置 IPO 类型，生成当前业务定义 / 场景业务的 A1 配置 IPO 实例建议，包含 I/P/O、支撑对象、25 类型引用、资产建议和初步方案。

A1 引擎配置 IPO 的形态模板：

| 项 | 写法 |
|----|------|
| IPO 名称 | `{业务定义/场景业务}的{25配置IPO类型名称实例}`，如“采购申请单的字段定义配置”“采购申请审批的流程节点配置” |
| 配置对象 | 需要被生成或支撑的业务定义、场景业务或公共业务定义 |
| 引擎模型与配置 IPO 类型 | 引用 25 中已确认或待确认的引擎模型 ID 与配置 IPO 类型 ID |
| Input | 配置页面及配置信息；必须列出关键配置项，如字段、布局、规则、流程节点、状态、权限、指标口径、数据来源、菜单入口等 |
| Process | 系统对配置项执行校验、解析、存储、编译、发布、版本管理和运行期装载，并将配置项绑定到对应业务定义或场景业务 |
| Output | 该业务定义或场景业务在运行期可被用户使用的业务能力；表现为操作页面、列表台账、流程动作、状态流转、权限控制、指标看板或其他可操作界面与运行机制 |
| 支撑对象引用 | 写明被支撑的 Bn 业务定义、A2 场景业务、公共业务定义和来源 STR 信息载体 |
| 资产建议 | 写明 25 引擎模型 / 配置 IPO 类型是否复用、扩展、新增或形成能力缺口反馈 |
| 初步方案 | 写明配置人员操作、系统校验 / 保存 / 发布、运行期装载、生效结果、异常或缺口处理 |

A1 引擎配置 IPO 初步方案至少包含：

1. 配置人员角色：谁使用配置页面维护该配置，如 IT 配置人员、业务配置人员、流程管理员、指标管理员等。
2. 配置入口：使用哪个配置页面、设计器或配置面板。
3. 配置步骤：按顺序说明填写 / 选择 / 绑定 / 排列 / 设置规则 / 设置权限 / 设置发布范围等配置动作。
4. 系统处理：说明系统如何校验、解析、保存、绑定业务定义或场景业务、生成版本、发布到运行期。
5. 运行期生效：说明该配置在业务用户页面、流程、台账、状态、规则、指标或集成能力中如何体现。
6. 依赖关系：列出前置配置 IPO、依赖引擎模型、依赖业务定义、依赖文档或资源资产。
7. 异常处理：说明配置缺失、校验失败、发布失败、运行期不生效或能力边界不足时的处理方式。
8. 资产与反馈：列出需要写回 / 引用 / 扩展 / 新增的 25 资产，以及是否形成 STR 缺口反馈。
9. 相关方需求：说明该引擎 IPO 服务的使用方——配置人员的配置操作需求（如字段定义、规则设置、发布审批等）和业务用户通过该引擎获得的运行能力需求（如页面录入、流程审批、台账查询、指标查看等）。

A1 引擎配置 IPO 典型样例：

| 样例 | Input | Process | Output |
|------|-------|---------|--------|
| 采购申请单的表单引擎配置 | 采购申请单配置页面、字段定义、字段校验、页面布局、附件规则 | 系统校验并发布表单配置，绑定到“采购申请单”业务定义 | 采购员可创建、编辑、提交采购申请单的操作页面及表单能力 |
| 采购申请审批的流程引擎配置 | 审批流程配置页面、节点、角色、条件、退回规则、超时规则 | 系统解析流程配置并发布审批流，绑定到采购申请业务定义 | 采购申请提交后可按规则流转、审批、退回、留痕的流程能力 |
| 采购订单台账的台账引擎配置 | 台账配置页面、列表字段、筛选条件、排序规则、状态字段、权限范围 | 系统生成台账视图配置并绑定采购订单业务定义 | 用户可查询、筛选、跟踪采购订单的台账页面和查询能力 |
| 采购履约指标的指标引擎配置 | 指标配置页面、指标公式、统计口径、数据来源、展示维度 | 系统校验指标口径并发布指标计算配置 | 管理者可查看采购及时率、异常率等指标看板和评价能力 |

A1 引擎配置建议方案至少包含：

1. 业务引擎名称
2. 引擎配置包
3. 引擎配置 IPO
4. 支撑的业务定义 / 场景业务
5. 生成或支撑机制
6. 24 业务定义影响
7. 25 引擎模型与配置 IPO 类型引用 / 扩展 / 新增建议
8. 来源类型、成熟度与补充依据
9. 是否形成 STR 缺口反馈建议
10. 若形成反馈建议，明确是配置扩展、能力改进还是新增业务引擎
11. 初步方案，至少覆盖配置人员操作、系统处理、运行期生效、依赖关系、异常处理和资产反馈
12. 相关方需求：配置人员的配置操作需求 + 业务用户通过该引擎获得的运行能力需求

## 五、运行结束输出（Step End）

Step End 的作用是把本轮设计结果翻译成人类下一次运行 `wft02a` 前的最小待办。AI 不只输出“方案已生成”，还必须根据当前运行结果，为人类确定一个且仅一个当前动作。

推荐交互方式：

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应节点的 `### 反馈区`、状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

### 单 ID 最小待办原则

1. 每次 Step End 只提示人类处理一个 ID。
2. 该 ID 可以是一个 BA 建议方案 ID、一个待选 STR 节点 ID、一个待进入 `wft02b` 的建议方案包 ID。
3. AI 必须同时给出剩余待办 ID 数量，帮助人类知道当前队列规模。
4. 若上一次 Step End 已提示某个当前待办 ID，且该 ID 尚未被人类确认完成、明确切换或变为不可操作，下次运行必须继续提示同一个 ID。
5. 人类对当前 ID 的回复可以是 `同意`、`修改：……`、`驳回：……`，或 `选择：……`。
6. 只有当前 ID 已完成、被人类明确切换，或因状态变化不再可操作时，AI 才能按阻断优先级、状态时间和推荐顺序自动选择下一个当前 ID，并说明切换原因。
7. 当前 ID 处理完成后，下一次运行 `wft02a` 再提示下一个 ID。
8. AI 可以附带其他待办摘要，但不得要求人类同时审阅或选择多个 ID。

### 当前待办 ID 连续性规则

1. Step End 输出的 `当前待办ID` 默认形成“待办锁定”。
2. AI 应在 `04-*.md` 对应 BA 节点反馈区、`02-*.md` 状态队列、`04-*.md` 待写回选择记录或本轮运行摘要中保留该 ID 的待办状态。
3. 若下一次运行时该 ID 仍处于待反馈、待确认、待补充或待选择状态，AI 必须继续提示该 ID，不得因为出现新的待办而切换。
4. 若人类回复 `换一个 / 跳过 / 选择：<另一个ID>`，AI 可以切换当前待办 ID，但必须先记录切换原因。
5. 若当前 ID 已经完成，AI 才从剩余待办中选择下一个 ID。
6. 若当前 ID 因上游状态变化不再可操作，AI 必须说明原 ID 不再可操作的原因，再选择新的当前待办 ID。

### Step End 待办类型

| 待办类型 | 何时出现 | 当前 ID | 人类只需做什么 | 下次运行 |
|---------|----------|---------|----------------|----------|
| `反馈建议方案` | 本轮已生成或修订 BA 建议方案，状态为 `待确认方案` | 一个 BA 建议方案 ID | 对该 ID 回复 `同意 / 修改 / 驳回` | `wft02a` |
| `选择进入wft02a` | 存在多个已获入口许可的 STR 节点（`可以分解分配` 或 `待补充分解分配`），且当前没有占用中的活动 STR 节点 | 一个 STR 节点 ID | 对该 ID 回复 `选择`、`选择进入wft02a` 或指定另一个 STR ID | `wft02a` |
| `选择进入wft02b` | 一个 BA 建议方案已被 `[同意]` 并推进到 `待资产写回`，或存在多个待写回建议方案包 | 一个建议方案包 ID | 对该 ID 回复 `选择进入wft02b` | `wft02b` |

### 输出结果

|---------|--------------|---------|------------|

### 最小待办输出格式

AI 在 Step End 应按以下格式输出：

```text
当前结果：等待建议方案反馈 / 需要补充后重跑wft02a / 等待选择进入wft02a / 等待选择进入wft02b / 已转交wft02b
当前待办ID：
是否延续上次待办ID：
剩余待办ID数量：
待办类型：
为什么先处理这个ID：
人类只需回复：
回复会写入：
下次运行：
其他待办摘要：
```

要求：

1. `当前待办ID` 必须是单个 ID，不得写成多个 ID 列表。
2. `是否延续上次待办ID` 必须写 `是 / 否`；若为 `否`，必须说明当前 ID 已完成、被人类切换或不再可操作的原因。
3. `剩余待办ID数量` 必须是数字；若无法精确统计，应写 `至少 N 个` 并说明统计口径。
4. `人类只需回复` 必须是一句话，且只包含当前 ID 的动作。
5. `其他待办摘要` 只能给数量和类型，不得要求人类同时处理。
6. 若人类直接在对话中回复，AI 应先转写到 `04-*.md` 对应 BA 节点反馈区、`02-*.md` 状态队列或 `04-*.md` 待写回选择记录，再执行下一步。
7. 若当前结果为 `已转交wft02b`，`当前待办ID` 可写被转交的建议方案包 ID，`剩余待办ID数量` 应写 `0`，`人类只需回复` 应写 `无需回复，运行 wft02b`。

2. 若有多个 `待确认方案`，选择最早进入该状态且无反馈的一个。

人类只需做什么：

1. 对当前 BA 建议方案 ID 回复 `同意`、`修改：……` 或 `驳回：……`。
2. AI 收到回复后，先转写到该 BA 节点 `### 反馈区`，再在下一次 `wft02a` 中处理。

[其他说明]

重点查看位置：

### 结果二：需要补充后重跑wft02a

适用：当前 BA 建议方案仍有阻断项，或只能形成 `阶段性方案`。

当前待办 ID：

1. 必须是当前阶段性 BA 建议方案 ID。
2. 不得同时要求人类选择其他 STR 节点或进入 `wft02b`。

人类只需做什么：

1. 查看 AI 输出的阻断项或缺口说明
2. 对当前 BA 建议方案 ID 回复 `修改：……`
3. 补充缺失的产品路径、业务对象、业务定义、引擎支撑方式、BA 完备性观察成熟度、升级判断或 STR 回补判断依据

重点查看位置：

1. AI 输出的阻断项清单
2. `04-*.md` 对应 BA 节点反馈区
3. `02-*.md` 对应 STR 场景业务与 wft01b 文档归属方案

### 结果三：等待选择进入wft02a

适用：当前没有待审阅 BA 建议方案，且存在可继续处理的 `可以分解分配` STR 节点。

口径说明：

1. `同意分解分配` 是上游状态队列中的正式入口许可，表示该 STR 节点允许进入 `wft02a`。
2. `选择进入wft02a` 有两种含义：从 `wft01b` 过来时，表示该 STR 已完成文档归属并被转交给 `wft02a`；在 `wft02a` 内部存在多个入口许可 STR 时，表示本轮处理对象的选择动作。
3. 只有不存在活动 STR 节点时，AI 才能提示 `等待选择进入wft02a`。
4. 若已经存在活动 STR 节点，即使其他 STR 节点已有 `同意分解分配`，也不得提示选择新 STR 节点。
5. 若多个 STR 节点同时具备入口许可，AI 必须先输出本结果，不能自动进入新增路径。
6. 若只有一个 STR 节点具备入口许可且不存在活动 STR 节点，AI 可以直接进入新增路径。
7. 若某个 STR 节点已有 `wft01b` 写入的 `选择进入wft02a` 转交记录，且不存在活动 STR 节点，AI 可以直接进入该 STR 的新增路径；不需要人类再补写 `同意分解分配`。

当前待办 ID：

1. AI 推荐一个 STR 节点 ID。
2. 推荐依据可为状态时间、业务优先级、同业务域连续性或人类先前指定范围。

人类只需做什么：

1. 对当前 STR 节点 ID 回复 `选择` 或 `选择进入wft02a`。
2. 若不接受推荐，可回复 `选择：STR-xxx` 指定另一个 STR ID。

重点查看位置：

1. `02-*.md` 状态队列中 `可以分解分配` 的 STR 节点
2. 推荐 STR 节点的场景业务归属基准和文档归属方案

### 结果四：等待选择进入wft02b

适用：当前 BA 建议方案已确认，或存在多个待 `wft02b` 处理的建议方案包。

锁定说明：

1. 当 BA 节点已推进到 `待资产写回`，但人类尚未选择进入 `wft02b` 时，活动 STR 节点仍保持锁定。
2. 人类对当前建议方案包 ID 回复 `选择进入wft02b` 后，AI 应先记录活动锁释放或转交状态，再由 `wft02b` 接手处理。
3. 活动锁释放前，`wft02a` 不得进入新的 STR 节点。
4. 若下一次运行 `wft02a` 时检测到人类已回复 `选择进入wft02b`，AI 只执行锁释放 / 转交记录，不得重新进入新增设计、方案反馈或方案改写。

释放 / 转交记录至少包含：

1. 活动 STR 节点 ID
2. 关联 BA 节点 ID
3. 被选择进入 `wft02b` 的建议方案包 ID
4. 当前状态到位说明：BA 节点已推进到 `待资产写回`
5. 人类选择动作：`选择进入wft02b`
6. 锁定状态：`已释放` 或 `已转交wft02b`
7. 释放 / 转交时间或轮次
8. 下一步 Skill：`wft02b`

当前待办 ID：

1. AI 推荐一个建议方案包 ID。
2. 建议方案包可以是 Bn 业务定义建议方案包、A2 场景业务建议方案包、A1 引擎配置建议方案包、24/25/26 资产建议包、STR 缺口反馈建议包或 `03` 详细定义写回包。

人类只需做什么：

1. 对当前建议方案包 ID 回复 `选择进入wft02b`。
2. 若不接受推荐，可回复 `选择进入wft02b：PKG-xxx` 指定另一个包 ID。
3. 该 `选择进入wft02b` 记录是下游 `wft02b` 可识别的入口选择记录；`wft02b` 不应要求人类对同一建议方案包重复确认。

重点查看位置：

1. `04-*.md` 状态队列和节点索引中该 BA 节点状态
2. 当前建议方案包 ID 的范围说明
3. 当前建议方案包 ID 的资产建议、STR 缺口反馈建议或详细定义写回说明
