# BA(A1) → SysReq · 引擎配置功能定义与资产化
**Skill ID**: wft03-eng-ba2sr

> **概述**：每次执行处理 ≥1 个 BA（业务架构）A1（平台引擎产品）节点（来自 wft02-eng，状态=`可以SR设计` / `待补充SR设计`），SR-F（功能性系统需求）反馈优先于新 BA 处理。将 BA 中每个 CU（配置单元）的引擎配置能力模型 IPO（输入-处理-输出）转化为 SysReq（系统需求）引擎能力功能定义（四要素：功能描述、功能流程、功能规则、功能接口），完成 SR-F 分类映射，对 A1 引擎配置类 SR-F 写回 `25-*.md`（平台引擎资产），推进 BA/SysReq 状态。引擎配置类 SR-F 必须保留 `业务支撑来源 → CU 配置页面/子页面 → 引擎处理 → 业务运行能力` 追溯；功能规则和接口部分留"到 PA（产品架构）阶段收敛"的空间，在功能定义中标注 `PA层收敛`。
>
> **输入 → 输出**：BA A1 节点（≥1 个，`可以SR设计` / `待补充SR设计`）/ 有反馈的 SR-F 节点 → SR-F 功能定义（四要素）+ 分类映射 + 25 写回（A1 引擎配置类）+ BA 推进为 `已SR设计` + SR-F 推进为 `待产品架构`
>
> **适用场景**：瀑布式系统设计第3步 · 引擎配置路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.1 | **修订**：2026-06-29

## 一、职责定位

### 本 Skill 目的与目标

> **主要目的**：取信于业务开发人员——「引擎能力可配置」。把 BA 的 CU 引擎配置能力模型转化为 SysReq 的引擎能力功能规格——逐 CU 划定系统行为边界：接收什么配置输入、如何处理配置、生成什么业务运行能力。
>
> **目标**：功能规格独立于实现——不绑定技术栈或组件方案，一步不落，一步不多余。本 Skill 只消费 wft02-eng 已确认的 CU IPO，不重新拆分 CU、不重新定义配置页面边界。引擎配置类功能规则和接口部分在 PA 阶段才能收敛的，标注 `PA层收敛`，不在此步强求确定。SysReq 是 wft05 映射 PA 组件的输入。

本 Skill 负责：

1. **接收**——接收人类选中的 ≥1 个 BA A1 节点（`可以SR设计` / `待补充SR设计`），校验状态/类型/数量/引擎分布，不合格则说明原因并退出
2. **匹配**——按 `EOS 引擎 + CU 类型 + 25-Q 结论 + Output 运行能力类别 + 关键扩展点` 复合键匹配已有 SR-F 节点——复合键一致且功能定义一致的 SR-F 直接归入已有节点（Q1=直接引用）、同键但能力需扩展则扩展归入（Q2=扩展），无可用节点则新建（Q3=新建）
3. **展开**——将 BA A1 节点的每个 CU IPO 转化为 SR-F 引擎能力功能定义四要素，沿三步推进：3a 逐 CU 生成功能定义 → 3b CU 间依赖链校验 → 3c 引擎能力边界确认与 PA层收敛标注
4. **分类**——基于功能定义，按 SR-F 分类规则执行分类映射（Layer 1 引擎能力类型 → Layer 3 SysReq 末级节点，默认无中间层）
5. **校验**——校验 SR-F 产出覆盖 BA 节点中所有 CU（消费完备性），校验分类合规性
6. **写回**——人类确认方案后统一将 A1 引擎配置类 SR-F 功能定义写回 `25-*.md`（含 PA层收敛项 + BA IPO 追溯），推进 BA/SysReq 状态；25-Q3 仅在上游已完成 25 模型扩展确认时正式写入 25，否则只在 SR-F 中保留 `[推断]` 阶段性功能定义和模型扩展待办
7. **推进**——人类确认时 BA 节点推进为 `已SR设计`，SR-F 节点推进为 `待产品架构`（wft05 可即时进入）

本 Skill 不负责：

1. 业务功能需求（A2（角色场景产品）/Bn（平台输出产品）BA → `wft03-biz-ba2sr`）
2. 非功能需求（STR-NFR，非功能相关方需求 → `wft04a`）
3. 用户可操作功能和资产管理功能的 SR-F 展开（→ `wft03-biz-ba2sr`）
4. PA（产品架构）组件设计与详细定义（由 wft05 承担）
5. CU 拆分/合并、配置页面/子页面边界重切、CU IPO 重写（回退 `wft02-eng-str2ba`）

## 二、输入与路由

### 标准输入包

1. 本轮处理对象（BA A1 节点 ≥1 个 / 有反馈的 SR-F 节点）
2. 上游追溯链（`04-*.md`（业务架构文档）BA A1 节点正文块 / CU IPO 清单 / 配置页面或子页面定位 / 业务支撑来源 / 25 对齐结论 / 引擎能力边界陈述 / 变更影响声明）
3. 已有 SR-F 架构资产（`05-*.md`（系统需求架构文档）树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`25`（平台引擎资产）EOS 引擎 + `23`（产品架构资产）产品架构）
5. 重入反馈

### Step Start 路由

#### 第一层：入口容错

| 条件 | 路由 |
|------|------|
| 无 BA A1 节点选中 + 无待处理反馈的 SR-F 节点 | 输出"无待处理对象"→ **退出** |
| BA A1 节点状态不是 `可以SR设计` / `待补充SR设计` | 输出状态分布，列出不合格节点及其当前状态 → **退出** |
| 选中 BA 节点含 A2/Bn 产品类型 | 输出类型分布（A2/Bn → `wft03-biz`），要求分路径 → **退出** |
| BA A1 节点缺少 CU IPO 或配置页面/子页面定位 | 输出"缺少 CU IPO / 配置页面边界，需回退 wft02-eng" → **退出** |
| CU IPO 缺少业务支撑来源或业务运行能力 Output | 输出缺项清单，要求回退 wft02-eng 补齐；若人类确认阶段性处理，则标注 `分解依据不足` |
| CU 为 `[待触发]` 占位 | 不生成 SR-F，Step End 列入占位 CU 清单 |
| BA A1 节点分属不同 EOS 引擎且无依赖关联 | 输出引擎分布，提示可能需分批处理 → 人类确认后继续或分批 |
| 选中 BA A1 节点 >5 个 | 输出"本轮 N 个 BA A1 节点，SR-F 设计工作量大，建议分批"，提示节点 ID 列表 → 人类确认后继续，不强制退出 |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]` 的 SR-F 节点 | 本轮先处理全部有反馈的 SR-F 节点 |
| 存在 `待确认方案` + 反馈区为空的 SR-F 节点 | 不阻塞新 BA；Step End 时提示 |

#### 第三层：新 BA A1 节点路由

SR-F 反馈处理完后，逐 BA A1 节点处理新设计 → Step 3。

#### 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 SR-F 节点的功能定义或分类归属，因此 Step 3 必须基于 Step 2 修改后的 SR-F 节点当前内容，不得使用 Step 1 加载时的缓存旧状态。

### BA 节点生命周期

#### 推进态（按顺序）

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `可以SR设计` | BA 方案已确认，wft03 可启动 SR-F 展开 | wft02-eng 人类确认方案 |
| `在SR设计` | wft03 正在展开 SR-F | 本 Skill 写入 |
| `已SR设计` | SR-F 展开完成，BA 消费版本与当前版本一致 | 本 Skill 处理完成且未检测到上游新版本 |

#### 回退态（上游修订触发）

当新 STR-E 汇入已有 BA 触发 wft02-eng 修订时，BA 内容变更、版本号递增——但本 Skill 可能已基于旧版本生成了 SR-F。修订后 BA 状态按本 Skill 的消费进度分流：

| 修订前状态 | 修订后 | 说明 |
|-----------|--------|------|
| `可以SR设计` | 保持，版本递增 | 本 Skill 尚未消费，不受影响——下次进入时直接读最新版本 |
| `在SR设计` | 保持，版本递增 | 本 Skill 正在处理中，重入时检测版本差异，按变更影响声明增量处理 |
| `已SR设计` | 降为 **`待补充SR设计`** | 本 Skill 之前生成的 SR-F 基于旧版本 BA，本轮新增的 CU 缺少对应 SR-F 节点，需重新介入补全 |

- **`待补充SR设计`**：已消费的 SR-F 节点不动（它们仍然有效），但 BA 新增了配置单元需要生成新的 SR-F。本 Skill 进入后按 BA 的变更影响声明确定增量映射范围，处理完成后若消费版本追上 BA 当前版本 → 推进回 `已SR设计`

BA 节点状态分工：wft02-eng 写入 `待确认方案` / `可以SR设计` / `待补充SR设计`，并在修订时更新版本号；本 Skill 写入 `在SR设计` / `已SR设计`。

#### 版本标记与消费追溯

- BA 节点初次生成时版本 = `v1`（由 wft02-eng 设置）
- 每次因新 STR-E 触发 wft02-eng 修订，版本号递增
- 本 Skill 首次消费 BA 时，在对应 SR-F 节点中记录 `BA 消费版本：v{N}`
- 本 Skill 重入同一 BA 时比对版本：若 BA 当前版本 > 已消费版本 → 触发变更感知，按变更影响声明增量处理

### Step 2 · 方案反馈处理

若本轮存在有反馈的 SR-F 节点（`待确认方案` + 反馈区有效），逐个处理。

#### 基本操作

| 操作 | 行为 |
|------|------|
| `[同意]` 或整体确认 | 归档反馈 → SR-F → `待产品架构`（wft05 此时即可进入）；BA → `已SR设计` |
| `[修改]`（局部） | 按反馈调整被指定的功能定义/分类归属 → 更新增量清单 → 回写 `待确认方案` |
| `[驳回]`（局部） | 移除被驳回的功能定义和分类归属 → 其他功能定义 → `已确认` → 回写 `待确认方案` |
| `[驳回]`（整体） | 重做 Step 3 → 回写 `待确认方案` |

> **写回时机**：`[同意]` 后的资产写回分两种情况——若本轮仅有反馈处理（无新 BA），直接进入 Step 5 写回 25 资产；若本轮同时有新 BA 待处理，先执行 Step 3~4 完成新设计，待全部内容经人类确认后由 Step 5 统一写回——避免分批次写回造成资产文件多次局部更新。

#### 分层反馈

SR-F 节点包含多个可独立决策的单元（各 CU 的功能定义四要素、分类归属、PA层收敛标注、BA IPO 追溯关系）。人类可以**局部修改**——整体确认一个 SR-F 节点的同时，要求调整其中某个 CU 的功能定义或某个分类归属，不需要因一个局部问题而驳回整个 SR-F 节点。

AI 读取反馈时按以下规则处理：

- **局部 `[修改]`**：按反馈调整被指定的对象，其涉及的分类归属重新执行。未提及的对象保持原结论不动，不触发全量重新分类。
- **局部修改导致级联**：功能定义变更 → 级联更新分类归属；CU 功能定义变更 → 级联更新依赖链校验结论。级联完成后重新执行受影响对象的一致性验证。
- **SR-F 节点推进条件**：全部反馈处理完成（无未处理的 `[修改]`/`[驳回]` 项）→ SR-F 推进为 `待产品架构`，BA 推进为 `已SR设计`。存在未处理项 → SR-F 保持 `待确认方案`。
- **`[驳回]`（局部）**：移除被驳回的功能定义和分类归属。剩余对象保持原结论。
- **`[驳回]`（整体）**：SR-F 主结构被推翻（"CU 划分根本不对"或"引擎能力边界错误"）→ 重做 Step 3。

#### 反馈区位置

反馈区位于 `05-*.md` 中**每个 SR-F 节点块的末尾**，作为 SR-F 节点正文的最后一个子节。

**AI 写入（Step 3 完成时）** —— 空反馈区：

```

**人类填写（或 AI 从对话转写）** —— 三种操作之一：

### Step 3 · 功能定义设计

将 BA A1 节点的每个 CU IPO 转化为 SR-F 引擎能力功能定义，沿三步推进：先逐 CU 生成功能定义（3a），再校验 CU 间依赖链（3b），最后确认引擎能力边界与 PA层收敛标注（3c）。本步不重新拆分 CU，也不调整配置页面/子页面边界；发现边界问题时回退 wft02-eng。

#### 功能定义四要素（引擎配置类）

| 要素 | 回答的问题 | 引擎配置类侧重 |
|------|-----------|---------------|
| 功能描述 | 系统做什么？ | "系统提供 XX 引擎能力" |
| 功能流程 | 系统几步做完？ | 配置页面接收→校验/解析/保存→编译/发布→运行期装载→业务运行页面生效 |
| 功能规则 | 系统怎么约束？ | 引擎行为约束（部分在 PA 层才能收敛，标注 `PA层收敛`） |
| 功能接口 | 系统跟谁交互？ | 对外 API、数据源、事件通知（内部接口标注 `PA层收敛`） |

**确定性差异**：引擎配置类功能规则和接口部分留"到 PA 阶段收敛"的空间，在功能定义中标注 `PA层收敛`——告知业务方"此处为引擎内部机制，在 PA 组件设计阶段才能确定具体实现方式"。功能描述和功能流程可写到基本确定。

#### 3a · 逐 CU 生成功能定义

对 BA 节点中每个需设计的 CU（排除 `[待触发]` 占位 CU），逐个生成 SR-F 功能定义：

1. 从 CU 的 Input（配置页面/子页面及配置信息）中提取 **功能流程**——系统视角的配置信息接收→处理→运行期表现管线
2. 从 CU 的 Process 和 25 引擎模型定义中提取 **功能规则**——配置合法性、依赖约束、发布约束、运行期约束。若规则在 PA 层才能确定，标注 `PA层收敛`
3. 从 CU 的依赖关系和 Output 中提取 **功能接口**——配置页面接口、元数据存储接口、运行期装载接口、业务运行页面调用接口。内部接口（引擎组件间调用）标注 `PA层收敛`
4. 综合形成 **功能描述**——该引擎能力如何把配置输入转换为业务运行能力，一句话收口
5. 保留追溯：每条 SR-F 必须记录来源 CU、配置页面/子页面、业务支撑来源和业务运行页面能力

**25 对齐延续**：BA 节点中每个 CU 已有 25-Q1/Q2/Q3 对齐结论（来自 wft02-eng Step 3.2）。SR-F 功能定义需延续此对齐关系：
- 25-Q1 的 CU → 功能定义直接引用 25 CU 类型定义中的标准能力项
- 25-Q2 的 CU → 功能定义包含扩展部分，标注扩展来源和方向
- 25-Q3 的 CU → 若已完成 25 模型扩展确认，按新 CU 类型生成正式功能定义；若未完成，仅生成 `[推断]` 阶段性功能定义，标注 `模型扩展待确认`

**Q1/Q2/Q3 SR-F 匹配**：在生成功能定义前，先按复合键匹配已有 SR-F 节点。复合键为 `EOS 引擎 + CU 类型 + 25-Q 结论 + Output 运行能力类别 + 关键扩展点`，避免同一 CU 类型在不同运行输出、页面承载或扩展模式下被误合并：

| 判定 | 条件 | 动作 |
|------|------|------|
| **Q1 直接匹配** | 已有 SR-F 节点复合键一致且功能定义一致 | 复用已有 SR-F → 追加 BA 引用 → 不重新生成 |
| **Q2 扩展匹配** | 已有 SR-F 节点复合键主体一致，但 Output 运行能力类别或关键扩展点需要扩展 | 扩展已有 SR-F → 全量重写功能定义（非增量追加），并记录新增扩展点 |
| **Q3 新建** | 无可用已有 SR-F 节点 | 标记待新建 → 生成功能定义 |

#### 3b · CU 间依赖链校验

基于 25 引擎模型中 CU 的依赖配置单元关系，校验 SR-F 产出是否满足依赖闭环：

| 检查项 | 内容 |
|--------|------|
| **依赖覆盖** | 被依赖的 CU 是否都有对应的 SR-F 节点 |
| **依赖方向** | SR-F 的功能接口是否正确引用了被依赖 CU 的 Output |
| **循环依赖** | CU 之间是否存在循环依赖（25 定义不允许，若发现则标记矛盾待人类裁决） |
| **占位 CU 依赖** | `[待触发]` 占位 CU 被其他 CU 依赖时，标注"依赖未触发 CU，待后续补全" |

#### 3c · 引擎能力边界确认与 PA层收敛标注

1. **边界确认**：校验每个 CU 的功能定义是否在 BA 的引擎能力边界陈述范围内（来自 wft02-eng 的 STR-E 引擎能力边界）。超出边界的标注 `[推断]`
2. **PA层收敛标注**：检查功能规则和功能接口中哪些项无法在本步确定，标注 `PA层收敛` + 收敛原因。PA层收敛项不得影响功能描述和功能流程的确定性
3. **CU 覆盖完备性**：校验 BA 节点中所有需设计的 CU 是否都有对应 SR-F（排除 `[待触发]` 占位 CU；顺序3候选 CU 仅在其候选 IPO 经人类确认允许阶段性展开时生成阶段性 SR-F，且不得被视为正式 25 模型能力）

说明：

1. 本 Skill 不重复做 CU 去重，直接消费 wft02-eng 已确认的首次出现结果
2. `[待触发]` 占位 CU 不生成 SR-F——待后续 STR-E 触发并由 wft02-eng 补充设计后，本 Skill 再补全对应 SR-F
3. 若关键缺口影响功能定义完整性，本轮应标记为 `阶段性方案`，标注阻断项

### Step 4 · 分类映射

基于 Step 3 已生成的功能定义，按 SR-F 分类规则执行分类映射。引用 [92-srf-classification-spec.md](92-srf-classification-spec.md)。

1. **确定第 1 层（引擎能力类型）**：按引擎能力类型归入（表单引擎能力、流程引擎能力、权限引擎能力等），与 25 EOS 引擎分类对齐
2. **确定第 3 层（SysReq 末级节点）**：功能定义本身即 SysReq 末级节点的核心内容
3. **默认不设中间层**：引擎配置类 SR-F 默认只有第 1 层 + 第 3 层，不设中间层（原则 8：引擎能力类不分中间层）
4. **例外补设**：同一引擎类型下 >7 个 SR-F 末级节点或 PA 边界混合时，按 92 spec 补设第 2 层（系统职责分类）
5. 执行 `复用 > 改进 > 新增` 分类宿主决策

**分类原则遵守检查：**

- [ ] 先有末级（功能定义），再做聚合（原则 1）
- [ ] 按系统职责聚合，不按业务对象（原则 2）
- [ ] 一节点只归一个主职责（原则 3）
- [ ] 非末级节点能写出一句统一定义（原则 4）
- [ ] 粒度适中（原则 5）
- [ ] 业务域只做上层分区，不替代系统职责（原则 6）
- [ ] 跨域可复用同一职责分类（原则 7）
- [ ] 引擎能力类不分中间层（原则 8）

### Step 5 · 资产写回与落账

#### 资产写回

- 25-Q2：扩展已有配置单元类型 → 追加引用 + 扩展说明 + PA层收敛项
- 25-Q3：若已完成 25 模型扩展确认 → 新建配置单元类型块 → 写入定义 + 引用 + PA层收敛项；若未完成 → 不写正式 25 类型块，仅在 SR-F 和资产写回记录中保留 `阶段性功能定义 + 模型扩展待确认`

**25 写回内容模板**（每个 CU 的功能定义写回）：

```
SR-F 功能定义引用：

  功能描述：<一句话>
