# BA(A1) → SysReq · 引擎配置功能定义与资产化
**Skill ID**: wft03-eng-ba2sr

> **概述**：每次执行处理 ≥1 个 BA（业务架构）A1（平台引擎产品）节点（来自 wft02-eng，状态=`可以SR设计` / `待补充SR设计`），SR-F（功能性系统需求）反馈优先于新 BA 处理。将 BA 中每个 CU（配置单元）的引擎配置能力模型 IPO（输入-处理-输出）转化为 SysReq（系统需求）引擎能力功能定义（四要素：功能描述、功能流程、功能规则、功能接口），完成 SR-F 分类映射，对 A1 引擎配置类 SR-F 写回 `25-*.md`（平台引擎资产），推进 BA/SysReq 状态。引擎配置类 SR-F 必须保留 `业务支撑来源 → CU 配置页面/子页面 → 引擎处理 → 业务运行能力` 追溯；功能规则和接口部分留"到 PA（产品架构）阶段收敛"的空间，在功能定义中标注 `PA层收敛`。
>
> **输入 → 输出**：BA A1 节点（≥1 个，`可以SR设计` / `待补充SR设计`）/ 有反馈的 SR-F 节点 → SR-F 功能定义（四要素）+ 分类映射；正式 SR-F 经人类确认后写回 25（A1 引擎配置类）、BA 推进为 `已SR设计`、SR-F 推进为 `待产品架构`；25-Q3 或待确认模型只形成阶段性 SR-F 和模型扩展待办
>
> **适用场景**：瀑布式系统设计第3步 · 引擎配置路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.7 | **修订**：2026-07-01

## 一、职责定位

### 1.1 本 Skill 目的与目标

> **主要目的**：取信于业务开发人员——「引擎能力可配置」。把 BA 的 CU 引擎配置能力模型转化为 SysReq 的引擎能力功能规格——逐 CU 划定系统行为边界：接收什么配置输入、如何处理配置、生成什么业务运行能力。
>
> **目标**：功能规格独立于实现——不绑定技术栈或组件方案，一步不落，一步不多余。本 Skill 只消费 wft02-eng 已确认的 CU IPO，不重新拆分 CU、不重新定义配置页面边界。引擎配置类功能规则和接口部分在 PA 阶段才能收敛的，标注 `PA层收敛`，不在此步强求确定。SysReq 是 wft05 映射 PA 组件的输入。

本 Skill 负责：

1. **接收**——接收人类选中的 ≥1 个 BA A1 节点（`可以SR设计` / `待补充SR设计`），校验状态/类型/数量/引擎分布/CU IPO 完整性，不合格则说明原因并退出；BA 缺少 CU IPO 或业务支撑来源时将其推进为 `需wft02修订`（附带缺失说明），交由 wft02-eng 修订
2. **匹配**——按 `EOS 引擎 + CU 类型 + 25-Q 结论 + Output 运行能力类别 + 关键扩展点` 复合键匹配已有 SR-F 节点——复合键一致且功能定义一致的 SR-F 直接归入已有节点（Q1=直接引用）、同键但能力需扩展则扩展归入（Q2=扩展），无可用节点则新建（Q3=新建）
3. **展开**——将 BA A1 节点的每个 CU IPO 转化为 SR-F 引擎能力功能定义四要素，沿三步推进：3a 逐 CU 生成功能定义 → 3b CU 间依赖链校验 → 3c 引擎能力边界确认与 PA层收敛标注
4. **分类**——基于功能定义，按 SR-F 分类规则执行分类映射（Layer 1 引擎能力类型 → Layer 3 SysReq 末级节点，默认无中间层）
5. **校验**——校验 SR-F 产出覆盖 BA 节点中所有 CU（消费完备性），校验分类合规性
6. **写回**——人类确认方案后统一将 A1 引擎配置类 SR-F 功能定义写回 `25-*.md`（含 PA层收敛项 + BA IPO 追溯），推进 BA/SysReq 状态；25-Q3 仅在上游已完成 25 模型扩展确认时正式写入 25，否则只在 SR-F 中保留 `[推断]` 阶段性功能定义和模型扩展待办
7. **推进**——人类确认正式 SR-F 时，BA 节点推进为 `已SR设计`，SR-F 节点推进为 `待产品架构`（wft05 可即时进入）；阶段性 SR-F 只能作为候选 PA 输入，需保留 `模型扩展待确认` 或 `待模型确认` 标记
8. **承接下游退回**——接收 wft05-eng 退回的 `需wft03修订` SR-F 节点，读取缺失说明，按人类改进方案修订 SR-F，人类确认后重新推进为 `待产品架构`；若缺失根因在 BA，可进一步将 BA 推进为 `需wft02修订`

本 Skill 不负责：

1. 业务功能需求（A2（角色场景产品）/Bn（平台输出产品）BA → `wft03-biz-ba2sr`）
2. 非功能需求（STR-NFR，非功能相关方需求 → `wft04a`）
3. 用户可操作功能和资产管理功能的 SR-F 展开（→ `wft03-biz-ba2sr`）
4. PA（产品架构）组件设计与详细定义（由 wft05 承担）
5. CU 拆分/合并、配置页面/子页面边界重切、CU IPO 重写（回退 `wft02-eng-str2ba`）

### 1.2 与 25 号资产的关系

25 号资产在本 Skill 中是 SR-F 功能定义的引擎模型参照和写回目标。wft03-eng 读取 25 的 CU 类型定义、依赖关系和状态，用于校验 BA IPO 是否可转化为系统功能；只在人类确认 SR-F 且模型状态允许时，才把 A1 引擎配置类功能定义引用写回 25。

| 25 / BA 情况 | wft03-eng 处理 |
|--------------|----------------|
| 25-Q1 且 CU 类型 `已确认` | 可生成正式 SR-F，确认后写回 25 功能定义引用 |
| 25-Q2 或 CU 类型 `待确认` | 生成带 `待模型确认` 的阶段性 SR-F；可进入 wft05 做阶段性 PA，但不得作为冻结依据 |
| 25-Q3 未完成模型扩展确认 | 只生成 `[推断]` 阶段性功能定义和模型扩展待办，不正式写回 25 |
| BA 缺 CU IPO / 业务支撑来源 / Output | 不补写 SR-F，退回 wft02-eng 修订 |

## 二、输入与路由

### 2.1 标准输入包

1. 本轮处理对象（BA A1 节点 ≥1 个 / 有反馈的 SR-F 节点 / `需wft03修订` 的 SR-F 节点）
2. 上游追溯链（`04-*.md`（业务架构文档）BA A1 节点正文块 / CU IPO 清单 / 配置页面或子页面定位 / 业务支撑来源 / 25 对齐结论 / 引擎能力边界陈述 / 变更影响声明）
3. 已有 SR-F 架构资产（`05-*.md`（系统需求架构文档）树画像 / 状态队列 / 节点索引）
4. 相关约束资产（`25`（平台引擎资产）EOS 引擎 + `23`（产品架构资产）产品架构）
5. 重入反馈

### 2.2 Step Start 路由

#### 2.2.1 第一层：入口容错

| 条件 | 路由 |
|------|------|
| BA A1 节点状态为 `需wft02修订` | 输出缺失说明，提示人类回到 wft02-eng 修订 BA → **退出** |
| BA A1 节点状态不是 `可以SR设计` / `待补充SR设计` | 输出状态分布，列出不合格节点及其当前状态 → **退出** |
| 选中 BA 节点含 A2/Bn 产品类型 | 输出类型分布（A2/Bn → `wft03-biz`），要求分路径 → **退出** |
| SR-F 节点状态为 `需wft03修订` 且人类未提供改进方案 | 输出 wft05-eng 缺失说明，提示人类提供改进方案或与 wft05-eng 协同确认 → **退出** |
| BA A1 节点缺少 CU IPO 或配置页面/子页面定位 | 输出"缺少 CU IPO / 配置页面边界" → BA → `需wft02修订`（附带缺失说明），交由 wft02-eng 修订 → **退出** |
| CU IPO 缺少业务支撑来源或业务运行能力 Output | 输出缺项清单 → BA → `需wft02修订`（附带缺失说明），交由 wft02-eng 修订 → **退出** |
| CU 为 `[待触发]` 占位 | 不生成 SR-F，Step End 列入占位 CU 清单 |
| BA A1 节点分属不同 EOS 引擎且无依赖关联 | 输出引擎分布，提示可能需分批处理 → 人类确认后继续或分批 |
| 选中 BA A1 节点 >5 个 | 输出"本轮 N 个 BA A1 节点，SR-F 设计工作量大，建议分批"，提示节点 ID 列表 → 人类确认后继续，不强制退出 |
| 无 BA A1 节点选中 + 无待处理反馈的 SR-F 节点 + 无 `需wft03修订` 的 SR-F 节点 | 输出"无待处理对象"→ **退出** |

入口容错只判断"能否进入本 Skill"。凡因 BA 状态、类型、CU IPO、业务支撑来源或运行能力 Output 缺失被拦截的对象，不得在本轮临时补写 SR-F；必须回到 wft02-eng 修订 BA，或由人类提供明确改进方案后重入。

#### 2.2.2 第二层：SR-F 重入优先

| 条件 | 路由 |
|------|------|
| 存在 `需wft03修订` 的 SR-F 节点 | 读取 wft05-eng 的缺失说明 → 按人类改进方案修订 SR-F → 人类确认 → SR-F → `待产品架构`；若缺失根因在 BA，可进一步将 BA → `需wft02修订` |
| 存在 `待确认方案` + 条目 `确认状态` 字段中有人类标注且无 `[已处理]` 的 SR-F 节点 | 本轮先处理全部有反馈的 SR-F 节点 |
| 存在 `待确认方案` + 所有条目均已 `[同意]` + `[已处理]` 的 SR-F 节点 | 不阻塞新 BA；Step End 时提示 |

#### 2.2.3 第三层：新 BA A1 节点路由

SR-F 重入处理完后，逐 BA A1 节点处理新设计 → Step 3。

#### 2.2.4 同次运行顺序约束

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 SR-F 节点的功能定义或分类归属，因此 Step 3 必须基于 Step 2 修改后的 SR-F 节点当前内容，不得使用 Step 1 加载时的缓存旧状态。

若同次运行既有反馈处理又有新 BA A1：先完成全部 SR-F 反馈处理；被确认的 SR-F 可进入 Step 5 写回队列，新 BA A1 则继续进入 Step 3~4 形成 `待确认方案`。Step 5 只写已确认 SR-F，不写本轮新生成但尚未确认的 SR-F。

## 四、引擎功能定义设计基础

### 4.1 SR-F 架构承接规则

SR-F 必须从 BA 的 CU IPO 逐项承接，不重新拆分 CU、不重切配置页面/子页面边界。每个 SR-F 至少保留 `来源 BA / 来源 CU / 配置页面或子页面 / 业务支撑来源 / Output 运行能力 / 25 对齐结论`，确保 wft05-eng 可以从功能定义追溯到业务运行能力和引擎模型。

### 4.2 25 写回与阶段性规则

25 写回只发生在 Step 2 人类确认后的正式 SR-F 上。若 CU 类型状态为 `待确认` 或 BA 为 25-Q3 候选，SR-F 只能标注为阶段性；该 SR-F 可帮助 wft05 识别 PA 候选组件，但不得推进为“最终可冻结”的 PA 输入。阶段性 SR-F 的 25 内容只记录扩展建议或待办，不落成正式类型定义。

### 4.3 PA 层收敛分类

`PA层收敛` 必须区分两类：`PA层必收敛` 和 `PA层可延后`。影响组件边界、接口契约、数据存储、事务一致性、权限校验、安全隔离、运行期装载路径的项目属于 `PA层必收敛`，必须明确传递给 wft05-eng；仅影响内部算法、字段级实现或开发期技术细节的项目属于 `PA层可延后`。若无法判定，应标注原因，供人类确认。

### BA 节点生命周期

#### 推进态（按顺序）

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `可以SR设计` | BA 方案已确认，wft03 可启动 SR-F 展开 | wft02-eng 人类确认方案 |
| `在SR设计` | wft03 正在展开 SR-F | 本 Skill 写入 |
| `已SR设计` | SR-F 展开完成，BA 消费版本与当前版本一致 | 本 Skill 处理完成且未检测到上游新版本 |

#### 上游变更态（BA 修订触发 SR-F 版本递增）

当新 STR-E 汇入已有 BA 触发 wft02-eng 修订时，BA 内容变更、版本号递增——但本 Skill 可能已基于旧版本生成了 SR-F。修订后 BA 状态按本 Skill 的消费进度分流：

| 修订前状态 | 修订后 | 说明 |
|-----------|--------|------|
| `可以SR设计` | 保持，版本递增 | 本 Skill 尚未消费，不受影响——下次进入时直接读最新版本 |
| `在SR设计` | 保持，版本递增 | 本 Skill 正在处理中，重入时检测版本差异，按变更影响声明增量处理 |
| `已SR设计` | 降为 **`待补充SR设计`** | 本 Skill 之前生成的 SR-F 基于旧版本 BA，本轮新增的 CU 缺少对应 SR-F 节点，需重新介入补全 |

- **`待补充SR设计`**：已消费的 SR-F 节点不动（它们仍然有效），但 BA 新增了配置单元需要生成新的 SR-F。本 Skill 进入后按 BA 的变更影响声明确定增量映射范围，处理完成后若消费版本追上 BA 当前版本 → 推进回 `已SR设计`

BA 节点状态分工：wft02-eng 写入 `待确认方案` / `可以SR设计` / `待补充SR设计` / `需wft02修订`，并在修订时更新版本号；本 Skill 写入 `在SR设计` / `已SR设计`。SR-F 节点状态由本 Skill 写入：`待确认方案` / `待产品架构` / `需wft03修订`。

#### 下游退回态（wft05-eng 退回触发）

下游退回 = wft05-eng 在 PA 设计中发现 SR-F 方案存在缺陷（如缺少来源 CU、配置页面/子页面、业务运行能力），无法继续 PA 设计，将 SR-F 退回本 Skill 修订。

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| **`需wft03修订`** | SR-F 方案存在缺陷，需本 Skill 修订后重新交付 | wft05-eng 写入，附带缺失说明 |

本 Skill 重入 `需wft03修订` 的 SR-F 时：读取 wft05-eng 的缺失说明 → 按人类改进方案修订 SR-F → 人类确认 → SR-F → `待产品架构`。若缺失根因在 BA，本 Skill 可进一步将 BA 推进为 `需wft02修订`。

#### 版本标记与消费追溯

- BA 节点初次生成时版本 = `v1`（由 wft02-eng 设置）
- 每次因新 STR-E 触发 wft02-eng 修订，版本号递增
- 本 Skill 首次消费 BA 时，在对应 SR-F 节点中记录 `BA 消费版本：v{N}`
- 本 Skill 重入同一 BA 时比对版本：若 BA 当前版本 > 已消费版本 → 触发变更感知，按变更影响声明增量处理

### Step 2 · 方案反馈处理

#### 人类审阅规则

人类对 SR-F 方案做两层审阅——条目级为权威留痕，整体确认为快捷补全。人类直接在条目 `确认状态` 字段标注意见，AI 处理完后追加 `[已处理]`。

1. **条目级反馈**（权威）：对 SR-F 节点的独立决策单元（各 CU 的功能定义四要素、分类归属、PA 层收敛标注、BA IPO 追溯关系）逐条标注。已 `[同意]` + `[已处理]` 的条目在后续轮次中默认跳过——人类若改变主意，直接写入新的 `[修改]` / `[驳回]` 即覆盖旧标注，覆盖后 `[已处理]` 自然消失，条目回到待处理状态
2. **整体确认**（快捷）：人类在对话中说「整体确认」→ AI 对**无条目级标注的条目**写入 `[同意]`（标注来源=`整体确认`），随后对这批条目追加 `[已处理]`——与条目级 `[同意]` 等效

**冲突消解**：条目级 > 整体确认。两者并存时，条目级标注保留原结论不受影响，整体确认仅补未标注的剩余条目。

#### AI 处理规则

若本轮存在 `待确认方案` 的 SR-F 节点，AI 扫描各条目的 `确认状态` 字段——有人类标注但无 `[已处理]` 的条目为本轮待处理对象：

1. **扫描待处理条目**——遍历 SR-F 节点各条目，筛选 `确认状态` 中有人类标记（`[同意]` / `[修改]` / `[驳回]`）但无 `[已处理]` 的条目
2. **收集整体确认**——若人类说了「整体确认」，对**无条目级标注的剩余条目**写入 `[同意]`（标注来源=`整体确认`），随后为该批条目追加 `[已处理]`
3. **执行**——逐条目按人类意见处理：

| 人类意见 | AI处理 |
|---------|------|
| `[同意]`（明示或整体确认补全） | 条目维持不变 → 在 `确认状态` 字段末尾追加 `[已处理]` |
| `[修改]`：修改内容 | 按反馈调整该条目 → 仅重对齐受影响部分 → 追加 `[已处理]` |
| `[驳回]`：驳回理由 | 移除被驳回的功能定义和分类归属 → 其余对象保持原结论 → 追加 `[已处理]`。若人类标注"整体驳回"，重做 Step 3 → 追加 `[已处理]` |

4. **级联处理**——局部修改导致级联时（功能定义变更 → 级联更新分类归属；CU 功能定义变更 → 级联更新依赖链校验结论），级联完成后重新执行受影响对象的一致性验证。
5. **判定 SR-F 节点状态**：
   - 全部条目 `确认状态` 含 `[同意]` + `[已处理]` → SR-F → `待产品架构`（wft05 此时即可进入）；BA → `已SR设计`
   - 任一条目含 `[修改]` 或 `[驳回]`（不论是否已追加 `[已处理]`）→ SR-F 保持 `待确认方案`
6. 写入 `### 反馈处理记录`（含本轮各条目人类意见+来源+处理动作+时间戳）

**多轮迭代**：`[同意]` + `[已处理]` 双标记完整的条目本轮默认跳过（已确认且已执行）。但上游 BA 修订可能改变已有 SR-F 对象——AI 在处理修订时先清除受影响条目的 `[已处理]` 标记（保留 `[同意]`），使条目回到待处理状态，供人类重新审阅。上游变更优先于人类确认。

**写回时机**：若本轮仅有反馈处理（无新 BA A1），进入 Step 5 按已确认 SR-F 方案执行 25 写回判定；25-Q1 可写回，25-Q2 需带扩展说明并满足人类确认，不需人类二次确认。若本轮同时有新 BA A1，先执行 Step 3~4 完成新增或修订设计；Step 5 仍只写本轮已确认的正式 SR-F，新产出的 `待确认方案` 或阶段性 SR-F 留待下一轮人类确认或模型补齐后再写资产。25-Q3 只有上游已完成 25 模型扩展确认时才正式写入 25，否则只作为 SR-F 节点内的阶段性功能定义和模型扩展待办保留。

#### 确认状态字段

各条目的 `确认状态` 字段位于 `05-*.md` SR-F 节点块各条目正文中，不再设独立的反馈区节。人类在字段中直接标注，AI 处理后追加 `[已处理]`。格式同 [wft02-biz Step 2](eos-wft02-biz-str2ba.md)。

### Step 3 · 功能定义设计

将 BA A1 节点的每个 CU IPO 转化为 SR-F 引擎能力功能定义，沿三步推进：先逐 CU 生成功能定义（3a），再校验 CU 间依赖链（3b），最后确认引擎能力边界与 PA层收敛标注（3c）。本步不重新拆分 CU，也不调整配置页面/子页面边界；发现边界问题时回退 wft02-eng。

#### 功能定义四要素（引擎配置类）

| 要素 | 回答的问题 | 引擎配置类侧重 |
|------|-----------|---------------|
| 功能描述 | 系统做什么？ | "系统提供 XX 引擎能力" |
| 功能流程 | 系统几步做完？ | 配置页面接收→校验/解析/保存→编译/发布→运行期装载→业务运行页面生效 |
| 功能规则 | 系统怎么约束？ | 引擎行为约束（部分在 PA 层才能收敛，标注 `PA层收敛`） |
| 功能接口 | 系统跟谁交互？ | 对外 API、数据源、事件通知（内部接口标注 `PA层收敛`） |

**确定性差异**：引擎配置类功能规则和接口部分留"到 PA 阶段收敛"的空间，在功能定义中标注 `PA层收敛`——告知业务方"此处为引擎内部机制，在 PA 组件设计阶段才能确定具体实现方式"。功能描述和功能流程可写到基本确定。凡影响组件边界、接口契约、数据存储、事务一致性、权限校验、安全隔离、运行期装载路径的项目，必须进一步标注为 `PA层必收敛`；其他实现细节可标注为 `PA层可延后`。

#### 3a · 逐 CU 生成功能定义

对 BA 节点中每个需设计的 CU（排除 `[待触发]` 占位 CU），逐个生成 SR-F 功能定义：

1. 从 CU 的 Input（配置页面/子页面及配置信息）中提取 **功能流程**——系统视角的配置信息接收→处理→运行期表现管线
2. 从 CU 的 Process 和 25 引擎模型定义中提取 **功能规则**——配置合法性、依赖约束、发布约束、运行期约束。若规则在 PA 层才能确定，标注 `PA层收敛`
3. 从 CU 的依赖关系和 Output 中提取 **功能接口**——配置页面接口、元数据存储接口、运行期装载接口、业务运行页面调用接口。内部接口（引擎组件间调用）标注 `PA层收敛`
4. 综合形成 **功能描述**——该引擎能力如何把配置输入转换为业务运行能力，一句话收口
5. 保留追溯：每条 SR-F 必须记录来源 CU、配置页面/子页面、业务支撑来源和业务运行页面能力

**25 对齐延续**：BA 节点中每个 CU 已有 25-Q1/Q2/Q3 对齐结论（来自 wft02-eng Step 3.2）。SR-F 功能定义需延续此对齐关系：
- 25-Q1 的 CU → 功能定义直接引用 25 CU 类型定义中的标准能力项
- 25-Q2 的 CU → 功能定义包含扩展部分，标注扩展来源和方向
- 25-Q3 的 CU → 若已完成 25 模型扩展确认，按新 CU 类型生成正式功能定义；若未完成，仅生成 `[推断]` 阶段性功能定义，标注 `模型扩展待确认`，并将 SR-F 标记为阶段性输入，不作为 PA 冻结依据

**Q1/Q2/Q3 SR-F 匹配**：在生成功能定义前，先按复合键匹配已有 SR-F 节点。复合键为 `EOS 引擎 + CU 类型 + 25-Q 结论 + Output 运行能力类别 + 关键扩展点`，避免同一 CU 类型在不同运行输出、页面承载或扩展模式下被误合并：

| 判定 | 条件 | 动作 |
|------|------|------|
| **Q1 直接匹配** | 已有 SR-F 节点复合键一致且功能定义一致 | 复用已有 SR-F → 追加 BA 引用 → 不重新生成 |
| **Q2 扩展匹配** | 已有 SR-F 节点复合键主体一致，但 Output 运行能力类别或关键扩展点需要扩展 | 扩展已有 SR-F → 全量重写功能定义（非增量追加），并记录新增扩展点 |
| **Q3 新建** | 无可用已有 SR-F 节点 | 标记待新建 → 生成功能定义 |

#### 3b · CU 间依赖链校验

基于 25 引擎模型中 CU 的依赖配置单元关系，校验 SR-F 产出是否满足依赖闭环：

| 检查项 | 内容 |
|--------|------|
| **依赖覆盖** | 被依赖的 CU 是否都有对应的 SR-F 节点 |
| **依赖方向** | SR-F 的功能接口是否正确引用了被依赖 CU 的 Output |
| **循环依赖** | CU 之间是否存在循环依赖（25 定义不允许，若发现则标记矛盾待人类裁决） |
| **占位 CU 依赖** | `[待触发]` 占位 CU 被其他 CU 依赖时，标注"依赖未触发 CU，待后续补全" |

#### 3c · 引擎能力边界确认与 PA层收敛标注

1. **边界确认**：校验每个 CU 的功能定义是否在 BA 的引擎能力边界陈述范围内（来自 wft02-eng 的 STR-E 引擎能力边界）。超出边界的标注 `[推断]`
2. **PA层收敛标注**：检查功能规则和功能接口中哪些项无法在本步确定，标注 `PA层收敛` + 收敛原因，并区分 `PA层必收敛` / `PA层可延后`。PA层收敛项不得影响功能描述和功能流程的确定性
3. **CU 覆盖完备性**：校验 BA 节点中所有需设计的 CU 是否都有对应 SR-F（排除 `[待触发]` 占位 CU；顺序3候选 CU 仅在其候选 IPO 经人类确认允许阶段性展开时生成阶段性 SR-F，且不得被视为正式 25 模型能力）

说明：

1. 本 Skill 不重复做 CU 去重，直接消费 wft02-eng 已确认的首次出现结果
2. `[待触发]` 占位 CU 不生成 SR-F——待后续 STR-E 触发并由 wft02-eng 补充设计后，本 Skill 再补全对应 SR-F
3. 若关键缺口影响功能定义完整性，本轮应标记为 `阶段性方案`，标注阻断项

### Step 4 · 分类映射

基于 Step 3 已生成的功能定义，按 SR-F 分类规则执行分类映射。引用 [92-srf-classification-spec.md](92-srf-classification-spec.md)。

1. **确定第 1 层（引擎能力类型）**：按引擎能力类型归入（表单引擎能力、流程引擎能力、权限引擎能力等），与 25 EOS 引擎分类对齐
2. **确定第 3 层（SysReq 末级节点）**：功能定义本身即 SysReq 末级节点的核心内容
3. **默认不设中间层**：引擎配置类 SR-F 默认只有第 1 层 + 第 3 层，不设中间层（原则 8：引擎能力类不分中间层）
4. **例外补设**：同一引擎类型下 >7 个 SR-F 末级节点或 PA 边界混合时，按 92 spec 补设第 2 层（系统职责分类）
5. 执行 `复用 > 改进 > 新增` 分类宿主决策

**分类原则遵守检查：**

- [ ] 先有末级（功能定义），再做聚合（原则 1）
- [ ] 按系统职责聚合，不按业务对象（原则 2）
- [ ] 一节点只归一个主职责（原则 3）
- [ ] 非末级节点能写出一句统一定义（原则 4）
- [ ] 粒度适中（原则 5）
- [ ] 业务域只做上层分区，不替代系统职责（原则 6）
- [ ] 跨域可复用同一职责分类（原则 7）
- [ ] 引擎能力类不分中间层（原则 8）

   - 三步设计流：3a（逐 CU 功能定义）→ 3b（依赖链校验）→ 3c（边界确认 + PA层收敛标注），替代原 3b 单一分支
   - 25 对齐延续：功能定义延续 wft02-eng 已建立的 25-Q1/Q2/Q3 对齐关系
   - BA 节点生命周期：采用版本标记 + 消费追溯模式（对齐 wft02-biz/eng），不用活动锁
