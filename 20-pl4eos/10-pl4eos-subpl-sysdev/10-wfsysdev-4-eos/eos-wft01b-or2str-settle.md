# OR → STR 资产化 · Skill B · 架构资产写回
**Skill ID**: wft01b · **Phase 2: OR → STR Asset Settlement**

> **概述**：接收 wft01a 已确认（状态=`可以资产化`) 且已完成资产对准的 STR 节点，按三条对称路径将原子单元写入对应架构资产——STR-F 的信息载体写回 23（产品架构）+ 26（文档定义）+ 22（场景指标），STR-E 的配置单元写回 25（引擎模型），STR-NFR 的量化指标写回 22（NFR 分类体系）。Q1 直接写回无需人类介入；Q2/Q3 形成资产建议后停下来等人类确认再写回。全部写回完成后推进 `01/02` 状态到 `待分解分配`。
>
> **输入 → 输出**：已确认且已完成资产对准的 STR 节点（状态=`可以资产化`）→ 原子单元写入对应架构资产（`23/26/22` 或 `25` 或 `22`）+ `01/02` 状态推进 + 追溯引用同步
>
> **适用场景**：瀑布式系统设计第1步 · Phase 2。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v2.0 | **修订**：2026-06-23

---

## 一、职责定位

Skill B 负责：

1. 接收已确认且已完成资产对准的 STR 节点
2. 按三条对称路径，将原子单元写入对应架构资产
3. Q1 直接写回；Q2/Q3 形成资产建议，等人类确认后写回
4. 维护 23 / 26 / 22 / 25 资产的文档引用、配置单元引用和指标引用
5. 同步维护资产全貌 / 分片索引 / 状态队列 / 追溯引用
6. 在 `02-*.md` 记录本轮写回摘要
7. 全部写回完成后推进 STR 节点到 `待分解分配`，对应 OR 条目到 `已分配`

Skill B 不负责：

1. 重做 OR→STR 主结构设计
2. 修改 Skill A 已确认的节点边界、承接关系和原子单元定义
3. 重新判定资产归属——对准已在 wft01a 完成
4. 生成下游详细定义
5. 业务人员确认

## 二、输入与路由

### 标准输入包

1. 已确认且已完成资产对准的 STR 节点（状态=`可以资产化`）
2. 当前 23 / 26 / 22 / 25 资产内容（按路径加载）
3. 树画像 / 资产全貌、状态队列、节点索引与本轮正文块定位信息
4. 资产建议反馈（如存在待确认资产建议）

### 进入条件

本轮处理的 STR 节点满足以下条件时可进入本 Skill：

1. 当前状态为 `可以资产化`
2. 原子单元已完成资产对准（含 Q1/Q2/Q3 判定和目标资产块引用）

若存量节点仍停留在 `待确认方案` 且反馈区已有 `[同意]`，AI 应先按 wft01a 规则将节点状态推进为 `可以资产化`，再进入本 Skill。

### 三路径数据路由

| 节点类型 | 路由 | 原子单元 | 目标资产 | 写回动作 |
|---------|------|---------|---------|---------|
| STR-F | 业务产品路径（§三） | 信息载体 + 场景指标 | 23 + 26 + 22 | 文档引用写回 23/26，场景指标写回 22 |
| STR-E | 平台引擎路径（§四） | 配置单元 | 25 | 配置单元引用写回 25 |
| STR-NFR | 质量需求路径（§五） | 量化指标 | 22 | 指标引用写回 22 |

### 不可修改项

进入 Skill B 后，AI 不能再修改以下内容；如需修改，必须回退 `wft01a`：

1. STR 节点清单、节点类型、节点边界
2. OR→STR 宿主分配
3. 复用 / 改进 / 新增决策
4. 原子单元的定义、边界和职责
5. 原子单元的 Q1/Q2/Q3 对准结果（AI 只能按对准结果执行写回，不能重新判定）

## 三、STR-F 业务产品路径（写回 23 + 26 + 22）

### Step F-Start · 前置校验与路由

| 条件 | 路由 |
|------|------|
| 无 `可以资产化` STR-F 节点 | 输出"无可资产化 STR-F 节点"，检查其他类型 |
| ≥1 个 STR-F 节点 = `可以资产化` | 处理人类选中的全部 STR-F 节点 |
| 该节点已有 `### 资产写回记录` 且所有原子单元均已写回 | 跳过，进入状态推进 |
| 该节点有 `### 资产建议反馈区` 且有待确认项 | 执行 Step F-4 |

### Step F-1 · 资产读取

1. 加载 `02-*.md` STR-F 节点正文块，读取信息载体清单（含 23-Q1/Q2/Q3 判定、23 产品类型/构件类型、26-Q1/Q2/Q3 判定、关系对象）和场景指标（含 22 对准结果）
2. 加载 `23-eos-output-architecture.md` 资产全貌层，定位 Q1 指向的产品类型 / 构件类型正文块
3. 加载 `26-eos-document-definitions.md` 资产全貌层，定位 Q1 指向的文档定义块
4. 加载 `22-eos-nfr-taxonomy.md` 资产全貌层，定位场景指标 Q1 指向的分类维度块
5. Q2/Q3 项只加载候选块全貌层，用于形成正式资产建议

### Step F-2 · 写回 23 + 26 + 22

**全 Q1 时自动写回全部原子单元，无需人类介入。**

#### 写回 23（产品架构资产）

按每个信息载体的 23-Q1 对准结果：

- **Q1**：定位产品类型 / 构件类型正文块 → 在构件类型的文档引用列表中追加该信息载体引用（去重），记录追溯引用 `STR-F-XXX → @comp-XXX`
- **Q2**：定位待扩展的产品类型 / 构件类型正文块 → 按 wft01a 给出的 23 资产建议形成扩展方案 → 进入 Step F-3
- **Q3**：按 wft01a 给出的 23 资产建议形成新增方案（含建议的业务域 / 产品类型 / 构件类型名称和定义）→ 进入 Step F-3

#### 写回 26（文档定义资产）

按每个信息载体的 26-Q1 对准结果：

- **Q1**：定位已有文档定义块 → 更新业务域、产品类型、构件类型、来源 STR 节点、来源 OR、追溯引用
- **Q2**：定位待扩展的文档定义块 → 形成扩展方案 → 进入 Step F-3
- **Q3**：形成新增文档定义方案（含稳定 BLOCK 名称、一句话定义、业务域、产品类型、构件类型）→ 进入 Step F-3

#### 写回 22（场景指标 · NFR 分类体系）

按每个场景指标的 22-Q1 对准结果：

- **Q1**：定位一级类目 / 分类维度 → 在对应分类维度下追加指标引用（含指标名称、指标类别、指标含义、主要数据来源），记录追溯引用
- **Q2**：形成扩展分类维度定义的建议 → 进入 Step F-3
- **Q3**：形成新增分类维度或一级类目的建议 → 进入 Step F-3

#### 写回要求

- 幂等：同一 STR 节点、同一原子单元、同一资产条目的引用不得重复追加
- 去重：23 中已有同产品类型 / 构件类型引用时去重后写入
- 局部写回：只更新本轮相关块，不得重写资产全文
- 全貌同步：所有新增或变更的资产块必须同步维护树画像 / 资产全貌、分片索引、状态队列和追溯引用

### Step F-3 · 资产建议形成与反馈

当存在 Q2 或 Q3 项时，AI 在 `02-*.md` 本轮 STR 节点块的 `### 资产建议反馈区` 中形成正式资产建议：

```text
### 资产建议反馈区

[操作] 同意 / 修改 / 驳回

[23 资产建议]
- 建议 ID：AS-001
- 类型：23-Q3 新增构件类型
- 内容：建议在产品类型"采购管理"下新增构件类型"供应商评估"
- 理由：来自 STR-F-XXX 的 IC-供应商评估表，23 资产无合适承接项
- 候选块 ID：—

[26 资产建议]
-

[22 资产建议]
-
```

反馈分支：

1. `[同意]`：执行写回 → 将资产建议标记为已写回 → 进入 Step F-5
2. `[修改]`：按人类反馈调整建议方案 → 更新 `### 资产建议反馈区` → 重新等待反馈
3. `[驳回]`：若驳回原因是原子单元本身不应存在，回退 `wft01a`；若驳回原因是归属另有目标，重新定位目标资产块并形成新建议

### Step F-4 · 反馈处理

当人类已有回复时，按 Step F-3 的反馈分支执行。

### Step F-5 · 状态推进

- 在 `02-*.md` 本轮 STR 节点块记录 `### 资产写回记录`（含写回日期、写回的资产块引用列表、原子单元→资产块映射）
- 全部原子单元写回完成后，在 `02-*.md` 树画像、状态队列和节点索引中将 STR 节点推进为 `待分解分配`
- 在 `01-*.md` 树画像、状态队列和本轮 OR 块中将对应 OR 条目推进为 `已分配`
- 在 `01/02` 追溯关系中确认 OR→STR→23/26/22 的引用链已记录
- 清空 `### 资产建议反馈区` 待办信号

## 七、资产建议反馈

### 反馈位置

`02-*.md` 对应 STR 节点下的 `### 资产建议反馈区`。

### 推荐交互方式

1. 人类优先通过对话回复
2. AI 先将对话反馈转写到 `### 资产建议反馈区`，再执行写回或调整
3. 对话是反馈输入通道，文档中的反馈区是权威留痕
4. AI 不得只凭对话上下文推进状态或写回资产，必须先完成规范转写

### 反馈格式

```text
### 资产建议反馈区

[操作] 同意 / 修改 / 驳回

[23 资产建议]
-
[25 资产建议]
-
[22 资产建议]
-
[其他说明]
-
```

### 反馈分支

| 操作 | 行为 | 后续路由 |
|------|------|---------|
| `[同意]` | 将资产建议标记为已确认，执行写回 | 进入状态推进 |
| `[修改]` | 按人类反馈调整资产建议，保持 `待反馈` | 重新等待反馈 |
| `[驳回]` | 原子单元归属另有目标 → 重新定位；原子单元不应存在 → 回退 wft01a | 重新写回或回退 |

## 八、Run End 输出

### 单 ID 最小待办原则

1. 每次 Run End 只提示人类处理一个 ID
2. 该 ID 可以是一个资产建议 ID、一个需要回退 wft01a 的 STR 节点 ID、一个待进入 wft02a 的 STR 节点 ID
3. AI 必须同时给出剩余待办 ID 数量
4. 若上一次 Run End 已提示某个当前待办 ID 且该 ID 尚未完成，下次运行必须继续提示同一个 ID
5. 人类回复可以是 `同意`、`修改：……`、`驳回：……`，或 `选择：……`
6. 只有当前 ID 已完成或被人类明确切换时，AI 才能选择下一个当前 ID，并说明切换原因

### Run End 待办类型

| 待办类型 | 何时出现 | 当前 ID | 人类只需做什么 | 下次运行 |
|---------|----------|---------|----------------|----------|
| `反馈资产建议` | 存在 Q2/Q3 资产建议待确认 | 一个资产建议 ID | 回复 `同意 / 修改 / 驳回` | `wft01b` |
| `回退wft01a` | 发现原子单元归属无法写回，需修改设计 | 一个 STR 节点 ID | 回复 `选择进入wft01a` | `wft01a` |
| `选择进入wft02a` | 三条路径全部写回完成，STR 节点已推进到 `待分解分配` | 一个 STR 节点 ID | 回复 `选择进入wft02a` | `wft02a` |

### 输出格式

```text
当前结果：反馈资产建议 / 回退wft01a / 选择进入wft02a
当前待办ID：
是否延续上次待办ID：
剩余待办ID数量：
待办类型：
为什么先处理这个ID：
人类只需回复：
回复会写入：
下次运行：
其他待办摘要：
```
