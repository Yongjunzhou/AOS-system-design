# OR → STR 架构 · Skill B · 文档归属与资产写回
**Skill ID**: wft01b · **Phase 2: OR → STR Settlement**

> **概述**：对 Skill A 已确认的 STR 场景定义执行广义文档节点归属与资产写回。Skill B 按十条信息链顺序逐链处理，每次只处理当前一条信息链上的广义文档 / 信息载体。信息链仅作为分析和分组维度，不作为产品归属单位；AI 必须对当前链上的每个广义文档节点分别判定其所属业务域、产品类型和构件类型，并记录归属依据。若 23 资产无合适承接项，则 AI 给出新增 / 扩展建议，经人类反馈确认后再写回资产。`wft01` 无 Phase 2 下半，全部信息链完成后进入 `wft02` 前准备态。
>
> **输入 → 输出**：已确认的 STR 场景定义 → 当前信息链的广义文档节点归属方案局部写入 `02-*.md` + `23/26/22` 资产按块写回 + 信息链归属状态推进；全部信息链完成后同步推进 `01/02` 状态、索引、追溯引用
>
> **适用场景**：瀑布式系统设计第1步 · Phase 2。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.16 | **修订**：2026-06-20

## 一、职责定位

Skill B 负责：

1. 接收已确认的 STR 场景定义方案
2. 按十条信息链顺序，针对当前信息链中的每个广义文档 / 信息载体节点，分别判定业务域、产品类型、构件类型
3. 按节点所属信息链类型，记录其与执行业务链节点、项目管理构件、资源提供方、公共协同 IPO 输入或被度量对象的关系
4. 基于 23 输出产品架构资产执行文档归属匹配；无合适承接项时提出新增 / 扩展建议
5. 维护 26 广义文档资产定义与归属
6. 维护 23 输出产品架构资产引用
7. 维护 22 非功能需求资产引用或候选线索
8. 推进当前信息链归属状态；全部信息链完成后推进 OR / STR 状态，使节点进入下游 `wft02` 的准备态

Skill B 不负责：

1. 重做 OR→STR 主结构设计
2. 修改 Skill A 已确认的节点边界与承接关系
3. 改变 Skill A 已确认的场景边界、角色、触发和主承接关系
4. 生成下游详细定义

## 二、输入与路由

### 标准输入包

1. 已确认的 STR 场景定义方案
2. 当前 23 / 26 / 22 资产内容
3. 文档归属反馈（如存在待确认归属项）
4. 树画像 / 资产全貌、状态队列、节点索引与本轮正文块定位信息
5. 信息链归属处理状态

### 进入条件

本轮处理的 STR 节点满足以下条件时可进入本 Skill：

1. 当前状态为 `待文档归属`

说明：`待文档归属` 表示该 STR 场景定义已经由 Skill A 完成方案确认，可以进入逐文档归属判定。若存量节点仍停留在 `待确认方案` 且反馈区已有 `[同意]`，AI 应先按 Skill A 规则将节点状态推进为 `待文档归属`，再进入本 Skill。

### 不可修改项

进入 Skill B 后，AI 不能再修改以下内容；如需修改，必须回退 `wft01a`：

1. STR 节点清单
2. 节点类型（STR-F / STR-NFR）
3. 节点边界
4. OR→STR 宿主分配
5. 复用 / 改进 / 新增 决策
6. 主承接关系
7. STR-F 场景定义中的信息载体清单与场景指标清单

## 三、操作流程

### Step 布局总览

| Step | 名称 | 产出 | 写资产 |
|------|------|------|--------|
| Start | 前置校验与路由 | 就绪 / 结束 | — |
| 1 | 资产读取 | 已读取资产内容 | — |
| 2 | 当前链文档清单读取 | 当前链待归属文档清单 | — |
| 3 | 当前链文档归属判定 | 当前链广义文档节点归属方案 | `02-*.md` |
| 4 | 当前链文档归属反馈处理 | 确认后的当前链归属方案 / 回退 | `02-*.md` |
| 5 | 当前链资产写回 | 当前链 23 / 26 / 22 资产更新与写回摘要 | `23-*.md` / `26-*.md` / `22-*.md` / `02-*.md` |
| 6 | 当前链状态推进 | 信息链归属状态更新；全部完成时 OR / STR 状态更新 | `01-*.md` / `02-*.md` |
| End | 运行结束输出 | 人类下一步操作提示 | 独立成章，见 §五 |

### Step Start · 前置校验与路由

通过 `02-*.md` 的树画像 / 状态队列 / 节点索引定位本轮 STR 节点正文块，读取该节点状态、`### 信息链归属处理状态`、`### 文档归属方案` 和 `### 文档归属反馈区`，按以下规则进入后续步骤：

| 条件 | 路由 |
|------|------|
| 节点状态不是 `待文档归属` | 输出不可进入说明后结束 |
| 信息链归属处理状态全部为 `已写回` | 执行 Step 6，推进到 `待分解分配` |
| 当前链状态为 `可写回` | 执行 Step 5 → Step 6 |
| 当前链无 `### 文档归属方案` | 执行 Step 1 → Step 2 → Step 3 |
| 当前链有 `### 文档归属方案`，且反馈区有 `[同意] / [修改] / [驳回]` | 执行 Step 4 |
| 当前链文档归属方案中所有条目均为 `可写回` | 执行 Step 5 → Step 6 |
| 当前链有 `### 文档归属方案`，且 `### 文档归属反馈区` 为空 | 输出 `等待当前链文档归属反馈` 后结束 |

说明：

1. 当前链是 `### 信息链归属处理状态` 中顺序最靠前、处理状态不是 `已写回` 的信息链。
2. 重新运行 Skill B 时，若当前链已存在文档归属方案，AI 不得覆盖该方案或清空反馈区。
3. 若需要重做当前链文档归属判定，只能由 Step 4 的 `[修改] / [驳回]` 分支触发。
4. 若检测到文档归属方案和反馈区内容冲突，应优先保留人类反馈，并输出需要人工确认的冲突说明。
5. 路由优先依据当前链状态；“当前链文档归属方案中所有条目均为 `可写回`”仅用于兼容旧方案或状态缺失场景。

信息链归属处理状态取值：

| 状态 | 含义 | Skill B 行为 |
|------|------|--------------|
| `待归属` | 当前链尚未形成文档归属方案 | 执行 Step 2 → Step 3 |
| `待反馈` | 当前链存在 `23-Q2 / 23-Q3` 或其他待确认归属项 | 等待人类反馈；有反馈时执行 Step 4 |
| `可写回` | 当前链归属方案已确认，可以写回资产 | 执行 Step 5 |
| `已写回` | 当前链资产写回完成 | 选择下一条未写回信息链 |
| `回退SkillA` | 当前链发现结构问题，需要修改场景定义 | 停止 Skill B，按回退规则运行 `wft01a` |

### Step 1 · 资产读取

1. 加载 `02-*.md` STR 树画像 / 状态队列 / 节点索引，读取本轮 STR 节点正文块、信息载体清单、场景指标、信息链归属处理状态、当前链文档归属方案和反馈区
2. 加载 `23-eos-output-architecture.md` 资产全貌层，按信息载体名称、业务域、典型文档、产品类型和构件类型筛选候选分片；只读取候选业务域 / 产品类型 / 构件类型块
3. 加载 `26-eos-document-definitions.md` 资产全貌层，按信息载体名称、文档性质和 26 方案级判断筛选候选文档定义块
4. 加载 `22-eos-nfr-taxonomy.md` 资产全貌层和与本轮场景指标、风险合规链、规则知识链相关的 NFR 分片
5. 加载 `01-*.md` 树画像 / 状态队列 / 节点索引，读取本轮 STR 来源 OR 块和状态队列相关行
6. 按追溯关系读取必要的上游 / 下游相关块，不默认全量读取大文档正文

读取要求：

1. 读取系统设计过程文档和资产文档时，必须遵守 [文档编写约定 §十四](../../../../00-generalspec/00-doc-conventions.md)：先读树画像 / 资产全貌、状态队列和节点索引，再读本轮正文块和必要相关块。
2. 23 号资产的产品类型 + 构件类型全貌层是文档归属判断的必读入口；末级明细、完整 I/O 和历史记录只按候选分片读取。
3. 只有在索引缺失、关系断裂或执行全局一致性检查时，才允许说明原因后扩大读取范围。

局部细节选择规则：

1. **场景线索**：根据本轮 STR 节点 ID，在 `02-*.md` 节点索引中定位场景定义正文块，读取已确认的信息载体清单、场景指标、信息链归属处理状态、当前链文档归属方案和反馈区；这是本轮最小必读范围。
2. **文档线索**：以每个信息载体为处理对象，根据载体名称、归属链、文档性质、产生方式、一句话职责、来源 OR 和 26 方案级判断，筛选 `26` 文档定义候选块。
3. **归属线索**：根据业务域、信息载体服务的业务对象、操作角色、典型文档、产品类型和构件类型，在 `23` 资产全貌中筛选候选业务域 / 产品类型 / 构件类型分片。
4. **指标线索**：根据场景指标的数据来源、指标类别和口径，筛选 `22` 中与效率、质量、成本、风险、协同、体验等相关的 NFR 分片；仅当指标需要写入或引用 22 资产时读取候选正文块。
5. **匹配线索**：对每个待归属文档，先用 `23` 的产品类型 + 构件类型全貌层判断是否存在直接承接项；只有出现同名 / 同义 / 相近构件、边界冲突或新增建议时，才读取候选构件的详细定义块。
6. **反馈线索**：若本轮为文档归属反馈处理，只读取文档归属反馈区指向的信息载体、23 资产建议和候选资产块；不得重新覆盖已形成的当前链文档归属方案。
7. **写回线索**：进入资产写回前，只读取将被写入的 `23/26/22` 本轮资产块及其索引行，用于去重、补充引用和维护状态；不得因写回而默认读取资产全文。
8. **扩大读取触发**：只有当候选归属无法判定、同名资产冲突、产品类型 / 构件类型边界不清、索引缺失、本轮写回块无法定位或需要做全局一致性检查时，才可扩大读取范围，并在输出中说明扩大读取的原因。

### Step 2 · 当前链文档清单读取

1. 从 `### 信息链归属处理状态` 选择当前链。
2. 从 STR-F 场景定义读取当前链的信息载体；若当前链为 `场景指标链`，读取场景指标。
3. 对当前链每个信息载体提取：载体名称、归属链、文档性质、产生方式、一句话职责、来源 OR、26 方案级判断。
4. 对当前链场景指标提取：指标名称、指标类别、指标含义、主要数据来源。
5. 形成本轮当前链待归属文档清单和指标线索清单。
6. 若发现当前链信息载体或场景指标本身需要新增、删除、改名或改变边界，立即触发结构问题回退 `wft01a`。

### Step 3 · 当前链文档归属判定

Step 3 的归属单位是“广义文档节点”，不是信息链。信息链只用于分组和说明节点来源；每个已确认进入场景定义的广义文档节点，都必须分别判定业务域、产品类型和构件类型，并记录归属依据。

#### 3.1 十类节点归属规则

对当前链每个待归属广义文档节点，按其所属信息链类型执行以下判定：

| 节点类型 | 归属锚点 | 归属规则 |
|------|---------|---------|
| 执行业务链节点 | 被执行、创建、处理或流转的产品数据 | 归属到该产品数据所在产品类型和构件类型；归属锚点是业务对象，不是页面、列表、表单等显示形态 |
| 管理业务链节点 | 被管理的业务对象、业务过程或管理控制对象 | 归属到执行业务链主产品类型下的项目管理构件，并进一步归入对应管理业务 WBS；跨多个构件时拆分为多个 WBS 引用，或定义跨构件管理 WBS 并列出覆盖范围 |
| 个人绩效链节点 | 当前场景内个人对业务/项目运行的贡献、达成、质量和效率 | 归属到执行业务链主产品类型下的项目管理构件，作为个人绩效 WBS；企业统一绩效考核场景另行记录向统一绩效产品的汇总/引用关系 |
| 组织绩效链节点 | 当前场景内组织、团队、部门或项目对业务/项目运行的贡献、达成、质量和效率 | 归属到执行业务链主产品类型下的项目管理构件，作为组织绩效 WBS；企业统一绩效考核场景另行记录向统一绩效产品的汇总/引用关系 |
| 规则知识链节点 | 所附属的执行业务链节点 | 作为执行业务链节点的质量支撑型附属文档处理，产品类型和构件类型继承所附属执行节点；内容包括准则、指南、范例、模板和知识说明 |
| 资源需求链节点 | 支持维资源提供方 | 由执行业务链节点发起，按资源类型归属到支持维对应资源提供类产品类型和构件类型，并由后者负责资源提供；执行业务链节点记录为需求发起方和场景引用方 |
| 风险合规链节点 | 所附属的执行业务链节点 | 与规则知识链同类处理，作为执行业务链节点的附属规则性文档；产品类型和构件类型继承所附属执行节点，内容为事先、事中、事后的风险与合规处理规则 |
| 状态事件链节点 | 所记录的执行业务链节点 | 作为执行业务链节点的系统过程数据和运行过程记录型附属文档，产品类型和构件类型继承所附属执行节点；若表达跨多个业务对象的状态汇总、监控或分析，则转入管理业务链处理 |
| 协同交互链节点 | 公共协同或目标承接方 | 通常作为系统公共协同 IPO 的输入，优先归属到系统公共协同类产品类型及构件类型；若协同事项明确进入某专业业务产品或外部协同/接口产品，则按目标承接方归属 |
| 场景指标链节点 | 被度量对象 | 独立于个人绩效和组织绩效，按每个指标被度量对象分别确定产品类型和构件类型；评价整个角色场景端到端运行质量的指标，归属到角色场景产品类型下的场景定义构件 |

#### 3.2 节点关系记录要求

每个广义文档节点除业务域、产品类型和构件类型外，还必须按节点类型记录必要关系：

| 节点类型 | 必须记录的关系 |
|------|---------------|
| 执行业务链节点 | 被执行产品数据 / 业务对象 |
| 管理业务链节点 | 被管理对象、项目管理构件、管理业务 WBS |
| 个人绩效链节点 | 评价的个人 / 角色、所属项目管理构件、个人绩效 WBS |
| 组织绩效链节点 | 评价的组织对象、所属项目管理构件、组织绩效 WBS |
| 规则知识链节点 | 所附属执行业务链节点、附属文档类型 |
| 资源需求链节点 | 需求发起执行节点、资源类型、资源提供产品类型 / 构件类型 |
| 风险合规链节点 | 所附属执行业务链节点、事先 / 事中 / 事后规则类型 |
| 状态事件链节点 | 所记录执行业务链节点、状态 / 事件 / 轨迹 / 日志类型 |
| 协同交互链节点 | 来源执行节点、公共协同或目标承接方、请求 / 通知 / 待办 / 回执 / 接口消息类型 |
| 场景指标链节点 | 被度量对象、指标口径、数据来源 |

#### 3.3 资产归属判定

对当前链每个广义文档节点，基于 23 输出产品架构资产判定三类归属：

1. 业务域
2. 产品类型
3. 构件类型

匹配规则：

| 判定 | 含义 | 动作 |
|------|------|------|
| 23-Q1 直接归属 | 23 中存在合适业务域、产品类型、构件类型 | 形成可直接写回的归属方案 |
| 23-Q2 扩展归属 | 23 中存在相近产品类型或构件类型，需扩展定义或别名 | 形成扩展建议，进入文档归属反馈 |
| 23-Q3 新增归属 | 23 中无合适产品类型或构件类型 | 形成新增建议，进入文档归属反馈 |
| 结构问题 | 文档本身不应属于当前场景，或需改变场景边界 | 停止本 Skill，回退 `wft01a` |

文档归属方案至少包含：

1. 文档 ID / 名称
2. 所属信息链
3. 关系对象：附属执行节点 / 项目管理 WBS / 资源提供方 / 公共协同输入 / 被度量对象等
4. 业务域
5. 产品类型
6. 构件类型
7. 匹配判定：23-Q1 / 23-Q2 / 23-Q3
8. 判定依据
9. 资产建议：直接引用 / 扩展已有 / 新增类型 / 新增构件
10. 确认状态：可直接写回 / 待确认归属

Step 3 必须先将当前链归属结果写入 `02-*.md` 本轮 STR 节点块的 `### 文档归属方案`，并同步更新该节点块的当前链文档归属处理摘要，再决定后续路由。若当前链所有文档均为 `23-Q1 直接归属` 且无其他阻断项，将当前链状态标记为 `可写回`，直接进入 Step 5。若存在 `23-Q2` 或 `23-Q3`，将当前链状态标记为 `待反馈`，并在 `### 文档归属反馈区` 提醒人类反馈。

写入结构：

```text
### 文档归属方案

#### 执行业务链

| 广义文档节点 | 所属信息链 | 关系对象 | 业务域 | 产品类型 | 构件类型 | 判定 | 资产建议 | 确认状态 |
|-------------|------------|----------|--------|----------|----------|------|----------|----------|

#### 状态事件链

<!-- 未处理的信息链保留标题，内容可为空，便于重新运行时按链定位 -->

### 文档归属反馈区

[操作] 同意 / 修改 / 驳回
```

写入要求：

1. `### 文档归属方案` 必须按信息链分组，分组标题使用 `#### 信息链名称`。
2. Skill B 只能新增或更新当前链分组，不得覆盖其他信息链分组。
3. 重新运行时，判断“当前链是否已有方案”必须只检查当前链分组，不得用其他链分组代替。
4. `### 文档归属方案` 中每一行的归属单位必须是广义文档节点，不得以整条信息链作为归属单位。
5. 当前链写回前，禁止存在“广义文档节点无产品类型 / 构件类型归属”或“关系对象缺失”的未收敛状态。

### Step 4 · 当前链文档归属反馈处理

当当前链文档归属方案存在待确认项时，本 Skill 进入文档归属反馈循环。

人类反馈位置：`02-*.md` 对应 STR 节点下的 `### 文档归属反馈区`。

反馈格式建议：

```text
### 文档归属反馈区

[操作] 同意 / 修改 / 驳回

[文档归属反馈]
-

[23资产建议反馈]
-

[其他说明]
-
```

反馈分支：

1. `[同意]`
   - 确认当前文档归属方案
   - 将文档归属方案中的待确认项标记为 `可写回`
   - 将当前链状态标记为 `可写回`
   - 在本轮 STR 节点块内归档反馈处理摘要，并生成新的空文档归属反馈区
   - 进入 Step 5
2. `[修改]`
   - 按人类反馈调整广义文档节点的关系对象、业务域、产品类型、构件类型或 23 资产建议
   - 保持 Skill A 已确认的信息载体清单不变
   - 在本轮 STR 节点块内更新当前链文档归属方案，状态仍为待确认归属
   - 将当前链状态保持为 `待反馈`
   - 重新等待反馈
3. `[驳回]`
   - 若驳回原因是归属错误但不影响场景定义，则重做 Step 3
   - 若驳回原因涉及信息载体本身、场景边界或承接关系，则将当前链状态标记为 `回退SkillA`，并回退 `wft01a`

### Step 5 · 当前链资产写回

Skill B 写资产时必须遵守 [文档编写约定 §十四](../../../../00-generalspec/00-doc-conventions.md) 的树形资产文档布局。写回范围按本轮相关块局部更新，不得重写大文档全文。

1. 写回 `26-*.md` 文档定义资产：
   - 在资产全貌 / 分片索引中定位同名、同义或候选文档定义块
   - 已有文档定义时，只更新该文档块的业务域、产品类型、构件类型、来源 STR 节点、来源 OR 和追溯引用
   - 新增文档定义时，新增稳定 `BLOCK`，同步新增树画像 / 资产全貌摘要、分片索引行、状态队列记录和追溯引用
2. 写回 `23-*.md` 输出产品架构资产：
   - 在资产全貌中定位本轮业务域、产品类型和构件类型块
   - 维护产品类型 / 构件类型的文档引用列表；对确认后的扩展或新增建议执行正式更新
   - 若新增或扩展产品类型 / 构件类型，同步更新树画像 / 资产全貌、分片索引、正文块、状态队列和追溯引用
3. 写回 `22-*.md` 非功能需求资产：
   - 将场景指标中具有非功能或质量属性含义的指标作为线索匹配到 22 条目
   - 可匹配时，在对应 NFR 块维护引用；不可匹配时，在候选线索区或建议区记录候选线索
   - 若新增候选线索，同步更新资产全貌摘要、索引行、状态队列和追溯引用
4. 更新 `02-*.md` 本轮 STR 节点块：
   - 将当前链文档归属方案中的可写回条目标记为已写回
   - 记录当前链 23 / 26 / 22 写回摘要和追溯引用
   - 将 `### 信息链归属处理状态` 中当前链状态推进为 `已写回`
   - 保留新的空 `### 文档归属反馈区`

写回要求：

1. 写回必须幂等：同一 STR 节点、同一文档、同一资产条目的引用不得重复追加。
2. 26 中已存在同名或同义文档时，应更新引用和归属信息，不得重复新增。
3. 23 中已存在同一产品类型 / 构件类型引用时，应去重后写入。
4. 若写回过程中发现仍有广义文档节点缺少产品类型、构件类型或必要关系对象，应停止写回并回到 Step 3；若需要改变场景边界或 OR 承接关系，应回退 `wft01a`。
5. 所有新增或变更的资产块必须同步维护树画像 / 资产全貌、分片索引、状态队列和追溯引用；不得只改正文块。

### Step 6 · 当前链状态推进

1. 若仍有下一条信息链状态不是 `已写回`，保持 STR 节点状态为 `待文档归属`，并在 `### 信息链归属处理状态` 中标出下一条当前链。
2. 若十条信息链全部为 `已写回`，在 `02-*.md` 树画像、状态队列和节点索引中将本轮 STR 节点推进为 `待分解分配`。
3. 若十条信息链全部为 `已写回`，在 `01-*.md` 树画像、状态队列和本轮 OR 块中将对应 OR 条目推进为 `已分配`。
4. 在 `01/02` 追溯关系中确认 OR→STR→23/26/22 的引用链已记录。
5. 清空已通过节点的反馈区待办信号和文档归属反馈区待办信号。

说明：

1. 对 `wft01` 而言，Skill B 完成后不再进入 Phase 2 下半
2. `待分解分配` 是本步完成后的下游准备态，供 `wft02` 选入

## 四、回退规则

### 结构问题触发条件

发现以下问题时，必须回退 `wft01a`：

1. STR 节点是否存在需要重判
2. STR 节点类型需要改
3. STR 节点边界需要重切
4. OR→STR 宿主分配需要重做
5. 复用 / 改进 / 新增 决策需要推翻
6. Skill A 已确认的信息载体清单或场景指标清单需要新增、删除、改名或改变语义边界

### 回退动作

1. 立即停止当前 Skill B
2. 不继续执行后续写回
3. 在 `02-*.md` 本轮 STR 节点反馈区写入回退记录，且必须包含 `[操作] 修改`，供 `wft01a` 重新运行时识别
4. 在 `02-*.md` 树画像、状态队列和节点索引中将节点状态回退为 `迭代中`
5. 输出下一步处理方式：`重新运行 wft01a`

回退记录建议格式：

```text
[操作] 修改

[结构反馈]
- Skill B 回退原因：
- 需要 Skill A 重新判断的内容：

[其他说明]
-
```

若结构问题在正式写回后才被发现，则必须先撤销本轮已写入的：

1. 23 资产引用更新
2. 26 资产引用更新
3. 22 资产引用更新
4. OR / STR 状态推进

## 五、运行结束输出（Step End）

Step End 的作用是把本轮运行结果翻译成人类下一步操作。AI 不只输出“完成 / 未完成”，还必须告诉人类：

1. 现在处于哪种结果
2. 人类下一步应该做什么
3. 可以直接通过对话反馈，还是需要编辑文档
4. 若需要留痕，AI 会写入哪个文档、哪个位置
5. 需要重点检查哪些内容

推荐交互方式：

1. 人类使用 Markdown 预览模式阅读当前链文档归属方案和资产建议。
2. 人类在对话中回复 `同意`、`修改：……` 或 `驳回：……`。
3. AI 先将对话反馈转写到 `### 文档归属反馈区`，再按反馈区规则执行资产写回、方案修改或回退处理。

约束：对话是反馈输入通道，`### 文档归属反馈区` 是权威留痕。AI 不得只凭对话上下文推进状态或写回资产，必须先完成反馈区转写。

### 输出结果

本 Skill 结束时，只允许输出以下四类结果之一：

| 输出结果 | 何时出现 | 人类下一步 |
|---------|----------|------------|
| `等待当前链文档归属反馈` | 当前链存在 `23-Q2 / 23-Q3`，需要人类确认文档归属或 23 资产建议 | 优先在对话中回复 `同意 / 修改 / 驳回`；AI 转写到 `文档归属反馈区` |
| `当前链完成，继续下一链` | 当前链已写回，但仍有后续信息链未写回 | 继续运行 `wft01b` 处理下一条信息链 |
| `已回退 Skill A` | 发现文档清单、场景边界、OR 承接等结构问题 | 查看回退记录，重新运行 `wft01a` |
| `本轮节点完成` | 十条信息链均已写回，STR 节点已推进到 `待分解分配` | 选入 `wft02`，或回到 `wft01a` 继续处理下一个 OR |

### 结果一：等待当前链文档归属反馈

人类当前应做什么：

1. 查看当前链 `### 文档归属方案` 中的 `23-Q2 / 23-Q3` 条目
2. 优先在对话中回复 `同意`、`修改：……` 或 `驳回：……`
3. 对 AI 提出的产品类型 / 构件类型新增或扩展建议给出确认或修改意见

AI 收到对话反馈后，应先转写为 `### 文档归属反馈区` 的规范记录，再继续处理。

对话反馈示例：

```text
修改：同意“采购申请单”的归属；但“采购预算占用信息”建议归入采购管理域，并新增预算控制构件建议。
```

文档反馈区备选格式：

```text
[操作] 同意 / 修改 / 驳回

[文档归属反馈]
-

[23资产建议反馈]
-

[其他说明]
-
```

重点查看位置：

1. `02-*.md` 对应 STR 节点当前链的 `### 文档归属方案`
2. `02-*.md` 对应 STR 节点的 `### 文档归属反馈区`
3. `02-*.md` 对应 STR 节点的 `### 信息链归属处理状态`
4. `23-eos-output-architecture.md` 中相关业务域、产品类型、构件类型

### 结果二：当前链完成，继续下一链

人类当前应做什么：

1. 查看 `### 信息链归属处理状态`，确认当前链已为 `已写回`
2. 查看下一条待处理信息链
3. 继续运行 `wft01b`，处理下一条信息链

重点查看位置：

1. `02-*.md` 对应 STR 节点的 `### 信息链归属处理状态`
2. 当前链的写回摘要
3. 下一条信息链的信息载体或场景指标

### 结果三：已回退 Skill A

人类当前应做什么：

1. 查看本轮处理的 STR 节点反馈区中的回退记录
2. 判断是否接受回退原因
3. 重新运行 `wft01a`，由 Skill A 修订场景定义、信息载体、场景指标或 OR 承接关系

重点查看位置：

1. `02-*.md` 对应 STR 节点反馈区
2. 回退记录中标注的结构问题
3. 受影响的信息载体、场景指标、场景边界或 OR 承接关系

### 结果四：本轮节点完成

人类当前可选择的动作：

1. 人类可在 `02-*.md` 状态队列中将 `待分解分配` 的 STR 节点选入 `wft02`
2. 人类也可回到 `wft01a` 继续选择下一个 OR 条目
3. 人类可查看 23 / 26 / 22 的写回摘要，确认资产更新是否符合预期

重点查看位置：

1. `02-*.md` 状态队列和节点索引中该 STR 节点状态是否已为 `待分解分配`
2. `26-eos-document-definitions.md` 中新增或更新的广义文档定义
3. `23-eos-output-architecture.md` 中新增或更新的产品类型 / 构件类型引用
4. `22-eos-nfr-taxonomy.md` 中与场景指标相关的引用或候选线索
