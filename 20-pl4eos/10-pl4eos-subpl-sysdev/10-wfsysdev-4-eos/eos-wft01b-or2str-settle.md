# OR → STR 架构 · Skill B · 文档归属与资产写回
**Skill ID**: wft01b · **Phase 2: OR → STR Settlement**

> **概述**：对 Skill A 已确认的 STR 场景定义执行文档归属与资产写回。Skill B 以场景定义中的每个广义文档 / 信息载体为处理对象，基于 23 输出产品架构资产判定其所属业务域、产品类型、构件类型；若 23 资产无合适承接项，则 AI 给出新增 / 扩展建议，经人类反馈确认后再写回资产。`wft01` 无 Phase 2 下半，本 Skill 完成后进入 `wft02` 前准备态。
>
> **输入 → 输出**：已确认的 STR 场景定义 → 文档归属方案 + 23/26/22 资产更新 + `01/02` 状态推进
>
> **适用场景**：瀑布式系统设计第1步 · Phase 2。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.9 | **修订**：2026-06-19

---

## 一、职责定位

Skill B 负责：

1. 接收已确认的 STR 场景定义方案
2. 针对 STR-F 场景定义中的每个广义文档 / 信息载体，判定业务域、产品类型、构件类型
3. 基于 23 输出产品架构资产执行文档归属匹配；无合适承接项时提出新增 / 扩展建议
4. 维护 26 广义文档资产定义与归属
5. 维护 23 输出产品架构资产引用
6. 维护 22 非功能需求资产引用或候选线索
7. 推进 OR / STR 状态，使节点进入下游 `wft02` 的准备态

Skill B 不负责：

1. 重做 OR→STR 主结构设计
2. 修改 Skill A 已确认的节点边界与承接关系
3. 新增、删除或重命名 Skill A 已确认的信息载体和场景指标
4. 生成下游详细定义

## 二、输入与路由

### 标准输入包

1. 已确认的 STR 场景定义方案
2. 当前 23 / 26 / 22 资产内容
3. 文档归属反馈（如存在待确认归属项）

### 进入条件

本轮处理的 STR 节点满足以下条件时可进入本 Skill：

1. 当前状态为 `待文档归属`

说明：`待文档归属` 表示该 STR 场景定义已经由 Skill A 完成方案确认，可以进入逐文档归属判定。若存量节点仍停留在 `待确认方案` 且反馈区已有 `[同意]`，AI 应先按 Skill A 规则将节点状态推进为 `待文档归属`，再进入本 Skill。

### 不可修改项

进入 Skill B 后，AI 不能再修改以下内容；如需修改，必须回退 `wft01a`：

1. STR 节点清单
2. 节点类型（STR-F / STR-NFR）
3. 节点边界
4. OR→STR 宿主分配
5. 复用 / 改进 / 新增 决策
6. 主承接关系
7. STR-F 场景定义中的信息载体清单与场景指标清单

## 三、操作流程

### Step 布局总览

| Step | 名称 | 产出 | 写资产 |
|------|------|------|--------|
| Start | 前置校验与路由 | 就绪 / 结束 | — |
| 1 | 资产读取 | 已读取资产内容 | — |
| 2 | 文档清单读取 | 待归属文档清单 | — |
| 3 | 文档归属判定 | 文档归属方案 | `02-*.md` |
| 4 | 文档归属反馈处理 | 确认后的归属方案 / 回退 | `02-*.md` |
| 5 | 资产写回 | 23 / 26 / 22 资产更新 | `23-*.md` / `26-*.md` / `22-*.md` |
| 6 | 状态推进 | OR / STR 状态更新 | `01-*.md` / `02-*.md` |
| End | 运行结束输出 | 人类下一步操作提示 | 独立成章，见 §五 |

### Step Start · 前置校验与路由

读取本轮处理的 STR 节点状态、`### 文档归属方案` 和 `### 文档归属反馈区`，按以下规则进入后续步骤：

| 条件 | 路由 |
|------|------|
| 节点状态不是 `待文档归属` | 输出不可进入说明后结束 |
| 无 `### 文档归属方案` | 执行 Step 1 → Step 2 → Step 3 |
| 文档归属方案中所有条目均为 `可写回` | 执行 Step 5 → Step 6 |
| 有 `### 文档归属方案`，且反馈区有 `[同意] / [修改] / [驳回]` | 执行 Step 4 |
| 有 `### 文档归属方案`，且 `### 文档归属反馈区` 为空 | 输出 `等待文档归属反馈` 后结束 |

说明：

1. 重新运行 Skill B 时，若已存在文档归属方案，AI 不得覆盖该方案或清空反馈区。
2. 若需要重做文档归属判定，只能由 Step 4 的 `[修改] / [驳回]` 分支触发。
3. 若检测到文档归属方案和反馈区内容冲突，应优先保留人类反馈，并输出需要人工确认的冲突说明。

### Step 1 · 资产读取

1. 加载 `02-*.md` 当前 STR 节点定义
2. 加载 `23-eos-output-architecture.md`
3. 加载 `26-eos-document-definitions.md`
4. 加载 `22-eos-nfr-taxonomy.md`
5. 加载 `01-*.md` 表C 与 OR 段落

### Step 2 · 文档清单读取

1. 从 STR-F 场景定义读取已确认的信息载体清单
2. 对每个信息载体提取：载体名称、归属链、文档性质、产生方式、一句话职责、来源 OR、26 方案级判断
3. 从场景指标中提取指标名称、指标类别、指标含义、主要数据来源
4. 形成本轮待归属文档清单和指标线索清单
5. 若发现信息载体或场景指标本身需要新增、删除、改名或改变边界，立即触发结构问题回退 `wft01a`

### Step 3 · 文档归属判定

对每个待归属文档，基于 23 输出产品架构资产判定三类归属：

1. 业务域
2. 产品类型
3. 构件类型

匹配规则：

| 判定 | 含义 | 动作 |
|------|------|------|
| 23-Q1 直接归属 | 23 中存在合适业务域、产品类型、构件类型 | 形成可直接写回的归属方案 |
| 23-Q2 扩展归属 | 23 中存在相近产品类型或构件类型，需扩展定义或别名 | 形成扩展建议，进入文档归属反馈 |
| 23-Q3 新增归属 | 23 中无合适产品类型或构件类型 | 形成新增建议，进入文档归属反馈 |
| 结构问题 | 文档本身不应属于当前场景，或需改变场景边界 | 停止本 Skill，回退 `wft01a` |

文档归属方案至少包含：

1. 文档 ID / 名称
2. 业务域
3. 产品类型
4. 构件类型
5. 匹配判定：23-Q1 / 23-Q2 / 23-Q3
6. 判定依据
7. 资产建议：直接引用 / 扩展已有 / 新增类型 / 新增构件
8. 确认状态：可直接写回 / 待确认归属

Step 3 必须先将归属结果写入 `### 文档归属方案`，再决定后续路由。若所有文档均为 `23-Q1 直接归属` 且无其他阻断项，将文档归属方案标记为 `可写回`，直接进入 Step 5。若存在 `23-Q2` 或 `23-Q3`，在 `### 文档归属反馈区` 提醒人类反馈。

写入结构：

```text
### 文档归属方案

| 文档 | 业务域 | 产品类型 | 构件类型 | 判定 | 资产建议 | 确认状态 |
|------|--------|----------|----------|------|----------|----------|

### 文档归属反馈区

[操作] 同意 / 修改 / 驳回
```

### Step 4 · 文档归属反馈处理

当文档归属方案存在待确认项时，本 Skill 进入文档归属反馈循环。

人类反馈位置：`02-*.md` 对应 STR 节点下的 `### 文档归属反馈区`。

反馈格式建议：

```text
### 文档归属反馈区

[操作] 同意 / 修改 / 驳回

[文档归属反馈]
-

[23资产建议反馈]
-

[其他说明]
-
```

反馈分支：

1. `[同意]`
   - 确认当前文档归属方案
   - 将文档归属方案中的待确认项标记为 `可写回`
   - 归档并清空文档归属反馈区
   - 进入 Step 5
2. `[修改]`
   - 按人类反馈调整业务域、产品类型、构件类型或 23 资产建议
   - 保持 Skill A 已确认的信息载体清单不变
   - 更新文档归属方案，状态仍为待确认归属
   - 重新等待反馈
3. `[驳回]`
   - 若驳回原因是归属错误但不影响场景定义，则重做 Step 3
   - 若驳回原因涉及信息载体本身、场景边界或承接关系，则回退 `wft01a`

### Step 5 · 资产写回

1. 写回 `26-*.md`：新增或更新广义文档定义，记录其业务域、产品类型、构件类型、来源 STR 节点与来源 OR
2. 写回 `23-*.md`：维护产品类型 / 构件类型的文档引用列表；对确认后的扩展或新增建议执行正式更新
3. 写回 `22-*.md`：将场景指标中具有非功能或质量属性含义的指标作为线索匹配到 22 条目；可匹配则维护引用，不可匹配则记录候选线索
4. 记录本轮变更摘要

写回要求：

1. 写回必须幂等：同一 STR 节点、同一文档、同一资产条目的引用不得重复追加。
2. 26 中已存在同名或同义文档时，应更新引用和归属信息，不得重复新增。
3. 23 中已存在同一产品类型 / 构件类型引用时，应去重后写入。
4. 若写回过程中发现需要改变文档清单、场景边界或 OR 承接关系，应停止写回并回退 `wft01a`。

### Step 6 · 状态推进

1. 在 `02-*.md` 表D 中将本轮处理的 STR 节点推进为 `待分解分配`
2. 在 `01-*.md` 表C 中将对应 OR 条目推进为 `已分配`
3. 清空已通过节点的反馈区待办信号和文档归属反馈区待办信号

说明：

1. 对 `wft01` 而言，Skill B 完成后不再进入 Phase 2 下半
2. `待分解分配` 是本步完成后的下游准备态，供 `wft02` 选入

## 四、回退规则

### 结构问题触发条件

发现以下问题时，必须回退 `wft01a`：

1. STR 节点是否存在需要重判
2. STR 节点类型需要改
3. STR 节点边界需要重切
4. OR→STR 宿主分配需要重做
5. 复用 / 改进 / 新增 决策需要推翻
6. Skill A 已确认的信息载体清单或场景指标清单需要新增、删除、改名或改变语义边界

### 回退动作

1. 立即停止当前 Skill B
2. 不继续执行后续写回
3. 在本轮处理的 STR 节点反馈区写入回退记录，且必须包含 `[操作] 修改`，供 `wft01a` 重新运行时识别
4. 节点状态回退为 `迭代中`
5. 输出下一步处理方式：`重新运行 wft01a`

回退记录建议格式：

```text
[操作] 修改

[结构反馈]
- Skill B 回退原因：
- 需要 Skill A 重新判断的内容：

[其他说明]
-
```

若结构问题在正式写回后才被发现，则必须先撤销本轮已写入的：

1. 23 资产引用更新
2. 26 资产引用更新
3. 22 资产引用更新
4. OR / STR 状态推进

## 五、运行结束输出（Step End）

Step End 的作用是把本轮运行结果翻译成人类下一步操作。AI 不只输出“完成 / 未完成”，还必须告诉人类：

1. 现在处于哪种结果
2. 人类下一步应该做什么
3. 可以直接通过对话反馈，还是需要编辑文档
4. 若需要留痕，AI 会写入哪个文档、哪个位置
5. 需要重点检查哪些内容

推荐交互方式：

1. 人类使用 Markdown 预览模式阅读文档归属方案和资产建议。
2. 人类在对话中回复 `同意`、`修改：……` 或 `驳回：……`。
3. AI 先将对话反馈转写到 `### 文档归属反馈区`，再按反馈区规则执行资产写回、方案修改或回退处理。

约束：对话是反馈输入通道，`### 文档归属反馈区` 是权威留痕。AI 不得只凭对话上下文推进状态或写回资产，必须先完成反馈区转写。

### 输出结果

本 Skill 结束时，只允许输出以下三类结果之一：

| 输出结果 | 何时出现 | 人类下一步 |
|---------|----------|------------|
| `等待文档归属反馈` | 存在 `23-Q2 / 23-Q3`，需要人类确认文档归属或 23 资产建议 | 优先在对话中回复 `同意 / 修改 / 驳回`；AI 转写到 `文档归属反馈区` |
| `已回退 Skill A` | 发现文档清单、场景边界、OR 承接等结构问题 | 查看回退记录，重新运行 `wft01a` |
| `本轮节点完成` | 23 / 26 / 22 写回完成，STR 节点已推进到 `待分解分配` | 选入 `wft02`，或回到 `wft01a` 继续处理下一个 OR |

### 结果一：等待文档归属反馈

人类当前应做什么：

1. 查看 `### 文档归属方案` 中的 `23-Q2 / 23-Q3` 条目
2. 优先在对话中回复 `同意`、`修改：……` 或 `驳回：……`
3. 对 AI 提出的产品类型 / 构件类型新增或扩展建议给出确认或修改意见

AI 收到对话反馈后，应先转写为 `### 文档归属反馈区` 的规范记录，再继续处理。

对话反馈示例：

```text
修改：同意“采购申请单”的归属；但“采购预算占用信息”建议归入采购管理域，并新增预算控制构件建议。
```

文档反馈区备选格式：

```text
[操作] 同意 / 修改 / 驳回

[文档归属反馈]
-

[23资产建议反馈]
-

[其他说明]
-
```

重点查看位置：

1. `02-*.md` 对应 STR 节点的 `### 文档归属方案`
2. `02-*.md` 对应 STR 节点的 `### 文档归属反馈区`
3. `23-eos-output-architecture.md` 中相关业务域、产品类型、构件类型

### 结果二：已回退 Skill A

人类当前应做什么：

1. 查看本轮处理的 STR 节点反馈区中的回退记录
2. 判断是否接受回退原因
3. 重新运行 `wft01a`，由 Skill A 修订场景定义、信息载体、场景指标或 OR 承接关系

重点查看位置：

1. `02-*.md` 对应 STR 节点反馈区
2. 回退记录中标注的结构问题
3. 受影响的信息载体、场景指标、场景边界或 OR 承接关系

### 结果三：本轮节点完成

人类当前可选择的动作：

1. 人类可在 `02-*.md` 表D 中将 `待分解分配` 的 STR 节点选入 `wft02`
2. 人类也可回到 `wft01a` 继续选择下一个 OR 条目
3. 人类可查看 23 / 26 / 22 的写回摘要，确认资产更新是否符合预期

重点查看位置：

1. `02-*.md` 表D 中该 STR 节点状态是否已为 `待分解分配`
2. `26-eos-document-definitions.md` 中新增或更新的广义文档定义
3. `23-eos-output-architecture.md` 中新增或更新的产品类型 / 构件类型引用
4. `22-eos-nfr-taxonomy.md` 中与场景指标相关的引用或候选线索
