# SR-F(A1) → PA · 引擎配置组件设计与资产化
**Skill ID**: wft05-eng-sr2pa

> **概述**：每次执行处理 ≥1 个 SR-F 引擎能力节点（来自 wft03-eng，状态=`待产品架构` / `待补充PA设计`），PA 反馈优先于新 SR-F 处理。基于 CU 的配置、处理和业务能力，从配置单元需求设计推导出支持配置和支持业务所需要的前后端组件需求——按 CU IPO 固定映射规则将 Input（配置页面/子页面及配置信息）映射为前端配置支持组件+后端配置定义处理组件，Process（引擎处理）映射为后端运行执行组件，Output（业务运行页面能力）映射为前端业务能力支持组件+后端业务能力服务组件，组织为 1 能力主组件 + 5 类承接组件。延续 wft03-eng 已建立的 25-Q1/Q2/Q3 对齐关系，承接 CU 间依赖链校验结果，收敛 PA层收敛项。完成跨域匹配、资产写回、SR 详细定义展开、PA 组件需求规格写入与状态推进。
>
> **输入 → 输出**：SR-F 引擎能力节点（≥1 个，`待产品架构` / `待补充PA设计`）/ 有反馈的 PA 方案节点 → PA 前后端组件方案 + 25 对齐延续 + 依赖链承接 + PA层收敛结论 + 跨域匹配写回 + SR 详细定义（`06-*.md`）+ PA 组件需求规格（`07-*.md`）+ SR-F 推进为 `已产品架构`
>
> **适用场景**：瀑布式系统设计第5步 · 引擎配置路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.3 | **修订**：2026-06-29

## 一、职责定位

### 本 Skill 目的与目标

> **主要目的**：取信于开发方——「能开工」。基于 CU 的配置、处理和业务能力，把配置单元需求设计映射为可构建的前后端组件需求——每个 CU 引擎配置能力对应到支持配置和支持业务的可认领组件，优先复用已有资产。
>
> **目标**：组件边界完整——职责清晰、依赖明确、产出可证，开发方能直接认领开发。一句话：wft05-eng 把"配置页面/子页面配置什么、引擎怎样处理、业务使用人员获得什么业务能力"翻译成"支持配置和支持业务各需要哪些前后端组件"。

本 Skill 负责：

1. **接收**——接收人类选中的 ≥1 个 SR-F 引擎能力节点（`待产品架构` / `待补充PA设计`），校验状态/类型/数量，不合格则说明原因并退出
2. **对齐**——延续 wft03-eng 已建立的 25-Q1/Q2/Q3 对齐关系，PA 组件设计直接引用/扩展/推断已有 25 配置单元类型定义
3. **映射**——按 CU IPO 固定映射规则将引擎配置能力映射为支持配置和支持业务的前后端组件（1 能力主组件 + 5 类承接组件），规则驱动，非创造性分解分配
4. **承接**——承接 wft03-eng 3b CU 间依赖链校验结果，PA 组件间引用关系与 CU 依赖链一致
5. **收敛**——收敛 wft03-eng 标注的 PA层收敛项，确定具体实现方式；无法收敛的标注 `[PA层未收敛]` + 原因
6. **落账**——执行 Q1/Q2/Q3 跨域匹配、正式写回 07 资产、更新反向引用（05/23/25）；25-Q3 仅在上游已完成 25 引擎模型扩展确认时作为正式新建，否则以 `[推断]` 阶段性组件方案落账，不反向固化 25 模型
7. **展开**——写入 SR 详细定义（`06-*.md`）和 PA 组件需求规格（`07-*.md`），达到开发可执行粒度
8. **推进**——人类确认后 SR-F 节点推进为 `已产品架构`

本 Skill 不负责：

1. 业务运行与治理侧 PA 组件设计（→ `wft05-biz-sr2pa`）
2. 非功能需求的量化与分类（由 wft04 承担，本 Skill 只消费 NFR 约束包）
3. 引擎能力缺口的所有处理（→ 参见 wft03-biz 的六不规则）

### 定位说明

wft05-eng 设计的是 **EOS 引擎本体侧的产品架构**——从配置单元的 Input / Process / Output 出发，推导支持配置输入、配置处理、业务能力运行所需的前后端组件需求。它可以设计为业务能力提供运行支撑的前端/后端组件，但不设计业务运行与治理侧的完整业务 PA 组件（菜单、场景页编排、资产目录、数据治理等），这些由 wft05-biz 独立设计。本 Skill 与 wft05-biz 之间通过 23/24/25 资产引用和引擎能力依赖标注衔接，不互相越界设计。

|------|------|

Step 2（反馈处理）先于 Step 3（新增设计）执行。Step 2 中 `[修改]` / `[驳回]` 可能改变 PA 组件方案，因此 Step 3 必须基于 Step 2 修改后的 PA 当前内容，不得使用 Step 1 加载时的缓存旧状态。

### Step 2 · 方案反馈处理

若本轮存在有反馈的 PA 方案节点（`待确认方案` + 反馈区有效），逐个处理。

#### 基本操作

| 操作 | 行为 |
|------|------|
| `[同意]` 或整体确认 | 归档反馈 → SR-F → `已产品架构`；PA → `已验证冻结` |
| `[修改]`（局部） | 按反馈调整组件边界/需求规格 → 更新增量清单 → 回写 `待确认方案` |
| `[驳回]`（局部） | 移除被驳回的组件和需求规格 → 其他组件 → `已确认` → 回写 `待确认方案` |
| `[驳回]`（整体） | 重做 Step 3 → 回写 `待确认方案` |

#### 分层反馈

PA 方案包含多个可独立决策的单元（各承接组件、组件边界、需求规格、PA层收敛结论）。人类可以**局部修改**——整体确认一个 PA 方案的同时，要求调整其中某个组件的边界或某个接口定义。

- **局部 `[修改]`**：按反馈调整被指定的对象，其涉及的承接关系重新检查。未提及的对象保持原结论不动。
- **PA 方案推进条件**：全部反馈处理完成 → SR-F 推进为 `已产品架构`，PA 推进为 `已验证冻结`。存在未处理项 → PA 保持 `待确认方案`。
- **`[驳回]`（整体）**：PA 主结构被推翻（"组件划分根本不对"或"映射关系错误"）→ 重做 Step 3。

#### 反馈区位置与格式

反馈区位于 `07-*.md` 中每个 PA 方案节点的末尾。格式同 wft03-biz 模式。

### Step 3 · PA 组件设计

将 SR-F 引擎能力功能定义映射为前后端组件方案。

> **设计姿态**：wft02-eng 的 CU IPO 已显式化配置页面/子页面、配置输入、引擎处理、业务运行页面能力；wft03-eng 的功能定义已显式化功能描述、流程、规则和接口（含 PA层收敛项），并建立了 25 对齐关系和 CU 间依赖链。本步按 CU IPO 到 PA 组件的固定映射规则做模式匹配，延续 25 对齐，承接依赖链，收敛 PA层收敛项。

#### 25 对齐延续

wft03-eng 已为每个 CU 建立了 25-Q1/Q2/Q3 对齐结论（来自 wft02-eng Step 3.2）。本步延续此对齐关系设计 PA 组件：

| 对齐结论 | PA 组件设计策略 |
|---------|---------------|
| **25-Q1**（已有匹配） | PA 组件直接引用 25 已有 CU 类型定义中的标准能力项，组件边界与 25 定义一致 |
| **25-Q2**（扩展已有） | PA 组件包含扩展部分，标注扩展来源和方向，与 25 已有定义对齐基础部分 |
| **25-Q3**（推断新建） | 若上游已完成 25 引擎模型扩展确认，则按新 CU 类型设计；否则仅形成 `[推断]` 阶段性 PA 组件方案，不反向固化 25 模型，待后续引擎模型确认后修订 |

25 对齐延续写入 PA 组件需求规格的"25 对齐"小节，与 wft03-eng 写回 25 的 SR-F 功能定义引用对齐。

#### 依赖链承接

wft03-eng Step 3b 已完成 CU 间依赖链校验。本步承接其结果设计 PA 组件间引用关系：

1. **依赖引用**：依赖 CU 的 PA 组件中，**后端运行执行组件**和**后端业务能力服务组件**应正确引用被依赖 CU 的 Output 组件
2. **依赖方向**：PA 组件间引用方向与 CU 依赖链方向一致（A 依赖 B → A 的组件引用 B 的 Output 组件）
3. **循环检测**：若 PA 组件间出现循环引用（25 定义不允许循环依赖），标记矛盾待人类裁决
4. **占位 CU 依赖**：被依赖的 CU 仍为 `[待触发]` 时，PA 组件中标注"依赖未触发 CU，待后续补全"

#### CU IPO → 前后端组件映射规则

| CU IPO 要素 | 要素说明 | 映射的 PA 组件 | 判断逻辑 |
|--------------|---------|---------------|---------|
| I（Input） | 配置页面/子页面及配置信息 | **前端配置支持组件** | 配置信息需要业务配置人员通过页面定义或维护 |
| | | **后端配置定义处理组件** | 配置信息需要被解析、校验、存储、版本化 |
| P（Process） | 引擎对配置信息的处理 | **后端运行执行组件** | 引擎运行时解释、调度和执行配置定义 |
| O（Output） | 引擎生成/支撑的业务运行能力 | **前端业务能力支持组件** | 业务使用人员需要在业务运行页面上操作/查看该能力；本组件提供该能力所需的运行控件、渲染、交互和状态呈现，具体页面编排由 wft05-biz 设计 |
| | | **后端业务能力服务组件** | 业务运行能力需要后端 API、事件、状态、数据读写或外部集成支撑 |

#### 组件结构：1 能力主组件 + 5 类承接组件

| 组件层 | 组件名称 | 职责 |
|------|---------|------|
| 主组件 | **能力主组件** | 该引擎能力 IPO 的核心承接单元 |
| 子组件1 | **前端配置支持组件** | 配置页面、设计器、参数编辑器 |
| 子组件2 | **前端业务能力支持组件** | 运行期控件、渲染、交互、状态呈现、页面嵌入点；不设计具体业务菜单/场景页编排 |
| 子组件3 | **后端配置定义处理组件** | 配置信息解析、校验、组装、版本化 |
| 子组件4 | **后端运行执行组件** | 定义解释、调度、执行、状态流转 |
| 子组件5 | **后端业务能力服务组件** | API、外部集成、消息、存储、事件、业务运行状态服务 |

#### 生成逻辑

1. 按 25 对齐结论确定设计策略（Q1 引用 / Q2 扩展 / Q3 推断）
2. 从 SR-F 功能描述和功能流程中提取**能力主组件**职责边界
3. 从 SR-F 来源 CU 的配置页面/子页面和功能接口中识别支持配置所需的前端能力 → **前端配置支持组件**
4. 从 SR-F 来源 CU 的业务运行页面能力、功能流程和 PA层收敛项中提取支持业务所需的前端能力 → **前端业务能力支持组件**；业务运行页面本体、菜单入口和场景编排回到 `wft05-biz`
5. 从 SR-F 功能流程和功能规则中提取配置信息处理逻辑 → **后端配置定义处理组件**
6. 从 SR-F 功能流程和 PA层收敛项中提取运行期执行逻辑 → **后端运行执行组件**
7. 从 SR-F 功能接口和 Output 中提取业务运行能力所需的 API、事件、数据读写、状态服务与集成需求 → **后端业务能力服务组件**
8. 按 CU 间依赖链结果，在组件间建立正确的引用关系
9. 装配 NFR 约束包：将 `组件类型` / `组件边界` / `接口机制` / `运行约束机制` / `共享归属` 类影响点写入对应组件需求规格；`阻断型` 约束必须改变组件边界、接口或运行机制，不能只作为备注
10. 对每个组件执行资产决策：**复用已有 > 改进已有 > 新增**
11. 新增组件须与现有资产风格一致（粒度/命名/边界模式/层级位置/依赖关系）

#### NFR 约束装配规则

| wft04 影响点 | 承接组件 / 写入位置 | 处理规则 |
|-------------|-------------------|---------|
| 组件类型 | 能力主组件 / 五类承接组件清单 | 决定是否新增、拆分、合并或复用组件；阻断型约束可推翻原组件划分 |
| 组件边界 | 组件职责边界 | 调整职责归属、前后端边界、同步/异步边界、共享/专用边界 |
| 接口机制 | 前端业务能力支持组件 / 后端业务能力服务组件 / 后端配置定义处理组件 | 明确 API、事件、鉴权、脱敏、幂等、限流、审计等接口机制 |
| 运行约束机制 | 后端运行执行组件 / 后端业务能力服务组件 | 明确性能、可靠性、可观测性、重试、补偿、隔离、降级等运行机制 |
| 共享归属 | 能力主组件 / 组件索引 | 决定组件是引擎内专用、跨 CU 共享、跨引擎共享还是平台级共享 |
| 配置项/配置规则/配置可达性/验收标准 | 不在 eng 侧展开为业务配置需求 | 标注为 `biz 约束`，仅保留引用；具体承接由 wft05-biz 处理 |

五类组件的 NFR 装配侧重点：

| 组件类型 | 重点消费的 NFR |
|---------|----------------|
| 前端配置支持组件 | 配置可用性、易用性、权限、审计、可访问性、兼容性 |
| 后端配置定义处理组件 | 配置校验、版本化、一致性、审计、幂等、权限、安全 |
| 后端运行执行组件 | 性能、可靠性、并发、隔离、重试/补偿、可观测性 |
| 前端业务能力支持组件 | 响应时间、可用性、可访问性、脱敏展示、权限可见性、兼容性 |
| 后端业务能力服务组件 | API 性能、安全、限流、审计、数据一致性、事件可靠性、集成可用性 |

同一 NFR 同时进入 biz 与 eng 时，wft05-eng 只消费组件需求相关部分，并保留 NFR 指标 ID，供 wft05-biz 检查口径一致。

#### PA层收敛项处理

wft03-eng 在 SR-F 功能规则和功能接口中标注了 `PA层收敛` 项。本步负责收敛这些项：

- 功能规则中的 PA层收敛项 → 在**后端运行执行组件**中确定具体实现方式
- 功能接口中的 PA层收敛项 → 在**后端业务能力服务组件**或**前端业务能力支持组件**中确定接口形态
- 收敛结果写入 PA 组件需求规格，标注 `[PA层已收敛]` + 收敛结论

**未收敛项处理**：

- 若 PA层收敛项在本步仍无法确定具体实现方式，标注 `[PA层未收敛]` + 原因说明
- `[PA层未收敛]` 不阻断 PA 方案推进——PA 方案仍可进入 `待确认方案`，但未收敛项须在 Step End 中显式提示人类
- 未收敛项后续处理路径：若与引擎内部机制相关，待 wft05-eng 产出后补全；若与业务侧配置相关，标注为开发阶段待确认项

#### Q1/Q2/Q3 PA 组件匹配

按 SR-F 分类结果 + CU 维度匹配已有 PA 组件节点：

| 判定 | 条件 | 动作 |
|------|------|------|
| **Q1 直接匹配** | 已有 PA 组件对应同一分类桶且职责一致 | 复用已有 PA → 追加 SR-F 引用 → 不重新生成 |
| **Q2 扩展匹配** | 已有 PA 组件但需扩展（新增组件类型/职责） | 扩展已有 PA → 全量重写组件方案（非增量追加） |
| **Q3 新建** | 无可用已有 PA 组件 | 标记待新建 → 生成组件方案 |

#### 到位粒度

必须达到**承接链级 + 组件边界级**：

1. 已判定需要哪些承接组件（5 类中哪些需要、哪些可合并、哪些暂不需要）
2. 已给出各组件的一句话职责边界
3. 已说明配置输入如何进入引擎、结果大致从哪里输出
4. 已完成关键需求条目到承接组件的分配
5. 已标注 25 对齐延续结论（Q1/Q2/Q3）
6. 已标注 PA层收敛项的收敛/未收敛状态

不在本步展开：API 字段级接口规格、配置解析详细步骤、执行引擎内部算法、存储结构/表结构/DTO 全量定义、重试/补偿/异常处理等开发级机制细节。

### Step 4 · 闭环检查

| 检查项 | 内容 |
|--------|------|
| **覆盖率** | 目标 SR-F IPO 的全部功能已有 PA 组件承接 |
| **1:1 承接** | 每条需求条目有唯一 PA 组件宿主 |
| **粒度与边界** | 组件边界清晰、可独立维护与验收 |
| **NFR 完整性** | 相关 `eng/共享` NFR 约束已映射到对应组件类型或组件需求规格，且阻断型组件约束已影响组件类型、边界、接口或运行机制 |
| **NFR 边界正确** | `biz` 配置类约束未在 eng 侧展开为业务配置需求，仅保留引用并提示由 wft05-biz 承接 |
| **biz/eng NFR 一致性** | 共享 NFR 的指标 ID、目标值、测量方法与 wft05-biz 装配口径一致 |
| **承接链一致性** | 与已有组件 I/O 关系无冲突（发现不一致标记为迭代建议，不阻断） |
| **依赖链一致性** | PA 组件间引用关系与 wft03-eng 3b 的 CU 间依赖链一致 |
| **25 对齐一致性** | PA 组件的 25 对齐延续与 wft03-eng 写回 25 的对齐结论一致 |
| **PA层收敛闭合** | 所有 PA层收敛项已在本步收敛或标注 `[PA层未收敛]`，无遗漏 |
| **分类一致性** | 同一分类桶的 SR-F 对应的 PA 组件归属一致 |

### Step 5 · 资产落账与写回

人类已在 Step 2 中整体确认方案，Step 5 统一执行落账与写回。

#### 跨域匹配（Q1/Q2/Q3）

| 匹配 | 内容 |
|------|------|
| **Q1: 域内复用** | 同一业务域内是否已有可复用组件 |
| **Q2: 跨域归并** | 其他业务域是否有同功能组件需归并 |
| **Q3: 共享决策** | 是否需要抽取为跨域共享组件 |

**权限边界**：Q1/Q2/Q3 只做落账级匹配（写到哪里、补什么引用），不做结构级改判（谁来承接、怎么切边界）。若匹配结果意味着主结构需要重定，必须回退 Step 3。

#### 写回操作

1. **写入 07**：PA 组件方案 + 跨域匹配结果 + 组件需求规格（功能清单、接口定义、数据定义、验收标准、NFR 约束承接、25 对齐延续、PA层收敛结论）
2. **写入 06**：SR 详细定义条目（将 SR-F 功能定义格式化为详细条目 + PA层收敛结论）
3. **更新 05**：反向引用更新（SR-F 节点 → `已产品架构`）
4. **更新 23**：PA 组件索引更新
5. **更新 25**：引擎模型引用更新（PA 组件承接关系 + 25 对齐延续 + PA层收敛结论写入 25 CU 引用块）

#### 状态推进

- `05` SR-F 节点推进为 `已产品架构`
- `07` PA 方案推进为 `已验证冻结`

写回要求：幂等去重、局部写回、全貌同步。
