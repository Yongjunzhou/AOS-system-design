# 引擎需求 → STR-E · 引擎定义设计与资产化
**Skill ID**: wft01-eng-or2str

> **概述**：每次执行处理 FR-ENG 类型 OR 条目（≤10 条），STR-E 反馈优先于新 OR 处理。完成 OR→STR-E 引擎定义设计、23/25 资产对准与写回，形成可进入 wft02a 的 STR-E 节点。以配置单元法展开引擎配置单元清单。引擎模型匹配即确定 23 产品类型/构件类型归属（引擎=构件类型）；配置单元逐单元对准 25 配置单元类型。
>
> **输入 → 输出**：FR-ENG OR 条目（≤10 条）/ 有反馈的 STR-E 节点 → STR-E 节点方案 + 23/25 资产对准与写回（状态=`待分解分配`）+ OR 条目推进为 `已分配`
>
> **适用场景**：瀑布式系统设计第1步 · 引擎需求路径。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.0 | **修订**：2026-06-23

---

## 一、职责定位

本 Skill 负责：

1. 接收人类选中的 FR-ENG OR 条目（≤10 条）
2. 匹配候选引擎模型，确定场景定义名称和抽象业务定义类型
3. 以配置单元法展开引擎配置单元清单（配置要素+处理说明+业务说明三栏）
4. 引擎模型 → 23 产品类型/构件类型对准（23-Q1/Q2/Q3）
5. 每个配置单元 → 25 配置单元类型对准（25-Q1/Q2/Q3）
6. 全 Q1 时自动写回 23/25 资产；Q2/Q3 时形成资产建议等人类确认后写回
7. 写回完成后推进 STR-E 节点到 `待分解分配`，对应 OR 到 `已分配`

本 Skill 不负责：

1. 业务需求（FR-BIZ → `wft01-biz-or2str`）
2. 非功能需求（NFR → `wft01-nfr-or2str`）
3. 下游 wft02a A1 路径的详细展开
4. 25 资产中引擎模型的正式定义——那是 25 资产自身的设计产出

### Step 3 · 新增引擎需求设计

#### Phase A：触发与场景边界确定

**A.1 触发识别**

从 OR 语义提取"哪个业务定义的配置需要启动了这项引擎定义需求"。触发 = "业务定义配置人员 + 需要让XX引擎具备YY能力 + 为了支撑ZZ业务定义"。

多引擎 OR 时列出所有候选引擎名称，由人类选择本轮处理哪一个。引擎需求不明确时标记需澄清。

**A.2 引擎模型匹配与 23 产品架构对准**

从 `25-eos-engine-models.md` 匹配候选引擎模型：

| 情形 | 处理 |
|------|------|
| 匹配到单个已知引擎模型 | 加载配置单元类型清单 |
| 匹配到多个引擎模型 | 列出候选，人类选择 |
| 匹配到但配置单元类型清单为空或全为 `模板` | 基于 OR 推断候选配置单元，标记 `[推断]` + `模型内容待填充` |
| 无匹配 | 提出新引擎模型方案建议（引擎名称/类型/定义/候选配置单元），全部标记 `[推断]` + `归属待定` |

状态敏感匹配：`已确认`→正常；`待确认`→标注 `[推断]`；`废弃`→不使用。

引擎模型匹配完成后，加载 `23` 资产全貌层，执行 **23 产品架构对准**：

| 情形 | 处理 |
|------|------|
| 23 中存在该引擎模型对应的产品类型/构件类型 | 23-Q1 + 产品类型 ID + 构件类型 ID |
| 23 中存在相近的可扩展 | 23-Q2 + 扩展建议 |
| 23 中无匹配 | 23-Q3 + 新增建议 |

引擎模型是 EOS 产品架构的一部分，应在 23 中注册为平台引擎类构件类型。

**A.3 场景定义名称确定**

定位于业务定义层——`xx业务配置运行`，xx 为抽象业务定义类型。同引擎多业务定义按"引擎模型+抽象业务定义"双键分组。

**A.4 STR-E 宿主匹配**

以双键搜索已有 STR-E 节点：Q1（复用）/ Q2（扩展，同引擎不同业务定义）/ Q3（新建）。

#### Phase B：配置单元展开

**B.1 候选配置单元加载**：从 25 资产获取该引擎模型的配置单元类型清单。候选集为空时按引擎模型来源分级处理（已确认→推断/待确认→推断+标注/AI建议→推断+归属待定）。

**B.2 多 OR 收敛检测**（复用路径时）：比对锚点为配置要素的职责范围。六类关系——确认/补充/矛盾/无新增/未触及/占位→实体。支持单元内部逐栏标注和粒度调整建议。

**B.3 逐配置单元处理**：

| 触发判定 | 处理 | 来源类型 | 增量标记 |
|---------|------|---------|---------|
| OR 明确触及 | 完整设计三栏 | `OR明示` | `[新增]` |
| OR 隐含触及 | 完整设计三栏，标注推断 | `OR隐含` | `[新增]` + `[推断]` |
| 模型标准单元，OR 未触及 | 占位 | `模型内置` | `[跨场景]` |
| OR 超出已知类型 | 完整设计三栏 + 25 资产扩展建议 | `推断` | `[新增]` + `[推断]` + `模型需扩展` |

**B.4 配置单元设计**（配置要素+处理说明+业务说明三栏）：
- 以 25 资产的职责边界为锚点限定写入范围
- I→P→O 连贯性当场验证，三栏缺一不可
- 含 `[阶段性]` 的不得推进到写回
- 超出部分追加 25 资产扩展建议

**B.5 配置单元排序与标识**：按 25 资产顺序/依赖拓扑/自然时序/OR 触及顺序四级优先级排列。

每个配置单元标识：
- 序号/名称/来源类型/确认状态
- **25 判定**（25-Q1/Q2/Q3）/ **25 引擎模型 ID** / **25 配置单元类型 ID** / **25 资产建议**（Q2/Q3时）

#### Phase C：验证与输出

**C.1 跨单元检查**：覆盖完整性 / 粒度自洽 / 单元间无重叠 / 集中性（同引擎同抽象业务定义）。

**C.2 引擎能力边界陈述**：`「业务定义配置人员」需要为「xx引擎」配置「yy抽象业务定义」的从配置到运行的完整能力，包含 N 个配置单元（…）`。

**C.3 推理依据**：配置常识推理 / 引擎机制推理 / 运行体验推理，按主导角度归类，编号引用。

**C.4 STR-E 完整要素**：

```
STR-E 节点：
├── 定义
├── 场景定义名称（xx业务配置运行）
├── 驱动源（目标抽象业务定义）
├── 候选引擎模型（25 资产引用）
├── 23 产品架构对准（23 产品类型/构件类型/23-Q判定/23资产建议）
├── 引擎能力边界陈述
├── 配置单元清单（含每个单元的 25-Q对准）
├── 推理依据 / 承接的 OR 末级
├── 符合性分析 / 设计指导（待下游生成）
└── 配置单元清单变更摘要
```

---

### Step 4 · 资产写回

#### 写回 23（产品架构资产——引擎模型注册）

- Q1：定位产品类型/构件类型 → 追加引擎模型引用（含引擎名称/定义/25引擎模型ID）→ 记录追溯引用
- Q2/Q3 → 进入 Step 5

#### 写回 25（引擎模型资产——配置单元引用）

每个配置单元：
- Q1：定位引擎模型→配置单元类型块 → 追加配置单元引用
- Q2/Q3 → 进入 Step 5

写回要求：幂等去重、局部写回、全貌同步。

---

### Step 5 · 资产建议反馈

存在 Q2/Q3 时形成 `### 资产建议反馈区`：[23 资产建议] / [25 资产建议]。

`[同意]`→写回；`[修改]`→调整后重新等待；`[驳回]`→重新对准或回退设计。

---

### Step 6 · 状态推进

- `02` 记录 `### 资产写回记录`
- STR-E → `待分解分配`，OR → `已分配`
- OR→STR-E→23/25 引用链记录

## 五、增量标记输出

STR-E 路径特有的标记：

| 标记 | 含义 |
|------|------|
| `[新增]` / `[确认]` / `[矛盾]` / `[推断]` / `[跨场景]` | 同通用定义 |
| `[粒度调整]` | 配置单元建议拆分或合并 |
| `[边界模糊]` | 两个配置单元职责重叠 |
| `[模型需扩展]` | 超出已知配置单元类型 |
| `[23-Q1/23-Q2/23-Q3]` | 引擎模型对 23 资产的对准结果 |
| `[25-Q1/25-Q2/25-Q3]` | 配置单元对 25 资产的对准结果 |

