## 一、跨步骤上下文管理

### 1.1 上下文传递方式

6 步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

### 1.2 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**（见本章末尾）中，供后续步骤引用。

### 1.3 回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见 AI决策框架 §7.2 跨步骤影响范围表）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

## 二、AI 经济原则

本步操作以下列顺序执行，AI 需注意：

1. **执行顺序不可逆**：分析活动 → SR-NFR 详细定义 → 加载 SysReq-NFR 资产 → NFR 架构组织（同步找宿主）→ 约束标注 + 闭环检查
   核心驱动力：NFR 不经过 BA 层（这是与功能分支的关键区别），SR-NFR 直接向 SysReq-NFR 转移
2. **基于资产优先**：SysReq-NFR 架构调整决策应先检查已有资产（现有 SysReq-NFR 分类体系），确实需要修改时再改进，最后才新增
3. **资产路径已知**：本步的资产是 `90-eos-system-product-data/05-eos-system-requirements-architecture.md` 中的非功能部分（v1 基线区）。`02-eos-stakeholder-requirements-architecture.md` 中的 SR-NFR 是本步的输入数据（待处理的 NFR），不是"资产"
4. **NFR 路径与功能路径的区别**：NFR 不经过 BA 层，从 SR-NFR 经 SysReq-NFR 直接分配到 PA 构件（N:1 映射）。量化指标应精确可验证，模糊术语按 L1 补齐规则处理

## 三、本步提示词模板

### 3.1 分析活动：非功能架构验证

```
请在开始非功能详细定义前，对相关方需求非功能部分架构定义执行分析活动。

相关方需求架构定义（非功能部分）：
[粘贴上一步SR-NFR部分的输出]

需求说明符合性分析：
1. 承接完整性 — 每个SR-NFR架构末级节点是否充分满足所承接的原始非功能需求末级
2. 分类正确性 — 非功能需求是否已按质量特性/环境适应性/可实现性三大类正确分类

需求性能符合性分析：
1. 指标量化检查 — 验证性能指标是否已量化（可测量、有数值+单位+条件）
2. 可验证性检查 — 指标是否可测试、有对应的验证方法

输出：分析报告
├─ 指标量化率：[具体数字]
├─ 可验证性：[通过/未通过]

```

### 3.2 SR-NFR 详细定义和量化

```
请对 SR-NFR 架构末级节点进行详细定义，将每条非功能需求量化。

SR-NFR 架构末级节点：[粘贴上一步SR-NFR部分的输出]

每条非功能需求详细定义应包含：
- 需求描述：系统应满足的质量属性说明
- 量化指标：具体数值、单位、测量条件（P99，1000并发下等）
- 测量方法：如何验证
- 应用范围：适用的场景/条件
- 优先级

⚠️ 量化指标补充规则（L1补齐规则）：
  如果原始需求中的指标模糊（如"高性能""高可用"）→
  ├─ 能找到合理行业参考标准（>80% confidence）→ 补充具体数值，标注"AI推断"
  ├─ 多个合理选项 → 列出选项，标注"待选择"
  └─ 无法确定 → 标注"待确认 — 缺失量化指标参考依据"

输出格式：
SR-NFR-XXX: [名称]
需求类型：[性能/安全/可用性等]
需求类别：[质量特性/环境适应性/可实现性]
描述：[质量属性说明]
量化指标：
  - [指标名]: [数值] [单位]（[条件]）
    例如：平均响应时间: < 200ms（1000并发下）
测量方法：[JMeter性能测试/安全审计等]
应用范围：[适用的系统范围]
优先级：[P0/P1/P2/P3] ⚠️ 待确认
```

### 3.3 NFR 架构组织（资产优先 + 同步找宿主）

> 先了解资产再动手——组织前加载 SysReq-NFR 现有架构（v1 基线），理解分类体系和已有节点覆盖情况。然后面对每条量化后 SR-NFR 时，已经知道"哪个类别下有节点、哪个类别还空着"，有方向地去归类。

```
请先加载现有 SysReq-NFR 架构资产（v1 基线·只读），理解其现有的 0-6 级分类结构和各 5 级节点的约束范围后，再逐条处理 SR-NFR。

量化后的 SR-NFR 清单：[粘贴3.2的输出]

—— 第一步：加载并理解 SysReq-NFR 架构资产（先做到心中有数）——

资产文件：90-eos-system-product-data/05-eos-system-requirements-architecture.md（非功能部分 v1 基线·只读）

加载所有现有 SysReq-NFR 5 级节点，对每个节点理解：
├─ 节点名称和定义 — 这个节点"约束什么非功能属性"？
├─ 所属类别 — 属于三大类（质量特性/环境适应性/可实现性）中的哪一类？什么类型（性能/安全/可用性等）？
├─ 现有约束范围 — 这个节点标识了影响哪些 PA 组件类型？
├─ 已承接的 SR-NFR — 这个节点目前已经承接了哪些类别的 NFR？

整体理解：
├─ 三大类覆盖情况：哪些大类已有节点，哪些还空着
├─ 各类型下的粒度：现有 SysReq-NFR 5 级节点是否颗粒度一致

—— 第二步：逐条 SR-NFR 找 SysReq-NFR 宿主 ——

0. 【分类前置判断】这条 SR-NFR 涉及多个非功能类型分支吗？
├─ 例：SR-NFR 同时涉及"性能"和"安全"两个不同维度的约束
├─ 如果是 → 先按类型分支拆分为独立条目，再分别执行下面的步骤
└─ 如果不是 → 直接进入步骤1

1. 【找宿主】用三个维度判断：

   ├─ Q1 非功能类型匹配：此 SR-NFR 的非功能类型（性能/安全/可用性等），与哪个 SysReq-NFR 5 级节点的类型一致？
   │  例："响应时间<200ms" — SysReq-NFR-性能? SysReq-NFR-可用性?
   │  判断依据：SysReq-NFR 节点定义中描述的非功能类型
   │
   ├─ Q2 覆盖范围偏差：此 SR-NFR 的具体内容，是否在 SysReq-NFR 节点定义的边界之内？
   │  例：SysReq-NFR-性能节点的定义是"系统响应时间和吞吐量" → 
   │  "并发用户数≥100"属于SysReq-NFR-可扩展性而非SysReq-NFR-性能
   │
   └─ Q3 同类承接情况：此 SysReq-NFR 节点是否已经在承接同类 SR-NFR？

   根据三个问题的答案做决策：
   ├─ 条件A — 复用已有节点：
   │  Q1命中 + Q2在边界内 + Q3同类
   │  → SR-NFR-XXX → SysReq-NFR-YYY（复用已有节点）
   │
   ├─ 条件B — 扩展节点定义后承接：
   │  Q1命中 + Q2略超边界 + Q3同类
   │  → 标记 SysReq-NFR-YYY（改进-扩展），记录扩展理由
   │
   ├─ 条件C — 新增 SysReq-NFR 节点：
   │  Q1无命中 + 不在任何 SysReq-NFR 节点边界内
   │  → 记录需新增节点，写明新增原因和所属大类+类型
   │
   └─ 条件D — ❓ 无法判断：
       Q1命中了多个类型 / Q2边界完全模糊 / Q3信息不足以对比
       → 标记❓，说明卡在哪个维度上，**不要强行推测**，留给人类

2. 【记录】同步记录本步决策：
├─ 承接的 SysReq-NFR 5 级：[SysReq-NFR-YYY / 新增]
├─ 资产决策：[复用 / 改进-扩展 / 新增]
└─ 决策理由：引用以上三个判断维度的结果

⚠️ 关键判断原则：
- "非功能类型匹配"的含义：SR-NFR 的非功能类型与 SysReq-NFR 节点的类型定义对应
  （如"平均响应时间<200ms"→性能类型、"可用性≥99.9%"→可用性类型）
- NFR 不经过 BA 层，SR-NFR 直接分配到 SysReq-NFR 5 级节点
- ⚠️ **扩展和新增的强内聚性检查**：
  ├─ 扩展前问"扩展后这个 SysReq-NFR 节点还描述一个单一非功能类型吗？"
  │  例：SysReq-NFR-性能节点扩展为"性能和可用性"→ 两个类型不同，应新增节点而非扩展
  ├─ 新增前问"这个新节点和同级兄弟节点的分类颗粒度一致吗？"
  │  例：相邻节点都是"某个具体非功能指标"的粒度，新节点不能是一个"大类"
  └─ 颗粒度标准：该 5 级节点应可对应一个可独立验证的非功能指标集合

—— 输出 ——

SR-NFR → SysReq-NFR 映射清单：
| SR-NFR | 描述 | 承接的 SysReq-NFR 5级 | 资产决策 | 决策理由（Q1/Q2/Q3） |
|--------|------|---------------------|---------|--------------------|
| SR-NFR-XXX | [描述] | SysReq-NFR-YYY | 复用 ✅ | Q1命中[类型]，Q2在边界内，Q3已有同类 |
| SR-NFR-YYY | [描述] | SysReq-NFR-ZZZ | 改进-扩展 ⚠️ | Q1命中，Q2略超[范围]，需扩展定义 |
| SR-NFR-ZZZ | [描述] | [新增] | 新增 ⚠️ | Q1无命中，无 SysReq-NFR 节点类型匹配 |
| SR-NFR-WWW | [描述] | — | ❓ 无法判断 | Q1命中多个[SysReq-NFR-AAA][SysReq-NFR-BBB]，无法确定主方向 |
```

## 五、审查修复流程

### 5.1 审查输入

AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

### 5.2 AI 修复流程

收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）→ 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题** → 参考AI决策框架的跨步骤影响范围表进行影响范围评估，列出所有受影响的下游节点

### 5.3 影响范围评估

| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 某步的架构定义 | 本层 + 下游已建立映射的节点 | 修改后同步更新映射 |
| 详细定义 | 下层方案 | 修改后检查下层是否受影响 |
| 映射关系 | 所有关联节点 | 需重新运行追溯验证 |
| 编号变更 | 全链路 | 需逐层更新引用，重新验证 |
