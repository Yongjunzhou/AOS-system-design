## 一、跨步骤上下文管理

### 1.1 上下文传递方式

8 步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

### 1.2 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**（见本章末尾）中，供后续步骤引用。

### 1.3 回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见 AI决策框架 §7.2 跨步骤影响范围表）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

## 二、AI 经济原则

本步操作以下列顺序执行，AI 需注意：

1. **执行顺序不可逆**：全链路合并验证 → 1:1 约束验证 → N:1 支持验证 → 双向追溯 → 逆向一致性验证 → 生成追溯矩阵 → 生成验证报告
   核心驱动力：从最底层（PA）开始，逐层向上验证逆向推导的自洽性。每一层的输出与前一层的输入必须可双向追溯
2. **基于资产优先**：各步的决策记录是本步的核心资产——验证不仅要检查"是否连上了"，还要检查"连得是否合理"。逆向推导特有的"制品差异"是验证重点
3. **逆向推导特点**：本步验证的关键区别在于检查"从产物归纳的架构推导链"是否自洽，而非"从需求分解的架构链"。特别关注推导矛盾（上下层之间的语义不一致）和空壳组件（有节点无实质内容）

## 五、审查修复流程

### 5.1 审查输入

AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

### 5.2 AI 修复流程

收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）→ 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题** → 参考AI决策框架的跨步骤影响范围表进行影响范围评估，列出所有受影响的下游节点

### 5.3 影响范围评估

| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 某步的架构定义 | 本层 + 下游已建立映射的节点 | 修改后同步更新映射 |
| 详细定义 | 下层方案 | 修改后检查下层是否受影响 |
| 映射关系 | 所有关联节点 | 需重新运行追溯验证 |
| 编号变更 | 全链路 | 需逐层更新引用，重新验证 |
