## 一、产品定位

### 1.1 产品定义

产品 A — **AOS 企业运营体系**（Enterprise Operation System），是企业实际运行所依赖的 IT/AI 化运营系统。**团队 A（AOS 开发者）** 负责开发和持续优化产品 A。

> **一句话**：AOS 是帮助企业实现运营管理 IT 化、AI 化的综合系统，涵盖流程管理、绩效管理、组织管理、决策支持等核心运营领域。

### 1.2 产品类型

AOS 属于 **IT/AI 化运营系统（软件产品）**。其产品架构分类体系为：

| 分类 | 说明 | 典型组件 |
|------|------|---------|
| **前端层** | 运营管理用户界面 | 运营门户、管理仪表盘、流程设计器、报表查看器 |
| **后端层** | 运营业务逻辑和服务 | 流程引擎、绩效计算引擎、规则引擎、审批服务 |
| **数据层** | 运营数据存储和处理 | 运营数据仓库、指标库、日志存储、缓存 |
| **基础设施层** | 系统支撑服务 | API 网关、统一认证、监控告警、消息中间件 |

### 1.3 产品开发阶段

AOS 产品开发分为四个阶段：

| 阶段 | 名称 | 说明 |
|------|------|------|
| 阶段1 | **系统设计** | 从 AOS 原始需求出发，经过五层结构（OR → SR → BA → SysReq → PA），开发出 AOS 产品架构 |
| 阶段2 | **构件开发** | 对 AOS 产品架构中所有需要新开发或改进的构件，根据其需求定义进行开发 |
| 阶段3 | **系统集成与交付** | 沿产品架构的树形层级从叶子节点逐级向上集成至根节点，交付用户 |
| 阶段4 | **系统运维** | 系统上线后的运维管理 |

当前工作范围严格限定在**阶段1（系统设计）**。

### 1.4 三层工作层面与工具来源

本目录下的 AI 辅助文档是产品B（系统设计流水线）的构件，属于**第2层工作（交付物构建）**。三层工作层面框架：

| 层面 | 内容 | 负责方 |
|------|------|--------|
| **第1层：自指设计** | 产品B用统一方法论对其自身需求进行系统设计，推导出产品B的产品架构 | 团队B |
| **第2层：交付物构建** | 将产品B的系统设计产物（准则、指南、AI辅助文档、任务定义）适配到产品A的上下文，形成团队A使用的设计工具 | 团队B |
| **第3层：使用工具设计产品A** | 团队A使用第2层产出的工具，对运营体系进行系统设计，产出产品A的产品数据 | 团队A（AOS 开发者） |

> **AOS 开发者视角**：AOS 设计工作对应第3层——使用产品B交付的工具（准则、指南、AI辅助文档、任务定义）来设计 AOS 运营体系。你的核心产出是产品数据文档（`10-aos-system-product-data/`）。

### 1.5 运营体系中的端到端业务框架

AOS 是为企业运营体系服务的系统。理解运营体系中的端到端业务框架，有助于 AI 在系统设计过程中准确把握每个项目的业务边界和接口关系。

#### 1.5.1 端到端的定义

**从需求提出到需求满足**。

#### 1.5.2 三类客户与三个业务维度

企业有三类客户，每类客户的需求构成一个业务维度：

| 客户类型 | 提出的需求 | 对应的业务维度 |
|---------|-----------|--------------|
| 产品用户/客户 | 产品订货合同、产品研发需求、售后服务需求 | **主价值链** |
| 上级/监管方 | 经营管理目标、合规及资质取证、年度审核 | **管理监管** |
| 内部员工 | 对人、资金、物料、设备、设施、工具、数据、知识资产等需求 | **支持服务** |

三个维度不是层级拆分，而是同一企业业务的三个视角。同一项目同时受三个维度作用——归属于某个维度，被其他维度管理，向其他维度提需求。

#### 1.5.3 项目间接口原则

需求在端到端项目A的具体业务运行中产生，接收方是端到端项目B整体（BA 2级），B接收后自主决定内部如何分解响应，A不需要了解B的内部结构。三种接口关系：

| 关系 | 本质 | 适用范围 |
|-----|------|---------|
| 上下游 | 链式传递 | 主价值链内部 |
| 管理/被管理 | 下达要求、监督考核 | 管理监管 → 其他维度 |
| 支持/被支持 | 提出需求、统筹满足 | 支持服务 → 其他维度 |

#### 1.5.4 端到端项目业务分解方法

端到端项目的业务分解从**被实现对象的系统架构**中工程化推导而来：

```
端到端项目（BA 2级）
├── 价值业务 ← 由项目的"产品"决定
│     └── 项目产出什么（广义产品/服务）
│           └── 该产品的系统架构 → 架构节点类型
│                 └── 每个节点类型 → 一个从需求提出到需求满足的端到端业务
└── 管理业务 ← 项目本身的管控活动（目标、进度、成本、质量等）
```

项目的系统设计（五层方法论）产出系统架构，架构节点类型定义了价值业务的边界。AOS 中每个端到端项目（如产品开发项目、生产交付项目）都可以用此方法分解。

详见 AOS 设计准则 [§1.9 运营体系中的端到端业务框架](../01-aos-system-design-standards/01-aos-system-design-standards.md#19-运营体系中的端到端业务框架)。

## 二、用户画像

### 2.1 三个关键角色

| 角色 | 职责 | 在瀑布式中的参与 |
|------|------|-----------------|
| **AOS 开发者**（团队A） | 提出 AOS 原始需求、审核设计方案、确认技术决策 | 需求提出、每步结果审核、关键决策 |
| **AI**（执行者） | 按指令执行规范化、分解、映射、验证等自动化任务 | 所有步骤的自动生成和检查 |
| **流水线开发者**（团队B） | 定义和优化产品 B（设计流水线）的准则、指南、AI 辅助文档 | 不直接参与 AOS 设计 |

### 2.2 角色交互方式

```
AOS 开发者 ──(提出原始需求)──→ AI
     │                            │
     │(审核/确认/决策)           │(调用产品B的构件)
     │                            ▼
     └───────── AI ──────────→ 执行系统设计任务
                                      │
                                      │(输出)
                                      ▼
                              AOS 设计文档集
```

## 三、产品全景

### 3.1 核心业务领域

AOS 企业运营体系覆盖以下核心业务领域：

| 业务领域 | 说明 | 典型流程 |
|---------|------|---------|
| **战略管理** | 企业战略制定、分解、跟踪 | 战略规划→关键结果分解→执行跟踪 |
| **流程管理** | 业务流程建模、执行、优化 | 流程定义→流程审批→流程执行→流程分析 |
| **绩效管理** | KPI 体系、绩效考核、绩效分析 | 指标定义→数据采集→考核评价→绩效分析 |
| **组织管理** | 组织架构、岗位职责、权限管理 | 组织设置→岗位定义→权限分配 |
| **运营监控** | 运营数据监控、异常预警、决策支持 | 数据采集→指标监控→异常告警→决策建议 |

| 场景 | 适用情况 | 周期 | 任务数 |

## 四、设计约束

### 4.1 版本策略

- 版本号格式：`v[主版本].[次版本]`
- 语义化版本号（主版本破坏兼容、次版本兼容变更）
- 准则的版本变更会级联影响指南、AI 辅助文档和任务定义

### 4.2 文档规范

- 所有文档使用 **Markdown + Mermaid** 格式
- 文件命名规范：`NN-descriptive-name.md`
- 中文为主体语言，术语附英文对照
- 每个文档包含头部元数据（标题/版本/日期）和尾部版本记录

### 4.3 设计方法论（核心框架）

**五层结构**：
```
第1层：原始需求及分析 (OR)          ← AOS 原始需求（业务部门提出）
    ↓ N:1 分配
第2层：相关方需求 (SR)              ← AOS 各相关方（管理者/执行者/IT）的需求
    ├─ 功能部分 → 第3层：业务架构 (BA)  ← AOS 运营业务流程（IPO 模型）
    └─ 非功能部分 → 第4层：系统需求-NFR
                    第4层：系统需求 (SysReq) ← BA IPO 去重后映射
                        ↓ N:1 分配
                    第5层：产品架构 (PA)     ← AOS 软件组件（前端/后端/数据/基础设施）
```

**三条核心规则**：
- **1:1 分配约束**：每条详细定义末级节点只能分配到下层方案中唯一一个架构末级节点
- **N:1 承接支持**：一个架构末级节点可以承接多个上层详细定义末级节点
- **双向追溯**：支持从上层需求追溯到下层方案，也支持下层方案反向追溯到上层需求
