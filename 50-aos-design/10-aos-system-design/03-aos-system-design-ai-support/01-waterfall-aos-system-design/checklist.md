# AOS 瀑布式系统设计：AI 自检清单
**AOS Waterfall System Design: AI Self-Check List**

**用途**：AI 完成每一步后，使用此清单自行验证产出质量。全部通过后方可提交给运营体系开发者审核。

---

## 一、跨步骤通用检查（适用于所有步骤）

### 完整性
- [ ] 每一步的所有输入文件是否存在且已读取
- [ ] 每一步的所有输出文件是否已生成且格式完整
- [ ] 是否标记了需要人工确认的待确认项（以 `⚠️ 待确认` 标记）

### 格式规范性
- [ ] 架构定义文档均包含两部分：第一部分（树形结构）+ 第二部分（映射关系表）
- [ ] 文档编号格式统一（OR-XXX / SR-F-XXX / SR-NFR-XXX / BA-XXX / SysReq-F-XXX / SysReq-NFR-XXX / PA-XXX）
- [ ] 树形结构层级缩进正确，末级节点清晰标注
- [ ] 超链接格式正确，指向有效目标

### 术语一致性
- [ ] "架构末级节点"和"详细定义末级节点"未混用
- [ ] "1:1 分配约束"和"N:1 承接支持"未混淆
- [ ] 功能需求和非功能需求的标识统一（SR-F vs SR-NFR / SysReq-F vs SysReq-NFR）

### 资产优先原则（新增）
- [ ] 每层分配前，是否先完成了需求侧的自治分析（可复用性/冲突性/规范性）？
- [ ] 分配下游方案时，是否遵循了复用→改进→新增的优先级？
- [ ] 废弃/合并的节点是否标注而非物理删除？
- [ ] 同一节点是否经历 3 次以上 Extend 却未审查 Split？

### 依据B对齐
- [ ] 输出的 AOS 设计文档是面向产品A（AOS运营体系），而非产品B（流水线）
- [ ] AOS 的设计内容使用了运营体系领域术语（业务流程、组织管理、绩效指标等），而非流水线术语
- [ ] AOS 产品架构的分类体系符合产品类型（AOS为IT/AI化运营系统 → 软件产品分类：前端/后端/数据/基础设施）
- [ ] 涉及引用示例时，示例内容反映 AOS 的业务场景

## 二、第1步自检：原始需求分析及相关方需求架构定义

### 原始需求处理
- [ ] 所有 AOS 原始需求已完成四项检查（完整性、清晰性、可测试性、独立性）
- [ ] 模糊术语已替换为精确表述
- [ ] 功能需求和非功能需求已明确分离
- [ ] 每条原始需求已分解到末级
- [ ] 每条末级需求满足 1:1 约束（只需一个 SR 架构末级节点承接）
- [ ] 冲突检测和重复检测已完成，结果已记录
- [ ] 所有 `⚠️ 待确认` 标记已标注

### 相关方需求架构定义
- [ ] 架构定义包含两个部分：树形结构 + 映射关系表
- [ ] 所有原始需求末级已映射到 SR 架构末级节点（覆盖率 100%）
- [ ] 功能需求（SR-F）和非功能需求（SR-NFR）分别处理
- [ ] 每个架构末级节点包含符合性分析和设计指导
- [ ] 每个架构末级节点包含性能指标
- [ ] SR-NFR 架构末级节点包含需求类型（性能/安全/可用性等）和需求类别（质量特性/环境适应性/可实现性）

## 三、第2步自检：相关方需求功能部分详细定义及业务架构定义

### 相关方需求功能部分详细定义
- [ ] 所有 SR-F 架构末级节点已分解到详细定义末级
- [ ] 每条详细定义末级只分配到一个业务架构 IPO（1:1 约束满足率 100%）
- [ ] 每条详细定义末级包含验收标准
- [ ] 每条详细定义末级标注了分配目标

### 业务架构 IPO 定义
- [ ] 每个 IPO 包含完整的 I（输入）/ P（处理）/ O（输出）三维定义
- [ ] IPO 去重已执行（复用→改进→新增），无重复 IPO
- [ ] 每个 IPO 包含符合性分析和设计指导
- [ ] 性能指标已从相关方需求正确继承
- [ ] BA 命名符合规范——IPO 名称使用"动词+名词"结构（如"流程建模""绩效计算""权限配置"）
- [ ] 每个 BA 节点已标注类型（配置方式/开发方式），对应正确的层级结构

### 同步设计
- [ ] 已同步建立系统需求功能部分 5 级节点结构骨架
- [ ] BA → SysReq 骨架映射关系已建立

## 六、第5步自检：系统需求详细定义及产品架构定义

### 系统需求详细定义
- [ ] 所有 SysReq-F 5 级节点已分解到 9 级场景活动
- [ ] 分解链完整：5级→6级→7级→8级→9级
- [ ] 每条 9 级场景活动只分配到一个 PA 组件（1:1 约束满足率 100%）
- [ ] 9 级场景活动粒度一致（可操作步骤级别）
- [ ] 正常/异常/替代场景已覆盖

### 产品架构定义
- [ ] 组件分类完整（AOS 为 IT/AI 化运营系统 → 软件产品分类：前端/后端/数据/基础设施）
- [ ] 每个 PA 组件有清晰的定义和职责边界
- [ ] 每个 PA 组件包含符合性分析和性能指标
- [ ] SysReq 9 级 → PA 映射覆盖率 100%
- [ ] 9 级活动中的 PA ID 与映射关系表中的 PA ID 一致（无编号引用错误）
- [ ] 每个 PA 组件标注了受哪些 NFR 约束
- [ ] 组件粒度和数量合理（一个组件承接 2-5 个 SysReq 9 级节点）
- [ ] NFR 冲突时有权衡决策记录

### 详细定义
- [ ] 每个 PA 组件有技术栈定义
- [ ] 每个 PA 组件有关键模块说明
- [ ] 每个 PA 组件有接口定义
- [ ] 每个 PA 组件有部署方式定义
- [ ] 每个 PA 组件有性能指标定义

## 七、第6步自检：双向追溯验证

### 1:1 分配约束
- [ ] 原始需求末级 → SR 架构末级：1:1 满足率 100%
- [ ] SR 详细定义末级 → BA IPO：1:1 满足率 100%
- [ ] BA IPO → SysReq 5 级：1:1 满足率 100%（去重后的 IPO）
- [ ] SysReq 9 级 → PA 末级：1:1 满足率 100%

### N:1 承接支持
- [ ] 每个 SR 架构末级节点承接 2-5 条原始需求末级（合理范围）
- [ ] 每个 BA IPO 承接 2-5 条 SR 详细定义末级（合理范围）
- [ ] 每个 SysReq 5 级节点承接 2-5 个 BA IPO（合理范围）
- [ ] 每个 PA 组件承接 2-5 个 SysReq 9 级节点（合理范围）
- [ ] 承接 1 条的已考虑合并，承接 6+ 条的已检查是否需要拆分

### 双向追溯
- [ ] 正向追溯完整：OR → SR → BA → SysReq → PA（100%）
- [ ] 反向追溯完整：PA → SysReq → BA → SR → OR（100%）
- [ ] 每个方案节点标注了承接来源（向上追溯）
- [ ] 每条需求标注了分配目标（向下追溯）

### 符合性分析
- [ ] 所有方案节点都有符合性分析（完整率 100%）
- [ ] 每个符合性分析说明了如何满足所承接的需求
- [ ] 符合性分析包含验证方法

### 性能指标跨层一致性
- [ ] 性能指标从 OR → SR → BA → SysReq → PA 逐层继承，无断层
- [ ] PA 组件的性能指标不低于 SysReq 的要求
- [ ] 所有量化指标包含数值、单位、测量条件
