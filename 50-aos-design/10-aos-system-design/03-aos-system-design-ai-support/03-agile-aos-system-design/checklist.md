# AOS 敏捷式系统设计：AI 自检清单
**AOS Agile System Design: AI Self-Check List**

**用途**：AI 完成每一步后，使用此清单自行验证产出质量。全部通过后方可提交给运营体系开发者审核。

---

## 一、通用检查（适用于所有步骤）

### 增量标记正确性
- [ ] 所有新增条目使用 `+` 前缀标记
- [ ] 所有修改条目使用 `~` 前缀标记
- [ ] 删除条目使用 `-` 前缀标记（极少使用，已确认）
- [ ] `+` 条目格式完整（定义/承接/符合性等全部字段）
- [ ] `~` 条目仅列出变更内容（未误用完整格式）

### 影响范围完整性
- [ ] 所有受影响的层已识别（无遗漏）
- [ ] 所有受影响的现有节点已标记 `~`（无遗漏）
- [ ] 新增节点与现有节点无重复
- [ ] 已检查是否需要同步更新下层架构骨架

### 格式规范性
- [ ] 文档编号格式统一（OR-XXX / SR-F-XXX / BA-XXX / SysReq-F-XXX / PA-XXX）
- [ ] 增量标记后的节点编号与现有体系一致
- [ ] 超链接格式正确，指向有效目标

### 术语一致性
- [ ] "架构末级节点"和"详细定义末级节点"未混用
- [ ] "1:1 分配约束"和"N:1 承接支持"未混淆
- [ ] 功能需求和非功能需求的标识统一（SR-F vs SR-NFR / SysReq-F vs SysReq-NFR）

### 资产优先原则（新增）
- [ ] 每层分配前，是否先完成了需求侧的自治分析（可复用性/冲突性/规范性）？
- [ ] 分配下游方案时，是否遵循了复用→改进→新增的优先级？
- [ ] 废弃/合并的节点是否标注而非物理删除？
- [ ] 同一节点是否经历 3 次以上 Extend 却未审查 Split？

### 依据B对齐
- [ ] 增量更新的 AOS 设计文档是面向产品A（AOS运营体系），而非产品B（流水线）
- [ ] AOS 的设计内容使用了运营体系领域术语（业务流程、组织管理、绩效指标等），而非流水线术语
- [ ] AOS 产品架构的分类体系符合产品类型（AOS为IT/AI化运营系统 → 软件产品分类：前端/后端/数据/基础设施）
- [ ] 涉及引用示例时，示例内容反映 AOS 的业务场景

## 八、第7步自检：完整性和一致性验证

### 1:1 分配约束
- [ ] 新 OR 末级 → SR 架构末级：1:1 满足率 100%
- [ ] 新 SR 详细定义末级 → BA IPO：1:1 满足率 100%
- [ ] BA IPO → SysReq 5 级：1:1 满足率 100%（去重后）
- [ ] SysReq 9 级 → PA 末级：1:1 满足率 100%

### N:1 承接支持
- [ ] 每个 SR 架构末级节点承接 2-5 条 OR 末级（合理范围）
- [ ] 每个 BA IPO 承接 2-5 条 SR 详细定义末级（合理范围）
- [ ] 每个 SysReq 5 级节点承接 2-5 个 BA IPO（合理范围）
- [ ] 每个 PA 组件承接 2-5 个 SysReq 9 级节点（合理范围）

### 双向追溯
- [ ] 正向追溯完整（新需求 → SR → BA → SysReq → PA）
- [ ] 反向追溯完整（PA → SysReq → BA → SR → 新需求）
- [ ] 每个新增/修改节点标注了承接来源
- [ ] 每条新增/修改需求标注了分配目标

### 增量一致性
- [ ] 同步设计一致性：上层详细定义的更新与下层架构定义的更新一致
- [ ] 无"更新了上层但漏掉了下层"的情况
- [ ] 新增节点 ID 与现有体系无冲突
- [ ] 修改节点 ID 在现有体系中存在
- [ ] 新增/修改的性能指标在 OR → SR → BA → SysReq → PA 间一致传递
- [ ] 新增/修改的 PA 组件性能指标不低于所承接 SysReq 的要求

### 符合性分析
- [ ] 所有新增/修改的方案节点都有符合性分析
- [ ] 符合性分析说明了如何满足所承接的需求
- [ ] 符合性分析包含验证方法
