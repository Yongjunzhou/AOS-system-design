## 📌 场景描述

### 适用条件

- ✅ 产品已上线运行
- ✅ 在使用过程中发现 bug 或紧急需求
- ✅ 需要快速响应（几小时到1-2天）
- ✅ 需要最小化变更，避免影响现有功能
- ✅ 适合 DevOps 团队快速迭代

### 不适用条件

- ❌ 需要进行大规模重构
- ❌ 影响范围超过50%
- ❌ 需要完整的系统设计

### 参与角色

本指南涉及三个核心参与者，在 DevOps 设计流程中各司其职：

| 角色 | 职责 | 参与环节 |
|------|------|---------|
| **AOS设计人员** | 定位运行问题，选择修复模式，验证方案有效性 | 所有步骤 |
| **AI** | 按指令执行影响范围评估、快速修复生成、映射关系更新等自动化任务 | 所有步骤的自动生成和检查 |
| **需求提出者** | 报告运行问题和变更需求，参与修复确认 | 第1步问题输入 |

## 📊 核心概念与准则映射

### 分析活动的 DevOps 适配

系统设计准则要求每层在架构定义完成后执行两类分析活动。在 DevOps 场景中，分析活动以**影响评估**的形式嵌入各模式：

| 准则的分析活动 | DevOps 适配 | 应用场景 |
|--------------|------------|---------|
| **需求说明符合性分析** | 影响范围验证（修改后的构件是否完整覆盖所承接的需求） | 三种模式均适用 |
| **需求性能符合性分析** | 回归验证（修改是否引入了性能退化） | 模式2/3重点关注 |

### 架构定义与详细定义在 DevOps 中的关注点

| 子文档 | 正向设计 | DevOps 快速修改 |
|-------|---------|---------------|
| **架构定义文档**（树形结构 + 映射关系表） | 先定义架构再分解 | 先更新映射关系表，再决定是否调整树形结构 |
| **详细定义文档**（对架构末级节点的分解） | 对架构末级节点分解 | 定位到具体的详细定义末级节点进行修改 |

### 三类影响范围的 DevOps 适配

| 影响范围 | 对应层 | 分析活动重点 | 同步设计重点 |
|---------|-------|------------|------------|
| **仅 PA** | 第5层 | 验证修改不破坏现有 PA→SysReq 映射 | 只需更新 PA 文档 |
| **SysReq + PA** | 第4-5层 | 验证修改后 SysReq↔PA 映射一致 | 同步更新两个文档的映射关系表 |
| **跨多层** | 第2-5层 | 每层验证修改不破坏追溯链路 | 按敏捷式指南执行，加速进行 |

### 两种工作流程的关系

系统设计准则定义了两阶段工作流程（§1.5.3）作为同步设计原则的具体操作方法。DevOps 的三种模式实际上是两阶段工作流程的简化变体：

| 设计准则 | DevOps 模式1 | DevOps 模式2 | DevOps 模式3 |
|---------|------------|------------|------------|
| 第一阶段：结构骨架约定 | 快速定位 PA 构件 | 快速定位 SysReq+PA 节点 | 快速评估各层骨架 |
| 第二阶段：详细内容填充 | 直接修改 PA 条目 | 修改 SysReq+PA 条目 | 跨层修改条目 |
| 同步设计 | 仅更新 PA | 同时更新 SysReq↔PA 映射 | 各层映射同步更新 |

## 📊 三种处理模式

### 模式1：快速修复（几小时）
**适用**：仅影响产品架构的 bug 或小改动

```
发现问题 → 快速定位 → 直接修改产品架构 → 快速验证 → 发布
```

**特点**：
- ✅ 不需要改动上层设计
- ✅ 最快的响应时间
- ✅ 最小的变更范围

**涉及规范**：[产品架构规范 §2 架构定义规范](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#二架构定义规范)

**示例**：
- 修复 UI 显示 bug
- 调整性能参数
- 修复前端逻辑错误

### 模式2：局部设计（1天）
**适用**：影响产品架构和系统需求的问题

```
发现问题 → 评估影响 → 快速设计 → 实现 → 验证 → 发布
```

**特点**：
- ✅ 需要修改系统需求和产品架构
- ✅ 不影响业务架构
- ✅ 相对快速的响应时间

**涉及规范**：
- [系统需求规范 §2 架构定义规范](../01-aos-system-design-standards/05-aos-system-requirements-specification.md#二架构定义规范)
- [产品架构规范 §2 架构定义规范](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#二架构定义规范)

**示例**：
- 修复系统功能 bug
- 实现小功能需求
- 优化系统性能

### 模式3：完整设计（2-3天）
**适用**：影响多层设计的问题

```
发现问题 → 完整评估 → 完整设计 → 实现 → 验证 → 发布
```

**特点**：
- ✅ 需要修改多层设计
- ✅ 可能影响业务架构
- ✅ 需要完整的系统设计流程

**涉及规范**：全部 02-06 层规范（依影响层级选择）

**示例**：
- 修复业务流程 bug
- 实现重要功能需求
- 架构级别的优化

## 📋 快速决策树

```
发现问题
  ↓
【第1步】快速影响范围评估（15分钟）
  ↓
问题是否仅影响产品架构？
  ├─ 是 → 【模式1：快速修复】
  │       └─ 直接修改PA → 验证 → 发布（几小时）
  │
  └─ 否 → 问题是否影响系统需求？
      ├─ 是 → 问题是否影响业务架构？
      │       ├─ 否 → 【模式2：局部设计】
      │       │       └─ 修改SysReq + PA → 验证 → 发布（1天）
      │       │
      │       └─ 是 → 【模式3：完整设计】
      │               └─ 完整设计流程 → 验证 → 发布（2-3天）
      │
      └─ 否 → 问题是否影响相关方需求？
          ├─ 是 → 【模式3：完整设计】
          │       └─ 完整设计流程 → 验证 → 发布（2-3天）
          │
          └─ 否 → 需要重新评估问题
```

## 📋 完整操作流程

### 第1步：快速影响范围评估（15分钟）

**目标**：快速判断问题的影响范围，选择合适的处理模式

**具体操作**：

1. **问题分类**
   - Bug 修复 vs 功能需求 vs 性能优化
   - 紧急程度：P0（立即修复）vs P1（今天修复）vs P2（本周修复）

2. **影响范围快速评估**
   - 问题是否仅影响产品架构（PA）？
   - 是否影响系统需求（SysReq）？
   - 是否影响业务架构（BA）？
   - 是否影响相关方需求（SR）？

3. **处理模式选择**
   - 仅 PA → 模式1（快速修复）
   - PA + SysReq → 模式2（局部设计）
   - 跨多层 → 模式3（完整设计）

**输出**：
- 问题分类和优先级
- 影响范围评估
- 选定的处理模式

---

### 第2步：快速修复（模式1，几小时）

**目标**：快速修复仅影响产品架构的问题

**涉及规范**：[产品架构规范 §3 详细定义规范](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#三详细定义规范)

**具体操作**：

1. **问题定位**
   - 定位到具体的产品构件（PA 架构末级节点）
   - 理解问题的根本原因

2. **分析活动：影响验证**
   - **需求说明符合性分析**：修改是否影响该 PA 构件所承接的 SysReq 9 级场景活动的覆盖完整性？是否有功能因修改而缺失？
   - **需求性能符合性分析**：修改是否引入了性能退化？响应时间、吞吐量是否仍在 SysReq 要求的范围内？**指标追溯**（修改后 PA 构件的性能指标仍可追溯到所承接的 SysReq）

3. **快速修改**
   ```
   ~ PA-XXX: [产品构件名称]（3级·架构末级节点）
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对实现方案的具体修改]（技术栈、接口、关键模块等）
   ├─ 承接的系统需求：[SysReq-XXX]（保持原有映射关系）
   ├─ 符合性分析：[修改后如何满足所承接的系统需求]
   └─ 验证方法：[如何验证修复]
   ```

   > **标记说明**：`~` 前缀表示修改现有条目。在快速修复场景下，PA 条目采用简化格式（聚焦原问题和修改内容），但**符合性分析**必须保留以确保追溯链路的完整性。

4. **文档结构同步更新**
   - 修改后同步更新产品架构文档中的两个子文档：
     - **架构定义文档**：确认树形结构无需调整，更新映射关系表中对应的符合性分析
     - **详细定义文档**：更新该构件的实现细节和技术方案描述

5. **快速验证**
   - 验证问题是否解决
   - 验证是否有副作用
   - 验证性能是否受影响

6. **发布**
   - 更新产品架构文档的版本号
   - 发布修复

**质量检查**：
- [ ] 问题是否完全解决
- [ ] 是否有副作用（回归测试通过）
- [ ] 性能是否在 SysReq 指标范围内
- [ ] 符合性分析已更新
- [ ] 产品架构文档的架构定义和详细定义同步更新

**输出**：
- 修复后的产品架构文档（架构定义 + 详细定义）
- 验证报告

---

### 第3步：局部设计（模式2，1天）

**目标**：快速设计和实现影响系统需求和产品架构的问题

**涉及规范**：
- [系统需求规范 §2 架构定义规范](../01-aos-system-design-standards/05-aos-system-requirements-specification.md#二架构定义规范)
- [系统需求规范 §3 详细定义规范](../01-aos-system-design-standards/05-aos-system-requirements-specification.md#三详细定义规范)
- [产品架构规范 §2 架构定义规范](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#二架构定义规范)

**具体操作**：

1. **分析活动：影响范围分析**
   - **需求说明符合性分析**：问题影响哪些 SysReq 5 级节点和 9 级场景活动？修改后 SysReq↔PA 的映射关系是否一致？
   - **需求性能符合性分析**：修改 SysReq 的性能指标或 PA 的实现方案后，性能指标是否仍然匹配？**指标追溯**（SysReq 节点的性能指标可追溯到 PA 构件或上层需求的指标要求）

2. **系统需求修改**

   **功能需求分支（0-9级结构）**：
   - 定位到受影响的 SysReq 5 级架构末级节点和 9 级详细定义末级节点
   ```
   ~ SysReq-F-XXX: [修改的系统功能需求]（5级·架构末级节点）
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对需求定义或性能指标的修改]
   ├─ 对应的PA构件：[PA-XXX]
   ├─ 符合性分析：[修改后如何满足所承接的需求]
   └─ 验证方法：[如何验证]

   （如影响9级场景活动，同步更新详细定义分解链）
   ~ SysReq-F-XXX.X.X.X.X（9级·场景活动）
   ├─ 修改内容：[活动步骤的修改]
   ├─ 分配到：PA-XXX[保持或更新映射目标]
   └─ 验收标准：[更新后的验收条件]
   ```

3. **产品架构同步修改**
   - 同步更新对应的 PA 构件定义
   - **同步设计原则**：修改 SysReq 详细定义时，同步检查 PA 构件是否能承接修改后的需求；反之亦然
   ```
   ~ PA-XXX: [产品构件名称]（3级·架构末级节点）
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对实现方案的修改]（技术栈、接口、关键模块等）
   ├─ 承接的系统需求：[SysReq-XXX]
   ├─ 符合性分析：[修改后如何满足所承接的系统需求]
   └─ 实现细节：[更新后的技术方案]
   ```

4. **文档结构同步更新**
   - 系统需求文档：
     - **架构定义文档**：更新 5 级节点的承接关系和符合性分析
     - **详细定义文档**：更新 6-9 级分解链中受影响的条目
   - 产品架构文档：
     - **架构定义文档**：更新映射关系表中对应 SysReq 的符合性分析
     - **详细定义文档**：更新构件的实现细节

5. **快速验证**
   - 验证 SysReq↔PA 映射关系是否正确
   - 验证是否有遗漏的影响
   - 验证功能是否正确实现

6. **发布**
   - 更新系统需求和产品架构文档的版本号
   - 发布修复

**质量检查**：
- [ ] 1:1 分配约束满足（每条9级场景活动分配到唯一PA构件）
- [ ] SysReq↔PA 映射关系一致
- [ ] 是否有遗漏的影响
- [ ] 是否需要更新上层设计（如影响BA则升级为模式3）
- [ ] 两个文档的架构定义和详细定义同步更新

**输出**：
- 修改后的系统需求文档（架构定义 + 详细定义）
- 修改后的产品架构文档（架构定义 + 详细定义）
- 验证报告

### 第4步：完整设计（模式3，2-3天）

**目标**：完整设计和实现影响多层的问题

**涉及规范**：依据影响层级选择对应的层规范

**具体操作**：

按照[敏捷式系统设计指南](03-agile-design-guide.md)的 8 步流程加速执行。以下提供各层的快速条目模板和关键对齐要点：

1. **分析活动：快速影响分析**（1小时）
   - **需求说明符合性分析**：问题跨层影响到哪些架构末级节点？逐层追溯完整性
   - **需求性能符合性分析**：跨层的性能指标传递是否一致；**指标追溯**（修改涉及的各层性能指标可逐层追溯一致性）

2. **快速设计**（1天）
   - 从影响点开始，逐层设计
   - 使用两阶段工作流程（结构骨架 + 具体设计）
   - 遵循**同步设计原则**：在修改上层详细定义的同时，同步调整下层方案的架构定义和映射关系

   **相关方需求（SR）快速模板**：
   ```
   ~ SR-F-XXX: [修改的相关方需求名称]（架构末级节点）
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对需求定义或性能指标的修改]
   ├─ 承接的原始需求末级：[保持或更新]
   ├─ 符合性分析：[修改后如何满足所承接的需求]
   └─ 设计指导：[对业务架构的指导]

   ~ SR-NFR-XXX: [修改的相关方非功能需求名称]（架构末级节点）
   ├─ 原问题：[问题描述]
   ├─ 需求类型：[性能/安全/可用性等]
   ├─ 需求类别：[质量特性/环境适应性/可实现性]
   ├─ 修改内容：[对需求定义或量化指标的修改]
   ├─ 符合性分析：[修改后如何满足所承接的需求]
   └─ 量化指标：[更新后的指标]
   ```
   > SR 文档同步更新**架构定义文档**（树形结构 + 映射关系表）和**详细定义文档**（对架构末级节点的分解）。

   **业务架构（BA）快速模板**：
   ```
   ~ BA-XXX: [修改的业务组件名称]（5级·架构末级节点）
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对IPO定义的修改]
   ├─ Input/Process/Output：[更新后的I/P/O]
   ├─ 承接的SR详细定义末级：[保持或更新]
   ├─ 符合性分析：[修改后如何满足所承接的需求]
   └─ 设计指导：[对系统需求的指导]
   ```
   > BA 的修改建议采用**三阶段工作流程**（详见[业务架构规范 §4.1](../01-aos-system-design-standards/04-aos-business-architecture-specification.md#41-三阶段工作流程)）：先快速评估创建占位符和骨架映射，再填充/更新IPO内容并去重，最后验证和冻结。
   > 
   > **BA 仅有架构定义（IPO清单），无详细定义**：BA 的架构末级节点（IPO）自身承担"需求"角色，直接映射到 SysReq。修改仅涉及 IPO 定义本身和映射关系。

   **系统需求（SysReq）快速模板**：
   ```
   ~ SysReq-F-XXX: [修改的系统功能需求]（5级·架构末级节点）
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对需求定义或性能指标的修改]
   ├─ 承接的BA：[保持或更新]
   ├─ 符合性分析：[修改后如何满足所承接的需求]
   ├─ 设计指导：[对产品架构的指导]
   └─ 性能指标：[更新后的指标]

   ~ SysReq-NFR-XXX: [修改的系统非功能需求]（5级·架构末级节点）
   ├─ 原问题：[问题描述]
   ├─ 类型：[所属类别，如：性能/安全/可测试性]
   ├─ 修改内容：[对量化指标或约束条件的修改]
   ├─ 量化指标：[更新后的指标]
   ├─ 验证方法：[如何验证该指标]
   └─ 约束条件：[更新后的约束]
   ```
   > SysReq 功能分支采用 **0-9 级层级结构**（5 级为架构末级，9 级为详细定义末级）；非功能分支采用 **0-6 级平行架构**。

   **产品架构（PA）快速模板**：
   ```
   ~ PA-XXX: [修改的产品构件名称]（3级·架构末级节点）
   ├─ 原问题：[问题描述]
   ├─ 修改内容：[对实现方案的修改]（文档格式、内容结构、版本策略、质量指标等）
   ├─ 承接的系统需求：[SysReq-XXX]
   ├─ 符合性分析：[修改后如何满足所承接的系统需求]
   └─ 实现细节：[更新后的技术方案或文档定义]
   ```

   > **标记说明**：`~` 前缀表示修改现有条目，`+` 前缀表示新增条目。完整模板格式请参考敏捷式系统设计指南。

3. **文档结构同步更新**（贯穿各层）
   - 每层文档同步维护**架构定义文档**（树形结构 + 映射关系表）和**详细定义文档**（对架构末级节点的分解）
   - 更新顺序按影响层级从高到低（SR→BA→SysReq→PA），确保上层修改驱动下层调整

4. **快速验证**（半天）
   - 验证 1:1 分配约束（每条详细定义末级分配到唯一架构末级节点）
   - 验证 N:1 承接合理性（每个架构末级节点承接 2-5 条，1条需考虑合并，6+条需拆分）
   - 验证双向追溯完整性（正向反向均可追溯）
   - **可实现性评估**：修改后的方案能否满足所有性能/质量指标要求
   - 验证符合性分析完整率100%
   - 验证同步设计一致性（每层修改后下层映射同步更新）

5. **发布**（半天）
   - 更新所有相关文档的版本号
   - 发布修复

**质量检查**：
- [ ] 影响范围涉及的所有层级均已修改
- [ ] 1:1 分配约束满足率100%
- [ ] N:1 承接数量合理
- [ ] 每层文档的架构定义和详细定义同步更新
- [ ] 符合性分析完整率100%
- [ ] 可实现性评估通过（修改后的方案可满足性能/质量指标要求）
- [ ] 同步设计一致性验证通过（上层修改驱动了下层调整）

**输出**：
- 修改后的系统设计文档集
- 验证报告

## 🎯 问题类型处理指南

### Bug 修复

**快速诊断**：
1. 问题现象是什么？
2. 在哪个组件中出现？
3. 根本原因是什么？

**处理流程**：
- 仅 UI bug → 模式1
- 系统功能 bug → 模式2
- 业务流程 bug → 模式3

**验证方法**：
- 问题是否完全解决
- 是否有副作用
- 是否影响其他功能

### 紧急需求

**快速评估**：
1. 需求的优先级是什么？
2. 需要修改哪些层级？
3. 预计工作量是多少？

**处理流程**：
- 小功能需求 → 模式2
- 大功能需求 → 模式3

**验证方法**：
- 功能是否正确实现
- 是否满足需求
- 是否有遗漏

### 性能优化

**快速诊断**：
1. 性能瓶颈在哪里？
2. 是否需要修改架构？
3. 优化的收益是多少？

**处理流程**：
- 参数调整 → 模式1
- 算法优化 → 模式2
- 架构优化 → 模式3

**验证方法**：
- 性能是否改善
- 是否有副作用
- 是否满足目标

## 🔗 跨模式一致性检查

以下检查项适用于所有三种模式，确保 DevOps 快速修改不破坏系统设计的一致性：

- [ ] **术语一致性**：全文使用统一定义（架构末级节点/详细定义末级节点）
- [ ] **映射流向一致**：分配方向统一向下（↓），承接方向统一向上（↑）
- [ ] **符合性分析完整**：每次修改后的方案节点都更新了符合性分析
- [ ] **1:1 分配约束**：修改不破坏现有映射的 1:1 约束
- [ ] **N:1 承接合理性**：每个架构末级节点承接 2-5 条为合理（1条需考虑合并，6+条需拆分）
- [ ] **同步设计执行**：上层详细定义修改后，下层架构已同步调整
- [ ] **映射覆盖率**：修改后无遗漏映射
- [ ] **超链接有效**：所有映射关系保留了双向导航的可追溯性
- [ ] **性能回退检查**：修改不引入性能退化
- [ ] **版本更新**：所有修改过的文档版本号已更新

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 快速修复引入新 bug | 问题扩大 | 充分的验证和回归测试 |
| 遗漏影响范围 | 设计不一致 | 使用决策树逐层评估 |
| 文档不更新 | 设计文档过时 | 强制每次发布前更新文档 + 版本号 |
| 模式选择错误 | 工作量增加 | 第1步充分评估，不确定时选择更保守的模式 |
| 上层修改后下层映射未同步 | 追溯链路断裂 | 严格执行同步设计原则 |

## 🎯 与其他场景的关系

### vs 敏捷式系统设计
- **相似**：都是在已有设计基础上的增量设计
- **差异**：DevOps 更强调**快速响应（小时级）** 和**最小化变更**
- **关系**：DevOps 是敏捷式在紧急场景下的加速变体。当 DevOps 模式3需要跨多层修改时，参考敏捷式指南的完整流程

### vs 瀑布式系统设计
- **相似**：都遵循系统设计准则的三条核心规则
- **差异**：DevOps 通过模式选择避免不必要的完整设计流程
- **关系**：DevOps 的完整设计（模式3）可视为瀑布式的简化版本

### vs 逆向工程
- **相似**：都需要理解现有设计
- **差异**：DevOps 是在理解基础上的快速修改，逆向工程是从代码重建文档
- **关系**：DevOps 假设已有完整的设计文档；如果文档缺失，先执行逆向工程再执行 DevOps

## 📌 关键原则

1. **快速响应**
   - 优先级高的问题要快速处理
   - 不要过度设计

2. **最小化变更**
   - 只修改必要的部分
   - 避免不必要的改动

3. **保持一致性**
   - 遵循系统设计准则的三条核心规则
   - 每层文档的架构定义和详细定义同步更新

4. **充分验证**
   - 验证问题是否解决
   - 验证是否有副作用
   - 验证映射关系和符合性分析完整

5. **双向追溯**
   - 每次修改后更新映射关系（1:1 分配约束 + N:1 承接支持）
   - 维护从原始需求到产品架构的完整追溯链路
   - 确保修改层级的符合性分析反映在相邻层级的映射中
