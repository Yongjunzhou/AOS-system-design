## 📌 指南范围

本指南针对**系统设计逆向工程**——从已有的产品架构（PA）及架构各节点（构件）的需求定义出发，沿五层结构反向追溯，生成完整的系统需求（SysReq）、业务架构（BA）、相关方需求（SR）乃至原始需求（OR）：

```
PA（产品架构 + 构件需求定义）→ SysReq → BA → SR → OR
```

**另一种逆向工程：组件开发逆向工程**（属于阶段2 构件开发）
从已完成组件的代码或图纸出发，逆向生成组件的详细设计方案、概要设计方案、组件需求定义。本指南不涉及此类型。

### 依据B：三层工作层面说明

本指南属于**第2层工作（交付物构建）**——产品L（系统设计流水线）的**设计指南构件**，对应的产品架构节点为 `02-aos-system-design-guidelines/02-reverse-engineering/`。

三层工作层面：
- **第1层（自指设计）**：产品L用方法论设计自身，产出产品L的产品数据
- **第2层（交付物构建）**：本指南属于此层——产品L的"设计指南"构件，由团队L开发、维护和优化，作为工具交付给团队M使用
- **第3层（使用工具设计产品M）**：团队M使用第2层工具设计运营体系，产出产品M的产品数据

产品L的架构节点对应四个目录——它们是产品L交付给AOS开发者使用的工具：

| 产品L的构件目录 | 用途 |
|----------------|------|
| `01-aos-system-design-standards/` | 设计准则（规范与标准） |
| `02-aos-system-design-guidelines/` | 设计指南（操作方法） |
| `03-aos-system-design-ai-support/` | AI 辅助文档（自动化支持） |
| `04-aos-system-design-tasks/` | 任务定义（操作规程） |

`10-aos-system-product-data/` 是团队M使用产品L的工具后产出的**AOS设计内容**，是产品L的**输出物**，而非产品L自身的构件。

### 逆向工程在产品L上的自应用

上述关系揭示了一个重要事实：产品M系统设计下的 01/02/03/04 目录中已存在的准则、指南、AI辅助文档、任务定义，就是产品L的产品架构（PA）节点。因此，本指南描述的逆向工程方法同样适用于**产品L自身**——以产品L的 PA 为起点，沿 `PA → SysReq → BA → SR → OR` 反向推导，生成产品L自身完整的系统设计文档链。

## 📌 场景描述

### 适用条件

#### 核心条件（启动逆向推导的前提）
- ✅ 产品架构（PA）及架构各节点的需求定义已存在（即使质量不足或不完整）
- ✅ 上游系统设计文档（SysReq/BA/SR/OR）缺失或不符合设计准则
- ✅ 缺少从需求到架构的完整追溯链路
- ✅ 需要生成符合准则的系统设计文档

#### 前置条件（仅在需要重建PA文档时需要）
- ✅ 有完整的产物目录或代码实现（当PA文档不存在或质量严重不足时）

### 不适用条件

- ❌ PA 文档极度缺失且无可用的产物目录或代码用于重建PA
- ❌ 原开发人员无法参与
- ❌ 需要快速完成（时间紧张）

### 参与角色

本指南涉及三个核心参与者，在逆向工程流程中各司其职：

| 角色 | 职责 | 参与环节 |
|------|------|---------|
| **AOS设计人员** | 分析PA及现有产物，推导系统设计，审核方案 | 所有步骤 |
| **AI** | 按指令执行 PA 分析、需求反向推导、映射关系建立等自动化任务 | 核心逆向步骤的自动生成，前置准备中的产物分析 |
| **需求提出者** | 提供系统背景、业务知识和确认 | 文档审核与确认 |

## 🎯 核心目标

通过逆向工程，从 PA（产品架构）及构件需求定义出发，沿五层结构反向追溯，生成完整的、符合准则的上游系统设计文档

```
PA（产品架构 + 构件需求定义） → SysReq（系统需求） → BA（业务架构） → SR（相关方需求） → OR（原始需求）
                     第5层                        第4层                  第3层                 第2层              第1层
```

## 📊 关键概念

### 逆向工程的流向特点

与正向（瀑布式）设计不同，逆向工程的推导方向是 **自底向上**（第5层→第1层）。这意味着：

- 每步的"分析活动"从"检查完整性"变为"验证推导一致性"——即衍生出的上层需求是否与下层实现一致
- **同步设计原则**在逆向中表现为：在推导上层时，需同步检查下层架构是否能合理承接
- **详细定义的双重身份**（既是本层细化又是下层需求）在逆向中帮助判断：从下层方案能否合理反推出上层需求

### 架构定义与详细定义在逆向中的关注点

每份方案文档由两个子文档构成（系统设计准则 §1.5），在逆向中需注意：

| 子文档 | 正向设计 | 逆向工程 |
|-------|---------|---------|
| **架构定义文档** | 先定义架构再分解 | 从产物目录/PA定义推导出架构树 |
| **详细定义文档** | 对架构末级节点分解 | 从PA定义/制品特征反推出分解条目 |

### 分析活动的逆向适配

系统设计准则要求每层在架构定义完成后执行两类分析活动。在逆向中适配为：

- **需求说明符合性分析**：从下层反推上层的需求说明，验证"下层实现是否完整对应于上层需求"
- **需求性能符合性分析**：从PA的实现特征反向推导应满足的性能指标要求

### 两阶段工作流程在逆向中的应用

由于逆向是自底向上推导，两阶段工作流程（§1.5.3）表现为：

- **第一阶段**：先快速从PA定义提取各层结构骨架（节点名称 + 映射关系），不做深度内容
- **第二阶段**：在各层骨架建立后，再并行填充详细内容
- 这与正向的区别在于：正向是先有需求再设计方案，逆向是先有实现再追溯需求

### 编号稳定性原则

在逆向工程中，各层节点的编号一经确定并冻结后，**应保持稳定**，不因后续修订而改变。这是因为：
- **已建立的映射关系依赖编号**：各层之间的映射关系基于编号建立，修改编号将导致追溯链路断裂
- **生成的文档是后续迭代的基线**：逆向产出的完整文档集将作为后续正向或敏捷设计的基线，基线编号应稳定
- **1:1 约束和 N:1 承接的引用基础**：所有分配关系和承接关系均以编号为锚点

编号变更仅限于：经评审确认编号冲突或架构调整时同步更新所有相关映射。

### 组织资产优先原则

在逆向工程的自底向上推导中，资产优先原则主要体现在反向聚合时的复用检查：

```
第1优先 ── 复用（Reuse）—— 反向聚合时检查上层是否已有可承接的节点
第2优先 ── 改进（Improve）—— 现有节点扩展后承接
第3优先 ── 新增（Add）—— 创建新节点
特殊情况 ── 废弃（Deprecate）—— 标注而非物理删除
```

**在逆向工程中的应用**：
- 第4步的 IPO 去重（复用→改进→新增）直接体现该原则：多个 SysReq 对应相同业务操作时，优先复用已识别的 IPO
- 合并建议机制：自底向上推导中发现重复节点时，标记合并建议而非立即删除
- 详见[AI决策框架中"组织资产优先"章节](../03-aos-system-design-ai-support/02-ai-decision-framework-aos.md#设计原则组织资产优先)

### 阶段一：前置准备（当PA文档不存在或质量不足时执行）

---

### 第1步：现有产物梳理与架构分析

**目标**：梳理现有产物（含PA文档、产品数据、已有技术文档及代码），理解产品全貌和架构现状

**涉及规范**：[产品架构规范 §2 架构定义规范](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#二架构定义规范)

**具体操作**：

1. **现有产物目录梳理**
   - 收集所有现有设计文档（PA、产品数据等）
   - 收集已有的系统设计文档（SysReq/BA/SR/OR，如有）
   - 识别文档覆盖盲区（哪些层有文档、哪些层缺失）
   - 对现有文档做质量评估（是否符合准则）

2. **产物类型与层级分类**
   - 按五层结构对现有产物分类
   - 识别各层架构定义的完整度
   - 分析各层详细定义的覆盖范围
   - 标注缺失或质量不足的层级和节点

3. **功能清单提取（辅助分析）**
   - 从现有产物和代码（如有）中提取已实现的功能
   - 按4大分类（前端/后端/数据/基础设施）进行归类
   - 记录每项功能涉及的PA构件

4. **分析活动：初版架构评估**
   - **需求说明符合性分析**：现有产物是否构成完整的五层架构？是否有明显缺失的层级或节点？
   - **需求性能符合性分析**：从现有产物的性能特征反推产品应满足的性能指标范围；**指标追溯**（每个识别出的功能点或性能特征可关联到产品架构的特定构件）

**输出**：
- 产物总览清单（含覆盖盲区标记）
- 功能模块清单（辅助参考）
- 关键技术决策记录

> **分析要点**：产物分析中识别的每个功能模块对应一个或多个产品架构构件。功能模块清单中的条目将作为第2步中产品架构构件的输入，建议先对功能模块按**4大分类**（前端应用、后端服务、数据存储、基础设施）进行归类，再映射到架构末级节点。

### 第2步：产品架构文档编写

> **前置说明**：本步骤仅在 PA 文档不存在或质量严重不足时执行。若 PA 文档已存在且质量满足要求，请跳过本步骤，直接进入第3步。

**目标**：编写符合准则的产品架构文档

**涉及规范**：
- [产品架构规范 §2 架构定义规范](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#二架构定义规范)
- [产品架构规范 §3 详细定义规范](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#三详细定义规范)
- [系统设计准则 §1.5 架构定义与详细定义的分离](../01-aos-system-design-standards/01-aos-system-design-standards.md#15-架构定义与详细定义的分离)

**具体操作**：

1. **产品构件识别和定义**
   - 从产物目录和功能清单中识别产品构件
   - 按**4大分类**确定构件类型（适用于AOS为软件产品的场景：前端应用、后端服务、数据存储、基础设施）
   - 按**构件划分原则**判断粒度是否合适（以"一个团队可在1-2周内完成开发"且"可独立部署和测试"为双重标准）
   ```
   PA-001: [产品构件名称]（3级·架构末级节点）
   ├─ 定义：[构件的职责和功能描述]
   ├─ 构件类型：[前端/后端/数据/基础设施]（因产品类型而异）
   ├─ 实现细节：[制品位置、关键类/函数、技术栈]
   ├─ 性能指标：[从PA及制品反推的性能特征]
   └─ 承接的系统需求：（待第3步完成后补充）
   ```

2. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构）**
   ```
   产品架构：AOS企业运营体系（0级）
   ├─ 前端应用层（1级）
   │  └─ 绩效管理前端（2级）
   │     ├─ 考核计划管理页面（3级·构件/末级）
   │     ├─ 评分录入页面（3级·构件/末级）
   │     └─ 绩效报告页面（3级·构件/末级）
   │
   ├─ 后端服务层（1级）
   │  └─ 绩效管理后端（2级）
   │     ├─ 考核计划管理服务（3级·构件/末级）
   │     ├─ 评分计算服务（3级·构件/末级）
   │     └─ 绩效报告生成服务（3级·构件/末级）
   │
   ├─ 数据层（1级）
   │  ├─ 考核计划数据库（3级·构件/末级）
   │  └─ 评分结果数据库（3级·构件/末级）
   │
   └─ 基础设施层（1级）
      └─ API网关（3级·构件/末级）
   ```


   **第二部分：需求映射视图（预留）**
   - 映射关系表暂时预留，将在第3步系统需求反向推导完成后填充
   - 显示每个 PA 构件承接了哪些 SysReq 9 级节点
   - 软件产品的映射关系表以"构件类型"标识每个PA构件的分层归属（前端/后端/数据/基础设施），详见[产品架构规范 §2.4](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#24-架构定义的文档结构)

3. **详细定义同步编写**
   - 每个产品构件补充**详细定义**（技术栈、功能模块、API端点、性能指标、安全要求等）
   - 详细定义是架构末级节点的实现细化，也是后续追溯的依据
   ```
   PA-001 详细定义示例：
   ├─ 技术栈: Vue 3 + Element Plus + Axios
   ├─ 功能模块: 考核计划创建、评分表配置、考核对象选择
   ├─ API端点: GET/POST /api/v1/assessment-plans
   ├─ 性能指标: 页面加载 < 2s, 列表查询 < 1s
   └─ 安全要求: RBAC权限控制（管理员/HR/考核执行人）
   ```

**输出**：
- 产品架构文档（架构定义 + 详细定义）
- 待下游步骤填充的映射关系表骨架

### 阶段二：核心逆向推导（PA → OR）

---

### 第3步：系统需求反向推导

**目标**：从产品架构反向推导系统需求

**涉及规范**：
- [系统需求规范 §2 架构定义规范](../01-aos-system-design-standards/05-aos-system-requirements-specification.md#二架构定义规范)
- [系统需求规范 §3 详细定义规范](../01-aos-system-design-standards/05-aos-system-requirements-specification.md#三详细定义规范)

**具体操作**：

1. **分析活动：PA→SysReq 追溯验证**
   - **需求说明符合性分析**：每个 PA 构件实现的每个功能，能否反推出一条合理的 SysReq 9 级场景活动？是否有功能实现了但缺少对应的 SysReq？
   - **需求性能符合性分析**：从 PA 构件的实现性能反推 SysReq 的性能指标范围；**指标追溯**（每个 SysReq 节点的性能指标可追溯到 PA 构件的实现特征）

2. **系统需求识别**
   - 分析每个产品构件实现了什么功能
   - 识别功能性需求（做什么）
   - 识别非功能性需求（性能、安全、可靠性等）——采用 **0-6 级平行架构**，按三大类组织
   - 记录每个系统需求的定义和验收标准

3. **系统需求条目定义**

   **功能需求部分（0-9级结构）**：
   - 从 PA 构件反推 SysReq 9 级场景活动（第9级是最细粒度的活动步骤）
   - 多条 9 级活动按场景聚合为 8 级 → 7 级 → 6 级 → 5 级节点
   - **5 级节点（组件实例）** 是架构末级节点，承接后续第4步的业务架构 IPO
   ```
   SysReq-001: [系统功能需求名称]（5级·架构末级节点）
   ├─ 定义：[系统需求的定义]
   ├─ 类型：功能需求
   ├─ 对应的PA构件：PA-001, PA-002
   ├─ 符合性分析：[如何通过产品架构实现]
   ├─ 设计指导：[对业务架构的指导]
   └─ 性能指标：[从PA反推的性能要求]

   （详细定义末级节点——中间分解层级不编号，末级节点平铺排列）
   SysReq-001.01（9级·场景活动）
   ├─ 描述：[活动步骤说明]
   ├─ 分配到：PA-001（产品构件）
   └─ 验收标准：[验收条件]
   ```

   **非功能需求部分（0-6级平行架构）**：
   - 从 PA 构件的非功能特征反推系统非功能需求
   - 按三大类分类：质量特性 / 环境适应性 / 可实现性
   ```
   SysReq-101: [系统非功能需求名称]（5级·架构末级节点）
   ├─ 定义：[非功能需求的定义]
   ├─ 类型：[性能/安全/可用性等]
   ├─ 量化指标：[从PA及制品反推的具体数值、单位、条件]
   ├─ 符合性分析：[如何通过产品架构实现]
   └─ 约束条件：[对产品架构的约束]
   ```

4. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构）**
   - 功能分支采用 **0-9 级层级结构**（5 级为架构末级，9 级为详细定义末级）
   - 非功能分支采用 **0-6 级平行架构**（独立分类组织）

   **第二部分：需求映射视图**
   - 显示每个 SysReq 末级节点对应的 PA 构件
   - 同时更新第2步 PA 文档中的映射关系表

5. **同步设计**
   - 完成 SysReq 定义后，更新第2步 PA 文档的"承接系统需求"字段
   - 同步在下层（业务架构）建立 5 级节点占位符（为第4步做准备）

**输出**：
- 系统需求文档（架构定义 + 详细定义）
- 需求映射关系表（PA ↔ SysReq）

### 第4步：业务架构反向推导

**目标**：从系统需求反向推导业务架构

**涉及规范**：
- [业务架构规范 §2 架构定义规范](../01-aos-system-design-standards/04-aos-business-architecture-specification.md#二架构定义规范)
- [业务架构规范 §3 IPO模型](../01-aos-system-design-standards/04-aos-business-architecture-specification.md#三ipo模型)
- [业务架构规范 §4.3 去重机制](../01-aos-system-design-standards/04-aos-business-architecture-specification.md#43-去重机制)

**具体操作**：

1. **分析活动：SysReq→BA 追溯验证**
   - **需求说明符合性分析**：从 SysReq 5 级节点是否能推导出合理的业务操作过程（IPO）？是否存在 SysReq 实现了但 BA 中说不清业务语义的情况？
   - **需求性能符合性分析**：业务层面的性能指标是否覆盖了 SysReq 的性能要求；**指标追溯**（每个 BA IPO 的性能指标可追溯到 SysReq 的性能指标）

2. **业务架构识别**
   - 分析系统需求的业务含义
   - 识别业务组件和业务流程
   - 每个业务组件使用 **IPO 模型**（Input-Process-Output）三维定义

3. **IPO 去重**
   - 逆向推导时，多个 SysReq 节点可能对应相同的业务操作
   - 采用**三阶段工作流程**（详见[业务架构规范 §4.1](../01-aos-system-design-standards/04-aos-business-architecture-specification.md#41-三阶段工作流程)）：先快速评估创建占位符和骨架映射，再填充/更新IPO内容并去重，最后验证和冻结
   - **去重三级策略**：复用→改进→新增（优先复用已识别的 IPO）—— 这是**组织资产优先原则**（详见[关键概念](#组织资产优先原则)）在 BA 层的直接应用
   - 去重后的 IPO 清单才是最终的业务架构输出

4. **业务架构条目定义**
   ```
   BA-001: [业务组件名称]（5级·架构末级节点）
   ├─ 定义：[业务组件的定义]
   ├─ Input（输入）：[输入的具体形式、数据结构、来源]
   ├─ Process（处理）：[规则、算法、步骤、约束条件]
   ├─ Output（输出）：[输出的具体形式、数据结构、验收标准]
   ├─ 对应的SysReq：SysReq-001, SysReq-002
   ├─ 符合性分析：[如何通过SysReq实现该业务操作]
   └─ 设计指导：[对系统需求的指导]
   ```

   > **业务架构仅包含架构定义（IPO清单），无详细定义**：业务架构的架构末级节点（IPO）自身承担"需求"角色，直接映射到系统需求。这与 SysReq 和 PA 不同——后两者既有架构定义也有详细定义。

5. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构，0-5级）**
   ```
   0级：架构根节点
   1级：业务域
   2级：端到端项目
   3级：项目业务对象架构
   4级：工作包（WBS）
   5级：业务组件（承接层）★ 末级
   └─ BA-001: [IPO名称]
        ├─ 定义：[业务组件定义]
        └─ Input / Process / Output
   ```

   **第二部分：需求映射视图**
   - 显示每个 BA IPO 对应的 SysReq 5 级节点
   - 同时更新第3步 SysReq 文档中的"承接 BA"字段

**输出**：
- 业务架构文档（架构定义 / IPO清单 + 映射关系表）
- 需求映射关系表（SysReq ↔ BA）

### 第5步：相关方需求反向推导

**目标**：从业务架构反向推导相关方需求

**涉及规范**：
- [相关方需求规范 §2 架构定义规范](../01-aos-system-design-standards/03-aos-stakeholder-requirements-specification.md#二架构定义规范)
- [相关方需求规范 §3 详细定义规范](../01-aos-system-design-standards/03-aos-stakeholder-requirements-specification.md#三详细定义规范)

**具体操作**：

1. **分析活动：BA→SR 追溯验证**
   - **需求说明符合性分析**：从 BA IPO 能否反推出清晰的相关方需求描述？业务操作是否都有对应的相关方诉求？
   - **需求性能符合性分析**：SR 层面的性能指标是否覆盖了 BA 的性能要求；**指标追溯**（每个 SR 节点的性能指标可追溯到 BA IPO 的性能指标）

2. **相关方需求识别**
   - 分析业务架构的相关方需求
   - 功能需求（SR-F）和非功能需求（SR-NFR）**分别处理**
   - 非功能需求不经过 BA 层，直接从PA及制品反推

3. **相关方需求条目定义**

   **功能需求部分（SR-F）**：
   ```
   SR-F-001: [相关方需求名称]（架构末级节点）
   ├─ 定义：[需求的定义]
   ├─ 承接的原始需求末级：（待第6步完成后补充）
   ├─ 符合性分析：[如何通过业务架构实现]
   ├─ 设计指导：[对业务架构的指导]
   └─ 性能指标：[从BA继承的性能要求]

   （详细定义末级节点）
   SR-F-001.01: [需求的分解条目]
   ├─ 描述：[详细说明]
   ├─ 分配到：BA-001（业务架构IPO）
   └─ 验收标准：[验收条件]
   ```

   **非功能需求部分（SR-NFR）**：
   ```
   SR-NFR-001: [相关方非功能需求名称]（架构末级节点）
   ├─ 定义：[非功能需求的定义]
   ├─ 量化指标：[具体数值、单位、条件]
   ├─ 测量方法：[如何验证]
   └─ 分配到：SysReq-101（系统非功能需求）
   ```

4. **架构定义文档的两部分结构**

   **第一部分：架构设计结果（树形结构）**
   - 功能需求分类 + 非功能需求分类（明确分离）

   **第二部分：需求映射视图**
   - 显示每个 SR 架构末级节点对应的 BA IPO
   - 同时更新第4步 BA 文档中的"承接 SR"字段

5. **详细定义的双重身份**
   - SR 的详细定义末级节点既是本层架构的细化（向内），又是 BA 的需求输入（向下传递）
   - 确保每个 SR 详细定义末级都能分配到**唯一**的 BA IPO（1:1 约束）

**输出**：
- 相关方需求文档（架构定义 + 详细定义）
- 需求映射关系表（BA ↔ SR）

### 第6步：原始需求反向推导

**目标**：从相关方需求反向推导原始需求

**涉及规范**：[原始需求规范 §3 分解规范](../01-aos-system-design-standards/02-aos-original-requirements-specification.md#三分解规范)

**具体操作**：

1. **分析活动：SR→OR 追溯验证**
   - **需求说明符合性分析**：从 SR 架构末级节点能否合理反推出原始需求的表述？每条 SR 是否有真实的业务来源？
   - **需求性能符合性分析**：原始需求的性能指标是否与 SR 的量化指标一致；**指标追溯**（原始需求的指标可从 SR 的量化指标追溯）

2. **原始需求识别**
   - 分析相关方需求的来源
   - 将 SR 架构末级节点聚合为原始需求条目
   - 识别原始需求的末级需求并分配到 SR 架构末级节点

3. **原始需求条目定义**
   ```
   OR-001: [原始需求名称]
   ├─ 规范化描述：[经过规范化处理的描述]
   ├─ 末级需求分解：
   │  ├─ OR-001.01 → SR-F-001
   │  ├─ OR-001.02 → SR-F-002
   │  └─ OR-001.03 → SR-NFR-001
   ├─ 类型：[功能需求/非功能需求]
   └─ 优先级：[P0/P1/P2/P3]
   ```

4. **原始需求文档结构**
   - 原始需求条目清单（规范化后的简洁格式）
   - 末级需求分解清单
   - 末级需求 → 相关方需求的分配关系

5. **同步设计**
   - 完成 OR 定义后，更新第5步 SR 文档中的"承接原始需求末级"字段
   - 至此，所有 5 层文档的映射关系均已闭环

**输出**：
- 原始需求文档（含末级需求分解）
- 完整的映射关系表（OR → SR）

### 第7步：完整性和一致性验证

**目标**：验证逆向工程结果的完整性和一致性

**涉及规范**：
- [系统设计准则 §5 验收标准](../01-aos-system-design-standards/01-aos-system-design-standards.md#五验收标准)
- [系统设计准则 §6 检查清单](../01-aos-system-design-standards/01-aos-system-design-standards.md#六检查清单)
- [系统设计准则 §3.2 三条核心规则](../01-aos-system-design-standards/01-aos-system-design-standards.md#32-三条核心规则)

**具体操作**：

1. **分析活动：全链路合并验证**
   - **需求说明符合性分析汇总**：逐层验证映射覆盖率 100%，从 OR→SR→BA→SysReq→PA 每个节点都有明确的承接/分配关系
   - **需求性能符合性分析汇总**：性能指标从 OR 到 PA 逐层继承验证，一致性确认；**可实现性评估**（产品架构能否满足所有性能要求）；**指标追溯**（端到端性能指标可逐层追溯）

2. **1:1 分配约束验证**
   - 验证每条需求都分配到唯一的方案架构末级节点
   - 范围：OR 末级→SR、SR 详细定义末级→BA IPO、SysReq 9 级→PA
   - 发现遗漏或重复的映射

3. **N:1 承接支持验证**
   - 验证每个方案架构末级节点的承接数量合理（2-5 条/节点）
   - 发现过度分解或过度聚合的情况

4. **双向追溯验证**
   - **正向**：从原始需求末级 → 产品架构（自顶向下验证）
   - **反向**：从产品架构 → 原始需求末级（自底向上验证——这是逆向工程的核心产出）
   - 验证所有超链接的有效性

5. **符合性分析完整率验证**
   - 所有方案节点（SR/BA/SysReq/PA）是否都有符合性分析
   - 每个符合性分析是否清晰说明了如何满足所承接的需求

6. **同步设计一致性验证**
   - 每层的架构定义与详细定义之间的映射是否一致
   - 各层之间没有"推导了上层但漏了下层"的情况

**输出**：
- 验证报告（完整性和一致性分析）
- 问题清单和改进建议

## 🔗 跨步骤一致性检查

以下检查项覆盖所有步骤，确保逆向推导的整体一致性：

- [ ] **映射流向一致**：分配方向统一向下（↓），承接方向统一向上（↑）
- [ ] **术语一致性**：全文使用统一定义（架构末级节点/详细定义末级节点/1:1分配约束/N:1承接支持）
- [ ] **层级结构完整**：每层从根节点到末级节点的树形结构完整（BA 0-5级，SysReq 功能0-9级/非功能0-6级）
- [ ] **1:1约束满足率100%**：每条详细定义末级分配到唯一的架构末级节点
- [ ] **N:1承接合理性**：每个架构末级节点承接 2-5 条详细定义末级为合理（1条需考虑合并，6+条需拆分）
- [ ] **映射覆盖率100%**：逐层映射无遗漏
- [ ] **符合性分析完整率100%**：所有方案节点都有符合性分析
- [ ] **详细定义的双重身份理解**：每一步的详细定义既是本层细化又是下层需求
- [ ] **IPO去重完成**：BA 层已执行去重（复用→改进→新增），无重复 IPO
- [ ] **超链接有效**：所有映射关系建立了双向超链接导航

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 产物文档不清晰 | 难以理解设计意图 | 邀请原开发人员参与，进行产物文档审查 |
| 产物文档过时 | 反向推导的架构不准确 | 与现有代码（如有）进行对比，发现差异 |
| 需求信息遗漏 | 反向推导的需求不完整 | 进行用户访谈，补充缺失的需求 |
| 映射关系复杂 | 难以建立清晰的映射 | 严格执行1:1约束和N:1承接规则 |
| 性能指标难以逆向 | 无法确定原始指标 | 从产物的实际性能监测数据反推，标注"反推值" |

## 📝 使用建议

1. **项目启动前**
   - 收集所有产物（PA文档、产品数据、代码等）
   - 组建跨职能的分析团队
   - 邀请原开发人员和AOS设计人员参与

2. **分析过程中**
   - 严格按照 8 步流程执行
   - 每步完成架构定义后先做分析活动，再进入下一步（避免推导偏差累积）
   - 及时记录发现的问题

3. **分析完成后**
   - 进行完整性和一致性验证
   - 生成需求映射矩阵
   - 建立文档维护机制
