## 📌 场景描述

### 适用条件

- ✅ 原始需求已经完全收集和确认
- ✅ 需要从头到尾进行完整的系统设计
- ✅ 设计周期相对固定，需求变更较少
- ✅ 适合大型项目或关键系统

### 不适用条件

- ❌ 需求还在不断变化
- ❌ 已有部分设计文档需要继承
- ❌ 需要快速迭代和反馈

### 参与角色

本指南涉及三个核心参与者，在瀑布式设计流程中各司其职：

| 角色 | 职责 | 参与环节 |
|------|------|---------|
| **AOS设计人员** | 提出功能和非功能需求，分析确认需求，审核设计产出 | 所有步骤 |
| **AI** | 按指令执行规范化、分解、映射、验证等自动化任务 | 所有步骤的自动生成和检查 |
| **需求提出者** | 提供原始需求和业务背景，参与需求确认 | 第1步需求输入 |

## 📊 关键概念

### 六步法工作流程

本指南遵循系统设计准则定义的系统设计流水线开发六步法（§3.1），按功能/非功能分支并行推进：

| 步骤 | 名称 | 分支 |
|------|------|------|
| 第1步 | 原始需求分析及相关方需求架构定义 | 统一处理 |
| 第2步 | 相关方需求功能部分详细定义及业务架构定义 | 功能分支 |
| 第3步 | 系统需求功能部分架构定义 | 功能分支 |
| 第4步 | 相关方需求非功能部分详细定义及系统需求非功能部分架构定义 | 非功能分支 |
| 第5步 | 系统需求详细定义及产品架构定义 | 分支合并 |
| 第6步 | 双向追溯验证 | 全链路验证 |

> **并行策略**：功能分支（第2-3步）和非功能分支（第4步）可完全并行推进。

### 架构定义与详细定义的分离

每份方案文档由两个子文档构成（系统设计准则 §1.5）：

| 子文档 | 作用 | 在瀑布式中的关注点 |
|-------|------|------------------|
| **架构定义文档** | 以架构末级节点为基本单位，构建方案的层级结构并建立对上层需求的映射 | 从零构建完整的树形结构和映射关系 |
| **详细定义文档** | 对架构末级节点进行详细设计分解，使每条末级条目在下层方案中有唯一的承接点 | 从架构末级逐层分解至原子粒度，确保1:1约束 |

### 分析活动的贯穿执行

系统设计准则要求每层在架构定义完成后、详细定义开始前，集中执行两类分析活动。在瀑布式中，分析活动是**质量保障的关键环节**：

- **需求说明符合性分析**：验证每个架构末级节点是否充分满足所承接的上层需求，检查覆盖度、一致性和追溯完整性
- **需求性能符合性分析**：验证性能指标是否从上层正确继承和量化，检查指标是否可测量、可验证

### 详细定义的双重身份

详细定义在流水线中扮演双重角色（系统设计准则 §1.5.2）：
- **本层身份**：作为当前方案**架构定义的细化**（向内分解）
- **下层身份**：作为流水线上**后续方案的需求**（向下传递）

在瀑布式设计中，这一概念指导每步的分解粒度：分解到每条详细定义末级只需一个下层架构末级节点承接（1:1约束）。

### 同步设计与两阶段工作流程

**同步设计原则**（§1.5.3）：在开展上层详细定义的同时，同步建立下层方案的架构结构骨架。具体操作方法为两阶段工作流程：

- **第一阶段：结构骨架约定** — 快速识别上层末级节点，在下游建立结构骨架和映射关系，冻结骨架
- **第二阶段：具体设计** — 并行填充详细内容，验证完整性和一致性

### 符合性分析

每个方案节点必须说明如何满足所承接的需求（系统设计准则 §3.2）：
- 如何满足每条承接的需求
- 包含具体的验证方法
- 逻辑清晰、可验证

## 📋 完整操作流程（6步）

### 第1步：原始需求分析及相关方需求架构定义

**目标**：从原始需求产出原始需求详细定义和相关方需求架构定义

**涉及规范**：
- [原始需求规范 §4.1 完整处理流程](../01-aos-system-design-standards/02-aos-original-requirements-specification.md#41-完整处理流程)
- [相关方需求规范 §2 架构定义规范](../01-aos-system-design-standards/03-aos-stakeholder-requirements-specification.md#二架构定义规范)
- [系统设计准则 §3.1 第1步](../01-aos-system-design-standards/01-aos-system-design-standards.md#第1步原始需求分析及相关方需求架构定义)

**具体操作**：

1. **原始需求收集和初步整理**
   - 从各渠道收集原始需求，按收集阶段格式完整记录（9字段）
   - **初步整理（可选，建议执行）**：对来自多渠道的零散碎片需求进行归并整理
     - 血缘归并、角色聚合、流程串联、父子补全四种方法
     - 遵循五条规则：不过度合并、保留原始出处、性能指标不合并、矛盾碎片不合并、无法判断标记待补全
   ```
   原始需求 OR-001
   ├─ 需求标题：[一句话概括]
   ├─ 需求描述：[合并后的完整需求描述]
   ├─ 原始碎片清单：
   │  ├─ [碎片1原始表述] — [提出方] [日期]
   │  └─ [碎片2原始表述] — [提出方] [日期]
   ├─ 包含的性能指标：[如有，单独列出]
   ├─ 依赖/冲突关系：[如有，标注]
   └─ 整理说明：[归并依据]
   ```

2. **原始需求规范化**
   - 格式统一（需求描述、提出方、优先级、类型）
   - 四项检查：完整性、清晰性、可测试性、独立性
   - 模糊术语替换为精确表述（"实时"→"延迟<100ms"）
   - 功能需求和非功能需求分离

3. **原始需求分解到末级**
   - 按原子性原则逐层分解
   - **分解停止准则**：每条末级只需一个相关方需求架构末级节点承接
   ```
   原始需求 OR-001
   ├─ 末级需求分解：
   │  ├─ 末级需求 1.1 → SR-F-001（需求规范化功能）
   │  ├─ 末级需求 1.2 → SR-F-001（需求规范化功能）
   │  └─ 末级需求 1.3 → SR-F-002（架构设计功能）
   └─ 优先级：[高/中/低]

   ✅ 每个末级只需一个SR节点承接（1:1约束满足）
   ❌ 如果一个末级需要SR-F-001+SR-F-002两个节点承接，说明还需分解
   ```

4. **分析活动：冲突和重复检测**
   - **需求说明符合性分析**：冲突检测（直接/间接矛盾）+ 重复检测（完全/部分/相似重复）
   - **需求性能符合性分析**：验证性能指标是否明确量化，检查指标是否有适用条件和验证方法
   ```
   | 需求A | 需求B | 问题类型 | 描述 | 建议 |
   | OR-001 | OR-005 | 直接冲突 | 一个允许删除，一个禁止删除 | 需确认 |
   ```

5. **相关方需求架构定义**
   - 分析末级需求的共性和关联性
   - 按功能需求（SR-F）和非功能需求（SR-NFR）分别分类
   - 定义架构末级节点 + **符合性分析** + **设计指导**

   **架构定义文档包含两个部分**：

   **第一部分：架构设计结果（树形结构）**
   ```
   相关方需求（根节点）
   ├─ 功能需求类（分类节点）
   │  └─ SR-F-001: [相关方需求名称]（架构末级节点）
   │       ├─ 定义：[需求的定义]
   │       ├─ 承接的原始需求末级：原始需求 1.1, 1.2
   │       ├─ 符合性分析：[如何满足所承接的需求]
   │       ├─ 设计指导：[对业务架构的指导]
   │       └─ 性能指标：[功能相关的性能要求]
   │
   └─ 非功能需求类（分类节点）
      └─ SR-NFR-001: [相关方非功能需求]（架构末级节点）
           ├─ 定义：[非功能需求的定义]
           ├─ 承接的原始需求末级：原始需求 N01, N02
           ├─ 符合性分析：[如何满足所承接的需求]
           └─ 量化指标：[具体的、可测量的指标值]
   ```

   **第二部分：需求映射视图（映射关系表）**
   ```
   | 原始需求末级 | 承接的SR架构末级节点 | 关联说明 | 超链接 |
   |-------------|---------------------|--------|--------|
   | 原始需求 1.1 | SR-F-001            | 考核计划管理 | [→] |
   | 原始需求 1.2 | SR-F-001            | 需求规范化 | [→] |
   | 原始需求 1.3 | SR-F-002            | 架构设计 | [→] |
   | 原始需求 N01 | SR-NFR-001          | 性能需求 | [→] |
   ```

**详细定义的双重身份说明**：
- 本步产出的原始需求详细定义，既是原始需求的细化和分解（本层身份），又是相关方需求架构定义的输入（下层身份）
- 设计准则将此称为"详细定义的双重身份"——详见 §1.5.2

**质量检查**：
- [ ] 所有原始需求已完成初步整理（碎片归并、补全语境）
- [ ] 每条原始需求都通过四项检查（完整性、清晰性、可测试性、独立性）
- [ ] 功能需求和非功能需求已明确分离
- [ ] 每条原始需求都分解到末级需求
- [ ] 末级需求是原子性的（1:1约束满足）
- [ ] 冲突和重复检测已完成，结果已记录
- [ ] 所有原始需求末级都已映射到相关方需求架构（覆盖率100%）
- [ ] 架构定义包含两部分（树形结构 + 映射关系表）
- [ ] 每个架构末级节点包含符合性分析和设计指导

**输出**：
- 原始需求文档（含末级需求分解）
- 相关方需求架构定义文档（树形结构 + 映射关系表）
- 冲突和重复检测报告

### 第2步：相关方需求功能部分详细定义及业务架构定义

**目标**：沿功能分支推进，产出相关方需求功能部分详细定义和业务架构定义

**涉及规范**：
- [相关方需求规范 §3 详细定义规范](../01-aos-system-design-standards/03-aos-stakeholder-requirements-specification.md#三详细定义规范)
- [业务架构规范 §2 架构定义规范](../01-aos-system-design-standards/04-aos-business-architecture-specification.md#二架构定义规范)
- [系统设计准则 §3.1 第2步](../01-aos-system-design-standards/01-aos-system-design-standards.md#第2步相关方需求功能部分详细定义及业务架构定义)
- [系统设计准则 §1.5.3 同步设计](../01-aos-system-design-standards/01-aos-system-design-standards.md#153-详细定义与下层架构定义的同步设计)

**具体操作**：

1. **分析活动：架构定义验证**
   - **需求说明符合性分析**：检查每个SR-F架构末级节点是否充分满足所承接的原始需求末级，验证功能/非功能分离正确性
   - **需求性能符合性分析**：验证性能指标是否从原始需求正确继承和量化，检查指标是否可测量、可验证
   - 分析活动在架构定义完成后、详细定义开始前集中执行

2. **相关方需求功能部分详细定义**
   - 对SR-F架构末级节点进行详细设计分解
   - **分解标准**：直到每条详细定义末级只需映射到业务架构的一个IPO（1:1约束）
   ```
   架构末级节点：SR-F-001（相关方需求规范化功能）

   详细定义末级节点：
   SR-F-001.1（考核计划功能）
   ├─ 描述：用户能够输入需求信息并创建新需求记录
   ├─ 分解依据：需求处理流程的第一个独立环节
   ├─ 分配目标：BA-考核计划流程 [1:1映射]
   ├─ 验收标准：考核计划创建成功率 > 99%
   └─ 符合性：直接满足SR-F-001中对考核计划管理的需求
   ```

   **详细定义的双重身份**：
   - **本层身份**：对SR架构末级节点的细化分解（向内）
   - **下层身份**：对业务架构的需求输入（向下传递）
   - 这意味着每条详细定义末级既是SR的"细化"，又是BA的"需求"

3. **业务架构IPO定义**
   - 采用**三阶段工作流程**（详见[业务架构规范 §4.1](../01-aos-system-design-standards/04-aos-business-architecture-specification.md#41-三阶段工作流程)）：第一阶段快速创建占位符和骨架映射，第二阶段详细填充IPO内容并去重，第三阶段验证和冻结
   - 根据详细定义结果识别IPO
   - 为每个IPO定义**I（输入）/ P（处理）/ O（输出）**
   - IPO去重：**复用 → 改进 → 新增**（优先复用已有IPO）
   ```
   BA-001: [IPO名称/业务组件名称]（5级·架构末级节点）
   ├─ 定义：[业务组件的定义]
   ├─ Input（输入）：[具体形式、数据结构、来源]
   ├─ Process（处理）：[规则、算法、步骤、约束条件]
   ├─ Output（输出）：[具体形式、数据结构、验收标准]
   ├─ 承接的相关方需求详细定义末级：
   │  ├─ SR-F-001.1, SR-F-001.2
   ├─ 符合性分析：[如何满足所承接的需求]
   ├─ 性能指标：[继承自相关方需求]
   └─ 设计指导：[对系统需求的指导]
   ```

4. **同步建立下层架构骨架**
   - 在开展业务架构详细填充的同时，**同步**在下层（系统需求）建立功能部分架构末级节点的结构骨架
   - 边分解边映射：每完成一个IPO，立即检查是否能建立从BA到SysReq的映射
   ```
   同步设计的实际操作：
   1. 在创建业务组件占位符时，同步创建系统需求5级节点占位符
   2. 在填充IPO详细内容时，同步建立BA → SysReq的映射关系
   3. 冻结映射关系后，并行进行内容深填
   ```

**质量检查**：
- [ ] 分析活动已完成（符合性分析完整率100%）
- [ ] 1:1分配约束满足率100%（每条详细定义末级只分配到一个IPO）
- [ ] 每个IPO都有清晰的I/P/O定义
- [ ] IPO去重已执行（复用→改进→新增）
- [ ] 每个业务组件包含符合性分析和设计指导
- [ ] 同步建立的系统需求架构骨架与IPO清单匹配
- [ ] 详细定义标注了分配目标（向下追溯）

**输出**：
- 相关方需求功能部分详细定义文档
- 业务架构定义文档（IPO清单 + 映射关系表）
- 系统需求功能部分架构骨架（同步建立）

### 第3步：系统需求功能部分架构定义

**目标**：将业务架构IPO去重并分类组织，形成系统需求功能部分的架构定义

**涉及规范**：
- [系统需求规范 §2.1 功能需求架构（0-9级）](../01-aos-system-design-standards/05-aos-system-requirements-specification.md#21-功能需求架构0-9级)
- [系统设计准则 §3.1 第3步](../01-aos-system-design-standards/01-aos-system-design-standards.md#第3步系统需求功能部分架构定义)

**具体操作**：

1. **IPO去重（三级策略）**
   - **第1步**：首先检查现有IPO中是否有相同或相似的
   - **第2步**：复用优先（有完全匹配的IPO直接复用，记录映射来源）
   - **第3步**：改进次之（现有IPO改进后可满足需求则改进后复用）
   - **第4步**：新增最后（确实需要新的IPO才新增）
   ```
   ✅ 正确去重：
     BA-001（考核计划管理）→ 包含"制定考核计划"IPO
     BA-002（评分管理）→ 包含"录入评分数据"IPO
     → 合并为SysReq-F-001（考核计划管理），同时承接BA-001和BA-002
     → 系统需求中"考核计划管理"只需实现一次

   ❌ 错误做法：
     直接将BA IPO清单复制为SysReq清单，未进行去重
     → BA-001和BA-002的"考核管理"在SysReq中出现两次
     → 导致产品架构中出现重复构件
   ```

2. **分析活动**
   - **需求说明符合性分析**：覆盖度检查（所有BA IPO是否都有对应的SysReq 5级节点承接），一致性校验（去重后语义一致），追溯验证（每个SysReq标注承接来源）
   - **需求性能符合性分析**：性能继承验证（是否继承了BA的性能指标），可行性评估，**指标追溯**（每个SysReq节点的性能指标是否能追溯到BA IPO或SR-NFR）

3. **功能需求架构组织（0-9级结构）**
   - 将去重后的IPO按系统功能分类组织
   - 采用 **0-9 级层级结构**（5级为架构末级承接BA，9级为详细定义末级分配到PA）
   ```
   0级：根节点
   1级：独立信息系统
   2级：功能需求分支
   3级：业务组件类型（流程/工单/计划/指标等）
   4级：业务组件小类
   5级：组件实例 ←── 承接BA 5级（N:1，架构末级）
   6级：功能模块（Tab/页面）
   7级：系统用例（按钮/菜单）
   8级：用例场景（正常/异常/替代）
   9级：场景活动 ──→ 分配到PA末级（1:1，详细定义末级）
   ```

   **功能部分架构定义示例**：
   ```
   系统需求（0级·根）
   └─ 绩效考核系统（1级·独立信息系统）
      └─ 功能需求分支（2级·功能分支）
         └─ 流程管理（3级·业务组件类型）
            └─ 流程实例管理（4级·业务组件小类）
               └─ SysReq-F-001: 绩效考核管理（5级·架构末级）
                    ├─ 定义：提供绩效考核的全生命周期管理
                    ├─ 承接BA：BA-001, BA-002
                    ├─ 符合性分析：通过流程引擎实现BA定义的操作流程
                    ├─ 设计指导：基于流程引擎配置，需包含流程定义/实例/监控
                    └─ 性能指标：流程启动时间 < 1s，支持 500 并发实例
   ```

   **需求映射视图（映射关系表）**：
   ```
   | BA末级节点 | 承接的SysReq架构末级 | 超链接 |
   |-----------|---------------------|--------|
   | BA-001    | SysReq-F-001        | [→] |
   | BA-002    | SysReq-F-001        | [→] |
   | BA-003    | SysReq-F-002        | [→] |
   ```

**质量检查**：
- [ ] 分析活动已完成（覆盖度和追溯验证通过）
- [ ] IPO去重完成率100%（采用三级策略）
- [ ] 功能需求来自IPO去重清单（100%覆盖）
- [ ] 架构层级符合0-9级结构
- [ ] 每个5级节点包含符合性分析和设计指导
- [ ] 映射关系表已建立（BA→SysReq）

**输出**：
- 系统需求功能部分架构定义（树形结构 + 映射关系表）

### 第4步：相关方需求非功能部分详细定义及系统需求非功能部分架构定义

**目标**：沿非功能分支独立推进，产出系统需求非功能部分架构定义

**涉及规范**：
- [系统需求规范 §2.2 非功能需求架构（0-6级）](../01-aos-system-design-standards/05-aos-system-requirements-specification.md#22-非功能需求架构0-6级)
- [系统设计准则 §3.1 第4步](../01-aos-system-design-standards/01-aos-system-design-standards.md#第4步相关方需求非功能部分详细定义及系统需求非功能部分架构定义)

> **并行说明**：本步（非功能分支）与第2-3步（功能分支）**完全独立并行**。两个分支可同时推进，不受彼此影响。

**具体操作**：

1. **分析活动**
   - **需求说明符合性分析**：检查每个SR-NFR架构末级节点是否充分满足所承接的原始非功能需求末级
   - **需求性能符合性分析**：验证性能指标是否从原始需求正确继承和量化，检查指标可测量性

2. **相关方需求非功能部分详细定义**
   - 对SR-NFR架构末级节点进行详细设计分解
   - 为每条非功能需求定义**量化指标**、**测量方法**、**应用范围**
   ```
   SR-NFR-001: 系统响应时间性能需求
   需求类型：性能 | 需求类别：质量特性需求
   描述：流水线在正常负载下应快速响应用户操作
   量化指标：
     - 平均响应时间：< 200ms
     - P99响应时间：< 1000ms（在1000并发下）
   测量方法：JMeter性能测试
   应用范围：所有关键流程操作
   ```

3. **非功能需求架构设计（0-6级平行架构）**
   - 按**三大类**组织：质量特性需求、环境适应性需求、可实现性需求
   - 采用 **0-6 级平行架构**（独立于功能需求分支）
   ```
   0级：非功能需求根节点
   1级：需求类别（三大类）
   2级：需求类型（性能/安全/可测试性等）
   3级：需求子类型（响应时间/吞吐量/数据加密等）
   4级：需求细分
   5级：非功能需求承接层 ←── 承接SR-NFR（架构末级）
   6级：操作序列/验证标准
   ```

   **非功能部分架构定义示例**：
   ```
   系统非功能需求（0级·根）
   └─ 质量特性需求（1级·需求类别）
      └─ 性能需求（2级·需求类型）
         └─ 响应时间（3级·需求子类型）
            └─ 系统响应时间（4级·需求细分）
               └─ SysReq-NFR-001: 系统响应时间（5级·架构末级）
                    ├─ 定义：系统在正常负载下应快速响应
                    ├─ 承接SR-NFR：SR-NFR-001
                    ├─ 量化指标：平均 < 200ms，P99 < 500ms
                    └─ 约束条件：需使用缓存层，数据库索引优化
   ```

**质量检查**：
- [ ] 分析活动已完成（性能指标量化率100%）
- [ ] 所有非功能需求都有明确的量化指标（包含数值、单位、条件）
- [ ] 非功能需求按三大类分类组织
- [ ] 架构层级符合0-6级平行架构
- [ ] 与功能分支无冲突
- [ ] 对产品架构的约束已明确

**输出**：
- 相关方需求非功能部分详细定义
- 系统需求非功能部分架构定义（树形结构 + 映射关系表）

### 第5步：系统需求详细定义及产品架构定义

**目标**：合并功能和非功能分支，产出系统需求详细定义和产品架构定义

**涉及规范**：
- [系统需求规范 §3 详细定义规范](../01-aos-system-design-standards/05-aos-system-requirements-specification.md#三详细定义规范)
- [产品架构规范 §2 架构定义规范](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#二架构定义规范)
- [系统设计准则 §3.1 第5步](../01-aos-system-design-standards/01-aos-system-design-standards.md#第5步系统需求详细定义及产品架构定义)
- [系统设计准则 §1.5.3 同步设计](../01-aos-system-design-standards/01-aos-system-design-standards.md#153-详细定义与下层架构定义的同步设计)

**具体操作**：

1. **分析活动：架构定义验证**（在SysReq详细定义开始前集中执行）
   - **需求说明符合性分析**：覆盖度检查（所有BA IPO→SysReq 5级映射完整），一致性校验（去重语义一致），追溯验证（每个SysReq节点标注承接来源）
   - **需求性能符合性分析**：性能继承验证（SysReq继承了BA/NFR的指标），可行性评估，**指标追溯**（每个SysReq节点的性能指标可追溯到BA IPO或SR-NFR）

2. **系统需求功能部分详细定义（5级→9级分解）**
   - 对SysReq-F 5级架构末级节点进行逐级分解至9级场景活动
   - **分解链**：5级（组件实例）→ 6级（功能模块）→ 7级（系统用例）→ 8级（用例场景）→ 9级（场景活动）
   - **分解停止准则**：每条9级场景活动只需分配到产品架构的一个构件（1:1分配约束）

   ```
   5级：SysReq-F-001（绩效管理）
    ↓ 按功能模块（Tab）分解
   6级：SysReq-F-001.1（考核计划列表）
    ↓ 按系统用例（按钮/菜单）分解
   7级：SysReq-F-001.1.1（制定考核计划）
    ↓ 按业务场景分解
   8级：SysReq-F-001.1.1.1（正常制定计划）
    ↓ 按活动步骤分解
   9级：SysReq-F-001.1.1.1.1（填写考核信息）→ PA-001（考核计划录入页面）
   9级：SysReq-F-001.1.1.1.2（提交考核计划）→ PA-002（考核计划API）
   → 每个9级活动只需一个PA构件承接 ✅
   ```

3. **产品架构定义**
   - 根据SysReq 9级场景活动，识别候选构件
   - 产品架构的**4大分类**：前端应用、后端服务、数据存储、基础设施
   - **构件划分原则**：以"一个团队可在1-2周内完成开发"且"可独立部署和测试"为双重标准



   **示例：业务软件产品**（如产品A·AOS企业运营体系）：
   ```
   产品架构：AOS企业运营体系（0级）
   ├─ 前端应用层（1级）
   │  └─ 绩效管理前端（2级）
   │     ├─ 考核计划管理页面（3级·构件/末级）
   │     ├─ 评分录入页面（3级·构件/末级）
   │     └─ 绩效报告页面（3级·构件/末级）
   │
   ├─ 后端服务层（1级）
   │  └─ 绩效管理后端（2级）
   │     ├─ 考核计划管理服务（3级·构件/末级）
   │     ├─ 评分计算服务（3级·构件/末级）
   │     └─ 绩效报告生成服务（3级·构件/末级）
   │
   ├─ 数据层（1级）
   │  ├─ 考核计划数据库（3级·构件/末级）
   │  └─ 评分结果数据库（3级·构件/末级）
   │
   └─ 基础设施层（1级）
      └─ API网关（3级·构件/末级）
   ```
   > 软件产品的映射关系表以"构件类型"标识每个PA构件的分层归属（前端/后端/数据/基础设施），详见[产品架构规范 §2.4](../01-aos-system-design-standards/06-aos-product-architecture-specification.md#24-架构定义的文档结构)。

4. **非功能约束验证**
   - 以系统需求非功能需求约束产品架构的符合性
   - 每个PA构件标注受哪些NFR约束，性能指标不低于SysReq要求
   - NFR冲突时记录权衡决策（如性能vs安全）

   **产品构件的详细定义应包含（AOS产品示例）**：
   ```
   PA-001: 考核计划录入页面
   ├─ 页面类型: Web交互表单
   ├─ 功能模块: 考核计划创建、评分表配置、考核对象选择、时间节点设置
   ├─ 交互流程: 创建计划 → 配置评分表 → 选择考核对象 → 发布
   ├─ 前端框架: Vue 3 + Element Plus
   ├─ 性能指标: 页面加载 < 2s, 表单提交响应 < 500ms
   ├─ 可用性要求: 支持断网草稿保存
   ├─ 非功能约束承接: SysReq-NFR-003（可用性）, SysReq-NFR-004（响应时间）
   └─ 质量要求: 字段校验覆盖率100%, 录入成功率 > 99.5%
   ```

   **产品构件的详细定义应包含（AOS产品示例——详细定义）**：
   ```
   PA-002: 考核计划管理服务
   ├─ 技术栈: Go 1.21 + Gin + PostgreSQL
   ├─ 功能模块: 考核计划CRUD、评分规则配置、考核对象管理、计划发布
   ├─ API端点: GET/POST /api/v1/assessment-plans, PUT/DELETE /api/v1/assessment-plans/:id
   ├─ 数据处理: 事务保证考核计划完整性，审核流状态机管理
   ├─ 性能指标: 查询 < 500ms (P99), 写入 < 200ms (P99)
   ├─ 安全要求: RBAC权限控制（管理员/HR/考核执行人）
   ├─ 非功能约束承接: SysReq-NFR-001（响应时间）, SysReq-NFR-005（权限控制）
   └─ 质量要求: 接口可用性 > 99.9%, 数据一致性100%
   ```

**质量检查**：
- [ ] 分析活动已完成（架构定义验证通过）
- [ ] 所有SysReq 5级节点已分解到9级场景活动
- [ ] 1:1分配约束满足率100%（每条9级活动分配到唯一PA构件）
- [ ] 前后端构件划分清晰，构件粒度合理
- [ ] 每个PA构件包含符合性分析和实现细节
- [ ] 每个PA构件标注了受哪些NFR约束
- [ ] NFR已评估影响，冲突有权衡决策记录
- [ ] 映射关系表已建立（SysReq 9级→PA）
- [ ] 非功能部分对产品架构的约束已体现

**输出**：
- 系统需求详细定义文档（含5级→9级分解）
- 产品架构定义文档（架构定义 + 详细定义 + 映射关系表）
- NFR符合性检查报告

### 第6步：双向追溯验证

**目标**：验证完整的追溯链路，产出追溯矩阵和验证报告

**涉及规范**：
- [系统设计准则 §5 验收标准](../01-aos-system-design-standards/01-aos-system-design-standards.md#五验收标准)
- [系统设计准则 §6 检查清单](../01-aos-system-design-standards/01-aos-system-design-standards.md#六检查清单)
- [系统设计准则 §3.2 三条核心规则](../01-aos-system-design-standards/01-aos-system-design-standards.md#32-三条核心规则)

**具体操作**：

1. **分析活动：合并验证**

   将第1~5步中各层的分析活动结果合并为完整的验证报告：

   **需求说明符合性分析汇总**：
   - 覆盖度检查：逐层验证映射覆盖率100%
   - 一致性校验：跨层语义一致，无矛盾
   - 追溯验证：正向反向均可追溯

   **需求性能符合性分析汇总**：
   - 性能继承验证：每层的性能指标是否从上层正确继承
   - 可实现性评估：产品架构能否满足所有性能要求
   - 权衡决策追溯：NFR冲突决策记录是否完整

2. **1:1分配约束验证**
   - 逐条验证**每条详细定义末级节点**是否分配到**唯一**的方案架构末级节点
   - 范围：原始需求末级→SR、SR详细定义末级→BA IPO、SysReq 9级→PA
   ```
   ✅ 通过：每个详细定义末级节点分配到一个架构末级节点
   ❌ 未通过：某个条目分配到多个节点 → 需继续分解或调整架构
   ```

3. **N:1承接支持验证**
   - 验证每个方案架构末级节点的承接数量合理（平均2-3条，最多5条）
   - 发现过度分解（承接1条，考虑合并）或过度聚合（承接6+条，需拆分）的情况

4. **双向追溯验证**
   - **正向**：从原始需求末级 → 相关方需求 → 业务架构 → 系统需求 → 产品架构
   - **反向**：从产品架构 → 系统需求 → 业务架构 → 相关方需求 → 原始需求末级
   - 验证每个节点是否标注了承接来源（向上）和分配目标（向下）

5. **符合性分析完整率验证**
   - 所有方案节点（SR/BA/SysReq/PA）是否都有符合性分析
   - 每个符合性分析是否清晰说明了如何满足所承接的需求

6. **文档完善和发布**
   - 处理验证中发现的问题
   - 更新版本号和日期
   - 生成需求追溯矩阵和验证报告

**质量检查**：
- [ ] 1:1约束满足率100%
- [ ] N:1承接数量合理（2-5条/节点，1条需考虑合并，6+条需拆分）
- [ ] 映射覆盖率100%（无遗漏、无冗余）
- [ ] 符合性分析完整率100%（所有方案节点都有符合性分析）
- [ ] 正向/反向追溯100%完整
- [ ] 性能指标跨层一致（产品架构不低于SysReq要求）
- [ ] 超链接有效率100%

**输出**：
- 需求追溯矩阵（端到端）
- 验证报告（含完整性和一致性分析结果）

## 🔗 跨步骤一致性检查

以下检查项覆盖所有步骤，确保整体设计的一致性：

- [ ] **术语一致性**：全文使用统一定义（架构末级节点/详细定义末级节点/1:1分配约束/N:1承接支持）
- [ ] **映射流向一致**：分配方向统一向下（↓），承接方向统一向上（↑）
- [ ] **层级结构完整**：每层从根节点到末级节点的树形结构完整（BA 0-5级，SysReq 功能0-9级/非功能0-6级）
- [ ] **1:1约束满足率100%**：每条详细定义末级分配到唯一的架构末级节点
- [ ] **N:1承接合理性**：每个架构末级节点承接 2-5 条详细定义末级为合理（1条需考虑合并，6+条需拆分）
- [ ] **映射覆盖率100%**：逐层映射无遗漏
- [ ] **详细定义的双重身份理解**：每一步的详细定义既是本层细化又是下层需求
- [ ] **同步设计执行**：每次开展上层详细定义时，已同步建立下层架构骨架
- [ ] **符合性分析贯穿**：每个方案节点都有符合性分析
- [ ] **超链接有效**：所有映射关系建立了双向超链接导航
- [ ] **版本号一致**：所有文档版本号正确对应

## ⚠️ 常见风险和应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 原始需求不完整或变更 | 后续设计返工 | 第1步充分执行初步整理（碎片归并、补全语境），与相关方确认后再进入详细设计 |
| 需求分解粒度过细或过粗 | 映射关系混乱 | 严格执行分解停止准则：每条末级只需一个下层方案节点承接（1:1约束） |
| 同步设计未执行 | 上下层映射断裂 | 每步完成架构定义后先做分析活动，再开始详细定义；建立下层架构骨架后再填充 |
| 性能指标未正确继承 | 方案无法满足性能要求 | 每步分析活动中执行性能符合性分析，确保指标逐层量化、可验证 |
| 功能与非功能分支不同步 | 合并时发现冲突 | 第2-3步与第4步保持沟通，第5步合并前做覆盖度检查 |

## 🚀 使用建议

### 1. 项目启动前
- 确认原始需求已完全收集
- 阅读[系统设计准则（总纲）](../01-aos-system-design-standards/01-aos-system-design-standards.md)的 §1（核心概念）和 §3（设计方法）
- 明确三个参与角色的分工（AOS设计人员、AI、需求提出者）
- 确定功能分支和非功能分支的并行团队分工

### 2. 设计过程中
- 严格按照6步流程执行
- **第2步和第4步可并行推进**（功能/非功能解耦）
- 每步开始前先阅读对应的层规范文档
- 每步完成架构定义后先做分析活动，再开始详细定义（避免返工）
- 定期进行质量检查

### 3. 设计完成后
- 进行端到端追溯验证
- 生成追溯矩阵和验证报告
- 建立文档维护机制
