## 📋 核心约束速记

### 约束1：1:1 分配约束
```
每条上层需求 → 唯一一个下层方案条目

检查方法：
✓ 需求是否分配了？
✓ 是否只分配到一个条目？
✓ 是否有多个条目都承接了这条需求？

常见问题：
❌ 一条需求分配到多个条目 → 需求分解不足
❌ 一条需求无法分配 → 方案设计不完整
```

### 约束2：N:1 承接支持
```
每个下层方案条目 ← 多条上层需求

合理范围：
- 最少：1条需求
- 最多：5条需求
- 平均：2-3条需求

检查方法：
✓ 承接的需求数量是否合理？
✓ 承接的需求之间是否有逻辑关联？
```

### 约束3：双向追溯
```
需求 ↔ 方案（通过超链接）

检查方法：
✓ 每条需求都标注了分配的方案条目？
✓ 每个方案条目都标注了承接的需求？
✓ 所有超链接都有效？
```

### 约束4：符合性分析
```
每个方案节点必须说明如何满足所承接的需求

检查方法：
✓ 每个架构末级节点都有符合性分析？
✓ 符合性分析说明了如何满足每条承接的需求？
✓ 符合性分析包含验证方法？
```

## 🏗️ 五层结构速记

```
第1层：原始需求及分析
  ↓ 分解到末级需求
第2层：相关方需求（原始需求→方案，兼具需求身份）
  ├─ 架构定义子文档：[第一部分：树形结构] + [第二部分：映射关系表]
  │   └─ 架构末级节点 = 需求映射的关键层级
  └─ 详细定义子文档（详细定义末级节点 = 分配到下层的源头）
  ↓ N:1 ↑ 详细定义的双重身份：既是本层细化，又是下层需求
第3层：业务架构（相关方需求功能部分的方案）
  └─ 架构定义子文档（IPO清单 + 映射关系表，无独立详细定义）
      ├─ IPO去重：复用→改进→新增
      └─ I（输入）/ P（处理）/ O（输出）
  ↓ N:1
第4层：系统需求（业务架构 + 相关方非功能需求的方案）
  ├─ 架构定义子文档：[第一部分：树形结构] + [第二部分：映射关系表]
  │   ├─ 功能需求分支（0-9级，5级=承接层，9级=场景活动→PA）
  │   └─ 非功能需求分支（0-6级平行架构，三大类）
  └─ 详细定义子文档（条目化需求，含符合性分析）
  ↓ N:1
第5层：产品架构（系统需求的方案）
  ├─ 架构定义子文档：[第一部分：树形结构] + [第二部分：映射关系表]
  │   └─ 4大分类（因产品类型而异）：软件→前端层/后端层/数据层/基础设施；知识/工具→规范准则/操作指南/AI辅助/任务定义
  └─ 详细定义子文档（条目化构件，含实现细节和性能指标）
```

> **同步设计原则**：在开展上层详细定义的同时，同步建立下层方案的架构结构骨架（末级节点名称+映射关系），确保每条详细定义末级节点都有明确的分配目标。

### 每层的角色

| 层 | 名称 | 角色 | 文档结构 |
|----|------|------|---------|
| 1 | 原始需求 | 需求 | 条目化需求 + 末级分解 |
| 2 | 相关方需求 | 方案→需求 | 架构定义子文档（两部分）+ 详细定义子文档 |
| 3 | 业务架构 | 方案 | 架构定义子文档（IPO清单，无详细定义） |
| 4 | 系统需求 | 方案→需求 | 架构定义子文档（两部分）+ 详细定义子文档 |
| 5 | 产品架构 | 方案 | 架构定义子文档（两部分）+ 详细定义子文档 |

## ✅ 操作检查清单（精简版）

### 分析活动
- [ ] 需求说明符合性分析：检查需求覆盖度和冲突/重复
- [ ] 需求性能符合性分析：验证性能指标是否量化、可验证

### 需求分解
- [ ] 末级需求是原子性的
- [ ] 分解停止准则：每条末级只需一个下层节点承接
- [ ] 粒度合理（平均2-3条）
- [ ] 每条都有定义和验收标准

### 需求映射
- [ ] 1:1约束满足（每条需求唯一分配）
- [ ] N:1支持满足（每个方案承接2-5条需求）
- [ ] 映射关系符合业务逻辑

### 符合性分析
- [ ] 每个方案节点都有符合性分析
- [ ] 说明了如何满足所承接的需求
- [ ] 包含验证方法
- [ ] 逻辑清晰、可验证

### 架构定义
- [ ] 架构定义包含两部分：树形结构 + 映射关系表
- [ ] 架构末级节点是需求映射的关键层级
- [ ] 功能分支使用0-9级结构，5级为承接层
- [ ] 非功能分支使用0-6级平行结构

### 同步设计
- [ ] 上层详细定义阶段同步建立下层架构结构骨架
- [ ] 每条详细定义末级节点都有明确的分配目标
- [ ] 结构骨架冻结后方可填充具体内容

### 双向追溯
- [ ] 需求标注了分配的方案
- [ ] 方案标注了承接的需求
- [ ] 所有超链接有效

## 🔍 常见问题速答

### Q1：末级需求应该分解到什么程度？
**A**：
- ✅ 原子性的、不可再分的
- ✅ **分解停止准则**：每条末级只需一个下层方案节点承接（1:1约束）
- ✅ 粒度合理（平均分解数2-3条）

### Q2：一个方案条目应该承接多少条需求？
**A**：
- 最少：1条
- 最多：5条
- 平均：2-3条

### Q3：如何判断映射是否合理？
**A**：
- ✅ 1:1约束满足（每条需求唯一分配）
- ✅ N:1支持满足（每个方案承接2-5条需求）
- ✅ 承接的需求之间有逻辑关联

### Q4：架构定义文档为什么要有两部分？
**A**：
- **第一部分：树形结构** — 展示方案的层级组织和分类，定义架构末级节点
- **第二部分：映射关系表** — 展示需求与方案之间的对应关系，支持双向追溯
- 两部分共同构成完整的架构定义

### Q5：什么是详细定义的双重身份？
**A**：
- **本层身份**：详细定义是对本层架构末级节点的细化分解（向内）
- **下层身份**：详细定义末级节点是下层方案设计的需求输入（向下传递）
- 这意味着修改某层详细定义时，需要同步评估对下游的影响

### Q6：如何建立有效的双向追溯？
**A**：
1. 在方案条目中标注承接的需求
2. 在需求中标注分配的方案条目
3. 使用超链接实现快速导航
4. 定期验证所有超链接的有效性

### Q7：架构末级节点和详细定义末级节点有什么关系？
**A**：
- **架构末级节点**：方案文档的架构定义子文档中的末级节点，是需求映射的关键层级
- **详细定义末级节点**：方案文档的详细定义子文档中的条目，是分配到下层方案架构末级节点的源头
- 两者共同构成方案文档的双层结构

## 🎓 最佳实践速记

### 场景：逆向工程
- ✅ 邀请原开发人员参与，进行代码审查
- ✅ 从代码反推时注意**IPO去重**（复用→改进→新增）
- ✅ 按4大分类整理PA构件：软件产品→前端层/后端层/数据层/基础设施；知识/工具类→规范准则/操作指南/AI辅助/任务定义
- ✅ 利用**详细定义的双重身份**判断反推的合理性

### 场景：瀑布式/新研产品
- ✅ 确保原始需求清晰完整，执行初步整理（碎片归并）
- ✅ 采用**同步设计**：开展上层详细定义时同步建立下层架构骨架
- ✅ 每个步骤执行**分析活动**（说明符合性 + 性能符合性）
- ✅ 使用**两阶段工作流程**（结构骨架约定→具体设计）

### 场景：敏捷式/增量需求
- ✅ 优先处理高优先级需求
- ✅ 分阶段处理，避免一次性大量变更
- ✅ **优先映射到现有方案**，最小化影响
- ✅ 使用增量标记（+新增/~修改）追踪变更
- ✅ 每步采用两阶段工作流程

### 场景：DevOps/快速修复
- ✅ 15分钟快速决策，选择最简模式
- ✅ 只修改必要的部分，避免过度设计
- ✅ 充分验证副作用（回归验证）
- ✅ 事后补全文档，保持追溯链路完整
