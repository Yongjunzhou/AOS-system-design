## 一、核心概念

### 1.1 本层的定义

**原始需求（Original Requirements）** 是 AOS 系统设计流程的起点，是未经规范化的、来自各相关方的原始表述。它可能是会议纪要中的一句话、邮件中的一段描述、或需求文档中的一条条目。

**核心特征**：
- **来源多样**：来自不同相关方（运营管理部、业务部门、信息系统部门等）、不同渠道
- **格式不一**：自然语言表述，风格和粒度参差不齐
- **质量不均**：可能包含模糊、矛盾、重复等问题
- **缺乏分类**：功能和性能常常混在一起表述

**原始需求的本质**：原始需求是"原材料"，不是"成品"。经过规范化、分解、分类等加工后，转化为结构化的相关方需求。

### 1.2 在5层结构中的位置

```
原始输入（邮件、会议纪要、文档...）
   ↓ 收集、整理
原始需求清单（第1层） ★ 本层
   ↓ 规范化、分解
原始需求详细定义（末级需求）
   ↓ N:1 分配
相关方需求架构末级节点（第2层）
```

| 维度 | 说明 |
|------|------|
| **在5层结构中的位置** | 第1层 |
| **文档类型** | 需求文档（描述系统应该做什么） |
| **承接关系** | 承接原始输入（未经规范化的表述） |
| **分配方向** | 分配到第2层相关方需求架构定义末级节点（N:1承接支持） |
| **架构定义** | 无（原始需求层只有详细定义） |

**关键特点**：原始需求是唯一不遵循"需求—方案配对"模型的层——它只有需求，没有方案。

### 1.3 原始需求与相关方需求的关系

| 维度 | 原始需求 | 相关方需求 |
|------|---------|-----------|
| **性质** | 原始表述，规范化前 | 规范化后的需求 |
| **格式** | 自由格式，风格不一 | 标准格式，统一规范 |
| **粒度** | 粗细不一 | 分解到末级（原子性） |
| **分类** | 功能/非功能混在一起 | 明确分离功能和性能 |
| **结构** | 平铺列表 | 架构树形结构 |

### 1.4 原始需求末级的向下传递角色

> **注**：本节的"双重角色"是功能层面的描述，与准则 §1.5.2 定义的"详细定义的双重身份"有区别。原始需求层是唯一没有架构定义的层（准则 §2.3），因此其末级需求不具备形式化的"双重身份"（即同时作为架构定义的细化和下层方案的需求）。但原始需求末级在功能上确实承担两个角色：在本层对原始需求做细化分解，同时作为下层（相关方需求）的输入。

```
原始输入（邮件、会议纪要、文档...）
   ↓ 收集、整理
原始需求清单
   ↓ 规范化、分解
原始需求详细定义（末级需求） ★ 本层角色：细化分解
   ↓ N:1 分配          ★ 向下传递：相关方需求输入
相关方需求架构末级节点
```

- **本层角色**：对原始需求的细化和分解（向内）
- **向下传递**：作为相关方需求架构定义的输入（向下传递）

### 1.5 本层的分析活动

原始需求层的分析活动包括**需求说明符合性分析**和**需求性能符合性分析**两类，具体体现为**冲突检测**和**重复检测**：

**需求说明符合性分析**：
- 冲突检测：识别语义直接矛盾、间接矛盾的需求
- 重复检测：识别完全重复、部分重复、相似重复的需求
- 覆盖度检查：规范化后的需求是否完整保留了原始语义，无遗漏

**需求性能符合性分析**：
- 验证性能指标是否明确量化（非功能需求必须有可测量的指标，如响应时间、完成周期、准确率等）
- 检查指标是否有明确的适用条件和验证方法
- 验证指标从原始需求到规范化描述的继承完整性

分析活动在规范化完成后、映射到相关方需求前集中执行。

## 二、收集和格式规范

### 2.1 收集渠道

| 渠道 | 说明 | 注意事项 |
|------|------|---------|
| **需求会议** | 会议纪要、讨论记录 | 确认参会方和记录准确性 |
| **邮件沟通** | 相关方的邮件需求 | 确认是否代表最终意见 |
| **已有文档** | 已有的需求规格说明书 | 确认文档版本和时效性 |
| **用户反馈** | 用户投诉、建议、改进意见 | 确认普遍性和优先级 |
| **法规合规** | 法律法规、行业标准 | 确认强制性和时间节点 |

### 2.2 收集阶段的格式（完整记录）

收集阶段保留完整的元数据，用于追溯：

| 字段 | 要求 | 说明 |
|------|------|------|
| **需求编号** | 唯一标识，格式 OR-XXX | 示例：OR-001 |
| **需求标题** | 一句话概括 | 清晰、简洁 |
| **提出方** | 提出需求的部门/角色 | 明确业务来源 |
| **提出时间** | 需求的提出日期 | 用于追溯和版本管理 |
| **原始表述** | 原始的、未经修改的表述 | 保留原始措辞 |
| **背景/原因** | 为什么需要这个需求 | 业务背景、痛点、驱动因素 |
| **优先级** | P0/P1/P2/P3 | P0：必须；P1：应该；P2：可以；P3：可选 |
| **来源渠道** | 需求来自哪个渠道 | 会议/邮件/文档/反馈 |
| **关联需求** | 与其他需求的关联关系 | 依赖/冲突/补充 |

### 2.3 规范化后的标准格式（简洁记录）

规范化后只保留核心信息，适用于正式文档：

```
OR-001: [需求标题]
├─ 规范化描述：[经过规范化处理后的描述]
├─ 类型：[功能需求/非功能需求]
├─ 优先级：[P0/P1/P2/P3]
└─ 验收标准：[可量化的验收条件]
```

**对比示例**：

```
收集阶段（完整记录）：
OR-001
├─ 标题：绩效考核评分功能
├─ 提出方：运营管理部
├─ 原始表述：希望系统能支持对员工进行绩效考核打分
├─ 优先级：P0
└─ 来源渠道：需求会议

规范化后（简洁记录）：
OR-001: 绩效考核评分
├─ 描述：部门主管选择考核周期和下属员工后，系统展示评分表，
│        主管逐项填入评分并提交，系统保存评分结果
├─ 类型：功能需求
├─ 优先级：P0
└─ 验收标准：评分提交后可在评分记录中查到，状态为"已提交"
```

## 三、分解规范

### 3.1 分解的定义和原则

**分解（Decomposition）** 是将复杂的原始需求逐层拆解为原子性的末级需求的过程。

**四大原则**：

| 原则 | 说明 | 判断方法 |
|------|------|---------|
| **原子性原则** | 每条末级需求不可再分 | 问：还能进一步拆解吗？ |
| **1:1分配约束** | 每条末级只需映射到相关方需求架构的一个末级节点 | 问：需要多个SR节点承接吗？ |
| **语义完整性** | 分解后不丢失原始语义，所有末级之和=原始需求 | 问：合并后等于原始需求吗？ |
| **粒度一致性** | 同层末级需求的粒度保持一致 | 问：与其他末级的粗细程度相当吗？ |
| **组织资产优先** | 优先复用已有分解模式，仅在必要时新增分解路径 | 问：现有架构中是否有可承接的SR节点？→ 按复用→改进→新增顺序检查 |

### 3.2 规范化规则（分解的前置步骤）

**规范性检查**：在分解前，先对原始需求进行四项检查。

**完整性检查**：
- ✅ 有清晰的**需求描述**（系统应该做什么）
- ✅ 有**类型**（功能需求/非功能需求）
- ✅ 有**优先级**（P0/P1/P2/P3）

**清晰性检查**：模糊术语替换为精确表述

| 模糊术语 | 应替换为 |
|---------|---------|
| 尽可能、尽量 | 具体数值或范围（响应时间 < 2s） |
| 良好的 | 具体指标（可用性 99.9%） |
| 实时的 | 具体延迟要求（延迟 < 100ms） |
| 快速的 | 具体时间（响应时间 < 1s） |

```
✅ 规范化后：
部门主管选择考核周期和员工后，系统应在 3 秒内展示评分表

❌ 规范化前：
绩效考核打分要流畅
```

**可测试性检查**：
- 功能需求：输入、处理逻辑、输出是否明确？
- 非功能需求：量化指标、适用条件、验证方法是否明确？

**独立性检查**：需求是否相对独立？是否与其他需求过度耦合？

### 3.3 分解方法和触发条件

遇到以下情况时需求需要分解：

| 条件 | 示例 | 分解方式 |
|------|------|---------|
| 包含"和/并且/同时"连接多个功能 | "支持考核计划创建、评分录入和报告生成" | 分解为3个独立需求 |
| 涉及多个角色或相关方 | "部门主管和HRBP都需要查看绩效考核结果" | 按角色分解 |
| 需要映射到多个SR末级 | 一个需求描述多个不同分类的功能 | 按功能分类分解 |

**示例**：

```
原始需求：系统应支持绩效考核全流程管理
├─ 末级需求 1.1：系统应支持考核计划创建功能
├─ 末级需求 1.2：系统应支持评分录入功能
├─ 末级需求 1.3：系统应支持考核结果确认功能
└─ 末级需求 1.4：系统应支持绩效报告查询功能
→ 每条末级只需映射到一个SR架构末级节点 ✅
```

### 3.4 分解停止准则

**判断方法**：在设计每条末级需求时，问自己：
> "这个条目能映射到相关方需求架构的一个末级节点吗？"

```
✅ 只需一个节点承接 → 分解完成，停止分解
  SR-F-004（绩效报告管理）能承接"绩效报告查询"这一条末级

❌ 需要多个节点承接 → 分解不充分，继续分解
  "绩效管理"既涉及SR-F-003（评分计算）又涉及SR-F-004（报告管理）
  → 应分解为"评分录入"和"绩效报告查询"两条
```

### 3.5 分解记录格式

以下示例以 AOS 运营绩效考核系统为背景，展示分解记录的格式。实际使用时，SR 节点名称应代入对应 AOS 项目的实际相关方需求架构。

```yaml
原始需求: OR-001（绩效考核评分）

末级分解:
  - ID: OR-001.01
    描述: 部门主管选择考核周期和下属员工
    分配到: SR-F-003（评分与计算功能）
    验收标准: 选择后系统正确展示该员工的评分表

  - ID: OR-001.02
    描述: 主管逐项填写评分并提交
    分配到: SR-F-003（评分与计算功能）
    验收标准: 提交后评分状态变为"已提交"，数据持久化

  - ID: OR-001.03
    描述: 系统保存评分记录并通知员工
    分配到: SR-F-005（通知与提醒功能）
    验收标准: 评分保存后，员工收到绩效评分已提交通知

分解决策:
  - OR-001.01: 考核范围选择是评分的前置步骤，独立分解
  - OR-001.02: 评分录入是核心功能，独立分解
  - OR-001.03: 通知功能归属于独立的通知模块
```

## 四、设计方法

### 4.1 完整处理流程

```
步骤1：需求收集
  └─ 从各渠道收集原始需求，按收集阶段格式完整记录
     └─ 输出：原始需求清单（含完整元数据）

步骤2：初步整理（可选，建议执行）
  └─ 对来自多渠道的零散碎片需求进行归并整理
     ├─ 血缘归并：将描述同一功能不同切面的碎片合并，还原完整表述
     ├─ 角色聚合：不同角色对同一功能的诉求合并为一条，标注角色差异
     ├─ 流程串联：同一流程不同步骤的碎片按操作顺序串联，保持完整性
     └─ 父子补全：以主需求为载体，补充细节作为子约束
     └─ 遵循五条规则：不过度合并、保留原始出处、性能指标不合并、矛盾碎片不合并、无法判断标记待补全
        └─ 输出：初步整理后的需求清单

步骤3：规范化
  └─ 四项检查（完整性、清晰性、可测试性、独立性）
     └─ 模糊术语替换，功能/非功能分离
        └─ 输出：规范化后的需求描述

步骤4：分解
  └─ 按分解原则将需求拆解到末级
     └─ 验证1:1约束（每条末级只需一个SR节点承接）
        └─ 输出：原始需求详细定义（末级需求清单）

步骤5：冲突和重复检测
  └─ 末级需求之间的冲突/重复交叉检查
     └─ 输出：冲突和重复检测报告

步骤6：映射到相关方需求
  └─ 按关联性分组，分配到SR架构末级节点
     └─ 输出：映射关系表
```

### 4.2 功能/非功能需求分离

在规范化时完成分离：

```
原始需求："绩效评分提交时，系统应在 3 秒内完成保存"

规范化后：
功能需求（OR-001.01）：
  - 描述：主管填写评分后点击提交，系统保存评分数据并更新状态
  - 验收标准：评分保存后状态变为"已提交"，数据库中可查到

非功能需求（关联 OR-001）：
  - 描述：评分提交操作应快速响应
  - 指标：提交响应时间 < 3s（包括数据持久化时间）
```

### 4.3 冲突和重复检测

**冲突类型**：

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| **直接冲突** | 语义直接矛盾，无法同时满足 | 由相关方确认保留哪条 |
| **间接冲突** | 同时实现可能行为不一致 | 确认是否调整业务规则 |
| **可合并冲突** | 可通过调整合并解决 | 建议合并为更完整的需求 |

**重复类型**：

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| **完全重复** | 两条需求完全相同 | 去重保留一条 |
| **部分重复** | 功能范围部分重叠 | 合并为更完整的描述 |
| **相似重复** | 表达不同但核心相同 | 确认是否为同一需求 |

**检测记录格式**：

```
| 需求A | 需求B | 问题类型 | 描述 | 建议 |
|-------|-------|---------|------|------|
| OR-001 | OR-005 | 直接冲突 | 一个要求评分结果必须审核，一个要求提交即生效 | 需确认 |
| OR-002 | OR-003 | 部分重复 | 都涉及绩效报告查询功能，50% 重叠 | 建议合并 |
```

### 4.4 到相关方需求的映射

**映射路径**：

```
原始需求末级需求清单
  ├─ 功能类末级 → N:1 → SR-F 架构末级节点
  └─ 非功能类末级 → N:1 → SR-NFR 架构末级节点
```

**映射规则**：

| 规则 | 说明 |
|------|------|
| **N:1承接支持** | 多个原始需求末级可映射到同一个SR架构末级 |
| **功能/非功能分离** | 功能类→SR-F，非功能类→SR-NFR |
| **1:1分配约束** | 每条原始需求末级只分配到唯一一个SR架构末级 |
| **完全映射** | 所有原始需求末级都必须映射，无遗漏 |

**映射表示例**（以 AOS 绩效考核系统为例）：

```
| 原始需求末级 | SR架构末级 | 类型 | 符合性说明 |
|-------------|-----------|------|-----------|
| OR-001.01    | SR-F-003  | 功能 | 评分选择功能与SR-F-003一致 |
| OR-001.02    | SR-F-003  | 功能 | 评分录入功能合并到同一节点 |
| OR-002.01    | SR-F-004  | 功能 | 报告查询需求映射到报告管理模块 |
| OR-003.01    | SR-NFR-001| 非功能 | 响应时间指标已量化 |
```

### 4.5 同步设计

**核心原则**：原始需求末级节点需要分配到相关方需求架构定义末级节点（1:1分配约束）。因此，在分解原始需求的同时，应同步建立相关方需求架构定义的结构骨架（即使只是末级节点名称和映射关系）。

**为什么要同步**：
- **提供分解的"靶子"**：相关方需求架构定义的存在，使分解过程有明确的分配目标，确保每条末级需求在分解时就能判断"是否只需要一个SR架构末级节点承接"
- **避免无目标返工**：如果SR架构定义尚未建立，原始需求分解到末级后可能发现没有对应的SR节点可承接，导致需要调整分解粒度，造成返工
- **为并行设计创造条件**：同步建立SR架构骨架后，上下游可并行填充详细内容

**实际操作**：
1. 在开始原始需求分解时，同步建立相关方需求架构定义的结构骨架（功能+非功能的末级节点名称）
2. 每完成一条末级需求，立即检查是否能分配到一个SR架构末级节点
3. 如果发现不能（需要多个），则说明分解不够，继续分解
4. 结构骨架冻结后，再并行填充SR架构末级节点的详细内容

> **参考**：准则 §1.5.3 详细定义了同步设计原则和两阶段工作流程（先约定结构骨架并建立映射关系，再并行填充详细内容）。

### 4.6 关键要求汇总

以下关键要求来自准则 §4.1，本规范的上述各章已逐一落实：

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误1：跳过规范化直接分解**

```
错误的做法：
原始表述："绩效考核打分要流畅"
直接分解为："OR-001.01：绩效考核打分"

问题：模糊的原始表述在分解后依然模糊
后果：模糊需求传递到下游，问题被放大
```

```
✅ 正确做法：
先规范化：
  "部门主管选择考核周期和员工后，系统应在 3 秒内展示评分表"
再分解：
  OR-001.01：评分表展示功能
  OR-001.02：响应时间 < 3s
```

**❌ 错误2：功能和非功能未分离**

```
错误的做法：
"绩效考核结果查询时，系统应该快速展示报告"
→ 作为一条功能需求处理

问题：非功能约束被当作功能需求
后果：经过业务架构层，设计混乱
```

```
✅ 正确做法：
功能需求：用户选择考核周期后，系统展示绩效报告
非功能需求：报告查询响应时间 < 3s，100人同时查询时 < 5s
→ 分别记录、分别处理
```

**❌ 错误3：分解不充分**

```
错误的做法：
OR-001.01：系统应支持绩效全流程管理
→ 映射到 SR-F-003 + SR-F-004 + SR-F-005

问题：一条末级需要三个SR节点承接，违反1:1约束
后果：追溯链路断裂
```

```
✅ 正确做法：
OR-001.01：评分录入功能 → SR-F-003（评分与计算）
OR-001.02：绩效报告查询 → SR-F-004（报告管理）
OR-001.03：评分通知推送 → SR-F-005（通知提醒）
→ 每条末级只映射到一个SR节点
```

**❌ 错误4：忽略资产优先原则，为每条新需求直接新增SR节点**

```
错误的做法：
新的原始需求到达时，直接在SR架构中新增架构末级节点来承接，
而不检查现有SR节点是否能复用或扩展后承接。

问题：忽视组织已有设计资产，导致SR架构膨胀
后果：SR节点数量快速增加，映射关系复杂化，维护成本上升
```

```
✅ 正确做法：
按 复用→改进→新增 的优先级依次检查现有SR节点。
仅当复用和改进都不可行时，才新增SR架构末级节点。
说明：保护组织已有设计资产，控制架构复杂度
```

### 7.2 最佳实践

**实践1：使用结构化格式记录**：每条原始需求使用统一的字段格式，便于自动化和批量处理。

**实践2：在规范化阶段就考虑分类**：提前识别功能/非功能、业务/平台分类，为后续工序准备好依据。

**实践3：边分解边考虑映射**：分解时同步考虑它应映射到哪个SR节点，确保分解粒度以映射为导向。

**实践4：保留原始表述**：无论规范化后的描述如何优化，始终保留原始表述作为追溯依据。

## 八、FAQ

**Q1：原始需求一定要收集所有元数据字段吗？**

收集阶段建议完整记录（9字段），便于追溯和审计。但规范化后的正式文档只需保留核心字段（编号、描述、类型、优先级、验收标准）。两阶段格式的分离正是为了兼顾追溯价值和文档简洁性。

**Q2：分解到什么程度才算"原子性"？**

判断标准是1:1约束——一条末级需求只需映射到**一个**相关方需求架构末级节点。如果还需要多个节点承接，说明不是原子性的。

**Q3：功能和非功能需求混在一起怎么办？**

在规范化阶段分离。识别原始需求中的"做什么"部分 → 作为功能需求；识别"有多好"部分 → 作为非功能需求。分离后标注相互关联关系。

**Q4：冲突检测和重复检测的结果谁来决定？**

规范负责识别和分类，最终决策由相关方确认。规范提供建议（如"建议合并"），但不替代人工判断。检测记录格式中应包含"待确认"的状态标注。

**Q5：原始需求映射到相关方需求后，原始需求还需要保留吗？**

需要保留。原始需求是追溯的源头，当后续设计出现争议时，需要通过原始表述确认最初的意图。保留原始表述是确保双向追溯完整性的基础。
