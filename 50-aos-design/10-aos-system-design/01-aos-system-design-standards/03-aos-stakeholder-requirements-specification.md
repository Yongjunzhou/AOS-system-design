## 一、核心概念

### 1.1 本层的定义

**相关方需求**是对原始需求（第1层）的**方案文档**，同时又是业务架构（第3层）的**需求输入**。它由两份子文档构成：

- **架构定义文档**：对需求的分类和组织（回答"有什么需求"）
- **详细定义文档**：对架构末级节点的分解（回答"每个需求具体包括什么"）

### 1.2 在5层结构中的位置

```
原始需求（第1层）
   ↓ 规范化、分解
相关方需求（第2层） ★ 本规范
  ├─ 架构定义：相关方需求条目 ← 承接原始需求末级（N:1承接支持）
  └─ 详细定义：对条目的分解 → 分配到业务架构末级（1:1分配约束）
       ↓ N:1承接支持
业务架构（第3层）
```

| 维度 | 说明 |
|------|------|
| **文档类型** | 方案文档（架构定义 + 详细定义） |
| **承接对象** | 原始需求末级（N:1承接支持） |
| **分配目标** | 业务架构末级/IPO（1:1分配约束） |
| **既是方案** | 对原始需求的方案（第1层的需求 → 第2层的方案） |
| **又是需求** | 对业务架构的需求（第2层的方案 → 第3层的需求） |

### 1.3 本层的分析活动

按系统设计准则的定义，本层通过两类分析验证设计的完整性和正确性：

**需求说明符合性分析**：
- 检查每个架构末级节点是否充分满足所承接的原始需求末级
- 验证 N:1 映射的完整性（无遗漏原始需求末级）
- 验证功能与非功能需求的分离正确性

**需求性能符合性分析**：
- 验证性能指标是否从原始需求正确继承和量化
- 检查指标是否可测量、可验证

分析活动贯穿设计全过程，在架构定义完成后和详细定义开始前集中执行。

### 1.4 详细定义的双重身份

详细定义在系统设计中扮演双重角色，这是设计能够层层递进的关键：

```
详细定义是：
1. 本层身份：对相关方需求架构末级节点的细化分解（向内分解）
2. 下层身份：对业务架构的需求输入（向下传递）
```

**示例**（以 AOS 绩效考核系统为例）：
```
架构末级：SR-F-003（评分与计算功能）
   ↓ [分解为]
详细定义末级：
  SR-F-003.1（评分表定义功能）→ 分配到 BA-评分表配置
  SR-F-003.2（评分录入功能）→ 分配到 BA-评分录入
  SR-F-003.3（自动计算功能）→ 分配到 BA-自动计算
```

这些详细定义末级既是对 SR-F-003 的分解说明，又是对业务架构的具体需求。

### 1.5 功能 vs 非功能需求的处理区别

| 方面 | 功能需求 | 非功能需求 |
|------|---------|----------|
| **分解方式** | 支持多层分解（SR-F-001.1.1等） | 不需要分层分解，直接到末级 |
| **映射目标** | 业务架构的IPO | 系统约束（不映射到业务架构） |
| **详细定义内容** | 具体的功能描述 | 量化指标和验证方法 |
| **分解终点** | 能映射到一个IPO | 能独立验证的约束条件 |

### 1.6 性能指标的跨层传递

性能指标**不是独立的需求**，而是需求的一个属性。每条需求都应包含**需求描述**和**性能指标**。

```
第2层：相关方需求 ← 定义性能指标的起点
  │  每条SR需求包含：需求描述 + 性能指标
  ↓  N:1分配
第3层：业务架构 ← 继承并细化性能指标
  ↓  IPO去重后作为系统需求输入
第4层：系统需求 ← 继承并技术化性能指标
  ↓  N:1分配
第5层：产品架构 ← 最终实现并验证性能指标
```

本规范聚焦于第2层性能指标的定义。后续层级的性能指标定义由相应层规范覆盖。

## 二、架构定义规范

> **注**：本章及第三章的示例以 AOS 运营绩效考核系统为背景。实际使用时，SR 节点名称应代入对应 AOS 项目的实际相关方需求架构。

### 2.1 架构末级节点的定义

**架构末级节点**是相关方需求分类体系中的最细粒度节点，是承接原始需求末级的关键层级。

#### 特征

- ✅ **可独立定义**：有清晰、完整的定义和描述
- ✅ **可独立验证**：有明确的验收标准和评估方式
- ✅ **承接多条原始需求**：体现N:1承接支持的映射关系
- ✅ **分配明确**：每条承接需求都有明确的分配关系

#### 功能需求架构末级节点（SR-F-XXX）

| 属性 | 要求 | 说明 |
|------|------|------|
| **ID格式** | SR-F-[号] | 示例：SR-F-001 |
| **标题** | 一个完整的功能或业务需求 | 评分与计算功能 |
| **定义** | 清晰的需求叙述 | 描述用户/系统应该能够做什么 |
| **承接需求** | N条原始需求末级 | 说明承接了哪些原始需求 |
| **符合性分析** | 如何满足所承接的需求 | 说明该节点如何满足上层需求 |
| **设计指导** | 为业务架构的指导 | 暗示业务架构应该如何设计 |
| **性能指标** | 功能相关的性能要求 | 处理效率、准确率等 |

#### 非功能需求架构末级节点（SR-NFR-XXX）

| 属性 | 要求 | 说明 |
|------|------|------|
| **ID格式** | SR-NFR-[号] | 示例：SR-NFR-001 |
| **需求类型** | 性能/安全/可用性等 | 明确的非功能需求类型 |
| **需求类别** | 质量特性/环境适应性/可实现性 | 三大类之一 |
| **定义** | 系统应满足的质量属性 | 系统应快速响应 |
| **承接需求** | N条原始非功能需求末级 | 说明承接的原始需求 |
| **符合性分析** | 如何满足所承接的需求 | 说明如何满足上层需求 |
| **量化指标** | 具体的、可测量的指标 | 包含数值、单位、条件 |

#### 颗粒度标准

通过对比确定最适粒度：

```
✅ 合适的粒度（应分为末级节点）：
  - SR-F-003: 评分与计算功能（代表一个完整的业务能力）
  - SR-NFR-001: 系统响应时间性能需求（一个独立的质量属性维度）

❌ 粒度过粗（应进一步分类）：
  - SR-F-001: 绩效全流程管理（涵盖多个明显不同的功能，太宽泛）

❌ 粒度过细（应合并到上级）：
  - SR-F-004: 报告标题生成（只是绩效报告管理的一小部分）
  - SR-NFR-002: 体验好（没有具体量化指标）
```

### 2.2 原始需求到架构末级节点的映射

**关系**：N:1承接支持 — 一个架构末级节点可承接多条原始需求末级。

**设计原则**：

| 原则 | 说明 |
|------|------|
| **关联性原则** | 承接的原始需求之间应有逻辑关联 |
| **完整性原则** | 架构末级节点应完整覆盖其承接的所有需求 |
| **清晰性原则** | 映射关系应清晰易懂，支持双向追溯 |

**操作步骤**：

```
第1步：列举所有原始需求末级
  → 需求-001（考核计划创建）、需求-002（评分录入）、需求-003（报告查询）...

第2步：按关联性分组
  → 组1（评分相关）：需求-001、002
  → 组2（报告相关）：需求-003、004

第3步：为每组定义架构末级节点
  → SR-F-003: 评分与计算功能（承接：需求-001、002）
  → SR-F-004: 绩效报告管理功能（承接：需求-003、004）
```

### 2.3 功能需求架构的分类体系

```
相关方需求（根）
└─ 功能需求
   ├─ 运营考核核心功能
   │  ├─ SR-F-001: 考核计划管理功能
   │  ├─ SR-F-002: 考核对象管理功能
   │  ├─ SR-F-003: 评分与计算功能
   │  ├─ SR-F-004: 绩效报告管理功能
   │  └─ SR-F-005: 通知与提醒功能
   │
   └─ 运营支撑功能
      ├─ SR-F-101: 组织架构管理功能
      ├─ SR-F-102: 用户权限管理功能
      └─ SR-F-103: 系统配置管理功能
```

**分类标准**：
- ✅ 按业务域分类：来自同一个业务部门/流程的需求聚合
- ✅ 按系统能力分类：涉及系统基础能力的需求聚合
- ❌ 避免过深分类体系：通常2-3层足够

### 2.4 非功能需求架构的分类体系

非功能需求**不经过业务架构**，直接在系统需求层面标记为系统约束。

```
相关方需求（根）
└─ 非功能需求
   ├─ 质量特性需求
   │  ├─ SR-NFR-001: 性能需求
   │  ├─ SR-NFR-002: 安全需求
   │  └─ SR-NFR-003: 可用性需求
   │
   ├─ 环境适应性需求
   │  └─ SR-NFR-101: 可扩展性需求
   │
   └─ 可实现性需求
      └─ SR-NFR-201: 可测试性需求
```

### 2.5 架构定义的文档结构

架构定义文档包含两个部分：

**第一部分：架构设计结果（树形结构）**

```
相关方需求（0级·根）

├─ 功能需求（1级·分类）
│  └─ 运营考核核心功能（2级·分类）
│     └─ SR-F-003: 评分与计算功能（3级·架构末级节点）
│        ├─ 定义：部门主管能够配置评分表、录入评分、系统自动计算绩效得分
│        ├─ 承接原始需求末级：需求-002, 需求-003, 需求-004
│        ├─ 符合性分析：通过评分配置、录入和自动计算满足原始需求对绩效考核评分的要求
│        ├─ 设计指导：业务架构应提供评分表配置、评分录入、自动计算等业务流程
│        └─ 性能指标：评分录入响应 < 2s，自动计算 < 5s（100人范围内）
│
└─ 非功能需求（1级·分类）
   └─ 质量特性需求（2级·分类）
      └─ SR-NFR-001: 性能需求（3级·架构末级节点）
         ├─ 定义：系统应在高负载下保持快速响应
         ├─ 承接原始需求末级：需求-N01, 需求-N02
         ├─ 符合性分析：通过性能优化和架构设计满足响应速度要求
         ├─ 量化指标：报告生成 < 10s，支持 5000+ 员工规模
         └─ 测量方法：性能测试
```

**第二部分：需求映射视图（映射关系表）**

```
| 原始需求末级 | 承接的架构末级节点 | 关联说明 | 超链接 |
|-------------|------------------|--------|--------|
| 需求-002    | SR-F-003         | 与评分录入直接相关 | [→] |
| 需求-003    | SR-F-003         | 与自动计算直接相关 | [→] |
| 需求-004    | SR-F-003         | 与评分表配置直接相关 | [→] |
| 需求-N01    | SR-NFR-001       | 报告生成时间约束 | [→] |
```

## 三、详细定义规范

### 3.1 功能需求详细定义

每条功能需求详细定义末级节点应包含：

| 字段 | 要求 | 说明 |
|------|------|------|
| **ID** | 唯一标识，格式SR-F-[上级]-[分解号] | SR-F-003.1 |
| **标题** | 一句话概括 | 评分录入 |
| **详细描述** | 完整的需求叙述 | 使用"系统应该..."或"用户能够..." |
| **分解依据** | 为什么进行这样的分解 | 说明该条目的独立意义 |
| **分配目标** | 映射到业务架构的末级节点 | BA-XXX或IPO名称 |
| **验收标准** | 可测试的标准 | 功能实现后如何验证 |
| **对上层的符合性** | 说明如何满足上层需求 | 链接到上层架构末级节点 |

**示例**（以 AOS 绩效考核系统为例）：

```
架构末级节点：SR-F-003（评分与计算功能）

详细定义末级节点：
SR-F-003.1（评分表配置功能）
├─ 描述：管理员能够定义评分表的考核维度、权重和评分标准
├─ 分解依据：评分处理流程的第一个独立环节
├─ 分配目标：BA-评分表配置流程 [1:1映射]
├─ 验收标准：评分表配置完成后可在评分时正确展示，权重总和 = 100%
└─ 符合性：直接满足SR-F-003中对评分配置的需求

SR-F-003.2（评分录入功能）
├─ 描述：主管能够逐项录入下属员工的各项评分
├─ 分配目标：BA-评分录入流程 [1:1映射]
├─ 验收标准：评分提交后状态变为"已提交"，数据持久化
└─ 符合性：满足SR-F-003中对评分录入的要求
```

### 3.2 非功能需求详细定义

非功能需求的详细定义 = **量化指标的完整定义**。

| 字段 | 要求 | 说明 |
|------|------|------|
| **ID** | 唯一标识，格式SR-NFR-[号] | SR-NFR-001 |
| **需求描述** | 系统应满足的质量属性说明 | 系统应快速响应 |
| **量化指标** | 具体的、可测量的指标 | 必须包含具体数值和单位 |
| **测量方法** | 如何测量/验证该指标 | 性能测试、安全审计 |
| **应用范围** | 指标适用的场景/条件 | 正常业务负载、高并发场景 |
| **优先级** | P0/P1/P2/P3 | 需求的优先级 |

**示例**：

```
SR-NFR-001: 系统响应时间性能需求
需求类型：性能 | 需求类别：质量特性需求
描述：AOS 绩效考核系统在正常负载下应快速响应用户操作
量化指标：
  - 平均响应时间：< 500ms
  - P99响应时间：< 2000ms（在500并发下）
测量方法：JMeter性能测试
应用范围：所有关键流程操作（评分录入、报告查询）
优先级：P0
```

### 3.3 分解标准

**何时需要分解**：
- 一个架构末级节点涉及**多个独立的功能流程**
- 一个需求涉及**多个用户角色的交互**
- 一个需求涉及**多个业务规则或条件**

**何时停止分解**（1:1约束判断法）：

在设计每条详细定义末级节点时，问自己：
> "这个需求能映射到业务架构的**一个**IPO吗？"

- ❌ **需要多个IPO承接** → 分解不充分，继续分解
- ✅ **只需一个IPO承接** → 分解完成，停止分解

**对比示例**：

```
✅ 分解充分：
SR-F-003.1（评分表配置）→ BA-评分表配置
SR-F-003.2（评分录入）→ BA-评分录入
→ 每个末级只映射到一个IPO，1:1约束满足

❌ 分解不充分：
SR-F-003.1（评分处理）→ BA-评分表配置 + BA-评分录入
→ 一个末级需要映射到多个IPO，违反1:1约束，需继续分解
```

### 3.4 格式模板（YAML）

```yaml
相关方需求: SR-F-003
架构末级节点: 评分与计算功能

详细定义末级节点:
  - ID: SR-F-003.1
    标题: 评分表配置
    描述: |
      管理员能够定义评分表的考核维度、权重
      和评分标准，系统保存评分表模板
    分解依据: 评分处理生命周期的第一个独立环节
    分配目标: BA-评分表配置流程
    验收标准: |
      1. 评分表配置成功率 > 99%
      2. 权重总和必须等于 100%
    对上层的符合性: 直接满足SR-F-003中对评分配置的要求
```

## 四、设计方法

### 4.1 两阶段设计流程

相关方需求设计分为两个阶段（先架构定义、再详细定义），与准则 §1.5.3 定义的跨层"两阶段工作流程"不同——本节描述的是相关方需求层内部的设计顺序：

```
第一阶段：架构定义（回答"有什么需求"）
  输入：原始需求末级清单
  步骤：
    1. 分析原始需求的关联性
    2. 按业务逻辑分组（功能/非功能分离）
    3. 定义架构末级节点（SR-F-XXX / SR-NFR-XXX）
    4. 标注N:1映射关系和符合性分析
  输出：架构定义文档（树形结构 + 映射视图）

第二阶段：详细定义（回答"每个需求具体包括什么"）
  输入：架构定义文档
  步骤：
    1. 选择架构末级节点
    2. 按功能或非功能规则分解
    3. 验证每条末级是否满足1:1约束
    4. 标注分配目标（业务架构IPO）
    5. 补充验收标准和符合性分析
  输出：详细定义文档
```

### 4.2 架构定义与详细定义的协作关系

```
架构定义层：组织和分类
  SR-F-003: 评分与计算功能 ← 架构末级节点
    ↓ 承接原始需求末级：需求-002, 003, 004

详细定义层：分解和细化
  SR-F-003.1: 评分表配置 ← 详细定义末级节点
  SR-F-003.2: 评分录入
  SR-F-003.3: 自动计算
    ↓ 每条都分配到业务架构的一个IPO（1:1）
    → BA-评分表配置流程、BA-评分录入流程等
```

### 4.3 同步设计实践

详细定义末级节点需要分配到业务架构IPO（1:1约束）。因此在开展详细定义的同时，应同步开展业务架构定义。

**实践方法**：

```
1. 在开始详细定义分解时，同步建立业务架构IPO结构骨架
2. 每完成一条详细定义末级，立即检查是否能分配到唯一IPO
3. 如果不能（需要多个），说明分解不充分，继续分解
4. 确认映射关系后，并行填充详细内容
```

### 4.4 迭代方法

> **与准则的关系**：本节描述的迭代三回合是准则 §1.5.3"两阶段工作流程"在相关方需求层的具体应用——第一轮对应第一阶段（约定结构骨架），第二、三轮对应第二阶段（填充内容 + 验证调整）。

实际设计中，两阶段设计通常是迭代进行的：

```
第一轮（快速约定结构）：
  架构定义 → 快速确定末级节点（占位符）
  详细定义 → 快速确定粒度（ID和标题）
  目标：建立映射关系骨架

第二轮（详细填充内容）：
  架构定义 → 补充完整的定义和符合性分析
  详细定义 → 补充完整的描述和验收标准
  目标：补充设计细节

第三轮（验证和调整）：
  检查1:1约束是否满足
  检查N:1承接支持是否合理
  目标：确保设计的充分性
```

## 五、验收标准

### 5.1 架构定义验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **原始需求映射覆盖率** | 100% | 所有原始需求末级都映射到某个架构末级节点 |
| **功能/非功能分离率** | 100% | 所有需求已按功能/非功能正确分类 |
| **架构末级节点可验证** | 100% | 每个末级节点都有清晰的验收标准 |

### 5.2 详细定义验收

| 检查项 | 标准 | 说明 |
|--------|------|------|
| **分解完整性** | 100% | 所有架构末级节点都已分解 |
| **1:1约束满足率** | 100% | 每条详细定义末级只分配到一个业务架构IPO |
| **验收标准完整率** | 100% | 每条详细定义末级都有可测量的验收标准 |
| **非功能指标明确率** | 100% | 每个非功能需求都有量化指标 |

| 检查项 | 标准 | 说明 |
|--------|------|------|

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误1：分解粒度不当**

```
错误的做法：
SR-F-003: 评分与计算功能
  ├─ SR-F-003.1（评分计算） ← 标题与上层几乎相同
  ├─ SR-F-003.2（流程处理）  ← 过于模糊
  └─ SR-F-003.3（系统管理）  ← 不相关

问题：分解标题未真正细化，读者无法通过分解获得更多信息
后果：业务架构设计时无法确定该分配什么IPO
```

```
✅ 正确的做法：
SR-F-003: 评分与计算功能
  ├─ SR-F-003.1（评分表配置）
  ├─ SR-F-003.2（评分录入）
  ├─ SR-F-003.3（自动计算）
  └─ SR-F-003.4（评分审核）

说明：明确了评分处理的四个具体环节，每个都能映射到独立IPO
```

**❌ 错误2：详细定义粒度与业务架构粒度不匹配**

> **注**：N:1承接支持（多条详细定义末级 → 一个BA IPO）本身是合法的。本错误关注的是粒度匹配问题——当多条详细定义代表明显不同的独立功能但都指向同一个"大而全"的IPO时，说明架构粒度不匹配。

```
可疑的做法（需判断）：
SR-F-003.1（评分表配置）→ BA-评分处理流程
SR-F-003.2（评分录入）→ BA-评分处理流程
SR-F-003.3（自动计算）→ BA-评分处理流程

问题：N:1承接本身合法，但若"BA-评分处理流程"是唯一IPO，则三个不同环节都挤在一个IPO中
判断：如果配置、录入、计算是三个可由不同人独立完成的工作，则应各对应独立IPO
```

```
✅ 正确的方案（二选一）：
方案A（合并详细定义——如果三个环节确是同一件事）：
  SR-F-003.1（评分生命周期管理）→ BA-评分处理流程
方案B（拆分业务架构——如果三个环节属于不同流程）：
  将BA-评分处理流程拆分为三个独立IPO（配置/录入/计算）
```

**❌ 错误3：非功能需求缺少量化指标**

```
错误的做法：
SR-NFR-001: 系统应该快速响应
量化指标：系统响应快

问题："快速"、"快"是主观词汇，无法验证
```

```
✅ 正确的做法：
SR-NFR-001: 系统响应时间性能需求
量化指标：
  - 平均响应时间 < 500ms
  - P99响应时间 < 2000ms
  - 在500并发用户下 < 1000ms

说明：具体数字和单位，可通过性能测试验证
```

### 7.2 最佳实践

**实践1：使用标准化的分解结构**

```
建议：对各类需求制定标准的分解模式
示例：
- 考核计划相关 → 创建/发布/执行/关闭
- 评分处理相关 → 配置/录入/计算/审核

优点：保持分解一致性，降低理解成本
```

**实践2：量化指标包含完整上下文**

```
建议：指标应包含 数值 + 单位 + 测量条件 + 应用范围

❌ 不完整：响应时间 < 2秒
✅ 完整：平均响应时间 < 500ms（在500并发、正常数据量下）
                P99响应时间 < 2秒（在500并发、最大数据量下）
```

**实践3：详细定义与业务架构同步设计**

```
流程：
1. 对架构末级节点进行初步的详细定义分解
2. 同步创建业务架构IPO的占位符
3. 验证每条详细定义末级能否1:1映射到一个IPO
4. 不能则调整（继续分解或调整业务架构）
5. 确认映射后，并行填充详细内容
```

**实践4：建立双向追溯**

```
在详细定义中标注分配目标（向下）
在业务架构中标注承接的详细定义（向上）
使用超链接实现快速导航
```

## 八、FAQ

**Q1：详细定义可以有多少层？**

没有固定限制，但应遵循1:1分配约束。功能需求通常1-3层（SR-F-001 → SR-F-001.1 → SR-F-001.1.1），非功能需求通常1-2层。关键是**每层都有明确的分解目的**，而非盲目分解。

**Q2：详细定义末级节点的数量有没有上限？**

单个架构末级节点的详细定义末级建议在3-10个。过少说明分解不足，过多（>10）说明粒度不当，应考虑是否需要重新组织分类。

**Q3：如果无法1:1映射到业务架构怎么办？**

有两种可能：① 详细定义分解不充分（该条目需继续分解）；② 业务架构设计不合理（缺少对应的末级节点）。处理：先检查详细定义是否还能进一步分解，如果已是原子级则调整业务架构设计。

**Q4：非功能需求一定要细化到具体数字吗？**

是的。非功能需求的详细定义本质上就是具体化量化指标。如果无法提供具体数字，可能原因：不了解用户真实需求（需沟通）、技术上未确定（可先提出预期）、需求不合理（需重新审视）。

**Q5：详细定义和业务架构设计可以并行吗？**

可以，并且**推荐并行**。第一轮快速约定结构（详细定义的ID和标题 + 业务架构的IPO名称），第二轮并行填充内容（描述和验收标准 + IPO的Input/Process/Output定义），第三轮验证映射。
