## 一、核心概念

### 1.1 本层的定义

**产品架构（Product Architecture）** 是将系统需求转化为具体构件实现方案的设计结果。它回答的问题是：**"系统应该如何被构建？构件之间如何交互？"**

产品架构在 AOS 系统设计中扮演着**方案文档**的角色：
- **是系统需求（第4层）的方案**：将功能和非功能要求转化为构件的实现方案
- **是产品研发的蓝图**：为开发团队提供明确的实现指导

> **注**：构件的实现方案形态因产品类型而异。对于 AOS（运营体系软件产品），构件实现方案体现为软件模块、微服务、数据库、界面等具体实现单元。

### 1.2 在5层结构中的位置

```
系统需求（第4层）
  ├─ 功能：SysReq-F 9级（场景活动）
  └─ 非功能：SysReq-NFR（系统约束）
       ↓ N:1 映射

产品架构（第5层） ★ 本规范
  ├─ 架构定义：产品构件清单 ← 承接SysReq 9级
  └─ 详细定义：实现方案、技术栈、接口定义、质量指标
       ↓
产品研发（构件实现）
```

| 维度 | 说明 |
|------|------|
| **文档类型** | 方案文档（架构定义 + 详细定义） |
| **承接对象** | SysReq 9级场景活动（N:1）+ SysReq-NFR |
| **输出对象** | 产品研发团队（开发实现） |

### 1.3 产品架构的组成

产品架构的组成因**产品类型**而异。AOS（企业运营体系）属于**软件产品**，其产品构件的组成如下：

| 组成 | 说明 | 示例 |
|------|------|------|
| **前端应用** | 用户交互界面 | AOS管理控制台 |
| **后端服务** | 业务逻辑处理 | 考核管理服务、评分计算服务 |
| **数据存储** | 数据持久化 | 考核计划数据库、评分结果数据库 |
| **基础设施** | 运行环境和支撑 | 应用服务器集群、监控系统 |

### 1.4 本层的设计活动

三个活动遵循准则 [§1.5.1 需求的分解、分配与分析](./01-aos-system-design-standards.md#151-需求的分解分配与分析)（含语义分解四条件、指标分解规则、1:1约束细则等完整规范），本层的具体要点如下。

> **层特有说明**：产品架构是最下层，**需求分配及分析不适用**。

**需求分解及分析**——本层要点：
- 对 PA 构件进行详细定义分解（实现策略/技术方案/质量指标）
- 分解时检查每个构件的职责边界是否清晰、是否可以独立开发和验收

**需求分配及分析**：不适用——产品架构是最下层，不需要再向下分配；构件的详细定义即为最终交付规范。

**架构承接需求分析**——本层要点：
- 覆盖度检查 — 所有 SysReq 9 级节点是否都有对应的 PA 构件承接
- 1:1 分配验证 — 每个 SysReq 9 级节点是否分配到唯一的 PA 构件
- N:1 承接合理性 — 每个 PA 构件承接 2-5 个 SysReq 9 级节点为合理
- 指标继承验证 — PA 构件的质量指标不低于所承接 SysReq 和 NFR 的要求
- 可实现性评估 — 结合技术栈和资源评估质量指标的可实现性
- 权衡决策记录 — 当多个 NFR 冲突时，权衡决策是否已记录

## 二、架构定义规范

### 2.1 架构末级节点

产品架构的末级节点是**产品构件**，每个构件应有清晰的职责边界。

每个产品架构末级节点包含：

| 属性 | 说明 | 示例 |
|------|------|------|
| **构件ID** | 唯一标识 | PA-001 |
| **构件名称** | 构件名称 | 考核计划管理界面 |
| **功能角色** | 前端/后端/数据/基础设施 | 前端 |
| **定义** | 构件的职责和用途描述 | 提供考核计划的创建、编辑、发布和关闭功能 |
| **承接的系统需求** | 承接了哪些SysReq 9级节点 | SysReq-F-001.1.1.1 |
| **符合性分析** | 如何满足所承接的系统需求 | 通过界面交互完整覆盖考核计划的全生命周期管理 |
| **实现方案** | 具体的技术实现方案 | Vue 3 + TypeScript 前端，Go RESTful API 后端 |
| **质量指标** | 该构件需要满足的质量目标 | 页面响应 < 2s，可用性 99.9% |

### 2.2 层级结构

产品架构的层级根据技术架构和业务领域划分，一般遵循以下模式：

| 层级 | 名称 | 说明 |
|------|------|------|
| **0级** | 产品根节点 | 整个产品的顶层 |
| **1级** | 技术架构分层 | 前端应用/后端服务/数据存储/基础设施 |
| **2级** | 子系统/模块 | 按业务领域划分 |
| **3级** | **构件（承接层）** | **★ 末级，承接SysReq 9级节点** |

> **注**：产品架构的层级数可根据产品复杂度适当调整。

### 2.3 构件分类

AOS（企业运营体系）作为软件产品，按技术架构和业务领域分类：

| 类别 | 子类型 | 说明 | 构件示例 |
|------|--------|------|---------|
| **前端应用** | 管理控制台 | 运营考核管理操作界面 | 考核计划管理界面、评分录入界面 |
| | 数据看板 | 数据展示和分析界面 | 绩效报告查看界面、运营数据看板 |
| **后端服务** | 业务服务 | 核心业务逻辑处理 | 考核计划服务、评分计算服务、报告生成服务 |
| | 支撑服务 | 基础协作功能 | 组织人员同步服务、权限管理服务 |
| **数据存储** | 业务数据 | 核心业务数据持久化 | 考核计划数据库、评分结果数据库 |
| | 基础数据 | 共享基础数据持久化 | 组织人员数据库、指标库 |
| **基础设施** | 运行环境 | 应用运行基础设施 | 应用服务器集群、负载均衡器 |
| | 可观测性 | 监控、日志和告警 | 监控告警系统、日志采集系统 |

### 2.5 构件间关系定义

AOS 作为软件产品，构件间关系如下：

| 关系类型 | 说明 | 示例 |
|---------|------|------|
| **依赖** | 构件A的运行依赖构件B | 评分录入界面依赖评分录入服务提供的 API |
| **调用** | 构件A调用构件B的接口 | 评分录入服务调用评分计算服务获取计算结果 |
| **组合** | 构件A由构件B和C组成 | 考核管理服务 = 考核计划服务 + 考核配置服务 |
| **关联** | 构件A与构件B存在数据结构关联 | 考核计划数据库与评分结果数据库通过 planId 外键关联 |

### 2.6 构件划分原则

构件划分的核心原则是**高内聚、低耦合**——每个构件应有明确的职责边界。

```
✅ 合适的构件划分：
  PA-003: 评分计算服务（一个完整的业务模块，独立部署单元）
  PA-007: 考核计划数据库（一个完整的数据存储单元，独立 schema）
  → 每个构件是一个独立可部署/可管理的单元

❌ 不合适的构件划分：
  PA-001: 所有后端服务合并为一个单体（粒度过粗，难以独立演进和部署）
  PA-002: 评分计算中的"加权平均算法"（粒度过细，应作为服务内部方法而非独立构件）
  → 粒度过粗导致紧耦合，粒度过细导致分布式通信开销
```

## 三、详细定义规范

### 3.1 详细定义的本质

产品架构详细定义是对每个产品构件进行深入的实现方案描述，使开发团队能够直接进入实现阶段。

对软件产品（AOS）而言，每个构件的详细定义包含：

| 内容 | 说明 | 示例（以"评分计算服务"构件为例） |
|------|------|------|
| **技术栈** | 编程语言、框架、数据库、中间件 | Go 1.22 + Gin + PostgreSQL + Redis + K8s |
| **接口定义** | 对外提供的 API（请求/响应格式） | RESTful API（swagger 文档）、gRPC 内部调用 |
| **数据模型** | 核心数据结构和数据流 | ScoreResult, ScoringRule, CalculationLog |
| **部署方式** | 容器化、实例数、资源需求 | Docker + K8s Deployment，3+ Pod |
| **性能指标** | 响应时间、吞吐量、一致性要求 | P99 < 2s，> 100 TPS，最终一致性 |
| **安全要求** | 鉴权方式、数据加密、合规 | RBAC + JWT，TLS 1.3，审计日志 |
| **监控告警** | 指标采集、告警规则、日志聚合 | Prometheus + Grafana，ELK Stack |

### 3.2 详细定义模板

```yaml
PA-003: 评分计算服务
├─ 技术栈
│   ├─ 编程语言: Go 1.22
│   ├─ 框架/库: Gin, GORM, Redis Client, gRPC
│   ├─ 数据库: PostgreSQL 15 + Redis 7
│   ├─ 消息队列: RabbitMQ
│   └─ 部署: Docker + Kubernetes (K8s)
├─ 接口定义
│   ├─ RESTful API:
│   │   ├─ POST /api/v1/scores/calculate
│   │   │   ├─ 请求体: { planId, employeeIds[], period, trigger }
│   │   │   └─ 响应体: { status, data: { taskId, totalRecords }, errors[] }
│   │   ├─ GET /api/v1/scores/{employeeId}?planId=X&period=Y
│   │   │   └─ 响应体: { employeeId, planId, scores[], totalScore, rank }
│   │   └─ POST /api/v1/scores/batch-calculate
│   │       └─ 异步任务，返回 taskId（通过 WebSocket 推送完成通知）
│   ├─ gRPC 内部接口:
│   │   ├─ rpc GetScoringRule(PlanIdRequest) returns (ScoringRuleList)
│   │   └─ rpc ValidateScore(ScoreValidationRequest) returns (ValidationResult)
│   └─ 事件:
│       ├─ ScoreCalculated { employeeId, planId, totalScore, period }
│       └─ CalculationFailed { taskId, errorCode, errorMessage }
├─ 数据模型
│   ├─ ScoreResult:
│   │   ├─ id: UUID (PK)
│   │   ├─ employeeId: UUID (FK → employees.id)
│   │   ├─ planId: UUID (FK → appraisal_plans.id)
│   │   ├─ period: VARCHAR(7)  -- "2026-05"
│   │   ├─ dimensionId: UUID (FK → scoring_dimensions.id)
│   │   ├─ rawScore: DECIMAL(5,2)
│   │   ├─ weight: DECIMAL(3,2)
│   │   ├─ weightedScore: DECIMAL(5,2)
│   │   ├─ calculatedAt: TIMESTAMP
│   │   └─ status: ENUM(pending, calculated, verified, disputed)
│   ├─ ScoringRule:
│   │   ├─ ruleId: UUID (PK)
│   │   ├─ type: ENUM(weighted_average, direct_sum, custom_formula)
│   │   ├─ formula: TEXT
│   │   └─ parameters: JSON
│   └─ CalculationLog:
│       ├─ logId: UUID (PK)
│       ├─ taskId: UUID
│       ├─ input: JSON
│       ├─ output: JSON
│       ├─ status: ENUM(success, failed)
│       └─ errorMessage: TEXT
├─ 部署方式
│   ├─ 部署形态: Docker 容器 + K8s Deployment
│   ├─ 实例数: 3+ Pod (HPA: CPU > 70% 自动扩容)
│   ├─ 资源需求: 2 vCPU / 4 GB RAM per Pod
│   ├─ 配置管理: ConfigMap + Vault（敏感数据）
│   └─ 网络策略: 仅允许评分录入服务和报告服务访问
├─ 性能指标
│   ├─ 响应时间: P99 < 2s（千人员工规模单次计算）
│   ├─ 吞吐量: > 100 TPS（并发评分计算请求）
│   ├─ 数据一致性: 最终一致性（异步计算场景）
│   ├─ 定时任务: 月度考核结果 T+1 内完成全量计算
│   └─ 离线容量: 支持万人规模的月度批量计算
├─ 安全要求
│   ├─ 访问控制: RBAC（仅考核管理员和系统管理员可用）
│   ├─ 数据加密: TLS 1.3 传输加密
│   ├─ 审计日志: 所有评分计算操作记录操作人、时间戳和变更内容
│   └─ 合规: 评分数据保留 7 年，支持数据导出
└─ 监控告警
    ├─ 指标采集:
    │   ├─ 业务指标: 计算请求量、成功率、平均处理时间
    │   ├─ 系统指标: CPU 使用率、内存、GC 暂停时间
    │   └─ 采集工具: Prometheus (/metrics 端点)
    ├─ 告警规则:
    │   ├─ 错误率 > 1% 持续 5 分钟 → P2 告警（钉钉/邮件通知）
    │   ├─ P99 延迟 > 5s 持续 3 分钟 → P2 告警
    │   ├─ 服务宕机 > 1 分钟 → P1 告警（电话+短信通知）
    │   └─ 批量计算失败率 > 5% → P2 告警
    └─ 日志聚合: ELK Stack（结构化 JSON 日志，按服务名索引）
```

### 3.3 详细定义深度对比

```
✅ 充分的详细定义：
  技术栈: Go 1.22 + Gin + PostgreSQL 15 + Redis 7 + K8s
  接口定义: RESTful API（含请求/响应格式）+ gRPC 内部接口
  数据模型: 含核心表结构、字段定义和外键关系
  部署方式: Docker + K8s、Pod 资源配置、HPA 策略
  性能指标: P99 < 2s、> 100 TPS、万人规模容量
  安全要求: RBAC + TLS 1.3 + 审计日志 + 合规保留
  → 开发工程师可基于此直接进入编码实现，无需额外沟通

❌ 不充分的详细定义：
  技术栈: Go
  接口定义: 待定
  数据模型: 待定
  性能指标: "高性能"
  → 信息太少，开发工程师需要反复确认技术选型和接口细节
```

## 四、设计方法

### 4.1 总体设计流程

```
步骤1：分析系统需求
  └─ 收集所有SysReq 9级节点和SysReq-NFR
     └─ 识别需求之间的关联关系
        └─ 输出：SysReq 9级清单 + NFR约束清单

步骤2：识别构件候选
  └─ 按功能相关性分组SysReq 9级节点
     └─ 按前端/后端/数据/基础设施归类
        └─ 评估构件的粒度和边界

步骤3：建立架构骨架
  └─ 采用技术架构分层（前端/后端/数据/基础设施）
     └─ 创建构件占位符，建立SysReq→PA初步映射
        └─ 输出：架构骨架 + 映射关系

步骤4：详细构件设计
  └─ 定义每个构件的技术栈、接口定义和数据模型
     └─ 定义构件间的依赖和调用关系
        └─ 输出：完整的架构定义 + 详细定义

步骤5：非功能约束验证
  └─ 检查每个NFR是否有对应的构件设计体现
     └─ 验证质量指标是否满足NFR要求
        └─ 输出：NFR符合性检查报告

步骤6：映射验证
  └─ SysReq→PA覆盖率 100%、1:1约束 100%
     └─ 输出：映射验证报告
```

### 4.2 两阶段工作流程

```
第一阶段：骨架建立（快速，1-2小时）
  1. 扫描SysReq 9级节点
  2. 创建PA构件占位符
  3. 建立SysReq→PA骨架映射
  4. 冻结骨架结构
  输出：构件占位符 + 映射关系

第二阶段：详细填充（深入，2-4天）
  1. 详细定义每个构件的实现方案
  2. 填充技术栈、接口定义、数据模型
  3. 验证非功能约束的可满足性
  4. 验证映射的完整性和一致性
  5. 冻结产品架构文档
  输出：完整的产品架构定义 + 详细定义
```

> **与准则的关系**：本节描述的是本层内部的两阶段工作流程。准则 §1.5.3 定义的"两阶段工作流程"是跨层级的：阶段一快速约定 PA 结构骨架并建立 SysReq→PA 映射关系（如 §4.3 所述），阶段二再并行填充 SysReq 详细定义和 PA 详细定义的详细内容。本节的"第一阶段：骨架建立"对应准则的阶段一在本层的操作，"第二阶段：详细填充"对应准则的阶段二中本层的操作。

### 4.3 同步设计

产品架构的初步结构应与系统需求详细定义**同步开展**。

**实际操作**：

```
1. SysReq 5级节点确定后 → 快速建立PA结构骨架（占位符）
2. SysReq 9级节点确定时 → 同步映射到PA构件
3. 冻结映射关系后 → 并行填充详细内容
4. 非功能约束 → 在构件设计阶段同步验证
```

**为什么同步**：SysReq 9级场景活动需要分配到PA构件。如果PA尚未建立，SysReq 9级分解就没有分配目标，可能导致分解粒度过粗或过细而需要返工。

### 4.4 非功能权衡决策

当多个非功能需求冲突时，需要进行权衡决策。AOS 常见权衡场景如下：

| 冲突场景 | 权衡决策 | 记录方式 |
|---------|---------|---------|
| 性能 vs 安全 | 传输加密增加延迟 | 记录最低加密等级要求和P99延迟上限 |
| 可用性 vs 成本 | 多活部署提升可用性但增加资源成本 | 记录可用性 SLA（99.9%）和成本预算上限 |
| 可维护性 vs 性能 | 通用查询接口降低编码量但可能影响查询性能 | 记录性能阈值和查询接口优化策略 |

### 4.5 设计原则

**N:1承接支持**：一个PA构件可承接多个SysReq 9级节点。2-5个合理，1个需考虑合并，6个以上需考虑拆分。

**1:1分配约束**：每个SysReq 9级节点分配到PA中唯一的构件。如果需要多个构件，说明9级活动还需继续分解。

**有序组织**：按技术架构分层（前端→后端→数据→基础设施）。核心要求：职责清晰、独立演进、便于复用。

**非功能约束满足**：每个PA构件标注受哪些NFR约束，构件的质量指标不低于SysReq的要求。

**组织资产优先原则**：新增PA构件前优先复用或扩展现有构件。信息处理模式差异是不可逾越的边界——即使资产优先，不同模式的需求不能强行塞入同一构件。详见[AOS系统设计准则 §3.3](01-aos-system-design-standards.md#33-组织资产优先原则)。

## 七、常见错误和最佳实践

### 7.1 常见错误

**❌ 错误1：构件粒度过大或过小**

```
错误的做法：
PA-001: 所有后端服务合并为一个大单体（粒度过粗，难以独立演进）
PA-002: 评分计算中"加权平均算法"（粒度过细，应作为方法而非独立构件）

问题：粒度过粗导致紧耦合和部署困难，粒度过细导致分布式通信开销
后果：团队无法独立开发部署，微服务治理复杂化
```

```
✅ 正确做法：
PA-001: 评分计算服务（粒度适中——一个完整的业务模块，独立部署单元）
PA-002: 考核计划数据库（粒度适中——一个完整的数据存储单元）
→ 每个构件是一个独立可部署/可管理的单元，职责边界清晰
```

**❌ 错误2：映射关系不完整**

```
错误的做法：
SysReq有20个9级节点，但PA只定义了15个构件承接
→ 5个SysReq 9级节点未映射

问题：部分功能需求没有对应的构件实现
后果：开发时遗漏功能或临时添加导致架构混乱
```

```
✅ 正确做法：
建立一个SysReq 9级 → PA构件的完整映射表
逐条验证：每个SysReq 9级都有对应的PA构件
→ 映射覆盖率 100%
```

**❌ 错误3：忽略非功能约束**

```
错误的做法：
PA构件定义中只写了业务逻辑描述，没有质量指标
→ "评分计算服务：提供评分计算功能"

问题：开发时不知道性能和安全要求
后果：上线后性能不达标需要重构
```

```
✅ 正确做法：
PA构件定义中包含明确的质量指标
→ "评分计算服务：P99 < 2s，> 100 TPS，RBAC 权限控制"
→ 开发时有明确的目标
```

**❌ 错误4：忽略资产优先原则，动辄新增PA构件**

```
错误的做法：
新的SysReq 9级活动到达时，直接在PA中新增构件来承接，
而不检查现有PA构件是否能扩展后承接。

问题：忽视已有PA设计资产，导致构件数量膨胀
后果：构件数量失控，产品架构过于复杂，维护成本上升
```

```
✅ 正确做法：
按 复用→改进→新增 的优先级依次检查现有PA构件。
仅当复用和改进都不可行时，才新增构件。
说明：保护组织已有设计资产，控制产品架构复杂度
```

### 7.2 最佳实践

**实践1：先骨架后填充**：先快速建立架构骨架（占位符），同步建立映射关系，再并行填充详细内容。

**实践2：构件以"独立部署单元"为粒度**：每个构件应是可独立部署和验证的最小单元。以"一个模块可在 1-2 周内由 2-3 人完成开发"为标准。

**实践3：将NFR纳入构件定义**：每个构件定义中都包含可测量、可验证的质量指标，在详细定义中说明NFR的满足方案。

**实践4：记录架构决策**：关键技术选择记录决策理由，NFR权衡决策记录在案。

## 八、FAQ

**Q1：产品架构的层级数是固定的吗？**

不是。产品架构的层级可根据产品复杂度适当调整。简单产品可能只有 2 级（根节点 → 构件），复杂产品可能需要 4 级或更多。关键是**保持层级的一致性**——同层节点应在同一抽象级别。

**Q2：一个PA构件承接多少个SysReq 9级节点合理？**

建议每个PA构件承接 2-5 个 SysReq 9 级节点（2-3 条最优，1 条需考虑合并，4-5 条可接受需检查过度聚合，6 条以上说明粒度过粗需考虑拆分）。承接 0 个说明构件可能多余。

**Q3：构件划分到多细才算合适？**

以"一个模块可在 1-2 周内由 2-3 人完成开发"和"可独立部署和测试"为双重标准。如果构件太大（超过 1 个月开发量）、版本耦合过紧，说明还需拆分。

**Q4：前端和后端构件可以分布在不同的层级中吗？**

可以。前端构件属于"前端应用"分支，后端构件属于"后端服务"分支，它们是同层（3级）的不同分类。同层意味着它们都直接承接 SysReq 9 级节点，但不一定属于同一技术领域。

**Q5：性能和安全冲突时怎么办？**

记录权衡决策。例如"评分结果需要传输加密（安全要求）但全量加密导致延迟超出 P99 上限（性能要求）。决策：评分结果负载中敏感字段（如个人评分明细）使用加密传输，聚合结果使用非加密传输以减少计算开销。"
