## 一、跨步骤上下文管理

### 1.1 上下文传递方式

6 步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

### 1.2 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**（见本章末尾）中，供后续步骤引用。

### 1.3 回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见 AI决策框架 §7.2 跨步骤影响范围表）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

## 二、AI 经济原则

本步操作以下列顺序执行，AI 需注意：

1. **执行顺序不可逆**：分析活动 → IPO 去重 → BA→SysReq 映射（同步）→ 架构组织 + 闭环检查 → PA 骨架
   核心驱动力：BA IPO 去重后直接映射到 SysReq 架构。SysReq 架构的调整（扩展/新增）由 BA IPO 驱动，不是预设的
2. **基于资产优先**：SysReq 架构调整决策应先检查已有资产（t02 骨架 + 现有 SysReq 架构），确实需要修改时再改进，最后才新增
3. **资产路径已知**：本步的资产是 `90-eos-pipeline-system-product-data/05-eos-pipeline-system-requirements-architecture.md`（含 v1 基线区）+ t02 同步建立的 SysReq 骨架映射。`04-eos-pipeline-business-architecture.md` 是本步的输入数据（待处理的 BA IPO），不是"资产"
4. **BA→SysReq 的特性**：与 t01（OR→SR N:1 分解）和 t02（SR→BA N:1 分配）不同，BA→SysReq 是**去重后的直接映射**（通用准则 §5.5），而非 N:1 分配。但"找宿主"的决策逻辑仍然适用——判断去重后 IPO 应分配到现有 SysReq 节点还是需新建节点

### 3.2 IPO 去重（三级策略确认版）

```
请对业务架构 IPO 清单执行最终去重确认。

业务架构 IPO 清单：[粘贴上一步IPO清单的输出]

三级去重策略：
第1步：检查 — 跨所有BA流程，检查是否有相同或相似的IPO
第2步：复用 — 完全相同的IPO合并为一个（记录所有承接来源）
第3步：新增 — 确实唯一的IPO保留独立

⚠️ IPO去重判断标准：
  - "完全重复" → 合并
  - "部分重复" → 拆分为通用+特化
  - "独立" → 保持独立

输出格式：
去重前IPO数：[N]
去重后IPO数：[M]
去重映射：
| 原始BA IPO | 去重后IPO | 去重说明 |
|-----------|----------|---------|
| BA-001    | IPO-去重-001 | 合并（与BA-002的"[功能]"相同） |
| BA-002    | IPO-去重-001 | 合并（与BA-001的"[功能]"相同） |
| BA-003    | IPO-去重-002 | 唯一，保留 |
```

### 3.3 BA IPO → SysReq 5 级（资产优先 + 同步映射）

> 先了解资产再动手——映射前加载现有 SysReq 架构（含 t02 骨架），理解每个 SysReq 5 级节点的职责边界。然后面对每条去重后 IPO 时，已经知道"哪个方向有节点、哪个方向还空着"，有方向地去映射。

```
请先加载现有 SysReq 架构资产（含 t02 骨架），理解其分类结构和各 5 级节点的职责边界后，再逐条处理去重后 IPO。

去重后 IPO 清单：[粘贴3.2的去重后IPO清单]

—— 第一步：加载并理解 SysReq 架构资产（先做到心中有数）——

资产来源一：t02同步建立的 SysReq 骨架映射（系统需求功能部分架构骨架）
├─ 已有骨架节点名称和占位符
├─ 已有 BA → SysReq 的骨架映射关系
└─ 骨架节点的分类归属（属于哪个功能模块）

资产来源二：05-eos-pipeline-system-requirements-architecture.md（v1 基线·只读）
├─ 加载所有现有 SysReq 5 级节点
├─ 理解每个节点的职责边界和覆盖范围
└─ 评估现有节点与 BA IPO 的匹配程度

整体理解：
├─ SysReq 功能架构按什么逻辑组织？（按功能模块？按产品构件？）
├─ 哪些功能区域已有节点覆盖？哪些还空着？
└─ 现有节点粒度是否适合 t05 详细分解（2-4 条 6-9 级末级）

—— 第二步：逐条去重后 IPO 找 SysReq 5 级宿主 ——

对每条去重后 IPO，执行以下操作：

0. 【前置判断】这条 IPO 涉及多个 SysReq 功能范围吗？
├─ 例如：IPO 同时描述"准则开发"和"模板维护" → 可能分属不同 SysReq 分类
├─ 如果是 → 先按涉及的功能范围分拆，再分别执行下面的步骤
└─ 如果不是 → 直接进入步骤1

1. 【找宿主】用三个维度判断：

   ├─ Q1 功能分类匹配：此 BA IPO 的核心功能意图，与哪个 SysReq 5 级节点的职责方向一致？
   │  例："准则开发 IPO" — SysReq-F-准则类? SysReq-F-模板类? SysReq-F-质量类?
   │  判断依据：SysReq 5 级节点定义中描述的功能范围和职责
   │
   ├─ Q2 覆盖范围偏差：此 BA IPO 描述的具体内容，是否在 SysReq 5 级节点定义的边界之内？
   │  例：SysReq 5 级节点定义是"准则体系管理功能" → 
   │  "模板格式规范化"可能在"准则体系管理"的边界之内（模板是准则的一种交付形态）
   │
   └─ Q3 同类承接情况：此 SysReq 5 级节点是否已经在承接同类 BA IPO？
      例：某 SysReq 5 级节点已承接多个"准则开发"类 IPO
      → 它在承接同类 IPO，职责范围一致

   根据三个问题的答案做决策：
   ├─ 条件A — 复用骨架关系（或现有节点）：
   │  Q1命中 + Q2在边界内 + Q3同类
   │  → BA-XXX → SysReq-F-YYY（复用骨架或现有节点）
   │  例："准则开发 IPO" → SysReq-F-准则管理（已在骨架中）
   │
   ├─ 条件B — 扩展 SysReq 节点后承接：
   │  Q1命中 + Q2略超边界 + Q3同类
   │  → 标记 SysReq-F-YYY（改进-扩展），记录扩展理由
   │
   ├─ 条件C — 新增 SysReq 5 级节点：
   │  Q1无命中 + 不在任何 SysReq 节点边界内
   │  → 记录需新增 SysReq 5 级节点，写明新增原因和分类归属
   │
   └─ 条件D — ❓ 无法判断：
       Q1命中了多个方向 / Q2边界完全模糊 / Q3信息不足以对比
       → 标记❓，说明卡在哪个维度上，**不要强行推测**，留给人类

2. 【记录】同步记录本步决策：
├─ 承接的 SysReq 5 级：[SysReq-F-YYY / 新增]
├─ 资产决策：[复用骨架 / 改进-扩展 / 新增]
└─ 决策理由：引用以上三个判断维度的结果

⚠️ 关键判断原则：
- "功能分类匹配"的含义：BA IPO 的业务活动类型与 SysReq 节点的功能分类对应
  （如"准则开发"→准则管理类功能、"质量检查"→质量管理类功能）
- BA→SysReq 是直接映射而非 N:1 分配，所以同一 IPO 只映射到唯一 SysReq 5 级节点
- ⚠️ **扩展和新增的强内聚性检查**：
  ├─ 扩展前问"扩展后这个 SysReq 节点还描述一个单一系统功能吗？"
  │  例：某节点负责"准则管理"，扩展为"准则+模板+模式管理"
  │  → 三个活动不同质，扩展后会变"杂"，应新增节点而非扩展
  ├─ 新增前问"这个新节点和同级兄弟节点的颗粒度一致吗？"
  │  例：相邻节点都是"某个系统功能模块"的粒度，新节点不能是一个"子系统"
  └─ 颗粒度标准：该 5 级节点应可自然分解为 2-4 条 6-9 级详细末级（t05 可操作）

—— 输出 ——

BA→SysReq 映射清单：
| 去重后 IPO | 描述 | 承接的 SysReq 5级 | 资产决策 | 决策理由（Q1/Q2/Q3） |
|-----------|------|-----------------|---------|--------------------|
| IPO-001   | [描述] | SysReq-F-YYY   | 复用骨架 ✅ | Q1命中[分类]，Q2在边界内，骨架已有映射 |
| IPO-002   | [描述] | SysReq-F-ZZZ   | 改进-扩展 ⚠️ | Q1命中，Q2略超[范围]，需扩展节点定义 |
| IPO-003   | [描述] | [新增]         | 新增 ⚠️ | Q1无命中，无SysReq节点职责方向匹配 |
| IPO-004   | [描述] | —              | ❓ 无法判断 | Q1命中多个[SysReq-F-AAA][SysReq-F-BBB]，无法确定主方向 |

资产决策汇总：
├─ 复用骨架/现有节点 ✅：[N个] — SysReq-F-XXX, SysReq-F-YYY, ...
├─ 改进（扩展）节点 ⚠️：[N个] — SysReq-F-ZZZ（扩展[内容]）
├─ 新增节点 ⚠️：[N个] — 在[分类]下新增，用于承接[IPO清单]
├─ ❓ 待人类裁定：[N个] — [IPO编号] + [卡在哪个维度]
└─ 未承接的 IPO：[清单]（理论上应为空）
```

### 3.4 SysReq 功能架构组织 + 闭环检查

> 本步产出 SysReq 功能架构定义子文档（05-*.md），遵循**详概分离**原则：每个 5 级架构末级节点只写四要素（定义/映射/符合性分析/设计指导），不写入 6-9 级详细内容（那是 t05 的工作）。

```

BA→SysReq 映射结果：[粘贴3.3的输出]

⚠️ 资产文件分区规则：
- v1 基线区（已冻结·只读）— **禁止修改**。复用节点直接引用 v1 定义
- v2 新增区（活跃·当前操作）— 扩展定义和新增节点写在此区
  ├─ 改进（扩展）节点 → v2 区补充扩展后的定义，标注"基于v1的SysReq-F-YYY扩展"
  ├─ 新增节点 → v2 区创建新节点条目
  └─ 复用节点 → 不写入新内容

—— 执行 ——

1. 按 0-9 级层级结构组织：
   0级：根节点
   1级：独立信息系统
   2级：功能需求分支

   4级：业务组件小类
   5级：组件实例 ←── **承接 BA IPO（架构末级）**
   6-9级：详细分解（空，留待 t05）

2. 完善所有 SysReq 5 级节点四要素

   SysReq-F-XXX: [名称]（5级·架构末级节点）
   ├─ 定义：[该节点定义的系统功能]
   ├─ 承接 BA：BA-XXX, BA-YYY（N:1）
   ├─ 符合性分析：[如何定性和定量满足所承接的 BA IPO]

   - 边界清晰：节点之间的职责不重叠

5. 性能指标继承检查

   - 指标数值是否不降级？

—— 输出 ——
```

## 五、审查修复流程

### 5.1 审查输入

AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

### 5.2 AI 修复流程

收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）→ 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题** → 参考AI决策框架的跨步骤影响范围表进行影响范围评估，列出所有受影响的下游节点

### 5.3 影响范围评估

| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 某步的架构定义 | 本层 + 下游已建立映射的节点 | 修改后同步更新映射 |
| 详细定义 | 下层方案 | 修改后检查下层是否受影响 |
| 映射关系 | 所有关联节点 | 需重新运行追溯验证 |
| 编号变更 | 全链路 | 需逐层更新引用，重新验证 |
