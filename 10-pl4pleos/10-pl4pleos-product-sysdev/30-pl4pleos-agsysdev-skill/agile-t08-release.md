> **概述**：完善增量更新后的文档，生成变更日志，发布新版本。

## 三、操作流程

本步按五个阶段的顺序执行：问题修复 → 变更汇总 → 生成日志 → 发布准备 → 终验。

核心驱动逻辑：先处理验证报告的问题，再汇总变更并生成发布物。确保所有问题闭环后再发布。

操作流程概览：

```
[阶段一] 审查修复               [阶段二] 变更汇总               [阶段三~四] 发布准备
  按验证报告逐项修复              按层分类汇总变更                 生成变更日志
  回溯到对应步骤                   影响分析摘要                    生成版本发布说明
[阶段五] 终验
  人类确认迭代完成
```

## 四、AI执行指南

### 4.1 跨步骤上下文管理

#### 上下文传递方式

8 步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

#### 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**（见§六）中，供后续步骤引用。

#### 回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见§七.2 跨步骤影响范围表）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

### 4.2 AI经济原则

本步操作以下列顺序执行，AI 需注意：

1. **执行顺序不可逆**：审查修复 → 变更汇总 → 生成日志 → 发布准备
   核心驱动力：先处理验证报告的问题，再汇总变更并生成发布物。确保所有问题闭环后再发布
2. **基于资产优先**：本次迭代的完整设计文档集为发布资产。v2 增量区与 v1 基线区合并，生成新的 v2 基线（冻结·只读），作为下一次迭代的资产
3. **增量设计特点**：本步是 8 步敏捷流程的最后一步，不涉及新设计工作。核心工作是：处理验证问题闭环、合并增量和基线、生成变更日志和版本发布说明。变更日志按层分类（OR/SR/BA/SysReq/PA），清晰记录本次迭代的范围

## 七、审查流程

### 7.1 审核要点

- [ ] 变更日志准确性（按层分类、无遗漏）
- [ ] 新版本号确认
- [ ] 验证报告问题闭环处理（所有问题已修复或标记）
- [ ] 所有文档版本号一致更新
- [ ] 超链接有效性（跨文档引用是否正确）

### 7.2 审查修复流程

#### 审查输入

AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

#### AI 修复流程

收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）→ 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题** → 参考AI决策框架的跨步骤影响范围表进行影响范围评估，列出所有受影响的下游节点

#### 影响范围评估

| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 某步的架构定义 | 本层 + 下游已建立映射的节点 | 修改后同步更新映射 |
| 详细定义 | 下层方案 | 修改后检查下层是否受影响 |
| 映射关系 | 所有关联节点 | 需重新运行追溯验证 |
| 编号变更 | 全链路 | 需逐层更新引用，重新验证 |
