# 完整性和一致性验证
**Skill ID**: agile-t07 · **Step 7: Completeness and Consistency Verification**

> **概述**：验证增量更新后的系统设计的完整性和一致性。
>
> **输入 → 输出**：task-01~task-06 的全部增量产出 → 增量验证报告（附录到 `09-*.md`）
>
> **适用场景**：敏捷式第7步。增量部分的全链路验证。
>
> **版本**：v1.1 | **修订**：2026-06-03

---

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：task-01~task-06 的全部增量产出，本次迭代前的完整基线文档
- **产出**：增量验证报告（附录到验证报告文档 `09-*.md`）

核心验证任务：
- 1:1 分配约束验证（增量部分）
- N:1 承接合理性验证（更新后）
- 双向追溯验证（增量部分）
- 增量一致性检查（有无"更新了上层漏了下层"）
- 版本一致性验证

### 与三条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **详概分离** | 仅验证增量部分的设计链一致性，不重新检查未受影响的基线内容 |
| **文档结对** | 验证 OR→SR→BA→SysReq→PA 全链路增量映射的完整性 |
| **基于资产** | 以本次迭代前的完整基线文档为基准对比增量产出 |

## 三、操作流程

本步按七个阶段的顺序执行：增量验证分析 → 1:1 约束验证 → N:1 承接验证 → 双向追溯验证 → 增量一致性检查 → 版本一致性验证 → 审查。

核心驱动逻辑：从增量产出出发，逐项验证设计链的完整性、一致性和正确性。验证范围限定在增量部分，不重新检查未受影响的基线内容。

操作流程概览：

```
[阶段一] 增量验证分析             [阶段二~六] 逐项验证             [阶段七] 审查
  需求说明符合性汇总               1:1 约束（增量）                 结果审核（人类审）
  需求性能符合性汇总               N:1 承接合理性
                                  双向追溯（增量）
                                  增量一致性
                                  版本一致性
```

## 四、AI执行指南

### 4.1 跨步骤上下文管理

#### 上下文传递方式

8 步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

#### 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**（见§六）中，供后续步骤引用。

#### 回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见§七.2 跨步骤影响范围表）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

### 4.2 AI经济原则

本步操作以下列顺序执行，AI 需注意：

1. **执行顺序不可逆**：增量验证分析 → 1:1 约束验证 → N:1 承接验证 → 双向追溯验证 → 增量一致性检查 → 版本一致性验证
   核心驱动力：从增量产出出发，逐项验证设计链的完整性、一致性和正确性。验证范围限定在增量部分，不重新检查未受影响的基线内容
2. **基于资产优先**：以本次迭代前的完整基线文档为基准，对比增量产出的正确性。资产文件分区（v1 基线区·只读 / v2 增量区·活跃）用于版本一致性验证
3. **增量设计特点**：本步是设计中唯一的纯验证步骤，不产出新设计内容。所有验证检查限定在增量部分（新增/修改的节点和映射），未受影响的基线内容仅做引用检查

## 七、审查流程

### 7.1 审核要点

- [ ] 验证报告未通过项（需人工确认修复方案）
- [ ] 增量一致性遗漏（"更新了上层漏了下层"）
- [ ] 1:1 约束回溯修复
- [ ] 版本一致性（所有文档版本号一致）

### 7.2 审查修复流程

#### 审查输入

AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

#### AI 修复流程

收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）→ 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题** → 参考AI决策框架的跨步骤影响范围表进行影响范围评估，列出所有受影响的下游节点

#### 影响范围评估

| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 某步的架构定义 | 本层 + 下游已建立映射的节点 | 修改后同步更新映射 |
| 详细定义 | 下层方案 | 修改后检查下层是否受影响 |
| 映射关系 | 所有关联节点 | 需重新运行追溯验证 |
| 编号变更 | 全链路 | 需逐层更新引用，重新验证 |
