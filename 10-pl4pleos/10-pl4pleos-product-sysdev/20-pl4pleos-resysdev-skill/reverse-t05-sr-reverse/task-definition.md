# 逆向工程流程第5步：相关方需求推导
**Reverse Engineering Step 5: Stakeholder Requirements Reverse Engineering**

**对应 AI 提示词**：`ai-prompts.md`（同目录）

## 一、目的

从 BA IPO 反向推导相关方需求（SR），包括架构定义和详细定义。非功能部分从 SysReq-NFR 反推（不经过 BA 层）。

本步在设计链中承担的角色：
- **输入**：业务架构文档（IPO 清单 + 去重报告），同步建立的相关方需求骨架
- **产出**：相关方需求架构定义（`02-*.md`），相关方需求详细定义（`03-*.md`）

### 本步与三条工程化设计原则的关系

| 原则 | 在逆向工程中的体现（本步） |
|------|--------------------------|
| **详概分离** | SR 层按架构 + 详细两层产出。架构定义（`02-*.md`）为树形结构 + 映射关系表；详细定义（`03-*.md`）为每条架构末级的详细分解。两条线并行：功能部分从 BA 反推，非功能部分从 SysReq-NFR 直接反推 |
| **文档结对** | BA IPO 与 SR 详细定义末级形成跨层结对。功能部分从 BA IPO 聚合到 SR-F 架构末级，非功能部分从 SysReq-NFR 独立推导 |
| **基于资产** | BA IPO 清单是推导 SR 的核心资产。从"需要什么业务步骤"反推"谁需要这些步骤"，即从业务操作特征推导各相关方对系统的期望 |

### 文档结对关系

本步执行 BA → SR 的跨层推导（功能部分）和 SysReq-NFR → SR-NFR 跨层推导（非功能部分）：

| 结对/关系 | 需求方 | 方案方 | 说明 |
|----------|--------|--------|------|
| 本步推导（功能） | BA IPO | SR-F 架构末级 | 从 IPO 聚合为 SR 需求描述 |

| 内部结对 | SR 架构定义 | SR 详细定义 | 架构→详细内部对偶 |

## 三、操作流程

本步按七个阶段的顺序执行：分析活动 → 反推 SR-F 架构 → SR 架构定义 → SR 详细分解 → NFR 推导 → 建映射 → 结果审核。

核心驱动逻辑：从"业务需要什么步骤"反推"谁在什么场景下提出什么需求"。功能部分从 BA IPO 反推，按 SR-F 分类聚合。

操作流程概览：

```
[阶段一] 分析活动         [阶段二] 反推SR-F架构     [阶段三] SR架构定义
 BA→SR追溯验证             从BA IPO聚合             树形结构+映射关系表

[阶段四] SR详细分解       [阶段五] NFR推导          [阶段六] 建映射
 每个末级节点详细分解       从SysReq-NFR反推          SR→BA追溯表

[阶段七] 结果审核
 人类确认SR合理性
```

## 四、审核要点

- [ ] SR 架构准确性
- [ ] 功能与非功能分离（SR-F 与 SR-NFR 是否正确分离）
- [ ] SR 详细定义 1:1 约束（每条详细末级追溯唯一 SR 架构末级）
- [ ] BA IPO 聚合的 SR 语义保持（聚合后是否保持了 BA 的业务语义）
- [ ] SR-NFR 量化指标合理性

## 五、质量门禁

- [ ] 分析活动已完成（BA→SR 追溯验证）
- [ ] 所有 BA IPO 已聚合到 SR-F 架构末级
- [ ] SR 架构定义完整（树形结构 + 映射关系表）
- [ ] SR 详细定义完成（每个架构末级已分解）
- [ ] SR-NFR 已从 SysReq-NFR 反推完成
- [ ] SR → BA 映射关系已建立
- [ ] 运行 rule-conformance-review 进行符合性审查，按审查修复流程处理问题
