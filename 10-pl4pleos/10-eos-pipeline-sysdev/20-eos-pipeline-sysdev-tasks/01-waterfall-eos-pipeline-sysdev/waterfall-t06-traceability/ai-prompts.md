## 一、跨步骤上下文管理

### 1.1 上下文传递方式

6 步流程中，每一步的产出是下一步的核心输入。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 合并/拆分、分类方式、层级结构 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

### 1.2 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**（见本章末尾）中，供后续步骤引用。

### 1.3 回溯修正

如果在第 N 步发现需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见 AI决策框架 §7.2 跨步骤影响范围表）
2. 定位到受影响的具体步骤和节点
3. 修改后检查下游映射是否需要同步调整
4. 标注变更记录

## 二、AI 经济原则

本步操作以下列顺序执行，AI 需注意：

1. **执行顺序不可逆**：分析活动 → 加载资产 → 1:1 验证 → N:1 验证 → 双向追溯验证（含 Q1/Q2/Q3）→ 符合性分析验证 → 产出矩阵+报告 → 闭环检查
   核心驱动力：先理解有哪些资产再验证（先加载后检查），先验证单个约束（1:1）再验证整体链路（双向追溯）
2. **基于资产优先**：验证前必须先加载全部 9 份产品数据文件和 5 份决策记录，理解各层节点和映射关系后再开始逐项验证
3. **资产路径已知**：本步的资产是 `90-eos-pipeline-system-product-data/` 下 01~09 号文件和 `04-eos-pipeline-system-design-tasks/` 下的决策记录。全部为输入数据，无论 v1 基线还是 v2 新增区
4. **t06 与 t01~t05 的本质区别**：t01~t05 是"创建/定义"步骤，t06 是"验证"步骤。t06 不产生新的架构节点或映射，而是检查已有映射的正确性和完整性。发现的断点不在此步修复，而是记录到验证报告中供回溯修正

```
请验证 N:1 承接的合理性。

验证范围（功能分支）：
1. 每个 SR 架构末级节点承接了多少条 OR 详细末级？
2. 每个 BA IPO 承接了多少条 SR 详细定义末级？
3. 每个 SysReq 5 级节点承接了多少个 BA IPO（去重后）？
4. 每个 PA 组件承接了多少个 SysReq 9 级节点？

验证范围（非功能分支）：
5. 每个 SysReq-NFR 5 级节点承接了多少条 SR-NFR？

合理范围：2-5 条/节点
- 1 条：⚠️ 偏少，检查是否需要合并
- 2-3 条：✅ 最优
- 4-5 条：✅ 可接受，检查是否有过度聚合
- 6 条以上：⚠️ 偏多，建议拆分

输出格式：
| 节点 | 承接数 | 评价 | 标记 | 建议 |
|------|-------|------|------|------|
| SR-F-XXX | 3 | ✅ 最优 | ✅ | - |
| PA-XXX | 1 | ⚠️ 偏少 | ⚠️ | 考虑合并 |
| SysReq-F-YYY | 7 | ⚠️ 偏多 | ⚠️ | 建议拆分为2个节点 |

⚠️ 承接1条 → 检查是否可合并；承接6+条 → 建议拆分并给出拆分方案
```

### 3.4 双向追溯验证（含 Q1/Q2/Q3）

> 先加载资产再验证——在 3.1 已加载全部资产、3.2~3.3 已完成逐层约束检查的基础上，进行正向和反向的链路完整性验证。功能分支和非功能分支分别执行。

```
请在已完成 1:1 和 N:1 验证的基础上，执行双向追溯验证。

加载的资产概览：[引用3.1的输出]
1:1 验证结果：[引用3.2的输出]
N:1 验证结果：[引用3.3的输出]

—— 执行 ——

从每个起点开始追踪，三个维度逐环节判断：

1. 【正向追溯】从 OR 出发，沿需求分解链向 PA 方向追踪：

   功能分支验证路径：OR → SR → BA → SysReq-F → PA
   非功能分支验证路径：SR-NFR → SysReq-NFR → PA

   对每个映射环节（如 OR→SR），用三个维度判断：

   ├─ Q1 正向链路完整性：出发节点是否标注了分配目标（→下层方案节点）？
   │  例：OR-001 节点 → 标注了 SR-F-001 且 SR-F-001 存在
   │  判断依据：节点是否包含"分配"、"承接来源"、"所对应的"等映射标注字段，
   │  且目标节点在方案方文档中确实存在
   │  如未标注分配目标 或 目标节点不存在 → ❌ 链路中断
   │
   ├─ Q2 反向链路完整性：目标节点是否标注了承接来源（←上层需求节点）？
   │  例：SR-F-001 节点 → 标注了"承接 OR：OR-001"且 OR-001 存在
   │  判断依据：节点是否包含"承接"、"来源于"等映射回上层需求的字段，
   │  且来源节点在需求方文档中确实存在
   │  如未标注承接来源 或 来源节点不存在 → ❌ 链路中断
   │
   └─ Q3 跨层一致性：出发节点和目标节点的语义描述是否一致？
      例：OR-001 描述"系统应支持 X 功能"→ SR-F-001 描述"X 功能的管理"
      → 语义一致，指标继承不降级 ✅
      OR-001 描述"响应时间 < 200ms"→ SR-F-001 描述"响应时间 < 500ms"
      → 指标降级 ⚠️，需确认
      判断依据：节点定义之间的语义匹配度，性能指标数值不变或更高

   根据三个问题的答案做决策：
   ├─ ✅ 链路完整 — Q1/Q2/Q3 均通过
   │  → 记录 OR→SR 追溯通过
   │
   ├─ ⚠️ 待确认 — Q1/Q2 通过但 Q3 语义略有偏差或指标降级
   │  → 记录映射关系存在但语义匹配度存疑，标记 ⚠️ 待人工确认
   │
   ├─ ❌ 链路中断 — Q1 或 Q2 未通过（缺少映射标注 或 目标/来源节点不存在）
   │  → 记录明确的链路断点，标记 ❌ 建议回溯修复
   │
   └─ ❓ 无法判断 — 信息不足以判断映射正确性
       → 标记 ❓，说明卡在哪个维度上，**不要强行推测**，留给人类

2. 【反向追溯】从 PA 出发，沿实现链向 OR 方向追踪：

   功能分支验证路径：PA → SysReq-F → BA → SR → OR
   非功能分支验证路径：PA → SysReq-NFR → SR-NFR

   同样适用 Q1/Q2/Q3 三维度判断标准（方向相反但判断逻辑相同）：
   ├─ Q1 反向链路完整性：方案节点是否标注了承接来源（←上层需求）？
   ├─ Q2 正向链路完整性：需求节点是否标注了分配目标（→下层方案）？
   └─ Q3 跨层一致性：语义是否一致、指标是否不降级？

—— 输出 ——

正向追溯：
├─ OR → SR：[通过率 N%，断点清单]
│  ├─ ✅ 链路完整：N 条
│  ├─ ⚠️ 待确认：N 条（编号+问题）
│  ├─ ❌ 链路中断：N 条（编号+问题）
│  └─ ❓ 无法判断：N 条（编号+原因）
├─ SR → BA：[通过率 N%，断点清单]
├─ BA → SysReq-F：[通过率 N%，断点清单]
├─ SysReq-F → PA：[通过率 N%，断点清单]
├─ SR-NFR → SysReq-NFR：[通过率 N%，断点清单]
└─ SysReq-NFR → PA：[通过率 N%，断点清单]

反向追溯：
├─ PA → SysReq-F：[通过率 N%，断点清单]
├─ SysReq-F → BA：[通过率 N%，断点清单]
├─ BA → SR：[通过率 N%，断点清单]
├─ SR → OR：[通过率 N%，断点清单]
├─ PA → SysReq-NFR：[通过率 N%，断点清单]
└─ SysReq-NFR → SR-NFR：[通过率 N%，断点清单]

符合性分析完整率：[百分比]
```

## 五、审查修复流程

### 5.1 审查输入

AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性）

### 5.2 AI 修复流程

收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、格式不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（某节为空、映射不完整）→ 补充内容，标注修复说明
3. **结构性问题**（粒度不匹配、分类错误）→ 提出修复方案，标注 ⚠️ 待确认
4. **跨步骤一致性问题** → 参考AI决策框架的跨步骤影响范围表进行影响范围评估，列出所有受影响的下游节点

### 5.3 影响范围评估

| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 某步的架构定义 | 本层 + 下游已建立映射的节点 | 修改后同步更新映射 |
| 详细定义 | 下层方案 | 修改后检查下层是否受影响 |
| 映射关系 | 所有关联节点 | 需重新运行追溯验证 |
| 编号变更 | 全链路 | 需逐层更新引用，重新验证 |
