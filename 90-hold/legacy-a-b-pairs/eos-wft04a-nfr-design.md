# STR-NFR → SysReq-NFR 架构 · Skill A · 非功能需求设计
**Skill ID**: wft04a · **Phase 1: STR-NFR → SysReq-NFR Design**

> **概述**：每次执行只处理一个目标对象：一个 STR-NFR 架构末级节点或一个迭代中的 SysReq-NFR 节点。Skill A 负责从 `02-*.md` 获取 STR-NFR 节点，量化定义、找宿主匹配、标注分层约束表，并标记候选写回 22 资产的条目，供人类在 `05-*.md` 审阅确认。
>
> **输入 → 输出**：`02-*.md` 表D 中一个 STR-NFR 架构末级节点 → AI 产出量化定义 + 宿主匹配 + 分层约束表 + 候选写回标记 → 写入 `05-*.md`（状态=待确认方案）→ 人类审阅确认 → 进入 wft04b 落账
>
> **适用场景**：瀑布式系统设计第4步 · Phase 1。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.2 | **修订**：2026-06-26

## 一、职责定位

### 本 Skill 目的与目标

> **主要目的**：取信于架构师——「指标合理，分层正确」。把 STR-NFR 的模糊期望转化为 SysReq-NFR 的量化约束——每个 NFR 有可验证的量化指标，有明确的约束层级。
>
> **目标**：量化不拍脑袋，分层不张冠李戴。全局约束不误标为局部，局部约束不误标为全局。SysReq-NFR 的分层约束表是 wft05a 的输入。

Skill A 负责：

1. 识别本轮唯一目标对象——一个 STR-NFR 架构末级节点（新增路径）或一个迭代中的 SysReq-NFR 节点（重入路径）
2. 记录链身份（平台链/业务链）——每个 STR-NFR 节点携带链标签，Step Start 识别并记录
3. 加载 t01 STR-NFR 架构产出 + `22-eos-nfr-taxonomy.md` 三层资产（已确认分类/具体需求模板/候选条目）+ SysReq-NFR v1 基线
4. 执行承接完整性验证与指标量化检查
5. 逐条 STR-NFR 量化定义（引用 22 具体需求模板作指标参考）
6. 按非功能三大类（质量特性/环境适应性/可实现性）归类，经 Q1/Q2/Q3 找 SysReq-NFR 5 级宿主
7. 标注分层约束表（全局/业务域/IPO + 影响级别/影响点）写入 SysReq-NFR 节点（供 wft05a 消费）
8. 执行三项自检（全覆盖/1:1/N:1 合理性）
9. **标记**候选写回 22 资产的具体需求条目（作为方案的一部分，不执行实际写回）
10. 判断当前方案状态（阶段性方案 / 可确认方案）
11. 提交人类进行门禁 1 方案审阅

Skill A 不负责：

1. 正式写入 `22-eos-nfr-taxonomy.md` 正文区（由 wft04b 承担）
2. 候选区匹配的执行操作（由 wft04b 承担）
3. 推进节点状态到下游（由 wft04b 承担）
4. 重做 t01 的 STR-NFR 架构设计
5. 修改 22 资产的已确认分类或模板
6. 展开 `03-*.md` 详细定义（由 wft04b 下半承担）

## 二、输入与路由

### 目标对象

每次执行只处理一个目标对象：

1. **新增路径**：`02-*.md` STR 状态表（表D）中 `当前状态=待分解分配 + 人类决策=同意分解分配` 的一个 **STR-NFR 架构末级节点**（仅取 NFR 类型）
2. **重入路径**：`05-*.md` 中 `当前状态=迭代中` 的一个 SysReq-NFR 节点

### 链身份识别

每个 STR-NFR 节点元数据中记录链身份（平台链 / 业务链 / 共享）。Step Start 读取该身份后记录到设计上下文中，影响：

1. Step 1 加载约束资产的作用域
2. Step 4 分层约束表中 `约束对象` 字段的取值
3. Step End 输出分链统计摘要

### 活动输入节点锁定协议

`wft04a` 同一时间只能允许一个上游输入节点处于活动处理状态。活动输入节点未处理结束前，不得让新的上游输入节点进入本 Skill。

**活动输入节点判定：**

以下任一情况成立时，应视为存在活动输入节点：

1. `05-*.md` 中已存在引用某个 STR-NFR 节点的 SysReq-NFR 方案节点，且该方案节点仍处于 `待确认方案`、`迭代中` 或其他未完成状态。
2. 状态队列、反馈区或运行摘要中存在该输入节点的待办锁定记录。
3. STR 状态队列（02 表D）中该输入节点已被标记为进入 `wft04a`，但尚未记录 `wft04a` 处理完成。

**活动输入节点完成判定：**

以下任一情况成立时，活动输入节点才可视为完成或释放：

1. 对应 NFR 方案已达到 `可确认方案`，且人类已确认进入 `wft04b`。
2. 该输入节点被人类明确撤销、废弃、跳过或退回上游。

**锁定规则：**

1. Step Start 必须先检查是否存在活动输入节点，再检查新的待进入节点。
2. 若存在一个活动输入节点，AI 只能重入该节点、处理该节点反馈、输出该节点当前待办，或提示进入正确的下游 Skill；不得选择新的输入节点。
3. 若存在多个活动输入节点，说明状态队列不一致，AI 必须停止新增处理，输出状态冲突说明，并要求人类先选择保留一个活动输入节点。
4. 只有不存在活动输入节点时，AI 才能从待进入队列中选择一个新的输入节点。
5. 新输入节点一旦进入本 Skill，AI 必须在状态队列、方案节点或运行摘要中记录活动锁定关系：`活动输入节点ID -> 方案节点ID / 当前待办ID`，同时记录链身份。

### 锁释放与转交记录格式

当活动输入节点完成并准备交接至 wft04b 时，AI 必须在 `05-*.md` 状态队列或运行摘要中写入转交记录，包含以下信息（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §四）：

1. 上游活动节点 ID（STR-NFR 节点 ID）
2. 下游待处理节点 ID（SysReq-NFR 方案节点 ID / 建议方案包 ID）
3. 状态到位说明（`可确认方案 + [同意] 已放行`）
4. 人类选择动作（`选择进入wft04b`）
5. 活动锁状态（`已释放` 或 `已转交wft04b`）
6. 释放时间或轮次
7. 下游 Skill 名称（`wft04b`）

### 标准输入包

1. 目标 STR-NFR 节点（含链身份、需求描述、初始指标、所属非功能大类、决策记录）
2. 当前 `22-eos-nfr-taxonomy.md` 三层资产（已确认分类 / 具体需求模板 / 候选条目）
3. `02-*.md` 树画像 / 状态队列 / 节点索引与 STR-NFR 节点正文块
4. `05-*.md` 树画像 / 状态队列 / 节点索引（工作目标文档）
5. `05-*.md` 当前 SysReq-NFR v1 基线
6. 重入反馈（若为重入路径）

### Step Start 路由

| 条件 | 路由 |
|------|------|
| 存在一个活动输入节点 | 只处理该活动节点对应的反馈、重入或当前待办，不得进入新节点 |
| 存在多个活动输入节点 | 停止执行，输出状态冲突说明，要求人类选择保留一个活动输入节点 |
| 存在多个 STR-NFR 节点 = `待分解分配 + 同意分解分配`，且不存在活动输入节点 | 输出 `等待选择进入wft04a`，当前待办 ID 为 AI 推荐的一个 STR-NFR 节点 ID |
| STR-NFR 节点 = `待分解分配 + 同意分解分配`，人类已回复 `选择进入wft04a` 或 `选择：<STR-NFR节点ID>`，且不存在活动输入节点 | 新增路径，写入活动输入节点锁定关系，记录链身份 |
| 仅存在一个 STR-NFR 节点 = `待分解分配 + 同意分解分配`，且不存在活动输入节点 | 新增路径，写入活动输入节点锁定关系，记录链身份 |
| SysReq-NFR 节点 = `迭代中` | 重入路径（Step 6） |
| SysReq-NFR 节点 = `待确认方案` 且反馈区为空 | 输出 `等待NFR方案反馈`，不得覆盖既有方案 |
| SysReq-NFR 节点 = `待确认方案` 且反馈区有 `[同意] / [修改] / [驳回]` | 方案反馈路径（Step 6） |
| SysReq-NFR 节点 = `待资产写回`，人类已回复 `选择进入wft04b`，但活动锁尚未释放或转交 | 只记录活动锁释放 / 转交到 `wft04b`，不再进入新增设计或反馈处理 |
| SysReq-NFR 节点 = `待资产写回`，且活动锁尚未释放或转交 | 输出 `等待选择进入wft04b`，当前待办 ID 为该 SysReq-NFR 方案 ID |
| SysReq-NFR 节点 = `待资产写回`，且活动锁已释放或已转交 `wft04b` | 输出 `已转交wft04b`，不得继续修改该节点 |
| SysReq-NFR 节点 = `已资产落账` 或 `已验证冻结` | 输出当前节点已完成，提示选择下一个 STR-NFR 节点 |
| 其他状态 | 输出不可操作说明后结束 |

> **说明**：路由表中 `迭代中` 入口仅由 wft04b 回退触发（wft04b §四 将节点状态回退为 `迭代中` 并写入 `[修改]` 反馈）。wft04a 自身的 Step 6 `[修改]` / `[驳回]` 在同一轮内完成处理并直接回写为 `待确认方案`，不经过 `迭代中` 中间状态。

## 三、操作流程

### Step 布局总览

| Step | 名称 | 产出 | 写资产 |
|------|------|------|--------|
| Start | 前置校验与路由 | 就绪 / 结束 | — |
| 1 | 资产加载与上下文建立 | 设计上下文 | — |
| 2 | 输入预检与方案可行性判断 | 预检结论 | — |
| 3 | NFR 量化定义（核心设计） | 量化清单 | `05-*.md` |
| 4 | 架构组织与约束标注 | 映射 + 约束表 | `05-*.md` |
| 5 | 方案整理与写回标记 | 完整方案（含候选写回标记） | `05-*.md` |
| 6 | 反馈重入处理 | 调整后的方案 | `05-*.md` |
| End | 运行结束输出 | 唯一路由 + 审查指引 | 独立成章，见 §五 |

### Step 1 · 资产加载与上下文建立

读取并建立本轮设计上下文：

1. 加载 `02-*.md` 中目标 STR-NFR 节点正文块（含链身份、需求描述、初始指标、所属非功能大类、决策记录）
2. 加载 `05-*.md` 当前 SysReq-NFR 状态表与已有方案
3. 加载 `05-*.md` SysReq-NFR v1 基线（树画像/状态队列/节点索引，理解 0-6 级分类结构）
4. 加载 `22-eos-nfr-taxonomy.md` 三层资产：
   - **已确认分类（@nfr-xxx）** → Step 4 归类基准
   - **具体需求模板（@nfr-xxx-REQ-yy）** → Step 3 量化参考
   - **候选条目（@cand-nfr-xxx）** → 作为未来写回标记的参考目标
5. 加载 `05-*.md` 中功能部分的 SysReq-F 节点 ID 清单——用于 Step 4c 分层约束表中 `约束对象=IPO` 时填写具体功能 IPO ID（供 wft05a 按 ID 匹配读取）
6. 加载目标链的相关约束资产（全局约束 + 同链已有 SysReq-NFR 节点）
7. 若为重入，加载目标 SysReq-NFR 节点反馈区的人类意见

加载要求（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §三）：

1. 读取系统设计过程文档和资产文档时，先读树画像 / 资产全貌、状态队列和节点索引，再读本轮正文块和必要相关块
2. 仅已确认资产可直接依赖；候选 / 疑似 / 待确认资产仅可作为参考
3. 若关键输入缺失，本轮最多只能形成 `阶段性方案`

### Step 2 · 输入预检与方案可行性判断

对目标 STR-NFR 节点执行统一分析：

1. 校验 STR-NFR → OR-NFR 追溯链是否成立（STR-NFR 架构末级是否充分满足所承接的原始非功能需求末级）
2. 承接完整性检查
3. 指标量化检查：
   - 量化率 100% → 通过
   - 有未量化指标 → 标注 ⚠️ 待确认（按 L1 补齐规则：可推测则补齐并标注 AI 推断；无法确定则标注待确认）
4. 识别关键缺口：
   - 量化缺口（模糊指标无法确定）
   - 宿主缺口（无合适分类宿主）
   - 追溯链缺口
   - 必要资产缺口（22 资产模板/候选区数据不可用或关键约束资产缺失）
5. 若关键缺口影响方案完整性，本轮标记为 `阶段性方案`

### Step 3 · NFR 量化定义（核心设计活动）

消费 STR-NFR 节点，逐条量化定义。

1. 加载 Step 1 已读取的 22 具体需求模板（@nfr-xxx-REQ-yy），提取指标集作为量化参考
2. 逐条 STR-NFR 完善详细定义，写入 `05-*.md` 作为量化建议小节：

   ```
   ### 量化定义（wft04a）
   
   | 条目 | 需求描述 | 量化指标 | 测量方法 | 应用范围 | 优先级 | 模板来源 | 备注 |
   |------|---------|---------|---------|---------|--------|---------|------|
   | STR-NFR-001 | ... | [指标名]: [数值] [单位]（[条件]） | JMeter/安全审计等 | 全局/业务域/IPO | 高/中/低 | @nfr-xxx-REQ-yy | AI推断/待确认 |
   ```

3. 处理模糊指标（如"高性能"）→ L1 补齐规则：可推测则补齐并标注 `AI推断`；无法确定则标注 `待确认 ⚠️`
4. 记录量化完成清单

### Step 4 · 架构组织与约束标注

**4a · 前置判断（归类）：**

1. 每条 STR-NFR 属于非功能三大类中的哪一类？
   - **质量特性**：性能、安全、可用性、可靠性等
   - **环境适应性**：可扩展性、兼容性、可移植性等
   - **可实现性**：可维护性、可测试性、可部署性等
2. 可归类 → 在该大类下找宿主
3. 不可归类 → 新增评审（维度独立性 / 可量化性 / 架构影响力）
   - 全部通过 → 新增分类
   - 任一不通过 → ❓ 留待人类裁定

**4b · Q1/Q2/Q3 找宿主：**

| 维度 | 判断问题 |
|------|---------|
| **Q1 非功能类型匹配** | SysReq-NFR 资产中已有同类型节点？ |
| **Q2 覆盖范围偏差** | 当前 STR-NFR 的覆盖范围是否在已有节点边界内/外？ |
| **Q3 同类承接情况** | 是否有已承接同类 NFR 的节点？ |

决策矩阵：

| Q1 | Q2 | Q3 | 决策 |
|----|----|----|------|
| 命中 | 边界内 | 有同类 | **复用 ✅** |
| 命中 | 略超边界 | 有同类 | **改进-扩展 ⚠️**（查强内聚性） |
| 无命中 | 不在边界 | 无同类 | **新增 ⚠️**（查颗粒度） |
| 多方向/模糊 | — | — | **❓ 留待人类裁定** |

**4c · 分层约束表标注：**

写入 SysReq-NFR 节点的分层约束表（供 wft05a 消费）。每个 SysReq-NFR 5 级节点必须带一张分层约束表：

| 约束范围 | 约束对象 | 影响级别 | 影响点 | 说明 |
|----------|----------|---------|-------|------|

字段取值规则（继承原 wft04 规范）：

1. `约束范围`：固定为 `全局` / `业务域` / `IPO`
2. `约束对象`：
   - `全局` 时写 `EOS`
   - `业务域` 时写业务域名或能力类名（链身份影响取值）
   - `IPO` 时写具体功能 IPO ID
3. `影响级别`：固定为 `阻断型` / `非阻断型` / `待判定`
4. `影响点`：固定为 `组件类型` / `组件边界` / `接口机制` / `运行约束机制` / `共享归属`
5. `说明`：一句话解释该 NFR 为什么对该对象产生该级别影响

填写约束：

1. 一行只表达一个约束对象的关系；一个 NFR 约束多个对象时写多行
2. 若 `影响级别=阻断型`，则 `影响点` 不可为空
3. 若 `影响级别=待判定`，则 `说明` 必须写明待判原因
4. 当一个 NFR 影响面很大时，应优先使用 `全局` 或 `业务域` 行，不强制展开全部 IPO

**4d · 三项自检：**

- **覆盖率**：全部 STR-NFR 条目已有 SysReq-NFR 宿主
- **1:1 检查**：每条 STR-NFR 有唯一宿主
- **N:1 合理性**：多个 STR-NFR 归入同一宿主时，保证语义内聚

### Step 5 · 方案整理与写回标记

将前面的分析结果整理为一份完整的 05 工作草案，写入 `05-*.md`：

```
### 量化定义（wft04a）              ← Step 3 已写入
### 分类映射与宿主决策               ← Step 4 已写入
### 分层约束表                      ← Step 4 已写入
### 完整方案摘要                    ← 本轮整理
### 候选写回22资产标记              ← 新增：需要进入22资产的具体需求ID列表
### 反馈区                         ← 人类审阅入口
```

**候选写回22资产标记的格式：**

| 序号 | 具体需求描述 | 指标集 | 匹配候选 ID | 操作类型 | 备注 |
|------|-------------|--------|-----------|---------|------|
| 1 | 响应时间≤200ms | P99 200ms / 1000并发 | @cand-nfr-perf-001 | 升级 | 指标集完整 |
| 2 | 年可用率≥99.9% | 99.9% / 年 | 无 | 新增 @nfr-avail-REQ-{seq} | — |
| 3 | — | — | @cand-nfr-xxx | 暂不写回 | 指标待确认 |

**"暂不写回"条目处理**：操作类型为"暂不写回"的条目不进入 wft04b 的实体提取清单（Step B）。这些条目仅在本轮方案中作为参考保留，待后续指标确认后可在新一轮 wft04a 中重新标记为"升级"或"新增"。

节点状态设为 `待确认方案`。在 SysReq-NFR 正文块写入可观察的 `方案状态` 字段，取值只能为 `阶段性方案` / `可确认方案`；若为 `阶段性方案`，必须写明阻断项 / 关键缺口。更新 `05-*.md` 树画像 / 状态队列 / 节点索引。

### Step 6 · 反馈重入处理

若目标为 `迭代中` SysReq-NFR 节点，按反馈区执行：

1. **`[同意]`**
   - 先检查方案状态是否为 `可确认方案`
   - 是 → 方案内容已在 05-*.md 中，无需重新写入；在 `05-*.md` 状态表将节点状态推进为 `待资产写回`，清空反馈区，将活动锁状态记为 `待进入wft04b` → 进入 End，输出 `等待选择进入wft04b`；只有人类回复 `选择进入wft04b` 或存在等效转交记录后，才记录活动输入节点锁释放 / 转交
   - 否（阶段性方案）→ 不得放行。**在反馈区写入归档记录**如下，不清除人类已输入的 `[同意]` 反馈：
     ```
     <!-- 归档：{日期} 收到 [同意]，因方案为阶段性方案，不构成有效放行。请人类补充缺失信息后回复 [修改：……] -->
     ```
     节点状态保持 `待确认方案` → 进入 End，输出 `需要补充后重跑wft04a`

2. **`[修改]`**
   - 按反馈调整量化定义（指标/条件/测量方法）、宿主分配、约束表
   - 若涉及宿主变更，需重新执行 Step 4b（Q1/Q2/Q3 找宿主）逻辑，重新验证分类决策合理性
   - 归档本次人类反馈和修改前方案摘要，清空旧反馈区并生成新空反馈区
   - 节点状态回写为 `待确认方案`
   - → 进入 End，重新评估方案状态并输出对应结果

3. **`[驳回]`**
   - 重新执行该节点对应的完整设计流程（可回到 Step 2 → Step 3 → Step 4 完整链路，也可按驳回指明的范围定位到具体步骤）
   - 原方案归档，写入新方案
   - 清空旧反馈区并生成新空反馈区
   - 节点状态回写为 `待确认方案`
   - → 进入 End，重新评估方案状态并输出对应结果

## 四、方案审阅（门禁 1）

### 审查口径

门禁 1 审查的是：

1. **NFR 量化指标是否完整合理**——指标数值、单位、条件、测量方法是否合理
2. **非功能需求分类是否正确**——是否与 22 资产分类一致
3. **扩展/新增宿主是否合理**——是否通过了强内聚性检查
4. **分层约束表是否完整**——影响级别、影响点、说明字段是否覆盖所有约束
5. **阻断型 NFR 约束是否已明确映射**——是否存在未完成的阻断型约束
6. **22 资产候选写回标记是否合理**——哪些需求确实可复用
7. **承接关系是否闭合**——STR-NFR → SysReq-NFR 追溯链成立

门禁 1 不审：

1. `22-*.md` 的正式写回结果（由 wft04b 负责）
2. `03-*.md` 的详细定义展开（由 wft04b 下半负责）
3. 功能分支与非功能分支的全面冲突（由人类在 End 环节整体判断）

### 方案状态

| 方案状态 | 含义 | 是否可放行至 wft04b |
|---------|------|-------------------|
| `阶段性方案` | 存在阻断项 / 量化缺口 / 宿主未决 / 阻断型约束未明确 | 否 |
| `可确认方案` | 量化完整、宿主明确、约束表完备、承接关系闭合 | 是 |

## 五、运行结束输出（Step End）

Step End 的作用是把本轮设计结果翻译成人类下一次运行 `wft04a` 前的最小待办。AI 不只输出"方案已生成"，还必须根据当前运行结果，为人类确定一个且仅一个当前动作。

推荐交互方式（对齐 [91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md) §四）：

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应节点的 `### 反馈区`、状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

### 单 ID 最小待办原则

1. 每次 Step End 只提示人类处理一个 ID。
2. 该 ID 可以是一个 SysReq-NFR 方案 ID、一个待选 STR-NFR 节点 ID、一个待进入 `wft04b` 的建议方案包 ID。
3. AI 必须同时给出剩余待办 ID 数量，帮助人类知道当前队列规模。
4. 若上一次 Step End 已提示某个当前待办 ID，且该 ID 尚未被人类确认完成、明确切换或变为不可操作，下次运行必须继续提示同一个 ID。
5. 人类对当前 ID 的回复可以是 `同意`、`修改：……`、`驳回：……`，或 `选择：……`。
6. 只有当前 ID 已完成、被人类明确切换，或因状态变化不再可操作时，AI 才能按阻断优先级、状态时间和推荐顺序自动选择下一个当前 ID，并说明切换原因。
7. 当前 ID 处理完成后，下一次运行 `wft04a` 再提示下一个 ID。
8. AI 可以附带其他待办摘要，但不得要求人类同时审阅或选择多个 ID。

### 方案状态与结果对照

| 方案状态 | End 输出结果 | 人类操作 | 下一轮动作 |
|---------|-------------|---------|-----------|
| 可确认方案 | `等待NFR方案反馈` | [同意] → Step 6 推进到待资产写回 → 输出 `等待选择进入wft04b`；人类回复 `选择进入wft04b` 后完成锁释放 / 转交 | wft04a 或 wft04b |
| 阶段性方案 | `等待NFR方案反馈` 或 `需要补充后重跑wft04a` | [修改] → Step 6 调整方案 → 回到待确认方案；[同意] 无效 | wft04a |

### 当前待办 ID 连续性规则

1. Step End 输出的 `当前待办ID` 默认形成"待办锁定"。
2. AI 应在 `05-*.md` 对应 SysReq-NFR 节点反馈区、STR 状态队列或本轮运行摘要中保留该 ID 的待办状态。
3. 若下一次运行时该 ID 仍处于待反馈、待确认、待补充或待选择状态，AI 必须继续提示该 ID，不得因为出现新的待办而切换。
4. 若人类回复 `换一个 / 跳过 / 选择：<另一个ID>`，AI 可以切换当前待办 ID，但必须先记录切换原因。
5. 若当前 ID 已经完成，AI 才从剩余待办中选择下一个 ID。
6. 若当前 ID 因上游状态变化不再可操作，AI 必须说明原 ID 不再可操作的原因，再选择新的当前待办 ID。

### Step End 待办类型

| 待办类型 | 何时出现 | 当前 ID | 人类只需做什么 | 下次运行 |
|---------|----------|---------|----------------|----------|
| `反馈NFR方案` | 本轮已生成或修订 NFR 方案，状态为 `待确认方案` | 一个 SysReq-NFR 方案 ID | 对该 ID 回复 `同意 / 修改 / 驳回` | `wft04a` |
| `选择进入wft04a` | 存在多个已获入口许可的 STR-NFR 节点，且当前没有活动输入节点 | 一个 STR-NFR 节点 ID | 对该 ID 回复 `选择` 或指定另一个 STR-NFR ID | `wft04a` |
| `选择进入wft04b` | 方案已被 `[同意]` 并推进到可进入 wft04b | 一个建议方案包 ID | 对该 ID 回复 `选择进入wft04b` | `wft04b` |
| `需要补充后重跑wft04a` | 当前方案为 `阶段性方案`，仍有阻断项 | 一个 SysReq-NFR 方案 ID | 对该 ID 回复 `修改：……` 并补充缺失信息 | `wft04a` |

**建议方案包 ID 定义**：建议方案包 ID 等同于已被 `[同意]` 放行的 SysReq-NFR 方案节点 ID。当该节点推进到 `待资产写回` 并准备进入 wft04b 时，使用"建议方案包 ID"称谓以强调该 ID 代表一个完整的可落账方案（含量化定义 + 宿主决策 + 分层约束表 + 候选写回标记）。

### 输出结果

本 Skill 结束时，只允许输出以下四类结果之一，每类结果必须带一个当前待办 ID：

| 输出结果 | 对应待办类型 | 当前 ID | 人类下一步 |
|---------|--------------|---------|------------|
| `等待NFR方案反馈` | `反馈NFR方案` | 一个 SysReq-NFR 方案 ID | 对该 ID 回复 `同意 / 修改 / 驳回` |
| `需要补充后重跑wft04a` | `需要补充后重跑wft04a` | 一个 SysReq-NFR 方案 ID | 对该 ID 回复 `修改：……` |
| `等待选择进入wft04a` | `选择进入wft04a` | 一个 STR-NFR 节点 ID | 对该 ID 回复 `选择` 或 `选择进入wft04a` |
| `等待选择进入wft04b` | `选择进入wft04b` | 一个建议方案包 ID | 对该 ID 回复 `选择进入wft04b` |

### 最小待办输出格式

AI 在 Step End 应按以下格式输出：

```text
当前结果：等待NFR方案反馈 / 需要补充后重跑wft04a / 等待选择进入wft04a / 等待选择进入wft04b
当前待办ID：
是否延续上次待办ID：
剩余待办ID数量：
待办类型：
为什么先处理这个ID：
关键方案摘要：
阻断项清单：（无 / 列出阻断项）
重点查看位置：（文档路径 + 章节/节点 ID）
人类只需回复：
回复会写入：
反馈入口：优先对话回复；备选直接编辑对应节点的 ### 反馈区
下次运行：
其他待办摘要：
[分链摘要] 平台链已处理N个，业务链已处理N个，剩余平台链N个，业务链N个
```

要求：

1. `当前待办ID` 必须是单个 ID，不得写成多个 ID 列表。
2. `是否延续上次待办ID` 必须写 `是 / 否`；若为 `否`，必须说明当前 ID 已完成、被人类切换或不再可操作的原因。
3. `剩余待办ID数量` 必须是数字；若无法精确统计，应写 `至少 N 个` 并说明统计口径。
4. `人类只需回复` 必须是一句话，且只包含当前 ID 的动作。
5. `其他待办摘要` 只能给数量和类型，不得要求人类同时处理。
6. `关键方案摘要` 必须包含本轮方案的核心决策（量化指标数、宿主决策类型、约束条数）。
7. `阻断项清单` 必须列出所有未闭合的阻断项；无阻断项时写 `无`。
8. `重点查看位置` 必须给出具体文档路径和章节或节点 ID。
