# OR → STR 架构 - Skill A - 设计开发
**Skill ID**: wft01a - **Phase 1: OR → STR Design**

> **概述**：每次执行处理一个角色的一种类型 OR 条目（≤10 条），STR 反馈优先于新 OR 处理。Skill A 负责完成 OR→STR 的主结构方案形成，支持三条路径——**业务定义路径**（FR-BIZ）：将同角色的 OR 分配到已有 STR-F 节点（四关系检测+来源升级+矛盾记录）或聚合为新的端到端场景（三步判断法+十链覆盖自查）；**引擎定义路径**（FR-ENG）：形成 STR-E 节点，以配置单元法识别引擎配置单元清单；**非功能需求路径**（NFR）：形成 STR-NFR 节点。不负责 22/23/24/25/26 资产正式写回。AI 对新增/改进内容以 `[新增]` / `[改进]` / `[确认]` / `[矛盾]` 标记提示人类阅读，待人类整体确认同意后将 STR 节点置为「可以资产化」。
>
> **输入 → 输出**：一个角色的一批 OR 条目（≤10 条，`待分配 + 同意分配`）/ 有反馈的 STR 节点 → 修订已有 STR-F 场景业务方案 / 新增 STR-F 场景业务方案 / STR-NFR 节点方案 / STR-E 引擎定义需求方案 局部写入 `02-*.md`（状态=`待确认方案` 或 `可以资产化`）+ OR 条目在 `01-*.md` 中同步更新分配、状态与追溯引用
>
> **适用场景**：瀑布式系统设计第1步 - Phase 1。
>
> **设计模式参考**：[91-doc-paired-skill-spec.md](91-doc-paired-skill-spec.md)
>
> **版本**：v1.41 | **修订**：2026-06-23

## 一、职责定位

Skill A 负责：

1. 接收人类选中的一批 OR 条目（同一角色，≤10 条），按角色聚合处理
2. 加载上游追溯链与相关约束资产
3. 将 OR 条目分配到已有 STR 节点（Q1/Q2）或聚合为新的端到端场景（Q3）
4. 对已有 STR 节点的修订，执行多 OR 收敛检测——四关系判定（确认/补充/矛盾/无新增）、来源类型升级、矛盾记录
5. 形成 STR-F / STR-NFR / STR-E 主结构方案
6. 对 STR-F 场景业务识别信息载体清单、场景指标、推理依据与补全说明
7. 对 STR-E 引擎定义需求，基于配置单元法识别引擎配置单元清单
8. 以 `[新增]` / `[改进]` / `[确认]` / `[矛盾]` 标记输出本轮增量内容清单，供人类逐项阅读
9. 提交人类审阅 STR 场景业务方案或 STR-E 引擎定义需求方案
10. 人类"整体确认同意"后，将 STR 节点生命周期状态置为「可以资产化」

Skill A 不负责：

1. 23 输出产品架构资产的正式引用写回
2. 22 非功能需求资产的正式引用写回
3. 26 广义文档资产的正式归属写回
4. 24 业务定义资产与 25 引擎模型资产的正式写回
5. 下游 `wft02` 的详细分解展开
6. 业务人员确认——这是人类线下活动，Skill 不包含业务确认清单的生成或触发机制

## 二、输入与路由

### 标准输入包

1. 本轮处理对象
2. 上游追溯链
3. 相关约束资产
4. 已有 STR 架构资产
5. 重入反馈
6. 树画像 / 资产全貌、状态队列、节点索引与本轮正文块定位信息
7. 若为引擎定义路径：25 引擎模型资产（含配置单元类型定义，按候选引擎模型筛选）
8. 若为引擎定义路径：24 业务定义资产（资产全貌层，用于判断引擎能力支撑的抽象业务定义类型）
9. 若为 NFR 路径：22 NFR 分类资产（含一级类目/分类维度/具体需求，按候选分类维度筛选）

### Step Start 路由

#### 第一层：入口容错检查

| 条件 | 路由 |
|------|------|
| 人类选中的 OR 分属**多个角色** | 输出"选了多个角色：角色A（N条）、角色B（M条）。请只选择一个角色，重新运行。"→ **退出** |
| 同角色 OR **>10 条** | 输出"选了 N 条 OR，超过单次上限 10 条。请分批，重新运行。"→ **退出** |
| 无 OR 选中 + 无待处理反馈的 STR 节点 | 输出"无待处理对象"→ **退出** |
| 存在选中 OR 且包含**多种类型**（FR-BIZ / FR-ENG / NFR） | 输出"本批 OR 包含多种类型，请分批次后重跑"→ **退出** |

#### 第二层：反馈优先

| 条件 | 路由 |
|------|------|
| 存在 `待确认方案` + 反馈区有 `[同意] / [修改] / [驳回]` 的 STR 节点 | 本轮**先执行 Step 2 处理全部有反馈的 STR 节点**，再执行 Step 3/4/5 处理新 OR |
| 存在 `待确认方案` + 反馈区为空的 STR 节点 | 不阻塞新 OR 进入；在 Run End 时提示人类有等待反馈的 STR 节点 |

#### 第三层：新 OR 路由

按 OR 类型数据路由：

| 类型 | 路由 |
|------|------|
| FR-BIZ（业务能力） | 执行 Step 3（新增业务需求设计） |
| FR-ENG（引擎能力） | 执行 Step 4（新增引擎需求设计） |
| NFR（非功能需求） | 执行 Step 5（新增非功能需求设计） |

OR 类型由上游 `eos-ort03-norm.md`（ORT 预处理链）在条目拆分时以 `类型` 字段标注，wft01a 直接读取使用。

#### 第四层：FR-ENG 类型复核

在进入 Step 4 之前，AI 应对 FR-ENG 类型的 OR 做一次合理性检查——不是重新做 ort03 的工作，而是用配置单元法的视角（"是否适合以配置单元展开"）做快速复核：

| 情形 | 处理 |
|------|------|
| OR 内容适合配置单元展开（可形成 I→P→O 连贯链） | 正常进入 Step 4 |
| OR 内容存在疑义（如实际描述的是运行时行为而非配置能力、语义更适合 FR-BIZ） | AI 输出疑义：`⚠ 类型疑义：ort03 标记为 FR-ENG，但 OR 描述为"<关键语义>"，AI 初步判断：建议更正为 <FR-BIZ / 保留 FR-ENG / 建议分割>，因为…`。由人类在对话中决定后继续 |

复核不阻断处理——AI 给出初步判断后即继续，最终分类由人类决策。

## 三、文档加载规则

读取系统设计过程文档和资产文档时，必须遵守 [文档编写约定 §十四](../../../../00-generalspec/00-doc-conventions.md)：先读树画像 / 资产全貌、状态队列和节点索引，再读本轮正文块和必要相关块。

### 全景加载

每次运行 wft01a 时，先加载基础全景层：

1. `01-*.md` 树画像 / 状态队列 / 节点索引，定位本轮 OR 条目正文块与 `#role-list`
2. `02-*.md` STR 树画像 / 状态队列 / 节点索引，定位本轮 STR 节点块

再按本轮数据路由加载路径特定资产：

| 数据路由 | 加载资产 |
|---------|---------|
| FR-BIZ（Step 3） | `23-eos-output-architecture.md` + `26-eos-document-definitions.md` + `22-eos-nfr-taxonomy.md`（场景指标链） |
| FR-ENG（Step 4） | `25-eos-engine-models.md` + `24-eos-business-definitions.md` |
| NFR（Step 5） | `22-eos-nfr-taxonomy.md` |

### 加载要求

1. 仅已确认资产可直接依赖
2. 候选 / 疑似 / 待确认资产仅可作为参考
3. 若关键输入缺失，AI 应在增量清单中说明缺口
4. 按追溯关系读取必要的上游 / 下游相关块，不默认全量读取大文档正文
5. 只有在索引缺失、关系断裂或执行全局一致性检查时，才允许说明原因后扩大读取范围

### 按需加载线索

以下线索规定在特定操作场景下如何从全景收窄到局部正文块。

**对象线索**：根据本轮 OR 条目 ID 或 STR 节点 ID，在节点索引中定位必须读取的本轮正文块。适用：Step 1。

**追溯线索**：若本轮处理对象已有上游、下游或相邻承接关系，只读取追溯关系指向的必要块；不得复制对方正文到本节点。适用：Step 1。

**结构线索**：根据本轮处理对象的父节点、角色、需求类型、业务域和状态，选择同角色、同业务域、同层级或同状态的候选 STR 块，用于判断复用、改进或新增。适用：Step 3、Step 4 A.4、Step 5 A.3。

**语义线索**：根据 OR 中的角色、动作、业务对象、流程阶段、约束条件、异常条件和指标口径，从资产全貌中筛选候选业务域、产品类型、构件类型、NFR 分类和文档定义分片。适用：Step 3、Step 4 Phase A、Step 5 Phase A。

**信息链线索**：按十链扫描法识别潜在信息载体和场景指标，再用候选载体名称、产生方式、状态事件、审批留痕、报表看板等关键词定位 `26` 文档定义分片和相关资产块。适用：Step 3 Phase C。

**反馈线索**：若本轮为方案反馈路径或重入路径，只读取该 STR 节点反馈区及反馈指向的相关块；不得重新扩大为首次新增设计的全量候选读取。适用：Step 2。

**引擎线索**：根据 OR 类型为 FR-ENG，在 `25` 资产全貌中筛选候选引擎模型和其配置单元类型；根据候选引擎模型名称定位相关分片；若 25 中无合适引擎模型，AI 提出新引擎模型方案建议（含引擎名称、类型、定义、候选配置单元），加载 25/24 资产全貌作为推断参考。适用：Step 4 引擎定义路径。

**NFR 线索**：根据 OR 类型为 NFR，在 `22` 资产全貌中筛选候选一级类目和分类维度；根据 NFR 语义定位相关分类维度分片；若 22 中无匹配分类维度，标记为候选分类维度待确认。适用：Step 5 NFR 路径。

**扩大读取触发**：只有当索引缺失、候选冲突、资产全貌不足以判断边界、追溯关系断裂或需要做全局一致性检查时，才可扩大读取范围，并在输出中说明扩大读取的原因。适用：全局。

### 写入规则

Skill A 写文档时必须遵守 [文档编写约定 §十四](../../../../00-generalspec/00-doc-conventions.md) 的树形资产文档布局。写入范围按本轮相关块局部更新，不得重写大文档全文。

1. 写入 `02-*.md` 正文节点块：
   - 复用 / 改进时，定位既有 STR 节点块，按节点类型更新对应内容：
     - STR-F：节点定义、信息载体清单、场景指标、推理依据、承接 OR 清单、反馈区、十链覆盖自查和变更摘要
     - STR-E：节点定义、配置单元清单、推理依据、承接 OR 清单、反馈区、引擎能力边界陈述和变更摘要
     - STR-NFR：节点定义、量化指标清单、推理依据、承接 OR 清单、反馈区、适用范围汇总和变更摘要
   - 新增时，在合适父节点下新增稳定 `BLOCK`，写入完整 STR-F / STR-NFR / STR-E 要素，并生成空 `### 反馈区`
   - 若方案存在阻断项或关键缺口，AI 必须在增量清单中说明
2. 更新 `02-*.md` 树画像 / 节点索引 / 状态队列：
   - 若新增节点，或既有节点的一句话边界、父节点、层级、名称、状态发生变化，同步更新树画像行
   - 复用 / 改进时，更新既有 STR 节点索引行的状态、来源 OR、块 ID 和一句话边界
   - 新增时，新增 STR 节点索引行，记录节点 ID、父节点 ID、节点类型、层级、状态、块 ID 和一句话边界
   - 将本轮 STR 节点状态写为 `待确认方案`
3. 更新 `01-*.md` 树画像 / OR 正文块 / 状态队列：
   - 若 OR 条目状态或一句话边界在树画像中有摘要记录，同步更新树画像行
   - 在本轮 OR 正文块的当前分配记录中写入承接 STR 节点编号和承接方式
   - 在 `01-*.md` 状态队列中将该 OR 条目状态写为 `待确认方案`
4. 更新追溯关系：
   - 在 `01` 与 `02` 的追溯区或本轮相关块承接关系中记录 OR→STR 引用
   - 只写引用关系，不复制对方正文
5. 写入反馈区：
   - 新方案必须保留 `### 反馈区`
   - 若已有未处理反馈，不得覆盖或清空
   - 若本轮消费了人类反馈，必须先归档反馈处理摘要，再生成新的空反馈区

写入分支：

1. **复用已有 STR 节点**：不新增节点；只在既有节点块内追加本 OR 到承接 OR 清单，并按节点类型重新检查对应要素是否需要补充（STR-F：功能说明、信息载体清单、场景指标、推理依据；STR-E：配置单元清单、推理依据；STR-NFR：量化指标清单、推理依据）；同步更新树画像、索引行和状态队列。
2. **改进已有 STR 节点**：在原节点块内更新节点定义与要素；保留旧版摘要和变更说明；更新该节点承接 OR 清单、树画像、索引行和状态队列。
3. **新增 STR 节点**：在合适父节点下新增稳定节点块；写入完整 STR-F / STR-E / STR-NFR 要素、反馈区、树画像行、节点索引行和状态队列记录。
4. 任一分支都不得覆盖人类尚未处理的反馈区内容。

状态回退规则：若命中的宿主 STR 节点当前状态为 `可以资产化` 或 `待分解分配`（即非 `待确认方案` 状态），AI 在写入新内容后，必须将 `02-*.md` 中该节点状态回退为 `待确认方案`，并在 `01-*.md` 状态队列中对应 OR 条目的状态同步为 `待确认方案`。回退后该节点需重新经历人类确认流程（含重新对准 23/25/22 资产，因为原子单元可能变化）。如果 wft01b 已开始写回该节点（`02` 中有非空 `### 资产写回记录`），则不应回退，应输出冲突提示由人类判断。

## 四、操作流程

### Step 布局总览

| Step | 名称 | 产出 | 写资产 |
|------|------|------|--------|
| Start | 前置校验与路由 | 就绪 / 结束 | — |
| 1 | OR 层已确认资产加载 | 设计上下文 | — |
| 2 | 方案反馈处理 | 调整后的 STR 节点方案 | `01-*.md` / `02-*.md` |
| 3 | 新增业务需求设计 | STR-F 节点方案 | `01-*.md` / `02-*.md` |
| 4 | 新增引擎需求设计 | STR-E 节点方案 | `01-*.md` / `02-*.md` |
| 5 | 新增非功能需求设计 | STR-NFR 节点方案 | `01-*.md` / `02-*.md` |
| End | 运行结束输出 | 人类下一步操作提示 | — |

### STR 节点生命周期状态

| 状态 | 含义 | 谁处理 |
|------|------|--------|
| `待确认方案` | 方案已生成或修订，等待人类反馈 | wft01a |
| `可以资产化` | 人类已整体确认同意，等待 wft01b 进行资产化写回 | wft01b |

### Step 1 - OR 层已确认资产加载

按 §三 全景加载规则加载各资产全景层，按对象线索定位本轮 OR/STR 正文块，按追溯线索读取上下游关系。

### Step 2 - 方案反馈处理

按 §三 反馈线索读取 STR 节点反馈区。

若本轮存在有反馈的 STR 节点（`待确认方案` + 反馈区有 `[同意]/[修改]/[驳回]`，或 `迭代中`），则逐个 STR 节点处理反馈。多个 STR 节点均有反馈时本轮全部处理。

人类确认时自行判断脑补载体的可接受程度——无算法门禁，人类主观判断即为最终决策。

保护规则：
- 若 STR 节点已处于 `待确认方案` 且反馈区为空，AI 不得重新生成或覆盖该节点方案。
- 若需要重做 STR 场景业务，只能由 Step 2 的 `[修改] / [驳回]` 分支触发，并必须在反馈区或变更记录中保留旧方案处理摘要与人类反馈记录。

反馈分支：

1. **整体确认同意**（人类整体确认本轮所有新增/改进内容）

   适用：人类在对话中回复"整体确认同意"或"全部同意"。

   - 归档反馈处理摘要并生成新的空反馈区，在 `02-*.md` 树画像、状态队列和节点索引中将 STR 节点生命周期状态推进为 `可以资产化`
   - 在 `01-*.md` 树画像、本轮 OR 块和状态队列中同步方案已确认状态
   - 该 STR 节点不再由 `wft01a` 处理，后续由 `wft01b` 接手资产化

2. `[同意]`（逐项确认）

   - 确认当前项已通过，继续等待其他项或全部确认完成；全部确认完成后执行上述"整体确认同意"的推进动作
3. `[修改]`
   - 按反馈类型调整节点定义与要素。反馈类型包括结构反馈、信息链反馈、指标反馈、推理依据反馈、其他说明
   - 必要时重做 OR→STR 承接关系
   - 重新生成推理依据与补全说明
   - 重新执行十链覆盖自查
   - 在当前 STR 节点块内归档本次人类反馈和修改前方案摘要，并生成新空反馈区
   - 更新 `02-*.md` 树画像 / 节点索引 / 状态队列，节点状态回写为 `待确认方案`
   - 若承接关系变化，同步更新 `01-*.md` 树画像、本轮 OR 块、状态队列和追溯引用
4. `[驳回]`
   - 重新执行该节点对应的 OR→STR 设计
   - 在当前 STR 节点块内归档原方案和本次人类反馈，写入新方案或按新宿主更新承接关系
   - 更新 `02-*.md` 树画像 / 节点索引 / 状态队列，节点状态回写为 `待确认方案`
   - 若改为其他 STR 宿主或新增节点，同步更新 `01-*.md` 树画像、本轮 OR 块、状态队列和追溯引用

注：STR-E 节点的反馈处理逻辑与 STR-F 相同，但 [修改] 的反馈类型包括配置要素反馈、处理机制反馈、业务能力反馈等；[驳回] 时重做 Step 4 的引擎定义扫描（Phase A→B→C），不执行十链覆盖自查。

注：STR-NFR 节点的反馈处理逻辑与 STR-F 相同，但 [修改] 的反馈类型包括分类归属反馈、量化指标反馈、测量方法反馈、适用范围反馈等；[驳回] 时重做 Step 5 的 NFR 分类匹配与量化设计（Phase A→B→C），不执行十链覆盖自查。

修订方案后，AI 必须输出增量清单（标记定义与格式见 Step 3 增量标记输出）。

---
### Step 3 - 新增业务需求设计

处理本角色的 业务定义类 OR 条目，分三个阶段：先分组（Phase A），再修订已有 STR（Phase B），最后新建 STR（Phase C）。

#### Phase A：OR → 目标 STR 分组

按 §三 结构/语义线索匹配候选 STR 和资产。

1. 沿用 `#role-list` 中已确认角色归属（本批次所有 OR 同角色，无需逐条确认）
2. 对每条 OR 独立执行 **Q1/Q2/Q3 宿主匹配**，判定其归属于已有 STR 节点还是需要新建：
   - **Q1（直接匹配）**：OR 的功能描述与已有场景业务在语义上一致，则复用该宿主。判定依据：OR 中的"动词 + 业务对象"能直接对应已有 STR 节点的场景职责边界陈述（即该场景业务的职责范围）。宿主候选池限定为 STR-F 节点，不论其当前生命周期状态（`待确认方案`、`可以资产化`、`待分解分配` 等均可匹配）；STR-E 和 STR-NFR 节点不参与 Q1/Q2 匹配
   - **Q2（扩展匹配）**：OR 的功能描述是已有场景业务的子集或超集边缘，可扩展后承接。边界判断：扩展不改变场景业务的核心职责边界——若扩展后变成大杂烩，应改为新增
   - **Q3（新增）**：OR 无任何可用宿主，或扩展后导致边界模糊。要求：新增场景业务的命名和粒度应与同层已有节点保持一致
匹配到 `待确认方案` 节点时：Q1/Q2 可正常匹配，不阻塞，但应在增量清单中标注 `[推断]` 提示人类"该匹配基于待确认的边界陈述，匹配结论可能随后续反馈变化"。匹配到 `可以资产化` 或 `待分解分配` 节点时无需标注。

3. **分组结果**：
   - Q1/Q2 命中的 OR → 归入对应已有 STR 节点的"本轮新增 OR 清单"
   - Q3（无匹配）的 OR → 进入"待聚合池"
4. 对"待聚合池"中的 OR，按**业务域 + 触发相似性**分组，每组将成为一个新的 STR-F 节点。无法与其他 OR 聚合的 OR（业务域/触发完全不同）独立成节点。

若命中的宿主 STR 节点当前状态不是 `待确认方案`（如为 `可以资产化` 或 `待分解分配`），AI 应在写入该节点新内容后，将其状态回退为 `待确认方案`，等待人类对新内容的反馈。回退前须检查 wft01b 是否已开始写回该节点（`02-*.md` 中存在非空的 `### 资产写回记录`）——若 wft01b 已开始写回，不应回退，应输出冲突提示由人类判断。

若本轮所有 OR 均为 Q3（无 Q1/Q2 命中），则跳过 Phase B，直接进入 Phase C。

#### Phase B：修订已有 STR（四关系检测）

对每个已有 STR-F 节点，先做**边界匹配检查**，再做逐载体四关系检测。

**边界匹配检查**：在逐载体比对之前，先检查新 OR 是否落在该 STR 节点的场景边界内。

| 检查结果 | 判定依据 | 处理 |
|---------|---------|------|
| OR 在边界内 | OR 描述的内容属于该场景的职责范围（同业务域、同职责段） | 继续逐载体四关系检测 |
| OR 超出边界 | OR 跨到其他业务域或职责段（判定依据同 2c 紧凑性检查——同一角色同一业务域） | 不进入四关系检测；该 OR 改为 Q3，退回待聚合池或独立成节点 |
| OR 在边界边缘 | OR 可能扩展边界，但尚在同一业务域内 | 继续四关系检测，但增量清单标注 `[边界扩展]` 供人类关注 |

边界匹配检查是 AI 的尽力检查，不是算法门禁——最终由人类在审阅增量清单时兜底。

退回 Q3 的 OR 若在 Phase C 中无法与其他 OR 聚合而独立成节点，AI 应在增量清单中标注 `[推断]`——提示人类该节点仅由单条 OR 驱动，边界可靠性较低。

**逐载体四关系检测**：对通过边界检查的 OR，将其与既有信息载体做比对——先按以下算法判定关系，再按表格执行对应的处理动作。

检测算法：对新 OR 的每个语义要素，匹配已有载体的"角色+业务对象+职责"三元组。三元组全匹配 → `确认`（脑补被证实）或 `无新增`（已明示）；部分匹配但职责/形态有差异 → `矛盾`；无匹配 → `补充`。

| 关系 | 判定条件 | 处理动作 |
|------|---------|---------|
| **确认** | 新 OR 语义与此前脑补的载体一致（同角色、同业务对象、同职责） | 来源类型升级（见下）；确认状态 `待确认`→`可确认`；追加 OR 引用；增量清单标记 `[确认]` |
| **补充** | 新 OR 提到此前未识别的载体 | 新增到信息载体清单；来源类型=`OR明示`；确认状态=`可确认`；增量清单标记 `[新增]` |
| **矛盾** | 新 OR 描述的载体形态/范围/关系与既有载体不同 | 在载体条目下追加矛盾记录（见下）；确认状态=`待确认`（需人类裁决）；增量清单标记 `[矛盾]` |
| **无新增** | 新 OR 内容完全被已有载体覆盖（含已 `OR明示` 确认的） | 仅追加 OR 引用到承接 OR 清单；不改变载体状态 |

**来源类型升级规则**（只升不降，`OR明示` 是终态，升级仅由 `[确认]` 或 `[补充]` 关系触发）：

| 原来源类型 | 升级条件 | 新来源类型 |
|-----------|---------|-----------|
| `OR隐含` | ≥1 条后续 OR 明确提到该载体 | `OR明示` |
| `同类场景推断` | 同场景的后续 OR 证实 | `OR明示` |
| `资产线索推断` | 后续 OR 证实该资产模式 | `OR明示` |

升级时同步更新确认状态 `待确认`→`可确认`。原推断依据（R#）保留作为设计历史，追加新 OR 引用。

**矛盾记录格式**（写入载体条目下）：

```text
⚠ 矛盾：A版（OR-XXX）："<原有描述>" vs B版（OR-YYY）："<新OR描述>" → 待人类裁决
```

矛盾必须由人类在反馈区裁决后才能推进。存在未解决矛盾时，AI 不得将该 STR 节点推进到 `可以资产化`。

**矛盾记录追加规则**：矛盾记录采用追加式，不覆盖。同一载体在多轮 OR 进入时反复产生矛盾时，每条矛盾独立追加，格式一致，以独立的 OR 来源编号区分。多条矛盾共存直到人类逐条裁决。追加不覆盖的好处：人类能追溯每一条矛盾的完整来龙去脉、裁决未完时不影响后续 OR 继续入站、裁决完成后有据可查。

修订完成后，更新 STR-F 要素（功能说明、信息载体清单、场景指标、推理依据、十链覆盖自查、承接 OR 清单等）及 `02-*.md` 树画像/节点索引/状态队列，按 §三 写入规则执行。

#### Phase C：新建 STR

对每个新场景（由待聚合池中的 1~N 条 OR 聚合而成），执行以下步骤。

1. **确定业务域**：从角色类型映射 23 资产业务域（如采购员→采购管理），显式写入 STR-F 节点

2. **按触发定义端到端业务闭环**——三步判断法

   本步骤将"端到端业务闭环"分解为三个可执行的子判断：识别触发、追踪自然终点、边界检查。三步依次执行，前一步的输出是下一步的输入。

   2a. **识别触发**

   从 OR 语义中提取"什么动作/事件启动了这个场景"。

   触发提取规则：

   - 触发通常是"角色 + 动作 + 业务对象"结构
     - 示例：采购员 + 提交 + 采购申请 → 触发=提交采购申请
     - 示例：质检员 + 收到 + 待检品通知 → 触发=收到待检通知
   - 当一个 OR 包含多个触发候选时：
     - 选最早发生的业务动作作为本轮触发
     - 后续触发作为场景内部环节，不独立成场景
     - 示例：OR="采购员需要能发起采购申请并跟踪订单执行" → 触发=发起采购申请（"跟踪订单"是申请后的环节，不是独立触发）
   - 跨场景判定：若后续触发在业务上独立于当前触发（不同角色、不同业务域、不同时间窗口），应标记为"跨场景线索——不在本场景边界内"，作为另一个 STR-F 节点的候选线索

   输出：触发陈述 + 触发来源（OR明示 / OR隐含 / 需澄清）

   组内触发不匹配处理：当 Phase A 分组后，2a 发现组内存在触发明显不同的 OR 时：
   - 若该 OR 的触发是独立场景起点（与组触发不同角色/业务域/业务对象）→ 从当前组移除，退回待聚合池，与其他待聚合 OR 重新尝试匹配
   - 若该 OR 的触发依赖于组触发（属于同一业务流内部的后续环节，如"查询订单进度"依赖于"提交订单"）→ 保留在组内，标注为"场景内部补充需求"，不独立成场景
   - 退回待聚合池的 OR 若无法与其他 OR 聚合，在 Phase C 中独立成节点

   2b. **追踪自然终点**

   从触发出发，沿业务流追踪到"该触发在本轮 OR 语义覆盖范围内的自然完成点"。

   **端到端约束**：STR-F 的"端到端"是**用户与 EOS 的一次完整交互闭环**——用户输入了什么（起端）→ EOS 处理了什么 → EOS 输出了什么、用户确认了什么（终端）。追踪的终端不得超过本轮 OR 语义覆盖的终点——OR 明示或强暗示的最后一个业务动作即为本轮追踪终点。AI 可以推断起端和终端之间的必经环节，但不得在终端之后扩展任何新环节。场景边界随多轮 OR 积累而逐步扩展，不由单轮推断一步到位。

   "强暗示"的判断标准：OR 中明示的业务对象的关键生命周期终点（如"采购申请"的生命周期终点是"审批通过/下达"）的最后一个业务动作。AI 不得将 OR 未提及的后续流转环节纳入场景范围。

   追踪规则：

   - 沿本角色在当前业务域内的动作链前进，直到达到本轮 OR 语义覆盖的终点
   - 自然终点的判断信号（满足任一即可停止）：
     - 已达到本轮 OR 明示或强暗示的最后一个业务动作——OR 说的"到最后一步"就是终点
     - 产出了一个可闭环节点的业务结果——订单已下达、合同已签署、验收单已签发等
     - 下一个动作已进入其他角色或其他业务域的职责范围——采购下单完成→供应商发货（发货是供应商职责，不是采购员职责）
     - 本角色在该业务域不再有后续主动动作——质检完成后，质检员在本业务的职责已结束
   - 若追踪链在同一个角色、同一个业务域内延伸过长（超过5个连续环节），检查是否包含多个不连续的职责段（见 2c 边界检查）

   跨角色/跨域截断规则：

   - 追踪到本角色职责结束时立即停止
   - 在场景业务中标注跨角色/跨域的交接点，格式：`完成A → [交接：角色B 执行B]`
   - 交接点不是场景边界开裂，只是标注"这里交给下一个角色"

   同时，在 2b 追踪业务流时，AI 自然识别流经的业务单据、表单、记录、通知等信息载体以及场景的运行评价指标。这些载体和指标将在步骤 5 中正式写入 STR-F 要素。

   输出：从触发到自然终点的环节链 + 自然终点陈述 + 交接点清单（如有）+ 初步信息载体与场景指标

   2c. **边界检查与职责陈述**

   检查 2b 输出是否构成一个合理的端到端场景业务。

   检查维度：

   | 检查项 | 通过条件 | 不通过处理 |
   |--------|---------|-----------|
   | 完整性 | 从触发到终点，角色能在这个场景内完成一件完整的事 | 未到自然终点 → 回到 2b 继续追踪 |
   | 一致性 | 所有环节属于同一个角色的同一个业务域 | 跨角色/跨域 → 截断到本角色职责范围，跨出部分标注为下游场景线索 |
   | 紧凑性 | 场景边界没有包含多个不连续的职责段（如同一角色在采购管理和供应商评估两个域的动作混在一个场景中） | 包含多个不连续职责段 → 拆分为多个 STR-F 候选，本轮处理第一个；剩余候选在增量清单中输出为 `[跨场景]` 线索，供人类下一轮选择相关 OR 进入 |
   | 粒度匹配 | 该场景的职责范围与已有同层 STR-F 节点（同角色/同业务域）的粒度相当；无已有节点时跳过此项检查 | 过粗/过细 → 调整追踪深度，参考已有节点风格调整边界；无已有节点时在增量清单标注 `[推断]`"新业务域，粒度需人类确认" |

   边界检查通过后，输出**场景职责边界陈述**——一句概括该场景"做什么"的话：

   格式：`「<角色>」在「<业务域>」中「从触发到终点的职责概括」`
   示例：`「采购员」在「采购管理」中从发起采购申请到完成收货入库的采购执行职责`

   边界陈述的作用：
   - 作为该 STR-F 节点的"一句话边界"，写入 02-*.md 的树画像行和节点索引行
   - 作为 Phase A Q1 宿主匹配的判断依据——Q1 判定中的"场景业务的职责范围"即指本步产出的边界陈述
   - 作为 wft02a 场景业务编排的输入基准

   输出：场景职责边界陈述 + 边界检查结论（通过 / 需调整 / 需拆分）
3. **推理依据与补全说明**：

   对所有超出 OR 直接明示的信息载体和场景指标，必须显性化 AI 推理依据。推理依据至少分为三类：

   | 推理类型 | 使用场景 | 输出要求 |
   |---------|----------|---------|
   | 业务常识推理 | 从业务动作、对象、角色职责推断常见业务单据、表单、记录 | 说明 OR 线索、业务规则、补全结果 |
   | 系统设计推理 | 从进度跟踪、状态变化、指标计算、自动触发推断系统衍生文档 | 说明实现该能力所需的状态、事件、日志或数据来源 |
   | 企业治理推理 | 从预算、供应商、质量、安全、审批、异常推断合规、规则、审计与证据载体 | 说明治理风险、控制点、补全结果 |

   推理依据采用编号引用，如 `R1`、`R2`。信息载体和场景指标条目应引用对应 `R#`，使人类可追溯每项补全的依据。

4. **派生 STR-F 节点名称**：即场景业务名称。本节点为新增 STR，命名和粒度应与同层已有节点保持一致。

5. **完善 STR-F 要素**：

   将三步判断法中识别的初步信息载体和场景指标整理为正式要素，写入 STR-F 节点。

   **十类信息载体归属对准规则**（AI 判定 23-Q1/Q2/Q3 的依据）：

   | 节点类型 | 归属锚点 | 归属规则 |
   |------|---------|---------|
   | 执行业务链节点 | 被执行、创建、处理或流转的产品数据 | 归属到该产品数据所在产品类型和构件类型；归属锚点是业务对象，不是页面、列表、表单等显示形态 |
   | 管理业务链节点 | 被管理的业务对象、业务过程或管理控制对象 | 归属到执行业务链主产品类型下的项目管理构件，并进一步归入对应管理业务 WBS；跨多个构件时拆分为多个 WBS 引用，或定义跨构件管理 WBS 并列出覆盖范围 |
   | 个人绩效链节点 | 当前场景内个人对业务/项目运行的贡献、达成、质量和效率 | 归属到执行业务链主产品类型下的项目管理构件，作为个人绩效 WBS；企业统一绩效考核场景另行记录向统一绩效产品的汇总/引用关系 |
   | 组织绩效链节点 | 当前场景内组织、团队、部门或项目对业务/项目运行的贡献、达成、质量和效率 | 归属到执行业务链主产品类型下的项目管理构件，作为组织绩效 WBS；企业统一绩效考核场景另行记录向统一绩效产品的汇总/引用关系 |

### Step 4 - 新增引擎需求设计

处理 FR-ENG 类型 OR 条目。按配置单元法识别引擎能力场景的配置单元清单。

**超出 25 资产的统一处理原则**：任何超出 25 资产范围的情形（无匹配引擎模型、配置单元类型不存在、粒度偏差），AI 不仅仅是上报问题，必须给出具体的归属或调整建议方案。所有超出范围项均标记 `[推断]`，等待人类在增量清单审阅时确认。AI 不得只上报缺口不给出建议。

#### Phase A：触发与场景边界确定

**A.1 触发识别**

从 OR 语义中提取"哪个业务定义的配置需要启动了这项引擎定义需求"。

触发提取规则：
- 触发通常是"业务定义配置人员 + 需要让XX引擎具备YY能力 + 为了支撑ZZ业务定义"
- 示例：
  - "需要一个能配置会签的审批引擎" → 触发=审批引擎需要会签能力，驱动源=审批业务定义需要会签
  - "需要一个表单引擎来支撑采购申请单的配置" → 触发=表单引擎需要支持采购申请单定义，驱动源=采购申请单业务定义需要表单引擎
- 当一个 OR 包含多个引擎需求候选时，选最核心的作为本轮触发，其余作为能力规格补充，不得独立成多个 STR-E 节点
- 若一个 OR 同时触及多个引擎（如审批引擎和表单引擎），AI 列出所有候选引擎名称，由人类选择本轮处理哪一个。剩余引擎在增量清单中以 `[跨场景]` 标记输出。若涉及的多引擎之间有强依赖，AI 标注依赖方向供人类参考
- 若引擎需求不明确指向具体引擎（如"需要更好的配置体验"），标记为需澄清，不得进入新增设计。此安全阀保留——当 25 资产引擎模型和配置单元类型定义成熟后，此情形将自然消除

**A.2 引擎模型匹配**

从 `25-eos-engine-models.md` 中匹配候选引擎模型：

| 情形 | 处理 |
|------|------|
| 匹配到单个已知引擎模型 | 加载该模型的**配置单元类型清单** |
| 匹配到多个引擎模型 | AI 列出所有候选引擎名称，由人类选择本轮处理哪一个。若多引擎之间有强依赖，标注依赖方向。剩余引擎以 `[跨场景]` 标记输出 |
| 匹配到引擎模型但配置单元类型清单为空或全为 `模板` | 基于 OR 语义推断候选配置单元，全部标记 `[推断]` + `模型内容待填充`。不阻断 Step 4 继续 |
| 无匹配 | AI 基于本轮 OR 语义 + 引擎常识，提出**新引擎模型方案建议**——包含建议的引擎名称、引擎类型（业务对象 / 流程与任务 / 规则与指标 / 集成与文档）、一句话定义、候选配置单元清单（从 OR 语义推断 I→P→O）和推理依据。全部标记 `[推断]` + `归属待定`。STR-E 节点可创建——候选引擎模型为 AI 建议方案（非 25 资产中已确认引擎），A.3/A.4/Phase B 正常继续，每一步均标记 `[推断]`。新引擎方案建议同时写入增量清单，供人类审阅决策 |

**状态敏感匹配规则**：

| 引擎模型状态 | 处理 |
|-------------|------|
| `已确认` | 正常加载和使用 |
| `待确认` | 可正常加载配置单元类型清单，但增量清单标注 `[推断]`——提示人类"该引擎模型尚待确认，基于其配置单元类型的配置单元设计可能随模型确认而变化" |
| `废弃` | 不得使用；输出"匹配到已废弃引擎模型 X，请确认是否应改用替代模型或新建引擎模型"。人类决定后：若改用替代模型→重新执行 A.2 匹配；若新建引擎模型→走无匹配路径（`[推断]` + `归属待定`）；若跳过→本轮不处理此 OR |

**引擎-业务定义匹配检查**：AI 应检查 OR 中隐含的抽象业务定义是否落在引擎模型"支持的业务定义"范围内。若不匹配，标注"引擎-业务定义不匹配"，继续搜索 25 资产中更合适的引擎模型。若 25 资产中无合适引擎模型，标注 `引擎模型待定义`。

配置单元类型清单是该引擎模型在 25 资产中预定义的标准配置单元集，每个类型包含名称、职责说明和资产引用路径。

**23 产品架构对准**：引擎模型匹配完成后，AI 加载 `23-eos-output-architecture.md` 资产全貌层，将引擎模型对准到 23 产品架构中的产品类型和构件类型：

| 情形 | 处理 |
|------|------|
| 23 中存在该引擎模型对应的产品类型 / 构件类型 | 记录 23-Q1 + 产品类型 ID + 构件类型 ID |
| 23 中存在相近产品类型 / 构件类型，可扩展 | 记录 23-Q2 + 扩展建议 |
| 23 中无匹配项 | 记录 23-Q3 + 新增建议（建议的产品类型名称 / 构件类型名称 / 定义） |

引擎模型是 EOS 产品架构的一部分，应在 23 中注册为平台引擎类构件类型。该对准结果作为 STR-E 节点级字段写入 02，供 wft01b 写回 23 资产。

**A.3 场景业务名称确定**

**EOS 引擎场景三层抽象**：

```
业务引擎 ──经配置生成──→ 业务定义 ──运行使用──→ 业务实例
（平台级能力，     （抽象业务类型，   （具体业务运行，
如审批引擎）        如审批业务）       如采购申请审批）
```

STR-E 的场景业务名称定位于**业务定义层**——`xx业务配置运行`，xx 为抽象业务定义类型。不得使用业务实例名称。

判定方法：若去掉业务场景名词（如"采购申请"），剩余部分可被其他业务场景复用，则取可复用部分为抽象业务定义。
- 示例：OR="需要一个能配置会签的审批引擎" → 场景名称=`审批业务配置运行`
- 示例：OR="需要一个表单引擎支撑采购申请单的配置" → 场景名称=`表单业务配置运行`
- 示例：OR="需要一个台账引擎来配置各种业务台账" → 场景名称=`台账业务配置运行`

**同引擎多业务定义的分组规则**：一个引擎支撑多个抽象业务定义（如审批引擎同时支撑审批业务、工单业务）。同批多个 FR-ENG OR 的分组依据是**引擎模型 + 抽象业务定义**的双键匹配，不是单键引擎匹配。同一引擎模型下的不同抽象业务定义应分入不同的 STR-E 节点——它们共享同一套配置单元类型库，但配置单元的实例化内容不同。

若 OR 未明确指定抽象业务定义类型，AI 应根据 OR 语义推断并标注来源类型为"OR隐含"或"需澄清"。

**A.4 STR-E 宿主匹配**

以双键（引擎模型 + 抽象业务定义）在 `02-*.md` 中搜索已有 STR-E 节点，确定本轮是复用还是新建：

| 判定 | 条件 | 路由 |
|------|------|------|
| Q1（复用） | 存在引擎模型相同 + 抽象业务定义相同的 STR-E 节点 | Phase B 走复用路径（先 B.2 四关系检测，再 B.3 逐单元处理） |
| Q2（扩展） | 存在引擎模型相同的 STR-E 节点，但抽象业务定义不同 | 同一引擎的不同抽象业务定义分入不同 STR-E 节点，走新建路径（跳过 B.2，直接 B.3） |
| Q3（新建） | 无匹配 STR-E 节点 | Phase B 走新建路径（跳过 B.2，直接进入 B.3） |

**Phase A 输出汇总**：

- A.1 输出：触发陈述 + 驱动源（为哪个抽象业务定义）+ 候选引擎名称
- A.2 输出：引擎模型匹配结论 + **23 产品架构对准**（引擎模型对应的 23 产品类型 / 构件类型 / 23-Q 判定）
- A.3 输出：场景业务名称 + 抽象业务定义类型
- A.4 输出：宿主匹配结论（Q1 复用 / Q2 扩展新建 / Q3 新建）+ 路由方向（复用路径或新建路径）

| 情形 | 处理 |
|------|------|
| 引擎模型在 25 资产中已确认，但配置单元类型清单为空或全为 `模板` | AI 基于 OR 语义 + 引擎常识推断候选配置单元，标记 `[推断]` + `模型内容待填充` |
| 引擎模型在 25 资产中 `待确认`，配置单元类型清单为空 | 同上，追加标注 `引擎模型待确认` |
| 引擎模型为 AI 建议方案，不在 25 资产中（A.2 无匹配路径）| AI 基于 OR 语义推断候选配置单元，标记 `[推断]` + `归属待定` |

区别：前两种情形在 25 资产中有引擎骨架，第三种是纯推断。

**B.2 多 OR 收敛检测（复用/改进既有 STR-E 时）**

当本轮为复用或改进已有 STR-E 节点时（即 OR 命中了既有的 STR-E 宿主），在逐配置单元处理之前，先执行简化版四关系检测。

比对单位：既有 STR-E 的配置单元 vs 本轮 OR 语义触及的配置能力。比对锚点：`配置要素` 的职责范围。

| 关系 | 判定条件 | 处理动作 |
|------|---------|---------|
| **确认** | 新 OR 触及的配置能力与既有配置单元的三栏语义一致 | 来源类型升级（`OR隐含`→`OR明示`），确认状态 `待确认`→`可确认`，追加 OR 引用，增量标记 `[确认]` |
| **补充** | 新 OR 触及了既有 STR-E 未覆盖的配置能力 | 新增配置单元到清单，来源=`OR明示`，确认状态=`可确认`，增量标记 `[新增]` |
| **矛盾** | 新 OR 对同一配置单元的三栏描述与既有版本存在**互斥**（如核心语义被推翻） | 在配置单元下追加矛盾记录（追加式不覆盖），确认状态=`待确认`，增量标记 `[矛盾]` |
| **无新增** | 新 OR 触及的配置能力完全被既有配置单元覆盖 | 仅追加 OR 引用；不改变配置单元状态 |
| **未触及** | 已有配置单元不在本轮 OR 的语义范围内 | 保持原样，无动作。不追加 OR 引用，不改变配置单元状态和来源类型 |
| **占位→实体** | 既有配置单元为占位（模型内置、三栏空置），本轮 OR 触及 | 从占位升级为完整设计三栏，来源=`OR明示`，确认状态=`可确认`，增量标记 `[确认]`（占位→实体的升级视为确认的一种） |

**矛盾判定细化**：矛盾不是"有任何差异"，而是"与既有版本在职责边界或关键机制上存在互斥"。

| 情形 | 判定 | 示例 |
|------|------|------|
| 新 OR 完全推翻既有栏的核心语义 | 矛盾 | 原说"串行审批"、新说"并行审批" |
| 新 OR 扩展了既有栏的范围但不颠倒核心语义 | 补充（该栏标注为扩展补充） | 原说"审批阈值"、新说"审批阈值需按部门分级" |
| 新 OR 在既有栏外增加了新维度 | 确认 + 补充 | 原说配置要素、新说处理说明 |

整单元四关系判定以配置要素职责为锚点，但允许**单元内部逐栏标注**——当整单元判定为"确认"或"补充"时，三栏内部可混合标注实际变化：

```text
配置单元 1：审批节点定义
  [确认] 配置要素（被 OR-005 证实）
  [补充] 处理说明（OR-005 新增了发布机制细节）
  [—] 业务说明（本轮无新增信息）
```

矛盾记录格式（写入配置单元条目下）：

```text
⚠ 矛盾：A版（OR-XXX）："<既有三栏摘要>" vs B版（OR-YYY）："<新 OR 描述的三栏差异>" → 待人类裁决
```

矛盾必须由人类裁决后才能推进。存在未解决矛盾时，AI 不得将该 STR-E 节点推进到 `可以资产化`。

**配置单元拆分/合并建议**：多轮 OR 积累可能导致既有配置单元粒度需要调整。

| 情形 | 处理 |
|------|------|
| 新 OR 对既有配置单元的某个子职责给出详细描述，且该子职责有独立的配置/发布/版本管理生命周期 | 增量清单输出 `[粒度调整]`——建议拆分。不影响本轮处理，作为建议供人类确认 |
| 新 OR 暗示两个既有配置单元应合并为一个（如两个单元有大量跨单元引用和共享配置） | 同上，建议合并 |

**冲突处理原则**：配置单元的边界通过人类审阅反馈迭代收敛。AI 不自动执行拆分/合并——仅在增量清单中输出 `[粒度调整]` 建议。

与 Step 3 Phase B 四关系检测的区别：
- 比对锚点从"角色+业务对象+职责"三元组 → 配置单元的"配置要素职责范围"
- 新增占位→实体的升级行和未触及行（OR 不触及的已有配置单元保持原样）
- 矛盾判定更收紧（互斥才矛盾，扩展算补充）
- 支持单元内部逐栏标注
- 新增粒度调整建议机制
- 不涉及边界匹配检查（配置单元没有跨业务域问题，引擎边界已由 A.2 引擎模型匹配和 C.1 集中性检查覆盖）

**B.3 逐配置单元处理**

**B.2 与 B.3 的分工**：当 B.2 已执行（复用路径）时，B.3 对 B.2 已处理的配置单元不重复处理——B.2 的关系判定和增量标记即为该单元的本轮结果。B.3 仅处理以下配置单元：
- 候选集中未被 B.2 覆盖的单元（B.2 仅处理已存在于 STR-E 中的配置单元；候选池中其他单元和新路径下的全部候选单元均由 B.3 处理）
- B.2"补充"关系新增的配置单元（在 B.2 中已新增到清单，B.3 中补充设计三栏）——按"OR 明确触及"行处理（完整设计三栏，来源=`OR明示`），不重复标记 `[新增]`（B.2 已标记）

当 B.2 未执行（新建路径）时，B.3 正常处理候选集中的全部配置单元。

对候选集中的每个配置单元，按以下规则处理：

| 触发判定 | 处理动作 | 来源类型 | 增量标记 |
|---------|---------|---------|---------|
| OR 明确触及 | 完整设计三栏（配置要素 + 处理说明 + 业务说明） | `OR明示` | `[新增]` |
| OR 隐含触及（语义推断） | 完整设计三栏，标注推断依据 | `OR隐含` | `[新增]` + `[推断]` |
| 引擎模型标准配置单元，但本轮 OR 未触及 | 仅占位——列出名称和职责说明，三栏空置 | `模型内置` | `[跨场景]` |
| OR 超出已知配置单元类型 | 完整设计三栏，并在推理依据中追加 25 资产扩展建议（格式见下） | `推断` | `[新增]` + `[推断]` + `模型需扩展` |

若引擎能力描述超出了 OR 明示范围且无推断依据，收窄到 OR 明示范围，在增量清单中说明。

**25 资产扩展建议格式**（写入该配置单元的推理依据中）：

```text
💡 25资产扩展建议：建议在 @engine-XXX 下新增配置单元类型"YYYY"：
  - 对应系统实体：<推断的运行时实体>
  - 建议的配置单元类型名称：<名称>
  - 理由：<为什么需要这个配置单元类型，引用 OR 语义>
  - 状态建议：待确认
```

该建议同时作为 wft01b 25 资产写回和 wft02a A1 展开时的输入线索。

**B.4 配置单元设计**

设计任一配置单元的三栏前，AI 先从 25 资产的配置单元类型定义中提取该单元的职责边界（"对应系统实体"和"职责说明"），以此为锚点限定三栏的写入范围。

若在设计中两个配置单元之间出现职责重叠或边界模糊（如校验规则中涉及字段定义、行为配置中涉及权限逻辑），在增量清单中标注 `[边界模糊]` 供人类关注，但不阻断继续设计。单元边界通过人类审阅反馈迭代收敛。

*配置要素*

回答"配置人员在此维度需要设置什么"：
- 需要配置哪些参数/字段（名称、类型、可选值）
- 配置项之间的依赖、互斥或顺序关系
- 配置的 UI 交互形态（下拉框/文本框/表格/树形/向导）
- 校验规则、默认值、模板预设等
- 是否需要版本管理、搜索、筛选等辅助能力

*处理说明*

回答"引擎如何处理这些配置"：
- 配置数据如何校验、存储、解析、编译、发布
- 运行时如何被调度、计算、路由、渲染
- 配置变更如何影响已有运行实例
- 是否需要版本回滚或兼容处理

*业务说明*

回答"配置完成后交付了什么业务能力"：
- 业务人员获得了什么新操作能力
- 能力以什么形态呈现（列表/看板/流程/报表/查询）
- 业务人员如何感知和使用该能力

**I→P→O 连贯性在每组内当场验证**：三栏缺一不可，当组锁定，不跨组。若某栏信息不足，在增量清单中标注缺口并标记 `[阶段性]`。

含 `[阶段性]` 配置单元的 STR-E 节点可生成并推进到 `待确认方案`，但**不得推进到 `可以资产化`**——三栏不全的配置单元不能资产化。

**B.5 配置单元排序与标识**

配置单元排序按以下优先级：

| 优先级 | 依据 | 规则 |
|--------|------|------|
| 1 | 25 资产定义了顺序 | 按资产定义的默认顺序排列 |
| 2 | 存在依赖关系（依赖配置单元前置） | 按依赖拓扑排序——被依赖的排前面 |
| 3 | 无明确顺序亦无依赖 | 按配置的自然时序排列：先结构（定义对象是什么）→ 再字段（对象包含什么）→ 再规则（有什么约束）→ 再行为/权限（怎么交互）→ 最后发布（交付运行） |
| 4 | 以上均不适用 | 按 OR 触及顺序排列，增量清单标注 `[推断]` 排序 |

新增的配置单元按上述规则插入合适位置。优先级 3 和 4 的排序在增量清单中标注 `[推断]` 排序。

每个配置单元的标识：
- 序号（按顺序：1、2、3……）
- 名称
- 来源类型
- 确认状态
- 25 引擎模型：匹配到的 25 资产引擎模型 ID
- 25 配置单元类型：匹配到的配置单元类型 ID
- 25 判定：`25-Q1` / `25-Q2` / `25-Q3`
- 25 资产建议：Q2/Q3 时的扩展/新增建议（Q1 留空）

25-Q1/Q2/Q3 判定规则：B.3 触发判定中的"OR 明确触及 / OR 隐含触及 / 模型内置"对应 25-Q1（配置单元类型已在 25 资产中存在），"超出已知配置单元类型"对应 25-Q3（需新增配置单元类型）。25-Q2 适用于需要扩展已有配置单元类型定义的场景。

#### Phase C：验证与输出

**C.1 跨单元检查**

| 检查项 | 通过条件 | 不通过处理 |
|--------|---------|-----------|
| 覆盖完整性 | OR 触及的引擎能力均已映射到至少一个配置单元 | 遗漏 → 若为复用路径则先经 B.2 四关系检测（判定"补充"），再进入 B.3 补充设计三栏；若为新建路径则直接 B.3 补充新单元 |
| 粒度自洽 | 各配置单元之间粒度一致，无"一个单元 vs 半个单元"的不对等 | 过粗 → 拆分；过细 → 合并（但若满足下方"OR 暗示粒度更细"条件，则不合并，按该规则处理） |
| 单元间无重叠 | 两个配置单元的描述不应覆盖同一组配置要素 | 重叠 → 重新划分单元边界 |
| 集中性 | 所有配置单元围绕同一个引擎模型和同一种抽象业务定义 | 涉及多引擎 → 拆分为多个 STR-E。本轮处理哪一个按 A.1/A.2 的人类选择结果（如无人类选择则默认取候选列表中的第一个）。剩余引擎在增量清单中输出 `[跨场景]` 标记，供人类下一轮选择相关 OR 进入 |

**粒度参考基准**（粒度自洽检查时使用）：

| 优先级 | 参考基准 | 适用条件 |
|--------|---------|---------|
| 1 | 25 资产中同一引擎模型的既有配置单元类型 | 25 资产有 ≥2 个配置单元类型——它们之间的粒度关系即期望粒度 |
| 2 | 引擎模型的"对应系统实体"颗粒度 | 25 资产仅 1 个或无配置单元类型——每个配置单元应对应一个可独立配置、独立发布、独立版本管理的系统实体 |
| 3 | 无可用参考 | 增量清单标注 `[推断]`"配置单元粒度待人类确认" |

**OR 暗示粒度比 25 既有类型更细时的处理**：粒度自洽检查时，若 OR 语义强烈暗示更细的粒度（非 AI 自行拆分），AI 不自动合并。保留细粒度配置单元，标记 `[推断]` + `粒度待确认`，并在推理依据中标注："OR 暗示粒度比 25 资产的既有配置单元类型更细，建议人类确认粒度"。人类确认后，形成 25 资产细化建议。

**C.2 引擎能力边界陈述**

边界检查通过后，输出引擎能力边界陈述：

格式：`「业务定义配置人员」需要为「xx引擎」配置「yy抽象业务定义」的从配置到运行的完整能力，包含 N 个配置单元（单元1 / 单元2 / ……）`

示例：`「业务定义配置人员」需要为「审批引擎」配置「审批业务」的从配置到运行的完整能力，包含 4 个配置单元（审批节点定义 / 流转条件定义 / 时效规则 / 通知规则）`

边界陈述的作用：
- 作为该 STR-E 节点的"一句话边界"，写入 02-*.md 的树画像行和节点索引行
- 作为下游 wft02a A1 路径设计引擎配置单元方案的输入基准

**引擎定义模糊时的处理**：若 25 资产中引擎模型的"定义"字段过于宽泛或不完整，AI 基于本轮 OR 语义 + 配置单元清单反向推导引擎能力边界。边界陈述末尾标注 `（引擎边界待确认：基于本轮 OR 推断，25 资产定义不完整）`，不阻断 STR-E 生成。

输出：引擎能力边界陈述 + 边界检查结论（通过 / 需调整 / 需拆分）

**C.3 推理依据**

按配置单元组织推理依据。每个配置单元的三栏设计独立编号（如 R1、R2 对应配置单元 1 的推理），跨单元推理另行编号。

| 推理类型 | 使用场景 | 输出要求 |
|---------|----------|---------|
| 配置常识推理 | 从配置页面、字段类型、校验规则推断引擎的配置元素需求 | 说明 OR 线索、配置惯例、补全结果 |
| 引擎机制推理 | 从配置→存储→发布→运行期的机制推断引擎需要什么处理能力 | 说明所需机制、实现方式、补全结果 |
| 运行体验推理 | 从业务人员的操作场景推断引擎应提供什么运行能力 | 说明用户场景、能力形态、补全结果 |

推理依据按**主导推理角度**归类（即推理链条中占比最大的那一类），不要求严格互斥。若一个推理依据明显跨两类，标注为复合类型（如 `R3（配置常识 + 引擎机制）`），但编号不分裂——一个推理依据一个 `R#`。

判定原则：问"这个推理的主语是谁"——主语是"配置人员/配置惯例"→配置常识；主语是"引擎/系统"→引擎机制；主语是"业务人员/操作场景"→运行体验。

推理依据采用编号引用。配置要素、处理说明、业务说明的条目应引用对应 R#。


变更摘要供人类追溯和 wft01b 读取，不替代增量清单。

### Step 5 - 新增非功能需求设计

处理 NFR 类型 OR 条目。按分类匹配→量化设计→跨场景验证的三阶段流程，形成 STR-NFR 节点方案。NFR 的核心操作是**定性→定量**转化——将 OR 中的定性描述转化为可测量的量化指标。

#### Phase A：NFR 分类匹配与分组

**A.1 22 资产加载**

从 `22-eos-nfr-taxonomy.md` 加载分类体系全貌——一级类目 → 分类维度 → 具体需求，作为匹配候选池。22 资产采用种子库策略（全量继承通用设计规范附录B + EOS 特有积累），AI 在匹配时应优先匹配 EOS 特有分类，其次附录B 种子分类。

**A.2 逐 OR 分类匹配**

对每条 NFR OR，在 22 资产的分类维度中匹配：

| 判定 | 条件 | 动作 |
|------|------|------|
| Q1（直接匹配） | OR 语义明确对应 22 中某个分类维度 | 归入该分类维度（是否已有 STR-NFR 节点由 A.3 宿主匹配判定） |
| Q2（扩展匹配） | OR 语义是某维度的子类或细化（如 OR 说"数据库查询响应时间"→属于 @nfr-001 系统性能的子类） | 归入该维度，扩展量化指标范围 |
| Q3（新增维度） | OR 语义在 22 现有分类维度中无匹配 | 提出新分类维度建议（含建议 ID、名称、定义、归属一级类目），标记 `[推断]` + `分类体系扩展`。AI 按建议维度继续后续 Phase，人类在方案审阅时一并确认分类建议 |
| Q3+（新增一级类目） | OR 语义连一级类目都不匹配 | 提出新一级类目 + 分类维度建议，标记 `[推断]` + `分类体系扩展`。同上，不单独设置分类确认门禁 |

超出 22 资产时，AI 始终给出分类归属建议方案（建议归属到哪个一级类目/分类维度），不空手上报。

**A.3 STR-NFR 宿主匹配**

以分类维度 ID 在 `02-*.md` 中搜索已有 STR-NFR 节点，确定本轮是复用还是新建：

| 判定 | 条件 | 路由 |
|------|------|------|
| Q1（复用） | 存在同分类维度的 STR-NFR 节点，且 OR 的量化指标在此节点中已有对应 | Phase B 走复用路径（B.1 四关系检测，判定为"确认"或"无新增"） |
| Q2（扩展） | 存在同分类维度的 STR-NFR 节点，但 OR 触及了新的子类指标 | 复用该节点，Phase B 走复用路径（B.1 判定为"补充"） |
| Q3（新建） | 无匹配 STR-NFR 节点 | Phase B 走新建路径（跳过 B.1） |

**A.4 分组聚合**

按分类维度分组——同一维度的 OR 归入同一 STR-NFR 节点。Q3/Q3+ 的 OR 按 AI 建议的新维度分组。无法与其他 OR 聚合的 OR 独立成节点。

若本轮所有 OR 均为 Q3/Q3+，则所有 STR-NFR 均为新增。

**Phase A 输出汇总**：

- A.2 输出：每条 OR 的分类匹配结论（Q1/Q2/Q3/Q3+）+ 对应分类维度 ID
- A.3 输出：每条 OR 的宿主匹配结论（Q1 复用 / Q2 扩展 / Q3 新建）+ 路由方向
- A.4 输出：分组结果——每个 STR-NFR 节点对应哪些 OR

#### Phase B：NFR 量化设计

**B.1 多 OR 收敛检测**（复用/改进既有 STR-NFR 时）

比对锚点：**NFR 分类维度 + 量化指标口径**。

| 关系 | 判定条件 | 处理动作 |
|------|---------|---------|
| **确认** | 新 OR 的 NFR 语义与此前 STR-NFR 的量化指标一致 | 来源升级（`OR隐含`→`OR明示`），追加 OR 引用，增量标记 `[确认]` |
| **补充** | 新 OR 在此维度下新增了指标、测量方法或适用范围 | 新增指标到 STR-NFR，来源=`OR明示`，追加 OR 引用，增量标记 `[新增]` |
| **矛盾** | 新 OR 的量化目标与既有指标存在互斥（如同一指标口径相同但目标值冲突） | 追加矛盾记录，增量标记 `[矛盾]`。存在未解决矛盾时不得推进到 `可以资产化` |
| **无新增** | 完全被覆盖 | 追加 OR 引用 |
| **未触及** | 已有指标不在本轮 OR 的语义范围内 | 保持原样，无动作。不追加 OR 引用，不改变指标状态 |

矛盾记录格式（写入 STR-NFR 节点指标条目下）：

```text
⚠ 矛盾：A版（OR-XXX）："<既有指标口径>" vs B版（OR-YYY）："<新 OR 的指标口径>" → 待人类裁决
```

矛盾必须由人类裁决后才能推进。存在未解决矛盾时，AI 不得将该 STR-NFR 节点推进到 `可以资产化`。

**B.2 量化指标提取**

从 OR 语义中提取可测量指标。核心动作是**定性→定量**转化：

| OR 描述类型 | 量化动作 | 来源标注 | 示例 |
|-----------|---------|---------|------|
| 含具体数值 | 直接采信 | `OR明示` | "响应时间<2秒"→指标=响应时间，目标值=<2s，单位=秒 |
| 含比较词（快/稳定/安全） | AI 基于 22 资产同类指标口径 + 行业常识 + 同类系统经验推断目标值 | `OR隐含` + `[推断]` | "系统要快"→参考同类系统指标推断 |
| 含定性约束（必须/禁止/支持） | 转化为布尔型指标或合规检查项 | `OR明示` | "必须支持 HTTPS"→指标=HTTPS支持，目标值=是 |
| 无任何量化线索 | 标注 `[阶段性]` + 缺口说明 | `待确认` | 不强推断——含 `[阶段性]` 指标的 STR-NFR 不得推进到 `可以资产化` |

**B.3 测量方法**

每个量化指标必须指定测量方法：压测 / 审计检查 / 日志统计 / 模拟 / 外部认证 / 人工检查 / 其他。AI 基于指标类型推断合适的测量方法，标注推断依据。

**B.4 适用范围**

标注该 NFR 约束哪些 STR-F 场景或 STR-E 引擎（通过 STR 节点 ID 引用）。从 OR 语义中提取：
- OR 明确指定适用范围 → 来源=`OR明示`
- OR 未指定 → 基于 NFR 维度的性质推断（如"响应时间"通常约束所有交互场景），标记 `[推断]` + `待确认`
- 完全无法推断 → 标注 `待确认`

#### Phase C：验证与输出

**C.1 跨场景一致性检查**

NFR 是跨场景属性——同一 NFR 维度可能约束多个 STR-F 场景。检查：

| 检查项 | 通过条件 | 不通过处理 |
|--------|---------|-----------|
| 口径一致 | 同一 NFR 维度在不同 STR-F 场景中的指标口径（名称+单位）一致 | 不一致 → 统一口径或标注差异供人类裁决 |
| 目标值不冲突 | 不同场景对同一指标的目标值不互斥（如场景A要求<1s、场景B要求<2s——不冲突，取严格值；场景A要求<1s、场景B要求<3s且≥2s——冲突） | 冲突 → 标注 `[矛盾]` |
| 分类归属 | STR-NFR 归属的分类维度在 22 资产中存在或有 Q3/Q3+ 新增建议 | 无归属 → 回 Phase A Q3 |

**C.2 22 资产扩展建议**

Phase A 产生 Q3/Q3+ 时，在推理依据中追加 22 资产扩展建议——格式与 Step 4 的 25 资产扩展建议对称：

```text
💡 22资产扩展建议：建议新增分类维度"YYYY"：
  - 建议 ID：@nfr-XXX
  - 归属一级类目：<一级类目名称>
  - 定义：<一句话定义>
  - 理由：<为什么需要这个分类维度，引用 OR 语义>
  - 状态建议：待确认
```

**C.3 完善 STR-NFR 要素**

```
STR-NFR 节点：
├── 定义
├── NFR 分类路径（一级类目 / 分类维度，含 22 资产 ID 引用）
├── 承接的 OR 末级
├── 量化指标清单：
│   ├── 指标 1：[名称]
│   │   ├── 目标值
│   │   ├── 单位
│   │   ├── 来源类型（OR明示 / OR隐含 / 推断）
│   │   ├── 测量方法
│   │   ├── 适用范围（约束的 STR-F / STR-E 节点 ID 列表）
│   │   ├── 22 一级类目：匹配到的 22 资产一级类目
│   │   ├── 22 分类维度：匹配到的分类维度 ID
│   │   ├── 22 判定（22-Q1 / 22-Q2 / 22-Q3 / 22-Q3+）
│   │   └── 22 资产建议（Q2/Q3/Q3+ 时的新增/扩展建议，Q1 留空）
│   └── ...
├── 推理依据
├── 适用范围汇总
├── 符合性分析（待下游 Skill 生成——wft01a 不填充此字段）
└── 设计指导（待下游 Skill 生成——wft01a 不填充此字段）
```

**C.4 STR-NFR 写入规则**

按 §三 写入规则执行。节点类型标记为 STR-NFR。当该 NFR 的适用范围涉及多个 STR-F 场景时，AI 应推荐人类在 Run End 中确认各场景的指标承接。多轮 OR 积累导致指标清单变更时，维护变更摘要（格式同 Step 4 C.5 配置单元清单变更摘要）。


### 增量标记输出

Step 3/4/5 新增方案时，以及 Step 2 修订方案后，AI 必须在本轮输出首部附加以 `===` 包裹的增量清单，明确标识本次新增/改进的内容。增量清单供人类阅读和反馈，不写入资产文档。

增量清单格式（新建 STR 时）：

```text
=== STR-F-XXXX 本轮更新清单 ===

[新增] 承接 OR-XXXX, OR-YYYY（共2条）
[新增] IC-XXX "载体名称" [23-Q1] → @comp-买管理:采购申请
[新增] KP-XXX "指标名称" [22-Q2] → 扩展 @nfr-效率

=== 以上为本次全部新内容 ===

请逐项确认，或回复"整体确认同意"。
```

增量清单格式（修订已有 STR 时，含四关系检测结果）：

```text
=== STR-F-XXXX 本轮更新清单 ===

[新增] 承接 OR-005, OR-006（共2条）
[确认] IC-003 "到货通知" OR隐含→OR明示（被 OR-005 证实）[23-Q1]
[矛盾] IC-007 "验收单" 与 OR-005 描述存在分歧
[新增] IC-008 "供应商评估表" 来自 OR-006 [23-Q3] → 建议新增构件

=== 以上为本次全部新内容 ===

请逐项确认，或回复"整体确认同意"。
对有 [矛盾] 标记的项，请回复裁决意见。
对有 [23-Q3] 标记的项，请确认资产新增建议。
```

增量清单格式（Phase B 与 Phase C 同轮执行时，按 STR 节点依次排列，前加总览行）：

```text
=== 本轮共处理 N 个 STR 节点（修订 M 个 + 新建 K 个） ===

=== STR-F-001 本轮更新清单 ===
[确认] IC-003 ...

=== STR-F-002 本轮新增清单 ===
[新增] 承接 OR-007（1条）
[新增] IC-010 ...

=== 以上为本次全部新内容 ===

请逐项确认，或回复"整体确认同意"。
```

八类标记（内容变更 + 资产对准）：

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| `[新增]` | 本次新增的内容 | 新 OR 进入、新增信息载体、新增指标、新增推理依据 |
| `[改进]` | 对既有内容的修改 | 已有节点的功能说明补充、口径调整、边界澄清 |
| `[确认]` | 既有脑补载体被新 OR 证实，来源类型已升级 | 多 OR 收敛——确认关系（`OR隐含`→`OR明示` 等） |
| `[矛盾]` | 新 OR 与既有载体存在分歧，需人类裁决 | 多 OR 收敛——矛盾关系 |
| `[推断]` | Q1/Q2 匹配到 `待确认方案` 节点，匹配基于待确认边界 | Phase A 宿主匹配时宿主边界尚未确认 |
| `[边界扩展]` | Phase B 边界检查中 OR 处于宿主边界边缘 | Phase B 边界匹配检查——边缘情形 |
| `[跨场景]` | Phase C 拆分出的剩余候选场景线索 | 2c 紧凑性检查拆分，本轮未处理 |
| `[删除]` | 移除既有内容 | 少见，一般需在反馈区说明原因 |
| `[23-Q1/23-Q2/23-Q3]` | 信息载体对 23 产品架构资产的对准结果 | STR-F Phase C 对准，在增量清单中逐载体展示 |
| `[25-Q1/25-Q2/25-Q3]` | 配置单元对 25 引擎模型资产的对准结果 | STR-E Phase B 对准，在增量清单中逐单元展示 |
| `[22-Q1/22-Q2/22-Q3]` | NFR 指标对 22 质量需求资产的对准结果 | STR-NFR Phase B 对准，在增量清单中逐指标展示

要求：

1. 增量清单作为 AI 本轮输出的一部分呈现给人类，不写入 `02-*.md` 或 `01-*.md`
2. 增量清单必须标注每项的来源（OR 条目 ID、资产线索等）
3. 人类确认全部内容后，STR 节点状态推进为 `可以资产化`



### Step End - 运行结束输出

Run End 的作用是把本轮运行结果翻译成人类下一步动作的最小提示。AI 不只输出”方案已生成”或”处理完成”，还必须根据当前运行结果，为人类提供一个当前可动作的提示。

### 推荐交互方式

1. 人类优先通过对话回复。
2. AI 先将对话反馈或选择转写到对应 STR 节点的 `### 反馈区`、`01/02` 状态队列或人类决策字段，再继续处理。
3. 对话是反馈输入通道，文档中的反馈区 / 状态队列是权威留痕。
4. AI 不得只凭对话上下文推进状态，必须先完成规范转写。

### Run End 三种提示

处理完本轮 STR 反馈或新 OR 设计后，始终输出以下三种提示（按实际情况标注”有”或”无”）：

```text
当前状态：完成 [角色名] 的 OR 处理
  修订 STR-F-XXX（+N 条 OR）
  修订 STR-F-YYY（+M 条 OR）
  新建 STR-F-ZZZ（聚合 K 条 OR）

你可以：
  A. 增加新的 OR → 选择 OR 条目，重新运行 wft01a
     - 待分配 OR 条目：N 个
  B. 对方案反馈或确认 → 回复「整体确认同意」/「修改：…」/「驳回：…」
     - STR-F-XXX（待确认方案）
     - STR-F-YYY（待确认方案）
     - STR-F-ZZZ（待确认方案）
  C. 运行 wft01b → 可资产化 STR 节点：
     - STR-F-WWW
     - STR-E-VVV
```

提示 A 列出当前可选的 OR 条目及其角色，供人类选择下一批。

提示 B 列出本轮产生或修订的全部 `待确认方案` STR 节点，人类可逐项反馈或整体确认。人类审阅时自行判断 `待确认` 载体的可接受程度——可接受则确认、需验证则线下找业务人员、需更多 OR 则选 A 继续积累。

提示 C 列出全部 `可以资产化` 的 STR 节点，供人类选择进入 wft01b。
