# DevOps 系统设计：提示词模板集
**DevOps System Design: Workflow Prompt Templates**

**用途**：AI 按 4 步流程依次执行，根据问题影响范围选择对应的处理模式。AI 将提示词中的 `[占位符]` 替换为实际内容后执行。

## 依据B说明

本提示词模板集适用于设计线（系统设计流水线）的DevOps场景，默认演示"软件产品"类型的快速修复和局部设计过程。使用时可根据目标产品类型选择相应的PA分类体系：

- **软件产品类型**（如EOS/EOS运营体系）：PA分类为 `前端/后端/数据/基础设施`
- **方法论/工具产品类型**（如设计线/流水线）：PA分类为 `规范与准则/设计指南/AI辅助文档/任务定义`

所有提示词模板中的 `[占位符]` 在执行时替换为实际内容。

---

## 跨步骤上下文管理说明

### 上下文传递方式

DevOps 4 步流程中，第1步选定模式后，第2-4步根据模式选择不同路径执行。AI 在执行每一步前，应确保完整继承了上一步的关键决策：

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 问题分析结果 | 问题分类、优先级、影响范围 | 第1步输出 |
| 处理模式 | 选定模式及选择依据 | 第1步输出 |
| 受影响节点清单 | 各层受影响的节点编号 | 第1步输出 |
| 增量修改内容 | 各模式的 ~/+ 修改条目 | 第2/3/4步输出 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |

### 决策记录机制

每步完成后，AI 将本步的关键决策记录在**本步决策记录模板**中，供后续步骤引用。

### 当需要回溯修正

如果在第 N 步发现问题需要修改第 M 步（M < N）的产出：
1. 评估影响范围（见[AI决策框架 -> 跨步骤影响范围表](../02-ai-decision-framework-eos-pipeline.md#72-跨步骤影响范围表)）
2. DevOps 场景中，回溯最常见的情况是：执行中发现影响范围超出原评估，需要升级模式
3. 确认后重启流程，从第1步重新评估影响范围
4. 标注变更记录

## 第1步：快速影响范围评估

### 目标
快速判断问题的影响范围，选择合适的处理模式（15 分钟内完成）。

### 上下文继承清单

```
从哪里来：
  - 直接输入：问题报告或紧急需求描述（由流水线开发者/EOS开发者提供）
  - 基线输入：现有完整系统设计文档集（5 层 + 映射矩阵）
  - 参考文档：[产品上下文](../01-product-context-eos-pipeline.md)，了解设计线的背景
  - 涉及规范：DevOps 设计指南 §快速决策树

预期产出：
  1. 问题分类和优先级报告
  2. 影响范围评估
  3. 选定的处理模式

本步决策对后续的影响：
  - 影响范围评估决定选用的处理模式
  - 处理模式决定执行路径（第2/3/4步之一）
  - 影响范围判断错误会导致后续全部返工
```

### 输入
- 问题报告或紧急需求描述
- 现有完整系统设计文档集

### 涉及规范

### 决策框架参考

| 决策场景 | 处理方法 | 参考章节 |
|---------|---------|---------|
| 问题分类 | Bug/功能需求/性能优化，按表现特征分类 | 按指南执行 |
| 紧急程度（P0/P1/P2） | 标注 ⚠️ 待确认 | 按指南执行 |
| 影响范围判断 | 逐层检查 PA→SysReq→BA→SR | 按决策树执行 |
| 需求歧义处理 | 可推测则补齐，多选项则列出 | [§4 需求歧义处理策略](../02-ai-decision-framework-eos-pipeline.md#四需求歧义处理策略) |

```

3. 定位受影响层——逐层检查：

   【第5层·产品架构】是否受影响？
   - 检查：是否有具体的 PA 组件出现异常？
   - 如确认 → 记录受影响的 PA 组件编号和名称

   【第4层·系统需求】是否受影响？

   【第3层·业务架构】是否受影响？
   - 检查：是否有 BA IPO 的业务逻辑需要调整？
   - 如确认 → 记录受影响的 BA IPO 编号

   【第2层·相关方需求】是否受影响？
   - 检查：是否有 SR 的需求描述或指标需要修改？
   - 如确认 → 记录受影响的 SR 节点编号

输出格式：
## 问题分类报告
├─ 问题类型：[Bug修复/功能需求/性能优化]
├─ 紧急程度：[P0/P1/P2] ⚠️ 待确认
├─ 问题摘要：[一句话描述]
├─ 现象：[具体现象]

## 影响范围
├─ 受影响的产品架构（PA）：PA-XXX（[原因]）
├─ 受影响的系统需求（SysReq）：SysReq-F-XXX（[原因]）
├─ 受影响的业务架构（BA）：BA-XXX（[原因]）
├─ 受影响的相关方需求（SR）：SR-F-XXX（[原因]）
└─ 受影响程度：[仅PA / PA+SysReq / 跨多层]
```

决策树：
1. 问题是否仅影响产品架构（PA）？

   │

            │       └─ 是 → 【模式3：完整设计】
            │               修改多层（SR/BA/SysReq/PA），2-3天完成

├─ 选定模式：[模式1/模式2/模式3]
├─ 选择依据：[为什么选择该模式]

├─ 涉及文档：[列出需要修改的文档]
└─ 同步设计范围：[列出需要同步更新的映射关系]

## 第2步：模式1 - 快速修复

### 目标
快速修复仅影响产品架构的问题，几小时内完成。

### 上下文继承清单

```
从哪里来：
  - 第1步产出：问题分类、影响范围、选定模式（模式1）
  - 基线输入：现有完整的 PA 文档

去哪里：
  - 本步产出 → 验证结果交付
  - 仅修改 PA 层，不影响上层

特别注意：
  模式1只修改 PA 层，不能涉及上层修改
  如果执行中发现影响超出 PA 范围 → 暂停并上报，升级模式
  使用 ~ 简化格式，但符合性分析必须保留
```

### 适用前提
- 第1步已选定模式1（仅 PA 受影响）
- 已有完整的 PA 文档

### 涉及规范
- [通用设计准则 §五.7 PA节点属性](../../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#57-pa节点属性)

### 决策框架参考

| 决策场景 | 处理方法 | 参考章节 |
|---------|---------|---------|
| 问题定位 | 定位到具体 PA 组件和字段 | 按指南执行 |
| 影响验证 | 检查功能覆盖完整性和性能回退风险 | 按指南执行 |
| 树形结构调整 | 标注 ⚠️ 待确认（模式1中极少使用） | [§3.3 PA构件粒度判断](../02-ai-decision-framework-eos-pipeline.md#33-pa-构件粒度判断) |

## 第3步：模式2 - 局部设计

### 目标
快速设计和实现影响系统需求和产品架构的问题，1 天内完成。

### 上下文继承清单

```
从哪里来：
  - 第1步产出：问题分类、影响范围、选定模式（模式2）
  - 基线输入：现有完整的 SysReq 和 PA 文档

去哪里：
  - 本步产出 → 验证结果交付
  - 修改 SysReq + PA 两层

特别注意：
  模式2不能涉及 BA/SR 层修改
  如执行中发现影响 BA/SR → 暂停并上报，升级为模式3
  务必同步更新 SysReq↔PA 映射关系
  SysReq 功能分支采用 0-9 级结构，非功能采用 0-6 级
```

### 适用前提
- 第1步已选定模式2（影响 SysReq + PA，不影响 BA/SR）
- 已有完整的 SysReq 和 PA 文档

### 涉及规范
- [通用设计准则 §五.6 SysReq条目格式](../../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#56-sysreq条目格式)
- [通用设计准则 §五.7 PA节点属性](../../../../00-general/10-general-system-design-standards/01-general-system-design-standards.md#57-pa节点属性)

### 决策框架参考

| 决策场景 | 处理方法 | 参考章节 |
|---------|---------|---------|
| 影响范围分析 | 定位受影响的 SysReq 节点和 PA 组件 | 按指南执行 |
| SysReq 9级→PA 1:1约束 | 每条9级活动只分配到一个PA组件 | [§2.3 分解停止判断](../02-ai-decision-framework-eos-pipeline.md#23-sysreq-5级9级的分解) |
| 同步设计 | 修改 SysReq 后同步调整 PA | 按指南执行 |
| N:1承接合理性 | 每个PA组件承接2-5个SysReq 9级为合理 | [§3.1 N:1 承接数触发的判断](../02-ai-decision-framework-eos-pipeline.md#31-n1-承接数触发的判断) |
| 资产优先原则 | 变更执行前先确认：能否仅修改现有节点？其次考虑扩展，最后才新增 | [设计原则：组织资产优先](../02-ai-decision-framework-eos-pipeline.md#设计原则组织资产优先) |

## 审查修复流程

### 审查输入
AI 完成指定步骤后，输出产物供 rule-conformance-review 审查。审查报告包含：
- 通过/未通过的项目清单
- 问题定位（涉及的具体文件、章节、编号）
- 问题类型（结构性/内容性/格式性/增量标记/回归影响）

### AI 修复流程
收到审查报告后，AI 按以下优先级处理：

1. **格式性问题**（编号错误、增量标记不规范）→ 直接修复，不需要人工确认
2. **内容缺失**（修改条目缺少符合性分析、验证方法）→ 补充内容，标注修复说明
3. **增量标记错误**（`~`/`+` 混用、遗漏标记）→ 以基线为准修正标记
4. **回归影响**（修改引入了新的问题或破坏了未修改节点）→ 评估后修正，标注 ⚠️ 待确认
5. **结构性问题**（模式选择不当、影响范围遗漏）→ 提出修复方案，标注 ⚠️ 待确认
6. **跨步骤一致性问题** → 参考[AI决策框架 -> 跨步骤影响范围表](../02-ai-decision-framework-eos-pipeline.md#72-跨步骤影响范围表)进行影响范围评估

### 影响范围评估
| 问题定位 | 影响范围 | 修复策略 |
|---------|---------|---------|
| 第1步影响评估错误 | 模式选择 + 后续全部修改链路 | 重新评估影响范围，修正模式选择后重启 |
| 第2步（模式1）修复错误 | 仅 PA 层 | 修正 PA 条目后重新验证 |
| 第3步（模式2）SysReq 修改错误 | PA 映射 | 修正 SysReq 后同步修正 PA 映射 |
| 第3步（模式2）PA 修改错误 | 仅 PA 层 | 局部修正 |
| 第4步（模式3）某层修改遗漏 | 下游全部受影响层 | 补充遗漏的修改后重新验证 |
| 验证发现问题 | 取决于问题定位 | 定位到对应步骤后回溯修复 |
