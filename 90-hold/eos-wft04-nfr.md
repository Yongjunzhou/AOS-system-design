# SR-NFR → SysReq 非功能架构
**Skill ID**: wft04 · **Step 4: SR Non-functional → SysReq Non-functional Architecture**

> **概述**：沿非功能分支独立推进，将两条设计链的 SR-NFR 分别详细定义并量化，在现有 SysReq-NFR 架构资产中找承接宿主，建立系统需求非功能部分架构。本步同步引用 `22-eos-nfr-taxonomy.md` 的非功能需求分类体系和具体需求模板作为量化参考，并将确认后的具体需求（含完整指标集）写回 22 资产，遵循三段式循环：**使用 22 资产归类 → 丰富 22 资产（写回具体需求）→ 修订标注**。

> **输入 → 输出**：SR-NFR 架构末级 + `22-eos-nfr-taxonomy.md` → SysReq-NFR 架构定义（含量化指标 + 分层约束表标注）+ `22-eos-nfr-taxonomy.md` 更新（已确认具体需求条目追加）
>
> **适用场景**：瀑布式系统设计第4步。与功能分支（t02+t03）完全独立并行。
>

### Step 4 补充：分层约束表规范

每个 SysReq-NFR 5 级节点必须带一张供 `wft05a` 读取的**分层约束表**：

| 约束范围 | 约束对象 | 影响级别 | 影响点 | 说明 |
|----------|----------|---------|-------|------|

字段取值规则：

1. `约束范围`：固定为 `全局` / `业务域` / `IPO`
2. `约束对象`：
   - `全局` 时写 `EOS`
   - `业务域` 时写业务域名或能力类名
   - `IPO` 时写具体功能 IPO ID
3. `影响级别`：固定为 `阻断型` / `非阻断型` / `待判定`
4. `影响点`：固定为 `组件类型` / `组件边界` / `接口机制` / `运行约束机制` / `共享归属`
5. `说明`：一句话解释该 NFR 为什么对该对象产生该级别影响

填写约束：

1. 一行只表达一个约束对象的关系；一个 NFR 约束多个对象时写多行
2. 若 `影响级别=阻断型`，则 `影响点` 不可为空
3. 若 `影响级别=待判定`，则 `说明` 必须写明待判原因
4. 当一个 NFR 影响面很大时，应优先使用 `全局` 或 `业务域` 行，不强制展开全部 IPO
