# SysReq 详细定义 → PA 架构
**Skill ID**: wft05 · **Step 5: SysReq Detailed Definition → Product Architecture**

> **概述**：读取 `05-*.md` 管理区 → SysReq 状态表，筛选**当前状态=`待产品架构` + 人类决策=`同意产品架构`** 的 SysReq 功能架构末级节点（IPO），并读取 `07-*.md` 管理区 → PA 状态表筛选**迭代中**的 PA 节点 → 在 `06-*.md` 中将 SysReq 末级节点展开为可分配的详细定义条目 → 读取现有 PA 资产与 `wft03` 建立的 PA 骨架 → 按 IPO 类型生成对应的 PA 末级组件：**业务定义类 IPO → 配置方案组件**，**业务引擎能力类 IPO → 前后端组件** → 映射 `wft04` 产出的 SysReq-NFR 约束 → 在 `07-*.md` 中完成 PA 架构定义、组件状态回写与反馈处理。
>
> **核心变化**：本步不再采用旧的“双链并行 + SysReq 5→9 级统一分解”叙事，而是统一消费 `wft03` 已收敛、并已获人类同意进入产品架构的 SysReq 末级 IPO，再按 IPO 类型执行两套下沉规则：业务定义类 IPO 生成配置方案组件，业务引擎能力类 IPO 生成前后端组件。同时明确：**业务定义类 IPO 所生成的配置方案信息，构成业务引擎能力类 IPO 的核心输入**。
>
> **输入 → 输出**：SysReq 架构定义（`05-*.md`） + SysReq-NFR 约束（`05-*.md` / `06-*.md`）→ SysReq 详细定义（`06-*.md`） + PA 架构定义（`07-*.md`）
>
> **适用场景**：瀑布式系统设计第5步。本步合并功能分支（`wft03`）与非功能分支（`wft04`），并完成 SysReq→PA 的桥接重组与组件生成。

## 一、设计角色

本步在设计链中承担的角色：
- **输入**：`05-*.md` 管理区 → SysReq 状态表中 **当前状态=`待产品架构` + 人类决策=`同意产品架构`** 的 SysReq 末级节点；同时读取 `07-*.md` 管理区 → PA 状态表中 **当前状态=`迭代中`** 的 PA 节点作为反馈重处理输入
- **产出1**：SysReq 详细定义（`06-*.md`）——围绕进入本轮的 SysReq 末级 IPO 展开为可分配条目
- **产出2**：PA 架构定义（`07-*.md`）——按 IPO 类型生成 PA 末级组件并组织产品架构树
- **产出3**：PA 组件状态更新——将本轮形成的 PA 组件写回 PA 状态表，供人类确认与后续 `t06` 追溯验证使用

本步按 IPO 类型生成两类 PA 末级组件：
- **业务定义类 IPO → 配置方案组件**：流程定义组件、表单定义组件、工单定义组件、计划定义组件、指标定义组件等
- **业务引擎能力类 IPO → 前后端组件**：前端配置组件、前端运行组件、后端定义处理组件、后端运行执行组件、后端接口与集成组件

### 与三条工程化设计原则的关系

| 原则 | 在本步中的体现 |
|------|--------------|
| **详概分离** | 本步产出 SysReq 层的**详细定义子文档**（`06-*.md`）和 PA 层的**架构定义子文档**（`07-*.md`）。`06-*.md` 承载可分配条目，`07-*.md` 承载组件树、组件详细定义、映射关系和状态表 |
| **文档结对** | 本步执行结对4（SysReq 详细定义末级 ↔ PA 架构）。先从 `wft03` 产出的 SysReq 末级 IPO 生成详细条目，再将条目分配到 PA 组件 |
| **基于资产** | PA 节点不是从零创建的；优先复用 `wft03` 同步建立的 PA 骨架与现有 `07-*.md` 组件资产。配置方案组件优先复用已有配置宿主；前后端组件优先复用已有引擎构件 |

### 基于已有资产的工作原则

`wft05` 不是“从需求直接发明组件”，而是基于已有资产开展的组件生成与组件落账工作。这里的已有资产至少包括：

1. `wft03` 已产出的 SysReq 末级 IPO；
2. `wft03` 已同步建立的 PA 骨架；
3. `07-*.md` 中已有的 PA 组件资产；
4. `23/24/25/26` 中已有的输出产品、业务定义、业务引擎和文档等相关资产。

因此，本步的基本动作不是“自由设计”，而是：

`读取就绪 IPO → 对照现有 PA 资产 → 决定复用 / 改进 / 新增 → 写回 PA`

### 4.4 PA 末级节点内容要求

PA 末级节点下的内容，不应只停留在“节点命名”和“一句定义”，而必须达到可直接支撑后续开发的粒度。

#### 4.4.1 配置方案组件末级节点的内容要求

对配置方案组件，至少应包含以下内容：

1. **组件定义**：说明该组件是什么、承接哪个业务定义类 IPO；
2. **承接的上游条目**：明确承接哪些 SysReq 详细定义条目；
3. **配置方案结构**：列出元数据、结构、规则、权限、展示、引用、校验发布等配置块；
4. **输入输出**：输入什么配置数据，输出什么业务定义结果或可加载定义；
5. **依附引擎**：明确依附哪个业务引擎运行；
6. **约束与校验**：明确完整性校验、边界约束、版本/发布条件；
7. **符合性分析**：说明为何足以承接上游需求；
8. **开发/配置指导**：给配置方案开发者可直接执行的指导。

一句话要求：配置方案组件末级节点应达到“**可开发的定义宿主规格**”粒度。

#### 4.4.2 前后端组件末级节点的内容要求

对前后端组件，至少应包含以下内容：

1. **组件定义**：明确这是前端配置组件、前端运行组件、后端定义处理组件、后端运行执行组件，还是接口与集成组件；
2. **承接的上游条目**：明确承接哪些 SysReq 详细定义条目；
3. **职责边界**：说明负责什么、不负责什么；
4. **输入**：尤其说明是否接收配置方案信息、运行上下文、外部事件或接口输入；
5. **输出**：说明输出页面能力、定义生成结果、执行结果、接口响应、事件等；
6. **内部处理逻辑**：说明解析、校验、组装、执行、调度、渲染、交互等关键逻辑；
7. **对外接口**：说明 API、消息、回调、存储关系、依赖服务；
8. **NFR 约束**：说明性能、安全、可用性、并发、审计等要求；
9. **开发指导**：给前端/后端开发者可直接展开的指导。

一句话要求：前后端组件末级节点应达到“**开发可执行定义**”粒度。

### 4.5 PA 非末级节点生成规则

PA 的非末级节点，不是预先想好一棵树再去硬塞末级节点，而是从已形成的末级组件集合中反推生成的稳定宿主结构。

#### 4.5.1 总原则

`先有末级组件集合，再生成非末级分类宿主。`

#### 4.5.2 生成规则

1. **先按组件对象类型分大类**：如配置方案组件类、前端组件类、后端组件类；
2. **再按职责稳定性分中类**：如业务定义配置组件、引擎配置前端组件、引擎运行前端组件、引擎定义处理后端组件、引擎运行执行后端组件、接口与集成后端组件；
3. **仅在边界稳定时生成非末级节点**：只有当一组末级节点已经具备稳定命名、稳定边界、稳定复用意义时，才抽象出非末级宿主；
4. **非末级节点同样遵循复用 > 改进 > 新增**：优先复用现有宿主，不够再改，最后才新增。

一句话要求：PA 非末级节点的本质，是“**对一组同类末级组件的稳定分类宿主**”。

## 四、AI 执行指南

### 4.1 跨步骤上下文管理

| 继承内容 | 说明 | 保存位置 |
|---------|------|---------|
| 架构决策 | 分类方式、IPO 类型、组件边界 | 上一步的决策记录 |
| 编号体系 | 所有已分配的 ID 和编号规则 | 各产品数据文件 |
| 待确认项 | 上一步标记的 ⚠️ 项 | 汇总清单 |
| AI推断记录 | AI 自行判断补齐的内容 | 决策记录文件的 AI推断项 |

### 4.2 AI 经济原则

1. **执行顺序不可逆**：前置校验 → 资产加载 → 分析活动 → 详细定义展开 → IPO→PA 组件生成 → PA 写回 + NFR 映射 → 闭环检查
2. **先判 IPO 类型，再生 PA 组件**：每个进入本轮的 SysReq 末级 IPO，都必须先判断是业务定义类 IPO 还是业务引擎能力类 IPO，再决定其 PA 末级组件类型
3. **业务定义类 IPO 生成配置方案组件**：主要产物是可被引擎加载的具体业务定义时，PA 末级节点应落为配置方案组件
4. **业务引擎能力类 IPO 生成前后端组件**：主要产物是可被多个业务定义复用的通用系统能力时，PA 末级节点应拆为前端组件、后端组件或两者组合
5. **配置方案信息是业务引擎核心输入**：业务定义类 IPO 所生成的配置方案信息，构成业务引擎能力类 IPO 在定义生成与运行承接阶段的主输入
6. **基于资产优先**：配置方案组件优先复用已有配置宿主；前后端组件优先复用已有引擎构件，确实不足时再改进或新增

### 4.3 IPO→PA 末级组件生成规则

#### 4.3.1 类型判定

对每个 SysReq 末级 IPO，先判定其对象类型：

| IPO 类型 | 判定标准 | PA 末级组件类型 |
|---------|---------|----------------|
| **业务定义类 IPO** | 目标是产出一个具体业务定义；需求主要落在“定义内容怎么配” | **配置方案组件** |
| **业务引擎能力类 IPO** | 目标是提供或增强一类业务引擎承接能力；需求主要落在“系统能力怎么实现” | **前后端组件** |

#### 4.3.2 业务定义类 IPO → 配置方案组件

处理步骤：

1. 识别其依附的业务引擎；
2. 识别要产出的业务定义对象；
3. 将 IPO 需求展开为配置方案信息；
4. 组织为一个或多个配置方案组件。

与业务引擎能力类 IPO 的关系：

1. 业务定义类 IPO 所生成的配置方案信息，构成后续业务引擎能力类 IPO 的核心输入；
2. 因此，本类组件不是孤立产物，而是“供业务引擎消费的定义信息载体”；
3. `wft05` 在继续处理业务引擎能力类 IPO 时，应将这些配置方案信息视为其定义生成与运行承接阶段的主输入。

建议优先采用“**1 个主定义组件 + 7 类子组件**”的结构：

| 组件层 | 组件名称 | 职责 |
|------|---------|------|
| 主组件 | **主定义组件** | 作为该业务定义的核心承载单元 |
| 子组件1 | **元数据组件** | 管理编号、名称、版本、状态、适用范围、归属域 |
| 子组件2 | **结构组件** | 管理字段结构、流程节点结构、指标维度结构等静态骨架 |
| 子组件3 | **规则组件** | 管理业务规则、流转规则、计算规则、校验规则 |
| 子组件4 | **权限组件** | 管理查看、配置、发起、审批、维护等权限范围 |
| 子组件5 | **展示组件** | 管理页面布局、列表视图、字段展示、菜单挂接、场景挂接 |
| 子组件6 | **引用集成组件** | 管理依赖的其他业务定义、引擎能力、外部数据源 |
| 子组件7 | **校验发布组件** | 管理完整性检查、冲突检查、发布条件、生效条件 |

一句话规则：`业务定义类 IPO + 需求 → 配置方案组件`。

#### 4.3.3 业务引擎能力类 IPO → 前后端组件

处理步骤：

1. 识别新增或增强的是哪类引擎能力；
2. 识别其核心输入，尤其是由业务定义类 IPO 提供的配置方案信息；
3. 判断该能力落在前端、后端，还是前后端共同承接；
4. 按技术职责拆成 PA 末级组件。

建议优先采用“**1 个能力主组件 + 5 类承接组件**”的结构：

| 组件层 | 组件名称 | 职责 |
|------|---------|------|
| 主组件 | **能力主组件** | 作为该业务引擎 IPO 的核心承接单元 |
| 子组件1 | **前端配置组件** | 承接配置页面、设计器、配置器、参数编辑器、规则编辑器 |
| 子组件2 | **前端运行组件** | 承接运行期输入页面、操作页面、结果页面、查看页面 |
| 子组件3 | **后端定义处理组件** | 承接配置信息解析、校验、组装、版本化、业务定义生成 |
| 子组件4 | **后端运行执行组件** | 承接业务定义解释、调度、执行、状态流转、结果产出 |
| 子组件5 | **后端接口与集成组件** | 承接前后端 API、外部集成、消息、存储访问、事件通知 |

一句话规则：`业务引擎能力类 IPO + 需求 → 前后端组件`。

#### 4.3.4 禁止跨类生成

1. 业务定义类 IPO 不直接落前后端组件；
2. 业务引擎能力类 IPO 不直接落配置方案组件；
3. 若某业务定义类 IPO 在上游仍存在引擎缺口，应按 `wft02` 的引擎缺口机制先回到 A1 路径处理，而不是在本步混写 PA 类型。
