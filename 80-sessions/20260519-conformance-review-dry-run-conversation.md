## AI

基于准则 v3.6，我的理解如下：

## 功能需求与非功能需求的定位

准则对 FR/NFR 的关系界定为**平行并列关系**，而不是派生关系。NFR 不从 FR 派生，也不从 FR 的性能指标派生。

### §1.9.1 两条共同前提

**任何需求（无论 FR 还是 NFR）都由两部分构成：**
- 需求名称及定性说明
- 性能指标（集合）

也就是说，"性能指标"是需求的通用属性，不是 NFR 的专属。

### §1.9.2 核心辨析——四条界限

| 要点 | 我的理解 |
|------|---------|
| NFR 不从 FR 派生 | 不能把 NFR 理解成 FR 的"附属品质"。例如"系统响应时间 < 2s"不是"需求录入功能"派生出来的，它是独立的需求条目 |
| FR 和 NFR 在同一层面并列 | 在 SR 层，FR 和 NFR 是 SR 架构下的两个平行分支（SR-F-XXX 和 SR-NFR-XXX），不是主从关系 |
| FR 自带性能指标与 NFR 自身指标并列 | 例如一条 FR 的"录入时间 < 1s"和一条 NFR 的"可用性 99.9%"互不隶属，是并列关系 |
| NFR 在 SR 层以后走不同的分配路径 | 这是最关键的设计决策——**NFR 不经过 BA**，直接到 SysReq-NFR |

### 两条路径的分离

```
SR 层：
├─ SR-F（功能部分）→ BA（第3层）→ SysReq（第4层功能）→ PA（第5层）
└─ SR-NFR（非功能部分）─→ SysReq-NFR（第4层非功能）→ PA（第5层）
```

这条分离路径的设计意图：防止 NFR 在 BA 层（业务过程说明）中被稀释或丢失，因为 BA 描述的是"用户如何用系统做事"，天然不承载"做得多好"这类约束。

### NFR 分类体系

准则 §1.9.3 将 NFR 分为三大类：

| 大类 | 视角 | 示例 |
|------|------|------|
| **质量特性** | 用户/运维视角（运行时品质） | 性能、安全性、可靠性、可用性、易用性 |
| **环境适应性** | 外部环境视角（变化适应） | 可扩展性、弹性、可移植性、互操作性 |
| **可实现性** | 开发/交付视角（生命周期便利性） | 可测试性、可部署性、可运维性、经济可承受性 |

### 在五层中的具体体现

| 层 | FR 的处理 | NFR 的处理 |
|---|----------|-----------|
| 第1层（OR） | 来源混杂，为初步分离做准备 | 来源混杂，标注性能指标 |
| 第2层（SR） | 架构末级 SR-F-XXX | 架构末级 SR-NFR-XXX，正式分类组织 |
| 第3层（BA） | IPO 承接 SR-F 详细定义 | **不经过 BA** |
| 第4层（SysReq） | IPO 去重 → 功能 SysReq（0-9 级） | 直接承接 SR-NFR → NFR SysReq（0-6 级平行架构） |
| 第5层（PA） | 构件承接 SysReq 详细定义 | NFR 约束构件对品质的符合性 |

### 实际操作中的误区

**最容易混淆的点**：在 OR 层，一条原始需求可能同时包含 FR 和 NFR。例如"需求录入应在 2 秒内保存"——这在 OR 层**不要急于分离**，保持一条完整记录。到 SR 层再正式拆分为：
- SR-F-XXX：需求录入功能（功能描述）
