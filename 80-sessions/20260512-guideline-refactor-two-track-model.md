## 🎯 核心改动：两轨需求模型

### 功能需求流程（Functional Requirements Track）

```
SR-F（相关方功能需求）
  ↓ [准则1：规范化、分解、初步BA设计]
BA（业务架构）
  ↓ [准则2：分配]
SysReq-F（系统功能需求）
  ↓ [准则3：分配]
PA（产品架构）
```

**特点**：
- 经过完整的四层架构
- 每层都有明确的分配关系
- 支持多对一映射和1:1约束
- 完整的需求追溯链

### 非功能需求流程（Non-Functional Requirements Track）

```
SR-NF（相关方非功能需求）
  ↓ [准则1：规范化、分解、标记为约束]
SysReq-NF（系统约束）
  ↓ [准则3：标记为约束]
PA（产品架构）
  ↓ [准则4：验证约束满足]
  ↓ [如不满足 → 反向迭代]
```

**特点**：
- 不经过业务架构
- 在系统需求层面标记为系统约束
- 作为产品架构的整体约束条件
- 支持反向迭代机制

## 📝 准则 1：相关方需求规范化与分类准则

### 主要改动

#### 1. 新增核心概念部分

**1.1 功能需求 vs 非功能需求**
- 功能需求：系统应该做什么（业务流程、业务规则）
- 非功能需求：系统应该如何做（性能、安全、可用性等）
- 关键区别表：展示两类需求的承接路线、分配方式、验收方式

**1.3 需求分类矩阵**
- 展示两轨流转路径的完整图示
- 功能需求 → 业务组件 → 配置改进/引擎改进/引擎开发
- 非功能需求 → 系统约束 → 产品架构调整

#### 2. 规范化规则分离

**2.1 功能需求规范化**
- 使用 SR-F-XXX 格式
- 包含 9 个必填字段
- 验收标准必须具体、可测量

**2.2 非功能需求规范化**
- 使用 SR-NF-XXX 格式
- 新增"需求类型"字段（性能/安全/可用性/可扩展性/其他）
- 必须包含具体的量化指标

#### 3. 分解规则分离

**3.1 功能需求分解**
- 达到原子需求级别
- 每个分解后的需求能映射到业务架构的唯一节点

**3.2 非功能需求分解**
- 按需求类型分解（性能、安全、可用性等）
- 按影响范围分解（系统级、模块级、组件级）
- 每个分解后的需求应有明确的量化指标

#### 4. 新增第 4 节：非功能需求标记为系统约束

- 非功能需求不经过业务架构
- 直接分配到系统需求架构中的非功能节点
- 作为产品架构的约束条件
- 包含约束标记表和记录格式

#### 5. 初步设计 BA 承接层（仅功能需求）

- 第 5 节明确说明仅针对功能需求
- 更新所有示例使用 SR-F-XXX 格式
- 三层优先级：配置改进 → 引擎改进 → 引擎开发

#### 6. 冲突/重复检测（仅功能需求）

- 第 6、7 节明确说明仅针对功能需求
- 非功能需求的冲突在系统需求层面处理

## 🔄 反向迭代机制（关键创新）

### 问题背景

在传统的需求流转模型中，产品架构是最终的实现层，一旦产品架构无法满足非功能约束，就会陷入困境。

### 解决方案

引入**反向迭代机制**：

```
产品架构调整
   ↓
影响系统需求 → 更新系统需求
   ↓
影响业务架构 → 更新业务架构
   ↓
影响相关方需求 → 更新相关方需求
   ↓
重新验证完整的追溯链
```

### 实现方式

- 准则 4 第 4 节明确说明约束不满足时的处理流程
- 准则 4 第 7 节同步设计步骤中添加"验证约束满足"
- 准则 4 Q6 常见问题中详细说明反向迭代的具体步骤

## 📦 业务组件与引擎组件详细说明

### 三层架构

```
引擎组件（基础层）
   ↓ [以UI配置方式生成]
业务组件定义（配置层）
   ↓ [实例化为具体业务]
业务实例（运行层）
```

### 引擎组件类型

| 引擎类型 | 功能说明 |
|--------|--------|
| 流程引擎 | 支持流程定义、流程实例、流程监控 |
| 计划引擎 | 支持计划定义、计划执行、计划追踪 |
| 工单引擎 | 支持工单定义、工单分配、工单跟踪 |
| 指标引擎 | 支持指标定义、指标计算、指标展示 |
| 表单引擎 | 支持表单定义、表单渲染、数据收集 |
| 关系引擎 | 支持关系定义、关系维护、关系查询 |

### 业务组件定义示例

**流程定义**（由流程引擎配置生成）
- 订单审批流程定义 → 实例化为具体订单审批流程
- 采购审批流程定义 → 实例化为具体采购审批流程

**计划定义**（由计划引擎配置生成）
- 销售计划定义 → 实例化为具体销售计划
- 生产计划定义 → 实例化为具体生产计划

**工单定义**（由工单引擎配置生成）
- 维修工单定义 → 实例化为具体维修工单
- 服务工单定义 → 实例化为具体服务工单

**指标定义**（由指标引擎配置生成）
- 销售指标定义 → 实例化为具体销售指标
- 成本指标定义 → 实例化为具体成本指标

### 关键概念

**业务组件定义**
- 通过引擎提供的配置界面定义业务规则、流程、指标等
- 是一个模板或配置，不是具体的业务
- 可以被多个业务实例复用

**业务实例**
- 业务组件定义必须实例化为具体的业务才能运行
- 是业务组件定义的具体运行实例
- 包含具体的业务数据和状态

**配置优于开发**
- 业务组件通过配置生成，无需代码开发
- 快速迭代和灵活调整
- 降低开发成本和风险

## 🎓 核心学习点

### 1. 两轨需求模型的必要性
- 功能需求和非功能需求有完全不同的处理路径
- 混淆两者会导致架构设计混乱

### 2. 业务架构的真正职责
- BA 代表用户视角下的业务流程
- BA 只能处理功能需求，不能处理非功能需求

### 3. 非功能约束的处理方式
- 非功能约束不分配到节点，而是作为整体约束
- 产品架构需要验证是否满足约束

### 4. 反向迭代的重要性
- 产品架构调整需要同步迭代上游
- 保证完整的追溯链一致性

### 5. 业务组件与引擎组件的关系
- 引擎是基础，业务组件定义是配置，业务实例是运行
- 体现"配置优于开发"的原则
